# Контекст приложения
Файл с контекстом приложения выглядит обычным образом
```xml
<?xml version="1.0" encoding="UTF-8"?>
<beans xmlns="http://www.springframework.org/schema/beans"
	xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
	xmlns:context="http://www.springframework.org/schema/context"
	xmlns:mvc="http://www.springframework.org/schema/mvc"
	xsi:schemaLocation="http://www.springframework.org/schema/beans 
	http://www.springframework.org/schema/beans/spring-beans.xsd
	http://www.springframework.org/schema/context 
	http://www.springframework.org/schema/context/spring-context.xsd
	http://www.springframework.org/schema/mvc 
	http://www.springframework.org/schema/mvc/spring-mvc-4.0.xsd">
 
<context:component-scan base-package="ru.my.project" />
  
<mvc:annotation-driven />
  
<bean id="templateResolver"
	class="org.thymeleaf.spring5.templateresolver.SpringResourceTemplateResolver">
	<property name="prefix" value="/WEB-INF/" />
	<property name="suffix" value=".html" />
</bean>
 
<bean id="templateEngine" class="org.thymeleaf.spring5.SpringTemplateEngine">
	<property name="templateResolver" ref="templateResolver" />
	<property name="enableSpringELCompiler" value="true" />
</bean>

<bean class="org.thymeleaf.spring5.view.ThymeleafViewResolver">
	<property name="templateEngine" ref="templateEngine" />
	<property name="order" value="1" />
	<property name="viewNames" value="*" />
</bean>

</beans>
```

Здесь используется ==НОВЫЙ== тег
> [!NOTE] Тег
> Тег `<mvc:annotation-driven/>` в конфигурации Spring MVC используется для автоматического ==включения поддержки== (НЕ ИЗБАВЛЯЕТ ОТ import) ==аннотаций==, связанных с обработкой HTTP-запросов. Этот элемент упрощает настройку приложения, позволяя активировать различные функции Spring MVC без необходимости явно настраивать их вручную.

==! Он включает в себя некоторые **основные** ФУНКЦИИ !:==
**Поддержка аннотаций контроллеров**:
- Автоматически включает поддержку таких аннотаций, как `@RequestMapping`, `@GetMapping`, `@PostMapping`, `@PathVariable`, `@RequestParam`, `@RequestBody`, `@ResponseBody` и других, которые используются для маршрутизации запросов и привязки данных к методам контроллера.
- Это позволяет вам использовать контроллеры, аннотированные с `@Controller` или `@RestController`, и автоматически связывать входящие HTTP-запросы с соответствующими методами.

А также создаются три бина. Эти бины необходимы для работы шаблонизатора Thymleaf 
==(СМ пометки в коде)==
```xml
<bean id="templateResolver" 
	class="org.thymeleaf.spring5.templateresolver.SpringResourceTemplateResolver">
	<property name="prefix" value="/WEB-INF/"/> !!!Указания МЕСТА с views!!!
	<property name="suffix" value=".html"/> !!! Расширение файлов представлений
</bean>

<bean id="templateEngine" class="org.thymeleaf.spring5.SpringTemplateEngine">
	<property name="templateResolver" ref="templateResolver"/>
	<property name="enableSpringELCompiler" value="true"/>
</bean>

<bean class="org.thymeleaf.spring5.view.ThymeleafViewResolver">
	<property name="templateEngine" ref="templateEngine"/>
	<property name="order" value="1"/>
	<property name="viewNames" value="*"/>
</bean>
```

![[Pasted image 20241026161220.png]]

> [!NOTE] ВАЖНО!
> Tomcat считывает информацию с файла `web.xml`. 
> В этом файле находится описание того, что сервер должен делать.

Сначала запрос будет отравляться на сервер Tomcat
> [!NOTE] Настройка Tomcat
> Необходимо сказать серверу, чтобы он все HTTP-запросы отправлял на Dispathcer сервлет, который в свою очередь перенаправит запрос в Spring приложение 
### Файл web.xml
Для начала необходимо начать с описания метаданных файла
```xml
<?xml version="1.0" encoding="UTF-8"?> 
<web-app xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" 
	xmlns="http://xmlns.jcp.org/xml/ns/javaee" 
	xsi:schemaLocation="http://xmlns.jcp.org/xml/ns/javaee 
	http://xmlns.jcp.org/xml/ns/javaee/web-app_3_1.xsd" 
	id="WebApp_ID" version="3.1">
```

**`<web-app>`** - Корневой элемент, который определяет веб-приложение. 
==Атрибуты внутри:==
- `xmlns`- описывает пространство имён для спецификаций Java EE.
	
- `xmlns:xsi`- Определяет пространство имён для схемы XML, что позволяет использовать `xsi:schemaLocation`.
	
- **`xsi:schemaLocation`**- Указывает местоположение схемы XML, которая используется для валидации файла `web.xml`.
	
- `id`- Идентификатор веб-приложения, может быть произвольным.
	
- `version`- Версия спецификации Servlet API, здесь это `3.1`.

Схема XML выполняет роль спецификации для XML-документов, чтобы обеспечить их правильную структуру и валидность.


- Далее идет Отображаемое имя приложения
```xml
<display-name>spring-mvc-app1</display-name>
```
`<display-name>`- Указывает отображаемое имя приложения. Это имя может отображаться в административных интерфейсах серверов приложений (например, в консоли управления Tomcat).

- Далее идет Абсолютный порядок загрузки
```xml
<absolute-ordering/>
```
**`<absolute-ordering/>`**: Элемент, который может использоваться для строгого определения порядка загрузки фрагментов веб-приложений. Здесь он пуст, что означает, что порядок загрузки будет обычным по умолчанию.

- ==Определение сервлета==
```xml
<servlet> 
	<servlet-name>dispatcher</servlet-name> 
	<servlet-class>org.springframework.web.servlet.DispatcherServlet</servlet-class> 
	<init-param> 
		<param-name>contextConfigLocation</param-name> 
		<param-value>/WEB-INF/applicationContextMVC.xml</param-value> 
	</init-param> 
	<load-on-startup>1</load-on-startup> 
</servlet>
```

 **`<servlet>`**: Описывает сервлет, который будет использован в приложении.
- **`<servlet-name>`**: Задаёт имя сервлета — здесь он называется `dispatcher`. Это имя будет использоваться для ссылки на этот сервлет в других частях конфигурации.
	
- **`<servlet-class>`**: Полное имя класса сервлета. В данном случае используется `DispatcherServlet` из Spring MVC, который является основным контроллером для маршрутизации HTTP-запросов.
	
- **`<init-param>`**: Параметр инициализации для сервлета:
    - **`<param-name>`**: Имя параметра инициализации — `contextConfigLocation`.
    - **`<param-value>`**: Значение параметра — ==путь к файлу== конфигурации Spring `/WEB-INF/applicationContextMVC.xml`. Этот файл загружается при старте приложения и используется для настройки ==Spring-контекста==.
    
- **`<load-on-startup>`**: Определяет приоритет загрузки сервлета. Значение `1` означает, что сервлет будет загружен сразу при запуске приложения (если значение больше 0), а не при первом обращении к нему. Чем меньше значение, тем раньше он загружается.

- ==Маппинг сервлета==
```xml
<servlet-mapping> 
	<servlet-name>dispatcher</servlet-name> 
	<url-pattern>/</url-pattern> 
</servlet-mapping>
```
**`<servlet-mapping>`**: Указывает, каким URL будет соответствовать данный сервлет:

- **`<servlet-name>`**: Ссылается на ранее определённый `dispatcher` сервлет.
- **`<url-pattern>`**: Определяет шаблон URL, который будет обрабатываться этим сервлетом. Здесь используется `/`, что означает, что `DispatcherServlet` будет обрабатывать все запросы к приложению.

## Замена файла web.xml
Начиная с версии Spring 3.0 данный файл можно заменить Java-классом
Такой ==класс== также ==будет считываться и работать== точно также ==как и web.xml==

> [!NOTE] Интерфейс заменяющий Web.xml
> Данный файл будет заменен на Java-класс, реализующий интерфейс ==WebApplicationInitializer== из пакета (org.springframework.web.WebApplicationInitializer)
> 
> Здесь есть два варианта:
> 
> 1.Реализовать интерфейс `WebApplicationInitializer` 
> В методе` void onStartup(ServletContext container)` описывается то же самое что и в web.xml (dispatcherServelt)
> 
> 2.Унаследовать абстрактный класс который уже реализовал этот интерфейс
>`AbstractAnnotationConfigDispatcherServletInitialiser` представленный в Spring 3.2

Для этого создадим ==новый пакет== в папке src/main/java и назовем его ==myconfig==, в нем будет находится наш конфигурационный класс заменяющий файл web.xml

```java
package myconfig;

import org.springframework.web.servlet.support.AbstractAnnotationConfigDispatcherServletInitializer;

public class MyWebInitializer extends AbstractAnnotationConfigDispatcherServletInitializer{

	@Override
	protected Class<?>[] getRootConfigClasses() {
		// TODO Auto-generated method stub
		return null;
	}
	@Override
	protected Class<?>[] getServletConfigClasses() {
		// TODO Auto-generated method stub
		return null;
	}
	@Override
	protected String[] getServletMappings() {
		// TODO Auto-generated method stub
		return null;
	}
}
```
Созданный класс унаследован от абстрактного класса (см выше), и должен реализовать его абстрактные методы

> [!NOTE] getServletConfigClasses
> Данный метод вернет класс, замещающий контекст приложения Spring (annotationContextMVC.xml) 

==Лишь после того как создадим конфигурационный класс, подставим его в метод getServletConfigClasses()== 

Теперь изменим метод
```java
protected Class<?>[] getServletConfigClasses() {
	// TODO Auto-generated method stub
	return new Class[] {MyApplicationContext.class};
}
```

В этой строке кода создается новый массив классов с помощью конструкции 
	`new Class[] { ... }`
Это означает, что вы создаете массив, который будет содержать типы классов, а не их экземпляры.
В данном случае массив содержит только один элемент: `MyApplicationContext.class`.

Также теперь нам необходимо привязать наш класс, заменяющий web.xml (==по сути заменяющий DispatcherServlet==), к определенному URL, который он будет обрабатывать
```java
@Override
protected String[] getServletMappings() {
	// TODO Auto-generated method stub
	return new String[] {"/"};
}
```

> [!NOTE] ==Маппинг Dispathcer Servlet==
> Если необходимо связать Dispatcher Servelt с определенным адресом (например `hello`) то необходимо написать:
> `/hello/*`

## Замена applicationContextMVC.xml
Для начала класс конфигуратор контекста необходимо создать
```java
@Configuration
@ComponentScan("ru.my.project")
@EnableWebMvc

public class MyApplicationContext implements WebMvcConfigurer{
	private final ApplicationContext applicationContext;
	@Autowired
	public MyApplicationContext(ApplicationContext applicationContext) {
		this.applicationContext = applicationContext;
	}
	@Bean
	public SpringResourceTemplateResolver templateResolver() {
		SpringResourceTemplateResolver templateResolver = new 
		SpringResourceTemplateResolver();
		
		templateResolver.setApplicationContext(applicationContext);
		templateResolver.setPrefix("/WEB-INF/views/");
		templateResolver.setSuffix(".html");
		return templateResolver;
	}
	@Bean
	public SpringTemplateEngine templateEngine() {
		SpringTemplateEngine templateEngine = new SpringTemplateEngine();
		
		templateEngine.setTemplateResolver(templateResolver());
		templateEngine.setEnableSpringELCompiler(true);
		return templateEngine;
	}
	@Override
	public void configureViewResolvers(ViewResolverRegistry registry) {
		ThymeleafViewResolver resolver = new ThymeleafViewResolver();
		resolver.setTemplateEngine(templateEngine());
		registry.viewResolver(resolver);
	}
}
```

Для поддержки БАЗОВОЙ конфигурации MVC используется аннотация
> [!NOTE] @EnableWebMVC
>Аннотация `@EnableWebMvc` в Spring используется для включения полной поддержки Spring MVC в конфигурации Java. 
> 
> Она активирует функциональность MVC и позволяет настраивать компоненты Spring MVC, такие как `ViewResolvers`, `MessageConverters`, обработчики маршрутов, и прочие настройки, обеспечивая поведение, аналогичное использованию XML-конфигурации MVC (например, через `<mvc:annotation-driven>`).
> 
> ==без использования аннотации== `@EnableWebMvc`. Вместо этого можно создать класс, который будет наследовать `WebMvcConfigurationSupport` и вручную настраивать все необходимые компоненты MVC.

Для настройки БАЗОВОЙ конфигурации необходимо реализовать интерфейс
> [!NOTE] Интерфейс WebMvcConfigurer
> `WebMvcConfigurer` предоставляет удобные методы для добавления форматтеров, настроек CORS, интерсепторов, обработчиков ресурсов и других настроек MVC.
> 
> Нужен для гибкой настройки базовой конфигурации MVC приложения, например заменить базовый шаблонизатор JSP на Thymleaf

С помощью этого интерфейса реализуем метод для реализации метода `configureViewResolvers` необходимый для описания того, что мы будем использовать шаблонизатор Tymleaf
### 1.`templateResolver()`
Этот метод создает и настраивает `SpringResourceTemplateResolver`, который ==отвечает за нахождение и обработку шаблонов== (HTML файлов). Вот что делает этот метод:
- **`setApplicationContext(applicationContext)`**: Связывает `templateResolver` с контекстом Spring, что позволяет ему находить ресурсы в приложении.
### 2.`templateEngine()`
Этот метод ==создает и настраивает== `SpringTemplateEngine`, который является ==ядром движка Thymeleaf==. Он отвечает за обработку шаблонов, используя `templateResolver` для их поиска.
- **`setTemplateResolver(templateResolver())`**: Связывает движок с ранее настроенным `templateResolver`.
- **`setEnableSpringELCompiler(true)`**: Включает компилятор выражений Spring EL, что может улучшить производительность при работе с выражениями внутри шаблонов.
### 3.`configureViewResolvers(ViewResolverRegistry registry)`
Этот метод ==настраивает резолвер представлений==, который Spring MVC использует для определения представления, возвращаемого контроллером.
- **Создание `ThymeleafViewResolver`**: `ThymeleafViewResolver` — это адаптер между Spring MVC и Thymeleaf, который преобразует логические имена представлений (например, строки, возвращаемые контроллерами) в конечные представления с использованием движка Thymeleaf.
- **`resolver.setTemplateEngine(templateEngine())`**: Связывает `ThymeleafViewResolver` с настроенным движком `templateEngine`, чтобы тот мог использовать шаблоны для рендеринга.
- **`registry.viewResolver(resolver)`**: Регистрирует резолвер представлений в Spring MVC.
# Развертывание приложения на сервере Tomcat

По умолчанию Tomcat не видит наше приложение MVC, для этого его необходимо добавить в модули, которые разворачиваются на сервере

Для этого можно нажать на сервер ПКМ и нажать ==Add and Remove==, в панели Availeable должен быть наш ==проект==, его нежно ==переместить в панель Configurated==

Теперь при запуске сервера, наше MVC приложение будет работать
