# 5. Валидация форм

Однако, в текущих формах мы можем передать пустые значения, и такая запись создастся в нашей БД.

Более того, мы можем ввести туда любые значения
- Очень длинное имя
- Отрицательный возраст
- Неправильная почта

Для решения этой проблемы будем использовать валидацию полей. Поля будут проверятся на допустимые значения

***
## 1. Установка зависимости в `pom.xml`
Для валидации форм, будем использовать ==Hibernate Validator==

Добавим следующий код в `pom.xml`
```xml
<dependency>
    <groupId>org.hibernate.validator</groupId>
    <artifactId>hibernate-validator</artifactId>
    <version>6.1.6.Final</version>
</dependency>
```

***
## 2. Установка ПРАВИЛ валидации
Поскольку мы вставляем данные в запись, а именно в класс `Person` , то в нем и будем устанавливать правила валидации

```java
public class Person {
	private int id;
	
	@NotEmpty(message = "Name can`t not be empty!")
	@Size(min = 2, max = 30, message = "Name can be length between 2 and 30 character!")
	private String name;
	
	@NotEmpty(message = "Email can`t not be empty!")
	@Email(message = "Email should be valid format!")
	private String email;
	
	@Min(value = 0, message = "Age can`t be negative value!")
	private int age;
	. . .
```

После подключения зависимости `Hibernate Validator` мы можем использовать аннотации для валидации значений полей класса `Person`

> [!NOTE] Аннотация `@NotEmpty`
> Данная аннотация проверяет, вводится ли пустое значение в это поле класса.
> Если ДА (правило нарушается), то выводится сообщение `message`

> [!NOTE] Аннотация `@Size`
> Аннотация проверяет, находится ли заданное значение в диапазоне между `min` и `max`.
> Если НЕТ (правило нарушается), то выводится сообщение `message`

> [!NOTE] Аннотация `@Email`
> Аннотация проверяет правильный ли формат Email вводится в поле.
> Если НЕТ - выводится сообщение `message`

> [!NOTE] Аннотация `Min`
> Аннотация УСТАНАВЛИВАЕТ минимально допустимое значение для поля
> Если вводится меньшее число чем установленное аннотацией - выводится `message`

***
## 3. `@Valid` Активация (включение) валидации в методах 
Для того чтобы эти правила начали работать, необходимо использовать ==аннотацию== `@Valid`

```java
@PostMapping()
public String addPersonInDateBase(@ModelAttribute("newperson") @Valid Person person, BindingResult bindingResult) 
{
	if(bindingResult.hasErrors())
		return "people/new";
	
	myPersonDAO.saved(person);
	return "redirect:/people";
}
```

Данная аннотация включает валидацию полей класса, в котором были установлены правила. 

#### ------------------------------------------------------------------------------------

> [!NOTE] ВААЖНООО ОЧЕНЬ
> Здесь ==необходимо использовать в модели атрибут с именем АНАЛОГИЧНЫМ== с тем, ==что используется в html-странице находящиеся в== `people/new`, поскольку имя атрибута ВШИТО в html-код, и именно в объект с этим атрибутом добавляются значения полей 
> 
> Если использовать другое имя атрибута, то ОШИБКИ валидации НЕ БУДУТ ВЫВОДИТЬСЯ, поскольку в методе при возникновении ошибки валидации, мы вернем страницу, которая ожидает атрибут `newPerson` (который удалится из модели при выходе из метода обрабатывающего GET-запросы)  от текущей модели.
#### ------------------------------------------------------------------------------------

Если эти правила нарушаются, то ошибки помещаются в объект класса `BindingResult`
У этого класса есть метод, который проверяет наличие ошибок. 

> [!NOTE] Класс `BindingResult`
> Объект этого класса, ==ДОЛЖЕН идти СРАЗУ== после аргумента, помеченного аннотацией `Valid`

Реализуем проверку на наличие ошибок - если есть, то ввернем окно для создания записи

То же самое ==сделаем и для метода контроллера==, предназначенного для принятие запроса ==на редактирование записи==

```java
@PatchMapping("/{id}")
public String updatePersonInDateBase(@ModelAttribute("updatePerson") @Valid Person person, BindingResult bindingResult, @PathVariable("id") int id) 
{
	if(bindingResult.hasErrors())
		return "people/edit";
	
	myPersonDAO.update(id, person);
	return "redirect:/people";
}
```

## 4. Отображение ошибок в представлении

Для того, ==чтобы ошибки валидации== (`message`) ==отображались в представлении==, необходимо добавить в код html-страницы ==тег Thymleaf==, который ==будет выводить это сообщение==, если ==ошибки валидации были обнаружены==

```html
<div th:if="${#fields.hasError('name')}" th:errors="*{name}">Name error</div>
```

Тег `th:if` используется для проверки условия
Тег `#fields` используется для доступа к полям класса указанного в `th:object`
Если ошибки в этом поле есть, то с помощью `th:errors` ошибка этого поля выведется на странице 

Данный код вставим после каждого тега `input`
```html
<body>
<form th:method="POST" th:action="@{/people}" th:object="${newPerson}">
	<label for="Name">Enter name person</label>
	<input type="text" th:field="*{name}" id="Name"/>
	<div style="color:red" th:if="${#fields.hasError('name')}" th:errors="*{name}">Name error</div>
	
	<br/>
	<label for="Age">Enter age person</label>
	<input type="text" th:field="*{age}" id="Age"/>
	<div style="color:red" th:if="${#fields.hasError('age')}" th:errors="*{age}">Name error</div>
	
	<br/>
	<label for="Email">Enter email person</label>
	<input type="text" th:field="*{email}" id="Email"/>
	<div style="color:red" th:if="${#fields.hasError('email')}" th:errors="*{email}">Name error</div>
	
	<br/>
	<input type="submit" value="Create!"/>
</form>
</body>
```
