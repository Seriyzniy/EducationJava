# 0. Логика работы SpringSecurity
Spring Security работает на основе фильтров. Когда HTTP-запрос поступает в ваше приложение, он проходит через цепочку фильтров Spring Security, где выполняется аутентификация и авторизация.
### Основные этапы:
1. **Фильтры Spring Security**:
    - Запрос проходит через цепочку фильтров, где выполняется аутентификация (проверка логина и пароля) и авторизация (проверка прав доступа).
        
2. **Аутентификация**:
    - Spring Security проверяет, аутентифицирован ли пользователь. Если нет, он перенаправляет его на страницу входа.
        
3. **Авторизация**:
    - Если пользователь аутентифицирован, Spring Security проверяет, имеет ли он доступ к запрашиваемому ресурсу.
***
### Настройка Spring Security?

В классе конфигурации `DispatcherServlet` вы ==создаете фильтр== `springSecurityFilterChain`. 

==Этот фильтр настраивается в классе== `SecurityConfig` (помеченный аннотацией`@EnableWebSecurity`) , где вы указываете:
✔ Какие URL-адреса требуют аутентификации, а какие нет.
✔Какую форму входа использовать.
✔ Что делать после успешного входа или выхода.
***
### Как Spring Security проверяет логин и пароль?

==Spring Security автоматически выполняет проверку логина и пароля== с помощью встроенных механизмов. Вот как это работает:
#### a) UserDetailsService:
- Вы реализуете интерфейс `UserDetailsService`, который загружает пользователя по его логину (username).
    
- Spring Security использует ваш `UserDetailsService`, чтобы получить объект `UserDetails`, который содержит информацию о пользователе, включая его пароль.

#### b) PasswordEncoder:
- Вы указываете `PasswordEncoder` (например, `BCryptPasswordEncoder` или `NoOpPasswordEncoder`) в конфигурации Spring Security.
    
- Spring Security использует `PasswordEncoder`, чтобы сравнить пароль, введенный пользователем на форме входа, с паролем, который хранится в базе данных (через `UserDetails`).

#### c) DaoAuthenticationProvider:
- Spring Security использует встроенный `DaoAuthenticationProvider`, который автоматически связывает `UserDetailsService` и `PasswordEncoder`.
    
- `DaoAuthenticationProvider` выполняет следующие шаги:
    
    1. Загружает пользователя с помощью `UserDetailsService`.
        
    2. Сравнивает введенный пароль с паролем из `UserDetails` с помощью `PasswordEncoder`.
        
    3. Если пароли совпадают, создает объект `Authentication`, который представляет аутентифицированного пользователя.

> [!NOTE] Реализация интерфейса `AuthenticationProvider`
> Если вам нужна ==кастомная логика== аутентификации (например, аутентификация через сторонний сервис), вам НУЖНО писать свой `AuthenticationProvider`.


> [!NOTE] Маппинг по умолчанию
> По умолчанию Spring Security принимает POST запросы с данными пользователя на адрес `/login`, и ожидает параметры `username` и `password`, а также проводит проверку введенных данных с БД
> 
> Этот адрес можно переопределить при помощи метода `loginProcessingUrl()` внутри `formLogin()`

## Добавления фильтра
Для создания такого фильтра, необходимо в классе конфигурации сервера (диспетчера сервлетов (web.xml)) создать метод, который добавить фильтр, ко всем сервлетам

```java
private void registerSecurityFilter(ServletContext servletContext) {
	servletContext.addFilter("springSecurityFilterChain", new
		DelegatingFilterProxy("springSecurityFilterChain"))
		.addMappingForUrlPatterns(null, true, "/*"); 
}
```

> [!NOTE] Специальный бин `springSecurityFilterChain`
> Данный бин АВТОМАТИЧЕСКИ создается Spring Security при помощи аннотации `@EnableWebSecurity` над классом, который будет использоваться для конфигурации
> 
> Этот бин представляет собой **цепочку фильтров**, которая отвечает за обработку безопасности в вашем приложении

Метод включения фильтра в загрузку
```java
@Override
public void onStartup(ServletContext servletContext) throws ServletException {
	super.onStartup(servletContext);
	registerSecurityFilter(servletContext);
}
```

## Конфигурация бина 
После того, как мы добавили бин (фильтр), необходимо его сконфигурировать 

Для этого ==необходимо создать класс==, который будет описывать конфигурацию фильтра
```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {
	@Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(auth -> auth
                .anyRequest().authenticated()  // Все запросы требуют авторизации
            )
            .formLogin(login -> login
                .loginPage("/login")  // Страница логина
                .permitAll()  // Страница логина доступна всем
            )
            .logout(logout -> logout
                .permitAll()  // Выход доступен всем
            );
        return http.build();
    }
	@Bean
    public UserDetailsService userDetailsService() {
        UserDetails user = User.withDefaultPasswordEncoder()
            .username("user")
            .password("password")
            .roles("USER")
            .build();
        return new InMemoryUserDetailsManager(user);
    }
}
```

> [!NOTE] Аннотация `@EnableWebSecurity`
> Класс, помеченный аннотацией `@EnableWebSecurity`, является **конфигурационным классом** для Spring Security. 
> 
> В этом классе вы определяете:
> - Правила безопасности (например, какие URL требуют аутентификации, а какие доступны всем).
> - Настройки аутентификации (например, форма входа, страница логина, настройки выхода).
> - Другие аспекты безопасности (например, защита от CSRF, настройки CORS и т.д.).
> 
> Эти настройки Spring Security использует для **конфигурации бина `springSecurityFilterChain`**.
> 
> 🔶Она говорит Spring, что нужно включить поддержку безопасности в приложении и использовать конфигурацию, представленную в классе, помеченном этой аннотацией.

## Добавление контроллера для перенаправленных запросов
Поскольку фильтр будет перенаправлять запросы на указанный в конфигурации URL, необходимо создать контроллер, который будет их обрабатывать

Класс контекста приложения
```java
@Override
public void addViewControllers(ViewControllerRegistry registry) {
    registry.addViewController("/login").setViewName("login");
}
```

С помощью объекта класса `ViewControllerRegistry` можно создать "простой" контроллер, без явного java-класса и аннотации `@Controller`
