
> [!NOTE] ⭐`НЕ ИЗМЕНЯЙ` `состояние` `ЧУЖОГО ОБЪЕКТА` в StreamAPI ⭐
> Stream API **предполагает чистые (stateless)** функции — без модификации внешнего состояния. 
> 
> `Работаем ТОЛЬКО С объектами`, которые `НАХОДЯТСЯ В ПОТОКЕ` 

❌ Пример неправильного кода❌
```java
List<Integer> numbers = List.of(1, 2, 3, 4);
List<Integer> result = new ArrayList<>();

numbers.parallelStream()
       .map(n -> n * 2)
       .forEach(result::add); // 🧨 race condition!
```

✅ Правильный способ✅
```java
List<Integer> result = numbers
	.parallelStream()
    .map(n -> n * 2)
    .collect(Collectors.toList());
```


> [!NOTE] Поток в (Stream API) - канал для данных. 
> Поток `оперирует` `на ИСТОЧНИКЕ данных`, при этом `попутно` перемещая данные, возможно с попутной фильтрацией, сортировкой или `другой обработкой`. 
> 
> Такая операция `НЕ МОДИФИЦИРУЕТ ИСТОЧНИК ДАНЫХ`, а наоборот, `СОЗДАЕТ НОВЫЙ ОБЪЕКТ`, для `хранения результата`
>

# 📕 1. Потоковые интерфейсы

> [!NOTE] `Базовый интерфейс` потокового API - `interface BaseStream`
> Определяет базовую функциональность ВСЕХ ПОТОКОВ в `java.util.stream`

```java
/**
 * Базовый интерфейс для всех потоков в Java Stream API.
 * Определяет основные операции для последовательной и параллельной обработки данных.
 * 
 * @param 💡<T> ТИП ЭЛЕМЕНТОВ в потоке
 * @param <S> ТИП самого ПОТОКА (для поддержки fluent interface)
 */
public interface BaseStream<T, S extends BaseStream<T, S>>
        extends AutoCloseable {
    
    /**
     * Возвращает итератор для последовательного обхода элементов потока.
     * Является терминальной операцией - после вызова поток считается consumed.
     */
    Iterator<T> iterator();

    /**
     * Возвращает spliterator для параллельной обработки элементов потока.
     * Spliterator позволяет разбивать поток на части для параллельного выполнения.
     */
    Spliterator<T> spliterator();

    /**
     * Возвращает признак параллельного потока.
     * true - если поток может выполняться параллельно, false - для последовательного выполнения.
     */
    boolean isParallel();

    /**
     * Возвращает последовательный (sequential) эквивалент текущего потока.
     * Все последующие операции будут выполняться последовательно, даже если исходный поток был параллельным.
     */
    S sequential();

    /**
     * ⭐Возвращает параллельный эквивалент текущего потока.
     * Все последующие операции могут выполняться параллельно (если поддерживается источником данных).
     */
    S parallel();

    /**
     * Возвращает неупорядоченный эквивалент текущего потока.
     * Снимает ограничения сохранения порядка элементов, что может улучшить производительность некоторых операций.
     */
    S unordered();

    /**
     * Возвращает поток с дополнительным обработчиком закрытия.
     * closeHandler будет вызван при закрытии потока методом close().
     * 
     * @param closeHandler обработчик, выполняемый при закрытии потока
     */
    S onClose(Runnable closeHandler);

    /**
     * Закрывает поток, освобождая все связанные с ним ресурсы.
     * Вызывает все зарегистрированные обработчики onClose() в порядке их добавления.
     * Для потоков, основанных на коллекциях, обычно не требует явного закрытия.
     * Обязательно вызывать для потоков, основанных на файлах или каналах ввода-вывода.
     */
    void close();
}
```

# 📕 2. ⭐⭐ Подинтерфейс Stream ⭐⭐
Существует несколько типов потоковых интерфейсов, производных от `BaseStream`. 

Наиболее универсальным из них является интерфейс `Stream`. 

```java
/**
 * Stream представляет собой последовательность элементов, поддерживающих последовательные и параллельные агрегатные операции.
 *
 * @param <T> тип элементов в потоке
 */
public interface Stream<T> extends BaseStream<T, Stream<T>>
```

Поскольку интерфейс ==Stream работает с объектными ссылками==, он не может напрямую оперировать с примитивными типами. 

==Для обработки== потоков ==примитивных типов== в потоковом API ==определены== следующие ==интерфейсы==: 
`DoubleStream` 
`IntStream` 
`LongStream`

Все перечисленные интерфейсы `расширяют` `BaseStream` и обладают та кими же возможностями, как Stream, но имеют дело с примитивными, а не ссылочными типами.

> [!NOTE]  ⭐⭐`После` завершения `ТЕРМИНАЛЬНОЙ` операции, `ПОТОК НЕЛЬЗЯ ПЕРЕИСПОЛЬЗОВАТЬ`⭐⭐
> 


В дополнение к методам, унаследованных от `BaseStream`, интерфейс определяет свои собственные:

> [!NOTE] ⭐⭐ Промежуточные VS Терминальные операции ⭐⭐
> - `Промежуточные` - формируют конвейер обработки, и возвращают Stream. ==Выполняются только при вызове терминальной операции==
> - 
> - `Терминальные` - ==запускают выполнение всего конвейера== ЗАКРЫВАЮТ поток (после вызова поток нельзя использовать повторно). Возвращают НЕ Stream, а void, boolean, Optional и т.д 

### Промежуточная операция - Фильтрация и преобразование
```java
/**
 * Фильтрует элементы потока по заданному условию.
 * 
 * 💡 predicate условие фильтрации (сохраняются элементы, для которых predicate возвращает true)
 * @return новый поток с отфильтрованными элементами
 */
Stream<T> filter(Predicate<? super T> predicate);


/**
 * Преобразует каждый элемент потока в другой объект с помощью заданной функции.
 * 
 * 💡 mapper функция преобразования элемента
 * 💡 <R> тип элементов в новом потоке
 * @return новый поток с преобразованными элементами
 */
<R> Stream<R> map(Function<? super T, ? extends R> mapper);

// Преобразование в примитивные потоки
IntStream mapToInt(ToIntFunction<? super T> mapper);
LongStream mapToLong(ToLongFunction<? super T> mapper);
DoubleStream mapToDouble(ToDoubleFunction<? super T> mapper);



/**
 * Преобразует каждый элемент потока в поток других элементов и объединяет все потоки в один.
 * 
 * mapper функция, преобразующая элемент в поток
 * <R> тип элементов в новом потоке
 * @return новый поток, содержащий все элементы из всех потоков, возвращенных mapper
 */
<R> Stream<R> flatMap(
	Function<? super T, ? extends Stream<? extends R>> mapper);

IntStream flatMapToInt(Function<? super T, ? extends IntStream> mapper);
LongStream flatMapToLong(Function<? super T, ? extends LongStream> mapper);
DoubleStream flatMapToDouble(Function<? super T, ? extends DoubleStream> mapper);
```

### Промежуточная операция -  Управление порядком и количеством
```java
// Удаление дубликатов элементов используя equals()
Stream<T> distinct();

// 💡Сортировка элементов💡 (естественным порядком или через КОМПАРАТОР или реализацию Comparable)
Stream<T> sorted(Comparator<? super T> comparator);
Stream<T> sorted();


// Ограничение количества элементов
Stream<T> limit(long maxSize);

// Пропуск первых N элементов
Stream<T> skip(long n);
```

### Промежуточная операция - Отладка и объединение
```java 
/**
* Выполняет действие для каждого элемента потока без изменения элементов.
* 
* action действие, выполняемое для каждого элемента
* @return тот же поток (для цепочки вызовов)
*/
Stream<T> peek(Consumer<? super T> action);


//  Создает поток, содержащий элементы текущего потока, за которыми следуют элементы другого потока.
Stream<T> concat(Stream<? extends T> other);
```

### Терминальная операция - Итерация и сбор
```java
// Выполнение действия для каждого элемента
void forEach(Consumer<? super T> action);
void forEachOrdered(Consumer<? super T> action);


// Преобразование в массив
Object[] toArray();
<A> A[] toArray(IntFunction<A[]> generator);

/*
supplier - Поставщик результата: создает новый пустой контейнер 
(например: ArrayList::new, StringBuilder::new)

accumulator - Аккумулятор: добавляет один элемент в контейнер
(например: List::add, StringBuilder::append)
 
combiner - Комбинатор: объединяет два контейнера (используется в 
параллельных операциях для слияния частичных результатов)
 
<R> - Тип результирующего контейнера
*/

<R> R collect(
	Supplier<R> supplier, 
	BiConsumer<R, ? super T> accumulator, 
	BiConsumer<R, R> combiner);

// Собирает элементы потока в коллекцию с помощью Collector.
// collector коллектор, определяющий способ сбора элементов
// <R> тип результата
// <A> тип аккумулятора коллектора	
<R, A> R collect(Collector<? super T, A, R> collector);
```

### Терминальная операция - Поиск и сравнение
```java
// Поиск минимального/максимального элемента
Optional<T> min(Comparator<? super T> comparator);
Optional<T> max(Comparator<? super T> comparator);


// Поиск любого/первого элемента
Optional<T> findFirst();
Optional<T> findAny();
```


### Терминальная операция - Проверка условий
```java
// Проверка условий для элементов потока
boolean anyMatch(Predicate<? super T> predicate);
boolean allMatch(Predicate<? super T> predicate);

// Проверяет, что НИ ОДИН элемент потока не соответствует заданному условию.
boolean noneMatch(Predicate<? super T> predicate);
```

### Терминальная операция - Агрегация и подсчет
```java
// Подсчет количества элементов
long count();

// Сводит элементы потока к одному значению с помощью аккумулирующей функции
// identity начальное значение
// accumulator функция аккумуляции
T reduce(
	T identity, 
	BinaryOperator<T> accumulator);
	
Optional<T> reduce(BinaryOperator<T> accumulator);


// Сводит элементы потока к одному значению с помощью аккумулирующей функции (с поддержкой разных типов).
// identity начальное значение
// accumulator функция аккумуляции
// combiner функция для комбинирования результатов (в параллельных операциях)
// <U> тип результата
<U> U reduce(
	U identity, 
	BiFunction<U, ? super T, U> accumulator, 
	BinaryOperator<U> combiner);
```

### Унаследовано от BaseStream

## Промежуточные операции с "ЗАПОМИНАНИЕМ"  и "ЗАБЫВАНИЕМ" состояния

> [!NOTE] Одни промежуточные операции не запоминают состояние, а другие его сохраняют.
> В операции, НЕ запоминающей состояние, `каждый элемент` `обрабатывается` `независимо` `от других`. 
> 
> В операции с сохранением состояния `обработка элемента` `может зависеть` `от аспектов других`элементов.

### **Операции БЕЗ состояния (Stateless)**
**Каждый элемент обрабатывается независимо:**
```java
// FILTER - не запоминает состояние
stream.filter(x -> x > 5)  // каждый элемент проверяется независимо

// MAP - не запоминает состояние  
stream.map(x -> x * 2)     // каждый элемент преобразуется независимо

// PEEK - не запоминает состояние
stream.peek(System.out::println)
```
**Как работает:**
- Элемент `A` → обработали → забыли
- Элемент `B` → обработали → забыли
    
- Не нужно знать про предыдущие или следующие элементы

#### Без состояния - легко параллелить:
```java
List<Integer> result = stream.parallel()
    .filter(x -> x > 0)    // каждый поток обрабатывает свою часть независимо
    .map(x -> x * 2)       // не нужна синхронизация между потоками
    .collect(toList());
```

### **Операции СО состоянием (Stateful)**
**Зависят от других элементов:**
```java
// SORTED - запоминает состояние
stream.sorted()            // нужно собрать ВСЕ элементы, чтобы их отсортировать

// DISTINCT - запоминает состояние  
stream.distinct()          // нужно помнить какие элементы уже встречались

// LIMIT - запоминает состояние
stream.limit(5)            // нужно считать сколько элементов уже прошло

// SKIP - запоминает состояние
stream.skip(3)             // нужно помнить сколько элементов пропущено
```

**Как работает:**
- Нужно собрать информацию о нескольких/всех элементах
- Элемент `A` → запомнили
- Элемент `B` → запомнили, сравнили с `A`
- Элемент `C` → запомнили, сравнили с `A` и `B`

#### С состоянием - сложнее параллелить:
```java
List<Integer> result = stream.parallel()
    .sorted()       // каждый поток сортирует свою часть,
                    // потом нужно объединить результаты - ДОРОГО!
    .limit(10)      // нужно координировать подсчет между потоками
    .collect(toList());
```

### Практические последствия:
**Эффективная цепочка:**
```java
// ХОРОШО: операции без состояния в начале
stream.filter(...)    // ← быстро в параллели
     .map(...)        // ← быстро в параллели  
     .sorted()        // ← дорогая операция (но на меньшем количестве данных)
     .limit(10)       // ← состояние (но на уже отсортированных данных)
```

**Неэффективная цепочка:**
```java
// ПЛОХО: операции с состоянием в начале
stream.sorted()       // ← сортируем ВСЕ данные (дорого)
     .filter(...)     // ← ❗❗❗отбрасываем много отсортированного (бесполезная работа)
     .limit(10)       // ← берем немного
```
