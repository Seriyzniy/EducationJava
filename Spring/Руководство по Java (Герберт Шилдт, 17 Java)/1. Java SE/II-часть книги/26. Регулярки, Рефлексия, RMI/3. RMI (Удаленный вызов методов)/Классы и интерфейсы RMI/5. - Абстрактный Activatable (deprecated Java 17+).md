
> [!NOTE] `Activatable` — это **абстрактный класс** из пакета `java.rmi.activation`
> предназначенный для `создания` `АКТИВИРУЕМЫХ удалённых объектов` в RMI.  
> 
> Он расширяет `RemoteServer` и предоставляет инфраструктуру для `ленивой` (по требованию) `активации` объектов, которые не всегда находятся в памяти.

Когда клиент вызывает метод на таком объекте:
- 💡если объект **не запущен**, RMI Activation System автоматически его **активирует** (запускает новый процесс или поднимает объект из архива);
    
- 💡затем выполняет вызов и, при необходимости, может выгрузить объект из памяти.
    

Таким образом, `Activatable` используется для реализации **долгоживущих распределённых сервисов**, которые активируются только при обращении, экономя ресурсы сервера.

## Конструктор - НУЖНО НАСЛЕДОВАТЬСЯ
```java
// Пояснение:
// ❗Регистрирует новый активируемый объект❗ в системе активации
// ❗и экспортирует❗ его для удалённого доступа.
//
// name — логическое имя объекта (в системе активации)
// data — сериализованные данные, передаваемые конструктору при активации
// restart — true, если объект должен автоматически перезапускаться при сбое активационной группы
// port — порт, на котором будет слушать RMI (0 — выбрать автоматически)
protected Activatable(
	String name, 
	MarshalledObject<?> data, 
	boolean restart, 
	int port) throws ActivationException, RemoteException;
        

// Пояснение:
// Используется системой активации ❗при восстановлении (активации)❗ объекта.
// Получает уникальный ActivationID, через который RMI связывает stub с этим объектом.
//
// id — идентификатор активации
// port — порт, на котором экспортировать объект
protected Activatable(ActivationID id, int port)
        throws RemoteException;

```

## Методы

### Управление экспортом и деактивацией
```java
// Пояснение:
// Экспортирует удалённый объект с заданным ActivationID и портом
public static Remote exportObject(Remote obj, ActivationID id, int port)
        throws RemoteException;

// Пояснение:
// Удаляет объект из RMI runtime (деактивирует)
public static boolean unexportObject(Remote obj, boolean force)
        throws NoSuchObjectException;


// Пояснение:
// ❗Активирует объект по заданному ActivationID.❗
// Обычно вызывается системой активации, а не вручную.
public static Remote activate(ActivationID id, boolean force)
        throws ActivationException, UnknownObjectException, RemoteException;

```

### Регистрация и деактивация в системе активации
```java
// Пояснение:
// ❗Регистрирует объект в системе активации❗ и возвращает его ActivationID
public static ActivationID register(ActivationDesc desc)
        throws ActivationException, UnknownGroupException, RemoteException;


// Пояснение:
// Отменяет регистрацию объекта в системе активации
public static void unregister(ActivationID id)
        throws ActivationException, UnknownObjectException, RemoteException;
```

### Информация о объекте
```java
// Пояснение:
// Возвращает ActivationID для текущего активируемого объекта
public ActivationID getID();
```

## 💡 Особенности и подводные камни 💡

1️⃣ 💡**Поддерживает ленивую активацию.**  
Объект ==создаётся== **только ==при первом обращении клиента==**, что снижает нагрузку на сервер.

2️⃣ **Требует активатора (`rmid`).**  
Для работы активируемых объектов необходимо запустить **RMI Activation System Daemon** (`rmid`), который управляет их жизненным циклом.

3️⃣ **Использует дескрипторы активации.**  
Каждый активируемый объект описывается через `ActivationDesc`, где задаётся имя, код, класс и параметры для активации.

4️⃣ 🧠**Регистрируется в реестре не напрямую.**  
Обычно регистрируется через систему активации (`ActivationSystem`), а не через `Naming.rebind()`.

5️⃣ **Сложнее, чем обычный UnicastRemoteObject.**  
Подходит только для случаев, ==где действительно нужна экономия ресурсов== или длительный жизненный цикл.

---
## 💎 Best Practices 💎

✅ 💡Используй `Activatable` только когда есть реальная необходимость в ленивой загрузке.  

✅ Всегда корректно реализуй `equals()` и `hashCode()` в `ActivationID`-зависимых классах.  

✅ Следи, чтобы `rmid` был запущен с правильными правами и CLASSPATH.  

✅ Храни `ActivationDesc` в конфигурации, а не в коде.  

✅ Используй `Activatable` совместно с `ActivationGroup` для логического объединения объектов.

## Пример
```java
import java.rmi.*;
import java.rmi.activation.*;
import java.rmi.server.*;
import java.io.Serializable;

// Удалённый интерфейс
public interface MyService extends Remote {
    String sayHello() throws RemoteException;
}

// Реализация активируемого объекта
public class MyServiceImpl extends Activatable implements MyService, Serializable {

    // Конструктор, вызываемый системой активации
    public MyServiceImpl(ActivationID id, MarshalledObject<?> data)
            throws RemoteException {
        super(id, 0); // экспорт по умолчанию
        System.out.println("Активируемый объект создан!");
    }

    @Override
    public String sayHello() {
        return "Hello from Activatable object!";
    }

    // main вызывается один раз — для регистрации объекта
    public static void main(String[] args) throws Exception {
        // Настройка активации
        ActivationGroupDesc.CommandEnvironment ace = null;
        ActivationGroupDesc groupDesc = new ActivationGroupDesc(null, ace);
        ActivationGroupID groupID = ActivationGroup
	        .getSystem()
	        .registerGroup(groupDesc);
	        
        ActivationGroup.createGroup(groupID, groupDesc, 0);

        // Дескриптор активации
        ActivationDesc desc = new ActivationDesc("MyServiceImpl", null, null);
        MyService stub = (MyService) Activatable.register(desc);

        // Регистрация stub в RMI Registry
        Naming.rebind("rmi://localhost/MyService", stub);

        System.out.println("Активируемый объект зарегистрирован.");
    }
}
```
