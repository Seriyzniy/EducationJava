

> [!NOTE] `RMIClientSocketFactory` — это **интерфейс** из пакета `java.rmi.server`
> который `определяет` `фабрику` для создания `клиентских сокетов`, `используемых RMI` при установлении соединения с удалённым объектом.
> 
> ⭐Его основная цель — `кастомизация сетевого взаимодействия` на стороне КЛИЕНТА  (например, использование SSL, нестандартных портов, туннелирования и т.п.).
> 
> Этот интерфейс может быть реализован разработчиком, чтобы RMI использовало нестандартные типы сокетов при соединении с сервером.

## Интерфейс
```java
public interface RMIClientSocketFactory
```

## Методы
### Создание сокета
```java
Socket createSocket(String host, int port) throws IOException;
```
- **Назначение:** создает клиентский сокет для подключения к удалённому объекту RMI.
    
- **Параметры:**
    - `host` — имя хоста или IP-адрес сервера, где расположен удалённый объект.
    - `port` — порт, на который будет произведено подключение.
        
- **Возвращает:** объект `java.net.Socket`, который RMI будет использовать для связи.
    
- **Исключения:** `IOException` — если при создании сокета возникли сетевые ошибки.

---
## 💡 Особенности и подводные камни 💡

1️⃣ ⭐⭐⭐**Сериализация обязательна.**  
==Реализации== `RMIClientSocketFactory` должны быть сериализуемыми, иначе RMI не сможет передать их по сети.

2️⃣ ⭐⭐⭐**Клиент и сервер должны быть согласованы.**  
Если клиент использует нестандартный `RMIClientSocketFactory`, сервер должен использовать совместимый `RMIServerSocketFactory`. Иначе соединение не установится.

3️⃣ ❗❗❗**equals() и hashCode() обязательны к переопределению.**  
RMI использует их для определения эквивалентности фабрик. Без правильной реализации возможны ошибки при регистрации/поиске объектов.

4️⃣ **Может использоваться с `UnicastRemoteObject`.**  
При экспорте удалённого объекта можно указать свою фабрику сокетов через конструктор `UnicastRemoteObject`.

5️⃣ **Не путать с RMIServerSocketFactory.**  
Первый отвечает за **клиентскую** сторону соединения, второй — за **серверную**.

---
## 💎 Best Practices 💎

✅ Реализуй `equals()` и `hashCode()` корректно — иначе объекты с одинаковыми фабриками будут считаться разными.  

✅ Используй SSL или другие защищённые сокеты при необходимости (например, `SSLSocketFactory`).  

✅ Проверяй сетевые параметры (таймауты, повторные подключения).  

✅ Тестируй фабрику в многопоточном окружении.  

✅ Явно логируй создание сокетов — это упростит отладку RMI.

## Пример
```java
import java.io.IOException;
import java.net.Socket;
import java.rmi.server.RMIClientSocketFactory;
import java.rmi.server.UnicastRemoteObject;

public class CustomClientSocketFactory implements RMIClientSocketFactory, java.io.Serializable {
    
    private int timeout;

    // Пояснение:
    // Конструктор позволяет задать таймаут подключения в миллисекундах
    public CustomClientSocketFactory(int timeout) {
        this.timeout = timeout;
    }

    // Пояснение:
    // ⭐Метод создает кастомный клиентский сокет и устанавливает таймаут
    @Override
    public Socket createSocket(String host, int port) throws IOException {
        Socket socket = new Socket(host, port);
        socket.setSoTimeout(timeout);
        return socket;
    }

    // Пояснение:
    // Корректное сравнение фабрик по таймауту
    @Override
    public boolean equals(Object obj) {
        if (this == obj) return true;
        if (!(obj instanceof CustomClientSocketFactory)) return false;
        return this.timeout == ((CustomClientSocketFactory) obj).timeout;
    }

    // Пояснение:
    // Возвращает hashCode, основанный на параметрах фабрики
    @Override
    public int hashCode() {
        return timeout;
    }
}
```

🔹 СЕРВЕР: Использование при экспорте удалённого объекта:
```java
MyRemoteImpl obj = new MyRemoteImpl();

RMIClientSocketFactory csf = new CustomClientSocketFactory(5000);
RMIServerSocketFactory ssf = new CustomServerSocketFactory();

// Экспортируем удаленный интерфейс и кастомный сокет
UnicastRemoteObject.exportObject(obj, 0, csf, ssf);

Naming.rebind("rmi://localhost/MyService", obj);
```

🔹 КЛИЕНТ: делает просто
```java
MyRemote stub = (MyRemote) Naming.lookup("rmi://server/MyService");

stub.remoteMethod();
```

## Зачем `RMIClientSocketFactory` должен быть сериализуемым?
*Объект удаленного интерфейса (stub)**, который RMI возвращает клиенту, содержит в себе:

- адрес удаленного объекта (host + port),
    
- и **экземпляр `RMIClientSocketFactory`** (если он есть).
    

Когда этот **stub сериализуется** (а это происходит при передаче ссылки на удаленный объект по сети — например, из сервера клиенту через реестр RMI или напрямую),  

RMI должен уметь сериализовать **всё его внутреннее состояние**, включая фабрику сокетов.

Примерно так выглядит структура сериализованного stub-а:
```json
RemoteStub {
    endpoint: (host, port)
    clientSocketFactory: CustomClientSocketFactory
}
```
➡️ Если фабрика **не сериализуема**, то при попытке передать `RemoteStub` клиенту возникнет `java.io.NotSerializableException`.

💡 Поэтому требование `implements java.io.Serializable` — не потому что фабрика сама «передает» данные,  
а потому что **она ==является частью метаданных удаленного объекта==**, которые нужно переслать клиенту, чтобы тот знал, **какие сокеты создавать при вызовах**.

## Экспорт на стороне клиента?

Это нужно **только в особых случаях**, например:

- 💡когда клиент **сам хочет быть удалённым объектом** (например, для callback-сценариев);
    
- когда сервер вызывает методы на клиенте (обратная связь);
