# 📕 1. Технология RPC (80-х годов) 
RMI основана на более ранней технологии удаленного вызова процедур ==Remote Procedure Call== (RPC), разработанной в 80-х годах и используемой для процедурного программирования. 

> [!NOTE] Remote Procedure Call (RPC)
> RPC `позволяет` `процедуре одного приложения` `вызывать функцию` `на другом компьютере`, 
> как будто эта функция является частью программы. 

Таким образом, RPC выполняет всю работу по организации сетевых взаимодействий и маршалинга данных (пакетирования параметров функций и возврата значений для передачи их по сети).

### RMI - продвинутый RPC
**RMI была разработана для устранения двух ключевых недостатков RPC:**
1. ❗**Ограниченная типовая система** — RPC поддерживает только простые типы данных и не может работать с объектами Java
2. ❗**Зависимость от IDL** — требует специального языка для описания удалённых интерфейсов


**RMI устраняет эти ограничения, обеспечивая:**
- 🟠Прямую передачу Java-объектов
- 🟠Использование стандартных Java-интерфейсов без необходимости в IDL

# 📕 2. Технология RMI

_RMI_ ==содержит набор объектов (классов)== для организации удаленного взаимодействия java-приложений.

![[Pasted image 20251014113937.png]]

### Серверное приложение
Серверное приложение, как правило, 
- 💡создает удаленные объекты (remote objects), 
- 💡делает доступные ссылки на эти объекты при помощи `кода` или `rmiregistry`
	
- находится в ожидании вызова методов этих объектов.

### Клиентское приложение
Клиентское приложение получает у сервера ссылку на удаленные объекты, после чего вызывает его методы

> [!NOTE] ⭐⭐ОБЩИЕ ДАННЫЕ должны иметь ОДИНАКОВУЮ СТРУКТУРУ ПАКЕТОВ, иначе `java.rmi.UnmarshalException`⭐⭐
> ✅**Одинаковая структура пакетов должна быть у всех классов, которые пересылаются между клиентом и сервером по RMI.**  
> 
> ⭐⭐То есть для:
> - интерфейсов `Remote`, 
> - всех DTO (Data Transfer Object), которые участвуют в аргументах или возвращаемых значениях этих методов.

## 📙1. Архитектура
RMI состоит из **трёх компонентов**:

| Компонент                                | Где находится                                 | Что делает                                                 |
| ---------------------------------------- | --------------------------------------------- | ---------------------------------------------------------- |
| 🟡**Интерфейс** (`Remote`)               | ==общий== (должен быть у клиента и у сервера) | описывает, какие методы можно вызвать удалённо             |
|                                          |                                               |                                                            |
| 🟢**Реализация** (`UnicastRemoteObject`) | на **сервере**                                | содержит конкретный код методов                            |
| 🟠**Stub (заглушка)**                    | на **клиенте**                                | объект-посредник, пересылающий вызовы через сеть на сервер |

### 📘 1. Интерфейс (общий для клиента и сервера)

> [!NOTE] `Контракт` между сервером и клиентом
> Это контракт, который и клиент, и сервер должны понимать одинаково.
> 
> ✅ Интерфейс должен ==наследоваться== от `Remote`.  
> ⚠️ ВСЕ МЕТОДЫ должны ==выбрасывать== `RemoteException`.

```java
package rmishared;

import java.rmi.Remote;
import java.rmi.RemoteException;

public interface BillingService extends Remote {
    double getBalance(String cardNumber) throws RemoteException;
}
```

### 📘2. Реализация на сервере

> [!NOTE] Класс должен НАСЛЕДОВАТЬСЯ от `UnicastRemoteObject`  И РЕАЛИЗОВАТЬ `КОНТРАКТ`
> Наследуемся от `UnicastRemoteObject`, чтобы этот объект мог быть "удалённым" (через TCP-сокеты).

```java
package rmiserver.server;

import java.rmi.RemoteException;
import java.rmi.server.UnicastRemoteObject;
import rmishared.BillingService;

public class BillingServiceImpl 
	extends UnicastRemoteObject // ✅НАСЛЕДОВАНИЕ 
	implements BillingService   // ✅КОНТРАКТ
{

    public BillingServiceImpl() throws RemoteException {
        super();
    }

    @Override
    public double getBalance(String cardNumber) throws RemoteException {
        return Math.random() * 1000; // просто тест
    }
}
```

#### Запуск RMI сервера
```java
package rmiserver.server;

import java.rmi.Naming;
import java.rmi.registry.LocateRegistry;

public class ServerMain {
    public static void main(String[] args) {
        try {
            // 1️⃣ Создаём объект-реализацию
            BillingService service = new BillingServiceImpl();

            // 2️⃣ Запускаем RMI Registry (на порту 1099)
            LocateRegistry.createRegistry(1099);

            // 3️⃣ Регистрируем объект в реестре под именем "BillingService"
            Naming.rebind("rmi://localhost:1099/BillingService", service);

            System.out.println("RMI Server started!");
        } catch (Exception e) {
            e.printStackTrace();
        }
    }
}
```

### ⭐⭐ 🧠  3. Что такое RMI Registry и URL ⭐⭐

RMI Registry — это как **“телефонная книга”** для удалённых объектов.  
Туда 
- ⭐сервер **регистрирует** объекты, 
- ⭐а клиент потом их **ищет**.

**URL** при регистрации или поиске имеет формат:
```http
rmi://<host>:<port>/<objectName>
```

🔹 Пример:  
`rmi://localhost:1099/BillingService`

- `localhost` — адрес сервера, где запущен RMI Registry
- `1099` — стандартный порт RMI
- `BillingService` — имя объекта в реестре

> [!NOTE] ⭐Утилита `rmiregistry` и ВАЖНАЯ особенность
> У `rmiregistry` есть неприятная особенность, `регистрировать объекты можно только с того же компьютера`, `с которого она запущена`. 
> При попытке зарегистрировать объект из другого компьютера выдается невнятная ошибка.


### ⭐⭐🧠 В RMI происходит **две независимые регистрации**⭐⭐

|Уровень|Что делает|Где происходит|Класс|
|---|---|---|---|
|**1. Экспорт объекта**|Делает объект удалённым (создаёт слушающий сокет, формирует stub, готовит инфраструктуру RMI runtime)|Внутри JVM сервера|`UnicastRemoteObject`|
|**2. Регистрация в реестре (Registry)**|Делает объект доступным по имени для клиентов, чтобы они могли его найти через `lookup()`|В отдельном процессе (RMI Registry)|`LocateRegistry` / `Registry`|

#### 🔁 Упрощённая аналогия
Представь:
- `UnicastRemoteObject` — это **человек**, который сел за телефон (экспортирован и слушает номер).
    
- `Registry.rebind()` — это **запись его номера в телефонную книгу**, чтобы другие знали, как ему позвонить.

### 📘 4. Клиент (ищет и вызывает)

> [!NOTE] `Stub` - `заглушка`, которая представляет `УДАЛЕННЫЙ ИНТЕРФЕЙС`, `при вызове метода` которого, `вызов передается по сети`
> Клиент вызывает `Naming.lookup()`, получает **stub**, и этот stub выглядит как обычный `BillingService`,  
> 
> но при вызове метода — он **отправляет данные по сети на сервер**, сервер выполняет метод и возвращает результат обратно.

```java
package rmiclient.client;

import java.rmi.Naming;
import rmishared.BillingService;

public class BillingClient {
    public static void main(String[] args) {
        try {
            String url = "rmi://localhost:1099/BillingService";
            BillingService service = 
	            (BillingService) Naming.lookup(url);

            double balance = service.getBalance("1234-5678-9012");
            System.out.println("Баланс карты: " + balance);
        } catch (Exception e) {
            e.printStackTrace();
        }
    }
}
```

#### 🧠 Что такое Stub

**Stub** — это “заместитель” удалённого объекта.  
Он существует **на клиентской стороне**, и делает следующее:

|Действие клиента|Что делает stub|
|---|---|
|`service.getBalance("1234")`|сериализует аргументы, отправляет их на сервер|
|Сервер вызывает метод `getBalance` и возвращает результат|stub получает результат и возвращает клиенту|
То есть stub ≈ "прозрачный сетевой посредник".

## 📙 2. Как всё работает по шагам
| Этап | Действие                                                 |
| ---- | -------------------------------------------------------- |
| 1    | Сервер создаёт объект и регистрирует его в RMI Registry  |
| 2    | Клиент через `Naming.lookup()` получает stub             |
| 3    | Stub подключается к серверу (по TCP)                     |
| 4    | Клиент вызывает метод → stub отправляет запрос на сервер |
| 5    | Сервер выполняет метод → возвращает результат            |
| 6    | Stub десериализует ответ и возвращает его клиенту        |

---
## 📙 3. Ключевые особенности:
- Реализована в пакете `java.rmi` (основной функционал сервера — в `java.rmi.server`)
    
- ⭐Обеспечивает **маршалинг данных** через сериализацию Java-объектов
    
- **Не требует IDL** — использует стандартные Java-интерфейсы
    
- Языково-специфичная технология (работает только с Java)

---
## ❓ «Зачем э то нужно?» ❓
==Задача== RMI — организация ==клиент-серверного взаимодействия==. 
⭐⭐Это значит, что вам НЕ ПРИДЕТСЯ беспокоится о:
- передаче данных
- предварительной обработке данных (протокол и т.д.). 

Удобно? Да. Но не во всех случаях. 

⭐Если в вашей клиент-серверной среде подразумевается работа программ, написанных не только на java, от RMI толку мало (хотя при большом желании можно попытаться «выкрутиться» при помощи JNI).
