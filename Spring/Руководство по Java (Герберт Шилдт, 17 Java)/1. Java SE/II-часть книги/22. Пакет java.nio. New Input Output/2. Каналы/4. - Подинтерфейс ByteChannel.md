
> [!NOTE] `ByteChannel` — это **подинтерфейс** `ReadableByteChannel` и `WritableByteChannel`.
> 
> То есть он представляет ==**ДВУнаправленный поток байтов**==, аналог **InputStream + OutputStream в одном объекте**.

Применяется для сущностей, где и чтение, и запись возможны одновременно:
- `SocketChannel` (TCP-соединение клиент ↔ сервер),
- `DatagramChannel` (UDP, хотя там специфика),
    
- `FileChannel` (работа с файлами).

## Интерфейс - маркерный
По сути, это **маркерный интерфейс**: он просто говорит — "этот канал может и читать, и писать".
```java
package java.nio.channels;

public interface ByteChannel 
        extends ReadableByteChannel, WritableByteChannel {
    // ❗❗Специальных методов нет.❗❗
    
    // Интерфейс наследует:
    //  - int read(ByteBuffer dst)
    //  - int write(ByteBuffer src)
    //  - boolean isOpen()
    //  - void close()
}
```

## 💡 Особенности и подводные камни 💡

1. **Симметрия**
    - В отличие от `InputStream` и `OutputStream`, `ByteChannel` объединяет оба направления в одном объекте.
        
    - Это критично для сетевых соединений (`SocketChannel`), где ты и читаешь, и пишешь через один канал.
        
2. **Поведение read/write**
    - `read(ByteBuffer)` возвращает число байтов или `-1` при EOF (как в `ReadableByteChannel`).
        
    - `write(ByteBuffer)` возвращает число записанных байтов (как в `WritableByteChannel`).
        
3. 💡**EOF и закрытие**
    - 💡После вызова `close()` оба направления закрываются.
        
    - В сокетах может быть ситуация: чтение завершено (EOF), но запись ещё возможна, или наоборот → это решается через `shutdownInput()/shutdownOutput()` у `SocketChannel`.
        
4. 💡**Неблокирующий режим**
    - 💡 Для сетевых каналов (`SocketChannel`, `DatagramChannel`) доступен неблокирующий режим.
        
    - 💡Тогда `read()` может вернуть `0`, а `write()` может не записать ничего.
        
5. **Расширение**
    - `ByteChannel` сам по себе не добавляет новых методов, но его расширяют:
        - `SeekableByteChannel` (добавляет позиционирование и `size()`),
            
        - `AsynchronousByteChannel` (для async I/O).

## ✅ Best Practices ✅
- 🧠Используй `ByteChannel`, когда тебе важен **универсальный интерфейс для чтения и записи** (например, если метод должен работать и с файлом, и с сокетом).
    
- При записи всегда проверяй `buffer.hasRemaining()`, потому что `write()` может записать меньше, чем содержится в буфере.
    
- 💡❗Обрабатывай `read() == -1` (EOF), особенно для файлов и сокетов.
    
- Для неблокирующего режима используй вместе с `Selector`.
    
- Для файлов — если нужно позиционирование, лучше использовать `SeekableByteChannel`.

## Пример
```java
import java.io.IOException;
import java.nio.ByteBuffer;
import java.nio.channels.ByteChannel;
import java.nio.file.Files;
import java.nio.file.Path;
import java.nio.file.StandardOpenOption;

public class ByteChannelFileExample {
    public static void main(String[] args) throws IOException {
        Path path = Path.of("example.txt");

        // try-with-resources закроет канал автоматически
        try (ByteChannel channel = Files.newByteChannel(
                path,
                StandardOpenOption.CREATE,
                StandardOpenOption.READ,
                StandardOpenOption.WRITE)
        ) {
            // Запись
            ByteBuffer buffer = ByteBuffer.wrap("Hello ByteChannel!\n".getBytes());
            
            while (buffer.hasRemaining()) {
                channel.write(buffer);
            }

            // Чтение
            buffer.clear();
            
            channel.read(buffer);
            buffer.flip();
            while (buffer.hasRemaining()) {
                System.out.print((char) buffer.get());
            }
        }
    }
}
```
