# ⭐ 0. Основные схемы HTTP-аутентификации

| Схема                           | Принцип                               | Особенности                           |
| ------------------------------- | ------------------------------------- | ------------------------------------- |
| **Basic**                       | Отправка `username:password` в Base64 | Простая, но небезопасная без HTTPS    |
| **Digest**                      | Хэш от пароля и nonce                 | Безопаснее Basic, без передачи пароля |
| ⭐**Bearer (Token)**⭐            | JWT или OAuth2 токен                  | Современный вариант (REST API)        |
| **Negotiate / NTLM / Kerberos** | Обмен challenge-response              | Используется в корпоративных AD-сетях |
| **Custom (e.g., API-Key)**      | Своя схема проверки токенов           | Для REST API и микросервисов          |

# Схема взаимодействия клиента и сервера

```sql
┌──────────────┐        ┌──────────────────────────┐
│ Java-клиент  │        │ HTTP-сервер (Spring/Tomcat) │
└──────┬───────┘        └───────────────┬──────────┘
       │                                │
       │ 1️⃣ GET /secure                 │
       ├───────────────────────────────►│
       │                                │ Проверка прав
       │                                │ → нет авторизации
       │                                │
       │ 2️⃣ 401 Unauthorized + WWW-Authenticate
       │◄───────────────────────────────┤
       │                                │
       │ 3️⃣ Authenticator возвращает PasswordAuthentication
       │                                │
       │ 4️⃣ GET /secure + Authorization: Basic ...
       ├───────────────────────────────►│
       │                                │ Проверка логина/пароля
       │                                │ → OK
       │ 5️⃣ 200 OK                      │
       │◄───────────────────────────────┤

```

# 📕 1. Аутентификация на клиенте
## 1️⃣ Клиент отправляет обычный HTTP/HTTPS-запрос
Ты вызываешь, например:
```java
HttpURLConnection conn = new URL(
	"https://example.com/secure")
	.openConnection();
	
conn.getInputStream();
```
🔸JVM создаёт TCP-соединение и отправляет HTTP-запрос (без `Authorization`-заголовка).  

🔸Если ресурс НЕ ТРЕБУЕТ АВТОРИЗАЦИИ — всё проходит успешно.

## 2️⃣ Сервер возвращает 401 или 407
Если сервер или прокси требует аутентификацию, он отвечает, например:
```http
HTTP/1.1 401 Unauthorized
WWW-Authenticate: Basic realm="Restricted Area"
```
или (если это прокси):
```http
HTTP/1.1 407 Proxy Authentication Required
Proxy-Authenticate: Basic realm="ProxyServer"
```

🔸Java получает этот ответ и видит заголовок `WWW-Authenticate` или `Proxy-Authenticate`.

## 3️⃣ Внутри сетевого стека срабатывает `AuthenticationInfo`
Внутренние классы `sun.net.www.protocol.http.AuthenticationInfo` и  
`sun.net.www.protocol.http.AuthCacheImpl` отвечают за кеширование и обработку авторизации.

❗❗Они проверяют:
- Есть ли уже кешированные логин/пароль для этого хоста/realm?
- Если **нет**, вызывают **`Authenticator.requestPasswordAuthentication()`**.

## 4️⃣ Вызов `Authenticator.getPasswordAuthentication()`

Вот здесь вступает в игру твой класс `Authenticator`, если он был задан:
```java
Authenticator.setDefault(new MyAuthenticator());
```

JVM вызывает его примерно так:
```java
PasswordAuthentication pa =
    Authenticator.requestPasswordAuthentication(
        host, 
        inetAddress, 
        port, 
        protocol, 
        prompt, 
        scheme, 
        url, 
        requestorType);
```

🔸 Этот ==👆статический метод 👆==  - **делегирует вызов** в твой переопределённый метод:
```java
protected PasswordAuthentication getPasswordAuthentication()
```

## 5️⃣ Твой код возвращает логин/пароль
Ты возвращаешь объект:
```java
return new PasswordAuthentication("user", "pass".toCharArray());
```
🔸JVM получает его и **кэширует** на время (в `AuthCache`).

## 6️⃣ Java автоматически повторяет запрос
Теперь стек `HttpURLConnection` или `HttpClient` делает **повторный запрос**:
- добавляет заголовок
```http
Authorization: Basic base64(user:pass)
```
или
```http
Proxy-Authorization: Basic base64(user:pass)
```
- и **повторно отправляет HTTP-запрос** к тому же URI.
    

Если сервер принимает авторизацию — ответ приходит с кодом `200 OK`.

## 7️⃣ Поведение при повторных запросах
Если тот же домен/realm встречается снова:
- данные **берутся из кеша (`AuthCacheImpl`)**,
- `Authenticator` **не вызывается повторно**,
- заголовки `Authorization` добавляются автоматически.
    

Если соединение закрыто и realm новый — `Authenticator` вызывается снова.

---
# 💡 Важные детали

1️⃣ **`Authenticator` ==НЕ ВЫЗЫВАЕТСЯ ЗАРАНЕЕ==** — только _в ответ_ на ошибку 401/407.  

2️⃣ **Не хранит пароли** — только возвращает их по запросу.  

3️⃣ **Потокобезопасен**, но должен быть быстрым — иначе блокирует сетевой поток.  

4️⃣ ❗❗==**Кеширование** делает система==, а не твой код.  

5️⃣ Если возвращаешь `null` → соединение прерывается (`IOException: Authentication failed`).  

6️⃣ Работает для всех протоколов, поддерживающих аутентификацию (`http`, `https`, `ftp`, `socks`).  

7️⃣ `Authenticator` можно комбинировать с `ProxySelector` и `CookieManager` для полной сетевой конфигурации.

# 📕 2. Аутентификация на сервере
## Механизм HTTP-аутентификации (RFC 7235)

## 1️⃣ Клиент делает обычный запрос
```http
GET /private/data HTTP/1.1
Host: example.com
```

## 2️⃣ Сервер проверяет права доступа
Если ресурс защищён, сервер отвечает:
```http
HTTP/1.1 401 Unauthorized
WWW-Authenticate: Basic realm="Private Area"
```

Этот заголовок говорит клиенту:  
«Ты не авторизован, я ожидаю Basic-аутентификацию».

## 3️⃣ Клиент повторяет запрос с заголовком `Authorization`
```http
GET /private/data HTTP/1.1
Host: example.com
Authorization: Basic YWRtaW46cGFzcw==
```

> [!NOTE] ⭐⭐Ключ авторизации через `Base64` (см `java.util`)⭐⭐
> где `YWRtaW46cGFzcw==` — это Base64("admin:pass").

## 4️⃣ Сервер декодирует и проверяет
Если данные совпадают с разрешёнными пользователями —  
отправляется ответ:
```http
HTTP/1.1 200 OK
```

иначе снова:
```http
HTTP/1.1 401 Unauthorized
```

# Под капотом (серверная логика)
На уровне Java сервер делает примерно так:
```java
String auth = request.getHeader("Authorization");

if (auth == null || !isAuthorized(auth)) {
    response.setHeader("WWW-Authenticate", "Basic realm=\"MyRealm\"");
    response.sendError(HttpServletResponse.SC_UNAUTHORIZED);
    return;
}
```

Если клиент отправил корректный `Authorization`:
```java
String credentials = new String(Base64.getDecoder()
    .decode(auth.substring("Basic ".length())));
    
String[] parts = credentials.split(":");

String username = parts[0];
String password = parts[1];

if (checkCredentials(username, password)) {
    chain.doFilter(request, response); // пропускаем дальше
} else {
    response.sendError(HttpServletResponse.SC_UNAUTHORIZED);
}

```

# 📕 3. Настройка на сервере (по типам технологий)
## 🔸 1. Встроенная аутентификация контейнера (например, Tomcat)

В Java EE / Jakarta EE аутентификация настраивается в `web.xml`:
```xml
<login-config>
    <auth-method>BASIC</auth-method>
    <realm-name>Secure Area</realm-name>
</login-config>

<security-constraint>
    <web-resource-collection>
        <web-resource-name>Private Data</web-resource-name>
        <url-pattern>/private/*</url-pattern>
    </web-resource-collection>
    <auth-constraint>
        <role-name>admin</role-name>
    </auth-constraint>
</security-constraint>
```

👉 Tomcat при доступе к `/private/*` вернёт `401` и заголовок `WWW-Authenticate`.
Он сравнит данные из `Authorization` с логинами в `tomcat-users.xml`:
```xml
<user username="admin" password="1234" roles="admin"/>
```

## 🔸 2. Spring Security (наиболее распространённый способ в современных Java-приложениях)
Spring Security перехватывает HTTP-запросы через `FilterChain`  
и сам управляет аутентификацией и заголовками.

Пример конфигурации:
```java
@Bean
SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    http
        .authorizeHttpRequests()
            .requestMatchers("/public/**").permitAll()
            .anyRequest().authenticated()
        .and()
        .httpBasic(); // включает Basic Authentication
    return http.build();
}
```

➡ Когда неавторизованный пользователь запрашивает `/secure/data`,  
Spring Security автоматически отвечает:
```http
HTTP/1.1 401 Unauthorized
WWW-Authenticate: Basic realm="Realm"
```
А при следующем запросе с `Authorization: Basic ...` —  
декодирует Base64, проверяет пользователя через `UserDetailsService`,  
и даёт доступ при совпадении.

## 🔸 3. Настройка на уровне веб-сервера (Apache, Nginx)

### Apache
```apache
<Directory "/var/www/secure">
    AuthType Basic
    AuthName "Restricted Area"
    AuthUserFile /etc/httpd/.htpasswd
    Require valid-user
</Directory>
```

Файл `.htpasswd` содержит хэшированные пароли (например, bcrypt или SHA).

### Nginx
```nginx
location /secure/ {
    auth_basic "Restricted Area";
    auth_basic_user_file /etc/nginx/.htpasswd;
}
```


## 🔸 ⭐⭐⭐4. Аутентификация на REST API (Token / JWT)⭐⭐
Современные API обычно не используют `WWW-Authenticate`,  
а работают через токены:
```http
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
```
Сервер (например, Spring Boot + JWT) проверяет подпись токена и срок действия.


## 🔸 5. Proxy-аутентификация
Прокси (например, Squid, корпоративный прокси, Apache mod_proxy)  
тоже может требовать аутентификацию.

Пример заголовков:
```http
HTTP/1.1 407 Proxy Authentication Required
Proxy-Authenticate: Basic realm="Proxy"
```

Тогда `Authenticator` на клиенте вернёт `PasswordAuthentication`,  
и Java добавит заголовок:
```http
Proxy-Authorization: Basic YWRtaW46cGFzcw==
```
