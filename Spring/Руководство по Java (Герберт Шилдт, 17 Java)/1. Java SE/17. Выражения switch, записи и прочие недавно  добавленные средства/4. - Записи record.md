### Основная идея
- В JDK 16 появился новый вид классов — `record`.
    
- Это ==**иммутабельные контейнеры== для данных**. (Не изменяемые)

> [!NOTE] Они ==автоматически генерируют==
> - ⭐Поля (`private final`).
> - Конструктор, с параметрами равным списку компонентов в `record MyRecord(список компонентов)`.
> - Геттеры.
> - ⭐Методы `equals()`, `hashCode()`, `toString()`.

***
# 📕 1. Основы записей (record)
Она объявляется с помощью ==контекстно-чувствительного ключевого слова==  - `record`, которое трактуется как ключевое слово только в контексте объявления записи.

Ниже показана общая форма объявления записи:
```java
record имя_записи(список компонентов) {
	//Необязательные операторы тела
}
```

> [!NOTE] ⭐Особенности:⭐
> `В списке компонентов` `ОПРЕДЕЛЯЮТСЯ ДАННЫЕ (поля класса)`, которые будут храниться в записи.
> 
> Тело записи является необязательным - ==компилятор автоматически предоставит элементы==, необходимые для хранения данных, 
> - создаст `конструктор` записи
> - `методы` получения, которые обеспечат доступ к данным, (💡ПРОСТО ИМЕНА, БЕЗ `get...()`💡)
> - `переопределит методы` `toString()` , `equals()` и `hashCode()`

# 📕 2. Пример и ключевые особенности
```java
record Employee(String name, int idNum) {}
```
Запись имеет имя `Employee` и состоит из двух компонентов: 
- строкового name 
- целочисленного idNum. 

```java
Employee emp = new Employee("Doe, John", 1047);

System.out.println("Идентификатором сотрудника " + 
		emp.name() + 
		" является " + 
		emp.idNum() 
	);
```

## Свойства
- ==Данные== **поверхностно ==неизменяемы==** (НЕЛЬЗЯ ПЕРЕПРИСВОИТЬ ПОЛЕ, но если там ссылка на объект, `объект можно мутировать`, но не ссылку).
    
- ⭐`record` НЕ МОЖЕТ НАСЛЕДОВАТЬ классы (всегда наследует `java.lang.Record`).
    
- ⭐Но ==может реализовывать интерфейсы==.
    
- Всегда `final` (==не наследуется==).
	
- Запись может быть ОБОБЩЕННОЙ
```java
class Bar {
	double money;
}

record Employee<T>(String name, int idNum, T clazz) {}

public static void main(String[] args) throws FileNotFoundException  {
	Employee<Bar> emp = new Employee<>("Tom", 1010, new Bar());
}
```
	
- ⭐Помимо полей, ассоциированных с компонентами записи, л==юбые другие поля должны быть статическими==.
```java
record Employee(String name, int idNum) {
	static double money; // ✅ Можно
	
	double price; // ❌ Ошибка, NON-STATIC field
}
```

## Конструкторы
- 🟢Канонический (по всем полям) создаётся автоматически, но можно переопределить.
```java
record Employee(String name, int idNum) {
	Employee(String name, int idNum){
		this.name = name;
		this.idNum = idNum;
	}
}

// ⭐ Особенности ⭐
// ✅ 1. Для канонического конструктора ТИПЫ И ИМЕНА ПАРАМЕТРОВ обязаны СОВПАДАТЬ С УКАЗАННЫМИ в объявлении ЗАПИСИ.
// ✅ 2. По завершению конструктора, ВСЕ ПОЛЯ ДОЛЖНЫ БЫТЬ ИНИЦИАЛИЗИРОВАНЫ
// ✅ 3. Конструктор должен быть, по меньшей мере, ТАКИМ ЖЕ ДОСТУПНЫМ, КАК в объявлении ЗАПИСЬ. (если public record, значит и конструктор public) 
// ✅ 4. НЕ МОЖЕТ СОДЕРЖАТЬ КОНСТРУКЦИЮ throwS и НЕ МОЖЕТ БЫТЬ ОБОБЩЕННЫМ
record Employee(String name, int idNum, double money) {
	public Employee(String name, int idNum, double money) {
		this.name = name;
		this.idNum = idNum;
		this.money = money;
		
		//Своя другая логика
		if(idNum < 0) {
			// ❗❗✅❗❗НО МОЖНО ВЫБРАСЫВАТЬ ИСКЛЮЧЕНИЯ ЧЕРЕЗ throw
			throw new RuntimeErrorException(null, "Heheh"); 
		}
	}
}

// ✅ 5. НЕ МОЖЕТ ВЫЗЫВАТЬ другой НЕканонический КОНСТРУКТОР
```

- Компактный (без параметров, значения доступны неявно).
```java
record Employee(String name, int idNum) {
	public Employee { // НЕТ ПАРАМЕТРОВ и ()
		name = name.trim();
		idNum = idNum+1;
	}
}
```
    
- 🟢⭐Неканонический (альтернативный, обязан вызвать другой конструктор через `this`).
```java
class Bar {
	private double money = 5;
	
	double getMoney() {
		return money;
	}
}

record Employee(String name, int idNum, double money) {

	public Employee(String name, int idNum, Bar obj) {
		
		//✅⭐Вызов канонического конструктора
		this(name, idNum, obj.getMoney()); 
		
		//Своя другая логика
	}
}


// ⭐ ОСОБЕННОСТИ ⭐
// ✅ 1. Основное требование - любой неканонический конструктор ОБЯЗАН СНАЧАЛА ВЫЗВАТЬ КАНОНИЧЕСКИЙ КОНСТРУКТОР через this 
```
## Методы получения

> [!NOTE] Можно переопределять, но нельзя нарушать неизменяемость.  
> Хотя и редко, но возникает необходимость в создании собственной реализации метода получения. 
> 
> ⭐В случае объявления метода получения, его НЕЯВНАЯ ВЕРСИЯ БОЛЬШЕ НЕ ПРЕДОСТАВЛЯЕТСЯ.

Одна из причин объявления собственного метода получения связана с возможностью генерации исключения
```java
record Employee(String name, int idNum) {

	public String name() { // ✅ Возвращаемое значение String, имя тоже name
		if(name == "Tom") {
			// ✅ throw можно
			// ❌ throwS в объявлении метода НЕЛЬЗЯ
			throw new RuntimeErrorException(null, "Hate Tom");
		}
		return this.name;
	}
}

// Особенности
// ✅ 1. Метод получения обязан иметь ТАКОЙ ЖЕ ВОЗВРАЩАЕМЫЙ ТИП И ИМЯ, как и компонент, который он получает
// ✅2. Должен быть ЯВНО ОБЪЯВЛЕН public
// ✅ 3. НЕЛЬЗЯ ИСПОЛЬЗОВАТЬ throwS
// ✅⭐4. Наконец, метод получения не должен быть ни обобщенным, ни статическим.
```


> [!NOTE] ✅⭐ Best Practices ⭐ ✅
> ==`Лучше ДОБАВЛЯТЬ НОВЫЕ МЕТОДЫ`== (например, `lastName()`), А НЕ МЕНЯТЬ ПОВЕДЕНИЕ СТАНДАРТНЫХ
