# 📕 1. Защищенное общее состояние (Protected Shared State)
В многопоточных программах у нас есть **общее изменяемое состояние** (например, поле класса). Если несколько потоков будут его читать и писать одновременно — появляются **состояния гонки (race condition)**.

👉 Этот паттерн говорит: _«==Если данные разделяются между потоками, нужно сделать так, чтобы все доступы к ним были защищены синхронизацией==»_
## Назначение
- ==Оградить **общие данные** от одновременного доступа==.
    
- Сделать класс **потокобезопасным**.
    
- Убрать возможность гонок, неконсистентности и visibility-проблем (невидимости изменений между потоками).
## 💡Реализация в Java
1. **Блокировка (lock)**:  
    Обычно используется встроенный монитор (`synchronized`), либо `ReentrantLock`.
    
2. **Защита всех путей доступа**:  
    Любая операция (get/set) должна быть защищена.
    
3. **Анотации (для понимания и анализа)**:
    - `@ThreadSafe` — класс потокобезопасен.
        
    - `@GuardedBy("this")` — поле защищено монитором текущего объекта (`synchronized`).
## Пример
```java
@ThreadSafe
public class ProtectedSharedState {
	
	//Чито косметическая аннотация, подсказывает что это поле используется 
	//несоклькими потоками
    @GuardedBy("this") 
    private int counter;

    // доступ только через synchronized методы
    public synchronized void increment() {
        counter++;
    }

    public synchronized int getCounter() {
        return counter;
    }
}
```

# 📕 2. Atomic Compound Actions (Атомарные составные действия)
## 🚩 Суть проблемы
- В многопоточке часто встречаются составные операции: **сначала проверить, потом сделать** или **прочитать-изменить-записать**.
    
- ==Если эти действия не выполняются **атомарно**, возникает _race condition_.==

### Классические случаи:
1. **check-then-act** (проверка перед действием)
```java
if (map.containsKey(key)) {   // check
    return map.get(key);      // act
}
```
🛑🛑🛑Между `containsKey` и `get` другой поток может изменить `map`.

## ✅ Решение: паттерн _Atomic Compound Actions_

Смысл: ==**выполняем составное действие внутри одной критической секции**== (через `synchronized` или `Lock`).

```java
@ThreadSafe
public class LazyInit {
    @GuardedBy("this")
    private ExpensiveObject instance;

    public synchronized ExpensiveObject getInstance() {
        if (instance == null) {     // check
            instance = new ExpensiveObject(); // act
        }
        return instance;
    }
}
```
→ здесь проверка и создание объекта выполняются атомарно.

## 💡 Главное правило
> [!NOTE] Несколько действий под одной блокировкой
>  Если действие состоит из нескольких шагов (проверка → чтение → изменение → запись), оно должно выполняться под одной блокировкой или через атомарные классы (`AtomicInteger`, `ConcurrentHashMap.computeIfAbsent` и т. д.).

# 📕3. **Lock Splitting (разделение блокировок)**.

## 🚩 Проблема
Если у класса есть несколько **независимых** общих переменных, часто делают так:
```java
@ThreadSafe
public class BadExample {
    private List<Object> hotState = new ArrayList<>();
    private Object anotherState;

    private final Lock lock = new ReentrantLock(); // 🛑Общий lock для двух полей

    public void add(Object o) {
        lock.lock();
        try {
            hotState.add(o);
        } finally {
            lock.unlock();
        }
    }

    public Object getAnotherState() {
        lock.lock();
        try {
            return anotherState;
        } finally {
            lock.unlock();
        }
    }
}
```
❌ Минус: один **общий лок** защищает всё, хотя `hotState` и `anotherState` не связаны.  
→ В итоге поток, который работает с `hotState`, **мешает** потоку, который работает с `anotherState`. Возникает **лишняя конкуренция за лок (lock contention)**.

## ✅ Решение: Lock Splitting
Если переменные **логически независимы**, мы защищаем каждую своей блокировкой.

В твоём примере сделано именно так:
- `hotState` защищается отдельным локом `lockState`
- `anotherState` — другим локом `lockOtherStates`

Это значит:
- Поток 1 может добавлять элементы в список `hotState`
- Поток 2 в это же время может читать/писать `anotherState`  
    → Они **не мешают** друг другу.

## Пример
### 🔶Hashtable (устаревший)
У него **одна глобальная блокировка** → параллельность низкая.
### 🔶ConcurrentHashMap
В старых версиях Java использовался **lock striping** (разделение карты на сегменты, у каждого — свой лок).  
Это и есть развитие идеи **Lock Splitting**.  
В новых версиях — ещё более тонкая гранулярность (CAS + узлы дерева/списка).

```java
@ThreadSafe
public class LockSplit {
	private List<Object> hotState; //условный список чего-то

	private Object anotherState; //условный логин
	private Object otherState; //условный пароль

	private Lock lockHotState = new ReentrantLock();
	private Lock lockOtherStates = new ReentrantLock();

	//Для List<Object> свой лок
	public List<Object> stateReader() {
		lockHotState.lock();
		try {
			return hotState;
		} finally {
			lockState.unlock();
		}
	}
	public void stateWriterMethod(Object param) {
		lockHotState.lock();
		try {
			this.hotState.add(param);
		} finally {
			lockState.unlock();
		}
	}
	
	//Для anotherState и otherState свой лок
	public Object anotherStateReader() {
		lockOtherStates.lock();
		try {
			return anotherState;
		} finally {
			lockOtherStates.unlock();
		}
	}

	public void anotherStateWriter(Object param) {
		lockOtherStates.lock();
		try {
			this.anotherState = param;
		} finally {
			lockOtherStates.unlock();
		}
	}
	
	//Для anotherState и otherState свой лок
	public Object otherStateReader() {
		lockOtherStates.lock();
		try {
			return otherState;
		} finally {
			lockOtherStates.unlock();
		}
	}

	public void otherStateWriter(Object param) {
		lockOtherStates.lock();
		try {
			this.otherState = param;
		} finally {
			lockOtherStates.unlock();
		}
	}
}
```

# 📕 4. **Thread Local Confinement (локализация на потоках)**.
## 🚩 Проблема
Многие классы в Java **не потокобезопасны**.  
Например:
- `SimpleDateFormat` (классический пример — гонки при парсинге/форматировании дат);
- `DecimalFormat`;
- любые коллекции без синхронизации (`ArrayList`, `HashMap`).

Если такие объекты разделять между потоками → получаем **race condition** и непредсказуемое поведение.

## ✅ Решение: Thread Local Confinement
Идея: у каждого потока **своя копия** объекта.  

🛑🛑🛑Используем `ThreadLocal<T>`:
- при первом вызове `get()` для потока → создаётся новая копия объекта (через `initialValue()`);
- в дальнейшем этот поток всегда работает со **своей версией**, независимой от других.

## Практичный пример

### 🔶 `SimpleDateFormat` без `ThreadLocal` (опасно!)
```java
public class BadDateFormatter {
    private static final SimpleDateFormat formatter = new SimpleDateFormat("yyyy-MM-dd");

    public String format(Date date) {
        return formatter.format(date); // гонки!
    }
}
```
### 🔶С `ThreadLocal` (безопасно ✅)
```java
public class SafeDateFormatter {
    private static final ThreadLocal<SimpleDateFormat> formatter =
        ThreadLocal.withInitial(() -> new SimpleDateFormat("yyyy-MM-dd"));

    public String format(Date date) {
        return formatter.get().format(date);
    }
}
```
## ⚠️ Важные моменты
1. **Утечки памяти**
    - `ThreadLocal` хранит значения в `ThreadLocalMap`, привязанной к потоку.
    - Если поток — это **пул потоков (например, `ExecutorService`)**, объекты будут жить, пока жив поток.  
        👉 Поэтому нужно **чистить `ThreadLocal`** вызовом `remove()`, если объект больше не нужен. (см Класс ThreadLocal)

2. **Не использовать для больших объектов**  
    Иначе можно «раздуть» память (каждый поток хранит свою копию).
    
3. **Современные альтернативы**  
    Иногда проще использовать уже потокобезопасные классы (`DateTimeFormatter` в Java 8+, который immutable → потокобезопасный).

# 📕⭐⭐⭐ 5. Producer-Consumer (Производитель-Потребитель)
**Producer-Consumer** — паттерн для организации **потокобезопасного обмена данными между потоками**, где есть отдельные **производители** и **потребители**.
## 🚩 Проблема
- В многопоточной среде часто есть логика **создания данных** (Producer) и **обработки данных** (Consumer).
- Если производитель и потребитель работают в одном потоке → блокировка и снижение производительности.
    
- Нужно разделить производство и потребление с минимальной синхронизацией.

## ✅Решение (идея паттерна)
1. Использовать **общую очередь** (обычно `BlockingQueue`) для передачи данных.
2. Производитель (`Producer`) помещает данные в очередь (`put()`).
3. Потребитель (`Consumer`) извлекает данные из очереди (`take()` или `poll()`).
    
4. Очередь автоматически **синхронизирует доступ**, блокируя потоки при необходимости:
    - Producer ждёт, если очередь заполнена.
    - Consumer ждёт, если очередь пуста.
        
5. Можно запускать **несколько производителей и потребителей**, они безопасно делят очередь.
## Компоненты
- `BlockingQueue<T`> — потокобезопасная очередь.
    
- `Producer (Runnable/Callable)` — создает данные и добавляет в очередь.
    
- `Consumer (Runnable/Callable)` — извлекает данные и обрабатывает их.
    
- `ExecutorService / Threads `— управляет потоками производителей и потребителей.

## Правила использования
1. **Использовать потокобезопасную очередь**, например:
    - `LinkedBlockingQueue`, `ArrayBlockingQueue`, `PriorityBlockingQueue`.
        
2. **Обрабатывать завершение**:
    - ==Consumer должен уметь завершаться, если данных больше нет== (например, через `poll(timeout)` или специальный «poison pill»).
        
3. **Можно иметь несколько производителей и потребителей**, чтобы масштабировать обработку.
    
4. **Не блокировать основной поток** — использовать ExecutorService или ThreadPool.

```java
public class ProducerConsumer {

	private BlockingQueue<String> data = new LinkedBlockingQueue<>();

	private Callable<Void> consumer = () -> {
		while (true) {
			var dataUnit = data.poll(5, TimeUnit.SECONDS);
			if (dataUnit == null)
				break;
			System.out.println("Consumed " + dataUnit + " from " + Thread.currentThread().getName());
		}
		return null;
	};

	private Callable<Void> producer = () -> {
		for (int i = 0; i < 90_000; i++) {
			var dataUnit = UUID.randomUUID().toString();
			data.put(dataUnit);
		}
		return null;
	};

	public void run(long forHowLong, TimeUnit unit) throws InterruptedException {
		var pool = Executors.newCachedThreadPool();
		pool.submit(producer);
		pool.submit(consumer);
		pool.submit(consumer);
		
		pool.shutdown();
		pool.awaitTermination(forHowLong, unit);
	}

	public static void main(String[] args) {
		var producerConsumer = new ProducerConsumer();
		try {
			producerConsumer.run(5, TimeUnit.SECONDS);
		} catch (InterruptedException e) {
			e.printStackTrace();
		}
	}
}
```

# 📕⭐⭐6. Safe Lazy Initialization (Безопасная отложенная инициализация) ⭐⭐

## 🚩Проблема
Классическая ленивая инициализация вроде такой:
```java
private static Object resource;

public static Object getResource() {
    if (resource == null) {
        resource = new Object();
    }
    return resource;
}
```
⚠️ Она **небезопасна** в многопоточной среде: два потока могут одновременно проверить `resource == null` и создать два экземпляра. Даже с `volatile` и double-checked locking (DCL) можно наделать ошибок, если не знать тонкости JMM (Java Memory Model).

## ✅ Решение через _Holder Idiom_
```java
public class SafeInitializationHolder {
    private static class ResourceHolder {
        static final Object resource = new Object();
    }

    public static Object getResource() {
        return ResourceHolder.resource;
    }
}
```
💡 **Почему это работает?**
1. Класс `ResourceHolder` ==загружается **только при первом обращении** к нему== (lazy).
    
2. По спецификации JVM:
    - ==инициализация статических полей выполняется **однократно**;==
        
    - гарантируется **happens-before**: все записи, сделанные при инициализации класса, становятся видимыми для любых потоков, обращающихся к нему позже.
        
3. **Гарантия видимости** - JVM обеспечивает видимость статической инициализации для всех потоков

> [!NOTE] Работает ТОЛЬКО если `static` поле объявлено как`final`
> Поскольку JVM захватывает блокировку во время инициализации [JLS 12.4.2], и эта блокировка захватывается каждым потоком, по крайней мере, один раз, чтобы гарантировать, что класс был загружен, записи в память, сделанные во время статической инициализации, автоматически видны всем потокам. 
> 
> Таким образом, статически инициализированные объекты не требуют явной синхронизации ни во время построения, ни при ссылке. 
> 
> ⭐⭐⭐Однако это относится только к состоянию as-constructed - ==если объект является изменяемым, синхронизация по-прежнему требуется как для чтения, так и для записи==, чтобы сделать последующие изменения видимыми и избежать повреждения данных.

## 💡🛑Отличие от других подходов🛑💡
- ✅ Быстрее и чище, чем `synchronized getInstance()`. 💡🛑
    
- ✅ Безопаснее, чем double-checked locking (DCL) без `volatile`.
    
- ✅ Автоматически потокобезопасно благодаря JVM, а не коду.

# 📕 ⭐7. Thread Task Cancel
**Thread Task Cancel** — паттерн для **отменяемых фоновых задач**, реализованных через `Thread` или `Runnable`.
## Проблема
- В фоновых потоках часто запускаются долгие или бесконечные задачи.
- Порой требуется **остановить задачу**, не дожидаясь её естественного завершения.
    
- Простое `stop()` в Java **небезопасно** и устарело → нужен корректный способ остановки.

## Решение (идея паттерна)
1. 🔶В задаче (`Runnable`) проверять **флаг прерывания**:
```java
while (!Thread.currentThread().isInterrupted()) {
    // работа
}
```

2. 🔶Для отмены задачи вызываем `thread.interrupt()`.
3. 🔶Поток сам завершает цикл и корректно выходит из задачи.
## Компоненты
- **Thread / Runnable** — выполняемая задача.
- **Цикл с проверкой `isInterrupted()`** — «точка выхода» из задачи.
    
- **Метод `cancel()`** — устанавливает флаг прерывания через `interrupt()`.

## Правила использования
1. 🔶**Проверять прерывание в цикле**:
```java
while (!Thread.currentThread().isInterrupted()) { ... }
```

2. 🔶**Не игнорировать InterruptedException**:
    Если поток блокируется в `sleep()`, `wait()`, `join()`, `BlockingQueue.take()`, нужно обработать `InterruptedException` и выйти.
    
3. 🔶**Не использовать устаревший Thread.stop()** — небезопасно.
    
4. 🔶**Использовать флаги прерывания только для Cooperative Cancellation** — поток сам решает, когда выйти.

```java
public class ThreadTaskCancel {

	private Thread thread;
	private Runnable task = () -> {
		while (!Thread.currentThread().isInterrupted()) {
			//action
		}
	};

	public void run() {
		thread = new Thread(task);
		thread.start();
	}

	public void cancel() {
		if (thread != null) {
			thread.interrupt();
		}
	}
}
```

# 📕⭐ 8. Safe Publishing (Безопасная публикация)

## Суть паттерна Safe Publishing
**Публикация объекта** = ==сделать ссылку на объект доступной для других потоков==.  
==**Опасность==** — если публикация сделана неправильно, другие потоки могут увидеть ==объект в **неполностью инициализированном состоянии** ==(например, поля ещё не проинициализированы). Это связано с тем, как работает **Java Memory Model (JMM)** и reorderings.
## Основные безопасные способы публикации:
1. 🔶**Статическая инициализация**  
==Статические поля и статические блоки ВЫПОЛНЯЮТСЯ ОДИН РАЗ при загрузке класса== → гарантирована видимость для всех потоков.
```java
public class SafePub1 {
    public static final Object obj = new Object(); // безопасно
}
```

2. 🔶 **Инициализация в конструкторе + final поля**  
Все `final`-поля и состояние, доступное из конструктора, становятся видимыми сразу после завершения конструктора.
```java
public class SafePub2 {
    private final Object obj;
    public SafePub2() {
        this.obj = new Object();
    }
    public Object get() { return obj; } // безопасно
}
```

3. 🔶 **Volatile-ссылка**  
Если ссылка объявлена как `volatile`, то запись в неё и последующее чтение из другого потока синхронизированы.
```java
public class SafePub3 {
    private static volatile Object obj;
    public static void init() {
        obj = new Object();
    }
    public static Object get() {
        return obj;
    }
}
```

4. 🔶 **AtomicReference**  
Делает то же самое, но явно.
```java
private static final AtomicReference<Object> ref = new AtomicReference<>();
```

5. 🔶 **Публикация через потокобезопасные коллекции**  
Например, `ConcurrentHashMap.put(key, object)` — гарантирует, что объект будет виден другим потокам.

# 📕⭐ 9.Explicit Condition Queues (Явные очереди условий)
## 💡 Идея паттерна
- В многопоточной среде часто нужно **ждать изменения состояния**, которое может произойти в другом потоке.
	    Например:
		- буфер может быть **пустым** или **полным**,
	    - счётчик может быть **достигнут** или **ещё нет**,
	    - ресурс может быть **доступен** или **занят**.
	    
- Для этого и нужны **очереди условий (Condition Queues)** — они ==позволяют потокам засыпать и просыпаться **по определённому предикату**==, а не крутиться в цикле.
    
В `ReentrantLock` мы можем создавать несколько независимых условий (`Condition`) для разных предикатов.

Одни потоки должны ждать, пока состояние станет подходящим, а другие — менять это состояние.  
==Если делать на `synchronized + wait/notify`, то есть только **одна очередь ожидания**, получается что все ждущие потоки перемешаны.==

```java
public class ExplicitConditionQueue {

	private static final int LIMIT = 5;
	private int messageCount = 0;
	
	private Lock lock = new ReentrantLock();
	private Condition limitReachedCondition = lock.newCondition();
	private Condition limitUnreachedCondition = lock.newCondition();

	public void stopMessages() throws InterruptedException {
		lock.lock();
		try {
			while (messageCount < LIMIT) {
				limitReachedCondition.await();
			}
			System.err.println("Limit reached. Wait 2s");
			Thread.sleep(2000);
			messageCount = 0;
			limitUnreachedCondition.signalAll();
		} finally {
			lock.unlock();
		}
	}

	public void printMessages(String message) throws InterruptedException {
		lock.lock();
		try {
			while (messageCount == LIMIT) {
				limitUnreachedCondition.await();
			}
			System.out.println(message);
			messageCount++;
			limitReachedCondition.signalAll();
		} finally {
			lock.unlock();
		}
	}

	public static void main(String[] args) {
		var queue = new ExplicitConditionQueue();
		// Will run indefinitely
		new Thread(() -> {
			while (true) {
				var uuidMessage = UUID.randomUUID().toString();
				try {
					queue.printMessages(uuidMessage);
				} catch (InterruptedException e) {
					e.printStackTrace();
				}
			}
		}).start();
		new Thread(() -> {
			while (true) {
				try {
					queue.stopMessages();
				} catch (InterruptedException e) {
					e.printStackTrace();
				}
			}
		}).start();
	}

}
```

# 📕⭐ 10. Controlled Concurrent Initialization
**Controlled Concurrent Initialization** — паттерн для **параллельной и управляемой инициализации ресурсов**, с гарантией того, что все ресурсы будут готовы перед использованием.
## Проблема
- Иногда ресурсы **дорогие в создании** (например, подключение к БД, загрузка данных, инициализация больших структур).
    
- ==Можно ускорить работу, запуская инициализацию **несколькими потоками одновременно**==
    
- Но нужно **гарантировать**, что все ресурсы будут инициализированы перед продолжением основной логики.
## Решение (идея паттерна)
1. Создать **несколько потоков**, каждый из которых инициализирует отдельный ресурс.
    
2. Использовать **CountDownLatch** с числом потоков = числу ресурсов.
3. Каждый поток после успешной инициализации вызывает `latch.countDown()`.
    
4. Основной поток вызывает `latch.await()` → **ждет, пока все потоки завершат работу**.
    
5. После этого безопасно использовать все инициализированные ресурсы.
## Компоненты
- **ExecutorService / ThreadPool** — управляет параллельным запуском инициализаций.
- **CountDownLatch** — барьер, который блокирует основной поток до завершения всех задач.
- **Runnable задачи** — инициализируют ресурсы и уменьшают счетчик `latch`.
- **Основная логика** — вызывается только после того, как все ресурсы готовы.

```java
public class ControlledInitialization {
	static class Resource1 {

	}

	static class Resource2 {

	}

	static class Resource3 {

	}

	private Resource1 resource1;
	private Resource2 resource2;
	private Resource3 resource3;
	private CountDownLatch latch = new CountDownLatch(3);

	// Задачи
	private Runnable initResource1 = () -> {
		try {
			// simulate wait
			Thread.sleep(4000);
			resource1 = new Resource1();
			latch.countDown();
		} catch (InterruptedException e) {
			e.printStackTrace();
		}
	};

	private Runnable initResource2 = () -> {
		try {
			// simulate wait
			Thread.sleep(4000);
			resource2 = new Resource2();
			latch.countDown();
		} catch (InterruptedException e) {
			e.printStackTrace();
		}
	};

	private Runnable initResource3 = () -> {
		try {
			// simulate wait
			Thread.sleep(4000);
			resource3 = new Resource3();
			latch.countDown();
		} catch (InterruptedException e) {
			e.printStackTrace();
		}
	};
	
	//Конструктор
	public ControlledInitialization() {
		initialize();
		try {
			latch.await();
		} catch (InterruptedException e) {
			e.printStackTrace();
		}
		doTask();
	}
	private void initialize() {
		System.out.println("=== Initializing Resources ===");
		var executor = Executors.newFixedThreadPool(3);
		executor.execute(initResource1);
		executor.execute(initResource2);
		executor.execute(initResource3);
		executor.shutdown();

	}
	
	private void doTask() {
		System.out.println("=== Resources Initialized ===");
		System.out.println("Resource 1 instance " + resource1);
		System.out.println("Resource 2 instance " + resource2);
		System.out.println("Resource 3 instance " + resource3);

	}

	public static void main(String[] args) {
		new ControlledInitialization();
	}
}
```

# 📕 11. Background Task Executor (Фоновое выполнение задач)
**Background Task Executor** — паттерн для выполнения задач в фоновом потоке, не блокируя основной поток.
## Проблема
- В приложениях часто есть **долгие операции** (отправка email, работа с сетью, обработка файлов, вычисления).
    
- Если выполнять их в основном потоке, пользовательский интерфейс или основной поток логики **замедлится или зависнет**.
    
- Нужно **отделить выполнение** этих задач, при этом иметь контроль: возможность отменить задачу, дождаться завершения, обработать ошибки.
## Решение (идея паттерна)
1. Создать **ExecutorService** (или другой пул потоков).
2. В методы Executor передавать **Callable** или **Runnable** задачи.

3. Получать **Future** для управления задачей:
    - ожидание завершения (`get()`),
    - отмена (`cancel(true)`),
    - проверка состояния (`isDone()`, `isCancelled()`).
        
4. Обеспечить **обработку ошибок и прерываний** через callback или проверку исключений.
    
5. Позволять **корректно завершать пул потоков** (`shutdown()`, `shutdownNow()`).
## Основные компоненты
- **ExecutorService** — потоковый пул, управляющий потоками.
- **Callable / Runnable** — задачи для фонового выполнения.
- **Future / FutureTask** — контроль выполнения задачи.
- **Callbacks / обработчики исключений** — для обработки ошибок или прерываний.
- **Методы shutdown / shutdownNow** — безопасное завершение работы пула потоков.

# 📕 12. Wait/Notify Queue
## Что это
==Использование **встроенного монитора объекта**== (`synchronized`) и его **очереди ожидания** (`wait/notify/notifyAll`) для организации ожидания наступления **состояния-предиката** и пробуждения потоков, когда предикат становится истинным.
## Компоненты
- **Общий объект-монитор**: на нём берут/отпускают монитор и вызывают `wait/notify`.
- **Состояние (shared state)**: набор полей, определяющих предикаты
- **Предикат(ы)**: ==булевы условия вида “можно продолжать?”== (например, `count > 0`, `!stopped`).

## Инварианты и гарантия видимости
- Все обращения к состоянию защищены **одним и тем же монитором**.
- Изменение состояния и вызов `notify/notifyAll` выполняются **под тем же монитором**.
    
- Освобождение/взятие монитора создаёт **happens-before**; пробуждённый поток видит актуальные изменения.

## Правила использования
1. 🔶**Всегда проверяйте предикат в цикле**:
```java
synchronized (lock) {
    while (!predicate()) {
	    lock.wait()
    };
    // действие
}
```

2. 🔶**Сначала меняйте состояние, потом сигнализируйте** (под монитором):
```java
synchronized (lock) {
    changeState();
    lock.notifyAll(); // или notify
}
```

3. 🔶 **Выбор notify vs notifyAll**:
	- `notify` — когда в очереди **однородные** ожидания и пробуждать нужно ровно **одного**.
	- `notifyAll` — когда ожидания **разные**/неизвестны или есть риск **зависания** из-за пробуждения “не того” потока.
	
4. 🔶 **Не выполняйте длительные/блокирующие операции под монитором**  
    (I/O, `sleep`, блокирующие очереди): держите критическую секцию минимальной.
    
5. 🔶**Обрабатывайте прерывания корректно**:
    - либо завершаем операцию,
    - либо `Thread.currentThread().interrupt()` и выходим вверх по стеку.
        
6. 🔶**Один монитор — одно состояние** (или чёткая дисциплина): не смешивайте не связанные состояния под одним замком без нужды.

```java
public class WaitNotifyQueue {

	private boolean continueToNotify;
	private BlockingQueue<String> messages;

	public WaitNotifyQueue(List<String> messages) {
		this.messages = new LinkedBlockingQueue<>(messages);
		this.continueToNotify = true;
	}

	public synchronized void stopsMessaging() {
		continueToNotify = false;
		notifyAll();
	}

	public synchronized void message() throws InterruptedException {
		while (!continueToNotify)
			wait();
		var message = messages.take();
		System.out.println(message);
	}

	public static void main(String[] args) {
		var messages = new LinkedList<String>();
		for (int i = 0; i < 130; i++) {
			messages.add(UUID.randomUUID().toString());
		}
		var waitNotifyQueue = new WaitNotifyQueue(messages);
		new Thread(() -> {
			try {
				while (true) {
					waitNotifyQueue.message();
					Thread.sleep(300);
				}
			} catch (InterruptedException e) {
				Thread.currentThread().interrupt();
				e.printStackTrace();
			}
		}).start();
		var random = new Random();
		new Thread(() -> {
			while (true) {
				int val = random.nextInt(100);
				System.out.println(val);
				if (val == 99) {
					break;
				}
				try {
					Thread.sleep(400);
				} catch (InterruptedException e) {
					Thread.currentThread().interrupt();
					e.printStackTrace();
				}
			}
			waitNotifyQueue.stopsMessaging();
			try {
				Thread.sleep(500);
			} catch (InterruptedException e) {
				Thread.currentThread().interrupt();
				e.printStackTrace();
			}
		}).start();
	}
}
```

# 📕 13. Resource Pool (Пул ресурсов)
## Идея паттерна Resource Pool
- Есть ограниченный набор ресурсов (например, соединения с БД, сетевые сокеты, файлы).
- Несколько потоков могут конкурировать за эти ресурсы.
- Если ресурсов нет → поток **ждёт**, пока кто-то освободит ресурс.
    
В Java обычно реализуется через:
- **Semaphore** → ограничение на количество одновременно занятых ресурсов.
- **BlockingQueue** → хранение ресурсов и упрощённое ожидание, если они заняты.

## ⭐ С GitHub
```java
public class ResourcePool<T> {

	private final static TimeUnit TIME_UNIT = TimeUnit.SECONDS;
	private Semaphore semaphore;
	private BlockingQueue<T> resources;

	public ResourcePool(int poolSize, List<T> initializedResources) {
		this.semaphore = new Semaphore(poolSize, true);
		this.resources = new LinkedBlockingQueue<>(poolSize);
		this.resources.addAll(initializedResources);
	}

	public T get() throws InterruptedException {
		return get(Integer.MAX_VALUE);
	}

	public T get(long secondsToTimeout) throws InterruptedException {
		semaphore.acquire();
		try {
			T resource = resources.poll(secondsToTimeout, TIME_UNIT);
			return resource;
		} finally {
			semaphore.release();
		}
	}

	public void release(T resource) throws InterruptedException {
		if (resource != null) {
			resources.put(resource);
			semaphore.release();
		}
	}
}
```

```java
public class ResourcePool<T> {

    private final Semaphore semaphore;
    private final Supplier<T> resourceFactory;

    public ResourcePool(int poolSize, Supplier<T> resourceFactory) {
        this.semaphore = new Semaphore(poolSize, true); // ограничиваем число ресурсов
        this.resourceFactory = resourceFactory;
    }

    // получить ресурс (создаётся "на лету")
    public T acquire() throws InterruptedException {
        semaphore.acquire(); // блокируем, если все занято
        return resourceFactory.get();
    }

    // освободить ресурс (фактически просто освобождаем слот)
    public void release(T resource) {
        if (resource != null) {
            semaphore.release();
        }
    }
}

public class ResourcePoolExample {
public static void main(String[] args) {
	// пул максимум из 3 ресурсов
	ResourcePool<String> pool = new ResourcePool<>(3, () -> "DB-Connection-" + System.nanoTime());

	Runnable task = () -> {
		try {
			String conn = pool.acquire();
			System.out.println(Thread.currentThread().getName() + " получил ресурс: " + conn);

			// эмуляция работы
			Thread.sleep(1000);

			pool.release(conn);
			System.out.println(Thread.currentThread().getName() + " освободил ресурс: " + conn);
		} catch (InterruptedException e) {
			Thread.currentThread().interrupt();
		}
	};

	// запускаем 5 потоков, а ресурсов всего 3
	for (int i = 0; i < 5; i++) {
		new Thread(task, "Поток-" + i).start();
	}
}
}
```

# 📕 14. Immutable Object + volatile reference (Неизменяемый объект с volatile ссылкой )
## 🚩 Проблема
- **Изменяемое состояние (mutable state)** в многопоточке → источник бед:
    - потеря обновлений (lost updates),
    - гонки при чтении-записи,
    - «устаревшие значения» (stale values) из-за кэширования потоками.
        
- Если мы используем **immutable объект** (например, `String` или `LocalDate`):
    - внутреннее состояние не меняется,
    - можно смело читать его из любого потока без синхронизации.
        
👉 Но остаётся вопрос: **как передавать новый экземпляр immutable объекта между потоками?**

## 🛡️ Назначение паттерна
Создать потокобезопасный объект, который:
1. **Не может изменяться** после создания
    
2. **Безопасно публикуется** через volatile ссылку
    
3. **Гарантирует видимость** изменений для всех потоков

```java
public class ImmutableObject {
	// MUTABLE ссылка на IMMUTABLE объект
	private volatile MyImmutableObject safeReference = new MyImmutableObject(0, "", false);

	public void updateField(int id, String newValue, boolean newAnotherValue) {
		this.safeReference = new MyImmutableObject(id, newValue, newAnotherValue);
	}

	public void printMyImmutableObject() {
		System.out.println(safeReference);
	}

	public final static class MyImmutableObject {
	    private final int id;           // final поля
	    private final String aValue;    // final поля
	    private final boolean anotherValue; // final поля
	    
	    // Конструктор инициализирует все поля
	    public MyImmutableObject(int id, String aValue, boolean anotherValue) {
	        this.id = id;
	        this.aValue = aValue;
	        this.anotherValue = anotherValue;
	    }
	    
	    // Только геттеры (без сеттеров)
	    public int getId() { return id; }
	    public String getaValue() { return aValue; }
	    public boolean getAnotherValue() { return anotherValue; }
	}
}
```

## 💡 Для чего volatile?
1. У нас есть `MyImmutableObject`, он **неизменяемый** → ✅ безопасен.
    
2. Но в коде мы ==можем **ПЕРЕПИСАТЬ ССЫЛКУ** на него, чтобы обновить состояние==:
```java
private MyImmutableObject safeReference;

public void update(...) {
    this.safeReference = new MyImmutableObject(...); // пересоздаём объект
}
```

Здесь ==уже два вопроса==:
- **Видимость:** увидит ли другой поток новый объект или у него останется старое значение в кеше CPU?
    
- **Порядок записи:** другой поток может увидеть «старую» версию ссылки, пока мы обновили её в одном месте.

> [!NOTE] ⭐⭐⭐ Даже если объект immutable, **сама ссылка на него mutable**.⭐⭐⭐
