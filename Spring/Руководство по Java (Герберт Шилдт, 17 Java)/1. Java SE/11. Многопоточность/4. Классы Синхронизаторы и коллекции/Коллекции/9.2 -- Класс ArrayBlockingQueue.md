
> [!NOTE] **`ArrayBlockingQueue<E>`** — это **ограниченная потокобезопасная очередь FIFO**, реализующая интерфейс `BlockingQueue<E>`.
> В отличие от `LinkedBlockingQueue`, она `хранит элементы` в `фиксированном МАССИВЕ`, а не в связанном списке.
> 

Очередь использует **единую блокировку (ReentrantLock)** для синхронизации операций вставки и извлечения.  

Обычно применяется в сценариях ==**Producer–Consumer**==, когда нужно ограничить количество элементов и обеспечить строгий порядок обработки.

## Конструкторы
```java
// Создает очередь указанной ёмкости
// capacity — максимальное количество элементов (должно быть > 0)
public ArrayBlockingQueue(int capacity)

// Создает очередь указанной ёмкости и управлением очередности доступа
// fair — если true, потоки обслуживаются по принципу FIFO (честная блокировка)
public ArrayBlockingQueue(int capacity, boolean fair)

// Создает очередь указанной ёмкости, содержащую элементы из указанной коллекции
// c — коллекция, элементы которой будут добавлены в очередь
public ArrayBlockingQueue(
	int capacity, 
	boolean fair, 
	Collection<? extends E> c)
```

## Методы
Класс **наследует методы** от `AbstractQueue<E>` и реализует `BlockingQueue<E>`,  
поэтому поддерживает все основные операции вставки, удаления и проверки.  

Ниже приведены **публичные методы**, реализованные или переопределённые в `ArrayBlockingQueue`.

###  Добавление элементов
```java
// Добавляет элемент, если есть место; иначе выбрасывает IllegalStateException
public boolean add(E e)

// Пытается вставить элемент, возвращает true, если успешно, false — если очередь заполнена
public boolean offer(E e)

// Пытается вставить элемент, ожидая не более указанного времени, если очередь заполнена
public boolean offer(E e, long timeout, TimeUnit unit) throws InterruptedException

// Блокирующе вставляет элемент в очередь, ожидая, если очередь заполнена
public void put(E e) throws InterruptedException
```

### Извлечение элементов
```java
// Извлекает и удаляет элемент из головы очереди, ожидая при необходимости
public E take() throws InterruptedException

// Пытается извлечь элемент, возвращает null, если очередь пуста
public E poll()

// Пытается извлечь элемент, ожидая не более указанного времени, если очередь пуста
public E poll(long timeout, TimeUnit unit) throws InterruptedException

// Возвращает элемент из головы очереди, не удаляя его; null, если очередь пуста
public E peek()
```

### Управление состоянием очереди
```java
// Возвращает текущее количество элементов в очереди
public int size()

// Возвращает количество свободных слотов
public int remainingCapacity()

// Удаляет все элементы из очереди
public void clear()
```

### Массовые операции
```java
// Перемещает все доступные элементы очереди в указанную коллекцию
public int drainTo(Collection<? super E> c)

// Перемещает не более maxElements элементов в указанную коллекцию
public int drainTo(Collection<? super E> c, int maxElements)
```

### Итераторы и представление
```java
// Возвращает итератор по элементам очереди (итератор слабосогласованный)
public Iterator<E> iterator()

// Возвращает массив, содержащий все элементы очереди
public Object[] toArray()

// Возвращает массив указанного типа с элементами очереди
public <T> T[] toArray(T[] a)
```

## 💡 Особенности и подводные камни 💡

1️⃣ 💡**Фиксированный размер** — размер задаётся один раз при создании и не может быть изменён.  

2️⃣ ⭐⭐⭐==**Единая блокировка**== — в отличие от `LinkedBlockingQueue`, используется **один общий `ReentrantLock`**, что увеличивает конкуренцию между потоками.  

3️⃣ **Память выделяется сразу**, поэтому нет накладных расходов на создание узлов при вставке.  

4️⃣ **Параметр `fair`** влияет на производительность: при `fair = true` уменьшается пропускная способность, но обеспечивается порядок FIFO между потоками.  

5️⃣ 🟠**Операции `put()` и `take()` ==блокируются==**, если очередь соответственно заполнена или пуста.  

6️⃣ 🟢**Методы `offer()` и `poll()` ==НЕ блокируются==**, если не указано время ожидания.  

7️⃣ **Не допускает `null`** — попытка вставки приведёт к `NullPointerException`.  

8️⃣ Итератор **слабосогласованный (weakly consistent)** — не бросает `ConcurrentModificationException`, но может не отражать актуальные изменения.

---

## 💎 Best Practices 💎

✅ Используйте `ArrayBlockingQueue` при **фиксированном размере пула** (например, ограничение очереди задач).  

✅ Для строгого порядка доступа между потоками используйте `fair = true`.

✅ Предпочитайте `LinkedBlockingQueue` при большом числе потоков-писателей и читателей (меньше конкуренции).  

✅ Применяйте `offer(e, timeout, unit)` для контролируемого ожидания вставки, чтобы избежать блокировки навсегда.  

✅ Не используйте `clear()` в критически многопоточных сценариях — возможны неожиданные эффекты при одновременных операциях.

## Пример
```java
import java.util.concurrent.*;

public class ArrayBlockingQueueExample {
    public static void main(String[] args) {
        BlockingQueue<String> queue = new ArrayBlockingQueue<>(2, true);

        Thread producer = new Thread(() -> {
            try {
                for (int i = 1; i <= 4; i++) {
                    queue.put("Task-" + i);
                    System.out.println("Produced: Task-" + i);
                }
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        });

        Thread consumer = new Thread(() -> {
            try {
                while (true) {
                    String task = queue.take();
                    System.out.println("Consumed: " + task);
                    Thread.sleep(700);
                }
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        });

        producer.start();
        consumer.start();
    }
}
```
