Методы `wait()`, `notify()` и `notifyAll()` — это ==**низкоуровневые механизмы синхронизации в Java**==, которые позволяют организовывать взаимодействие между потоками внутри **монитора объекта**. 

Они ==реализуют **модель "ожидание-уведомление"**== (wait-notify pattern).

## Эти методы определены в классе `java.lang.Object`, а не в `Thread`!
Потому что **монитор принадлежит объекту**, а не потоку. Только владелец монитора может вызывать `wait()` и `notify()`. 

🛑 Являются `final` методами класса `Object`

| Метод         | Назначение                                                                 |
| ------------- | -------------------------------------------------------------------------- |
| `wait()`      | Поток ==**временно освобождает монитор**== и **встаёт в очередь ожидания** |
| `notify()`    | ==Пробуждает **один СЛУЧАЙНЫЙ поток**== из очереди ожидания                |
| `notifyAll()` | ==Пробуждает **все потоки**,== ожидающие монитор этого объекта             |
| `sleep()`     | НЕ ОСВОБОЖДАЕТ МОНИТОР, просто переводит в `TIMED_WAITING` поток           |
> [!NOTE] 🛑 Их можно ==ВЫЗВАТЬ ТОЛЬКО из СИНХРОНИЗИРОВАННОГО контекста== 🛑
> 

## Пример
В данном примере кода, есть общий класс Q, который содержит разделяемую переменную, в которую поток `Producer` записывает значение, а `Consumer` читает это значение переменной.

Чтобы они 100% работали по очереди (никто никого не перекрывал, и не выполнял больше одного действия с `монитором` объекта Q), необходимо использовать методы `wait()` `notify()`

Первым должен отправить сообщение `Producer`, поэтому `Consumer` должен ожидать своей очереди на использование монитора объекта `Q`. Поэтому используется метод `wait()` в самом начале.
Тем временем, `Producer` изменяет значение переменной, и "пробуждает" следующий в очереди поток (`Consumer`), а сам переходит в ожидание.
```java
class Q {
	int n;
	boolean valueSet = false;
	
	synchronized int get() {
		while(!valueSet) {
			try {
				wait();
			} catch (InterruptedException e) {
				System.out.println("Перехвачено InterruptedEx");
			}
		}
		System.out.println("Получено: " + n);
		valueSet = false;
		notify();
		return n;
	}
	
	synchronized void put(int n) {
		while(valueSet) {
			try {
				wait();
			} catch (InterruptedException e) {
				System.out.println("Перехвачено InterruptedEx");
			}
		}
	
		this.n = n;
		valueSet = true;
		System.out.println("Отправлено: " + n);
		notify();
	}
}

class Producer implements Runnable{
	Q q;
	Thread t;
	
	public Producer(Q q) {
		this.q = q;
		t = new Thread(this, "Производитель");
	}

	@Override
	public void run() {
		int i = 0;
		while(i < 10) {
			q.put(i++);
		}
	}
}

class Consumer implements Runnable {
	Q q;
	Thread t;
	
	public Consumer(Q q) {
		this.q = q;
		t = new Thread(this, "Потребитель");
	}

	@Override
	public void run() {
		int n = 0;
		while(n < 9) {
			n = q.get();
		}
	}
	
}

public class MyMain {
	public static void main(String[] args) {
		Q q = new Q();
		Producer p = new Producer(q);
		Consumer c = new Consumer(q);
		
		p.t.start();
		c.t.start();
	}
}
```

### 💡 Почему `while`, а не `if`?
Это **важный шаблон**:
```java
while (!условие) {
    wait();
}
```
- Потоки могут **ложно пробудиться (spurious wakeup)**.
    
- `while` гарантирует, что условие будет проверено снова.

# 💡 Подводные камни 💡

### ☢ `IllegalMonitorStateException`

Вызывается, если `wait()`, `notify()` или `notifyAll()` вызваны **вне `synchronized` блока**.

### ☢ Зависания

Если `notify()` вызывается **до `wait()`**, поток может "зависнуть" навсегда, потому что пропустил сигнал.
> [!NOTE] Title
> 👉 Если `producer()` выполнится **раньше** `consumer()`, то `notify()` никого не разбудит, и `consumer()` **зависнет** в `wait()`.


### ☢ Многопоточность требует `volatile` или `synchronized`

Поле-флаг (например, `ready`) должно быть либо `volatile`, либо изменяться в `synchronized` блоке, иначе **поток не увидит изменения** из-за кэширования.
