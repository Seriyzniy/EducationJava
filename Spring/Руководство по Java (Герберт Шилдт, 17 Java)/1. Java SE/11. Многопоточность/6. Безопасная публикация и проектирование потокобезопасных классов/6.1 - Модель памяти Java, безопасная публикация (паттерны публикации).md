В отсутствие синхронизации не существует никаких гарантий, что один поток увидит изменения, сделанные другим потоком. Причины — многоуровневая иерархия кэшей процессора и агрессивные оптимизации компилятора и процессора.

**Подробное объяснение:**
- **Проблема:** `aVariable = 3;` — когда другой поток увидит это значение?
    
- **Причины "невидимости" изменений:**
    - **Компилятор** может переупорядочить инструкции или хранить переменные в регистрах.
    - **Процессор** может выполнять инструкции не по порядку (out-of-order execution).
    - **Кэши** процессоров могут не синхронизироваться с основной памятью немедленно.
        
- **В однопоточном коде** эти оптимизации скрыты и безопасны (as-if-serial semantics).
	
- **В многопоточном коде** эти оптимизации приводят к проблемам видимости и упорядочивания. 

> [!NOTE] ⭐ JMM - абстракция, дающая гарантии
> JMM — это абстракция, которая скрывает от нас сложности аппаратных архитектур и дает четкие гарантии при использовании синхронизации.

## 16.1.1 Основа моделей памяти

==Аппаратное обеспечение обеспечивает разные уровни согласованности памяти== (cache coherence). 

> [!NOTE] ⭐⭐JMM устанавливает барьеры памяти ПРИ ИСПОЛЬЗОВАНИИ блокировок в коде⭐⭐
> ==JMM унифицирует эти различия==, вставляя необходимые **барьеры памяти (memory barriers)** там, где это требуется семантикой Java (например, при использовании `volatile` или `synchronized`).

**Подробное объяснение:**
- **Согласованность кэша (Cache Coherence):** В многопроцессорной системе каждый процессор имеет свой кэш. Разные архитектуры по-разному решают проблему синхронизации этих кэшей.
    
- **Барьеры памяти (Memory Barriers/Fences):** Это специальные инструкции процессора, которые заставляют его выполнить все операции чтения/записи до барьера, прежде чем выполнять операции после барьера. JVM использует их для реализации гарантий JMM.
    
- **Роль JMM:** Быть посредником между слабыми гарантиями железа и сильными гарантиями, необходимыми для написания корректных многопоточных программ на Java

## 16.1.2 Переупорядочивание

**Компилятор и процессор могут **переупорядочивать** инструкции для оптимизации. 
В пределах одного потока это незаметно, но для другого потока такие переупорядочивания могут выглядеть так, будто операции выполняются в случайном порядке.

```java
public class PossibleReordering {
    static int x = 0, y = 0;
    static int a = 0, b = 0;

    public static void main(String[] args) throws InterruptedException {
        Thread one = new Thread(new Runnable() {
            public void run() {
                a = 1;
                x = b;
            }
        });
        
        Thread other = new Thread(new Runnable() {
            public void run() {
                b = 1;
                y = a;
            }
        });
        
        one.start();
        other.start();
        one.join();
        other.join();
        System.out.println("(" + x + ", " + y + ")");
    }
}
```

На рис. 16.1 показано возможное чередование с переупорядочением, приводящее к печати (0, 0).
![[Pasted image 20250920120951.png]]

**Подробное объяснение (на примере `PossibleReordering` из Листинга 16.1):**
- **Код:** Два потока записывают значения в `a`/`b` и читают значения из `b`/`a` в переменные `x`/`y`.
    
- **Ожидаемые результаты:** (1,0), (0,1), (1,1).
    
- **Неожиданный результат (0,0):** Возможен из-за переупорядочивания!
    - Поток A может выполнить `x = b` (0) _до_ `a = 1`.
        
    - Поток B может выполнить `y = a` (0) _до_ `b = 1`.
        
    - Результат: `x = 0`, `y = 0`.
        

**Вывод:** Рассуждать о порядке выполнения без синхронизации невероятно сложно. Синхронизация (блокировки, `volatile`) создает барьеры памяти, которые предотвращают переупорядочивание операций вокруг точек синхронизации.

## 16.1.3 О модели памяти Java менее чем в 500 словах
**Ключевая идея:** JMM определяется через отношение **happens-before** — формальное описание того, какие действия одного потока гарантированно видны другому.

**Правила happens-before:**

1. **Порядок программы:** Действия в одном потоке упорядочены так, как указано в коде.
    
2. **Монитор (Блокировка):** Разблокировка монитора happens-before последующей блокировкой того же монитора.
    
3. **`volatile`:** Запись в `volatile`-поле happens-before последующего чтения того же поля.
    
4. **Запуск потока:** `Thread.start()` happens-before всем действиям запущенного потока.
    
5. **Завершение потока:** Все действия потока happen-before тому моменту, когда другой поток обнаружит его завершение (через `join()` или `isAlive()`).
    
6. **Прерывание:** Вызов `interrupt()` happens-before обнаружению прерывания целевым потоком.
    
7. **Финализация:** Завершение конструктора объекта happens-before запуску его финализатора.
    
8. **Транзитивность:** Если A happens-before B, и B happens-before C, то A happens-before C.
    

**Корректно синхронизированная программа** — это программа, в которой нет гонок данных (все доступы к общим данным упорядочены отношением happens-before). Такая программа ведет себя так, как будто все действия всех потоков выполняются в некотором строгом глобальном порядке (**последовательная согласованность**).

# 📕 ⭐⭐16.2 Публикация⭐⭐📕
 Безопасная публикация объекта — это не просто сделать ссылку на него видимой другим потокам. 
 Нужно также гарантировать, что его _внутреннее состояние_ будет видно корректно. 


> [!NOTE] ⭐ Безопасная публикация НЕ ОТМЕНЯЕТ синхронизацию ⭐
> **Безопасная публикация (safe publication)** ==гарантия, что **объект, однажды созданный**== и опубликованный, ==будет виден другим потокам в **полностью сконструированном виде**== (все поля будут иметь актуальные значения).
> 
> 👉 ==Это всего лишь про _момент передачи объекта другим потокам_==.  
> Это **не равно** потокобезопасность в дальнейшем.

 Это достигается установлением отношения happens-before между публикацией и потреблением.
## 16.2.1 ❌ Небезопасная публикация ❌

**Ключевая идея:** Публикация ссылки на объект без установления отношения happens-before (например, через гонку данных) может привести к тому, что другой поток увидит **частично сконструированный объект**.

```java
public class UnsafeLazyInitialization {
    private static Resource resource;

    public static Resource getInstance() {
        if (resource == null) {
            resource = new Resource(); // unsafe publication
        }
        return resource;
    }
}
```

**Подробное объяснение (на примере `UnsafeLazyInitialization` из Листинга 16.3):**
- `resource = new Resource();` — это не атомарная операция. Она состоит из:
    1. Выделение памяти под объект.
        
    2. Вызов конструктора (инициализация полей).
        
    3. Присваивание ссылки переменной `resource`.
        
- **Переупорядочивание:** Шаги 2 и 3 могут быть поменяны местами! (Это разрешено JMM при отсутствии happens-before).
    
- **Сценарий:**
    - Поток A выполняет шаги 1 и 3, но не 2. Теперь `resource` не `null`.
        
    - Поток B видит, что `resource != null`, и возвращает его. Но поля объекта еще не проинициализированы конструктором! Поток B видит объект в недопустимом состоянии.
        

## 16.2.2 ⭐⭐⭐ Безопасная публикация ⭐⭐⭐

**Ключевая идея:** Безопасная публикация требует установления отношения happens-before между записью ссылки и ее последующим чтением. 

#### ⭐⭐⭐ Инициализация из статического инициализатора.
- 🔶 `Static`-блок
```java
static final Map<String, String> settings;

static { //🛑🛑Т.к static-блоки выполняются раньше, чем поток сможет использовать класс
	Map<String, String> tmp = new HashMap<>();
	tmp.put("mode", "test");
	settings = tmp;   // безопасная публикация
}
```
- 🔶`Static` инициализация при объялении
```java
class Constants {
    static final Connection conn = new Connection("db-url"); // безопасная публикация
}
```
- 🔶 Класс-холдер
```java
class Singleton {
    private Singleton() {}

    private static class Holder {
	    // создаётся ТОЛЬКО при загрузке Holder
        static final Singleton INSTANCE = new Singleton(); 
    }

    public static Singleton getInstance() {
        return Holder.INSTANCE;
    }
}
```
✅Это работает так как static инициализация происходит до фактической загрузки класса, тем самым потоки увидят актуальное значение
#### Хранение ссылки в `volatile`-поле.
**Volatile и happens-before** связаны тем, что **==запись в volatile-переменную происходит до любого ПОСЛЕДУЮЩЕГО (не параллельного) считывания== этой переменной**.
```java
public class MyMain {
	private static volatile boolean running = true; 
	
    public static void main(String[] args) throws InterruptedException {
    	Thread worker = new Thread(() -> {
            System.out.println("Worker started");
            while (running) {
            }
            System.out.println("Worker stopped");
        });

        worker.start();

        Thread.sleep(1000); // даём поработать
        System.out.println("Main thread stops worker...");
        running = false; // сигнал остановки

        worker.join();
        System.out.println("Main finished");
    }
}

Worker started
Main thread stops worker...
Worker stopped
Main finished

//🛑Без volatile цикл не закончится никогда
```
#### Хранение ссылки в `final`-поле правильно сконструированного объекта.

```java
class Config {
    private final String mode;
    private final int port;

    public Config(String mode, int port) {
        this.mode = mode;
        this.port = port;
    }

    public String getMode() { return mode; }
    public int getPort() { return port; }
}

public class Demo {
    static Config config;

    public static void main(String[] args) throws InterruptedException {
        // Поток 1 публикует объект
        Thread t1 = new Thread(() -> {
            config = new Config("prod", 8080);
        });

        // Поток 2 читает объект
        Thread t2 = new Thread(() -> {
            while (config == null) { } // ждём публикации
            System.out.println("Mode=" + config.getMode() + ", Port=" + config.getPort());
        });

        t1.start();
        t2.start();
        t1.join();
        t2.join();
    }
}
```
Если убрать `final`, другой поток может увидеть:
- `config != null`
- но `config.getPort()` = `0` (значение по умолчанию),
    
- потому что ==запись в поле могла быть переупорядочена== после публикации ссылки. (==НО  НЕВЕРОЯТНО РЕДКОЕ ЯВЛЕНИЕ==)
#### Хранение ссылки в поле, защищенном замком.

Когда поток **A** создал объект `X` и "опубликовал" его (например, записал в `volatile` переменную или поместил в `ConcurrentHashMap`), другой поток **B**, читающий этот объект, **увидит гарантированно корректное состояние именно этого объекта X**.

Но!  
❓❗Safe publication **не гарантирует**, что поток B увидит _все остальные изменения_ переменных, которые сделал поток A до публикации.  
Например:
```java
int a = 0;
MyObj x = new MyObj();

a = 42;         // 1. запись в другую переменную
x.field = 99;   // 2. инициализация объекта
published = x;  // 3. публикация через volatile
```

> [!NOTE] Поток B, прочитавший `published`, гарантированно увидит `x.field == 99`,  но насчёт `a == 42` гарантий нет.

Возьмём `BlockingQueue` из `java.util.concurrent`.
Когда поток **A** вызывает `queue.put(X)`, то в спецификации Java указано:  
`put()` _happens-before_ соответствующий `take()`.

Это означает, что:
1. Всё, что сделал поток **A** до вызова `put(X)`, будет видно потоку **B** после `take()`.
2. Не только состояние объекта `X`, но и **все остальные изменения памяти, сделанные до `put()`**.

Это значит:
- **Отношение HB устанавливается между операцией `put` (A) и операцией `take` (B)**.
    
- То есть очередь выступает как "посредник" (memory synchronization point).
    
- Таким образом:
    - _действия A до put_ → _put(X)_ → HB → _take(X)_ → _действия B после take_.

Очередь гарантирует, что **данные, которые были "до put" у A, видны "после take" у B**.

## 16.2.3 Идиомы безопасной инициализации

### Safe Lazy Initialization
**Ключевая идея:** Ленивую инициализацию нужно делать правильно.
```java
public class SafeLazyInitialization {
    private static Resource resource;

    public synchronized static Resource getInstance() {
        if (resource == null) {
            resource = new Resource();
        }
        return resource;
    }
}
```
**`SafeLazyInitialization` (Листинг 16.4):** Простое решение — `synchronized`. Работает правильно, но может быть не самым производительным.

### Safe Eager Initialization
**`EagerInitialization` (Листинг 16.5):** Ранняя инициализация. Просто и эффективно, если объект почти всегда нужен.

> [!NOTE] ⭐Не требует синхронизации вообще
> Использование ранней (eager) инициализации, показанной в листинге 16.5, исключает затраты на синхронизацию при каждом вызове метода getInstance класса SafeLazyInitialization.

```java
// Вариант 1: Ранняя инициализация (Eager Initialization)
public class EagerInitialization {
    private static final Resource resource = new Resource();
    
    public static Resource getResource() {
        return resource;
    }
}
```

### ⭐⭐⭐Идиома класса-холдера - always to use ⭐⭐⭐
**Идиома класса-холдера (Листинг 16.6):** Лучшее из двух миров. 

Класс `ResourceHolder` загружается и инициализируется JVM только при первом обращении к `ResourceHolder.resource`. Статическая инициализация потокобезопасна по умолчанию. 
⭐⭐⭐**Это предпочтительный способ ленивой инициализации.**⭐⭐⭐

> [!NOTE] Так как поле ресурса обернуто в внутренний класс, этот класс будет загружен только при первом использовании
>

```java
// Вариант 2: Ленивая инициализация с классом-холдером (Initialization-on-demand holder idiom)
public class ResourceFactory {
    private static class ResourceHolder {
        static final Resource resource = new Resource();
    }
    
    public static Resource getResource() {
        return ResourceHolder.resource;
    }
}
```

# 16.3 Безопасность инициализации

**Ключевая идея:** **Неизменяемые объекты (immutable)** обладают особым свойством: они являются потокобезопасными независимо от способа их публикации, даже если публикация была небезопасной. Это гарантируется JMM для **правильно сконструированных объектов** с `final`-полями.

```java
import java.util.HashMap;
import java.util.Map;

public class SafeStates {
    private final Map<String, String> states;

    public SafeStates() {
        states = new HashMap<String, String>();
        states.put("alaska", "AK");
        states.put("alabama", "AL");
        ...
        states.put("wisconsin", "WI");
        states.put("west virginia", "WV");
        states.put("wyoming", "WY");
    }

    public String getAbbreviation(String stateName) {
        return states.get(stateName.toLowerCase());
    }
}
```

**Подробное объяснение (на примере `SafeStates` из Листинга 16.8):**

- ⭐⭐⭐**Условия "правильно сконструированного объекта":**⭐⭐⭐
    - Все его поля объявлены как `final`.
        
    - ⭐⭐⭐**Ссылка на объект не "убегает" из конструктора** (т.е. `this` не публикуется до завершения конструктора).

- **Гарантия JMM:** Для такого объекта любая публикация его ссылки (даже через гонку данных) гарантирует, что любой поток, получивший эту ссылку, увидит все значения, записанные в `final`-поля в конструкторе, в их конечном, проинициализированном состоянии. Более того, все объекты, достижимые через эти `final`-поля (например, `HashMap` в примере), также будут видны корректно.
    
- ❗❓❗**Что нарушает безопасность инициализации?**
    - Не-`final` поля.
        
    - Изменение состояния после конструктора.
        
    - "Убегание" `this` из конструктора.
