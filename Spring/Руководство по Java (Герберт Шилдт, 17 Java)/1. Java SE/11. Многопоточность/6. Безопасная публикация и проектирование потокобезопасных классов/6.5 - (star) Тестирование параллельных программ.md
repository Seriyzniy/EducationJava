Глава фокусируется на двух основных типах тестов: 
1. на **корректность** (безопасность и живучесть) 
2. на **производительность** (пропускная способность, задержка, масштабируемость). 

Ключевой посыл: ни один метод тестирования не может найти все ошибки, поэтому необходим комплексный подход

> [!NOTE] Понятия
> - **Инвариант** — это **логическое выражение некоторого условия, истинность которого необходимо соблюдать**. Определяет глобальные свойства некоторого класса, которые должны соблюдаться после его создания на протяжении всего времени жизни.
> 
> - **Постусловие** — это **инвариант, соблюдаемый на выходе из операции**. Должно быть выполнено после исполнения действия.

## 12.1 Тестирование на корректность
 Цель тестов на корректность — убедиться, что класс ведет себя в соответствии со своей спецификацией в многопоточной среде. 
 
 Основные категории:
 - **безопасность** ("ничего плохого не происходит") 
 - **живучесть** ("что-то хорошее когда-нибудь происходит").

С тестами на живучесть связаны тесты производительности. 
Производительность можно измерить несколькими способами, в том числе: 
- Пропускная способность: скорость выполнения набора параллельных задач; 
- Отзывчивость: задержка между моментом поступления запроса и выполнением какого-либо действия (также называемая “временем ожидания”, latency); 
- Масштабируемость: улучшение пропускной способности (или отсутствие такового) по мере увеличения доступности ресурсов (обычно ЦП).

### Интересный пример для тестирования
```JAVA
public class BoundedBuffer<E> {
    private final Semaphore availableItems, availableSpaces;
    @GuardedBy("this") private final E[] items;
    @GuardedBy("this") private int putPosition = 0, takePosition = 0;
    
    public BoundedBuffer(int capacity) {
        availableItems = new Semaphore(0);
        availableSpaces = new Semaphore(capacity);
        items = (E[]) new Object[capacity];
    }
    
    public boolean isEmpty() {
        return availableItems.availablePermits() == 0;
    }
    
    public boolean isFull() {
        return availableSpaces.availablePermits() == 0;
    }
    
    public void put(E x) throws InterruptedException {
        availableSpaces.acquire();
        doInsert(x);
        availableItems.release();
    }
    
    public E take() throws InterruptedException {
        availableItems.acquire();
        E item = doExtract();
        availableSpaces.release();
        return item;
    }
    
    private synchronized void doInsert(E x) {
        int i = putPosition;
        items[i] = x;
        putPosition = (++i == items.length) ? 0 : i;
    }
    
    private synchronized E doExtract() {
        int i = takePosition;
        E x = items[i];
        items[i] = null;
        takePosition = (++i == items.length) ? 0 : i;
        return x;
    }
}
```

Класс `BoundedBuffer` реализует очередь на основе массива фиксированной длины с блокирующими методами `put` и `take`, управляемыми парой подсчитывающих семафоров. 

Семафор `availableItems` представляет собой количество элементов, которое может быть удалено из буфера, и изначально равен нулю (так как буфер изначально пуст). 

Аналогично, семафор `availableSpaces` представляет собой количество элементов, которое может быть добавлено в буфер, и инициализируется размером буфера. 

Операция `take` требует, чтобы сначала было получено разрешение от семафора `availableItems`. Эта операция выполняется немедленно, если буфер не пустой, в противном случае операция блокируется, до появления элементов в буфере. Как только разрешение получено, следующий элемент из буфера удаляется, и освобождается разрешение семафора `availableSpaces`. 

Операция `put` определяется обратным образом, так что при выходе из методов `put` или `take` сумма счетчиков обоих семафоров всегда равна значению границы


> [!NOTE] `Semaphore(0)` можно захватить увеличив счетчик через `release()`
> Ключевой момент заключается в том, что **семафоры динамически меняют свои значения** в процессе работы.
> ```java 
> public void put(E x) throws InterruptedException {
>     availableSpaces.acquire();  // Захватываем 1 разрешение (теперь
>     availableSpaces = 2)
>     doInsert(x);                // Вставляем элемент в массив
>     availableItems.release();   // Освобождаем 1 разрешение (теперь availableItems
>     = 1)
>    }
> ```

## 12.1.1 Простые модульные тесты
💡**Ключевая идея:** Начните с базовых последовательных тестов, чтобы ==проверить инварианты класса в изоляции.== Это помогает ==исключить простые ошибки, НЕ связанные с параллелизмом==, прежде чем переходить к сложным многопоточным тестам.

**Пример:** Для класса `BoundedBuffer` простые тесты проверяют, что новый буфер пуст, а после добавления `N` элементов в буфер емкости `N` он становится полным

```java
class BoundedBufferTest extends TestCase {
    
    void testIsEmptyWhenConstructed() {
        BoundedBuffer<Integer> bb = new BoundedBuffer<Integer>(10);
        assertTrue(bb.isEmpty());
        assertFalse(bb.isFull());
    }
    
    void testIsFullAfterPuts() throws InterruptedException {
        BoundedBuffer<Integer> bb = new BoundedBuffer<Integer>(10);
        for (int i = 0; i < 10; i++) {
            bb.put(i);
        }
        assertTrue(bb.isFull());
        assertFalse(bb.isEmpty());
    }
}
```
Некоторые инварианты из тех, что быстрее всего приходят на ум, это то, что только что созданный буфер должен идентифицировать себя и как пустой, и как не полный. Аналогичен, но чуть более сложен, тест на безопасность при вставке N элементов в буфер с емкостью N (который должен успешно завершиться без блокировки), и проверить, что буфер распознает, что он полон (а не является пустым).

**Ключевые моменты:**
- Тесты должны проверять **инварианты** (например, `isEmpty()` и `isFull()`).
    
- Последовательные тесты — это первый и необходимый шаг.

## 12.1.2 Тестирование блокирующих операций
💡**Ключевая идея**: ==Тестирование методов, которые должны блокироваться при определенных условиях== (например, `take()` из пустого буфера), требует создания отдельного потока и управления его жизненным циклом из основного потока теста.

```java
void testTakeBlocksWhenEmpty() {
    final BoundedBuffer<Integer> bb = new BoundedBuffer<Integer>(10);
    Thread taker = new Thread() {
        public void run() {
            try {
                int unused = bb.take();
                fail(); // Если мы попадаем сюда, тест провален
            } catch (InterruptedException success) {
                // Expected behavior - thread was interrupted while blocked
            }
        }
    };
    
    try {
        taker.start();
        Thread.sleep(LOCKUP_DETECT_TIMEOUT);
        taker.interrupt();
        taker.join(LOCKUP_DETECT_TIMEOUT);
        assertFalse(taker.isAlive());
    } catch (Exception unexpected) {
        fail();
    }
}
```

> [!NOTE] 🧠Важно
> Заманчиво использовать метод Thread.getState, для проверки того, что поток фактически заблокирован на условии ожидания, но этот подход не надежен. Нет такого требования, чтобы заблокированный поток всегда входил в состояние WAITING или TIMED_WAITING, так как JVM, вместо этого, может выбрать для реализации блокировки ожидание спина.

**Ключевые моменты:**
- Для проверки блокировки используется **прерывание** потока.
    
- **Не полагайтесь на `Thread.getState()`** для определения состояния потока, так как оно ненадежно (из-за spin-waiting, ложных пробуждений и задержек планировщика).
    
- Используйте таймауты в методах типа `join`, чтобы тест не завис навсегда.

## 12.1.3 Тестирование безопасности
**Ключевая идея:** Чтобы выявить ошибки безопасности (гонки данных), ==нужно создать множество потоков, интенсивно работающих с общим ресурсом==, и затем проверить, что не было нарушений инвариантов.

**Пример:** Класс `PutTakeTest`  запускает `N` потоков-производителей и `N`потоков-потребителей, которые обмениваются данными через `BoundedBuffer`. Каждый поток вычисляет свою **контрольную сумму** (XOR) помещаемых или извлекаемых элементов. После завершения работы суммарная контрольная сумма производителей должна равняться суммарной контрольной сумме потребителей. Если это не так, произошло повреждение данных.

```java
public class PutTakeTest {
    private static final ExecutorService pool = Executors.newCachedThreadPool();
    private final AtomicInteger putSum = new AtomicInteger(0);
    private final AtomicInteger takeSum = new AtomicInteger(0);
    private final CyclicBarrier barrier;
    private final BoundedBuffer<Integer> bb;
    private final int nTrials, nPairs;

    public static void main(String[] args) {
        new PutTakeTest(10, 10, 100000).test(); // sample parameters
        pool.shutdown();
    }

    PutTakeTest(int capacity, int npairs, int ntrials) {
        this.bb = new BoundedBuffer<Integer>(capacity);
        this.nTrials = ntrials;
        this.nPairs = npairs;
        this.barrier = new CyclicBarrier(npairs * 2 + 1);
    }

    void test() {
        try {
            for (int i = 0; i < nPairs; i++) {
                pool.execute(new Producer());
                pool.execute(new Consumer());
            }
            barrier.await(); // wait for all threads to be ready
            barrier.await(); // wait for all threads to finish
            assertEquals(putSum.get(), takeSum.get());
        } catch (Exception e) {
            throw new RuntimeException(e);
        }
    }

    class Producer implements Runnable {
        public void run() {
            try {
                int seed = (this.hashCode() ^ (int) System.nanoTime());
                int sum = 0;
                barrier.await();
                for (int i = nTrials; i > 0; --i) {
                    bb.put(seed);
                    sum += seed;
                    seed = xorShift(seed);
                }
                putSum.getAndAdd(sum);
                barrier.await();
            } catch (Exception e) {
                throw new RuntimeException(e);
            }
        }
    }

    class Consumer implements Runnable {
        public void run() {
            try {
                barrier.await();
                int sum = 0;
                for (int i = nTrials; i > 0; --i) {
                    sum += bb.take();
                }
                takeSum.getAndAdd(sum);
                barrier.await();
            } catch (Exception e) {
                throw new RuntimeException(e);
            }
        }
    }

    // Генератор псевдослучайных чисел среднего качества
    static int xorShift(int y) {
        y ^= (y << 6);
        y ^= (y >>> 21);
        y ^= (y << 7);
        return y;
    }
}
```

(см как работает `CyclicBarrier`) Когда любой из потоков (`Producer` или `Consumer`) доходит до `await()` он уменьшает счетчик. Когда счетчик доходит до 0 то код продолжает выполнение

💡 **Ключевые моменты:**
- Используйте **нечувствительные к порядку операции** (сложение, XOR) для подсчета контрольных сумм, чтобы избежать синхронизации между потоками.
    
- **Генерируйте тестовые данные псевдослучайно** (например, с помощью `xorShift` из листинга 12.4), чтобы компилятор не мог предугадать и предвычислить результат.
    
- Для координации старта и финиша всех потоков используйте **`CyclicBarrier`**. Это гарантирует, что потоки начнут работу одновременно, что увеличивает вероятность выявления гонок.
    
- ⭐ Запускайте тесты на **многопроцессорных системах** и с **количеством потоков > количества ядер**, чтобы увеличить переключения контекста и разнообразие сценариев выполнения.

## 12.1.4 Тестирование управления ресурсами
💡 **Ключевая идея:** Классы должны не только правильно работать, но и корректно управлять ресурсами (памятью, файловыми дескрипторами и т.д.), не допуская утечек.

**Пример:** Тест `testLeak` (листинг 12.7) помещает большие объекты в буфер, а затем извлекает их. После этого делается "снимок" потребляемой памяти (например, с помощью профайлера). Если реализация буфера не обнуляет ссылку на извлеченный элемент (в методе `doExtract`), то объекты не будут собраны сборщиком мусора, и потребление памяти между двумя снимками будет разным.

```java
class Big { 
    double[] data = new double[100000]; 
}

void testLeak() throws InterruptedException {
    final int CAPACITY = 100;
    final int THRESHOLD = 10000; // Пороговое значение для утечки памяти
    
    BoundedBuffer<Big> bb = new BoundedBuffer<Big>(CAPACITY);
    
    // Принудительная сборка мусора и получение исходного размера кучи
    System.gc();
    Thread.sleep(100); // Даем время на сборку мусора
    long heapSize1 = Runtime.getRuntime().totalMemory() - Runtime.getRuntime().freeMemory();
    
    // Заполняем буфер большими объектами
    for (int i = 0; i < CAPACITY; i++) {
        bb.put(new Big());
    }
    
    // Очищаем буфер
    for (int i = 0; i < CAPACITY; i++) {
        bb.take();
    }
    
    // Снова принудительная сборка мусора и замер памяти
    System.gc();
    Thread.sleep(100); // Даем время на сборку мусора
    long heapSize2 = Runtime.getRuntime().totalMemory() - Runtime.getRuntime().freeMemory();
    
    // Проверяем, что память вернулась к исходному состоянию
    long memoryDiff = Math.abs(heapSize1 - heapSize2);
    assertTrue("Memory leak detected: " + memoryDiff + " bytes", memoryDiff < THRESHOLD);
}
```

**Ключевые моменты:**
- Утечки памяти — частая проблема в структурах данных, которые должны повторно использовать ресурсы.
    
- Явное обнуление ссылок (`items[i] = null`) часто необходимо только в таких случаях.

## 12.1.5 Использование обратных вызовов
💡 **Ключевая идея:** Многие классы (например, `ThreadPoolExecutor`) предоставляют точки расширения в виде обратных вызовов (callback). Это можно использовать для тестирования их внутреннего поведения.

**Пример:** `TestingThreadFactory` (листинг 12.8) подсчитывает количество созданных потоков. Это позволяет протестировать, что пул потоков правильно расширяется при нагрузке (листинг 12.9): мы отправляем много долгоиграющих задач и проверяем, что было создано ровно `MAX_SIZE` потоков.

```java
class TestingThreadFactory implements ThreadFactory {
    public final AtomicInteger numCreated = new AtomicInteger();
    private final ThreadFactory factory = Executors.defaultThreadFactory();

    public Thread newThread(Runnable r) {
        numCreated.incrementAndGet();
        return factory.newThread(r);
    }
}

public void testPoolExpansion() throws InterruptedException {
    int MAX_SIZE = 10;
    TestingThreadFactory threadFactory = new TestingThreadFactory();
    ExecutorService exec = Executors.newFixedThreadPool(MAX_SIZE, threadFactory);

    for (int i = 0; i < 10 * MAX_SIZE; i++)
        exec.execute(new Runnable() {
            public void run() {
                try {
                    Thread.sleep(Long.MAX_VALUE);
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                }
            }
        });

    for (int i = 0;
         i < 20 && threadFactory.numCreated.get() < MAX_SIZE;
         i++)
        Thread.sleep(100);

    assertEquals(threadFactory.numCreated.get(), MAX_SIZE);
    exec.shutdownNow();
}
```

💡 **Ключевые моменты:**
- Инструментирование фабрик и других точек расширения — мощный способ тестирования политик класса.

## 12.1.6 Увеличение степени чередования
**Ключевая идея:** Чтобы ==увеличить шанс проявления редких ошибок==, связанных с временными соотношениями, можно ==искусственно увеличить количество переключений контекста== в критических секциях.

**Пример:** В листинге 12.10 в метод `transferCredits` добавлен случайный вызов `Thread.yield()`. Это может заставить планировщик переключиться на другой поток в середине неатомарной операции и выявить ошибку, которая иначе проявилась бы только при большом количестве запусков.

```java
public synchronized void transferCredits(Account from, 
                                        Account to, 
                                        int amount) 
{ 
    from.setBalance(from.getBalance() - amount); 
    if (random.nextInt(1000) > THRESHOLD) 
        Thread.yield(); 
    to.setBalance(to.getBalance() + amount); 
}
```

**Ключевые моменты:**
-  **`Thread.yield()`** - вызывается случайным образом (с вероятностью ~(1000-THRESHOLD)/1000)
    - ==Заставляет планировщик потоков переключиться на другой поток==
    - Увеличивает вероятность проявления гонок данных
    
- Более надежный, но медленный способ — использовать короткий `sleep`.
    
- Такие вставки можно удобно добавлять и убирать с помощью **аспектно-ориентированного программирования (AOP)**.

# 12.2 📕 Тестирование производительности
💡 **Ключевая идея:** Тесты производительности измеряют пропускную способность, задержку и масштабируемость. Они должны быть основаны на реальных сценариях использования и включать базовую проверку корректности.

## 12.2.1 Расширение класса PutTakeTest с добавлением учёта времени
**Ключевая идея:** Тот же ==тест, что используется для проверки корректности, можно расширить для измерения производительности.==

Основное расширение, которое мы должны сделать в классе PutTakeTest, это добавить возможность измерения времени, затрачиваемого на выполнение. 

Вместо того, чтобы пытаться измерить время, затрачиваемое на выполнение одной операции, мы получаем более точное измерение, ==путем учёта времени, в целом затрачиваемого на выполнение и деления полученного значения на количество операций== (put/take), чтобы рассчитать время, затрачиваемое на выполнение одной операции.

**Пример:** Класс `BarrierTimer` (листинг 12.11) измеряет время между преодолением двух барьеров `CyclicBarrier` (старт и финиш всех потоков). Модифицированный `TimedPutTakeTest` (листинг 12.12) вычисляет среднее время на операцию (`nsPerItem`). Драйвер (листинг 12.13) запускает тест с разными параметрами (емкость буфера, количество пар производитель-потребитель).
```java
public class BarrierTimer implements Runnable {
    private boolean started;
    private long startTime, endTime;
    
    public synchronized void run() {
        long t = System.nanoTime();
        if (!started) {
            started = true;
            startTime = t;
        } else {
            endTime = t;
        }
    }
    
    public synchronized void clear() {
        started = false;
    }
    
    public synchronized long getTime() {
        return endTime - startTime;
    }
}

public class TimedPutTakeTest extends PutTakeTest {
    private final BarrierTimer timer = new BarrierTimer();
    
    public TimedPutTakeTest(int capacity, int npairs, int ntrials) {
        super(capacity, npairs, ntrials);
        this.barrier = new CyclicBarrier(npairs * 2 + 1, timer); //+ Таймер
    }
    
    public void test() {
        try {
            timer.clear();
            for (int i = 0; i < nPairs; i++) {
                pool.execute(new Producer());
                pool.execute(new Consumer());
            }
            barrier.await();
            barrier.await();
            
            long nsPerItem = timer.getTime() / (nPairs * (long)nTrials);
            System.out.print("Throughput: " + nsPerItem + " ns/item");
            assertEquals(putSum.get(), takeSum.get());
        } catch (Exception e) {
            throw new RuntimeException(e);
        }
    }
    
    public static void main(String[] args) throws Exception {
        int tpt = 100000; // trials per thread
        for (int cap = 1; cap <= 1000; cap *= 10) {
            System.out.println("Capacity: " + cap);
            for (int pairs = 1; pairs <= 128; pairs *= 2) {
                TimedPutTakeTest t = new TimedPutTakeTest(cap, pairs, tpt);
                System.out.print("Pairs: " + pairs + "\t");
                t.test();
                System.out.print("\t");
                Thread.sleep(1000);
                t.test();
                System.out.println();
                Thread.sleep(1000);
            }
        }
        pool.shutdown();
    }
}
```

⭐⭐**Ключевые компоненты:**
1. ⭐**`BarrierTimer`** - измеряет время между двумя срабатываниями барьера
    - ⭐`run()` вызывается при срабатывании барьера
        
    - Первый вызов фиксирует время начала, второй - время окончания
        
    - `getTime()` возвращает общее время выполнения
        
2. **`TimedPutTakeTest`** - наследник `PutTakeTest` с таймером
    - Барьер инициализируется с таймером как действием при срабатывании
        
    - `test()` вычисляет время на одну операцию (ns/item)

## 12.2.2 Сравнение нескольких алгоритмов
💡 **Ключевая идея:** Сравнивайте производительность вашей реализации с известными аналогами из `java.util.concurrent`.

**Пример:** На Рис. 12.2 сравниваются `BoundedBuffer`, `ArrayBlockingQueue` и `LinkedBlockingQueue`.

- `LinkedBlockingQueue` часто масштабируется лучше, чем `ArrayBlockingQueue`, несмотря на дополнительные затраты на выделение памяти под узлы. Это происходит потому, что операции над головой и хвостом связанного списка могут выполняться более независимо, снижая конкуренцию.
    
- `BoundedBuffer` на основе семафоров проигрывает из-за большего количества точек конкуренции (семафор + явная блокировка в `doInsert`/`doExtract`).


💡**Ключевые моменты:**
- **Интуиция может подводить.** Алгоритм, который выполняет больше операций (например, выделяет память), но делает это с меньшей конкуренцией, может масштабироваться лучше.

## 12.3 📕⭐Как избежать ошибок тестирования производительности
Тестирование производительности на JVM чревато ловушками. Результаты могут быть искажены сборкой мусора, динамической компиляцией и агрессивными оптимизациями компилятора.

### 12.3.1 Сборка мусора
**Проблема:** Запуск сборщика мусора во время теста может сильно исказить результаты.

**Решение:**
1. **Запускать тест достаточно долго**, чтобы произошло несколько сборок мусора. Это даст результаты, более соответствующие реальности.
    
2. **Убедиться, что сборка мусора не происходит** во время теста (с помощью флагов JVM, например, `-verbose:gc`), но это менее предпочтительно.

### 12.3.2 Динамическая компиляция
**Проблема:** JVM сначала интерпретирует байт-код, а затем компилирует "горячие" методы в машинный код. Измерение производительности интерпретируемого кода или во время компиляции бессмысленно.

> [!NOTE] 🧠
> При первой загрузке класса JVM выполняет его, интерпретируя байт-код. В какой-то момент, если метод выполняется достаточно часто, динамический компилятор убирает байт-код и преобразует его в машинный код; после завершения компиляции он переключается с интерпретации на прямое выполнение.

**Решение:**
- **Прогрев (Warm-up):** Запускать тест некоторое время до начала измерений, чтобы весь критический код был скомпилирован.
    
- **Запускать тест достаточно долго** (минуты), чтобы время компиляции было незначительной частью общего времени выполнения.
    
- Использовать флаг `-XX:+PrintCompilation` в HotSpot, чтобы убедиться, что компиляция завершилась до начала замеров.

## 12.4 📕 Комплементарные подходы к тестированию
### 12.4.2 Инструменты для проведения статического анализа
**Ключевая идея:** Специальные инструменты (например, **FindBugs**) анализируют байт-код на предмет распространенных **шаблонов ошибок (bug patterns)**.

**Примеры шаблонов ошибок, которые ищет FindBugs:**
- **Несогласованная синхронизация:** Поле иногда защищается блокировкой, а иногда нет.
    
- **Неосвобождённая блокировка:** Явная блокировка (`Lock`) не освобождается в `finally`-блоке.
    
- **Пустой `synchronized` блок:** Бессмысленное использование.
    
- **Блокировка с двойной проверкой (Double-checked locking):** Классическая ошибка инициализации.
    
- **Запуск потока из конструктора:** Потенциальная утечка ссылки `this`.
    
- **Ошибки уведомления:** Вызов `notify()` без изменения состояния.
    
- **Ошибки ожидания условия:** Вызов `wait()` не в цикле и без проверки условия.
    
- **Засыпание с блокировкой:** Вызов `Thread.sleep()` с удержанием блокировки.

### 12.4.3 Аспектно-ориентированные подходы к тестированию
**Ключевая идея:** **AOP** можно использовать для ненавязчивого добавления проверок в код. Например, аспект может проверять, что методы Swing вызываются только из потока событий (EDT), не модифицируя исходный код самих методов.

### 12.4.4 Профилировщики и инструменты мониторинга
**Ключевая идея:** Профилировщики (VisualVM, JProfiler, Async-Profiler) и **JMX** предоставляют ценную информацию о поведении программы в реальном времени: состояние потоков (выполняется, заблокирован, ждет I/O), конкуренция за блокировки, использование CPU. Это помогает идентифицировать "узкие места" (bottlenecks).

#### 1. ⭐⭐⭐JMX (Java Management Extensions)
🟠 **Программный доступ к мониторингу:**
```java
public class ThreadMonitoringExample {
    public static void main(String[] args) throws Exception {
        // Получение MXBean для мониторинга потоков
        ThreadMXBean threadMXBean = ManagementFactory.getThreadMXBean();
        
        // Включение мониторинга конкуренции
        threadMXBean.setThreadContentionMonitoringEnabled(true);
        
        ScheduledExecutorService scheduler = Executors.newScheduledThreadPool(1);
        scheduler.scheduleAtFixedRate(() -> {
            // Мониторинг состояния потоков
            long[] threadIds = threadMXBean.getAllThreadIds();
            for (long threadId : threadIds) {
                ThreadInfo threadInfo = threadMXBean.getThreadInfo(threadId);
                if (threadInfo != null) {
                    System.out.printf("Thread %s: %s, blocked: %dms, waited: %dms%n",
                            threadInfo.getThreadName(),
                            threadInfo.getThreadState(),
                            threadInfo.getBlockedTime(),
                            threadInfo.getWaitedTime());
                }
            }
        }, 0, 5, TimeUnit.SECONDS);
    }
}
```
