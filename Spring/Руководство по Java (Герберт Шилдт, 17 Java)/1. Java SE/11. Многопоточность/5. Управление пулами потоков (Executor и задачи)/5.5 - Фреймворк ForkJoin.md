# 📕 1. Что такое Fork/Join Framework

`ForkJoinPool` — это специализированный пул потоков, появившийся в **Java 7**, предназначенный для:
- выполнения ==задач, которые **можно рекурсивно разделить на подзадачи (fork)**==,
- ==и затем **объединить результаты (join)**==.

Это реализация паттерна **"разделяй и властвуй"** (**Divide and Conquer**) с поддержкой **work-stealing**

Используется для задач:
- обработки больших массивов / коллекций,
- рекурсивных вычислений (например, параллельный `merge sort`),
- поиска (поиск элемента в дереве, файле, структуре).

***
# 📕 2. Основные классы

1. `ForkJoinPool`
    - 🛑 аналог `ExecutorService`, но оптимизирован под рекурсивные задачи.
    - управляет пулом потоков и балансировкой нагрузки (work-stealing). Выполняет задачу `ForkJoinTask`
    
2. `ForkJoinTask<V>`
    - базовый абстрактный класс для задач.
    - два подкласса:
        - 🔴`RecursiveAction` — задача **без возвращаемого результата**.
            
        - 🟢`RecursiveTask<V>` — задача **с возвращаемым результатом**.

***
# 📕 3. Как работает

1. Задача подаётся в `ForkJoinPool`.
    
2. Она делится на подзадачи (метод `fork()`).
    
3. Каждая подзадача выполняется параллельно.
    
4. Результаты объединяются (`join()`).
    

Ключ: **work-stealing** — если у потока закончились задачи, он может «украсть» подзадачи у других потоков → равномерная загрузка.

## Когда появляются рабочие потоки в `ForkJoinPool`

Когда ты создаёшь `ForkJoinPool` **без параметров**:
```java
ForkJoinPool pool = new ForkJoinPool();
```

> [!NOTE] он ПО УМОЛЧАНИЮ инициализируется с **parallelism = ЧИСЛО ДОСТУПНЫХ ЯДЕР процессора (Runtime.getRuntime().availableProcessors())**.
> 

## Основные методы
- `fork()` — отправляет подзадачу в пул.
    
- `join()` — ожидает результат подзадачи.
    
- `compute()` — 🛑==основной метод, где пишется логика рекурсивного деления==.🛑
    
- `invoke(task)` — запустить задачу и дождаться результата.
    
- `invokeAll(task1, task2, ...)` — запустить несколько задач и дождаться всех.

# 💡 4. Пример
## ✅ 1. Простая задача (без результата)
```java
public class MyForkJoinFramework {
	public static void main(String[] args) {
		ForkJoinPool pool = new ForkJoinPool();
        pool.invoke(new PrintTask(1, 20));
        pool.shutdown();
	}
}

class PrintTask extends RecursiveAction {
    private final int start, end;

    public PrintTask(int start, int end) {
        this.start = start;
        this.end = end;
    }

    @Override
    protected void compute() {
        if (end - start <= 5) { // базовый случай
            for (int i = start; i <= end; i++) {
                System.out.println(Thread.currentThread().getName() + " prints " + i);
            }
        } else {
            int mid = (start + end) / 2;
            invokeAll(new PrintTask(start, mid), new PrintTask(mid + 1, end));
        }
    }
}
```

Здесь создается задача без возврата результата `PrintTask`. В ней определяется метод `compute()` в котором ПРОПИСЫВАЕТСЯ УСЛОВИЕ рекурсивного разделения задачи.

Сама задача заключается в том, чтобы вывести все числа от `start` до `end`. Если разница между ними больше 5, то тогда находится медиана, и задачи разбиваются на две (от start до mid и mid+1 до end). И так рекурсивно.


```java
import java.util.concurrent.*;

class SumTask extends RecursiveTask<Long> {
    private final int[] arr;
    private final int start, end;

    public SumTask(int[] arr, int start, int end) {
        this.arr = arr;
        this.start = start;
        this.end = end;
    }

    @Override
    protected Long compute() {
        if (end - start <= 10) { // базовый случай
            long sum = 0;
            for (int i = start; i < end; i++) sum += arr[i];
            return sum;
        } else {
            int mid = (start + end) / 2;
            SumTask left = new SumTask(arr, start, mid);
            SumTask right = new SumTask(arr, mid, end);
            left.fork();
            long rightResult = right.compute(); // оптимизация: выполняем одну задачу сами
            long leftResult = left.join();
            return leftResult + rightResult;
        }
    }
}

public class ForkJoinSumExample {
    public static void main(String[] args) {
        int[] array = new int[100];
        for (int i = 0; i < array.length; i++) array[i] = i + 1;

        ForkJoinPool pool = new ForkJoinPool();
        long result = pool.invoke(new SumTask(array, 0, array.length));
        System.out.println("Sum = " + result);
        pool.shutdown();
    }
}
```

- **Главный поток** (обычно `main`) вызывает `invoke(...)`.
    
- Метод `invoke` передаёт задачу в пул.
    
- Пул **не выполняет задачу в главном потоке** (если это не `commonPool` и не `ForkJoinTask.invokeAll` внутри задачи).  
    Вместо этого он отдаёт её одному из **рабочих потоков пула** (`Worker-1`).
    
- Когда внутри `compute()` ты вызываешь `fork()`:
    
    - новая подзадача кладётся в очередь `Worker-1`;
        
    - если есть **свободный поток** (например, `Worker-2`), он может украсть эту задачу и выполнить её.
        
- Таким образом, `Worker-2` и другие потоки **создаются сразу при инициализации пула** — не динамически на каждую задачу.

# 💡 Подводные камни и неочевидные моменты

1. 🔶**Не используем ForkJoinPool для мелких задач**
    - слишком много накладных расходов на fork/join.
        
    - правило: **если задача < 1000 операций — лучше обычный ExecutorService**.
        
2. 🔶 **Рекурсия может быть глубокой**
    - риск переполнения стека при неправильной стратегии.
        
    - обычно делают «порог» (threshold), ниже которого задача выполняется линейно.
        
3. 🔶 **Вызов `join()` блокирует поток**
    - ❌плохой стиль: `left.fork(); right.fork(); left.join(); right.join();` → может повесить систему.
        
    - ✅хорошая практика: `left.fork(); right.compute(); left.join();` → один поток работает, другой «украден».
	    - Если бы мы сделали:
		```java
		left.fork();
		right.fork();
		long leftResult = left.join();
		long rightResult = right.join();
		```
		→ то оба задания бы ушли в пул, а текущий поток сидел бы без работы, ожидая.  
		Это **неэффективно**, потому что меньше параллельной загрузки CPU.
        
1. 🔶**Work-stealing ≠ бесплатный**
    - есть расходы на координацию и очереди задач.
        
    - выигрывает только при больших вычислительных задачах.
        
2. 🔶 **Проблема блокировок внутри ForkJoinPool**
    - если в задаче используется `synchronized` или `lock`, потоки могут блокироваться → пул неэффективен.
        
    - лучше использовать алгоритмы без блокировок (CAS, атомарные классы).
