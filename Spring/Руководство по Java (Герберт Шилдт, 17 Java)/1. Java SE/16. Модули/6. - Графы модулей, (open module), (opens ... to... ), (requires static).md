==Компилятор и среда выполнения строят **граф модулей**, который отображает все зависимости между модулями== (прямые и транзитивные). 
Это гарантирует, что все необходимые модули будут доступны. Графы полезны для визуализации сложных зависимостей в больших приложениях.

# 📕 1. Три специальных характерных черты модулей
Описываются три дополнительные возможности для специфических сценариев.

1. **Открытые модули (`open module`)**:
    - Объявление `open module` делает ==_все_ пакеты модуля доступными во время выполнения== (через Reflection API), *даже если они не экспортированы.* На этапе компиляции действуют обычные правила экспорта.
        
    - Основная цель — обеспечить работу фреймворков (например, Spring, Hibernate), которые активно используют рефлексию.
        
2. **Оператор `opens`**:
    - Позволяет ==открыть для рефлексии только конкретный пакет, А НЕ ВЕСЬ МОДУЛЬ: `opens package.name;`.
        
    - Можно также использовать `opens ... to ...` для ограничения круга модулей.
        
3. **Оператор `requires static`**:
    - Объявляет зависимость, которая ==ОБЯЗАТЕЛЬНАЯ НА ЭТАПЕ КОМПИЛЯЦИИ==, НО является ==**ОПЦИОНАЛЬНОЙ** ВО ВРЕМЯ ВЫПОЛНЕНИЯ==.
        
    - Полезно для функциональности, которая может быть доступна, но не является строго необходимой для работы программы.
