## 📦 Основные области памяти JVM

Когда ==JVM== запускается, она ==**резервирует** определённые области памяти== в процессе Java-программы. Эти области можно разделить на ==две большие категории==:

# 🔴 1. **Heap (Куча)**

> [!NOTE] ❗❗❗
> Область памяти, где **размещаются все объекты и массивы**, созданные через `new`.

☢Особенности:
- Управляется **сборщиком мусора (GC)**
- Расширяется или сужается по мере необходимости
- Основная часть heap разделена на **"поколения"**:

#### 🔸 Young Generation (молодое поколение)
- Где **появляются все новые объекты**
- Подразделяется на:
    - **Eden** — все новые объекты попадают сюда
    - **Survivor (S0, S1)** — выжившие объекты из Eden
        
- Часто собирается (minor GC)

#### 🔸 Old Generation (Tenured)
- Сюда перемещаются объекты, которые **долго живут**
- Сборка происходит реже, но тяжелее (major GC)

#### 🔸 (Опционально) Humongous / Large Object Space
- Используется G1 GC для больших объектов (например, большие массивы)

***
# 🔴 2. **Non-Heap (не-куча)**

> [!NOTE] ❗❗❗
> Области памяти, используемые самой JVM ==для **хранения служебной информации**==, **метаданных**, **интерпретации кода**, **JIT-компиляции** и т.п.

#### 🔸 Metaspace (Java 8+)
- Хранит **метаданные классов**, включая:
    - Список методов, полей
    - Таблицы виртуальных методов
    - Структуры `Klass`
        
- Приходит на замену **PermGen** (который был в Java ≤ 7)
    
- Использует **нативную память** (off-heap), не управляется GC напрямую

#### 🔸 Code Cache
- Машинный код, сгенерированный JIT-компилятором
- Позволяет выполнять байткод как **оптимизированный машинный код**
    
#### 🔸 Thread Stacks

- У каждой нити (потока) — **свой стек**, в котором хранятся:
    - локальные переменные
    - параметры методов
    - адрес возврата
        
- Это не heap, и не управляется GC

#### 🔸 GC Structures, JVM internal caches
- Таблицы GC, JIT-профайлы, inline-кеши и прочее

***
# 📊 Сводная таблица: Heap vs Non-Heap

| Параметр                | Heap             | Non-Heap                              |
| ----------------------- | ---------------- | ------------------------------------- |
| Что хранит              | Объекты, массивы | Метаданные классов, JIT-код           |
| Управляется GC          | ✅ Да             | Частично (например, Code Cache – нет) |
| Расширяется динамически | ✅ Да             | Да, но в пределах JVM-настроек        |
| Расположение            | В managed heap   | В нативной памяти (Metaspace)         |

# 🛑❌🛑 Off-Heap
## 🔹 Что такое off-heap?

**Off-heap (вне кучи)** — это ==область памяти, **выделенная вне стандартной Java Heap**==, то есть **вне управления сборщиком мусора (GC)**.

☢Особенности:
- Эту память JVM **не отслеживает и не очищает автоматически**.
	
- Ответственность за её выделение и освобождение лежит на **разработчике или библиотеке**, использующей её.
	
- Это **нативная память**, выделенная средствами ОС напрямую (обычно через `malloc` или аналогичные вызовы).

***
## 🔸 Heap vs Off-Heap

|Свойство|Heap (куча)|Off-Heap|
|---|---|---|
|Где хранится|Внутри JVM|В нативной памяти (ОС)|
|Управляется GC|✅ Да|❌ Нет|
|Влияет на паузы GC|✅ Да|❌ Нет|
|Размер ограничен|Параметрами JVM (`-Xmx`, и т.п.)|Зависит от ОС|
|Доступ|Через обычные объекты|Через указатели, `ByteBuffer`, JNI и пр.|
|Скорость доступа|Обычно быстрее|Может быть медленнее (но стабильнее)|
***
## 🧠 Зачем нужен off-heap?

1. **Избежать пауз GC**
    - Объекты в куче живут до тех пор, пока GC их не убьёт.
    - При большом объёме данных (например, кэш, буферы) GC может "подвисать".
    - Off-heap позволяет **сократить давление на GC** и **избежать stop-the-world пауз**.
        
2. **Более эффективное использование памяти**
    - JVM heap может быть ограничен, особенно в облачных окружениях.
    - Off-heap может использовать **всю доступную системную память**.
        
3. **Совместимость с нативными библиотеками**
    - Например, при использовании **JNI**, **Direct I/O**, **Zero-Copy**.
        
4. **Управление памятью вручную**
    - Возможность создавать свои аллокаторы, пулы и схемы хранения.

***
## 🛠 Как использовать off-heap в Java?

Есть несколько способов:

🔹 1. `ByteBuffer.allocateDirect(int capacity)`
```java
ByteBuffer buffer = ByteBuffer.allocateDirect(1024);
buffer.putInt(42);
```

- Выделяет память вне heap
- Используется в NIO (высокоскоростной ввод-вывод)
- Работает быстрее при взаимодействии с файлами и сетями (Zero-Copy)
	
- 🔴НО: GC **не освобождает** её! Она очищается только при `finalize()` или `cleaner` (с Java 9)
***

🔹 2. Использование `Unsafe`
```java
Unsafe unsafe = getUnsafe();
long address = unsafe.allocateMemory(1024);
unsafe.putInt(address, 42);
```

☢Особенности:
- Прямой доступ к нативной памяти
- **Очень мощный и опасный API** — можно нарушить безопасность JVM
