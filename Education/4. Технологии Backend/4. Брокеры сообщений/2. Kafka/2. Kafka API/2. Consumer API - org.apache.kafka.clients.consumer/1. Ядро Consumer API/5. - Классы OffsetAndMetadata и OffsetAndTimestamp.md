# 📕 1. Класс **OffsetAndMetadata**


> [!NOTE] `OffsetAndMetadata` 
> **неизменяемый (immutable)** класс, **представляющий** **смещение (offset)**, ==которое Kafka Consumer зафиксировал (commit) в топике== `__consumer_offsets`.  
> 
> Кроме самого смещения, он может содержать **дополнительные метаданные (metadata)** — строку, которую можно использовать для отладки, хранения состояния и т. п.
> 

Используется в методах:
```java
KafkaConsumer.commitSync(Map<TopicPartition, OffsetAndMetadata>)
```
    
```java
KafkaConsumer.commitAsync(
	Map<TopicPartition, OffsetAndMetadata>, 
	Callback)
```

## Конструктор
```java
// Создаёт экземпляр с указанным смещением.
// Параметры:
// - offset — номер следующего сообщения, которое нужно будет прочитать.
public OffsetAndMetadata(long offset)

// Создаёт экземпляр с указанным смещением и метаданными.
// Параметры:
// - offset — номер следующего сообщения
// - metadata — произвольная строка для описания состояния
public OffsetAndMetadata(long offset, String metadata)

// Создаёт экземпляр с указанным смещением, метаданными и временем лидерства.
// Параметры:
// - offset — смещение
// - leaderEpoch — эпоха лидера партиции (опционально, может быть null)
// - metadata — пользовательская строка
public OffsetAndMetadata(
	long offset, 
	Optional<Integer> leaderEpoch, 
	String metadata)
```

## Методы
### Основные методы доступа
```java
// Возвращает номер смещения
public long offset()

// Возвращает строку метаданных
public String metadata()

// Возвращает эпоху лидера, если она есть
public Optional<Integer> leaderEpoch()
```

## 💡 Особенности и подводные камни 💡

1️⃣ 💡`OffsetAndMetadata` используется ==при РУЧНОЙ ФИКСАЦИИ смещений (commit)==.  

2️⃣ Смещение хранится как **"следующее сообщение"** — то есть **уже обработанное + 1**.  

3️⃣ Если указать неверное смещение, при следующем `poll()` сообщения могут быть **пропущены или продублированы**.  

4️⃣ Метаданные не интерпретируются Kafka — можно хранить любую информацию (например, время последней обработки). 

5️⃣ Объект immutable, поэтому после создания его нельзя изменить.

---
## 💎 Best Practices 💎

✅ ❗❗❗Фиксируй offset **только после успешной обработки сообщения**.  

✅ 💡Храни в metadata полезные данные — например, `timestamp` или `batchId`. 

✅ Не создавай `OffsetAndMetadata` с отрицательным offset — это вызовет `IllegalArgumentException`.  

✅ 🧠Используй leaderEpoch только при интеграции с API, где нужно контролировать версии партиций.

---
## Пример
```java
import org.apache.kafka.clients.consumer.*;
import org.apache.kafka.common.TopicPartition;

import java.util.Collections;

public class CommitExample {
    public static void main(String[] args) {
        KafkaConsumer<String, String> consumer = ...;

        // После успешной обработки сообщения
        long nextOffset = 105; // следующее сообщение
        TopicPartition partition = new TopicPartition(
	        "orders", 
	        0);

        OffsetAndMetadata offsetMeta =
                new OffsetAndMetadata(
	                nextOffset, "Processed order batch #42");

        consumer.commitSync(
	        Collections.singletonMap(partition, offsetMeta));
    }
}
```

# 📕2. Класс OffsetAndTimestamp

> [!NOTE] `OffsetAndTimestamp`
> то **контейнер для связи offset-а с временной меткой (timestamp)**.  
> 
> Используется в методах, возвращающих смещения, соответствующие определённому времени, например: 
> ```java
> KafkaConsumer.offsetsForTimes(
> 	Map<TopicPartition, Long> timestampsToSearch)
> ```

Kafka хранит timestamp для каждого сообщения (create или log append time), и ==этот класс позволяет узнать==, **какое смещение соответствует заданному времени**.

## Конструктор
```java
// Создаёт экземпляр, связывающий смещение и временную метку.
// Параметры:
// - offset — смещение сообщения
// - timestamp — временная метка (в миллисекундах с эпохи)
public OffsetAndTimestamp(long offset, long timestamp)

// Создаёт экземпляр с указанием эпохи лидера.
// Параметры:
// - offset — смещение
// - timestamp — время
// - leaderEpoch — эпоха лидера (опционально)
public OffsetAndTimestamp(
	long offset, 
	long timestamp, 
	Optional<Integer> leaderEpoch)
```

## Методы
### Основные методы доступа
```java
// Возвращает смещение
public long offset()

// Возвращает временную метку сообщения (в миллисекундах)
public long timestamp()

// Возвращает эпоху лидера, если она присутствует
public Optional<Integer> leaderEpoch()
```

---
## 💡 Особенности и подводные камни 💡

1️⃣ Используется ==только ДЛЯ ПОЛУЧЕНИЯ offset по времени== (а не для фиксации).  

2️⃣ 💡Значение `timestamp` — это **время записи или создания сообщения**, зависит от `log.message.timestamp.type`.  

3️⃣ 💡Если брокер не хранит сообщений с таким временем — метод `offsetsForTimes()` вернёт `null` для партиции. 

4️⃣ Этот класс часто применяется при **позиционировании потребителя по времени** — например, «читать с 10:00».

5️⃣ Объект immutable — нельзя изменить offset или timestamp после создания.

---
## 💎 Best Practices 💎

✅ 💡Используй `offsetsForTimes()` для восстановления чтения после простоев — это безопаснее, чем фиксированные offset.  

✅ Проверяй `null`, возвращаемый методом `offsetsForTimes()`.

✅ ❗❗❗Не путай `OffsetAndTimestamp` (==поиск offset по времени==) и `OffsetAndMetadata` (==фиксация offset в группе==).  

✅ 💡Храни timestamp в UTC, чтобы избежать ошибок при переходах часовых поясов.

---
## Пример
```java
import org.apache.kafka.clients.consumer.*;
import org.apache.kafka.common.TopicPartition;

import java.time.Instant;
import java.util.*;

public class OffsetTimeExample {
    public static void main(String[] args) {
        KafkaConsumer<String, String> consumer = ...;

        // Определяем время, начиная с которого нужно читать
        long targetTime = Instant
	        .parse("2024-10-01T10:00:00Z").toEpochMilli();

        Map<TopicPartition, Long> query = Map.of(
	        new TopicPartition("orders", 0), 
	        targetTime);

        // Получаем смещение для времени
        Map<TopicPartition, OffsetAndTimestamp> offsets = consumer.offsetsForTimes(query);

        offsets.forEach((tp, oat) -> {
            if (oat != null) {
                System.out.printf(
	                "Партиция %s: читать с offset=%d" + 
	                "(время=%s)%n",
                    tp.partition(), 
                    oat.offset(), 
                    Instant.ofEpochMilli(oat.timestamp())
                );
                consumer.seek(tp, oat.offset());
            } else {
                System.out.printf("Для партиции %s нет сообщений за это время%n", tp.partition());
            }
        });
    }
}
```
