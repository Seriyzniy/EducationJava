
> [!NOTE] Интерфейс `Consumer<K, V>`
> Интерфейс `Consumer` определяет API для:
> - чтения сообщений из Kafka, 
> - управления подписками, 
> - смещениями, 
> - партициями 
> - жизненным циклом потребителя.  
> 
> 💡Реализуется классом **`KafkaConsumer`** (а также `MockConsumer` для тестов).

## Интерфейс
```java
package org.apache.kafka.clients.consumer;

public interface Consumer<K, V> extends Closeable {
}
```
Интерфейс **расширяет `Closeable`**, поэтому требует корректного закрытия через `close()` для освобождения ресурсов и выхода из consumer group.

## Методы

### Управление подписками и партициями
```java
// ❗❗Подписка на список топиков❗❗
void subscribe(Collection<String> topics);

// Подписка с обработчиком ребалансировки
void subscribe(
	Collection<String> topics, 
	ConsumerRebalanceListener callback);

// Подписка по регулярному выражению (например, "test-.*")
void subscribe(Pattern pattern);

// Подписка по шаблону с обработчиком ребалансировки
void subscribe(
	Pattern pattern, 
	ConsumerRebalanceListener callback);

// Текущий набор подписанных топиков
Set<String> subscription();


// ❗❗Ручное назначение партиций❗❗(💡без участия consumer group💡)
void assign(Collection<TopicPartition> partitions);

// Текущий набор назначенных партиций
Set<TopicPartition> assignment();


// Отписка от всех топиков
void unsubscribe();
```
> Эти методы управляют тем, **откуда именно потребитель получает данные** — через автоматическую подписку (в группе) или ручное назначение партиций.

### Чтение и получение данных
```java
// Читает записи (с тайм-аутом ожидания)
ConsumerRecords<K, V> poll(Duration timeout);

// Более старая версия (устарела)
@Deprecated
ConsumerRecords<K, V> poll(long timeoutMs);
```
> Метод `poll()` — **главный механизм получения сообщений**. Он не только извлекает данные, но и выполняет "heartbeat" для поддержания связи с координатором группы.

### Управление смещениями
```java
// Синхронная фиксация ❗❗всех текущих смещений❗❗
// Consumer отправляет commit-запрос на брокер и джет ACK от брокера о подтверждении сохранения offset'a в топику __consumer_offsets
void commitSync();

// Синхронная фиксация ❗❗указанных смещений❗❗
void commitSync(Map<TopicPartition, OffsetAndMetadata> offsets);


// Асинхронная фиксация всех смещений
void commitAsync();

// Асинхронная фиксация указанных смещений
void commitAsync(
	Map<TopicPartition, OffsetAndMetadata> offsets,
	OffsetCommitCallback callback);



// Получить текущие позиции чтения
long position(TopicPartition partition);

// Установить смещение вручную
void seek(TopicPartition partition, long offset);

// Установить смещение по времени (например, на начало дня)
void seekToBeginning(Collection<TopicPartition> partitions);
void seekToEnd(Collection<TopicPartition> partitions);
```

### Метаданные и информация о партициях
```java
// Возвращает конец лога (последний offset)
Map<TopicPartition, Long> endOffsets(
	Collection<TopicPartition> partitions);

// Возвращает начало лога (самый старый offset)
Map<TopicPartition, Long> beginningOffsets(
	Collection<TopicPartition> partitions);



// Возвращает offset по timestamp (для поиска по времени)
Map<TopicPartition, OffsetAndTimestamp> offsetsForTimes(
	Map<TopicPartition, 
	Long> timestampsToSearch
);

// Возвращает offset для старейшего или новейшего сообщения
Map<TopicPartition, OffsetAndTimestamp> offsetsForLeaderEpoch(
	Map<TopicPartition, OffsetAndMetadata> offsets
);



/**
 * Проверяет, включено ли автоматическое подтверждение (commit) смещений.
 * Возвращает true, если потребитель настроен на автоматическое подтверждение
 * полученных сообщений с заданным интервалом.
*/
boolean isAutoCommitEnabled();

// Получить метаданные о группе (id, generation, memberId)
ConsumerGroupMetadata groupMetadata();
```

### Управление жизненным циклом
```java
/**
 * Принудительно прерывает текущую операцию poll() и вызывает WakeupException
 * Используется для безопасного завершения работы потребителя из другого потока
 */
void wakeup();

/**
 * Корректно закрывает потребителя, отписывается от топиков и освобождает ресурсы
 * Выполняет финализацию смещений и завершает сетевые соединения
 */
void close();

/**
 * Закрывает потребителя с ограничением по времени выполнения операции
 * @param timeout максимальное время ожидания завершения операций
 * @throws TimeoutException если операция не завершилась за указанное время
 */
void close(Duration timeout);

/**
 * @deprecated Начиная с 0.11.0.0, используйте {@link #commitSync()} или {@link #commitAsync()}
 * Внутренний метод для ручного коммита смещений (устарел в пользу явных методов)
 */
@Deprecated
void commitOffsets();
```

### Дополнительные
```java
/**
 * Возвращает метрики производительности потребителя
 * @return Map где ключ - название метрики, значение - сама метрика
 */
Map<MetricName, ? extends Metric> metrics();

/**
 * Возвращает информацию о всех топиках в кластере
 * @return Map где ключ - название топика, значение - список информации о партициях
 */
Map<String, List<PartitionInfo>> listTopics();

/**
 * Возвращает информацию о партициях указанного топика
 * @param topic название топика
 * @return список с информацией о партициях топика
 */
List<PartitionInfo> partitionsFor(String topic);

/**
 * Возвращает уникальный идентификатор экземпляра потребителя
 * @return client instance ID (формат: consumer-{group.id}-{counter})
 */
String clientInstanceId(Duration timeout);
```

## 💡 **Особенности и подводные камни** 💡

1️⃣ **`poll()` должен вызываться регулярно**
- Иначе координатор удалит consumer из группы (`session.timeout.ms`)
- ⚠️ **И также превысится `max.poll.interval.ms`** - это ДВА разных таймаута!
    

2️⃣ **`wakeup()` - безопасный способ завершения**
- Можно вызывать из другого потока для graceful shutdown
    
- Выбрасывает `WakeupException` в текущем вызове `poll()`
    

3️⃣ **Не смешивайте ручные и авто-коммиты**
```java
// ❌ ОПАСНО: авто-коммит + ручной коммит
enable.auto.commit=true
consumer.commitSync();  // Двойной коммит!


// ✅ ЛУЧШЕ: один из подходов
enable.auto.commit=true   // Только авто-коммит
// ИЛИ
enable.auto.commit=false  // Только ручные коммиты
consumer.commitSync();
```

4️⃣ **Consumer - одноразовый**
- После `close()` нельзя переиспользовать
- Создавайте новый экземпляр для новой сессии

5️⃣ **`assign()` vs `subscribe()` - взаимоисключающие**
```java
// ❌ НЕЛЬЗЯ вместе
consumer.subscribe(Arrays.asList("topic"));
consumer.assign(partitions);  // Exception!


// ✅ ЛИБО динамическая подписка
consumer.subscribe(Pattern.compile("test-.*"));

// ✅ ЛИБО ручное назначение партиций  
consumer.assign(Collections.singletonList(partition));
```

## 💎 **Best Practices** 💎

✅ **Всегда закрывайте consumer**
```java
// try-with-resources (Java 21+)
try (KafkaConsumer<String, String> consumer = new KafkaConsumer<>(props)
) {
    consumer.subscribe(List.of("topic"));
    // работа с consumer...
}  // auto-close


// ИЛИ традиционный finally
KafkaConsumer consumer = new KafkaConsumer<>(props);
try {
    consumer.subscribe(List.of("topic"));
    // работа...
} finally {
    consumer.close(Duration.ofSeconds(5));  // с таймаутом!
}
```

✅ **Graceful shutdown с WakeupException**
В `poll()` цикле обрабатывайте `WakeupException` и завершайте корректно.
```java
try {
    while (running) {
        ConsumerRecords<String, String> records = 
	        consumer.poll(Duration.ofMillis(100));
        // обработка...
    }
} catch (WakeupException e) {
    // Игнорируем - это нормальное завершение
} finally {
    consumer.close();
}
```

✅ **Коммит после успешной обработки**
```java
ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
for (ConsumerRecord<String, String> record : records) {
    try {
        processRecord(record);  // бизнес-логика
    } catch (Exception e) {
        // ❌ НЕ КОММИТИТЬ при ошибках!
        break;  
    }
}
// ✅ Коммит только после успешной обработки ВСЕХ записей
consumer.commitSync();
```


✅ **Обработка ошибок в commitAsync**
В `commitAsync` обязательно используйте `OffsetCommitCallback` для обработки ошибок.
```java
consumer.commitAsync((offsets, exception) -> {
    if (exception != null) {
        log.error("Commit failed for offsets {}", offsets, exception);
        // Ваша логика восстановления
    }
});
```

✅ **Правильные таймауты**
```properties
# Для быстрой обработки
max.poll.interval.ms=30000      # 30 секунд на обработку батча
session.timeout.ms=10000        # 10 секунд для обнаружения отвала
heartbeat.interval.ms=3000      # 3 секунды между heartbeat'ами

# Для долгой обработки (ML, тяжелые вычисления)
max.poll.interval.ms=300000     # 5 минут
session.timeout.ms=45000        # 45 секунд
```

## Пример
```java
Properties props = new Properties();
props.put(
	ConsumerConfig.BOOTSTRAP_SERVERS_CONFIG, 
	"localhost:9092");
props.put(
	ConsumerConfig.GROUP_ID_CONFIG, 
	"example-group");
props.put(
	ConsumerConfig.KEY_DESERIALIZER_CLASS_CONFIG,
	StringDeserializer.class.getName());
props.put(
	ConsumerConfig.VALUE_DESERIALIZER_CLASS_CONFIG,
	StringDeserializer.class.getName());
props.put(
	ConsumerConfig.AUTO_OFFSET_RESET_CONFIG, 
	"earliest");

try (Consumer<String, String> consumer = new KafkaConsumer<>(props)
) {
    consumer.subscribe(Collections.singletonList("test-topic"));

    while (true) {
        ConsumerRecords<String, String> records = 
	        consumer.poll(Duration.ofMillis(100));
	      
		//Симуляция обработки данных  
        for (ConsumerRecord<String, String> record : records) {
            System.out.printf(
	            "topic=%s partition=%d" + 
	            "offset=%d key=%s value=%s%n",
                record.topic(), 
                record.partition(), 
                record.offset(), 
                record.key(), 
                record.value());
        }
        consumer.commitSync(); // фиксация смещений после обработки
    }
}
```
