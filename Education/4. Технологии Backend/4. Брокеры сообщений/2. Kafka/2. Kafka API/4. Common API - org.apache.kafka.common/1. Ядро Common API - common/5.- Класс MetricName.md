
> [!NOTE] **`MetricName`** 
> это **неизменяемый (immutable)** класс, который ==инкапсулирует имя метрики, её группу, описание и теги== (атрибуты).  
> 
> Он используется как **идентификатор** для метрик Kafka.
> Contents
> 
> Пример: `"record-send-rate"`, `"bytes-consumed-total"`, `"request-latency-avg"` и т.д.

Каждая метрика Kafka имеет:
- уникальное имя (`name`)
- логическую группу (`group`)
- описание (`description`)
- набор тегов (`tags`) — обычно `client-id`, `topic`, `partition` и др.

---
## Конструктор
```java
/**
 * Создаёт новый экземпляр MetricName.
 *
 * @param name        — имя метрики (например, "record-send-rate").
 * @param group       — группа метрик (например, "producer-metrics").
 * @param description — человекочитаемое описание метрики.
 * @param tags        — карта тегов, уточняющих контекст метрики (например, {"client-id": "producer-1"}).
 */
public MetricName(
	String name, 
	String group, 
	String description, 
	Map<String, String> tags)
```

Пример:
```java
MetricName metricName = new MetricName(
    "record-send-rate",
    "producer-metrics",
    "Средняя скорость отправки сообщений продюсером",
    Collections.singletonMap("client-id", "producer-1")
);
```

## Методы
### Доступ к свойствам
```java
// Возвращает имя метрики
public String name();

// Возвращает группу, к которой относится метрика
public String group();

// Возвращает описание метрики
public String description();

// Возвращает карту тегов
public Map<String, String> tags();
```

### Работа с тегами
```java
// Возвращает значение конкретного тега по имени
public String tags(String key);

// Возвращает true, если тег с указанным ключом присутствует
public boolean hasTag(String key);
```

## 💡 Особенности и подводные камни 💡

1️⃣ `MetricName` — **immutable**: после создания поля нельзя изменить (что гарантирует корректность при многопоточности).  

2️⃣ 💡💡Экземпляры `MetricName` **часто создаются автоматически** внутри Kafka (например, в `Metrics` или `KafkaProducer`). 

3️⃣ Пользователю редко нужно создавать их вручную — только если вы добавляете **кастомные метрики**.  

4️⃣ Для единообразия имен метрик Kafka использует **`MetricNameTemplate`** — шаблон, из которого создаются `MetricName`.  

5️⃣ Теги (`tags`) очень важны: они помогают различать метрики с одинаковыми именами, но разным контекстом (например, по `client-id` или `topic`).  

6️⃣ Kafka различает метрики **только по полному набору атрибутов (name + group + tags)** — если хоть одно отличается, это считается другой метрикой.

---
## 💎 Best Practices 💎

✅ Используй `MetricNameTemplate`, если создаёшь несколько однотипных метрик с разными тегами.  

✅ Не добавляй избыточные теги — это может раздуть количество метрик.  

✅ Используй человекопонятные описания — они отображаются в системах мониторинга. 

✅ При добавлении пользовательских метрик через `Metrics.addMetric()` передавай **тот же** `MetricName`, чтобы не плодить дубликаты.  

✅ Избегай хардкода имён — Kafka придерживается стандарта `snake_case` (`request-rate`, `incoming-byte-total`, `records-lag-max`).

## Пример
```java
import org.apache.kafka.common.metrics.*;

import java.util.Collections;

public class CustomMetricExample {
    public static void main(String[] args) {
        Metrics metrics = new Metrics();

        // Создание уникального имени для метрики
        MetricName metricName = new MetricName(
            "custom-events-processed",
            "application-metrics",
            "Количество обработанных событий приложением",
            Collections.singletonMap("client-id", "app-1")
        );

        // Добавление метрики (Gauge)
        metrics.addMetric(metricName, (config, now) -> {
            // Здесь можно вернуть динамическое значение, например, размер очереди
            return 42.0;
        });

        // Получение текущего значения
        KafkaMetric metric = metrics.metric(metricName);
        System.out.println(metric.metricValue());
    }
}

```
