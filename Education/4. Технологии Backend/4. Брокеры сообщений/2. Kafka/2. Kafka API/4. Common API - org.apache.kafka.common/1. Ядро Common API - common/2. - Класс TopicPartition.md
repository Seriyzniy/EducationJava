
> [!NOTE] TopicPartition
> это **небольшой, неизменяемый (immutable)** объект, который просто связывает:
> ```bash
> (topic name) + (partition number)
> ```

📦 То есть он **однозначно идентифицирует партицию конкретного топика** в кластере Kafka.

## Определение класса (упрощённо)
```java
public final class TopicPartition implements Serializable {
    private final String topic;
    private final int partition;
    
    // конструктор
    public TopicPartition(String topic, int partition) { ... }

    // геттеры
    public String topic() { return topic; }
    public int partition() { return partition; }

    // equals(), hashCode(), toString() переопределены
}
```

---
# Как используется внутри Kafka

Kafka всё время работает не просто с “топиками”, а с конкретными **партициями** —  
потому что **отправка, чтение, смещение, лидерство и репликация** происходят именно ==на уровне партиции==.

`TopicPartition` используется в десятках мест для идентификации этих партиций.

## Пример в продюсере
Когда продюсер отправляет сообщение, Kafka решает:
> "В какую партицию этого топика записать сообщение?"

После этого сообщение ассоциируется с `TopicPartition`, например:
```java
TopicPartition tp = new TopicPartition("orders", 2);
```

Продюсер хранит буферы (RecordAccumulator) **по ключу `TopicPartition`**:
```java
Map<TopicPartition, Deque<ProducerBatch>> batches;
```

## Пример в консюмере

Консюмер **подписывается** на набор партиций (а не просто на топики):
```java
List<TopicPartition> partitions = List.of(
    new TopicPartition("orders", 0),
    new TopicPartition("orders", 1)
);

consumer.assign(partitions);
```

Далее оффсеты, коммиты и позиции **всегда хранятся и передаются через `TopicPartition`**:
```java
consumer.position(new TopicPartition("orders", 0));

consumer.commitSync(
	Map.of(
		tp, // TopicPartition
		new OffsetAndMetadata(offset)
	)
);
```
