# 1. Интерфейс Metric


> [!NOTE] **`Metric`** 
> это **интерфейс**, который описывает базовое поведение любой метрики в Kafka. 
> 
> Он используется как **унифицированный контракт**, чтобы можно было получить текущее значение метрики и её описание.
> 


Проще говоря:

> `Metric` — это интерфейс для чтения метрик,  
> `KafkaMetric` — это его стандартная реализация,  
> `MetricName` — это идентификатор метрики (её имя, группа и теги).

## Интерфейс
```java
package org.apache.kafka.common.metrics;

/**
 * Интерфейс метрики, отслеживаемой для целей мониторинга.
 * Позволяет получить имя метрики и текущее значение.
 */
public interface Metric {

    /**
     * Возвращает имя метрики.
     * 
     * @return имя метрики (объект MetricName, включающий группу, описание и теги)
     */
    MetricName metricName();

    /**
     * Возвращает текущее значение метрики.
     * 
     * @return значение метрики как Object (часто это Double)
     */
    Object metricValue();
}
```

# 2. Класс Metrics


> [!NOTE] **`Metrics`** — это основной класс для **регистрации, хранения и управления метриками** Kafka.
> 
> Он отвечает за создание сенсоров (`Sensor`), добавление метрик (`KafkaMetric`), обновление их значений и предоставление доступа к собранным данным.

Проще говоря:
> `Metrics` — это “менеджер” метрик Kafka, через который всё создаётся и отслеживается.


## Конструктор
```java
// Создаёт реестр метрик с настройками по умолчанию
public Metrics()

// Создаёт реестр метрик с кастомной конфигурацией
public Metrics(MetricConfig defaultConfig)

// Создаёт реестр метрик с заданной конфигурацией и набором репортёров
public Metrics(
	MetricConfig defaultConfig, 
	List<MetricsReporter> reporters, 
	Time time)

// Создаёт реестр метрик с полной кастомизацией
public Metrics(MetricConfig defaultConfig,
               List<MetricsReporter> reporters,
               Time time,
               MetricNameTemplateRegistry templateRegistry)

```

 💬 Пояснения к конструкторам:
- **`MetricConfig defaultConfig`** — настройки метрик по умолчанию (например, время обновления, тип усреднения).
    
- **`List<MetricsReporter> reporters`** — список объектов, которые экспортируют метрики во внешние системы (JMX, Prometheus и т. д.).
    
- **`Time time`** — источник времени (обычно `SystemTime`).
    
- **`MetricNameTemplateRegistry templateRegistry`** — реестр шаблонов имён метрик (используется внутри Kafka для унификации).

## Методы

### Работа с метриками

```java
// Возвращает неизменяемую карту всех зарегистрированных метрик
public Map<MetricName, KafkaMetric> metrics()  

// Добавляет новую метрику в реестр 
public void addMetric(KafkaMetric metric) 
 
// Удаляет метрику по имени 
public void removeMetric(MetricName metricName)
```

### Работа с сенсорами

```java
// Создаёт или возвращает существующий сенсор с данным именем
public Sensor sensor(String name)  

// Создаёт сенсор с указанием уровня доступа (для группировки) 
public Sensor sensor(
	String name, 
	Sensor.RecordingLevel recordingLevel)  

// Создаёт сенсор с родительскими сенсорами (для иерархической агрегации) 
public Sensor sensor(String name, Sensor... parents)
```

### Конфигурация и служебные методы

```java
// Возвращает конфигурацию метрик по умолчанию 
public MetricConfig config()  

// Устанавливает новую конфигурацию 
public void updateMetricConfig(
	MetricName name, 
	MetricConfig config)  

// Закрывает все метрики и репортеры 
@Override 
public void close()
```

## 💡 Особенности и подводные камни 💡

1️⃣ `Metrics` — потокобезопасен. Все операции синхронизированы, чтобы избежать гонок при добавлении/удалении метрик.

2️⃣ При создании `Sensor` он **автоматически** регистрирует связанные `KafkaMetric` внутри `Metrics`.

3️⃣ `Metrics` управляет временем агрегации и обновления метрик (через `MetricConfig`).

4️⃣ `MetricsReporter` объекты (например, `JmxReporter`) **автоматически уведомляются** о добавлении/удалении метрик.

5️⃣ Если удалить сенсор, все его метрики также будут удалены из реестра.

6️⃣ Метрики можно получать в виде `Map<MetricName, KafkaMetric>` — ключом является объект `MetricName`, который содержит имя, группу и теги.

---
## 💎 Best Practices 💎

✅ Используй **один экземпляр `Metrics`** на всё приложение Kafka Client (Producer/Consumer/Admin).  

✅ Всегда закрывай `metrics.close()` при завершении работы клиента.  

✅ Используй **осмысленные группы и теги** (`MetricName`) — это облегчает мониторинг.  

✅ Не добавляй метрики вручную без `Sensor`, если ты не понимаешь, как они будут обновляться.  

✅ ❗❗❗Добавляй `MetricsReporter` для интеграции с системами наблюдения (Prometheus, Grafana, JMX).

## Пример
```java
import org.apache.kafka.common.metrics.*;
import org.apache.kafka.common.utils.Time;

import java.util.Collections;

public class MetricsExample {
    public static void main(String[] args) {
        // Создаём систему метрик с дефолтной конфигурацией
        Metrics metrics = new Metrics(
	        new MetricConfig(), 
	        Collections.singletonList(new JmxReporter()), 
		    Time.SYSTEM
		);

        // Создаём сенсор
        Sensor sensor = metrics.sensor("request-latency");

        // Добавляем метрику среднего значения
        sensor.add(
            new MetricName("avg-latency", "producer-metrics", "Средняя задержка запросов", Collections.emptyMap()),
            new Avg()
        );

        // Добавляем метрику максимального значения
        sensor.add(
            new MetricName("max-latency", "producer-metrics", "Максимальная задержка запросов", Collections.emptyMap()),
            new Max()
        );

        // Имитируем обновление данных
        sensor.record(100);
        sensor.record(200);
        sensor.record(50);

        // Получаем и выводим все метрики
        metrics.metrics().forEach((name, metric) -> {
            System.out.printf("%s = %s%n", name.name(), metric.metricValue());
        });

        metrics.close();
    }
}
```
