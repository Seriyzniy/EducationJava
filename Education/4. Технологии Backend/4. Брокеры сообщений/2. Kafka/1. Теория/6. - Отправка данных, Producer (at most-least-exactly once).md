## 📙2.Семантика доставки

**Семантике доставки сообщений (delivery semantics)** — то _как система гарантирует доставку сообщений между продюсером и консюмером_, и что происходит при сбоях.

### 1. At most once — «не больше одного раза»
💡 Сообщение может быть **доставлено один раз или не доставлено вовсе.**

#### Суть
- Продюсер **отправляет** сообщение.
    
- Если в процессе доставки произойдёт ошибка (сбой сети, сбой брокера и т.п.), продюсер **не будет повторять отправку**.
    
- Следовательно, **некоторые сообщения могут потеряться**.

#### Как включить в Kafka
- Продюсер не использует подтверждения:
```bash
acks=0
```
    
- Или не делает retry при ошибках:
```bash
retries=0
```


### 2. At least once — «по крайней мере один раз»
💡 Сообщение **гарантированно будет доставлено**,  
но **возможны дубликаты.**
#### Суть
- Продюсер повторно отправляет сообщение, если не получил подтверждение от брокера.
    
- Это обеспечивает **надёжную доставку**,  
    но если сообщение **успело записаться**, а подтверждение не пришло — **оно будет записано дважды.**

#### Как включить в Kafka
- Продюсер ждёт подтверждение от всех реплик:
```bash
acks=all
```

- И разрешает повторные отправки:
```bash
retries > 0
```

- Консюмер **должен коммитить offset только после успешной обработки сообщения**, иначе при перезапуске он получит то же сообщение снова.


### 3. **Exactly once** — _ровно один раз_
💡 Сообщение **доставляется один и только один раз**,  
даже при сбоях и ретраях.
#### Суть
Kafka добивается этого с помощью **идемпотентности продюсера** и **транзакций**.

##### Идемпотентность продюсера
Kafka присваивает каждому продюсеру **уникальный ID (PID)** и каждому сообщению — **sequence number**.  

Если одно и то же сообщение придёт повторно — брокер увидит повторяющийся sequence и **отбросит дубликат**.

Включается просто:
```bash
enable.idempotence=true
```
(❗❗❗в Kafka 3.x — включено по умолчанию)

Это даёт **exactly-once** _для записи в один топик и партицию._

---
##### Транзакции
Для atomic write across multiple partitions / topics:

1. Продюсер начинает транзакцию (`beginTransaction()`)
    
2. Отправляет несколько сообщений в разные топики/партиции
    
3. Завершает транзакцию (`commitTransaction()`)
    

Если произойдёт сбой, незавершённая транзакция **откатывается**, и ни одно сообщение из неё не станет видимым.

Пример конфигурации:
```yaml
enable.idempotence=true 
transactional.id=my-tx-producer 
acks=all
```

## 📙2. Сериализация сообщения
Kafka — это **брокер сообщений**, который **работает только с байтами**.  

Она не знает, что такое `String`, `Integer`, `User`, `OrderEvent` и т.д


Когда продюсер хочет отправить сообщение, он передаёт два значения:
```java
ProducerRecord<K, V>
```
где:
- `K` — ключ (key);
- `V` — значение (value).

Kafka **не может отправить объект `K` или `V` как есть** — нужно их **преобразовать в массив байтов** (`byte[]`).  
Вот тут и вступают в игру сериализаторы.

![[Pasted image 20251106093605.png]]

### Параметры key.serializer и value.serializer
❗❗(см [[3. - Kafka Message (Record, Event)]])❗❗

Это параметры продюсера, которые указывают **классы сериализаторов** для ключа и значения.
> Как сериализовать значение ключа и значения, как строку, число и т.д


Примеры стандартных сериализаторов из `org.apache.kafka.common.serialization`:

|Тип данных|Класс сериализатора|
|---|---|
|String|`StringSerializer`|
|Integer|`IntegerSerializer`|
|Long|`LongSerializer`|
|Double|`DoubleSerializer`|
|byte[]|`ByteArraySerializer`|

⭐⭐Если используешь кастомные объекты — создаёшь **свой класс**, реализующий интерфейс `Serializer<T>`.

## 📙5. Механизм накопления
Когда продюсер (`KafkaProducer`) отправляет сообщения, **он не отправляет их сразу по сети одно за другим**.  

Вместо этого он:
- группирует (накапливает) сообщения в памяти,
- и отправляет их **пакетами (batch)**.

Эти пакеты называются **"record batches"** (или "accumulation batches" — когда речь идёт о буфере перед отправкой).

🧠 Идея: отправлять **100 сообщений одним сетевым запросом** гораздо быстрее, чем 100 отдельных запросов.  
Это **уменьшает сетевые издержки** и повышает **пропускную способность (throughput)**.

![[Pasted image 20251106095511.png]]

### Настройка параметров batching
Kafka управляет этим поведением с помощью **двух ключевых параметров**:

| Параметр     | Тип          | Назначение                                                                     |
| ------------ | ------------ | ------------------------------------------------------------------------------ |
| `batch.size` | байты        | Максимальный **размер одного батча** для одной партиции<br>(По умолчанию 16Кб) |
| `linger.ms`  | миллисекунды | **Время ожидания, прежде чем отправить батч**, даже если он неполный           |

### Суммарный объем batch
Если суммарный объем bath для двух (и более) разных партиций (НО ДЛЯ ОДНОГО брокера) ПРЕВЫШАЕТ batch.size, то данные отправляются, даже если по отдельности их размер меньше batch.size и даже если linger.ms еще не истек
