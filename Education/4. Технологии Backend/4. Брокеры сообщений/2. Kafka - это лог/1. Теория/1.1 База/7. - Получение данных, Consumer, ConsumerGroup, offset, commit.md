**Consumer** - высокопроизводительный получатель сообщений

Читать данные, Consumer может только в лидеры партиций (на рисунке не указаны реплики)

🔶Чтение происходит при помощи метода `poll()`. При этом
- poll читает сообщения ПАЧКАМИ (batch)

# 📕2.Группа Consumer

> [!NOTE] Consumer group
> Consumer group = логическая группа потребителей, которые совместно читают данные из одного или нескольких топиков, деля между собой партиции.
> 

Если нам нужно МАСШТАБИРОВАТЬ какой-либо из сервисов, то каждый из них будет считывать одно и то же сообщение (с одной и той же партиции). 
А нам это не нужно
![[Pasted image 20251112111727.png]]

Мы бы хотели, чтобы дубликаты Consumer, читали другие сообщения.
Для этого и есть Consumer Group
![[Pasted image 20251112112039.png]]

> [!NOTE] ⭐==ОДНА ПАРТИЦИЯ для ОДНОГО КОНСЮМЕРА== из группы⭐
>Kafka гарантирует, что каждый **потребитель внутри группы получает уникальный набор партиций** — то есть события распределяются равномерно, а не дублируются.

Несколько Consumer объединяются в группу "Consumer Group", для этого есть параметр `group.id`

**При запуске нескольких Consumer** мы можем **указать**, к какой **группе они относятся**, при помощи указания одинакового group.id

## Балансировка партиций между Consumer
За балансировку партиций между Consumer Group отвечают КОНТТРОЛЛЕРЫ на кластере Kafka

Каждый Consumer отправляет специальное `Heart Beats` сообщение, ПО УМОЛЧАНИЮ каждые 3 секунды. 
Тем самым говоря, "Я живой и могу обрабатывать данные"
![[Pasted image 20251112112619.png]]

Если Consumer не прислал сообщение (в течении этого времени), то Kafka Cluster считает, что этот Consumer "упал", и производит перебалансировку партиций


## Консюмеры одной группы читают одну партицию
Если несколько консюмеров с одним **group.id** подключаются к топику с **одной партицией**, только **один** из них получает данные.  
Последний подключившийся перехватывает «указатель» чтения и продолжает вычитывать сообщения после завершения предыдущего.

![[Pasted image 20251106203708.png]]

Такой режим — разновидность **монопольного потребителя**.  
В отличие от JMS-очередей, где сообщения равномерно распределяются между консюмерами, в Kafka одна партиция всегда обслуживается только **одним консюмером** из группы.

# 📕2. Offset в Consumer

## ❌Какая проблема?
Допустим, Producer записал в топикА в партицию 0 несколько сообщений

А на стороне получателя есть несколько Consumer, объединенных в группу. Каждый Consumer читает определенную партицию

И вот, один из Consumer считал дынные из партиции 0
![[Pasted image 20251106102625.png]]

И при этом, Consumer успел обработать эти сообщения.

Но что делать, если этот консюмер - упал? 
- ❗❗Данные в партиции остались (см [[4. - Унифицированная модель адресата (Топик и партиции, offset и тд)]]), сообщения не удаляются, а ждут истечение TTL сегмента
	
- ❓❓Другому Consumer из этой группы теперь нужно мониторить и партицию падшего Consumer? Но тогда данные, которые уже обработались придется ПОВТОРНО считать
![[Pasted image 20251106103152.png]]

## ✅Решение
Для решения этой проблемы, в Kafka есть такие понятия как:
- offset
- commit

![[Pasted image 20251106103417.png]]

В Kafka Брокере, есть ==ОТДЕЛЬНЫЙ ТОПИК== **__consumer_offsets**

В этой топике, есть свой партиция, которая хранит в себе информацию о
- Партиции, откуда были считаны данные
- Группа консюмера, который считывал данные
- Номер сообщения (offset), которое уже было считано 

![[Pasted image 20251106103740.png]]


> [!NOTE] 🔥 Offset по умолчанию хранится 24 часа 🔥
> 

### Как происходит работа в таком случае?
Если Consumer считал данные и упал, то другой Consumer из группы берет на себя обработку партиции "падшего" Consumer

Но, он сначала читает из offset-топика информацию о том, что успел обработать "падший" Consumer
![[Pasted image 20251106104322.png]]

Из этого offests-топика, он видит, что "падший" Consumer обрабатывал партицию-0, и успел обработать сообщение с offset=2. 
Значит следующие сообщение нужно обрабатывать начиная с offset=3

# 📕3. Типы коммитов в offset-topic
![[Pasted image 20251106104631.png]]

Существуют следующие типы коммитов:

## 🔶Auto-commit
**Обеспечивает**: at most-once (доставка максимум 1 раз - потеря сообщений)

**Происходит когда Consumer ТОЛЬКО ПОЛУЧИЛ данные, НО НЕ ОБРАБОТАЛ**. 
То есть как только Consuemr получил данные, то сразу отправляет коммит.

Из-за этого сообщения могут потеряться, так как коммит был сделан. А Consumer мог упасть, хотя данные не успел обработать, а коммит в offset-топике уже есть

## 🔶Manual commit - ручной коммит
**Обеспечивает**: at least-onse (минимум 1 раз доставка - дубликаты)

Мы сами определяем когда отправить коммит (ОБЫЧНО ПОСЛЕ ОБРАБОТКИ)

Но из-за этого могут быть дубликаты, так как мы коммит сразу не отправляем, а сначала обрабатываем данные. 

Допустим мы обработали данные и упали, но коммит не успели сделать. Тогда при следующем чтении, начнем читать с тех же данных

### Exactly once
**Обеспечивает**: Exactly once (ровно один раз)

Для этого необходимо обеспечить собственное хранение offset (в БД, файле и т.д)
Такое сохранение нужно выполнять в транзакции

# 📕3. Пропажа offset
Может случиться ситуация, когда данные offset удаляются из offsets-топика для группы Consumer

Это регулируется при помощи параметров
- 🔶offsets.retention.minutes (default 7 days)
Если Consumer группа, не подключалась к партиции больше чем 7 дней, то offset удаляется из топика-offsets

- 🔶auto.offset.reset (earliest или latest)
Настройка определяющая, что делать Consumer, который был отключен в течении некоторого времени, и его offset был удален.
Есть два значения:
- earliest - самое первое которое уже лежит в топике
- latest - самое последнее сообщение
