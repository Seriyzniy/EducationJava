## Балансировка партиций
> [!NOTE] Причина такой балансировки?
> ❗❗❗Kafka учитывает ==КОЛИЧЕСТВО ВСЕХ ПАРТИЦИЙ ВСЕХ ТОПИКОВ НА БРОКЕР==❗❗❗
> 
> 👉Размазывает все партиции из всех топиков, на имеющихся брокеров равномерно

==Балансировка партиций== (partition balancing) — это процесс, при котором Kafka распределяет **партиции топика** по **разным брокерам** в кластере так, чтобы:
- нагрузка была **равномерной** (каждый брокер хранит примерно одинаковое количество данных),
    
- данные были **дублированы** (реплицированы) для отказоустойчивости,
    
- и при сбое брокера кластер мог **автоматически восстановить** баланс.

Kafka будет размещать партиции так, чтобы на каждом брокере
- Broker1

# 2. Репликация данных replication-factor

Если бы определенные партиция находились ТОЛЬКО на одном из брокеров, то при сбое данные из этой партиции терялись
![[Pasted image 20251106083137.png]]

### replication-factor
Поэтому были введены репликации партиций, которые регулируются параметром `replication-factor (>1)`

> Этот параметр говорит, **сколько реплик должно быть для каждой партиции**

> [!NOTE] 🔥 Правило для репликаций 🔥
> ==Реплики== одной партиции ==НЕ ДОЛЖНЫ находиться на одном брокере==

![[Pasted image 20251106083235.png]]

В таком случае, если упадет один из брокеров, то данные из "падших" партиций, будут на репликах на других брокерах

НО есть нюансы
- отставание реплики
- понятие главная реплика

## Лидерство и репликация
Чтобы обеспечиь согласованность данных, в Kafka есть Master-Slave система (Главный и ведомый)

![[Pasted image 20251106083841.png]]
А остальные назначаются ведомыми

Каждая **партиция** топика хранится на **нескольких брокерах**:
- 🔶**Лидер (Leader)** — брокер, который ==обслуживает все операции записи и чтения== для этой партиции.
    
- 🔶**Реплики (Followers)** — брокеры, которые **асинхронно копируют** данные лидера, чтобы быть готовыми принять управление, если лидер выйдет из строя.

💬Количество реплик задаётся параметром **replication.factor** при создании топика.

> 💡Например:💡
> Если replication.factor = 3, то каждая партиция хранится на трёх брокерах — один лидер и две реплики.

## Кто назначает лидера - Controller Broker
Распределением лидеров и реплик по брокерам управляет **контроллер кластера (Controller Broker)**. 

`Контроллер` — это **один из брокеров**, выбранный ZooKeeper (в старых версиях) или **KRaft controller** (в новых версиях Kafka ≥ 2.8 без ZooKeeper).

# 📕3. Отставание данных в ведомых партициях - ISR
Так как сообщения ВСЕГДА ПОПАДАЮТ В ЛИДЕРА ПАРТИЦИИ, то в репликах данные сами по себе не появятся

Для этого ведомые партиции (реплики) **С НЕКОТОРОЙ ПЕРЕОДИЧНОСТЬЮ опрашивают** лидера партиции: 
	"А что у тебя там нового есть"
![[Pasted image 20251106085038.png]]

И поскольку этот опрос происходит периодично, то Follower(ведомый) партиции могут отставать по данным от лидера

## ❌Проблема, лидер упал
![[Pasted image 20251106085257.png]]

Если в такой ситуации лидер упадет, то нужно выбрать другого лидера из ведомых

НО, у ведомых МОГУТ ОТСУТСВОВАТЬ все данные

## ✅In-sync replicas (ISR реплики)
Для того чтобы решить проблему, было введено понятие 
	"In-sync replicas (ISR)"

Этот параметр регулируется при помощи
	`min.insync.replicas`


> [!NOTE] ⭐СУТЬ⭐
> При записи в лидера, ==ДАННЫЕ СИХНРОННО ЗАПИСЫВАЮТСЯ В ISR-реплики==
> 
> 💡ОБЫЧНО ставят параметр НА ЕДИНИЦУ МЕНЬШЕ чем количество партиций
> 
> ❗❗❗НО такие реплики тормозят запись из-за синхронизации

![[Pasted image 20251106090355.png]]
(3 - потому что лидер и 2 фоловера)

Каждое новое сообщение лидер **сначала записывает у себя**, потом рассылает фолловерам, и только после того как **все ISR-реплики подтвердят запись**, лидер может вернуть продюсеру `ACK`.

### Пример, почему много ISR плохо
Представь:
- Есть **3 брокера** (`B1`, `B2`, `B3`);
    
- Есть **1 партиция** (`P0`);
    
- У партиции `replication.factor=3`, значит, все 3 брокера хранят её копии;
    
- ISR = `[B1, B2, B3]`.
    
Лидер, допустим, — `B1`.

#### Как идёт запись (при `acks=all`)
1. Продюсер отправляет сообщение лидеру (`B1`);
    
2. `B1` записывает сообщение на диск;
    
3. `B1` отправляет запись фолловерам (`B2`, `B3`);
    
4. Когда **все** из ISR ответили, что записали — лидер возвращает `ACK`.

🧠 То есть Kafka гарантирует:
> Сообщение записано на **все синхронные реплики** → значит, оно не потеряется.

#### Что случается, если **один брокер из ISR падает**

Допустим, `B3` упал. 

Тогда:
- Лидер (`B1`) не может получить подтверждение от всех из `[B1, B2, B3]`;
    
- Соответственно, **новые записи “зависают” — `acks=all` никогда не выполнится**, потому что один брокер не отвечает.

То есть Kafka ждёт, что **все реплики из ISR подтвердят запись**, а одна из них мертва → система блокируется.

#### Как Kafka решает проблему
Kafka не зависает навсегда.  

Через параметр `replica.lag.time.max.ms` брокер замечает, что `B3` давно не отвечает, и **исключает его из ISR**.  
Теперь ISR = `[B1, B2]`, и снова можно писать.

Но на это уходит время (по умолчанию — секунды).  
И пока брокер не исключён, **новые записи не подтверждаются**, а продюсеры ждут.
