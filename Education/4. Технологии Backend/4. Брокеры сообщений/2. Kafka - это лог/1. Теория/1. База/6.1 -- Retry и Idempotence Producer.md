# 📕1. Retry при отправке сообщений
**Retry** — это автоматический повтор попытки отправки сообщения в Kafka, когда первая попытка завершилась ошибкой.

**Retry вызывается в следующих случаях:**
1. **Временные ошибки сети** (таймауты, разрывы соединений)
    
2. **Недоступность брокеров** (брокер перезагружается или временно недоступен)
    
3. **Ошибка "Leader not available"** (лидер партиции переизбирается)

> [!NOTE] Вызывается АВТОМАТИЧЕСКИ
> Producer вызывает retry **сам, автоматически**, без какого-либо вмешательства разработчика.

> ПО УМОЛЧАНИЮ
> Producer пытается выполнять Retry в течении 2-х минут, а потом timeout-exception

## Два типа ошибок Retry
### Retriable Errors (Повторяемые ошибки)
Эти **ошибки считаются временными** и Kafka Producer автоматически повторяет отправку:

|Ошибка|Описание|Когда возникает|
|---|---|---|
|`NetworkException`|Проблемы с сетью|Разрыв соединения, таймауты|
|`TimeoutException`|Таймаут операции|Брокер не ответил за `request.timeout.ms`|
|`NotEnoughReplicasException`|Недостаточно реплик|Меньше реплик, чем требуется по `min.insync.replicas`|
|`NotLeaderForPartitionException`|Лидер недоступен|Перевыборы лидера партиции|
|`CorruptRecordException`|Поврежденная запись|Временные проблемы с сериализацией|
|`UnknownTopicOrPartitionException`|Топик/партиция неизвестны|Метаданные устарели, обновляются|


### Non-Retriable Errors (Неповторяемые ошибки)
Эти **ошибки считаются фатальными** - повторная отправка бессмысленна:

| Ошибка                        | Описание                  | Почему не повторяется               |
| ----------------------------- | ------------------------- | ----------------------------------- |
| `RecordTooLargeException`     | Сообщение слишком большое | Размер превышает `max.request.size` |
| `SerializationException`      | Ошибка сериализации       | Данные невозможно сериализовать     |
| `AuthenticationException`     | Ошибка аутентификации     | Неверные учетные данные             |
| `AuthorizationException`      | Ошибка авторизации        | Нет прав на запись в топик          |
| `UnsupportedVersionException` | Неподдерживаемая версия   | Несовместимость протоколов          |
| `InvalidTopicException`       | Невалидное имя топика     | Нарушение правил именования         |

# 📕 2. Idempotence

[13. Idempotence producer (Kafka - полный курс)](https://www.youtube.com/watch?v=7BpUvwg9nDs&list=PLt91xr-Pp57Q50WsXz9r-zmxy5ceu_hp_&index=13)

**Идемпотентность** — это свойство операции, **при котором многократное выполнение этой операции с одинаковыми входными данными не изменяет результат** после первого выполнения. 

Иными словами, независимо от того, сколько раз выполнена операция, итог будет одним и тем же.

> В нашем случае, НЕ БУДЕТ ДУБЛИКАТОВ

## Без Idempotence
Если Idempotence не настроен, то возможна ситуация, когда сообщение СОХРАНИТСЯ на брокере несколько раз

## С Idempotence
Если ИДЕМПОТЕНТНОСТЬ настроена, то сообщение не будет сохранено брокером дважды
![[Pasted image 20251112093405.png]]

Kafka РЕАЛИЗУЕТ идемпотентность через механизм **идентификаторов продюсера (Producer ID, PID)** и **порядковых номеров сообщений (sequence numbers)**.

**Цель:**  
даже если продюсер из-за ошибки или таймаута повторно отправит то же сообщение, брокер сможет **распознать дубликат** и **не записать его повторно** в лог партиции.

## Параметры для включения Idempotence
Основной параметр:
```bash
enable.idempotence = true
```
- Включает идемпотентность.
- После включения Kafka автоматически настраивает некоторые другие параметры, чтобы сохранить консистентность.

### Важные параметры, связанные с идемпотентностью:
| Параметр                                      | Описание                                                                     | Рекомендованное значение с идемпотентностью                                                                  |
| --------------------------------------------- | ---------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------ |
| `acks`                                        | Количество подтверждений, которое ждёт продюсер                              | `all` (то есть leader + все ISR)                                                                             |
| `retries`                                     | Количество повторов при ошибках                                              | Обычно `Integer.MAX_VALUE` (бесконечные повторы)                                                             |
| `max.in.flight.`<br>`requests.per.connection` | Сколько запросов может быть "в полёте" одновременно<br>(запросов без ответа) | <= 5 (для Kafka < 2.0) или <= 5–10 в новых версиях. Если больше, возможна перестановка сообщений при ретраях |
| `delivery.timeout.ms`                         | Максимальное время доставки                                                  | По умолчанию 2 минуты, обычно не трогают                                                                     |
| `transactional.id`                            | Только для транзакционного продюсера                                         | Не обязателен для обычного идемпотентного режима                                                             |


## Подводные камни и нюансы
1. **Совместимость версий**
    - Идемпотентность работает начиная с Kafka **0.11.0.0*
        
    - Для старых версий `enable.idempotence` просто игнорируется.
        
2. **`max.in.flight.requests.per.connection`**
    - Если >5 на Kafka <2.0, могут появиться **out-of-order сообщения** при повторной отправке.
        
    - На Kafka 2.0+ можно безопасно ставить до 5–10.
        
3. **Логика retries**
    - Идемпотентный продюсер не потеряет сообщения, если включены `retries`.
        
    - ❗❗❗Но важно, чтобы `delivery.timeout.ms` было больше, чем максимальное время всех повторов.
        
4. **Транзакции vs идемпотентность**
    - Идемпотентность гарантирует **single-produce**, но **не гарантирует атомарность** нескольких топиков или партиций.
        
    - Для атомарных операций нужно использовать **транзакции (`transactional.id`)**.
        
5. **Ошибки после ретраев**
    - Если продюсер получает `FencedException` (например, при смене `producerId` в транзакции), нужно **создать новый экземпляр продюсера**.
        
    - Повторное использование заблокированного продюсера приведёт к отказу.
        
6. **Производительность**
    - Идемпотентность чуть снижает throughput, так как брокер хранит последовательность сообщений.
        
    - В большинстве случаев это не критично.
