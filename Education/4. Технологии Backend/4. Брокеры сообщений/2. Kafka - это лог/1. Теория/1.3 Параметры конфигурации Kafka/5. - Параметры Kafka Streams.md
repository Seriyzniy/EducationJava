
## Базовые обязательные настройки
```properties
# Идентификатор приложения (обязательный). Все экземпляры данного приложения должны использовать один и тот же.
application.id=my‐streams‐app

# Адреса bootstrap-брокеров Kafka (обязательный).
bootstrap.servers=broker1:9092,broker2:9092
```

## Гарантии и отказоустойчивость
```properties
# Уровень гарантии обработки: at_least_once или exactly_once_v2
processing.guarantee=exactly_once_v2

# Число резервных копий (standby replicas) задач, если используется состояние
num.standby.replicas=1

# Репликация создаваемых внутренними темами топиков (changelog/repartition)
replication.factor=3
```

## Хранилище состояния и директория
```properties
# Директория на локальном диске, где будут храниться локальные state stores
state.dir=/var/lib/kafka-streams/state

# Максимальный лаг (в смещениях) восстановления, при котором экземпляр считается catch-up
acceptable.recovery.lag=1000
```
- `state.dir` важно, если у вас stateful приложение с локальным хранилищем (например, RocksDB) — не рекомендуется использовать `/tmp` в продакшене. [docs.confluent.io+1](https://docs.confluent.io/platform/current/streams/developer-guide/config-streams.html?utm_source=chatgpt.com)
    
- `acceptable.recovery.lag` контролирует, насколько далеко behind по смещениям может быть экземпляр перед тем, как его считать отставшим. [docs.confluent.io](https://docs.confluent.io/platform/6.2/streams/developer-guide/config-streams.html?utm_source=chatgpt.com)

## Сердце потоковой обработки (потребители/производители)
```properties
# По умолчанию Serde (сериализатор/десериализатор) для ключей
default.key.serde=org.apache.kafka.common.serialization.Serdes$StringSerde

# По умолчанию Serde для значений
default.value.serde=org.apache.kafka.common.serialization.Serdes$StringSerde

# Таймаут коммита состояния обработки (ms)
commit.interval.ms=3000

# Максимальное время ожидания “пустоты” при ожидании данных в join/merge для сохранения порядка
max.task.idle.ms=0
```
- Параметры `default.key.serde` и `default.value.serde` позволяют указать типы по умолчанию для ключей и значений.
    
- `commit.interval.ms` — как часто приложение будет выполнять commit состояния (и смещений) внутрь. Значение влияет на задержку и нагрузку.
    
- `max.task.idle.ms` важен, если у вас объединение (join) нескольких источников — задаёт максимальное время, которое задача простаивает в ожидании данных с “медленного” входа, чтобы сохранить порядок. [docs.confluent.io+1](https://docs.confluent.io/platform/6.2/streams/developer-guide/config-streams.html?utm_source=chatgpt.com)

## Настройки внутренней топологии и ребаланса
```properties
# Политика перебалансировки (group protocol) — classic или streams
group.protocol=streams

# Максимальное число “прогревочных” резервных задач (warmup replicas)
max.warmup.replicas=1

# Интервал между проверками “пробного” ребаланса (ms)
probing.rebalance.interval.ms=10000
```
- `group.protocol` управляет использованием нового протокола распределения партиций (Streams custom). [docs.confluent.io](https://docs.confluent.io/platform/current/streams/developer-guide/config-streams.html?utm_source=chatgpt.com)
    
- `max.warmup.replicas` и `probing.rebalance.interval.ms` контролируют поведение высокодоступных приложений при миграции задач между экземплярами. [docs.confluent.io](https://docs.confluent.io/platform/6.2/streams/developer-guide/config-streams.html?utm_source=chatgpt.com)

## Потребители/производители конфигурируемые через Streams
```properties

consumer.max.poll.records=1000

# Префикс producer-конфигурации (пример: переопределение linger.ms)
producer.linger.ms=50

# Префикс admin-клиента
admin.client.id=my‐streams‐admin
```
