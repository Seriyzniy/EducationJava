Stateless обработка проста
Данные никаким образом не сохраняются, данные поступают на вход, обрабатываются и поступают на выход

# 2. Steteful
Теперь, если нам необходимо обрабатывать данные потоково, И каким-то образом нужно собрать информацию по всем обработанным данным, то необходима Steteful обработка

Для этого в Kafka Streams используется ==Хранилище состояний==
(см [[10.1 -- Компоненты]])

> [!NOTE] ⭐По умолчанию используется RocksDB ⭐
> - Похож по структуре на TreeMap из Java
> - Хранится к ОЗУ (куче)
> - Хранит данные в виде ключ=значение

Условно, Stateful обработку можно представить так
![[Pasted image 20251110133638.png]]
Компоненты:
- Источник данных (обычно топик)
- Обработчик преобразования (processor который выполняет map)
- Обработчик преобразования (processor который выполняет sum)
	- Использует LocalStore для получения значений из уже обработанных данных
- Синхронизатор (processor который отправляет данные куда-либо - обычно в топик)

## Changelog-топик
Kafka Streams "под капотом" создают ДОПОЛНИТЕЛЬНЫЙ ТОПИК - ==chanhgelog==-топик

В этот топик попадают **данные содержащие информацию ОБ ИЗМЕНЕНИЯХ В LocalStore** (RocksBD).
Топик содержит то же количество партиций, которое обрабатывает Kafka Streams

Допустим у нас есть потребитель, который считывает данные с 3-х топиков. Данные в этом топике представляют информацию о фруктах на складе и их количестве.

![[Pasted image 20251110134644.png]]

Так как Consumer обрабатывает данные потоково, то Kafka Streams будет использовать RocksDB, для хранения результатов сообщений по ключу (бананы, яблоки, груши).

НО ПРИ ЭТОМ, Kafka Streams "под капотом" СОЗДАСТ ТОПИК changelog, с именем совпадающим с ID приложения, указанного в StreamConfig (группа Consumer)

Теперь, допустим поступили новые данные в топик (Source), и эти данные обработались в Kafka Streams, и изменения поместились в `cahngelog`-топик
![[Pasted image 20251110135431.png]]

### Проблема?
Но что если, мы добавим еще одного CONSUMER в той же группе? 
Kafka перебалансирует партиции топика между Consumer'ами одной группы.

Тогда новый Consumer будет обрабатывать партицию с грушами (допустим так произошла балансировка).
- Но у нового Consumer в БД - ПУСТО! 
- А ему допустим нужно ПОСЧИТАТЬ СУММУ всех груш
- В `__consumers_offsets` хранится offset последнего обработанного сообщения, но никак не сумма
- Придется вычислять все заново?
![[Pasted image 20251110135411.png]]

### Решение
Пересчитывать заново не придется. Kafka Streams прочитает changelog, и получит результат вычислений предыдущего Consumer

И после этого пойдет обрабатывать сообщения по последнему offset, который закомичен в `__consumers_offsets`
![[Pasted image 20251110140321.png]]
