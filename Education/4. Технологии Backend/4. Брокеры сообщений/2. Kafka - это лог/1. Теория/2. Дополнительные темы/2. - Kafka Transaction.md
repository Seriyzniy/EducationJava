# 📕 1. Проблема
Допустим у нас есть такой микросервис, который получает запрос на перевод денег с одного счета на другой

(синтетический пример, на самом деле можно было бы объединить снятие и перевод денег в один микросервис)
![[Pasted image 20251112161326.png]]

У этого Микросервиса есть 
- Consumer, который считывает запросы на перевод денег
- Producer, который передает сообщение в другие микросервисы

![[Pasted image 20251112161622.png]]

Может случиться ситуация, когда логика внутри сервиса выполнилась, и пришла очередь выполнения отправки в топки Kafka. 
Но при этом, в один топик мы сообщение поместили, а в другой нет.
![[Pasted image 20251112162102.png]]


## Когда делать commit для offset?
Нам нужно зафиксировать обработку сообщения путем смещения offset

Когда его коммитить? 
Очевидно только после того, как сообщение ГАРАНТИРОВАНО отравилось в оба топика - так как это признако успешно обработанного сообщения в данном случае

Но что если в один топик сообщение отправилось, а в другой по какой-то причине не отправилось? 
Тогда offset мы не будем коммитить, но сообщение уже ушло в один топик!!! 
И это сообщение кто-то обработает, хотя данные могут быть в несогласованном состоянии

---
Теперь, допустим наш микросервис поднялся после падения, и начинает работу.
Так как offset не был закоммичен, он считает сообщение еще раз
![[Pasted image 20251112163342.png]]

> [!NOTE] ❗❗❗ КОСЯК ❗❗❗
> Хотя если Producer идемпотентен, то сообщение дубликат не будет попадать в ОРАНЖЕВЫЙ топик

---
# 2. Решение - Kafka Transaction

Мы хотим, чтобы данные попадали
- либо в два топика сразу
- либо вообще не попадали

Для решения, используется Kafka Transaction.

В этом случае, когда Producer добавляет сообщения в топик, то это сообщение будет находится в статусе `Uncommited`

А Consumer'ы настроены на чтение только закомиченных сообщений `Read commited`

![[Pasted image 20251112163629.png]]


Тогда если сервис отправит сообщение в один из топиков, и упадет, то сообщение будет помечено как `Uncommited`. 
![[Pasted image 20251112164044.png]]

А когда сервис поднимется, и отправит другое сообщение в другой топик, сообщения перейдут в состояние `Commited`, а сообщения с этим состоянием Consumer сможет считать
![[Pasted image 20251112164155.png]]

## Реализация транзакций в Kafka
### Уровни участников
1. **Transactional Producer** — продюсер, который работает в рамках транзакции (`transactional.id` обязателен).
    
2. **Transaction Coordinator** — специальная роль на брокере, которая управляет жизненным циклом транзакции.
    
3. **Kafka Broker** — обычные брокеры, хранящие данные топиков.

### Как выбирается координатор
Когда продюсер создаётся и у него есть `transactional.id`, Kafka делает следующее:

1. ❗❗❗Берёт `transactional.id` (для каждого продюсера свой координатор)
    
2. Хэширует его (`hash(transactional.id) % N`), где `N` — количество партиций служебного топика `__transaction_state`
    
3. Находит **партицию этого топика**, в которой будет храниться состояние этой транзакции
    
4. Лидер этой партиции = **Transaction Coordinator** для данного `transactional.id`

Таким образом:
> координатор определяется детерминированно, на основе `transactional.id`.

📌 **Служебный топик `__transaction_state`** — это внутренняя системная таблица Kafka, в которой хранятся все состояния транзакций: `Begin`, `PrepareCommit`, `Commit`, `Abort`, и т. д.


### ⭐⭐Механика⭐⭐
#### 1️⃣1. ⭐Инициализация
Продюсер создаётся с параметром `transactional.id`.

Kafka присваивает ему **уникальный `producerId` и `producerEpoch`**, которые будут использоваться между сессиями 
(в отличие от обычного идемпотентного продюсера, где `producerId` временный).

```java
props.put("transactional.id", "my-tx-id"); 

// включается автоматически
props.put("enable.idempotence", "true"); 
```

#### 2️⃣2. Начало транзакции
Продюсер вызывает:
```java
producer.initTransactions(); 

producer.beginTransaction();
```

#### 3️⃣3. В процессе
Продюсер:
- **читает сообщения** из input-topic (через consumer),
    
- **отправляет** новые сообщения в один или несколько output-топиков,
    
- ⭐**помечает оффсеты** прочитанных сообщений как обработанные в рамках транзакции.
Это делается методом:
```java
producer.sendOffsetsToTransaction(offsets, consumerGroupId);
```
👉 Это ключевой момент — Kafka позволяет **атомарно записать** и результат обработки, и факт “прочтения”.


#### 4️⃣4. Завершение транзакции
✅Если всё успешно:
```java
producer.commitTransaction();
```
→ Transaction Coordinator записывает специальную **“commit marker” запись** в служебный топик `__transaction_state` и отправляет маркер в каждую затронутую партицию.


❌Если произошла ошибка:
```java
producer.abortTransaction();
```
→ Transaction Coordinator записывает **“abort marker”**, и все сообщения, отправленные в рамках транзакции, **логически игнорируются** потребителями, читающими в режиме `read_committed`. (ПОМЕЧАЮТСЯ КАК `Uncommited`)


### Пример: Read → Process → Write (Java)
```java
KafkaConsumer<String, String> consumer = 
	new KafkaConsumer<>(consumerProps);
KafkaProducer<String, String> producer = 
	new KafkaProducer<>(producerProps);

producer.initTransactions();

while (true) {
    ConsumerRecords<String, String> records = 
	    consumer.poll(Duration.ofMillis(100));

    if (records.isEmpty()) continue;

    producer.beginTransaction();
    try {
        for (ConsumerRecord<String, String> record : records) {
            // Обработка данных
            String transformed = record.value().toUpperCase();

            // Отправляем в первый топик
            producer.send(new ProducerRecord<>(
	            "output-topic-1",
	            transformed)
	        );

            // Отправляем во второй топик
            producer.send(new ProducerRecord<>(
	            "output-topic-2", 
	            transformed)
	        );
        }

        // Фиксируем оффсеты чтения как часть транзакции
        Map<TopicPartition, OffsetAndMetadata> offsets = 
	        currentOffsets(records);
        
        // Отправка
        producer.sendOffsetsToTransaction(
	        offsets, 
	        consumer.groupMetadata()
	    );

        // Коммит транзакции
        producer.commitTransaction();

    } catch (Exception e) {
        producer.abortTransaction(); // откат
    }
}

```

### Как брокер обеспечивает атомарность
- Брокер **не делает двухфазного commit'а** в привычном виде, но реализует _подобие 2PC_ на уровне Kafka логов.
    
- Все сообщения внутри транзакции имеют флаг `isTransactional`.
    
- Пока транзакция не закоммичена, consumer с `isolation.level=read_committed` их **не видит**.
    
- Если транзакция “abort’нута” — эти записи **навсегда игнорируются**, даже если физически в логе остаются.


### Ключевые параметры конфигурации

| Параметр                                      | Назначение                                                                                             | Типичное значение                                 |
| --------------------------------------------- | ------------------------------------------------------------------------------------------------------ | ------------------------------------------------- |
| `transactional.id`                            | Уникальный ID продюсера (идентификатор транзакции). Должен быть постоянным между рестартами приложения | Например, `"processor-1"`                         |
| `enable.idempotence`                          | Включает идемпотентность (для транзакций обязательна)                                                  | `true`                                            |
| `acks`                                        | Гарантия записи всех реплик                                                                            | `all`                                             |
| `retries`                                     | Повторы при сбоях                                                                                      | `Integer.MAX_VALUE`                               |
| `max.in.flight.`<br>`requests.per.connection` | Максимум "в полёте" запросов                                                                           | ≤ 5                                               |
| `transaction.timeout.ms`                      | Максимальная длительность транзакции, по истечении которой брокер сам её прервет                       | По умолчанию 1 минута (можно увеличить до 15 мин) |
| `isolation.level` (для consumer)              | Поведение чтения при транзакциях                                                                       | `read_committed` (или `read_uncommitted`)         |

#### Подводные камни

|Проблема|Описание|
|---|---|
|🔸 `transactional.id` должен быть уникален|Если два продюсера используют один ID, Kafka “заблокирует” старого (fencing).|
|🔸 Транзакции не бесплатны|Каждая транзакция = координация с `__transaction_state` → нагрузка на кластер. Лучше объединять несколько сообщений в одну транзакцию.|
|🔸 Время жизни транзакции (`transaction.timeout.ms`)|Если превышено — Kafka сама её abort’нет.|
|🔸 Совместимость|Exactly-once доступно начиная с Kafka **0.11.0.0**, но стабильно работает с **2.5+**.|
