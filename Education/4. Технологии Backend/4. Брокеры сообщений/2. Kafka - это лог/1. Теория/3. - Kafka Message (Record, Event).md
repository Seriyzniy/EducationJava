Сообщение (в Kafka часто называют **Record** или **Event**) — это **основная единица данных**, которая передаётся от продюсера к брокеру и далее читается консюмером.

# 📕1. Сообщение в Kafka
**Kafka Record (сообщение)** — это _не просто строка или JSON_, а структурированный объект, который содержит:

| Поле          | Назначение                                                                     |
| ------------- | ------------------------------------------------------------------------------ |
| **Topic**     | Имя топика, куда отправляется сообщение                                        |
| **Partition** | Партиция, в которую будет записано сообщение                                   |
| **Offset**    | 💡==Уникальный номер== (смещение) в пределах партиции💡                        |
|               |                                                                                |
| **Key**       | ==Ключ сообщения== (опционально, используется для маршрутизации и группировки) |
| **Value**     | ==Основное содержимое сообщения== (payload, данные события)                    |
| **Timestamp** | Метка времени (создания или записи на брокер)                                  |
| **Headers**   | 💡Пользовательские метаданные (аналог HTTP-заголовков, опционально)            |

## Пример записи (Kafka Record)
```json
{
  "topic": "user-events",
  "partition": 2,
  "offset": 105,
  "key": "user123",
  "value": {
    "eventType": "login",
    "timestamp": "2025-10-31T12:00:00Z"
  },
  "headers": {
    "source": "web",
    "version": "1.0"
  }
}
```

# 📕2.Компоненты Kafka Record

### Key
- Необязательный, используется для распределения сообщений по кластеру
	
- ❗❗Используется ==для выбора партиции==:  
    `partition = hash(key) % numPartitions`
    (используется Murmur2 в качестве хеш-фукнции)
	
- Гарантирует, что **все сообщения с одинаковым ключом попадут в одну партицию**,  
    а значит — **сохранят порядок** между собой.


Пример: все события одного пользователя (`userId`) идут в одну партицию.

---
### Value
- Основное ==тело сообщения==.
    
- Может быть строкой, JSON, бинарным массивом и т.д.
    
- ❗❗❗Kafka сама **не интерпретирует** содержимое value —  
    это ответственность сериализатора и потребителя.

---
### Timestamp
- В Kafka у каждого сообщения есть timestamp:
    - `CreateTime` — когда продюсер создал сообщение.
        
    - `LogAppendTime` — когда брокер записал сообщение в лог.
        
- Используется для TTL, ретеншена и анализа задержек.
    

---
### Headers
- Появились с Kafka 0.11+.
    
- Набор пар «ключ → значение» ==для дополнительной информации== (например, trace id, тип сообщения и т.д.).


> [!NOTE] ВСЕГДА ИСПОЛЬЗУЙ одну и ту же версию Kafka для Producer и Consumer
> Иначе более новые могут использовать Headers а старые версии не понимают этот формат
