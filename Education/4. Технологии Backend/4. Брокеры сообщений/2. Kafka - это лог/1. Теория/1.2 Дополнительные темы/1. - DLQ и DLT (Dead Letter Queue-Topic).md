
`DeserializationException` возникает на стороне **Consumer**, когда Kafka не может корректно десериализовать сообщение из байтов в объект Java.

Типичные причины:
- несовпадение схемы сериализации/десериализации (например, Avro/JSON);
    
- изменение модели данных без миграции;
    
- неверный `value.deserializer` или `key.deserializer`;
    
- повреждённое сообщение в топике;
    
- в Spring Kafka — неверная настройка `ErrorHandlingDeserializer`

# ❌ 1. Проблема
Допустим у нас есть топик, в котором отправляются сообщения определенного формата.
Этот формат можно десериализовать при помощи JSON Deserializer
![[Pasted image 20251112100007.png]]

Но если в этот же топик, кто-то отправит другой формат сообщения, **то Consumer не сможет его десериализовать, и зависнет в бесконечной попытке** его обработать 
(так как offset он не сместит)

# ✅2. Решение в Apahce Kafka (без Spring)
Когда consumer читает сообщение, Kafka вызывает `Deserializer` для `key` и `value`.  

Если десериализация не удалась (например, формат не тот или JSON битый) →  выбрасывается `SerializationException` / `DeserializationException`.

По умолчанию consumer **упадёт** и **перестанет обрабатывать** партицию.  

💡Чтобы этого не происходило — в Kafka есть механизм **Deserialization Exception Handler**.

## 1. Решение DeserializationException через конфигурацию Consumer

Существует ==параметр==:
```yaml
default.deserialization.exception.handler
```
который определяет **класс**, который будет вызван, если Kafka Consumer не смог десериализовать ключ или значение сообщения.

### Доступные стандартные обработчики (Kafka client API)
Классы находятся в пакете `org.apache.kafka.clients.consumer`

| Класс                                | Описание                                                                                    |
| ------------------------------------ | ------------------------------------------------------------------------------------------- |
| **`LogAndContinueExceptionHandler`** | Логирует ошибку <br>и _пропускает сообщение_, двигаясь дальше. <br>Consumer не падает.      |
| **`LogAndFailExceptionHandler`**     | Логирует ошибку <br>и _останавливает consumer_ <br>(дефолтное поведение).                   |
| **(кастомный класс)**                | Можно написать свой обработчик, <br>реализовав интерфейс `DeserializationExceptionHandler`. |

### Пример конфигурации
```properties
bootstrap.servers=localhost:9092
group.id=my-consumer-group

key.deserializer=
	org.apache.kafka.common.serialization.StringDeserializer
	
value.deserializer=
	org.apache.kafka.common.serialization.StringDeserializer

# при ошибке десериализации — просто логировать и двигаться дальше
default.deserialization.exception.handler=
org.apache.kafka.clients.consumer.LogAndContinueExceptionHandler
```

## 2. DLQ / DLT — Dead Letter Queue / Topic
**DLQ (Dead Letter Queue)** или **DLT (Dead Letter Topic)** — это **резервный топик**, куда ==отправляются сообщения, которые не удалось обработать== 
(десериализация, валидация, бизнес-ошибка и т.д.).

> DLQ ≈ DLT, но в Spring Kafka чаще говорят **DLT (Dead Letter Topic)**.

![[Pasted image 20251112102538.png]]

А этот DLT может слушать другой Consumer, который будет обрабатывать эти сообщения

### DLQ и DLT пишешь сам
Kafka “в чистом виде” не имеет встроенного механизма DLQ. 

Ты реализуешь его **сам**:
```java
try {
    MyMessage msg = deserialize(record.value());
    process(msg);
} catch (Exception e) {
    producer.send(
	    new ProducerRecord<>(
		    "my-topic-DLQ", 
		    record.key(), 
		    record.value()
		)
	);
}
```

### Какой сериализатор использовать для отправки в DLT?
Ты хочешь отправить **битое или неизвестного формата** сообщение в DLQ.  

НО:
- чтобы отправить сообщение в Kafka — нужен **Producer**
- Producer требует **Serializer**
- Но ты **не знаешь**, чем сериализовать, ведь исходная «плохая» запись не десериализуется.

> ==в DLQ мы НЕ сериализуем оригинальный payload==
> 
> Так как мы не смогли десериализовать сообщение, значит и сериализовывать его не нужно

Поэтому, используем самый просто сериализатор:
```java
new KafkaProducer<>(props,
        new StringSerializer(),
        new ByteArraySerializer()
)
```
То есть:
- ключ → String (или тоже ByteArray)
- значение → ByteArray

#### Почему ByteArraySerializer?
Потому что ты отправляешь «сырой» byte[] из ConsumerRecord

## Итого по "чистой" Kafka
| Что решает проблему                           | Где настраивается    | Что делает                                     |
| --------------------------------------------- | -------------------- | ---------------------------------------------- |
| default.deserialization.<br>exception.handler | Kafka Consumer props | Управляет поведением при ошибке десериализации |
| `LogAndFailExceptionHandler`                  | встроенный           | Падает при ошибке                              |
| `LogAndContinueExceptionHandler`              | встроенный           | Пропускает сообщение                           |
| Собственный `DeserializationExceptionHandler` | кастом               | Любое поведение (лог, DLQ, метрики и т.д.)     |
| DLQ вручную                                   | свой код             | Отправка ошибочного сообщения в другой топик   |

# ✅3. Решение с Spring Kafka
Spring Kafka добавляет **обёртки и расширенные механизмы** поверх стандартного клиента.  

Он даёт:
- безопасную десериализацию (`ErrorHandlingDeserializer`);
    
- централизованный `DefaultErrorHandler`;
    
- автоматическую поддержку DLT.

## 1. ErrorHandlingDeserializer
Это «обёртка» над любым десериализатором (JSON, Avro, String и т.п.). 

Если десериализация падает → consumer не умирает, а получает **объект-ошибку**, который дальше можно обработать.

> [!NOTE] ⭐Можно указать для key\value Deserializer⭐
> 

## 2. Dead Letter Publishing Recoverer
Spring Kafka автоматически создаёт **DLT (Dead Letter Topic)**

Сообщение публикуется туда с заголовками:

|Header|Значение|
|---|---|
|`kafka_dlt-exception-message`|сообщение об ошибке|
|`kafka_dlt-exception-stacktrace`|stacktrace|
|`kafka_dlt-original-topic`|оригинальный топик|
|`kafka_dlt-original-offset`|offset сбойного сообщения|

```java
DeadLetterPublishingRecoverer recoverer = 
	new DeadLetterPublishingRecoverer(
        template,
        // Функция выбора DLT топика
        (record, ex) -> new TopicPartition(
	        record.topic() + ".DLT",
	        record.partition()
	    )
    );
```

📌 В этом примере DLT топик будет называться:
```css
<имя_оригинального_топика>.DLT
```
и попадёт в ту же партицию, что и оригинал.

## 3. DefaultErrorHandler (ранее SeekToCurrentErrorHandler)
Spring вызывает его при **любых** ошибках (не только десериализация, но и бизнес-логика).

> [!NOTE] Внутри использует DeadLetterPublishingRecoverer
> 

```java
@Bean
public DefaultErrorHandler errorHandler(
	KafkaTemplate<Object, Object> template) 
{
    DeadLetterPublishingRecoverer recoverer = 
	    new DeadLetterPublishingRecoverer(
		    template,
		    (r, e) -> new TopicPartition(
				r.topic() + ".DLT", 
				r.partition()
			)
		);

    DefaultErrorHandler handler = 
	    new DefaultErrorHandler(recoverer);
	    
    handler.addNotRetryableExceptions(
	    DeserializationException.class
	);
    handler.setBackOffFunction(
	    (r, e) -> new FixedBackOff(1000L, 2)); // 2 ретрая по 1с

    return handler;
}
```

Этот обработчик, передается в фабрику, где создаются Listner'ы (слушатели топика)
