# 📕 1. KRaft

==KRaft== (Kafka Raft Metadata Mode) — это **новый режим** Kafka **без ZooKeeper**.

В KRaft каждый брокер может иметь **две роли**:
1. **broker** — хранит и обслуживает топики, партиции, принимает и отдает сообщения.
    
2. **controller** — управляет кластерной метаинформацией:
    - какие топики существуют
    - где партиции
    - кто лидер партиции
    - создание/удаление топиков


> В старом Kafka с ZooKeeper роль контроллера выполнял отдельный процесс.  
> 
> В KRaft *контроллер встроен прямо в брокер*, НО НУЖЕН **отдельный listener**, *чтобы брокеры и клиенты не мешались с контроллером*.

## Security Protocol
Это реальный сетевой протокол безопасности:

| Протокол       | Значение                            |
| -------------- | ----------------------------------- |
| **PLAINTEXT**  | обычное TCP без шифрования          |
| SASL_PLAINTEXT | TCP без шифрования + аутентификация |
| SSL            | TLS шифрование                      |
| SASL_SSL       | TLS + аутентификация                |

## Listener
**Listener** — это интерфейс/порт, на котором Kafka слушает соединения.

Формат:
```css
<listener_name>://<host>:<port>
```
- **listener_name** — произвольное имя (например, `PLAINTEXT`, `CONTROLLER`)
    
- **host:port** — на каком адресе и порту принимать соединения

Kafka поддерживает несколько listener’ов сразу:
- один *для внешних клиентов*
- один *для других брокеров*
- один *для внутреннего контроллера*


### Пример
|Listener|Host:Port|Для кого|
|---|---|---|
|PLAINTEXT|0.0.0.0:29092|Kafka-UI, другие контейнеры Docker|
|PLAINTEXT_HOST|0.0.0.0:9092|Java-клиент на хосте (localhost)|
|CONTROLLER|0.0.0.0:9093|Контроллер KRaft, внутренние процессы Kafka|

**PLAINTEXT_HOST** → Java-клиент на твоей машине должен видеть Kafka как `localhost:9092`.

**PLAINTEXT** → Kafka-UI в Docker видит Kafka как `kafka:29092`
  
**CONTROLLER** → KRaft контроллеры общаются друг с другом, управляют метаданными, без вмешательства клиентов.

> PLAINTEXT - имя listenerʼа
> 0.0.0.0 - *слушать на всех интерфейсах* внутри контейнера
> 29092 / 9092 - порты внутри контейнера

## Кворум
В Kafka 3.x и выше можно запускать Kafka без ZooKeeper — в режиме **KRaft (Kafka Raft Metadata)**.

> *Метаданные теперь хранятся* не в ZooKeeper, а в **встроенном Raft-кворуме**.

> [!NOTE] Кворум 
> это **группа управляющих узлов** (контроллеров), которые *хранят и реплицируют метаданные Kafka* через Raft.
> 

Похожая идея:
- в ZooKeeper был набор узлов
- в KRaft — группа контроллеров

Кворум нужен для:
- согласованного хранения топиков, партиций, их лидеров
- распределения ролей
- избрания лидера контроллеров
- устойчивости к сбоям


> Raft требует **нечётное число узлов в кворуме** (1, 3, 5…).

### Кворум в режиме _single node_
Когда ты запускаешь Kafka в Docker в режиме одиночного брокера:
```yaml
KAFKA_PROCESS_ROLES=broker,controller
KAFKA_CONTROLLER_QUORUM_VOTERS=1@kafka:9093
```

Это означает:
- кворум состоит из 1 узла
- этот узел — контроллер
- он слушает по адресу `kafka:9093`  

То есть кластер минимальный, но полноценный, хотя устойчивости нет (при падении брокера всё падает).

### Распределённый кластер (3 контроллера)
```yaml
KAFKA_CONTROLLER_QUORUM_VOTERS=
	1@node01:9093,2@node02:9093,3@node03:9093
```

# 📕2. Основные параметры конфигурации KRaft

## Параметры, определяющие роль ноды

### KAFKA_PROCESS_ROLES (process.roles)

Роли узла:
- `broker` — обслуживает топики/партиции
    
- `controller` — участвует в Raft-кворуме и хранит метаданные
 

Примеры:
❌ Только брокер (не работает в KRaft):
```yaml
broker
```

❌ Только контроллер:
```yaml
controller
```

✔ Single-node KRaft:
```yaml
broker,controller
```

---
## Идентификация узла в кворуме

### KAFKA_NODE_ID
Уникальный ID узла.

Используется в Raft-кворуме.  
Каждый узел = уникальный ID.

```yaml
KAFKA_NODE_ID: 1
KAFKA_CONTROLLER_QUORUM_VOTERS: 1@kafka:9093
```
Указываем ID, и через @ указываем адрес внутри кластера

---
## Параметры кворума (вместо ZooKeeper)

### KAFKA_CONTROLLER_QUORUM_VOTERS
Указывает всех контроллеров:
```css
<node.id>@<host>:<port>

<node.id>@<host>:<port><node2>@<host>:<port>,...
```

### KAFKA_CONTROLLER_LISTENER_NAMES
Слушатель, по которому работают контроллеры.

Всегда должен присутствовать в `KAFKA_LISTENERS`.
Обычно:
```yaml
CONTROLLER
```

### KAFKA_CONTROLLER_QUORUM_BOOTSTRAP_SERVERS (альтернатива)

Можно использовать вместо `voters`, но только в dev.

---
## Параметры listener’ов
### KAFKA_LISTENERS
Список интерфейсов, на которых Kafka будет слушать:
```yaml
PLAINTEXT://0.0.0.0:29092,
PLAINTEXT_HOST://0.0.0.0:9092,
CONTROLLER://0.0.0.0:9093
```

Каждый listener обслуживает свои задачи.

### KAFKA_ADVERTISED_LISTENERS
Как Kafka сообщает клиентам, ==КУДА подключаться==.
```yaml
PLAINTEXT://kafka:29092
PLAINTEXT_HOST://localhost:9092
```

### KAFKA_LISTENER_SECURITY_PROTOCOL_MAP
Необходимая карта listener → протокол.

Для KRaft обязательно:
```yaml
PLAINTEXT:PLAINTEXT
PLAINTEXT_HOST:PLAINTEXT
CONTROLLER:PLAINTEXT
```

---
## Системные параметры
### KAFKA_LOG_DIRS
Где хранить data + metadata.

Для KRaft всегда должен быть установлен.
