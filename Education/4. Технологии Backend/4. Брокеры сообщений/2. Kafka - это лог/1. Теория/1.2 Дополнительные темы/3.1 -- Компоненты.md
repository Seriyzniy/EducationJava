# 📕1. Поток и "таблица"
Kafka Streams оперирует двумя основными абстракциями: **KStream** и **KTable**:
🔶Поток(**KStream**) — это **непрерывный поток событий** (пар «ключ–значение»), представляющий упорядоченную последовательность неизменяемых записей. 
Подходит для обработки данных по мере их поступления — например, логов событий, транзакций, кликов и т.п.

🔶Таблица(**KTable**) — собой **снимок**, который **фиксирует последнее значение ❗❗ДЛЯ КАЖДОГО КЛЮЧА**❗❗ в потоке в любой момент времени.
> Напоминает таблицу БД, обновляемую при поступлении новых данных. 
> ❗❗Основана на журнале изменений (==changelog topic==) и позволяет работать с текущим состоянием системы.

| **Характеристика**       | **KStream (Поток)**                                                | **KTable (Таблица)**                                               |
| ------------------------ | ------------------------------------------------------------------ | ------------------------------------------------------------------ |
| **Что представляет**     | Последовательность неизменяемых событий/записей                    | Снимок текущего состояния для каждого ключа                        |
| **Природа данных**       | Неограниченный, упорядоченный, воспроизводимый лог событий         | Непрерывно обновляемая таблица «ключ-последнее значение»           |
| **Типичные операции**    | Фильтрация, преобразование (map), группировка, объединение потоков | Агрегация, объединение с потоками/таблицами, интерактивные запросы |
| **Пример использования** | Логи, клики пользователей, транзакции, показания сенсоров          | Текущий баланс счета, статус заказа, профиль пользователя          |

Эта дуальность **KStream** и **KTable** образует мощную парадигму, объединяющую обработку событий и управление состоянием.  

Она позволяет решать задачи без внешних БД: 
> например, 
> поток заказов (**KStream**) можно обогащать актуальными данными о пользователях (**KTable**), которая постоянно обновляется из другого потока.  

Такой подход упрощает архитектуру, устраняет необходимость во внешних хранилищах состояния и повышает производительность.

# 📕2. Топология

> [!NOTE] Топология в Kafka Streams 
> представляет собой ориентированный ациклический **граф обработки потока**, который состоит из:
> - источников, 
> - процессоров
> - хранилищ состояния. 
> 
> Топология определяет, как данные будут обрабатываться и перемещаться по системе.
> 

Топология может состоять из следующих компонентов:
- **Источник (Source Processor)**: Источник — это точка входа в топологию, из которой считываются данные, чем обычно являются Kafka топики.
    
- **Потоковый процессор (Stream Processor)**: Потоковые процессоры — это **узлы в топологии, которые обрабатывают данные**. Они могут выполнять различные операции, такие как фильтрация, преобразование и агрегации данных.
    
- **Хранилища состояния (State Store)**: Хранилища состояния позволяют процессорам сохранять и извлекать данные. Это полезно для операций, которые требуют знания о предыдущем состоянии, например, для агрегации или join операций.
    
- **Синхронизатор (Sink Processor)**: Синхронизатор отправляет полученные записи от его верхнеуровневого процесса в указанный Kafka топик.

Вычислительная логика приложения **Kafka Streams** определяется **топологией процессора** — графом, где 
- **узлы представляют операции обработки** данных (_map, filter, join, aggregate_), 
- а **рёбра** — **потоки данных** между ними.

![[Pasted image 20251109170735.png]]

Топологию можно задать двумя способами:
1. **DSL (Domain-Specific Language)** — декларативный, функциональный API, рекомендованный для большинства случаев. Позволяет описывать обработку данных в несколько строк с помощью встроенных операций.
    
2. **Processor API** — **низкоуровневый**, императивный API для более гибкой и детальной настройки логики, включая прямое управление состоянием.
    
### DSL
В DSL доступны ключевые операции потоковой обработки:  
`map` — преобразование,  
`filter` — фильтрация,  
`groupBy` — группировка,  
`aggregate` — агрегация,  
`join` — объединение потоков,  
`windowedBy` — оконные вычисления.

## 📙1.Источник (Source Processor)

> [!NOTE] Источник (Source Processor) в **Kafka Streams**
> это начальный узел топологии, который **читает данные из Kafka-топиков** и передаёт их дальше на обработку потоковым процессорам.
> 

Когда вы создаёте поток через `builder.stream("topic")` или `builder.table("topic")`, Kafka Streams автоматически создаёт источник, связанный с указанным топиком.

Пример:
```java
StreamsBuilder builder = new StreamsBuilder();

KStream<String, String> source = builder.stream(
	"input-topic",
    Consumed.with(Serdes.String(), Serdes.String())
);
```
(см [[3. - Интерфейс Serde и Фабрика Serdes]])

Здесь создаётся **KStream**, читающий пары _ключ–значение_ из `input-topic`.  
Источник всегда является **неизменяемой начальной точкой топологии**, не имеет предшественников и определяет, откуда поступают данные в поток.

## 📙2. Потоковый процессор (Stream Processor)

> [!NOTE] **Потоковый процессор (Stream Processor)** 
> это ключевой компонент в обработке данных, представляющий собой узел в топологии, который берет данные, проходящие через него, и выполняет над ними различные операции.
> 

Отвечает за **выполнение операций над потоком данных**, поступающих из источников (Source Processors). 

> Каждая операция Kafka Streams (`filter`, `map`, `flatMap`, `groupBy`, `aggregate` и др.) ==соответствует созданию одного или нескольких потоковых процессоров== внутри топологии.

### ОСОБЕННОСТЬ
> [!NOTE] ❗❗ ВАЖНОЕ ПРАВИЛО ❗❗
> Операторы Kafka Streams (map, filter, transform, join, process и другие) должны выполнять только быстрые, неблокирующие, детерминированные операции. 
> 
> Запрещается выполнять длительные или блокирующие действия, такие как обращения к базе данных, вызовы внешних REST/SOAP сервисов, операции с файловой системой, ожидания (sleep), либо любой другой ввод-вывод, который может задержать поток обработки.

- **Каждый Stream Task работает однопоточно** — любая блокировка останавливает обработку всех сообщений данного таска.
    
- **Блокировка превышающая commit interval нарушает семантику Exactly-Once**.
    
- **Длительные операции приводят к rebalancing**, что может вызвать:
    - потерю владения партициями
        
    - CommitFailedException
        
    - зависание state store (RocksDB)
        
- **Работа с внешними системами нарушает идемпотентность Kafka Streams**, так как побочные эффекты нельзя откатить.

### Основные типы операций:

#### 1. Фильтрация (`filter`)
Отбирает только записи, удовлетворяющие условию.
```java
// Creating a topology
StreamsBuilder builder = new StreamsBuilder();

// Reading data from the "transactions-topic" Kafka topic to a KStream (source)
KStream<String, Double> transactions = builder
	.stream(
		"transactions-topic", 
		Consumed.with(Serdes.String(), Serdes.Double())
	);
// Filtering transactions
KStream<String, Double> largeTransactions = transactions
	.filter((id, transaction) -> 
		transaction.getAmount() > 1000.00
	);
```

**Назначение:** оставляет только нужные данные для последующей обработки.  
**Пример:** фильтрация транзакций по сумме.

#### 2. Преобразование (`map`, `mapValues`)
Преобразует записи потока.

- 🔶`mapValues` изменяет **ТОЛЬКО ЗНАЧЕНИЕ** (value), **не меняет ключ**, не вызывает перераспределения данных.
```java
// Creating a topology
StreamsBuilder builder = new StreamsBuilder();

// Reading data from the "transactions-topic" Kafka topic to a KStream (source)
KStream<String, Double> transactions = builder
	.stream(
		"transactions-topic", 
		Consumed.with(Serdes.String(), Serdes.Double())
	);

// Mapping transactions
KStream<String, String> transactionEventDescriptions = transactions
	.mapValues((userId, transactionValue) -> {
		return String.format(
			"User %s has made a transaction of $%s", 
			userId, 
			transactionValue
		);
	});
```

- 🔶`map` изменяет **КЛЮЧ и/или ЗНАЧЕНИЕ**, что может вызвать **repartitioning** (перераспределение по партициям).
```java
KStream<String, String> transactionEventDescriptions = transactions
	.map((userId, transactionValue) -> {
	    String message = String.format(
		    "User %s has made a transaction of $%s",
		    userId, 
		    transactionValue
		);    
		return KeyValue.pair(userId, message);
	});
```

**Назначение:** преобразует структуру данных в нужный формат для последующих шагов.

#### 3. Агрегация (`groupByKey`, `count`, `aggregate`)
Позволяет выполнять группировку и сводные операции.

```java
KTable<String, Long> txCount = transactions.groupByKey().count();
```

**Назначение:** получение агрегированных результатов — подсчёта, суммы, среднего и т.п.  
**Результат:** формируется `KTable`, представляющая текущие агрегированные состояния.

### Особенности:
- Каждый Stream Processor — это **изолированный узел** в топологии.
    
- Связан входными и выходными потоками с другими процессорами.
    
- Работает **на стороне Consumer-приложения** (Kafka Streams runtime), а не на брокере Kafka.
    
- Может хранить промежуточное состояние в **State Store**, если операция требует этого (например, при агрегации или join).

## 📙3. Синхронизатор (Sink Processor)

> [!NOTE] **Синхронизатор (Sink Processor)** 
> это узел в топологии обработки данных, который **отвечает за отправку обработанных данных в Kafka топик.** 
> Он представляет собой конечную точку в потоке обработки данных и не имеет нижестоящих процессоров.
> 

Пример создания Sink Processor в Kafka Streams DSL:

```java
// Creating a topology
StreamsBuilder builder = new StreamsBuilder();

// Creating a Source Processor
KStream<String, Double> transactions = builder
	.stream(
		"transactions-topic", 
		Consumed.with(Serdes.String(), Serdes.Double())
	);

// Filtering transactions
KStream<String, Double> largeTransactions = transactions
	.filter((id, transaction) -> 
		transaction.getAmount() > 1000.00
	);

// Creating a Sink Processor, which sends data to the output Kafka 
topiclargeTransactions.to("large-transactions-topic");
```

## 📙3. Хранилища состояний (State Stores)

> [!NOTE] **Хранилища состояний (State Stores)**
> это **локальные хранилища Kafka Streams**, используемые потоковыми процессорами **для хранения данных**.
> 
> Это *необходимо* в сценариях, *где операции обработки требуют знания о предыдущем состоянии*, например, для операций агрегации или join. (ПРИ ОБРАБОТКЕ С СОХРАНЕНИЕМ СОСТОЯНИЯ)
> (например, при `count()`, `aggregate()`, `join`, оконных операциях).
> 
> При использовании Kafka Streams DSL, **такие хранилища состояний создаются автоматически**, когда вы вызываете операторы, требующие хранения состояния, такие как count() или aggregate(), или когда вы выполняете оконное разбиение потока.

Каждое приложение Kafka Streams может иметь одно или несколько таких хранилищ 
- **в памяти** (in-memory) 
- **на диске** (по умолчанию — RocksDB).

> ==RocksDB== — встроенная key–value база, обеспечивающая высокую скорость чтения и записи без сетевых вызовов.
>  
> ⭐⭐ПОХОЖ НА TreeMap из Java, и хранится в куче⭐⭐
> 
> Она хранит данные локально, а Kafka Streams периодически делает **чекпоинты и репликацию** этих данных ==в специальные changelog-топики==, что позволяет восстановить состояние при сбое.
