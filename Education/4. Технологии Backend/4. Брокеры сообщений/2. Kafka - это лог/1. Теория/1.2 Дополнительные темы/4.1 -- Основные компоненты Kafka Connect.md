# 📕1. Connectors
> [!NOTE] Connectors
> По сути коннекторы ==определяют логику копирования данных В/ИЗ Kafka==. 
> Классы, реализующие данную логику определены в плагинах (connector plugins). 
> 

Kafka Connect поддерживает **2 типа коннекторов**: 
- 🔶Source Connectors - из ВНЕШКИ в Kafka
- 🔶Sink Connectors - из Kafka во ВНЕШКУ

![[Pasted image 20251111114413.png]]

> ⭐⭐Существует множество уже [написанных плагинов](https://www.confluent.io/product/connectors), однако при желании можно написать свой.⭐⭐

#### Примеры Source Connector
|Коннектор|Назначение|
|---|---|
|**Debezium PostgreSQL**|CDC из PostgreSQL (чтение изменений таблиц через WAL)|
|**Debezium MySQL**|CDC из MySQL binlog|
|**JDBC Source Connector**|Периодически вычитывает строки из SQL-таблицы|
|**FileStreamSourceConnector**|Читает строки из файла и отправляет их в Kafka|
|**MongoDB Source Connector**|Читает изменения из MongoDB Change Streams|

#### Примеры Sink Connector
|Коннектор|Назначение|
|---|---|
|**JDBC Sink Connector**|Записывает сообщения из Kafka в SQL-таблицы|
|**Elasticsearch Sink Connector**|Индексирует данные из Kafka в Elasticsearch|
|**S3 Sink Connector**|Записывает сообщения в Amazon S3 в виде файлов|
|**FileStreamSinkConnector**|Записывает сообщения из Kafka в файл|
|**HTTP Sink Connector**|Отправляет данные POST-запросами на REST API|

# 📕 2. Tasks
> [!NOTE] Tasks - это _единица работы_ внутри Kafka Connect
> Можно сказать, что **коннектор = логика + набор задач (Tasks)**.
> - Коннектор (`SourceConnector` или `SinkConnector`) — это **“мозг”**, который знает, _как_ подключаться к внешней системе (например, к PostgreSQL, Elasticsearch и т.п.).
> - А **Tasks** — это **“рабочие руки”**, которые реально выполняют копирование данных.

Каждый инстанс коннектора координирует набор Tasks, которые занимаются копированием данных. 

## Состояние в топиках
> Tasks сами **не хранят состояние** (например, “до какого момента я уже вычитал данные”). 
 
Чтобы при сбое всё восстановилось, ==Kafka Connect хранит это состояние== в **служебных топиках Kafka**:
- `config.storage.topic` - хранит конфигурации всех коннекторов и tasks
	
- `status.storage.topic` - хранит статусы (RUNNING, FAILED, PAUSED)
	
- `offset.storage.topic` - хранит, до какого offset’а task обработала данные

## Ребалансировка
Ребалансировка — это процесс **перераспределения задач (`Tasks`) между воркерами** в кластере Connect.

Она происходит в трёх случаях:
### 1️⃣При первом запуске коннектора
Когда ты впервые создаёшь коннектор (через REST API), кластер Connect:
- создаёт нужное число Tasks,
- равномерно распределяет их по доступным Worker-ам.
```bash
3 Worker-а, 6 Tasks

→ по 2 Task на каждый Worker
```

### 2️⃣ При изменении конфигурации коннектора
Если ты меняешь параметр, например:
```json
"tasks.max": 8
```
Kafka Connect поймёт, что теперь нужно больше Tasks 
	→ запускает **ребалансировку**, чтобы распределить 8 задач по Worker-ам.

### 3️⃣ При падении или добавлении Worker-а
Если один Worker падает — его Tasks **автоматически перераспределяются** между оставшимися.  
А если ты добавляешь новый Worker в кластер — часть задач **передаётся ему** для балансировки нагрузки.

> 💡Однако **при падении самой Task**, ребалансировка н**е происходит**,bтак как это считается исключительным случаем, поэтому перезапуск Task должен осуществляться через [REST API](https://docs.confluent.io/platform/current/connect/monitoring.html).

# 📕3. Workers
Коннекторы и задачи - логические элементы, которые исполняются в ==процессах-воркерах==.

- **Worker** — это JVM-процесс, в котором могут работать **несколько Tasks** (в разных потоках).
    
- **Task** — это логическая единица работы (чтение/запись данных), но **без состояния**.
    
- **Connector** — координирует свои Tasks, решает, как их создавать и сколько их должно быть.

# 📕4.Converters

> [!NOTE] Конвертеры 
> обязательный элемент системы, который необходим для того, **чтобы ДАННЫЕ были корректно СЕРИАЛИЗОВАНЫ / ДЕСЕРИАЛИЗОВАНЫ _при копировании в / из Kafka_**. 

По умолчанию доступны следующие конвертеры:
- `org.apache.kafka.connect.json.JsonConverter`  
- `org.apache.kafka.connect.storage.StringConverter`  
- `org.apache.kafka.connect.converters.ByteArrayConverter`  
- `org.apache.kafka.connect.converters.DoubleConverter`  
- `org.apache.kafka.connect.converters.FloatConverter`  
- `org.apache.kafka.connect.converters.IntegerConverter`  
- `org.apache.kafka.connect.converters.LongConverter`  
- `org.apache.kafka.connect.converters.ShortConverter`

Также можно настроить и другие конвертеры:
- `io.confluent.connect.avro.AvroConverter`
- `io.confluent.connect.protobuf.ProtobufConverter`
- `io.confluent.connect.json.JsonSchemaConverter`

# 📕5. Transformers 

> [!NOTE] Transformers 
> Single Message Transformation (SMT) - **модуль преобразования**, который применяется **к каждому сообщению** (record), проходящему через Kafka Connect.
> 
> SMT — это **“фильтры” или “промежуточные шаги”**, встроенные между источником данных и Kafka, либо между Kafka и системой-приёмником.
> 
> 💡Это опциональный элемент💡 в пайплайне Kafka Connect.

Например, можно изменить тип данных, добавить поле к записи или удалить поле, изменить название топика, в который отправляется сообщение. 

## Где именно применяются SMT?
Позиция SMT **зависит от типа коннектора**:

| Тип коннектора       | Место применения SMT                     |
| -------------------- | ---------------------------------------- |
| **Source Connector** | после конвертера (перед записью в Kafka) |
| **Sink Connector**   | до конвертера (после чтения из Kafka)    |
