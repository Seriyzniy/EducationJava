# 📕1. Копартицирование (Co-partitioning)

> [!NOTE] **Копартицирование** 
> это требование, чтобы **все входные топики**, участвующие **в операции join** или aggregation, **имели одинаковое число партиций и одинаковую стратегию разбиения** по ключу (partitioner.
> 

## 🎯 Зачем это нужно

Kafka Streams использует **ключ (`record key`)** как основу для определения, какая партиция будет обрабатывать конкретное событие.  

Когда ты делаешь join между потоками (например, `KStream.join(KTable)` или `KStream.join(KStream)`),  
Kafka Streams должен быть уверен, что **все записи с одинаковыми ключами попадают в одну и ту же партицию**.

Если это не выполняется — join не сработает корректно: нужные данные окажутся в разных потоках (на разных инстансах).

## ✅Пример - все хорошо
💡Сценарий
У нас есть два потока:
- `orders` (заказы)
- `payments` (платежи)

Оба топика **имеют одинаковое количество партиций (3)** и используют **одинаковое правило разбиения по ключу (`orderId`)**.
```
          ┌──────────────────────────────┐
          │     Kafka cluster            │
          │                              │
          │   Topic "orders" (3 партиции)│
          │   Topic "payments" (3 партиции)
          └──────────────────────────────┘

               ↓ копартиционированы ↓

 KStream orders  → [P0]─┐
                        │ join по orderId
 KStream payments → [P0]┘
 
 
 KStream orders  → [P1]─┐
                        │ join по orderId
 KStream payments → [P1]┘
 
 
 KStream orders  → [P2]─┐
                        │ join по orderId
 KStream payments → [P2]┘
```

✅ Здесь **каждая партиция** ==обрабатывает только те заказы== и платежи, ==у которых одинаковый ключ==.

✅ Kafka Streams знает, что join можно выполнить **локально**, без обмена между нодами.  

⭐**Быстро, дешево и правильно.**

```java
KStream<String, Order> orders = builder.stream("orders");
KTable<String, Customer> customers = builder.table("customers");

KStream<String, EnrichedOrder> enriched =
    orders.join(
	    customers, 
	    (order, customer) -> new EnrichedOrder(order, customer)
	);
```

Чтобы этот join сработал корректно:
- **orders** и **customers** должны быть **копартиционированы**,  
    то есть:
    - иметь одинаковое число партиций (например, 3),
        
    - использовать одинаковый `partitioner` (обычно `DefaultPartitioner` по ключу).
        

Если одно из условий нарушено — Kafka Streams выдаст ошибку:
```java
Invalid topology: Streams with different number of partitions cannot be joined
```

## ❌Пример - разное число партиций

### Проблема

Топики `orders` и `payments` имеют **разное число партиций** или **разный partitioner**.

```bash
orders (2 партиции)         payments (3 партиции)
     ┌──────┬──────┐             ┌────┬────┬────┐
     │ P0   │ P1   │             │ P0 │ P1 │ P2 │
     └──────┴──────┘             └────┴────┴────┘
```

Если ты попробуешь сделать `orders.join(payments)` —  
==Kafka Streams НЕ СМОЖЕТ гарантировать, ЧТО ЗАПИСИ с одинаковым ключом ПОПАДУТ В ОДНУ НОДУ== - это **экземпляр приложения Kafka Streams**, то есть **один запущенный процесс (JVM)**..


Streams делит всю обработку на **StreamTask**’и — по одной на партицию входного топика.

|Streams Task|Партиция `orders`|Партиция `payments`|
|---|---|---|
|Task 0|P0|P0|
|Task 1|P1|P1|
|Task 2|—|P2 ❌ (проблема)|

Для `Task 2` join невозможен → Streams не может построить корректный граф топологии.  
Поэтому — ошибка.

❌ Результат — join будет **некорректен** (❗❗❗часть данных потеряется или пойдёт не туда❗❗❗).  
📛 Streams выбросит ошибку:
```java
TopologyException: number of partitions must match for join
```

# 📕2. Репартицирование (Repartitioning)

> [!NOTE] **Репартицирование** 
> это процесс **перезаписи потока данных во временный топик**, чтобы перераспределить ключи по партициям в соответствии с новым правилом (partitioner).
> 
> ⭐Проще говоря, Kafka Streams иногда создаёт **временный внутренний топик**, чтобы “перехешировать” данные и привести их к нужному виду для join’ов, groupBy и т. д.
> 

## Когда происходит репартицирование автоматически
Kafka Streams **автоматически создаёт внутренние репартиционные топики** в нескольких случаях:

|Операция|Когда требуется репартицирование|
|---|---|
|`groupBy()`|Когда ключ изменяется (новое разбиение по ключу)|
|`join()` между потоками|Когда потоки не копартиционированы|
|`repartition()` явно|Когда ты вызываешь этот метод вручную|
|`aggregate()`|Если предварительно изменён ключ (`selectKey`, `map`)|

## Пример
```java
KStream<String, String> input = builder.stream("input");

KStream<String, String> repartitioned = input
    .selectKey((oldKey, value) -> extractNewKey(value))  // изменяем ключ
    .repartition();  // создаёт внутренний топик "_repartition"
```

🧠 После вызова `.repartition()` Kafka Streams создаст внутренний топик, например:
```java
<application-id>-<node-name>-repartition
```
В него будут записаны перераспределённые записи, а затем считаны обратно с новыми ключами.

## ⚠️ Подводные камни репартицирования
1️⃣ **Это дорого по производительности** — данные записываются и читаются заново.  
2️⃣ **Создаются временные топики** — увеличивают нагрузку на Kafka cluster.  
3️⃣ **Неправильная стратегия ключей** может вызвать **избыточные репартиции**.  
4️⃣ Лучше **планировать ключи заранее**, чтобы избежать репартиционирования.
