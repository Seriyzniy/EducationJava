# 2. Добавление serde
Как и при работе с обычным Consumer, для Kafka Streams необходимы параметры конфигурации. Их мы можем передать при помощи `StreamsConfig`
(см [[1. - Класс StreamsConfig]])

В этом классе мы указываем
- адреса для подключения к Kafka
- специальный ID приложения
- директорию для локального хранилища
- классы сериализации и десериализации, объединенные в `Serde`
	(см [[3. - Интерфейс Serde и Фабрика Serdes]])

Эти Serde будут использоваться Kafka Streams для 
- чтения данных из топиков
- записи данных в другой топик (см теорию)

## Почему стоит писать конкретные Serde под каждый топик?
Поскольку **Kafka Streams под капотом использует топики Kafka для своей работы** (вспомни changelog топик), то он наверняка **будет отправлять данные туда и получать данные из этого топика**.

Для этого ему нужны классы Serde 
(объединяет классы сериализации и десериализации в один, и вызывает нужный)

Но **если** мы передадим ему **новый Serde**, в котором будут нами написанные **классы сериализации и десериализации общие для двух типов данных** - то будет ==много проблем==

Почему?

А потому что, мы их писали специально так хитро, чтобы не писать два отдельных продюсера под каждый топик. А именно, 
- `сериализатор` - **сериализует только тело сообщения**, а заголовок мы добавляли через созданного Producer
- `десериалзитор` - **десериализовывал данные, ОСНОВЫВАЯСЬ на заголовке**, который добавил Producer

А так как **в Kafka Streams** используется ==собственный Producer==, который работает под капотом, ***у нас НЕ будет возможности добавить заголовок*** к телу сообщения! 

А значит в процессе работы, **когда Kafka Streams потребуется записать и считать данные с топика changelog** или repartition, то Kafka Streams
- ❗❗❗ воспользуется сериализатором, который НИЧЕГО не знает про заголовок
- ❗❗❗ воспользуется десериализатором, и увидит 
	>"У **сообщения нет заголовка**, значит нужно вызвать метод десериализации, который подходит по сигнатуре и НЕ ИМЕЕТ заголовков"

В результате, Kafka Streams не сможет определить правильно тип данных, который был передан изначально.


Поэтому и нужны новые классы сериализации и десериализаци ПОД каждый топик

# 4. Написание Consumer с использованием Kafka-Streams
Теперь опишем Conuser, который используя Kafka Streams будет ПОТОКОВ обрабатывать данные с двух топиков, объединяя данные в новый DTO и после отправлять в другой топик

## Краткая вводная
Ключевыми понятиями в Kafka Streams являются
- Поток данных KStream
- Таблица данных KTable, которая отображает последнее значение для каждого ключа

Весь процесс обработки данных строится вокруг `ТОПОЛОГИИ` (граф), каждая операция `map(), filter(), join()` и прочие, СОЗДАЮТ новый `ПРОЦЕССОР`(узел графа)
(см теорию по компонентам [[3.1 -- Компоненты]])

Для создания топологии используется `StreamBuilder`, а для запуска всей топологии `KafkaStreams`
(см [[3. - Класс StreamsBuilder]])

## Проецирование на задачу
Теперь попробуем применить парадигму из KStream и KTable на нашу задачу.

Что нам нужно?
У нас есть два топика
- В один поступают данные о покупателях
- В другой поступают данные о товарах


Возникает вопрос, а кого из них можно представить как поток(KStream) данных, а кого как таблицу(KTable)

> Таблица представляет собой МНОЖЕСТВО из уникальных ключей (нет дубликатов)

Очевидно, что ОДИН ПОЛЬЗОВАТЕЛЬ может купить НЕСКОЛЬКО ТОВАРОВ, но только ОДИН ТОВАР принадлежит ОДНОМУ ПОКУПАТЕЛЮ(тот кто оплачивает)

Получается, ПОЛЬЗОВАТЕЛЬ (покупатель) УНИКАЛЕН, а товар нет(может дублироваться)

И логично, **если придет один и тот же покупатель**, нам **не нужно обновлять данные** по нему

Соответственно
- Пользователя можно представить как таблицу
- Товары как поток

Теперь, как мы определились с данными, нам нужно будет объединить две структуры данных. 
Для этого есть метод `join()` с несколькими перегрузками

> [!NOTE] ❗❗❗ ПРАВИЛО ОБЪЕДИНЕНИЯ ❗❗❗
> При объединении ПОТОКИ ДОЛЖНЫ БЫТЬ ОДИНАКОВО ТИПИЗИРОВАНЫ
> (см [[1.1 -- Ветвление потоков]])

Так как у нас данные наследуются от Data, то все нормально, базовый тип у них общий

## Описание класса
Теперь опишем класс Consumer, который использует Kafka Streams по описанной выше логике
```java
package consumer;  
  
import model.DataProduct;  
import model.DataUser;  
import model.EnrichedData;  
import org.apache.kafka.clients.consumer.ConsumerConfig;  
import org.apache.kafka.common.serialization.Serdes;  
import org.apache.kafka.streams.KafkaStreams;  
import org.apache.kafka.streams.StreamsBuilder;  
import org.apache.kafka.streams.StreamsConfig;  
import org.apache.kafka.streams.kstream.*;  
import util.kafkastreams.DataProductSerde;  
import util.kafkastreams.DataUserSerde;  
import util.kafkastreams.EnrichedDataSerde;  
  
import java.io.FileNotFoundException;  
import java.io.FileReader;  
import java.io.IOException;  
import java.util.Properties;  
  
public class CustomKafkaStreamsConsumer {  
    private final Properties props = new Properties();  
  
	// Метод выполнения Kafka Streams
    public static void main(String[] args) {  
        try(FileReader fileReader = 
	        new FileReader("src/main/resources/kafka-config.properties")
	    ){  
            Properties props = new Properties();  
            props.load(fileReader);  
  
            CustomKafkaStreamsConsumer kafkaStreamsConsumer = 
	            new CustomKafkaStreamsConsumer(
		            props.getProperty("kafka.bootstrap-servers"));  
  
            kafkaStreamsConsumer.consumeInStream(  
                    props.getProperty("kafka.topic-users"),  
                    props.getProperty("kafka.topic-products"),  
                    props.getProperty("kafka.topic-result"));  
  
        } catch (FileNotFoundException e) {  
            System.out.println("File not found in consumer");  
            throw new RuntimeException(e);  
        } catch (IOException e) {  
            throw new RuntimeException(e);  
        }  
    }  
  
    public CustomKafkaStreamsConsumer(String bootstrapServers) {  
        props.put(  
            StreamsConfig.APPLICATION_ID_CONFIG,  
            "application.id"  
        );  
        props.put(  
            StreamsConfig.BOOTSTRAP_SERVERS_CONFIG,  
            bootstrapServers  
        );  
        props.put(  
            StreamsConfig.STATE_DIR_CONFIG,  
            "local-streams-directory"  
        );  
        props.put(  
            StreamsConfig.consumerPrefix(ConsumerConfig.AUTO_OFFSET_RESET_CONFIG),  
            "earliest"  
        );  
    }  
  
    public void consumeInStream(
	    String userTopic, 
	    String productTopic, 
	    String resultTopic
	){  
		//1️⃣ Создание топологии
        StreamsBuilder builder = new StreamsBuilder();  
	  
		//2️⃣ Получение ТАБЛИЦЫ из топика пользователей
        KTable<String, DataUser> userKTable = builder.table(  
                userTopic,  
                Consumed.with(Serdes.String(), new DataUserSerde()));  
  
		//3️⃣ Получение потока дыннх из топика товаров
        KStream<String, DataProduct> productKStream = builder.stream(  
                productTopic,  
                Consumed.with(Serdes.String(), new DataProductSerde()));  
  
		//4️⃣ Для объединения двух потоков, 
		// НЕОБХОДИМО чтобы у потоков данных совпадали КЛЮЧИ
		// иначе Kafka Streams НЕ ЗНАЕТ как объединять
		//
		// Для этого меняй ключ у потока товаров на ключ пользователя
        KStream<String, DataProduct> productKStreamByUserId = 
	        productKStream  
				.selectKey(
					(key, value) -> 
					String.valueOf((value.getUserId())));  
  
        // Выводим для отладки  
        userKTable.toStream().peek((key, value) ->  
                System.out.println(
	                "KTable User - Key: " + key + ", Value: " + value)  
        );  
  
        // Выводим для отладки  
        productKStreamByUserId.peek((key, value) ->  
                System.out.println(
	                "Product stream - Key: " + key + ", Value: " + value) 
        );  
  
		// 5️⃣ Объединяем потоки
        KStream<String, EnrichedData> enrichedStream = 
	        productKStreamByUserId  
                .leftJoin(  
                    userKTable,  
                    (product, user) -> {  
                        // Создаем обогащенный объект  
                        EnrichedData enriched = new EnrichedData();  
						
                        if(user != null){  
                            enriched.setUserName(user.getUserName());  
                            enriched.setCountry(user.getCountry());  
                        } else {  
                            enriched.setUserName("UNKNOW or NOT READY");  
                            enriched.setCountry("UNKNOW or NOT READY");  
                        }  
                        
                        enriched.setId(
	                        product.getPurchaseId()); 
	                         
                        enriched.setProductName(
	                        product.getProductName());  
	                        
                        enriched.setAmount(
	                        product.getAmount());  
  
                        return enriched;  
                    },  
                    //При объединении используется changelog топик, 
                    // туда читаются и записываются данные
                    // (дублируются из RockDB)
                    Joined.with(
	                    Serdes.String(), 
	                    new DataProductSerde(), 
	                    new DataUserSerde())  
            );  
  
        // Выводим результат для отладки  
        enrichedStream.peek((key, value) ->  
			System.out.println(
				"Enriched data - Key: " + key + ", Value: " + value)  
        );  
  
		// Отправка в топик
        enrichedStream.to(
	        resultTopic, 
	        Produced.with(Serdes.String(), new EnrichedDataSerde()));  
  
  
		// Запуск топологии Kafka Streams
        KafkaStreams streams = new KafkaStreams(builder.build(), props);  
        streams.start();  
  
        Runtime.getRuntime().addShutdownHook(new Thread(streams::close));  
    }  
}
```

### Пояснение
После того, как мы получили таблицу пользователей и поток товаров, необходимо их объединить, и в процессе объединения создать новый результирующий DTO

> При создании таблицы/потока указываем классы Serde. Их Kafka Stream будет использовать для чтения/записи в топики.
> 
> При этом, **ВЕЗДЕ в топологии при работе с этими объектами, НУЖНО использовать ТОЛЬКО ЭТИ Serde**. Иначе ошибка компиляции
> 
> 💡 Можно указать в конфигурации 
> - `DEFAULT_KEY_SERDE_CLASS_CONFIG` и
> - `DEFAULT_VALUE_SERDE_CLASS_CONFIG`
> Но тогда данные из всех топиков должны быть одинакового типа

Чтобы объединить два потока, нам необходимо чтобы они совпадали по ключу 
> Сам подумай, как бы можно было объединить два потока данных, не заглядывая во внутрь этих данных, так как там могут быть неочевидные связанные ключи или вообще не быть

==В качестве ключа используется ключ сообщения передаваемый в Kafka==

Далее вызываем метод join (или leftJoin по аналогии с SQL), указываем с какой таблицей объединяем, и внутри создаем новый объект

После чего вызываем метод `to()` на новом потоке, и используем Serde для сериализации данных в топик
### Предыдущая проблема
При разработке, возникла **проблема из-за использования заголовков** при десериализации данных

Данные считывались и десериализовывались нормально, пока не добавил метод join

Почему?

Потому что загрузка данных в KTable и KStream выполняется АСИНХРОННО, и поток выполнения кода, доходил до выполнения метода `join`

А тот в свою очередь должен как-то объединить данные. 
Чтобы это сделать, он дожидается первых данных из KStream и KTable, и для KTable сохраняет записи в RockDB 

> Так как **KTable хранит в себе только последнее значение для каждого ключа**, то новые данные "затирают" старые.
> 
> И **чтобы не потерять актуальные данные**, Kafka Streams сохраняет эти актуальные данные в RockDB, А ТАЖКЕ **отправляет их в** `changelog`, чтобы при падении, просто прочитать оттуда актуальные данные

А так как ранее использовались де/сереализаторы, которые работали с заголовками
- сериализатор ничего не знал о заголовке, и преобразовывал только тело сообщения
- десериализатор использовал заголовок, переданный Producer'ом вместе с сообщением для получения сообщения
... то Kafka Streams использовал этот серализатор для записи сообщения `DataUser` в `changelog`, при этом игнорирую заголовок (у него Producer ничего не знает про заголовок), а при десериализации Kafka Stream не мог получить тело сообщения, так как вызывал метод десериалиазатора, который не использовал заголовки (в нашем случае возвращал null)

#### Данные с KTable появляются позже
1. **KStream начинает обработку СРАЗУ**
    - Данные уже в топике продуктов
    - Обработка начинается немедленно
        
    - `peek()` выполняется для каждого сообщения
        
2. **KTable требует инициализации State Store**
    - Должен *загрузить данные из changelog* топика
    - Должен *восстановить состояние*
        
    - Только ПОСЛЕ этого начинает обрабатывать `peek()`
##### 1. **State Store инициализация** (для JOIN)
```java
// ПЕРВИЧНОЕ считывание для State Store
KTable<String, Data> userKTable = builder.table("users-topic");

// Под капотом:
// 1. Читает из users-topic → заполняет RocksDB State Store
// 2. State Store сразу готов для JOIN операций
// 3. Changelog используется только для бэкапа
```

##### 2. **KTable.toStream()** - отдельный поток данных
```java
// ВТОРИЧНАЯ обработка для peek()
userKTable.toStream().peek((key, value) -> 
    System.out.println("KTable User - Key: " + key + ", Value: " + value)
);
```
