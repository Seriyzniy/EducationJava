> [!NOTE] Интерфейс `Admin`
> Это **высокоуровневый API для управления и получения метаданных Kafka-кластера**.  
> 
> Он позволяет программно выполнять административные операции, такие как:
> - создание и удаление топиков,
> - изменение конфигураций,
> - управление ACL, 
> - получение информации о брокерах, группах, и т. д

**Реализуется классом:**  
`org.apache.kafka.clients.admin.KafkaAdminClient` — основная реализация интерфейса `Admin`.

**Наследует**
```java
java.io.Closeable
java.lang.AutoCloseable
```
→ Это значит, что экземпляр `Admin` должен закрываться (обычно через try-with-resources).

## Интерфейс
```java
public interface Admin extends AutoCloseable
```

### ⭐Статические фабричные методы⭐
```java
// Создает новый AdminClient с указанными свойствами
static Admin create(Properties props);

// То же самое, но с Map-конфигурацией
static Admin create(Map<String, Object> conf);

// Возвращает AdminClientBuilder (Kafka 3.8+)
static AdminClientBuilder builder();
```

Пример:
```java
Properties props = new Properties();

props.put(
	AdminClientConfig.BOOTSTRAP_SERVERS_CONFIG,
	"localhost:9092"
);

try (Admin admin = Admin.create(props)) {
    // использование admin API
}
```

## 💡 **Особенности и подводные камни 💡**

1️⃣ **Все ❗❗методы асинхронны**❗❗  
Возвращают ⭐объекты `...Result`, которые содержат⭐ 
⭐`KafkaFuture`⭐
Для получения результата нужно вызвать `get()` (блокирует поток).

2️⃣ ❗❗**AdminClient использует сетевые ресурсы.**❗❗  
Нужно **всегда вызывать `close()`** после использования, иначе останутся открытые соединения к брокерам.

3️⃣ **Многие операции требуют привилегий.**  
Без нужных прав (ACL) — получите `ClusterAuthorizationException`.

4️⃣ 💡**Методы не гарантируют мгновенного эффекта.**💡  
Например, `createTopics()` может вернуть успех до того, как топик реально будет доступен для продюсеров.

5️⃣ **Версионная совместимость.**  
Некоторые методы могут быть недоступны в старых версиях брокера (например, `incrementalAlterConfigs()`).

6️⃣ **Thread-safe, но не предназначен для частого создания.**
💡Экземпляр `Admin` желательно создавать один на приложение.💡

---
## 💎 **Best Practices 💎**

✅ 💡Используйте `try (Admin admin = Admin.create(props)) { ... }` для безопасного закрытия.  

✅ 💡Создавайте один экземпляр `Admin` на приложение💡(Singleton).  

✅ Обрабатывайте `KafkaFuture` асинхронно или с тайм-аутом.  

✅ 💡Проверяйте совместимость версий клиента и брокера. 

✅ После создания или удаления топика — делайте `describeTopics()` для валидации.  

✅ Используйте `incrementalAlterConfigs` вместо `alterConfigs` (новый, безопасный API).

## Пример
```java
import org.apache.kafka.clients.admin.*;
import java.util.*;

public class AdminCreateExample {
    public static void main(String[] args) throws Exception {
        // Конфигурация клиента
        Properties props = new Properties();
        props.put(
	        AdminClientConfig.BOOTSTRAP_SERVERS_CONFIG,
	        localhost:9092");
        
        props.put(
	        AdminClientConfig.CLIENT_ID_CONFIG, 
	        "admin-demo");

        // Создание экземпляра Admin через статический метод
        try (Admin admin = Admin.create(props)) {
            // Получить информацию о кластере
            DescribeClusterResult clusterInfo = 
	            admin.describeCluster();
            
            System.out.println(
	            "Cluster ID: " 
	            + clusterInfo.clusterId().get());
            System.out.println(
	            "Controller: " + 
	            clusterInfo.controller().get().id());

            // Создать новый топик
            NewTopic topic = new NewTopic(
	            "demo-topic", 
	            3, 
	            (short)1
	        );
            admin.createTopics(List.of(topic)).all().get();
            System.out.println("Топик создан!");
        }
    }
}
```
