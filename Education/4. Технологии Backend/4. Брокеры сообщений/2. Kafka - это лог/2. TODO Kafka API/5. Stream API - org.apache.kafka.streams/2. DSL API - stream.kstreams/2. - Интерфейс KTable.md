
> [!NOTE] `KTable<K, V>` 
> это ==абстракция табличного представления данных== (table abstraction) в Kafka Streams. 
> 
> ❗❗Каждый ключ (`K`) в `KTable` представляет **текущее состояние** (последнее значение) этого ключа. 
> 
> Можно думать о `KTable` как о **материализованной проекции потока (`KStream`)**, где каждое новое событие с тем же ключом **перезаписывает** предыдущее значение.
> 

🧠 Пример:  
Если поток содержит записи:
```bash
key=A, value=1
key=A, value=2
key=B, value=10
```

то в `KTable` останется:
```bash
A → 2
B → 10
```

> [!NOTE] `KTable` реализуется как лог (changelog topic) + состояние (state store).
> 

---
## **Фабричные методы** 
```java
// Из Kafka-топика
KTable<K, V> table(String topic, Consumed<K, V> consumed);

// Из существующего потока KStream
KTable<K, V> table = stream.toTable();

// Из агрегирующих операций
KTable<K, Agg> aggregated = stream
    .groupByKey()
    .aggregate(
        () -> 0L,
        (key, value, agg) -> agg + value,
        Materialized.as("aggregate-store")
    );
```

## Методы

### Категория: Преобразования (Transformations)
```java
// Преобразует каждое значение (value) в новое значение
// Не изменяет ключ
KTable<K, VR> mapValues(
	ValueMapperWithKey<
		? super K, 
		? super V, 
		? extends VR> mapper);

// Преобразует таблицу в поток (каждое обновление становится событием)
KStream<K, V> toStream();

// Преобразует таблицу в поток с пользовательским преобразованием записи
KStream<KR, VR> toStream(
	KeyValueMapper<
		? super K, 
		? super V, 
		? extends KeyValue<? extends KR, ? extends VR>> mapper);
```

### Категория: Группировка и агрегация
```java
// Группирует по ключу (например, для дальнейшего count/aggregate)
KGroupedTable<K, V> groupBy(
	KeyValueMapper<
		? super K, 
		? super V, 
		KeyValue<K, V>> selector);

// Группирует по новому ключу
KGroupedTable<KR, V> groupBy(
    KeyValueMapper<
	    ? super K, 
	    ? super V, 
	    KeyValue<KR, V>> selector,
    Grouped<KR, V> grouped
);
```

🗒 Обычно используется для агрегации:
```java
KTable<String, Long> counts = table
    .groupBy((key, value) -> KeyValue.pair(value, value))
    .count();
```

### Категория: Объединение (Joins)

```java
// Inner Join
KTable<K, VR> join(
    KTable<K, VO> other,
    ValueJoiner<? super V, ? super VO, ? extends VR> joiner
);

// Left Join
KTable<K, VR> leftJoin(
    KTable<K, VO> other,
    ValueJoiner<? super V, ? super VO, ? extends VR> joiner
);

// Outer Join
KTable<K, VR> outerJoin(
    KTable<K, VO> other,
    ValueJoiner<? super V, ? super VO, ? extends VR> joiner
);
```

💬Эти join'ы **по ключу**, и объединяют текущее состояние обеих таблиц.  
Обновления автоматически пересчитываются при изменении входных данных.

### Категория: Фильтрация
```java
// Оставляет только записи, удовлетворяющие предикату
KTable<K, V> filter(Predicate<? super K, ? super V> predicate);

// Удаляет записи, удовлетворяющие предикату
KTable<K, V> filterNot(
	Predicate<? super K, ? super V> predicate);
```

### Категория: Вывод (Sinks)
```java
// Записывает содержимое таблицы в Kafka-топик
// Каждое изменение таблицы (insert/update/delete) будет отправлено
void toStream().to(String topic, Produced<K, V> produced);

// Сериализация значений с использованием Serdes по умолчанию
void toStream().to(String topic);
```

### Категория: Взаимодействие с состоянием
```java
// Возвращает имя materialized state store, если таблица материализована
String queryableStoreName();
```

## 💡 Особенности и подводные камни 💡
1️⃣ **KTable ≠ просто коллекция**  
— ==Это стрим обновлений состояния (changelog)==, а не snapshot.  
Каждый **новый ключ перезаписывает старое значение**.

2️⃣ **KTable и KStream различаются семантикой**
- `KStream` = поток событий ("каждое событие — факт").
    
- `KTable` = поток изменений состояния ("каждое событие — апдейт состояния").

3️⃣ **Joins работают по последнему известному значению**  
— Join двух `KTable` использует **актуальное состояние** обоих источников.

4️⃣ **Агрегации материализуются**  
— Любая операция `count()`, `aggregate()` или `reduce()` создаёт state store и changelog topic.

5️⃣ **toStream() может “взорвать” поток**  
— Если таблица часто обновляется, `toStream()` генерирует **много событий** (каждое обновление → новое сообщение).

---
## 💎 Best Practices 💎

✅ Используй `KTable` для **дедупликации** и **stateful join’ов** (например, user profile join). 

✅ Материализуй таблицы (`Materialized.as("store")`), если планируется интерактивный доступ (`Interactive Queries`). 

✅ Не забывай про **changelog topics** — они нужны для восстановления состояния при сбое.  

✅ Если данные изменяются редко, а обрабатываются часто — `KTable` даст огромный выигрыш по производительности. 

✅ Если поток событий только добавляется (append-only) — лучше использовать `KStream`.

## Пример
```java
StreamsBuilder builder = new StreamsBuilder();

// Читаем поток заказов
KStream<String, Order> orders = builder.stream("orders");

// Преобразуем поток заказов в таблицу (где ключ — ID заказа)
KTable<String, Order> ordersTable = orders.toTable();


// Читаем таблицу клиентов
KTable<String, Customer> customers = builder.table("customers");


// Объединяем заказы с информацией о клиентах
KTable<String, EnrichedOrder> enriched = ordersTable.join(
    customers,
    (order, customer) -> new EnrichedOrder(order.id(), customer.name(), order.total())
);

// Преобразуем таблицу обратно в поток для публикации
enriched
    .toStream()
    .peek((key, value) -> System.out.println("Отправляем заказ: " + value))
    .to("enriched-orders", Produced.with(Serdes.String(), Serdes.serdeFrom(EnrichedOrder.class)));

```
