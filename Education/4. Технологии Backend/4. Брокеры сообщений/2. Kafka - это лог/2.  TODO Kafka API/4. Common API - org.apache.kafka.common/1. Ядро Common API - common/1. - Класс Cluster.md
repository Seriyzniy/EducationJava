
> [!NOTE] Класс `Cluster` — **неизменяемое представление** части или всего **Kafka-кластера**. 
> 
> Он ==используется клиентами Kafka для получения метаданных== (например, чтобы продюсер понял, на какой брокер отправить сообщение, или чтобы консьюмер знал, у кого забирать данные).
> 

## Конструктор - ОБЫЧНО СОЗДАЕТСЯ ВНУТРИ автоматически
```java
// Создаёт новый экземпляр Cluster.
// Параметры:
// clusterId — идентификатор кластера, может быть null
// nodes — список всех узлов (брокеров) кластера
// partitions — список информации о партициях (PartitionInfo)
// unauthorizedTopics — топики, к которым клиент не имеет доступа
// invalidTopics — некорректные (ошибочные) топики
// internalTopics — внутренние служебные топики Kafka
// controller — контроллер кластера (может быть null)
public Cluster(String clusterId,
               Collection<Node> nodes,
               Collection<PartitionInfo> partitions,
               Set<String> unauthorizedTopics,
               Set<String> invalidTopics,
               Set<String> internalTopics,
               Node controller) {
    ...
}
```

### Фабричные методы
```java
// Создаёт пустой кластер (без узлов и партиций)
public static Cluster empty();

// Создаёт кластер с заданным списком узлов
public static Cluster bootstrap(List<Node> nodes);

// Создаёт кластер с заданным идентификатором и списком узлов
public static Cluster cluster(
	String clusterId, 
	Collection<Node> nodes,
	Collection<PartitionInfo> partitions,
    Set<String> unauthorizedTopics,
    Set<String> invalidTopics,
    Set<String> internalTopics);
```

## Методы
### Метаданные о кластере
```java
// Возвращает уникальный идентификатор кластера
public String clusterId();

// Возвращает контроллер кластера (или null, если неизвестен)
public Node controller();

// Возвращает список всех узлов (брокеров) в кластере
public List<Node> nodes();

// Возвращает true, если кластер пуст (нет брокеров)
public boolean isBootstrapConfigured();

// Возвращает список всех топиков в кластере
public Set<String> topics();
```

### Метаданные о партициях
```java
// Возвращает список всех партиций для указанного топика
public List<PartitionInfo> partitionsForTopic(String topic);

// Возвращает все партиции во всём кластере
public List<PartitionInfo> partitions();

// Возвращает список всех партиций, у которых есть лидер
public List<PartitionInfo> availablePartitionsForTopic(
	String topic);

// Возвращает объект PartitionInfo для конкретной TopicPartition
public PartitionInfo partition(TopicPartition tp);
```

### Работа с узлами и лидерами
```java
// Возвращает узел-лидер для указанной партиции
public Node leaderFor(TopicPartition tp);

// Возвращает узел по его ID (или null, если не найден)
public Node nodeById(int id);
```

### Недоступные и ошибочные топики
```java
// Возвращает набор топиков, к которым клиент не имеет доступа
public Set<String> unauthorizedTopics();

// Возвращает набор некорректных топиков (ошибочных)
public Set<String> invalidTopics();

// Возвращает набор внутренних (служебных) топиков Kafka
public Set<String> internalTopics();
```

# 📕1. Как `Cluster` связан с реальной Kafka ?

`Cluster` — это **объект-представление (snapshot) метаданных** о реальном Kafka-кластере.

Он **создаётся внутри клиента** (`KafkaProducer`, `KafkaConsumer`, `AdminClient`) на основе **ответов от брокеров** на запросы `MetadataRequest`.

## Процесс построение Cluster
### 1️⃣ Конфигурация клиента
Когда ты создаёшь `KafkaProducer` или `KafkaConsumer`, ты указываешь:
```java
props.put("bootstrap.servers", "broker1:9092,broker2:9092");
```
Это — **точка входа**.  
Эти адреса нужны, чтобы клиент мог впервые “достучаться” до кластера.

### 2️⃣ Клиент делает запрос `MetadataRequest`
Сразу после запуска, продюсер или консьюмер **отправляет сетевой запрос** `MetadataRequest` одному из брокеров из `bootstrap.servers`.

Он спрашивает:
> “Какие топики существуют?  
> Какие у них партиции?  
> Кто лидер каждой партиции?  
> Кто контроллер кластера?”

### 3️⃣ Брокер возвращает `MetadataResponse`
Брокер отвечает структурой `MetadataResponse`, которая содержит всю информацию:
- ID кластера (`clusterId`)
    
- список всех брокеров (`Node`)
    
- партиции (`PartitionInfo`)
    
- лидеры, реплики, ISR и т.д.

### 4️⃣ На основе ответа создаётся объект `Cluster`
Клиент **преобразует этот ответ в объект `Cluster`**:
```java
Cluster cluster = new Cluster(
    clusterId,
    nodes,
    partitionInfos,
    unauthorizedTopics,
    invalidTopics,
    internalTopics,
    controllerNode
);
```
> Этот `Cluster` живёт **только в памяти клиента**, и обновляется при изменении метаданных (например, при реабалансировке, изменении лидеров и т.п.).

### 5️⃣ Использование `Cluster`
Теперь продюсер/консьюмер **использует `Cluster`**, чтобы:
- определить, **на какой брокер** отправлять сообщение (по ID лидера партиции);
    
- выбрать правильную партицию (`partitionForTopic`);
    
- обнаружить новые топики или изменения структуры.
    

### 6️⃣ Периодическое обновление
Клиенты Kafka **регулярно обновляют метаданные** — то есть, пересоздают объект `Cluster` на основе новых данных от брокеров.

Например:
- Продюсер видит `NOT_LEADER_FOR_PARTITION` → обновляет метаданные.
    
- Консьюмер делает ребаланс → обновляет `Cluster`.
    
- AdminClient вызывает `describeCluster()` → получает актуальный `Cluster`.
