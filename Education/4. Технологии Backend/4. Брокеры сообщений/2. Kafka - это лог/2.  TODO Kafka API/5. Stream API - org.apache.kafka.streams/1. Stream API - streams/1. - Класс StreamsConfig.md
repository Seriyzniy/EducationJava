
> [!NOTE] `StreamsConfig` 
> это **конфигурационный класс**, который ==инкапсулирует все параметры для приложения Kafka Streams==: 
> - настройки продюсеров, консьюмеров, состояния, топологии, сериализаторов и т. д.  
> 
> При создании Kafka Streams экземпляр этого класса передаётся в конструктор `KafkaStreams`.
> 

Не реализует интерфейсы напрямую, но наследуется от `AbstractConfig`, что даёт ему все возможности по управлению и валидации конфигурационных параметров.

---
## 📙1.Конструктор
```java
// Создаёт объект StreamsConfig с указанными настройками
public StreamsConfig(Map<?, ?> props)
```

**Пояснение:**  
Конструктор принимает `Map` с параметрами конфигурации.  
Это может быть обычный `Properties`, `HashMap` и т. п.

Все ключи — это строки с именами параметров (например `"application.id"`, `"bootstrap.servers"`),  
а значения — объекты (чаще всего строки или числа).

---
## ⭐2.Константы класса `StreamsConfig`
**Назначение:** каждая константа соответствует имени конфигурационного параметра,  
который можно передать через `Properties` или `Map` при создании `KafkaStreams`.

---
### Базовые параметры приложения
```java
public static final String APPLICATION_ID_CONFIG = "application.id";
```
**Пояснение:**  
Уникальный ==идентификатор стрим-приложения==.  
Используется Kafka для разделения состояния и consumer group'ов между приложениями.

❗❗**Обязательный параметр.**  
Если два приложения имеют одинаковый `application.id`, они будут делить одно состояние и одну группу консьюмеров.

---
```java
public static final String BOOTSTRAP_SERVERS_CONFIG = "bootstrap.servers";
```
**Пояснение:**  
==Список брокеров Kafka==, к которым подключается приложение.

Формат: `"host1:9092,host2:9092"`.
❗❗**Обязательный параметр.**

---
```java
public static final String STATE_DIR_CONFIG = "state.dir";
```
**Пояснение:**  
==Каталог на локальной машине==, где Kafka Streams хранит ==состояние== (RocksDB, changelog и offset’ы).

По умолчанию: `/tmp/kafka-streams`.
❗❗❗Изменяй при работе в продакшене, чтобы сохранять состояние между перезапусками.

---
### Параметры обработки
```java
public static final String PROCESSING_GUARANTEE_CONFIG = "processing.guarantee";
```
**Пояснение:**  
Определяет ==семантику обработки сообщений==:
- `"at_least_once"` — по умолчанию, возможны дубликаты
    
- `"exactly_once_v2"` — обеспечивает строгую семантику (без дубликатов, но медленнее)

Используется для контроля над надёжностью и производительностью.

---
```java
public static final String NUM_STREAM_THREADS_CONFIG = "num.stream.threads";
```
**Пояснение:**  
==Количество потоков== (threads), обрабатывающих топологию.

💡Каждый поток = отдельный поток выполнения, обрабатывающий свою партицию.

Обычно 1–4 в зависимости от числа CPU и партиций топиков.

---
```java
public static final String DEFAULT_KEY_SERDE_CLASS_CONFIG = "default.key.serde";

public static final String DEFAULT_VALUE_SERDE_CLASS_CONFIG = "default.value.serde";
```
**Пояснение:**  
Определяют ==классы для сериализации/десериализации== ключей и значений сообщений,  
если не указано явно при создании `KStream` или `KTable`.
Например:
```JAVA
props.put(
	StreamsConfig.DEFAULT_KEY_SERDE_CLASS_CONFIG, 
	Serdes.String().getClass());
	
props.put(
	StreamsConfig.DEFAULT_VALUE_SERDE_CLASS_CONFIG, 
	Serdes.Long().getClass());
```

---
```java
public static final String DEFAULT_DESERIALIZATION_EXCEPTION_HANDLER_CLASS_CONFIG =
    "default.deserialization.exception.handler";
```
**Пояснение:**  
Определяет ==обработчик ошибок десериализации==.

💡Можно указать свой класс, реализующий интерфейс  
`DeserializationExceptionHandler`, чтобы управлять реакцией на ошибки (например, пропуск сообщения или завершение работы).

---
### Параметры состояния
```java
public static final String COMMIT_INTERVAL_MS_CONFIG = "commit.interval.ms";
```
**Пояснение:**  
==Интервал между фиксациями состояния== (state store и offsets) в миллисекундах.

По умолчанию — 30 секунд.

💡Чем меньше интервал, тем быстрее восстанавливается состояние после сбоя,  
но тем выше нагрузка на диск и брокер.

---
```java
public static final String CACHE_MAX_BYTES_BUFFERING_CONFIG = "cache.max.bytes.buffering";
```
**Пояснение:**  
Максимальный ==размер кэша (в байтах)== для буферизации записей перед коммитом.

По умолчанию 10 МБ.

Если поставить 0 — отключается кэширование, и все операции будут выполняться сразу (медленнее, но полезно для отладки).

---
### Параметры восстановления и топологии
```java
public static final String TOPOLOGY_OPTIMIZATION_CONFIG = "topology.optimization";
```
**Пояснение:**  
Управляет ==оптимизацией топологии== (соединение операций, сокращение ненужных state stores).

Возможные значения:
- `"none"` — без оптимизаций
- `"all"` — включает все доступные оптимизации

Используй `"all"` в продакшене для уменьшения overhead'а.

---
```java
public static final String UPGRADE_FROM_CONFIG = "upgrade.from";
```
**Пояснение:**  
Указывает, с какой версии Kafka Streams происходит обновление.

Применяется ==при миграции между версиями Kafka Streams==, чтобы включить режим совместимости (например, при обновлении формата состояния).

---
### Параметры продюсеров, консьюмеров и администраторов
```java
public static final String PRODUCER_PREFIX = "producer.";
public static final String CONSUMER_PREFIX = "consumer.";
public static final String ADMIN_CLIENT_PREFIX = "admin.";
```
**Пояснение:**  
Позволяют задавать параметры для внутренних клиентов Kafka Streams.

Например:
```java
props.put("producer.retries", 10);
props.put("consumer.max.poll.records", 1000);
props.put("admin.request.timeout.ms", 30000);
```

---
### 🔐 Безопасность
```java
public static final String SECURITY_PROTOCOL_CONFIG = "security.protocol";
```
**Пояснение:**  
Определяет, какой ==протокол безопасности== используется:  
`"PLAINTEXT"`, `"SSL"`, `"SASL_PLAINTEXT"`, `"SASL_SSL"`.

Применяется для подключения к защищённым брокерам Kafka.

---
### Прочие параметры
```java
public static final String APPLICATION_SERVER_CONFIG = "application.server";
```
**Пояснение:**  
Используется для **Interactive Queries** — если нужно обращаться к локальному состоянию через REST.

Значение — `"host:port"`, например `"localhost:8080"`.

Тогда другие инстансы могут искать, на каком хосте находится нужное состояние.

---
```java
public static final String METRICS_RECORDING_LEVEL_CONFIG = "metrics.recording.level";
```
**Пояснение:**  
Определяет уровень детализации метрик: `"INFO"`, `"DEBUG"`, `"TRACE"`.

Используется для мониторинга производительности Kafka Streams.

## 📙3. Методы
### Основные методы получения конфигураций
```java
// Возвращает идентификатор приложения
public String getString(String key)

// Возвращает числовое значение параметра
public int getInt(String key)

// Возвращает логическое значение параметра
public boolean getBoolean(String key)
```

### Потоковые параметры
```java
// Возвращает ID приложения (application.id)
public String applicationId()

// Возвращает список bootstrap-серверов Kafka
public String bootstrapServers()

// Возвращает директорию для состояния (state.dir)
public String stateDir()
```
💬**Пояснение:**  
Эти методы позволяют получить ключевые параметры, определяющие контекст работы приложения:
- `application.id` — уникальный идентификатор стрим-приложения
    
- `bootstrap.servers` — список брокеров Kafka
    
- `state.dir` — путь к директории для хранения локального состояния (RocksDB и changelog)

### Методы интеграции с Kafka Clients
```java
// Создаёт объект конфигурации продюсера
public Map<String, Object> getProducerConfigs(String clientId)

// Создаёт объект конфигурации консьюмера
public Map<String, Object> getConsumerConfigs(String clientId)

// Создаёт объект конфигурации администратора Kafka
public Map<String, Object> getAdminConfigs(String clientId)
```
💬**Пояснение:**  
Эти методы извлекают подмножество настроек, относящихся к клиентам Kafka (Producer, Consumer, AdminClient).  
Например, можно получить все параметры продюсера, используемые Kafka Streams внутри, или изменить их при необходимости.
