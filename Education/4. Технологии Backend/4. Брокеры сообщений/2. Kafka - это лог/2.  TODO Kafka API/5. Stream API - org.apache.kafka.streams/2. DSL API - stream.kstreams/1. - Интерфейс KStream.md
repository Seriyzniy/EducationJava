
> [!NOTE] `KStream<K, V>`
> Интерфейс, представляющий **непрерывный поток записей** (событий) вида `key → value`.  
> 
> Каждая запись в `KStream` — это **НЕЗАВИСИМОЕ СОБЫТИЕ**, а не обновление состояния, как в `KTable`.
> 
> Используется для **обработки потоков событий в реальном времени** — фильтрации, трансформаций, объединений и агрегаций.

## Фабричные методы (через StreamsBuilder)
```java
//Пояснение:
// Создаёт KStream ❗❗из одного Kafka-топика.❗❗
// Параметр topic — имя топика.
// Параметр consumed — объект, определяющий Serde для ключей/значений и параметры подписки.
<K, V> KStream<K, V> StreamsBuilder.stream(
	String topic, 
	Consumed<K, V> consumed)

//Пояснение:
// Создаёт KStream ❗❗из нескольких топиков.❗❗
// Параметр topics — список имён топиков.
// Параметр consumed — объект, определяющий Serde для ключей/значений и параметры подписки.
<K, V> KStream<K, V> StreamsBuilder.stream(
	Collection<String> topics, 
	Consumed<K, V> consumed)
```

## Методы `KStream`
### Трансформации (изменение ключей и значений)
```java
//Пояснение:
// Преобразует каждую запись в новую пару ключ-значение.
<KR, VR> KStream<KR, VR> map(
	KeyValueMapper<
		? super K, 
		? super V, 
		KeyValue<? extends KR, ? extends VR>> mapper)

//Пояснение:
// Преобразует только значение каждой записи.
<VR> KStream<K, VR> mapValues(
	ValueMapper<? super V, ? extends VR> mapper)



//Пояснение:
// Преобразует одну запись в несколько пар ключ-значение.
<KR, VR> KStream<KR, VR> flatMap(
	KeyValueMapper<
		? super K, 
		? super V, 
		Iterable<KeyValue<? extends KR, ? extends VR>>> mapper)

//Пояснение:
// Преобразует значение в несколько значений.
<VR> KStream<K, VR> flatMapValues(
	ValueMapper<
		? super V, 
		? extends Iterable<? extends VR>> processor)



//Пояснение:
// Изменяет ключ каждой записи на новый.
<KR> KStream<KR, V> selectKey(
	KeyValueMapper<
		? super K, 
		? super V, 
		? extends KR> mapper)
```

### Фильтрация
```java
//Пояснение:

//Пояснение:
// Противоположность filter() — оставляет записи, НЕ удовлетворяющие условию.
KStream<K, V> filterNot(
	Predicate<? super K, ? super V> predicate)
```

### Объединения (Joins)
```java

// Объединяет два KStream по ключу с окном времени.
<V1, VR> KStream<K, VR> join(
	KStream<K, V1> otherStream, 
	ValueJoiner<? super V, ? super V1, ? extends VR> joiner, 
	JoinWindows windows, 
	StreamJoined<K, V, V1> joined)


//Пояснение:
// Левое объединение двух потоков.
<V1, VR> KStream<K, VR> leftJoin(
	KStream<K, V1> otherStream, 
	ValueJoiner<? super V, ? super V1, ? extends VR> joiner, 
	JoinWindows windows, 
	StreamJoined<K, V, V1> joined)


//Пояснение:
// Объединяет поток и таблицу (KTable).
<VO, VR> KStream<K, VR> join(
	KTable<K, VO> table, 
	ValueJoiner<? super V, ? super VO, ? extends VR> joiner, 
	Joined<K, V, VO> joined)

//Пояснение:
// Левое объединение потока и таблицы.
<VO, VR> KStream<K, VR> leftJoin(
	KTable<K, VO> table, 
	ValueJoiner<? super V, ? super VO, ? extends VR> joiner, 
	Joined<K, V, VO> joined)
```

### Терминальные операции, вывод и побочные эффекты
```java
//Пояснение:
// Выполняет действие над каждой записью (например, логирование или запись в внешнюю систему).
void foreach(ForeachAction<? super K, ? super V> action)

// ❗❗❗Записывает поток в Kafka-топик❗❗❗
// Параметры:
// - topic — имя выходного топика
// - Produced<K, V> — параметры продюсера (серде, partitioner, и т.д.)
void to(String topic, Produced<K, V> produced);
void to(String topic, Produced<K, V> produced)

// Записывает поток в Kafka-топик с использованием стандартных настроек Serdes (по умолчанию)
void to(String topic);

//Пояснение:
// Печатает поток в stdout (для отладки).
void print(Printed<K, V> printed)
```

💬**Комментарий:**  
`to()` — это **терминальный метод**. Он завершает цепочку операций и указывает, что результаты обработки (`KStream`) нужно отправить в Kafka-топик.  
После вызова `to()` поток не может быть продолжен — это финальная точка.

## 💡 Особенности и подводные камни 💡

1️⃣ `KStream` — **immutable**. Каждая операция возвращает _новый поток_. 

2️⃣ Все операции **ленивые (lazy)** — вычисление выполняется только при запуске `KafkaStreams`.  

3️⃣ При использовании `groupBy()` → Kafka создаёт **внутренний топик для переразделения (repartition)**.  

4️⃣ ❗❗❗При join’ах обязательно нужно указать окно (`JoinWindows`), иначе не скомпилируется.  

5️⃣ `flatMap()` и `flatMapValues()` часто вызывают _взрыв данных_ — нужно следить за объёмом.  

6️⃣ Если указать неверные Serde при `Consumed.with()` или `Produced.with()`, можно получить `SerializationException`.

---
## 💎 Best Practices 💎

✅ Используй `mapValues()` вместо `map()`, если не меняешь ключ — это дешевле.  

✅ ❗❗❗Определяй `Named` в операциях (`mapValues(Named.as("normalize-values"))`) — поможет в метриках и дебаге.  

✅ Всегда задавай `Serde` явно при вводе/выводе (`Consumed.with()`, `Produced.with()`). 

✅ Для сложных join-операций — всегда добавляй окна (`JoinWindows.ofTimeDifferenceWithNoGrace(...)`).  

✅ Для тестов — используй `TopologyTestDriver` и `TestInputTopic/TestOutputTopic`.

## Пример
```java
StreamsBuilder builder = new StreamsBuilder();

KStream<String, String> source = builder.stream(
    "input-topic",
    Consumed.with(Serdes.String(), Serdes.String())
);

source
    .filter((key, value) -> value.contains("order"))
    .mapValues(value -> value.toUpperCase())
    .groupByKey()
    .count()
    .toStream()
    .to("output-topic", 
	    Produced.with(Serdes.String(), Serdes.Long())
	);

KafkaStreams streams = new KafkaStreams(
	builder.build(), 
	new StreamsConfig(props));
	
streams.start();
```
