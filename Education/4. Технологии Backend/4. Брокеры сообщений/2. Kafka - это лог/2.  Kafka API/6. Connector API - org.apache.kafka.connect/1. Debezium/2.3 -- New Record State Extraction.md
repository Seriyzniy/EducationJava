Debezium-коннекторы генерируют сообщения об изменениях данных, которые представляют каждую операцию, зафиксированную в исходной базе данных. 
**Сообщения, которые коннектор отправляет в Apache Kafka, имеют СЛОЖНУЮ СТРУКТУРУ**, точно отражающую детали исходного события в БД.

Несмотря на то, что этот сложный формат сообщения детализирует информацию об изменениях в системе, **он может быть неудобен для некоторых downstream-потребителей**. 
Sink-коннекторы или другие **компоненты Kafka-экосистемы могут требовать**, ==чтобы сообщения имели упрощённую, «плоскую» структуру ИМЕН и ЗНАЧЕНИЙ ПОЛЕЙ.==

Чтобы упростить формат событий, которые создают коннекторы Debezium, можно использовать **Single Message Transformation (SMT) по "выравниванию" событий** — ==Event Flattening== SMT. 

Настройте эту трансформацию, если потребители требуют более простой формат Kafka-сообщений. 
В зависимости от необходимости SMT ***можно применить либо к коннектору-источнику (Debezium), либо к sink-коннектору***. Чтобы Kafka сохраняла исходные события Debezium, настройте SMT на sink-коннекторе.

Трансформация "event flattening" является SMT для Kafka Connect.

> Информация в этой главе описывает SMT для SQL-коннекторов Debezium. Эквивалентная SMT для MongoDB описана в разделе _MongoDB New Document State Extraction_.

---
# **Структура события изменений**
Debezium генерирует события об изменениях с достаточно сложной структурой. Каждое событие состоит из трёх частей:

### **1. Метаданные**, которые включают:
- тип операции (CREATE/UPDATE/DELETE)
- информацию об источнике, например, имя БД и таблицы
- временную метку изменения
- опциональную информацию о транзакции

### **2. Данные строки _до_ изменения (`before`)**

### **3. Данные строки _после_ изменения (`after`)**
Пример структуры UPDATE-события:
```json
{
  "op": "u",
  "source": { ... },
  "ts_ms": "...",
  "ts_us": "...",
  "ts_ns": "...",
  "before": {
    "field1": "oldvalue1",
    "field2": "oldvalue2"
  },
  "after": {
    "field1": "newvalue1",
    "field2": "newvalue2"
  }
}
```

После обработки этим SMT сообщение превращается в:
```json
{
  "field1": "newvalue1",
  "field2": "newvalue2"
}
```

---
# **Поведение**
**Event Flattening SMT извлекает поле** `after` из события изменения Debezium **и заменяет исходное событие простым Kafka-сообщением**, содержащим только данные после изменения.

SMT можно настроить *либо на Debezium-коннекторе, либо на sink-коннекторе*. 

> Преимущество настройки на sink-коннекторе — сохранение в Kafka **полных** событий Debezium.

SMT можно настроить так, чтобы:
### ✔️ Добавлять метаданные события в упрощённое сообщение
(по умолчанию — метаданные _не добавляются_)

### ✔️ Сохранять записи об операциях DELETE
(по умолчанию — такие записи удаляются, так как большинство потребителей не умеют их обрабатывать)

==При DELETE== Debezium создаёт 2 сообщения:
1. Сообщение с `"op": "d"` и полем `before`
    
2. ==Tombstone-сообщение==
```json
key = <тот же ключ>
value = null
```


Вместо удаления этих сообщений SMT может быть настроен:
### _Вариант 1:_ оставить сообщение и заменить его значение на `"value": "null"`

### _Вариант 2:_ оставить сообщение и преобразовать `before` в прямой набор полей + флаг
```json
"__deleted": "true"
```
Также можно настроить сохранение tombstone.

# **Конфигурация**
Для включения стандартного поведения ==используй== `ExtractNewRecordState`:
```json
transforms=unwrap,
...
transforms.unwrap.type=io.debezium.transforms.ExtractNewRecordState
```

Полная пример-конфигурация:
```json
transforms=unwrap,...
transforms.unwrap.type=io.debezium.transforms.ExtractNewRecordState

transforms.unwrap.drop.tombstones=false
transforms.unwrap.delete.handling.mode=rewrite
transforms.unwrap.add.fields=table,lsn
```

### Значения опций:
- **drop.tombstones=false**  
    Не удалять tombstone-сообщения.
    
- **delete.handling.mode=rewrite**  
    Преобразовать DELETE-сообщение в плоский объект с полем `"__deleted": "true"`.
    
- **add.fields=table,lsn**  
    Добавить метаданные (`__table`, `__lsn`) в сообщение.

---
# **Добавление метаданных**
Можно добавить метаданные в:
### ✔️ заголовки сообщения (`add.headers`)

### ✔️ значение сообщения (`add.fields`)

Например:
```json
transforms.unwrap.add.fields=op,table,lsn,source.ts_ms

transforms.unwrap.add.headers=db
```

Это добавит:
```json
"__op": "c",
"__table": "MY_TABLE",
"__lsn": "123456789",
"__source_ts_ms": "123456789"
```

# **Селективное применение SMT**
**Debezium генерирует разные типы сообщений** 
- heartbeat, 
- tombstone, 
- события транзакций, 
- события схемы. 

**Их структура отличается от структуры data-change событий**, поэтому SMT лучше применять только к тем сообщениям, которые содержат изменения данных.

Для этого можно использовать **SMT predicates**.

Подробнее: _Configure an SMT predicate for the transformation_.

# Конфигурационные свойства
## **drop.tombstones**
**Описание (перевод):**  
Debezium генерирует tombstone-запись для каждой операции DELETE. По умолчанию SMT удаляет такие записи из потока. Чтобы сохранять tombstone-записи, укажите `drop.tombstones=false`.  
Опция устарела и будет удалена в будущих версиях. Её заменяет `delete.tombstone.handling.mode`.

**Значение по умолчанию:** `true`

---
## **delete.handling.mode**
**Описание (перевод):**  
Debezium создаёт запись события об изменении для DELETE. По умолчанию SMT удаляет такие записи.

Чтобы сохранять записи DELETE, установите:
- `none` — оставить запись DELETE, значение будет `"value": "null"`
    
- `rewrite` — оставить запись и преобразовать: добавить в value поля из `before` + `__deleted: true`
    

Опция устарела и заменяется на `delete.tombstone.handling.mode`.

**Значение по умолчанию:** `drop`

---
## **delete.tombstone.handling.mode**
**Описание (перевод):**  
Определяет, как SMT должен обрабатывать DELETE-события.  
Эта настройка имеет приоритет над устаревшими `drop.tombstones` и `delete.handling.mode`.

Варианты:

- **drop** — удалить событие DELETE и TOMBSTONE
    
- **tombstone (default)** — оставить TOMBSTONE (значение: `"value": "null"`)
    
- **rewrite** — оставить событие DELETE, добавить данные из `before` + `__deleted: true`, удалить TOMBSTONE
    
- **rewrite-with-tombstone** — как `rewrite`, но TOMBSTONE тоже сохраняется
    

**Значение по умолчанию:** _нет (но поведение по умолчанию — tombstone)_

---
## **route.by.field**
**Описание (перевод):**  
Позволяет направлять события в топики на основе значения поля из `after` (или `before` для DELETE).  
Например, `route.by.field=destination` отправляет запись в топик, имя которого равно `after.destination`.

Это может быть полезно в sink-коннекторах, если название топика определяет таблицу назначения.

**Значение по умолчанию:** отсутствует

---
## **add.fields.prefix**
**Описание (перевод):**  
Строка-префикс, которая будет добавлена перед именами метаданных, добавляемых в value.

**Значение по умолчанию:** `__` (двойное подчеркивание)

---
## **add.fields**
**Описание (перевод):**  
Список через запятую полей метаданных, которые нужно добавить в value упрощённого сообщения.  
Можно указывать путь к полю через struct (например: `source.ts_ms`).  
Можно переименовывать поле: `version:VERSION`.

Добавляемые метаданные будут иметь префикс `__`.

Если указанное поле отсутствует в записи — оно всё равно будет создано.

**Значение по умолчанию:** отсутствует

---
## **add.headers.prefix**
**Описание (перевод):**  
Строка-префикс, добавляемая перед именами полей метаданных, включаемых в заголовки Kafka-сообщений.

**Значение по умолчанию:** `__`

---
## **add.headers**
**Описание (перевод):**  
Список через запятую полей метаданных, которые должны быть добавлены в header сообщения.  
Поддерживает struct (`source.ts_ms`) и переименование (`connector:CONNECTOR`).  
Если поле отсутствует в событии — оно _не_ будет добавлено в заголовок.

**Значение по умолчанию:** отсутствует

---
## **drop.fields.header.name**
**Описание (перевод):**  
Имя заголовка Kafka-сообщения, в котором передаётся список полей исходного сообщения, которые необходимо удалить из результата.

**Значение по умолчанию:** отсутствует

---
## **drop.fields.from.key**
**Описание (перевод):**  
Определяет, нужно ли удалять поля, указанные в `drop.fields.header.name`, из ключа события.

**Значение по умолчанию:** `false`

---
## **drop.fields.keep.schema.compatible**
**Описание (перевод):**  
Определяет, можно ли удалять поля, помеченные как обязательные (non-optional), перечисленные в `drop.fields.header.name`.  
По умолчанию SMT удаляет только опциональные поля.

**Значение по умолчанию:** `true`
