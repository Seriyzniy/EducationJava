# 1. Базовые параметры конфигурации
## 1. Основные параметры коннектора
Эти параметры задают базовую конфигурацию самого коннектора.
### name
Уникальное имя коннектора. Повторная регистрация с тем же именем завершится ошибкой.  
**Значение по умолчанию**: Нет значения по умолчанию

### connector.class
Имя Java-класса коннектора. Для PostgreSQL всегда использовать `io.debezium.connector.postgresql.PostgresConnector`.  
**Значение по умолчанию**: Нет значения по умолчанию

### tasks.max
Максимальное количество задач. PostgreSQL коннектор всегда использует одну задачу.  
**Значение по умолчанию**: 1

### topic.prefix
Префикс топика Kafka для данного сервера или кластера PostgreSQL. Должен быть уникальным и содержать только буквы, цифры, дефисы, точки и подчёркивания.  
**Значение по умолчанию**: Нет значения по умолчанию

# 2. Advanced Debezium PostgreSQL connector configuration properties

## **Converters & Custom Converters**

### **converters**
Перечисляет через запятую символические имена пользовательских конвертеров, которые может использовать коннектор.  
Этот параметр **обязателен**, если вы хотите использовать хоть один кастомный конвертер.

Например:
```
converters=isbn
```

После добавления конвертера **нужно также указать** свойство `<name>.type` с полным именем класса, реализующего интерфейс конвертера:
```
isbn.type=io.debezium.test.IsbnConverter
```

Если нужно передать конвертеру дополнительные параметры — они также задаются с префиксом `<name>.*`:
```
isbn.schema.name=io.debezium.postgresql.type.Isbn
```

**Значение по умолчанию**: _нет значения_

### `<converter>.type`
Класс Java, реализующий интерфейс конвертера.  

Формат:
```
<symbolicName>.type
```

Например:
`isbn.type=io.debezium.test.IsbnConverter`

Для дополнительной настройки поведения можно добавлять параметры конфигурации с префиксом символического имени конвертера.

**Значение по умолчанию**: _нет значения (обязательно при использовании converter)_

---

## Snapshot — Основные настройки

### **snapshot.mode**
Указывает, выполняет ли коннектор снимок данных ==при запуске== — и если да, то как.

Доступные режимы:
#### **always**
Делает snapshot **каждый запуск**, затем начинает стриминг.

#### **initial** (по умолчанию)
Делает snapshot только если **нет сохранённых offset’ов**.

#### **initial_only**
Делает snapshot и **завершает работу**, не переходя в стриминг.

#### **no_data**
Не делает snapshot **никогда**.  
Если есть сохранённый LSN — продолжает с него.  
Если нет — начинает стрим с позиции создания слота.

#### **never**
Устарело, эквивалентно **no_data**.

#### **when_needed**
Делает snapshot, только если:

- отсутствуют offset’ы, или
    
- offset указывает на позицию WAL, которой уже нет.
    

#### **configuration_based**

Поведение snapshot полностью задаётся набором свойств `snapshot.mode.configuration.based.*`.

#### **custom**

Работа snapshot определяется пользовательским классом, заданным через `snapshot.mode.custom.name`, реализация интерфейса:

`io.debezium.spi.snapshot.Snapshotter`

**Значение по умолчанию**: `initial`

## Snapshot Locking

### **snapshot.locking.mode**
Как коннектор удерживает блокировки таблиц при snapshot.

Возможные значения:
#### **shared**

Коннектор удерживает shared lock во время чтения метаданных. После этого, при чтении данных, блокировки не требуются.

#### **none** (по умолчанию)
Полностью избегает блокировок.  
**НЕЛЬЗЯ использовать, если структура таблиц может измениться во время snapshot!**

#### **custom**
Реализация задаётся через `snapshot.locking.mode.custom.name`.

**Значение по умолчанию**: `none`

---
### **snapshot.locking.mode.custom.name**

## Event processing & Queue settings**

### **event.processing.failure.handling.mode**
Определяет, как коннектор должен реагировать на ошибки при обработке событий.

Доступные значения:

#### **fail** (по умолчанию)
Прерывает работу коннектора, выбрасывает исключение и указывает offset проблемного события.

#### **warn**
Логирует offset проблемного события, пропускает его и продолжает работу.

#### **skip**
Игнорирует ошибку полностью, пропускает событие и продолжает работу (без логов).

**Значение по умолчанию**: `fail`

---
### **max.batch.size**
Максимальное количество событий в одном батче, который обрабатывается коннектором.

Это _не_ количество сообщений в очереди — это количество, которое Debezium передаёт вниз по pipeline за один цикл.

**Значение по умолчанию**: `2048`

---
### **max.queue.size**
Максимальное число событий в очереди (blocking queue), куда Debezium пишет сообщения, прежде чем отправить их в Kafka.

Важно:

- очередь создаёт backpressure (от обратного давления)
    
- должна быть **строго > max.batch.size**
    
- если Kafka недоступна, очередь заполняется → Debezium перестаёт читать WAL
    

**Значение по умолчанию**: `8192`

---

### **max.queue.size.in.bytes**
Максимальный объём очереди в байтах.  
По умолчанию ограничения по объёму нет.

Если **оба параметра** установлены:

`max.queue.size max.queue.size.in.bytes`

— блокировка записи в очередь сработает, когда будет превышено **любой** из лимитов.

Пример:

- `max.queue.size=1000`
    
- `max.queue.size.in.bytes=5000`
    

→ очередь перестанет принимать записи либо после 1000 элементов, либо после 5000 байт — в зависимости от того, что наступит раньше.

**Значение по умолчанию**: `0` (отключено)

---

### **poll.interval.ms**

Сколько миллисекунд Debezium ждёт, прежде чем начать формировать батч событий.

Если изменений мало, коннектор будет простаивать именно это время.

**Значение по умолчанию**: `500`

## Unknown datatypes & Initial statements & Heartbeat & Status updates

### **include.unknown.datatypes**
Определяет, что делать с полями, чей тип БД Debezium не знает.

#### `false` (по умолчанию)
- поле исключается из event
- пишется предупреждение в лог

#### `true`
- поле включается в событие в виде **сырого бинарного значения**
- формат контролируется настройкой `binary.handling.mode`
    

⚠️ Опасно: может сломать совместимость downstream-консьюмеров при обновлении Debezium, когда тип станет поддерживаемым.

**Значение по умолчанию**: `false`

---
### **database.initial.statements**
Список SQL-запросов, которые Debezium выполняет при установке JDBC-соединения.

❗ Не выполняются при открытии соединения для чтения WAL.  
❗ Полезно только для установки session settings (например, `SET search_path=...`).

Запросы разделяются `;`.  
Чтобы использовать точку с запятой как символ — пишется `;;`.

**Значение по умолчанию**: _нет_

---

### **status.update.interval.ms**
Частота отправки сообщений о статусе репликационного соединения PostgreSQL.

Также влияет на скорость обнаружения того, что БД «умерла».

**Значение по умолчанию**: `10000` (10 секунд)

---
### **heartbeat.interval.ms**
Как часто коннектор отправляет heartbeat-сообщения в heartbeat-topic.

Heartbeat полезны:
- для мониторинга активности коннектора
    
- для обновления LSN, когда нет событий о данных
    
- чтобы избежать переполнения WAL на PostgreSQL (важно!)
    

#### Значения:
- `0` — отключено
- > 0 — интервал между heartbeat в миллисекундах

**Значение по умолчанию**: `0`

---

### **heartbeat.action.query**
SQL-запрос, который Debezium выполняет в момент отправки heartbeat.

Используется для "принудительного" появления изменений в WAL, особенно если:
- база низконагружена
    
- WAL копится из-за того, что нет событий, которые Debezium может отправить
    

Пример из документации:

`INSERT INTO test_heartbeat_table (text) VALUES ('test_heartbeat')`

**Значение по умолчанию**: _нет_

## Schema refresh, delays, snapshot fetch size**

### **schema.refresh.mode**
Когда Debezium должен обновлять in-memory схему таблиц.

Варианты:
#### **columns_diff** (по умолчанию)
Самый безопасный режим — схема обновляется при _любых_ расхождениях.

#### **columns_diff_exclude_unchanged_toast**
Схема обновляется только если расхождения не могут быть объяснены неизменившимися TOAST-полями.


Это повышает производительность, но:
⚠️ In-memory схема может устареть, если поля TOAST будут удалены.

**Значение по умолчанию**: `columns_diff`

---
### **snapshot.delay.ms**
Время в миллисекундах, которое Debezium ждёт перед началом snapshot.

Использование:
- запуск нескольких коннекторов одновременно
    
- предотвращение перегрузок и перестроений Connect cluster
    
- синхронизация snapshot-процессов
    

**Значение по умолчанию**: _нет_

---

### **streaming.delay.ms**
Задержка перед переходом в стриминг после завершения snapshot.

Полезно, если:
- snapshot завершился
    
- offset.flush ещё не выполнен
    
- в этот момент происходит сбой, и snapshot запускается повторно
    

Рекомендация:
`streaming.delay.ms > offset.flush.interval.ms`

**Значение по умолчанию**: `0`

---
### **snapshot.fetch.size**
Количество строк, считываемых Debezium за один SELECT при snapshot.

Больше число означает:
- меньше запросов
- более быстрый snapshot

Но:
− больше потребление памяти

**Значение по умолчанию**: `10240`

## Репликация PostgreSQL

### slot.stream.params
Список параметров для передачи в настроенный плагин логического декодирования, разделенных точкой с запятой.  
**Значение по умолчанию**: Нет значения по умолчанию

### slot.max.retries
Максимальное количество последовательных попыток подключения при сбое соединения со слотом репликации.  
**Значение по умолчанию**: `6`

### [slot.retry.delay.ms](https://slot.retry.delay.ms/)
Количество миллисекунд ожидания между попытками повторного подключения при сбое коннектора при подключении к слоту репликации.  
**Значение по умолчанию**: `10000`

### unavailable.value.placeholder
Определяет константу, которую предоставляет коннектор, чтобы указать, что исходное значение является "toasted"-значением, не предоставляемым базой данных.  
**Значение по умолчанию**: `__debezium_unavailable_value`

### provide.transaction.metadata
Определяет, генерирует ли коннектор события с границами транзакций и обогащает события изменения метаданными транзакций.  
**Значение по умолчанию**: `false`

### flush.lsn.source
Определяет, должен ли коннектор фиксировать LSN обработанных записей в исходной базе данных PostgreSQL, чтобы журналы WAL могли быть удалены. Установите `false`, чтобы отключить это. Внимание: если установлено `false`, LSN не будет подтвержден Debezium, что может привести к проблемам с дисковым пространством.  
**Значение по умолчанию**: `true`

### retriable.[restart.connector.wait.ms](https://restart.connector.wait.ms/)
Количество миллисекунд ожидания перед перезапуском коннектора после возникновения исправимой ошибки.  
**Значение по умолчанию**: `10000`

### skipped.operations
Список через запятую типов операций, которые будут пропущены во время потоковой передачи. Операции включают: `c` для вставок/создания, `u` для обновлений, `d` для удалений, `t` для усечений и `none`, чтобы не пропускать никакие операции.  
**Значение по умолчанию**: `t`

### [xmin.fetch.interval.ms](https://xmin.fetch.interval.ms/)
Определяет, как часто (в миллисекундах) XMIN будет считываться из слота репликации. Значение XMIN предоставляет нижнюю границу, с которой может начаться новый слот репликации. Значение по умолчанию `0` отключает отслеживание XMIN.  
**Значение по умолчанию**: `0`

### [database.query.timeout.ms](https://database.query.timeout.ms/)
Определяет время в миллисекундах, которое коннектор ждет завершения запроса. Установите значение `0`, чтобы убрать ограничение по времени.  
**Значение по умолчанию**: `600000`

## Инкрементальные снимки

### incremental.snapshot.chunk.size
Максимальное количество строк, которое коннектор извлекает и считывает в память во время фрагмента инкрементального снимка.  
**Значение по умолчанию**: `1024`

### incremental.snapshot.watermarking.strategy
Определяет механизм "водяных знаков", который коннектор использует во время инкрементального снимка для дедупликации событий.  
**Значение по умолчанию**: `insert_insert`

### read.only
Определяет, записывает ли коннектор "водяные знаки" в коллекцию сигнальных данных для отслеживания прогресса инкрементального снимка. Установите значение `true`, чтобы разрешить коннектору с доступом только для чтения использовать стратегию "водяных знаков", не требующую записи в коллекцию сигнальных данных.  
**Значение по умолчанию**: `false`

## Наименование топиков

### topic.naming.strategy
Имя класса `TopicNamingStrategy`, который должен использоваться для определения имени топика для событий изменения данных, схемы, транзакций, heartbeat и т.д.  
**Значение по умолчанию**: `io.debezium.schema.SchemaTopicNamingStrategy`

### topic.delimiter
Разделитель для имени топика.  
**Значение по умолчанию**: `.`

### topic.cache.size
Размер, используемый для хранения имен топиков в ограниченной параллельной хэш-карте. Этот кэш поможет определить имя топика, соответствующее заданной коллекции данных.  
**Значение по умолчанию**: `10000`

### topic.heartbeat.prefix
Определяет имя топика, в который коннектор отправляет heartbeat-сообщения. Шаблон имени топика: `topic.heartbeat.prefix.topic.prefix`.  
**Значение по умолчанию**: `__debezium-heartbeat`

### topic.transaction
Определяет имя топика, в который коннектор отправляет сообщения с метаданными транзакций. Шаблон имени топика: `topic.prefix.topic.transaction`.  
**Значение по умолчанию**: `transaction`
