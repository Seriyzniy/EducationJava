## ConnectionFactory  
Это интерфейс верхнего уровня, используемый **для установления соединений с брокером**. 

В типичном приложении обмена сообщениями существует единственный экземпляр этого интерфейса. 
==В ActiveMQ — это ActiveMQConnectionFactory==. На верхнем уровне эта конструкция сообщает местонахождение брокера сообщений, вместе с низкоуровневыми деталями того, как следует взаимодействовать с ним. 
> Как следует из названия, ConnectionFactory — это механизм, с помощью которого **создаются объекты Connection**.  
  
## Connection  
Это долгоживущий объект, который грубо похож на TCP-соединение — после создания он обычно **существует В ТЕЧЕНИЕ ВСЕГО ЖИЗНЕННОГО ЦИКЛА приложения до его закрытия.** 

> Connection — потокобезопасный и может работать с несколькими потоками одновременно. 
> Объекты Connection **позволяют создавать объекты Session** 
  
## Session  
> [!NOTE] Session
> Это дескриптор потока при взаимодействии с брокером.

Объекты Session ==НЕ ЯВЛЯЮТСЯ ПОТОКОБЕЗОПАСНЫМИ==, что означает, что они не могут быть доступны нескольким потокам одновременно. 

Session — это **основной транзакционный дескриптор**, с помощью которого программист **может закоммитить и откатить (rollback) операции обмена сообщениями**, если он работает в транзакционном режиме. 

> Используя этот объект, вы **создаете объекты** 
> - **Message**, 
> - **MessageConsumer** и **MessageProducer**, 
> - а также получаете указатели (дескрипторы) на объекты **Topic** и **Queue**.  
  
## MessageProducer  
Этот интерфейс ==позволяет отправлять сообщение== адресату.  
  
## MessageConsumer  
Этот интерфейс позволяет разработчику ==получать сообщения.== 

Существует **два механизма извлечения** сообщения:  
- 🔶Регистрация **MessageListener**. 
Это реализованный вами интерфейс обработчика сообщений, который будет последовательно обрабатывать любые сообщения, выдаваемые брокером, используя один поток.

- 🔶 Опрос (polling) на наличие сообщений с помощью метода `receive()`.

  
## Message  
Это, вероятно, самая важная структура, поскольку она переносит ваши данные.

**Сообщения в JMS** состоят **из двух аспектов**:  
- 🔶 ==Метаданные сообщения.== 
Сообщение содержит `заголовки` и `свойства`. 
И то, и то может рассматриваться, как элементы мапы (КЛЮЧ=ЗНАЧЕНИЕ). 

- **Заголовки** — это хорошо известные элементы, определенные спецификацией JMS и доступные напрямую через API, такие как **JMSDestination** и **JMSTimestamp**. 
	
- **Свойства** — это **произвольные фрагменты информации о сообщении**, которые задаются для упрощения обработки или маршрутизации сообщений без необходимости считывания самого пейлоада сообщения. 
	💡Вы можете, например, задать заголовок AccountID или OrderType.

- 🔶 ==Тело сообщения==. 
Из **Session** может быть создано несколько различных типов сообщений в зависимости от типа содержимого, которое будет отправлено в теле, наиболее распространенными из которых являются:
- **TextMessage** для строк 
- **BytesMessage** для двоичных данных.

# 📕 1. Общий принцип обмена сообщениями
## Участники

|Компонент|Роль|
|---|---|
|**Producer (производитель)**|Отправляет сообщения.|
|**Broker (ActiveMQ)**|Принимает, сохраняет, маршрутизирует и доставляет сообщения.|
|**Consumer (потребитель)**|Получает сообщения.|
|**Destination**|Очередь (`Queue`) или тема (`Topic`) — "адрес" сообщений.|

## 1. Установка соединения (TCP-сокеты)
1. **Producer** (или **Consumer**) подключается к брокеру через **TCP**:
```
tcp://localhost:61616
``` 

По умолчанию используется **протокол OpenWire** — собственный двоичный протокол ActiveMQ.
    
2. Создаётся ==постоянное двустороннее TCP-соединение== (socket).  
Оно не закрывается после отправки — используется постоянно для обмена сообщениями и подтверждений (ACK).

3. Producer создаёт объекты **Connection**, **Session** и **MessageProducer** (через JMS API).

---
## 2. Отправка сообщения
**Producer создаёт сообщение** (объект `javax.jms.Message`) и указывает:
- `Destination` — это либо **Queue (очередь)**, либо **Topic (тема)**;
- метаданные (заголовки, приоритет, TTL, delivery mode, correlationId и т. д.);
- полезную нагрузку (`body`).

Затем:
1. Producer вызывает:
```
producer.send(message);
```
    
2. Сообщение сериализуется в протокольный формат OpenWire и передаётся брокеру по TCP.
    
3. Брокер получает сообщение и смотрит его флаги:
    - **persistent = true** → сообщение должно пережить сбой (записывается на диск);
        
    - **persistent = false** → хранится только в оперативной памяти.


---
## 3. Приём сообщения в брокере

**Broker принимает сообщение** и:
- определяет, **куда его положить**, исходя из `Destination`;
    
- создаёт внутреннюю запись сообщения в **журнале хранения (message store / journal)**;
    
- добавляет метаданные о маршрутизации (в какую очередь/топик принадлежит сообщение);

Если сообщение **персистентное**, оно записывается в **журнал (journal)**.

### Что такое журнал (Journal)

> Журнал — это **последовательный лог-файл** на диске, куда брокер _добавляет новые сообщения_ и _записи об их подтверждениях_.  
> 💡Его особенность — **запись всегда в конец файла**, без случайного доступа.

В ActiveMQ журнал используется в адаптерах **KahaDB** или **LevelDB**.  
Обычно он состоит из набора файлов (например, по 32 МБ).

### Процесс:
1. Брокер сериализует сообщение в двоичный формат.
    
2. Записывает его в конец файла журнала.
    
3. Обновляет индекс: где хранится это сообщение (offset в файле).
    
4. Возвращает Producer'у **ACK (подтверждение приёма)**.

💡 Такая структура ускоряет запись — диск пишет последовательно, без лишних seek-операций (особенно эффективно на HDD и SSD).

---
## 4. Хранение и ожидание доставки

Брокер имеет **реестр (registry)** всех активных `Destination`:
- Очереди (Queue)
- Топики (Topic)

Каждая `Destination` содержит:
- список **подписчиков** (consumers);
- список **непрочитанных сообщений**;
- ==внутреннюю очередь ожидания== (message cursor).

### ➤ Для Queue:
- Сообщение сохраняется в **очереди**, пока один из потребителей его не прочитает.
    
- Если подключено несколько потребителей, брокер применяет **load balancing (round-robin)**.
    
- После подтверждения получения (`ack`) сообщение помечается как доставленное и **удаляется из хранилища**.

### ➤ Для Topic:
- Сообщение **дублируется** для каждого активного подписчика.
    
- Если подписчик **не онлайн** и **тема не durable** — сообщение **теряется**.
    
- Если подписчик durable (устойчивая подписка), брокер сохраняет сообщение в **отдельную очередь подписчика**, пока он не прочитает.


В памяти брокер поддерживает **очереди** и **индексы**, указывающие на позиции сообщений в журнале.

---
## 5. Получение сообщения Consumer’ом
1. **Consumer** подключается к брокеру через TCP (аналогично Producer).
    
2. Создаёт подписку на нужную очередь или топик.
    
3. Брокер находит непрочитанные сообщения и:
    - извлекает их из памяти (или читает из журнала);
        
    - отправляет Consumer’у по тому же TCP-соединению.
        
4. Consumer обрабатывает сообщение и отправляет **ACK (подтверждение)** обратно брокеру.

---
## 6. Удаление (подтверждение) сообщения

Когда брокер получает подтверждение (ACK) от Consumer:
1. Он помечает сообщение как доставленное.
    
2. В журнал записывается запись об **удалении/подтверждении**.
    
3. Если все сообщения из файла журнала прочитаны, этот файл **удаляется или архивируется**.

Таким образом, журнал остаётся компактным и не засоряется старыми данными.

---
## 7. Поведение при сбое

Если брокер внезапно выключится (например, отключилось питание):
- При следующем запуске ActiveMQ **восстановит состояние из журнала**, перечитав лог и пересоздав индексы.
    
- Все **персистентные сообщения**, не подтверждённые Consumer’ами, будут снова доступны для доставки.
