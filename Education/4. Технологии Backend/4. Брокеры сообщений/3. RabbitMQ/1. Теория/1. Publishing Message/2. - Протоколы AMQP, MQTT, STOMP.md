
> [!NOTE] Публикация ВСЕГДА через Exchage
> Публикация сообщений в RabbitMQ **работает почти одинаково во всех поддерживаемых протоколах**: 
> Издатель -> Exchanger(Обменник) -> Очередь -> Получатель
> 
> Отличается лишь тип Обменника

**Все протоколы поддерживают publisher acknowledgements**, позволяя издателю понимать, принял брокер сообщение или нет и при необходимости повторно отправить его.  

Основные различия касаются **терминологии и набора свойств сообщения**, а не общей логики работы.

# 1. AMQP 0-9-1
## 1. Архитектура
В **AMQP 0-9-1** публикация происходит **на канале (channel) в обменник (exchange)**. 

Обменник (`Exchange`) использует топологию маршрутизации, заданную путем определения **привязок (bindings)** между одной или несколькими очередями и этим обменником, или между исходным обменником и обменником-получателем. 

Успешно маршрутизированные сообщения сохраняются в очередях.

Роль каждой сущности описана в руководстве по концепциям AMQP 0-9-1.

## 2. Ошибки публикаций
Существует несколько распространенных ==типов ошибок публикации==, которые обрабатываются с помощью различных функций протокола:
### 1. Публикация в несуществующий обменник
**Публикация в несуществующий обменник** приводит к ==ОШИБКЕ КАНАЛА (channel error)==, который закрывается, так что дальнейшая публикация (или любая другая операция) на нем становится невозможна.

### 2. Сообщение не может быть маршрутизирована ни в одну очередь, и `Mandatory=false`
- **Когда опубликованное сообщение не может быть маршрутизировано ни в одну очередь** (например, потому что для целевого обменника не определены привязки), 
	
- и издатель установил свойство сообщения `mandatory` в значение `false` (это значение по умолчанию)

Тогда сообщение ==ОТБРАСЫВАЕТСЯ== ИЛИ ==ПЕРЕПУБЛИКУЕТСЯ== **в альтернативный обменник** (alternate exchange), если он настроен.

### 3. Сообщение не может быть маршрутизирована ни в одну очередь, и `Mandatory=true`
- **Когда опубликованное сообщение не может быть маршрутизировано ни в одну очередь**, 
	
- и издатель установил свойство сообщения `mandatory` в значение `true`, 

Тогда ==сообщение будет ВОЗВРАЩЕНО (returned)== ИЗДАТЕЛЮ. 

> Издатель должен иметь **обработчик возвращенных сообщений (returned message handler)**, чтобы обработать возврат (например, залогировав ошибку или повторив отправку через другой обменник).

# 2. AMQP 1.0
В AMQP 1.0 публикация происходит в контексте **линка (link)**.

**Link (Линк)** — в AMQP 1.0 это однонаправленный канал между двумя узлами (например, клиентом и брокером) для отправки или получения сообщений. Более абстрактная и гибкая концепция по сравнению с каналами в AMQP 0-9-1.

# 3. MQTT
## 1. Архитектура
В MQTT сообщения публикуются **в топик по установленному соединению**, а сервер RabbitMQ маршрутизирует их через **topic-exchange** в очереди.

> При **QoS 1** брокер подтверждает публикацию пакетом **PUBACK**.
> [MQTT Version 5.0](https://docs.oasis-open.org/mqtt/mqtt/v5.0/os/mqtt-v5.0-os.html#_Toc3901121)


Издатели могут указать серверу, что опубликованное сообщение в топике должно быть **сохранено (retained)** — сохраняется только **последнее** сообщение на каждый топик.

## 2. Коды подтверждения (acks) в пакете PUBACK
В **MQTT 5.0** PUBACK содержит **reason code**, по которому RabbitMQ сообщает результат:
- **0 – Success**: сообщение успешно доставлено во все очереди;
    
- **16 – No matching subscribers**: не найдено подходящих привязок;
    
- **131 – Implementation specific error**: брокер отклонил сообщение (например, очередь недоступна).

## 3. Обнаружение ошибок публикаций
В **MQTT 3.1/3.1.1** ошибки публикации можно обнаружить только по **разрыву соединения** — других механизмов нет.

Подробнее см. в руководствах по MQTT и MQTT-over-WebSockets.

# 3. STOMP
## 1. Архитектура
Клиенты STOMP публикуют сообщения **по установленному соединению** и указывают **destination** — это строка, которая определяет, куда именно RabbitMQ должен направить сообщение.

В RabbitMQ `destination` может означать разное в зависимости от настроек STOMP-плагина:
- очередь,
- обменник,
- или маршрут в обменник.

То есть семантика `destination` **не жёстко определена протоколом**, а зависит от того, как RabbitMQ интерпретирует путь.

## 2. Уведомление об ошибке
STOMP позволяет издателю узнать, **успешно ли сервер обработал publish**.  

Его вариация подтверждений для издателей называется **квитанциями (receipts)** — это функция, которую клиенты активируют при публикации.

Как это работает:
1. Клиент отправляет сообщение и добавляет заголовок `receipt:<id>`.
    
2. RabbitMQ обрабатывает публикацию.
    
3. RabbitMQ отправляет обратно кадр `RECEIPT` с тем же `<id>`.
    
4. Если `RECEIPT` не пришёл — сервер не смог обработать запрос. 

То есть это аналог publisher confirms, но **управляется клиентом и работает иначе**, более "вручную".

Подробнее см. в руководстве по STOMP, STOMP-over-WebSockets и в спецификации STOMP 1.2.

# 2. Маршрутизация
## 1. AMQP 0-9-1  
В AMQP 0-9-1 ==маршрутизация выполняется обменниками== (exchanges). 

Обменники — это именованные таблицы маршрутизации. Элементы этих таблиц называются **привязками (bindings)**. Более подробно это описано в руководстве по концепциям AMQP 0-9-1.

Существует несколько **встроенных типов обменников**:
- **Topic (Топик)**
    
- **Fanout (Рассылка)**
    
- **Direct (Прямой)**, включая обменник по умолчанию (default exchange)
    
- **Headers (По заголовкам)**
    

Первые три типа рассмотрены с примерами в учебных руководствах (tutorials). 
(см [[1. - Publiher, Exchange, Bindings]])

**Дополнительные типы обменников могут быть предоставлены плагинами.** Плагины обменников, которые поставляются вместе с RabbitMQ:
- Consistent hashing exchange (Обменник с согласованным хешированием)
    
- Random routing exchange (Обменник со случайной маршрутизацией)
    
- Internal event exchange (Обменник внутренних событий)
    
- Delayed message exchange (Обменник с отложенными сообщениями)
    

Как и все плагины, **их необходимо включить, прежде чем использовать.**
