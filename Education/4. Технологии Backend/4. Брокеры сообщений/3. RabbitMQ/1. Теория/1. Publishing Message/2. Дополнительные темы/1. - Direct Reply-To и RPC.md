# 1. Мотивация
**Direct Reply-To** — механизм RabbitMQ, позволяющий реализовать RPC (запрос-ответ) **без создания отдельной очереди ответов**.

Это было описано в руководстве 6 [RabbitMQ Tutorials | RabbitMQ](https://www.rabbitmq.com/tutorials#6-rpc)

## **Проблема обычного RPC:**
### Обычный RPC
**RPC (Remote Procedure Call)** — это ==вызов удалённой процедуры (метода), как будто она локальная.==

Участвуют две стороны:
- **Client** (клиент, инициатор) — хочет вызвать метод.
- **Server** (сервер, исполнитель) — выполняет метод и возвращает результат.

Классический RPC _ничего не знает о брокерах сообщений_ — исторически это просто сетевой вызов (gRPC, JSON-RPC, XML-RPC, Thrift и т.д.).

### RPC с использование RabbitMQ
RabbitMQ предоставляет шаблон «request–reply»:
- Клиент отправляет сообщение-запрос.
    
- Сервер читает запрос, обрабатывает и отправляет ответ.
    
- Клиент получает ответ.

💡 **Это всё тот же RPC, просто транспорт — RabbitMQ**, который перенаправляет сообщения между ними

RPC (запрос/ответ) — это распространенный паттерн при работе с брокерами сообщений, такими как RabbitMQ. В руководстве 6 показаны несколько реализаций клиента. 

- Обычно сторона, инициирующая запрос (**RPC-клиент**), **отправляет запросы, которые маршрутизируются в долгоживущую, известную очередь запросов.** 
- Сторона, обрабатывающая запрос (**RPC-сервер**), потребляет сообщения из этой очереди и отправляет ответы, используя имя очереди, указанное в свойстве `reply-to` сообщения запроса.

Откуда берется очередь для ответов инициатора? 

Инициатор мог бы объявлять **одноразовую очередь для каждой пары запрос-ответ, но это НЕЭФФЕКТИВНО**: даже нереплицируемая очередь относительно дорога в создании и удалении по сравнению со стоимостью получения ответа. 
В кластерах накладные расходы выше, потому что все узлы должны согласовывать создание, тип, параметры репликации и другие метаданные.

==Лучший подход== — создать **одну очередь ответов на каждого инициатора** и повторно использовать ее для множества запросов. 

Свойства этой очереди зависят от случая использования:
- **Эксклюзивные очереди** распространены, когда ответы потребляет один клиент, и очередь удаляется при отключении.
    
- **Неэксклюзивные, долгоживущие очереди** подходят для долго выполняющихся задач, где ответы должны сохраняться при кратковременном отключении клиента.

## **Direct Reply-To решает это так:**  
RabbitMQ **вообще не создаёт очередь для ответов**.  

> ==RabbitMQ предоставляет "виртуальную" очередь==`"amq.rabbitmq.reply-to"` — **НЕ настоящую очередь**, а канал для прямой доставки.

Ответ доставляется **напрямую** от обработчика запроса к инициатору через внутренние процессы брокера — без буферизации, без сохранения, без фактической очереди.

**Преимущества:**
- нет записи метаданных очереди в Khepri;
- нет очередей для отображения в UI;
- нет процессов Erlang под очереди;
- меньше нагрузки на кластер.

### Поток сообщений

#### Запрос:
client → RabbitMQ → очередь сервера → сервер

#### Ответ:
server → RabbitMQ → **НЕ очередь** → сразу client


То есть:
- **запрос** проходит через обычную очередь
- **ответ** минует реальную очередь ответа и идёт напрямую клиенту

## **Важно:**  
==Это не прямое соединение между клиентами== — сообщение всё равно проходит через брокер, просто минуя очередь.

# 2. Использование Direct Reply-To
## 1. **Когда использовать Direct Reply-To**
Используйте Direct Reply-To, если:
- **Очень много клиентов** (десятки тысяч), делающих RPC.
    
- Соединения часто создаются и разрываются (high churn).
    
- Важно **минимизировать накладные расходы** на создание/удаление очередей.
    
- Допустимо, что ответ может потеряться (**at-most-once**).  
    Если ответ не пришёл → клиент просто повторяет запрос.

## 2. **Когда избегать Direct Reply-To**
Избегайте Direct Reply-To, если:
- Вам нужны гарантии **at-least-once** (ответ нельзя терять).
    
- Ответы должны **буферизоваться и сохраняться** брокером.
    
- Один клиент генерирует **большой поток RPC-ответов** (сотни+ в секунду). Очередь лучше справится благодаря буферизации.
    
- Вы используете **долгоживущие соединения**, делающие много RPC-вызовов — классическая очередь ответов будет столь же эффективной и иногда предпочтительнее.

# 3.**Детали реализации на стороне брокера**
Внутренне, RabbitMQ ==реализует Direct Reply-To с использованием типа очереди== `rabbit_volatile_queue`. "Волатильная" (volatile) описывает ее семантику: 
- **не устойчивая к перезапуску (non-durable)**, 
- **с нулевым буфером**, 
- **at-most-once** (не более одного раза), 
- **может терять сообщения**, 
- и **не хранится в хранилище метаданных**.

Вы увидите `rabbit_volatile_queue` только в нескольких местах. Ее экземпляры **не отображаются** в интерфейсе управления (Management UI) или в выводе команды `rabbitmqctl list_queues`.

Одно из мест, где она появляется — это метрики Prometheus, например:`rabbitmq_global_messages_delivered_total{protocol="amqp10",queue_type="rabbit_volatile_queue"}` — количество сообщений ("прямых ответов"), доставленных инициаторам запросов (requesters) по протоколу AMQP 1.0. 
`rabbitmq_global_messages_delivered_total{protocol="amqp091",queue_type="rabbit_volatile_queue"}` — количество сообщений ("прямых ответов"), доставленных инициаторам запросов по протоколу AMQP 0.9.1. 
`rabbitmq_global_messages_dead_lettered_maxlen_total{queue_type="rabbit_volatile_queue",dead_letter_strategy="disabled"}` — количество сообщений ("прямых ответов"), отброшенных RabbitMQ.

# 4. Использование
Direct Reply-To поддерживается для протоколов **AMQP 1.0** и **AMQP 0.9.1**. 

Он также работает **между протоколами** (например, инициатор запроса на AMQP 1.0 с обработчиком на AMQP 0.9.1, или наоборот).

## 1. **Использование в AMQP 1.0**
### 1. Клиент создает link для получения ответов
**Клиент (requester) создаёт link для получения ответов**

**Присоединение** (с источником, source) д**олжно устанавливать определенные поля**. 

Если ваша клиентская библиотека AMQP 1.0 поддерживает Direct Reply-To, она установит их за вас (см. примеры). 

В противном случае:
- `snd-settle-mode = settled` — ответы всегда считаются подтверждёнными.
    
- `address = null` — RabbitMQ сам создаст адрес.
    
- `durable = none` — состояние не сохраняется.
    
- `expiry-policy = link-detach` — всё удаляется при отсоединении линка.
    
- `rabbitmq:volatile-queue = true` — указание, что очередь виртуальная (или в `capabilities`).
    
- `timeout = 0` — никаких таймеров.

### 2. RabbitMQ возвращает псевдо-адрес
RabbitMQ возвращает сгенерированный брокером **псевдо-адрес очереди** ==в поле== `address` ==ответа== на присоединение (attach response).

В ответ на attach клиент получает адрес вида:
```bash
/queues/amq.rabbitmq.reply-to.<opaque-suffix>
```
где `<opaque-suffix>` не имеет смысла для клиентов.

Это **не настоящая очередь**, а виртуальный канал ответов.

### 3. Первый запрос
Перед отправкой первого запроса инициатор должен предоставить **кредит линка (link credit)** для этой псевдо-очереди.

Для каждого запроса **установите следующие СВОЙСТВА СООБЩЕНИЯ**:
- `message-id`: глобально уникальное значение. (Обработчик установит `correlation-id` ответа равным этому значению.)
    
- `reply-to`: адрес, ==полученный В ОТВЕТЕ НА ПРИСОЕДИНЕНИЕ== (attach response).

### 4. Сервер (responder) отправляет ответ
**Обработчик (responder) считывает** `reply-to` из запроса и отправляет ответ на этот адрес одним из двух способов:

#### **Вариант A (предпочтительный):**
1. *Присоединить отправляющий линк* к **анонимному терминалу (anonymous terminus)** (целевой адрес `null`), как описано в _Target Address v2_ ([AMQP 1.0 | RabbitMQ](https://www.rabbitmq.com/docs/amqp#target-address-v2)), 
	
2. и установить адрес ответа в свойстве `to` сообщения. Полезно при ответе многим различным инициаторам (без линка для каждого инициатора).

#### **Вариант B:**
Создать отправляющий линк напрямую к предоставленному адресу. 

В этом случае RabbitMQ проверяет, все ли еще подключен инициатор; если нет, RabbitMQ отклоняет линк.

### 5. Если обработка долгая
Если обработчик будет выполнять ресурсоемкую работу, он может заранее проверить, присутствует ли инициатор, отправив **HTTP GET через AMQP**. 

Статус 200 указывает, что инициатор все еще подключен (см. примеры).

## 2. **Особенности и ограничения AMQP 1.0**
- **Ответы не буферируются.** Если инициатор отключится или отклонит сообщение, вернуть его некуда — Direct Reply-To не имеет настоящей очереди.
    
- **Ответы можно получать только на том же connection и session**, где был создан receiving link. В другой сессии читать нельзя.
    
- **Ответы должны быть settled.**  
    Если обработчик отправит _unsettled_ ответ, RabbitMQ всё равно сразу пометит его как _accepted_ и может позже выбросить.
    
- **Нет отказоустойчивости.** Ответы могут быть потеряны, если:
    - У инициатора AMQP 1.0 **заканчивается кредит линка (link credit)**. Ответственность за предоставление достаточного кредита лежит на инициаторе.
    
	- **Управление потоком сессии AMQP 1.0** запрещает доставку. Инициатор должен поддерживать свой `incoming-window` достаточно большим.
    
	- **Процесс записи AMQP 1.0 брокера** не может отправлять ответы инициатору достаточно быстро.
        
- **Если потеря ответов недопустима — нужен классический RPC через обычные очереди.**

## 3. Примеры: AMQP 1.0
```java
String requestQueue = "request-queue";

// ------------------------------------------------------------
// Создаём RESPONDER (RPC-сервер).
// Responder — это обработчик запросов. Он слушает очередь
// request-queue, получает сообщения и отправляет ответы.
// При этом он сам будет отправлять ответы через Direct Reply-To,
// если инициатор указал reply-to = amq.rabbitmq.reply-to.
// ------------------------------------------------------------
Responder responder = connection.responderBuilder()
	// очередь, из которой Responder получает запросы 
    .requestQueue(requestQueue) 
    
    // обработчик входящих RPC-запросов
    .handler((ctx, req) -> { 
        // ------------------------------------------------------------
        // (НЕ обязательно) Проверяем, жив ли requester.
        // Это имеет смысл, т.к. Direct Reply-To не буферизует ответы.
        // Если requester отключился — нет смысла формировать ответ.
        // ------------------------------------------------------------
        if (ctx.isRequesterAlive(req)) {
	        // читаем тело запроса
            String in = new String(req.body(), UTF_8);
            
            // формируем ответ
            String out = "*** " + in + " ***";
            
            // создаём ответное сообщение
            return ctx.message(out.getBytes(UTF_8));          
        } else {
            // requester уже отключён — ответ не создаём
            return null;
        }
    }).build();


// ------------------------------------------------------------
// Создаём REQUESTER (RPC-клиент).
// Важно: Requester по умолчанию использует Direct Reply-To.
// То есть reply-to будет = amq.rabbitmq.reply-to, и ответы
// будут приходить напрямую, минуя любую очередь.
// ------------------------------------------------------------
Requester requester = connection.requesterBuilder()
    .requestAddress().queue(requestQueue)   // куда отправлять запросы (очередь RPC)
    .requester()
    .build();

// ------------------------------------------------------------
// Создаём RPC-запрос — обычное сообщение
// ------------------------------------------------------------
Message request = requester.message("hello".getBytes(UTF_8));

// ------------------------------------------------------------
// Отправляем запрос Responder'у.
// Поскольку Direct Reply-To включён по умолчанию,
// requester автоматически слушает amq.rabbitmq.reply-to.
// publish() возвращает CompletableFuture<Message> —
// мы получим ответ асинхронно.
// ------------------------------------------------------------
CompletableFuture<Message> responseFuture = requester.publish(request);

// ------------------------------------------------------------
// Получаем ответ, ожидая максимум 10 секунд.
// Если responder ответит — future завершится и вернёт Message.
// ------------------------------------------------------------
Message response = responseFuture.get(10, TimeUnit.SECONDS);

```
