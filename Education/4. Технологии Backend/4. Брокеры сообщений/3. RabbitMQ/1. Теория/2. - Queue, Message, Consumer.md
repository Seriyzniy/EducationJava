# 1. Queue
![[Pasted image 20251209150932.png]]

**Queue** — это последовательная структура, куда RabbitMQ помещает сообщения до того, как их прочитает Consumer.
- Messages → очередь → Consumer
    
- Обработка строго по порядку (FIFO), если не используются специальные опции.

## 1. Персистентность сообщений
1. `delivery_mode` **у Publisher**:
    - `1` — transient (не сохранять на диск, быстрее)
    - `2` — persistent (сохранять на диск, надёжнее)
    
2. **Durable/Transient у Queue**:
    - **Durable queue** — очередь выживает после перезагрузки RabbitMQ
        
    - **Transient queue** — исчезает при перезагрузке

> [!NOTE] ПЕРСИСТЕНТНОСТЬ сообщений только если очередь тоже персистентная
> Важно: даже если сообщения persistent (`delivery_mode=2`), но очередь **не durable**, после перезагрузки **всё пропадёт**.

## 2. Типы очередей
|Тип очереди|Особенности|Когда использовать|
|---|---|---|
|**Classic**|обычная очередь, стандартный FIFO|большинство случаев|
|**Quorum**|гарантирует консистентность через кворум в кластере|надёжное хранение, отказоустойчивость|
|**Stream**|новая очередь с принципами Kafka, поддержка больших потоков|редкое использование, потоковые сценарии, аналитика|

## 3. Durable/Transient
- ==Durable== queue = очередь **выживает после рестарта RabbitMQ**
    
- ==Transient== queue = исчезает при рестарте

**Сообщения persistent** без durable queue = потеря данных при рестарте

# 2. Message
**Message** — это объект, который Publisher отправляет в Exchange, а Consumer получает из очереди.  

## 1. Основные поля сообщения
- **payload**
    - Полезная нагрузка: строка, JSON, Base64, бинарные данные (картинки, файлы)
        
    - Практический размер одного сообщения: **до 128 MB** (хотя теоретически ~2 GB)
        
    - Совет: не злоупотреблять большими сообщениями — может сильно нагрузить сеть и брокер
        
- **routing key**
    - Ключ маршрутизации (строка), который Exchange использует для доставки сообщения
        
    - **Один routing key на сообщение**
        
    - Работает с Direct и Topic Exchange
    
- **delivery_mode**
    - Признак персистентности сообщения:
        - `1` — transient (не сохранять на диск)
        - `2` — persistent (сохранять на диск)
            
    - Работает в связке с durable очередями
    
- **headers**
    - Набор ключ-значение
        
    - Используется для:
        - **Headers Exchange** (маршрутизация по заголовкам)
            
        - TTL (время жизни сообщения)
            
        - Дополнительные метаданные для приложений

# 3. Consumer
![[Pasted image 20251209151921.png]]

**Consumer** — это приложение или демон, который **получает сообщения из очереди и выполняет логику обработки**:
- отправка уведомлений
- запись в базу данных
- генерация отчётов
- пересылка сообщений в другие очереди

Он замыкает весь путь сообщения от Publisher через Exchange и Queue.

## 1. Декларирование сущностей
Consumer **может декларировать практически все сущности** — exchanges, queues, bindings и тд. 

> На практике мы стараемся декларировать все сущности именно Consumer, но решать нужно для каждого проекта индивидуально.

## 2. Подключение и канал
- Consumer создаёт **соединение (connection)** по AMQP
    
- Внутри соединения создаёт **канал (channel)**
    
- Через канал инициирует **consuming** сообщений из очереди

### Подписка на ОДНУ очередь
❗❗❗ Обычно **Consumer подписывается на одну очередь**.  

Если нужно получать из нескольких очередей — лучше **смаршрутизировать их потоки в одну очередь**, а не создавать кучу Consumer внутри приложения.

> Для этого тогда необходимо правильно настроить Bindings

## 3. Push-модель доставки
- RabbitMQ **проталкивает сообщения в Consumer по мере их появления**
    
- Нет периодического опроса — это **быстро и удобно**

## 4. Prefetch count
- Определяет **количество НЕПОДТВЕРЖДЁННЫХ сообщений**, которые Consumer может держать одновременно
    
- По умолчанию = 0 → Rabbit проталкивает **все сообщения сразу**
    
- ❗❗❗Рекомендуется ставить = 1 → Rabbit отправляет следующее сообщение **только после ack предыдущего**
    
- При мультитредовой обработке можно увеличить для параллелизма

## 5. Подтверждение сообщений
1. **Acknowledge (ack)**
    - Consumer подтверждает успешную обработку
    - Если подтверждения нет, то СООБЩЕНИЕ НЕ УДАЛЯЕТСЯ ИЗ ОЧЕРЕДИ
      
2. **Negative Acknowledge (nack)**
    - Consumer сообщает о неудаче, ==сообщение возвращается в очередь==
        
    - ==Автоматически срабатывает при разрушении канала==, что удобно при остановке Consumer
    
3. **AutoAck**
    - ==Rabbit считает сообщение обработанным сразу после отправки==
        
    - Быстро, но **нет гарантий**, что сообщение реально обработано
