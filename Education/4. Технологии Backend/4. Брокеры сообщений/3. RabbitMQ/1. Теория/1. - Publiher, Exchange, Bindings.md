# 1. Publisher

> [!NOTE] **Publisher**
> `внешнее приложение`, которое публикует сообщения в RabbitMQ по протоколу **AMQP**.
> 

![[Pasted image 20251208223325.png]]

Чтобы понять, что делает Publisher, важно разобраться в базовых абстракциях RabbitMQ.

## Что такое Connection и Channel

### ✔ Connection
Это **физическое TCP-соединение** между приложением и RabbitMQ.  
Оно «дорогое» — создание требует handshake, аутентификации, выдачи параметров сессии, инициализации работы с брокером.

Закрывать/открывать Connection часто — _очень плохо_. Это тяжёлая операция.

### ✔ Channel
Channel — это **лёгкая виртуальная сессия** внутри одного Connection.  
> Можно представить, что Connection — это трубопровод, а Channel — это «поток внутри трубопровода», по которому идут отдельные команды AMQP.

Зачем нужны каналы?
- Чтобы параллелить работу в одном TCP-соединении
    
- Чтобы несколько потоков могли «логически» работать независимо
    
- Чтобы уменьшить нагрузку на установку TCP-соединений

Каждый канал создает отдельный `Erlang` процесс. 
Одно подключение может иметь множество каналов ([multiplexing](https://ru.wikipedia.org/wiki/%D0%9C%D1%83%D0%BB%D1%8C%D1%82%D0%B8%D0%BF%D0%BB%D0%B5%D0%BA%D1%81%D0%B8%D1%80%D0%BE%D0%B2%D0%B0%D0%BD%D0%B8%D0%B5)). 

Для каждого канала существуют некие структуры и объекты в памяти. 
Поэтому чем больше каналов имеется в рамках соединения, тем **больше памяти использует RabbitMQ** для управления таким соединением.


> ❗ Но ВАЖНО:  
> Хотя AMQP разрешает множество каналов в одном Connection, **официальная документация RabbitMQ не рекомендует использовать десятки/сотни каналов одновременно**, особенно при активной публикации.
> 
> RabbitMQ плохо переносит ситуацию, когда приложение безостановочно создаёт заново каналы и соединения («флаппинг»).


## «Флаппинг» каналов - плохо
Это когда Publisher для каждого сообщения делает так:
1. connect
2. open channel
3. publish
4. close channel
5. close connection

Так нельзя.  
RabbitMQ начинает «умирать» уже при **≈300 таких операций в секунду**.

Причина:  
▪ каждый канал — это структура в памяти сервера;  
▪ открытие/закрытие — дорогая операция;  
▪ частое создание TCP-соединений ещё тяжелее.

#### ✔ Правильно
Создать connection один раз → создавать 1–2 channel → публиковать тысячи сообщений без закрытия → закрыть только по завершению работы.

#### Что делать, если Publisher изменить нельзя?
Использовать **amqproxy** — ==прокси==, которое держит постоянные соединения с RabbitMQ и работает как «буфер» для плохих издателей.

❗❗❗Но **нельзя** использовать его для Consumer — разрушение соединений в одну сторону приводит к неустойчивости.

## Publisher всегда пишет в exchange (никогда — в очередь)
Метод отправки сообщения
```java
void basicPublish(
    String exchange,      // 1. Обменник
    String routingKey,    // 2. Ключ маршрутизации (Тема сообщения)
    BasicProperties props, // 3. Свойства сообщения
    byte[] body           // 4. Тело сообщения (данные)
) throws IOException;
```

Даже если ты указываешь:
```java
channel.basicPublish(
	"", // Exchange
	QUEUE_NAME, 
	null, 
	message.getBytes());
```
на самом деле это:
- отправка в **default exchange** `""`
- с routing key = `"myQueue"`

==Очередь получает сообщение только через exchange.==

## delivery_mode — персистентность сообщения
Publisher определяет, сохранять ли сообщение на диске.
- **delivery_mode = 1** — _не сохранять_: быстрее, но сообщение потеряется при рестарте RabbitMQ
    
- **delivery_mode = 2** — _сохранять на диск_: медленнее, но надёжнее

> Используется при durable-очередях.

## Routing Key
**Publisher всегда задаёт routing key** — это «адрес» внутри exchange. 

Exchange определяет само, в какие очереди отправить сообщение по этому routing key.

## Publisher confirms (подтверждения издателя)
Это механизм, при котором RabbitMQ **говорит издателю**, что сообщение точно записано в очередь/на диск.

Зачем нужно?  

Если, например, у RabbitMQ **закончится место на диске**, он некоторое время продолжает _принимать_ publish-команды, но **не сохраняет** данные.  
Без confirms Publisher будет думать, что всё ОК.

С confirms издатель точно знает, какие сообщения дошли.

Минусы:
- снижает скорость (Rabbit ждёт подтверждений от сервера)
    
- неудобно в однопоточных языках (Python, Ruby) — нужно асинхронное ожидание ack/nack

## Флаг mandatory
Если `mandatory=true`:
- сообщение **не может быть отброшено**, если не нашлось очередей по маршруту
    
- оно возвращается издателю через `basic.return`

Или помещается в специальный exchange (alternate exchange).

> Используется редко, в основном для логгирования «потерянных» сообщений.

# 2. Exchange
**Exchange — это центральный роутер сообщений в RabbitMQ.**  
Все сообщения от Publisher **всегда сначала попадают в Exchange**, а уже затем — в очереди по правилам маршрутизации.

Exchange **не хранит сообщения**. Он только решает, куда их направить
![[Pasted image 20251209140406.png]]

## Exchange — неизменяемый объект
Exchange имеет параметры (тип, durability, аргументы).  
Но эти параметры **нельзя изменить после создания**.

Если нужно поменять:
- тип (fanout, topic, direct, headers),
- durability,
- auto-delete,
- другие свойства,

то нужно:
1. удалить exchange
2. создать заново с новыми настройками.

Это поведение заложено в AMQP-протокол.

## Binding — это отдельная сущность
**Binding** — это связка:
```css
Exchange  →  Queue
```

Binding определяет _правило маршрутизации_ (**например, routing key**). 

Binding **не является частью Exchange**, поэтому:
- можно добавлять / удалять binding в любое время,
    
- менять настройки binding без пересоздания exchange (**НО НУЖНО ПЕРЕСОЗДАТЬ САМ Binding)**.

Таким образом:
- Exchange — это роутер,
- Binding — его «правила маршрутизации».

## Доставка сообщений
Когда Exchange получает сообщение:
1. Он ищет все очереди, у которых есть **подходящие binding**.
    
2. Для каждой из этих очередей отправляет _ровно одно_ сообщение (даже если несколько binding у одной и той же очереди подходят).

Пример:  
Если очередь `A` имеет два подходящих routing key — всё равно попадёт **одно сообщение**, не два.

## Durable vs Transient
Это признак **ПЕРСИСТЕНТНОСТИ EXCHANGE**, а НЕ СООБЩЕНИЙ.
- 🔶**Durable = true** → Exchange сохранится после перезагрузки RabbitMQ.
- 🔶**Durable = false** → пропадёт при рестарте брокера.

#### Важно: 
durable **НЕ ДЕЛАЕТ СООБЩЕНИЯ PERSISTENT** — этим занимается `delivery_mode` у Publisher.

## Exchange НЕ хранит сообщения
Exchange — **не очередь**.

Он не умеет:
- буферизовать сообщения,
- хранить их,
- удерживать до появления подходящей очереди.

==Если для сообщения нет подходящего маршрута: → Exchange сразу отбрасывает сообщение==
(если не включены механизмы наподобие alternate exchange).

Восстановить такое сообщение невозможно, оно просто исчезает.

# 3. Binding
**Binding — это правило маршрутизации**, связывающее:
```css
Exchange  →  Queue
	binding: "routing_key1"
    binding: "routing_key2"
```

Binding говорит Exchange:  
> «Если приходит сообщение, соответствующее этим условиям, отправь его в эту очередь».

То есть Binding — это **статический маршрут**, по которому Exchange доставляет сообщение.

## Binding — неизменяемый объект
==Binding не поддерживает изменение параметров==.  

Если нужно изменить:
- routing key,
- headers,
- аргументы,

то:
1. старый binding удаляется,
2. создаётся новый с нужными параметрами.

Это **сделано специально — чтобы правила маршрутизации были чётко определены** и предсказуемы.

## Несколько binding к одной паре Exchange–Queue
Между одним Exchange и одной Queue может быть **несколько binding**, но:
- каждый binding должен иметь **уникальные параметры**,
- ==два одинаковых binding создать нельзя==.


Пример для topic exchange:
```java
exchange → queue_name
    binding: "user.*"
    binding: "order.*"
```
Оба действуют независимо.

Но создать два binding с одинаковым routing key нельзя (это просто дублирование).

## Параметры Binding
Параметры зависят от типа Exchange:
### ✔ Для direct / topic exchange
Используется **routing key**. 

Binding определяется строкой (или шаблоном)
```http
routing key = "user.created"
routing key = "user.*"
routing key = "order.#"
```

### ✔ Для headers exchange
Используются **headers** — набор ключ/значение; Exchange будет маршрутизировать по соответствию заголовков.

Например:
```http
x-match = any
type = "payment"
country = "US"
```

# 4. Типы Exchange
_После разбора binding вернёмся к типам Exchange, так как их работа неразрывно связана._ 

Выделяют **четыре типа Exchange**:
- Fanout;
- Direct;
- Topic;
- Headers.

## 1. Fanout Exchange — "рассылка всем"
**Fanout** — самый простой и «тупой» тип обменника.  

Он **НЕ СМОТРИТ НИ НА ROUTING KEY, НИ НА ЗАГОЛОВКИ**.

Работает по принципу:
> Пришло сообщение → отправить его во ВСЕ очереди, которые привязаны binding'ом к этому exchange.

Это буквально **broadcast** (рассылка) внутри RabbitMQ.

> По скоростям выдает на тестах около 30000mps, но столько же выдает и тип **Direct.**

## 3. Topic
**Topic Exchange** похож на Direct Exchange, НО ПОЗВОЛЯЕТ ИСПОЛЬЗОВАТЬ **шаблоны (wildcards)** в binding key, а не только точное совпадение.

### Wildcards
1. `*` — совпадает **ровно с одним словом**
    - слова разделяются точкой `.`
пример:
```JAVA
binding key = "user.*.created"

message key = "user.admin.created" → совпадение
message key = "user.admin.deleted" → нет
```

2. `#` — совпадает с **любым количеством слов** (включая ноль)
пример:
```java
binding key = "order.#"

message key = "order" → совпадение
message key = "order.new" → совпадение
message key = "order.new.fast" → совпадение
```

## 4. Headers Exchange — маршрутизация по заголовкам
**Headers Exchange** — самый гибкий тип обменника в RabbitMQ.
- Вместо **routing key** он использует **заголовки сообщений** и правила binding.
    
- Binding содержит набор заголовков и параметр **x-match**, который определяет логику совпадения.

### Параметр x-match
1. `x-match = all`
    - Для попадания сообщения в очередь **должны совпасть все заголовки**, указанные в binding.
        
Пример:
```json
binding: {type: "payment", country: "US", x-match=all}

message: {type: "payment", country: "US"} → совпадение ✅
message: {type: "payment", country: "CA"} → ошибка ❌
```


2. `x-match = any`
	- Для попадания достаточно, чтобы **хотя бы один заголовок совпал**.

Пример:
```json
binding: {type: "payment", country: "US", x-match=any}


message: {type: "payment", country: "CA"} → совпадение ✅ (type совпадает)

message: {type: "invoice", country: "US"} → совпадение ✅ (country совпадает)

message: {type: "invoice", country: "CA"} → ошибка ❌ (нет совпадений)
```
