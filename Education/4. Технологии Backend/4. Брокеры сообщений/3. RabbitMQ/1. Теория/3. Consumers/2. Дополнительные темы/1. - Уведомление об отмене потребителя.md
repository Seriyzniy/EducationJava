Если потребитель перестаёт получать сообщения не по своей инициативе (например, очередь удалена или упал узел кластера), **канал по умолчанию не получает уведомления**, что вызывает проблемы.
(То есть, Consumer работает нормально, но что-то отвалилось у брокера, из-за чего сообщения не потребляются)

Чтобы брокер **сообщал клиенту** о таких неожиданных отменах, он отправляет ему `basic.cancel` асинхронно.

Это работает **только если клиент при подключении** объявляет ключ `consumer_cancel_notify`:
```ini
consumer_cancel_notify = true
```
(в `client-properties`).

> Обычная отмена, инициированная самим клиентом (`basic.cancel`), не вызывает таких уведомлений.

---
## Java,.NET, Go, Python клиенты по умолчанию включают этот флаг
Поддерживаемые RabbitMQ клиенты (Java, .NET, Go, Python и др.) **по умолчанию включают capability**:
```java
consumer_cancel_notify = true
```

👉 **Брокер имеет право отправлять клиенту асинхронный basic.cancel**, если потребитель был отменён по причинам, не связанным с самим клиентом.

Для этого в Java-клиенте есть метод:
```java
public void handleCancel(String consumerTag)
```

Например, в нашем Java-клиенте интерфейс `Consumer` имеет колбэк `handleCancel`, который можно переопределить, создав подкласс `DefaultConsumer`:
```java
channel.queueDeclare(queue, false, true, false, null);

Consumer consumer = new DefaultConsumer(channel) {
    @Override
    public void handleCancel(String consumerTag) throws IOException {
        // Потребитель был неожиданно отменен
    }
};
channel.basicConsume(queue, consumer);
```

Не является ошибкой, если клиент отправит `basic.cancel` для потребителя, который уже был неожиданно отменен (например, из-за удаления очереди). 

По определению, возможна гонка (race condition) между отправкой `basic.cancel` клиентом и отправкой асинхронного уведомления брокером. В таких случаях брокер не генерирует ошибку при получении `basic.cancel` и отвечает `basic.cancel-ok`, как обычно.

## **Отмена потребителя и реплицированные очереди**
Клиенты, поддерживающие уведомление об отмене потребителя, всегда будут получать информацию, когда очередь удаляется или становится недоступной. 

Потребители могут запросить, чтобы они были отменены при изменении лидера (leader) реплицированной очереди.
