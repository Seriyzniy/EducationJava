
# 1. Что такое обменник для недопоставленных сообщений (Dead Letter Exchange)?
Сообщения из очереди могут быть помечены как "недоставленные" (dead-lettered), что означает их повторную публикацию в обменник при возникновении любого из следующих четырех событий:

1. Сообщение отклонено получателем AMQP 1.0 с использованием исхода `rejected` или потребителем AMQP 0.9.1 с использованием `basic.reject` или `basic.nack` с параметром `requeue`, установленным в `false`, ИЛИ
    
2. Срок действия с**ообщения истек из-за TTL** (время жизни) для конкретного сообщения, ИЛИ
    
3. Сообщение удалено, потому что его очередь превысила ограничение по длине, ИЛИ
    
4. Сообщение возвращено в очередь кворума больше раз, чем установлено в `delivery-limit`.

> **Важно**: Если срок действия всей очереди истекает, *сообщения в очереди не помечаются как недоставленные*.

Обменники для недопоставленных сообщений (DLX) — это обычные обменники. Они могут быть любого из обычных типов и объявляются как обычные.

---
# 2. Как настраивается механизм недоставленных сообщений
Для любой очереди DLX может быть определен клиентами с использованием политик. 

Существует несколько ключей политики, связанных с DLX, включая некоторые, поддерживаемые только очередями кворума, **но два основных**:
- `dead-letter-exchange`: Имя используемого DLX
    
- `dead-letter-routing-key`: Ключ маршрутизации, используемый при помещении сообщений в недоставленные

> **Важное примечание**: Ключи политики также могут быть установлены приложениями во время объявления очереди ==через дополнительные аргументы (x-arguments).==
> 
> **Настоятельно не рекомендуется использовать жестко заданные x-аргументы**, поскольку их нельзя обновить без переразвертывания приложений, в то время как политики можно обновлять в любой момент.
> 
> В случае, если и политика, и аргументы указывают DLX, аргумент, указанный в аргументах, переопределяет указанный в политике.

В дополнение к целевому имени DLX указывается ключ маршрутизации, который будет использоваться при помещении сообщений в недоставленные. 
> Если ключ маршрутизации не задан, используются собственные ключи маршрутизации сообщения.

Когда указан обменник для недоставленных сообщений, в дополнение к обычным правам на настройку объявленной очереди, пользователь должен иметь права на чтение для этой очереди и права на запись для обменника недоставленных сообщений. 
Права проверяются во время объявления очереди.

## Настройка обменника для недоставленных сообщений с использованием политики
Чтобы указать DLX с помощью политики, добавьте ключ `"dead-letter-exchange"` в определение политики:
```bash
# rabbitmqctl с bash
rabbitmqadmin policies declare \
    --name "DLX" \
    --pattern ".*" \
    --definition '{"dead-letter-exchange":"my-dlx"}' \
    --apply-to "queues" \
    --priority 7
```

```http
PUT /api/policies/%2f/DLX  
{
	"pattern": ".*",  
	"definition": {"dead-letter-exchange":"my-dlx"},  
	"priority": 7,  
	"apply-to": "queues"
}
```
(больше способов смотри [Dead Letter Exchanges | RabbitMQ](https://www.rabbitmq.com/docs/dlx))

В предыдущем примере объявляется *политика с именем "DLX"*, которая **ПРИМЕНЯЕТСЯ КО ВСЕМ ОЧЕРЕДЯМ** (независимо от типа), и настраивает обменник с именем "my-dlx" в качестве цели для недоставленных сообщений. 
> Это всего лишь пример; на практике часто используются несколько политик, каждая из которых применяется к подмножеству очередей.

Аналогичным образом, явный ключ маршрутизации может быть указан путем добавления ключа `"dead-letter-routing-key"` в политику:
```bash
rabbitmqadmin policies declare --name=DLX --pattern=".*" \
    --definition='{"dead-letter-exchange":"my-dlx", "dead-letter-routing-key":"my-routing-key"}' \
    --apply-to=queues --priority=7
```

## Настройка обменника для недоставленных сообщений с использованием дополнительных аргументов очереди


> [!WARNING]
> **Предупреждение**: Настоятельно ==не рекомендуется использовать жестко заданные x-аргументы==, поскольку их нельзя обновить без переразвертывания приложений и удаления очереди перед ее повторным объявлением, в то время как политики можно обновлять в любой момент.

Чтобы **установить DLX для очереди**, укажите дополнительный аргумент `x-dead-letter-exchange` при объявлении очереди. 

Значение должно быть именем обменника в том же виртуальном хосте:
```java
channel.exchangeDeclare("some.exchange.name", "direct");

// Важно: предпочтительно использовать политики вместо жестко заданных x-аргументов
Map<String, Object> args = new HashMap<String, Object>();
args.put("x-dead-letter-exchange", "some.exchange.name");

channel.queueDeclare("myqueue", false, false, false, args);
```

Приведенный *выше код объявляет новый обменник* с именем `some.exchange.name` *и устанавливает этот новый обменник* в ***качестве обменника для недоставленных сообщений*** для вновь созданной очереди. 

Обратите внимание, что обменник не обязательно должен быть объявлен при объявлении очереди, но о**н должен существовать к моменту, когда сообщения нужно будет поместить в недоставленные**.

> Если он отсутствует, сообщения будут тихо удалены.

В дополнение к целевому имени DLX можно указать ключ маршрутизации, используемый при помещении сообщений в недоставленные:
```java
// Важно: предпочтительно использовать политики вместо жестко заданных x-аргументов
Map<String, Object> args = new HashMap<String, Object>();
args.put("x-dead-letter-exchange", "some.exchange.name");
args.put("x-dead-letter-routing-key", "some-routing-key");
```

# 3. Маршрутизация недоставленных сообщений
***Недоставленные сообщения маршрутизируются в их обменник*** для недоставленных сообщений:
1. **С ключом маршрутизации, указанным для очереди**, в которой они находились; ИЛИ, если он не был установлен,
    
2. **С теми же ключами маршрутизации**, *с которыми они были* ***первоначально опубликованы***


Например, если вы публикуете сообщение в обменник с ключом маршрутизации `foo`, и это сообщение помечается как недоставленное, оно публикуется в его обменник для недоставленных сообщений с ключом маршрутизации `foo`. 

Если очередь, в которую сообщение изначально попало, объявлена с `x-dead-letter-routing-key`, установленным в `bar`, то сообщение публикуется в его обменник для недоставленных сообщений с ключом маршрутизации `bar`.

> [!NOTE]  
> Если конкретный ***ключ маршрутизации не был установлен для очереди***, **сообщения в ней помечаются как недоставленные** со **всеми их первоначальными ключами маршрутизации**. 
> 
> Это включает ключи маршрутизации, добавленные заголовками CC и BCC (подробнее см. в разделе "Распределение с выбором отправителя").[Sender-selected Distribution | RabbitMQ](https://www.rabbitmq.com/docs/sender-selected)

## Цикл недоставленных сообщений
***Возможно образование цикла*** **помечания сообщений как недоставленных**, при котором одно и то же сообщение достигает одной и той же очереди дважды. 

Например, это может произойти, когда *очередь помечает сообщения как недоставленные* **в обменник по умолчанию без указания ключа маршрутизации** для недоставленных сообщений. 

Чтобы предотвратить автоматическое бесконечное зацикливание сообщений в RabbitMQ, RabbitMQ обнаружит цикл и удалит сообщение, если во всем цикле не было отклонений.

# 4. Безопасность
Помечание сообщений как недоставленных — это форма публикации сообщений, и, как и любая форма публикации, **оно может завершиться неудачей в определенных сценариях**. 

Например, *если для помечания сообщений как недоставленных* **настроено использование очереди кворума**, ***в которой нет активного кворума***, **публикация завершится неудачей**, и узел, выполняющий помечание как недоставленное, зарегистрирует сообщение, похожее на следующее:
```
Cannot forward any dead-letter messages from source quorum queue 'qq.input' in vhost 'my-vhost'
with configured dead-letter-exchange exchange 'amq.topic' in vhost 'my-vhost'
```

## Повторная публикация с подтверждениями издателя
По умолчанию, **недоставленные сообщения повторно публикуются без активации внутренних подтверждений издателя**. 

Поэтому использование обменников для недоставленных сообщений (DLX) в кластерной среде RabbitMQ не гарантирует безопасность.

**Сообщения удаляются из исходной очереди** ==сразу после публикации в целевую очередь DLX==. 
Это гарантирует отсутствие риска чрезмерного накопления сообщений, которое могло бы истощить ресурсы брокера. 
> Однако сообщения могут быть потеряны, если целевая очередь недоступна для приема сообщений.

Очереди кворума поддерживают помечание сообщений как недоставленных с гарантией **«как минимум один раз»**, где сообщения повторно публикуются с активированными внутренними подтверждениями издателя.

# 5. Влияние помечания как недоставленного на сообщения
*Помечание сообщения как недоставленного* **ИЗМЕНЯЕТ ЕГО ЗАГОЛОВКИ**:
- *имя обменника заменяется на имя последнего обменника* для недоставленных сообщений;
    
- **ключ маршрутизации может быть заменён на указанный в очереди**, выполняющей помечание как недоставленное (т. е. настроенный `dead-letter-routing-key`);
    
- если происходит указанное выше, заголовок `CC` также будет удалён;
    
- заголовок `BCC` будет удалён в соответствии с распределением по выбору отправителя. 

> *Одно сообщение* **может быть помечено** как недоставленное **несколько раз**. 
> 
> **Каждый раз**, когда сообщение помечается как недоставленное, **это событие записывается в заголовок** сообщения. 
> 
> Чтобы предотвратить неограниченный рост заголовка, история событий о недоставленных сообщениях сжимается по паре **{Очередь, Причина}**.

- Сообщение **AMQP 1.0** будет содержать аннотацию сообщения с символьным ключом `x-opt-deaths`, а значением будет массив карт (словарей). 
	
- Сообщение **AMQP 0.9.1** будет ==содержать заголовок== `x-death`, значением которого ==является массив мап== ([array](https://docs.oasis-open.org/amqp/core/v1.0/os/amqp-core-types-v1.0-os.html#type-array) of [map](https://docs.oasis-open.org/amqp/core/v1.0/os/amqp-core-types-v1.0-os.html#type-map)s). Массив как в AMQP 1.0, так и в AMQP 0.9.1 упорядочен по новизне, то есть самое последнее событие помечания как недоставленного записывается в первом элементе массива.

В следующей таблице ==описаны пары «ключ-значение» карты== AMQP 1.0 и таблица элементов массива AMQP 0.9.1. Все ключи AMQP 1.0 имеют тип `symbol`. Клиенты AMQP 1.0 не должны зависеть от порядка пар «ключ-значение» в карте.

| Ключ AMQP 1.0  | Тип значения AMQP 1.0 | Ключ AMQP 0.9.1       | Тип значения AMQP 0.9.1 | Описание                                                                                                                                                         |
| -------------- | --------------------- | --------------------- | ----------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `queue`        | `string`              | `queue`               | `longstr`               | Имя очереди, из которой это сообщение было помечено как недоставленное.                                                                                          |
| `reason`       | `symbol`              | `reason`              | `longstr`               | Почему это сообщение было помечено как недоставленное (описано ниже).                                                                                            |
| `count`        | `ulong`               | `count`🔥             | `long`                  | Сколько раз это сообщение было помечено как недоставленное из этой очереди по этой причине.                                                                      |
| `first-time`   | `timestamp`           | —                     | —                       | Когда это сообщение было впервые помечено как недоставленное из этой очереди по этой причине.                                                                    |
| `last-time`    | `timestamp`           | —                     | —                       | Когда это сообщение было в последний раз помечено как недоставленное из этой очереди по этой причине.                                                            |
| `time`         | `timestamp`           | —                     | —                       | Когда это сообщение было впервые помечено как недоставленное из этой очереди по этой причине.                                                                    |
| `exchange`     | `string`              | `exchange`            | `longstr`               | Обменник, в который это сообщение было опубликовано до того, как оно было впервые помечено как недоставленное из этой очереди по этой причине.                   |
| `routing-keys` | массив `string`       | `routing-keys`        | массив `longstr`        | Ключи маршрутизации (включая `CC`, но исключая `BCC`) этого сообщения до того, как оно было впервые помечено как недоставленное из этой очереди по этой причине. |
| `ttl`          | `uint`                | —                     | —                       | Заголовок `ttl` AMQP 1.0 (время жизни в миллисекундах) до того, как это сообщение было впервые помечено как недоставленное из этой очереди по этой причине.      |
| —              | —                     | `original-expiration` | `longstr`               | Исходное свойство `expiration` этого сообщения до того, как оно было впервые помечено как недоставленное из этой очереди по этой причине.                        |

`ttl` AMQP 1.0 и `original-expiration` AMQP 0.9.1 являются необязательными и записываются, потому что **исходный TTL сообщения удаляется** ***из сообщения при помечании как недоставленного***, чтобы предотвратить его повторное истечение в любых очередях, в которые оно маршрутизируется.

Причина (`reason`) — это имя, описывающее, ***почему сообщение было помечено как недоставленное***, и является одним из следующих:
- **`rejected`** — сообщение было отклонено;
    
- **`expired`** — срок действия TTL сообщения истёк;
    
- **`maxlen`** — превышена максимально допустимая длина очереди;
    
- **`delivery_limit`** — сообщение возвращено больше раз, чем установленный лимит (задаётся аргументом политики `delivery-limit` для очередей кворума).


Кроме того, **для самого первого события помечания как недоставленного добавляются следующие шесть аннотаций сообщения** AMQP 1.0 или заголовков AMQP 0.9.1:
- **`x-first-death-queue`** — первая очередь, из которой это сообщение было помечено как недоставленное;
    
- **`x-first-death-reason`** — почему это сообщение было впервые помечено как недоставленное;
    
- **`x-first-death-exchange`** — обменник, в который это сообщение было опубликовано до того, как оно было впервые помечено как недоставленное;
    
- **`x-last-death-queue`** — последняя очередь, из которой это сообщение было помечено как недоставленное;
    
- **`x-last-death-reason`** — почему это сообщение было в последний раз помечено как недоставленное;
    
- **`x-last-death-exchange`** — обменник, в который это сообщение было опубликовано до того, как оно было в последний раз помечено как недоставленное.

Аннотации **`x-first-*`** ==никогда не изменяются==. Всякий раз, когда сообщение впоследствии помечается как недоставленное, аннотации **`x-last-*`** обновляются.
