В AMQP 0-9-1 **exchange** — это объект внутри RabbitMQ, который принимает сообщения от издателя и по правилам маршрутизации направляет их в очереди.

Все элементы топологии (exchanges, queues, bindings) существуют **в пределах виртуального хоста (Virtual Host, vhost)**.

---
## Что такое Virtual Host (vhost)?
**Virtual Host (виртуальный хост)** — это _логически изолированная область_ внутри RabbitMQ.  
Можно представить vhost как **отдельный мини-сервер RabbitMQ внутри большого сервера**.

В каждом vhost свои:
- очереди
- обменники
- привязки
- права доступа пользователей

То есть **одни и те же имена** ресурсов в разных vhost — **не конфликтуют**.

---
## Системные обменники (system exchanges)
При создании vhost RabbitMQ **автоматически создаёт несколько встроенных (system) обменников**, например:
- `amq.direct`
- `amq.topic`
- `amq.fanout`
- `amq.match`
- `amq.headers`
- и **default exchange** с пустым именем `""`.

Это служебные обменники, которые всегда существуют и обеспечивают базовую функциональность маршрутизации.

## Отличие
### Virtual Host — это:
- изолированное **пространство ресурсов** (очереди, exchange, bindings)
- единица управления доступом
- механизм разделения окружений/приложений

### System Exchanges — это:
- автоматически **созданные обменники в каждом vhost**
- служебные и базовые exchange (direct/topic/fanout/headers)
- особенно важный — default exchange `""`, привязанный ко всем очередям

---
## Предварительно объявленные обменники (Pre-Declared Exchanges)
### Согласно спецификации AMQP 0-9-1  
Согласно спецификации AMQP 0-9-1, **каждый виртуальный хост содержит ряд предварительно объявленных обменников:**
1. **Обменник по умолчанию (default exchange)**
    
2. **Ряд обменников `amq.*`**, по одному каждого типа, например `amq.fanout` или `amq.topic`
    

**Системные обменники (System Exchanges)**
1. **`amq.rabbitmq.log`** — системный топик-обменник, используемый **опциональной функцией логирования**
    
2. **`amq.rabbitmq.event`** — системный топик-обменник, используемый **механизмом внутренних событий**

---
# 1. Имена обменников (Exchange Names)
Каждый обменник должен иметь имя.

Один специальный обменник, называемый **обменником по умолчанию (default exchange)** (см. ниже), использует **пустую строку** в качестве имени, когда он указывается в качестве цели публикации [[1. - Основы Publisher в Rabbit|1. - Основы Publisher в Rabbit]]

В других контекстах, например, в управлении правами доступа, тот же обменник может обозначаться как **"amq.default"**, но это имя **нельзя использовать при публикации сообщений.**

---
# 2. Типы обменников
Обменники могут быть разных типов. **Тип обменника определяет, как он маршрутизирует сообщения, опубликованные в него.** Например, один тип обменника может использовать маршрутизацию на основе топика (шаблона), а другой может направлять все сообщения во все привязанные очереди без условий.

RabbitMQ поставляется с несколькими типами обменников:
1. **Fanout** (рассылка): рассмотрено в tutorial 3
2. **Topic** (топик): рассмотрено в tutorial 5
3. **Direct** (прямой): рассмотрено в tutorial 4
4. **Default direct exchange** (прямой обменник по умолчанию): встроенный прямой обменник с особыми характеристиками
    
5. **Local Random** (локальный рандомный)
6. **JMS Topic** (JMS топик)
7. **Consistent Hashing exchange** (обменник с согласованным хешированием)
8. **Random exchange** (случайный обменник)
9. **Recent history exchange** (обменник с историей)
    
10. **Headers** (по заголовкам)

Типы обменников могут предоставляться плагинами.

---
## **Fanout** (Рассылка)
Fanout обменники **направляют копию каждого опубликованного в них сообщения в каждую очередь, поток или обменник, привязанный к нему. Ключ маршрутизации сообщения полностью игнорируется.**

См. tutorial 3, чтобы узнать, как используется этот тип обменника. ([RabbitMQ Tutorials | RabbitMQ](https://www.rabbitmq.com/tutorials))

---
## **Topic** (Топик)
Topic обменники используют **сопоставление шаблонов** ключа маршрутизации сообщения с шаблоном ключа маршрутизации (привязки), использованным при привязке.

Для целей маршрутизации ключи разделяются на сегменты с помощью `.`. 

Некоторые сегменты заполняются конкретными значениями, а другие — **подстановочными знаками**:
- `*` — для **ровно одного** сегмента
    
- `#` — для **нуля или более** (включая несколько) сегментов
    

**Например:**
1. Шаблон привязки (ключ маршрутизации) `"regions.na.cities.*"` будет соответствовать ключам маршрутизации сообщений:
    - `"regions.na.cities.toronto"` ✓
        
    - `"regions.na.cities.newyork"` ✓
        
    - Но НЕ `"regions.na.cities"` ❌, потому что `*` — это подстановочный знак, который соответствует **ровно одному** сегменту
        
2. Шаблон привязки `"audit.events.#"` будет соответствовать:
    - `"audit.events"` ✓
        
    - `"audit.events.users.signup"` ✓
        
    - `"audit.events.orders.placed"` ✓
        
    - Но НЕ `"audit.users"` ❌, потому что второй сегмент не совпадает
        
3. Шаблон привязки `"#"` будет соответствовать **любому** ключу маршрутизации и заставляет topic exchange вести себя как fanout для привязок, использующих такой шаблон.

См. tutorial 5, чтобы узнать, как используется этот тип обменника. ([RabbitMQ Tutorials | RabbitMQ](https://www.rabbitmq.com/tutorials))

---
## **Direct** (Прямой)
Direct обменники направляют сообщения в одну или несколько привязанных очередей, потоков или обменников, используя **точное совпадение** ключа маршрутизации привязки.

**Например,** ключ привязки (маршрутизации) `"abc"` будет соответствовать `"abc"` и **только `"abc"`**.

См. tutorial 4, чтобы узнать, как используется этот тип обменника. ([RabbitMQ Tutorials | RabbitMQ](https://www.rabbitmq.com/tutorials))

---
## **Default Exchange** (Обменник по умолчанию)
Default exchange — это прямой обменник, который имеет несколько особых свойств:
1. Он **всегда существует** (предварительно объявлен)
    
2. Его имя для клиентов AMQP 0-9-1 — **пустая строка (`""`)**
    
3. Когда очередь объявляется, RabbitMQ **автоматически привязывает эту очередь к default exchange**, используя ее (очереди) имя в качестве ключа маршрутизации

Это предоставляет приложениям AMQP 0-9-1 механизм, который позволяет удобно публиковать сообщения **"напрямую" в очередь**, используя только ее имя, хотя под капотом все еще участвует прямой обменник.

> **Совет:** Default exchange используется из-за своих особых свойств. 
> 
> Он **не предназначен для использования как "обычный" обменник**, для которого приложения явно создают привязки.

Для случаев, когда необходим прямой обменник и пользовательская топология, рассмотрите возможность объявления и использования отдельного прямого обменника.

## Local Random Exchange (Локальный случайный обменник)
**Local Random Exchange** — новый тип обменника в RabbitMQ 4.0, *созданный для сценариев* **запрос-ответ (RPC)**. 

> Его особенность:  
> ==Сообщение всегда доставляется в локальную очередь на том же узле, где находится издатель.== 
> 
> Это **минимизирует задержку**, потому что сообщение **не нужно пересылать по кластеру**.

Если таких локальных очередей несколько — **обменник выбирает одну случайным образом**.

### **Мотивация**
> **Совет:**  
> Тип обменника "Local Random Exchange" предназначен для совместного использования с **эксклюзивными очередями (exclusive queues)**, чтобы обеспечить комбинацию с меньшей задержкой для рабочих нагрузок типа запрос-ответ ("RPC").

> **Важно:**  
> Конструкция этого типа обменника предполагает, что **количество потребителей запрос-ответ** (**не меньше**) количества узлов в кластере. 
> 
> Если это не так, некоторые опубликованные сообщения будут потеряны.

### RPC через RabbitMQ + Local Random Exchange (LRE)
Запрос/ответ, часто называемый RPC, — это популярный паттерн для реализации с помощью брокера сообщений, такого как RabbitMQ. 

Один из распространенных случаев использования — это архитектура на основе микросервисов, где один сервис запрашивает данные у другого.

#### Ключевые требования для поддержки такого сценария включают:
##### **1. Низкая задержка**
Издателю важно быстро получить ответ.  
Обычные ***обменники могут отправить сообщение в очередь на другом узле*** → появляется сетевой хоп и задержка.  
**LRE всегда доставляет сообщение только в локальную очередь**, что минимизирует задержку.  
Если локальных очередей нет — сообщение теряется.
##### **2. Масштабируемость**
Добавляя новые экземпляры сервиса (новые очереди), ты автоматически увеличиваешь пропускную способность — LRE(Local Random Exchange) рандомно распределяет запросы по всем привязанным локальным очередям.  
Обычно используют **эксклюзивные очереди**, чтобы каждая очередь гарантированно находилась на том же узле, что и потребитель.
##### **3. Нет межузловой пересылки**
Связка LRE + эксклюзивные очереди гарантирует, что сообщение публикуется и потребляется **на одном узле** без пересылки через сеть.  
Это снижает задержку до минимума.

Но важно требование: 
→ **Количество RPC-потребителей должно быть ≥ количеству узлов в кластере.**  
Иначе на некоторых узлах не будет локальных очередей, и сообщения туда просто не доставятся.

# 3. Свойства обменников (Exchange Properties)
## 1. Долговечность (Durability)  
Обменники могут быть **durable** или **transient**.  
Transient используются крайне редко и планируются к удалению в будущих версиях.  

**Эмпирическое правило:** Рассмотрите возможность использования долговечных обменников по следующим причинам:

1. Обычно всегда создают **durable exchange**, потому что приложения редко нуждаются во временных пользовательских обменниках и используют готовые (например, `amq.topic`).

2. **В дальнейшем в серии 4.x поддержка временных (не долговечных) сущностей будет удалена**, когда Khepri станет единственной поддерживаемой системой хранения метаданных.

## **2. Автоудаление (Auto-Deletion)**  
Auto-delete exchange удаляется, когда исчезает его **последняя привязка**.  
Если обменник никогда не привязывался — он не будет удалён.

## **3. Необязательные аргументы (Optional Arguments / x-arguments)**  
Это словарь ключ–значение, задаваемый при объявлении обменника.  

Используются для расширенных функций, например:
- `alternate exchange`
- `headers exchange`
- возможные расширения через плагины

Эти аргументы также можно задавать **политиками (policies)** как предпочтительный способ — без изменения кода приложения.

### Способы задания x-arguments
Способы задания x-arguments:
- через политики (рекомендуется) — на группу exchange'ов
    
- при объявлении exchange клиентом — когда параметр должен быть известен заранее

## Приоритет аргументов
1. **Операторская политика (operator policy)** — максимальный приоритет
    
2. **Аргументы клиента** — выше, чем у обычных политик
    
3. **Обычные политики (policy)**

Для числовых параметров операторские политики навязывают **верхние пределы**:  
клиент может задать меньшее число, но не большее.

# Exchange-to-Exchange bindings, Alternate Exchanges и System Exchanges
## 1. **Привязки обменник-к-обменнику (Exchange-to-Exchange Bindings)**
Помимо очередей и потоков, **обменник может быть привязан к другому обменнику** с использованием расширения протокола AMQP 0-9-1 в RabbitMQ, называемого **привязками обменник-к-обменнику (exchange-to-exchange bindings)**, или сокращенно **E2E**.

Приложения затем публикуют сообщения в **исходный обменник (source exchange)**, который маршрутизирует сообщения в **обменник-назначение (destination exchange)**.

### Нюансы
Важно:
- Сообщение **не перепубликуется** — маршрутизация происходит _однократно_ через объединённый набор правил всех связанных обменников.
    
- Ingress-метрики _destination exchange_ **не увеличиваются**, поскольку фактически сообщение туда не «приходит».
    
- Метрики очередей/стримов увеличиваются как обычно.

E2E используется для сложных схем маршрутизации, агрегации нескольких exchange'ов или построения модульных топологий.

### Пример
Используйте метод `Channel#exchangeBind`. 

В следующем примере обменник "destination" привязывается к обменнику "source" с ключом маршрутизации "routingKey".
```java
Channel ch = conn.createChannel();
ch.exchangeBind("destination", "source", "routingKey");
```

## 2. Альтернативные обменники (Alternate Exchanges, AE)
Если exchange **не смог маршрутизировать сообщение** (нет подходящих binding'ов), он может передать сообщение на **альтернативный exchange**.  

Используется для:
- сбора всех _unroutable_ сообщений;
    
- миграции топологии;
    
- аудита/логирования.

### Определение альтернативного обменника
**Альтернативный обменник (AE)** можно назначить для любого обменника, ==кроме _exchange по умолчанию_==.  

> Рекомендуется задавать AE через **политику (policy)** — это упрощает управление и изменение настроек.

**Exchange по умолчанию** — особый механизм, встроенный в RabbitMQ, поэтому **он не поддерживает AE**.

AE также можно задать как **аргумент при объявлении обменника**.  
Если и политика, и аргументы указывают AE, **приоритет имеют аргументы**.

#### Настройка альтернативного обменника через политику (рекомендуется)
Добавьте ключ **`alternate-exchange`** в policy и примените её к нужным обменникам:
```bash
rabbitmqctl set_policy AE "^my-direct$" '{"alternate-exchange":"my-ae"}' --apply-to exchanges
```
Так AE _my-ae_ будет назначен обменнику _my-direct_. Политику можно задать и через UI.

#### Настройка через аргументы клиента (не рекомендуется)
При объявлении обменника клиент может указать аргумент **`alternate-exchange`** в методе `exchange.declare()`, но этот способ хуже, чем policy.

Для работы требуется:
- право **configure** на объявляемый обменник,
- право **read** на него,
- право **write** на AE.

Лучше использовать политики.

Пример
```java
Map<String, Object> args = new HashMap<String, Object>();
args.put("alternate-exchange", "my-ae");

channel.exchangeDeclare("my-direct", "direct", false, false, args);
channel.exchangeDeclare("my-ae", "fanout");

channel.queueDeclare("routed");
channel.queueBind("routed", "my-direct", "key1");

channel.queueDeclare("unrouted");
channel.queueBind("unrouted", "my-ae", "");
```
В приведенном выше фрагменте кода на Java мы создаем прямой обменник (direct) 'my-direct', для которого настроен AE с именем 'my-ae'. 
Последний объявляется как веерный обменник (fanout). Мы привязываем очередь 'routed' к 'my-direct' с ключом привязки 'key1' и очередь 'unrouted' к 'my-ae'.

## 3. Системные обменники (System Exchanges)
RabbitMQ имеет несколько встроенных exchange'ов:
- `amq.rabbitmq.log` — логирование RabbitMQ
    
- `amq.rabbitmq.event` — внутренние события (через плагин)
    
- `amq.rabbitmq.trace` — трассировка сообщений
    

Они позволяют собирать логи, события и трейсинг, используя обычный AMQP-клиент.

# 4. Bindings
**Привязка (binding)** — это правило маршрутизации, связывающее 
- **обменник (exchange)** с
	- **очередью**, 
	- **потоком (stream)** 
	- или другим **обменником**.  

Без привязок обменник — это просто пустая таблица маршрутизации.

## Зачем нужны привязки?
Чтобы сообщить обменнику: _куда и при каких условиях отправлять сообщения_.

## Свойства привязки
Каждая привязка содержит:
- **Source name** — имя источника (exchange)
    
- **Destination name** — имя назначения (queue, stream или другой exchange)
    
- **Destination type** — `queue` или `exchange`
    
- **Arguments Map** (опционально) — используются некоторыми типами обменников (например, headers)

## Долговечность (Durability)
Привязка **наследует долговечность** от обменника и назначения:
### Привязки бывают:
1. **Полностью долговечными (Fully durable):** и их обменник, и назначение (очередь/поток) являются долговечными
    
2. **Полудолговечными (Semi-durable):** на практике это означает, что их обменник долговечен, а назначение (очередь/поток) — временное
    
3. **Полностью временными (Fully transient):** и их обменник, и назначение (очередь/поток) являются временными

> В будущем RabbitMQ удалит поддержку полудолговечных и временных привязок.

#### ВАЖНЫЙ НЮАНС
Если узел с **нереплицированными (classic)** _временными очередями_ выключается:
- временные очереди удаляются
    
- полудолговечные привязки тоже удаляются
    
- при перезапуске удаление и повторное создание привязок может идти параллельно → **несогласованная маршрутизация** 

Это может приводить к странным ошибкам маршрутизации.
