# 1. Развёртывание кластера
Используя документацию 
[Clustering Guide | RabbitMQ](https://www.rabbitmq.com/docs/clustering)

Можно узнать, как правильно создать кластер RabbitMq
Для этого нам необходимо
- Разные имена нод в формате префикс@имя_хоста
- Одинаковый cookie файл

Это описано в разделах
- Node Names (Identifiers)
- How CLI Tools Authenticate to Nodes (and Nodes to Each Other): the Erlang Cookie

А для объединения в кластер, будем использовать CLI утилиту - `rabbitmqctl`

## Docker-compose
```yaml
version: "3.5"  
  
services:  
  rabbit1-service:  
    image: rabbitmq:3-management  
    container_name: rabbitmq-container-1  
    hostname: rabbit1  
    ports:  
      - "5671:5672"  
      - "15671:15672"  
    environment:  
      RABBITMQ_NODENAME: rabbit@rabbit1  
      RABBITMQ_ERLANG_COOKIE: "secret_cookie"  
  
  rabbit2-service:  
    image: rabbitmq:3-management  
    container_name: rabbitmq-container-2  
    hostname: rabbit2  
    ports:  
      - "5672:5672"  
      - "15672:15672"  
    environment:  
      RABBITMQ_NODENAME: rabbit@rabbit2  
      RABBITMQ_ERLANG_COOKIE: "secret_cookie"
```

> [!IMPORTANT] Обязательный `hostname`
> Так как имя ноды RabbitMq описывается как `prefix@hostname`, то проще всего указать явно hostname в docker-compose
> (localhost не подойдет, так как это адресация к самому себе, а не к другому контейнеру)

В переменных окружения мы указываем:
- имя ноды
- значение куки (хотя лучше через volume и сделать общее значение)

> **РЕКОМЕНДАЦИЯ**
> Вообще рекомендуется использовать НЕЧЕТНОЕ количество узлов, так как используется протокол Raft, который составляет кворум по формуле
> `Кворум = N/2+1`
> где N - число узлов
> 
> В нашем случае при 2 узлах, получится что они же и будут в составе кворума. И если одна из нод упадет, то система останавливается


> [!NOTE] Про Кворум
> Представьте совет из 5 человек, который принимает решения большинством голосов. **Кворум = 3 человека** (большинство от 5). 
> 
> Если на заседании присутствуют только 2 человека, решения не могут быть приняты — нет кворума.


## Присоединение к кластеру
Теперь присоединим ноду к кластеру 

> [!NOTE] ПРАВИЛО
> Перед тем как присоединить ноду к кластеру, ее необходимо 
> - остановить
> - сбросить данные (для чистого присоединения)
> И потом только подключить к кластеру
> 
> При этом, **ЭТО ДЕЛАЕТСЯ на ноде, К КОТОРОЙ ПРИСОЕДИНЯЕМСЯ (от этого зависит имя кластера)**
> 
> В версиях 4.1.0 команда `rabbitmqctl join_cluster` делает это автоматически

Так как мы используем образ 3, то нам необходимо выполнить эту процедуру с остановкой и сбросом
Мы будем присоединять вторую ноду к первой. Поэтому подготовим первую ноду для присоединения

```bash
rabbitmqctl stop_app
rabbitmqctl reset

rabbitmqctl join_cluster rabbit@rabbit2

rabbitmqctl start_app
```

### Проверка статуса кластера
```bash
rabbitmqctl cluster_status
```

![[Pasted image 20251223203201.png]]
Как видно из скрина, то мы видим кластер RabbitMq, с именем первого контейнера. А также участников кластера - активные ноды

Также если посмотреть чуть ниже, то видны порты, которые слушают ноды
![[Pasted image 20251223203554.png]]
- 4369: [epmd](https://www.rabbitmq.com/docs/networking#epmd), демон для поиска помощников, используемый узлами RabbitMQ и инструментами CLI
	
- 6000–6500: используется для репликации [потока RabbitMQ](https://www.rabbitmq.com/docs/stream)
	
- 25672: используется для связи между узлами и инструментами CLI (порт распределительного сервера Эрланга) и выделяется из динамического диапазона (по умолчанию ограничен одним портом), вычисляется как AMQP-порт + 20000). Если только внешние соединения на этих портах не действительно необходимы (например, кластер использует [федерацию](https://www.rabbitmq.com/docs/federation) или инструменты CLI применяются на машинах вне подсети), Эти порты не должны быть публично раскрыты. Подробности смотрите [в руководстве по нетворкингу](https://www.rabbitmq.com/docs/networking).
	
- 35672-35682: используется инструментами CLI (клиентские порты распределения Erlang) для связи с узлами и выделяется из динамического диапазона (вычисляется как распределительный порт сервера + 10000 через Порт распределения серверов + 10010).

А именно интересен порт 25672, который используется для связи между узлами и инструментами CLI
