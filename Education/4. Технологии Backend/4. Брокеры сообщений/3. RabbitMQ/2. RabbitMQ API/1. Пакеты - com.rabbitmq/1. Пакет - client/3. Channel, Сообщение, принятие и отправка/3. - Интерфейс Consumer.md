
> [!NOTE] Интерфейс `Consumer`
> представляет собой **ОБЪЕКТ-ОБРАТНЫЙ ВЫЗОВ**, который получает **уведомления и сообщения** из очереди, на которую подписан через `Channel.basicConsume(...)`

Он **не является функциональным интерфейсом Java 8** — это обычный интерфейс с несколькими методами, которые вызываются клиентом **в отдельном потоке исполнения**, отличном от основного потока соединения

Важный момент: интерфейс **не реализует** ни `AutoCloseable`, ни `Closeable` — он не связан напрямую с механизмами закрытия через `try-with-resources`

### Подинтерфейсы и реализации
#### **Подинтерфейсы:**
- `RpcServer.RpcConsumer` — специализированный подинтерфейс для RPC-серверов (детали в API). 

#### **Известные реализующие классы:**
- `DefaultConsumer` — наиболее распространённый базовый класс для реализации `Consumer`

# Методы
## События регистрации и отмены
```java
// Called when the consumer is registered by basicConsume.
// Вызывается при успешной регистрации потребителя.
void handleConsumeOk(String consumerTag);

// Called when the consumer is cancelled due to basicCancel.
// Вызывается при отмене потребителя через basicCancel.
void handleCancelOk(String consumerTag);
```
💬 Эти методы используются для подтверждения того, что служба потребления была успешно зарегистрирована или отменена по вашему запросу.

## Отмена по причине брокера
```java
// Called when the consumer is cancelled for reasons other than basicCancel.
// Вызывается, если потребитель отменён брокером (например, очередь удалена).
void handleCancel(String consumerTag) throws IOException;
```
💬 В отличие от `handleCancelOk`, этот метод сигнализирует о **без вашего прямого участия** закрытии подписки.

## Сигналы Shutdown
```java
// Called when the channel or connection is shut down.
// Вызывается, когда канал или соединение закрываются.
void handleShutdownSignal(
	String consumerTag, 
	ShutdownSignalException sig);
```
💬 Позволяет обработать неожиданные или программные закрытия канала/соединения.

## Восстановление (recovery)
```java
// Called when basic.recover-ok is received after a basic.recover.
// Вызывается после восстановления доставок сообщений.
void handleRecoverOk(String consumerTag);
```
💬 Используется в механизмах автоматического восстановления.

## Доставка сообщений
```java
// Called when a basic.deliver is received.
// Вызывается при получении сообщения.
void handleDelivery(String consumerTag,
                    Envelope envelope,
                    AMQP.BasicProperties properties,
                    byte[] body) throws IOException;
```
💬 Основной метод: сюда приходят данные сообщения, свойства и метаданные.

# 💡 Особенности и подводные камни 💡
1️⃣ **Методы могут быть вызваны из пул потоков клиента**, а не из того же потока, где вы создали `Channel` — это может влиять на синхронизацию. [rabbitmq.github.io](https://rabbitmq.github.io/rabbitmq-java-client/api/current/com/rabbitmq/client/Consumer.html?utm_source=chatgpt.com)  

2️⃣ `handleDelivery` реализует логику обработки, но **само подтверждение (ack/nack)** должно быть вызвано вручную, если `autoAck=false`. [rabbitmq.github.io](https://rabbitmq.github.io/rabbitmq-java-client/api/current/com/rabbitmq/client/Consumer.html?utm_source=chatgpt.com)  

3️⃣ В `handleCancel` нужно корректно освобождать ресурсы, если очередь была удалена. [rabbitmq.github.io](https://rabbitmq.github.io/rabbitmq-java-client/api/current/com/rabbitmq/client/Consumer.html?utm_source=chatgpt.com)  

4️⃣ Ошибки внутри `handleDelivery` **могут привести к закрытию канала**, если не обработаны. [rabbitmq.github.io](https://rabbitmq.github.io/rabbitmq-java-client/api/current/com/rabbitmq/client/Consumer.html?utm_source=chatgpt.com)  

5️⃣ Если используется **автоматическое восстановление**, потребитель будет восстановлен, и будет вызван `handleRecoverOk`. [rabbitmq.github.io](https://rabbitmq.github.io/rabbitmq-java-client/api/current/com/rabbitmq/client/Consumer.html?utm_source=chatgpt.com)

# 💎 Best Practices 💎
✅ Обычно **наследуют `DefaultConsumer`**, а не реализуют `Consumer` напрямую — это избавляет от необходимости реализовывать все методы вручную. [rabbitmq.github.io](https://rabbitmq.github.io/rabbitmq-java-client/api/current/com/rabbitmq/client/Consumer.html?utm_source=chatgpt.com)  

✅ В `handleDelivery` лучше **не выполнять долгие блокирующие операции** — иначе это задержит доставку других сообщений в том же канале. [rabbitmq.github.io](https://rabbitmq.github.io/rabbitmq-java-client/api/current/com/rabbitmq/client/Consumer.html?utm_source=chatgpt.com)  

✅ Всегда обрабатывать ошибки внутри `handleDelivery` и других callback’ов, чтобы не допустить закрытия канала. [rabbitmq.github.io](https://rabbitmq.github.io/rabbitmq-java-client/api/current/com/rabbitmq/client/Consumer.html?utm_source=chatgpt.com)  

✅ Для простых случаев используйте лямбда-ориентированные API (`DeliverCallback`, `CancelCallback`) вместо прямой реализации `Consumer`.

# Пример использования
```java
Channel channel = connection.createChannel();

Consumer consumer = new DefaultConsumer(channel) {
    @Override
    public void handleDelivery(String consumerTag,
                               Envelope envelope,
                               AMQP.BasicProperties properties,
                               byte[] body)
            throws IOException 
    {
        String message = new String(body, StandardCharsets.UTF_8);
        System.out.println("Received: " + message);
        // подтверждаем вручную
        channel.basicAck(envelope.getDeliveryTag(), false);
    }
};

channel.basicConsume("my.queue", false, consumer);

```
