
> [!NOTE] Docker Compose — это инструмент, который `упрощает работу с` `многоконтейнерными` Docker-приложениями.  
> 
> Он позволяет `описать всю инфраструктуру приложения (контейнеры, сети, тома, переменные окружения и т.д.`) `в одном YAML-файле`, а затем **одной командой запустить всё приложение**.
> 
> ⭐==Описывается в YAML-файле==: `docker-compose.yml`
> 

## 🟡 Структура YAML файла 🟡
### **Основные элементы синтаксиса** 
```yaml
# Комментарии начинаются с решетки

key: value  # Пары ключ-значение

list_item:  # Списки
  - item1
  - item2
  - item3
    
nested_structure: # Вложенные структуры (словарь)
  key1: value1
  key2: value2
```

### **Типы данных**
```yaml
string: "Hello World"        # Строка
number: 42                   # Число
float: 3.14                  # Дробное число
boolean: true                # Логическое значение
null_value: null             # Пустое значение
date: 2024-01-15             # Дата
```

### **Многострочные строки**
```yaml
single_line: "Одна строка"
multiline: |                 # Сохраняет переносы строк
  Первая строка
  Вторая строка
  Третья строка
folded: >                    # Сворачивает переносы в пробелы
  Это длинная строка
  которая будет свернута
```

## Зачем нужен Docker Compose? 

Без Compose тебе пришлось бы:
- ❗вручную запускать каждый контейнер с кучей аргументов `docker run ...`;
- ❗помнить порты, имена сетей, тома, зависимости между контейнерами;
- ❗перезапускать всё вручную при изменениях.

Compose решает это:
- ✅хранит всю конфигурацию в **`docker-compose.yml`**;
- ✅позволяет управлять проектом **одной командой** (`docker compose up`);
- ✅автоматически ==❗❗СОЗДАЕТ  **СЕТЬ** ❗❗между контейнерами и даёт им имена==;
- ✅упрощает **развертывание**, **отладку** и **тестирование** многосервисных систем.

### Как это работает:
Ты создаёшь файл **`docker-compose.yml`**, в котором описываешь:
- какие `образы` (или Dockerfile) использовать;
- какие `порты` пробрасывать;
- какие `тома` монтировать;
- какие `переменные` окружения нужны;
- `зависимости` между контейнерами (`depends_on`).

Запускаешь:
```bash
docker compose up
```
💡Compose сам:
- `создаст сеть` для всех контейнеров;
- `поднимет каждый сервис` в нужном порядке;
- выведет `общие логи` в одном потоке.

# 📕 1. Структура docker-composer.yml
## 📙 Пример 1 - Один контейнер
Возьмем пример запуска контейнера из [[4. - Приложение для записи-хранения текстовых файлов + Volumes + сборка war]]

Если раньше нам приходилось каждый раз вводить эту команду для сборки:
```bash
docker run 
--name java-servlets-container
-v D:\volum_docker:/tmp 
--env SAVE_FILE_PATH=/tmp/ 
-p 7070:8080  
java-servlets-image
```
Хотя возможны другие более ОБЪЕМНЫЕ КОМАНДЫ ЗАПУСКА


То в `docker-compose.yml` это можно записать следующим образом
```yaml
version:"3.5"
services:
	my-servlets-service:
		image: java-servlets-image
		container_name: java-servlets-container
		
		volumes:
			- D:\volum_docker:/tmp
		
		environment:
			- SAVE_FILE_PATH: /tmp/
		
		ports:
			-"7070:8080"
			
		restart: unless-stopped # always/no/on-failure
```
### Пояснение содержимого
#### 🔶Шапка docker-compose
##### version:"3.5"
- Указывает версию формата docker-compose файла
##### services
- Секция, содержащая описание сервисов (контейнеров)

#### 🔶Сервис: my-servlets-service
- Имя сервиса в docker-compose

- `image: java-servlets-image` - использует образ с именем "java-servlets-image"
- `container_name: java-servlets-container` - задаёт конкретное имя контейнеру

##### volumes
- `- D:\volum_docker:/tmp` - монтирует директорию с хоста в контейнер:
    - `D:\volum_docker` - директория на хост-машине (Windows)
    - `/tmp` - точка монтирования внутри контейнера

##### environment
- `- SAVE_FILE_PATH=/tmp/` - устанавливает переменную окружения
    
- Приложение будет сохранять файлы по пути `/tmp/` внутри контейнера

##### ports
- `- "7070:8080"` - пробрасывает порты:
    
    - `7070` - порт на хост-машине
        
    - `8080` - порт внутри контейнера (обычно порт Tomcat/Jetty)
##### restart: unless-stopped
- Политика перезапуска контейнера:
    - `unless-stopped` - при перезагрузки сервера, контейнер ==БУДЕТ ИМЕТЬ ТО ЖЕ СОСТОЯНИЕ что и ДО ПЕРЕЗАГРУЗКИ СЕРВЕРА== (был остановлен, значит и будет остановлен)

## 📙 Пример 2 - Несколько контейнеров
Допустим у нас приложение состоит из нескольких сервисов
- Backend: php
- Database: postgres
- Cache: redis

```bash
# 🟢Серсив PHP
docker run
--name app
-p 80:80
--net appnet
php: apache

# 🟢Серсив Postgres
docker run
--name app-postgres
-e POSTGRES_PASSWORD=mypas
--net appnet
postgres

# 🟢Серсив Redis
docker run
--name app-redis
--net appnet
redis
```

Логичнее будет разделить сети так, чтобы пользователи могли получить доступ только напрямую к `PHP`-приложению, а к БД нет. 
Поэтому, поместим сервис приложения PHP в Docker-сеть `internet`, а БД и кеш в сеть appnet

![[Pasted image 20251024093147.png]]


Теперь опишем `dsocekr-compose.yml`
```yml
version:"3.5"

services:
	php-app:
		image: php:apache
		container_name: app
		ports:
			- "80:80"
		restart: unless-stopped
		
		depends_on:
			- app-db
			- app-redis
			  
		network:
			- internet
			- appnet

	app-db:
		image: postgres
		container_name: app-postgres
		restart: unless-stopped
		environment:
			- POSTGRES_PASSWORD: mypass
		network:
			- appnet
	
	app-redis:
		image: redis
		container_name: app-redis
		restart: unless-stopped
		network:
			- appnet

network:
	internet:
		name: internet
		driver: bridge
	appnet:
		name: appnet
		driver: bridge
```

## depends_on
Когда сервисы зависят друг от друга, ВАЖЕН ПОРЯДОК ИХ ЗАПУСКА. Так как сервисы мы упаковываем в контейнеры, то нужно запустить контейнеры в правильном порядке.

> [!NOTE] ⭐⭐`depends_on` - указывает зависимость от других сервисов⭐⭐
> При помощи `depends_on`, можно указать БЕЗ КАКИХ ГОТОВЫХ СЕРВИСОВ, НЕЛЬЗЯ ЗАПУСТИТЬ ЭТОТ СЕРВИС

# 📕 2. Утилита docker compose

> [!NOTE] ⭐⭐Все команды выполняются из папки, где лежит `docker-compose.yml`.⭐⭐
> 

## 🔶 **1. Запуск и остановка**

```css
docker compose up
```

> Запускает все сервисы, указанные в файле, и показывает логи.  
> При первом запуске автоматически создаёт образы, сети и тома.

---
С флагами:

```bash
docker compose up -d
```
> Запуск в фоновом режиме (detached mode).

---

```bash
docker compose down
```
> Останавливает и **==удаляет==** все контейнеры, сети и временные тома, созданные `up`

---
## 🔶**2. Управление контейнерами**

```bash
docker compose start 
docker compose stop 
docker compose restart
```
> Запускают, останавливают или перезапускают контейнеры без пересоздания.

---

```bash
docker compose ps
```
> Показывает список контейнеров, созданных этим проектом (состояние, порты и т.д.)

---

```bash
docker compose exec <service> <command>
```
> Выполняет команду внутри контейнера (как `docker exec`).

Пример:
```bash
docker compose exec app bash
```

---

```bash
docker compose logs 
docker compose logs -f app
```
> ==Просмотр ЛОГОВ== всех сервисов или конкретного (с `-f` — потоковый вывод).

---

```bash
docker compose up -d --scale service_name=number_of_containers
```
> ==Масштабировать сервисы== до определенного количества контейнеров

Пример 
```bash
docker compose up -d --scale web=3
```

---

```bash
docker compose pause service_name # Приостоновить сервис
docker compose unpause service_name # Возобновить сервис
```
>==Приостановить/Возобновить== определенный контейнер

---

## 🔶**3. Сборка и обновление**

```css
docker compose build
```
> Собирает образы из `Dockerfile`, указанных в `build:` секциях.

---

```bash
docker compose pull
```
> Загружает свежие образы из Docker Hub.

---

```bash
docker compose up --build
```
> Пересобирает образы **перед запуском** — удобно при изменениях в коде.

---

## 🔶 **4. Очистка**
```bash
docker compose down -v
```
> Останавливает контейнеры и **удаляет тома**.

---

```bash
docker compose rm
```
> Удаляет остановленные контейнеры (если не нужны).

---

## 🔶 **5. Проверка и отладка**

```bash
docker compose config
```
> Проверяет синтаксис YAML и показывает итоговую конфигурацию после подстановки `.env`.

---

```bash
docker compose images
```
> Показывает, какие образы используются сервисами.

---

```bash
docker compose top
```
> Показывает процессы, запущенные внутри контейнеров Compose-проекта.
