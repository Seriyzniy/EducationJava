
Существует два подхода для управления нодами
- Императивный - явно указываем что  сделать (через консоль вводим команду)
- Декларативный - "что нужно сделать" описано в YAML-файле


# 📕 1. Deployment - декларативный подход

> [!NOTE] `Deployment` — это декларативный объект Kubernetes, 
> Описывает: Как должно быть развернуто и управляться приложение (Pod’ы и их версии).
> 
> ⭐Описывается в `deployment.yaml`


> [!NOTE] Такой же формат используется для описания:
> - Подов (pod.yml) (в части spec описываем контейнеры как в template)
> 


## 📙 1. Зачем нужен Deployment
❗❗Без него ты бы должен был:
- сам запускать Pod’ы вручную;
- следить, чтобы они не упали;
- удалять старые и создавать новые при обновлении версии;
- вручную масштабировать приложение.


✅Deployment делает всё это **автоматически**:
- запускает нужное количество Pod’ов (через ReplicaSet),
- обновляет их с нулевым простоем (rolling update),
- следит за состоянием (self-healing),
- позволяет откатить изменения (rollback).

## 📙 2. Структура deployment.yml
Состоит из двух частей
![[Pasted image 20251027194701.png]]

### 🔶 Общая структура `deployment.yaml`
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  ...
spec:
  ...
```

У манифеста Kubernetes (включая Deployment) **всегда есть три** `обязательных верхних блока`:

| Раздел       | Что описывает                                                                  |
| ------------ | ------------------------------------------------------------------------------ |
| `apiVersion` | Версия API, к которой относится ресурс <br>🧠(для Deployment всегда apps/v1)🧠 |
| `kind`       | Тип ресурса (в нашем случае — Deployment)                                      |
| `metadata`   | Метаданные (имя, метки, аннотации)                                             |
| `spec`       | Спецификация — то, _как должен работать Deployment_                            |

#### 🔶 `apiVersion`
![[Pasted image 20251028091226.png]]

#### 🔶 `kind`
```yaml
kind: Deployment
```
Тип создаваемого объекта.  

Другие варианты: `Pod`, `Service`, `ConfigMap`, `Ingress` и т.д.  
Для деплоймента — всегда `Deployment`.

#### 🔶`metadata`
Метаданные, не влияющие на логику, но важные для идентификации:
```yaml
metadata:
  name: my-deployment
  namespace: default
  labels:
    app: my-app
  annotations:
    description: "Deployment for nginx app"
```

| Поле          | Описание                                                                      |
| ------------- | ----------------------------------------------------------------------------- |
| `name`        | Имя деплоймента (уникально в namespace)                                       |
| `namespace`   | Пространство имён, где развернуть (по умолчанию — default)                    |
| `labels`      | `Метки — ключ-значение` <br>(❗используются для поиска и связывания объектов❗) |
| `annotations` | Произвольная метаинформация, не влияющая на работу                            |

#### 🔶`spec`  
Внутри `spec` описывается ==всё поведение деплоймента==:  
сколько подов, как обновлять, что именно запускать и т.д.
```yaml
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 1
  template:
    ...
```

#### 🔹 `replicas`
Количество Pod’ов, которые должно быть запущено.
```yaml
replicas: 3
```
❗❗Если один Pod упадёт — Kubernetes автоматически поднимет новый.❗❗

#### 🔹 `selector`
Описывает, **какие Pod’ы принадлежат этому Deployment**.
```yaml
selector:
  matchLabels:
    app: my-app
```
⚠️ Важно: `selector.matchLabels` должен **совпадать** с метками в `template.metadata.labels`.

#### 🔹 `strategy`
Задаёт ❗❗**стратегию обновления**❗❗ Pod’ов при изменении Deployment’а.

- Вариант 1: RollingUpdate (по умолчанию)
```yaml
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxUnavailable: 1   # максимум 1 Pod может быть недоступен
    maxSurge: 1         # максимум 1 Pod сверх нормы при обновлении
```

- Вариант 2: Recreate
```yaml
strategy:
  type: Recreate
```
Все Pod’ы удаляются, потом создаются новые (используется, если старые и новые версии конфликтуют, например из-за БД).

#### 🔹 `revisionHistoryLimit`
Сколько старых ReplicaSet хранить для возможного отката:
```yaml
revisionHistoryLimit: 5
```

#### 🔹 `progressDeadlineSeconds`
Через сколько секунд считать обновление неудачным:
```yaml
progressDeadlineSeconds: 600
```

#### 🔹 `minReadySeconds`
Минимальное время, в течение которого Pod должен быть "готов", чтобы считаться успешным:
```yaml
minReadySeconds: 10
```

#### 🔹 `paused`
Можно временно "заморозить" обновления Deployment’а:
```yaml
paused: true
```


### 🔶 `template`
Поле `template` — это **шаблон Pod’а**, из которого будут создаваться все Pod’ы этого Deployment’а.

```yaml
template:
  metadata:
    labels:
      app: my-app
  spec:
    containers:
      - name: nginx
        image: nginx:1.25
        ports:
          - containerPort: 80
        resources:
          limits:
	        # Половина от процессора (1000m - один процессор)
            cpu: "500m"
            memory: "256Mi"
          requests:
            cpu: "250m" 
            memory: "128Mi"
        env:
          - name: ENV
            value: "prod"
        volumeMounts:
          - name: html-volume
            mountPath: /usr/share/nginx/html
    volumes:
      - name: html-volume
        configMap:
          name: nginx-config
```

#### 🔹 `template.metadata`
Метки для Pod’ов (должны совпадать с `spec.selector.matchLabels`).

#### 🔹 `template.spec.containers`
Список контейнеров внутри Pod’а.  
Можно запустить несколько контейнеров в одном Pod’е (например, sidecar для логирования).

Ключи внутри контейнера:

| Поле                                              | Описание                                              |
| ------------------------------------------------- | ----------------------------------------------------- |
| `name`                                            | Имя контейнера                                        |
| `image`                                           | Docker-образ                                          |
| `ports`                                           | Открытые порты контейнера                             |
| `env`                                             | Переменные окружения                                  |
| `resources`                                       | ❗❗Лимиты и запросы ресурсов❗❗                         |
| `volumeMounts`                                    | Подключаемые тома                                     |
| `livenessProbe`, `readinessProbe`, `startupProbe` | Проверки “живости”                                    |
| `command` / `args`                                | Переопределение точки входа                           |
| `imagePullPolicy`                                 | Политика скачивания образа (`IfNotPresent`, `Always`) |

#### 🔹 `template.spec.volumes`
Определяет, какие **тома (volumes)** монтируются в Pod.

Примеры:
```yaml
volumes:
  - name: config
    configMap:
      name: nginx-config
  - name: cache
    emptyDir: {}
```

#### 💡 Дополнительные возможности `template.spec`
| Поле                        | Назначение                                                |
| --------------------------- | --------------------------------------------------------- |
| `initContainers`            | Контейнеры, которые запускаются **до основных**           |
| `affinity` / `nodeSelector` | Правила, на каких нодах размещать Pod                     |
| `tolerations`               | Исключения для taints                                     |
| `serviceAccountName`        | Какой service account использовать                        |
| `restartPolicy`             | Политика перезапуска Pod (для Deployment всегда `Always`) |
| `dnsPolicy` / `hostNetwork` | Настройки сети                                            |
| `imagePullSecrets`          | Секреты для приватных registry                            |
