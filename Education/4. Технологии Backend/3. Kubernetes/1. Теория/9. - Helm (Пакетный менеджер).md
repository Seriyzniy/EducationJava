
> [!NOTE] **Helm** — это **менеджер пакетов для Kubernetes**, 
> аналогичный тому, как `apt` используется в Ubuntu. 
> 
> Он упрощает **установку, обновление, удаление и управление приложениями** в Kubernetes, ==избавляя от необходимости вручную писать== и поддерживать сложные ==YAML-манифесты==.
> 

## Основная идея
Основная идея заключается в том, чтобы запаковать приложение в пакет (Charts), и потом с легкостью устанавливать его на различные кластера K8s

**Чарты (charts)** — это пакеты, описывающие всё, что нужно для деплоя приложения в кластер:
- Deployment
- Service
- ConfigMap 
- Secret
- Ingress
- И другие Kubernetes-ресурсы

**Структура пакета (Чарта)**
```
mychart/
├── Chart.yaml        # Метаданные чарта (имя, версия, описание)
├── values.yaml       # ⭐⭐Значения ПО УМОЛЧАНИЮ⭐⭐
├── templates/        # ❗❗Шаблоны манифестов Kubernetes❗❗
│   ├── deployment.yaml
│   ├── service.yaml
│   └── _helpers.tpl  # Вспомогательные функции и шаблоны
└── charts/           # Зависимости (другие чарты)
```

## Ключевые понятия Helm

|Понятие|Описание|
|---|---|
|**Chart**|Пакет Helm, содержащий шаблоны YAML-манифестов и файлы значений|
|**Release**|Конкретный экземпляр чарта, установленный в кластер|
|**Repository (repo)**|Хранилище, где лежат чарты (например, официальный `bitnami/`, `stable/`)|
|**Values**|Файл `values.yaml`, содержащий параметры конфигурации для шаблонов чарта|
|**Template**|YAML-файлы, в которых используются переменные и функции (Go-шаблоны)|

# 1. Порядок работы
Для начала необходимо установить Helm CLI (утилиту).

## 📙 Развертывание ГОТОВОГО чарта из репозитория
Helm использует репозитории чарта — хранилища пакетов, подобных `bitnami`, `grafana`, `prometheus-community` и др.

#### Добавление репозитория чарта
```bash
helm repo add bitnami https://charts.bitnami.com/bitnami
helm repo update
```

#### Поиск нужного если не знаем точного имени
Например, ищем PostgreSQL:
```bash
helm search repo postgresql
```

### Установка найденного чарта
Пример установки NGINX из Bitnami:
```bash
helm install my-nginx bitnami/nginx
```
==Helm создаст все Kubernetes-ресурсы== (Deployment, Service и т. д.) на основе шаблонов чарта.


Проверим:
```bash
helm list
kubectl get all
```

---
## 📙 Развертывание СОБСТВЕННОГО чарта
Если ты деплоишь **своё приложение**, а не готовый пакет:

### Создание каркаса чарта
Для начала создается пакет (Chart)
```bash
helm create myapp
```
создаёт **каркас (шаблонную структуру)** Helm-чарта.  

```
myapp/
├── Chart.yaml
├── values.yaml
├── templates/
│   ├── deployment.yaml
│   ├── service.yaml
│   ├── ingress.yaml
│   └── _helpers.tpl
```

> То есть Helm НЕ СКАНИУРЕТ твои существующие YAML-манифесты и не добавляет их автоматически — он просто ==создаёт готовую структуру== с примером приложения ==ШАБЛОН== (обычно — Nginx Deployment + Service).


### Настрой шаблоны и ==values.yaml==
В `values.yaml` ты задаёшь параметры, которые будут подставляться в шаблоны:
```yaml
image:
  repository: myregistry/myapp
  tag: "1.0.0"

service:
  type: ClusterIP
  port: 8080
```

> Можно создать собственный файл volues.yaml, и при помощи helm install -f <мой_volues.yaml> использовать его

В шаблонах (`templates/deployment.yaml`) используешь переменные:
```yaml
image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
```

### Проверка чарта перед установкой
Проверка синтаксиса:
```bash
helm lint myapp/
```

Посмотреть, какие YAML будут применены:
```yaml
helm template myapp/
```

### Установка чарта в кластер
```bash
helm install myapp ./myapp
```
Helm создаст **release** — установленный экземпляр чарта.


Проверим:
```bash
helm list
```

### Обновление приложения
Измени `values.yaml` (например, новую версию образа):
```yaml
image:
  tag: "1.1.0"
```

Применение обновления:
```bash
helm upgrade myapp ./myapp
```


---
### [Доп] Проверка состояния
```bash
helm status myapp
kubectl get pods
```

### [Доп] Откат на предыдущую версию
Если обновление неудачное:
```bash
helm rollback myapp 1
```

### [Доп] Удаление приложения
```bash
helm uninstall myapp
```

### [Доп] Упаковка чарта для распространения
```bash
helm package myapp
```

Появится архив:
```
myapp-0.1.0.tgz
```
Ты можешь опубликовать его в своём Helm-репозитории.

# Ручное добавление файлов манифестов

Ты **добавляешь файлы вручную в `templates/` только если**:
1. Тебе нужны **дополнительные ресурсы**, которых нет по умолчанию  
    (например, `ConfigMap`, `Secret`, `PersistentVolumeClaim`, `CronJob` и т.д.)
    
2. Твоя архитектура **существенно отличается** от шаблона Helm  
    (например, ты деплоишь StatefulSet, а не Deployment)
    
3. Ты хочешь **вынести специфические ресурсы** отдельно (например, отдельный `configmap.yaml` или `rbac.yaml`).

## Через папку `templates/`
Все YAML-манифесты Kubernetes нужно **ДОБАВЛЯТЬ ВРУЧНУЮ** в каталог:
```bash
mychart/templates/
```
Примеры файлов:
- `deployment.yaml`
- `service.yaml`
- `configmap.yaml`
- `secret.yaml`
- `ingress.yaml`
- и т.д.

Helm при установке (или при команде `helm template`) просто **рендерит шаблоны из этой папки** в чистые YAML-манифесты Kubernetes, подставляя значения из `values.yaml`.

#### Пример добавления собственного манифеста
Например, у тебя есть обычный манифест `configmap.yaml`:
```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: myapp-config
data:
  APP_ENV: "prod"
```

Ты можешь просто положить его в `mychart/templates/configmap.yaml` —  
и он станет частью чарта.

Helm при установке:
```bash
helm install myapp ./mychart
```
автоматически ==создаст== `ConfigMap` ==в кластере вместе с остальными ресурсами==.
