Соберем архитектуру Kubernetes
![[Pasted image 20251028105309.png]]

Представим, что я хочу задеплоить nginx в кластере Kubernetes. Я введу в терминале нечто такое:
```bash
kubectl run --image=nginx --replicas=3
```
… и нажму на Enter.

Через несколько секунд увижу 3 пода с nginx, распределённые по всем рабочим узлам. 
Работает — словно по волшебству, и это здорово! 

Но что на самом деле происходит под капотом?

---
# 📕 1. kubectl

Итак, начнём. Мы только что нажали на Enter в терминале. И что теперь?
## 📙 Валидация и генераторы

В первую очередь kubectl выполнит валидацию на стороне клиента 
- проверяет корректность команды и YAML (чтобы не отправлять в кластер заведомо неверные запросы).

Затем kubectl 
- формирует **HTTP-запрос** к **kube-apiserver**, через который проходят все изменения и обращения к состоянию кластера. API в свою очередь общается с etcd

kubectl использует **генераторы (generators)** — механизмы, преобразующие введённые параметры в конкретные runtime-объекты (например, Pod, Deployment и т.д.). 

> (в статье[[Что происходит в Kubernetes при запуске kubectl run? Часть 1 / Хабр](https://habr.com/ru/companies/flant/articles/342658/)]: генераторы - абстракцию, реализующую сериализацию)

Если явно не указан тип ресурса, kubectl определяет его автоматически:

- `--restart-policy=Always` → создаётся **Deployment**,
    
- `--restart-policy=Never` → создаётся **Pod**.
    

Разобравшись, что мы хотим создать Deployment, kubectl воспользуется генератором `DeploymentV1Beta1` для создания [runtime-объекта](https://github.com/kubernetes/kubernetes/blob/7650665059e65b4b22375d1e28da5306536a12fb/pkg/kubectl/run.go#L59) из предоставленных параметров. 

> [!NOTE] ⭐`Runtime object` — это обобщающий термин для ресурса.⭐
> 

## 📙 Группы API и «переговоры о версиях»
> Kubernetes использует **версионные API**, сгруппированные по **API groups**, чтобы упростить работу с разными типами ресурсов
> Например, у `Deployment` группа — **apps**, а версия — **v1** (раньше `v1beta2`)

После генерации runtime-объекта kubectl выполняет **“переговоры о версии” (version negotiation)** — определяет нужную группу и версию API, обращаясь к `/apis` на kube-apiserver. 

Сервер возвращает описание API в формате **OpenAPI** (эндпоинты для работы с ресурсами), чтобы клиент знал структуру ресурсов.
(см [[1. OpenApi в Kubernetes]])


Для ускорения kubectl **кэширует схемы API** в `~/.kube/schema`. При удалении этого каталога kubectl при следующем запуске снова загрузит схемы, что можно увидеть при запуске с флагом `-v`.

Затем kubectl отправляет **HTTP-запрос** к **API Server**, получает ответ и выводит сообщение об успешном создании ресурса.

## 📙 Аутентификация клиента
Перед отправкой запроса kubectl проходит этап **аутентификации**.

Учётные данные берутся из ❗❗**файла kubeconfig**❗❗, который ищется в таком порядке:
1. Указанный через `--kubeconfig`;
2. Определённый в переменной `$KUBECONFIG`;
3. По умолчанию — `~/.kube/config`.
    

Из файла выбираются текущий **контекст**, **кластер** и **пользователь**.  
Если заданы параметры (например, `--username`), они имеют приоритет.

После этого kubectl формирует HTTP-клиент с нужным способом аутентификации:

- сертификаты X.509 — через `tls.TLSConfig`;
    
- токены — в заголовке `Authorization`;
    
- логин и пароль — через basic auth;
    
- токен OpenID — также в `Authorization`.

# 📕 2. kube-apiserver

Итак, запрос был отправлен, ура! Что дальше? В дело вступает kube-apiserver.


> [!NOTE] kube-apiserver
> kube-apiserver — основной интерфейс, используемый клиентами и системными компонентами для сохранения и получения состояния кластера.

## 📙 Аутентификация
После отправки запроса его принимает **kube-apiserver**, который первым делом выполняет **аутентификацию** — проверку, кто делает запрос.

При запуске сервер формирует цепочку **аутентификаторов** на основе переданных флагов:

- `--client-ca-file` → проверка клиентских сертификатов (x509);
    
- `--token-auth-file` → проверка токенов из заголовка `Authorization`;
    
- `--basic-auth-file` → базовая HTTP-аутентификация.


Запрос последовательно проходит через все аутентификаторы, пока один из них не сработает успешно
1. [обработчик x509](https://github.com/kubernetes/apiserver/blob/51bebaffa01be9dc28195140da276c2f39a10cd4/pkg/authentication/request/x509/x509.go#L60) верифицирует, что HTTP-запрос зашифрован с TLS-ключом, подписанным корневым сертификатом удостоверяющего центра;
2. [обработчик токенов](https://github.com/kubernetes/apiserver/blob/51bebaffa01be9dc28195140da276c2f39a10cd4/pkg/authentication/request/bearertoken/bearertoken.go#L38) верифицирует, что предоставленный токен (определён в HTTP-заголовке `Authorization`) существует в файле на диске, указанном директивой `--token-auth-file`;
3. [обработчик basicauth](https://github.com/kubernetes/apiserver/blob/51bebaffa01be9dc28195140da276c2f39a10cd4/plugin/pkg/authenticator/request/basicauth/basicauth.go#L37) похожим образом убедится, что учётные данные для базовой аутентификации в HTTP-запросе соответствуют локальным данным.

❗❗Если ни один не сработает — возвращается ошибка.  

При успешной аутентификации данные о пользователе добавляются в контекст запроса, а заголовок `Authorization` удаляется — дальше запрос идёт на этап **авторизации** и **admission контроллеров**.

## 📙 Авторизация
После аутентификации **kube-apiserver** проверяет, **имеем ли мы право** выполнять запрошенную операцию — это этап **авторизации**.

Как и при аутентификации, сервер собирает цепочку **авторизаторов (authorizers)** из переданных флагов 
(команду которую мы хотим выполнить в k8s). 

Каждый запрос проходит через них по очереди:
- если **все** авторизаторы отклоняют — возвращается `Forbidden`;
- если **хотя бы один** разрешает — запрос проходит дальше.
    

Примеры авторизаторов:
- **Webhook** — проверка через внешний HTTP(S)-сервис;
- **ABAC** — политики из статического файла;
- **RBAC** — контроль доступа на основе ролей (через ресурсы Kubernetes);
- **Node** — ограничивает доступ компонентов узлов (например, kubelet) только к их собственным ресурсам.

## 📙 Контроль допуска

> [!NOTE] **Admission controllers**
> Окей, мы аутентифицированы и авторизованы kube-apiserver. Что осталось? 
> 
> Сам kube-apiserver доверяет нам и разрешает продолжить, 
> 
> НО у других частей системы в Kubernetes могут быть собственные глубокие убеждения по поводу того, что разрешено, а что — нет.

После авторизации запрос проходит через **admission controllers** — последний уровень проверки перед записью объекта в **etcd**.

Они проверяют, **соответствует ли запрос политикам и ограничениям кластера**, предотвращая нежелательные последствия. В отличие от авторизаторов, **одного отказа** достаточно, чтобы отклонить запрос.

💡Контроллеры реализованы как **плагины** (в plugin/pkg/admission), 
и сгруппированы по категориям 
- управление ресурсами, 
- безопасность, 
- значения по умолчанию 
- консистентность.


Примеры:
- `InitialResources` устанавливает лимиты по умолчанию для ресурсов контейнера, основываясь на предыдущем использовании;

- `LimitRanger` устанавливает значения по умолчанию для запросов и лимитов контейнера, ==гарантирует верхние границы== для определённых ресурсов (512 Мб памяти по умолчанию, но не более 2 Гб);

- `ResourceQuota` считает количество объектов (подов, rc, балансировщиков нагрузки сервисов) и общие потребляемые ресурсы (процессор, память, диск) в пространстве имён и предотвращает их превышение.

# 3. 📕 etcd
  
К этому моменту Kubernetes полностью одобрил входящий запрос и разрешил двигаться дальше.

Следующим шагом **kube-apiserver** десериализует HTTP-запрос, ==создаёт объект ресурса (например ПОДА)== и сохраняет его в **etcd** — центральное хранилище состояния кластера.

При запуске kube-apiserver выполняет инициализацию:
1. ⭐ **Создаёт server chain** (цепочку серверов) — основу для агрегации API
> **цепочка REST-серверов**, через которую kube-apiserver **обрабатывает входящие HTTP-запросы** от клиентов (kubectl, контроллеров, kubelet и т.д.).
    
2. Затем **инициализирует generic apiserver** — реализацию по умолчанию 
(Аналог DispathcerServlet в Spring MVC)
    
3. **Загружает OpenAPI-схему** в конфигурацию
    
4. ⭐ После этого **настраивает storage providers** для всех групп API — унифицированный интерфейс к данным (поставщика хранилища - _storage provider_). 
> ❗❗С ним kube-apiserver взаимодействует, когда обращается к состоянию ресурса или изменяет его.
    
5. **Регистрирует REST-маршруты** для всех версий API, связывая HTTP-запросы с обработчиками


К этому моменту kube-apiserver знает, какие существуют маршруты и имеет внутренний mapping, указывающий на то, какие обработчики и поставщики хранилища должны быть вызваны при соответствии запроса. 

Предположим, в него попал наш HTTP-запрос:
1. **Маршрутизация** - запрос сопоставляется с зарегистрированными маршрутами. И затем вызывается обработчик
    
2. **Вызов createHandler** - для POST-запросов:
    - Декодирование и базовая валидация JSON
    - Аудит и финальные проверки
        
3. **Сохранение в etcd** - через storage provider (ключ: `<namespace>/<name>`)
    
4. **Подтверждение создания**:
    - Проверка успешности операции при помощи метода `get` (похоже через ORM в БД проверяет сохранение объекта)
        
    - Вызов post-create обработчиков
        
    - Финализация декораторами
        
5. **Формирование ответа** - отправка HTTP-ответа клиенту
    

Если маршрут не найден - возвращается 404.


⭐⭐Итог: объект (например, **Deployment**) теперь хранится в **etcd**, НО ПОКА ЕЩЕ НЕ ЗАПУЩЕН в кластере — этим займутся контроллеры.

# 📕 4. Инициализаторы

Когда объект сохранён в хранилище данных, он не является полностью видимым apiserver и не попадает в планировщик, пока не отработает набор [инициализаторов](https://kubernetes.io/docs/admin/extensible-admission-controllers/#initializers) _(intializers)_


> [!NOTE] Инициализатор 
> Это специальные контроллеры, выполняющие предварительные действия над ресурсом до его «публикации» в кластере.  
> 

❗❗❗Если для ресурса инициализаторы не заданы — он становится видим сразу.❗❗❗

**Примеры использования:**
- Автоматическая вставка **sidecar-контейнера** по условию (например, при открытом порте 80).
    
- Добавление **тома с сертификатами** во все поды в namespace.
    
- Блокировка создания **секретов** с коротким паролем.


Какие инициализаторы применять к конкретным ресурсам, задаётся через объект **InitializerConfiguration**.

### 🧠 Доп. инфа
Представьте, что мы хотим запускать свой инициализатор при каждом случае создания пода. Тогда мы сделаем нечто такое:
```yaml
apiVersion: admissionregistration.k8s.io/v1alpha1
kind: InitializerConfiguration
metadata:
  name: custom-pod-initializer
initializers:
  - name: podimage.example.com
    rules:
      - apiGroups:
          - ""
        apiVersions:
          - v1
        resources:
          - pods
```
После создания этого конфига в поле ожидания (`metadata.initializers.pending`) каждого пода будет добавлен `custom-pod-initializer`. Контроллер инициализатора будет уже развёрнут и начнёт регулярно сканировать кластер на новые поды. Когда инициализатор обнаружит под со своим _(т.е. инициализатора)_ названием в поле ожидания, он исполнит свои действия.
