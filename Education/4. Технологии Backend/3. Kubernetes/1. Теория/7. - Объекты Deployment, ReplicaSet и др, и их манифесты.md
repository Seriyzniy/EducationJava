# 📕 1. Deployment

**Deployment** — объект в Kubernetes, который описывает желаемое состояние приложения: какой образ использовать, сколько реплик должно быть, стратегии обновления и т.д.

📄 Пример Deployment:
```yaml
apiVersion: apps/v1            # Группа и версия API

kind: Deployment               # Тип ресурса

metadata:
  name: my-deployment          # Имя деплоймента
  labels:                      # Метки для селектора
    app: myapp


spec:
  replicas: 3 # Сколько подов должно быть запущено
  
  selector: # ❗❗Какие поды принадлежат этому деплойменту❗❗
    matchLabels:
      app: myapp
      
      
  strategy:            # Стратегия обновления
    type: RollingUpdate
    rollingUpdate:
	  # Макс. число подов, которые могут быть недоступны при обновлении
      maxUnavailable: 1  
      # Макс. число дополнительных подов во время обновления
      maxSurge: 1 
    
   
  template:                    # Шаблон пода
    metadata:
      labels: # ❗❗Метка для подов этого деплоймента❗❗
        app: myapp
        
    spec:
      containers:              # Описание контейнеров
        - name: nginx
          image: nginx:latest
          ports:
            - containerPort: 80
              
          # 🔍 Проверка "живости" контейнера
          livenessProbe:
            httpGet:
              path: /healthz
              port: 8080
            initialDelaySeconds: 10
            periodSeconds: 5
            failureThreshold: 3

          # ⚡ Проверка "готовности"
          readinessProbe:
            httpGet:
              path: /ready
              port: 8080
            initialDelaySeconds: 5
            periodSeconds: 3
            failureThreshold: 2

          # 🕓 Проверка для долгого старта (опционально)
          startupProbe:
            httpGet:
              path: /startup
              port: 8080
            failureThreshold: 30     # сколько попыток перед тем, как считать, что не запустился
            periodSeconds: 10
           
          resources:
            limits:            # Ограничения ресурсов
              cpu: "500m"
              memory: "256Mi"
            requests:          # Минимально требуемые ресурсы
              cpu: "250m"
              memory: "128Mi"
          
          env:                 # Переменные окружения
            - name: ENV
              value: prod
          volumeMounts:
            - name: config
              mountPath: /etc/config
              
      volumes:
        - name: config
          configMap:
            name: my-config
```

## Как работает Deployment

1. Когда ты создаёшь Deployment, Kubernetes (а именно контроллер Deployment) создаёт **ReplicaSet** под капотом.
    
2. ReplicaSet (а именно контроллер ReplicaSet) создаёт **нужное число Pod’ов**.
    
3. Если ты меняешь, например, `image: nginx:1.27.2 → nginx:1.28`,  
    Контроллер Deployment создаёт **новый ReplicaSet** с новой версией Pod’а.
    
4. Kubernetes **постепенно** (rolling update) уменьшает старый ReplicaSet и увеличивает новый.

## Основные возможности Deployment
|Возможность|Описание|
|---|---|
|`replicas`|Сколько Pod’ов держать|
|`rolling update`|Плавное обновление без простоев|
|`rollback`|Откат к предыдущей версии|
|`pause/resume`|Приостановить/возобновить обновление|
|`scale`|Масштабировать количество Pod’ов|
|`revision history`|Хранит историю изменений ReplicaSet’ов|

## Типы стратегий обновления
В секции `strategy` можно указать:
### RollingUpdate (по умолчанию)
Постепенно заменяет Pod’ы:
```yaml
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxUnavailable: 1
    maxSurge: 1
```

> Одновременно максимум 1 Pod может быть недоступен, и максимум 1 новый может быть запущен сверх нормы.

### Recreate
Удаляет все старые Pod’ы, потом создаёт новые:
```yaml
strategy:
  type: Recreate
```
> Используется, если старые и новые версии не могут работать вместе (например, при изменении схемы БД).

# 📕2. ReplicaSet

**ReplicaSet** следит, чтобы в любой момент времени было запущено **нужное количество одинаковых Pod’ов**.


> [!NOTE] ❗❗❗Обычно **создаётся автоматически** ❗❗❗через Deployment, но можно и вручную
> 


📄 Пример ReplicaSet:
```yaml
apiVersion: apps/v1

kind: ReplicaSet

metadata:
  name: my-replicaset
  labels:
    app: myapp
    
    
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
      
  template:  # ❗❗❗Шаблон подов, как в Deployment❗❗❗
    metadata:
      labels:
        app: myapp
    spec:
      containers:
        - name: nginx
          image: nginx:latest

```

# 📕3.Pod

**Pod** — минимальная единица развертывания в Kubernetes.  
Это группа из одного или нескольких контейнеров, которые:
- делят общую сеть и файловую систему,
- запускаются и перезапускаются как единое целое.

```yaml
apiVersion: v1

kind: Pod

metadata:
  name: my-pod
  labels:
    app: myapp
    
    
spec:
  containers:
    - name: nginx
      image: nginx:latest
      ports:
        - containerPort: 80
      env:
        - name: ENV
          value: prod
          
  restartPolicy: Always         # Always | OnFailure | Never
  
  volumes:                      # (опционально)
    - name: config
      configMap:
        name: my-config
```

## 💡 Sidecar-контейнер 💡
`Sidecar (боковой контейнер)` — это **дополнительный контейнер внутри Pod-а**, который работает _вместе_ с основным контейнером приложения и **расширяет его функциональность**.

❗❗Он — часть логики Pod-а, а НЕ СИСТЕМНЫЙ ЭЛЕМЕНТ Kubernetes.

Kubernetes никак специально не помечает sidecar — это просто обычный контейнер, который **выполняет вспомогательные задачи**.

|Сценарий|Что делает sidecar|
|---|---|
|**Логирование**|Читает логи из `/var/log/app.log` и отправляет их в Elasticsearch или Loki.|
|**Прокси / сервис-меш**|Контейнер с Envoy или Istio proxy, перехватывающий весь трафик.|
|**Init-операции**|Загружает конфигурацию или токен до старта основного контейнера.|
|**Мониторинг**|Собирает метрики или экспортирует их в Prometheus.|
|**Синхронизация данных**|Обновляет файлы конфигурации через git-sync.|

### Пример Pod-а с sidecar
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: app-with-sidecar
spec:
  containers:
  - name: main-app
    image: nginx:latest
    volumeMounts:
      - name: shared-logs
        mountPath: /var/log/nginx

  - name: log-exporter # ❗❗Sidecar - ДОПОЛНИТЕЛЬНЫЙ КОНТЕЙНЕР❗❗
    image: busybox
    command: ["sh", "-c", "tail -n+1 -F /var/log/nginx/access.log"]
    volumeMounts:
      - name: shared-logs
        mountPath: /var/log/nginx

  volumes:
  - name: shared-logs
    emptyDir: {}
```

🧠 Здесь `main-app` — основное приложение (Nginx),  
а `log-exporter` — sidecar, который "подглядывает" в общий том и что-то делает с логами.

# 📕 4.Service
**Service** — обеспечивает **сетевой доступ** к группе подов. 

Дает **постоянный IP** и **DNS-имя**, даже если поды пересоздаются.

```yaml
apiVersion: v1

kind: Service

metadata:
  name: my-service
  
spec:
  selector:
    app: myapp  # К каким подам привязывается
    
  ports:
    - protocol: TCP
      port: 80                  # Порт, доступный внутри кластера
      targetPort: 8080          # Порт внутри пода
      
  type: ClusterIP  # ClusterIP | NodePort | LoadBalancer | ExternalName
```

**Типы сервисов:**
- `ClusterIP` — доступен только внутри кластера (по умолчанию)
    
- `NodePort` — доступ снаружи по `nodeIP:port`
    
- `LoadBalancer` — внешний балансировщик (в облаках)
    
- `ExternalName` — DNS-алиас на внешний ресурс

# 📕 5.Ingress
Ingress — это объект, который задаёт **HTTP/HTTPS-маршрутизацию** на уровне кластера.  
Он позволяет:

- работать с **доменными именами** (DNS);
    
- маршрутизировать по **путям (path)**;
    
- **терминировать TLS (HTTPS)**;
    
- обеспечивать **единый входной шлюз** для множества сервисов.

❗❗Работает поверх `Ingress Controller` (Nginx, Traefik и т.д.)❗❗

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress

metadata:
  name: my-ingress         # Имя ресурса Ingress
  namespace: default       # (опционально) пространство имён
  annotations:
	# Аннотации управляют поведением контроллера (например, NGINX)
    nginx.ingress.kubernetes.io/rewrite-target: /   
    
    
spec:
   # Указывает, какой ❗❗Ingress-контроллер должен обрабатывать этот Ingress❗❗
  ingressClassName: nginx   
  
  # Правила маршрутизации входящих запросов   
  rules: 
      # (опционально) конкретный домен (если не указан — правило применяется ко всем)
    - host: myapp.example.com     
      http:
        paths: # Пути (URI), которые будут обрабатываться
          - path: /  # URL-путь (например, / или /api)
            pathType: Prefix  # Тип совпадения пути: Exact | Prefix | ImplementationSpecific
            
            backend:
              service: # 💡Указываем сервис, на который будет перенаправлен запрос💡
                name: my-service # Имя Service
                port:
                  number: 80 # Порт Service, на который нужно направить трафик


  tls:  # (опционально) секция TLS для HTTPS
    - hosts:
        - myapp.example.com
      secretName: my-tls-secret   # Секрет с сертификатом и приватным ключом

```

# 📕6. ConfigMap и Secret. Способы из передачи в Pod 

## 📙1. ConfigMap

**ConfigMap** хранит **настройки** и **параметры конфигурации** в виде пар `ключ-значение`. 

Хранит **неконфиденциальные данные конфигурации** (например, настройки, пути, URL, имена баз данных, лог-уровень и т.д.).

📦 Он нужен, чтобы **не хардкодить конфигурацию в Pod’ах или образах**.


Контейнеры могут использовать их как:
- переменные окружения,
- файлы, смонтированные в под.

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: my-config
  
data:
  APP_MODE: production
  APP_PORT: "8080"
  config.json: |
    {
      "debug": false,
      "max_connections": 100
    }
```

Использование в поде:
```yaml
envFrom:
  - configMapRef:
      name: my-config
```
или
```yaml
volumes:
  - name: config
    configMap:
      name: my-config
```

---
## 📙2. Secret
`Secret` хранит **конфиденциальные данные** — пароли, токены, ключи, сертификаты.

Он также передаётся подам **через переменные окружения или volume**.

⚠️ Данные в Secret хранятся в **base64-кодировке**, а не в шифрованном виде (поэтому доступ к etcd нужно ограничивать!).

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: db-secret
type: Opaque
data:
  DB_USER: dXNlcg==        # base64("user")
  DB_PASSWORD: cGFzc3dvcmQ= # base64("password")
```

## 3. Способы передачи
### ✅ Способ 1. Через переменные окружения (`env`)
Позволяет **взять отдельные ключи** из ConfigMap или Secret.
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: myapp
spec:
  containers:
  - name: app
    image: nginx
    env:
      - name: APP_ENV
        valueFrom:
          configMapKeyRef:
            name: app-config
            key: APP_ENV
      - name: DB_PASSWORD
        valueFrom:
          secretKeyRef:
            name: db-secret
            key: DB_PASSWORD
```
💡 Используй, когда тебе нужно **только несколько конкретных переменных**.

### ## ✅ Способ 2. Через `envFrom` — подключить **все ключи сразу**
Позволяет автоматически импортировать **все пары ключ-значение** из ConfigMap или Secret как переменные среды.
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: myapp
spec:
  containers:
  - name: app
    image: nginx
    envFrom:
      - configMapRef:
          name: app-config
      - secretRef:
          name: db-secret

```

📘 В контейнере будут доступны:
```ini
APP_ENV=production
APP_DEBUG=false
DATABASE_URL=jdbc:postgresql://db-service:5432/mydb
DB_USER=user
DB_PASSWORD=password
```

### ✅ Способ 3. Через Volume (монтирование файлов)

ConfigMap и Secret можно **смонтировать как файлы** в контейнер.

#### Пример: монтирование ConfigMap
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: configmap-volume-pod
  
spec:
  containers:
    - name: app
      image: nginx
      volumeMounts:
        - name: config-volume
          mountPath: /etc/config
  volumes:
    - name: config-volume
      configMap:
        name: app-config
```

📁 Внутри контейнера:
```
/etc/config/APP_ENV
/etc/config/APP_DEBUG
/etc/config/DATABASE_URL
```

#### Пример: монтирование Secret
```yml
apiVersion: v1
kind: Pod
metadata:
  name: secret-volume-pod
spec:
  containers:
    - name: app
      image: nginx
      volumeMounts:
        - name: secret-volume
          mountPath: /etc/secret
          readOnly: true
  volumes:
    - name: secret-volume
      secret:
        secretName: db-secret
```

📁 В контейнере:
```
/etc/secret/DB_USER
/etc/secret/DB_PASSWORD
```
