
> [!NOTE] ⭐⭐YAML - формат файла для описания объектов⭐⭐
> Kubernetes использует **YAML-файлы** для описания состояния объектов кластера (Pod, Deployment, Service, ConfigMap и т.д.).

# 📕 1. Общая структура YAML-манифеста

Любой YAML-файл в Kubernetes имеет **4 верхних уровня**, одинаковых для всех ресурсов:
```yaml
apiVersion: apps/v1 # 1-й

kind: Deployment    # 2-й

metadata:           # 3-й
  name: my-nginx
  labels:
    app: nginx
    
spec:               # 4-й
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
        - name: nginx
          image: nginx:latest
          ports:
            - containerPort: 80

```

---
## Уровень 1️⃣ — `apiVersion`
```yaml
apiVersion: apps/v1
```

**Что это:**  
Версия API, через которую Kubernetes понимает, к какой группе ресурсов относится объект.

**Примеры:**

|Ресурс|apiVersion|
|---|---|
|Pod|v1|
|Deployment|apps/v1|
|Service|v1|
|Ingress|networking.k8s.io/v1|
|ConfigMap|v1|

> ❗❗Это важно, потому что разные версии API могут содержать разные поля и правила.

---
## Уровень 2️⃣ — `kind`

```yaml
kind: Deployment
```

**Что это:**  
Тип ресурса, который ты создаёшь.  

Это может быть 
- `Pod`, 
- `Service`, 
- `Deployment`, 
- `ReplicaSet`, 
- `ConfigMap` и т.д

> Kubernetes по этому полю понимает, какой контроллер должен обработать объект.

---
## Уровень 3️⃣ — `metadata`
```yaml
metadata:
  name: my-nginx
  labels:
    app: nginx
  namespace: default
```

**Что это:**  
Метаданные — уникальные идентификаторы ресурса.

Основные директивы:

| Поле          | Назначение                                                                          |
| ------------- | ----------------------------------------------------------------------------------- |
| `name`        | Имя ресурса <br>(❗❗обязательно❗❗, уникально в пределах namespace).                  |
| `namespace`   | Пространство имён, в котором создаётся объект.                                      |
| `labels`      | Метки (key:value) — для группировки и выбора ресурсов.                              |
| `annotations` | Аннотации — произвольные данные (описания, версии и т.д.), не участвующие в логике. |
> 📘 `labels` активно используются селекторами 
> (`selector.matchLabels`) для связи объектов (например, Deployment → Pod).


---
## Уровень 4️⃣ — `spec`
```yaml
spec:
  ... 
```

**Что это:**  
Самое важное — описание _желаемого состояния_ (desired state) объекта.

**Содержимое `spec` зависит от типа ресурса (`kind`):**

### 🔶 Пример для `Deployment`
```yaml
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
        - name: nginx
          image: nginx:latest
          ports:
            - containerPort: 80
```

|Поле|Назначение|
|---|---|
|`replicas`|Сколько подов должно быть запущено.|
|`selector`|Как контроллер определяет, какие поды принадлежат этому Deployment.|
|`template`|Шаблон пода, который нужно развернуть.|

⭐⭐А вот внутри `template` — снова **`metadata`** и **`spec`** (но уже для пода).

#### Внутри `template.spec` (у пода)
|Поле|Назначение|
|---|---|
|`containers`|Список контейнеров, которые нужно запустить.|
|`containers[].name`|Имя контейнера.|
|`containers[].image`|Docker-образ контейнера.|
|`containers[].ports`|Порты контейнера.|
|`containers[].env`|Переменные окружения.|
|`containers[].volumeMounts`|Тома, примонтированные к контейнеру.|
|`volumes`|Определение общих томов (например, ConfigMap, PVC).|

### 🔶 Пример для `Service`
```yaml
spec:
  selector:
    app: nginx
  ports:
    - port: 80
      targetPort: 80
  type: ClusterIP
```

|Поле|Назначение|
|---|---|
|`selector`|К каким подам направлять трафик.|
|`ports`|Внешний и внутренний порты.|
|`type`|Тип сервиса (`ClusterIP`, `NodePort`, `LoadBalancer`).|

### 🔶 Пример для `ConfigMap`
```yaml
spec:   # отсутствует
data:
  key1: value1
  key2: value2
```
У `ConfigMap` вместо `spec` используется `data`.
