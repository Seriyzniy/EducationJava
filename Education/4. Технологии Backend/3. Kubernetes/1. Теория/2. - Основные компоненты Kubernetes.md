# 📕 2. Нода (Node - узел)
**Нода (узел) в Kubernetes** — это **физическая или виртуальная машина**, на которой развёртываются и запускаются контейнеры с приложениями. 

Это представление одной машины в вашем кластере
![[Pasted image 20251027100723.png]]


> [!NOTE] ⭐⭐`Каждый узел` (даже мастер-нада) содержит сервисы, необходимые для запуска подов⭐⭐
> 
![[Pasted image 20251027102948.png]]

🔶Container Runtime - отвечает за создание и контроль контейнеров на каждом узле (даже мастер-ноде). МОЖЕТ БЫТЬ Docker, ContainerD, CRI-O

🔶Kublet - сервис отвечающий за СОЗДАНИЕ ПОД'а и коммуникацию между узлами (нодами) в рамках кластера. ЧЕРЕЗ НЕГО рабочие ноды ВЗАИМОДЕЙСТВУЮТ с МАСТЕР-НОДОЙ

🔶Kube-proxy - отвечает за сетевые ресурсы каждого нода

# 📕 3. Кластер
**Cluster (кластер)** - Совокупность машин (нод), на которых работают ваши приложения

Кластер балансирует нагрузку по нодам 
![[Pasted image 20251027101452.png]]

## 📙 Роли нодов

**Master Node (Control Plane)** — управляет кластером: 
- планирование, 
- масштабирование, 
- мониторинг. 
(МОЖЕТ БЫТЬ НЕСКОЛЬКО (как минимум 3 штуки))
> [!NOTE] ⭐⭐⭐Мастер-нода — это **“мозг” кластера**.  ⭐⭐⭐
> Все команды управления (создать Pod, масштабировать Deployment, удалить Service и т.п.) проходят **через неё**.


**Worker Nodes** — где реально крутятся контейнеры.
![[Pasted image 20251027101051.png]]

## 📙Структура Мастер-ноды (Control plane - другое название)

![[Pasted image 20251027181146.png]]

Кроме Container Runtime, kubelet, kube-proxy на мастер ноде есть:

🔶 Первое и самое простое — etcd. Это база данных формата «ключ — значение»‎, которая содержит текущую конфигурацию и состояние кластера Kubernetes. 
Например, в ней хранится информация о развертываемых приложениях, настройках и ресурсах.

🔶 Есть scheduler, который определяет, на какую из воркер-нод деплоим приложение. Если на какой-то ноде есть свободная память, значит туда задеплоим приложение

🔶 Далее — ==API-сервер==. С ним общаются все: от разработчика до нод. 
Никто напрямую в базу ETCD не лезет и ей не управляет. API — это единственная точка входа в кубер. 

🔶 И последнее — Controller manager. Программа, которая следит за тем, чтобы все, что попросили, было сделано.

🧠Также есть Cloud Controller Manager - для управления кластером в облаке

> [!NOTE] ⭐Также на МАСТЕР-ноде `есть DNS-сервис`, для взаимодействия между сервисами внутри кластера 
> 

`kubectl` - интерфейс в командной строке для работы с API

# 📕 4. Принцип работы
Допустим мы хотим развернуть приложение NGINX, который показывает html страницу

## 📙 1. Обращение к Master Node через kubectl
![[Pasted image 20251027190227.png]]

Мы обращаемся к мастер ноде, и говорим "Разверни поду с приложением NGINX"
```bash
kubectl apply -f nginx-pod.yaml

ИЛИ

kubectl run nginx --image=nginx
```

### ❓ЧТО ЗА ФАЙЛ nginx-pod.yaml❓
Это **манифест Kubernetes** — обычный текстовый файл в формате **YAML**, который описывает:
> “Какое состояние ты хочешь получить в кластере”.

Когда ты делаешь:
`kubectl apply -f nginx-pod.yaml`
ты передаёшь Kubernetes **декларативное описание ресурса** (Pod, Deployment, Service и т.д.).  

Kubernetes читает этот файл, проверяет структуру и создаёт объект с указанными параметрами.

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod
  labels:
    app: nginx
spec:
  containers:
    - name: nginx-container
      image: nginx:latest
      ports:
        - containerPort: 80
```

### Kubectl → API Server (мастер-нода)
`kubectl` использует файл **==kubeconfig==**, где указано:
- читает YAML-манифест,
- формирует HTTP-запрос к **kube-apiserver** (это REST API Kubernetes),
- аутентифицируется через **kubeconfig** (сертификаты, токены),
- и отправляет JSON-запрос вида на IP-сервера УКАЗАННОГО в **kubeconfig**:
    > “Создай Pod с такими-то параметрами”.

Kubectl отправляет HTTP(S)-запрос в **kube-apiserver** (это REST API).  
Например:
```http
POST /api/v1/namespaces/default/pods
```
Тело запроса (твоя YAML-манифестация Pod’a) передаётся в JSON.

## 📙 2. API Server валидирует и сохраняет объект в etcd

`kube-apiserver`:
- проверяет синтаксис, права доступа (RBAC),
- валидирует структуру манифеста,
- если всё корректно — сохраняет объект `Pod` в **etcd**.

`etcd` — это **распределённое key-value хранилище**, в котором лежит всё “желаемое состояние” кластера:
> “Должен существовать Pod nginx с образом nginx:latest”.


На этом этапе **контейнер ещё не создан** — только зарегистрирована “цель”.

## 📙 3. Controller Manager замечает новое “желаемое состояние”

**kube-controller-manager** следит за записями в `etcd`.  
Он видит, что теперь в кластере _должен быть Pod nginx_, но его пока нет.

> [!NOTE] ❗❗ СЛЕДИТ ЗА ЖИЗНЕННЫМ ЦИКЛОВ ПОДЫ ❗❗
> 

Контроллер сообщает планировщику (scheduler):
> “Нужно разместить новый Pod”.


## 📙 4. Scheduler выбирает подходящую Worker Node

**kube-scheduler**:
- получает список всех доступных worker-нод,
- анализирует ресурсы (CPU, RAM, taints, nodeSelector и т.д.),
- выбирает оптимальную ноду, например `node-2`,
    
- записывает в `etcd`:
    > “Pod nginx назначен на node-2”.
    
![[Pasted image 20251027191427.png]]

## 📙 5. kubelet на выбранной ноде получает задание

На каждой worker-ноде работает агент **kubelet** — он периодически опрашивает API Server:
> “Есть ли для меня новые Pod’ы?”

Когда kubelet видит, что ему назначен новый Pod:
- скачивает (через CRI) образ `nginx` из Docker Hub или другого реестра,
- создаёт контейнер через ❗❗**container runtime**❗❗ (например `containerd`),
- запускает его,
- настраивает сеть (через CNI — Container Network Interface),
- и монтирует нужные volume’ы.

![[Pasted image 20251027191731.png]]

## 📙 6. kubelet сообщает мастеру о статусе

После запуска контейнера:
- kubelet уведомляет API Server: “Pod nginx → статус: Running”.
- API Server обновляет запись в `etcd`.

Теперь команда `kubectl get pods` покажет:
```bash
NAME     READY   STATUS    RESTARTS   AGE
nginx    1/1     Running   0          10s
```

## 📙 7. kube-proxy настраивает сеть

Если Pod должен быть доступен по сети (например, через Service),  
компонент **kube-proxy** на каждой ноде:

- настраивает правила iptables / IPVS,
    
- чтобы трафик на `Service`-IP перенаправлялся к нужному Pod’у.
    

---

## 📙8. Мониторинг и самовосстановление

Если Pod упадёт:
- kubelet заметит, что контейнер завершился,
- Controller Manager увидит, что “желаемое состояние (1 Pod) ≠ текущее (0 Pod)”,
    
- и создаст новый Pod (или перезапустит старый, в зависимости от контроллера — Deployment, ReplicaSet и т.д.).

# 📕 5. Общий вид
## 1️⃣ Физический кластер Kubernetes
> 💡 Каждый **физический сервер** = **нода (node)** кластера.

```
[Physical Server 1] —> worker-node
[Physical Server 2] —> worker-node
[Physical Server 3] —> master-node (или control-plane)
```

На каждом сервере установлены:
- **ОС (обычно Linux)**
- **kubelet** — агент, управляющий контейнерами на этой ноде
- **container runtime** — например, containerd или CRI-O (чтобы запускать контейнеры)
- **kube-proxy** — отвечает за сетевое взаимодействие Pod’ов

Внутри **ноды** уже создаются:
```
Node (сервер)
 └── Pod
       └── Container (например, nginx)
```

🔹 **Контейнеры запускаются напрямую на хостовой ОС** — **никаких промежуточных виртуальных машин** нет.  
🔹 Kubernetes сам распределяет Pod’ы по этим физическим нодам.

#### Итого по физическому кластеру:

|Уровень|Что это|
|---|---|
|Сервер|Node|
|Node|Работает kubelet, запускает Pod’ы|
|Pod|Группа контейнеров|
|Контейнер|Приложение (nginx, java-сервис и т.д.)|

## 2️⃣ Виртуальный кластер Kubernetes

> 💡 Здесь узлы (nodes) — это **виртуальные машины**, созданные либо в облаке, либо на локальном гипервизоре.

### Структура:

```
[Physical Server]
 └── [Virtual Machine 1] → worker-node
 └── [Virtual Machine 2] → worker-node
 └── [Virtual Machine 3] → master-node
```

На уровне Kubernetes для него **ничего не меняется** —  
он просто видит “ноды”, не зная, что это ВМ.


Внутри виртуальной машины:
```
VM (node)
 └── Pod
      └── Container (например, nginx)
```

То есть:
- **Кластер = набор виртуальных машин (нод)**.
- **В каждой VM — kubelet, container runtime, и Pod’ы**.
    
- **Контейнеры всё равно работают на уровне ОС ВМ**, а не поверх других контейнеров.
