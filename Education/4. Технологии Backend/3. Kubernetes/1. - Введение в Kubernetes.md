# 📕 1. Как было до Kubernetes
Раньше какое-то приложение/сервис выглядел как то так
![[Pasted image 20251027091859.png]]
То есть он состоял из разных сервисов, каждый из которых выполнял свою задачу.

## 💥 Проблема роста нагрузки сервисов
НО возникает проблема, НАГРУЗКА РАСТЕТ и при этом НЕ РАВНОМЕРНО. Какой-то из сервисов загружен больше, чем другие (например люди чаще делают заказы а не авторизируются)

Поэтому мы масштабируем сервисы, и распределяем их ПО НЕСКОЛЬКИМ серверам, чтобы если один из серверов "упал", наше приложение хоть как-то функционировало
![[Pasted image 20251027092255.png]]

## 💥Проблема распределения нагрузки между серверами и масштибирования
Теперь возникает проблема распределения нагрузки МЕЖДУ СЕРВЕРАМИ. 
Нужно как-то распределять нагрузку между всеми серверами. 

Для этого устанавливается БАЛАНСИРОВЩИК
Однако, приходится вручную указывать
- IP-адреса серверов
- порты на которых работают приложения

При  этом возникает еще одна проблема - масштабирование сервисов (каждый дубликат сервиса связываем с балансировщиком вручную)

![[Pasted image 20251027092431.png]]

## 💥Проблема падения сервисов и выкатывание новых версий сервисов
### 🔶"Падение" сервисов
Если сервисы на каком-либо сервере "падают", из-за чего либо (утечка памяти, сервер сгорел и т.д), то:
- Балансировщик об этом не знает, и продолжает отправлять туда запросы
- Приходится вручную поднимать сервисы

### 🔶"Выкатывание" новых версий сервисов
Допустим, нам необходимо ОБНОВИТЬ какой-то сервис, из-за нового функционала.
Для этого мы обновляем каждый сервис по очереди (чтобы некоторые из старых сервисов работали)
![[Pasted image 20251027093400.png]]

Все это мы делаем вручную. Однако, если в новой версии опять что-то пошло не так, приходится откатывать на старую версию сервиса, опять вручную

И это лишь малая часть. В реальных кампаниях таких сервером сотни.

Всем эти проблемы решает Kubernetes

# 📕 2. Kubernetes (k8s)

> [!NOTE] Kubernetes — система управления КОНТЕЙНЕРАМИ
> Используется для автоматизации:
> -  развертывания,
> - масштабирования 
> - управления КОНТЕНЕРизированными приложениями.  
> Её создала Google (на основе своего внутреннего проекта **Borg**), а сейчас поддерживает сообщество **Cloud Native Computing Foundation (CNCF)**.
> 

ГРУБО говоря:
> Kubernetes — это операционная система для вашего дата-центра, которая управляет контейнерами так же, как ОС управляет процессами.


## 📙 Основные задачи Kubernetes
Kubernetes решает несколько ключевых проблем, возникающих при эксплуатации контейнеров в продакшене:

| Проблема                                        | Что делает Kubernetes                                            |
| ----------------------------------------------- | ---------------------------------------------------------------- |
| Сложно управлять множеством контейнеров вручную | Автоматически распределяет контейнеры по серверам (нодам)        |
| Нужно балансировать нагрузку                    | Сам распределяет трафик между контейнерами                       |
| Контейнеры падают                               | Автоматически перезапускает упавшие контейнеры                   |
| Нужно масштабировать приложение                 | Легко увеличивает/уменьшает число контейнеров                    |
| Нужен деплой без простоя                        | Делает rolling updates и откаты (rollback)                       |
| Разные среды (dev, stage, prod)                 | Позволяет описывать инфраструктуру декларативно (манифесты YAML) |

## 📙 Что Kubernetes НЕ ДЕЛАЕТ
- НЕ ХРАНИТ данные
- НЕ ХРАНИТ образы (registry)
- НЕ занимается СБОРКОЙ ОБРАЗОВ
- НЕ занимается СБОРОМ ЖУРАНЛОВ и МЕТРИК

## 📙⭐⭐ Инструменты вокруг Kubernetes ⭐⭐

- **kubectl** — CLI для взаимодействия с кластером.
    
- **Minikube / Kind** — запуск локального кластера.
    
- **Helm** — менеджер пакетов для Kubernetes (аналог apt/npm).
    
- **Kustomize** — инструмент для настройки манифестов.
    
- **Lens / Portainer / Rancher** — графические интерфейсы для управления кластером.
