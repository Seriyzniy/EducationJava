# 📕 1. Под (Pod)
Минимальная единица развертывания в Kubernetes.  

![[Pasted image 20251027095703.png]]

Это **ОБЕРТКА над ОДНИМ или НЕСКОЛЬКИМИ контейнерами**, которые:
- живут вместе (на одной ноде),
- делят сеть и хранилище.

# 📕 2. Нода (Node - узел)
**Нода (узел) в Kubernetes** — это **физическая или виртуальная машина**, на которой развёртываются и запускаются контейнеры с приложениями. 

Это представление одной машины в вашем кластере
![[Pasted image 20251027100723.png]]


> [!NOTE] ⭐⭐`Каждый узел` (даже мастер-нада) содержит сервисы, необходимые для запуска подов⭐⭐
> 
![[Pasted image 20251027102948.png]]

🔶Container Runtime - отвечает за создание и контроль контейнеров на каждом узле (даже мастер-ноде). МОЖЕТ БЫТЬ Docker, ContainerD, CRI-O

🔶Kublet - сервис отвечающий за коммуникацию между узлами (нодами) в рамках кластера. ЧЕРЕЗ НЕГО рабочие ноды ВЗАИМОДЕЙСТВУЮТ с МАСТЕР-НОДОЙ

🔶Kube-proxy - отвечает за сетевые ресурсы каждого нода

# 📕 3. Кластер
**Cluster (кластер)** - Совокупность машин (нод), на которых работают ваши приложения

Кластер балансирует нагрузку по нодам 
![[Pasted image 20251027101452.png]]

## 📙 Роли нодов

**Master Node (Control Plane)** — управляет кластером: 
- планирование, 
- масштабирование, 
- мониторинг. 
(МОЖЕТ БЫТЬ НЕСКОЛЬКО (как минимум 3 штуки))
> [!NOTE] ⭐⭐⭐Мастер-нода — это **“мозг” кластера**.  ⭐⭐⭐
> Все команды управления (создать Pod, масштабировать Deployment, удалить Service и т.п.) проходят **через неё**.


**Worker Nodes** — где реально крутятся контейнеры.
![[Pasted image 20251027101051.png]]

## 📙Структура Мастер-ноды

![[Pasted image 20251027102052.png]]

Кроме Container Runtime< kubelet, kube-proxy на мастер ноде есть:

🔶 Первое и самое простое — etcd. Это база данных формата «ключ — значение»‎, которая содержит текущую конфигурацию и состояние кластера Kubernetes. Например, в ней хранится информация о развертываемых приложениях, настройках и ресурсах.

🔶 Есть scheduler, который определяет, на какую из воркер-нод деплоим приложение. Если на какой-то ноде есть свободная память, значит туда задеплоим приложение

🔶 Далее — ==API-сервер==. С ним общаются все: от разработчика до нод. 
Никто напрямую в базу ETCD не лезет и ей не управляет. API — это единственная точка входа в кубер. 

🔶 И последнее — Controller manager. Программа, которая следит за тем, чтобы все, что попросили, было сделано.

🧠Также есть Cloud Controller Manager - для управления кластером в облаке

> [!NOTE] ⭐Также на МАСТЕР-ноде `есть DNS-сервис`, для взаимодействия между сервисами внутри кластера 
> 

`kubectl` - интерфейс в командной строке для работы с API
