# 📕 1. Команда: `git cherry-pick`

> [!NOTE] `git cherry-pick` - "выбрать лучшие вишенки", мы выбираем нужные коммиты не трогая ветку 
> **`git cherry-pick`** — это команда, которая **берёт один (или несколько) конкретных коммитов** из другой ветки и **добавляет их в текущую ветку**.  
> 
> При этом Git **создаёт новые коммиты с новыми хешами**, даже если содержимое полностью совпадает.
> 
> 💡Это очень простой и прямолинейный способ сказать, что ты хочешь копировать несколько коммитов на место, где сейчас находишься (`HEAD`)


> [!NOTE] ❗❗❗Копируемый коммит НЕ ДОЛЖЕН БЫТЬ предком `HEAD❗❗❗`
> Важно помнить, что cherry-pick поместит любой коммит сразу после HEAD (только если этот коммит не является предком HEAD)


## 📙 Синтаксис
```bash
git cherry-pick <commit-hash>
```

Можно выбрать сразу несколько:
```bash
git cherry-pick <commit1> <commit2> <commit3>
```

Или диапазон:
```bash
git cherry-pick A..B
```
(все коммиты **после A** и **включая B**)

# 📕 2. - Команда `git rebase -i`

> [!NOTE] `git rebase -i` (interactive rebase) 
> Это **интерактивный режим rebase**, где ты **сам выбираешь, что сделать с каждым коммитом**.
> 
> 💡 Используется для "чистки" истории перед публикацией — чтобы ветка выглядела аккуратно и логично, без лишних коммитов вроде “fix typo” или “oops”.
> 

## 📙 Синтаксис
```bash
git rebase -i <base-commit>
```

или, короче:
```bash
git rebase -i HEAD~N
```
где **N** — количество последних коммитов, которые ты хочешь "отредактировать".

## Пример
Допустим, у тебя есть история:
```css
A --- B --- C --- D --- E (HEAD)
```

Ты запускаешь:
```bash
git rebase -i HEAD~3
```

Это откроет редактор с тремя последними коммитами (C, D, E):
```css
pick C Commit message 1
pick D Commit message 2
pick E Commit message 3
```

Теперь ты можешь изменить слово `pick` на другое действие, чтобы указать Git, что делать с каждым коммитом.

|Команда|Что делает|
|---|---|
|`pick`|Оставить коммит как есть.|
|`reword`|Оставить изменения, но **изменить сообщение** коммита.|
|`edit`|Приостановить rebase на этом коммите — можно **изменить содержимое** (например, добавить или удалить файлы, изменить код).|
|`squash`|Объединить этот коммит с предыдущим — сообщения коммитов будут объединены.|
|`fixup`|Объединить с предыдущим, **игнорируя сообщение** текущего.|
|`drop`|Удалить коммит.|
|`exec <cmd>`|Выполнить указанную команду shell после применения коммита (редко используется).|

## ⚠️ Осторожно!

- `rebase -i` **переписывает историю**: создаются **новые коммиты с новыми хешами**.
    
- Не стоит использовать его **в публичных ветках**, где другие уже взяли твои старые коммиты — это приведёт к рассинхронизации истории.
    
- Лучше всего применять `rebase -i` **до пуша в общий репозиторий** (или только в личных ветках).

# 📕 3. - Изменение прошлого коммита

## 📙 1. Использование `rebase -i`
Пусть история такая:
![[Pasted image 20251025222518.png]]

и ты понял, что **в коммите B** забыл добавить файл или исправить код.

### 1️⃣ Открываешь интерактивный rebase

Ты хочешь редактировать `B`, то есть **второй** коммит от HEAD.  
Значит указываем три последних коммита (чтобы включить `B` в список):

```bash
git rebase -i HEAD~3
```

---
### 2️⃣ Git откроет список коммитов в редакторе

Например:

```bash
🟢 pick B Add user model
pick C Add validation 
pick D Fix typo
```

Меняешь команду `pick` на `edit` у коммита, который хочешь исправить:
```bash
🟢 edit B Add user model 
pick C Add validation 
pick D Fix typo
```

Сохраняешь и закрываешь файл.

---
### 3️⃣ Git приостановит rebase на этом коммите

Git «откатит» HEAD в состояние коммита **B** и скажет:

```
Stopped at B... Add user model
You can amend the commit now, with:

    git commit --amend

Once you are satisfied with your changes, run:

    git rebase --continue

```

---
### 4️⃣ Вноси изменения
Вноси любые правки в код:
```bash
vim User.java
```

И добавь их в индекс:
```bash
git add User.java
```

---
### 5️⃣ Обнови коммит

```bash
git commit --amend
```

> 💡 Это перезапишет коммит B, добавив новые изменения (и, при желании, новое сообщение).

---
### 6️⃣ Продолжи rebase

```bash
git rebase --continue
```

Git применит оставшиеся коммиты (C и D) поверх нового B′ и завершит процесс.

---
## 🧠 Что происходит "под капотом"

> [!NOTE] 🧠Если кратко 🧠
> Git **временно “вытаскивает” коммит из прошлого**, делает его актуальным (`HEAD` на нём), вносятся изменения пользователем, а потом Git **строит заново всю историю**, где этот изменённый коммит возвращается на своё “логическое место

### **Шаг 1. Git находит базу (base commit)**

> 💡 База — это коммит, который является **родителем самого старого из выбранных коммитов**.

В нашем случае:
- самый старый коммит из диапазона — `B`,
- его родитель — `A`.

**Значит, база = A.**

---
### **Шаг 2. Git собирает список коммитов для “переигрывания”**

Git строит **список коммитов после базы (A)**, то есть `B`, `C`, `D`.

Каждый коммит — это объект (снимок состояния дерева и метаданные), и Git **запоминает**, что их нужно будет применить _поверх нового родителя_ (A или его потомков).

То есть Git как бы говорит:

> “Окей, я запомнил, что у нас есть 3 коммита (B, C, D).  
> Сейчас я буду их применять один за другим, начиная от A”.

⚙️ Эти “временные коммиты” — **не создаются сразу**.  
Git лишь **сохраняет их хеши** и **инструкции**, как их воспроизвести.  

---
### **Шаг 3. Git “отматывает” HEAD к базе**

Git перемещает HEAD и рабочую директорию к коммиту **A**:

`HEAD → A`

> При этом **все изменения из коммитов B, C, D исчезают** из рабочей директории (временно).

---
### **Шаг 4. Git начинает “переигрывать” коммиты один за другим**

Теперь Git берёт **каждый коммит из списка (B, C, D)** и:

1. **Берёт diff** между коммитом и его родителем (например, между A и B).
    
2. Применяет этот diff к текущей рабочей директории.
    
3. Создаёт **новый коммит** (КОПИЮ), но уже с другим родителем — текущим HEAD.  
    То есть теперь у нас:
    
    `A --- B'   (копия B)`
    
    🔸 `B'` — это **новый коммит** (КОПИЯ), который повторяет изменения B,  
    но у него **другой родитель (A)** и **другой хеш**.
    
4. HEAD перемещается на новый коммит (B′).  
    Теперь текущая база — B′.
    
---
### **Шаг 5. Если ты указал `edit`**

Когда Git доходит до коммита, помеченного как `edit`, он **останавливается** после применения diff-а, **не создавая следующий коммит** из списка.

Он говорит тебе:

> “Я применил изменения, соответствующие этому коммиту (B), и== теперь можешь что-то поменять.”

В этот момент:
- HEAD указывает на **новый коммит B′**,
- рабочая директория содержит все изменения из B,
- Git приостанавливает “переигрывание”.

---
### **Шаг 6. Ты делаешь `git commit --amend`**

Когда ты вызываешь `--amend`:

- Git **не редактирует старый коммит B′**,
    
- он **создаёт ещё один новый коммит B″**,  
    который включает изменения из B′ + твои новые правки.
    

⭐⭐После этого:⭐⭐

`A --- B″`

и старый B′ больше не нужен — Git заменил его новым коммитом (новый объект с новым хешем).

---
### **Шаг 7. Git продолжает “переигрывание”**

Теперь `rebase --continue` говорит Git:

> “Окей, продолжай применять остальные коммиты (C, D),  
> но теперь делай это поверх нового B″.”

Git применяет диффы C и D, создавая новые коммиты C′ и D′:

`A --- B″ --- C′ --- D′`

## 📙 1. Использование `cherry-pick`
Тоже самео можно сделать при помощи `git cherry-pick`

1) Скопировать необходимый коммит для изменения
2) Внести изменения, а затем `commit --amend`
3) Скопировать все остальные коммиты (дочерние), которые были у предыдущей версии измененного коммита
