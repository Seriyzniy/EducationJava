
> [!NOTE] Слияние ветвей
> **Слияние веток** — это процесс объединения изменений из одной ветки в другую.

Ты, например, работаешь в ветке `feature`, а основная ветка — `main`.  
Когда твоя задача готова, ты хочешь объединить свои изменения в `main`:
```bash
git checkout main
git merge feature
```

📦 В результате Git “вплетает” коммиты из `feature` в историю `main`.

## 📙 Как Git выполняет merge под капотом
1. Git ищет **общего предка** обеих веток (общий коммит, откуда они разошлись).
    
2. **Сравнивает изменения ОТ ПРЕДКА до каждой ветки** (ссылки на последний коммит).
    
3. Объединяет их вместе в новый **merge commit** (если это нужно).

## 📙 Алгоритм выбора изменений Git (по каждому файлу)📗
Для каждого файла Git делает следующее:
1. Если файл **НЕ ИЗМЕНИЛСЯ НИ В ОДНОЙ ИЗ ВЕТОК** относительно `base` →  
    ✅ просто `копируется как есть` (ничего не меняется).
    
2. Если файл **ИЗМЕНИЛСЯ только В ОДНОЙ ВЕТКЕ** →  
    ✅ Git ❗❗`БЕРЕТ ИЗМЕНЕННУЮ ВЕРСИЮ`❗❗ (из той ветки, где было изменение).
    
3. Если файл изменился **В ОБЕИХ ВЕТКАХ, но В РАЗНЫХ МЕСТАХ** →  
    ✅ Git автоматически `объединяет изменения`.
    
4. Если файл изменился **В ОБЕИХ ВЕТКАХ В ОДНОМ И ТОМ ЖЕ МЕСТЕ** →  
    ⚠️ Git не может решить сам → возникает ❗❗❗`конфликт`❗❗❗.
 
> [!NOTE] ⭐⭐Логика: `БРАТЬ последние ИЗМЕНЕНИЯ`, если нет конфликтов⭐⭐
> Git **не выбирает ветку целиком**, он **сливает пофайлово и построчно**. 
> 
> Он сохраняет **все изменения из обеих веток**, кроме тех случаев, где изменения пересекаются — тогда спрашивает тебя.

# 📕 1. Виды слияние в Git
| Вид слияния                     | Команда                                | Что делает                                  | Когда применяется                                                           |
| ------------------------------- | -------------------------------------- | ------------------------------------------- | --------------------------------------------------------------------------- |
| 🟢 **Fast-forward merge**       | `git merge` (по умолчанию, если можно) | Просто сдвигает указатель ветки вперёд      | Когда ветка не расходилась<br>(main является ПРЯМЫМ РОДИТЕЛЕМ другой ветки) |
| 🟡 **Recursive (по умолчанию)** | `git merge`                            | Объединяет две ветки, создавая merge-коммит | Когда ветки расходились                                                     |
| 🔵 **Squash merge**             | `git merge --squash`                   | Сжимает все коммиты ветки в один            | Когда нужно сохранить изменения, но не всю историю                          |

## 📙 1. Fast-forward merge (“перемотка вперёд”)

> [!NOTE] 🧠 Идея
> Если твоя ветка **прямо продолжает** другую — Git может просто “переместить указатель” без создания нового коммита.
> 

Пример
![[Pasted image 20251025170341.png]]

Если **в main** не было новых коммитов, то при слиянии:
```bash
git checkout main
git merge feature
```

> Ищет ОБЩИЙ коммит, ДО РАЗДЕЛЕНИЯ - это коммит B
> Сравнивает файлы по обоим веткам об общего коммита, и объединяет ветви. 
> (Так как после B файлы менялись только в C и D, то изменения в C и D несчем сравнивать)
> После объединения `перемещаем main`

Git видит, что `feature` — просто "дальше по линии", и:
- **не создаёт merge-коммит**
- просто сдвигает `main` на коммит `D`

![[Pasted image 20251025170437.png]]


> [!NOTE] Создание merge-коммита даже при fast-forward
> ```bash
> git merge --no-ff feature
> ```
> ```css
> A --- B --- C --- D  
> 			\     \ 
> 			  \_____M (main)
> ```

## 📙 2. Recursive merge (рекурсивное слияние)

> [!NOTE] 🧠 Идея
> Когда ветки **разошлись** — Git не может просто переместить указатель.  
> Он должен **объединить изменения с обеих сторон**, сравнив их с общим предком.
> 
### 🔍 Почему “recursive”
Если история сложная и ветви сливались несколько раз, Git может **рекурсивно** вычислить общий базовый коммит (base), если есть несколько возможных предков.

Пример
![[Pasted image 20251025170702.png]]
Коммиты `C` и `E` расходятся, общий предок — `B`.

Git выполняет **трёхстороннее слияние (three-way merge)**:
1. Берёт **общего предка** (`B`)
2. Сравнивает с `C` и с `E`
3. Объединяет изменения

![[Pasted image 20251025171159.png]]

`M` — merge-коммит ==с двумя родителями== (`C` и `E`).
Ветка feature удаляется (подтянуть нельзя, так как есть ветвление)

## 📙 3. Squash merge (слияние с “сжатием”)

> [!NOTE] 🧠Идея
> Иногда не нужно тянуть в `main` всю историю разработки ветки — **только результат**
> 
> Squash объединяет все коммиты ветки в **один большой diff** и добавляет его в текущую ветку как **один коммит**.

Пример
![[Pasted image 20251025172811.png]]

Делаем 
```bash
git checkout main
git merge --squash feature
git commit -m "Add new feature"
```

Результат:
![[Pasted image 20251025172928.png]]

📦 `F` содержит все изменения из `feature`,  
но самих `C`, `D`, `E` в истории **нет** (они “сжаты”).

# 📕 2. Стратегии слияния и опции разрешения конфликтов

🔹 **Стратегии слияния** (`-s <strategy>`) — это **алгоритмы**, определяющие _как Git объединяет ветки_ (например, как ищется общий предок, какие коммиты участвуют, нужен ли merge-коммит).  

🔹 **Опции стратегии** (`-X <option>`) — это **дополнительные параметры**, которые управляют _тем, как Git разрешает конфликты_ или _какие приоритеты выбрать_ внутри конкретной стратегии (обычно `recursive`).

> - `-s` → определяет **алгоритм слияния (что делает Git)**
> - `-X` → управляет **поведением при конфликте (как Git решает спор)**

## 📙 1. Основные стратегии слияния (`-s <strategy>`)

### 🔶 1. `recursive` (по умолчанию)

🔸 Применяется при слиянии **двух веток**.  
🔸 Git ищет **общего предка** (merge base) и делает **трёхходовое слияние (3-way merge)**:
- сравнивает общий предок, HEAD и ветку;
- объединяет изменения построчно;
    
- при конфликте — требует ручного разрешения (если не заданы опции).

```bash
git merge -s recursive -X <option> <branch>
```

| Опция                       | Что делает                                                               | Пример                                         |
| --------------------------- | ------------------------------------------------------------------------ | ---------------------------------------------- |
| 💡`ours`                    | При конфликте оставляет версию из **текущей ветки**                      | `git merge -X ours feature`                    |
| 💡`theirs`                  | При конфликте выбирает версию из **сливаемой ветки**                     | `git merge -X theirs feature`                  |
|                             |                                                                          |                                                |
| `ignore-space-change`       | Игнорирует различия только в количестве пробелов                         | `git merge -X ignore-space-change feature`     |
| `ignore-all-space`          | Полностью игнорирует различия в пробелах                                 | `git merge -X ignore-all-space feature`        |
| `patience`                  | Использует более "умный" алгоритм сравнения строк (медленнее, но точнее) | `git merge -X patience feature`                |
| `diff-algorithm=<алгоритм>` | Задаёт конкретный алгоритм сравнения (`myers`, `histogram`, `patience`)  | `git merge -X diff-algorithm=patience feature` |
| `renormalize`               | Приводит файлы к одинаковому виду (EOL, атрибуты), если они различаются  | `git merge -X renormalize feature`             |

### 🔶2. `ours`
🔸 Не выполняет реального слияния содержимого.  
🔸 Git просто создаёт merge-коммит, в котором:
- сохраняется содержимое **текущей ветки**;
- **вся другая ветка считается объединённой логически**.

📘 То есть конфликты даже не проверяются — просто игнорируются.

```bash
git merge -s ours feature
```
Результат:
- файлы остаются как в `main`;
- ветка `feature` помечается как объединённая;
- **никакого анализа строк или конфликтов не происходит**.

### 🔶 3. `octopus`
🔸 Используется для **слияния более двух веток сразу**:
```bash
git merge -s octopus feature1 feature2 feature3
```
⚠️ Ограничения:
- Не умеет разрешать конфликты — если они есть, merge отменяется.
- Работает только при **автоматическом слиянии без конфликтов**.
