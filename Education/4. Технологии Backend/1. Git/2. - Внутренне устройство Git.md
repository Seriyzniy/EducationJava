# 📕 3. Статусы отслеживания файлов
Существует 4 статуса отслеживания файла
- 🔶`Untracked` (Неотслеживаемый)
Просто добавили файл в рабочую директорию

- 🔶`Staged` (Подготовленный)
Файлы добавленные в индекс (подготовили к коммиту).
Эти файлы отправляются в репозиторий

- 🔶`Unmodified` (Немодифицированный)
==Новые файлы, которые попали в репозиторий==, помечаются как `Unmodified` (Немодифицированный)

- 🔶`Modified` (Модифицированный)
Если изменить файл, который находится и в рабочей директории и в репозитории, тогда он становится `Modified` (Модифицированный)

# 📕 4. Типы объектов в Git
В Git существует **4 фундаментальных типа объектов**, которые хранятся в `.git/objects/`:

## 🔶 Blob (Binary Large Object)
> **Blob** — это просто “контейнер” ==для содержимого файла==.  
> Git не хранит имена файлов в blob — только **содержимое**.

SHA-1 (или SHA-256) от содержимого создаёт **уникальный идентификатор** blob.  
Если файл не изменился, его blob останется тем же — даже в другом коммите.

---
## 🔶 Tree
> **Tree** — это “==каталог==”, который связывает имена файлов с их blob или другими tree.

Пример структуры дерева:
```css
Tree (root)
├── src → tree (другая папка)
│   └── Main.java → blob
└── README.md → blob
```

Каждая запись в tree содержит:
```php
<права доступа> <имя файла> <SHA-1 объекта>
```
Таким образом, ==tree описывает структуру проекта на момент коммита==.

---
## 🔶 Commit
> **Commit** — это объект, который связывает **tree (==снимок состояния==)** и метаданные (кто, когда, зачем).
> (ССЫЛКА НА ДЕРЕВО)

Содержимое commit-объекта:
```php
tree <SHA-1 дерева>
parent <SHA-1 предыдущего коммита>   # может быть несколько — при merge
author <имя и email>
committer <имя и email>
message <сообщение коммита>
```

> [!NOTE] ❗❗❗ **Важно:**  ❗❗❗
> `Commit` ==НЕ ХРАНИТ изменения== (diff), а ==лишь **ссылку на состояние дерева**.==  
> Git сам вычисляет различия между снимками при необходимости.
> 

---
## 🔶Annotated Tag
> **Tag** — это объект для аннотированных тегов.

Содержимое tag-объекта:
```css
object <SHA-1 коммита>
type commit
tag v1.0
tagger <имя и email>
message "Release version 1.0"
```
Отличие от «легковесных» тегов:
- **Легковесный тег** — просто ссылка (`refs/tags/v1.0 → commit`).
- **Аннотированный тег** — полноценный объект с метаданными и подписью.


🎯 Зачем вообще нужны теги
> **Тег (tag)** — это просто **ярлык (==метка==)**, который ==указывает на конкретный коммит==
> Он помогает быстро найти и ссылаться на этот коммит в будущем.

❓Где теги применяются на практике
- **В CI/CD** — для автоматического деплоя по тегу (например, `v1.0` → билд релиза).
- **В changelog’ах** — чтобы определить, какие коммиты были между версиями.
- **В релизах на GitHub/GitLab** — релиз создаётся _на основе тега_.
- **Для отката к стабильной версии** — можно быстро сделать checkout

# 📕 5. Как git работает с данными

Каждый раз, когда мы сохраняем состояние проекта, Git запоминает, как выглядел каждый файл в проекте (`репозитории`) в этот момент времени, как бы делает снимок (`коммит`) репозитория. 
И сохраняет ссылку на этот коммит

![[Pasted image 20251024144242.png]]

## Переиспользование файлов
Если какие-то ФАЙЛЫ НЕ БЫЛИ ИЗМЕНЕНЫ вовсе, то Git то ИХ он НЕ ЗАПОМИНАЕТ, а просто ==ПОДТЯГИВАЕТ ССЫЛКИ== с предыдущих версий этого файла (коммитов)


> [!NOTE] ⭐⭐ Вычисление `hash-суммы` ⭐⭐
> Перед тем как сохранить файл, Git вычисляет hash-сумму файла, НА ОСНОВЕ ЕГО СТРУКТУРЫ (tree, blob).
> - Если `hash отличается` от предыдущей версии, то создается ==НОВЫЙ== объект.
> - Если `hash совпадает` с предыдущей версии, то ==ПОДТЯГИВАЮТСЯ ССЫЛКИ==


![[Pasted image 20251024144354.png]]

## Коммиты образуют цепочку
Каждый раз, когда мы делаем коммит,  Git ==ВЫЧИСЛЯЕТ HASH-сумму== этого коммита, И ТОЛЬКО ==ПОТОМ СОХРАНЯЕТ== ИНФОРМАЦИЮ

![[Pasted image 20251024163812.png]]

Каждый коммит содержит:
- Размер коммита
- данные автора
- ==ССЫЛКУ НА РОДИТЕЛЬСКИЙ КОММИТ== (хеш сумма родителя)
- сообщение коммита
- ссылку на дерево

И если хеш-сумма отличается, тогда происходит запись ИЗМЕНЕНЫХ ФАЙЛОВ
