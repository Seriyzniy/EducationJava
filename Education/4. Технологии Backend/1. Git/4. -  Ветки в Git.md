Как уже упоминалось, каждый коммит ЗНАЕТ СВОЕГО родителя.

# 📕 1. Указатель HEAD

> [!NOTE] **`HEAD`** — это `указатель на` `ТЕКУЩУЮ ветку` или `КОНКРЕТНЫЙ коммит`, с которым ты сейчас работаешь.
> 

## ✅ По умолчанию указывает НА ВЕТКУ
В обычном состоянии он ==указывает на текущую ветку==, а ветка уже на ПОСЛЕДНИЙ КОММИТ

💡Поэтому, когда ты делаешь новый `commit`, Git:
1. создаёт новый объект commit,
2. перемещает указатель `main` на него, (ветка ссылается на коммит)
    
3. `HEAD` остаётся прикреплённым к `main`

## ❓❗ HEAD указывает НА КОММИТ напрямую
> [!NOTE] Это состояние называется `detached HEAD` (отделённый HEAD).
> 

Допустим есть иcтория коммитов:
```css
A --- B --- C   ← main
```
Ветка `main` указывает на последний коммит, а HEAD соответственно на ветку

Например, если ты делаешь: (Переключились на коммит)
```bash
git checkout B
```

Теперь:
```css
main → C
HEAD → B (коммит напрямую)
```
Ты **не находишься ни на одной ветке**.  
Если ты сейчас сделаешь новый коммит, он не будет “привязан” к ветке.

### 📙 Что будет, если сделать коммит в detached HEAD?
Предположим, ты находишься в состоянии:
```css
A --- B --- C (main)
      ↑
     HEAD
```

Теперь ты изменяешь код и делаешь коммит:
```bash
git commit -m "experimental commit"
```
Git ==создаст новый коммит== `D`, **но не двинет ветку `main`**, потому что HEAD не прикреплён к ней.

Теперь история выглядит так:
```css
A --- B --- C (main)
      \
       D (HEAD detached)
```
То есть `D` существует, но **ни одна ветка на него не ссылается**.  


### 📙 Потерянный коммит
Если ты теперь вернёшься на `main`, например:
```bash
git checkout main
```
и не сохранишь `D`, он ==будет “потерян” при сборке мусора== (через несколько дней).


### 📙 "Спасение" потерянного коммита
Если тебе понравились изменения в detached HEAD, ==просто создай ветку==:
```bash
git switch -c experiment

либо
git branch имя_новой_ветки
```

Теперь:
```css
A --- B --- C (main)
      \
       D (experiment, HEAD)
```
Всё безопасно ✅  
Твой новый коммит закреплён в ветке `experiment`.

# 📕 2. Ветка
> [!NOTE] `Ветка` - ==всегда ссылается==  на `ПОСЛЕДНИЙ КОММИТ`
> Ссылка в ветке АВТОМАТИЧЕСКИ МЕНЯЕТСЯ при создании коммита

## 📙 1. Создание веток

> [!NOTE] 💡Ветка создается от того коммита, ГДЕ НАХОДИТСЯ `HEAD`
> Для создания веток используются команды `git branch, git checkout -b, git switch -c`

```bash
git branch [имя_ветки] 
# Управление ветками  


git branch myvetka 
# Создаст ветку myvetka
```

```bash
git checkout [ветка | commit | commit_hash] 
# Переключение между ветками или коммитами  

-b <имя>  # ⭐⭐СОЗДАТЬ⭐⭐ новую ветку и сразу перейти в неё  


# Современный вариант: 
git switch [ветка] 
git switch -c [новая_ветка]   # ⭐⭐СОЗДАТЬ и переключиться⭐⭐
```

## 📙 2. Перемещение по веткам

> [!NOTE] Перемещение между ветками
> Для перемещения между ветками используются команды `git branch, git switch`
> 
> ⭐⭐Эти команды ==ПЕРЕМЕЩАЮТ УКАЗАТЕЛЬ== `HEAD`⭐⭐

## 📙 3. Перемещение по "шагам" в истории коммитов 
Чтобы перемещаться по "шагам" по ветке коммитов, используется следующий синтаксис


> [!NOTE] Перемещение на один шаг назад в истории коммитов
> ```bash
> # Перемещение на ШАГ НАЗАД (К родителю ветки main)
> git checkout main^ 
> 
> # Перемещение на ВТОРОГО РОДИТЕЛЯ (когда текущий коммит - merge-commit) 
> git checkout main^2
> 
> # Перемещение на 2 ШАГа НАЗАД (К прародителю ветки main)
> git checkout main^^
> 
> # Перемещение НАЗАД на 4 шага
> git checkout HEAD~4
> ```


## 📙 4 История коммитов при перемещении
При этом, в истории коммитов будут находиться коммиты текущей ветки и коммиты до родительского коммита
![[Pasted image 20251025121048.png]]

Так например, если вызвать команду из ветки `feature-2` 
```bash
git log
```

То В ИСТОРИИ КОММИТОВ, будут коммиты ТЕКУЩЕЙ ВЕТКИ, и коммиты родителя и до родителя
