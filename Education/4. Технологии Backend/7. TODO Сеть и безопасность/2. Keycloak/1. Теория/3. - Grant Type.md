> [!NOTE] Grant Type
> Grant Type - это **способ, которым клиент получает Access Token** от сервера авторизации в OAuth 2.0 / OpenID Connect. 
> 
> ***(По сути у самого KeyCloak)***

Именно тип разрешения (grant) определяет, как и при каких условиях клиент получает токены. 
> Keycloak, как реализация Identity Provider и Authorization Server, поддерживает разные grant types.

(см [[3.4 -- OAuth2 и OpenID Connect]])
Типы grant types:
- Authorization Code
- PKCE
- Client Credentials
- Device Code
- Refresh Token

❌Устаревшие: 
- Implicit Flow
- Password Grant

---
## Authorization Code Flow
**Описание**: Стандартный безопасный flow для веб-приложений, где сервер клиента может хранить секреты.
**Тип приложений**: Веб-приложения (backend + frontend).

![[Pasted image 20260120121604.png]]

**Клиент формирует POST-запрос к token-эндпоинту** авторизационного сервера с обязательными параметрами:
`grant_type` -> Authorization Code	
`authorization_code` -> code, полученный на шаге 6
`redirect_uri` ->	тот же URI, что использовался при авторизации
`client_id` -> Идентификатор клиента
`client_secret` -> Секрет клиента

> [!NOTE] Проще говоря
> Похоже на обычное получение токена от KeyCloak

## Authorization Code + PKCE (Proof Key for Code Exchange)
**Описание**: Модификация Authorization Code Flow для публичных клиентов (например, мобильных и SPA), чтобы предотвратить атаку перехвата кода.
**Тип приложений**: Мобильные приложения, SPA (Single Page Applications).

Authorization Code = безопасно для серверов с хранением Client Secret.
Authorization Code + PKCE = безопасно для мобильных и браузерных клиентов без секретов.

Без PKCE	
- Нужен Client Secret	
- Уязвимость на этапе редиректа	
- Подходит для серверных приложений	
С PKCE
- Нужен только Code Verifier
- Защита на этапе редиректа
- Подходит для мобильных и SPA**

### Принцип работы
![[Pasted image 20260120122338.png]]
Клиент перед началом авторизации:
- **Генерирует случайный код** (code_verifier),
- ***Строит на его основе ХЕШ - code_challenge (обычно SHA-256)***,
- При первом запросе отправляется code_challenge.
- При обмене Authorization Code на Access Token отправляется code_verifier.
- Authorization Server сверяет code_challenge и code_verifier:
- Если всё сходится, только тогда выдает Access Token.

Это защищает от атаки перехвата кода, потому что даже если злоумышленник украдет Authorization Code, без правильного code_verifier он не сможет обменять его на Access Token.

## Client Credentials Flow
**Описание**: Классический machine-to-machine сценарий, когда приложение запрашивает токен от собственного имени, **без пользователя.**
**Тип приложений:** Сервисные взаимодействия (backend to backend), микросервисы.

## **Device Code Flow**
**Описание**: Для устройств без полноценной клавиатуры (например, Smart TV). Пользователь вводит код на другом устройстве.
**Тип приложений**: Устройства с ограниченным вводом - SmartTV, консоли, IoT.

![[Pasted image 20260120123801.png]]

---
## **Refresh Token Flow**
**Описание**: После получения Access Token можно получить новый токен без повторной аутентификации пользователя.
**Тип приложений:** Любые клиенты с долгоживущими сессиями.

![[Pasted image 20260120123957.png]]

## Implicit Flow (Устаревший)
**Описание**: Устаревший поток для браузерных приложений, ***когда токен выдавался сразу без обмена кода***. Небезопасен!

**Тип приложений**: Старые SPA (сейчас не рекомендуется использовать).

![[Pasted image 20260120124229.png]]

## Resource Owner Password Credentials (ROPC) Flow (Устаревший)
**Описание**: ***Клиент напрямую запрашивает логин/пароль пользователя*** и обменивает их на токен. Очень уязвимо.

**Тип приложений:** Доверенные приложения (сейчас категорически не рекомендуется использовать).
