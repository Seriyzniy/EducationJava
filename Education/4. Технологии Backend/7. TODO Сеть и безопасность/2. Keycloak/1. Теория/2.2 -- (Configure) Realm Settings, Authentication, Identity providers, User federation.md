> **Realm Settings** — это ==центр УПРАВЛЕНИЯ ВСЕМИ НАСТРОЙКАМИ ВАШЕГО REALM(домена безопасности).== 
> Здесь настраивается всё: от аутентификации и тем оформления до безопасности и интеграций.

## Опасные настройки (⚠️ Будьте осторожны!)
1. **Disable SSL** — только для тестовых сред
    
2. **Disable brute force protection** — никогда в production
    
3. **Very long token lifespans** — риски безопасности
    
4. **Allow duplicate emails** — может сломать workflows
    
5. **Disable email verification** — спам-регистрации

# 2. Authentication

> [!NOTE] Authentication
> В Keycloak, раздел Authentication **используется для настройки и управления потоками аутентификации**, ***обязательными действиями, и политиками входа***.
> 
> Authentication flow (поток аутентификации) - ==это набор шагов, через которые проходит пользователь== или клиент **при входе, регистрации, сбросе пароля и т.д.**
> 
> .
> *Каждый поток состоит из действий* (executions): например, 
> - проверка пароля, 
> - OTP, 
> - условные переходы, 
> - выбор IdP и др.

## Основные потоки аутентификации
### Browser flow
**Основной поток для входа пользователей через браузер**. 
Включает: 
- формы логина, 
- OTP (двухфакторную аутентификацию), 
- CAPTCHA и другие шаги

---
### Client authentication flow
**Поток аутентификации для клиентов** (*используется в client credentials grant*). 
> Работает без участия пользователя.

---
### Direct grant flow
***Поток для ресурсного владельца*** (Resource Owner Password Credentials Grant). 
**Используется, когда пользователь напрямую вводит логин и пароль** (например, через curl или мобильное приложение).

---
### Docker authentication flow
**Поток аутентификации Docker-клиентов** при доступе к приватному Docker Registry через Keycloak.

---
### First broker login flow
Поток, **срабатывающий при первом входе пользователя** ==через внешний Identity Provider== (например, Google, GitHub). 

Позволяет привязать внешний аккаунт к локальному пользователю Keycloak.

---
### Registration flow
**Поток регистрации новых пользователей.** 
Включает 
- форму ввода данных, 
- подтверждение по email 
- и проверки по политике безопасности (например, сила пароля).

---
### Reset credentials flow
**Поток для восстановления доступа**, *если пользователь забыл пароль.* 

> Может включать подтверждение по email, OTP и другие шаги для подтверждения личности.

# 4. User federation

> [!NOTE] User federation
> В Keycloak раздел User Federation **используется для ПОДКЛЮЧЕНИЯ К ВНЕШНИМ ИСТОЧНИКАМ ПОЛЬЗОВАТЕЛЕЙ**, таким как LDAP, Active Directory или Kerberos. 
> 
> Это ==позволяет аутентифицировать пользователей без хранения их в базе Keycloak.==
> 
> User Federation - это механизм интеграции Keycloak с существующими системами управления пользователями.

![[Pasted image 20260120105817.png]]

С помощью федерации:
- **пользователи остаются в LDAP/AD/Kerberos;**
- Keycloak может их аутентифицировать;
- можно синхронизировать или кэшировать данные (имя, email, роли и т.д.);
- поддерживается одноразовый вход (SSO), если внешняя система это позволяет.

## Add Kerberos providers
- Подключение к Kerberos (например, Active Directory с Kerberos-аутентификацией).
- Позволяет пользователям входить в Keycloak, не вводя логин и пароль, если они уже прошли Kerberos-аутентификацию на своём компьютере (SPNEGO / GSSAPI).
- Используется в доменных Windows-сетях для SSO (автоматический вход).

### Что такое Kerberos?

> [!NOTE] Kerberos
> Kerberos - это **протокол аутентификации**, ***основанный на криптографии и "билетах".***
> 
> Kerberos = безопасный вход в систему + SSO

![[Pasted image 20260120111809.png]]

Позволяет пользователю один раз войти в систему (SSO) и получать доступ к другим сервисам без повторного ввода пароля.

Часто используется в доменных Windows-сетях.

**Основан на централизованном сервере** (KDC - Key Distribution Center).

#### KDC - Key Distribution Center
> [!NOTE] KDC
> **Центральный сервер** *в протоколе Kerberos*, который о**твечает за аутентификацию пользователей и выдачу `билетов`** (tickets) для доступа к другим сервисам.
> 
> KDC - это ==сердце Kerberos, которое:==
> - аутентифицирует пользователей;
> - управляет безопасной выдачей доступа;
> - работает по принципу "один раз вошёл - везде доступен" (SSO).

##### Компоненты KDC
KDC состоит из двух основных компонентов:
- `Authentication Server` (AS)
**Проверяет логин и пароль пользователя** ==и выдает TGT== - ticket-granting ticket (билет доступа к другим билетам).
> *ТОЛЬКО ТУТ ПОЛЬЗОВАТЛЕЬ УКАЗЫВАЕТ ЛОГИН И ПАРОЛЬ*

- `Ticket Granting Server` (TGS)
**Получает TGT от пользователя и выдает сервисные билеты** (service tickets) - они ***используются для доступа к конкретным приложениям,*** например, файловому серверу или веб-приложению.

> **АНАЛОГИЯ**
> Это как парк развлечений:
> Сначала общий пропуск, а потом *билет* на определенный аттракцион
## Add LDAP providers
- Подключение к LDAP-серверам, включая OpenLDAP и Microsoft Active Directory.
- Keycloak может:
	- импортировать пользователей;
	- аутентифицировать их;
	- синхронизировать изменения;
	- редактировать (если включено).
	
Часто используется в корпоративных системах, где уже есть LDAP-инфраструктура.

### Что такое LDAP?
> [!NOTE] Lightweight Directory Access Protocol
> LDAP - это **протокол доступа к каталогу пользователей**, используемый для хранения и получения информации о пользователях, группах, ролях и другой организационной информации.
> 
> 🔥LDAP = база пользователей и их свойств

Пример: корпоративная база сотрудников (имена, логины, email, отделы).
Часто используется с Active Directory, OpenLDAP и другими системами.
Не занимается безопасной аутентификацией сам по себе - **только хранит данные.**

## Пример сценария:
Мы подключаем корпоративный LDAP, и пользователи могут логиниться в Keycloak (и в наши приложения), не создавая отдельные аккаунты.
