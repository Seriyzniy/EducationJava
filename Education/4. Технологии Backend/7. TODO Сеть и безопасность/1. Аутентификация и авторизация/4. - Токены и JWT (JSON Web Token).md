В современном веб-разработке для управления доступом к ресурсам широко применяются токены.

В этом разделе мы подробно разберём структуру, особенности и
практические аспекты использования токенов, а также рассмотрим важные вопросы безопасности.

---
# 1. Токен

> [!NOTE] 
> Токен - это **строка**, которая *подтверждает, что пользователь прошёл аутентификацию.*

> Он **передаётся** клиентом **ПРИ КАЖДОМ ЗАПРОСЕ к серверу**, чтобы получить доступ к защищённым ресурсам.

Используется в системах без сохранения состояния (stateless) и может содержать:
- ID пользователя
- Время жизни
- Подпись для защиты от подделки

## Stateless vs Stateful
В системах аутентификации есть 2 подхода как сервер может хранить информацию о том кто такой пользователь и какие у него есть права

### Stateless
Stateless:
- **Сервер не хранит сессию**
- ==Вся информация о пользователе - в токене==
- Удобно при масштабировании и распределении нагрузки
- Быстрая проверка - сервер проверяет токен на корректность и подпись не орбращаясь к БД
- **МИНУС** - отзыв соединения

> **Аналогия**
> Вы пришли в офис и получили пропуск С ЗАШИФРОВАННОЙ ИНФОЙ О ВАШЕМ ИМЕНИ и прочем.
> Охранник проверяет этот пропуск (проверяет подпись)
> 

### Stateful
Stateful:
- **Сервер хранит состояние** (*например, в БД* или Redis)
- ==Токен - лишь ссылка на данные==
- 🔥Легче отозвать доступ (удалить сессию)
- **МИНУС** -Если много открытых сессий, то большая нагрузка на память
- **МИНУС** - Проблема с масштабированием, чтобы сессии синхронизировались между серверами

> **Аналогия**
> Вы пришли в офис и получили пропуск, а охранник записал ваше имя и время входа в журнал.
> Каждый раз когда вы заходите в офис, охранник сверяется с этим журналом, чтобы проверить что пропуск выдан именно вам.

В нашем случае, сервер хранит СЕССИЮ - уникальную запись с данными пользователя (ID время входа и т.д).
Когда вы делаете запрос, сервер смотрит в журнале есть ли сессия с таким ID

# 2. JSON Web Tokew (JWT) 

> Посмотреть [JSON Web Tokens - jwt.io](https://www.jwt.io/)
## 1. Структура
JWT состоит из трёх частей, **разделённых точками:**
![[Pasted image 20260111131523.png]]
- `Header` - тип токена и алгоритм подписи (например, HS256)
- `Payload` - полезная нагрузка (userId, exp и т.д.) - Claims
- `Signature` - подпись, защищающая токен от подделки. **Подпись происходит при помощи ЗАКРЫТОГО ключа - ШИФРУЕТСЯ ХЕШ от header и payload.** Если изменить header или payload то подпись перестанет совпадать

Все части кодируются в base64 и легко читаются - токен можно
декодировать, но не подделать без ключа.

Сервер сначала смотрит на HEADER, чтобы понять, как проверять подпись.

## 2. Какие данные можно (и нельзя) класть в токен
JWT удобно *использовать для хранения минимально необходимой* информации:
- ID пользователя, 
- его роль (НЕ СЛЕДУЕТ - ПЛОХАЯ ПРАКТИКА), 
- дату истечения действия токена.

Но важно помнить: всё содержимое токена доступно для чтения - даже если оно подписано.

**Можно класть:**
- userId (sub), флаги 
- служебные поля: iat, exp, iss, aud

**Нельзя класть:**
- пароли или их хэши
- персональные данные

# 3. HMAC и RSA для ПОДПИСИ
Токены можно подписывать двумя способами:
## HMAC (HS256 и т.п.) - симметричная подпись:
Используется **один общий секрет**. 
И сервер, и проверяющая сторона должны знать этот ключ. 

> Простой и быстрый, но не подходит, если токен должен проверяться сторонним сервисом.

### Пример
Создается секретный ключ, который используется для создания подписи и ее проверки. **Хранится ТОЛЬКО НА СЕРВЕРЕ**
#### Создание подписи
Сервер берет HEADER и PAYLOAD токена, соединяет их и создает ХЕШ (ШИФРУЕТ) с помощью секретного ключа.
- Полученный хеш - подпись

Эту подпись СЕРВЕР прикрепляет к токену и передает клиенту для пользования

#### Проверка токена
Сервер, когда получает токен, он **заново вычисляет подпись** при помощи переданного в запросе HEADER и PAYLOAD и сравнивает ВЫЧЕСЛЕННУЮ ПОДПИСЬ с ПЕРЕДАННОЙ в токене

### ИСПОЛЬЗОВАНИЕ
Используется когда токен проверяет лишь один сервис

---
## RSA (RS256 и т.п.) - ассиметричная подпись
Применяется **пара ключей:** 
- приватный для подписи, 
- публичный - для проверки. 

Безопаснее в распределённых системах: можно отдать только публичный ключ другим сервисам, не раскрывая секрет.

### Пример
Сервер генерирует два ключа
- приватный
- публичный

Для создания подписи используется приватный ключ. А для проверки подписи используется публичный ключ

> Проверить подпись приватным ключом не получится (особенности алгоритма)

### ИСПОЛЬЗОВАНИЕ
Используется когда необходимо чтобы сторонние сервисы проверяли токен

# 4. Access Token vs Refresh Token
## Типы токенов
В системе обычно используются два типа токенов:
### Access Token
**Короткоживущий токен**, *с которым клиент получает доступ* к защищённым ресурсам. 
Обычно хранится в памяти или HttpOnly cookie. 
Если он утечёт - вред ограничен по времени.

### Refresh Token
**Более долгоживущий**, используется только **для обновления**
**access-токена.** 
==Хранится безопасно, например, в http-only cookie== или в базе данных на стороне сервера.

> **Refresh Token МОЖНО ИСПОЛЬЗОВАТЬ ТОЛЬКО ОДИН РАЗ!**

#### Принцип работы
Когда время действия Access Token подходит к концу, клиент отправляет Refresh Token на сервер, для получения нового Access Token.

Сервер проверяет Refresh Token, и если он валиден - отправляет новый Access Token, иначе - требуется повторная аутентификация

### Общий принцип работы токенов
#### Шаг 1: **Первичная аутентификация (логин)**
```
Пользователь вводит логин/пароль
    ↓
Сервер проверяет данные
    ↓
Сервер выдает:
┌─────────────────┬─────────────────┐
│ Access Token    │ Refresh Token   │
│ (15 минут)      │ (30 дней)       │
│ Для API запросов│ Только для      │
│                 │ получения нового│
│                 │ Access Token    │
└─────────────────┴─────────────────┘
```

#### Шаг 2: **Использование Access Token**
```
Пользователь хочет получить список друзей
    ↓
Отправляет запрос: GET /api/friends
    ↓
В заголовке: Authorization: Bearer {ACCESS_TOKEN}
    ↓
Сервер проверяет:
1. Подпись токена ✓
2. Не истек ли срок ✓
3. Есть ли права на /api/friends ✓
    ↓
Возвращает список друзей ✅
```

#### Шаг 3: **Access Token истек**
```
Через 15 минут...
    ↓
Пользователь отправляет запрос
    ↓
Сервер проверяет токен
    ↓
ОШИБКА: Token expired (истек)
    ↓
Клиент получает ошибку 401
```

#### Шаг 4: **Обновление Access Token**
```
Клиент видит ошибку 401
    ↓
Отправляет Refresh Token на специальный endpoint:
POST /api/refresh
Body: { "refresh_token": "xxxxx" }
    ↓
Сервер проверяет:
1. Действителен ли Refresh Token ✓
2. Не отозван ли он ✓
3. Есть ли у пользователя доступ ✓
    ↓
Сервер выдает НОВУЮ пару:
┌─────────────────┬─────────────────┐
│ Новый Access    │ Новый Refresh   │
│ Token (еще на   │ Token (еще на   │
│ 15 минут)       │ 30 дней)        │
└─────────────────┴─────────────────┘
    ↓
Клиент повторяет исходный запрос
с новым Access Token ✅
```

**Refresh Token МОЖНО ИСПОЛЬЗОВАТЬ ТОЛЬКО ОДИН РАЗ!**

После каждого успешного использования refresh токена для получения новой пары (access + refresh):
1. **Старый refresh токен УДАЛЯЕТСЯ из БД** (помечается как использованный)
    
2. **Выдается НОВЫЙ refresh токен**
    
3. Теперь для следующего обновления нужен **ЭТОТ НОВЫЙ ТОКЕН**

---
## Утечка токенов и её последствия
Если access-токен или refresh-токен попадёт в руки злоумышленника - он сможет получить доступ к учётной записи.

Утечка access-токена:
- Позволяет выполнить любые действия от лица пользователя.
- Н*о вред ограничен временем жизни токена*.

Утечка refresh-токена:
- **Гораздо опаснее**: *позволяет постоянно получать новые access-токены.*
- Используется для перехвата сессии - «тихого» захвата аккаунта.

# 5. Blacklist и revocation
JWT по своей природе не имеет встроенного механизма отзыва, ведь он проверяется локально и не требует обращения к серверу. 

Но отозвать токен всё же можно:
## 1. Серверный blacklist:
Сохраняем идентификатор токена (или его jti) в БД, помечая его как недействительный. 
*Каждый запрос сверяется с этим списком*.

**Минус**: теряется преимущество автономности токена.

## 2. Ротация refresh-токенов
Каждый раз при обновлении access-токена создаётся новый
refresh-токен. Старый считается недействительным.
### Токен украли
```
Пользователь: AT1 + RT1
Злоумышленник: [УКРАЛ RT1!]
    ↓
Пользователь ничего не знает
Злоумышленник ждет удобного момента
```
### Злоумышленник действует ПЕРВЫМ
```
Злоумышленник отправляет RT1 на сервер:
POST /refresh -> RT1
    ↓
Сервер проверяет: RT1 валиден ✓
    ↓
Сервер:
1. УДАЛЯЕТ RT1 из БД (отмечает как использованный)
2. Генерирует НОВУЮ ПАРУ:
   AT2 (15 мин) + RT2 (30 дней)
    ↓
Отправляет злоумышленнику: AT2 + RT2
```
### Настоящий пользователь пытается обновить токен
```
Пользователь отправляет RT1 на сервер:
POST /refresh -> RT1
    ↓
Сервер проверяет RT1 в БД
    ↓
ОШИБКА! RT1 уже использован/удален
    ↓
Сервер: 
1. Возвращает ошибку "Invalid refresh token"
2. ЛОГИРУЕТ событие: "Попытка использования уже использованного RT"
3. Отправляет уведомление пользователю:
   "Обнаружена подозрительная активность"
```

### Что теперь?
```
Злоумышленник имеет: AT2 + RT2 (работает)
Пользователь имеет: НИЧЕГО (не может обновить токены)

НО ЭТО ХОРОШО!
Потому что:
1. Пользователь УЗНАЕТ о взломе сразу
2. Может принудительно разлогинить все сессии авторизировавшись занаво и получив новые токены
3. Злоумышленник получил доступ лишь НА 15 МИНУТ (AT2)
```
