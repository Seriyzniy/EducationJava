Существует множество подходов, которые различаются по уровню защиты, удобству и области применения. 

Современные системы часто комбинируют несколько методов для повышения надёжности и улучшения пользовательского
опыта. 

Правильный выбор способов аутентификации зависит от целей
безопасности и особенностей сервиса.

# 1. Логин и пароль
Наиболее распространённый метод, при котором пользователь вводит свой логин и заранее установленный пароль.

Это базовая форма аутентификации, используемая практически во всех цифровых сервисах.

Преимущества:
- Прост в реализации и интеграции в системы.
- Пользователи уже знакомы с этой моделью.

Недостатки:
- Подвержен фишинговым атакам.
- Пользователи часто используют слабые и одинаковые пароли.

## Добавление других методов
Обычно, только этот метод не используют, а ДОБАВЛЯЮТ другие способы аутентификации (Двухфакторная аутентификация, одноразовый код, биометрия)

## Добавление соли - best practice
Добавление соли к хешу пароля предотвращает атаку Rainbow Tables (словарь)

### Rainbow Tables
**Rainbow Tables (радужные таблицы)** — это предварительно вычисленные таблицы хэшей паролей, которые позволяют быстро находить оригинальный пароль по его хэшу.

```python
# Упрощенный пример создания rainbow table
rainbow_table = {}

# Для каждого возможного пароля вычисляем хэш
for password in all_possible_passwords:
    hash_value = md5(password)  # или другой алгоритм
    rainbow_table[hash_value] = password

# Результат: огромная таблица [хэш → пароль]
```

### Как соль (salt) защищает от Rainbow Tables?
**Соль** — это случайные данные, которые добавляются к паролю перед хэшированием.

```java
// ❌ Без соли (уязвимо)
hash = bcrypt(password)

// ✅ С солью (защищено)
salt = random_bytes(16)
hash = bcrypt(salt + password)
```

```java
// У каждого пользователя УНИКАЛЬНАЯ соль
Пользователь А: 
  соль = "A1B2C3"
  хэш = bcrypt("A1B2C3" + "secret") = "HASH_A"

Пользователь B:
  соль = "X7Y8Z9"  ← ДРУГАЯ СОЛЬ!
  хэш = bcrypt("X7Y8Z9" + "secret") = "HASH_B"  ← ДРУГОЙ ХЭШ!

// Хэши разные, несмотря на одинаковые пароли!
```

## **Рекомендуемые алгоритмы хэширования** ПАРОЛЕЙ
| Алгоритм        | Стойкость          | Рекомендуется                         |
| --------------- | ------------------ | ------------------------------------- |
| **bcrypt**      | Высокая            | ✅ Да (для большинства случаев)        |
| **Argon2**      | Очень высокая      | ✅ Да (современный стандарт)           |
| **PBKDF2**      | Средняя-высокая    | ⚠️ Да, с большим количеством итераций |
| **SHA-256/512** | Низкая для паролей | ❌ Нет (слишком быстрый)               |
| **MD5, SHA-1**  | Очень низкая       | ❌ Категорически нет                   |

# 2. Email + код (magic link)
Пользователю **отправляется временный код** или **уникальная ссылка на почту.**

Он *вводит этот код или переходит по ссылке*, ***чтобы подтвердить свою личность.***

Преимущества:
- Уменьшает зависимость от паролей.
- Упрощает вход для пользователей, особенно с мобильных устройств.

Недостатки:
- Зависимость от доступа к email.
- Возможность компрометации почты открывает доступ к сервисам.

---
# 3. SMS-код / OTP (One-time password)
Пользователю **на номер телефона отправляется одноразовый код** (ОТР),который нужно ввести для входа. 

> Часто используется как второй фактор, номожет выступать и как основной.

Преимущества:
- Удобно для большинства пользователей.
- Дополнительный уровень защиты при 2FA.

Недостатки:
- Уязвимость к атакам через подделку SIM-карт и перехват SMS.
- Потеря доступа к номеру = потеря доступа к аккаунту.

# 4. Биометрия: отпечатки, Face ID, retina
Аутентификация с использованием уникальных биометрических характеристик пользователя. В основном применяется на мобильных устройствах и ноутбуках.

Данные биометрии также шифруются

Преимущества:
- Высокая скорость и удобство.
- Очень сложно подделать или украсть.

Недостатки:
- Потеря или утечка биометрических данных - серьёзный риск.
- Требует совместимого оборудования.

# 5. Push-уведомления
Пользователь пытается войти на аккаунт на ноутбуке/компьютере. Данный запрос попадает на сервер, который на привязанный смартфон или приложение отправляется уведомление с запросом «Разрешить вход?».

Пользователь подтверждает доступ одним нажатием.

Преимущества:
- Очень удобно, особенно для двухфакторной аутентификации.
- Позволяет мгновенно реагировать на попытки входа.

Недостатки:
- Необходимо установить приложение и включить уведомления.
- Уязвимость при компрометации телефона.

# 6. QR-авторизация
QR-авторизация - это метод входа в систему, при котором пользователь сканирует на экране специальный QR-код с помощью мобильного устройства.

Часто используется для входа в веб-приложения, когда пользователь уже авторизован в мобильном приложении сервиса.

## Как генерируется и используется QR-код?
*При попытке входа в систему* на экране компьютера **генерируется уникальный QR-код**, ==содержащий временный одноразовый токен или ссылку==, **связанную с сессией.**

Пользователь открывает мобильное приложение, сканирует этот код, и приложение отправляет на сервер подтверждение личности.

Таким образом, веб-сайт получает подтверждение о том, что владелец мобильного устройства - это тот самый пользователь, которому разрешен вход.

## Технические детали протокола
QR-код содержит уникальный идентификатор сессии и/или одноразовый **токен с коротким сроком действия**, чтобы предотвратить повторное использование и атаки воспроизведения.

Связь между мобильным приложением и сервером обычно происходит через защищенный канал (HTTPS), а обмен данными часто реализуется с помощью REST API или WebSocket.

> Сервер проверяет, что сканирование и подтверждение пришли от зарегистрированного устройства, и активирует сессию пользователя на компьютере.

# 7. USB-ключи и физические токены
USB-ключи и физические токены - это аппаратные устройства, которые подключаются к компьютеру или смартфону через USB, NFC или Bluetooth.

Являются ФИЗИЧЕСКИМ НОСИТЕЛЕМ закрытого криптографического ключа, который НИКОГДА НЕ ПОКИДАЕТ устройство

Они используются для подтверждения личности пользователя и обеспечивают высокий уровень безопасности.

Преимущества:
- Очень высокая степень защиты.
- Защита от фишинга и удалённого доступа.

Недостатки:
- Потеря или повреждение устройства = потеря доступа.
- Требуют дополнительной настройки и поддержки со стороны сервиса.

## Протоколы U2F и FIDO2
### Протокол U2F
Протокол U2F (Universal Second Factor) - это *стандарт*, разработанный *для обеспечения второго фактора аутентификации* **с использованием физических устройств.** 
Он **позволяет значительно повысить безопасность** по сравнению с
паролями, **ПОСКОЛЬКУ ТРЕБУЕТ ФИЗИЧЕСКОГО ПОДТВЕРЖДЕНИЯ через устройство**, например, нажатие кнопки на ключе.

### Протокол FIDO2
Позже был создан более универсальный стандарт FID02, который включает в себя два ключевых компонента: 
1) Web Authentication API (WebAuthn) и 
2) Client To Authenticator Protocol (СTAP).

#### Web Authentication API (WebAuthn) 
Это **интерфейс, встроенный в современные браузеры**, позволяющий веб-сайтам безопасно взаимодействовать с аппаратными устройствами для аутентификации пользователей.
#### СТАР 
Это **протокол**, который отвечает за связь между устройством
пользователя и аутентификатором, таким как USB-ключ. 

Вместе они дают возможность реализовать безопасную и удобную безпарольную аутентификацию.

#### Как работает
Когда пользователь пытается войти в аккаунт, **сервер генерирует**
**уникальный challenge - случайную строку**, *которую нужно подписать.*

**Пользователь подтверждает действие**, например, нажатием на кнопку USB-ключа.
*Устройство подписывает challenge своим приватным ключом* **и возвращает подпись серверу.** 

**Сервер проверяет подпись с помощью публичного ключа**, зарегистрированного ранее, и если всё совпадает, разрешает вход.

# 8. Сторонние сервисы (OAuth2)
Вход ==через сторонние сервисы== - Google, Github, Apple и др. Приложение получает доступ к аккаунту пользователя через протокол OAuth2.

Преимущества:
- Быстрый вход без регистрации.
- Удобно пользователям: не нужно запоминать новый пароль.
- Надёжные провайдеры обеспечивают безопасность.

Недостатки:
- Зависимость от стороннего сервиса.

# 9. SSO - единый вход
Единый вход для множества сервисов. *Пользователь проходит аутентификацию один раз и получает доступ к целой экосистеме.* 

Часто используется в компаниях.

SSO (Single Sign-On) - это технология, которая позволяет:
- Войти один раз - пользоваться множеством сервисов.
- Больше не вводить пароли повторно.
- Переходить между системами бесшовно.

## Где это полезно?
Особенно актуально в компаниях и организациях, где используются:
- Почта, CRM, ERP, облачные сервисы.
- Корпоративные порталы, внутренние системы.

Иначе бы пришлось логиниться каждый раз в очередном сервисе, и каждый раз придумывать пароль, что занимает много времени

## Как это работает?
1. Пользователь вводит логин и пароль один раз.
2. Сервис перенаправляет его на SS0-сервер (Identity-provider) (например, login.company.ru).
3. После успешного входа пользователь получает токен.
4. Этот токен принимают все доверенные сервисы.
5. Повторный вход - не нужен.

Токенами могут быть например
- XML в SAML
- JSON в OpenID Connect

## Что такое IdP и SP?
- IdP (Identity Provider) - проверяет, кто ты.
- SP (Service Provider) - даёт доступ к сервису.

**SP доверяет IdP** - нет нужды вводить пароль заново.

## Какие протоколы используются?
1. `SAML` (старый и сложный) - XML-протокол, часто в компаниях и госорганах.
2. `OpenID Connect` - построен поверx OAuth2, удобно для веба и мобильных. Использует JSON и REST API
3. `Kerberos` - часто используется в сетях Windows. Используется в локальных сетях
4. `Shibboleth` - федеративная аутентификация (вузовская среда). Реализует SAML

## Преимущества и недостатки
### Плюсы SSO для бизнеса
1. Централизованное управление доступом.
2. Повышение безопасности при правильной реализации.
3. Интеграция с Active Directory и другими системами.

### Минусы и риски
1. Один вход - один риск: если украден доступ, скомпрометированы все сервисы.
2. Требуется настройка взаимодействия между IdP и SP.
3. Нужно тщательно тестировать безопасность.

## SSO + 2FA = безопаснее
Чтобы снизить риски:
- Используют двухфакторную аутентификацию.
- Применяют биометрию, SMS-коды или аппаратные ключи

# 10. Аутентификация по IP/геолокации
Дополнительная проверка пользователя по IP-адресу или географическому положению. 

Часто используется для анализа подозрительной активности.

Преимущества:
- Повышает безопасность без участия пользователя.
- Можно применять как невидимый фактор.

Недостатки:
- Неточность и ложные срабатывания.
- Использование VPN или прокси может мешать.
