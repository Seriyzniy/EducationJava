Пароль - это всё ещё один из самых распространённых методов
аутентификации, но именно он чаще всего становится причиной взломов.

Современная безопасность не может опираться только на «логин + пароль».

Важно понимать, как пароли устроены, какие у них уязвимости, и как обеспечить их надёжное хранение.

---
# 1. Проблема слабых паролей
1. Пользователи выбирают лёгкие пароли - 123456, qwerty,
password.
2. Часто используют один и тот же пароль на всех сайтах.
3. Такой подход делает взлом делом пары минут.

## Базы утекших паролей
Пароли из взломанных сервисов часто выложены в сеть.

Примеры:
- RockYou.txt - 14+ млн паролей
- Collection #1 - 773 млн email-адресов

Хакеры используют их для атак по словарю.

> Проверить свой email можно на https://haveibeenpwned.com

## Генераторы паролей
Надёжный способ получить сложный пароль:
- не содержит слов из словаря
- случайный, длинный
- не используется повторно
 
Где генерировать:
- Менеджеры паролей (1Password, Bitwarden)
- Командная строка (pwgen, openssl rand)
- Локальные генераторы - безопаснее всего

---
# 2. Хеширование паролей
Пароль в базе не должен храниться в виде текста. Он преобразуется в хеш
- одностороннюю зашифрованную строку.
## Безопасные алгоритмы:
- bcrypt
- Argon2

## Устаревшие и небезопасные:
- MD5
- SHA1 / SHA256 без соли

## Как взламывают хеши
Основные методы:
- `Brute Force` - полный перебор возможных комбинаций
- `Dictionary Attack` - атака по базе популярных паролей
- `Rainbow Tables` - готовые хeши популярных паролей

# 3. Соль и перец
Чтобы даже одинаковые пароли имели разные хеши

## Соль
`Соль (salt)` - уникальная строка, добавляется к каждому паролю перед хешированием
### Цель соли:
- **Делает каждый хэш уникальным**, даже при одинаковых паролях
    
- **Защищает от rainbow tables** - нужна отдельная таблица для каждой соли
    
- **Предотвращает групповые атаки** - нельзя атаковать всех пользователей одновременно

```java
// Пример с солью
public class SaltExample {
    
    // Генерация случайной соли для КАЖДОГО пользователя
    public String generateSalt() {
        SecureRandom random = new SecureRandom();
        byte[] salt = new byte[16]; // 128 бит
        random.nextBytes(salt);
        return Base64.getEncoder().encodeToString(salt);
    }
    
    // Хэширование с солью
    public String hashWithSalt(String password, String salt) {
        String saltedPassword = salt + password;
        BCryptPasswordEncoder encoder = new BCryptPasswordEncoder(12);
        return encoder.encode(saltedPassword);
    }
}
```

## **Перец (Pepper)**
`Перец (pepper)` - **общий секрет**, добавляется отдельно, ==хранится вне БД==, а в защищенной переменной окружения или конфигурации сервера
### Цель перца:
- **Дополнительный секрет**, ==известный только приложению==
    
- **Защищает при компрометации БД** - даже с данными из БД нельзя вычислить пароли
    
- **Требует компрометации и БД, и конфигурации приложения**

```java
// Пример с перцем
public class PepperExample {
    
    // Pepper - общий секрет для ВСЕХ пользователей
    private final String GLOBAL_PEPPER = "my_super_secret_pepper_123!";
    
    // Или получаем из безопасного хранилища
    private String getPepper() {
        return System.getenv("APP_PEPPER");
        // Или из HashiCorp Vault, AWS KMS, etc.
    }
    
    
    // Хэширование с солью и перцем
    public String hashWithSaltAndPepper(String password, String salt) {
        String pepper = getPepper();
        String combined = salt + password + pepper;
        
        BCryptPasswordEncoder encoder = new BCryptPasswordEncoder(12);
        
        return encoder.encode(combined);
    }
}
```


## Основные различия

| Характеристика      | **Соль (Salt)**                    | **Перец (Pepper)**                  |
| ------------------- | ---------------------------------- | ----------------------------------- |
| **Назначение**      | Защита от rainbow tables           | Дополнительный секретный ключ       |
| **Уникальность**    | Уникальна для каждого пользователя | Общая для всех пользователей        |
| **Хранение**        | В БД рядом с хэшем                 | Вне БД (конфиг, KMS, env variables) |
| **Изменение**       | Можно менять индивидуально         | Меняется для всех сразу             |
| **Основная защита** | Против precomputed attacks         | Против компрометации БД             |

# 4. Политики сложности паролей
Надёжный пароль должен быть:
- длиной от 12 символов
- содержать буквы в разных регистрах, цифры и спецсимволы
- не содержать личных данных
