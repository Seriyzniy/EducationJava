> [!IMPORTANT] Сторонний сервис
> У стороннего сервиса (Google), **есть специальный сервис**, *который **позволяет** другим сервисам/компаниям/сайтам* **авторизовывать пользователя** 
> 

Тем самым, сторонний сервис (Google) передает ваши данные другим серсивам

### Но как Google знает кому передавать?
Чтобы какой-то сервис/приложение могло использовать OAuth через сторонний сервис, оно **(сервис/приложение) должно зарегистрироваться у этого стороннего сервиса**, через специальную панель, в которой указывается Callback

## 3. OAuth или OpenIDConnect (OIDC)
Далее есть два варианта
- использовать сторонний сервис для "быстрого входа"
- использовать сторонний сервис для выполнения действий на нем от имени пользователя (получение доступа к ресурсам пользователя)

### 1. Использование для быстрого входа (OIDC)

> [!NOTE] OIDC (OpenIdConnect)
> OIDC используется **только для аутентификации**
> С его помощью можно получить JWT токен с информацией о пользователе

![[Pasted image 20260118160830.png]]

Сторонний сервис дает доступ
- ФИО
- Почта
- другие публичные данные (аватарка и прочее)

С помощью этого, можно узнать, кто зашел в наш сервис, не регистрируя пользователя

### 2. Доступ к ресурсам OAuth2.0

> [!NOTE] OAuth2.0
> При помощи OAuth2.0 сервис, **получает доступ к ОПРЕДЕЛЕННЫМ ресурсам пользователя** (которые сервис запрашивает у пользователя)

![[Pasted image 20260118161156.png]]

В результате, сервис может выполнять действия с полученными ресурсами пользователя (планировать события в Google Calendar и прочее)

## 4. Редирект на сервис
Далее. после того, как пользователь зашел в аккаунт на стороннем сервисе (Googe), **происходит редирект на наш BACKEND сервис (сайт) с специальным кодом**
![[Pasted image 20260118162019.png]]

С этим кодом, наш **Backend сервис, дергает специальную API-ручку для получения токена**, и передает этот специальный код

> [!WARNING] Специальный код на Backend а не на Frontend
> Иначе бы этот код можно было бы украсть

## 5. Сервис делает запрос на получение токенов
Далее, сервис после получения кода, отправляет запрос на Googe(сторонний сервис) вместе с этим кодом, для получение токенов

Для этого, в запросе сервис передает
- Полученный код
- `ClientID` (*полученный при регистрации приложения в Google OAuth2.0*)
- `ClientSecret` (*полученный при регистрации приложения в Google OAuth2.0*)

Без `ClientID` и `ClientSecret` сторонний сервис (Google) не смог бы понять, что это именно наше приложение получит данные пользователя. а не какой-то мошенник

# 2. Детали реализации
## 1. Frontend
В приложении где находится ссылка на сторонний сервис (Google), когда пользователь нажимает "Войти через Google", то **вместе с запросом передается** `ClientID` **полученный ранее сервисом**

Также указывается scope (доступ к ресурсам) 

## 2. Backend
После получения токена, доступ к ресурсам пользователя происходит через backend (если дали доступ и был ранее запрос на это)

### Варианты использования
#### 1. Дальнейшее взаимодействие с пользователем
После авторизации пользователя через OAuth2.0, **для дальнейшего взаимодействия пользователя и сервиса** используются:
- Session and Cookie
- ==НОВОГО токена==, который сервис генерирует сам

> [!IMPORTANT] Токен от Google используется только для работы с Google
> 

**Если пользователь заходил на сайт через Google** ==то бэкенд создает и отдает пользователю сессию/токен== для авторизации

#### 2. Привязка аккаунта
Если пользователь привязывал свой Google аккаунт к уже существующему аккаунту на сайте, то может ничего не вернуться либо новый токен/сессия
