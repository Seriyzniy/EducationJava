Когда пользователь входит в систему - сайт должен "запомнить", кто он такой. 

Для этого и существуют сессии.

В этой главе разберём, как они работают:
- где и как хранятся,
- как долго живут,
- как обеспечивают безопасность,
- и чем отличаются от токенов.

# 1. Сессия - Statefull подход

> [!NOTE] 
> Сессия - это **способ «помнить» пользователя** *между запросами.*

### Принцип работы
Когда вы логинитесь на сайте, а **сервер генерирует уникальный идентификатор сессии** `Session ID` (UUID или другое) *и отправляет клиенту при помощи* специального **куки**, который автоматически будет прикрепляться к каждому последующему запросу

*Каждый следующий запрос сопровождается специальным идентификатором сессии*, по которому сервер понимает: 
- «Ага, это тот самый пользователь».

## Где хранятся сессии на сервере
Сессии можно хранить по-разному:
- В оперативной памяти сервера - быстро, но при перезапуске всё стирается.
- **В базе данных** (например, Redis, PostgreSQL) - устойчиво и масштабируемо.
- В самих cookie - сессия не на сервере, а в браузере (это ужеближе к токенам).

## Когда начинается и когда заканчивается
Сессия **начинается после логина**. Сервер создаёт уникальный идентификатор (session ID) и отдаёт его клиенту - обычно через cookie.

Сессия заканчивается, когда:
- пользователь вручную выходит
- истекает время жизни
- сессия удалена админом

## Что хранить в сессии
Сессия - это не просто факт входа пользователя. **Это объект с полезной информацией**, которая помогает системе понять, кто перед ней.

Обычно содержит:
1. Идентификатор пользователя
2. Уникальный токен сессии
3. IP-адpес и user-agent
4. Дата создания

## Плюсы и минусы
### **Плюсы Session-based Auth**

#### **1. Безопасность**
- **Сессия хранится на сервере** — клиент получает только идентификатор
    
- **Простой инвалидация** — удалил сессию на сервере, и доступ прекращён
    
- **Контроль срока жизни** — можно устанавливать разные политики:
    - По времени (30 минут неактивности)
        
    - По абсолютному времени (сессия на 24 часа)
        
    - Вечная (помнить меня)
#### **2. Хранение контекста**
Сессия может содержать временные данные пользователя:
- Корзина покупок
    
- Данные многошаговой формы
    
- Настройки UI (тема, язык)
    
- Временные файлы

### **Минусы и проблемы**

#### **1. Масштабируемость**
```java
// Проблема: несколько серверов
Пользователь → Сервер А (создал сессию) → OK
Пользователь → Сервер Б (не знает о сессии) → 401

// Решение: общее хранилище сессий
Все серверы → Redis/БД с сессиями → Работает
```

#### **2. Производительность**
Каждый запрос требует обращения к хранилищу:
```
Без сессий:    Запрос → Проверка подписи JWT → Ответ
С сессиями:    Запрос → Запрос к Redis/БД → Ответ
```

#### **3. Хранение состояния сервером**
Это противоречит REST-принципу stateless, что усложняет:
- Горизонтальное масштабирование
- Кэширование
- Отказоустойчивость

# 2. Чем сессия отличается от токена
Сессия - это ==данные, которые хранятся на сервере==. 
У клиента - только session ID. 
Можно в любой момент прервать сессию

Токен (например, JWT) - **это всё, включая данные**, ==хранится на стороне клиента.== Сервер проверяет подпись и ничего не хранит у себя.
Токен нельзя прервать 

Сессии - stateful (сервер запоминает),
Токены - stateless (сервер не хранит ничего).

# 3. Cookie
**Cookie** — это пара «ключ=значение», которую сервер отправляет браузеру, а браузер возвращает её обратно при следующих запросах.

![[Pasted image 20260118133817.png]]

Пример:
```bash
Set-Cookie: username=Alexey; sessionId=abc123; theme=dark
```

##### **1. Сервер отправляет Cookie**
```http
HTTP/1.1 200 OK
Content-Type: text/html
Set-Cookie: sessionId=abc123; Max-Age=3600; Path=/
Set-Cookie: username=Alex; HttpOnly

<html>...</html>
```
##### **2. Браузер сохраняет их**
Браузер сохраняет в специальном хранилище:
```bash
sessionId=abc123 (действует 1 час, для всего сайта)
username=Alex (только для сервера, не для JavaScript)
```
##### **3. Браузер автоматически отправляет их обратно**
```http
GET /profile HTTP/1.1
Host: example.com
Cookie: sessionId=abc123; username=Alex
```
##### **4. Сервер читает и "узнаёт" вас**
Сервер видит `sessionId=abc123` → ищет в своей базе → находит ваши данные → показывает ваш профиль.
