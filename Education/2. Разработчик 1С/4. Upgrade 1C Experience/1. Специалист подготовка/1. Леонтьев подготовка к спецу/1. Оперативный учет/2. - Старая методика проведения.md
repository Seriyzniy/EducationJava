# 3. Старая методика проведения - "Проверь, потом пиши"

> [!IMPORTANT] "Сначала ПРОВЕРЯЕМ остаток, потом СПИСЫВАЕМ"
> `Старая методика проведения` — это классический алгоритм, который использовался в системе долгие годы. 
> 
> Принцип старой методики контроля остатков следующий: 
> - **Сначала Проверяем, есть ли остаток товаров** в нужном количестве. 
> - Если есть — списываем, 
> - если нет — сообщаем об ошибке.
> 

Обычно используется для контроля отрицательных остатков (чтобы нельзя было списать товаров больше чем есть на складе)

---
## 3.1 Когда использовать и алгоритм

> [!NOTE] Отличие от НОВОЙ МЕТОДИКИ
> В **СТАРОЙ методике, мы сразу же получаем нужные данные** из нужных нам таблиц, **и затем по большой выборке делаем проверку**, и по результатам проверки записываем движения
>  
> А в **НОВОЙ методике, мы же сначала получаем данные** из одной таблицы, **формируем движения, и записываем их**. 
> - **И ТОЛЬКО ПОСЛЕ ЗАПИСИ проверяем** валидность записей п**о дополнительной таблице**

> [!NOTE] БЛОКИРОВКА
> В **СТАРОЙ используется** `управляемая блокировка` (вручную выбираем поля для блокировки)
> 
> В **НОВОЙ используется** `автоматическая блокировка`, которая блокирует вообще все

### 3.1.1 Когда использовать?
Старая методика ***используется в основном***, **когда данных в одной таблице НЕ ХВАТАЕТ ДЛЯ СОСТАВЛЕНИЯ ДВИЖЕНИЯ**, и нужно обращаться к другим таблицам.
> Допустим в движения по одному регистру, нужны данные из другого регистра связанные с данным документом

Обычно говорят, что старую методику используем когда нам нужно:
- Посчитать СЕБИСТОИМОСТЬ
- Вести учет по ПАРТИЯМ

### 3.1.2 Алгоритм
**Алгоритм работы старой методики:**
1. Программа устанавливает **управляемую блокировку** на данные (например, на сочетание Номенклатуры и Склада). Это необходимо, чтобы другие пользователи не могли изменить остатки, пока мы их анализируем.
	
2. Выполняется запрос к регистру накопления для получения текущих остатков.
	
3. В коде модуля проводится анализ: достаточно ли товара для списания?
	
4. Если товара хватает, рассчитывается себестоимость (обычно по методу «средней»).
	
5. Формируются и записываются движения в регистр.

## 3.4 МЕТОДИКА - Удаление движений и Блокировки
> [!IMPORTANT] ЗАМЕЧАНИЕ. МОМЕНТ РАСХОД В РЕГИСТРЕ = МОМЕНТ СПИСАНИЯ
> ВАЖНО, чтобы данные движений УДАЛЯЛИСЬ ПРЕД НАЧАЛОМ ПЕРЕпроведения.
> 
> Иначе, может случиться такая ситуация, когда 
> - **РАСХОД В РЕГИСТРЕ** ***зафиксировался ровно в тот момент, когда мы производим списание***
> - Мы проводим списание, **происходит проверка, что ДО НАШЕГО МОМЕНТА ВРЕМЕНИ (МОМЕНТ- 1 секунда)**, товары есть в регистре. 
> - И так как РАСХОД В РЕГИСТР ЗАПИСАЛСЯ ИМЕННО РОВНО В НАШ МОМЕНТ ВРЕМЕНИ, **то секунду назад там товары были**
> 
> И это не правильное поведение. Поэтому, нужно удалять старые движения при перепроведении, для того, чтобы можно было изменить данные

### 3.4.2 Блокировки
Блокировка нужна на тот случай, если НЕСКОЛЬКО ПОЛЬЗОВАТЕЛЕЙ будут проводить один и тот же документ в один и тот же момент времени

Если не использовать блокировки, то могут возникнуть аномалии данных (см [[2. - Уровни изоляций транзакций]])

Особенности блокировки в старой методике, заключается в том, чтобы ТОЧЕЧНО БЛОКИРОВАТЬ ЗАПИСИ, которые сейчас используются

Для блокировки используется элемент `БлокировкаДанных`
, которая состоит из элементов блокировки (см [[2. - Блокировки]]), которые мы определяем сами

![[Pasted image 20260308173839.png]]
> 💡 Готовый код можно получить из синтаксиса помощника, только затем изменить поля

Здесь мы создаем новый объект блокировки данных
Затем:
1. Указываем поля КАКОГО ОБЪЕКТА будем блокировать - регистр остатков
2. Указываем режим блокировки (нельзя rw пока работаем)
3. Указываем значение поля, строки которого будем блокировать *(ПОЛЕ РЕГИСТРА, ЗНАЧЕНИЕ ПОЛЯ ДОКУМЕНТА)*

Так как номенклатура у нас находится в Табличной Части, ***то для блокировки строки***, з**начение которой совпадает из массива данных,** необходимо указать `ИсточникДанных` - табличная часть

После этого указываем, значение из какого поля, должно совпадать с полем `ИсточникДанных`, чтобы заблокировать именно эту строку

По итогу, мы блокируем строки регистра, которые 
- Содержат значение из переменной Склад
- Содержат номенклатуру, которая есть в массиве данных "ИсточникДанных"

Ну и в конце вызываем метод `Заблокировать()`
