# 1. Константы - ИЗМЕНЯЕМЫЕ переменные, но общие на всю базу
**Константы в 1С** — это ==глобальные одиночные значения==, которые хранятся в информационной базе и доступны из любого места системы. 

> Это "синглтон" значения на всю базу данных.

## **Почему их называют "константами"?**
Термин **"константа"** в 1С означает **"постоянство в пространстве"** (одно на всю базу), а **НЕ "постоянство во времени"** (неизменяемое).

Лучшая аналогия — **"глобальная переменная"**, а не математическая константа вроде π.

## **Суть простыми словами:**
Это "коробка с одной важной настройкой" для всей системы. Например:
- **Название компании** — одно на всю базу
    
- **Основная валюта** — рубль, доллар, евро
    
- **Дата начала работы** — когда начали вести учет

# 2. Справочники - набор однотипных объектов какой-то сущности (Как коллекция в Java)
**Справочники в 1С** — это основа любой конфигурации. 

Это таблицы "мастер-данных" (master data), которые ==описывают долговременные сущности бизнеса.==

> [!NOTE] Мастер-данные
> **Мастер-данные** (Master Data) — термин, который означает **критически важную бизнес-информацию, которая используется в различных процессах компании**. 
> 
> В отличие от транзакционных данных (например, заказов или платежей), мастер-данные меняются редко и служат основой для аналитики и операционной деятельности.

## **Суть простыми словами:**
Это "база знаний" вашей системы. Например:
- **Каталог товаров** — что мы продаем
    
- **Картотека клиентов** — кому продаем
    
- **Штатное расписание** — кто работает
    
- **Список складов** — где храним

# 3. Документы - объект фиксации факта во времени
**Документы в 1С** — это **объект фиксации факта во времени**. 

Это не просто записи, а полноценные "команды" системы, которые меняют состояние учета.

## **Суть документа простыми словами**
Представьте документ как **бухгалтерскую проводку в бизнес-логике**
- **Накладная** → изменились остатки товаров
    
- **Платежное поручение** → изменился баланс счета
    
- **Акт выполненных работ** → появилась выручка


> [!NOTE] Суть
> **Ключевая идея:** Документ = событие, которое что-то меняет в системе.

## ДВИЖЕНИЕ ДОКУМЕНТА
Документ может находиться в 3 состояниях
- Записан - черновик
- Проведен - актуальный и проверенный
- Помечен на удаление - мусор
## Проведение и запись, разница

|**Запись**|**Проведение**|
|---|---|
|Сохранение **факта**|Учет **последствий**|
|Менеджер создал заказ|Бухгалтер провел его|
|Можно редактировать|Часто запрещает изменения|
|Не влияет на отчеты|Меняет отчеты сразу|

==Проведение = бизнес-транзакция:==
- проверка условий 
- расчёт данных
- запись в регистры
- блокировки

### **Пример из жизни:**
1. **Менеджер** создает счет (записывает) → клиент его видит
    
2. **Бухгалтер** проверяет (не проводит если ошибки)
    
3. **Клиент** оплачивает → бухгалтер проводит
    
4. **Остатки** на складе уменьшаются только после проведения

# 6. Отчёты - механизм анализа данных (по сути результат выполнения SQL)
**Отчёты в 1С** — это механизм анализа данных: агрегация, группировка, визуализация. 
Это read-only представления, которые показывают "что происходит" в системе.

**Важно:** Отчёт **не хранит** результаты. Он **вычисляет** их при каждом открытии.

## **СКД (Система Компоновки Данных) — основной движок**
СКД — это декларативный язык описания отчётов внутри 1С:
```yaml
📦 СКД-схема:
 ├── 📋 Источники данных:
 │    ├── Регистр.Продажи
 │    ├── Справочник.Номенклатура
 │    └── Справочник.Контрагенты
 │
 ├── 🎛️ Настройки:
 │    ├── Поля для вывода
 │    ├── Группировки (по месяцам, товарам)
 │    ├── Отборы (период, контрагент)
 │    └── Сортировка
 │
 └── 📊 Макет вывода:
      ├── Таблица
      ├── Диаграмма
      └️ Дерево
```

Обычно используется конструктор запросов, а не ручное написание SQL-запроса на русском языке

## **Архитектура отчёта: 3 слоя**
### **1. Слой данных (Data Layer)** 
SQL только на русском языке, или запрос составленный при помощи конструктора
```sql
-- Что происходит внутри:
SELECT 
    Товар,
    SUM(Количество) as Продано,
    SUM(Сумма) as Выручка
FROM РегистрПродаж
WHERE Дата BETWEEN @НачалоПериода AND @КонецПериода
GROUP BY Товар
```

### **2. Слой логики (Business Logic Layer)**
- **Расчетные поля** (наценка, маржа, процент)
    
- **Условное оформление** (красный если убыток)
    
- **Итоги и подытоги**

### **3. Слой представления (Presentation Layer)**
- **Таблицы** с группировками
    
- **Диаграммы и графики**
    
- **Сводные таблицы**
    
- **Экспорт** в Excel/PDF

# 8. Планы видов характеристик - динамическое добавление реквизитов(атрибутов) к объектам
**Планы видов характеристик (ПВХ) в 1С** — это 
- механизм для создания "расширяемых справочников"
- или системы динамических атрибутов. 

Это ==способ добавлять новые свойства объектам без изменения структуры== базы данных.

### Зачем нужны
Когда набор свойств:
- неизвестен заранее
- зависит от бизнеса
- должен расширяться без доработки

## Суть
Обычные реквизиты: **программист заранее решает**, какие поля будут у всех объектов.

ПВХ: **пользователь сам решает**, какие поля нужны каждому типу объектов, и может менять это в процессе работы.

## Реализация
У каждого объекта (справочники, документы) и т.д есть 1 атрибут (реквизит) - Характеристика, которое содержит ссылку на таблицу ПВХ.

Как только пользователь заполняет дополнительный атрибут (реквизит ПВХ) для объекта, то в поле Характеристика подставляется ссылка на этот атрибут из ПВХ таблицы

```
🔧 Механизм:
  1. Каждый товар → имеет ОДНУ ссылку на "паспорт"
  2. "Паспорт" → описывает ВСЕ характеристики товара
  3. Пользователь создаёт новые "паспорты" → новые записи в каталоге
  4. Система предлагает ВСЕ "паспорта" для товара
  5. При выборе → сохраняется только ССЫЛКА на "паспорт"
```

# 9. Планы счетов
> План счетов в системе «1С:Бухгалтерия» — это нормативно утверждённая структура всех бухгалтерских счетов компании. 
> 
> Она определяет:
> - назначение и содержание каждого счёта;
> - его нумерацию и порядок учёта операций;
> - возможности детализации через аналитические разрезы (субконто).


**План счетов** в 1С — это **метаописание бухгалтерской модели учёта**, которое определяет _какие счета вообще существуют_ и _какую аналитику по ним можно вести_.  
Сам план счетов **не хранит движения**, он задаёт **структуру**, по которой потом работают регистры бухгалтерии.

Каждый счёт в плане счетов — это не просто номер. Это объект с набором свойств:
- признак активный / пассивный / активно-пассивный;
    
- возможность использования валютного учёта;
    
- список допустимых аналитик (субконто);
    
- иерархия (родительские и дочерние счета)

## Как мыслить разработчику
План счетов — это **контракт бухгалтерского домена**.  
Если ты его меняешь — ты меняешь _семантику всех проводок в системе_.

# 10. Регистры - результат изменений (ИСТОРИЯ с обновлением значений как в Kafka) 

> [!NOTE] 
> **Документы — это events**  
> **Регистры — это state**

Регистры — это **основа всей учётной модели 1С**. Если упростить, то:
- документы **инициируют изменения**,
- регистры **хранят результат** этих изменений.

Регистры в 1С — это ==ОТДЕЛЬНЫЕ оптимизированные таблицы==, которые:
1. 🔥**Хранят предварительно рассчитанные агрегаты** (суммы, остатки, средние)
    
2. 🔥**Обновляются в реальном времени** при проведении документов
    
3. **Позволяют получать итоги мгновенно**, без сканирования всех документов
    
4. **Имеют прямую связь** с документами-источниками через Registrar_ID

### **Когда документ проводится:**
```
📄 Документ "Продажа" от 15.01.2024:
  iPhone - 10 шт

⚙️ Система делает ДВА действия:
  1. Записывает документ в таблицу "Документы_Продажи"
  2. ОБНОВЛЯЕТ регистр "ИтогиПродаж":
     Было: iPhone, Янв 2024 = 140 шт
     Прибавляем: +10 шт
     Стало: iPhone, Янв 2024 = 150 шт ← ОБНОВЛЕНО!
```

## 1. Регистры сведений - Что?Какой?Когда?

> [!WARNING] Ключевой вопрос
> Если регистру можно задать вопросы Что?Какой?Когда?, и получить один ответ - то тогда используется регистр сведений

**Регистры сведений в 1С** — это простейший тип регистров, который ==хранит "просто данные" без автоматической агрегации==. Это как умные таблицы с дополнительными возможностями.

> Принципиально важно: 
> **платформа ничего не считает автоматически**.  


Если тебе нужен «актуальный курс валюты» — ты сам определяешь:
- по максимальной дате ≤ нужной;
- по дополнительным условиям.

Поэтому регистры сведений используются как:
- справочные данные во времени (курсы, тарифы);
- состояния (настройки, статусы);
- параметры, которые «действуют с такого-то момента».

## 2. Регистры накопления - Сколько?

> [!WARNING] Ключевой вопрос
> Если регистру можно задать вопросы Сколько, и получить обобщеный ответ (число) - то тогда используется регистр накоплений

Регистры накопления — это уже **активный учёт**.  
Они хранят **движения**, но главное — **платформа сама считает остатки**

Представьте **бухгалтерскую книгу учета товаров**, которая:
1. Автоматически подсчитывает остатки
2. Мгновенно показывает итоги
3. Контролирует, чтобы не ушёл в минус
4. Хранит всю историю изменений

> **Регистр накопления = "живая книга", которая сама себя обновляет.**


Технически регистр накопления состоит из:
- измерений (например, Номенклатура + Склад);
- ресурсов (Количество, Сумма);
- движений двух типов: приход и расход.

### Очень важная деталь:  
**остатки не хранятся напрямую**, они _вычисляются_ по движениям (с оптимизациями внутри СУБД).

Поэтому:
- нельзя «поправить остаток» напрямую;
- нельзя обойти документ;
- всё идёт через движения.

## 3. Регистры бухгалтерии
Регистры бухгалтерии — это **строго формализованный вариант регистра накопления**, но с обязательной **двойной записью**.

Каждое движение содержит:
- счёт дебета;
- счёт кредита;
- аналитику по обоим счетам;
- сумму;
- период.

### Архитектурный смысл
Это **строгий, регламентированный state**, где свобода разработчика ограничена нормативами.

---
## 4. Регистры расчёта
Регистры расчёта — самые сложные концептуально. 

Они предназначены для **зависимых, периодических расчётов**, где:
- один расчёт может влиять на другой;
- возможны перерасчёты задним числом;
- важен порядок выполнения.

Представьте **цепочку домино для вычислений**:
1. Толкаете одно домино (меняете исходные данные)
2. Остальные падают автоматически в правильном порядке
3. Система знает, какие домино зависят от каких

> **Регистр расчёта = "интеллектуальный калькулятор", который сам разбирается с зависимостями и пересчётами.**


Классический пример — зарплата:
- оклад;
- премия;
- налог;
- удержания.

==Если меняется одно== — платформа знает, что ==нужно пересчитать всё зависимое.==

### Архитектурный смысл
Это **calculation engine**, а не просто хранилище данных.
