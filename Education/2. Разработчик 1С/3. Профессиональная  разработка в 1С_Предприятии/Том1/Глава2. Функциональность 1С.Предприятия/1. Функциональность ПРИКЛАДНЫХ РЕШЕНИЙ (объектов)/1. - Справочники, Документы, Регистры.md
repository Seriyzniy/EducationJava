***Значительная часть функциональности*** **прикладных решений**, создаваемых на платформе «1С:Предприятие», **определяется теми возможностями**, **которые содержит базовая реализация используемых объектов конфигурации.** 

Не претендуя на полное и глубокое изложение, рассмотрим лишь основные функциональные возможности некоторых объектов конфигурации. Кроме этого, рассмотрим также и некоторые общие объекты технологи ческой платформы, не реализующие конкретной прикладной специфики, но обеспечивающие важные функциональные возможности для прикладных решений

# 1. Справочники - базовые сущности на которых строится решение
Справочники описывают ***каталоги с относительно постоянным содержимым***: перечень продукции, список клиентов, валюты и т.д. 

Они обеспечивают **поддержку иерархических структур**, позволяют группировать данные и предоставляют ряд сервисных возможностей.

> [!NOTE] Ссылка на книгу
> ПОДРОБНЕЕ Глава 9 «Хранение информации», стр. 435.

## **Иерархия:**
- Включается простой активизацией соответствующего свойства в метаданных.
	![[Pasted image 20260226163512.png]]

- Поддержка иерархии автоматически распространяется на все аспекты использования объекта:
    - В объектной модели — определение уровня, контроль зацикливания.
        
    - В интерфейсе — отображение в виде иерархического списка или дерева с навигацией и интерактивным изменением.
        
    - В отчётах — формирование иерархических документов и получение многоуровневых итогов при группировке.

## Владелец
Вкладка **«Владелец»** в справочнике определяет **иерархическую зависимость одного справочника от другого**. 
> Простыми словами: *элементы одного справочника* (подчиненного) ***не могут существовать сами по себе***, они всегда привязаны к конкретному элементу другого справочника (владельца).

```java
// Класс-владелец (Контрагент) - ничего не знает о договорах
public class Kontragent {
    private UUID id;
    private String name;
    private String inn;
    // ... другие реквизиты
    // НЕТ поля List<Dogovor> dogovory!
}
// Подчиненный класс (Договор) - хранит ссылку на владельца
public class Dogovor {
    private UUID id;
    private String number;
    private Date date;
    
    private Kontragent owner;  // ← Вот здесь ссылка на владельца!
    // ... другие реквизиты
}
```

> [!NOTE] 
> 1. **Со стороны БД** (то, что я описал) — действительно, ссылка хранится в **дочернем** справочнике. Каждый договор "знает" своего контрагента через поле `OwnerID`.
> 2. **Со стороны интерфейса** (то, что вы увидели) — 1С автоматически создает **обратную связь** для удобства пользователя.

**Классический пример из жизни:**
- **Справочник-владелец:** "Контрагенты" (список фирм-клиентов).
    
- **Подчиненный справочник:** "Договоры контрагентов". У каждого договора обязательно должен быть "хозяин" — конкретный контрагент. Нельзя создать договор "просто так", без привязки к фирме.

# 2. Документы - факт совершения действия
(см [[2.2 -- Прочие вкладки]])

Документы ==отражают события в жизни предприятия==: поступление материалов, банковские операции, приём сотрудников и т.д. 
Их шаблон обеспечивает отражение в учётных механизмах, контроль последовательности событий и сквозную нумерацию объектов разных типов.

> [!NOTE] Ссылка на книгу
> ПОДРОБНЕЕ Глава 10 «Документы и последовательности», стр. 495.

## 2.1 Проведение документа
Ключевой механизм — **проведение документов**. Он предлагает стандартную модель связи между событиями предприятия и учётными механизмами. 
Любая информация, введённая в виде документов, может отражаться в планировании, управленческом и бухгалтерском учёте. 

Разработчику достаточно указать связь в метаданных и описать алгоритм проведения.

Система автоматически выполняет проведение и отмену проведения, поддерживает отражение событий в реальном времени и восстановление последовательности при изменениях задним числом. Это обеспечивает единую предсказуемую модель поведения всех прикладных решений, упрощая их освоение и поддержку.

# 3. Механизм характеристик (Планы Видов характеристик)
***Механизм характеристик*** **позволяет хранить свойства объектов** (справочников, документов и т.д.), **которые неизвестны на момент разработки** прикладного решения.

> [!NOTE] Ссылка на книгу
> ПОДРОБНЕЕ Раздел «Использование плана видов характеристик», стр. 483.

Например, для номенклатуры ***пользователь может самостоятельно вводить*** **новые** ***свойства***: цвет, размер, габариты, мощность.

Для разных групп номенклатуры можно создавать свои наборы свойств:
- для холодильников — объём морозильной камеры, число компрессоров, уровень шума;
    
- для компьютеров — объём оперативной памяти, объём жёсткого диска;
    
- для одежды — размер, рост, цвет.

Платформа позволяет описать дополнительные характеристики в дереве метаданных (свойство _Планы Видов Характеристики_). В дальнейшем на их основе можно строить отчёты, анализировать продажи и получать другую информацию для принятия решений. характеристик

# 4. Механизм сведений - РЕГИСТР СВЕДЕНИЙ
(см [[5.3 -- Периодические регистры сведений]])

> [!NOTE] 
> Механизм сведений (регистры сведений) позволяет хранить **произвольные данные** ==в разрезе нескольких измерений==.

> [!NOTE] Ссылка на книгу
> ПОДРОБНЕЕ Раздел «Хранение информации в регистрах сведений», стр. 455.

**Примеры:**
- курсы валют в разрезе валют;
- цены в разрезе номенклатуры и типа цен.

Ключевая особенность — возможность задать **периодичность хранения**. 
Это позволяет хранить не только текущие значения, но и **историю их изменений** во времени. 
> Например, для одной и той же номенклатуры можно хранить разные цены, действовавшие в разные периоды.

# 5. Механизм учета движения средств - РЕГИСТРЫ НАКОПЛЕНИЙ
(см [[5.1 -- Регистры Накоплений]] и [[5.2 -- Оборотные регистры накопления]])

Механизм (финансы, товары, материалы) автоматизирует складской учёт, взаиморасчёты, планирование. 
Его основа — **регистры накопления**.

> [!NOTE] Ссылка на книгу
> ПОДРОБНЕЕ Глава 11 «Учет движения средств», стр. 545

Регистр накопления образует многомерную систему измерений и ==накапливает числовые данные в их разрезе.== 
> Например: остатки товаров по номенклатуре и складам, объёмы продаж по номенклатуре и подразделениям.

Данные регистров используются в экономических и аналитических отчётах. Для ускорения работы с большими объёмами (сотни тысяч и миллионы записей) в платформе реализован **механизм агрегатов** для оборотных регистров. 

Система автоматически анализирует статистику запросов пользователей и выбирает оптимальный состав поддерживаемых агрегатов.

# 6. Механизм бухгалтерского учета
Механизмы бухгалтерского учета реализуют систему двойной записи, но не навязывают жёстких принципов — они позволяют создавать модели учета, применимые как в России, так и в других странах.

> [!NOTE] Ссылка на книгу
> ПОДРОБНЕЕ Глава 12 «Бухгалтерский учет», стр. 613

**Основные возможности:**
- ведение многоуровневых планов счетов с произвольной иерархией и переменной или фиксированной разрядностью кодов;
    
- аналитический учёт в нескольких разрезах и уровнях;
    
- одновременная работа с несколькими планами счетов;
    
- консолидированный учёт по нескольким юридическим лицам;
    
- возможность задания для отдельных разрезов аналитики различных видов учёта: количественный, суммовой, валютный и др.

# 7. Механизм сложных периодических расчетов - РЕГИСТРЫ РАСЧЕТА
Механизм позволяет реализовывать различные модели расчета заработной платы и других периодических расчетов со сложными взаимосвязями. 

> [!NOTE] Ссылка на книгу
> ПОДРОБНЕЕ Том 2, глава 13 «Сложные периодические расчеты».

Работа основана на двух составляющих:
1. **Описание видов расчета** — например, оклад, алименты, штрафы. Можно задавать правила влияния одних видов расчета на другие.
    
2. **Хранение данных** — как промежуточных, используемых для вычислений, так и конечных результатов.

Хотя наиболее типичное применение — расчёт зарплаты, механизм не привязан именно к этой задаче. Он успешно используется и для других целей, требующих периодических расчётов со сложными зависимостями: расчёт дивидендов, стоимости коммунальных услуг и т.д.
