# 1. Модуль объекта
Каждый прикладной объект конфигурации, данные которого могут быть модифицированы в режиме 1С:Предприятие, имеет свой модуль. Этот модуль исполняется при создании объекта встроенного языка, который позволяет модифицировать данные объекта конфигурации. Соответствующий объект встроенного языка создаётся, например, при вводе нового объекта, при копировании, при получении данных существующего объекта.

## 1.1 Названия модулей для различных объектов
![[Pasted image 20260301145620.png]]

### 1.1.1 Что за объекты "ДокументОбъект, СправочникОбъект"
Когда вы создаёте в конфигурации объект (справочник, документ, отчёт и т.д.), ***платформа автоматически создаёт соответствующий*** **объект встроенного языка**. 

> [!IMPORTANT] Объект - объект текущего прикладного объекта 
> Это программный объект, через который вы можете работать с данными этого объекта конфигурации в коде.

Эти объекты имеют имена вида:
- `СправочникОбъект.<ИмяСправочника>`
    
- `ДокументОбъект.<ИмяДокумента>`
    
- `ОтчетОбъект.<ИмяОтчета>`
    
- и так далее для каждого типа объектов

### 1.1.2 Объект vs Ссылка
Важно понимать разницу между **ссылкой** и **объектом**:

|                      | Ссылка                                               | Объект                                          |
| -------------------- | ---------------------------------------------------- | ----------------------------------------------- |
| **Тип**              | `СправочникСсылка.<Имя>`                             | `СправочникОбъект.<Имя>`                        |
| **Назначение**       | Указывает на элемент, но ==не позволяет его менять== | Позволяет ==читать и изменять== данные элемента |
| **Где используется** | В реквизитах, в отчётах, ***для быстрого доступа***  | ***Для изменения данных***, записи в базу       |
| **Методы**           | Только чтение данных                                 | Полный доступ + методы записи, удаления         |

```java
// Получаем ССЫЛКУ на контрагента
Ссылка = Справочники.Контрагенты.НайтиПоКоду("0001");
Имя = Ссылка.Наименование; // ✅ Можно читать
Ссылка.Наименование = "Новое имя"; // ❌ Ошибка! Ссылка не позволяет изменять

// Получаем ОБЪЕКТ для изменения
Объект = Ссылка.ПолучитьОбъект(); // Получаем ДокументОбъект.Контрагенты
Объект.Наименование = "Новое имя"; // ✅ Можно изменять
Объект.Записать(); // Сохраняем изменения в базу
```

### 1.1.3 Что значит "расширяется модулем"
Когда *вы пишете код в* **модуле объекта** (например, в модуле справочника), этот ==код становится частью объекта== `СправочникОбъект.<Имя>`.

1. Платформа создаёт базовый объект `СправочникОбъект` со стандартными свойствами и методами (Наименование, Код, Родитель, Записать(), Удалить() и т.д.)
    
2. Ваш модуль объекта **расширяет** этот базовый объект:
    - добавленные вами реквизиты становятся свойствами объекта;
        
    - написанные вами экспортируемые процедуры и функции становятся методами объекта.

---
## 1.2 Контекст модуля объекта
Контекст модуля объекта образуется:

- ***глобальным контекстом***, включая экспортируемые функции и процедуры общих модулей, скомпилированных на сервере (со свойством _Сервер_);
    
- свойствами и методами объекта встроенного языка, контекст которого расширяется модулем;
    
- **реквизитами объекта конфигурации**, которому принадлежит модуль;
    
- ***локальным контекстом самого модуля*** объекта.

![[Pasted image 20260301150233.png]]

## 1.3 Экспорт
Если переменные, процедуры или функции модуля объекта определены как экспортируемые, они становятся доступны в качестве свойств и методов соответствующих объектов встроенного языка.

### 1.3.1 Пример
В модуле справочника _ТиповыеАнкеты_ определена экспортируемая процедура:
```
Процедура ПроверитьКорректностьЗаполненияАнкеты() Экспорт
    // Алгоритм проверки
КонецПроцедуры
```

Вызов этой процедуры из модуля внешней обработки:
```
Анкета = 
Справочники.ТиповыеАнкеты.НайтиПоКоду("000000001").ПолучитьОбъект(); 

Анкета.ПроверитьКорректностьЗаполненияАнкеты();
```


## 1.4 События модуля объекта
Помимо описания переменных и основной программы, модуль объекта может содержать описание процедур-обработчиков событий.

> [!IMPORTANT] ОБЩИЕ СОБЫТИЯ ДЛЯ ВСЕХ ОБЪЕКТОВ
> Состав событий различен для разных объектов, но есть три общих события, вызываемых для всех объектов.

1. **ОбработкаПроверкиЗаполнения** 
	
2. **ПередЗаписью** 
    
3. **ПриЗаписи** 

![[Pasted image 20260301150307.png]]
### 1.4.1 Событие ОбработкаПроверкиЗаполнения
Вызывается **перед записью данных объекта, до начала транзакции записи**.

В обработчике разработчик может:
- реализовать собственные алгоритмы проверки заполнения реквизитов;
    
- добавить в массив проверяемых реквизитов дополнительные реквизиты для проверки платформой;
    
- очистить массив проверяемых реквизитов, отказавшись от стандартной проверки;
    
- установить параметр _Отказ_ в значение _Истина_, чтобы отказаться от записи объекта при невыполнении условий.
    ![[Pasted image 20260301150447.png]]

### 1.4.2 Событие ПередЗаписью
Вызывается **после начала транзакции записи, но до начала непосредственной записи данных в базу данных**.

Разработчик имеет возможность отказаться от записи данных, установив параметр _Отказ_, если, например, не выполнены требуемые условия.

### 1.4.3 Событие ПриЗаписи
Вызывается **после записи данных в базу данных, но до окончания транзакции записи**.

В обработчике выполняются действия, неразрывно связанные с данными объекта, которые не могут быть изменены отдельно от основных данных. Разработчик также может отказаться от записи данных, если в результате записи нарушаются какие-либо условия.

# 2. Модуль менеджера объекта

## 2.1. Назначение и функции
Для каждого прикладного объекта существует менеджер, предназначенный ==для управления этим объектом как объектом конфигурации.== 

> [!IMPORTANT] Менеджер - модуль для управления прикладный объектом
> С помощью менеджера можно создавать объекты, работать с формами и макетами.

Модуль менеджера позволяет расширить функциональность менеджеров, предоставляемых системой, за счёт написания процедур и функций на встроенном языке. 

Фактически **это позволяет описать методы для объекта конфигурации**, которые ***относятся*** не к конкретному экземпляру объекта базы данных, а **к самому объекту конфигурации.**

---
## 2.2. Контекст модуля менеджера
Контекст модуля менеджера образуется:
- глобальным контекстом, включая экспортируемые функции и процедуры общих модулей, скомпилированных на сервере (со свойством _Сервер_);
    
- локальным контекстом самого модуля менеджера.

![[Pasted image 20260301151804.png]]
### 2.2.1. Особенности исполнения
Модуль менеджера ==исполняется только на сервере== и **не может иметь переменных и тела модуля.**

---
## 2.3. Экспорт
Если функции или процедуры модуля менеджера объявлены как экспортируемые, к ним можно получить доступ через менеджера объекта.

### 2.3.1. Пример
В модуле менеджера справочника _Контрагенты_ определена экспортируемая функция:
```java
Функция ПолучитьСписокДебиторов() Экспорт
    // ...
КонецФункции
```

Вызов этой функции из модуля внешней обработки:
```java
Дебиторы = Справочники.Контрагенты.ПолучитьСписокДебиторов();
```

---
## 2.4. События модуля менеджера
В модулях менеджеров объектов располагается обработчик события `ОбработкаПолученияДанныхВыбора`. 

Это событие ***возникает на сервере*** **перед стандартным формированием списка** при 
- вводе по строке, 
- автоподборе текста 
- и быстром выборе.

# 3. Модуль управляемой формы

## 3.1. Общее описание
Каждая форма, определённая в конфигурации, имеет свой собственный модуль. 

> [!IMPORTANT] Исполняется при создании формы
> Модуль управляемой формы исполняется **при создании объекта** `УправляемаяФорма` встроенного языка. 

Он может создаваться при открытии формы прикладного объекта (например, при открытии формы элемента справочника) либо явно получаться средствами встроенного языка.

### 3.1.1. Пример получения формы
```java
ФормаСписка = ПолучитьФорму("Справочник.Номенклатура.ФормаСписка");
```

---
## 3.2. Контекст модуля управляемой формы
Модуль управляемой формы ==имеет два различных контекста== в зависимости от места исполнения: **клиентский и серверный.**

![[Pasted image 20260301152337.png]]
### 3.2.1. Контекст клиентских процедур и функций
Контекст клиентской части модуля управляемой формы образуется:
- *свойствами и методами глобального контекста*, доступными в управляемых клиентах (тонкий клиент, веб-клиент, толстый клиент в режиме управляемого приложения);
    
- ***экспортируемыми функциями и процедурами клиентских*** общих модулей (со свойством _Клиент (управляемое приложение)_) и неглобальных серверных общих модулей со свойством _Вызов сервера_;
    
- экспортируемыми переменными, процедурами и функциями модуля управляемого приложения;
    
- свойствами и методами расширения формы, определяемого основным реквизитом формы;
    
- свойствами и методами объекта `УправляемаяФорма` встроенного языка;
    
- **реквизитами формы, которой принадлежит модуль**;
    
- локальным контекстом самого модуля формы, скомпилированным в управляемых клиентах.

### 3.2.2. Контекст серверных процедур и функций
Контекст серверной части модуля управляемой формы образуется:

- *глобальным контекстом*, включая экспортируемые функции и процедуры общих модулей, скомпилированных на сервере (со свойством _Сервер_);
    
- свойствами и методами расширения формы, определяемого основным реквизитом формы;
    
- свойствами и методами объекта **УправляемаяФорма** встроенного языка;
    
- **реквизитами формы, которой принадлежит модуль**;
    
- **серверными процедурами и функциями самого модуля управляемой формы**, а также переменными и телом модуля, скомпилированными на сервере.

---
## 3.3. Доступ к объекту основного реквизита

> [!IMPORTANT] НЕТ ДОСТУПА к **МЕТОДАМ И СВОЙСТВАМ** `Модуля Объекта`, который является реквизитом формы
> В контексте управляемой формы недоступны свойства и методы объекта, который назначен основным реквизитом формы.
> 
> Получить к ним доступ можно только из серверной процедуры модуля формы, предварительно преобразовав данные формы в объект.

### 3.3.1. Пример
Пусть **в модуле объекта справочника** определена экспортируемая процедура:
```java
Процедура ПересчитатьЦены() Экспорт
    // ...
КонецПроцедура
```

В серверной контекстной процедуре модуля формы элемента этого справочника можно преобразовать данные формы в объект, выполнить метод объекта, а затем преобразовать обратно:

```java
&НаСервере
Процедура ОбновитьДанные()
    СправочникОбъект = РеквизитФормыВЗначение("Объект");
    
    СправочникОбъект.ПересчитатьЦены();
    
    ЗначениеВРеквизитФормы(СправочникОбъект, "Объект");
    
КонецПроцедуры
```

---
## 3.4. Исполнение модуля формы на клиенте и на сервере

### 3.4.1. Двойственная природа формы
**Форма** как программный объект ==существует одновременно на клиенте и на сервере.== Она состоит из двух взаимодействующих частей, каждая из которых выполняется в разных контекстах.

Все процедуры и функции в модуле формы должны иметь явное указание контекста исполнения с помощью **директив компиляции**.

### 3.4.2. Директивы компиляции

| Директива                               | Контекст          | Доступ к форме                                    | Применение                                                  |
| --------------------------------------- | ----------------- | ------------------------------------------------- | ----------------------------------------------------------- |
| **&НаКлиенте**                          | Клиент            | Полный доступ к реквизитам, элементам, параметрам | Обработчики клиентских событий, локальные команды           |
| **&НаСервере**                          | Сервер            | Полный доступ к контексту формы                   | Обработчики серверных событий, контекстные серверные вызовы |
| **&НаСервере<br>БезКонтекста**          | Сервер            | Контекст формы недоступен                         | Внеконтекстные серверные вызовы                             |
| **&НаКлиенте<br>НаСервереБезКонтекста** | Клиент или сервер | Контекст недоступен                               | Редко, для одинаковых действий на клиенте и сервере         |

> [!WARNING] ПО УМОЛЧАНИЮ директива `&НаСервере` **для любого метода в модуле формы**
> Если директива не указана, платформа по умолчанию использует **&НаСервере**.

### 3.4.3. Пример процедуры с директивой

```java
&НаСервере
Процедура ПриСозданииНаСервере(Отказ, СтандартнаяОбработка)

    Если Объект.Ссылка = ПланыОбмена.УдаленныеСклады.ЭтотУзел() Тогда
        Элементы.Главный.Доступность = Ложь;
    КонецЕсли;
    
КонецПроцедуры
```

---
## 3.5. Переменные модуля формы

### 3.5.1. Директивы для переменных
==Перед определением переменных также должны указываться директивы== компиляции.

|Директива|Время жизни|Доступность|
|---|---|---|
|**&НаКлиенте**|С момента создания до закрытия формы|Только в клиентской части|
|**&НаСервере** (или без директивы)|Только на время серверного вызова|Уничтожается после возврата на клиента|

![[Pasted image 20260301153327.png]]

### 3.5.2. Важные особенности
- Без указания директивы переменная получает директиву **&НаСервере**.
    
- ==Значение серверной переменной сохраняется только в пределах одного серверного вызова==. При разных вызовах переменная инициализируется заново.
    
- **Клиентские переменные** (`&НаКлиенте`) ==живут всё время жизни формы== и могут использоваться для хранения данных.
    
- ***Передавать значения с клиента на сервер через переменные модуля формы*** **нельзя**. Для этого используются:
    - реквизиты формы (при контекстных серверных вызовах);
        
    - **параметры вызываемой процедуры** или функции.

## 3.6. Экспортируемые процедуры формы

### 3.6.1. Назначение
Разработчик может создать **в модуле формы экспортируемые процедуры или функции.** 
> Они становятся доступны как методы программного объекта `Форма` во встроенном языке.

### 3.6.2. Директивы для экспортируемых процедур
Для экспортируемых процедур также необходимо указывать директивы компиляции. 

> [!NOTE] Директива для экспортируемых методов
> Практический смысл имеет использование только директивы **&НаКлиенте**.

**Объяснение:**
- Объект формы создаётся методами `ОткрытьФорму()` или `ПолучитьФорму()`, которые доступны **только в контексте клиента.**
    
- ==Созданная форма существует в контексте клиента==, поэтому её экспортируемые процедуры должны быть определены в этом же контексте.

**Важно:** Если директива не указана, платформа по умолчанию использует **&НаСервере**, и вызвать такую процедуру на клиенте будет невозможно.

# 4. Модуль команды

## 4.1. Общее описание
***У прикладных объектов конфигурации*** **существуют подчинённые объекты** `Команды`, а также общие объекты `Общая команда`. 

Каждая команда имеет свой модуль, в котором можно написать предопределённую процедуру `ОбработкаКоманды()`, описывающую действия, выполняемые системой при вызове команды.

Модуль команды, как и модуль управляемой формы, ==существует одновременно на сервере и на клиенте.==

---
## 4.2. Директивы компиляции
В модуле команды **могут использоваться следующие директивы**:

|Директива|Контекст исполнения|
|---|---|
|**&НаКлиенте**|Исполнение в управляемом клиенте|
|**&НаСервере**|Исполнение на сервере|
|**&НаКлиентеНаСервере**|Исполнение и в клиенте, и на сервере|

### 4.2.1. Особенности процедуры ОбработкаКоманды
Процедура `ОбработкаКоманды()` ==обязательно должна предваряться директивой== `&НаКлиенте`, так как **выполнение команды происходит в клиентском приложении.**

Из клиентских процедур модуля команды можно вызывать серверные процедуры, после их выполнения управление возвращается на клиента. 

Из серверных процедур вызывать клиентские **нельзя**.

---
## 4.3. Контекст модуля команды

![[Pasted image 20260301154338.png]]
### 4.3.1. Контекст клиентских процедур
Контекст клиентских процедур модуля команды образуется:
- свойствами и методами глобального контекста, доступными в управляемых клиентах (тонкий клиент, веб-клиент, толстый клиент в режиме управляемого приложения);
    
- экспортируемыми функциями и процедурами клиентских общих модулей (со свойством _Клиент (управляемое приложение)_) и неглобальных серверных общих модулей со свойством _Вызов сервера_;
    
- экспортируемыми переменными, процедурами и функциями модуля управляемого приложения;
    
- локальным контекстом самого модуля команды.

### 4.3.2. Контекст серверных процедур
Контекст серверных процедур модуля команды образуется:
- свойствами и методами глобального контекста;
    
- экспортируемыми функциями и процедурами общих модулей, скомпилированных на сервере (со свойством _Сервер_);
    
- серверными методами самого модуля команды.

---
## 4.4. Ограничения и особенности
- Модуль команды может содержать ==только описание процедур и функций.==
    
- В модуле команды **нельзя** использовать переменные и тело модуля.
    
- ❗❗❗Хотя в модуле команды можно описывать методы с ключевым словом **Экспорт**, из встроенного языка невозможно получить доступ к командам, а следовательно, и к их контексту.
