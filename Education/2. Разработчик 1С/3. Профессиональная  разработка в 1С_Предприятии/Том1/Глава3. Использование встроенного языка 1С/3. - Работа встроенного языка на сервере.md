# 1. Организация выполнения кода на сервере или на клиенте
Используя возможность управления компиляцией модулей, исполнение кода передаётся с клиента на сервер. 

После выполнения вызванной процедуры или функции система продолжает исполнение кода на клиенте.


Архитектура системы ориентирована на максимальный перенос выполнения всей функциональности на сервер и максимальное «облегчение» клиента:
- **На сервере** выполняется вся работа с прикладными объектами, подготовка к отображению форм (чтение объектов из базы, заполнение данных формы, расположение элементов, запись данных после изменения), подготовка командного интерфейса, формирование отчётов.
    
- **На клиенте** выполняется только отображение информации, подготовленной на сервере, взаимодействие с пользователем и вызовы серверных методов для выполнения необходимых действий.


После запуска прикладного решения выполнение кода всегда начинается на клиенте. 

Передача выполнения на сервер может быть выполнена путём:
- вызова серверных процедур из клиентских процедур модуля формы или команды (с директивами **&НаСервере**, **&НаСервереБезКонтекста**);
    
- вызова процедуры общего модуля, исполняющегося на сервере.

После завершения вызываемой процедуры выполнение возвращается на клиента.

# 2. Клиентские и серверные процедуры в модуле формы, команды

## 2.1. Основные принципы
***Модуль формы и модуль команды*** **существуют одновременно на клиенте и на сервере.** 

Процедуры в этих модулях исполняются 
- либо в контексте клиента, 
- либо в контексте сервера 
в зависимости от указанной директивы.

### 2.1.1 Ключевое правило:
- Из клиентских процедур (**&НаКлиенте**) можно вызывать серверные (**&НаСервере**, **&НаСервереБезКонтекста**), передавая выполнение на сервер.
    
- После выполнения серверных процедур исполнение возвращается на клиента.
    
- Принудительно передать выполнение с сервера на клиента **невозможно** — ==из серверных процедур нельзя вызывать клиентские.==

---
## 2.2. Пример
Требуется при изменении товара в накладной подставить цену из регистра сведений. Работа с регистром на клиенте невозможна.

**Клиентская процедура (обработчик события):**
```java
&НаКлиенте
Процедура ТоварыТоварПриИзменении(Элемент)
    // ...
    ЦенаТовара = РозничнаяЦена(Товар, Дата);
КонецПроцедуры
```

**Серверная функция (вызывается с клиента):**
```java
&НаСервереБезКонтекста
Функция РозничнаяЦена(ВыбранныйТовар, АктуальнаяДата)
    // ...
    Возврат Цена;
КонецФункции
```

> [!IMPORTANT] ОПТИМИЗАЦИЯ
> Если процедура используется часто, её целесообразно поместить в общем серверном модуле и вызывать из серверной процедуры модуля формы.
> 
> Короче, если часто используется метод сервера с указанием директивы, то лучше перенеси метод в общий модуль, и вызывать оттуда (ПРОСТО УМЕНЬШАЕТСЯ ДУБЛИРОВАНИЕ КОДА)

---
## 2.3 Рекомендации по оптимизации клиент-серверного взаимодействия
Для эффективной работы прикладного решения следует соблюдать следующие рекомендации:
- основной объём кода должен содержаться в **серверных процедурах** (с директивами **&НаСервере**, **&НаСервереБезКонтекста**);
    
- клиентские процедуры (**&НаКлиенте**) должны содержать небольшой и несложный для исполнения код;
    
- необходимо стремиться к **минимизации серверных вызовов** (ситуаций, когда из клиентских процедур вызываются серверные);
    
- везде, где возможно, нужно использовать **внеконтекстные серверные вызовы** (**&НаСервереБезКонтекста**) вместо контекстных (**&НаСервере**), так как это быстрее и эффективнее.

# 3. Вызов процедур из серверных модулей

## 3.1. Передача выполнения кода на сервер через общие модули
***Передача выполнения кода на сервер*** м**ожет быть осуществлена посредством вызова экспортируемой процедуры общего модуля**, 
> у которого установлены свойства `Сервер` и `Вызов сервера`. 

> [!IMPORTANT] Код из общего модуля - компилируется только на сервере
> Экземпляр этого модуля будет скомпилирован только на стороне сервера.

При вызове в модуле формы отчёта этой экспортной процедуры выполнение передаётся на сервер, в экспортируемую процедуру общего модуля. 
После её завершения управление возвращается на клиента.

### 3.1.1. Пример
Вызов процедуры неглобального серверного общего модуля (со свойствами _Сервер_ и _Вызов сервера_) из клиентского обработчика команды в модуле формы:

```java
&НаКлиенте
Процедура ВыполнитьНаСервере(Команда)
    Серверный.ВыполнитьНаСервере();
КонецПроцедуры
```

## 3.2. Цепочка вызовов на сервере
После того ***как выполнение передано на сервер***, **все последующие вызовы процедур** и функций **будут выполняться также на стороне сервера.**

![[Pasted image 20260301162517.png]]
### 3.2.1. Пример цепочки вызовов
Рассмотрим общий модуль **Расчеты**, который компилируется как на стороне сервера, так и на стороне клиента, и содержит экспортируемую процедуру `Пересчитать()`.
1. На клиенте в модуле формы вызывается экспортируемая процедура общего серверного модуля **Серверный**, и управление передаётся на сервер.
    
2. Из этой процедуры вызывается процедура `Пересчитать()`.
    
3. Выполнение продолжается на сервере, и процедура `Пересчитать()` вызывается из серверного экземпляра модуля **Расчеты**.
    
4. После её завершения управление возвращается на клиента.
    
5. Затем из модуля формы снова вызывается процедура `Пересчитать()` — теперь она будет вызвана из **клиентского** экземпляра модуля **Расчеты**.

---
## 3.3. Важные ограничения
**Разработчик не может управлять передачей выполнения с сервера обратно на клиента**. 

> Выполнение возвращается только после завершения вызванной процедуры или функции.

Если выполнение осуществляется на сервере, и вызываемая процедура не найдена в коде, скомпилированном на стороне сервера, ==будет выдано сообщение об ошибке== — **даже** в том случае, **если эта процедура присутствует в экземпляре, скомпилированном на стороне клиента.**

# 4. Передача параметров на сервер и возврат значений
При передаче выполнения кода на сервер **обычно выполняется передача параметров** вызываемой процедуре или функции. 

В случае вызова функции также происходит передача результата работы с сервера на клиента.

## 4.1. Ограничения на передаваемые типы
**НЕ ВСЕ ТИПЫ значений и объекты**, доступные на сервере, **МОГУТ БЫТЬ ПЕРЕДАНЫ С КЛИЕНТА НА СЕРВЕР и обратно**. 

Точная информация о возможности использования типов и их передаче содержится в документации. 

> В синтакс-помощнике ***объекты, которые могут передаваться*** между клиентом и сервером, **помечены фразой** _«Возможен обмен с сервером»_.


> [!IMPORTANT] Передавай ссылку на сервер, а на сервере по ссылке получай объект
> Поскольку на клиенте невозможна работа с объектными данными, на сервер можно передавать **ссылку** на объект, а в серверной процедуре получать по этой ссылке объект и работать с ним.

## 4.2. Временное хранилище для больших данных
Для обмена «специфическими» данными (например, файлами, картинками, табличными частями документа) предназначено **временное хранилище**:
- Метод **ПоместитьВоВременноеХранилище()** сохраняет сериализуемое значение во временном хранилище.
    
- Метод **ПолучитьИзВременногоХранилища()** извлекает значение по указанному адресу.

Временное хранилище может использоваться для передачи данных между формами, минуя клиента, что особенно эффективно при больших объёмах передаваемых данных.

# 5. Привилегированный и безопасный режимы исполнения кода
В процессе функционирования прикладного решения могут возникать ситуации, ***когда некоторые действия нужно выполнить максимально быстро***, **без проверки прав пользователя на используемые данные**, а при выполнении некоторых действий, наоборот, нужно оградить систему от выполнения «ненадёжного» кода, представляющего потенциальную угрозу для её безопасности. 

Для этого платформа поддерживает привилегированный и безопасный режимы исполнения кода.

## 5.1. Привилегированный режим
В привилегированном режиме код исполняется на сервере **без проверки прав**. 

Использование привилегированного режима позволяет:
- ускорить работу, так как не накладываются ограничения на доступ к данным;
    
- выполнять операции с данными от лица пользователей, которым эти данные недоступны.

### 5.1.1. Способы включения привилегированного режима
**Через свойства конфигурации:**
- Для документов можно задать использование привилегированного режима при проведении и при отмене проведения.
    
- Для бизнес-процессов можно задать использование привилегированного режима при создании задач.
    
- Эти свойства стандартно устанавливаются платформой для новых объектов.


**Программно:**
- Процедура `УстановитьПривилегированныйРежим()` — включает/выключает режим.
    
- Функция `ПривилегированныйРежим()` — определяет, включен ли режим.


**Привилегированные общие модули:**  
Можно ***создавать общие модули со свойством*** `Привилегированный`. Код в таких модулях выполняется без проверки прав. В них рекомендуется переносить операции, использующие данные, на которые у текущего пользователя нет прав.

### 5.1.2. Пример использования
Кладовщик имеет доступ к документу продажи, но не имеет доступа к регистру продаж, в который документ записывает движения. 

Использование привилегированного режима при проведении документа:
- не делает данные регистра доступными кладовщику;
    
- позволяет провести документ и сохранить движения в регистре.

---
## 5.2. Безопасный режим
Безопасный режим предназначен **для защиты системы от выполнения на сервере «ненадёжного» программного кода**. 

Потенциальную опасность представляют:
- внешние обработки;
    
- программный код, вводимый пользователем для использования в методах `Выполнить()` и `Вычислить()`.

Безопасный режим рекомендуется включать при выполнении на сервере кода, надёжность которого разработчик гарантировать не может.

### 5.2.1. Управление безопасным режимом
- Процедура `УстановитьБезопасныйРежим()` — включает/выключает режим.
    
- Функция `БезопасныйРежим()` — определяет, включен ли режим.
    
- При создании и подключении внешних обработок и отчётов платформа стандартно устанавливает параметр _БезопасныйРежим_ в значение _Истина_.

### 5.2.2. Ограничения в безопасном режиме
В безопасном режиме накладываются следующие ограничения:
- привилегированный режим отменяется (если был установлен);
    
- попытки перехода в привилегированный режим игнорируются;
    
- запрещены операции с COM-объектами;
    
- запрещена загрузка и подключение внешних компонентов;
    
- запрещён доступ к файловой системе (кроме временных файлов);
    
- запрещён доступ к Интернету.

# 6. Работа с метаданными
Встроенный язык «1С:Предприятия» позволяет работать с метаданными. 

> [!NOTE] Метаданные только для чтения
> Важная концептуальная особенность: доступ к метаданным предоставляется **только на чтение**.

Средствами встроенного языка разработчик не может модифицировать метаданные — изменение структуры возможно только средствами визуального конструирования в режиме Конфигуратор.

## 6.1. Объект Метаданные
Для доступа к метаданным из встроенного языка используется объект **ОбъектМетаданныхКонфигурация**, доступный через свойство глобального контекста **Метаданные**.

Этот объект предоставляет доступ:
- к свойствам самой конфигурации;
    
- к отдельным коллекциям объектов метаданных, описывающих различные виды прикладных объектов.

### 6.1.1. Доступ к коллекциям объектов
```java
// Доступ к коллекции документов
МетаданныеДокументов = Метаданные.Документы;

// Доступ к коллекции справочников
МетаданныеСправочников = Метаданные.Справочники;
```

---
## 6.2. Иерархическая структура метаданных
Объекты метаданных образуют иерархическую структуру. 

Элементами коллекций являются объекты метаданных, которые позволяют получить доступ как к специфическим свойствам объектов, так и к общим для различных объектов свойствам.

### 6.2.1. Получение описаний реквизитов и табличных частей

```java
// Для объекта метаданных "Документ.АккредитивПереданный":
// описание реквизита "ВалютаДокумента"
Объект = Метаданные.Документы.АккредитивПереданный.Реквизиты.ВалютаДокумента;

// описание табличной части "РасшифровкаПлатежа"
Объект = Метаданные.Документы.АккредитивПереданный.ТабличныеЧасти.РасшифровкаПлатежа;

// описание реквизита табличной части "ДоговорКонтрагента"
Объект = Метаданные.Документы.АккредитивПереданный.
    ТабличныеЧасти.РасшифровкаПлатежа.Реквизиты.ДоговорКонтрагента;
```

Реквизиты и табличные части объектов метаданных также представляют собой коллекции объектов метаданных.

---
## 6.3. Доступ к элементам коллекций
Все коллекции объектов метаданных допускают:
- обращение к элементам по **имени** или **индексу**;
    
- перебор с помощью итератора **Для Каждого**.

### 6.3.1. Пример перебора всех справочников

```java
Сообщение = Новый СообщениеПользователю;

Для Каждого Справочник из Метаданные.Справочники Цикл
    Сообщение.Текст = Символы.ПС + Справочник.ПолноеИмя();
    Сообщение.Сообщить();
    
    МенеджерСправочника = Справочники[Справочник.Имя];
    Выборка = МенеджерСправочника.Выбрать();
    Пока Выборка.Следующий() Цикл
        Сообщение.Текст = Выборка.Наименование;
        Сообщение.Сообщить();
    КонецЦикла;
КонецЦикла;
```

---
## 6.4. Методы для работы с метаданными
Объекты описания метаданных имеют набор стандартных методов (рис. 3.22), а объект **ОбъектМетаданныхКонфигурация** содержит два важных метода:

### 6.4.1. Метод НайтиПоТипу()
Позволяет найти объект метаданных, описывающий некоторое значение или тип.

### 6.4.2. Метод Содержит() коллекции объектов
Позволяет проверить, содержит ли коллекция указанный объект метаданных.

### 6.4.3. Метод Родитель()
Позволяет получить объект метаданных, находящийся на вышестоящем уровне иерархии.

## 6.5. Примеры использования
### 6.5.1. Определение типа регистра по движению документа

```java
// Выбирается первая запись из первого набора записей в движениях документа
Сообщение = Новый СообщениеПользователю;
Движения = Документы.АвансовыйОтчет.НайтиПоНомеру(
	"ТК000009", 
	'20110101000000').ПолучитьОбъект().Движения[0];
	
Движения.Прочитать();

ОбъектМетаданных = Метаданные.НайтиПоТипу(ТипЗнч(Движения[0]));

Если Метаданные.РегистрыНакопления.Содержит(ОбъектМетаданных) Тогда
    Сообщение.Текст = "Это движения по регистру накопления";
    
ИначеЕсли Метаданные.РегистрыСведений.Содержит(ОбъектМетаданных) Тогда
    Сообщение.Текст = "Это движения по регистру сведений";
    
ИначеЕсли Метаданные.РегистрыБухгалтерии.Содержит(ОбъектМетаданных) Тогда
    Сообщение.Текст = "Это движения по регистру бухгалтерии";
    
ИначеЕсли Метаданные.РегистрыРасчета.Содержит(ОбъектМетаданных) Тогда
    Сообщение.Текст = "Это движения по регистру расчета";
    
КонецЕсли;

Сообщение.Сообщить();
```

### 6.5.2. Определение принадлежности табличной части
```java
// Получается табличная часть "Товары" документа "АвансовыйОтчет"
Сообщение = Новый СообщениеПользователю;
ТабличнаяЧасть = Документы.АвансовыйОтчет.НайтиПоНомеру(
	"ТК000009", 
	'20110101000000').ПолучитьОбъект().Товары;
	
ОбъектМетаданных = Метаданные.НайтиПоТипу(ТипЗнч(ТабличнаяЧасть)).Родитель();

Если Метаданные.Справочники.Содержит(ОбъектМетаданных) Тогда
    Сообщение.Текст = "Это табличная часть справочника";
    
ИначеЕсли Метаданные.Документы.Содержит(ОбъектМетаданных) Тогда
    Сообщение.Текст = "Это табличная часть документа";
    
КонецЕсли;
Сообщение.Сообщить();
```
