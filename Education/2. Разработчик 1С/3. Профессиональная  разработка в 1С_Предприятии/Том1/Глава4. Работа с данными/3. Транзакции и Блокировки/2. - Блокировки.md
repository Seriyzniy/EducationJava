# 1. Возможные проблемы при многопользовательском доступе к одним и тем же данным
Работа в многопользовательской среде требует компромисса между требованиями целостности данных и параллельностью работы. При одновременном чтении и изменении одних и тех же данных могут возникать следующие проблемы.

## 1.1. Проблема потерянного изменения
Возникает, когда две транзакции изменяют одни и те же данные, взяв за основу начальное значение. В системе остаются изменения только последней транзакции.

**Пример:**
- Транзакция 1 читает цену товара: 420.
    
- Транзакция 2 читает цену того же товара: 420.
    
- Транзакция 1 изменяет цену на 450 и записывает.
    
- Транзакция 2 изменяет цену на 500 и записывает (поверх изменения транзакции 1).


**Результат:** Цена стала 500. Изменение, сделанное транзакцией 1, потеряно.

**Вывод:** нельзя одновременно изменять одни и те же данные без контроля.

## 1.2. Проблема «грязного» чтения
Возникает, когда одна транзакция читает данные, изменённые другой транзакцией, но ещё не зафиксированные. Если вторая транзакция откатывается, первая использовала несуществующие данные.

**Пример:**
- Транзакция 1 изменяет цену с 420 на 450 (не зафиксировано).
    
- Транзакция 2 читает цену (получает 450) и использует в расчётах.
    
- Транзакция 1 откатывается, цена возвращается к 420.
    

**Результат:** Транзакция 2 работала с данными, которых в итоге не стало в системе.

**Вывод:** нельзя читать изменённые, но ещё не записанные данные.

## 1.3. Проблема неповторяемого чтения
Возникает, когда одна транзакция несколько раз читает одни и те же данные, а другая транзакция изменяет эти данные между чтениями. При повторном чтении первая транзакция получает другое значение.

**Пример:**
- Транзакция 1 читает цену товара: 420.
    
- Транзакция 2 изменяет цену на 450 и фиксирует.
    
- Транзакция 1 снова читает цену: теперь 450.
    

**Результат:** В рамках одной транзакции одно и то же поле дало разные значения.

**Вывод:** нельзя допускать изменения данных между чтениями в одной транзакции.

## 1.4. Проблема чтения фантомов (Фантомное чтение)
Возникает, когда одна транзакция выполняет выборку данных по условию, а другая транзакция добавляет или удаляет записи, попадающие под это условие. При повторной выборке первая транзакция получает другой набор записей.

**Пример (расчёт премии по отделам):**
- Транзакция 1: выбрать всех сотрудников отдела А для расчёта премии.
    
- Транзакция 2: добавить нового сотрудника в отдел А.
    
- Транзакция 1 повторно выбирает сотрудников отдела А (появляется «фантом» — новая запись).
    

**Результат:** Одна и та же выборка в рамках одной транзакции дала разное количество записей.

**Вывод:** нельзя добавлять или удалять данные, которые могут попасть в уже прочитанный набор, до завершения транзакции.

## 1.5. Заключение
Приведённые проблемы демонстрируют необходимость механизмов, контролирующих одновременный доступ к данным. В 1С для этого используется **механизм управляемых блокировок**, который позволяет разработчику выбирать баланс между параллельностью работы и целостностью данных.

# 2. Уровни изоляции транзакций
Для увеличения производительности системы необходимо разрешить параллельное выполнение транзакций, но при этом обеспечить целостность данных. 

Строгость ограничений может быть различной в зависимости от задачи. В современных СУБД эта возможность реализуется через **уровни изоляции транзакций**.

## 2.1. READ UNCOMMITTED (незавершённое чтение)
Низший уровень изоляции, обеспечивающий максимальную параллельность выполнения транзакций.

**Особенности:**
- Защищает данные, изменяемые одной транзакцией, от изменений конкурирующими транзакциями.
    
- Если одна транзакция изменяет данные, другая транзакция, желающая изменить те же данные, должна ожидать завершения первой.
    
- Однако **чтение данных разрешено** даже незавершённых изменений.
    

**Допускает проблемы:**

- «грязное» чтение;
    
- неповторяемое чтение;
    
- фантомы.

## 2.2. READ COMMITTED
Уровень изоляции, запрещающий «грязное» чтение.

**Особенности:**
- Если одна транзакция начала изменять данные, конкурирующая транзакция не может ни изменить, ни **прочитать** их до завершения изменений первой.
    
- После завершения изменений другие транзакции могут читать данные, не дожидаясь окончания всей первой транзакции.
    

**Допускает проблемы:**

- неповторяемое чтение;
    
- фантомы.
    

## 2.3. REPEATABLE READ (повторяемое чтение)
Уровень изоляции, обеспечивающий повторяемость чтения данных.

**Особенности:**
- Если первая транзакция начала читать данные, другая транзакция не может их изменить до окончания первой транзакции.
    

**Допускает проблемы:**
- фантомы.

## 2.4. SERIALIZABLE (последовательное выполнение)
Максимальный уровень изоляции, обеспечивающий полную изоляцию транзакций друг от друга.

**Особенности:**
- Транзакции выполняются так, как если бы они выполнялись последовательно, одна за другой.
    
- Решает **все** рассмотренные проблемы:
    - потерянное изменение;
        
    - «грязное» чтение;
        
    - неповторяемое чтение;
        
    - фантомы.
        

## 2.5. Зависимость от СУБД
В зависимости от используемого уровня изоляции СУБД накладывает различные типы блокировок на объекты базы данных на разное время. Выбор уровня изоляции определяет баланс между производительностью и целостностью данных.

# 3. Режимы блокировок в 1С
В 1С:Прдеприятии есть 2 режима управления блокировоками
- `Управляемые` - управляются разработчиком
- `Автоматические` - управляются системой

Режим который используется в конфигурации, УКАЗЫВАЕТСЯ собственно в конфигурации
![[Pasted image 20260308172926.png]]

> [!NOTE] Он указывает режим блокировки, который используется для всех объектов
> 

## 3.1 ЗАМЕЧАНИЕ
Однако, у каждого объекта, можно установить собственный режим блокировки, который может отличаться от режима установленного в Конфигурации

НО
	Если в конфигурации установлен ОДИН РЕЖИМ 
	(`Управляемый`ИЛИ `Автоматический`)
	То в объектах будет использоваться именно тот режим, который указан в конфигурации, НЕСМОТРЯ НА СОБСТВЕННЫЕ НАСТРОЙКИ

ВТОРОЕ НО
	Если в конфигурации установлен ДВОЙНОЙ РЕЖИМ
	(`Управляемый` И `Автоматический`)
	То теперь, режим который установлен в объектах играет роль, и будет использоваться именно их локальный режим блокировки, а не тот что в конфигурации

# 4. Режим управляемых блокировок

Система «1С:Предприятие» стандартно поддерживает **управляемый режим блокировок данных** в транзакции. 

> [!IMPORTANT] Используется МИНИМАЛЬНО допустимый уровень изоляции
> При работе в этом режиме используется достаточно низкий уровень изоляции транзакций в СУБД, позволяющий достичь высокой параллельности работы пользователей.

![[Pasted image 20260302200312.png]]

Однако этот уровень изоляции транзакций СУБД не может сам по себе обеспечить целостность и непротиворечивость данных во всех случаях.

## 4.2. Собственный менеджер блокировок платформы
При модификации данных методами встроенного языка (например, методом `Записать()` для объектных данных) ==платформа устанавливает== **собственные управляемые блокировки** в транзакции, которые обрабатываются собственным менеджером транзакционных блокировок.

![[Pasted image 20260302200542.png]]

Эти блокировки также могут быть установлены разработчиком:
- 🔥**разделяемая блокировка** — для обеспечения неизменности считываемых в транзакции данных;
  _"Я сейчас смотрю на этот документ, чтобы принять решение. **Посмотреть с ним могут все**, **но пока я смотрю, никто не может в нем ничего исправить**."_
    
- 🔥**исключительная блокировка** — для запрета чтения данных другими транзакциями.
  _"Я переписываю этот договор. **Пока я не закончу, его нельзя не только менять, но даже читать**, чтобы никто случайно не увидел черновик."_

![[Pasted image 20260302201011.png]]

## 4.3. Преимущества управляемых блокировок
Управляемые блокировки «1С:Предприятия» учитывают логическую структуру прикладного решения, что позволяет:

- максимально точно блокировать необходимые области данных;
    
- избегать возникновения избыточных блокировок;
    
- блокировать только действительно необходимые области данных.

## 4.4. Обработка запросов
Любой запрос к данным обрабатывается следующим образом:
1. Сначала запрос проверяется собственным менеджером транзакционных блокировок «1С:Предприятия».
    
2. Если конфликт управляемых блокировок не обнаруживается, запрос передаётся на исполнение СУБД.
    
3. СУБД использует свой механизм блокировок с низким уровнем изоляции транзакций.

> [!WARNING] 2x проверка на изоляцию
> Сначала проверяется блокировка на объект ПЛАТФОРМОЙ, а затем СУБД (но с низкой изоляцией)


## 4.5. Установка РЕЖИМА управления блокировкой данных

> [!NOTE] Title
> Режим блокировок в конфигурации устанавливается с помощью свойства **Режим управления блокировкой данных**. Это свойство есть как у конфигурации в целом, так и у каждого прикладного объекта конфигурации.

![[Pasted image 20260302201801.png]]

Для платформы «1С:Предприятие» стандартным является режим управляемых блокировок. Поэтому для новых конфигураций свойство _Режим управления блокировкой данных_ устанавливается в значение **Управляемый**.

# 5. Установка управляемых блокировок
Средствами встроенного языка **установка управляемых блокировок** внутри явной или неявной транзакции ==выполняется с помощью специального объекта== `БлокировкаДанных`.

Новый экземпляр объекта создаётся конструктором ==и представляет собой коллекцию== `ЭЛЕМЕНТОВ БЛОКИРОВКИ ДАННЫХ`. 

***Изначально коллекция пуста*** — **разработчик добавляет в неё необходимое количество элементов**.

![[Pasted image 20260302202233.png]]

## 5.1. Пространства блокировок
При добавлении элемента блокировки необходимо указать **пространство блокировок**, которое будет блокировать данный элемент. 

Пространства блокировок определены в платформе и соответствуют структуре прикладных объектов конфигурации.
![[Pasted image 20260302202313.png]]

### **Особенности:**
- Для объектных данных (справочники, документы) определено единственное пространство блокировки — сам объект данных.
    
- Для необъектных данных определено два пространства блокировок:
    - с суффиксом **НаборЗаписей** — для блокировки самих записей объекта (например, ***при добавлении новых записей***);
        
    - без суффикса — для анализа данных объекта или операций, изменяющих ***существующие данные***.

---
## 5.2. Настройка элемента блокировки
После добавления элемента необходимо:
1. Установить **режим блокировки** (разделяемый или исключительный).
    
2. Определить **значения полей блокировки** — какие именно «записи» будут заблокированы.

![[Pasted image 20260302201011.png]]

Условия можно задавать двумя способами:
- явным указанием имени поля и его значения;
    
- с помощью источника данных, содержащего необходимые значения.

### 5.2.1. Важное замечание - БЛОКИРОВКИ НА ПЛАТФОРМЕ, СУБД ничего не знает о них
Управляемые блокировки описываются в терминах объектов метаданных и их полей, ==но никак не связаны с реальной структурой хранения данных в СУБД==. Это лишь записи о том, что заблокировано «нечто».

Для объектных данных блокировка объекта с указанной ссылкой будет соответствовать блокировке всех записей, содержащих эту ссылку, во всех таблицах объекта.

---
## 5.3. Совместимость блокировок
При установке новых блокировок менеджер анализирует имеющиеся:
- если режимы совместимы — новая блокировка устанавливается;
    
- если не совместимы — ожидает снятия существующей блокировки.

---
## 5.4. Способы задания условий

### 5.4.1. Явное указание имени поля и значения
Используется метод **УстановитьЗначение()** объекта `ЭлементБлокировкиДанных`:

```java
// Создать объект блокировка данных.
БлокировкаДанных = Новый БлокировкаДанных;

// Добавить новый элемент блокировки, блокирующий "нечто" в данных регистра
// накопления Остатки номенклатуры. 
ЭлементБлокировки = БлокировкаДанных.Добавить("РегистрНакопления.ОстаткиНоменклатуры");

// Установить режим блокировки – исключительный. Другие транзакции, устанавливающие 
// управляемые блокировки, не смогут даже начать чтение этих данных.
ЭлементБлокировки.Режим = РежимБлокировкиДанных.Исключительный;

// Указать, что именно мы блокируем в данных регистра Остатки номенклатуры – все "записи", у которых значение измерения Склад равно значению, содержащемуся в переменной Склад.
ЭлементБлокировки.УстановитьЗначение("Склад", Склад);
```

### 5.4.2. Задание диапазона значений
Для полей типа `Дата` или `Число` можно задать диапазон через объект **Диапазон**:

```java
// Создать объект блокировка данных.
БлокировкаДанных = Новый БлокировкаДанных;

// Добавить новый элемент блокировки, блокирующий "нечто" в данных регистра 
// накопления Продажи. 
ЭлементБлокировки = БлокировкаДанных.Добавить("РегистрНакопления.Продажи");

// Установить режим блокировки – разделяемый. Эти данные гарантированно не будут 
// изменены другими транзакциями до окончания существующей транзакции.
ЭлементБлокировки.Режим = РежимБлокировкиДанных.Разделяемый;

// Указать, что именно мы блокируем в данных регистра Продажи – все "записи", у которых значение измерения Контрагент равно значению, содержащемуся в переменной Контрагент.
ЭлементБлокировки.УстановитьЗначение("Контрагент", Контрагент);

// Создать объект Диапазон, описывающий интервал от начала месяца, к которому 
// принадлежит указанная дата, до указанной даты.
Диапазон = Новый Диапазон(НачалоМесяца(Дата), Дата);

// Указать, что именно мы блокируем в данных регистра Продажи – все "записи", у 
// которых значение измерения Контрагент равно значению, содержащемуся в переменной 
// Контрагент, и значение поля Период содержится в указанном диапазоне.
ЭлементБлокировки.УстановитьЗначение("Период", Диапазон);
```

### 5.4.3. Использование источника данных
Сначала задаётся свойство **ИсточникДанных**, затем метод **ИспользоватьИзИсточникаДанных()** настраивает соответствие полей:

```java
// Создать объект блокировка данных.
БлокировкаДанных = Новый БлокировкаДанных;

// Добавить новый элемент блокировки, блокирующий "нечто" в данных регистра 
// накопления Остатки номенклатуры. 
ЭлементБлокировки = БлокировкаДанных.Добавить("РегистрНакопления.ОстаткиНоменклатуры");

// Установить режим блокировки – исключительный. Другие транзакции, устанавливающие 
// управляемые блокировки, не смогут даже начать чтение этих данных.
ЭлементБлокировки.Режим = РежимБлокировкиДанных.Исключительный;

// Указать, что именно мы блокируем в данных регистра Остатки номенклатуры – все 
// "записи", у которых значение измерения Склад равно значению, содержащемуся в 
// переменной Склад.
ЭлементБлокировки.УстановитьЗначение("Склад", Склад);

// Указать источник данных, который содержит данные для установки ограничений на 
// другие поля этого элемента блокировки – в данном случае таблица значений 
// СписокНоменклатуры.
ЭлементБлокировки.ИсточникДанных = СписокНоменклатуры; 

// Указать, что именно мы блокируем в данных регистра Остатки номенклатуры – все 
// "записи", у которых значение измерения Склад равно значению, содержащемуся в 
// переменной Склад и у которых значение измерения Номенклатура равно какому-либо 
// значению, содержащемуся в колонке Номенклатура указанного источника данных.
ЭлементБлокировки.ИспользоватьИзИсточникаДанных("Номенклатура", "Номенклатура");
```

В качестве источника можно использовать:
- результат запроса;
- табличную часть;
- набор записей;
- таблицу значений.
    

Объект `Диапазон` также может быть значением поля источника данных.

## 5.5. Установка блокировок - метод `Заблокировать()`
Для применения всех созданных блокировок используется метод **Заблокировать()** объекта `БлокировкаДанных`:
- если метод вызывается **внутри транзакции** (явной или неявной) — блокировки устанавливаются немедленно ==и снимаются автоматически при окончании транзакции==;
    
- ❗❗❗если метод вызывается **вне транзакции** — блокировки не устанавливаются.
