# 1. Транзакции
Независимо от выбранного варианта работы (файловый или клиент-серверный) система «1С:Предприятие» обеспечивает работу с базой данных с использованием механизма **транзакций**.

## 1.1. Понятие транзакции
**Транзакция** — это неделимая с точки зрения воздействия на базу данных последовательность операций манипулирования данными, выполняющаяся по принципу «всё или ничего». 

Она переводит базу данных из одного целостного состояния в другое целостное состояние. Если какое-либо действие транзакции невыполнимо или происходит сбой, база данных возвращается в состояние до начала транзакции (выполняется **откат**).

Главное свойство транзакции — **атомарность** (неделимость).

## 1.2. Пример необходимости транзакции
Рассмотрим интерактивную пометку на удаление группы справочника. 

Фактически это последовательность операций:
1. Установить значение поля _ПометкаУдаления_ для группы.
    
2. Записать группу.
    
3. Выбрать все элементы справочника, находящиеся в иерархии помечаемой группы.
    
4. Для каждого из них установить _ПометкаУдаления_.
    
5. Записать каждый элемент.


Нет гарантии, что каждая операция будет успешна. Например, один из подчинённых элементов может редактироваться другим пользователем, и система не сможет его изменить. В результате часть элементов окажется помеченной, а часть — нет. С точки зрения системы такая ситуация недопустима и нарушает целостность базы данных: либо все элементы должны быть помечены, либо ни один.

Для обеспечения такого поведения система выполняет пометку на удаление **в транзакции**. Если в процессе происходит ошибка (например, элемент заблокирован), выполняется откат, и все изменения отменяются. Только при успешном завершении всех операций изменения фиксируются в базе.

## 1.3. Использование транзакций
### **Неявное использование:**  
==Система автоматически вызывает транзакции при любых действиях, связанных с модификацией данных в базе.== Например, все обработчики событий в модулях объектов и наборов записей, связанные с изменением данных, выполняются в транзакции.

### **Явное использование:**  
Разработчик может управлять транзакциями явно с помощью процедур глобального контекста:
- **НачатьТранзакцию()**
    
- **ЗафиксироватьТранзакцию()**
    
- **ОтменитьТранзакцию()**

---
# 2. Использование явного вызова транзакций

## 2.1. Основные процедуры управления транзакциями
- **НачатьТранзакцию()** — открывает транзакцию. Все последующие изменения базы данных могут быть либо целиком приняты, либо целиком отвергнуты.
    
- **ЗафиксироватьТранзакцию()** — принимает все изменения, выполненные в открытой транзакции.
    
- **ОтменитьТранзакцию()** — отменяет все изменения, выполненные в открытой транзакции, возвращая базу к состоянию до её начала.

### 2.1.1. Пример простой схемы работы с транзакцией
```java
НачатьТранзакцию();

// Последовательность операций
Если не <условие> Тогда
    ОтменитьТранзакцию();
КонецЕсли;
// Продолжение операций

ЗафиксироватьТранзакцию();
```

Если условие не выполняется, происходит откат всех изменений. Если всё успешно — изменения фиксируются.

# 3. Обработка ошибок базы данных в транзакции
Разработчик не всегда может предусмотреть все возможные ошибки. Для перехвата и обработки исключительных ситуаций используется конструкция **Попытка … Исключение … КонецПопытки**.

## 3.1. Общая схема с обработкой ошибок
```java
НачатьТранзакцию();

Попытка
    // Операции, которые могут вызвать ошибку
Исключение
    ОтменитьТранзакцию();
КонецПопытки;

ЗафиксироватьТранзакцию();
```

При использовании этой схемы важно учитывать, что ошибки базы данных делятся на два типа.

---
## 3.2. Категории ошибок базы данных

### 3.2.1. Невосстановимые ошибки
Ошибки, при которых нормальное функционирование системы может быть нарушено (например, порча данных).

При их возникновении выполнение программы прекращается в любом случае. Если такая ошибка произошла в транзакции, система автоматически выполняет откат.

### 3.2.2. Восстановимые ошибки
Ошибки, не вызывающие серьёзных нарушений. Операция, вызвавшая ошибку, прекращается, и возникает исключение, которое можно перехватить конструкцией _Попытка…Исключение_. Система **не выполняет автоматический откат** транзакции — это остаётся на усмотрение разработчика.

**Возможные сценарии:**
- Если ошибка не связана с базой данных, можно продолжить транзакцию.
    
- Если ошибка вызвана базой данных, дальнейшее продолжение или фиксация транзакции становятся невозможны. Единственное допустимое действие — отмена транзакции, после чего можно повторить попытку.
### 3.2.3. Пример цикла повторных попыток
```java
Записано = Ложь;

Пока НЕ Записано Цикл
    Попытка
        НачатьТранзакцию();
        Данные.Записать();
        ЗафиксироватьТранзакцию();
        Записано = Истина;
    Исключение
        ОтменитьТранзакцию();
    КонецПопытки;
    
КонецЦикла;
```

# 4. Вложенный вызов транзакций
В рамках уже выполняемой транзакции можно вызывать процедуры `НачатьТранзакцию()`, `ЗафиксироватьТранзакцию()` и `ОтменитьТранзакцию()`. 

Например:
```java
НачатьТранзакцию();
    // ... операции ...
    НачатьТранзакцию();  // "вложенный" вызов
        // ... операции ...
    ЗафиксироватьТранзакцию();
    // ... операции ...
ЗафиксироватьТранзакцию();
```

> [!IMPORTANT] ВСЕГДА ОДНА ТРАНЗАКЦИЯ 
> Однако **«1С:Предприятие» не поддерживает вложенных транзакций**. 
> Такое обращение не означает начала новой транзакции — всегда действует только транзакция самого верхнего уровня. 
> 
> И если "откатить" внутреннюю транзакцию, то откатится вся внешняя транзакция
> 

## 4.1. Механизм счётчика транзакций
Поведение транзакций управляется внутренним **счётчиком транзакций**:
- **НачатьТранзакцию()** — увеличивает счётчик на 1. Новая транзакция реально начинается только при увеличении счётчика с 0 до 1.
    
- **ЗафиксироватьТранзакцию()** — уменьшает счётчик на 1. Реальная фиксация данных происходит только при уменьшении счётчика с 1 до 0.
    
- **ОтменитьТранзакцию()** — при значении счётчика > 1 не отменяет изменения немедленно, а устанавливает специальный признак. Последующая фиксация при счётчике = 1 приведёт к отмене всей транзакции верхнего уровня.> Все вызовы внутри неё относятся к той же транзакции.

---
## 4.2. Важные выводы
- Нельзя быть уверенным, что `НачатьТранзакцию()` действительно начинает новую транзакцию — это зависит от текущего значения счётчика.
    
- Аналогично, `ЗафиксироватьТранзакцию()` и `ОтменитьТранзакцию()` не всегда завершают транзакцию.
    
- Из-за этой сложности при возникновении ошибки в транзакции часто разумнее:
    
    - сообщить пользователю об ошибке;
        
    - предоставить ему возможность повторить операцию или устранить причину ошибки перед повторной попыткой.

# 6. Влияние транзакций на работу программных объектов
Механизм транзакций обеспечивает атомарность и согласованность изменений в базе данных. 

Работа с данными осуществляется через программные объекты, доступные во встроенном языке. Результат такого опосредованного взаимодействия не всегда очевиден.

## 6.1. Прозрачность программных объектов для транзакций
Программные объекты **абсолютно прозрачны** для транзакций базы данных. 

Это означает, что транзакции могут вызываться при выполнении методов объектов, но действия базы данных (например, откат) никак не влияют на соответствующие программные объекты.

### 6.1.1. Пример: изменение наименования в транзакции с последующим откатом
Рассмотрим пошагово пример изменения наименования элемента справочника в транзакции, которая затем отменяется.

**Шаг 1. Получение объекта справочника**
```java
Товар = Справочники.Номенклатура.НайтиПоКоду("000000001").ПолучитьОбъект();
// В базе данных и в свойстве программного объекта — "Старое наименование"
```
В этот момент и в базе данных, и в программном объекте хранится одинаковое значение — "Старое наименование".


**Шаг 2. Изменение наименования в транзакции**
```java
НачатьТранзакцию();

Товар.Наименование = "Новое наименование";
// В программном объекте — "Новое наименование", 
// в базе — "Старое наименование"
```
После изменения свойства программного объекта наименования в базе и в объекте разошлись.


**Шаг 3. Запись объекта**
```java
Товар.Записать();
// После успешной записи и в базе, и в объекте — "Новое наименование"
```
Запись фиксирует изменения в базе данных — теперь наименования совпадают.


**Шаг 4. Отмена транзакции**
```java
ОтменитьТранзакцию();
// В базе данных — "Старое наименование" (откат)
// В программном объекте — "Новое наименование" (не изменилось)
```
После отмены транзакции база данных возвращается к исходному состоянию, но программный объект сохраняет изменённое значение.

## 6.2. Проверка рассинхронизации
Сравним значения в базе и в программном объекте:
```java
Запрос = Новый Запрос("ВЫБРАТЬ
    | Номенклатура.Наименование
    |ИЗ
    | Справочник.Номенклатура КАК Номенклатура
    |ГДЕ
    | Номенклатура.Код = ""000000001""");
    
Выборка = Запрос.Выполнить().Выбрать();

Выборка.Следующий();
НаименованиеВБазе = Выборка.Наименование;      // "Старое наименование"
НаименованиеВПамяти = Товар.Наименование;      // "Новое наименование"

Сообщить("В памяти: " + НаименованиеВПамяти);
Сообщить("В базе: " + НаименованиеВБазе);
```

## 6.3. Восстановление синхронизации
**Для приведения** программного объекта **в соответствие с базой** данных ==необходимо перечитать данные==:
```java
Товар.Прочитать();  // Теперь Товар.Наименование = "Старое наименование"
```

## 6.4. Важный вывод
При отмене транзакций базы данных разработчик ==должен самостоятельно обеспечивать синхронизацию== программных объектов с актуальным состоянием базы, если это необходимо. 

Это можно сделать:
- перечитыванием всех данных объекта (метод `Прочитать()`);
    
- выборочным изменением реквизитов программного объекта (например, для корректного отображения в интерфейсе).

## 6.5. Исключения
В некоторых случаях, в силу прикладной специфики программных объектов, откат изменений в базе данных **может** влиять на значения свойств соответствующих программных объектов. Такие ситуации являются исключениями из общего правила и требуют отдельного рассмотрения.

---
# 7. Исключения: влияние отката транзакции на свойства объектов
**В некоторых случаях** ==откат транзакции всё же влияет на свойства соответствующих программных объектов.== 

Рассмотрим наиболее важные примеры.

## 7.1. Восстановление признака проведённости документа
При отмене транзакции признак **Проведен** документа восстанавливает значение, которое было до начала транзакции.

```java
НачатьТранзакцию();
    Накладная = Документы.ПриходнаяНакладная.СоздатьДокумент();
    Накладная.Дата = ТекущаяДата();
    Накладная.Записать(РежимЗаписиДокумента.Проведение);
    
    Проведен = Накладная.Проведен;  // Истина
ОтменитьТранзакцию();

Сообщить(Накладная.Проведен);  // Нет (значение восстановлено)
Сообщить(Проведен);            // Да (сохранённая переменная)
```

Несмотря на откат транзакции, свойство программного объекта изменилось — вернулось к исходному состоянию.

## 7.2. Очистка ссылки на объект
==Если объект был создан в транзакции==, то при её откате значение **ссылки** очищается (становится пустой ссылкой), хотя сам объект продолжает существовать в памяти.

```java
НачатьТранзакцию();
    Товар = Справочники.Номенклатура.СоздатьЭлемент();
    Товар.Записать();
    Ссылка = Товар.Ссылка;  // ссылка на новый элемент
ОтменитьТранзакцию();

Сообщить(Товар.Ссылка = Справочники.Номенклатура.ПустаяСсылка());  // Истина
Сообщить(Ссылка);  // <Объект не найден> (идентификатор)
```

Ссылка в объекте стала пустой, а сохранённая в переменной ссылка указывает на несуществующий объект.

## 7.3. Очистка кода/номера объекта
Если объект создавался вне транзакции, а запись с автоматической генерацией кода выполнялась внутри транзакции, то при откате код объекта очищается.

```java
// создан вне транзакции
Товар = Справочники.Номенклатура.СоздатьЭлемент();

НачатьТранзакцию();
    Товар.Записать();            // автоматически генерируется код
    Код = Товар.Код;              // "000000008"
ОтменитьТранзакцию();

Сообщить(Товар.Код);  // "" (очищен)
Сообщить(Код);        // "000000008" (сохранён в переменной)
```

## 7.4. Вывод
Эти примеры показывают, что в некоторых случаях откат транзакции всё же изменяет свойства программных объектов. 

Разработчику важно учитывать такие особенности при работе с транзакциями и, при необходимости, самостоятельно синхронизировать состояние объектов после отката.
