# 1. Объект

> [!IMPORTANT] Объект - для МОДИФИКАЦИИ (rw)
> Тип **Объект** 
> (например, `СправочникОбъект.<имя>`, `ДокументОбъект.<имя>`) **предназначен для модификации данных** — чтения и изменения. 
> 
> Остальные типы встроенного языка позволяют только читать данные.

Объект представляет собой совокупность:
- записи основной таблицы;
    
- всех строк табличных частей, относящихся к этому объекту.

---
## 1.1 Особенности сравнения объектов
Значения типа Объект можно сравнивать между собой, но, в отличие от ссылок, они равны только если являются одним и тем же экземпляром программного объекта:

```java
// Один и тот же экземпляр — равны
Объект1 = Справочники.Номенклатура.НайтиПоКоду("001").ПолучитьОбъект();
Объект2 = Объект1;  // Объект1 = Объект2

// Разные экземпляры для одного элемента БД — не равны
Объект1 = Справочники.Номенклатура.НайтиПоКоду("001").ПолучитьОбъект();
Объект2 = Справочники.Номенклатура.НайтиПоКоду("001").ПолучитьОбъект();  // Объект1 <> Объект2
```

---
## 1.2 Способы получения объекта
1. **Через менеджер объекта** (например, `СоздатьЭлемент()`, `СоздатьДокумент()`) — создаётся новый экземпляр, не соответствующий ни одному объекту в БД. После записи появляется новый объект в базе.
    
2. **Из ссылки методом `ПолучитьОбъект()`** — создаётся экземпляр с чтением данных из базы (или из кеша). Считываются все реквизиты объекта и всех его табличных частей.
    
3. **Из выборки методом `ПолучитьОбъект()`** — создаётся экземпляр без чтения из базы, данные заполняются непосредственно из программного объекта выборки.

---
## 1.3 События объекта
Тип Объект поддерживает ряд событий для определения собственных алгоритмов:

|Событие|Момент возникновения|Назначение|
|---|---|---|
|**ПередЗаписью**|Перед записью в БД, после начала транзакции, до непосредственной записи|Анализ необходимости записи, возможность отказа|
|**ПриЗаписи**|После записи в БД, до окончания транзакции|Действия, которые должны быть выполнены только при успешной записи; возможность отказа|
|**ПриКопировании**|У нового объекта при создании копированием|Специфическая обработка при копировании|
|**ОбработкаЗаполнения**|У нового объекта при создании, вводе на основании, выполнении метода `Заполнить()`|Начальное заполнение реквизитов в зависимости от объекта-основания|
|**ОбработкаПроверкиЗаполнения**|Перед записью, до начала транзакции|Проверка заполнения реквизитов, добавление или очистка массива проверяемых реквизитов, возможность отказа|
|**ПередУдалением**|В транзакции удаления, перед непосредственным удалением|Действия перед удалением, возможность отмены удаления|

# 2. Кеширование представлений и объектов
> [!IMPORTANT] Объекты кешируются
> При работе с объектными данными система выполняет кеширование считываемых данных в оперативной памяти с использованием **кеша объектов**. 
> 

Кеш объектов состоит ==из двух частей==:
- **транзакционный кеш** — используется при чтении в рамках транзакции;
    
- **обычный кеш** — используется при чтении вне транзакции.


==При любых обращениях к объектным данным== система сначала обращается к кешу.
![[Pasted image 20260301192938.png]]

В кеш считываются два вида данных:
- все данные объекта целиком;
    
- значения полей, необходимые для формирования представления ссылки на объект.

## 2.1. Рекомендации по проектированию
> [!NOTE] 
> ***При обращении к любому реквизиту объекта через точку*** от ссылки **выполняется полное чтение всех данных объекта**. 

Это важно учитывать при проектировании: **большие данные (картинки, файлы, тексты) рекомендуется выносить в отдельные справочники** или регистры сведений, **чтобы избежать считывания больших объёмов при работе с объектами**.

---
## 2.2. Обычный кеш - очередь
Если ==требуемых данных нет в обычном кеше==:
1. выполняется **чтение из базы данных**;
2. данные **сохраняются в кеше**.

Уникальный ***идентификатор для кеша — ссылка на объект***. 
Данные могут храниться в двух видах: полностью или только представление.

> [!IMPORTANT] КЕШ - последовательная ОЧЕРЕДЬ
> Все считанные данные помещаются в последовательную очередь, и, поскольку объем кеша ограничен, наиболее старые данные будут вытесняться из кеша последними считанными данными.


**Сценарии работы:**
- При обращении за представлением: если в кеше есть данные, представление берётся из кеша; если нет — считываются только поля для представления.
    
- При обращении к реквизиту через точку от ссылки:
    - если в кеше весь объект — значение берётся из кеша;
        
    - если только представление — оно удаляется, считываются все данные;
        
    - если данных нет — считываются все поля объекта.


Считанные ***данные хранятся в кеше до наступления*** одного ***из событий:***
- вытеснение более новыми данными (***переполнение кеша***);
    
- **обнаружение изменения данных в базе**;
    
- **истечение 20 секунд с последнего обращения** (проверка версии); 
    
- истечение 20 минут с последнего считывания;
    
- изменение данных в базе.

Если обращение к кешу происходит в течение 20 секунд после загрузки данных, они считаются валидными без проверки. 
По истечении 20 секунд система проверяет версию данных в кеше с версией в базе. 
При несовпадении версий (данные изменились) кеш очищается и выполняется повторное чтение из базы, после чего отсчёт 20 секунд начинается заново.****

> [!IMPORTANT] Каждые 20 секунд кеш сверяет версии объектов
> При обращении в пределах 20 секунд данные считаются валидными. 
> 
> После 20 секунд проверяется соответствие версий; при несовпадении данные удаляются из кеша и считываются ИЗ БД заново.

### ПРИМЕР
Рассмотрим последовательное выполнение двух операторов (листинг 4.5), где 
- Номенклатура – это ссылка на объект справочника, 
- а Наименование и ВидНоменклатуры – реквизиты справочника Номенклатура.
![[Pasted image 20260301194737.png]]

На выполнение второго оператора будет тратиться гораздо меньше времени, поскольку 
- в первом случае (с большой долей вероятности) будет выполняться обращение к базе данных и чтение всех данных объекта, 
- а во втором – чтение из оперативной памяти (кеша объектов)

> [!NOTE] 
> При получении данных через объектную модель, мы считываем данные из кеша, и если там нет, только тогда отправится запрос в БД.
> 
> Или же сразу считаем актуальные данные, из-за обновления кеша каждые 20 секунд

## 2.3. Транзакционный кеш
**При обращении к данным в рамках транзакции** используется транзакционный кеш:
- данные гарантированно актуальны (валидны);
    
- ==при считывании устанавливается блокировка в базе==, исключающая изменения до конца транзакции;
    
- данные хранятся до вытеснения или окончания транзакции.


По окончании транзакции:
- при успешном завершении (**Commit**) — данные переносятся из транзакционного кеша в обычный; А транзакционные кеш очищается
    
- при откате (**Rollback**) — транзакционный кеш просто очищается.
