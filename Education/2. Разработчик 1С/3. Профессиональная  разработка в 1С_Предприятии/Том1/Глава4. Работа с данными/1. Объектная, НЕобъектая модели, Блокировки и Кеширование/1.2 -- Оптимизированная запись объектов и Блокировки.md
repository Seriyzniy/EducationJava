# 1. Оптимизированная запись объектов
**При записи объектов** в базу данных, **выполняется оптимизация**:
- *если реквизиты объекта не менялись* — з**аписывается только версия объекта;**
    
- если строки табличной части не менялись — табличная часть не записывается;
    
- если менялись отдельные строки или добавлялись новые — записываются только изменённые и добавленные строки;
    
- при изменении порядка строк или удалении — записываются все строки табличной части.

## 1.2 Создание новых объектов
Зачастую при разработке или модификации прикладных решений требуется выполнять некоторые действия, сопровождающие создание тех или иных новых объектов базы данных. 

Например, при создании новой приходной накладной может потребоваться автоматически заполнять поле Склад, если известно, что все поступающие товары должны приходоваться только на один определенный склад. 

Также может потребоваться установка каких-либо других реквизитов документа по умолчанию.


Для автоматического заполнения новых объектов можно использовать:
- свойства реквизитов **Значение заполнения** и **Заполнять из данных заполнения** (без программирования);
    
- если этих возможностей недостаточно — **обработчик события** `ОбработкаЗаполнения` в модуле объекта.

Событие возникает при:
- интерактивном создании нового объекта;
- вводе на основании (интерактивном или программном);
- выполнении метода объекта **Заполнить()**.

Параметр `ДанныеЗаполнения` может содержать:
- ссылку (при вводе на основании);
- структуру (при создании из списка с отбором);
- Неопределено (при создании из панели действий).


> *При копировании* (интерактивном или программном) событие **ОбработкаЗаполнения** ***не вызывается***.

# 2. Блокировки
Система обеспечивает два вида блокировок для целостной работы с объектами при одновременном доступе нескольких пользователей.

## 2.1. Пессимистическая блокировка
Механизм пессимистической блокировки ==запрещает изменение данных объекта другими сессиями== или данной сессией до тех пор, **пока блокировка не будет снята этим объектом встроенного языка**

### 2.1.1 **Использование системой:**
- ==Автоматически устанавливается== расширениями форм **при начале редактирования объекта.**
    
- При попытке редактирования другим пользователем выдаётся сообщение о невозможности блокировки.
    
- Снимается при закрытии формы.

### 2.1.2 **Программное управление:**
**Методы формы** `ЗаблокироватьДанныеФормыДляРедактирования()` и `РазблокироватьДанныеФормыДляРедактирования()` — блокировка привязывается к конкретной форме;


**Методы глобального контекста** `ЗаблокироватьДанныеДляРедактирования()` и `РазблокироватьДанныеДляРедактирования()` — 
- ОБЯЗАТЕЛЬНО указать ссылку на объект 
- и, при необходимости, идентификатор формы, если блокировка устанавливается с привязкой к конкретной форме,

Блокировки с идентификатором формы и без него несовместимы друг с другом.

Перед модификацией данных следует попытаться их заблокировать. При успехе — выполнять изменения, при неудаче — обрабатывать исключение.

### 2.1.3 Пример
Перед модификацией данных нужно попытаться их заблокировать.

```java
&НаСервере
Функция ПримерМодификации()

    ТоварСсылка = Справочники.Товары.НайтиПоКоду("000000001");
    
    Попытка
        ЗаблокироватьДанныеДляРедактирования(ТоварСсылка);
        
        // Можно выполнять модификацию данных объекта.
        ТоварОбъект = ТоварСсылка.ПолучитьОбъект();
        ТоварОбъект.Наименование = "Новое наименование";
        ТоварОбъект.Записать();
        
        Возврат Истина;
        
    Исключение
        // Нельзя модифицировать данные объекта.
        Сообщение = Новый СообщениеПользователю;
        Сообщение.Текст = "Данные объекта уже заблокированы";
        Сообщение.Сообщить();
        
        Возврат Ложь;
        
    КонецПопытки;
    
КонецФункции
```

## 2.2 Оптимистическая блокировка - CAS
**Запрещает запись объекта**, ***если после его считывания*** **данные в базе изменились.** (CAS = Compare And Swap)

Реализуется через проверку версии объекта при записи или блокировке:
- при считывании сохраняется версия объекта из базы;
    
- если перед записью версия в базе изменилась (объект изменён другой сессией), возникает ошибка;
    
- гарантирует, что изменения одного пользователя не «затрут» изменения другого.

## 2.3 БЛОКИРОВКИ не в БД, а в менеджере платформы

> [!NOTE] Блокировки НА УРОВНЕ ПЛАТФОРМЫ А НЕ БД
> ***Оба вида блокировок*** **реализуются собственным менеджером блокировок платформы**, работающим над уровнем базы данных. 

Блокировки прозрачны для транзакций, за исключением: если пессимистическая блокировка установлена в транзакции и выполняется откат, блокировка снимается автоматически.
