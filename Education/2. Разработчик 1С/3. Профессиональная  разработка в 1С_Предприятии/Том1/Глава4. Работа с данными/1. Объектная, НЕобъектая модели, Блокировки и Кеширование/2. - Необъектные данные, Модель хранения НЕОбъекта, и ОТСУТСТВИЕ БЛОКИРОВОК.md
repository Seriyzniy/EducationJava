К необъектным данным относятся:
- регистры сведений, регистры накопления;
    
- регистры расчёта, перерасчёты;
    
- регистры бухгалтерии, последовательности;
    
- константы.


Необъектные данные представляют собой записи, которые **полностью описываются значениями своих полей** 
> (других вспомогательных полей нет). 

Они:
- не имеют собственной ценности вне этих значений;
    
- **не поддерживают внутренние уникальные идентификаторы**;
    
- не изменяют состояние базы данных при удалении и создании записи с теми же значениями.

---
# 1. Модель хранения НЕОБЪЕКТНЫХ данных

## 1.1. Природа необъектных данных
Необъектные данные представляют собой **набор записей, хранящихся в таблицах**. 

Каждая **запись полностью описывается значениями своих полей** и не обладает самостоятельной значимостью — важны только сами значения.

Ключевые отличия от объектных данных:
- запись можно удалить и создать новую с теми же значениями — состояние базы данных не изменится;
    
- ==изменение значений полей создаёт другую запись== (в отличие от объекта, который остаётся тем же).

Константы выделяются тем, что для каждой хранится одно значение, и работа с ними не вызывает сложностей.

---
## 1.2. Подчинение регистратору
==Основное свойство== записей необъектных данных — **подчинение регистратору** (документу, с которым связаны записи). 
> Исключение: независимый режим записи для регистра сведений.

Каждая запись содержит обязательное поле **Регистратор**. 
> Регистратор – это некоторый документ, с которым связаны записи необъектных данных.

Записи, относящиеся к одному регистратору, называются его **движениями**.

![[Pasted image 20260301203141.png]]

### 1.2.1. Связь с регистратором
- ***Движения не являются частью документа***, **при записи и чтении документа они не считываются и не записываются**.
    
- Могут создаваться при проведении, записи документов, интерактивно или как набор записей с отбором по регистратору.
    
- **Не могут существовать отдельно от регистратора.** ***При удалении документа система ищет и удаляет все его движения во всех регистрах***, где этот документ может быть регистратором.


Наличие или отсутствие движений не зависит от проведённости документа или пометки на удаление:
- непроведённый документ может иметь движения;
    
- проведённый может не иметь;
    
- помеченный на удаление также может иметь движения.


Дата движений не жёстко связана с датой регистратора — документ может иметь движения разными датами.

### 1.2.2. Модели использования
Наиболее простая модель — создание движений в обработчике проведения регистратора. 

В этом случае остальная логика поддерживается системой автоматически. 

Другие варианты взаимодействия требуют самостоятельной разработки в зависимости от прикладной задачи.

## 1.3. Уникальность записей - PK из БД
Для каждого объекта метаданных определён **ключ записи**. 

==Данные не могут содержать записи с одинаковыми значениями ключа==
***Ключ формируется из значений нескольких полей***, и его **состав различается для разных объектов** и может меняться в зависимости от свойств.

**Для необъектных сущностей**, подчинённых регистратору, ключ включает:
- ==ссылку на регистратор==;
    
- ==номер строки== (обеспечивает уникальность и упорядочивание записей).

(см [[5. - Регистры (Общее)]] для разницы ключей для разных регистров)

==Для неподчинённых сущностей== в состав ключа ***могут входить другие поля*** (**период, измерения**).

Наличие уникального ключа необходимо для различных задач системы, например, для позиционирования в табличном поле.

## 1.4. Набор записей
***Для модификации необъектных данных*** используется **набор записей** — коллекция отдельных записей, принадлежащих некоторому объекту конфигурации. 

Он позволяет читать, изменять и удалять необъектные данные.

### 1.4.1. Отбор в наборе записей
Состав набора записей определяется свойством **Отбор**. 
> Элементы отбора создаются платформой, разработчик может только устанавливать условия равенства нужным значениям.

Состав отбора для различных объектов конфигурации:

![[Pasted image 20260301204341.png]]

**Важная особенность:** если среди элементов отбора есть отбор по регистратору, он **обязательно должен быть установлен**, иначе при записи набора будет ошибка.

Для большинства объектов допускается модификация данных только частями — наборами записей, относящимися к определённому регистратору. Исключение — независимые регистры сведений, где можно модифицировать сразу все данные (если отбор не задан).

---
### 1.4.2. Запись набора записей и удаление
Для наборов записей ***не существует понятия удаления***. НАБОР ЗАПИСЕЙ МОЖНО ТОЛЬКО ЗАПИСАТЬ

Запись может выполняться в двух режимах:
- **С добавлением** — новые записи добавляются к существующим.
    
- **С замещением** — существующие записи заменяются новыми.
    

Для удаления записей необъектных данных нужно записать с замещением **пустой набор записей** (не содержащий ни одной записи).

---
### 1.4.3. Тип записи
Набор записей представляет собой коллекцию объектов типа **запись**. Перебор записей выполняется конструкцией `Для Каждого … Из … Цикл`. Тип записи используется только в наборах записей и предоставляет доступ к значениям полей конкретной записи.

---
## 1.5. Интерактивное редактирование наборов записей - НЕТ БЛОКИРОВОК, ВОЗМОЖНЫ ПОТЕРИ ДАННЫХ
==Для необъектных данных== **не существует механизма объектных блокировок** (оптимистической и пессимистической). 

При использовании набора записей для интерактивного редактирования нужно учитывать, что данные могут быть изменены другой сессией или другим набором записей в той же сессии в промежутке между чтением и записью.

Это может привести к ситуации, когда **изменения одного пользователя будут потеряны из-за перезаписи** старыми данными другого пользователя.

> [!IMPORTANT] ВОЗМОЖНА ПОТЕРЯ ДАННЫХ В НЕОБЪЕКТНЫХ ДАННЫХ
> В регистрах и др. необъектных данных НЕ ИСПОЛЬЗУЮТСЯ БЛОКИРОВКИ 
> 
> А так как ***блокировки управляются менеджером блокировок***, который работает над уровнем БД, ***и эти данные НЕОБЪЕКТЫ***, **то захватить блокировку на НЕОБЪЕКТ - НЕВОЗМОЖНО**
