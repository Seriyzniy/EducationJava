В отличие от примитивных типов, определённых на уровне технологической платформы, **прикладные типы** ***создаются автоматически при добавлении в конфигурацию объектов метаданных.*** 

Они позволяют работать с данными, хранящимися в структурах, описываемых этими объектами.

## 0.1. Образование типов для различных объектов
Для каждого объекта конфигурации платформа создаёт набор типов данных. 

Например, при добавлении справочника _Номенклатура_ появляются типы:
- `СправочникМенеджер.Номенклатура`
    
- `СправочникСсылка.Номенклатура`
    
- `СправочникОбъект.Номенклатура`
    
- `СправочникВыборка.Номенклатура`
    

При добавлении регистра сведений _КурсыВалют_ становятся доступны:
- `РегистрСведенийМенеджер.КурсыВалют`
    
- `РегистрСведенийВыборка.КурсыВалют`
    
- `РегистрСведенийМенеджерЗаписи.КурсыВалют`
    
- `РегистрСведенийНаборЗаписей.КурсыВалют`
    
- `РегистрСведенийЗапись.КурсыВалют`
    
- `РегистрСведенийКлючЗаписи.КурсыВалют`

---
## 0.2. Группы прикладных типов
> [!NOTE] 
> Типы, создаваемые для разных объектов, объединяются в группы (`<вид объекта>Выборка.<имя>`) со схожей функциональностью. 

Например, все типы `<вид объекта>Выборка.<имя>` работают аналогично. 

Это существенно облегчает разработку: освоив работу с выборкой справочника, разработчик легко сможет работать с выборкой регистра сведений или бизнес-процесса.

# 1. Менеджер объектов
Для каждого вида объектов конфигурации (справочники, документы и т.д.) в системе определён тип **менеджера этих объектов** (например, `СправочникиМенеджер`, `ДокументыМенеджер`). 

Эти типы существуют всегда, независимо от наличия конкретных объектов в прикладном решении.

Каждый из этих типов имеет единственное значение в конкретном прикладном решении, доступное через соответствующие свойства глобального контекста (`Справочники`, `Документы`).

## 1.1. Основные задачи
> [!IMPORTANT] Цель
> Главная задача менеджера объектов — предоставить доступ к **менеджерам конкретных объектов конфигурации** 
> (например, `СправочникМенеджер.Контрагенты`, `ДокументМенеджер.ПриходнаяНакладная`).
> 
> А эти менеджеры нужны для управления объектом конфигурации, а не конкретным объектом

![[Pasted image 20260302171929.png]]

**Менеджеры** конкретных **объектов** ==доступны двумя способами==:
- как свойства менеджера объектов (например, `Справочники.Контрагенты`);
    
- перебором коллекции значений, которой является менеджер объектов (элементы коллекции — менеджеры отдельных объектов).

---
## 1.2. Проверка принадлежности к типам ссылок
Менеджер объектов позволяет проверить, является ли тип значения типом ссылки на объект этого вида:
![[Pasted image 20260302172254.png]]
- метод **ТипВсеСсылки()** возвращает объект `ОписаниеТипов`, содержащий типы ссылок на все объекты данного вида (например, на все справочники);
    
- метод **СодержитТип()** этого описания проверяет, входит ли анализируемый тип в состав описания.

# 2. Менеджер

> [!IMPORTANT] Менеджер - точка входа в объект конфигурации
> Менеджер прикладного объекта является «точкой входа» в конкретный объект конфигурации в объектной модели встроенного языка. 


Он предоставляет доступ к общим действиям, относящимся к конкретному объекту (например, к справочнику _Контрагенты_ или документу _Авансовый отчет_).

Основные возможности менеджера:
- получение выборок из данных объекта;
    
- создание и поиск элементов данных;
    
- доступ к предопределённым элементам;
    
- выполнение общих действий, поддерживаемых объектом;
    
- получение макетов.


В конкретном прикладном решении всегда существует только один экземпляр каждого менеджера.

---
# 3. Объект
Тип объекта создаётся только для объектов, хранящих объектные данные (справочники, документы, планы счетов и т.д.). 

Только **с помощью объекта может быть выполнена модификация данных в базе данных**; другие типы объектных данных позволяют только чтение.

(см [[1.1 -- Объект, его события и кеширование]])

---
# 4. Ссылка
Тип ссылки создаётся для тех же объектов, что и тип объекта. 

Ссылка служит **для однозначной идентификации объекта данных** в базе данных. Значением ссылки является уникальный внутренний идентификатор, хранящийся в поле _Ссылка_ таблиц объекта.

Ссылка позволяет обращаться к свойствам объекта и получать сам объект.

(см [[1. - Объектные данные, Модель хранения объекта]] пункт про Ссылку)

---
# 5. Набор записей
Тип набора записей предназначен **для работы с необъектными данными** (например, наборы записей регистров). 

Как и объект для объектных данных, набор записей ==является ЕДИНСТВЕННЫМ СРЕДСТВОМ МОДИФИКАЦИИнеобъектных данных==.

---
# 6. Динамический список
Объект **ДинамическийСписок** предназначен ==для динамического просмотра данных== объекта конфигурации ==в элементе формы== _Таблица_.

Он ***считывает данные из базы*** **порциями** по мере навигации пользователя или обращения из встроенного языка.

На динамическом списке построены все формы списка объектов конфигурации, содержащие основной реквизит типа _ДинамическийСписок_. 

Список формируется запросом к основной таблице либо произвольным запросом. Он использует систему компоновки данных и позволяет задавать группировку, отбор, порядок и условное оформление через свойства реквизита (интерактивно или программно) или настройкой в режиме 1С:Предприятие.

## 6.1. Основные свойства
При создании формы списка, конструктором форм, платформа добавляет основной реквизит типа _ДинамическийСписок_ и заполняет его свойства:

- **ОсновнаяТаблица** — таблица, к которой автоматически формируется запрос;
    
- **ДинамическоеСчитываниеДанных** — данные считываются небольшими порциями только при необходимости отображения.


Реквизит _Список_ содержит поля (стандартные и созданные разработчиком), а также настраиваемые свойства: _Отбор_, _Порядок_, _УсловноеОформление_, _Группировка_, _Параметры_.

---
## 6.2. Отображение полей
В форме списка по умолчанию отображаются только поля _Дата_ и _Номер_. 
> Для отображения других полей разработчик может перетащить их в дерево элементов формы.

**Важная особенность:** запросом считываются только значения полей, отображаемых в форме, чтобы не передавать лишнюю информацию на клиента. 

Если поле должно считываться всегда, можно:
- установить свойство _Использовать всегда_ в конфигураторе;
    
- установить программно: `Список.УстановитьОбязательноеИспользование("Ссылка", Истина)`.

---
## 6.3. Произвольный запрос
При установке свойства `ПроизвольныйЗапрос` становится доступным ручное формирование запроса (через окно _Настройка списка_ или программно). 

Это позволяет выбирать данные из нескольких таблиц, использовать параметры, вычисляемые поля и т.д.

### 6.3.1. Пример программного создания
В обработчике _ПриСозданииНаСервере_ формы списка можно задать произвольный запрос:

```java
Если Параметры.Свойство("Номенклатура") Тогда
    Список.ПроизвольныйЗапрос = Истина;
    Список.ТекстЗапроса = "ВЫБРАТЬ ... ГДЕ ... = &Номенклатура";
    Список.ОсновнаяТаблица = "Документ.РасходнаяНакладная";
    Список.Параметры.УстановитьЗначениеПараметра("Номенклатура", Параметры.Номенклатура);
КонецЕсли;
```

Если форма открывается без параметра, используется стандартный запрос к основной таблице.

---
## 6.4. Настройка списка
Разработчик может настроить группировку, отбор, порядок и условное оформление в конфигураторе (окно _Настройка списка_) или программно.

### 6.4.1. Пример программного условного оформления
```java
УО = Список.УсловноеОформление.Элементы;

ЭлементУО = УО.Добавить();
ЭлементУО.Оформление.УстановитьЗначениеПараметра("ЦветФона", WebЦвета.Пурпурный);
ЭлементУсловия = ЭлементУО.Отбор.Элементы.Добавить(Тип("ЭлементОтбораКомпоновкиДанных"));
ЭлементУсловия.ЛевоеЗначение = Новый ПолеКомпоновкиДанных("Товары.Количество");
ЭлементУсловия.ВидСравнения = ВидСравненияКомпоновкиДанных.Больше;
ЭлементУсловия.ПравоеЗначение = 3;
ОформляемоеПоле = ЭлементУО.Поля.Элементы.Добавить();
ОформляемоеПоле.Поле = Новый ПолеКомпоновкиДанных("Номер");
```

Пользователь также может настраивать список в режиме 1С:Предприятие (команда _Все действия → Настроить список…_), изменяя, отключая или удаляя заданные разработчиком настройки.

# 7. Выборка - обход всех элементов прикладного объекта
Выборка предназначена ==для динамического обхода элементов данных==, хранящихся в структуре объекта конфигурации. 

Она не считывает данные целиком в память, **а получает их блоками по мере обхода методом** `Следующий()`. Это позволяет обрабатывать очень большие объёмы информации.

*Каждый **блок содержит 25 записей***. Для объектных таблиц каждая запись включает все данные объекта (все поля и табличные части), поэтому при обращении к реквизитам выборки или получении объекта повторного обращения к базе не происходит.

```java
Сообщение = Новый СообщениеПользователю; 
ВыборкаВалют = Справочники.Валюты.Выбрать(); //Перебор всех записей в справочнике Валюты

Пока ВыборкаВалют.Следующий() Цикл 
	Сообщение.Текст = ВыборкаВалют.Наименование;
	Сообщение.Сообщить(); 
КонецЦикла;
```
## 7.1. Параметры выборки
Выборка осуществляет считывание данных в определённом порядке, который задаётся параметрами метода **Выбрать()**:
- **Отбор** — может выполняться только по одному полю, для которого построен индекс (индексируемые по умолчанию поля или поля с установленным признаком индексирования). Отбор задаётся структурой с единственным элементом: ключ — имя поля, значение — значение отбора.
    
- **Порядок** — задаётся с использованием полей, имеющих индекс. Например: `"Дата Убыв"` для сортировки по убыванию даты.
    
- **Родитель** — для иерархических данных. В обычной выборке возвращаются элементы, родитель которых равен указанному. В иерархической выборке — все элементы в иерархии указанного родителя (все уровни). При пустой ссылке выбираются элементы верхнего уровня.
    ![[Pasted image 20260302175220.png]]
    ![[Pasted image 20260302175236.png]]
    
- **Владелец** — для подчинённых справочников отбираются записи, владелец которых равен указанному.

---
## 7.2. Иерархическая выборка
Для иерархических данных (справочники, планы видов характеристик, планы счетов) используется метод **ВыбратьИерархически()**. 

Такая выборка обходит записи в соответствии с их иерархией:
- после элемента выдаётся подчинённый ему элемент (если существует);
    
- если подчинённых нет — следующий элемент.  
    Каждое считывание подчинённых элементов реализовано как отдельный блок данных.

---
## 7.3. Особенности использования выборки
Выборка считывает данные блоками в определённой последовательности сортировки. 

При обходе методом **Следующий()** очередной блок получается как множество записей, следующих в указанном порядке после последней записи предыдущего блока.

> [!WARNING] Нет контроля при считывании блоками - возможны аномалии
> Разработчик не имеет контроля над актуальностью получаемой информации — даже в транзакции невозможно определить, взята ли запись из ранее полученного блока или считана только что в составе нового. 
> 
> Это может приводить к различным аномалиям.

### 7.3.1. Возможные проблемы
❗❗❗**Получение данных удалённого объекта:**  
Если после получения блока одна из его записей была удалена из базы, к моменту окончания обхода выборки она будет отсутствовать в базе, но продолжит присутствовать в полученной выборке.

**Множественное попадание записи:**  
При изменении данных (например, переименовании) порядок следования записи может измениться. Если она переместится в ещё не считанный блок, при получении этого блока запись будет считана повторно.

**Непопадание записи в выборку:**  
В противоположной ситуации запись может переместиться в уже считанный блок и вообще не попасть в выборку.

---
### 7.3.2. Удаление иерархических данных
**При удалении родительского элемента удаляются и все его подчинённые элементы.** 

Если использовать обычную выборку, удаление родителя может затронуть элементы, уже считанные в текущем блоке. При попытке работы с таким объектом возникнет ошибка, так как он уже не существует в базе.

Для удаления иерархических данных следует использовать **иерархическую выборку**:
- подчинённые элементы располагаются в одном или нескольких блоках сразу за родительским;
    
- после удаления родителя (и его подчинённых) выполняется чтение следующего по порядку блока.

---
### 7.3.3. Рекомендации
Следует внимательно относиться к изменению объектов в процессе обхода динамической выборки — это может повлиять на порядок их включения. 

Динамические выборки рекомендуется использовать:
- для задач, не требующих ответственного чтения данных;
    
- для регламентных задач, выполняемых в монопольном режиме;
    
- когда объём данных очень велик, и получение выборки другими способами (например, запросом) неэффективно.
