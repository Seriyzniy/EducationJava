# 1. SQL инъекция
SQL инъекция один из самых распространенных ==способов взлома== сайтов и программ работающих с БД

Так как ==строки из HTML формы напрямую КОНКАТИНИРУЮТСЯ в SQL-запросе==, злоумышленник может подобрать такую строку, которая навредит БД


> [!NOTE] SQL инъекция в проекте
> Если в поле `email`ввести следующую строку: `tom@gmail.com'); DROP TABLE persons; --`
> 
> То запрос который отправится в БД будет выглядеть так:
> `INSERT INTO persons` 
> `VALUES (1, 'tom', 24, 'tom@gmail.com'); DROP TABLE persons; --')`

В данном случае SQL запрос выполнит вставку в таблицу, а затем УДАЛИТ таблицу `persons`

НО этого не случится в данном случае, поскольку у нас есть валидация формы, в которой проверяется формат введенной строки.

ОДНАКО, если же в последнем поле мы вводили бы фамилию, то SQL инъекция успешно бы прошла
#### 2. Еще один вид SQL инъекций

SQL инъекции бывают разные.

Например, мы хотим получить информацию о пользователе по его ID. 
В приложении мы бы делали следующую конкатенацию строк

```sql
SELECT * FROM persons WHERE id = значение_из_формы
```

Однако, если мы введем в поле `ID` значение `105 OR 1=1`
```sql
SELECT * FROM persons WHERE id = 105 OR 1=1
```

Злоумышленник бы получим все записи из таблицы

# 2. Защита от SQL инъекций

![[Pasted image 20241125111354.png]]

SQL запрос компилируется один раз и не может быть изменен. Данные от пользователя могут быть ВСТАВЛЕНЫ лишь в ОПРЕДЕЛЕННЫЕ места запроса и не могут изменить сам SQL-запрос

Под каждый вопросительный знак, будет вставляться значение по индексу знака `?`

#### 1. Защита от инъекции
Если в поле `email`ввести следующую строку: `tom@gmail.com'); DROP TABLE persons; --`, то какие бы кавычки мы бы не ставили, ==введенные данные будут восприниматься как строка==.

Это происходит благодаря данному коду
```java
preparedStatement.setStrin(3, person.getEmail())
```
Все что введет пользователь в поле `email`, будет восприниматься как строка

#### 2. Преимущества PreparedStatement 

- Удобнее чем Statement (избавляет от ручной конкатенации строк)
- Безопаснее
- Быстрее чем Stetement (потому что ==компилируется один раз на стороне БД==, а след. разы просто вставляет значения на места `?`)
