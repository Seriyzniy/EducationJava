Асинхронные потребители в Spring AMQP используют несколько типов потоков. 
- В `SimpleMessageListenerContainer` вызов `MessageListener` **происходит в потоках** `TaskExecutor`. 
	Если он не задан явно, используется `SimpleAsyncTaskExecutor`. При использовании пула потоков важно, чтобы его размер соответствовал уровню параллелизма (`concurrency`). 
	
- В `DirectMessageListenerContainer` **слушатель вызывается прямо в потоке клиента** RabbitMQ, а `TaskExecutor` применяется только для служебных задач мониторинга.

## Имена потоков
У `SimpleAsyncTaskExecutor` имена потоков **автоматически получают префикс имени бина контейнера**, что удобно для анализа логов. 

> Если `TaskExecutor` задаётся вручную, рекомендуется самостоятельно настроить понятные имена потоков, так как контейнер их не изменяет.

## Собственный Executor в SimpleMessageListenerContainer и его реализациях
`Executor`, настроенный в `CachingConnectionFactory`, передаётся клиенту RabbitMQ и используется для доставки сообщений в контейнер слушателя. 

Если он не задан, клиент создаёт собственный пул потоков размером примерно `availableProcessors() * 2` на одно соединение. 

При большОм числе фабрик соединений или при использовании `CacheMode.CONNECTION` имеет смысл применять общий `ThreadPoolTaskExecutor` с достаточным количеством потоков.

## Параллелизм в DirectMessageListenerContainer
Для `DirectMessageListenerContainer` особенно важно, чтобы фабрика соединений имела `TaskExecutor` с пулом, способным поддерживать нужный уровень параллелизма всех контейнеров. Размер пула по умолчанию также равен `availableProcessors() * 2`.

## ThreadFactory 
Отдельно стоит учитывать, что клиент RabbitMQ использует `ThreadFactory` для создания потоков низкоуровневого I/O (работа с сокетами). Изменить её можно только через настройку базовой клиентской `ConnectionFactory` RabbitMQ.
