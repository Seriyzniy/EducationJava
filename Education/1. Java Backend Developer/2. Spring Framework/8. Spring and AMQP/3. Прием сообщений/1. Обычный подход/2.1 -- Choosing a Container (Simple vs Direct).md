# 1. Описание контейнеров
## SimpleMessageListenerContainer
```bash
[Поток 1] → Consumer → Channel → Connection → RabbitMQ
[Поток 2] → Consumer → Channel → Connection → RabbitMQ
[Поток N] → Consumer → Channel → Connection → RabbitMQ
```

До версии 2.0 в Spring AMQP существовал только `SimpleMessageListenerContainer` (SMLC). 
Он использует 
- **внутреннюю очередь** 
- и **отдельный поток** на каждого потребителя. 

Если контейнер слушает несколько очередей, один и тот же поток обрабатывает сообщения из всех этих очередей. Параллелизм управляется через `concurrentConsumers` и связанные с ним свойства. 
- Сообщения, поступающие от клиента RabbitMQ, сначала передаются во внутреннюю очередь, а затем обрабатываются потоком потребителя. 
- Такая схема была обусловлена ограничениями ранних версий клиента RabbitMQ, которые не поддерживали параллельную доставку.

## DirectMessageListenerContainer
```bash
[Пул потоков] → 
    [Канал 1] → Consumer → Connection → RabbitMQ
    [Канал 2] → Consumer → Connection → RabbitMQ
    [Канал N] → Consumer → Connection → RabbitMQ
```

С появлением обновлённой модели потоков в клиенте RabbitMQ стало возможным реализовать `DirectMessageListenerContainer` (DMLC). 

> [!IMPORTANT] 
> В нём **слушатель вызывается напрямую в потоке клиента** RabbitMQ, *без промежуточной очереди и отдельного потока* потребителя.

За счёт этого архитектура DMLC проще, однако часть возможностей SMLC в нём недоступна. 

**Параллелизм здесь задаётся через** `consumersPerQueue` и ==фактически ограничивается пулом потоков клиентской библиотеки==; 
> свойства `concurrentConsumers` и связанные механизмы отсутствуют.

# 2. АНАЛОГИЯ
## SimpleMessageListenerContainer (SMLC) - как **ПОЧТАЛЬОНЫ**
Представьте **отделения почты** в городе:
```bash
[Почтальон 1] → Обходит свой участок → Забирает письма → Доставляет вам
[Почтальон 2] → Обходит свой участок → Забирает письма → Доставляет вам
[Почтальон 3] → Обходит свой участок → Забирает письма → Доставляет вам
```

**Как работает:**
1. Каждый почтальон (поток) **регулярно ходит на почту** (опрос брокера)
    
2. **Спрашивает**: "Есть ли для меня письма?" (pull-модель)
    
3. Если есть → забирает партию и разносит
    
4. Если нет → идет обратно и ждет следующего обхода
    

**Проблемы:**
- Почтальоны **ходят даже когда писем нет** (тратят силы) (указываем min количество потоков)
    
- Каждому нужен **свой маршрут и униформа** (отдельный поток)
    
- Если почтальон заболел → его участок простаивает
    
- **Медленнее реагирует** на новые письма (ждет следующего обхода)

## DirectMessageListenerContainer (DMLC) - как **ДОСТАВЩИКИ-КУРЬЕРЫ**
Представьте **службу доставки типа Яндекс.Еды**:
```bash
[Диспетчерский центр] → 
    [Курьер 1] ← Новый заказ! → Едет забирать
    [Курьер 2] ← Новый заказ! → Едет забирать
    [Курьер N] ← Новый заказ! → Едет забирать
```

**Как работает:**
1. Есть **центральный диспетчер** (==RabbitMQ брокер==)
    
2. Когда приходит письмо → **диспетчер САМ ЗВОНИТ курьеру** (push-модель)
    
3. ==Ближайший свободный курьер едет забирать==
    
4. Курьеры **не дежурят зря** - отдыхают пока нет заказов
    

**Преимущества:**
- **Реагирует мгновенно** - как только письмо пришло
    
- **Экономит ресурсы** - курьеры не ездят "вхолостую"
    
- **Гибкое распределение** - один курьер может взять несколько заказов
    
- **Автомасштабирование** - больше заказов = больше курьеров

# 3. Отличия
## Отличия функционала
Следующие функции доступны в SMLC, но не в DMLC:
- `batchSize`: В SMLC вы можете установить это значение **для управления количеством сообщений, доставляемых в транзакции**, или для уменьшения количества подтверждений (acks), но это может привести к увеличению количества повторных доставок после сбоя. 
	- (**DMLC действительно имеет** `messagesPerAck`, который можно использовать для уменьшения количества ack, так же как `batchSize` в SMLC, **но его нельзя использовать с транзакциями** — каждое сообщение доставляется и подтверждается (ack) в отдельной транзакции).
    
- `consumerBatchEnabled`: включает **пакетную обработку отдельных сообщений в потребителе**; для получения дополнительной информации см. "[Message Listener Container Configuration :: Spring AMQP](https://docs.spring.io/spring-amqp/reference/amqp/containerAttributes.html)".
    
- `maxConcurrentConsumers` и интервалы или триггеры масштабирования потребителей (consumer scaling intervals or triggers) — 
	- в DMLC нет автоматического масштабирования. Однако он позволяет вам программно изменять свойство `consumersPerQueue`, и количество потребителей корректируется соответствующим образом.
    

Тем не менее, DMLC имеет следующие преимущества перед SMLC:
- **Добавление и удаление очередей во время выполнения более эффективно.** 
	- В ==SMLC== весь поток потребителя перезапускается (==все потребители отменяются и создаются заново==). 
	- В **DMLC** **незатронутые потребители не отменяют**ся.
    
- Избегается переключение контекста между потоком клиента RabbitMQ и потоком потребителя.
    
- Потоки разделяются между потребителями, а не имеют выделенный поток для каждого потребителя, как в SMLC. Однако см. ВАЖНОЕ примечание о конфигурации фабрики соединений в разделе "Потоки и асинхронные потребители".

См. "[Message Listener Container Configuration :: Spring AMQP](https://docs.spring.io/spring-amqp/reference/amqp/containerAttributes.html)" для получения информации о том, какие свойства конфигурации применяются к каждому контейнеру.

##  Сравнительная таблица

| Аспект                      | SimpleMessage<br>ListenerContainer (SMLC)                                            | DirectMessage<br>ListenerContainer (DMLC)                          |
| --------------------------- | ------------------------------------------------------------------------------------ | ------------------------------------------------------------------ |
| **Архитектура**             | ==Pull-модель== <br>(потребители опрашивают брокер)                                  | ==Push-модель== (брокер доставляет сообщения)                      |
| **Потоки**                  | Использует отдельные потоки <br>для каждого потребителя <br>(один Consumer на поток) | Использует общий пул потоков (один Connection, несколько Channels) |
| **Масштабирование**         | Добавляет Consumer'ов <br>(требует больше потоков)                                   | Добавляет каналы (Channels) в существующих потоках                 |
| **Производительность**      | Выше накладные расходы, <br>ниже пропускная способность                              | Ниже накладные расходы, выше пропускная способность (до 2-3x)      |
| **Prefetch**                | Устанавливается <br>через `setPrefetchCount()`                                       | Управляется через `setPrefetchCountPerConsumer()`                  |
| **Управление транзакциями** | Поддерживает <br>внешние транзакции (JMS-style)                                      | Не поддерживает внешние транзакции                                 |
| **Подтверждения (Ack)**     | Поддерживает <br>`AcknowledgeMode.NONE`, <br>`AUTO`, <br>`MANUAL`                    | Только `MANUAL` (явное подтверждение)                              |
| **Блокировки**              | Может блокировать сообщения<br>при конкурентном доступе                              | Меньше блокировок, более эффективное распределение                 |
| **Перезапуск**              | Требует пересоздания всех Consumer'ов                                                | Более легкий перезапуск, т.к. Connection остается                  |

## Когда что использовать
### **Используйте SimpleMessageListenerContainer если:**
- Требуются транзакции (например, с `@Transactional`)
    
- Нужны batch-подтверждения сообщений
    
- У вас небольшое количество очередей (< 10)
    
- Требуется совместимость со старым кодом
    
- Нужны сложные сценарии перезапуска/восстановления

### **Используйте DirectMessageListenerContainer если:**
- Требуется высокая производительность (> 10K сообщений/сек)
    
- Много очередей (> 10) нужно обрабатывать
    
- Важна эффективность использования ресурсов
    
- Не нужны внешние транзакции
    
- Используется Spring Boot 2.0+ (он использует DMLC по умолчанию)

---
## ⚠️ Важные замечания
1. **DMLC требует ручного подтверждения** (`AcknowledgeMode.MANUAL`)
    
2. **SMLC лучше для транзакционных сценариев** (JTA, XA)
    
3. **DMLC не поддерживает `ChannelAwareMessageListener`** в старом стиле
    
4. **Spring Boot 2.x+ использует DMLC по умолчанию**
    
5. **При миграции с SMLC на DMLC проверьте логику подтверждения сообщений**
