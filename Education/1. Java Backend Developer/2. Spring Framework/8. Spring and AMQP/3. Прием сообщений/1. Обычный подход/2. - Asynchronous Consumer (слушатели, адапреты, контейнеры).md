
> [!IMPORTANT] IMPORTANT
> Spring AMQP также **поддерживает аннотированные конечные точки (end-point)** слушателя (listener endpoints) **с помощью аннотации** `@RabbitListener` и предоставляет открытую инфраструктуру для программной регистрации конечных точек. 
> 
> На данный момент это самый удобный способ настройки асинхронного потребителя. 
> 
> Для получения более подробной информации см. "Конечные точки слушателя на основе аннотаций [Annotation-driven Listener Endpoints :: Spring AMQP](https://docs.spring.io/spring-amqp/reference/amqp/receiving-messages/async-annotation-driven.html)".


> [!IMPORTANT] IMPORTANT
> Значение `prefetch` *по умолчанию раньше было 1*, что могло приводить к неполной загрузке эффективных потребителей. 
> 
> Начиная с версии 2.0, значение `prefetch` **по умолчанию теперь 250**, что должно поддерживать занятость потребителей в большинстве распространенных сценариев и, следовательно, повышать пропускную способность.

Тем не менее, *существуют сценарии, в которых значение* `prefetch` **должно быть низким**:
- Для больших сообщений, особенно если их обработка медленная (сообщения могут накапливаться в большом объеме памяти в клиентском процессе).
    
- Когда необходим строгий порядок сообщений (в этом случае значение `prefetch` следует вернуть к 1).
    
- Другие особые случаи.
    

Кроме того, при низком объеме обмена сообщениями и нескольких потребителях (включая параллелизм в рамках одного экземпляра контейнера слушателя) вы можете уменьшить `prefetch`, чтобы добиться более равномерного распределения сообщений между потребителями.

См. "[Message Listener Container Configuration :: Spring AMQP](https://docs.spring.io/spring-amqp/reference/amqp/containerAttributes.html)".

# 1. Message Listener
> **Для асинхронного получения** сообщений (`Message`) задействован **специальный компонент** (НЕ `AmqpTemplate`). 

Этот ==компонент является контейнером== **для колбэка**, потребляющего сообщения. 
Позже в этом разделе мы рассмотрим контейнер и его свойства. 

Но сначала *следует взглянуть на колбэк*, так как именно в нем ваш код приложения интегрируется с системой обмена сообщениями. 

## 1. Интерфейс MessageListener
Существует несколько вариантов колбэка, начиная с реализации интерфейса `MessageListener`, который показан в следующем списке:
```java
public interface MessageListener {
    void onMessage(Message message);
}
```

## 2. Интерфейс ChannelAwareMessageListener
**Если логика** вашего колбэка по какой-либо причине **зависит от экземпляра AMQP** `Channel`, вы можете использовать `ChannelAwareMessageListener`. 

Он выглядит похоже, но имеет дополнительный параметр. 

Следующий список показывает определение интерфейса `ChannelAwareMessageListener`:
```java
public interface ChannelAwareMessageListener {
    void onMessage(
	    Message message, 
	    Channel channel) throws Exception;
}
```
> В версии 2.1 этот интерфейс переместился из пакета `o.s.amqp.rabbit.core` в `o.s.amqp.rabbit.listener.api`.

# 2. MessageListenerAdapter

> [!NOTE] ПОЯСНЕНИЕ о чем речь
> Речь идёт о том, **как Spring “подключает” твой обычный Java-класс (POJO) к RabbitMQ**, не заставляя тебя напрямую работать с:
> - `Message`
> - `Channel`
> - AMQP API (`basicAck`, `basicReject`, `Envelope` и т.п.)
> 
> 👉 ==Адаптеры — это прослойка между RabbitMQ API и твоей бизнес-логикой.==


*Если вы предпочитаете поддерживать* **более СТРОГОЕ РАЗДЕЛЕНИЕ МЕЖДУ ЛОГИКОЙ вашего приложения И API ОБМЕНА СООБЩЕНИЯМИ**, вы можете положиться на реализацию адаптера, предоставляемую фреймворком. 
> Это часто называют поддержкой "Message-driven POJO".


> [!IMPORTANT] `@RabbitListener`
> В версии 1.5 был введен **более гибкий механизм для обмена сообщениями с POJO** — аннотация `@RabbitListener`. 
> 
> Для получения дополнительной информации см. "Конечные точки слушателя на основе аннотаций ([Annotation-driven Listener Endpoints :: Spring AMQP](https://docs.spring.io/spring-amqp/reference/amqp/receiving-messages/async-annotation-driven.html))


При использовании адаптера вам ==нужно предоставить только ссылку на экземпляр==, **который сам адаптер должен вызывать**.

Следующий пример показывает, как это сделать:
```java
MessageListenerAdapter listener = new MessageListenerAdapter(somePojo);

listener.setDefaultListenerMethod("myMethod");
```

Вы можете создать подкласс адаптера и предоставить реализацию `getListenerMethodName()` для динамического выбора различных методов на основе сообщения. 

Этот метод имеет два параметра: 
- `originalMessage` 
- и `extractedMessage`, 
последний является результатом любого преобразования. По умолчанию настроен `SimpleMessageConverter`. 

> Для получения дополнительной информации и сведений о других доступных конвертерах см. [Message Converters :: Spring AMQP](https://docs.spring.io/spring-amqp/reference/amqp/message-converters.html#simple-message-converter)

## 1. Нововведения
Начиная с версии 1.4.2, **ИСХОДНОЕ СООБЩЕНИЕ имеет свойства** `consumerQueue` и `consumerTag`, которые можно использовать для определения очереди, из которой было получено сообщение.

Начиная с версии 1.5, вы *можете настроить карту соответствия очереди или тега потребителя имени метода*, **чтобы динамически выбирать вызываемый метод**. 
> Если в карте нет записи, мы возвращаемся к методу слушателя по умолчанию. 
> Метод слушателя по умолчанию (если не задан) — `handleMessage`.

Начиная с версии 2.0, предоставлен удобный `FunctionalInterface`. Следующий список показывает его определение:
```java
@FunctionalInterface
public interface ReplyingMessageListener<T, R> {

    R handleMessage(T t);

}
```

Этот интерфейс упрощает удобную настройку адаптера с использованием лямбда-выражений Java 8, как показано в следующем примере:
```java
new MessageListenerAdapter((ReplyingMessageListener<String, String>) data -> {
    ...
    return result;
}));
```

Начиная с версии 2.2, 
- метод `buildListenerArguments(Object)` **считается устаревшим**, 
- и вместо него введен **новый метод** `buildListenerArguments(Object, Channel, Message)`. 

Новый метод помогает слушателю получать аргументы `Channel` и `Message`, чтобы сделать больше, например, вызвать `channel.basicReject(long, boolean)` в режиме ручного подтверждения (manual acknowledge). 

Следующий пример показывает самый базовый вариант:
```java
public class ExtendedListenerAdapter extends MessageListenerAdapter {

    @Override
    protected Object[] buildListenerArguments(
	    Object extractedMessage, 
	    Channel channel, 
	    Message message) 
	{
        return new Object[]{extractedMessage, channel, message};
    }

}
```

Теперь вы можете настроить `ExtendedListenerAdapter` так же, как `MessageListenerAdapter`, **если вам нужно получать "channel" и "message".** 

Параметры слушателя должны быть установлены так, как возвращает `buildListenerArguments(Object, Channel, Message)`, как показано в следующем примере слушателя:
```java
public void handleMessage(
	Object object, 
	Channel channel, 
	Message message) throws IOException 
{
    ...

```

# 3. Контейнер
**Контейнер слушателя (Listener Container)** — это компонент Spring AMQP, который берет на себя _активную работу_:
- управляет подключением к RabbitMQ,
- подписывается на очереди,
- получает сообщения,
- и передаёт их в `MessageListener`.


Сам `MessageListener` при этом остаётся **пассивным** — только обрабатывает сообщение.

Контейнер:
- является **компонентом жизненного цикла** (`start/stop`);
- связывает **очередь (Queue)** и **слушатель (MessageListener)**;
- требует `ConnectionFactory` и список очередей для потребления.


==Контейнеры в Spring AMQP==:
- **до версии 2.0** — только `SimpleMessageListenerContainer`;
- **с версии 2.0** — два варианта:
    - `SimpleMessageListenerContainer`
    - `DirectMessageListenerContainer`


Выбор между ними зависит от требований и описан в разделе **“Choosing a Container”**.

Следующий пример показывает самый базовый вариант, использующий `SimpleMessageListenerContainer`:
```java
SimpleMessageListenerContainer container = 
	new SimpleMessageListenerContainer();
	
container.setConnectionFactory(rabbitConnectionFactory);
container.setQueueNames("some.queue");

container.setMessageListener(
	new MessageListenerAdapter(somePojo)
);
```

## ОПРЕДЕЛЕНИЕ БИНА КОНТЕЙНЕРА
### 1. XMl
> [!IMPORTANT] Контейнер создавать как бин
> **Как "активный" компонент**, контейнер слушателя чаще всего **создается с определением бина**, чтобы он мог работать в фоновом режиме.

Следующий пример показывает один из способов сделать это с помощью XML:
```xml
<rabbit:listener-container connection-factory="rabbitConnectionFactory">
    <rabbit:listener 
	    queues="some.queue" ref="somePojo" method="handle"/>
</rabbit:listener-container>
```

Следующий пример показывает другой способ сделать это с помощью XML:
```xml
<rabbit:listener-container 
connection-factory="rabbitConnectionFactory" type="direct">

    <rabbit:listener 
	    queues="some.queue" ref="somePojo" method="handle"/>
</rabbit:listener-container>
```

Оба приведенных выше примера создают `DirectMessageListenerContainer` (обратите внимание на атрибут `type` — по умолчанию он `simple`).

### 2. Java
В качестве альтернативы вы можете предпочесть использовать конфигурацию на Java, которая выглядит похоже на предыдущий фрагмент кода:
```java
@Configuration
public class ExampleAmqpConfiguration {

    @Bean
    public SimpleMessageListenerContainer messageListenerContainer() {
    
        SimpleMessageListenerContainer container = 
	        new SimpleMessageListenerContainer();
	        
        container.setConnectionFactory(rabbitConnectionFactory());
        container.setQueueName("some.queue");
        container.setMessageListener(exampleListener());
        
        return container;
    }

    @Bean
    public CachingConnectionFactory rabbitConnectionFactory() {
    
        CachingConnectionFactory connectionFactory =
            new CachingConnectionFactory("localhost");
            
        connectionFactory.setUsername("guest");
        connectionFactory.setPassword("guest");
        
        return connectionFactory;
    }

    @Bean
    public MessageListener exampleListener() {
        return new MessageListener() {
            public void onMessage(Message message) {
                System.out.println("received: " + message);
            }
        };
    }
}
```

# 4. Приоритет потребителя (Consumer Priority)
Начиная с версии RabbitMQ 3.2, **брокер теперь поддерживает приоритет потребителя** (см. [Using Consumer Priorities with RabbitMQ | RabbitMQ](https://www.rabbitmq.com/blog/2013/12/16/using-consumer-priorities-with-rabbitmq)). 

Это включается путем установки аргумента `x-priority` для потребителя. 

Теперь `SimpleMessageListenerContainer` поддерживает установку аргументов потребителя, как показано в следующем примере:
```java
container.setConsumerArguments(Collections.
<String, Object> singletonMap("x-priority", Integer.valueOf(10)));
```

**Для удобства** *пространство имен предоставляет атрибут* `priority` в элементе `listener`, как показано в следующем примере:
```xml
<rabbit:listener-container connection-factory="rabbitConnectionFactory">
    <rabbit:listener queues="some.queue" ref="somePojo" method="handle" priority="10" />
</rabbit:listener-container>
```

Начиная с версии 1.3, вы можете изменять очереди, которые контейнер слушает, во время выполнения. См. "Очереди контейнера слушателя ([Listener Container Queues](https://docs.spring.io/spring-amqp/reference/amqp/listener-queues.html#listener-queues).)".

# 5. Очереди с автоматическим удалением (auto-delete Queues)
Если контейнер слушает **auto-delete очереди***, 
- очередь имеет опцию `x-expires` 
- или на брокере настроена политика "Время жизни" (Time-To-Live)
**брокер удаляет их при остановке контейнера** (*когда отключается последний consumer*).  

> Раньше (до версии **1.3**) контейнер нельзя было перезапустить, потому что очередь уже была удалена.

Начиная с **Spring AMQP 1.3**, при старте контейнера **RabbitAdmin автоматически переобъявляет отсутствующие очереди**.

Также можно:
- использовать **условное объявление (conditional declaration)** ([Conditional Declaration](https://docs.spring.io/spring-amqp/reference/amqp/broker-configuration.html#conditional-declaration))
    
- настроить `RabbitAdmin` с `auto-startup="false"`,  
    чтобы **объявлять очередь только при запуске контейнера**, а не при старте приложения.

Следующий пример показывает, как это сделать:
```xml
<rabbit:queue id="otherAnon" declared-by="containerAdmin" />


<rabbit:direct-exchange name="otherExchange" auto-delete="true" declared-by="containerAdmin">
    <rabbit:bindings>
        <rabbit:binding queue="otherAnon" key="otherAnon" />
    </rabbit:bindings>
</rabbit:direct-exchange>


<rabbit:listener-container id="container2" auto-startup="false">
    <rabbit:listener id="listener2" ref="foo" queues="otherAnon" admin="containerAdmin" />
</rabbit:listener-container>


<rabbit:admin id="containerAdmin" connection-factory="rabbitConnectionFactory"
    auto-startup="false" />
```

В этом случае очередь и обменник объявляются с помощью `containerAdmin`, у которого установлено `auto-startup="false"`, чтобы элементы не объявлялись во время инициализации контекста. Кроме того, контейнер не запускается по той же причине. 

Когда контейнер позже запускается, он использует свою ссылку на `containerAdmin` для объявления элементов.
