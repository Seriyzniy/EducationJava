В конвейере обработки перед вызовом слушателя есть два шага конвертации.

## 1. Первый шаг конвертации
Первый шаг использует `MessageConverter` 
- **для преобразования входящего сообщения Spring AMQP** `Message` 
- **в сообщение Spring Messaging** `Message`. 


> [!IMPORTANT] 
> При вызове целевого метода **полезная нагрузка** сообщения (payload), ==если необходимо==, **преобразуется в тип параметра метода**.

==Конвертером по умолчанию== для первого шага является `SimpleMessageConverter` Spring AMQP, который обрабатывает преобразование в `String` и объекты `java.io.Serializable`. 

Все остальные остаются как `byte[]`. В дальнейшем обсуждении мы будем называть его "конвертером сообщений".

## 2. Второй шаг конвертации
==Конвертером по умолчанию== для второго шага является `GenericMessageConverter`, который делегирует сервису конвертации (экземпляру `DefaultFormattingConversionService`). 
> В дальнейшем обсуждении мы будем называть его "конвертером аргументов метода".

### Описание кастомного конвертора
Чтобы изменить конвертер сообщений, вы **можете добавить его как свойство в бин фабрики контейнера**. 

Следующий пример показывает, как это сделать:
```java
@Bean
public SimpleRabbitListenerContainerFactory rabbitListenerContainerFactory() 
{
    SimpleRabbitListenerContainerFactory factory = 
	    new SimpleRabbitListenerContainerFactory();
    ...
    factory.setMessageConverter(new Jackson2JsonMessageConverter());
    ...
    return factory;
}
```
Это *настраивает Jackson2 конвертер*, который *ожидает наличия информации в заголовках* для управления преобразованием.

Вы также можете использовать `ContentTypeDelegatingMessageConverter`, который может обрабатывать преобразование различных типов контента.

## Указание конвертера в аннотации
Начиная с версии 2.3, вы можете переопределить конвертер фабрики, указав имя бина в свойстве `messageConverter`.
```java
@Bean
public Jackson2JsonMessageConverter jsonConverter() {
    return new Jackson2JsonMessageConverter();
}

@RabbitListener(..., messageConverter = "jsonConverter")
public void listen(String in) {
    ...
}
```
Это позволяет избежать необходимости объявлять другую фабрику контейнеров только для изменения конвертера.

В большинстве случаев нет необходимости настраивать конвертер аргументов метода, за исключением случаев, когда, например, вы хотите использовать пользовательский `ConversionService`.

## Выведение типа сообщения, если нет заголовка
В версиях **до 1.6** *информация о типе* для конвертации JSON **должна была предоставляться в заголовках сообщений**, или требовался пользовательский `ClassMapper`. 

Начиная **с версии 1.6**, *если нет заголовков* с информацией о типе, **тип может быть выведен из аргументов целевого метода**.

> Этот вывод типа работает только для `@RabbitListener` на уровне метода.

См. `Jackson2JsonMessageConverter` для получения дополнительной информации.

Если вы хотите настроить конвертер аргументов метода, вы можете сделать это следующим образом:
```java
@Configuration
@EnableRabbit
public class AppConfig implements RabbitListenerConfigurer {

    ...

    @Bean
    public DefaultMessageHandlerMethodFactory myHandlerMethodFactory() {
        
        DefaultMessageHandlerMethodFactory factory = 
	        new DefaultMessageHandlerMethodFactory();
	        
        factory.setMessageConverter(
	        new GenericMessageConverter(myConversionService()));
	        
        return factory;
    }

    @Bean
    public DefaultConversionService myConversionService() {
    
        DefaultConversionService conv = new DefaultConversionService();
        
        conv.addConverter(mySpecialConverter());
        return conv;
    }

    @Override
    public void configureRabbitListeners(
	    RabbitListenerEndpointRegistrar registrar) 
	{
        registrar
	        .setMessageHandlerMethodFactory(myHandlerMethodFactory());
    }
    ...

}
```

Для слушателей с несколькими методами (см. "[Multi-method Listeners :: Spring AMQP](https://docs.spring.io/spring-amqp/reference/amqp/receiving-messages/async-annotation-driven/method-selection.html)") выбор метода основан на полезной нагрузке сообщения после конвертации сообщения. Конвертер аргументов метода вызывается только после выбора метода.
