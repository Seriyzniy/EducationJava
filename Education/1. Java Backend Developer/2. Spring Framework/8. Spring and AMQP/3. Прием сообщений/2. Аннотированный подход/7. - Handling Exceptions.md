По умолчанию, если аннотированный *метод слушателя выбрасывает исключение*, **оно передается контейнеру,** 
- и сообщение повторно ставится в очередь и доставляется заново, 
- отбрасывается или направляется в обменник "мертвых писем" (dead letter exchange), в зависимости от конфигурации контейнера и брокера. 

Отправителю ничего не возвращается.

## Атрибуты аннотации @RabbitListener для обработки ошибок
Начиная с версии 2.0, аннотация `@RabbitListener` **имеет два новых атрибута:** `errorHandler` и `returnExceptions`.

Они не настроены по умолчанию.

### errorHandler для указания бина
Вы можете использовать `errorHandler`, чтобы указать имя бина, реализующего `RabbitListenerErrorHandler`. 

Этот функциональный интерфейс имеет один метод:
```java
@FunctionalInterface
public interface RabbitListenerErrorHandler {

    Object handleError(
	    Message amqpMessage, 
	    org.springframework.messaging.Message<?> message,
        ListenerExecutionFailedException exception) throws Exception;

}
```

Как видите, **у вас есть доступ** 
- к сырому сообщению, полученному из контейнера, 
	
- объекту `Message<?>` Spring Messaging, созданному конвертером сообщений, 
	
- и исключению, которое было выброшено слушателем (обернутому в `ListenerExecutionFailedException`). 

Обработчик ошибок может 
- **либо вернуть некоторый результат** (который отправляется в качестве ответа), 
- **либо выбросить исходное или новое исключение** (которое выбрасывается в контейнер или возвращается отправителю, в зависимости от настройки `returnExceptions`).

### returnExceptions - булевый атрибут
Атрибут `returnExceptions`, когда установлен в `true`, **приводит к ВОЗВРАЩЕНИЮ ИСКЛЮЧЕНИЙ ОТПРАВИТЕЛЮ.** 

Исключение оборачивается в объект `RemoteInvocationResult`. 

На стороне отправителя доступен `RemoteInvocationAwareMessageConverterAdapter`, который, если он настроен в `RabbitTemplate`, повторно выбрасывает исключение со стороны сервера, обернутое в `AmqpRemoteException`. 

Трассировка стека исключения сервера синтезируется путем объединения стеков сервера и клиента.

> Этот механизм, как правило, работает только с конвертером по умолчанию `SimpleMessageConverter`, который использует Java сериализацию. 
> 
> **Исключения**, как правило, 
> - **НЕ "дружественны к Jackson"** 
> - и не могут быть сериализованы в JSON. 
> 
> Если вы используете JSON, рассмотрите возможность использования `errorHandler` для возврата другого дружественного к Jackson объекта `Error` при возникновении исключения.

> В версии 2.1 этот интерфейс переместился из пакета `o.s.amqp.rabbit.listener` в `o.s.amqp.rabbit.listener.api`.

## Доступ к Channel в заголовке сообщения
Начиная **с версии 2.1.7**, `Channel` доступен в заголовке сообщения messaging; 
- это позволяет вам подтвердить (ack) 
- или отклонить (nack) неудачное сообщение при использовании `AcknowledgeMode.MANUAL`:

```java
public Object handleError(
	Message amqpMessage, 
	org.springframework.messaging.Message<?> message,
    ListenerExecutionFailedException exception) 
{
	...
	message
		.getHeaders()
		.get(AmqpHeaders.CHANNEL, Channel.class)
		.basicReject(
			message
				.getHeaders()
				.get(AmqpHeaders.DELIVERY_TAG, Long.class),
			true);
}
```

## Ошибки связанные с конвертацией
Начиная с версии 2.2.18, если возникает исключение конвертации сообщения, **обработчик ошибок будет вызван с** `null` в аргументе `message`. 

Это позволяет приложению отправить некоторый результат вызывающей стороне, указывая, что было получено некорректно сформированное сообщение. Ранее такие ошибки выбрасывались и обрабатывались контейнером.
