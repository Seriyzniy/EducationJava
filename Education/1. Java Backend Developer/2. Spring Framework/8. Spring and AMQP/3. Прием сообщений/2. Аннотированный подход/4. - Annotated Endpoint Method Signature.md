Пока что мы внедряли простую `String` в наш endpoint, но на самом деле **метод может иметь очень гибкую сигнатуру**. 

Следующий пример переписывает его для внедрения `Order` с пользовательским заголовком:
```java
@Component
public class MyService {

    @RabbitListener(queues = "myQueue")
    public void processOrder(\
	    Order order, 
	    @Header("order_type") String orderType) 
	{
        ...
    }
}
```

В следующем списке **перечислены аргументы**, которые *можно сопоставить с параметрами* в endpoint'ах слушателя:
1. Необработанный объект `org.springframework.amqp.core.Message`.
    
2. `MessageProperties` из необработанного сообщения `Message`.
    
3. `com.rabbitmq.client.Channel`, на котором было получено сообщение.
    
4. `org.springframework.messaging.Message`, преобразованный из входящего сообщения AMQP.
    
5. Аргументы метода, аннотированные `@Header`, для извлечения конкретного значения заголовка, включая стандартные заголовки AMQP.
    
6. Аргумент, аннотированный `@Headers`, который также должен быть приводим к типу `java.util.Map` для получения доступа ко всем заголовкам.
    
7. Преобразованная полезная нагрузка (payload).
    
8. Неаннотированный элемент, который не относится к поддерживаемым типам (то есть не `Message`, `MessageProperties`, `Message<?>` или `Channel`), сопоставляется с полезной нагрузкой. Вы можете сделать это явным, аннотировав параметр `@Payload`. Вы также можете включить валидацию, добавив дополнительную аннотацию `@Valid`.

## Использование Message
*Возможность внедрять абстракцию сообщений Spring* **особенно полезна, чтобы использовать всю информацию**, **хранящуюся в транспортно-специфичном сообщении**, без зависимости от транспортно-специфичного API. 

Следующий пример показывает, как это сделать:
```java
@RabbitListener(queues = "myQueue")
public void processOrder(Message<Order> order) 
{ 
	...
}
```

## DefaultMessageHandlerMethodFactory
Обработка аргументов метода обеспечивается `DefaultMessageHandlerMethodFactory`, который вы можете дополнительно настроить для поддержки дополнительных аргументов метода. Поддержку конвертации и валидации также можно настроить там же.

### Валидация перед обработкой
Например, если мы **хотим убедиться**, что наш `Order` **действителен перед обработкой**, мы можем аннотировать полезную нагрузку с помощью `@Valid` и настроить необходимый валидатор следующим образом:
```java
@Configuration
@EnableRabbit
public class AppConfig implements RabbitListenerConfigurer {

    @Override
    public void configureRabbitListeners(
	    RabbitListenerEndpointRegistrar registrar
	) {
        registrar
	        .setMessageHandlerMethodFactory(myHandlerMethodFactory());
    }

    @Bean
    public DefaultMessageHandlerMethodFactory myHandlerMethodFactory() {
        
        DefaultMessageHandlerMethodFactory factory = 
	        new DefaultMessageHandlerMethodFactory();
	        
        factory.setValidator(myValidator());
        return factory;
    }
}
```
