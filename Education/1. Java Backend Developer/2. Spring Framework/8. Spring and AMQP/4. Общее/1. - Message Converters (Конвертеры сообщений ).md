# Интерфейс MessageConverter
`AmqpTemplate` предоставляет методы для отправки и получения сообщений, делегируя преобразование данных `MessageConverter`. 

`MessageConverter` определяет два метода: 
- один для преобразования объекта в `Message`, 
- другой — для обратного преобразования из `Message`. 

При создании `Message` можно дополнительно задать его свойства, а передаваемый объект обычно используется как тело сообщения.

В следующем листинге показано определение интерфейса `MessageConverter`:
```java
public interface MessageConverter {

    Message toMessage(
	    Object object, 
	    MessageProperties messageProperties)
            throws MessageConversionException;

    Object fromMessage(
	    Message message) throws MessageConversionException;

}
```

## Методы отправки
Методы отправки в `AmqpTemplate`, работающие с объектами, проще, так как не требуют явного создания `Message`. 

Формированием сообщения занимается `MessageConverter`: он преобразует переданный объект в байтовое представление тела сообщения и добавляет заданные `MessageProperties`.

В следующем листинге показаны определения различных методов:
```java
void convertAndSend(Object message) throws AmqpException;

void convertAndSend(
	String routingKey, 
	Object message) throws AmqpException;

void convertAndSend(
	String exchange, 
	String routingKey, 
	Object message) throws AmqpException;

void convertAndSend(
	Object message, 
	MessagePostProcessor messagePostProcessor)throws AmqpException;

void convertAndSend(
	String routingKey, 
	Object message,
    MessagePostProcessor messagePostProcessor) throws AmqpException;

void convertAndSend(
	String exchange, 
	String routingKey, 
	Object message,
    MessagePostProcessor messagePostProcessor) throws AmqpException;
```

## Методы приема
На стороне приема есть только два метода: 
- один, который принимает имя очереди, 
- и другой, который полагается на свойство "queue" шаблона, которое должно быть установлено. 

В следующем листинге показаны определения двух методов:
```java
Object receiveAndConvert() throws AmqpException;

Object receiveAndConvert(String queueName) throws AmqpException;
```

`MessageListenerAdapter`, упомянутый в разделе "Асинхронный потребитель", также использует `MessageConverter`.

# 1. SimpleMessageConverter
`SimpleMessageConverter` — **стандартная реализация** `MessageConverter`, используемая `RabbitTemplate` по умолчанию. 
> Он поддерживает 
> - строки, 
> - сериализуемые Java-объекты 
> - и массивы байтов.

## При преобразовании ИЗ СООБЩЕНИЯ:  
Если `content-type`, входящего сообщения (`Message`), начинается с `text`(например, "text/plain"), тело сообщения преобразуется в `String` с учётом `content-encoding` (по умолчанию UTF-8, можно переопределить через `defaultCharset`). 

При `application/x-java-serialized-object` выполняется Java-десериализация, однако этот подход не рекомендуется из-за жёсткой связи между продюсером и консьюмером и невозможности интеграции с не-Java системами. 

Во всех остальных случаях возвращается массив байтов.

## При преобразовании В СООБЩЕНИЯ:  
При преобразовании в `Message` из произвольного Java-объекта `SimpleMessageConverter` аналогичным образом работает с массивами байтов, строками и сериализуемыми экземплярами. 

Он **преобразует каждый из них в байты** (в случае массивов байтов преобразовывать нечего) и соответствующим образом устанавливает свойство `content-type`. 

Если преобразуемый объект не соответствует одному из этих типов, тело сообщения (`Message body`) будет `null`.

# 2. SerializerMessageConverter
Этот конвертер аналогичен `SimpleMessageConverter`, за исключением того, что его можно настроить с другими реализациями `Serializer` и `Deserializer` Spring Framework для преобразований `application/x-java-serialized-object`.

> См. "Java десериализация" для получения важной информации.

# 3. JacksonJsonMessageConverter
Этот раздел посвящен использованию `JacksonJsonMessageConverter` для преобразования в `Message` и из `Message`. 

Он содержит следующие подразделы:
- Преобразование **в сообщение** (Message)
- Преобразование **из сообщения** (Message)

> Классы `AbstractJackson2MessageConverter`, их реализации и связанный API `Jackson2JavaTypeMapper` **помечены как устаревшие** (deprecated) для удаления в версии 4.0 ==в пользу соответствующих классов, основанных на Jackson 3==. См. JavaDocs устаревших классов для получения соответствующего руководства по миграции.

## 1. Преобразование в сообщение (Message)
Использовать Java-сериализацию для преобразования сообщений обычно не рекомендуется. Более универсальной и переносимой альтернативой является JSON. 

В Spring AMQP для этого применяется `JacksonJsonMessageConverter`, который можно настроить в `RabbitTemplate`, заменив стандартный `SimpleMessageConverter`. 

==Он использует библиотеку Jackson 3.x== и по умолчанию добавляет информацию о типе объекта в `MessageProperties` через `DefaultClassMapper`.

Следующий пример настраивает `JacksonJsonMessageConverter`:
```xml
<bean class="org.springframework.amqp.rabbit.core.RabbitTemplate">
    <property name="connectionFactory" ref="rabbitConnectionFactory"/>
    <property name="messageConverter">
        <bean class="org.springframework.amqp.support.converter.JacksonJsonMessageConverter">
            <!-- при необходимости переопределить DefaultClassMapper -->
            <property name="classMapper" ref="customClassMapper"/>
        </bean>
    </property>
</bean>
```
Как показано выше, `JacksonJsonMessageConverter` по умолчанию использует `DefaultClassMapper`. 
**Информация о типе добавляется в** `MessageProperties` (и извлекается из них).

### Если тип известен заранее
Если входящее сообщение *не содержит информацию о типе в* `MessageProperties`, **но вы знаете ожидаемый тип**, вы *можете настроить статический тип, используя свойство* `defaultType`, как показано в следующем примере:
```java
<bean id="jsonConverterWithDefaultType"
      class="o.s.amqp.support.converter.JacksonJsonMessageConverter">
    <property name="classMapper">
        <bean class="org.springframework.amqp.support.converter.DefaultClassMapper">
            <property name="defaultType" value="thing1.PurchaseOrder"/>
        </bean>
    </property>
</bean>

```

### Из заголовка TypeId
Также можно настроить пользовательские сопоставления значений заголовка `TypeId` с Java-классами:
```java
@Bean
public JacksonJsonMessageConverter jsonMessageConverter() {

    JacksonJsonMessageConverter jsonConverter = 
	    new JacksonJsonMessageConverter();
	    
    jsonConverter.setClassMapper(classMapper());
    
    return jsonConverter;
}

@Bean
public DefaultClassMapper classMapper() {
    DefaultClassMapper classMapper = new DefaultClassMapper();
    
    Map<String, Class<?>> idClassMapping = new HashMap<>();
    idClassMapping.put("thing1", Thing1.class);
    idClassMapping.put("thing2", Thing2.class);
    
    classMapper.setIdClassMapping(idClassMapping);
    return classMapper;
}
```

> [!NOTE] 
> В этом случае, **если** отправитель устанавливает заголовок `TypeId=thing1`, конвертер создаёт объект `Thing1`.

### Нововведение с contentEncoding
Начиная с версии 2.4.3, если `supportedMediaType` содержит параметр `charset`, конвертер не добавляет отдельное свойство `contentEncoding`, а использует charset напрямую. 

Для этого введён метод `setSupportedMediaType`:
```java
String utf16 = "application/json; charset=utf-16";

converter.setSupportedContentType(MimeTypeUtils.parseMimeType(utf16));
```

## 2. Преобразование из сообщения (Message)
При преобразовании **из сообщения** входящий JSON преобразуется в объект на основе информации о типе, переданной отправляющей системой в заголовках. 

Начиная с версии 2.4.3, если `contentEncoding` отсутствует, 
- конвертер пытается извлечь `charset` из `contentType`; 
- если его нет, используется `charset` из `supportedMediaType`, 
- а в крайнем случае — `defaultCharset`.

```java
String utf16 = "application/json; charset=utf-16";

converter.setSupportedContentType(MimeTypeUtils.parseMimeType(utf16));
```
> До версии 1.6 отсутствие информации о типе приводило к ошибке. 
> 
> Начиная с 1.6, в этом случае JSON десериализуется стандартными настройками Jackson (обычно в `Map`).

### Выведение типа
Кроме того, начиная **с версии 1.6**, **при использовании аннотаций** `@RabbitListener` (на методах) ==выведенная информация о типе добавляется в== `MessageProperties`. 
> Это позволяет конвертеру преобразовывать данные в тип аргумента целевого метода.

Это работает, если 
- есть один параметр без аннотаций 
- или один параметр с `@Payload`; 
- параметры типа `Message` в расчёт не берутся. 

#### Приоритет выведенного типа над типом указанном в заголовке
По умолчанию **выведенный тип C ИМЕЕТ ПРИОРИТЕТ НАД ЗАГОЛОВКАМИ** `TypeId`, что позволяет потребителю преобразовывать сообщения в собственную доменную модель. 

Это применяется **ТОЛЬКО ЕСЛИ ТИП параметра ЯВЛЯЕТСЯ КОНКРЕТНЫМ** (не абстрактным и не интерфейсом) или он из пакета `java.util`. **Во всех остальных случаях используется** `TypeId` и связанные заголовки.

---
#### Настройка на получение типа ТОЛЬКО из TypeId

*Бывают случаи*, когда вы можете захотеть переопределить поведение по умолчанию и *всегда использовать информацию* `TypeId`

Например, предположим, у вас есть `@RabbitListener`, принимающий аргумент `Thing1`, НО СООБЩЕНИЕ содержит `Thing2`, **который является подклассом** `Thing1` (который является конкретным).
Выведенный тип будет неверным.

> Чтобы справиться с этой ситуацией, **установите свойство** `TypePrecedence` в `JacksonJsonMessageConverter` в значение `TYPE_ID` вместо значения по умолчанию `INFERRED`. 
> 
> (Свойство фактически находится в `DefaultJacksonJavaTypeMapper` конвертера, но для удобства в конвертере предоставлен сеттер.) 
> 
> Если вы внедряете пользовательский маппер типов, вам следует установить свойство в маппере.

---
#### Требование соответсвия

**Для корректного преобразования** `contentType` ==должен быть совместим с JSON==. 
(используется проверка `contentType.contains("json")`)

Начиная с версии 2.2, 
- при отсутствии `contentType` 
- или при значении `application/octet-stream` 
**предполагается** `application/json`. 

> Чтобы вернуть старое поведение (возврат `byte[]`), нужно установить `assumeSupportedContentType=false`. 
> 
> Если тип контента не поддерживается, 
> - записывается *предупреждающее сообщение в лог* `Could not convert incoming message with content-type […​]`,
> 	
> - и `message.getBody()` возвращается как есть — как `byte[]`

---
#### Требования к Producer

Таким образом, чтобы соответствовать требованиям `JacksonJsonMessageConverter` на стороне потребителя, **производитель должен** 
- добавить свойство сообщения `contentType` — например, как `application/json` или `text/x-json`, 
	
- или использовать `JacksonJsonMessageConverter`, который устанавливает заголовок автоматически.


Примеры использования конвертера с `@RabbitListener`:
```java
@RabbitListener
public void thing1(Thing1 thing1) {...}

@RabbitListener
public void thing1(
	@Payload Thing1 thing1, 
	@Header("amqp_consumerQueue") String queue) {...}

@RabbitListener
public void thing1(
	Thing1 thing1, 
	o.s.amqp.core.Message message) {...}

@RabbitListener
public void thing1(
	Thing1 thing1, 
	o.s.messaging.Message<Foo> message) {...}


// ОШИБКА, НЕЛЬЗЯ ОПРЕДЕЛИТЬ КАКОЙ АРГУМЕНТ - ПОЛЕЗНАЯ НАГРУЗКА
@RabbitListener
public void thing1(
	Thing1 thing1, 
	String bar) {...}



@RabbitListener
public void thing1(
	Thing1 thing1, 
	o.s.messaging.Message<?> message) {...}
```

В первых четырёх случаях конвертер пытается преобразовать данные в `Thing1`. 

Пятый пример недопустим, так как невозможно определить, какой параметр является полезной нагрузкой. 

В шестом случае используются стандартные настройки Jackson из-за обобщённого (`wildcard`) типа.

> Однако *вы можете создать пользовательский конвертер* и использовать свойство сообщения `targetMethod`, чтобы решить, в какой тип преобразовывать JSON.

#### ОГРАНИЧЕНИЯ
==Вывод типа возможен только== **при использовании** `@RabbitListener` ==на уровне метода==. 
> При аннотации на уровне класса тип используется для выбора метода `@RabbitHandler`. 

Для пользовательских сценариев инфраструктура предоставляет свойство сообщения `targetObject`, доступное в кастомном конвертере.

### Доверенные пакеты и Optional
Начиная с версии 1.6.11, 
- `JacksonJsonMessageConverter` 
- и, следовательно, `DefaultJacksonJavaTypeMapper` (`DefaultClassMapper`) 
**предоставляют опцию** `trustedPackages` для преодоления уязвимости "Serialization Gadgets". 

> Для совместимости с предыдущими версиями по умолчанию `JacksonJsonMessageConverter` доверяет всем пакетам — то есть использует `*` для этой опции.

С версии 2.4.7 **конвертер можно настроить так,** чтобы при результате десериализации `null` он **возвращал** `Optional.empty()`:
```java
@RabbitListener(queues = "op.1")
void listen(@Payload(required = false) Thing payload) {
    handleOptional(payload);
}

@RabbitListener(queues = "op.2")
void listen(Optional<Thing> optional) {
    handleOptional(optional.orElse(this.emptyThing));
}
```

Чтобы включить эту функцию, установите `setNullAsOptionalEmpty` в `true`; когда `false` (по умолчанию), конвертер возвращается к исходному телу сообщения (`byte[]`).
```java
@Bean
JacksonJsonMessageConverter converter() {

    JacksonJsonMessageConverter converter = 
	    new JacksonJsonMessageConverter();
	    
    converter.setNullAsOptionalEmpty(true);
    
    return converter;
}
```

## Десериализация абстрактных классов
До версии 2.2.8, **если выведенный тип для** `@RabbitListener` **был абстрактным классом (включая интерфейсы**), 
конвертер возвращался к поиску информации о типе 
- в заголовках и, если она присутствовала, использовал ее; 
- если ее не было, ***он пытался создать абстрактный класс***. 

Это **вызывало проблему**, когда использовался пользовательский `ObjectMapper`, настроенный с пользовательским десериализатором для обработки абстрактного класса, но входящее сообщение имело неверные заголовки типов.

Начиная с версии 2.2.8, предыдущее поведение сохраняется по умолчанию. 

Если у вас есть такой пользовательский `ObjectMapper` и в***ы хотите игнорировать заголовки типов, всегда используя выведенный тип для преобразования***, установите `alwaysConvertToInferredType` в `true`

Это необходимо для обратной совместимости и чтобы избежать накладных расходов на попытку преобразования, когда оно будет неудачным (со стандартным `ObjectMapper`).

## Использование интерфейсов проекции Spring Data
Начиная с версии 2.2, ***вы можете преобразовывать JSON в интерфейс проекции Spring Data*** **вместо конкретного типа**. 

Это позволяет осуществлять очень избирательную и слабо связанную привязку к данным, включая поиск значений из нескольких мест внутри JSON-документа. 

Например, следующий интерфейс может быть определен как тип полезной нагрузки сообщения:
```java
interface SomeSample {

  @JsonPath({ "$.username", "$.user.name" })
  String getUsername();

}


@RabbitListener(queues = "projection")
public void projection(SomeSample in) {
    String username = in.getUsername();
    ...
}
```
Методы доступа (Accessor methods) будут использоваться по умолчанию ***для поиска имени свойства как поля в полученном JSON-документе**.* 

Выражение `@JsonPath` **позволяет настроить поиск значения** и даже определить несколько выражений JSON path для поиска значений **из нескольких мест**, пока выражение не вернет фактическое значение.

Чтобы включить эту функцию, установите `useProjectionForInterfaces` в `true` ==в конвертере сообщений==. 
Вы также должны добавить 
- `spring-data:spring-data-commons` 
- и `com.jayway.jsonpath:json-path` в class path.

При использовании в качестве параметра метода `@RabbitListener` тип интерфейса автоматически передается конвертеру как обычно.

## Преобразование из сообщения с помощью RabbitTemplate
Как упоминалось ранее, **информация о типе передается в заголовках сообщения**, чтобы помочь конвертеру при преобразовании из сообщения. 

***В большинстве случаев это работает нормально***. 

ОДНАКО **при использовании** универсальных типов (**generic types**) он может преобразовывать только простые объекты и известные «контейнерные» объекты (списки, массивы и карты). 

Начиная **с версии 2.0**, `JacksonJsonMessageConverter` реализует `SmartMessageConverter`, что позволяет использовать его с новыми методами `RabbitTemplate`, принимающими аргумент `ParameterizedTypeReference`. 

Это позволяет преобразовывать сложные универсальные типы, как показано в следующем примере:
```java
Thing1< Thing2<Cat, Hat> > thing1 =
    rabbitTemplate.receiveAndConvert(
	    new ParameterizedTypeReference< Thing1<Thing2<Cat, Hat>> >() { });
```

# 3. MarshallingMessageConverter
Еще один вариант — `MarshallingMessageConverter`. Он делегирует реализациям интерфейсов стратегий `Marshaller` и `Unmarshaller` из библиотеки **Spring OXM**. 
Подробнее об этой библиотеке можно прочитать [Marshalling XML by Using Object-XML Mappers :: Spring Framework](https://docs.spring.io/spring-framework/reference/data-access/oxm.html). 

В терминах конфигурации наиболее распространено предоставление только аргумента конструктора, поскольку большинство реализаций `Marshaller` также реализуют `Unmarshaller`. 

Следующий пример показывает, как настроить `MarshallingMessageConverter`:
```xml
<bean class="org.springframework.amqp.rabbit.core.RabbitTemplate">
    <property name="connectionFactory" ref="rabbitConnectionFactory"/>
    <property name="messageConverter">
        <bean class="org.springframework.amqp.support.converter.MarshallingMessageConverter">
            <constructor-arg ref="someImplementationOfMarshallerAndUnmarshaller"/>
        </bean>
    </property>
</bean>
```

# 4. JacksonXmlMessageConverter
Этот класс был представлен в версии 2.1 и может использоваться **для преобразования сообщений в XML и из XML.**

И `JacksonXmlMessageConverter`, и `JacksonJsonMessageConverter` имеют один и тот же базовый класс: `AbstractJacksonMessageConverter`.

`JacksonXmlMessageConverter` ***использует библиотеку Jackson 3.x***.

Вы можете использовать его так же, как `JacksonJsonMessageConverter`, за исключением того, что он поддерживает XML вместо JSON. 

Следующий пример настраивает `JacksonJsonMessageConverter`:
```xml
<bean id="xmlConverterWithDefaultType"
        class="org.springframework.amqp.support.converter.JacksonXmlMessageConverter">
    <property name="classMapper">
        <bean class="org.springframework.amqp.support.converter.DefaultClassMapper">
            <property name="defaultType" value="foo.PurchaseOrder"/>
        </bean>
    </property>
</bean>
```

См. [[#3. JacksonJsonMessageConverter]] для получения дополнительной информации.

> Начиная с версии 2.2, предполагается `application/xml`, если нет свойства `contentType` или оно имеет значение по умолчанию `application/octet-stream`. 
> 
> Чтобы вернуться к предыдущему поведению (возвращать неконвертированный `byte[]`), установите свойство конвертера `assumeSupportedContentType` в `false`.

# 5. ContentTypeDelegatingMessageConverter
Этот класс был представлен в версии 1.4.2 и позволяет делегировать определенному `MessageConverter` **на основе свойства** `contentType` в `MessageProperties`. 

По умолчанию он делегирует `SimpleMessageConverter`, если нет свойства `contentType` или есть значение, которое не соответствует ни одному из настроенных конвертеров. 

Следующий пример настраивает `ContentTypeDelegatingMessageConverter`:
```xml
<bean id="contentTypeConverter" class="ContentTypeDelegatingMessageConverter">
    <property name="delegates">
        <map>
            <entry key="application/json" value-ref="jsonMessageConverter" />
            <entry key="application/xml" value-ref="xmlMessageConverter" />
        </map>
    </property>
</bean>
```

# 6. Java десериализация
Десериализация Java-объектов из сообщений с типом `application/x-java-serialized-object` **потенциально опасна при работе с недоверенными источниками**. 
> Это относится к `SimpleMessageConverter` и `SerializerMessageConverter`, использующим `DefaultDeserializer`.

**По умолчанию список** разрешённых классов **пуст**, п***оэтому десериализация запрещена.*** 
Для её включения нужно явно задать разрешённые пакеты или классы с помощью шаблонов (например, `thing1.`, `thing1.thing2.Cat`, `.MySafeClass`). 
Шаблоны проверяются по порядку, и при отсутствии совпадений выбрасывается `SecurityException`.

Н==астройка выполняется через свойство== `allowedListPatterns`. 
Если все отправители сообщений доверенные, можно разрешить десериализацию для всех классов, установив переменную окружения `SPRING_AMQP_DESERIALIZATION_TRUST_ALL` или системное свойство `spring.amqp.deserialization.trust.all` в `true`.

# 7. Message Properties Converters

> [!IMPORTANT] 
> `MessagePropertiesConverter` **отвечает за преобразование между** 
> - `BasicProperties` клиента RabbitMQ 
> - и `MessageProperties` Spring AMQP. 
> 
> Обычно достаточно стандартной реализации `DefaultMessagePropertiesConverter`, но при необходимости её можно заменить.

По умолчанию конвертер преобразует `LongString` в `String`, если их размер не превышает 1024 байта (лимит настраивается через конструктор). 

Начиная **с версии 1.6**, более длинные заголовки остаются `LongString` и доступны через `getBytes()`, `toString()` или `getStream()`. 
> Это исправило старое поведение, при котором такие значения фактически представлялись как `DataInputStream` и некорректно отображались при выводе. 

Для возврата к поведению до 1.6 добавлен специальный конструктор:
```java
/**
 * Создать экземпляр, где LongString будут возвращены
 * неконвертированными или как java.io.DataInputStream, если они длиннее этого лимита.
 * Используйте этот конструктор с 'true', чтобы восстановить поведение до версии 1.6.
 * @param longStringLimit лимит.
 * @param convertLongLongStrings LongString, если false,
 * DataInputStream, если true.
 * @since 1.6
 */
public DefaultMessagePropertiesConverter(
	int longStringLimit, 
	boolean convertLongLongStrings) { ... }

```

С версии 1.6 в `MessageProperties` добавлено свойство `correlationIdString`, чтобы избежать лишних преобразований между `byte[]` и `String`. 
> Ранее при преобразовании в `BasicProperties`, используемые клиентом RabbitMQ, и обратно выполнялось ненужное преобразование `byte[]` <→ `String`, 
> 
> потому что 
> - `MessageProperties.correlationId` — это `byte[]`, 
> - а `BasicProperties` использует `String`.

Для совместимости введена политика `correlationIdPolicy` (по умолчанию `BYTES`), определяющая, какое из свойств (`correlationId`, `correlationIdString` или оба) использовать при входящих и исходящих сообщениях.

Для входящих сообщений:
- `STRING`: Сопоставляется только свойство `correlationIdString`
    
- `BYTES`: Сопоставляется только свойство `correlationId`
    
- `BOTH`: Сопоставляются оба свойства
    

Для исходящих сообщений:
- `STRING`: Сопоставляется только свойство `correlationIdString`
    
- `BYTES`: Сопоставляется только свойство `correlationId`
    
- `BOTH`: Учитываются оба свойства, при этом свойство `String` имеет приоритет


Также с версии 1.6 входящие свойства `deliveryMode` и `userId` больше не копируются напрямую в исходящие сообщения: они сохраняются в `receivedDeliveryMode` и `receivedUserId`, чтобы избежать их непреднамеренного распространения.

Начиная с версии 2.2, пользовательские заголовки типа `Class<?>` преобразуются с помощью `getName()`, а не `toString()`, что упрощает обработку на стороне потребителя. 

При поэтапных обновлениях системы потребителям может потребоваться временно поддерживать оба формата.
