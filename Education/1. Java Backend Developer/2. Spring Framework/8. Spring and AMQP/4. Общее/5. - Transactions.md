Spring AMQP умеет **автоматически управлять транзакциями** при работе с RabbitMQ — как в синхронных (`RabbitTemplate`), так и в асинхронных (`SimpleMessageListenerContainer`) сценариях.

Цель: **связать обработку сообщений и бизнес-логику (например, БД)** в единое корректное поведение при ошибках.

---
# 1. Семантика транзакций
Есть два способа указать фреймворку желаемую семантику транзакций. 
### 1-й способ `channelTransacted = true`
- RabbitMQ-канал становится **транзакционным**
    
- Все операции:
    - либо **commit**
    - либо **rollback** при исключении
    
- Если `false` (по умолчанию):
    - сообщения **автоматически подтверждаются**
    - откат бизнес-логики **не влияет** на RabbitMQ
    

Это **настройка конфигурации**, фиксируется при старте приложения.
### 2-й способ - Внешняя транзакция Spring (`PlatformTransactionManager`)
- Обычная Spring-транзакция (`@Transactional`)
    
- Может быть декларативной или императивной
    
- Более **динамична**, т.к. зависит от текущего потока выполнения
    

💡 Если **есть внешняя транзакция** и `channelTransacted=true`:

- commit/rollback RabbitMQ **откладывается** до завершения Spring-транзакции

Внешняя транзакция в принципе более динамична, поскольку система реагирует на состояние текущего потока во время выполнения. 
> Однако на практике это также часто является настройкой конфигурации, когда транзакции накладываются на приложение декларативно.

---
## Синхронный сценарий (RabbitTemplate)
```java
@Transactional
public void doSomething() {
    String incoming = rabbitTemplate.receiveAndConvert();
    // выполнить некоторую обработку в базе данных...
    String outgoing = processInDatabaseAndExtractReply(incoming);
    
    rabbitTemplate.convertAndSend(outgoing);
}
```

В предыдущем примере полезная нагрузка `String` принимается, конвертируется и отправляется как тело сообщения внутри метода, помеченного как `@Transactional`. 

***Если обработка в базе данных*** **завершается неудачно** с исключением, 
- входящее **сообщение возвращается брокеру**, 
- **а исходящее** сообщение **не отправляется**. 

Это применяется ко всем операциям с `RabbitTemplate` внутри цепочки транзакционных методов (если, например, `Channel` не манипулируется напрямую для раннего коммита транзакции).

## Асинхронный сценарий (MessageListenerContainer)
Для асинхронных случаев использования с `SimpleMessageListenerContainer`, если требуется внешняя транзакция, **контейнер должен запросить ее при настройке слушателя.** 


***Чтобы указать, что требуется внешняя транзакция,*** пользователь **предоставляет реализацию** `PlatformTransactionManager` контейнеру при его конфигурации. 

В следующем примере показано, как это сделать:
```java
@Configuration
public class ExampleExternalTransactionAmqpConfiguration {

    @Bean
    public SimpleMessageListenerContainer messageListenerContainer() {
        SimpleMessageListenerContainer container = 
	        new SimpleMessageListenerContainer();
	        
        container.setConnectionFactory(rabbitConnectionFactory());
        
        container.setTransactionManager(transactionManager());
        container.setChannelTransacted(true);
        
        container.setQueueName("some.queue");
        container.setMessageListener(exampleListener());
        
        return container;
    }
}
```
> В предыдущем примере менеджер транзакций добавляется как зависимость, внедренная из другого определения бина (не показано), и флаг `channelTransacted` также установлен в `true`.


Эффект заключается в том, что ***если слушатель завершается неудачно с исключением***, **транзакция откатывается**, **и сообщение также возвращается брокеру**. 

Важно отметить, что **если транзакция не может быть зафиксирована** (например, из-за ошибки ограничения базы данных или проблемы с подключением), **AMQP-транзакция также откатывается, и сообщение возвращается брокеру**. 
> Это иногда называют "Лучшие усилия однофазного коммита" ("Best Efforts 1 Phase Commit") и является очень мощным паттерном для надежного обмена сообщениями. 

***Если бы*** в предыдущем примере флаг `channelTransacted` **был установлен в** `false` (по умолчанию), **внешняя транзакция все равно предоставлялась** бы для слушателя, **НО все операции обмена сообщениями автоматически подтверждались**, поэтому эффект заключается в фиксации операций обмена сообщениями даже при откате бизнес-операции.

# 2. Условный откат (Conditional Rollback)
**До версии 1.6.6** ***добавление правила отката*** в `transactionAttribute` контейнера при использовании внешнего менеджера транзакций (например, JDBC) **не имело эффекта**. 
**Исключения всегда приводили к откату транзакции**.

Кроме того, при использовании совета по транзакциям (transaction advice) в цепочке советов контейнера условный откат был не очень полезен, поскольку все исключения слушателя оборачиваются в `ListenerExecutionFailedException`.


Начиная с **1.6.6**, эта проблема исправлена: правила отката теперь **применяются корректно**. 

Для этого введён класс `ListenerFailedRuleBasedTransactionAttribute` — подкласс `RuleBasedTransactionAttribute`, который учитывает `ListenerExecutionFailedException` и применяет правила отката **к реальной причине исключения**, а не к обёртке. 

Этот атрибут можно использовать как напрямую в контейнере, так и через transaction advice.

Пример: транзакция **не откатывается**, если причиной ошибки является `DontRollBackException`:

```java
@Bean
public AbstractMessageListenerContainer container() {
    ...
    container.setTransactionManager(transactionManager);
    
    
    RuleBasedTransactionAttribute transactionAttribute =
        new ListenerFailedRuleBasedTransactionAttribute();
        
    transactionAttribute.setRollbackRules(Collections.singletonList(
        new NoRollbackRuleAttribute(DontRollBackException.class)));
        
        
    container.setTransactionAttribute(transactionAttribute);
    ...
}
```

Итого: с версии 1.6.6 в Spring AMQP стало возможно **реально управлять откатом транзакций по типу исключений**, даже с учётом обёртки `ListenerExecutionFailedException`.

## Примечание об откате полученных сообщений
==AMQP-транзакции распространяются== **только на отправку сообщений и подтверждения**. 
Поэтому ***при откате Spring-транзакции*** *после получения сообщения* Spring AMQP вынужден 
- **явно отклонить сообщение** (аналог `nack`). 
- откатить транзакцию

Что произойдёт дальше, **не связано с транзакциями** и определяется свойством `defaultRequeueRejected` (по умолчанию `true`: сообщение возвращается в очередь).

> До версии RabbitMQ **2.7.0** такие сообщения помещались в **конец очереди**; 
> начиная с 2.7.0 — в **начало очереди**, как при откате в JMS.

Ранее поведение при откате было **несогласованным**:  
- при локальных AMQP-транзакциях применялась обычная логика requeue (`defaultRequeueRejected`, `AmqpRejectAndDontRequeueException`),  
	
- при использовании `TransactionManager` **сообщение ВСЕГДА ВОЗВРАЩАЛОСЬ В ОЧЕРЕДЬ**.

Начиная с версии **2.0**, поведение унифицировано: **обычные правила requeue применяются в обоих случаях**. 

Старое поведение можно вернуть, установив `alwaysRequeueWithTxManagerRollback=true` в контейнере.

# 3. Использование `RabbitTransactionManager`
`RabbitTransactionManager` — это **альтернатива внешним транзакциям** (например, JDBC) для работы с RabbitMQ. Он реализует `PlatformTransactionManager` и управляет **только AMQP-транзакциями**, работая с **одной `ConnectionFactory`**.

> Важно: этот подход **не поддерживает XA-транзакции**, поэтому **нельзя объединить** транзакции RabbitMQ и базы данных в одну атомарную операцию.

Для участия в транзакции, **код должен получать канал через**
- `ConnectionFactoryUtils.getTransactionalResourceHolder(...)`, 
- а не создавать его напрямую `Connection.createChannel()`.
При использовании `RabbitTemplate` это происходит **автоматически** — он подхватывает привязанный к потоку транзакционный `Channel`.

Конфигурация через Java:
```java
@Bean
public RabbitTransactionManager rabbitTransactionManager() {
    return new RabbitTransactionManager(connectionFactory);
}
```

Или через XML:
```xml
<bean id="rabbitTxManager"
      class="org.springframework.amqp.rabbit.transaction.RabbitTransactionManager">
    <property name="connectionFactory" ref="connectionFactory"/>
</bean>
```

`RabbitTransactionManager` удобен для **чисто AMQP-сценариев**, но не подходит, если требуется согласованность с БД.

# 4. Синхронизация транзакций (Transaction Synchronization)
Синхронизация транзакции RabbitMQ с другой транзакцией (например, БД) реализует семантику **Best Efforts 1-Phase Commit**

При этом возможно, что транзакция RabbitMQ **не зафиксируется на этапе завершения**, — такая ошибка **логируется** инфраструктурой `spring-tx`, НО **НЕ ПРОБРАСЫВАЕТСЯ*** в вызывающий код.

Начиная с версии **2.3.10**, это поведение можно отследить: после коммита транзакции в том же потоке можно вызвать  
`ConnectionUtils.checkAfterCompletion()`. 
> Если ошибка была — будет выброшено `AfterCompletionFailedException`, содержащее статус завершения синхронизации.

Функция включается глобально для всех потоков:
```java
ConnectionFactoryUtils
	.enableAfterCompletionFailureCapture(true);
```

по умолчанию ошибки завершения RabbitMQ-транзакции **тихие**, но с 2.3.10 их можно **явно обнаружить и обработать**.
