Многие операции Java-клиента RabbitMQ выбрасывают **checked-исключения** (чаще всего `IOException`). 

Компоненты Spring AMQP 
- `RabbitTemplate`, 
- `SimpleMessageListenerContainer` и др.
**перехватывают их и преобразуют** в исключения иерархии `AmqpException` (`org.springframework.amqp`).

Если исключение выбрасывается **внутри listener’а**, оно оборачивается в `ListenerExecutionFailedException`. По умолчанию сообщение **rejected и requeued** (сообщение отклоняется и ставится в очередь)

Поведение можно изменить:
- `defaultRequeueRejected=false` → сообщение отбрасывается или уходит в DLQ
    
- `AmqpRejectAndDontRequeueException` / `ImmediateRequeueAmqpException` → условный контроль из кода listener’а ([Resilience: Recovering from Errors and Broker Failures :: Spring AMQP](https://docs.spring.io/spring-amqp/reference/amqp/resilience-recovering-from-errors-and-broker-failures.html#async-listeners))


## Проблема неустранимых ошибок
***Существуют ошибки, которые происходят*** **до вызова пользовательского кода** (например, ошибка конвертации сообщения). 

В этом случае listener не может повлиять на ack/nack, и при `defaultRequeueRejected=true` сообщение будет доставляться бесконечно.

До версии 1.3.2 пользователям нужно было писать пользовательский `ErrorHandler`, как обсуждалось в разделе "Обработка исключений", чтобы избежать этой ситуации.

## ConditionalRejectingErrorHandler (c 1.3.2)
Начиная **с версии 1.3.2**, ***обработчиком ошибок по умолчанию*** теперь является `ConditionalRejectingErrorHandler`, **который отклоняет (и не ставит обратно в очередь) сообщения**, которые завершаются неустранимой (irrecoverable) ошибкой.

В частности, он отклоняет сообщения, которые завершаются следующими ошибками:
- `o.s.amqp…​MessageConversionException`: Может быть выброшено ***при конвертации полезной нагрузки*** входящего сообщения с использованием `MessageConverter`.
    
- `o.s.messaging…​MessageConversionException`: Может быть выброшено *службой конвертации*, ***если требуется дополнительная конвертация*** при маппинге на метод `@RabbitListener`.
    
- `o.s.messaging…​MethodArgumentNotValidException`: Может быть выброшено, ***если в слушателе используется валидация*** (например, `@Valid`) и ***валидация не проходит***.
    
- `o.s.messaging…​MethodArgumentTypeMismatchException`: Может быть выброшено, если входящее с***ообщение было сконвертировано в тип, не подходящий для целевого метода***. 
	- Например, параметр объявлен как `Message<Foo>`, но получено `Message<Bar>`.
    
- `java.lang.NoSuchMethodException`: Добавлено в версии 1.6.3.
    
- `java.lang.ClassCastException`: Добавлено в версии 1.6.3.

> Вы можете настроить экземпляр этого обработчика ошибок с `FatalExceptionStrategy`, чтобы пользователи могли предоставить свои собственные правила для условного отклонения сообщений — например, делегируя реализацию `BinaryExceptionClassifier` из Spring Retry (раздел "Слушатели сообщений и асинхронный случай"). 

Кроме того, `ListenerExecutionFailedException` **теперь имеет свойство** `failedMessage`, которое вы можете использовать при принятии решения. 
> Если метод `FatalExceptionStrategy.isFatal()` возвращает `true`, обработчик ошибок выбрасывает `AmqpRejectAndDontRequeueException`

По умолчанию `FatalExceptionStrategy` логирует предупреждение, когда исключение определяется как неустранимое.

## Кастомизация фатальных ошибок
Поведение можно расширить:
- задать собственный `FatalExceptionStrategy`
    
- использовать `ListenerExecutionFailedException.getFailedMessage()`
    
- с версии **1.6.3** — унаследоваться от  
    `ConditionalRejectingErrorHandler.DefaultExceptionStrategy`  
    и переопределить `isUserCauseFatal(...)`
    

## DLQ, TTL и зацикливание
Распространённый паттерн — DLQ + TTL + возврат в основную очередь.  
Проблема: **фатальные сообщения начинают зацикливаться**.

Начиная с **2.1**:
- ***если сообщение*** с неустранимой ошибкой ==содержит заголовок== `x-death`, оно **логируется и отбрасывается**
    
- старое поведение можно вернуть установив 
    `discardFatalsWithXDeath=false` в `ConditionalRejectingErrorHandler` 
    

## MANUAL ack и фатальные ошибки
Начиная с **2.1.9**:
- сообщения с неустранимыми ошибками **отклоняются И НЕ requeue’ятся**, даже при `AckMode.MANUAL`
    
- причина: такие ошибки возникают **до вызова listener’а**, и сообщение иначе зависает в _unacked_
    

Для возврата старого поведения:
- `rejectManual=false` в `ConditionalRejectingErrorHandler`
