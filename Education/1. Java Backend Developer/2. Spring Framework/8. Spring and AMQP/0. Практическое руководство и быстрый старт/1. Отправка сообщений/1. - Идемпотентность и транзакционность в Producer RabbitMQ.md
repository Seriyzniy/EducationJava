**В RabbitMQ нет встроенной поддержки exactly-once delivery** как в Kafka. Вместо этого используются комбинированные подходы:

|Концепция|**В Kafka**|**В RabbitMQ**|
|---|---|---|
|**Идемпотентность**|Встроенная (`enable.idempotence`)|Нет встроенной, реализуется на уровне приложения|
|**Транзакции**|Поддерживаются (`transactional.id`)|Есть, но **только для single-broker**|
|**Exactly-once**|Возможно с транзакциями|Невозможно в распределенном виде|

# 1. Настройка транзакционности Producer
RabbitMQ поддерживает транзакции на уровне **канала (Channel)**, но только для одного брокера.

> [!NOTE] ТРАНЗАКЦИИ РЕСУРСОЁМКИЕ
> Using standard AMQP 0-9-1, the only way to guarantee that a message isn't lost is by using transactions -- make the channel transactional then for each message or set of messages publish, commit. 
> 
> In this case, transactions are unnecessarily heavyweight and decrease throughput by a factor of 250. To remedy this, a confirmation mechanism was introduced. It mimics the consumer acknowledgements mechanism already present in the protocol.
> [Consumer Acknowledgements and Publisher Confirms | RabbitMQ](https://www.rabbitmq.com/docs/confirms)
>
> **Перевод**
> При использовании стандарта AMQP 0-9-1 единственный способ гарантировать, что сообщение не будет потеряно — это использование транзакций: сделать канал транзакционным, затем для каждого сообщения или набора сообщений выполнять публикацию и подтверждение (commit).
> 
> В этом случае **транзакции являются избыточно ресурсоёмкими и снижают пропускную способность в 250 раз**. Для решения этой проблемы был введён механизм подтверждений (confirmations). Он повторяет механизм подтверждений (acknowledgements) потребителем, уже существующий в протоколе.

```java
@Bean
public RabbitTemplate transactionalRabbitTemplate(
        ConnectionFactory connectionFactory) {
    
    RabbitTemplate template = new RabbitTemplate(connectionFactory);
    
    // 1. Включение транзакционного режима
    template.setChannelTransacted(true); // Ключевой параметр!
    
    // 2. Настройка подтверждений
    template.setConfirmCallback((correlationData, ack, cause) -> {
        if (!ack) {
            // Транзакция откатится автоматически
            log.error("Confirm failed: {}", cause);
        }
    });
    
    // 3. MessageConverter для сериализации
    template.setMessageConverter(jsonMessageConverter());
    
    return template;
}
// В результате:
// 1. Перед отправкой: channel.txSelect()
// 2. После успешной отправки: channel.txCommit()
// 3. При ошибке: channel.txRollback()
```

## **Spring @Transactional с RabbitMQ**
### **Интеграция с JDBC транзакциями (Chained Transactions)**
```java
@Configuration
@EnableTransactionManagement
public class TransactionConfig {
    
    @Bean
    public PlatformTransactionManager transactionManager(
            DataSource dataSource) 
    {
        return new DataSourceTransactionManager(dataSource);
    }
    
    @Bean
    public RabbitTemplate transactionalRabbitTemplate(
            ConnectionFactory connectionFactory) 
    {
        RabbitTemplate template = new RabbitTemplate(connectionFactory);
        
        template.setChannelTransacted(true); 
        
        return template;
    }
}

// Использование в сервисе
@Service
public class OrderService {
    
    @Transactional // Использует DataSourceTransactionManager
    public void createOrder(Order order) {
        // 1. Сохраняем в БД
        orderRepository.save(order);
        
        // 2. Отправляем в RabbitMQ в той же транзакции
        // Spring синхронизирует Rabbit транзакцию с JDBC
        rabbitTemplate.convertAndSend(
            "orders-exchange",
            "order.created",
            order
        );
        
        // Если здесь будет исключение - откатятся и БД и RabbitMQ
    }
}
```


> [!NOTE] Ограничения Spring транзакций с RabbitMQ:
> 1. Rabbit транзакции **работают только с JDBC/Hibernate транзакциями**
> 2. ***Не работают с JTA/XA*** распределенными транзакциями
> 3. ==Требуют channelTransacted = true==
> 4. Не поддерживают двухфазный commit (2PC) (как в кафке, либо все клиенты коммитят, либо никто)
> 5. Если нужно распределенные транзакции - используйте паттерны Saga/Outbox

## **Паттерн Transactional Outbox**
Для распределенных транзакций в микросервисах:
### **Реализация Outbox Pattern:**
```java
@Component
public class OutboxMessageProducer {
    
    @Transactional
    public void publishWithOutbox(Order order) {
        // 1. Сохраняем бизнес-данные
        orderRepository.save(order);
        
        // 2. Сохраняем сообщение в outbox таблицу (в ту же БД транзакцию)
        OutboxMessage outboxMessage = new OutboxMessage();
        outboxMessage.setAggregateId(order.getId());
        outboxMessage.setAggregateType("ORDER");
        outboxMessage.setEventType("ORDER_CREATED");
        outboxMessage.setPayload(objectMapper.writeValueAsString(order));
        outboxMessage.setStatus("PENDING");
        
        outboxRepository.save(outboxMessage);
        // Коммит БД транзакции
    }
}

// Отдельный процесс (scheduler) отправляет из outbox
@Component
public class OutboxPublisher {
    
    @Scheduled(fixedDelay = 5000)
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void publishPendingMessages() {
        List<OutboxMessage> pending = outboxRepository
            .findByStatus("PENDING", PageRequest.of(0, 100));
        
        for (OutboxMessage message : pending) {
            try {
                // Отправляем в RabbitMQ
                rabbitTemplate.convertAndSend(
                    "events-exchange",
                    message.getEventType(),
                    message.getPayload()
                );
                
                // Помечаем как отправленное
                message.setStatus("SENT");
                outboxRepository.save(message);
                
            } catch (Exception e) {
                log.error("Failed to publish message: {}", message.getId(), e);
                message.setStatus("FAILED");
                outboxRepository.save(message);
            }
        }
    }
}
```

# 2. Идемпотентность
Так как встроенной идемпотентности нет, используются паттерны:

## **Паттерн 1: Идемпотентный Consumer с deduplication**
Producer отправляет сообщение с уникальным ключом
```java
// Producer отправляет с уникальным ID
public void sendIdempotentMessage(Order order) {
    String messageId = UUID.randomUUID().toString();
    
    rabbitTemplate.convertAndSend(
        "orders-exchange",
        "order.created",
        order,
        message -> {
            // Добавляем уникальный ID сообщения
            message.getMessageProperties()
                   .setMessageId(messageId);
            message.getMessageProperties()
                   .setHeader("idempotency-key", messageId);
            message.getMessageProperties()
                   .setTimestamp(new Date());
            return message;
        }
    );
}
```

А COnsumer используя БД, находит дубликаты сообщений и выполняет соответсвующие действия
(аналогично как расказано в [[7.2 -- Idempotence Consumer и обеспечение Exactly-once]])

## **Паттерн 2: Persistent Messages + Publisher Confirms**
```java
@Bean
public RabbitTemplate reliableRabbitTemplate(
        ConnectionFactory connectionFactory) {
    
    RabbitTemplate template = new RabbitTemplate(connectionFactory);
    
    // 1. Обязательные подтверждения
    template.setConfirmCallback((correlationData, ack, cause) -> {
        if (ack) {
            // Удалить из локальной БД отправленных сообщений
            messageLogService.markAsSent(correlationData.getId());
        } else {
            // Повторить отправку или пометить как ошибку
            messageLogService.markAsFailed(
                correlationData.getId(), cause);
        }
    });
    
    // 2. Persistent messages
    template.setChannelTransacted(false); // Для perfomance
    
    template.setMessagePostProcessor(message -> {
        message.getMessageProperties()
               .setDeliveryMode(MessageDeliveryMode.PERSISTENT);
        message.getMessageProperties()
               .setPriority(0);
        return message;
    });
    
    return template;
}
```


## Паттерн 3: Idempotent Producer
### **Реализация с локальным кэшем отправленных сообщений:**
```java
@Component
public class IdempotentRabbitProducer {
    
    private final Cache<String, Boolean> sentMessagesCache = 
        CacheBuilder.newBuilder()
            .expireAfterWrite(1, TimeUnit.HOURS)
            .maximumSize(10000)
            .build();
    
    public void sendIdempotent(String idempotencyKey, Object message) {
        
        // Проверяем, не отправляли ли уже
        if (sentMessagesCache.getIfPresent(idempotencyKey) != null) {
            log.warn("Duplicate idempotency key: {}", idempotencyKey);
            return; // Уже отправлено, игнорируем
        }
        
        try {
            // Отправляем с подтверждением
            CorrelationData correlationData = 
                new CorrelationData(idempotencyKey);
            
            rabbitTemplate.convertAndSend(
                "exchange",
                "routing.key",
                message,
                correlationData
            );
            
            // Ждем подтверждения
            Confirm confirm = correlationData.getFuture()
                .get(5, TimeUnit.SECONDS);
            
            if (confirm.isAck()) {
                // Кэшируем как успешно отправленное
                sentMessagesCache.put(idempotencyKey, true);
            } else {
                throw new AmqpException("Message not confirmed");
            }
            
        } catch (Exception e) {
            // Не кэшируем при ошибке - можно повторить
            throw new AmqpException("Send failed", e);
        }
    }
}
```
