**Spring AMQP состоит из двух модулей** (каждый представлен JAR-файлом в дистрибутиве):
- **`spring-amqp`** — содержит универсальные абстракции AMQP (пакет `org.springframework.amqp.core`), не зависящие от конкретного брокера или клиентской библиотеки. Это позволяет писать переносимый код на уровне AMQP-модели.
    
- **`spring-rabbit`** — конкретная реализация этих абстракций для **RabbitMQ**.


Абстракции `spring-amqp` теоретически могут быть реализованы для других AMQP-брокеров (например, проверялись с Apache Qpid), но на практике **официально поддерживается и тестируется только RabbitMQ**.

# 1. Сообщение Message
Спецификация AMQP 0-9-1 **не определяет** класс или интерфейс Message (Сообщение). 
Вместо этого, при выполнении операции, такой как `basicPublish()`, содержимое передается как аргумент в виде массива байтов (`byte-array`), а дополнительные свойства передаются в виде отдельных аргументов.

**Spring AMQP определяет класс `Message`** как часть более общей модели домена AMQP. Цель класса `Message` — **инкапсулировать тело сообщения и его свойства в едином экземпляре**, чтобы API, в свою очередь, мог быть проще.

Следующий пример показывает определение класса `Message`:
```java
public class Message {

    private final MessageProperties messageProperties;

    private final byte[] body;

    public Message(byte[] body, MessageProperties messageProperties) {
        this.body = body;
        this.messageProperties = messageProperties;
    }

    public byte[] getBody() {
        return this.body;
    }

    public MessageProperties getMessageProperties() {
        return this.messageProperties;
    }
}
```

## Интерфейс MessageProperties
Интерфейс `MessageProperties` определяет несколько общих свойств, таких как:
- `messageId` (идентификатор сообщения)
- `timestamp` (временная метка)
- `contentType` (тип содержимого)
... и многие другие.

Вы также можете **расширить эти свойства пользовательскими ЗАГОЛОВКАМИ**, вызывая метод `setHeader(String key, Object value)`.

## Замечание
Начиная с версий 1.5.7, 1.6.11, 1.7.4 и 2.0.0, если тело сообщения представляет собой сериализованный `Serializable` Java-объект, оно больше **не десериализуется по умолчанию** при выполнении операций `toString()` (например, в лог-сообщениях). Это сделано для **предотвращения небезопасной десериализации**.

По умолчанию десериализуются только классы из пакетов **`java.util`** и **`java.lang`**.

Чтобы вернуться к предыдущему поведению, вы можете добавить разрешённые шаблоны классов/пакетов, вызвав `Message.addAllowedListPatterns(…)`. Поддерживается простой символ подстановки `*` (wildcard), например `com.something.*`, `*.MyClass`. Тела сообщений, которые не могут быть десериализованы, в лог-сообщениях представляются как `byte[<размер>]`.

# 2. Exchange
Интерфейс `Exchange` представляет собой **AMQP Exchange (обменник)**, куда отправляет сообщения продюсер (Message Producer).

Каждый Exchange в пределах виртуального хоста брокера имеет:
- **Уникальное имя**
- Несколько других свойств

## Интерфейс Exchange
```java
public interface Exchange extends Declarable {

    String getName();

    String getType();

    boolean isDurable();

    boolean isAutoDelete();

    Map<String, Object> getArguments();

}
```

Exchange также имеет **'тип'**, представленный константами, определенными в `ExchangeTypes`. Основные типы:
1. **direct** (прямой)
2. **topic** (тематический)
3. **fanout** (широковещательный)
4. **headers** (по заголовкам)


В **базовом пакете можно найти РЕАЛИЗАЦИИ ИНТЕРФЕЙСА** `Exchange` ==для каждого из этих типов==.

## ConsistentHashExchange (начиная с версии 3.2)
Начиная с версии 3.2, для удобства на этапе конфигурации приложения был введен тип **`ConsistentHashExchange`**. Он предоставляет такие опции, как `x-consistent-hash` для типа exchange. Позволяет настроить аргументы определения exchange `hash-header` или `hash-property`. Соответствующий плагин RabbitMQ `rabbitmq_consistent_hash_exchange` должен быть включен на брокере. Подробнее о назначении, логике и поведении Consistent Hash Exchange можно узнать в официальной документации RabbitMQ.

## Exchange по умолчанию
Спецификация AMQP также требует, чтобы любой брокер предоставлял **"default" (по умолчанию) direct exchange**, который **не имеет имени**. Все объявленные очереди привязываются к этому Exchange по умолчанию с их именами в качестве ключей маршрутизации.

Вы можете узнать больше об использовании Exchange по умолчанию в Spring AMQP в разделе `AmqpTemplate`.

# 3. Queue
Класс **`Queue`** представляет **компонент, из которого потребитель сообщений (message consumer) получает сообщения.** 

Подобно различным классам Exchange, наша реализация предназначена для абстрактного представления этого основного типа AMQP.
```java
public class Queue {

    private final String name;       // Имя очереди
    private final boolean durable;   // Устойчивость (durable)
    private final boolean exclusive; // Эксклюзивность
    private final boolean autoDelete; // Автоудаление

    /**
     * Очередь является durable (устойчивой), non-exclusive (неэксклюзивной) 
     * и non auto-delete (неавтоудаляемой).
     *
     * @param name имя очереди.
     */
    public Queue(String name) {
        this(name, true, false, false);
    }

    // Геттеры и сеттеры опущены для краткости
}
```

## Особенности конструктора
Обратите внимание, что конструктор принимает **имя очереди**. В зависимости от реализации, административный шаблон (admin template) может предоставлять методы для генерации **уникально названной очереди**.

## Автоматически генерируемые очереди
Такие автоматически сгенерированные очереди могут быть полезны в качестве:
- **Адреса "reply-to"** (для ответов в RPC-паттерне)
- В других **временных ситуациях**


По этой причине свойства **'exclusive' (эксклюзивность)** и **'autoDelete' (автоудаление)** автоматически сгенерированной очереди будут установлены в **'true'**:
- `exclusive = true` - очередь доступна только текущему соединению
    
- `autoDelete = true` - очередь автоматически удаляется при отключении потребителя

## Дополнительная информация
См. раздел об очередях в "*Configuring the Broker*" для получения информации об объявлении очередей с использованием поддержки пространства имен, включая аргументы очередей.

# 4. Binding
Учитывая, что продюсер отправляет сообщения в **exchange (обменник)**, а потребитель получает из **queue (очереди)**, **привязки (bindings)**, которые соединяют очереди с обменниками, являются **критически важными** для связи продюсеров и потребителей через систему обмена сообщениями.

В Spring AMQP мы определяем класс **`Binding`** для представления этих соединений. Этот раздел рассматривает основные варианты привязки очередей к обменникам.

## Использование BindingBuilder для Fluent API
Мы предоставляем **`BindingBuilder`** для облегчения стиля "fluent API", как показано в следующем примере:
```java
Queue queue = ...;

// Привязать очередь к DirectExchange с фиксированным ключом маршрутизации
Binding directBinding = BindingBuilder.bind(queue)
    .to(new DirectExchange("someDirectExchange"))
    .with("foo.bar");

// Привязать очередь к TopicExchange с шаблоном маршрутизации
Binding topicBinding = BindingBuilder.bind(queue)
    .to(new TopicExchange("someTopicExchange"))
    .with("foo.*");

// Привязать очередь к FanoutExchange без ключа маршрутизации
Binding fanoutBinding = BindingBuilder.bind(queue)
    .to(new FanoutExchange("someFanoutExchange"));
```

Для ясности предыдущий пример показывает класс `BindingBuilder`, но этот стиль **хорошо работает при использовании статического импорта для метода** `'bind()'`.

## Природа экземпляра Binding
**Сам по себе экземпляр класса `Binding` только хранит данные о соединении.** Другими словами, это **не "активный" компонент**.

Однако, как вы увидите позже в разделе "Configuring the Broker", класс `AmqpAdmin` может использовать экземпляры `Binding` для фактического **запуска действий привязки на брокере**.

## Определение Binding через конфигурацию Spring
Как вы можете видеть в том же разделе, вы можете определять экземпляры `Binding`, используя аннотации Spring `@Bean` внутри классов `@Configuration`.

Также есть **удобный базовый класс**, который дополнительно упрощает этот подход для создания определений бинов, связанных с AMQP, и распознает очереди, обменники и привязки так, что они все объявляются на AMQP-брокере при запуске приложения.

## AmqpTemplate
`AmqpTemplate` также определен в основном пакете. Будучи одним из основных компонентов, участвующих в фактическом обмене сообщениями AMQP, он подробно обсуждается в своем собственном разделе (см. `AmqpTemplate`).
