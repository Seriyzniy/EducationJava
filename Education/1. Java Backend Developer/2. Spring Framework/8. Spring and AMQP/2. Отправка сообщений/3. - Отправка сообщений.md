При отправке сообщения вы можете использовать любой из следующих методов:
```java
void send(Message message) throws AmqpException;

void send(String routingKey, Message message) throws AmqpException;

void send(String exchange, String routingKey, Message message) throws AmqpException;
```

Мы можем начать обсуждение с последнего метода в приведенном списке, так как он фактически является наиболее явным. 
Он позволяет указать имя обменника AMQP (вместе с ключом маршрутизации) во время выполнения. 

Последний параметр — это колбэк, отвечающий за фактическое создание экземпляра сообщения.

Пример использования этого метода для отправки сообщения может выглядеть так: **Следующий пример показывает, как использовать метод send для отправки сообщения**:
```java
amqpTemplate.send(
	"marketData.topic", 
	"quotes.nasdaq.THING1",
    new Message("12.34".getBytes(), someProperties));
```



Вы можете установить свойство `exchange` в самом шаблоне, если планируете использовать этот экземпляр шаблона для отправки в один и тот же обменник большую часть или все время. 

В таких случаях вы можете использовать второй метод из приведенного выше списка. 

Следующий пример функционально эквивалентен предыдущему:
```java
amqpTemplate.setExchange("marketData.topic");

amqpTemplate.send(
	"quotes.nasdaq.FOO", 
	new Message("12.34".getBytes(), someProperties));
```



Если в шаблоне установлены оба свойства `exchange` и `routingKey`, вы можете использовать метод, который принимает только `Message`. Следующий пример показывает, как это сделать:
```java
amqpTemplate.setExchange("marketData.topic");
amqpTemplate.setRoutingKey("quotes.nasdaq.FOO");

amqpTemplate.send(new Message("12.34".getBytes(), someProperties));
```


> [!INFO] ВАЖНО
> Свойства **exchange** и **routingKey** в `RabbitTemplate` следует понимать так: **явные параметры метода всегда переопределяют значения шаблона**. 
> 
> При этом значения по умолчанию существуют всегда, даже если вы их не задаёте явно.

И для имени обменника, и для ключа маршрутизации значением по умолчанию является **пустая строка**, и это осознанный выбор. 
- Ключ маршрутизации нужен не всегда (например, у `fanout`-exchange), 
- а очередь может быть легально привязана к обменнику с пустым routing key.

Пустая строка для имени обменника имеет особый смысл: в AMQP она обозначает **обменник по умолчанию**. 

Все очереди автоматически привязаны к нему как к `direct`-exchange с ключом, равным имени очереди. 

Поэтому для простой point-to-point отправки достаточно указать имя очереди в качестве `routingKey`, используя обменник по умолчанию, без дополнительной конфигурации.

Следующий пример показывает, как это сделать:
```java
RabbitTemplate template = new RabbitTemplate(); // используем обменник по умолчанию без имени

template.send(
	"queue.helloWorld", 
	new Message("Hello World".getBytes(), someProperties));
```

В качестве альтернативы можно создать шаблон, который будет использоваться **для публикации преимущественно или исключительно в одну очередь.** 

Следующий пример показывает, как это сделать:
```java
RabbitTemplate template = new RabbitTemplate(); // используем обменник по умолчанию без имени

template.setRoutingKey("queue.helloWorld"); // но мы всегда будем отправлять в эту очередь

template.send(new Message("Hello World".getBytes(), someProperties));
```

# 1. API построителя сообщений (Message Builder API)
Начиная с версии 1.3, API построителя сообщений предоставляется классами `MessageBuilder` и `MessagePropertiesBuilder`. 

Эти методы обеспечивают удобный "плавный" (fluent) способ создания сообщения или свойств сообщения. 

В следующих примерах показан fluent API в действии:
```java
Message message = MessageBuilder.withBody("foo".getBytes())
    .setContentType(MessageProperties.CONTENT_TYPE_TEXT_PLAIN)
    .setMessageId("123")
    .setHeader("bar", "baz")
    .build();
```

```java
MessageProperties props = MessagePropertiesBuilder.newInstance()
    .setContentType(MessageProperties.CONTENT_TYPE_TEXT_PLAIN)
    .setMessageId("123")
    .setHeader("bar", "baz")
    .build();
    
Message message = MessageBuilder.withBody("foo".getBytes())
    .andProperties(props)
    .build();
```

Каждое из свойств, определенных в `MessageProperties`, может быть установлено. 
Другие методы включают 
- `setHeader(String key, String value)`, 
- `removeHeader(String key)`, 
- `removeHeaders()` 
- и `copyProperties(MessageProperties properties)`. 

Для каждого метода установки свойства существует вариант `set*IfAbsent()`. **В случаях, когда существует начальное значение по умолчанию, метод называется** `set*IfAbsentOrDefault()`.

## Статические методы для создания MessageBuilder
Предоставлено пять статических методов для создания начального построителя сообщений:
1. `public static MessageBuilder withBody(byte[] body)` — сообщение, созданное построителем, имеет тело, которое является прямой ссылкой на аргумент.
    
2. `public static MessageBuilder withClonedBody(byte[] body)` — сообщение, созданное построителем, имеет тело, которое является новым массивом, содержащим копию байтов аргумента.
    
3. `public static MessageBuilder withBody(byte[] body, int from, int to)` — сообщение, созданное построителем, имеет тело, которое является новым массивом, содержащим диапазон байтов из аргумента. Для получения более подробной информации см. `Arrays.copyOfRange()`.
    
4. `public static MessageBuilder fromMessage(Message message)` — сообщение, созданное построителем, имеет тело, которое является прямой ссылкой на тело аргумента. Свойства аргумента копируются в новый объект `MessageProperties`.
    
5. `public static MessageBuilder fromClonedMessage(Message message)` — сообщение, созданное построителем, имеет тело, которое является новым массивом, содержащим копию тела аргумента. Свойства аргумента копируются в новый объект `MessageProperties`.

## Статические методы для создания MessagePropertiesBuilder
Предоставлено три статических метода для создания экземпляра `MessagePropertiesBuilder`:

1. `public static MessagePropertiesBuilder newInstance()` — новый объект свойств сообщения инициализируется значениями по умолчанию.
    
2. `public static MessagePropertiesBuilder fromProperties(MessageProperties properties)` — построитель инициализируется предоставленным объектом свойств, и `build()` вернет именно этот объект.
    
3. `public static MessagePropertiesBuilder fromClonedProperties(MessageProperties properties)` — свойства аргумента копируются в новый объект `MessageProperties`.

## Перегрузка метода send с CorrelationData
В реализации `AmqpTemplate` с помощью `RabbitTemplate` каждый из методов `send()` **имеет перегруженную версию, принимающую дополнительный объект** ==CorrelationData==.

*Когда включены подтверждения* от издателя (publisher confirms), **этот объект возвращается в колбэке**, описанном в разделе про `AmqpTemplate`. 

Это ==позволяет отправителю связать подтверждение (ack или nack) с отправленным сообщением.==


Начиная с версии 1.6.7, был введен интерфейс `CorrelationAwareMessagePostProcessor`, *позволяющий изменять данные корреляции* **после конвертации сообщения**. 

Следующий пример показывает, как его использовать:
```java
Message postProcessMessage(Message message, Correlation correlation);
```

### Теперь Correlation устаревший (переместили функционал)
В версии 2.0 этот интерфейс считается устаревшим (deprecated). 

Метод был перемещен в `MessagePostProcessor` с реализацией по умолчанию, которая делегирует вызов `postProcessMessage(Message message)`.

### Новый интерфейс колбека
Также, начиная с версии 1.6.7, предоставлен ==новый интерфейс колбэка== `CorrelationDataPostProcessor`. 

Он вызывается после всех экземпляров `MessagePostProcessor` (предоставленных в методе `send()`, а также тех, что заданы через `setBeforePublishPostProcessors()`).
==ПЕРЕД САМОЙ ПУБЛИКАЦИЕЙ==

Он получает готовое `Message` и исходный `CorrelationData` (если он был передан в `send()`), и может:
- изменить существующий `CorrelationData`,
    
- либо полностью заменить его новым.

Таким образом, общий смысл такой:  
- `CorrelationData` — это ключ для связывания сообщения и подтверждения, 
- а `CorrelationDataPostProcessor` — рекомендуемый способ централизованно управлять этой корреляцией, не вмешиваясь в сам код отправки сообщений.

Следующий пример показывает, как использовать метод `postProcess`:
```java
CorrelationData postProcess(Message message, CorrelationData correlationData);
```

# 2. Publisher Returns
*Если у шаблона включено свойство* `mandatory`, **возвращённые сообщения передаются в колбэк**, описанный в разделе про `AmqpTemplate`.

Начиная с версии **1.4**, `RabbitTemplate` поддерживает свойство **`mandatoryExpression`** на основе SpEL. 
> Это выражение вычисляется для каждого отправляемого сообщения (сообщение является корневым объектом выражения) и должно возвращать логическое значение. В выражениях можно использовать ссылки на бины, например: `@myBean.isMandatory(#root)`

Механизм publisher returns также может использоваться самим `RabbitTemplate` во внутренних операциях `send` и `receive`. Подробнее см. раздел **Reply Timeout**.

# 3. **Batching (пакетная отправка)**
В версии **1.4.2** был добавлен класс **`BatchingRabbitTemplate`** — наследник `RabbitTemplate`, который переопределяет метод `send` и группирует сообщения в пакеты согласно стратегии `BatchingStrategy`

Сообщения отправляются в RabbitMQ только после формирования полного пакета.

## Интерфейс BatchingStrategy
Интерфейс `BatchingStrategy` определяет логику пакетирования:
- добавление сообщения в пакет,
    
- момент следующей отправки,
    
- выпуск готовых пакетов.

==Пакетируемые данные хранятся в памяти==, поэтому при сбое системы неотправленные сообщения могут быть потеряны.

## Реализация - SimpleBatchingStrategy
В комплекте поставляется **`SimpleBatchingStrategy`**, **которая поддерживает отправку сообщений только в один exchange** или с одним routing key. 

Она настраивается тремя параметрами:
- `batchSize` — количество сообщений в пакете,
    
- `bufferLimit` — максимальный размер пакета в байтах (имеет приоритет над `batchSize`),
    
- `timeout` — время бездействия, после которого частичный пакет отправляется.

`SimpleBatchingStrategy` формирует пакет, **добавляя перед каждым сообщением 4-байтовый заголовок с длиной**. Эта информация передаётся получателю через свойство сообщения `springBatchFormat` со значением `lengthHeader4`.

Контейнеры слушателей по умолчанию автоматически распаковывают такие пакеты, ориентируясь на заголовок `springBatchFormat`. Если хотя бы одно сообщение из пакета отклоняется, отклоняется весь пакет целиком. Дополнительные детали см. в разделе **@RabbitListener with Batching**.
