# 1. Publishing Dead-letter Records
Publishing Dead-letter Records - Механизм отправки сообщений, которые **не удалось обработать после N попыток**, в специальный топик — **DLT (Dead Letter Topic)**. 

> [!NOTE] Класс `DeadLetterPublishingRecoverer`
> В Spring Kafka этим занимается класс **DeadLetterPublishingRecoverer** (DLPR). 
 
Он подключается в 
- **DefaultErrorHandler** или 
- **DefaultAfterRollbackProcessor**.

Пример настройки DeadLetterPublishingRecoverer в DefaultErrorHandler
```java
CommonErrorHandler handler =
    new DefaultErrorHandler(
	    new DeadLetterPublishingRecoverer(template), backOff);
```

## Базовые требования
🔶Класс требует наличия для отправки сообщения:
```java
KafkaTemplate<Object, Object>
```

🔶Можно настроить:
- `BiFunction<ConsumerRecord<?, ?>, Exception, TopicPartition>` — определяет в какой топик и партицию публиковать failed-запись.

---
## Поведение по умолчанию
### Назначение топика и раздела:
- По умолчанию запись отправляется в топик с именем `<originalTopic>-dlt`
    
- Сообщение направляется в тот же раздел, что и исходная запись
	
- Если указан **отрицательный номер раздела** - Kafka сама выбирает раздел
    
- Начиная с версии 2.2.4, используется `groupId` консьюмера для более точного выбора DLT

**Важное ограничение:** топик мертвых писей должен иметь как минимум столько же разделов, сколько исходный топик


**Особенности версии 2.2.4+:**
- `ListenerExecutionFailedException` расширяется свойством `groupId`
- Резолвер назначения может использовать эту информацию вместе с данными `ConsumerRecord`


### Пользовательский резолвер назначения
```java
DeadLetterPublishingRecoverer recoverer = 
	new DeadLetterPublishingRecoverer(
		template,
        (r, e) -> {
            if (e instanceof FooException) {
                return new TopicPartition(
	                r.topic() + ".Foo.failures", 
	                r.partition());
            }
            else {
                return new TopicPartition(
	                r.topic() + ".other.failures", 
	                r.partition());
            }
        });
        
CommonErrorHandler errorHandler = 
	new DefaultErrorHandler(
		recoverer, // DLT Resolver 
		new FixedBackOff(0L, 2L));
```

## Какие заголовки добавляются в DLT-записи
Запись, отправленная в топик мертвых писей, улучшается следующими заголовками:
### Основные заголовки исключений:
- **KafkaHeaders.DLT_EXCEPTION_FQCN**: Имя класса исключения (обычно `ListenerExecutionFailedException`, но может быть другим).
    
- **KafkaHeaders.DLT_EXCEPTION_CAUSE_FQCN**: Имя класса причины исключения, если присутствует (начиная с версии 2.8).
    
- **KafkaHeaders.DLT_EXCEPTION_STACKTRACE**: Трассировка стека исключения.
    
- **KafkaHeaders.DLT_EXCEPTION_MESSAGE**: Сообщение исключения.

### Заголовки для ошибок ключа:
- **KafkaHeaders.DLT_KEY_EXCEPTION_FQCN**: Имя класса исключения (только для ошибок десериализации ключа).
    
- **KafkaHeaders.DLT_KEY_EXCEPTION_STACKTRACE**: Трассировка стека исключения (только для ошибок десериализации ключа).
    
- **KafkaHeaders.DLT_KEY_EXCEPTION_MESSAGE**: Сообщение исключения (только для ошибок десериализации ключа).

### Метаданные исходной записи: 
- **KafkaHeaders.DLT_ORIGINAL_TOPIC**: Исходный топик.
    
- **KafkaHeaders.DLT_ORIGINAL_PARTITION**: Исходный раздел.
    
- **KafkaHeaders.DLT_ORIGINAL_OFFSET**: Исходное смещение.
    
- **KafkaHeaders.DLT_ORIGINAL_TIMESTAMP**: Исходная временная метка.
    
- **KafkaHeaders.DLT_ORIGINAL_TIMESTAMP_TYPE**: Исходный тип временной метки.
    
- **KafkaHeaders.DLT_ORIGINAL_CONSUMER_GROUP**: Исходная группа потребителей, которая не смогла обработать запись (начиная с версии 2.8).

То есть в результирующее сообщение помещается **вся контекстная информация о том, где и почему произошла ошибка**.

## Способы добавления дополнительных заголовков
### 1. Переопределить `createProducerRecord()`
Если нужно полный контроль над тем, какие данные попадут в сообщение (есть доступ ко _всем уже собранным стандартным заголовкам_).

Создать подкласс `DeadLetterPublishingRecoverer` и переопределить метод, где можно модифицировать ProducerRecord.

#### Где настраивается
В собственном наследнике:
```java
class MyRecoverer extends DeadLetterPublishingRecoverer {
    @Override
    protected ProducerRecord<Object, Object> createProducerRecord(...) {
        ProducerRecord pr = super.createProducerRecord(...);
        
        pr.headers()
			.add("my", "x".getBytes());
        return pr;
    }
}
```

### 2. Установить `headersFunction()`
Если нужно быстро добавить заголовки, без расширения класса.

BiFunction `(ConsumerRecord, Exception) → Headers`, заголовки будут добавлены к итоговой записи.

#### Где настраивается
```java
recoverer.setHeadersFunction((rec, ex) -> {
    Headers h = new RecordHeaders();
    h.add("error", ex.getClass().getSimpleName().getBytes());
    return h;
});
```

## Поддержка ErrorHandlingDeserializer
Ситуация: нужно полное сообщение, которое «сломало» десериализацию.


**ErrorHandlingDeserializer** - Восстанавливает `record.value()` до исходных байтов, которые не удалось десериализовать (раньше был `null`).

Ранее `value()` был `null`, и пользовательский код должен был декодировать `DeserializationException` из заголовков сообщения.

## Использование нескольких `KafkaTemplate`
Чтобы публиковать разные типы (например String и byte\[]) в зависимости от значения.

В recoverer можно передать **набор шаблонов по типам значений.**
```java
//Бин, который сконфигурирует DLT Recover и вернет его
@Bean
public DeadLetterPublishingRecoverer publisher(
	KafkaTemplate<?, ?> stringTemplate,
    KafkaTemplate<?, ?> bytesTemplate) 
{
    Map<Class<?>, KafkaOperations<?, ?>> templates = 
	    new LinkedHashMap<>();
	    
    templates.put(String.class, stringTemplate);
    templates.put(byte[].class, bytesTemplate);
    return new DeadLetterPublishingRecoverer(templates);
}
```

- Издатель использует ключи карты для поиска шаблона, подходящего для `value()`, который будет опубликован. 
- Рекомендуется `LinkedHashMap`, чтобы ключи проверялись по порядку.
- При публикации значений `null` и наличии нескольких шаблонов, восстановитель будет искать шаблон для класса `Void`; 
- если ни один не присутствует, будет использован первый шаблон из `values().iterator()`.

## Поведение при ошибке публикации
### Поведение при ошибках публикации
#### setFailIfSendResultIsError(true)
Бросает ошибку, если публикация в DLT не удалась.

#### setWaitForSendResultTimeout(x)
Таймаут ожидания результата отправки.

### Поведение при ошибке recoverer’а
#### resetStateOnRecoveryFailure
**true по умолчанию** — BackOff сбрасывается при неудачной попытке восстановления.

#### resetStateOnExceptionChange
**false по умолчанию** — при изменении типа исключения можно перезапустить цикл retry.

## Повторная публикация в исходный топик вместо DLT
Если сообщения взаимозависимы, иногда лучше отправлять ошибочные сообщения в конец оригинального топика:
```java
return retries > 5 ? topic-dlt : topic;
```

```java
import org.apache.kafka.common.header.Header
@Bean
public ErrorHandler eh(KafkaOperations<String, String> template) {
    return new DefaultErrorHandler(
	    new DeadLetterPublishingRecoverer(
		    template,
            (rec, ex) -> {
                Header retries = rec
	                .headers()
	                .lastHeader("retries");
	                
                if (retries == null) {
                    retries = new RecordHeader(
	                    "retries", new byte[] { 1 });
                    rec.headers().add(retries);
                }
                else {
                    retries.value()[0]++;
                }
                return retries.value()[0] > 5
                        ? new TopicPartition(
	                        "topic-dlt", rec.partition())
                        : new TopicPartition(
	                        "topic", rec.partition());
            }), new FixedBackOff(0L, 0L));
}
```

# 2. Управление заголовками DLT сообщений (Managing Dead Letter Record Headers)

При публикации ошибочной записи в **DLT (Dead Letter Topic)** класс `DeadLetterPublishingRecoverer` **добавляет к записи специальные заголовки:**
- информацию об исключении,
    
- стек-трейс,
    
- оригинальные заголовки,
    
- историю маршрута. 

При повторном перепубликовывании (например, запись уже из DLT тоже упала) **заголовки могут** **накапливаться**, увеличивая размер сообщения вплоть до `RecordTooLargeException`.

Этот раздел описывает **как контролировать добавление, удаление и модификацию таких заголовков**.

## Основные свойства DeadLetterPublishingRecoverer
### appendOriginalHeaders (default = true)
Добавляет _все оригинальные заголовки_ (с именем `ORIGINAL`) в новую запись.
#### Когда отключать (`false`)
— Если заголовки становятся слишком большими.  
— Если не нужно хранить историю прохождения.

### stripPreviousExceptionHeaders (default = true с версии 2.8)
Удаляет все старые заголовки об исключениях (с именем `EXCEPTION`) перед записью новых.

Чтобы:
- не копить огромные стеки,
- не допустить превышения размера записи,
- не мешать дальнейшей обработке.

---
## Особенности работы с заголовками в Apache Kafka
Apache Kafka поддерживает несколько заголовков с тем же именем; 
- чтобы получить "последнее" значение, вы можете использовать `headers.lastHeader(headerName)`; 
- чтобы получить итератор по нескольким заголовкам, используйте `headers.headers(headerName).iterator()`.

---
## Расширенный контроль заголовков (версия 2.8.4+)

### Управление стандартными заголовками:
Появилась возможность контролировать, какие стандартные заголовки добавляются в выходную запись
    
**Используется перечисление** `HeadersToAdd` с общими именами 10 стандартных заголовков
    
Это абстракция - фактические имена заголовков устанавливаются методом `getHeaderNames()`, который можно переопределить

### Исключение заголовков из списка
Чтобы исключить заголовки, используйте `excludeHeaders()`. 

Например, чтобы убрать трассировку стека:
```java
DeadLetterPublishingRecoverer recoverer = 
	new DeadLetterPublishingRecoverer(template);
	
recoverer.excludeHeaders(HeaderNames.HeadersToAdd.EX_STACKTRACE);
```

## Кастомизация заголовков исключений
### Использование ExceptionHeadersCreator:
Для полной кастомизации заголовков исключений используйте `setExceptionHeadersCreator()` — это отключает все стандартные EXCEPTION-заголовки:
```java
DeadLetterPublishingRecoverer recoverer = 
	new DeadLetterPublishingRecoverer(template);
	
recoverer.setExceptionHeadersCreator(
	(kafkaHeaders, exception, isKey, headerNames) -> {
	    kafkaHeaders.add(new RecordHeader(..., ...));
});
```
**Особенности подхода:**
- Полностью *отключает все стандартные заголовки* исключений
    
- Предоставляет полный контроль над созданием заголовков
    
- Параметры включают заголовки Kafka, исключение, флаг ключа и имена заголовков

# 3.Реализация ExponentialBackOffWithMaxRetries
Spring Framework предоставляет ряд реализаций `BackOff`. 

По умолчанию `ExponentialBackOff` будет повторять бесконечно; 
чтобы сдаться после некоторого количества попыток повторения, требуется вычисление `maxElapsedTime`. 

Начиная с версии 2.7.3, Spring for Apache Kafka предоставляет `ExponentialBackOffWithMaxRetries`, который является подклассом, и который получает свойство `maxRetries` и автоматически вычисляет `maxElapsedTime`, что немного удобнее.

```java
@Bean
DefaultErrorHandler handler() {
    ExponentialBackOffWithMaxRetries bo = 
	    new ExponentialBackOffWithMaxRetries(6);
	    
    bo.setInitialInterval(1_000L);
    bo.setMultiplier(2.0);
    bo.setMaxInterval(10_000L);
    return new DefaultErrorHandler(myRecoverer, bo);
}
```

Это будет повторять через 1, 2, 4, 8, 10, 10 секунд перед вызовом восстановителя.
