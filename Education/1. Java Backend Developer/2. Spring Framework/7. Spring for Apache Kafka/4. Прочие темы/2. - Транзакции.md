Версия клиентской библиотеки 0.11.0.0 добавила поддержку транзакций. 
## 4 способа использования транзакций
Spring for Apache Kafka добавляет **поддержку следующими способами:**
### 🔶KafkaTransactionManager
Используется с обычной поддержкой транзакций Spring (`@Transactional`, `TransactionTemplate` и др.)

Работает так же, как обычные Spring-транзакции
```java
@Transactional
public void process() {
    kafkaTemplate.send(...);
}
```
Все отправки в рамках метода выполнятся в **одной Kafka транзакции**

---
### 🔶 Transactional KafkaMessageListenerContainer
Это **listener-контейнер**, который может:
- читать сообщения
- обрабатывать их
    
- **в той же транзакции отправлять компенсирующие сообщения**
    
- коммитить офсет **как часть транзакции**


---
### 🔶 Локальные транзакции с KafkaTemplate
Можно вручную:
```java
template.executeInTransaction(t -> {
    t.send(...);
    t.send(...);
});
```

Это транзакция **только продюсера**, без участия Spring @Transactional.

---
### 🔶 Синхронизация транзакций с другими менеджерами транзакций
Например, если у вас:
- PostgreSQL
- Kafka

Вы можете сделать **одну распределённую транзакцию**, где Kafka участвует через:
```java
KafkaTransactionManager
DataSourceTransactionManager
ChainedTransactionManager
```

## 1) Включение транзакций 
Транзакции включаются путем предоставления **DefaultKafkaProducerFactory** с префиксом `transactionIdPrefix`.

```java
factory.setTransactionIdPrefix("my-tx-");
```

В этом случае вместо управления одним общим `Producer`, фабрика поддерживает кэш транзакционных продюсеров. 

Когда пользователь вызывает `close()` на продюсере, он возвращается в кэш для повторного использования вместо фактического закрытия.

### Кеш транзакционных продюсеров
Когда KafkaTemplate начинает транзакцию:
1. нужен продюсер
    
2. берется продюсер из кэша (или создаётся новый)
    
3. ему назначается `transactional.id`:
```java
transactional.id = transactionIdPrefix + n
```

4. после выполнения транзакции, когда вызывается `close()`:
    - он **НЕ закрывается реально**
    - возвращается в кэш  
        → для будущих транзакций

#### Почему так?
Потому что транзакционный продюсер — **тяжёлый объект**.  
Его нельзя создавать каждый раз заново → это очень дорого для Kafka.

#### Как выбирается transactional.id?
Пример при `transactionIdPrefix="tx-"`:
```java
tx-0
tx-1
tx-2
...
```
Каждый новый продюсер получает свой ID.


❗ Для **нескольких инстансов приложения**:
> transactionIdPrefix должен быть уникальным  
> иначе Kafka посчитает продюсера "зомби" (fenced)

Например:
```java
tx-app1-
tx-app2-
```

### Spring Boot автоконфигурация
Если в `application.yml` указать:
```yaml
spring.kafka.producer.transaction-id-prefix: tx-
```

То Boot:
- создаёт `KafkaTransactionManager`
	
- подключает его в Listener Container
    
- включает поддержку транзакций автоматически

### Параметр maxAge (Spring Kafka 2.5.8+)
Начиная с версии 2.5.8, вы теперь можете настраивать свойство `maxAge` на фабрике продюсеров. 

Это полезно при использовании **транзакционных продюсеров, которые могут ПРОСТАИВАЮТ БОЛЬШЕ**, чем `transactional.id.expiration.ms` брокера.

С текущими `kafka-clients` это может вызвать `ProducerFencedException` без перебалансировки. 
Установив `maxAge` меньше, чем `transactional.id.expiration.ms`, фабрика обновит продюсер, если он превысил свой максимальный возраст.

# 1) KafkaTransactionManager

> [!NOTE] `KafkaTransactionManager` 
> это **реализация PlatformTransactionManager** от Spring Framework. 
> 

В его **конструктор передается ссылка на фабрику продюсеров.** 
==Если вы предоставляете пользовательскую фабрику== продюсеров, ==она должна поддерживать транзакции==. 

> (См. `ProducerFactory.transactionCapable()`.)

Вы можете использовать `KafkaTransactionManager` с обычной поддержкой транзакций Spring 
(`@Transactional`, `TransactionTemplate` и другие). 

Если транзакция активна, любые операции `KafkaTemplate`, выполненные в рамках транзакции, используют `Producer` транзакции. Менеджер фиксирует или откатывает транзакцию в зависимости от успеха или неудачи.

Вы должны настроить `KafkaTemplate` для использования той же `ProducerFactory`, что и менеджер транзакций.

## 1) Принцип работы
Когда вы вызываете метод, отмеченный `@Transactional`, Spring начинает транзакцию.  

Если транзакция активна, то **все вызовы `kafkaTemplate.send()` внутри неё используют один и тот же транзакционный продюсер**.

KafkaTransactionManager:
1. **открывает транзакцию Kafka**
    
2. привязывает транзакцию к текущему Spring-треду
    
3. при завершении:
    - ✅если ошибок нет → делает `producer.commitTransaction()`
    - ❌если была ошибка → `producer.abortTransaction()`


Это полностью аналогично работе с транзакциями для БД.

# 2) Синхронизация транзакций

Сценарий:
- Вы хотите **в одном бизнес-процессе**
    - отправить сообщения в Kafka
    - **и** обновить базу данных
        
- и хотите, чтобы **всё было атомарно**:
    - если упадёт БД → Kafka не отправит
    - если упадёт Kafka → БД не обновится
        
    - или наоборот — в зависимости от выбранной стратегии

==Это делается с помощью обычной Spring-транзакции== + **авто-синхронизации KafkaTemplate.**

## 1) Механика: как работает синхронизация?
Если:
- KafkaTemplate **использует транзакционный ProducerFactory** (то есть есть `transactionIdPrefix`)
- Метод помечен `@Transactional` (обычная Spring-транзакция с JDBC)
    
- ==KafkaTemplate делает== `send()` ==внутри активной транзакции==

❗❗Тогда Spring **автоматически включает Kafka в эту же транзакцию**❗❗

Причём:
- **сначала** коммитится транзакция БД
- **потом** — коммит Kafka

Это называется ==DB-first commit chain.==

```java
@Transactional
public void process(List<Thing> things) {
    things.forEach(
	    thing -> this.kafkaTemplate.send("topic", thing));
    updateDb(things);
}
```
Что реально происходит:
1. @Transactional → Spring открывает JDBC-транзакцию
    
2. KafkaTemplate понимает, что транзакция активна → начинает Kafka-транзакцию
    
3. Вы вызываете `send()` → записи буферизуются в транзакционном Kafka-producer
    
4. Вы обновляете БД
    
5. Метод завершается
    
6. Spring делает:
```sql
1) commit DB
2) commit Kafka
```

Если коммит Kafka **сломается**, ==вылетит исключение наружу==  
— это поведение появилось в Spring Kafka 2.5.17+.
При помощи этого исключения, можно написать код, который удалит данные с БД

> До Spring Kafka 2.5.17 ошибка просто логировалась

## 2) Смена порядка транзакций
По умолчанию:
```sql
DB commit → Kafka commit
```

Но иногда нужно наоборот:
```sql
Kafka commit → DB commit
```

Например:
- вы хотите быть уверены, что сообщение _точно отправится_,  
    и только потом обновлять БД

Тогда используется **вложенная транзакция**:

```java
@Service
public class Service {

    @Transactional("dbTxManager")
    public void outerMethod() {
        // сначала Kafka
        innerKafkaMethod();
        // потом DB
        updateDb();
    }

    @Transactional("kafkaTransactionManager")
    public void innerKafkaMethod() {
        kafkaTemplate.send("topic", "value");
    }
}
```

Порядок:
1. `innerKafkaMethod()` начинает Kafka-транзакцию
    
2. Она успешно коммитится
    
3. Возвращаемся в DB-транзакцию
    
4. Коммитим БД
    

Это — ==Kafka-first схема.==

# 3) Использование транзакций, инициированных потребителем
Consumer-initiated transactions (транзакции, инициированные **контейнером слушателя**) нужны **когда нужно не только прочитать данные, но и что-то отправить в Kafka** и при этом обеспечить **exactly-once semantics**.

Пример:
- Слушатель получает сообщение из `topic-A`
    
- Обрабатывает его
    
- Отправляет результат в `topic-B`
    
- После этого фиксирует offset, чтобы не было повторного чтения

Если мы **не используем транзакцию**, может случиться:
```java
Consumer прочитал сообщение
Listener отправил в topic-B
Произошёл сбой до commit offset
```
→ offset не закоммичен → consumer повторно прочитает сообщение  
→ отправка в topic-B произойдёт ещё раз → дублирование

**Цель транзакции:** сделать **одну атомарную операцию**:
```java
offset commit + producer.send = атомарно
```

То есть **либо всё прошло, либо ничего** — exactly-once.

## 1.Как это работает в контейнере
1. Контейнер listener-а вызывает **producer.beginTransaction()** (KafkaTemplate, связанный с контейнером)
    
2. Вы выполняете `@KafkaListener` метод, где:
    - читаете данные
        
    - делаете какие-то действия
        
    - отправляете записи в другие топики через KafkaTemplate
        
3. Контейнер в конце автоматически делает:
```java
producer.sendOffsetsToTransaction(offsets, consumerGroupId)
producer.commitTransaction()
```
- offsets + producer.records **фиксируются одной атомарной транзакцией**
    
- если что-то упало → транзакция откатывается → сообщения будут обработаны повторно, без дубликатов в output topic

# 4. Локальные транзакции KafkaTemplate (KafkaTemplate Local Transactions)
Вы можете использовать `KafkaTemplate` для выполнения серии операций в рамках локальной транзакции. 

Следующий пример показывает, как это сделать:
```java
boolean result = template.executeInTransaction(t -> {
    t.sendDefault("thing1", "thing2"); //Отправка в топик
    t.sendDefault("cat", "hat"); //Отправка в топик
    return true;
});
```

Аргумент в callback - это сам шаблон (this). Если callback завершается нормально, транзакция фиксируется. Если выбрасывается исключение, транзакция откатывается.

Если есть транзакция `KafkaTransactionManager` (или синхронизированная) в процессе, она не используется. **Вместо этого используется новая "вложенная" транзакция**.

## 1. Как это работает под капотом
1. KafkaTemplate использует **ProducerFactory**, который создаёт транзакционные продюсеры.
    
2. Когда вызывается `executeInTransaction(t -> { ... })`:
    - Создаётся **новый транзакционный продюсер** (или берётся из пула, если транзакции поддерживаются)
        
    - Вызывается `producer.beginTransaction()`
        
    - Выполняются **все send операции внутри callback**
        
    - Если callback завершился нормально → `producer.commitTransaction()`
        
    - Если callback выбросил исключение → `producer.abortTransaction()`
        
3. После завершения транзакции продюсер возвращается в пул для повторного использования (не закрывается).

# 5. Фиксированный суффикс TransactionId (TransactionIdSuffix Fixed)
Kafka **требует уникальный `transactional.id`** для каждого транзакционного продюсера.
	
В Spring Kafka **DefaultKafkaProducerFactory** может создавать несколько продюсеров для параллельной работы или повторного использования.
	
Чтобы обеспечить уникальность `transactional.id`, добавляют **суффикс**, например:
```java
transactional.id = my.txid.0
transactional.id = my.txid.1
transactional.id = my.txid.2
...
```

Раньше суффиксы генерировались **на лету**, увеличивая счётчик.

## 1. Новая стратегия в версии 3.2
**Интерфейс `TransactionIdSuffixStrategy`** позволяет контролировать, как генерировать суффикс:

- **DefaultTransactionIdSuffixStrategy(maxCache)** — стандартная реализация
    - Если `maxCache > 0` → продюсеры **повторно используют transactional.id в диапазоне `[0, maxCache-1]`**
        
    - Если все transactional.id заняты → выбрасывается `NoProducerAvailableException`
        
    - Если `maxCache = 0` → суффиксы генерируются на лету (увеличивающийся счётчик)


Пример:
```java
TransactionIdSuffixStrategy ss = new 
	DefaultTransactionIdSuffixStrategy(5);
	
pf.setTransactionIdSuffixStrategy(ss);
```
- Суффиксы: `my.txid.0`, `my.txid.1`, `my.txid.2`, `my.txid.3`, `my.txid.4`
    
- После этого система **повторно использует свободные transactional.id** из кэша, вместо создания бесконечного числа новых суффиксов.

# 6. Транзакционная и нетранзакционная публикация через KafkaTemplate
По умолчанию, если **KafkaTemplate настроен с транзакционной фабрикой продюсеров**, то:

- **Все отправки должны происходить внутри транзакции**
    - Запущенной через:
        - `TransactionTemplate` (с настроенным KafkaTransactionManager)
	        
        - аннотацию `@Transactional`
            
        - `KafkaTemplate.executeInTransaction(...)`
            
        - контейнер слушателя с `KafkaTransactionManager`
            
- Если попытаться **отправить сообщение вне транзакции**, шаблон выбросит `IllegalStateException`.

## 1. Новое свойство `allowNonTransactional` (с версии 2.4.3)

Можно включить `allowNonTransactional = true`
Тогда KafkaTemplate **разрешит выполнять отправку вне транзакции**

Используется отдельный **non-transactional продюсер**, который создаётся через `ProducerFactory.createNonTransactionalProducer()`

Этот продюсер **также кэшируется** или привязывается к потоку для повторного использования
```java
KafkaTemplate<String, String> template = 
	new KafkaTemplate<>(producerFactory);
	
template.setAllowNonTransactional(true);
template.send("topic", "message"); // сработает, даже если транзакция не активна
```

# 7.Транзакции со слушателями пачек (Batch Listeners)
**Слушатель пачек (batchListener=true)** получает сразу несколько записей (`List<ConsumerRecord>`)
    
- Если слушатель выбрасывает исключение:
    - `AfterRollbackProcessor` пытается выполнить **повторную доставку**
        
    - **Проблема:** не ясно, какая запись в пачке не удалась → **вся пачка повторно доставляется**

---

## 1.BatchToRecordAdapter (с версии 2.4.2)
Решение проблемы: обрабатывать **ошибки по одной записи**, даже если используется batch listener.
- `BatchToRecordAdapter` разбивает пачку на отдельные записи
    
- Listener вызывается **по одной записи за раз**, но всё ещё поддерживается batch listener
    
- Можно настроить **ConsumerRecordRecoverer** (например, `DeadLetterPublishingRecoverer`) для обработки ошибок конкретных записей
    

Пример из документации:
```java
public static class TestListener {

    final List<String> values = new ArrayList<>();

    @KafkaListener(id = "batchRecordAdapter", topics = "test")
    public void listen(String data) {
        values.add(data);
        if ("bar".equals(data)) {
            throw new RuntimeException("reject partial");
        }
    }

}

@Configuration
@EnableKafka
public static class Config {
    ConsumerRecord<?, ?> failed;

    @Bean
    public TestListener test() {
        return new TestListener();
    }

    @Bean
    public ConsumerFactory<?, ?> consumerFactory() {
        return mock(ConsumerFactory.class);
    }

    @Bean
    public 
	    ConcurrentKafkaListenerContainerFactory<String, String> 
	    kafkaListenerContainerFactory() 
	{
        ConcurrentKafkaListenerContainerFactory factory = 
	        new ConcurrentKafkaListenerContainerFactory();
	        
        factory.setConsumerFactory(consumerFactory());
        factory.setBatchListener(true);
        
        factory.setBatchToRecordAdapter(
	        new DefaultBatchToRecordAdapter<>((record, ex) ->  {
	            this.failed = record;
	        })
	    );
        return factory;
    }
}
```
- `DefaultBatchToRecordAdapter` вызывает callback для **неудачной записи**
    
- Остальные записи продолжают обрабатываться нормально
    
- Можно сохранять неудачные записи или отправлять их в **Dead Letter Topic**

### Как это работает на практике
1. Поступает пачка записей `[A, B, C, D]`
	
2. Batch listener с обычным AfterRollbackProcessor: если B упала → вся пачка `[A, B, C, D]` повторно доставляется
    
3. Batch listener с `BatchToRecordAdapter`:
    - Listener вызывается для каждой записи отдельно
        
    - B упала → `ConsumerRecordRecoverer` обрабатывает B
        
    - A, C, D продолжают обрабатываться без повторной доставки
