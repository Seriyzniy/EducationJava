Настройка Consumer на транзакционность используется вместе с настройкой транзакционного Producer - ДЛЯ ОБЕСПЕЧЕНИЯ семантики доставки `Exactly-once`

Но, одной транзакционностью не обойтись, для обеспечения `exactly-once` доставки необходимо
- Идемпотентный продюсер
- Идемпотентный консюмер
- Настроенная работа с транзакциями в обоих

# 1. Настройка транзакционности Consumer
Чтобы Consumer работал в паре с транзакционным Producer, необходимо всего лишь установить параметр ИЗОЛЯЦИИ ТРАНЗАКЦИЙ

По сути отвечающий на вопрос: какие сообщения можно читать?
- "закомиченные Producer" - `read_committed`
- "незакомиченные Producer" - `read_uncommitted`

В случае использования транзакций, Consumer настраивается на чтение ТОЛЬКО ЗАКОМИЧЕННЫХ сообщений
```java
isolation.level=read_committed
```

Это делаяется для того, чтобы при отправке сообщений продюсером, когда он успел отрпавиь в один топик, но не успел отправить в другой топик (упал), то Consumer не читал сообщения, из топика
(см [[7.2 -- Idempotence Consumer и обеспечение Exactly-once]])

Для этого, в конфигурационные параметры добавляется этот параметр
```java
props.put(  
	ConsumerConfig.ISOLATION_LEVEL_CONFIG,  
	"read_committed"  
);  
```

Во всем остальном, это обычный Consumer
(см также [[1. - Транзакционный и Идемпотентный Producer]])

# 2. Обеспечение идемпотентности 
Для обеспечения идемпотентности (гарантии того, что не будут обрабатываться дубликаты сообщений) используется одна хитрость

Чтобы сделать обработку сообщений идемпотентной, необходимо использование БД, в которую будет записываться уникальный ID сообщения ПОСЛЕ полной обработки.

А при получении сообщения, проверять БД на наличие ID. 
- Если ID не найден - сообщение еще не обрабатывали
- Если ID найден - пропускаем обработку и фиксирует offset
