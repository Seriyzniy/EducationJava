На стороне получателя могут возникнуть ошибки, которые можно разделить на следующие группы:

## Группа 1 — Ошибки ДО вызова Listener (до уровня твоего кода)
То есть Consumer **даже не успел попасть в @KafkaListener**.

Это:
### 1.1. **Ошибка десериализации ключа (Key Deserialization Error)**
Ключ невозможно десериализовать с KeyDeserializer.

### 1.2. **Ошибка десериализации значения (Value Deserialization Error)**
Обычно JSON parse error, Avro schema mismatch, EOFException.

### 1.3. **Ошибка в Header Deserialization**
Если маппер заголовков не может преобразовать header → объект.

### 1.4. **Ошибка на уровне Consumer API**
Например:
- commit провалился
- heartbeat session timeout
- потеря партиции
- разрыв соединения
- неверный offset (OffsetOutOfRangeException)

### ПОВЕДЕНИЕ ПО УМОЛЧАНИЮ
❌ _По умолчанию Listener НЕ вызывается._
Spring Kafka получает **ConsumerRecord<byte[], byte[]>**,  
пытается вызвать:
```java
valueDeserializer.deserialize(topic, data);
```

Если exception → контейнер выбивает **SerializationException**,  
**consumer-поток падает и контейнер перезапускается** в цикле:
```java
poll() → SerializationException → container restart → poll() → SerializationException → ...
```

⚠️Это очень опасно — ==consumer зависает в бесконечном цикле==.
Именно поэтому Spring создал `ErrorHandlingDeserializer`.


## **Группа 2 — Ошибки ВНУТРИ Listener (уровень твоего кода)**

### 2.1. **Ошибки бизнес-логики**
Пример: NPE, IllegalStateException, RuntimeException.

### 2.2. **Ошибки вызова внешних сервисов**
Например RestTemplate timeout → HttpServerErrorException.

### 2.3. **Ошибки в batch listener’ах**
❗❗ Исключение в одной записи ломает entire batch.

### ПОВЕДЕНИЕ ПО УМОЛЧАНИЮ
Если Listener бросил исключение, по умолчанию работает:
`DefaultErrorHandler` 
(до Spring Kafka 2.8 – SeekToCurrentErrorHandler)

Что происходит при исключении:
- контейнер делает **seekToCurrent** → перематывает offset назад
    
- сообщение будет прочитано заново
    
- бесконечно

То есть consumer зацикливается на проблемном сообщении:
```java
msg → exception → retry → exception → retry → ...
```
Это дефолт! И это правильно, потому что Kafka не теряет данные.

# 1. Обработка ошибок десериализации 
Это ошибки **до того**, как сообщение попало в ваш `@KafkaListener`.

Например:
- JSON невалидный
- Неверная схема
- Повреждённые байты
- Невозможно создать объект / нет конструктора


## 1. **ErrorHandlingDeserializer**
Spring Kafka предлагает встроенную стратегию, которая:
🔹 Не падает  
🔹 Пропускает плохие сообщения  
🔹 Передаёт их вашему Listener-у в виде `ErrorHandlingDeserializer`

Этот десериализатор - **ОБЕРТКА НАД обычным десериализатором**

Он помещает информацию об ошибке в **Headers**:
```java
header: "springDeserializerExceptionValue"
header: "springDeserializerExceptionKey"
```
Listener ПОЛУЧАЕТ ConsumerRecord, но `value == null`.


Пример конфигурации:
```java
Map<String, Object> props = new HashMap<>();
props.put(
	ConsumerConfig.KEY_DESERIALIZER_CLASS_CONFIG,
    ErrorHandlingDeserializer.class);
props.put(
	ConsumerConfig.VALUE_DESERIALIZER_CLASS_CONFIG,
    ErrorHandlingDeserializer.class);

// реальные десериализаторы
props.put(
	ErrorHandlingDeserializer2.KEY_DESERIALIZER_CLASS,
    StringDeserializer.class);
props.put(
	ErrorHandlingDeserializer2.VALUE_DESERIALIZER_CLASS,
    JsonDeserializer.class);

props.put(JsonDeserializer.TRUSTED_PACKAGES, "*");
```

Теперь Listener работает:
```java
@KafkaListener(...)
public void listen(User user, @Headers MessageHeaders headers) {

    if (user == null) {
        // ошибка десериализации
        Exception e = 
	        (Exception) headers.get("springDeserializerExceptionValue");
        log.error("Десериализация не удалась: ", e);
        return;
    }

    // нормальная обработка
}
```

## 2. DefaultErrorHandler
Можно сказать: _если ошибка десериализации — отправить в DLT_.


> [!NOTE] Важный факт:
> **БЕЗ ErrorHandlingDeserializer у вас НЕТ способа отправить такое сообщение в DLT**, 
> потому что Listener не вызывается →
> → ErrorHandler не вызывается 
> → recoverer не работает.
>

# 2. ГЛОБАЛЬНЫЙ обработчик ошибок
Для обработки ошибок внутри listener можно использовать `DefaultErrorHandler` - реализация `CommonErrorHandler`

## 📌 Поведение DefaultErrorHandler
Он делает retry в рамках consumer thread:
```java
exception → retry → exception → retry → ...  
пока не закончится backoff (если задан)
```

Пример настройки с использование DLT, и ретраем в 3 попытки:
```java
@Bean
public CommonErrorHandler errorHandler(
	KafkaTemplate<Object, Object> template) {

    DeadLetterPublishingRecoverer recoverer =
            new DeadLetterPublishingRecoverer(template);

    FixedBackOff backOff = new FixedBackOff(2000L, 3L); // 3 retry

    DefaultErrorHandler handler = 
	    new DefaultErrorHandler(
				recoverer, 
				backOff);

    // Чтобы IllegalArgumentException сразу шёл в DLT:
    handler.addNotRetryableExceptions(IllegalArgumentException.class);

    return handler;
}
```

Внутри него можно настроить:
- ✔ **Retry**
`FixedBackOff`, `ExponentialBackOff`

- ✔ **Skip / ignore**
`addNotRetryableExceptions()`

- ✔ **Dead-Letter Topic**
`DeadLetterPublishingRecoverer`
Создаёт:
```java
topic → topic.DLT
```

- ✔ **Пользовательский Recoverer**
```java
BiConsumer<ConsumerRecord<?,?>, Exception> myRecoverer = 
(record, ex) -> {
    log.error("Не удалось обработать запись {}", record, ex);
};
```

И затем, этот обработчик передать фабрике контейнеров
```java
factory.setCommonErrorHandler()
```

### Создание собственных исключений
Ты можешь выбросить свой exception, чтобы изменить поведение обработчика ошибок.

Для этого стоит создать собственные исключения
```java
public class FatalBusinessException extends RuntimeException {}
public class NonRetryableBusinessException extends RuntimeException {}
```

И добавить их как:
- NonRetryableException
- RetryableException
- FatalExceptions

в обработчик через методы:
```java
handler.addNotRetryableExceptions(
	NonRetryableBusinessException.class, 
	OtherException.class, ...);
	
handler.addRetryableExceptions(
	RetryableBusinessException.class, 
	OtherException.class, ...);

handler.addFatalExceptions(
	FatalBusinessException.class, 
	OtherException.class, ...);
```

А в коде обработки сообщения в catch выбрасывать созданные исключения в зависимости от типа ошибки.
- Если **ошибка не критична (Retryable)**, то можно **залогировать ошибку**, НО при этом **offset не коммитить**, в результате Consumer **еще раз попробует обработать** сообщение с Kafka **определённое количество раз**, указанное в `FixedBackOff`
	
- Если **ошибка критична (NonRetryable)**, и нет смылса пробовать еще раз обрабатывать, то вызовется `Recover` который был указал в DefauleErrorHandler

Пример
```java
if (externalServiceDown) {
    throw new RetryableBusinessException();
}

if (invalidInput) {
    throw new NonRetryableBusinessException();
}

if (stateCorrupted) {
    throw new FatalBusinessException();
}
```

# Дополнительная инфа

## 1. **Как обрабатываются offset-ы при ошибках**
Если **listener бросил exception** → offset **НЕ коммитится**.

Это значит:
- сообщение будет снова прочитано после перезапуска
    
- или после seek
    
- или после backoff
    

Если же recoverer (DLT) успешно обработал сообщение → offset коммитится.

Это важно, чтобы consumer не добился бесконечного цикла после DLT.


## 2. **Отличие ErrorHandlingDeserializer от ErrorHandler**

|Механизм|Когда работает|Что делает|
|---|---|---|
|ErrorHandlingDeserializer|ДО listener|предотвращает падение и передаёт ошибку в headers|
|DefaultErrorHandler|ПОСЛЕ listener|управляет retry, backoff, DLT|
