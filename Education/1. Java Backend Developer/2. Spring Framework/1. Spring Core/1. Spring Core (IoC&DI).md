Все в Spring вертится вокруг Spring Core, а именно вокруг ==**Inversion of Control**== (IoC - инверсия управления) и **==Dependency Injection==** (DI - внедрение зависимостей)

# Object Dependencies (зависимости объекта)

 **Зависимости** - объекты других классов, с которыми работает текущий объект и использует их функциональность

Dependency Injection - одна из реализаций IoC, посредством которой созданием объекта и компоновкой его зависимостей занимается другой объект (фреймворк)

Фреймворк определяет и внедряет зависимости объекта через:
- Параметры конструктора 
	
- Свойства объекта (set methods)
	
- Множество конфигураций:
	- factory methods
	- scope
	
- XML, аннотации или Java-код
	
- Автоматизация Autowiring

![[Img_Example1.png]]

Например: 
- Для UserService зависимость это UserRepository. 
	
- В свою очередь UserRepository использует класс ConnectionPool, для получения соединения для отправки запроса к базе данных.
	
- ConnectionPool содержит множество соединений, которые знают как отправлять запросы к БД.

Такие зависимости в проектах очень часто встречаются. Рассмотрим другой пример

![[Img_Example2.png]]

Опишем это в Java. для этого создадим проект на основе Maven:
![[Img_MavenStruct.png]]

Теперь создадим описанные выше классы и интерфейс:
```java
package ru.studyspring.myproject;

public interface GenreMusic {
	public String plaing();
}
```

```java
package ru.studyspring.myproject;

public class ClassicMusic implements GenreMusic{
	@Override
	public String plaing() {
		return "Classic Music";
	}
}
```

```java
package ru.studyspring.myproject;

public class RockMusic implements GenreMusic{
	@Override
	public String plaing() {
		return "AC-DC music";
	}
}
```
Опишем главный класс MusicPlayer:

```java
package ru.studyspring.myproject;

public class MusicPlayer {
	private final GenreMusic music = new ClassicMusic();
}
```

В результате получим, MusicPlayer УПРАВЛЯЕТ тем, как создается  класс реализующий GenreMusic, а это не правильно. 


> [!NOTE] Зависимости
> Эти **_зависимости_** должны предоставляться ИЗВНЕ, потому что ClasicMusic/RockMusic также может иметь зависимости

Получается MusicPlayer должен знать как создавать или получать ЗАВИСИМОСТИ для ClasiMusic, для того чтобы предоставить их внутреннему объекту.

Поэтому необходимо предоставить информацию о ClassicMusic нашему MusicPlayer в конструкторе.

```java
package ru.studyspring.myproject;

public class MusicPlayer {
	private final GenreMusic music;
	
	public MusicPlayer(GenreMusic gmusic) {
		this.music = gmusic;
	}
	public void PlaingMusic() {
		System.out.println("Playing: " + music.plaing());
	}
}
```
Теперь MusicPlayer не управляет как создавать ClassicMusic.

 Теперь каждый класс может сосредоточиться за выполнение своей задачи, но ни как не за создание зависимых объектов.

```java
package ru.studyspring.myproject;

public class ApplicationRunner {
	public static void main(String[] args) {
		GenreMusic cmusic = new ClassicMusic();
		GenreMusic rmusic = new RockMusic();
		MusicPlayer musicPlayer = new MusicPlayer(cmusic);
	}
}
```
Однако внедрение через конструктор не удобно, поскольку придется каждый раз создавать зависимые объекты.

# Inversion of Control (инверсия управления)

Однако, при усложнении приложения, придется ВРУЧНУЮ добавлять все новые зависимости и внедрять их в объекты. А количество таких зависимостей и объектов может достигать сотен и тысячи.

**Inversion of Control** - принцип программирования, при котором управление выполнением программы передается фреймворку, а не программисту.

Происходит потеря контроля над выполнением кода: фреймворк управляет кодом программиста, а не программист управляет фреймворком

Inversion of Control Spring подразумевает Dependency Injection, то есть фреймворк управляет внедрением зависимостей

Dependency Injection - одна из реализаций IoC, посредством которой созданием объекта и компоновкой его зависимостей занимается другой объект (фреймворк)

То есть, существует некоторый класс, представляющий ==контейнер объектов==. Он ==отвечает== за создание объектов и ==внедрением в них зависимостей==

```java
package ioc;  
  
public class ContainerIOC {  
    public <T> T get(Class<T> clazz){  
        return null;  
    }  
}
```

В данном случае мы создали класс, который будет возвращать объект необходимого нам класса уже с необходимыми зависимостями (реализацию писать не будем).

Теперь в ApplicationRunner создадим этот контейнер, и  через него будем просить его вернуть необходимый нам класс с зависимостями
```java
import ioc.ContainerIOC;  
import pool.ConnectionPool;  
import repository.UserRepository;  
import service.UserService;  
  
public class ApplicationRunner {  
  
    public static void main(String[] args) {  
        ContainerIOC containerIOC = new ContainerIOC();  
          
        ClassicMusic cmusic = containerIOC.get(ClassicMusic.class);  
        MusicPlayer musicPlayer = containerIOC.get(UserRepository.class);  
        //TODO использование musicPlayer  
    }  
}
```

> [!NOTE] Примечание
> Инверсия управления заключается в том, что теперь мы не управляем созданием объектов и зависимостями. Нам нужен просто функционал UserService, потому:

```Java
import ioc.ContainerIOC;  
import pool.ConnectionPool;  
import repository.UserRepository;  
import service.UserService;  
  
public class ApplicationRunner {  
  
    public static void main(String[] args) {  
        ContainerIOC containerIOC = new ContainerIOC();    
        MusicPlayer musicPlayer = containerIOC.get(MusicPlayer.class);  
        //TODO использование musicPlayer  
    }  
}
```
Мы просто просим у контейнера нужным нам класс, а об о всех зависимостях он позаботится сам.
## IoC Container (контекст приложения)

Описанный в предыдущем разделе контейнер существует в Spring, и называется он IoC Container.

**IoC Container** - объект, который занимается созданием (==***через Reflection API***==) других объектов и внедрением в них зависимостей (DI).
Контейнер представлен в виде ассоциативного массива (Map). Где ключом является String (имя класса), а значение объект ==Bean==

**_Bean_** - объект со всеми необходимыми зависимостями, который был создан с помощью IoC Container (Controller, Service, Repository)

==Приложение Spring выглядит следующим образом:==

![[Img_SpringApp.png]]

Ниже представлен код, показывающий работу контекста приложения
```java
MusicPlayer musicPlayer = appContext.getBean("musicPlayer", MusicPlayer.class);
```

В параметре метода getBean(), указывается второй параметр кроме `id`, поскольку ==по умолчанию контекст приведет тип бина к типу Object.==

## Spring приложение состоит из:
- IoC Container - вокруг него держится все приложение
	 Реализует один из двух интерфейсов:
	 - BeanFactory
	 - ==ApplicationContext ==(наиболее используемый, реализует интерфейс BeanFactory)
	
- Metadata - метаинформация 
	- Metadata представляет собой совокупность объектов (==Bean Definitions== ). которые говорят IoC Container как создавать Beans, конфигурировать и внедрять их зависимости
	- Bean - объект
	- Bean Definition - сам класс (чертеж)
	
- POJOs (Plain Old Java Object) - обычный объект, с набором свойств, конструкторов, get/set методов, не содержащие логики.
