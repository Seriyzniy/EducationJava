# 1. Создание фильтра `DelegatingFilterProxy`

Spring Security работает с помощью фильтров (см теорию 0.Введение)
#### Конфигурация XML в `web.xml`
```xml
<filter> 
   <filter-name>springSecurityFilterChain</filter-name> 
   <filter-class>org.springframework.web.filter.DelegatingFilterProxy</filter-class> 
</filter> 

<filter-mapping> 
   <filter-name>springSecurityFilterChain</filter-name> 
   <url-pattern>/*</url-pattern> // Проверяет все запросы начианая с /
</filter-mapping>
```
#### Конфигурация Java
В классе конфигурации сервера

==- Метод создания фильтра==
```java
private void registerSecurityFilter(ServletContext servletContext) {
    servletContext.addFilter("springSecurityFilterChain",
        new DelegatingFilterProxy("springSecurityFilterChain"))
        .addMappingForUrlPatterns(null, true, "/*");
}
```

> [!NOTE] Интерфейс `ServletContext`
> `ServletContext` — это объект, который контейнер сервлетов создает для каждого веб-приложения. Этот объект доступен во всей жизни приложения и предоставляет несколько методов для настройки его работы.
> 

 **Методы, доступные через `ServletContext`**:
1. **`addFilter`** — используется для регистрации фильтра.
2. **`getServletContext()`** — позволяет получить `ServletContext` в любом месте твоего кода, если он передан в метод.
3. **`getAttribute(String name)`** — позволяет получить атрибуты, связанные с контекстом приложения.
4. **`setAttribute(String name, Object object)`** — позволяет установить атрибуты в контекст.
5. **`getServletRegistration(String servletName)`** — позволяет получить информацию о зарегистрированном сервлете.

==- Метод, который говорит что делать при запуске сервера==
```java
@Override public void onStartup(ServletContext servletContext) throws ServletException 
{ 
	super.onStartup(servletContext); 
	registerSecurityFilter(servletContext);
}
```

# 2. Класс конфигуратор Spring Security
После добавления фильтра (бина `springSecurityFilterChain`) необходимо создать класс, который НАСТРОИТ фильтр (бин)

🟠Для конфигурации фильтра НЕОБХОДИМО определить:
✔Бин securityFilterChain - настраивает правила обработки каждого запроса с точки зрения безопасности
✔Бин passwordEncoder - необходим для определения правила сравнения паролей 

🟢Дополнительные бины
✔Бин userDetailsService - используется для получения пользователя из БД и формирование UserDetails (обычно реализуют собственный)

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {
	@Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(auth -> auth
                .anyRequest().authenticated()  // Все запросы требуют авторизации
            )
            .formLogin(login -> login
                .loginPage("/login")  // Страница логина
                .permitAll()  // Страница логина доступна всем
            )
            .logout(logout -> logout
                .permitAll()  // Выход доступен всем
            );
        return http.build();
    }
	@Bean
    public UserDetailsService userDetailsService() {
        UserDetails user = User.withDefaultPasswordEncoder()
            .username("user")
            .password("password")
            .roles("USER")
            .build();
        return new InMemoryUserDetailsManager(user);
    }
}
```

## **Метод `securityFilterChain(HttpSecurity http)`**

Этот метод конфигурирует ПРАВИЛА безопасности для HTTP-запросов. Он возвращает бин типа `SecurityFilterChain`, который управляет поведением безопасности на уровне HTTP.

- **`http`** — это объект типа `HttpSecurity`, который позволяет настроить различные аспекты безопасности веб-приложения, такие как правила доступа, аутентификацию и другие параметры.
##### Разбор настроек внутри метода `securityFilterChain`:

- 🟢`authorizeHttpRequests(auth -> auth.anyRequest().authenticated())`:
    - Это правило говорит, что **все запросы** (кроме тех, которые явно исключены) должны требовать аутентификации.
    - Это означает, что доступ к любому URL на сайте будет возможен только для аутентифицированных пользователей.

- 🟢`formLogin(login -> login.loginPage("/login").permitAll())`:
    - Конфигурация формы для аутентификации.
    - **`loginPage("/login")`** — указывает на кастомную страницу входа. Если пользователь не аутентифицирован и пытается зайти на защищенную страницу, его перенаправит на эту страницу для входа.
    - **`permitAll()`** — говорит, что страницу входа можно посещать всем пользователям, независимо от того, аутентифицированы они или нет.

- 🟢`logout(logout -> logout.permitAll())`:
    - Настройка выхода из системы.
    - **`permitAll()`** — говорит, что функциональность выхода доступна всем пользователям, независимо от их статуса аутентификации.

- 🟢`http.build()`:
    - Этот метод завершает настройку цепочки фильтров безопасности и создает объект `SecurityFilterChain`, который Spring будет использовать для защиты HTTP-запросов.

## **Метод `userDetailsService()`**

Этот метод ==создает бин, который управляет ДАННЫМИ пользователей==. Он возвращает объект типа `UserDetailsService`, который используется для поиска и аутентификации пользователей.

- **`UserDetailsService`** — это интерфейс, который отвечает за загрузку данных пользователя для аутентификации и авторизации в приложении.
##### Разбор кода в `userDetailsService`:

- **`User.withDefaultPasswordEncoder()`**:
    - Это статический метод, который создает пользователя с **паролем**, используя дефолтный **кодировщик пароля** (что не рекомендуется для продакшн-окружений, лучше использовать более безопасные методы для кодирования паролей, такие как `BCryptPasswordEncoder`).
    
- **`username("user").password("password").roles("USER").build()`**:
    - Создает пользователя с именем **`user`**, паролем **`password`** и ролью **`USER`**.
    
- **`new InMemoryUserDetailsManager(user)`**:
    - Создает объект `InMemoryUserDetailsManager`, который хранит пользователя в памяти. 
    - Это значит, что информация о пользователе хранится НЕ в базе данных, а непосредственно в приложении. Это хорошо для простых приложений или тестов, но для реальных приложений нужно использовать более стойкое хранилище (например, базу данных).

# 3. Добавление контроллера для переадресации `/login`
Поскольку фильтр будет перенаправлять НЕавторизированных  пользователей на адрес `/login`, необходимо создать контроллер

В классе конфигурации контекста
```java
@Override // Обязательно
public void addViewControllers(ViewControllerRegistry registry) {
    registry.addViewController("/login").setViewName("login");
}
```

С помощью объекта класса `ViewControllerRegistry` можно создать "простой" контроллер, без явного java-класса и аннотации `@Controller`
