
> [!NOTE] Контракт
> «Я умею ловить неперехваченные исключения в потоке».


Статический интерфейс `Thread.UncaughtExceptionHandler` реализуется классами, которым необходимо обрабатывать неперехваченные исключения. 

Он реализован классом `ThreadGroup`.

В интерфейсе `Thread.UncaughtExceptionHandler` объявлен только один метод:
```java
void uncaughtException(Thread t, Throwable e)
```

## Пример
```java
public class MyHandler implements Thread.UncaughtExceptionHandler {
    @Override
    public void uncaughtException(Thread t, Throwable e) {
        System.err.println("Поток " + t.getName() + " сдох с ошибкой: " + e);
    }
}

public class Test {
    public static void main(String[] args) {
        Thread t = new Thread(() -> { 
	        throw new RuntimeException("Boom!"); 
	    });
	    
        t.setUncaughtExceptionHandler(new MyHandler());
        t.start();
    }
}

// Вывод
Поток Thread-0 упал: java.lang.RuntimeException: Ошибка в потоке!
```

# ✅⭐Best Practices⭐✅
Если ты пишешь многопоточный код и не используешь `UncaughtExceptionHandler` — у тебя ==точно теряются ошибки==. Потоки падают молча, а ты потом ломаешь голову, почему «ничего не работает».

Грамотная практика — ставить дефолтный хендлер через `Thread.setDefaultUncaughtExceptionHandler()`, чтобы ловить все фатальные ошибки.
