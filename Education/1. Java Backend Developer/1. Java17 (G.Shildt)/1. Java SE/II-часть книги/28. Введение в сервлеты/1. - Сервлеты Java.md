
> [!NOTE] Сервлеты
> Сервлеты - это `небольшие программы`, которые выполняются `на серверной стороне` веб подключения.

# 📕 1. Происхождение сервлетов

Чтобы понять, **зачем вообще придумали сервлеты**, нужно вспомнить, как **браузер и сервер** взаимодействуют.

Когда ты вводишь адрес сайта (URL) — браузер делает **HTTP-запрос** к серверу.  
Сервер находит нужный **файл (HTML, изображение, видео и т.д.)** и возвращает его в ответ.  
Это называется ==**статическое содержимое**== — то есть заранее готовый файл.

Пример MIME-типа (указание, что за данные приходят):
- `text/plain` — обычный текст
- `text/html` — HTML-страница  
    Браузер по MIME-типу понимает, **как** отображать полученные данные.


> [!NOTE] MIME-типы
> **Internet Media Types**, также известные как **MIME-типы** (Multipurpose Internet Mail Extensions), представляют собой `стандарты для обозначения типов данных`, которые `могут быть переданы посредством сети Интернет` с применением стандарта MIME


## Проблема статического содержимого
Когда сайты стали интерактивными (интернет-магазины, базы данных, заказы и т.д.), нужно было:
- показывать **актуальные данные** (например, остатки товара, цена, заказы),
    
- 💡то есть страницы должны были **генерироваться динамически** — прямо при запросе.

---
## CGI (Common Gateway Interface)

Первые ==динамические страницы== создавались через механизм **CGI** — «Общий шлюзовой интерфейс».

🧩 Суть:  
Когда клиент делает запрос, веб-сервер запускает отдельную **внешнюю программу** (например, написанную на C, C++ или Perl),  
которая:

- получает данные запроса (через stdin);
    
- делает, например, запрос к БД;
    
- формирует HTML;
    
- и отправляет результат серверу, который пересылает его клиенту.
    
### Недостатки CGI
CGI оказалось **очень неэффективным**:

1. Для каждого запроса создавался **новый процесс** → это огромные накладные расходы на процессор и память.
    
2. Каждый раз заново открывалось соединение с базой данных.
    
3. Программы зависели от платформы (разные ОС, разные компиляторы).
    
4. Из-за этого всё работало **медленно и не масштабировалось**.

---
## Появление сервлетов

Чтобы решить эти проблемы, придумали **сервлеты** — специальные Java-классы, работающие **внутри веб-сервера (или контейнера сервлетов)**.

Сервлет — это **Java-программа**, которая принимает HTTP-запросы и формирует HTTP-ответы, но:

- она **не запускается заново при каждом запросе**,
    
- работает **в одном процессе** вместе с сервером,
    
- и использует **потоки**, а не процессы — что гораздо быстрее и экономнее.

# 📕 2. Жизненный цикл сервлета

Центральное место в жизненном цикле сервлета занимают три метода: `init()` , `service()` и `destroy()` . 
Они реализуются каждым сервлетом и ==ВЫЗЫВАЮТСЯ сервером== в определенное время.


> [!NOTE] В Tomcat ПРИ ПЕРВОМ ОБРАЩЕНИИ К СЕРЛЕТУ
> 
## Основные этапы жизненного цикла сервлета
| Этап                                   | Метод       | Когда вызывается                               | Что делает                                                                     |
| -------------------------------------- | ----------- | ---------------------------------------------- | ------------------------------------------------------------------------------ |
| **1. Загрузка и инициализация**        | `init()`    | Один раз при первой загрузке сервлета в память | Подготавливает сервлет: инициализирует переменные, открывает соединения и т.д. |
| **2. Обработка запросов**              | `service()` | `Для` каждого `входящего` `HTTP-запроса`       | Обрабатывает запрос клиента и формирует ответ                                  |
| **3. Завершение работы (уничтожение)** | `destroy()` | Один раз перед выгрузкой сервлета из памяти    | Освобождает ресурсы: закрывает файлы, соединения, сохраняет данные             |
## 🔸 Этап 1. `init()` — инициализация сервлета
Когда ==ВПЕРВЫЕ приходит HTTP-запрос== к данному сервлету:
1. Контейнер **загружает класс сервлета** (если он ещё не загружен).
2. **Создаёт объект** сервлета (КОНСТРУКТОР).
	
3. **Вызывает метод `init()`**.

📌 Этот метод вызывается **только один раз** за всё время существования сервлета.  

Обычно здесь:
- 💡читают параметры конфигурации (из `web.xml` или аннотаций),
- 🧠создают подключения к БД,
- подготавливают ресурсы.

```java
@Override
public void init() throws ServletException {
    // Инициализация ресурсов
    dbConnection = DriverManager.getConnection(...);
    System.out.println("Servlet initialized");
}
```

## 🔸 Этап 2. `service()` — обработка HTTP-запросов
==Каждый раз==, когда клиент (браузер) отправляет ==запрос==:
- контейнер вызывает **`service()`** у уже инициализированного сервлета.
    

Метод `service()` получает два объекта:
```java
public void service(
	ServletRequest req, // Запрос
	ServletResponse res // Ответ
)
```

🟢В случае **HttpServlet** он вызывает один из методов:
- `doGet()` — если запрос типа GET
- `doPost()` — если POST
- `doPut()`, `doDelete()`, и т.д.
    
📌 Сервлет **не создаётся заново** при каждом запросе — используется один и тот же объект.  
Это повышает производительность.

```java
@Override
protected void doGet(
	HttpServletRequest req, 
	HttpServletResponse resp) throws ServletException, IOException 
{
    resp.setContentType("text/html");
    resp.getWriter().println("<h1>Hello, Servlet!</h1>");
}
```

## 🔸 Этап 3. `destroy()` — завершение работы
Когда сервер решает выгрузить сервлет из памяти (например, при остановке приложения или очистке ресурсов), контейнер вызывает метод `destroy()`.

📌 Этот метод вызывается **один раз** — перед удалением сервлета.

Обычно здесь:
- 💡закрывают соединения с базой данных,
    
- 💡останавливают фоновые потоки,
	
- 💡сохраняют данные.

```java
@Override
public void destroy() {
    // Освобождение ресурсов
    dbConnection.close();
    System.out.println("Servlet destroyed");
}
```
После этого объект сервлета **удаляется сборщиком мусора (GC)**.


```scss
        ┌──────────────┐
        │  Загружается │
        └──────┬───────┘
               │
         init() вызывается
               │
               ▼
        ┌──────────────┐
        │ Обработка    │
        │ service()    │ ← вызывается при каждом запросе
        └──────┬───────┘
               │
         destroy() вызывается
               │
               ▼
        ┌──────────────┐
        │  Выгрузка    │
        └──────────────┘
```
