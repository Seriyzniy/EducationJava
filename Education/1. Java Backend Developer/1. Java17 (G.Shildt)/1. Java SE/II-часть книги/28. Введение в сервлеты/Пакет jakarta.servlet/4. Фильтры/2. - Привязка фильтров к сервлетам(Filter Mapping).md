
> [!NOTE] Filter Mapping
> **Filter Mapping** — это `способ`, с помощью которого `веб-контейнер` (Tomcat, Jetty, и т.д.) `решает`,  
> `какие фильтры` должны применяться `к какому запросу`,  и `в каком порядке` их вызывать.

То есть именно через **filter-mapping** мы говорим контейнеру:
> "Для всех запросов, подходящих под этот URL-шаблон или идущих к этому сервлету, вызывай вот этот фильтр."

## ГДЕ задаётся mapping
==Mapping== фильтров можно задать ==двумя способами==:
1. 📄 **В `web.xml`** (deployment descriptor — традиционный способ)
2. 💬 **Через аннотации** `@WebFilter` (современный способ)

## КАК: Способ привязки
Каждый фильтр можно "привязать" двумя способами:

|Тип привязки|Что это значит|
|---|---|
|**по имени сервлета** (`<servlet-name>`)|Фильтр применяется только к конкретному сервлету|
|**по URL-шаблону** (`<url-pattern>`)|Фильтр применяется ко всем запросам, чей путь совпадает с шаблоном|

📍 Например:
```xml
<filter-mapping>
    <filter-name>AuthFilter</filter-name>
    <url-pattern>/secure/*</url-pattern>
</filter-mapping>
```

→ Любой запрос к URL, начинающемуся с `/secure/`, ==будет проходить через== `AuthFilter`.


### Привязка по ЦЕПОЧКЕ
```xml
<!-- 1️⃣ Авторизация -->
    <filter>
        <filter-name>AuthFilter</filter-name>
        <filter-class>com.example.servlets.AuthFilter</filter-class>
        <async-supported>true</async-supported>
    </filter>

    <!-- 2️⃣ Логирование -->
    <filter>
        <filter-name>LoggingFilter</filter-name>
        <filter-class>com.example.servlets.LoggingFilter</filter-class>
        <async-supported>true</async-supported>
    </filter>

    <!-- 3️⃣ Сжатие ответа -->
    <filter>
        <filter-name>CompressionFilter</filter-name>
        <filter-class>com.example.servlets.CompressionFilter</filter-class>
        <async-supported>true</async-supported>
    </filter>


    <!-- ==================== FILTER MAPPINGS ==================== -->

    <!-- Важно: ❗порядок в web.xml определяет порядок вызова❗ -->
    <filter-mapping>
        <filter-name>AuthFilter</filter-name>
        <url-pattern>/secure/*</url-pattern>
    </filter-mapping>

    <filter-mapping>
        <filter-name>LoggingFilter</filter-name>
        <url-pattern>/secure/*</url-pattern>
    </filter-mapping>

    <filter-mapping>
        <filter-name>CompressionFilter</filter-name>
        <url-pattern>/secure/*</url-pattern>
    </filter-mapping>


    <!-- ==================== SERVLET ==================== -->

    <servlet>
        <servlet-name>MainServlet</servlet-name>
        <servlet-class>com.example.servlets.MainServlet</servlet-class>
        <async-supported>true</async-supported>
    </servlet>

    <servlet-mapping>
        <servlet-name>MainServlet</servlet-name>
        <url-pattern>/secure/main</url-pattern>
    </servlet-mapping>
```

## Порядок выполнения фильтров (Filter Chain)
![[Pasted image 20251018101355.png]]
Когда несколько фильтров применяются к одному ресурсу, контейнер создаёт **цепочку фильтров (Filter Chain)**.

- 💡**Порядок вызова фильтров** зависит от **порядка их описания** в `web.xml`.
    
- Первый в списке — первый, кто "перехватывает" запрос.
    
- После него управление передаётся дальше вызовом:
```java
chain.doFilter(request, response);
```
- ⭐⭐⭐Когда _последний фильтр в цепи_ вызывает `chain.doFilter()`, _управление уходит в **сервлет**_.
    
- ⭐⭐⭐После ==завершения сервлета== — управление возвращается обратно ==по цепочке (в обратном порядке).==

```scss
FilterA 
	⬇
FilterB 
	⬇
Servlet 
	⬇
FilterB (обратный проход) 
	⬇
FilterA (обратный проход)
```

## Примеры XML
### Привязка по URL
```XML
<filter>
    <filter-name>LoggingFilter</filter-name>
    <filter-class>com.example.LoggingFilter</filter-class>
</filter>

<filter-mapping>
    <filter-name>LoggingFilter</filter-name>
    <url-pattern>/*</url-pattern>
</filter-mapping>
```
🟢 Здесь фильтр `LoggingFilter` будет вызываться **для всех URL** (`/*`).


### Пример filter-mapping с dispatcher types
```xml
<filter-mapping>
    <filter-name>AuthFilter</filter-name>
    <url-pattern>/secure/*</url-pattern>
    
    <dispatcher>REQUEST</dispatcher>
    <dispatcher>FORWARD</dispatcher>
</filter-mapping>
```

Здесь `AuthFilter` будет срабатывать:
- и при прямом запросе `/secure/...`
- и при `forward` из других сервлетов.

## Dispatcher Types (Типы диспетчеризации)

> [!NOTE] Dispatcher Types
> **Dispatcher Types** —  это флаг, определяющий **в каких случаях** фильтр должен применяться.

Когда фильтр сопоставляется с ресурсом, мы можем уточнить:  
- применять ли его **только при прямом запросе от клиента**,  
- или также при **внутренних перенаправлениях и ошибках**.

|Тип|Когда применяется|Пример|
|---|---|---|
|`REQUEST`|Запрос пришёл **напрямую от клиента**|Обычный GET/POST|
|`FORWARD`|Запрос был **переадресован** (`RequestDispatcher.forward()`)|Контроллер → JSP|
|`INCLUDE`|Ресурс **включён** через `include()`|Вставка шапки/футера|
|`ERROR`|Обработка **страницы ошибки** (`error-page` в web.xml)|При 404 или 500|
|`ASYNC`|Асинхронная обработка (`AsyncContext.dispatch()`)|Servlet 3.0+ async|

> [!NOTE] ❗ПО УМОЛЧАНИЮ - `REQUEST`❗
> 📌 Если не указать `dispatcher`, то по умолчанию используется **`REQUEST`**.
