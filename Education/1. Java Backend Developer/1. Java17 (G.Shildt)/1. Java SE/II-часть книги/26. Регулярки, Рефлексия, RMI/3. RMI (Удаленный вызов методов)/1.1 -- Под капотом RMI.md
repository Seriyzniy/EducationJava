# 1. RMI Runtime
**RMI Runtime** — это **внутренний механизм Java**, встроенный в JVM,  
который ==управляет всей инфраструктурой удалённых вызовов==:  
- создаёт, 
- хранит и маршрутизирует объекты, stub’ы, сокеты и вызовы.

Можно сказать, что это «среда выполнения RMI», аналог **JVM для удалённых вызовов**.  

Она живёт **в каждой JVM**, участвующей в RMI — и на стороне **сервера**, и на стороне **клиента**.

# 2. Архитектура (уровни RMI)
```mathematica
        Клиентская JVM                         Серверная JVM
   ┌─────────────────────────┐           ┌─────────────────────────┐
   │       RMI Runtime       │           │       RMI Runtime       │
   │                         │           │                         │
   │   Stub  ───────────────►│──────────►│ Skeleton / Dispatcher   │
   │   RemoteCall (TCP, JRMP)│           │                         │
   │                         │           │   LiveRef ↔ Объект      │
   └─────────────────────────┘           └─────────────────────────┘
```
**Пояснение:**

- Stub (заглушка) на клиенте вызывает метод → создаёт `RemoteCall`.
    
- RMI runtime сериализует параметры и посылает запрос по TCP.
    
- На сервере RMI runtime получает вызов, находит объект по `LiveRef`, вызывает метод.
    
- Результат возвращается обратно клиенту.

## Внутренние компоненты RMI Runtime
| Класс                     | Назначение                                                                  |
| ------------------------- | --------------------------------------------------------------------------- |
| `LiveRef`                 | Уникальный `идентификатор удалённого объекта` (включает хост, порт, ObjID)  |
| `ServerRef`               | `Управляет вызовами` для конкретного `удалённого объекта`                   |
|                           |                                                                             |
| `ObjID`                   | Уникальный ID каждого экспортированного объекта                             |
|                           |                                                                             |
| `RemoteRef`               | Интерфейс, реализуемый stub'ом, содержит методы удалённого вызова           |
|                           |                                                                             |
| `Transport`               | Отвечает за физическую передачу сообщений (по JRMP, TCP)                    |
| `Dispatcher` / `Skeleton` | Принимает вызовы, находит объект, вызывает метод                            |
| `DGCImpl`                 | Distributed Garbage Collector — управляет временем жизни удалённых объектов |

## Пример жизненного цикла объекта в RMI Runtime

1️⃣ **Создание**
```java
HelloService service = new HelloServiceImpl();
```
→ `UnicastRemoteObject` вызывает `exportObject()`,  
→ RMI runtime создаёт `LiveRef` и `ServerRef`, открывает порт.


2️⃣ **Регистрация в Registry**
```java
registry.rebind("HelloService", service);
```
→ stub сохраняется в реестре.


3️⃣ **Клиент делает lookup**
```java
HelloService stub = (HelloService) registry.lookup("HelloService");
```
→ RMI runtime клиента получает stub с `LiveRef` (адресом и ObjID).
- Клиент делает запрос **в RMI Registry**, который работает на **сервере** (например, `host:1099`).
    
- Этот запрос говорит:
    «Отдай мне ссылку на удалённый объект с именем `HelloService`».
    
- Registry возвращает **stub (заглушку)** — объект, реализующий тот же интерфейс `HelloService`.

> 📦 Stub возвращается в сериализованном виде (через сеть), клиент его десериализует — и теперь у него есть локальный прокси.


4️⃣ **Удалённый вызов**
```java
stub.sayHello("Seriy");
```
→ RMI runtime ==клиента== сериализует аргументы → отправляет TCP-запрос.  
→ RMI runtime ==сервера== принимает → ищет объект по LiveRef → вызывает метод.


5️⃣ **Возврат результата**  
→ результат сериализуется и возвращается клиенту.
