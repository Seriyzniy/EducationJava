Способ обработки событий, принятый в исходной версии `Java (1.0)`, значительно изменился во всех последующих версиях `Java`, начиная с `1.1`. 
Хотя метод обработки событий ==версии 1.0== по-прежнему поддерживается, ==использовать его в новых программах не рекомендуется.==

1. ❌**Старая модель (Java 1.0)** — каждый компонент сам обрабатывает свои события.  
    ⚠️ Сейчас не используется.
    
2. ✅**Современная модель делегирования событий (Delegation Event Model)** —  
    источник события передаёт его **слушателю (listener)**, который реализует обработку.
# 📕 1. Модель делегирования обработки событий

Модель основана на ==трёх ключевых компонентах==:
1. **Источник** — генерирует события.
    
2. **Прослушиватель** — регистрируется для получения конкретных событий и обрабатывает их.
    
3. **Событие** — сам объект с информацией о произошедшем действии.

В модели делегирования обработки событий `ПРОСЛУШИВАТЕЛИ должны регистрироваться В ИСТОЧНИКЕ`, чтобы получать уведомления о поступлении событий, что обеспечивает важное преимущество: уведомления отправляются только тем прослушивателям, которые желают их получать

# 📕 2. Компоненты модели
## 📙 1. Событие

> [!NOTE] Событие представляет собой `объект`, `описывающий изменение` `состояния` `источника`. 
> Событие может быть сгенерировано как следствие взаимодействия человека с элементами графического пользовательского интерфейса. 
> 
> Действия, которые вызывают генерацию событий, включают `нажатие кнопки`, `ввод символа с клавиатуры`, `выбор элемента в списке` и `щелчок` кнопкой мыши.
> 

Также `могут происходить события`, которые напрямую `не связаны с взаимодействием` `с `пользовательским `интерфейсом`. 

Например, событие может генерироваться, когда истекает время таймера, значение счетчика превышает установленное значение, возникает программный или аппаратный сбой или операция завершается. 

Можно свободно определять события, подходящие для разрабатываемого приложения.


> [!NOTE] ⭐Каждый `тип события` имеет `собственный метод` `addXListener()`.
>

## 📙 2. Источник

> [!NOTE] Источник представляет собой `объект`, `который генерирует событие`. 
> Это происходит, когда внутреннее состояние данного объекта каким-либо образом изменяется. 
> 
> 💡Источники могут генерировать более одного типа событий.
> 

==Источник ДОЛЖЕН ЗАРЕГИСТРИРОВАТЬ ПРОСЛУШИВАТЕЛЕЙ==, чтобы они могли получать уведомления о событиях конкретного типа. 

> [!NOTE] Для `каждого ТИПА СОБЫТИЯ` предусмотрен `СВОЙ МЕТОД регистрации`
> 

### Регистрация прослушивателя - ОБЩАЯ форма
```JAVA
public void addTypeListener(TypeListener el)
```
В `Туре` указывается имя события, 
а в `el` - ссылка на прослушивателя событий.

Например, метод, который регистрирует прослушиватель ==событий клавиатуры==, называется 
	add==Key==Listener() , 

а метод, регистрирующий прослушиватель ==движения мыши==, имеет имя 
	add==МouseMotion==Listener()

Когда `происходит событие`, `все зарегистрированные прослушиватели` `уведомляются` и `ПОЛУЧАЮТ КОПИЮ объекта события`, что называется групповой рассылкой события. 

Во всех случаях ==уведомления отправляются== только ==тем прослушивателям==, ==которые зарегистрировались== для их получения.
### Отмена регистрации
==Источник== также ==обязан предоставлять метод==, который позволяет прослушивателю ==ОТМЕНЯТЬ регистрацию для события== конкретного типа и имеет следующую общую форму: 
```java
public void removeTypeListener(TypeListener el)
```

Скажем, чтобы удалить прослушиватель событий клавиатуры, потребуется вызвать метод remove==Key==Listener().

## 3. Прослушиватели событий

> [!NOTE] Прослушиватель - это объект, который уведомляется при возникновении события
> К нему предъявляются `два основных требования`
> 1. **==Регистрация в источниках==** - должен быть зарегистрирован в одном или нескольких источниках для получения уведомлений о конкретных типах событий
> 	
> 2. **==Реализация методов обработки==** - должен предоставлять обработчики событий через реализацию методов для получения и обработки уведомлений

### Техническая реализация:
- ==Методы== для обработки событий ==определены В наборе ИНТЕРФЕЙСОВ== (например, в пакете java.awt.event)
    
- 💡Пример: интерфейс `MouseMotionListener` содержит два метода для обработки событий перетаскивания и перемещения указателя мыши
    
- Любой объект может стать обработчиком событий, если реализует соответствующий интерфейс прослушивателя

### Критически важное требование к обработчику:

- Обработчик события **должен быстро возвращать управление**
    
- Графический интерфейс не должен сохранять управление в течение длительного времени
    
- После обработки события управление должно быть немедленно возвращено исполняющей сред
