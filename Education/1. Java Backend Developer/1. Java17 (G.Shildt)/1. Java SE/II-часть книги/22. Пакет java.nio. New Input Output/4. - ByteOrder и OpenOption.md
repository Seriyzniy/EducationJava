# 📕 1. ByteOrder


> [!NOTE] `ByteOrder`
> Определяет **==ПОРЯДОК расположения БАЙТОВ== в многобайтовых типах данных** (например, `int`, `long`, `float`) при записи в `ByteBuffer` или чтении из него.
> 
> Это важно, потому что разные процессоры и протоколы **по-разному интерпретируют байты**.
## Что такое порядок байтов

- **Big-endian** → старший байт идёт **первым**  
    (`0x12345678` хранится как `12 34 56 78`)
    
- **Little-endian** → младший байт идёт **первым**  
    (`0x12345678` хранится как `78 56 34 12`)

Примеры архитектур:

|Тип|Примеры CPU|
|---|---|
|Big-endian|PowerPC, SPARC|
|Little-endian|x86, x86-64, ARM (в современных системах)|

## Статические поля и методы
```java
// ✅ Константы порядка байтов
public static final ByteOrder BIG_ENDIAN;
public static final ByteOrder LITTLE_ENDIAN;

// ✅ Возвращает порядок байт платформы, на которой выполняется JVM
public static ByteOrder nativeOrder();
```
💡 `nativeOrder()` полезен, если ты работаешь с `MappedByteBuffer`, `DirectByteBuffer` или JNI —  
то есть с памятью, разделяемой с нативным кодом.

## Пример
```java
ByteBuffer buffer = ByteBuffer.allocate(8);

// Устанавливаем порядок little-endian
buffer.order(ByteOrder.LITTLE_ENDIAN);
buffer.putInt(0x12345678);

buffer.flip();
System.out.printf("%02X %02X %02X %02X%n",
    buffer.get(), buffer.get(), buffer.get(), buffer.get());

```

🧠 Вывод на типичной x86-системе:
```
78 56 34 12
```

## 💡 Особенности и подводные камни 💡

1️⃣ **По умолчанию ByteBuffer big-endian**, как в сетевых протоколах (network byte order).  
2️⃣ Если ты работаешь с нативным кодом, нужно **совпадение порядка байт** (`nativeOrder()`).  
3️⃣ Несоответствие endianness между клиентом и сервером → краш в бинарных протоколах.  
4️⃣ При чтении бинарных файлов (например, форматы изображений, WAV, ZIP) всегда указывай `ByteOrder`.  
5️⃣ Если ты используешь `DirectByteBuffer` для общения с JNI или `MappedByteBuffer`, **устанавливай nativeOrder()** для ускорения (без байтового реверса).

# 📕 2. OpenOption

> [!NOTE] `OpenOption`
> `Интерфейс-маркер`, описывающий **==режимы открытия файла или канала==**(чтение, запись, создание, удаление и т.д.).
> 
> 
> Все конкретные варианты ==реализованы в== `StandardOpenOption` (enum).

Используется при работе с `FileChannel`, `AsynchronousFileChannel` и `Files.newByteChannel()`.

## Интерфейс
```java
public interface OpenOption { }
```

## Стандартные значения (`StandardOpenOption`)
```java
public enum StandardOpenOption implements OpenOption {
    
    // ✅ Режимы доступа ✅
    // Открыть для чтения
    READ,
    // Открыть для записи
    WRITE,
    // Добавлять в конец файла
    APPEND,


    // ✅ Поведение ПРИ ОТКРЫТИИ существующего файла ✅
    // Очистить файл при открытии (если существует)
    TRUNCATE_EXISTING,


    // ✅ Поведение ПРИ СОЗДАНИИ файла ✅
    // Создать новый, если не существует
    CREATE,
    // Создать новый, но бросить исключение, если уже есть
    CREATE_NEW,


    // ✅ Специальные режимы ✅
    // Удалить при закрытии
    DELETE_ON_CLOSE,
    // Создать "разрежённый" файл (для систем, поддерживающих)
    SPARSE,


    // ✅ Гарантии целостности данных ✅
    // Принудительный flush метаданных и содержимого на диск
    SYNC,
    // Flush только данных (без метаданных)
    DSYNC
}
```

## Пример
```JAVA
import java.nio.*;
import java.nio.channels.*;
import java.nio.file.*;
import java.util.EnumSet;

public class OpenOptionExample {
    public static void main(String[] args) throws Exception {
        Path path = Path.of("example.txt");

        try (FileChannel channel = FileChannel.open(path,
                EnumSet.of(StandardOpenOption.CREATE,
                           StandardOpenOption.WRITE,
                           StandardOpenOption.TRUNCATE_EXISTING))) {

            ByteBuffer buffer = ByteBuffer.wrap("Hello, OpenOption!".getBytes());
            channel.write(buffer);
        }
    }
}
```
