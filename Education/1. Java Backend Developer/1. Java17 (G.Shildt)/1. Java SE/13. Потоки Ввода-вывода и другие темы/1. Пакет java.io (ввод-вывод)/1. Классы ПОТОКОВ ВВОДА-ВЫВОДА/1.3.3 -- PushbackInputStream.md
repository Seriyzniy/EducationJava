`PushbackInputStream` — это специализированный поток ввода, который расширяет `FilterInputStream`. 

> [!NOTE] ⭐⭐ Ключевая особенность: ⭐⭐
> Позволяет ==**"откатывать" (возвращать) байты обратно в поток**== методом `unread(...)`.

Зачем?
- При написании парсеров (например, XML, JSON, собственные протоколы).
    
- Когда мы читаем 1 байт "на опережение", но понимаем, что он принадлежит другому токену — и нужно вернуть его обратно.
    
- Работает как буфер с "откатом", глубина которого задаётся в конструкторе (по умолчанию 1 байт).
    
⚠️ Важно: `PushbackInputStream` **не поддерживает mark/reset**, потому что `unread()` — фактически его альтернатива.

### Как это устроено внутри

- У `PushbackInputStream` ==есть **собственный буфер** (массив байтов), куда он кладёт "откаченные" данные.==
    
- Когда вызываешь `unread(...)`, байт(ы) кладутся в этот буфер.
    
- ⭐⭐⭐А ==при следующем== `read()` поток ==**сначала проверяет буфер отката**==. Если там что-то есть → оно вернётся первым. Только когда буфер пуст — читается из исходного потока (`in`).

***
## Конструкторы
```java
public PushbackInputStream(InputStream in)
// in — исходный поток, из которого читаем
// ⭐буфер отката имеет размер 1 байт (можно вернуть только один символ назад)

public PushbackInputStream(InputStream in, int size)
// in — исходный поток
// ⭐ size — глубина буфера отката (сколько байтов можно "откатить" подряд)
```

## Методы
```java
// Читает 1 байт из потока (0–255).
// Возвращает -1 если достигнут EOF.
// Если в буфере отката есть байты — они читаются первыми.
public int read() throws IOException

// Читает до len байтов в массив b[], начиная с позиции off.
// Возвращает количество реально прочитанных байтов или -1 при EOF.
// Сначала выгребает данные из буфера отката, потом из "in".
public int read(byte[] b, int off, int len) throws IOException


// ✅ МЕТОД ОТКАТА ✅
// Откатывает один байт обратно в поток.
// Если буфер переполнен — IOException.
public void unread(int b) throws IOException

// ✅✅Откатывает массив байтов в поток.
public void unread(byte[] b) throws IOException

// ✅✅Откатывает часть массива (от off до off+len).
public void unread(byte[] b, int off, int len) throws IOException


// Возвращает количество доступных для чтения байт (из in + из буфера).
public int available() throws IOException

// Закрывает поток (и сам, и вложенный in).
public void close() throws IOException

// Возвращает true, если mark/reset не поддерживаются.
public boolean markSupported()
```

***
## 💡 Особенности 💡

- 📌 🛑🛑🛑**Буфер отката** фиксированного размера — больше, чем в него влезет, "откатить" нельзя.
    
- 📌 Сначала читается всё, что было "откачено", потом уже данные из исходного потока.
    
- 📌 Используется вместо `mark/reset`, когда нужно немного "заглянуть вперёд".
    
- ❌ ==Не потокобезопасен.==

***
## Пример
```java
import java.io.*;

public class PushbackInputStreamExample {
    public static void main(String[] args) {
        String data = "abc123";
        byte[] bytes = data.getBytes();

        try (PushbackInputStream pis =
                     new PushbackInputStream(new ByteArrayInputStream(bytes), 2)) {

            int ch;

            while ((ch = pis.read()) != -1) {
                if (Character.isDigit(ch)) {
                    // Если встретили цифру, возвращаем её обратно
                    pis.unread(ch);
                    System.out.println("Найдена цифра: " + (char) pis.read());
                } else {
                    System.out.println("Буква: " + (char) ch);
                }
            }

        } catch (IOException e) {
            e.printStackTrace();
        }
    }
}

Буква: a
Буква: b
Буква: c
Найдена цифра: 1
Найдена цифра: 2
Найдена цифра: 3
```

## Best-Practices

- ✅ Используй `PushbackInputStream`, когда нужно реализовать **lookahead** (заглянуть вперёд) при парсинге.
    
- ✅ Размер буфера (`size`) подбирай под глубину возможного "отката". Для JSON/XML часто хватает 1–2 байта.
    
- ❌ Не забывай: `mark/reset` тут **не работают**. Не пытайся их использовать.
    
- ✅ Лучше комбинировать с `BufferedInputStream` для производительности, если основной поток — файл или сеть.
    
- ❌ Никогда не "откатывай" больше байтов, чем размер буфера — будет `IOException`.

***
## Смысл отката

Откат (`unread`) нужен не для того, чтобы "вернуть и снова прочитать то же самое", а для случаев, когда:

1. **Ты прочитал слишком много для своей текущей задачи**  
    Например, ты читал цифры, пока встречаются, и на символе `+` понял:
    
    - "Окей, число закончилось".
        
    - Но `+` сам по себе важен для следующего шага (это оператор, а не часть числа).
        
    - Если бы не было отката — оператор `+` **потерялся бы**, потому что уже "считан".
        
    - С откатом — ты возвращаешь его в поток и даёшь возможность следующему этапу (лексеру/парсеру) обработать его "как будто он ещё не был считан".
        
2. **Ты заглянул наперёд, но понял, что пока рано**  
    Иногда нужно прочитать символ(ы), чтобы понять контекст. Пример:
    
    - У тебя код: `>=`.
        
    - Читаешь `>`, думаешь: "Это оператор `>`".
        
    - Но потом видишь `=` и понимаешь: "Ага, это не `>`, а `>=`".
        
    - В таком случае ты откатываешь и `=`, и `>`, чтобы парсер выше уровня обработал их как одно целое.

### Рассмотренный пример
Строка: `123+456`

1. Читаем `1`, `2`, `3` → собираем число `123`.
    
2. Читаем `+` → понимаем: "Это уже не цифра, число закончилось".
    - Если мы не сделаем `unread`, то число `123` мы собрали, но символ `+` уже **потерян** (его прочитал текущий уровень кода, но дальше он нужен следующему этапу).
        
    - С `unread` мы возвращаем `+` в поток, чтобы следующий шаг мог с ним работать.
        
3. Следующий этап парсинга начнётся именно с `+`.
