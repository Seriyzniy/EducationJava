`PushbackReader` — это **с==имвольный поток**== (наследник `FilterReader`), который ==позволяет **откатывать назад уже прочитанные символы**== и возвращать их обратно в поток.

## Конструкторы
```java
public PushbackReader(Reader in)
// in — поток, который оборачиваем.
// Размер буфера отката по умолчанию = 1 символ.

public PushbackReader(Reader in, int size)
// in — оборачиваемый Reader.
// size — размер pushback-буфера (сколько символов можно "вернуть назад").
```


## Методы
```java
// Читает один символ из потока.
// Возвращает символ (0–65535) или -1 при EOF.
public int read() throws IOException

// Читает блок символов в массив cbuf.
// Возвращает количество реально прочитанных символов или -1 при EOF.
public int read(char[] cbuf, int off, int len) throws IOException



// "Откат" символа назад: помещает один символ в pushback-буфер.
// Следующий вызов read() вернёт именно этот символ.
public void unread(int c) throws IOException

// ✅✅"Откат" массива символов (целиком).
public void unread(char[] cbuf, int off, int len) throws IOException

// ✅✅"Откат" всего массива символов.
public void unread(char[] cbuf) throws IOException



// Проверяет, доступен ли поток для чтения без блокировки.
public boolean ready() throws IOException

// Закрывает поток (и внутренний Reader).
public void close() throws IOException

// Пропускает n символов вперёд.
// Возвращает фактическое количество пропущенных символов.
public long skip(long n) throws IOException
```

***
## 💡 Особенности 💡
- ⭐`PushbackReader` **использует внутренний буфер** для хранения "откаченных" символов.
    
- ⭐Размер буфера задаётся конструктором. Если ты откатишь больше символов, чем позволяет буфер, получишь `IOException`.
    
- Откат символов — это не "магия", а запись в буфер, который потом возвращается в `read()`.
    
- Очень полезен для **лексических анализаторов**, парсеров JSON/XML, компиляторов.

***
## Пример
```java
import java.io.*;

public class PushbackReaderExample {
    public static void main(String[] args) throws IOException {
        String text = "123+456";
        
        try (PushbackReader pr = new PushbackReader(new StringReader(text), 10)) 
	    {
            int c;
            while ((c = pr.read()) != -1) {
                if (Character.isDigit(c)) {
                    System.out.print((char) c);
                } else {
                    // Если встретили не цифру — возвращаем символ в поток
                    pr.unread(c);
                    System.out.println("\nНаткнулся на символ: " + (char) pr.read());
                }
            }
        }
    }
}
```

## Best-Practices

- ✅Используй `PushbackReader`, когда нужен **откат символов** (лексеры, парсеры).
    
- Размер буфера подбирай с запасом — если "откатишь" больше, чем позволяет буфер, словишь `IOException`.
    
- ✅Для работы с **байтовыми** потоками аналог есть — `PushbackInputStream`.
    
- ❌❗✅Не используй для простого чтения строк — это оверхед. Лучше `BufferedReader`.
    
- Оборачивай в `try-with-resources`, чтобы гарантированно закрывать базовый поток.

## Смысл отката

Откат (`unread`) нужен не для того, чтобы "вернуть и снова прочитать то же самое", а для случаев, когда:

1. **Ты прочитал слишком много для своей текущей задачи**  
    Например, ты читал цифры, пока встречаются, и на символе `+` понял:
    
    - "Окей, число закончилось".
        
    - Но `+` сам по себе важен для следующего шага (это оператор, а не часть числа).
        
    - Если бы не было отката — оператор `+` **потерялся бы**, потому что уже "считан".
        
    - С откатом — ты возвращаешь его в поток и даёшь возможность следующему этапу (лексеру/парсеру) обработать его "как будто он ещё не был считан".
        
2. **Ты заглянул наперёд, но понял, что пока рано**  
    Иногда нужно прочитать символ(ы), чтобы понять контекст. Пример:
    
    - У тебя код: `>=`.
        
    - Читаешь `>`, думаешь: "Это оператор `>`".
        
    - Но потом видишь `=` и понимаешь: "Ага, это не `>`, а `>=`".
        
    - В таком случае ты откатываешь и `=`, и `>`, чтобы парсер выше уровня обработал их как одно целое.

### Рассмотренный пример
Строка: `123+456`

1. Читаем `1`, `2`, `3` → собираем число `123`.
    
2. Читаем `+` → понимаем: "Это уже не цифра, число закончилось".
    - Если мы не сделаем `unread`, то число `123` мы собрали, но символ `+` уже **потерян** (его прочитал текущий уровень кода, но дальше он нужен следующему этапу).
        
    - С `unread` мы возвращаем `+` в поток, чтобы следующий шаг мог с ним работать.
        
3. Следующий этап парсинга начнётся именно с `+`.
