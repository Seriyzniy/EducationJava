`try-with-resources` — это специальная форма `try`, в которую ты **встраиваешь ресурс**, и он будет **автоматически закрыт**, когда блок `try` завершится (включая случаи с `return` или `throw`).

> [!NOTE] 🛑==Ресурс== объявленный в оператора `try` НЕЯВНО ==является== `finall`🛑
> 

```java
try (Resource res = new Resource()) {
    // работа с res
} catch (Exception e) {
    // обработка исключений
}
```
> [!NOTE] 🔒 `res.close()` вызовется **автоматически** — даже если будет исключение.

***
# 📕 1. Требование: ресурс должен реализовывать интерфейс `AutoCloseable`
```java
public interface AutoCloseable {
    void close() throws Exception;
}
```
> [!NOTE] 🛑Также подойдёт `java.io.Closeable`, который расширяется (кем?) `AutoCloseable` и ==предназначен для потоков==. 🛑

# 💡Неочевидные особенности💡
## 1. **Порядок закрытия — в _обратном_ порядке**
```java
try (A a = new A(); B b = new B()) {
    // работа
}
```
🔹 Сначала закроется `b.close()`, потом `a.close()`. Это важно, если ресурсы зависят друг от друга.
> [!NOTE] 🛑Можно ==использовать НЕСКОЛЬКО РЕСУРСОВ== в в блоке `try with resources`
> 

## 2. ПОДАВЛЕННЫЕ исключения

> [!NOTE] ⭐Подавленное исключение, работает ТОЛЬКО с `try-with-resources`⭐
> Подавленные исключения (`Suppressed Exceptions`) — это ==механизм, который придумали **специально для== `try-with-resources`** (Java 7).  
> 
> Смысл: если при закрытии ресурса (`close()`) возникло исключение, оно **не должно затирать основное исключение из блока `try`**. 
> Поэтому JVM добавляет его в список подавленных (`addSuppressed`).
> 
> ⭐⭐ То есть ==обычный==`try-finally` ==такой магии не делает==. ⭐⭐

Сначала ==выбрасывается основное исключение==, **а 🛑исключение из `close()` будет подавлено**.🛑

```java
class Connection implements Closeable {
    @Override
    public void close() throws IOException {
        throw new IOException(); // При закрытии всегда бросается исключение
    }
}

public class App {
    public static void main(String[] args) {
        try (Connection c = new Connection()) {
            throw new RuntimeException(); // Бросаем RuntimeException в try
        } catch (IOException e) {
            e.printStackTrace(); 
        }
    }
}
```

В данном примере в блоке try выбрасывается исключение RntimeException, которое становится ОСНОВНЫМ.
Сразу же после этого вызывается метод `close()`, так как используется `try with resources`. В этом методе также выбрасывается исключение.

JVM видит, что после основного RuntimeException, выбрасывается еще одно, поэтому JVM добавляет второе исключение (IOException) ==КАК ПОДАВЛЕННОЕ в RuntimeException через== `addSuppressed()`.

После чего, ход выполнения доходит до `catch`, поскольку в catch обрабатывается только IOException, а у нас основное исключение RuntimeException, то в блок catch мы не попадаем. Из-за чего вызовется стандартный обработчик исключений

==ОДНАКО==

> [!NOTE] `finally` перекрывает абсолютно все исключения
> 
```java
public class MyMain {
	public static void main(String[] args) {
		try {
    		TestConnection c = new TestConnection();
			c.test();
		} catch (IOException e) {
			Throwable[] throwable = e.getSuppressed();
			
			if(throwable.length <= 0) {
				System.out.println("Array of Throwable is EMPTY");
			} else {
				RuntimeException re = (RuntimeException) throwable[0];
				
				System.out.println(re.getMessage());
			}
			
		}
	}
}

class Connection {
	void test() throws IOException {
		try { //🛑 Обычный try 🛑
			throw new ArithmeticException("Ex arith in try");
		} catch (IndexOutOfBoundsException e) {
			System.out.println("Не попадем в catch");
		} finally{
			throw new IOException("Ex IO in finally");
		}
	}
}

Array of Throwable is EMPTY
```

# 🛑ВАЖНОЕ ПРАВИЛО🛑

> [!NOTE] Подавленные исключения в `try-with-resources` БЕЗ `finally`
> Если в теле `try` выбрасывается **исключение `A`**, а затем при закрытии одного или нескольких ресурсов выбрасываются **другие исключения `B1`, `B2`, ...**, то:
> 
> - Исключение `A` становится **основным**.
>     
> - Все `B1`, `B2`, ... ==добавляются как **подавленные** ==(via `addSuppressed()`).
>     
> - Если **в теле `try` не было исключения**, но при закрытии ресурса оно появилось — **тогда оно становится основным**.


> [!NOTE] ⭐⭐⭐`finally` ВСЕГДА имеет НАИВЫСШИЙ ПРИОРИТЕТ перед другими ⭐⭐⭐
> - `try-with-resources` умеет **сохранять исключения из `close()` как suppressed**(подавленные), НО **ТОЛЬКО ЕСЛИ НЕТ `finally` с исключением**.
> .
>  - Если в `finally` выбросить исключение → оно ==**перекрывает абсолютно всё** (даже suppressed).==
> - `finally` всегда имеет **высший приоритет**.
