Система модулей также поддерживает расширения во время выполнения. 
Одной из наиболее важных является ==возможность создания образа времени выполнения==, который специально приспособлен к вашему приложению. 

Для этого можно применять инструмент JDK под на званием `jlink`, объединяющий группу модулей в оптимизированный образ времени выполнения. 

Инструмент `jlink` можно использовать для связывания модульных файлов JAR, файлов JMOD или даже модулей в их неархивированной форме с "развернутым каталогом"

***
# 📕1. Связывание файлов в развернутом каталоге (с файлов `module-info.class`)

Для начала ==соберём **ОБРАЗ ВРЕМЕНИ ВЫПОЛНЕНИЯ== (runtime image)** из **неархивированных модулей** (т.е. НЕ `.jar`, А уже СОБРАННЫХ `.class` внутри папок).

При работе в среде Windows показанная далее команда связывает модули для первого примера из этой главы.

## 📙1. Утилита jlink
```bash
jlink [OPTIONS] --module-path <paths> --add-modules <modules> --output <dir>
```
> [!NOTE] Сборка из `module-info.class`
> `jlink` берёт модульные артефакты (`.jmod` или «exploded modules» — каталоги с `module-info.class`) и собирает из них минимальный образ JRE + твоё приложение.

==Соберем ОБРАЗ на основе готового проекта из предыдущих глав== (см [[5. - Использование служб (Services), ЛУЧШИЙ пример]])
```bash
jlink 
  --launcher MyScriptsToStart=modAppDemo/org.appstart.ModAppDemo 
  --module-path "%JAVA_HOME%"/jmods;MyServiceBuild 
  --add-modules modAppDemo 
  --output mylinkedmodapp
```

Запустим собранное приложение
> [!NOTE] 🧠Утилита создаст скрипты для запуска под Linux и Windows (bat).
> ==Запустить== батник для винды можно с указанием ==полного пути к батнику, ЗАКЛЮЧЕННОГО В КАВЫЧКИ==

### 🔶 Параметр --launcher
```bash
--launcher MyScriptsToStart=modAppDemo/org.appstart.ModAppDemo
```
==СОЗДАЕТ **СКРИПТ ЗАПУСКА**== внутри runtime-образа.

Формат:
```bash
<имя_скрипта>=<имя_модуля>/<полностью.квалифицированный.Класс>
```

В нашем случае:
- 🟢`MyScriptsToStart` → имя скрипта (появится как `bin/MyScriptsToStart` в собранном образе).
- 🔴`modAppDemo/org.appstart.ModAppDemo` → ==какой модуль запускать и какой класс внутри него использовать как== `main`.

То есть после сборки можно будет запускать так:
```bash
mylinkedmodapp/bin/MyScriptsToStart
```
***
### 🔶 Параметр --module-path
```bash
--module-path "%JAVA_HOME%"\jmods;MyServiceBuild
```
Список путей, где искать модули.
🧠🧠Пути разделяются `;` (Windows) или `:` (Linux/Mac).🧠🧠

⭐Обычно указывают:⭐
1. `$JAVA_HOME/jmods` → здесь лежат стандартные `.jmod` из JDK (обязательно для базовых модулей типа `java.base` и т.д.). (ТУТ БЕРЕТСЯ ИЗ СИСТЕМНОЙ ПЕРЕМЕННОЙ ОС, если утсановлена)

2. `MyServiceBuild` → каталог с твоими собственными модулями (exploded form — папки с `module-info.class` внутри).

> [!NOTE] Таким образом `jlink` найдёт и `системные`, и `пользовательские модули`.
> 

***
### 🔶 Параметр --add-modules
```bash
--add-modules modAppDemo
```
==Список модулей==, которые ==будут включены в runtime-образ==.

> [!NOTE] 🧠Указать главный модуль, а остальное подтянет `jlink`🧠
> Указываешь **главный модуль** приложения (`modAppDemo`), а `jlink` автоматически подтянет все зависимости (`modMathFuncs`, `java.base`, `java.logging` и т.д.).

Можно перечислить через запятую:
```bash
--add-modules modAppDemo,modMathFuncs
```

***
### 🔶 Параметр --output mylinkedmodapp
Папка, куда будет создан готовый runtime-образ.
Структура внутри похожа на JDK/JRE:
```bash
mylinkedmodapp/
  bin/           <- здесь твой скрипт "MyScriptsToStart"
  conf/
  include/
  legal/
  lib/
```

> [!NOTE] ⭐⭐Это по сути минимизированная JRE + твоё приложение.⭐⭐
> 

***
### 🔶 Дополнительно полезные опции

- `--strip-debug` → удаляет debug-информацию из классов.
    
- `--compress=2` → сжимает ресурсы (2 = ZIP).
    
- `--no-header-files` → убрать include-заголовки (для JNI).
    
- `--no-man-pages` → убрать man-страницы.

# 📕2. Связывание модульных файлов JAR
Несмотря на удобство связывания модулей из развернутого каталога, при работе с реальным кодом чаще придется иметь дело с файлами JAR. 

> [!NOTE] (Вспомните, что `JAR` означает `Java ARchive` (архив Java) и ``является файловым форматом``, обычно применяемым `при развертывании приложений`.)

==В случае модульного кода== будут ==использоваться модульные файлы JAR==. 

`Модульный файл JAR` содержит файл `module-info.class`. Начиная с версии JDK 9, инструмент `jar` позволяет создавать модульные файлы `JAR`.

После создания модульных файлов JAR вы можете с помощью jlink связать их в образ времени выполнения.

## 📙 1. Создание JAR файлов приложения
Ниже приведены команды jar, которые создают модульные файлы JAR для программы MyModAppDemo. 
Каждая команда должна вводиться в каталоге, расположенном непосредственно над проектом.

> [!NOTE] ❗❗СНАЧАЛА ==СОЗДАТЬ ПАПКУ, КУДА БУДУТ ПОМЕЩАТЬСЯ== `.jar` файлы, иначе `NoSuchFileException`❗❗
> 

```bash
# 1) modMathFunc
jar --create --file=MyServiceBuild/jars/modMathFunc.jar -C MyServiceBuild/modMathFuncs .

# 2) userfuncs
jar --create --file=MyServiceBuild/jars/userfuncs.jar -C MyServiceBuild/userfuncs .

# 3) userfuncsimpl
jar --create --file=MyServiceBuild/jars/userfuncsimpl.jar -C MyServiceBuild/userfuncsimpl .

# 4) modAppDemo (главный модуль с запуском)
jar --create --file=MyServiceBuild/jars/modAppDemo.jar --main-class=org.appstart.ModAppDemo -C MyServiceBuild/modAppDemo .
```
👉 Здесь:
- `--create` - сообщает jar о необходимости ==создания нового файла JAR==
	
- `--file=...` - указывается ==имя нового файла JAR==
    
- `-C MyServiceBuild/<module>` - ==откуда брать файлы== (`module-info.class` + классы)
	
- `--main-class` - ==Класс, содержащий метод main()==, указывается ИМЯ ПАКЕТА класса
	
- `.` - означает "все содержимое папки".

## 📙 2. Связывание JAR файлов
```bash
jlink 
	--launcher MyJarScripts=modAppDemo/org.appstart.ModAppDemo 
	--module-path "%JAVA_HOME%"\jmods;MyServiceBuild/jars 
	--add-modules modAppDemo 
	--output mytestlinkjar
```

`--module-path "%JAVA_HOME%"\jmods;MyServiceBuild/jars ` указываем папку где лежат файлы `.jar`

# 📕 3. Файлы JMOD
Файлы `JMOD` Инструмент `jlink` также способен связывать файлы в более новом формате `JMOD`, появившемся в версии JDK 9. 

Файлы JMOD могут включать эле менты, которые неприменимы к файлу JAR, ==и создаются посредством нового инструмента `jmod`==. 

Хотя в большинстве приложений по-прежнему будут использоваться модульные файлы JAR, файлы JMOD окажутся полезными в специализированных ситуациях

> [!NOTE] 🧠Интересный факт `jpackage`
> Инструмент `jlink` также может быть задействован недавно добавленным инструментом `jрасkage`, который позволяет создавать собственное устанавливаемое приложение.

# 📕🧠 4. **Кратко об уровнях и автоматических модулях**

**Содержание:** Две продвинутые концепции, полезные в основном при миграции legacy-кода или создании сложных систем.

- **Уровни модулей**: Позволяют связывать модули с разными загрузчиками классов.
    
- **Автоматические модули**: Позволяют включить обычный (немаодульный) JAR-файл в модульный путь. Имя модуля генерируется автоматически из имени JAR-файла. Это временное решение для постепенного перехода на модули.
