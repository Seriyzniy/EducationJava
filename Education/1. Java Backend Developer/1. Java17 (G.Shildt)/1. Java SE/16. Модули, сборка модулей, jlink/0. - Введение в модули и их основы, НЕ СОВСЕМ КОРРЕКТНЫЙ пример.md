В версии JDK 9 появилось новое и важное средство, называемое модулями. 

> [!NOTE] Модуль - группа пакетов/ресурсов
> Модуль — это именованная коллекция (группа) пакетов/ресурсов с явным описанием зависимостей и экспортируемых пакетов. 
> 
> 💡ФАЙЛ МОДУЛЯ называется - ТОЛЬКО `module-info.java`, а внутри любое имя (принято НАЗВАНИЕ ПАКЕТА, в котором он расположен БЕЗ ПОЛНОГО ПУТИ)
> 
> Модули дают явную инкапсуляцию между частями большой системы, позволяют контролировать видимость пакетов, сокращают набор используемых API

Как правило, модули наиболее полезны в крупных приложениях, поскольку они помогают сократить сложность управления, часто связанную с большой программной системой.


> [!NOTE] 🧠 myPackage.com => com.myPackage 🧠
> На момент написания книги рекомендовалось применять метод обратных доменных имен, при котором обратное имя до мена, "владеющего" проектом, используется в качестве префикса для модуля. 
> 
> Скажем, в проекте, ассоциированном с `herbschildt.com`, для префикса модуля будет применяться `com.herbschildt`. (То же самое касается имен паке тов.)

## ⭐⭐КЛЮЧЕВЫЕ МОМЕНТЫ⭐⭐

> [!NOTE] ❗ВАЖНО❗
> 1. Файл модуля ВСЕГДА называется `module-info.java`
> 2. Имя самого модуля BEST PRACTICES = НАЗВАНИЕ ПАКЕТА в которой он лежит
> 3. ОДИН МОДУЛЬ - ОДИН ОТДЕЛЬНЫЙ ПРОЕКТ
> 4. Имена пакетов разных модулей должны быть разные
> 5. Для ВИДИМОСТИ двух модулей, зависимый модуль, должен настроить переменную `Module path` в настройках проекта
> 6. В путях использовать ОБРАТНЫЕ СЛЕШИ `/`, а НЕ ПРЯМЫЕ `\`
> 7. При компиляции в качестве целевых папок для СКОМПИЛИРОВАННЫХ модулей, ЛУЧШЕ ВЫБИРАТЬ ИМЕНА = ФАКТИЧЕСКОЕ ИМЯ МОДУЛЯ в файле `module-info`
> 8. Ключи `--module-path` `-m` РАБОТАЮТ С МОДУЛЯМИ, а не папками в которых они лежат, поэтому ПРИ СБОРКЕ УКАЗЫВАТЬ ФАКТИЧЕСКОЕ ИМЯ МОДУЛЯ, а не имя папки в которой лежит модуль

# 📕 1. Основы модулей
Модуль представляет собой группу пакетов и ресурсов, на которые можно коллективно ссылаться по имени модуля. 

В объявлении модуля указывается имя модуля и определяется отношение модуля и его пакетов с другими модулями. 

==Объявления модулей записываются в виде операторов== в файле исходного кода Java и поддерживаются ==несколькими ключевыми словами==, связанными с модулями:
```java
exports   provides    transitive
module    requires    uses
open      to          with
opens
```

> [!NOTE] 🛑 Ключевые слова только для модулей! 🛑
> Важно понимать, что `перечисленные выше ключевые слова распознаются как ключевые слова только в контексте объявления модуля.` В других ситуа циях они интерпретируются как идентификаторы (имя переменной).

> [!NOTE] 💡Модуль объявляется как файл `.java`
> Объявление модуля содержится в файле по имени `module-info.java`, т.е. модуль определяется в файле исходною кода `Java`. 
> 
> Файл `module-info.java` затем компилируется с помощью `javac` в файл класса и известен как его ==*дескриптор модуля*==. 
> (Компилируется в `module-info.class`) 
> 
> ⭐Файл `module-info.java` должен содержать `только определение модуля`, но не другие виды объявлений.

## Структура модуля
```java
module имя_модуля {
    // определение модуля
}
```

# 📕 2. Простой пример модуля
В основе возможностей модуля лежат две ключевые особенности.:
1. 🔶Первая из них - способность модуля сообщать о том, что ему требуется другой модуль.
	💡==Отношение зависимости задается с помощью оператора== `requires`. По умолчанию наличие необходимого модуля проверяется как на этапе компиляции, так и во время выполнения

2. 🔶Второй ключевой особенностью является ==способность модуля контролировать, какие из ero пакетов доступны другому модулю==, что достигается с помощью ключевого слова `exports`
	💡Открытые и защищенные типы внутри пакета доступны другим модулям только в том случае, если они явно экспортированы.

### 🟣Написание исходников 
После настройки каталогов можете заняться созданием файлов исходного кода приложения.

❗❗В текущем примере будут использоваться четыре файла исходного кода.

🔶Содержимое первого файла, `SimpleMathFuncs.java`
```java
// Простые математические функции.
package com.mathfuncs;

public class SimpleMathFuncs {
	public static boolean isFactor(int a, int b) {
        if((b % a) == 0) return true;
        return false;
    }
	
    // Возвратить наименьший положительный делитель, общий для a и b.
    public static int lcf(int a, int b) {
        // Разложить на множители, используя положительные значения.
        a = Math.abs(a);
        b = Math.abs(b);
        int min = a < b ? a : b;
        
        for(int i = 2; i <= min/2; i++) {
            if(isFactor(i, a) && isFactor(i, b))
                return i;
        }
        return 1;
    }
    
    // Возвратить наибольший положительный делитель, общий для a и b.
    public static int gcf(int a, int b) {
        // Разложить на множители, используя положительные значения.
        a = Math.abs(a);
        b = Math.abs(b);
        int min = a < b ? a : b;
        
        for(int i = min/2; i >= 2; i--) {
            if(isFactor(i, a) && isFactor(i, b))
                return i;
        }
        return 1;
    }
}
```

🔶Далее показано содержимое второго файла исходного кода, `ModAppDemo.java`, ==в котором задействованы методы класса== `SimpleMathFuncs`.
```java
// Простой пример приложения, основанного на модулях.
package org.appstart;

import com.mathfuncs.SimpleMathFuncs;

public class ModAppDemo {
	public static void main(String[] args) {

        if (SimpleMathFuncs.isFactor(2, 10)) {
            System.out.println("2 является делителем 10");
        }

        System.out.println(
	        "Наименьший общий делитель для 35 и 105 равен " + 
	        SimpleMathFuncs.lcf(35, 105));

        System.out.println(
	        "Наибольший общий делитель для 35 и 105 равен " +
	        SimpleMathFuncs.gcf(35, 105));
    }
}
```

>  Для этого необходимо 
>  - настроить файлы `module-info.java` для обоих проектов, 
>  - а также для "зависимого проекта" добавить переменную `Module path`, в настройках проекта

### 🟣Описание файлов модулей 
Затем для каждого модуля понадобится ==добавить файлы== `module-info.java,` которые ==содержат определения модулей==.

🔶Первым делом добавьте файл, определяющий модуль `modMathFuncs`, и поместим его по пути `src`:
```java
// Определение модуля математических функций.
module modMathFuncs {
	// ❗✅Экспортировать пакет com.mathfuncs.
    // Полное имя пакета ИЗ ФАЙЛА С КЛАССОМ
	exports com.mathfuncs;
}
```
> [!NOTE] Модуль `modMathFuncs` ==ЭКСПОРТИРУЕТ ПАКЕТ== `com.mathfuncs`, ==что делает его доступным для других== модулей.
> 
> ⭐⭐Должен находится в КОРНЕ ВСЕХ ПАКЕТОВ⭐⭐, чтобы было удобно экспортировать всевозможные пакеты

🔶Наконец, ниже представлено содержимое файла `module-info.java` для модуля `org.start`. Поместим его по пути `src\`
❗❗❗Обратите внимание, что `modAppDemo` требует модуля `modMathFuncs`.
```java
// Определение модуля главного приложения.
module modAppDemo {
	// ❗✅ Требует 💡ИМЯ💡 модуля appfuncs.
	requires modMathFuncs;
}
```

# 2. Компиляция и запуск первого примера модуля

> [!NOTE] ⭐⭐СНАЧАЛА КОМПИЛИМ ИСХОДНИКИ, затем `module-info.java`⭐⭐
> 1) В пути ==использовать ОБРАТНЫЕ СЛЕШИ== `/` а не прямые `\`
> 2) При компиляции СНАЧАЛА НЕЗАВИСИМЫЕ МОДУЛИ, а потом зависимые

В версии JDK 9 компилятор `javac` был обновлен с целью поддержки модулей. 

Таким образом, подобно всем другим программам на Java программы на основе модулей компилируются с применением `javac`

Процесс прост, а основное отличие заключается в том, что ==обычно будет явно указываться путь к модулю==, который `сообщает компилятору местоположение для скомпилированных файлов`.

> [!NOTE] `javac` и ключи утилиты
>  `-d <директория>` - Это ключ «**куда положить скомпилированные `.class`-файлы**».
> 
`--module-path <директория>` - Это ключ «**где искать модули, от которых зависит текущий**».

## Компиляция пакета `com.mathfuncs`
```bash
javac -d MyModulesBuild/modfuncs MyModuleFuncs/src/com/mathfuncs/SimpleMathFuncs.java MyModuleFuncs/src/module-info.java
```

## Компиляция пакета `org.appstart` 
Здесь указываем `--module-path MyModulesBuild/modfuncs` а не просто `--module-path MyModulesBuild`, так как имя папки с модулем (`modfuncs`), отличается от фактического имени модуля в `module-info.java` (`modMathFuncs`)
```bash
javac --module-path MyModulesBuild/modfuncs -d MyModulesBuild/modappstart MyModuleAppStart/src/org/appstart/ModAppDemo.java MyModuleAppStart/src/module-info.java
```

## Запуск
> [!NOTE] ⭐⭐ `java` и ключи утилиты ⭐⭐
> `--module-path <директория>` - директория, где хранятся модули. Java просматривает вложенные директории и ищет модуль. ПОЭТОМУ `имяпапки`, должно совпадать с фактическим `именем_модуля` из файла `module-info`
> (Если разные, то тогда можно указать путь прямиком до главного модуля)
> 
> `-m` - ИМЯ_МОДУЛЯ из файла `module-info.java` ,А НЕ ПАПКИ где лежит структура `module-info.class` + `package`

```bash
java --module-path MyModulesBuild -m modAppDemo/org.appstart.ModAppDemo
```
