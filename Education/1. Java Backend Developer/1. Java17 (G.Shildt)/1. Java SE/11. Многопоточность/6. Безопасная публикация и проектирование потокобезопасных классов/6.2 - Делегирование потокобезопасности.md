# 📕 1. Суть делегирования потокобезопасности

Когда ты используешь во внешнем классе **внутренний потокобезопасный объект** (например, `ConcurrentHashMap`, `BlockingQueue`, `AtomicInteger` и т.д.), то:

- если **внешний класс просто передаёт вызовы внутрь** (не добавляет логики между вызовами), то внешнему классу **не нужно** добавлять дополнительную синхронизацию. Потокобезопасность "делегируется".
```java
class SafeCounter {
    private final AtomicInteger count = new AtomicInteger(0);

    public void increment() {
        count.incrementAndGet(); // потокобезопасно
    }

    public int get() {
        return count.get(); // потокобезопасно
    }
}
```

👉 Здесь `SafeCounter` сам по себе потокобезопасен, ПОТОМУ ЧТО все его методы **делегируют работу** потокобезопасному `AtomicInteger`.

# 📕2. Когда нужна дополнительная синхронизация

⚠️ Если во внешнем классе появляется **СОСТОВНАЯ ОПЕРАЦИЯ** (несколько вызовов к внутреннему объекту, которые должны быть атомарными вместе), то **НУЖНО ДОБАВИТЬ СИНХРОНИЗАЦИЮ во внешнем классе**, потому что внутренний объект гарантирует потокобезопасность только для отдельных операций.

```java
class UnsafeCache<K, V> {
    private final ConcurrentHashMap<K, V> map = new ConcurrentHashMap<>();

    // хотим "добавить, если ключа нет"
    public void putIfAbsent(K key, V value) {
        if (!map.containsKey(key)) {   // шаг 1
            map.put(key, value);       // шаг 2
        }
    }
}
```

Здесь два потока могут одновременно пройти проверку и положить разные значения.

🔧 Решение — использовать уже готовый потокобезопасный метод:
```java
map.putIfAbsent(key, value);
```

***
# 📌 Правило:

- Если наружный класс просто **оборачивает потокобезопасный класс без логики** → **синхронизация не нужна**.
    
- Если ты **комбинируешь несколько операций** — нужно либо использовать атомарный метод API, либо синхронизировать самому.
