# 1. **Против Взаимоблокировок (Deadlock)**

| Проблема                                                   | Решение (Best Practice)                      | Суть решения                                                                                                                                                              |
| ---------------------------------------------------------- | -------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Разный порядок блокировок** (Lock-Ordering Deadlock)     | **Глобально согласованный порядок захвата.** | Всегда захватывайте блокировки в одном и том же порядке во всей программе. <br><br>Например, сначала блокируйте объект с меньшим `System.identityHashCode()`.             |
| **Динамический порядок** (зависит от входных данных)       | **Индуцирование порядка.**                   | Сравнивайте хэш-коды объектов. <br><br>При равенстве используйте третью, "гардеробную" блокировку (`tieLock`) для устранения неопределенности.                            |
| **Взаимодействие объектов** (вызов чужого метода под lock) | **Открытые вызовы (Open Call).**             | ==**Никогда не вызывайте чужой метод, удерживая блокировку**==. <br><br>Сужайте `synchronized`-блоки до минимума, отпускайте lock _перед_ вызовом методов других классов. |
| **Блокировка ресурсов** (пулы соединений, семафоры)        | **Упорядочивание и таймауты.**               | Запрашивайте ресурсы в согласованном порядке. <br><br>Используйте `tryLock()` с таймаутом для возможности восстановления.                                                 |

# 2. **Против Голодания (Starvation) и Плохой Отзывчивости**

| Проблема                                | Решение (Best Practice)              | Суть решения                                                                                                                      |
| --------------------------------------- | ------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------- |
| **Неправильные приоритеты потоков**     | **Избегайте изменения приоритетов.** | Используйте `Thread.NORM_PRIORITY` для всех потоков. ==Настройка приоритетов непереносима== и ведет к непредсказуемому поведению. |
| **Долгие операции в UI/Service потоке** | **Вынос в фоновые потоки.**          | Используйте `ExecutorService` и `SwingWorker` (для GUI) для выполнения долгих задач, чтобы не блокировать поток событий.          |
| **Долгие критические секции**           | **Дробление задач.**                 | Дробите длинные операции на части и отпускайте блокировки между ними. Делайте критические секции как можно короче.                |
**Ключевой вывод:** Долгая работа в критической секции или потоке, отвечающем за отклик, — главный враг живучести. Делите и властвуйте.

# 3. **Против Динамической Взаимоблокировки (Livelock)**

| Проблема                                                                 | Решение (Best Practice)                    | Суть решения                                                                                                                                               |
| ------------------------------------------------------------------------ | ------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------- |
| ==**Бесполезное повторение операции**== (напр., "отравленное сообщение") | **Рандомизация и экспоненциальный откат.** | Добавляйте случайную задержку (`Thread.sleep(randomDelay)`) между повторами. При каждой новой ошибке увеличивайте задержку в 2 раза (exponential backoff). |
**Ключевой вывод:** Если операция повторяется, она не должна делать этого в жестком цикле. Случайность и рост задержки — ключ к выходу из livelock.

# 4. **Инструменты и Тактики Диагностики**

| Задача                                  | Инструмент/Метод                                         | Комментарий                                                                                                             |
| --------------------------------------- | -------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------- |
| **Экстренная диагностика в продакшене** | **Дамп потоков** (`jstack <pid>`, `kill -3`, JVisualVM). | **Первый и главный инструмент.** <br><br>JVM сама найдет и укажет на deadlock'и в стандартном выводе.                   |
| **Программная проверка на deadlock**    | **`ThreadMXBean.findDeadlockedThreads()`**               | Позволяет встроить мониторинг deadlock'ов прямо в приложение.                                                           |
| **Предотвращение и восстановление**     | **`ReentrantLock.tryLock(timeout)`**                     | ==Не используйте бесконечное ожидание==. Таймаут дает шанс освободить ресурсы, залогировать ошибку и повторить попытку. |
| **Поиск проблем в коде**                | **Аудит кода и ревью.**                                  | Поиск всех мест с вложенными блокировками и вызовов "чужих" методов под lock.                                           |
**Ключевой вывод:** Ваша первая линия защиты — это профилактика (Open Calls, порядок блокировок). Вторая линия — это возможность восстановления (таймауты). Дамп потоков — это ваш скальпель для вскрытия проблемы уже после ее возникновения.

# 🧠 Итоговая памятка разработчика

1. **Пишите код с Open Calls.** Это самое важное правило.
    
2. **Всегда захватывайте блокировки в одинаковом порядке.**
    
3. **Избегайте `synchronized` методов** — используйте `synchronized` _блоки_ минимальной длины.
    
4. **Для долгих операций используйте `tryLock` с таймаутом.**
    
5. **Не меняйте приоритеты потоков.**
    
6. **Не блокируйте UI/сервисные потоки** — выносите работу в `Executor`.
    
7. **При повторах используйте задержки и экспоненциальный откат.**
    
8. **Знайте, как снять и прочитать дамп потоков.** Это must-have навык.
