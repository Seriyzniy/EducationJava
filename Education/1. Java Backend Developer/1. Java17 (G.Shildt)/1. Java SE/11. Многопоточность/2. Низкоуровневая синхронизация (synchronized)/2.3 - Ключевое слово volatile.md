`volatile` — это ==**модификатор переменной**==, который говорит JVM:

> [!NOTE] Volotile
> "==НЕ кэшируй это значение В ЛОКАЛЬНОМ ПОТОКЕ== — всегда читай и пиши напрямую в основную (общую) память."
> 
> и **ЗАПРЕЩАЕТ ПЕРЕСОРТИРОВКУ инструкций**

Так как у каждого ядра процессора есть свой кеш, то туда может записаться значение часто-используемой переменной, для ускорения работы
![[Pasted image 20250629132753.png]]

В результате, если два потока часто используют(изменяют) одну и ту же переменную, то процессор может закешировать значение этой переменной в кеш. 

В результате первый поток, может не увидеть изменения переменной, которые произошли во втором потоке

# 📕1. Что гарантирует `volatile`:

1. **Гарантия видимости** — если один поток изменил `volatile` переменную, **другие потоки увидят новое значение сразу**.

## ❌Пример БЕЗ `volatile`
```java
class Shared {
    boolean running = true;

    void run() {
        while (running) {
            // бесконечный цикл — никогда не выйдет!
        }
        System.out.println("Stopped");
    }

    void stop() {
        running = false;
    }
}
```
Поток может **не увидеть `running = false`**, потому что значение кэшировалось в CPU-регистре или в L1-кэше.

## ✅ С `volatile`:
```java
volatile boolean running = true;
```
Теперь, при каждом чтении переменной `running`, ==поток обращается к **главной памяти**==, и изменения становятся **видимы другим потокам немедленно**.

***
# 📕 2. Что **НЕ делает** `volatile`

1. ==**НЕ делает операции атомарными!**==  
    Пример: `counter++` не становится потокобезопасным.
```java
volatile int counter = 0;

void increment() {
    counter++; // НЕ атомарно: читается -> инкрементируется -> записывается
}
```
И даже если `count` — `volatile`, **всё равно возможна гонка потоков** (race condition), потому что между чтением и записью данные могут быть изменены другим потоком.

2. ❌ ==**НЕ блокирует доступ другим потокам**==
Если один поток записывает в `volatile`-переменную, **другой поток может в это же время читать/записывать**. То есть:
> `volatile` **не синхронизирует потоки** и **не создаёт критическую секцию**.

***
# 💡 Модель памяти и volatile 💡

`volatile` встраивает **memory barriers**:
- **Before write → Store barrier** (flush всех записей до неё)
    
- **After read → Load barrier** (обновление всех кэшей после неё)

🔒 Это обеспечивает **упорядоченность и видимость**, но **не синхронизацию доступа**.
