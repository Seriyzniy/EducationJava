Как говорилось ранее, при работе с потоками, важной задачей является их синхронизация с разделяемым ресурсом. 
Для этого используется `монитор`, который есть у каждого объекта. Монитором может владеть лишь одни поток.

> [!NOTE] 🛑`Synchronized` ДАЕТ ГАРАНТИИ ВИДИМОСТИ как и `volatile`🛑
> В Java `synchronized` блоки и методы обеспечивают не только атомарность, но и **синхронизацию памяти** (memory synchronization), которая включает в себя те же гарантии видимости, что и `volatile`.


> [!NOTE] 🧠 Справедливая и Несправедливая блокировки
>  "Справедливая" (`fair`) блокировка гарантирует, что потоки получат блокировку строго в порядке очереди (FIFO). 
>  
>  "Несправедливая" (`nonfair`, по умолчанию) блокировка позволяет потоку "баржировать" (впереть себя в очередь) и захватить освободившуюся блокировку, даже если есть ждущие потоки.

Одной из причин, из-за которой баржированные блокировки работают намного лучше, чем справедливые блокировки, при интенсивной конкуренции, является то, что между возобновлением приостановленного потока и его фактическим запуском может быть значительная задержка. 

Предположим, поток `A` удерживает блокировку, а поток `B` запрашивает эту блокировку. Так как блокировка занята, поток `B` приостанавливается (suspended).

Когда поток `A` освобождает блокировку, выполнение потока `B` возобновляется (resumed), чтобы он мог повторить попытку. Тем временем, однако, если поток `C` запрашивает блокировку, есть хороший шанс того, что `C`сможет захватить блокировку, использовать ее и отпустить ещё до того, как поток `B` закончит просыпаться. 

В этом случае выигрывают все: поток `B` захватывает блокировку не позже, чем получил бы её в ином случае, поток `C` захватывает блокировку намного раньше, а пропускная способность улучшается.

### 💡 Как работает под капотом

##### JVM использует:
- **монитор (ObjectMonitor)** — скрытая структура данных.
    
- **заголовок объекта (Mark Word)** — в него временно записывается информация о блокировке.
##### При входе в `synchronized`:
1. Поток **пытается захватить монитор** объекта (или класса).
    
2. Если свободен — захватывает, продолжает выполнение.
    
3. Если занят — поток **становится в очередь ожидания**.
    
4. После выхода из блока — **монитор освобождается**, другие потоки могут войти.

# 📕1. Synchronized методы
Для захвата монитора потоком, используется ключевое слово `synchronized` рядом с методом. После захвата монитора одним потоком, другие будут ожидать

Для освобождения монитора, главный поток просто возвращает управление из синхронизированного метода (просто выходит из метода)

> [!NOTE] 🛑Захватывается объект, чей `synchronized` метод вызывается 🛑
> 

```java
public synchronized void increment() {
    // весь метод заблокирован монитором объекта this
}

Эквивалентно:
public void increment() {
    synchronized(objectRef) {
        // тело метода
    }
}
```

# 📕2. Synchronized блоки
```java
public void increment() {
    synchronized (objectRef) {
        count++;
    }
}
```
или
```java
synchronized(SomeClass.class) {
    // для синхронизации по классу
}
```

# ⚠️Подводные камни

## 🔸1. Синхронизация по нестабильному объекту
```java
synchronized(someString) // ПЛОХО: строки — интернированы!
```
 **Это опасная практика**, потому что строки в Java **интернированы** (кэшируются в пуле строк), что может привести к неожиданным блокировкам и дедлокам.
```java
String lock1 = "lock";
String lock2 = "lock";

System.out.println(lock1 == lock2); // true (один и тот же объект!)
```
Если два разных потока сделают `synchronized(lock1)` и `synchronized(lock2)`, они **будут блокировать один и тот же монитор**, хотя, возможно, программист ожидал разные блокировки.

## 🔸2. Потеря производительности
`Synchronized` может замедлить работу, если критическая секция слишком большая.

## 🔸3. Возможность **взаимоблокировок (deadlock)**:
Допустим, у нас есть **два потока** (`Thread1` и `Thread2`), которые работают так:

 **Thread1:**
```java
synchronized(obj1) {      // 1. Захватывает obj1
    synchronized(obj2) {  // 3. Ждёт, пока Thread2 отпустит obj2
        // Код...
    }
}
```

 **Thread2:**
```java
synchronized(obj2) {      // 2. Захватывает obj2
    synchronized(obj1) {  // 4. Ждёт, пока Thread1 отпустит obj1
        // Код...
    }
}
```

# 💡 **Условия возникновения deadlock** (Условия **Коффмана**)

==Для deadlock необходимо **все 4 условия**== (известные как **условия Коффмана**):

1. **Взаимное исключение** (Mutual Exclusion)
    - Ресурсы (`obj1`, `obj2`) могут использоваться только одним потоком за раз.
        
2. **Удержание и ожидание** (Hold and Wait)
    - Поток удерживает один ресурс и ждёт другой.
        
3. **Отсутствие вытеснения** (No Preemption)
    - Ресурс нельзя отобрать у потока, он должен освободить его сам.
        
4. **Круговая зависимость** (Circular Wait)
    - Потоки образуют кольцо ожидания (Thread1 ждёт Thread2, Thread2 ждёт Thread1).
        

Если **нарушить хотя бы одно условие**, deadlock **не произойдёт**.

## **Решение**:
- Всегда блокируйте объекты **в одинаковом порядке**.
	
- Используйте `tryLock` с таймаутом.
	
- Минимизируйте вложенные `synchronized`.
