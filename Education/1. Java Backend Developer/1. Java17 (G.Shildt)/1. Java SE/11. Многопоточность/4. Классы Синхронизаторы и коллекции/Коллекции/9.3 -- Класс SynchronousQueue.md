
> [!NOTE] **`SynchronousQueue<E>`** — это потокобезопасная очередь `БЕЗ` внутреннего `БУФЕРА`
> 💡Реализуюет интерфейс `BlockingQueue<E>`.
> 
> `SynchronousQueue` часто используется в **производительно–потребительских системах**, `где задачи нужно` `передавать напрямую` без накопления.

> [!NOTE] НЕТ буфера как у обычной очереди, `ПЕРЕДАЧА РУКА-В-РУКУ`
> Главная особенность этой очереди: **в ней нет внутреннего буфера** — ==каждый `put` должен быть напрямую «схвачен» соответствующим `take`.==
> Это делает её уникальной по сравнению с `LinkedBlockingQueue` и `ArrayBlockingQueue`.

❗❗Каждая операция вставки (`put`) **блокируется**, ПОКА ДРУГОЙ ПОТОК не выполнит соответствующую операцию извлечения (`take`), и наоборот.  

Таким образом, это **точка прямого обмена данными между потоками** — элемент никогда не хранится внутри очереди.

## Конструкторы
```JAVA
// Создает SynchronousQueue с нефейрной политикой (по умолчанию)
// "Нефейрная" версия работает быстрее, так как использует стек (LIFO)
public SynchronousQueue()

// Создает SynchronousQueue с указанной политикой честности
// fair — если true, очередь работает по принципу FIFO (честная блокировка)
public SynchronousQueue(boolean fair)
```

## Методы
Класс реализует интерфейс `BlockingQueue<E>`, но с особыми ограничениями —  
==размер очереди всегда равен 0==, и нельзя использовать bulk-операции или итераторы.  

Методы, связанные с емкостью и итерацией, работают иначе, чем в других очередях.
### Добавление элементов
```java
// Пытается вставить элемент, ожидая, пока другой поток не выполнит take()
// Блокируется, пока не произойдет "передача" элемента
public void put(E e) throws InterruptedException

// Пытается вставить элемент немедленно; возвращает false, если нет ожидающего потока take()
public boolean offer(E e)

// Пытается вставить элемент, ожидая заданное время, если нет получателя
public boolean offer(E e, long timeout, TimeUnit unit) throws InterruptedException
```

### Извлечение элементов
```java
// Извлекает элемент, ожидая, пока другой поток не выполнит put()
// Блокируется, пока элемент не будет "передан"
public E take() throws InterruptedException

// Пытается извлечь элемент немедленно; возвращает null, если нет ожидающего потока put()
public E poll()

// Пытается извлечь элемент, ожидая заданное время, если нет отправителя
public E poll(long timeout, TimeUnit unit) throws InterruptedException
```

### Управление состоянием очереди
```java
// Возвращает 0 (всегда), так как очередь не хранит элементы
public int size()

// Возвращает 0, так как очередь не имеет буфера
public int remainingCapacity()

// Удаляет все элементы (не делает ничего, так как очередь всегда пуста)
public void clear()

// Возвращает false — итератор не поддерживается, так как очередь пуста
public Iterator<E> iterator()
```

### Массовые операции
```java
// Возвращает 0 — невозможно переместить элементы, так как очередь пуста
public int drainTo(Collection<? super E> c)

// Возвращает 0 — невозможно переместить элементы, так как очередь пуста
public int drainTo(Collection<? super E> c, int maxElements)
```

## 💡 Особенности и подводные камни 💡
1️⃣ ⭐⭐⭐**Нет буфера** — элементы не накапливаются, они сразу передаются от производителя к потребителю.  

2️⃣ 🟠**Методы `put()` и `take()` блокируются**, пока не произойдет парная операция.  

3️⃣ 🟢**`offer()` и `poll()` немедленно возвращают** `false` или `null`, если нет встречной операции.  

4️⃣ **Параметр `fair`**:
- `false` (по умолчанию) — используется стек LIFO (быстрее, меньше синхронизаций);
- `true` — используется очередь FIFO (честная политика, но ниже производительность).  

5️⃣ **size() всегда возвращает 0**, даже если поток ожидает пару.  

6️⃣ **Не допускает `null`** — попытка вставки `null` вызывает `NullPointerException`.  

7️⃣ **drainTo() и iterator() бессмысленны** — очередь ничего не хранит.  

8️⃣ Возможен **взаимный блок (deadlock)**, если оба потока одновременно ждут встречной операции.

## 💎 Best Practices 💎

✅ 🧠Используйте `SynchronousQueue` для **небуферизованной передачи данных** между потоками (например, при балансировке задач).  

✅ Хорошо подходит для **динамических пулов потоков** (`newCachedThreadPool()`), где поток создается под каждую задачу.  

✅ Используйте **`fair = true`**, если нужно гарантировать строгий порядок очередности между потоками.  

✅ Не применяйте `SynchronousQueue` там, где нужно хранить несколько элементов — используйте `ArrayBlockingQueue` или `LinkedBlockingQueue`.  

✅ Всегда проектируйте обмен с учетом возможности блокировки — `put()` и `take()` могут ждать бесконечно.

## Пример
```java
import java.util.concurrent.*;

public class SynchronousQueueExample {
    public static void main(String[] args) {
        SynchronousQueue<String> queue = new SynchronousQueue<>();

        Thread producer = new Thread(() -> {
            try {
                for (int i = 1; i <= 3; i++) {
                    System.out.println("Producer waiting to transfer: Task-" + i);
                    queue.put("Task-" + i);
                    System.out.println("Produced: Task-" + i);
                }
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        });

        Thread consumer = new Thread(() -> {
            try {
                while (true) {
                    Thread.sleep(1000); // имитация обработки
                    String task = queue.take();
                    System.out.println("Consumed: " + task);
                }
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        });

        producer.start();
        consumer.start();
    }
}

```
