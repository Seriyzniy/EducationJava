
> [!NOTE] Класс **`LinkedBlockingQueue`** — это `потокобезопасная очередь` с блокировкой,
> Реализует интерфейсы `BlockingQueue<E>`, `Queue<E>`, `Collection<E>`, `Iterable<E>` и `Serializable`

Она ==основана на **СВЯЗНОМ СПИСКЕ** ==и используется для обмена задачами между потоками (например, в пулах потоков).  

Поддерживает **необязательное ограничение ёмкости**, что позволяет контролировать количество элементов в очереди и предотвращать переполнение памяти.

## Конструкторы
```JAVA
// Создает неограниченную очередь (размер по умолчанию — Integer.MAX_VALUE)
public LinkedBlockingQueue() 

// Создает очередь с указанной максимальной ёмкостью
// capacity — максимальное количество элементов в очереди (должно быть > 0)
public LinkedBlockingQueue(int capacity)

// Создает очередь, содержащую элементы из указанной коллекции
// c — коллекция, элементы которой добавляются в очередь
public LinkedBlockingQueue(Collection<? extends E> c)
```

## Методы

Класс **наследует методы** от `AbstractQueue<E>`, `BlockingQueue<E>` и `Collection<E>`,  
такие как `add()`, `remove()`, `size()`, `iterator()`, `clear()`, `contains()` и т. д.  
Ниже приведены **основные публичные методы**, характерные именно для `LinkedBlockingQueue`.

### Добавление и вставка элементов
```java
// Добавляет элемент, если есть место; иначе выбрасывает IllegalStateException
public boolean add(E e)

// Добавляет элемент, возвращает true, если удалось; иначе false
public boolean offer(E e)

// Добавляет элемент, ожидая освобождения места при необходимости
// e — элемент для добавления
// timeout — время ожидания
// unit — единицы времени
public boolean offer(E e, long timeout, TimeUnit unit) throws InterruptedException

// Блокирующе вставляет элемент в очередь, ожидая при необходимости
public void put(E e) throws InterruptedException
```

### Извлечение и удаление элементов
```java
// Удаляет и возвращает элемент из головы очереди, ожидая при необходимости
public E take() throws InterruptedException

// Удаляет и возвращает элемент из головы очереди, 
// либо null, если очередь пуста
public E poll()

// Удаляет и возвращает элемент из головы очереди, ожидая заданное время
public E poll(long timeout, TimeUnit unit) throws InterruptedException

// Возвращает элемент из головы очереди, не удаляя его, либо null, если пуста
public E peek()
```

### Методы для управления состоянием очереди
```java
// Возвращает текущий размер очереди
public int size()

// Возвращает количество доступных "свободных слотов"
public int remainingCapacity()

// Удаляет все элементы из очереди
public void clear()
```

### Массовые операции и потокобезопасность
```java
// Перемещает все элементы из очереди в указанную коллекцию
// c — коллекция, в которую перемещаются элементы
public int drainTo(Collection<? super E> c)

// Перемещает не более maxElements элементов из очереди в указанную коллекцию
public int drainTo(Collection<? super E> c, int maxElements)
```

### Итераторы и представление
```java
// Возвращает итератор по элементам очереди (не гарантирует моментальную консистентность)
public Iterator<E> iterator()

// Возвращает массив, содержащий элементы очереди
public Object[] toArray()

// Возвращает массив указанного типа, содержащий элементы очереди
public <T> T[] toArray(T[] a)
```

## 💡 Особенности и подводные камни 💡

1️⃣ ❗❗❗**Блокировка на двух замках** — используется два `ReentrantLock`: 
- один для ==операций вставки==, 
- другой ==для извлечения==, 

что уменьшает конкуренцию между потоками.  

2️⃣ **Неограниченная очередь** (`new LinkedBlockingQueue()`) может привести к переполнению памяти, если производитель быстрее потребителя.  

3️⃣ ❗❗❗**Методы `put()` и `take()` ==блокируются==**, если очередь соответственно полна или пуста. 

4️⃣ ❗❗❗**Методы `offer()` и `poll()` ==НЕ блокируются==** (или блокируются ограниченное время, если передан timeout).  

5️⃣ **Итератор слабосогласованный (weakly consistent)** — не бросает `ConcurrentModificationException`, но может не отражать все актуальные изменения.  

6️⃣ **Отдельные блокировки** для головы и хвоста обеспечивают более высокую производительность, чем у полностью синхронизированных очередей.

## 💎 Best Practices 💎

✅ Используйте **ограниченную ёмкость** (`new LinkedBlockingQueue<>(capacity)` в продакшене, чтобы избежать `OutOfMemoryError`.  

✅ Применяйте **`take()`/`put()`** при необходимости строгой синхронизации потоков (Producer-Consumer).  

✅ Используйте **`offer(e, timeout, unit)`** для контролируемого ожидания при переполнении очереди.  

✅ Не используйте `clear()` в многопоточной среде — может нарушить ожидаемую логику работы потребителей.  

✅ Для логгирования и мониторинга используйте `remainingCapacity()` и `size()` с осторожностью — значения мгновенные и могут устареть сразу после вызова.

## Пример
```java
import java.util.concurrent.*;

public class ProducerConsumerExample {
    public static void main(String[] args) {
        BlockingQueue<String> queue = new LinkedBlockingQueue<>(3);

        // Поток-производитель
        Thread producer = new Thread(() -> {
            try {
                for (int i = 1; i <= 5; i++) {
                    queue.put("Task-" + i);
                    System.out.println("Produced: Task-" + i);
                }
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        });

        // Поток-потребитель
        Thread consumer = new Thread(() -> {
            try {
                while (true) {
                    String task = queue.take();
                    System.out.println("Consumed: " + task);
                    Thread.sleep(500);
                }
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        });

        producer.start();
        consumer.start();
    }
}
```
