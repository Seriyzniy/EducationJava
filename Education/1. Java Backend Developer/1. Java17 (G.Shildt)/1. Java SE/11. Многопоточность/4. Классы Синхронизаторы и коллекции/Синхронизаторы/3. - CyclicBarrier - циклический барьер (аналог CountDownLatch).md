
> [!NOTE] **CyclicBarrier** — это синхронизатор из пакета `java.util.concurrent`,  позволяющий `группе потоков` `ожидать друг друга` `в определенной точке` (барьере).
> После того как все потоки достигли барьера, они могут продолжить выполнение.  
> 
> ⭐⭐Ключевое отличие от `CountDownLatch`: **CyclicBarrier можно использовать повторно**, после срабатывания барьера он автоматически сбрасывается
> 
> Часто используется для координации потоков в циклических вычислениях или параллельных этапах.
> 
> Реализует интерфейс: `java.io.Serializable`.  

## Конструкторы
```java
// Создает барьер, ожидающий заданное количество участников.
// parties — количество потоков, которые должны достичь барьера, чтобы он открылся.
public CyclicBarrier(int parties)

// Создает барьер с заданным количеством участников и Runnable-действием,
// которое выполняется один раз, когда все потоки достигнут барьера.
public CyclicBarrier(int parties, Runnable barrierAction)
```

## Методы
### Основные операции
```java
// Ожидает, пока все участники не достигнут барьера.
// Если текущий поток — последний участник, выполняется barrierAction (если он задан),
// и все участники освобождаются.
// Возвращает индекс потока в барьере (от 0 до parties-1).
// Может выбросить InterruptedException или BrokenBarrierException.
public int await() throws InterruptedException, BrokenBarrierException

// Ожидает, пока все участники не достигнут барьера, но не дольше заданного времени.
// Возвращает индекс потока в барьере, либо выбрасывает TimeoutException.
// timeout — время ожидания.
// unit — единицы измерения.
public int await(long timeout, TimeUnit unit)
        throws InterruptedException, BrokenBarrierException, TimeoutException
```

### Информация о состоянии барьера
```java
// Возвращает количество участников, которые должны достичь барьера.
public int getParties()

// Возвращает количество участников, которые уже достигли барьера.
public int getNumberWaiting()

// Проверяет, разорван ли барьер.
public boolean isBroken()

// Сбрасывает барьер в исходное состояние,
// прерывая все ожидающие потоки.
public void reset()
```

---
## 💡 Особенности и подводные камни 💡

1️⃣ ⭐⭐**Повторно используемый** — после того как все участники достигли барьера, он автоматически сбрасывается. 

2️⃣ ⭐⭐Может иметь **==barrierAction==** — задаваемый Runnable, ==выполняющийся один раз при достижении барьера== всеми участниками.  

3️⃣ ❗❗❗Если один из потоков прерывается или не успевает достичь барьера (timeout) — барьер **разрушается (broken)**, и другие потоки получают `BrokenBarrierException`.  

4️⃣ ❗❗`await()` возвращает индекс участника — полезно для определения порядка потоков при продолжении работы.  

5️⃣ Барьер не гарантирует порядок освобождения потоков.  

6️⃣ Используется для **синхронизации циклов вычислений** и **параллельных этапов обработки**.  

7️⃣ Не подходит для одиночных событий — для этого лучше `CountDownLatch`.

---
## 💎 Best Practices 💎
✅ Используйте, когда нужно синхронизировать группу потоков **на каждой итерации цикла**.  

✅ 💡barrierAction подходит для сбора промежуточных результатов или логирования.  

✅ 💡Проверяйте `isBroken()` перед использованием, чтобы обработать сбой барьера.  

✅ Не используйте слишком малое время ожидания, если потоки выполняют продолжительные задачи.  

✅ Если один поток может "зависнуть", планируйте обработку `TimeoutException`

## Пример
```java
CyclicBarrier barrier = new CyclicBarrier(3, () ->
	System.out.println("Все прибыли, начинаем выполнение!")
);

for (int i = 1; i <= 3; i++) {
	int id = i;
	new Thread(() -> {
		System.out.println("Поток " + id + " ждёт у барьера");
		try {
			Thread.sleep(id * 1000);
			barrier.await(); // ждет других
			System.out.println("Поток " + id + " прошёл барьер");
		} catch (Exception e) { e.printStackTrace(); }
	}).start();
}

//Вывод
Поток 3 ждёт у барьера
Поток 2 ждёт у барьера
Поток 1 ждёт у барьера

Все прибыли, начинаем выполнение!

Поток 2 прошёл барьер
Поток 1 прошёл барьер
Поток 3 прошёл барьер
```
