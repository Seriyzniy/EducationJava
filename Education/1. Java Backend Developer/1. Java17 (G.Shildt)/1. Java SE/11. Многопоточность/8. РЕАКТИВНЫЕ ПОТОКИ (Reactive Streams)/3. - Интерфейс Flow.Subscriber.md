
> [!NOTE] `Flow.Subscriber<T>` — это интерфейс, представляющий `приёмник данных` (подписчика) в реактивном потоке.  
>
> Он `подписывается` на `Publisher`, `получает данные через метод` `onNext()`, 
> 
> 💡Управляет потоком запросов (через `Subscription`) и обрабатывает завершение или ошибки.
> 

## Пример создания
```java
// Интерфейс не имеет конструкторов
// Экземпляр создаётся путём реализации интерфейса Subscriber
Flow.Subscriber<T> subscriber = new Flow.Subscriber<>() {
    @Override
    public void onSubscribe(Flow.Subscription subscription) {
        // Реакция на начало подписки
    }
    @Override
    public void onNext(T item) {
        // Обработка поступившего элемента
    }
    @Override
    public void onError(Throwable throwable) {
        // Обработка ошибки
    }
    @Override
    public void onComplete() {
        // Действия при завершении потока данных
    }
};
```

## Методы

Интерфейс определяет **четыре метода**, которые представляют жизненный цикл подписки.
### Методы жизненного цикла подписчика
```java
// Вызывается издателем (Publisher) при подписке.
// Подписчик должен сохранить Subscription, чтобы управлять потоком данных.
void onSubscribe(Flow.Subscription subscription);
```
🟩 **Пояснение:**
- Этот метод ==вызывается **один раз**==, когда Publisher принимает подписчика.
	
- Подписчик должен ==**сохранить ссылку** на объект== `Subscription`.
    
- Через `subscription.request(n)` он управляет количеством запрашиваемых элементов.
    
- Можно сразу запросить первый элемент, например `subscription.request(1)`.
---
```java
// Вызывается при поступлении нового элемента данных от издателя.
void onNext(T item);
```
🟩 **Пояснение:**
- Метод вызывается Publisher’ом **==по одному разу на каждый элемент==**.
    
- Подписчик должен обработать элемент и при необходимости запросить следующий через `subscription.request(1)`.
    
- 💡Вызывается **только после onSubscribe()**.
    
- ❗❗❗Если вызывается после onError() или onComplete() — это нарушение контракта.
---
```java
// Вызывается, если во время передачи данных произошла ошибка.
void onError(Throwable throwable);
```
🟩 **Пояснение:**

- Завершает поток с ошибкой.
    
- После вызова `onError()` **никакие другие методы (onNext/onComplete)** не должны вызываться.
    
- Обычно используется для логирования или аварийного завершения обработки.
---
```java
// Вызывается, когда Publisher закончил передачу всех данных.
void onComplete();
```
🟩 **Пояснение:**
- Сигнализирует о **нормальном завершении потока**.
    
- После вызова `onComplete()` **никакие другие методы не вызываются**.
    
- Можно использовать для освобождения ресурсов, закрытия соединений и т.д.

---
## 💡 Особенности и подводные камни 💡

1️⃣ 💡Подписчик сам решает, **сколько элементов запрашивать**, через `subscription.request(n)`.  
Если не вызвать `request()`, то данные не будут поступать!

2️⃣ 💡Все методы (`onNext`, `onError`, `onComplete`) вызываются **в том же потоке**, где работает `Publisher`, если не предусмотрено иное.

3️⃣ Вызовы должны быть **последовательными** — `Publisher` обязан гарантировать, что `onNext()` не вызывается одновременно из разных потоков.

4️⃣ После вызова `onError()` или `onComplete()` никаких других вызовов быть не должно.

5️⃣ 💡Если в `onNext()` произойдёт исключение — это может прервать поток, если не предусмотрено восстановление.

6️⃣ Подписчик не должен запрашивать отрицательное количество элементов (`request(-1)` вызовет `IllegalArgumentException`).

---

## 💎 Best Practices 💎

✅ Сразу в `onSubscribe()` сохраняй `subscription` в поле и делай первый вызов `request(1)`.

✅ ⭐Обрабатывай `onNext()` быстро — не блокируй поток! Используй асинхронные очереди или пулы потоков для тяжёлых операций.

✅ В `onError()` всегда логируй исключения, чтобы не терять информацию о сбоях.

✅ В `onComplete()` корректно освобождай ресурсы и завершай работу.

✅ ⭐Не пытайся вызывать `onNext()` вручную — это делает только Publisher.

## Пример
```java
import java.util.concurrent.Flow;
import java.util.concurrent.SubmissionPublisher;

public class SubscriberExample {
    public static void main(String[] args) throws InterruptedException {
        SubmissionPublisher<Integer> publisher = new SubmissionPublisher<>();

        Flow.Subscriber<Integer> subscriber = new Flow.Subscriber<>() {
            private Flow.Subscription subscription;
            private int received = 0;

            @Override
            public void onSubscribe(Flow.Subscription subscription) {
                this.subscription = subscription;
                System.out.println("Подписка оформлена!");
                subscription.request(1); // Запрашиваем первый элемент
            }

            @Override
            public void onNext(Integer item) {
                System.out.println("Получено число: " + item);
                received++;
                if (received < 5) {
                    subscription.request(1); // Запрашиваем следующий
                } else {
                    subscription.cancel(); // Отменяем подписку после 5 элементов
                    System.out.println("Достигнут лимит — подписка отменена.");
                }
            }

            @Override
            public void onError(Throwable throwable) {
                System.out.println("Ошибка: " + throwable.getMessage());
            }

            @Override
            public void onComplete() {
                System.out.println("Все данные получены!");
            }
        };

        publisher.subscribe(subscriber);

        // Публикуем данные
        for (int i = 1; i <= 10; i++) {
            publisher.submit(i);
        }

        publisher.close();
        Thread.sleep(1000);
    }
}

//Вывод
Подписка оформлена!
Получено число: 1
Получено число: 2
Получено число: 3
Получено число: 4
Получено число: 5
Достигнут лимит — подписка отменена.
```
