
> [!NOTE] **Реактивные потоки (Reactive Streams)** — это `способ` организации `асинхронной передачи данных` между компонентами приложения,  
> при котором ==данные передаются== **не сразу все**, ==а **поштучно, ПО ЗАПРОСУ**==.
> 

👉 То есть ==подписчик== сам ==**регулирует скорость получения данных*==*, чтобы не "захлебнуться" от потока информации.  

Это и есть ключевая идея: **управляемый асинхронный поток данных с поддержкой “backpressure”**.
## Основная идея — _Backpressure_

_(контроль давления в потоке данных)_

Когда издатель (`Publisher`) производит данные быстрее, чем подписчик (`Subscriber`) может обработать,  
**backpressure** позволяет подписчику сказать:

> «Пока хватит, дай мне ещё только 10 элементов».


## Проблема, которую решают реактивные потоки

До появления реактивной модели существовали только два варианта:

1️⃣ **Синхронный (pull)** — потребитель сам запрашивает данные, пока источник готов их отдавать.  
 ❌ Неэффективно при большом количестве источников или сетевых операциях (ждём I/O).

2️⃣ **Асинхронный (push)** — источник сам “толкает” данные подписчику.  
 ❌ Если подписчик не успевает — переполнение буфера, потеря данных, зависание.


💥 **Reactive Streams** объединили оба подхода:  
👉 данные "толкаются" (push), но ==подписчик **сам запрашивает их количество**== (pull).


## Компоненты реактивного потока Flow

| Компонент          | Роль                                        | Основные методы                                          |
| ------------------ | ------------------------------------------- | -------------------------------------------------------- |
| **Publisher**      | Источник данных                             | `subscribe(Subscriber<? super T> s)`                     |
| **Subscriber**     | Получатель данных                           | `onSubscribe()`, `onNext()`, `onError()`, `onComplete()` |
|                    |                                             |                                                          |
| **Subscription**   | Контракт между ними (==управляет потоком==) | `request(long n)`, `cancel()`                            |
|                    |                                             |                                                          |
| **Processor<T,R>** | Промежуточный преобразователь               | объединяет Publisher и Subscriber                        |

# 📕1. Базовая схема работы стримов
Все начинается с подписки `Subscriber` на `Publisher` посредством вызова метода `Publisher.subscribe()`. 

`Publisher` использует переданный объект `Subscriber`, вызывая метод `onSubscribe()`и ==передавая в `Subscriber` объект `Subscription`==. 

Через этот объект (`Subscription`)`Subscriber` будет взаимодействовать с `Publisher`.

![[Pasted image 20251009195716.png]]

⭐⭐Теперь `Subscriber`, используя полученный `Subscription`, будет запрашивать значения у `Publisher`. 

Это важный момент, ==НЕ `Publisher` инициализирует отправку данных подписчикам когда хочет, это ПОДПИСЧИКИ ЗАПРАШИВАЮТ необходимое количество данных у `Publisher`. Таким образом реализуется контроль потока данных (Backpressure).==

![[Pasted image 20251009195836.png]]

Метод `onNext(T t)` вызывается, когда `Subcriber` запрашивает значения у `Publisher`, используя метод `Subscription.request()`. Он ==передаёт данные по одному,== но не больше, чем было запрошено подписчиком.  
  
Метод `onError()` вызывается, когда ошибка происходит на стороне `Publisher`, оповещая таким образом о проблеме `Subscriber`, передавая объект исключения.  
  
Метод `onComplete()` оповещает подписчиков, что у `Publisher` не осталось элементов для передачи. Данный метод, как и `onError()` вызывается лишь один раз.
