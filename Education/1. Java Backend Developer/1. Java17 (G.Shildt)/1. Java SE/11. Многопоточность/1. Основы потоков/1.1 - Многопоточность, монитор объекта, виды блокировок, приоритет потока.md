### Что такое поток (Thread)?

Поток — это **независимая единица исполнения**, которая выполняется параллельно с другими потоками внутри одного процесса (программы JVM).

- Все потоки ==делят **общую память (heap)**==.
    
- ==Каждый поток имеет СОБСТВЕННЫЙ **стек вызовов (stack)**.==
    
- Потоки могут выполняться **параллельно** (на многопроцессорных системах) или **псевдопараллельно** (через переключение контекста на одном ядре).

***
# 📕 1. Приоритет потоков
Приоритет потока - представляют собой целые числа, определяющие относительный приоритет одного потока над другим

Java каждый поток имеет **целочисленный приоритет**:
```java
public static final int MIN_PRIORITY = 1;
public static final int NORM_PRIORITY = 5;
public static final int MAX_PRIORITY = 10;
```


> [!NOTE] Изменение приоритета потока
> Для этого используется метод `setPriority()` класса `Thread`
> ```java
> final void setPriority(int level);
> ```
> 
> Для получения текущего приоритета используется метод `getPriority()`
> ```java
> final int getPriority();
> ```

## 💡 Что нужно знать:
- Планировщик потоков **может** использовать приоритет при выборе следующего потока, но **не обязан**.
    
- ==Java НЕ ГАРАНТИРУЕТ строгую приоритетную модель== — поведение **зависит от JVM и ОС**.
    
- На некоторых платформах **все потоки одного приоритета исполняются по кругу (round-robin)**.


==**Переключение контекста**== — это процесс, при котором **операционная система (или виртуальная машина)** **приостанавливает один поток и начинает выполнение другого**, сохраняя состояние первого.


> [!NOTE] `Переключение` может происходить по `одному из двух типов правил`:
> 
### 🔶1. **Добровольное** (cooperative)
- Поток **сам уступает** выполнение, вызывая, например:
    - `Thread.sleep()`
        
    - ⭐`Thread.yield()` _(намёк планировщику уступить ресурс)_
        

### 🔶 2. **Принудительное** (preemptive)
- ОС/планировщик **сам решает**, что поток должен быть прерван, по таким причинам:
    - время кванта (тайм-слайс) истекло;
        
    - появился поток с более высоким приоритетом;
        
    - текущий поток блокируется (`wait()`, `join()`, `I/O` и т. д.).

# 📕 2. Синхронизация
Синхронизация нужна, чтобы **одновременно только один поток мог обращаться к критической секции** — части кода, которая работает с общими ресурсами.

В Java ==синхронизация реализована через механизм **МОНИТОРОВ== (monitor locks)**.

***
## 📙 Виды блокировок

| Состояние                      | Что в Mark Word                             | Объяснение                                                 | Биты |
| ------------------------------ | ------------------------------------------- | ---------------------------------------------------------- | ---- |
| **Unlocked (не заблокирован)** | Биты указывают, что объект свободен         | Можно захватить                                            | 01   |
| **Lightweight Lock**           | Mark Word указывает на стек текущего потока | Используется CAS и т.н. lock record                        | 00   |
| **Heavyweight Lock**           | Mark Word указывает на **Monitor Object**   | ==При высокой конкуренции (создаётся структура monitor)==  | 10   |
| **Biased Locking**             | В Mark Word закодирован ID потока-владельца | Быстрая блокировка, если используется только одним потоком | 101  |
> [!NOTE] ⚠️ С версии Java 15 **Biased Locking отключена по умолчанию**.

### 🔶 1. **Biased Lock** (смещённая блокировка)

> [!NOTE] 💡**Идея**: если объект почти всегда ==захватывается одним потоком== — ==зачем делать синхронизацию==?

- При создании объект имеет `Mark Word` с флагом **biasable** (`101`).
    
- Как только **один поток захватывает `synchronized`**, `Mark Word` записывает его `Thread ID`.
    
- Повторные заходы этим потоком **не требуют CAS и системных вызовов**

### 💡Особенности 💡
✅ Очень дёшево  
❌ Если **другой поток попытается захватить этот объект**, происходит **отмена biase** и ==возможная **эскалация**.==
***
### 🔶2. **Lightweight Lock** (легковесная блокировка)

> [!NOTE] 💡Идея: "Может быть, `T2` просто немного подождёт, пока `T1` закончит?"

Если bias невозможен (например, потоки меняются), JVM пытается использовать **легковесную блокировку**:

- JVM создает **lock record в стеке потока**.
    
- Выполняется **CAS (compare-and-swap)** на `Mark Word`, заменяя его на указатель на stack lock.
    
- Если успешно — поток "захватил" объект.

##### Как работает:
1. Поток `T1` входит в `synchronized(lock)`:
    - JVM записывает **в `MarkWord` объекта ссылку на стек `T1` (LockRecord)**.
        
    - Объект "заблокирован", но **всё это локально в стеке**, без ядра.
        
2. Через пару микросекунд приходит поток `T2`:
    - Он тоже хочет зайти в `synchronized(lock)`, **пытается через CAS**.
        
    - JVM видит, что объект уже захвачен.
        
3. ==`T2` НЕ бежит сразу в операционную систему**== (как в heavyweight)!
    - Он говорит: "Я немного подожду, может `T1` быстро освободит".
        
    - JVM ==запускает **короткое ожидание== (spin)** — ==T2 крутится в цикле== (Spin-wait).
    
4. Если `T1` быстро выходит:
    - JVM возвращает старый `MarkWord` из стека,
        
    - `T2` спокойно заходит.


🟡 **Spin lock**: если другой поток тоже захочет объект, он **будет "крутиться" (spinning)** в цикле, ожидая освобождения.

✅ Быстро  
❌ Уместно **только при низком уровне конкуренции**
***
### 🔶 3. **Heavyweight Lock** (тяжёлая блокировка)

> [!NOTE] 💡 Идея: "Окей, тут очередь. Встань в хвост и жди. Будем будить по порядку."
>

Если `lightweight` не сработала (например, **слишком долго крутится spinlock**, или **слишком много потоков**) — JVM **эскалирует в heavyweight**:

- Создаёт объект `ObjectMonitor` в куче.
    
- Заменяет `Mark Word` на **указатель на `ObjectMonitor`**.
    
- Потоки блокируются через **ядро ОС** (вызов `park()` / `unpark()` или `pthread_mutex`).

##### Как работает:
1. Поток `T1` заходит в `synchronized(lock)`.
    - JVM создаёт **`ObjectMonitor` в куче**.
        
    - `MarkWord` теперь указывает на `ObjectMonitor`.
        
2. Приходит поток `T2`:
    - Понимает, что объект захвачен,
        
    - **не ждёт в цикле**,
        
    - ==**отправляется "спать" (через вызов ядра ОС== — `park()`/`wait()`), в очередь внутри `ObjectMonitor`**.
        
3. Когда `T1` выходит из блока:
    - JVM берёт **следующий поток из очереди**,
        
    - **будит его**,
        
    - тот захватывает `lock`.

✅ Гарантированно корректно  
❌ Медленно (системные вызовы, контекстные переключения)

# 📕 3. Монитор в Java

**Монитор** — это ==механизм, обеспечивающий **взаимное исключение== (mutual exclusion)** и **координацию потоков**. 

🛑Каждый ==объект в Java ассоциирован с монитором==. Его можно представить как «замок» (lock), который можно **захватить** или **освободить**. 🛑

Когда ты пишешь:
```java
synchronized (obj) {
    // критическая секция
}
```
Java:
1. **получает монитор, связанный с объектом `obj`** (внутри JVM).
    
2. Если монитор свободен, поток его захватывает и продолжает выполнение.
    
3. Если монитор уже захвачен другим потоком — текущий поток блокируется и ждёт.
    
4. Когда поток выходит из `synchronized` — **монитор освобождается**, и его может захватить следующий поток.

## 💡 Где хранится монитор? В заголовке объекта, НО ...💡 

Так как MarkWord это изменяемые данные, которые зависят от типа блокировки (указываются в младших 2-3 битах), то в случае ТЯЖЕЛОВЕСНОЙ блокировки, ==ВЕСЬ MarkWord содержи УКАЗАТЕЛЬ на структуру== `ObjectMonitor`

Когда JVM "эскалирует" блокировку до **heavyweight lock**, `Mark Word` перестаёт содержать какие-либо данные о потоке или хэш-код — **в него записывается чистый указатель на `ObjectMonitor`**, выровненный с окончанием на `10` (последние два бита указывают, что это heavy lock).

## 📙 Объект `ObjcetMonitor` 
`ObjectMonitor` — это ==**внутренняя C++-структура**==, определённая ==в исходниках HotSpot JVM== (написана на C++), которая используется для управления **тяжёлыми (heavyweight)** блокировками объектов в Java.

Когда поток не может захватить объект через **легковесный путь**, он попадает в очередь ожидания у `ObjectMonitor`.

### 📍 Где хранится `ObjectMonitor`?
Он ==НЕ ЯВЛЯЕТСЯ частью объекта Java== напрямую, ==и выделяется в **куче JVM**==, в области, называемой **monitor cache** или **monitor pool**.

🔸 В `Mark Word` объекта в этом случае хранится **указатель на соответствующий `ObjectMonitor`**, который будет использоваться для синхронизации.

### Структура `ObjectMonitor` (в общих чертах)

Вот приблизительно что содержит `ObjectMonitor` внутри:

|Поле|Назначение|
|---|---|
|`owner`|Поток, владеющий монитором|
|`EntryList`|Очередь потоков, ожидающих захвата|
|`WaitSet`|Потоки, которые вызвали `wait()`|
|`recursions`|Счётчик рекурсивных захватов|
|`object`|Указатель на объект, с которым связан монитор|
|`is_busy` / `flag`|Служебные флаги|
### Как `ObjectMonitor` используется JVM

1. Поток вызывает `synchronized` и не может получить монитор.
    
2. JVM проверяет: легковесная блокировка невозможна (например, другой поток уже захватил монитор).
    
3. JVM **создаёт (или переиспользует)** `ObjectMonitor` из пула.
    
4. Сохраняет **указатель на `ObjectMonitor` в Mark Word** объекта.
    
5. Поток переходит в очередь `EntryList` или `WaitSet`.
### 💡 Что дальше? 💡

- Как только монитор освобождается — JVM будит один из потоков из `EntryList`.
    
- Поток получает `owner`, и `recursions` обновляется, если блокировка рекурсивная.
