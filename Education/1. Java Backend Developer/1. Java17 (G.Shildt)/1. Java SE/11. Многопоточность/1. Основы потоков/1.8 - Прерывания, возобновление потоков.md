Методы `suspend()`, `resume()` и `stop()` относятся к ==**устаревшим методам управления потоками в Java**==, которые входили в старые версии API `Thread`. Эти методы **доступны, но ==использовать их строго не рекомендуется**==, потому что они **опасны и могут привести к непредсказуемому поведению программы**.

В Java прерывание потока — это **механизм кооперативной остановки работы**.

⚠️ Поток **никогда не убивается принудительно** (как в `Thread.stop()`, который давно объявлен устаревшим и опасным). 
Вместо этого другой поток может "попросить" поток завершиться через `thread.interrupt()`
# 💡 Как правильно завершать потоки:
## 🔶 Через флаг
```java
class MyThread extends Thread {
    private volatile boolean running = true;

    public void run() {
        while (running) {
            // работа
        }
    }

    public void shutdown() {
        running = false;
    }
}
```

## 🔶 Или через `interrupt()`:
```java
class MyThread extends Thread {
    public void run() {
        while (!Thread.currentThread().isInterrupted()) {
            // работа
        }
    }
    public void cancel() { interrupt(); }
}
```
→ И можно вызвать `t.interrupt()` снаружи.

> [!NOTE] `interrupt()` - запрашивает прерывание, а не резко прерывает поток
> Хороший способ думать о прерывании – это представлять себе, что оно фактически не прерывает выполняющийся поток; оно просто запрашивает, чтобы поток прервал сам себя при следующей удобной возможности (эти возможности называются точками отмены (cancellation points)).


***
# 📕2. Как обрабатывать `InterruptedException` — две «правильные» стратегии

## 🔶 1. **Пропуск `InterruptedException` выше по стеку**
Идеально, когда метод **может пробросить исключение наружу**.  

То есть:
- Мы ловим исключение и **сразу выбрасываем его дальше**.
- Это позволяет внешнему коду решить, что делать.
```java
void doWork() throws InterruptedException {
    while (true) {
        // что-то делаем
        Thread.sleep(1000); // может выбросить InterruptedException
    }
}
```

Если поток прервали → метод сразу завершится исключением.

✅ Это "правильный" вариант, если твой метод сам по себе **не должен управлять жизненным циклом потока**, а просто делегирует ответственность вызывающему коду.

***
## 🔶 2. **Восстановление прерывания**
Иногда ты **не можешь пробросить исключение**, потому что:

- твой метод не объявлен с `throws InterruptedException` (например, он ==реализует `Runnable.run()`==, который ничего не выбрасывает);
    
- или твой метод по контракту обязан "проглотить" исключения

В этом случае — **нельзя потерять информацию о прерывании!**  
👉==Ты перехватываешь исключение, делаешь локальную обработку (лог, очистка ресурсов)==, а потом **восстанавливаешь флаг прерывания** вызовом:
```java
Thread.currentThread().interrupt();
```

### Пример
```java
class MyTask implements Runnable {
    @Override
    public void run() {
        try {
            while (!Thread.currentThread().isInterrupted()) {
                // работа
                Thread.sleep(1000);
            }
        } catch (InterruptedException e) {
            // локальная очистка
            System.out.println("Прерывание получено, завершаюсь...");
            Thread.currentThread().interrupt(); // ⭐ восстановили флаг
        }
    }
}
```

✅ Теперь вышестоящий код (например, ExecutorService) всё ещё может увидеть, что поток прерван, и корректно завершить задачу.

***
## 💡 Что будет, если этого не делать? 💡
❌Если ты "проглотишь" исключение `InterruptedException` и **не восстановишь флаг**, то:
- внешний код "подумает", что поток работает нормально;
    
- система, которая ждёт остановки (например, `ExecutorService.shutdownNow()`), не сможет корректно завершить задачу;
    
- могут возникнуть **зависшие потоки**, которые не реагируют на прерывание.

***
## 💡 Как выбирать стратегию? 💡

1. Если метод **может пробросить `InterruptedException`** → **пробрасывай дальше**. Это лучший вариант.
    
2. Если метод **не может пробросить** → **лови и восстанавливай флаг** (через `Thread.currentThread().interrupt()`).
