`Condition` — это ==продвинутая альтернатива== `wait()`, `notify()` и `notifyAll()` из синхронизации на основе `synchronized`. 

Она ==работает **в связке с ==`Lock` (обычно с `ReentrantLock`)** и даёт **более гибкое и контролируемое** управление потоками.

# 📕1. Что такое `Condition`?

`Condition` — это ==интерфейс== из пакета `java.util.concurrent.locks`, предназначенный для **координации потоков**: один поток может **ждать сигнала**, другой — **подавать сигнал**.

Это **аналог мониторных методов**:
- `condition.await()` ≈ `object.wait()`
> [!NOTE] ⭐⭐⭐`await()` делает три вещи:  ⭐⭐⭐
> - Освобождает блокировку (другие потоки могут его получить)-
> - Переводит поток в режим ожидания (ждет signal())
> - После signal() автоматически перезахватывает блокировку

- `condition.signal()` ≈ `object.notify()`
    
- `condition.signalAll()` ≈ `object.notifyAll()`

Но работает **только в связке с `Lock`**, чаще всего — `ReentrantLock`.

# 📕 2. Best practice
## 🔶 1. **Один ==Condition на одно логическое условие**==
```java
Lock lock = new ReentrantLock();
Condition notFull = lock.newCondition();
Condition notEmpty = lock.newCondition();
```

## 🔶 2. **Используй `while`, а не `if` при ожидании**
```java
while (условие ожидания) {
    condition.await();
}
```

## 🔶 3. Блокируй `lock.lock()` перед `await()` и `signal()`
🔒 Методы `await()` и `signal()` можно вызывать **только внутри критической секции**, иначе `IllegalMonitorStateException`.

## 🔶 4.  **Используй `finally { lock.unlock(); }`**
⚠️ Это важно, чтобы **всегда разблокировать**, даже при исключениях.
## Пример
```java
class BoundedBuffer<T> {
    private final T[] buffer;
    private int count = 0, putPtr = 0, takePtr = 0;
    
    private final Lock lock = new ReentrantLock();
    private final Condition notFull  = lock.newCondition(); //Создание через Lock
    private final Condition notEmpty = lock.newCondition();

    @SuppressWarnings("unchecked")
    public BoundedBuffer(int size) {
        buffer = (T[]) new Object[size];
    }

    public void put(T x) throws InterruptedException {
        lock.lock();
        try {
            while (count == buffer.length)
                notFull.await(); // ждём, пока появится место
            buffer[putPtr] = x;
            putPtr = (putPtr + 1) % buffer.length;
            count++;
            notEmpty.signal(); // сигнал потребителю
        } finally {
            lock.unlock();
        }
    }

    public T take() throws InterruptedException {
        lock.lock();
        try {
            while (count == 0)
                notEmpty.await(); // ждём, пока появятся данные
            T x = buffer[takePtr];
            takePtr = (takePtr + 1) % buffer.length;
            count--;
            notFull.signal(); // сигнал производителю
            return x;
        } finally {
            lock.unlock();
        }
    }
    
    public void main(){
	    
    }
    
    public static void main(String[] args) {
		BoundedBuffer<Integer> boundedBuffer = new BoundedBuffer<>(5);
		
		new Thread(()->{
			for(int i = 0; i < 10; i++) {
				try {
					boundedBuffer.put(i);
					System.out.println("Отправлено в буфер i = " + i);
				} catch (InterruptedException e) {
					e.printStackTrace();
				}
			}
		}).start();
		
		new Thread(()->{
			int x = 0;
			for(int i = 0; i < 10; i++) {
				try {
					x = boundedBuffer.take();
					System.out.println("Получено из буфера i = " + x);
				} catch (InterruptedException e) {
					e.printStackTrace();
				}
			}
		}).start();
	}
}
```

# 💡 Почему `Condition` лучше `wait/notify`? 💡

| Возможность                | `wait/notify`           | `Condition`                                  |
| -------------------------- | ----------------------- | -------------------------------------------- |
| Работа без `synchronized`  | ❌                       | ✅ (работает с `Lock`)                        |
| Несколько условий ожидания | ❌ (только один монитор) | ✅ (==несколько== `Condition` на один `Lock`) |
| Прерываемое ожидание       | Ограниченно             | ✅ `await()`, `awaitNanos()`                  |
| Гибкий контроль потоков    | ❌                       | ✅                                            |
Использование нескольких `Condition` ==позволяет пробуждать поток определенного типа== (`Producer/Consumer`). 

Если же не использовать `Condition`, то при попытке вызвать `notify`, доступ к монитору получит следующий поток на очереди. То есть, может получиться так, что `Producer` вызовет `notify` и будет ожидать что проснется именно какой-нибудь `Consumer`, НО ПРОСНЕТСЯ`Producer`

## Пример
У нас есть **две очереди**:
- 🧑‍🍳 Очередь **поваров (Producers)** — они готовят еду.
- 🍽️ Очередь **едоков (Consumers)** — они хотят есть.

И есть **один общий звонок (монитор)**. Кто-то из поваров (или едоков), когда заканчивает работу, **нажимает на кнопку `notify()`**, чтобы разбудить кого-то, кто спит.

## ☢ Проблема с `notify()`
Вот что может случиться:

1. **Все повара устали** и легли спать (`wait()`), потому что кухня полная — больше нельзя ничего готовить.
    
2. Все едоки тоже спят, потому что на кухне **нет еды** — им нечего есть.
    
3. Приходит последний повар `producer5`, кладёт последнюю порцию, и говорит:  
    👉 «Я положил еду! Пусть кто-нибудь проснётся!» — и он вызывает `notify()`.
    
4. **Кого разбудит звонок `notify()`?**  
    Он не знает, кого будить — **повара или едока**. Это просто случайный выбор.
    
    💥 И вот **будит другого повара!**  
    А тот тоже не может ничего приготовить (кухня всё ещё полная). Он снова засыпает...
    
    **В итоге все спят, и никто не ест. Еда пропадает. Беда!**

## 🐞 Вот как появляется баг (повара спят, т.к еда готова, едаки ждут - еды нет)

1. **ProducerONE вызывает `put()`**  
    — `isFull == false`, он кладёт товар и вызывает `notify()`
    
2. В буфере теперь **что-то есть**, `isFull == true`
    
3. **ConsumerONE вызывает `take()`**  
    — всё хорошо: он берёт товар, устанавливает `isFull = false` и вызывает `notify()`
    
4. Теперь буфер пуст — `isFull == false`  
    (⚠️ Но **оба потока**, Producer2 и Consumer2, **могут проснуться** после `notify()`)
    
5. **ConsumerTWO успевает раньше Producer2 и вызывает `take()`**  
    — `isFull == false`, он засыпает (`wait()`)
    
6. **ProducerTWO вызывает `put()`**  
    — кладёт товар, `isFull = true`, вызывает `notify()`
    
7. ⚠️ ==**ProducerTHREE просыпается** (а не Consumer2)  ==
    — но `isFull == true`, значит он уходит обратно в `wait()`
    

👉 Теперь:
- буфер полон
- Consumer2 спит
- все производители спят
- **но Consumer не проснулся** — **мертвый тупик**
