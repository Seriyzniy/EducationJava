`ReentrantLock` — это класс из пакета `java.util.concurrent.locks`, ==предоставляющий **расширенный механизм блокировки**==, альтернативный ключевому слову `synchronized`. 

> [!NOTE] 🛑`ReentrantLock` реализует интерфейс `java.util.concurrent.locks.Lock` 🛑
> **==`Lock` — это интерфей==с, описывающий контракт блокировки**,  ==а `ReentrantLock`== — одна из ==**конкретных реализаций** этого интерфейса==.

```java
public class ReentrantLock implements Lock {
	// Создает экземпляр ReentrantLock с НЕЧЕСТНЫЙ (non-fair) политикой доступа.
	// Нет гарантии порядка получения блокировки
	public ReentrantLock() {
        this(false); // Вызывает основной конструктор с fair = false
    }
    
    // Создает экземпляр ReentrantLock с указанной политикой доступа.
    // Блокировка предоставляется потокам в порядке FIFO
    // ⭐ Ниже производительность из-за накладных расходов на поддержание очереди
    public ReentrantLock(boolean fair) {
        sync = fair ? new FairSync() : new NonfairSync();
    }
}
```

Он особенно полезен в более **гибких** и **сложных многопоточных сценариях**, где нужны дополнительные возможности, которых нет у `synchronized`.

```java
private final Lock lock = new ReentrantLock();

public void doWork() {
	lock.lock(); // захват блокировки
	try {
		System.out.println("Критическая секция");
	} finally {
		lock.unlock(); // обязательно освободить
	}
}
```

# 📕 1. Отличия от `synchronized`
| Возможность                         | `synchronized` | `ReentrantLock`                           |
| ----------------------------------- | -------------- | ----------------------------------------- |
| Повторный захват                    | ✅              | ✅                                         |
| Явное управление                    | ❌              | ✅ (`lock()/unlock()`)                     |
| Прерываемое ожидание                | ❌              | ✅ `lockInterruptibly()`                   |
| Попытка захвата без блокировки      | ❌              | ✅ `tryLock()`                             |
| Захват с таймаутом                  | ❌              | ✅ `tryLock(timeout)`                      |
| Явное опрос состояния блокировки    | ❌              | ✅ `isLocked()`, `isHeldByCurrentThread()` |
| Использование в условных переменных | ❌              | ✅ `Condition`                             |
## 💡 Варианты захвата

### 🔶 1. `lock()`
- ==Захватывает блокировку.== Если занята — поток блокируется до освобождения.
### 🔶 2. `tryLock()`
- Пытается захватить замок **без ожидания**. Возвращает `true`, если успешно.
```java
if (lock.tryLock()) {
    try {
        // критическая секция
    } finally {
        lock.unlock();
    }
}
```

### 🔶 3. `tryLock(long time, TimeUnit unit)`
- ⭐⭐⭐Пытается ==захватить блокировку в течение заданного времени.==
```java
if (lock.tryLock(1, TimeUnit.SECONDS)) {
    ...
}
```

### 🔶 4. `lockInterruptibly()`
- ⭐⭐⭐Позволяет **прервать поток**, ожидающий захвата.
```java
try {
    lock.lockInterruptibly();
} catch (InterruptedException e) {
    // обработка
}
```
- **Проблема `synchronized`:** Ожидание входа в `synchronized`-блок _непрерываемо_. Если поток заблокирован навсегда, вы не можете его прервать.
    
- **Реакция на отмену:** `lockInterruptibly` интегрирует механизм блокировок с механизмом прерывания потоков Java, делая задачи, использующие явные блокировки, корректно отменяемыми.
#### Ключевые особенности `lockInterruptibly()`:
1. **Прерываемость**: ==Поток может быть прерван во время ожидания блокировки==
    
2. **Исключение**: Бросает `InterruptedException` при прерывании
    
3. **Безопасность**: Всегда должен использоваться в try-catch блоке

```java
import java.util.concurrent.locks.Lock;
import java.util.concurrent.locks.ReentrantLock;

public class InterruptibleLockExample {
    private final Lock lock = new ReentrantLock();
    
    public void performTask() {
        try {
            System.out.println(Thread.currentThread().getName() + 
            ": Пытаюсь получить блокировку...");
            
            // Используем lockInterruptibly вместо обычного lock()
            lock.lockInterruptibly();
            
            try {
                System.out.println(Thread.currentThread().getName() + 
                ": Получил блокировку!");
                // Имитация длительной работы
                Thread.sleep(5000);
                System.out.println(Thread.currentThread().getName() + 
                ": Завершил работу");
            } finally {
                lock.unlock();
                System.out.println(Thread.currentThread().getName() + 
                ": Освободил блокировку");
            }
            
        } catch (InterruptedException e) {
            System.out.println(Thread.currentThread().getName() + 
            ": Был прерван во время ожидания блокировки!");
            Thread.currentThread().interrupt(); // Восстанавливаем статус прерывания
        }
    }
    
    public static void main(String[] args) throws InterruptedException {
        InterruptibleLockExample example = new InterruptibleLockExample();
        
        Thread thread1 = new Thread(example::performTask, "Thread-1");
        Thread thread2 = new Thread(example::performTask, "Thread-2");
        
        thread1.start();
        Thread.sleep(100); // Даем время thread1 захватить блокировку
        
        thread2.start();
        Thread.sleep(1000); // Ждем, пока thread2 начнет ожидать блокировку
        
        // Прерываем второй поток, который ждет блокировку
        System.out.println("Прерываем Thread-2...");
        thread2.interrupt();
        
        thread1.join();
        thread2.join();
    }
}
```

### 🔶 5. `newCondition()`
- Возвращает Condition instance для использования с этой блокировкой.
```java
public Condition newCondition() {
	return sync.newCondition();
}
```
# 💡 Обязательно `unlock()` в `finally` 💡

Никогда не забывайте ==**освобождать блокировку**== в `finally`:
```java
lock.lock();
try {
    // критическая секция
} finally {
    lock.unlock(); // иначе → deadlock
}
```

Гибкое управление захватом и освобождением монитора позволяет:

## 1. Захватить монитор в одном методе, а освободить в другом
```java
class Example {
    private final Lock lock = new ReentrantLock();

    public void firstMethod(Object object) {
    	this.lock.lock(); // Захват монитора 
    }
    
	public void secondMethod(Object object) {
    	this.lock.unlock(); // Освобождение монитора 
	}
}
```

# Выбор между `synchronized` и `ReentrantLock`
==Используйте `synchronized` по умолчанию==. Используйте `ReentrantLock` только тогда, когда вам нужны его уникальные возможности.

**Преимущества `synchronized`:**
1. **Прост и компактен:** Меньше кода, меньше chance для ошибок.
    
2. **Безопасен:** Автоматическое освобождение блокировки при выходе из блока (через механизм finally в bytecode).
    
3. **Хорошая интеграция с инструментами:** Дампы потоков в старых версиях JVM показывали информацию о `synchronized`-блокировках, что помогало в отладке deadlock-ов. (В Java 6+ это исправлено и для `ReentrantLock`).
    
4. **Потенциал для оптимизаций JVM:** JVM может проводить оптимизации вроде устранения блокировки (lock elision) или сорасщения блокировок (lock coarsening) именно со встроенными блокировками.

**Причины выбрать `ReentrantLock`:**
1. Вам нужны его расширенные функции: `tryLock()`, `lockInterruptibly()`, fair locking.

// 🟢 === БЛОКИРОВКА === 🟢
/** Захватывает блокировку (блокирующий вызов) */
public void lock() { sync.acquire(1); }

/** Пытается захватить блокировку без ожидания */
public boolean tryLock() { return sync.nonfairTryAcquire(1); }

/** Пытается захватить блокировку с таймаутом */
public boolean tryLock(long time, TimeUnit unit) throws InterruptedException {
	return sync.tryAcquireNanos(1, unit.toNanos(time));
}

/** Захватывает блокировку с возможностью прерывания */
public void lockInterruptibly() throws InterruptedException { sync.acquireInterruptibly(1); }

/** Освобождает блокировку */
public void unlock() { sync.release(1); }

// 🟢 === CONDITION === 🟢
/** Создаёт новый Condition для работы с этим lock */
public Condition newCondition() { return sync.newCondition(); }

// 🟢 === СОСТОЯНИЕ ===🟢 
/** Количество захватов блокировки текущим потоком */
public int getHoldCount() { return sync.getHoldCount(); }

/** Проверяет, владеет ли текущий поток блокировкой */
public boolean isHeldByCurrentThread() { return sync.isHeldExclusively(); }

/** Проверяет, захвачена ли блокировка любым потоком */
public boolean isLocked() { return sync.isLocked(); }

/** Проверяет, используется ли честная политика (fair) */
public final boolean isFair() { return sync instanceof FairSync; }

/** Поток, который в данный момент владеет блокировкой */
protected Thread getOwner() { return sync.getOwner(); }

/** Потоки, ожидающие блокировку (для отладки) */
protected Collection<Thread> getQueuedThreads() { return sync.getQueuedThreads(); }

/** Количество потоков, ожидающих блокировку */
public final int getQueueLength() { return sync.getQueueLength(); }

//🟢  === WAITERS === 🟢
/** Проверяет, есть ли потоки, ожидающие указанное Condition */
public boolean hasWaiters(Condition condition) {
	if (condition == null) throw new NullPointerException();
	if (!(condition instanceof AbstractQueuedSynchronizer.ConditionObject))
		throw new IllegalArgumentException("not owner");
	return sync.hasWaiters((AbstractQueuedSynchronizer.ConditionObject)condition);
}

/** Количество потоков, ожидающих указанное Condition */
public int getWaitQueueLength(Condition condition) {
	if (condition == null) throw new NullPointerException();
	if (!(condition instanceof AbstractQueuedSynchronizer.ConditionObject))
		throw new IllegalArgumentException("not owner");
	return sync.getWaitQueueLength((AbstractQueuedSynchronizer.ConditionObject)condition);
}

// 🟢 === ПОЛЯ === 🟢
private final Sync sync; // внутренний синхронизатор (AQS)
