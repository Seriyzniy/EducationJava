Существует альтернатива блокировкам (synchronized, Lock) для обеспечения потокобезопасности — это неблокирующие алгоритмы, основанные на атомарных операциях процессора (CAS - compare-and swap). 

Они сложнее в реализации, но могут обеспечить лучшую производительность и масштабируемость, особенно при высоких нагрузках, а также избегают проблем живучести (deadlock, priority inversion).

# 15.1 📕 Недостатки блокировки
💡**Ключевая идея:** ==Блокировки (synchronized, `ReentrantLock`) имеют серьезные недостатки в производительности и живучести==, особенно при высокой конкуренции.

**Подробное объяснение:**
1. 🔶**Высокие накладные расходы:** Когда поток не может получить блокировку, операционная система приостанавливает его (==переводит в состояние ожидания==). Это приводит к ==дорогостоящим операциям переключения контекста== (context switch), что может быть очень затратно, если сама операция, защищаемая блокировкой, короткая (например, инкремент счетчика).
    
2. 🔶**Проблемы живучести:**
    - **Взаимоблокировки (Deadlock):** Потоки блокируют друг друга, ожидая ресурсов.
        
    - **Инверсия приоритетов (Priority Inversion):** Поток с низким приоритетом удерживает блокировку, а поток с высоким приоритетом вынужден его ждать, что эффективно снижает его приоритет.
        
    - **Голодание (Starvation):** Если поток, удерживающий блокировку, надолго "завис" (бесконечный цикл, ошибка страницы), все ожидающие потоки не могут прогрессировать.
        
3. 🔶**Ограниченная детализация:** Блокировка защищает весь критический участок. Часто нам нужно атомарно изменить всего одну переменную, и блокировка для этого слишком "тяжеловесна".
    

> [!NOTE] **Вывод:** ==Нужен более легковесный== и детализированный ==механизм для простых операций.==

# 15.2 📕 Аппаратная поддержка параллелизм
💡 **Ключевая идея:** Современные процессоры предоставляют специальные инструкции, которые позволяют реализовывать оптимистичную неблокирующую синхронизацию.

## 15.2.1 Сравнить и обменять
💡**Ключевая идея:** CAS — это атомарная инструкция процессора, которая является основой для неблокирующих алгоритмов. Она работает по принципу "оптимистичной" проверки.

**Подробное объяснение:**
- **Что делает:** Инструкция CAS имеет три операнда: адрес памяти `V`, ожидаемое старое значение `A` и новое значение `B`.
    
    - Она атомарно проверяет, равно ли текущее значение по адресу `V` значению `A`.
        
    - Если да, то она записывает новое значение `B`.
        
    - Если нет (значение было изменено другим потоком), то она ничего не делает.
        
    - В любом случае, она возвращает текущее значение в `V` (или флаг успеха).
        
- **Семантика:** "Я думаю, что в `V` должно быть значение `A`. Если это так, установи `B`. Иначе, скажи мне, что я ошибся".

```java
public class SimulatedCAS {
    @GuardedBy("this")
    private int value;

    public synchronized int get() {
        return value;
    }

    public synchronized int compareAndSwap(int expectedValue, int newValue) {
        int oldValue = value;
        if (oldValue == expectedValue) {
            value = newValue;
        }
        return oldValue;
    }

    public synchronized boolean compareAndSet(int expectedValue, int newValue) {
        return (expectedValue == compareAndSwap(expectedValue, newValue));
    }
}
```

Когда несколько потоков пытаются обновить одну и ту же переменную с помощью CAS одновременно, ==один выигрывает и обновляет значение переменной, а остальные проиграют. ==

Но ==проигравшие не наказываются отстранением==, как это могло бы быть, если бы они не смогли захватить блокировку; вместо этого им говорят, что они не выиграли гонку на этот раз, но ==могут попробовать еще раз. ==

Поскольку ==поток, который проигрывает в операции CAS, НЕ блокируется==, он может решить, хочет ли он повторить попытку, выполнить какое-либо другое действие по восстановлению или не делать ничего

## 15.2.2 Неблокирующий счётчик
💡**Ключевая идея:** CAS позволяет реализовать потокобезопасные структуры без блокировок, перекладывая обработку конфликтов на вызывающий код.

Класс CasCounter из листинга 15.2, реализует потокобезопасный счетчик с помощью операции CAS. 
Если операция CAS проваливается, операция немедленно повторяется. Осуществление повторной попытки, как правило, является разумной стратегией, хотя в случаях жёсткой конкуренции может быть желательным подождать некоторое время или откатится, прежде чем повторять попытку

```java
public class CasCounter {
	//См предыдущий листинг
	//volatile не нужен, т.к в предыдущем классе уже используется синхронизация 
    private SimulatedCAS value; 
    
    public int getValue() {
        return value.get();
    }
    
    public int increment() {
        int v;
        do {
            v = value.get();
        } while (v != value.compareAndSwap(v, v + 1));
        return v + 1;
    }
}
```

**Преимущество перед блокировкой:** Потоки не приостанавливаются при конфликте. Они просто повторяют попытку. Это может быть значительно быстрее, так как избегает дорогостоящих переключений контекста.

> [!NOTE] ✅ЛУЧШЕ использовать готовые реализации классов `Atomic***`
> **На практике:** Не нужно писать свой `CasCounter`. Следует использовать готовый `AtomicInteger.incrementAndGet()`.

## 15.2.3 Поддержка операций CAS в среде JVM
💡**Ключевая идея:** Начиная ==с Java 5.0, JVM напрямую использует аппаратные инструкции CAS (если они есть на процессоре)== для предоставления высокоэффективных атомарных операций Java-коду.

**Подробное объяснение:**
- **До Java 5.0:** Для использования CAS нужно было писать нативный (C/C++) код.
    
- **После Java 5.0:** Появились классы в пакете `java.util.concurrent.atomic` (`AtomicInteger`, `AtomicLong`, `AtomicReference`).
    
- **Как работает:** JVM компилирует вызовы методов этих классов (например, `compareAndSet`) в наилучшие доступные низкоуровневые инструкции (например, `cmpxchg` на x86). Если процессор не поддерживает CAS, JVM эмулирует его с помощью спин-блокировок.

# 15.3 📕 Классы атомарных переменных (Atomic классы)

**Ключевая идея:** Классы из пакета `java.util.concurrent.atomic` — это обертка вокруг volatile-переменных, которая добавляет к ним атомарные операции типа "считать-изменить-записать" (read-modify-write).

> [!NOTE] ⭐`Atomic`-классы дают ту же гарантию что и `volatile`⭐
> **Да, все атомарные классы (из пакета `java.util.concurrent.atomic`) гарантируют те же эффекты памяти (видимость и упорядочивание), что и `volatile` переменные, и даже больше.**

> [!NOTE] CAS быстрее блокировок, т.к не тратит время на восстановление
> С алгоритмами, основанными на атомарных переменных вместо блокировок, потоки с большей вероятностью смогут продолжить работу без задержек и затратить меньшее время на восстановление, если столкнутся с конкуренцией.

Существует ==двенадцать классов атомарных переменных==, разделенных на ==четыре группы==:  
1. скаляры (scalars), 
2. средства обновления полей (field updaters), 
3. массивы 
4. составные переменные

### Группа 1: Скаляры (Scalars)

Это самые часто используемые атомарные классы. Они обеспечивают атомарные операции над одиночными переменными.

**1. 🔶`AtomicInteger`**
- **Назначение:** Атомарная операция над значением `int`.
- **Ключевые методы:**
    - `get()`, `set(int newValue)`
    - `compareAndSet(int expect, int update)` (CAS)
    - `getAndIncrement()` (аналог `i++`), `incrementAndGet()` (аналог `++i`)
    - `getAndAdd(int delta)`, `addAndGet(int delta)`
    - `getAndUpdate(IntUnaryOperator updateFunction)`, `updateAndGet(...)` (атомарное применение функции)
        
- **Пример:** Счетчик, генератор ID.
```java
AtomicInteger counter = new AtomicInteger(0);
int nextId = counter.incrementAndGet(); // Потокобезопасно
```

**2. 🔶`AtomicLong`**
- **Назначение:** Атомарная операция над значением `long`.
- **Ключевые методы:** Аналогичны `AtomicInteger`.
    
- ❗❗❗**Особенность:** На 32-битных системах операции с `long` не являются атомарными по умолчанию. `AtomicLong` решает эту проблему.
    
- **Пример:** Подсчет байтов в сетевом соединении.
```java
AtomicLong totalBytes = new AtomicLong(0L);
totalBytes.addAndGet(newBytes.length);
```

**3. 🔶`AtomicBoolean`**
- **Назначение:** Атомарная операция над значением `boolean`.
- **Ключевые методы:**
    - `compareAndSet(boolean expect, boolean update)`
    - `getAndSet(boolean newValue)`
        
- **Пример:** Флаг инициализации, который должен быть установлен только один раз.
 ```java
 AtomicBoolean isInitialized = new AtomicBoolean(false);
public void init() {
	if (isInitialized.compareAndSet(false, true)) {
		// Инициализация выполнится только один раз
	}
}
 ```

**4. 🔶⭐⭐⭐`AtomicReference<V>`**⭐⭐⭐
- ⭐⭐**Назначение:** Атомарная операция над ссылкой на объект типа `V`.⭐⭐

> [!NOTE] То есть он может быть `final` НО при этом изменять объект внутри себя
> Это **объект-контейнер**, который предоставляет методы для атомарного изменения ссылки, которую он хранит _внутри себя_.
```java
public class AtomicReference<V> {
    private volatile V value; // Вот эта внутренняя ссылка меняется!

    public final boolean compareAndSet(V expect, V update) {
        // Магия на уровне процессора, которая атомарно проверяет,
        // что this.value == expect, и если да, то sets this.value = update
    }
    // ... другие методы
}
```

- **Ключевые методы:**
    - `get()`, `set(V newValue)`
    - `compareAndSet(V expect, V update)`
    - `getAndUpdate(UnaryOperator<V> updateFunction)`
        
- **Пример:** Реализация неблокирующего стека (как в книге), кэширование самого свежего значения.
```java
AtomicReference<Data> latestData = new AtomicReference<>();

public void updateData(Data newData) {
	latestData.set(newData); // Атомарная публикация
}  
```

### Группа 2: Массивы (Arrays)
Эти классы обеспечивают атомарность для _отдельных элементов_ массива. Важно: это не атомарность операций над всем массивом целиком.

**1. 🔶`AtomicIntegerArray`**
- **Назначение:** Массив `int[]`, элементы которого можно обновлять атомарно.
- **Ключевые методы:** Аналогичны `AtomicInteger`, но с индексом: 
	- `get(int i)`, 
	- `compareAndSet(int i, int expect, int update)`, 
	- `getAndAdd(int i, int delta)`.

**2. 🔶`AtomicLongArray`**
- **Назначение:** Массив `long[]` с атомарными элементами.
    
**3. 🔶`AtomicReferenceArray<E>`**
- **Назначение:** Массив ссылок `E[]`, элементы которого можно обновлять атомарно.
- **Пример:** Низкоуровневая реализация пула ресурсов или кольцевого буфера.
```java
AtomicIntegerArray voteCounts = new AtomicIntegerArray(10); // 10 кандидатов
voteCounts.getAndIncrement(candidateId); // Атомарно добавить голос за кандидата N
```

### ⭐⭐⭐Группа 3: Средства обновления полей (Field Updaters)

Это самые "экзотические" классы. Они ==позволяют ПРИМЕНЯТЬ АТОМАРНЫЕ операции CAS к ОБЫЧНЫМ **volatile** полям существующих объектов==. 
Это нужно ==для экономии памяти, так как не требует создавать объект-обертку== (`AtomicInteger`) для каждого поля.

> [!NOTE] ⭐⭐Создание ТОЛЬКО ЧЕРЕЗ фабричный метод `newUpdater()⭐⭐`
> Классы-апдейтеры не имеют конструкторов; для их создания необходимо вызвать фабричный метод newUpdater, указав класс и имя поля.

**1. 🔶`AtomicIntegerFieldUpdater<T>`**
- **Назначение:** Атомарное обновление volatile `int` поля класса `T`.
    
- ⭐⭐⭐**Создание:** Через статический фабричный метод.⭐⭐⭐ (Один updater на весь класс)
```java
public class MyClass {
    private volatile int count;
    
    private static final AtomicIntegerFieldUpdater<MyClass> UPDATER =
            AtomicIntegerFieldUpdater.newUpdater(MyClass.class, "count");
            
    public void increment() {
        UPDATER.getAndIncrement(this);
    }
}
```

**2. 🔶`AtomicLongFieldUpdater<T>`**
- **Назначение:** Атомарное обновление volatile `long` поля.
    

**3. 🔶⭐⭐⭐`AtomicReferenceFieldUpdater<T,V>`**
- **Назначение:** Атомарное обновление volatile ссылочного поля типа `V` в классе `T`.
    
- **Пример использования:** Именно этот класс используется внутри `ConcurrentLinkedQueue` (как описано в вашем отрывке из книги) для обновления поля `next` узлов (`Node`) очереди, чтобы не создавать для каждого узла свой `AtomicReference`.

### Группа 4: Составные переменные

Эти классы решают проблему ABA, описанную в конце главы. Они хранят не просто значение, а пару **(значение + метка)**.

**1. 🔶`AtomicStampedReference<V>`**
- **Назначение:** Атомарная операция над парой `(reference: V, stamp: int)`. 
Метка (`stamp`) — это обычно номер версии (version number).
    
- ⭐**Решает проблему ABA:** Даже если ссылка вернулась к прежнему значению `A`, ее метка (версия) будет другой, и CAS завершится неудачей.
    
- **Ключевые методы:**
    - `compareAndSet(V expectedReference, V newReference, int expectedStamp, int newStamp)`
    - `get(int[] stampHolder)` (возвращает и ссылку, и текущую метку в массив)
        
- **Пример:** Алгоритмы, чувствительные к истории изменений (например, non-blocking stack/queue в средах без GC).
    

**2. 🔶`AtomicMarkableReference<V>`**
- **Назначение:** Упрощенная версия `AtomicStampedReference`. Атомарная операция над парой `(reference: V, mark: boolean)`.
    
- **Использование:** Когда нужен просто флаг. Классический пример — реализация алгоритмов с логическим удалением узлов из списка. Поле `mark` может означать "элемент удален".
    
- **Ключевые методы:**
    - `compareAndSet(V expectedReference, V newReference, boolean expectedMark, boolean newMark)`
        
    - `get(boolean[] markHolder)`

## 15.3.1 ⭐⭐⭐ Atomics as “better volatiles” ⭐⭐⭐
💡 **Ключевая идея:** ==Атомарные переменные можно использовать== не только для сложных неблокирующих алгоритмов, но и ==как замену volatile-переменным, когда нужны атомарные составные операции.==

```java
public class CasNumberRange {
    @Immutable
    private static class IntPair {
        final int lower; // Invariant: lower <= upper
        final int upper;

        public IntPair(int lower, int upper) {
            this.lower = lower;
            this.upper = upper;
        }
    }

    private final AtomicReference<IntPair> values = 
        new AtomicReference<IntPair>(new IntPair(0, 0));

    public int getLower() { 
        return values.get().lower; 
    }
    
    public int getUpper() { 
        return values.get().upper; 
    }

    public void setLower(int i) {
        while (true) {
            IntPair oldv = values.get();
            if (i > oldv.upper) {
                throw new IllegalArgumentException(
                    "Can't set lower to " + i + " > upper");
            }
            IntPair newv = new IntPair(i, oldv.upper);
            if (values.compareAndSet(oldv, newv)) {
                return;
            }
        }
    }

    // Аналогичная реализация для setUpper
    public void setUpper(int i) {
        while (true) {
            IntPair oldv = values.get();
            if (i < oldv.lower) {
                throw new IllegalArgumentException(
                    "Can't set upper to " + i + " < lower");
            }
            IntPair newv = new IntPair(oldv.lower, i);
            if (values.compareAndSet(oldv, newv)) {
                return;
            }
        }
    }
}
```

**Проблема:** У нас есть инвариант: `lower <= upper`. Нужно атомарно изменять `lower` и `upper` так, чтобы инвариант никогда не нарушался для других потоков.

❌❗❗**Неверное решение:** Использовать два `AtomicInteger` для `lower` и `upper` по отдельности. Это не сработает, так как между проверкой инварианта (`if (i > oldv.upper)`) и установкой нового значения (`compareAndSet`) другой поток может изменить `upper`, что нарушит инвариант.

✅**Верное решение (Immutable + AtomicReference):**
1. Создать **неизменяемый** (immutable) класс-контейнер `IntPair`, хранящий обе границы.

2. Хранить состояние не в двух отдельных переменных, а в одном `AtomicReference<IntPair>`. ⭐⭐⭐Так как `AtomicReference` можно сравнить "старый" объект на который он ссылается и "новый" который идет в `compareAndSwap`⭐⭐⭐. Таким образом, если даже один из потоков `A` поменяет объект между проверкой инварианта и CAS другого потока `B`, то при CAS в `B`, произойдет проверка ссылки на ожидаемый старый объект (который уже заменил `A`) и новый, который подготовил `B`

3. Для изменения границы:
	- В цикле читаем текущее состояние (`oldv`).
		
	- Проверяем инвариант для _нового_ значения.
		
	- Создаем **новый** неизменяемый объект `IntPair` с новыми значениями.
		
	- Пытаемся атомарно заменить старый `IntPair` на новый с помощью `compareAndSet`.
		
	- Если `CAS` неуспешен (другой поток уже изменил состояние), повторяем попытку.
		

**Вывод:** Объединяя несколько связанных переменных в один неизменяемый объект и используя `AtomicReference` для него, можно атомарно обновлять группу переменных, поддерживая инварианты.

## 15.3.2 Производительность: блокировки против атомарных переменных
**Ключевая идея:** Производительность блокировок и атомарных переменных зависит от уровня конкуренции. 
- Атомарные переменные обычно масштабируются ==лучше при низкой и умеренной конкуренции==
- Блокировки могут ==лучше== справляться ==с экстремально высокой конкуренцией==.

**Подробное объяснение:**
- 🔶**Высокая конкуренция (Рис. 15.1):** Блокировки (`ReentrantLock`) показывают лучшую производительность. Почему? Потоки, не получившие блокировку, приостанавливаются, что снижает общую нагрузку на шину памяти и CPU, позволяя потоку-победителю быстрее выполнить работу.
    
- 🔶**Умеренная/низкая конкуренция (Рис. 15.2):** Атомарные переменные (`AtomicInteger`) показывают **значительно** лучшую производительность и масштабируемость. Почему? Они избегают накладных расходов на переключение контекста и работают на более низком уровне.
    
- 💡**Лучший вариант (Кривая `ThreadLocal`):** Во многих случаях лучшая стратегия — вообще избегать разделяемого состояния, используя `ThreadLocal`. Это идеально для генераторов случайных чисел, где каждому потоку нужна своя собственная последовательность.
    

> [!NOTE] Умеренная конкуренция - Atomic, высокая конкуренция - блокировки
> **Вывод:** Для большинства реальных сценариев (умеренная конкуренция) атомарные переменные предпочтительнее блокировок из-за лучшей масштабируемости.

# 15.4 📕 Неблокирующие алгоритмы
**Ключевая идея:** ==Неблокирующие алгоритмы — это алгоритмы, в которых неудача или приостановка одного потока не может привести к неудаче или приостановке другого==. 

==Алгоритм называется СВОБОДНЫМ== от блокировок (lock-free), ==если при выполнении каждого шага какой либо поток сможет прогрессировать==. 

Алгоритмы, использующие операции CAS исключительно для координации между потоками, при правильном построении могут быть как неблокирующими, так и свободными от блокировок

## 15.4.1 Неблокирующий стек
**Ключевая идея:** Можно построить неблокирующий стек, используя `AtomicReference` для вершины стека (head) и спекулятивно обновляя его с помощью CAS.

```java
public class ConcurrentStack<E> {
    private final AtomicReference<Node<E>> top = new AtomicReference<Node<E>>();

    public void push(E item) {
        Node<E> newHead = new Node<E>(item);
        Node<E> oldHead;
        do {
            oldHead = top.get();
            newHead.next = oldHead;
        } while (!top.compareAndSet(oldHead, newHead));
    }

    public E pop() {
        Node<E> oldHead;
        Node<E> newHead;
        do {
            oldHead = top.get();
            if (oldHead == null) {
                return null;
            }
            newHead = oldHead.next;
        } while (!top.compareAndSet(oldHead, newHead));
        return oldHead.item;
    }

    private static class Node<E> {
        public final E item;
        public Node<E> next;

        public Node(E item) {
            this.item = item;
        }
    }
}
```


**Подробное объяснение (на примере `ConcurrentStack` из Листинга 15.6):**
- **Структура:** Стек реализован как связный список. `top` — это `AtomicReference<Node<E>>`, указывающий на вершину.
    
- **Метод `push`:**
    1. Создается новый узел.
        
    2. В цикле читается текущая вершина (`oldHead`).
        
    3. Новый узел ссылается на `oldHead` (`newHead.next = oldHead`).
        
    4. Пытаемся CAS-ом заменить `top` с `oldHead` на `newHead`.
        
    5. Если CAS успешен — элемент добавлен. Если нет — другой поток изменил стек, повторяем попытку.
        
- **Метод `pop`:**
    1. В цикле читаем текущую вершину (`oldHead`).
        
    2. Если стек пуст (`oldHead == null`), возвращаем `null`.
        
    3. Читаем следующий элемент (`newHead = oldHead.next`).
        
    4. Пытаемся CAS-ом заменить `top` с `oldHead` на `newHead`.
        
    5. Если успешно — возвращаем данные из `oldHead`. Если нет — пробуем снова.
        
- **Потокобезопасность:** Гарантируется атомарностью и память-семантикой (видимостью) операции `compareAndSet`, которая действует как volatile запись/чтение.

## 15.4.2 Неблокирующий связанный список (очередь)
💡**Ключевая идея:** Построение неблокирующих алгоритмов для структур с несколькими указателями (очередь имеет head и tail) сложнее. 
❗❓Решение — позволить потокам помогать друг другу завершать операции.

```java
public class LinkedQueue<E> {
    private static class Node<E> {
        final E item;
        final AtomicReference<Node<E>> next;

        public Node(E item, Node<E> next) {
            this.item = item;
            this.next = new AtomicReference<Node<E>>(next);
        }
    }

    private final Node<E> dummy = new Node<E>(null, null);
    private final AtomicReference<Node<E>> head = new AtomicReference<Node<E>>(dummy);
    private final AtomicReference<Node<E>> tail = new AtomicReference<Node<E>>(dummy);

    public boolean put(E item) {
        Node<E> newNode = new Node<E>(item, null);
        while (true) {
            Node<E> curTail = tail.get();
            Node<E> tailNext = curTail.next.get();
            
            if (curTail == tail.get()) { // Проверяем, не изменился ли tail
                if (tailNext != null) {
                    // Очередь в промежуточном состоянии, продвигаем tail
                    tail.compareAndSet(curTail, tailNext);
                } else {
                    // Очередь в стабильном состоянии, пробуем вставить новый узел
                    if (curTail.next.compareAndSet(null, newNode)) {
                        // Вставка удалась, пробуем продвинуть tail
                        tail.compareAndSet(curTail, newNode);
                        return true;
                    }
                }
            }
        }
    }
}
```

**Подробное объяснение (на примере `LinkedQueue` из Листинга 15.7 и Рис. 15.3-15.5):**

- **Проблема:** Чтобы добавить элемент в очередь, нужно атомарно обновить два указателя: `next` последнего элемента и `tail`. Сделать это одной CAS-инструкцией невозможно.
    
- **Решение (Алгоритм Майкла-Скотта):**
    1. **Определение состояния:** Очередь может быть в двух состояниях:
        - **Спокойное (Quiescent):** `tail.next == null`.
            
        - **Промежуточное (Intermediate):** `tail.next != null` (другой поток начал вставку, но не завершил ее).
            
    2. **Помощь (Helping):** Если поток видит, что очередь в промежуточном состоянии, он не ждет, а _помогает_ конкурирующему потоку: пытается переместить `tail` на правильное место (выполняет второй шаг за него).
        
    3. **Затем** только после этого пытается вставить свой собственный элемент.
        
- **Преимущество:** Алгоритм не блокируется. Даже если поток "умрет" на середине операции, другой поток завершит ее за него.

## 15.4.3 Обновления атомарного поля - `AtomicReferenceFieldUpdater<T,V>`
**Ключевая идея:** Классы `AtomicReferenceFieldUpdater` позволяют применить семантику CAS к обычным volatile-полям, что экономит память, избегая создания объекта-обертки `AtomicReference` для каждого узла структуры данных.

В листинге 15.7 показан алгоритм, используемый классом `ConcurrentLinkedQueue`, но фактическая реализация немного отличается. 
Вместо представления каждого экземпляра `Node` с помощью атомарной ссылки, класс `ConcurrentLinkedQueue` использует `volatile` ссылку и обновляет ее с помощью основанного на reflection класса `AtomicReferenceFieldUpdater`, как показано в листинге 15.8.

```java
private static class Node<E> {
    private final E item;
    private volatile Node<E> next;

    public Node(E item) {
        this.item = item;
    }
}

private static final AtomicReferenceFieldUpdater<Node, Node> nextUpdater = 
    AtomicReferenceFieldUpdater.newUpdater(
        Node.class,     // Класс, содержащий поле
        Node.class,     // Тип поля
        "next"          // Имя поля
    );
```

**Подробное объяснение (на примере Листинга 15.8):**
- **Зачем?** В `ConcurrentLinkedQueue` узлы (`Node`) создаются и удаляются очень часто. Создание для каждого узла своего объекта `AtomicReference` — это значительные накладные расходы на память и GC.
    
- **Как?** С помощью рефлексии создается `AtomicReferenceFieldUpdater` (в данном случае `nextUpdater`), который "умеет" применять `compareAndSet` к обычному volatile-полю `next` любого объекта `Node`.
    
- ❗❗**Предостережение:** Атомарность гарантируется только если все обновления поля производятся через этот апдейтер. Если поле изменяется напрямую, атомарность теряется.

## 15.4.4 Проблема ABA

**Ключевая идея:** Проблема ABA возникает, когда значение переменной меняется с `A` на `B` и обратно на `A`. CAS, видя `A`, ошибочно считает, что ничего не изменилось, хотя это может быть не так.

**Подробное объяснение:**
- **Сценарий:** Поток 1 читает значение головы стека `A`. Затем его вытесняют.
    
    - Поток 2 выталкивает `A`, меняя голову на `B`.
        
    - Поток 2 (или 3) выталкивает `B`.
        
    - Поток 2 (или 3) помещает в стек новый узел, который по стечению обстоятельств был аллоцирован по тому же адресу, что и старый `A` (такое возможно при ручном управлении памятью или из-за работы GC). Теперь голова снова указывает на `A`, но состояние стека изменилось!
        
    - Поток 1 возвращается, делает CAS: "голова все еще `A`? Да!" и успешно записывает свое новое значение, портя структуру данных.
        
- ⭐**Решение:** `AtomicStampedReference` и `AtomicMarkableReference`. Они добавляют к ссылке "метку" (stamp) — обычно номер версии (integer) или флаг (boolean). CAS проверяет не только ссылку, но и эту метку. Даже если ссылка вернулась к `A`, ее метка (номер версии) будет другой.
