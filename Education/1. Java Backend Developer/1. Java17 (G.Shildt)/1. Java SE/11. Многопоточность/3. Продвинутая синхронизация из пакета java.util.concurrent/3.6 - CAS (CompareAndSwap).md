Захват монитора, хоть и является удобным при организации работы с разделяемым ресурсом, однако имеет недостаток, связанный с тем, что потоки ждут, пока освободиться монитор.

CAS — это **==АТОМАРНАЯ== НЕБЛОКИРУЮЩАЯ инструкция** уровня процессора. 
## 💡 Что значит "атомарная" в данном случае?
Это значит, что **CAS выполняется в один машинный цикл**:  
✅ Чтение текущего значения  
✅ Сравнение  
✅ Условная запись  
**всё одновременно, без возможности "вклиниться" между действиями**.


Она ==сравнивает текущее значение переменной с ОЖИДАЕМЫМ СТАРЫМ значением==, и **если они совпадают**, то **заменяет** его на новое.

> CAS — "сравни и замени, если не изменилось".

## Где используется CAS?
- `Atomic*` классы (`AtomicInteger`, `AtomicReference`)
    
- `ConcurrentHashMap`
    
- `LongAdder`
    
- `StampedLock`
    
- Алгоритмы типа lock-free queue, stack и др.

# 📕 1. Сигнатура CAS в абстрактном виде:
```java
boolean compareAndSwap(int expectedValue, int newValue) {
	int oldValue = extractvaluesfromcell();
	if(oldValue == expectedValue){
		putNewValueInCell(newValue);
		return true;
	}
	return false;
}
```
- Если **текущее значение == expectedValue**, то → заменить на `newValue`, вернуть `true`
    
- Иначе → ничего не делать, вернуть `false`

## Пример с `AtomicInteger`
```java
AtomicInteger atomic = new AtomicInteger(5);

boolean updated = atomic.compareAndSet(5, 10); // ✅ true → обновилось
System.out.println(atomic.get()); // 10

updated = atomic.compareAndSet(5, 20); // ❌ false → не обновилось
System.out.println(atomic.get()); // 10
```

Начальное значение 5. Допусти два потока пытаются изменить переменную. Допустим первый поток начал работать первым:
1. Извлекает значение из переменной
2. Сравнивает ОЖИДАЕМОЕ старое значение, с тем которое есть по факту
	- Если совпало - изменяет переменную
	- Иначе - отказ в изменении

Затем чуть позже, стартует второй поток. Он также считывает значение 5. После чего пытается провести операцию `CAS`. Но в этот момент первый поток уже изменил значение на 10.
Второй поток в ожидаемое старое значение установит 5, а в новое свое значение. После проверки окажется, что по факту в переменной находится 10, а поток передал ожидаемое 5. Значит изменения не применяются

# 📕 2. Недостатки
## 🔶1. **Отсутствие блокировок = Нет ожидания**

### Что это?
CAS — это **активное ожидание** (busy-waiting): поток постоянно пытается выполнить операцию, пока не получится. Он **не блокируется и не засыпает**.
### Почему это плохо?
- Расходуется **CPU** даже если нет реального прогресса
- Потоки конкурируют за ресурсы, не уступая управление

## 🔶2. **Ограниченность к единичным значениям**

### Что это?
CAS **может безопасно обновить только одно значение**.
### Почему это плохо?
Если нужно **атомарно обновить несколько переменных**, CAS **не поможет**, потому что нет возможности проверить и заменить _группу значений_ как единое целое.

# 📕 3. Проблемы
## 🔶 1.  **ABA-проблема**

**Суть:** Значение, которое вы хотите поменять, могло **измениться и вернуться обратно**, и вы об этом **не узнаете**.

```
1. Thread1 читает A
2. Thread2 меняет A → B → A
3. Thread1 делает CAS(A, X) — успешно! Но A уже "другая"
```

⛔ Это может привести к ==**логическим ошибкам**==, особенно ==если значение имеет **смысловую нагрузку**== (например, ==ссылка== на объект, версия и т.д.).

### ✅ Решения ABA-проблемы:

**Использование счётчиков или версий (stamp)**  
→ Вместо простого значения, храним: `значение + версия`

Добавляют к значению **метку** (stamp или версию), которая изменяется при каждой операции.
- 📌`AtomicStampedReference` хранит пару: `(value, int stamp)`.
    
- 📌`AtomicMarkableReference` хранит: `(value, boolean mark)`.

```java
AtomicStampedReference<String> ref = new AtomicStampedReference<>("A", 0);

int[] stampHolder = new int[1];
String current = ref.get(stampHolder); // current = "A", stamp = 0

// Поток 1: Пытается изменить "A" → "B" только если метка не изменилась
boolean success = ref.compareAndSet("A", "B", stampHolder[0], stampHolder[0] + 1);
```
### Когда ABA — это не проблема?
1. **Примитивные типы** (например, `int`):  
    Если значение `A` физически равно новому `A` (нет скрытого состояния), ABA безопасно.
    
2. **Неизменяемые объекты**:  
    Если `A` — это immutable-объект, его повторное использование не вызывает проблем.

## 🔶 2. **Живучий спин (Live Lock)**

**Суть:** Два потока **постоянно проигрывают CAS**, мешая друг другу и не продвигаясь вперёд.
```java
while (!atomic.compareAndSet(expected, update)) {
    // retry
}
```
### ✅ Решения:
- Использовать **ограниченное число повторов** + `Thread.yield()` или `sleep()`
- Иногда лучше применить **блокировки**, особенно при высокой конкуренции

- Использовать **`Backoff`-алгоритмы** (экспоненциальная задержка перед повтором)
