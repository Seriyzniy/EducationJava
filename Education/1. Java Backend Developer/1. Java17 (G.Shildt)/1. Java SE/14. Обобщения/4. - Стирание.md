# 📕 1. Почему и как работают обобщения?
Важным ограничением при добавлении обобщений в Java, заключалось то, что обобщенный код обязан был быть совместимым с ранее существовавшим необобщенным кодом.

==Способ, которым в Java реализованы обобщения==, удовлетворяющий упомянутому ограничению, предусматривает применение `СТИРАНИЕ`.

# 📕 2. Стирание - параметры типов заменяются их ограничениями на этапе компиляции
Рассмотрим, каким образом работает стирание. 

> [!NOTE] **Стирание** - `процесс удаления информации об обобщенных` типах на этапе компиляции:
> 1. Параметры типов заменяются их ограничениями (обычно Object, если не указано ограничение)
> 2. Приведения вставляются автоматически 
> 3. Обеспечивает совместимость с унаследованным кодом

## 📙 2.1 Мостовые методы
В Java ==**дженерики реализованы через стирание типов== (type erasure)**.  
То есть при компиляции информация о параметрах типа **частично теряется**.
### Пример
```java
class Gen<T> {
    T ob;

    Gen(T o) { ob = o; }

    T getOb() { return ob; }
}
```

После **стирания типов** ==метод в байткоде выглядит так==:
```java
Object getOb() { return ob; }
```
⭐потому что `T` заменяется на `Object` (если нет ограничений).

### Ситуация с переопределением

Теперь мы пишем подкласс:
```java
class Gen2 extends Gen<String> {
    Gen2(String o) { super(o); }

    @Override
    String getOb() {
        System.out.print("Вызван метод getOb(), специфичный для String: ");
        return ob;
    }
}
```
Что происходит?
- В базовом классе `Gen<T>` метод после стирания выглядит как `Object getOb()`.
    
- В `Gen2` мы пишем метод `String getOb()`.
    
⚠️ Но по правилам JVM сигнатуры методов должны совпадать при переопределении.  
А здесь **возвращаемые типы разные**: `Object` (после стирания) vs `String`.

## ⭐ Решение компилятора: мостовой метод
Чтобы всё корректно работало, компилятор добавляет в `Gen2` **мостовой метод**:
```java
public Object getOb() {   // мостовой метод
    return getOb();       // ✅вызывает реальный getOb(), возвращающий String✅
}
```

Таким образом в классе `Gen2` будет **два метода**:
1. `String getOb()` — то, что мы написали.
2. 🛑⭐`Object getOb()` — автоматически сгенерированный мостовой метод.

## Зачем это нужно?
1. Чтобы **сохранить совместимость** с кодом, где базовый класс используется в «сырых» (raw) или обобщённых формах.  
    Например:
```java
Gen obj = new Gen2("Test");
Object o = obj.getOb(); // работает благодаря мостовому методу
```

2. Без мостового метода вызов `getOb()` у `obj` ожидал бы возвращаемый тип `Object`, а у нас только `String` → несостыковка.
    
3. Чтобы JVM могла корректно обрабатывать переопределения **после стирания типов**.
