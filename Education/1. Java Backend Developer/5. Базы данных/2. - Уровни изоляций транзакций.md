## 🎯 Цель уровней изоляции
Обеспечить букву **I** в ACID — _Isolation_:  
ограничить влияние параллельных транзакций друг на друга и ==предотвратить конкурентные аномалии==:
- **Dirty read** — чтение незакоммиченных данных
    
- **Non-repeatable read** — строка изменилась между двумя SELECT
    
- **Phantom read** — появляются новые строки, подходящие под WHERE
    
- **Lost update** — потеря обновления из-за параллельной записи


## Механизмы реализации уровней изоляций
Реальные механизмы под капотом:
- **MVCC** — многоверсионность (PostgreSQL, InnoDB)
    
- **Read/Write Locks** — блокировки строк, диапазонов
    
- **Snapshot** — логические снимки базы
    
- **Predicate Locks** (PostgreSQL)
    
- **Next-key locks** (MySQL)
    
- **SSI (Serializable Snapshot Isolation)** — слежение за конфликтами (PostgreSQL)

Каждый уровень изоляции — это _определённая комбинация этих методов_.

---
# 1. Уровни изоляций (SQL стандарт)
## 1 READ UNCOMMITTED
Самый низкий уровень. 
> Чтение не закомиченных данных

**Что допускает:**  
✔ dirty reads  
✔ non-repeatable reads  
✔ phantom reads  
✔ lost updates (обычно да, если нет блокировок)

**Как работает:**  
Не ставит нормальные блокировки на чтение — можно читать чужие незафиксированные данные.

📌 _В PostgreSQL фактически не поддерживается — READ UNCOMMITTED превращается в READ COMMITTED._

---
## 2. READ COMMITTED (дефолт в PostgreSQL, Oracle)
> Чтение только зафиксированных данных.

**Что допускает:**  
✔ non-repeatable reads  
✔ phantom reads

**Что НЕ допускает:**  
❌ dirty reads  
❌ lost updates (обычно предотвращается за счёт блокировок на запись)

### 🧩 Под капотом (PostgreSQL):
- MVCC snapshot создаётся **каждый раз при выполнении SELECT**
    
- Каждый запрос видит состояние БД “на момент себя”  
    → поэтому последовательные SELECT могут видеть разные данные

---
## 3. REPEATABLE READ
Гарантия повторяемости чтения.

**Что НЕ допускает:**  
✘ dirty reads  
✘ non-repeatable reads

**Что допускает:**  
✔ phantom reads — в SQL стандарте  
Однако:  
📌 _В PostgreSQL phantom read предотвращается с помощью MVCC + predicate locks._

### 🧩 Под капотом (PostgreSQL):
- **Один snapshot** создаётся **на весь транзакционный контекст**
    Все чтения в транзакции происходят в одном “снимке” (snapshot).  
	То есть два SELECT подряд возвращают одни и те же строки.

- PostgreSQL использует:
    - MVCC → скрывает строки, появившиеся позже
        
    - Predicate locks → логически защищает диапазоны

---
## 4. SERIALIZABLE
Его цель — **заставить параллельные транзакции работать так, как будто они выполняются строго по очереди**.

**Что НЕ допускает:**  
❌ dirty reads  
❌ non-repeatable reads  
❌ phantom reads  
❌ lost updates  
❌ write skew и другие сложные аномалии

📌 Достигает максимальной корректности, но за счёт большего количества откатов.
