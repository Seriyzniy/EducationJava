По умолчанию коннектор Debezium отправляет каждую запись об изменении в топик, имя которой формируется из имени базы данных и имени таблицы, в которой было внесено изменение. 

Другими словами, **топик получает записи для одной физической таблицы**. 

Если вы хотите, чтобы топик получала записи для нескольких физических таблиц, необходимо настроить коннектор Debezium для перенаправления записей в эту топик.

## Логические таблицы
Логическая таблица — это объединение нескольких физических таблиц с одинаковой схемой (например, шардированных).

Таблицы вроде `db_shard1.my_table` и `db_shard2.my_table` физически разные, но логически составляют одну таблицу.  

Чтобы получать события из них в **одной теме Kafka**, используют SMT Topic Routing.

## **Секционированные таблицы PostgreSQL**
По умолчанию Debezium создаёт отдельную тему для каждого партиции секционированной таблицы PostgreSQL.  
Чтобы объединить все события в одну тему:

1. включают SMT Topic Routing;
    
2. отключают добавление уникализирующих полей:  
    **key.enforce.uniqueness = false** — потому что ключи в секционированной таблице уже гарантированно уникальны.

## Пример
Чтобы направить записи событий изменения для **нескольких физических таблиц в одну и ту же тему**, настройте преобразование маршрутизации тем в конфигурации Kafka Connect для коннектора Debezium. 

Для настройки маршрутизации тем SMT ==необходимо указать регулярные выражения==, которые определяют:
- **Таблицы**, для которых маршрутизируются записи. Все эти таблицы должны иметь одинаковую схему.
- **Название темы** назначения.

```css
transforms=Reroute transforms.Reroute.type=io.debezium.transforms.ByLogicalTableRouter transforms.Reroute.topic.regex=(.*)customers_shard(.*) transforms.Reroute.topic.replacement=$1customers_all_shards
```

# Свойства
## **topic.regex**
Указывает регулярное выражение, которое применяется к каждому событию изменения, чтобы определить, нужно ли маршрутизировать его в конкретную тему.  
**Значение по умолчанию:** _(нет значения по умолчанию)_

---
## **topic.replacement**
Указывает регулярное выражение, представляющее имя целевой темы.  

Каждая запись, совпадающая с `topic.regex`, направляется в тему, имя которой формируется на основе этого выражения.  
Выражение может ссылаться на группы, захваченные регуляркой в `topic.regex` — через `$1`, `$2` и т.д.  
**Значение по умолчанию:** _(нет значения по умолчанию)_

---
## **key.enforce.uniqueness**
Определяет, нужно ли добавлять поле в ключ события изменения, чтобы обеспечить уникальность ключей между несколькими таблицами, записи которых идут в одну тему.  
Это предотвращает коллизии ключей у записей с одинаковым ключом, но из разных исходных таблиц.

Установите `false`, если не хотите добавлять такое поле.  
Например, при маршрутизации из секционированной PostgreSQL-таблицы можно поставить `false`, так как уникальность ключей гарантируется самой БД.  
**Значение по умолчанию:** `true`

---
## **key.field.name**
Имя поля, которое добавляется в ключ события.  
Значение этого поля содержит имя исходной таблицы.  
SMT добавляет это поле только если `key.enforce.uniqueness = true`.  
**Значение по умолчанию:** `__dbz__physicalTableIdentifier`

---
## **key.field.regex**
Указывает регулярное выражение, которое применяется к имени целевой темы (по умолчанию), чтобы выделить группы символов.  
Используется только если `key.enforce.uniqueness = true`.  
**Значение по умолчанию:** _(нет значения по умолчанию)_

---
## **key.field.replacement**
Указывает выражение, определяющее значение добавляемого ключевого поля.  
Может ссылаться на группы, захваченные `key.field.regex`.  
Используется только если `key.enforce.uniqueness = true`.  
**Значение по умолчанию:** _(нет значения по умолчанию)_

---
## **schema.name.adjustment.mode**
Определяет, как должны корректироваться имена схем ключей сообщений, полученные из имени темы, чтобы быть совместимыми с конвертером данных:

- `none` — не применять корректировку (по умолчанию)
    
- `avro` — заменять недопустимые для Avro символы на подчёркивание
    

**Значение по умолчанию:** `none`

---
## **logical.table.cache.size**
Размер LRU-кеша, используемого для хранения максимального числа записей.

Кеш хранит:
- старую и новую схемы для логического ключа и значения таблицы
    
- результаты вычислений ключей
    
- результаты применения topic regex
    

Это ускоряет трансформацию исходных записей.  
**Значение по умолчанию:** `16`
