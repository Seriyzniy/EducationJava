
> [!NOTE] `RecordMetadata`
> это **неизменяемый (immutable)** объект, который ==описывает результат успешной отправки сообщения== продюсером в Kafka.  (ПРИ ПОДТВЕРЖДЕНИИ ACK)
> 
> Он **содержит служебную информацию** о том, **куда именно попало сообщение** (в какой топик, партицию, с каким смещением, временем и т. д.).
> 

Экземпляры этого класса ==возвращаются из метода==:
```java
Future<RecordMetadata> send(ProducerRecord<K, V> record)
```
или передаются в колбэке 
```java
Callback.onCompletion(
	RecordMetadata metadata, 
	Exception exception)
```

## Конструктор - НЕ ИСПОЛЬЗУЕТСЯ НАПРЯМУЮ

> Конструктор **публичный**, но используется в основном внутренне — пользователю обычно ==не нужно создавать== `RecordMetadata` вручную.  
> 
> Объекты создаются внутри клиента Kafka, когда брокер подтверждает приём сообщения.

```java
// Создает объект метаданных записи после успешной отправки продюсером.
public RecordMetadata(
    TopicPartition topicPartition,  // Топик и партиция
    long baseOffset,                // Смещение первой записи в батче
    long relativeOffset,            // Смещение текущей записи относительно baseOffset
    long timestamp,                 // Временная метка записи
    Long checksum,                  // Контрольная сумма CRC32 (устарела)
    int serializedKeySize,          // Размер сериализованного ключа (в байтах)
    int serializedValueSize,        // Размер сериализованного значения (в байтах)
    long batchSize,                 // Размер батча сообщений
    long logAppendTime,             // Время добавления в лог (если поддерживается брокером)
    long logStartOffset             // Начальное смещение сегмента лога
)
```
### 💬 Пояснения к параметрам:

- **`topicPartition`** — объект `TopicPartition`, указывающий имя топика и номер партиции.
    
- **`baseOffset`** — смещение начала батча сообщений (если продюсер отправил несколько).
    
- **`relativeOffset`** — смещение внутри батча.
    
- **`timestamp`** — фактическая временная метка, присвоенная брокером.
    
- **`checksum`** — контрольная сумма CRC32 (устарела начиная с Kafka 0.11).
    
- **`serializedKeySize`** — размер сериализованного ключа, в байтах.
    
- **`serializedValueSize`** — размер сериализованного значения, в байтах.
    
- **`batchSize`** — размер пакета (батча) сообщений, в байтах.
    
- **`logAppendTime`** — время, когда сообщение физически записано в лог на брокере.
    
- **`logStartOffset`** — первое смещение текущего лог-сегмента на брокере.

## Методы

### Основные геттеры
```java
// Возвращает имя топика, куда было отправлено сообщение.
public String topic()

// Возвращает номер партиции, куда было записано сообщение.
public int partition()

// Возвращает глобальное смещение (offset) записи в партиции.
public long offset()

// Возвращает временную метку (timestamp) записи.
public long timestamp()

// Возвращает размер сериализованного ключа.
public int serializedKeySize()

// Возвращает размер сериализованного значения.
public int serializedValueSize()

// Возвращает объект TopicPartition.
public TopicPartition topicPartition()
```

### Расширенные методы
```java
// Возвращает смещение начала батча (baseOffset).
```

## 💡 Особенности и подводные камни 💡

1️⃣ ❗❗❗`RecordMetadata` создается **только ПОСЛЕ ПОДТВЕРЖДЕНИЯ (ACK)** от брокера. 

2️⃣ 💡Если `acks=0`, метаданные не будут возвращены (так как подтверждения нет).  

3️⃣ Поле `offset` гарантированно уникально внутри одной партиции — оно указывает точное место сообщения.  

4️⃣ `timestamp` может быть либо:
- временем создания на продюсере (`CreateTime`),
    
- либо временем записи брокером (`LogAppendTime`) — зависит от конфигурации топика.  
- 
5️⃣ В Kafka до версии 0.11 использовалось поле `checksum` — сейчас оно **deprecated**. 

6️⃣ Метаданные можно использовать для **аудита, логирования, мониторинга** успешных отправок.  

7️⃣ Если используется **batching**, `baseOffset` и `relativeOffset` помогут понять, где именно сообщение находится в батче.
    

---
## 💎 Best Practices 💎

✅ 💡Используйте `RecordMetadata` в колбэке, чтобы логировать успешные отправки.  

✅ Храните `offset` и `partition`, если нужно обеспечить повторную доставку (идемпотентность).  

✅ Избегайте ручного создания `RecordMetadata` — это делает Kafka-клиент.  

✅ При тестировании можно мокать этот класс, если вы проверяете поведение вашего Callback.  

✅ Используйте `timestamp()` для метрик latency (сравнение времени отправки и подтверждения).

## Пример
```java
import org.apache.kafka.clients.producer.*;

import java.util.Properties;

public class RecordMetadataExample {
    public static void main(String[] args) {
        // Конфигурация продюсера
        Properties props = new Properties();
        props.put(
	        ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, 
	        "localhost:9092");
        props.put(
	        ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, "org.apache.kafka.common.serialization.StringSerializer");
        props.put(
	        ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, "org.apache.kafka.common.serialization.StringSerializer");


        try (KafkaProducer<String, String> producer = new KafkaProducer<>(props)
        ) {
            ProducerRecord<String, String> record =
                    new ProducerRecord<>(
	                    "orders", 
	                    "user123", 
	                    "Order created #54321");

            // ⭐⭐Отправляем сообщение и ждем результат⭐⭐
            RecordMetadata metadata = producer.send(record).get();

            // Логируем результат
            System.out.printf("""
                    ✅ Сообщение записано:
                    Топик: %s
                    Партиция: %d
                    Смещение: %d
                    Время: %d
                    Размер ключа: %d байт
                    Размер значения: %d байт
                    """,
                    metadata.topic(),
                    metadata.partition(),
                    metadata.offset(),
                    metadata.timestamp(),
                    metadata.serializedKeySize(),
                    metadata.serializedValueSize()
            );
        } catch (Exception e) {
            e.printStackTrace();
        }
    }
}
```
