
> [!NOTE] Класс `StreamsBuilder`
> это **высокоуровневый КОНСТРУКТОР ПОТОКОВОЙ ТОПОЛОГИИ (Topology)**.  
> 
> Он предоставляет **DSL (Domain Specific Language)** для описания логики обработки данных в декларативной форме.
> 

> 📘 Проще говоря, `StreamsBuilder` используется для создания **потоков (`KStream`, `KTable`, `GlobalKTable`)**, указания их источников и приёмников, а также описания всех трансформаций и соединений.  
> 
> После построения топологии вызывается `builder.build()`, и ❗❗результат передаётся в `KafkaStreams`.❗❗


---
## Конструктор
```JAVA
// Создаёт новый экземпляр StreamsBuilder.
// Обычно вызывается без параметров — это стартовая точка для определения топологии.
public StreamsBuilder()
```
Этот конструктор подготавливает внутреннюю структуру `Topology` и связывает её с DSL API.  
После создания можно добавлять источники, процессоры и приёмники (sinks).

---
## Методы
### Создание потоков и таблиц
```java
// Создаёт KStream из указанного топика.
// topics — одно или несколько имён топиков, которые будут источником данных.
public <K, V> KStream<K, V> stream(String... topics)

// Создаёт KStream из топика с пользовательским описанием (например, десериализация, timestampExtractor и т.п.)
// topic — имя топика.
// consumed — объект Consumed<K,V> с настройками источника.
public <K, V> KStream<K, V> stream(
	String topic, 
	Consumed<K, V> consumed)



// Создаёт KTable из топика (стрим будет преобразован в таблицу по ключу).
// topic — имя топика.
// consumed — настройки источника.
// materialized — параметры для хранения состояния (например, имя стора).
public <K, V> KTable<K, V> table(
	String topic, 
	Consumed<K, V> consumed, 
	Materialized<K, V, KeyValueStore<Bytes, byte[]> >
materialized)

// Создаёт GlobalKTable — глобальную таблицу, дублируемую во всех инстансах приложения.
// topic — имя топика.
// consumed — настройки источника.
public <K, V> GlobalKTable<K, V> globalTable(
	String topic, 
	Consumed<K, V> consumed)
```

💬 Комментарий
- `KStream` — поток (каждое событие обрабатывается как отдельное).
    
- `KTable` — таблица (обновления по ключу).
    
- `GlobalKTable` — реплицированная таблица для join'ов.


### Добавление произвольных источников, процессоров, синхнонизаторов
```java
// Добавляет источник данных (source node) в топологию.
// name — имя узла источника.
// topics — список топиков, из которых будет читаться поток.
public synchronized void addSource(String name, String... topics)


// Добавляет процессор в топологию на основе ProcessorSupplier.
// processorName — имя узла в топологии.
// supplier — фабрика, создающая экземпляры процессора.
// parentNames — имена родительских узлов, которые подают данные в этот процессор.
public synchronized void addProcessor(
	String processorName, 
	ProcessorSupplier<?, ?, ?, ?> supplier, 
	String... parentNames)

// Добавляет приёмник (sink) — узел, отправляющий данные в Kafka.
// name — имя узла.
// topic — имя выходного топика.
// parentNames — узлы, чьи данные поступают на выход.
public synchronized void addSink(
	String name, 
	String topic, 
	String... parentNames)
```

⚠️ Эти методы — **низкоуровневые** и используются редко, если ты работаешь с DSL.  
> ⭐⭐Обычно DSL (`stream()`, `table()`, `.map()`, `.filter()` и т.д.) автоматически создаёт такие узлы под капотом.

### Построение топологии
```java
// Строит финальную топологию на основе всех добавленных потоков и операций.
// Возвращает объект Topology, который можно передать в KafkaStreams.
public Topology build()
```
💬 После вызова `build()` дальнейшее добавление узлов в StreamsBuilder недопустимо — топология считается финализированной.

### Тестирование
```java
// Возвращает внутренний объект InternalStreamsBuilder (в основном используется фреймворком тестирования).
public InternalStreamsBuilder internalStreamsBuilder()
```
⚙️ Применяется в тестовых и внутренних сценариях, редко — вручную.

## 💡 Особенности и подводные камни 💡

1️⃣ **DSL — это декларативный слой**  
`StreamsBuilder` не запускает обработку. Он ==только описывает, _что_ нужно делать, но не _как_==. 
Исполнение происходит после 
```java
new KafkaStreams(builder.build(), props)
```

2️⃣ **Нельзя переиспользовать один StreamsBuilder для нескольких приложений**  
Каждый ==экземпляр должен соответствовать одной топологии.==

3️⃣ **Топология становится неизменяемой после build()**  
Попытка модифицировать после `build()` вызовет исключение.

4️⃣ ❗❗❗**Сильная связка с типами Serde (сериализаторы/десериализаторы)**  
Если не указать корректные Serde для ключей и значений, возможны runtime-ошибки сериализации.

5️⃣ **KTable и KStream ведут себя по-разному**  
Например, `KStream.to()` просто публикует новые записи, а `KTable.toStream()` преобразует изменения состояния таблицы в поток обновлений (changelog).

---
## 💎 Best Practices 💎

✅ **Используй `Consumed.with()` и `Produced.with()`**, чтобы явно указать типы Serde и избежать ошибок сериализации.

✅ **Храни итоговую топологию в переменной (`Topology topology = builder.build()`)** — удобно для отладки и визуализации.  

✅ **Не вызывай `build()` раньше времени** — добавлять узлы после этого нельзя.  

✅ **Используй `Materialized.as()` для KTable** — чтобы явно задать имя state store.  

✅ **Разделяй топологии логически** — большие цепочки лучше оформлять в отдельных методах для читаемости.

## Пример
```java
import org.apache.kafka.streams.*;
import org.apache.kafka.streams.kstream.*;
import java.util.*;

public class WordCountTopology {
    public static void main(String[] args) {
        Properties props = new Properties();
        props.put(
	        StreamsConfig.APPLICATION_ID_CONFIG, 
	        "wordcount-app");
        props.put(
	        StreamsConfig.BOOTSTRAP_SERVERS_CONFIG, 
	        "localhost:9092");
        props.put(
	        StreamsConfig.DEFAULT_KEY_SERDE_CLASS_CONFIG, 
	        Serdes.String().getClass());
        props.put(
	        StreamsConfig.DEFAULT_VALUE_SERDE_CLASS_CONFIG, 
	        Serdes.String().getClass());



        StreamsBuilder builder = new StreamsBuilder();

        // Создаём поток слов из топика "input-topic"
        KStream<String, String> textLines = builder
	        .stream("input-topic");

        // Преобразуем поток: разбиваем строки на слова, группируем и считаем количество
        KTable<String, Long> wordCounts = textLines
            .flatMapValues(value -> 
	            List.of(value.toLowerCase().split("\\W+")))
            .groupBy((key, word) -> word)
            .count(Materialized.as("word-counts-store"));

        // Отправляем результат в топик "output-topic"
        wordCounts
            .toStream()
            .to(
	            "output-topic", 
	            Produced.with(Serdes.String(), Serdes.Long())
	        );

        // Строим топологию
        Topology topology = builder.build();

        // Запускаем приложение
        KafkaStreams streams = new KafkaStreams(topology, props);
        
        Runtime.getRuntime().addShutdownHook(new Thread(streams::close));
        streams.start();
    }
}
```
