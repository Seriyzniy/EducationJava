# 1. Квота (Quota)
**Квота (quota)** в контексте **Apache Kafka** — это механизм **ограничения ресурсов**, который используется для того, чтобы:
> Контролировать, сколько данных (байт/запросов/операций) может отправлять или получать клиент (продюсер, консумер, администратор) за единицу времени.


Если не ограничивать использование, **один "жадный" клиент может перегрузить брокеров** и замедлить всех остальных.  

Чтобы этого не случилось, Kafka вводит **квоты на использование ресурсов**.

## Типы квот

|Тип квоты|Кого ограничивает|Что ограничивает|Настройка|
|---|---|---|---|
|producer_byte_rate|Продюсера|Скорость записи (байт/с)|broker / user / client ID|
|consumer_byte_rate|Консумера|Скорость чтения (байт/с)|broker / user / client ID|
|request_percentage|Клиента|Кол-во запросов|user / client ID|
|controller_mutation_rate|Админ-клиента|Операции create/delete|на брокере|
|alterClientQuotas|Любого клиента|Любые квоты|Admin API|
### 1️⃣ **Квоты для Producer**
- **Ограничивают скорость записи данных** (throughput) в байтах/сек.
    
- Позволяют предотвратить перегрузку брокеров из-за слишком высокой скорости продюсера.

🧩 _Пример:_  
Если квота `producer_byte_rate = 1048576`, то продюсер может писать максимум **1 МБ/с**.

---
### 2️⃣ **Квоты для Consumer**
- **Ограничивают скорость чтения** (fetch) данных с брокеров.
    
- Если консумер потребляет слишком быстро, Kafka начнёт искусственно задерживать его запросы.

🧩 _Пример:_  
`consumer_byte_rate = 2097152` → консумер может читать **до 2 МБ/с**.

---
### 3️⃣ **Квоты для Admin операций (в Kafka 2.6+)**
- Контролируют **скорость административных запросов**, например `DescribeTopics` или `ListOffsets`.
    
- Это защищает контроллер и метаданные от перегрузки из-за слишком частых вызовов Admin API.


🧩 _Пример:_  
`controller_mutation_rate` ограничивает количество мутаций (например, `createTopic`) в секунду.

---
### 4️⃣ **Квоты для пользователей / клиентов**
- Можно настроить **индивидуальные квоты для конкретного пользователя** (SASL-логин) или IP.
    
- Настраивается через **Client Quota API** (`Admin.alterClientQuotas()` / `describeClientQuotas()`).

🧩 _Пример:_  
Можно выдать пользователю `userA` квоту 1 МБ/с, а `userB` — 10 МБ/с.

---
## Где задаются квоты
Квоты задаются в конфигурациях брокера (`server.properties`) или динамически через Admin API

### Примеры статических конфигураций:
```properties
# Квоты по умолчанию
producer_byte_rate=1048576       # 1 MB/s
consumer_byte_rate=2097152       # 2 MB/s
controller_mutation_rate=10.0    # 10 операций/сек

# Включение квот для пользователей
client.quota.callback.class=org.apache.kafka.server.quota.ClientQuotaCallback
```

### Динамические квоты (через Admin API)
Можно программно изменить квоты для конкретных клиентов:
```java
import org.apache.kafka.clients.admin.*;
import java.util.*;

try (Admin admin = Admin.create(
	Map.of("bootstrap.servers", "localhost:9092"))
) {
    ClientQuotaEntity user = new ClientQuotaEntity(
	    Map.of("user", "userA")
	);
    
    ClientQuotaAlteration alter = new ClientQuotaAlteration(
        user,
        List.of(
	        new ClientQuotaAlteration.Op(
		        "producer_byte_rate", 1048576.0)
		    )
    );
    
    admin.alterClientQuotas(List.of(alter)).all().get();
}
```

# 2. ACL в Apache Kafka

> [!NOTE] **ACL (Access Control List)** 
> это механизм **контроля доступа** в Kafka.
> 
> Он определяет, **кто (какой пользователь)** может выполнять **какие действия** над **какими ресурсами** в кластере Kafka.
> 

По умолчанию Kafka открыта для всех — ==любой клиент может==:
- создать топик,
- писать и читать сообщения,
- удалять группы и т. д.
    

⚠️ Это **опасно в продакшене**, особенно в multi-tenant системах (где несколько сервисов используют один кластер).

➡️ Поэтому включают **аутентификацию (SASL / SSL)** и **авторизацию через ACL**, чтобы:
- ограничить права пользователей,
- разграничить продюсеров и консумеров,
- предотвратить случайные или злонамеренные действия.

## Структура ACL
Каждая запись ACL описывается структурой:

|Компонент|Описание|Пример|
|---|---|---|
|**Principal**|Кто выполняет действие (пользователь)|`User:alice`|
|**Permission Type**|Разрешено (`ALLOW`) или запрещено (`DENY`)|`ALLOW`|
|**Operation**|Какое действие разрешено/запрещено|`READ`, `WRITE`, `CREATE`, `DELETE`|
|**Resource Type**|К какому ресурсу относится правило|`TOPIC`, `GROUP`, `CLUSTER`, `TRANSACTIONAL_ID`|
|**Resource Name / Pattern Type**|Имя ресурса и тип шаблона|`topicA`, `PREFIXED`|
|**Host**|С какого хоста разрешено действие|`*` (все) или конкретный IP|

## Типы ресурсов к которым можно назначить ACL
|Тип ресурса|Пример|Что контролирует|
|---|---|---|
|**TOPIC**|`my-topic`|Доступ к чтению/записи в топик|
|**GROUP**|`my-group`|Доступ к группе консумеров|
|**CLUSTER**|`kafka-cluster`|Общие операции, например, создание топиков|
|**TRANSACTIONAL_ID**|`my-tx`|Транзакционные продюсеры|
|**DELEGATION_TOKEN**|`tokenA`|Авторизация через токены (Kafka ≥ 2.0)|

## Примеры операций (Operation)
|Операция|Применимо к|Описание|
|---|---|---|
|**READ**|Topic, Group|Чтение сообщений / получение смещений|
|**WRITE**|Topic|Отправка сообщений|
|**CREATE**|Topic, Cluster|Создание ресурса|
|**DELETE**|Topic, Group|Удаление ресурса|
|**DESCRIBE**|Все|Получение метаданных|
|**ALTER**|Все|Изменение конфигурации|
|**CLUSTER_ACTION**|Cluster|Операции контроллера|
|**IDEMPOTENT_WRITE**|Cluster|Для идемпотентных продюсеров|

## Типы шаблонов (PatternType)
Kafka поддерживает ==**три способа задания ресурса**==:

|Тип|Описание|Пример|
|---|---|---|
|**LITERAL**|Точное совпадение|`TOPIC:payments`|
|**PREFIXED**|Все ресурсы с данным префиксом|`TOPIC:payments-*`|
|**ANY**|Совпадает с любым ресурсом (обычно при фильтрации ACL)|—|

## Пример установки ACL
### Через консоль CLI
Создадим ACL, разрешающую пользователю `alice` писать и читать топик `payments`:
```bash
kafka-acls.sh \
  --authorizer-properties zookeeper.connect=localhost:2181 \
  --add \
  --allow-principal User:alice \
  --operation WRITE \
  --operation READ \
  --topic payments
```

Посмотреть все ACL:
```bash
kafka-acls.sh --list --authorizer-properties zookeeper.connect=localhost:2181
```

### Через Admin API
Программно можно добавлять и удалять ACL через `Admin` интерфейс:
```java
import org.apache.kafka.clients.admin.*;
import org.apache.kafka.common.acl.*;
import org.apache.kafka.common.resource.*;

import java.util.List;

public class AclExample {
    public static void main(String[] args) throws Exception {
        try (Admin admin = Admin.create(
	        Map.of("bootstrap.servers", "localhost:9092"))
	    ) {
            // Разрешить User:alice писать в топик payments
            AclBinding acl = new AclBinding(
                new ResourcePattern(
	                ResourceType.TOPIC, 
	                "payments", 
	                PatternType.LITERAL),
	                
                new AccessControlEntry(
	                "User:alice", 
	                "*", 
	                AclOperation.WRITE, 
	                AclPermissionType.ALLOW)
            );

            admin.createAcls(List.of(acl)).all().get();
            System.out.println("ACL добавлен!");
        }
    }
}
```
