
> [!NOTE] Consumer group
> Consumer group = логическая группа потребителей, которые совместно читают данные из одного или нескольких топиков, деля между собой партиции.
> 

Если нам нужно МАСШТАБИРОВАТЬ какой-либо из сервисов, то каждый из них будет считывать одно и то же сообщение (с одной и той же партиции). 
А нам это не нужно
![[Pasted image 20251112111727.png]]

Мы бы хотели, чтобы дубликаты Consumer, читали другие сообщения.
Для этого и есть Consumer Group
![[Pasted image 20251112112039.png]]

> [!NOTE] ⭐==ОДНА ПАРТИЦИЯ для ОДНОГО КОНСЮМЕРА== из группы⭐
>Kafka гарантирует, что каждый **потребитель внутри группы получает уникальный набор партиций** — то есть события распределяются равномерно, а не дублируются.

Несколько Consumer объединяются в группу "Consumer Group", для этого есть параметр `group.id`

**При запуске нескольких Consumer** мы можем **указать**, к какой **группе они относятся**, при помощи указания одинакового group.id

# 1. Балансировка партиций между Consumer
За балансировку партиций между Consumer Group отвечают КОНТТРОЛЛЕРЫ на кластере Kafka - ==Group Coorditanor==

> [!NOTE] **Group Coordinator** — специальный брокер Kafka, назначенный для каждой Consumer Group.
> - Каждый consumer в группе поддерживает heartbeat-сессию с координатором.
> - Когда что-то меняется (консюмер упал, появился новый, изменилось число партиций, изменились топики подписки), **Group Coordinator** инициирует процедуру ребалансировки.
> 

Каждый Consumer отправляет специальное `Heart Beats` сообщение, ПО УМОЛЧАНИЮ каждые 3 секунды. 
Тем самым говоря, "Я живой и могу обрабатывать данные"
![[Pasted image 20251112112619.png]]

Если Consumer не прислал сообщение (в течении этого времени), то Kafka Cluster считает, что этот Consumer "упал", и производит перебалансировку партиций

## КАК происходит ребалансировка (пошагово)
### 1. Выбор координатора группы
![[Pasted image 20251114115338.png]]
При запуске потребитель отправляет запрос `FindCoordinator` с `group.id`. 
Брокер хэширует `group.id` и ==вычисляет номер раздела== в топике `__consumer_offsets` через модульное деление:
```bash
hash(group.id)%кол-во_партиций(consumer_offsets)
```

Лидер этого раздела становится координатором группы. Брокер возвращает потребителю адрес координатора.


### 2. Присоединение участников
![[Pasted image 20251114115624.png]]
Затем потребители отправляют координатору запрос `JoinGroup` с информацией о своих подписках.  

```
JoinGroup Request =>  
	group_id => имя_группы  
	session_timeout_ms => общий_таймаут_сессии  
	rebalance_timeout_ms => таймаут_ребалансировки  
	member_id => идентификатор_консьюмера  
	protocols => // а это уже массив протоколов и их метаданные  
		name => имя_протокола (н-р.: range, roundrobin, cooperative-sticky, etc)    
		metadata => это те самые метаданные – наша заинтересованность в топиках
```

Координатор выбирает одного потребителя (обычно первого) лидером группы.  

В ответ он выдаёт каждому `memberId`, а лидеру — дополнительно полный список участников и их подписки, чтобы лидер мог выполнить распределение партиций с помощью выбранной стратегии.

### 3. Назначение разделов
![[Pasted image 20251114120633.png]]
После получения полного списка участников и информации о подписках, **лидер группы распределяет разделы** между потребителями **с помощью партиционера**. 

Затем лидер отправляет координатору запрос `SyncGroupRequest` с назначениями, а остальные потребители отправляют только свои ID. 

Координатор возвращает каждому потребителю его назначенные разделы, после чего начинается потребление данных.

### 4. Стратегии разделения партиций
#### **1. RangeAssignor (по диапазонам)**
- Делит партиции **по диапазонам** для каждого топика.
    
- Может привести к **неравномерному** распределению, если у топиков разное число партиций.
    
- Быстрее и проще всего.
    

#### **2. RoundRobinAssignor (круговой)**
- Распределяет партиции **равномерно по кругу**, независимо от топика.
    
- Более **равномерное** распределение между всеми консюмерами.
    
- Хорош для сценариев, когда все топики одинаково важны.
    

#### **3. StickyAssignor (липкое распределение)**
- Ставит цель **минимизировать перебалансировку** при изменении состава группы.
    
- Сохраняет существующее распределение как можно дольше.
    
- При этом старается обеспечить **равномерность**.
    
- Часто считается **оптимальной стратегией по умолчанию**.



## Консюмеры одной группы читают одну партицию
Если несколько консюмеров с одним **group.id** подключаются к топику с **одной партицией**, только **один** из них получает данные.  
Последний подключившийся перехватывает «указатель» чтения и продолжает вычитывать сообщения после завершения предыдущего.

![[Pasted image 20251106203708.png]]

Такой режим — разновидность **монопольного потребителя**.  
В отличие от JMS-очередей, где сообщения равномерно распределяются между консюмерами, в Kafka одна партиция всегда обслуживается только **одним консюмером** из группы.
