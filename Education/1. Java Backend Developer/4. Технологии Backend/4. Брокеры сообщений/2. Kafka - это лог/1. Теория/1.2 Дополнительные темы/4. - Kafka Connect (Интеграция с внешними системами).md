
# 📕1. Для чего нужен? 
## ❌ 1.Проблема: интеграция систем вручную
Представим задачу:  
тебе нужно выгружать данные из PostgreSQL (или другого источника) и отправлять их в Kafka.

### Как ты делаешь это кодом (без Connect)
Ты создаёшь микросервис, который:
1. Подключается к базе данных через JDBC.
    
2. Периодически выполняет SQL-запрос (или слушает CDC 
	>**CDC (Change Data Capture)** - все изменения в БД записываются в лог, а при получении данных читается этот лог
    
3. Преобразует результат в JSON.
    
4. Отправляет сообщение через `KafkaProducer`.
    
5. Хранит offset (где остановился, чтобы не читать заново).
    
6. Обрабатывает ошибки сети / Kafka / БД.
    
7. Реализует retry с backoff.
    
8. Логирует события и пишет метрики.
    
9. Масштабируется (чтобы читать быстрее).
    
10. Гарантирует _at-least-once_ (или _exactly-once_).
    

👎 **Проблемы такого подхода:**
- Нужно писать **много кода инфраструктуры**, не связанного с бизнес-логикой.
    
- Каждый новый источник (Postgres, Oracle, MongoDB, FTP) требует нового микросервиса.
    
- Нужно **самостоятельно реализовывать**:
    - управление offset'ами,
        
    - повторные попытки (retry),
        
    - мониторинг,
        
    - масштабирование,
        
    - согласованность данных.
        
- Тестировать и обновлять становится **дорого и рискованно**

## ✅ 2.Решение: Kafka Connect
**Kafka Connect** — это встроенный фреймворк Kafka для интеграции с внешними системами.  

Он берет на себя всё, что в ручном подходе ты должен был бы реализовывать кодом.
### Что делает Connect
- Управляет **чтением и записью** данных (источник / приёмник).
    
- Сам **хранит offset’ы** в Kafka (`connect-offsets` topic).
    
- Обеспечивает **масштабирование** через worker'ы и задачи (tasks).
    
- Делает **retry, backoff, dead-letter queue**.
    
- Предоставляет **REST API** для управления и мониторинга.
    
- ==Работает с готовыми плагинами-коннекторами==:
    - `PostgresSourceConnector`
        
    - `S3SinkConnector`
        
    - `ElasticsearchSinkConnector`
        
    - и десятки других.

💡 Connect = готовая **"шина интеграции"**, где ты просто указываешь _откуда_ и _куда_ гонять данные.  
А не пишешь каждый раз код.

| Подход                  | Ты управляешь                 |
| ----------------------- | ----------------------------- |
| 🧩 **Kafka Connect**    | Конфигурацией (декларативно)  |
| 💻 **Свой микросервис** | Кодом и логикой (императивно) |

### Без Connect (код микросервиса)
```java
while (true) {
    ResultSet rs = statement.executeQuery("SELECT * FROM orders WHERE id > ?");
    
    while (rs.next()) {
        String json = toJson(rs);
        producer.send(
	        new ProducerRecord<>("orders-topic", json)
	    );
        saveOffset(rs.getInt("id"));
    }
    Thread.sleep(1000);
}
```

### С Kafka Connect (конфигурация JSON)

|Задача|Свой микросервис|Kafka Connect|
|---|---|---|
|Чтение из источника|Нужно писать код (JDBC, REST, CDC)|Делает коннектор|
|Сериализация|Пишешь вручную (JSON, Avro)|Через `Converter` (JSON, Avro, Protobuf)|
|Отправка в Kafka|`KafkaProducer` вручную|Встроенный продюсер|
|Управление offset|Реализуешь сам|Встроенный (топик `connect-offsets`)|
|Retry / Backoff|Сам|Автоматически|
|Масштабирование|Сам (через Kafka Consumer Groups)|Через Connect cluster и tasks|
|Мониторинг|Сам (Prometheus, логи)|REST API + JMX|
|Изменение логики|Меняешь код|Меняешь JSON-конфиг|
|Поддержка 10 источников|10 сервисов|10 коннекторов (конфиги)|

# 📕2. Концепции Kafka Connect

> [!NOTE] **Kafka Connect** 
> это _фреймворк_ внутри Kafka для:
> - чтения данных **из внешних систем → в Kafka** (**Source Connectors**) 
> - и записи данных **из Kafka → во внешние системы** (**Sink Connectors**).
> 

👉 То есть это =="МОСТ"== между Kafka и другими системами (БД, файловыми хранилищами, REST API и т.д.), который:
- выполняет всё **надёжно, масштабируемо и без ручного кода**,
    
- автоматически обрабатывает **offset**, **ошибки**, **рестарты**, **распределение задач**,
    
- и позволяет работать **в продакшене без кастомных сервисов**

## 📙1. Режимы работы Kafka Connect
> [!NOTE] Режимы работы
> Kafka Connect можно запускать в **двух режимах** 
> - **standalone** 
> - **distributed**.
> 
### Standalone mode (однопроцессный)
📍 **Сценарий использования:** тестирование, отладка, локальная разработка.
- Запускается **один процесс** (один _Connect Worker_) на одной машине.
    
- Все коннекторы (source и sink) работают **внутри этого процесса**.
    
- Конфигурация хранится **в локальных файлах**, например:
```bash
connect-standalone.sh 
connect-standalone.properties 

my-source.properties 
my-sink.properties
```

❗Если процесс упадёт — все коннекторы останавливаются.

### Distributed mode (кластерный)
📍 **Сценарий использования:** продакшен, большие нагрузки, высокая доступность.

![[Pasted image 20251111114144.png]]
- Kafka Connect ==работает как кластер из нескольких Worker-ов== (обычно Docker-контейнеров или JVM-процессов).
    
- Все Worker-ы **объединяются одним параметром** `group.id`.
    
- Конфигурация коннекторов хранится **внутри Kafka-топиков**, а не на диске:
    - `connect-configs` — конфиги коннекторов
        
    - `connect-offsets` — информация о смещениях (offsets)
        
    - `connect-status` — статусы задач
        
- Kafka Connect распределяет **tasks (подзадачи)** между Worker-ами автоматически.
    
- Если один Worker упадёт — **его задачи перераспределяются** между другими.
    
- Управление коннекторами выполняется через **REST API**:
```http
POST /connectors
GET /connectors
DELETE /connectors/{name}
```
