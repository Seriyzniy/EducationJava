# 1. Параметры из книги + лекций на ютубе
**acks** - определяет надежность доставки:
- `0` - максимальная скорость, возможна потеря сообщений
    
- `1` - подтверждение от лидера раздела (баланс скорости и надежности)
    
- `all` - подтверждение от всех реплик (максимальная надежность)
 

**buffer.memory** - объем памяти для буферизации сообщений перед отправкой. 
При переполнении вызовы `send()` блокируются или генерируют исключение.


**compression.type** - алгоритм сжатия:
- `snappy` - хороший баланс скорости и сжатия
    
- `gzip` - лучшее сжатие, но выше нагрузка на CPU
    
- `lz4` - альтернативный алгоритм сжатия
 

**retries** - количество попыток повторной отправки ==при временных ошибках== (Retriable). Рекомендуется настраивать с учетом времени восстановления кластера.


**batch.size** - максимальный размер пакета сообщений в байтах. Увеличение размера улучшает пропускную способность, но использует больше памяти.


**[linger.ms](https://linger.ms/)** - время ожидания дополнительных сообщений перед отправкой пакета. Увеличение уменьшает накладные расходы, но увеличивает задержку.


**client.id** - идентификатор клиента для логирования и мониторинга.


**max.in.flight.requests.per.connection** - Управляет количеством сообщений, которые производитель может отправить серверу, не получая ответов. 
Значение 1 гарантирует порядок доставки при повторных отправках.

**Таймауты:**
- `request.timeout.ms` - 💡время ожидания ответа от брокера💡
    
- `metadata.fetch.timeout.ms` - время ожидания метаданных
    
- `max.block.ms` - время блокировки при вызове `send()`
    

**max.request.size** - максимальный размер одного запроса, ограничивает размер сообщений и пакетов.

**receive.buffer.bytes/send.buffer.bytes** - размеры TCP-буферов для оптимизации сетевого взаимодействия.

# 2. Остальные параметры 
**Пояснения**:
- Комбинация `acks=all` + `enable.idempotence=true` + `retries>0` даёт высокую гарантию доставки без дубликатов.
    
- `batch.size`, `linger.ms`, `compression.type` отвечают за пропускную способность (throughput) и задержку (latency).
    
- `buffer.memory`, `max.block.ms` контролируют поведение при взрыве нагрузки — если буферы переполнены, `send()` может блокировать.
    
- Если используете транзакции — `transactional.id` обязателен, иначе транзакции не работают.
    
- `metadata.max.age.ms` регулирует, как часто продюсер обновляет информацию о партициях/лидерах.

## Общие настройки подключения
```yaml
# Список брокеров Kafka, к которым продюсер подключается при старте.
bootstrap.servers=broker1:9092,broker2:9092

# Уникальный идентификатор продюсера (для логов и метрик брокера).
client.id=order-service-producer
```

## Сериализация сообщений
```yaml
# Класс сериализатора для ключа сообщения.
key.serializer=org.apache.kafka.common.serialization.StringSerializer

# Класс сериализатора для значения сообщения.
value.serializer=org.apache.kafka.common.serialization.StringSerializer
```

## Надёжность и гарантии доставки
```yaml
# 🟢Уровень подтверждения от брокеров:
# 0 — не ждать ответа,
# 1 — ждать подтверждения лидера,
# all — ждать подтверждения всех реплик в ISR.
acks=all

# Включает идемпотентность, чтобы сообщения не дублировались при повторных попытках.
enable.idempotence=true

# 🟢Максимальное количество повторных попыток при временных ошибках.
retries=5

# Минимальная задержка между повторными попытками.
retry.backoff.ms=100

# Максимальное количество неотправленных сообщений в полёте на одно соединение.
# Для идемпотентности должно быть ≤ 5.
max.in.flight.requests.per.connection=5

# Максимальное время ожидания подтверждения доставки сообщения (включая все retry).
delivery.timeout.ms=120000
```

## Буферизация и пакетная отправка
```yaml
# Максимальный размер batch (в байтах).
# Чем больше, тем выше пропускная способность, но и задержка.
batch.size=16384

# Задержка перед отправкой batch (в мс), чтобы накопить больше сообщений.
linger.ms=5

# 🟢Общий размер буфера памяти продюсера (в байтах).
buffer.memory=33554432

# Тип сжатия сообщений: none, gzip, snappy, lz4, zstd.
compression.type=lz4
```

## Транзакции (Transactional Producer)
```yaml
# Уникальный идентификатор продюсера для транзакций. Обязателен для transactional API.
transactional.id=tx-producer-1

# Максимальное время выполнения транзакции, после которого Kafka её прервет.
transaction.timeout.ms=60000
```

## Размеры сообщений и лимиты
```yaml
# Максимальный размер одного запроса к брокеру (в байтах).
max.request.size=1048576

# Таймаут ожидания ответа от брокера.
request.timeout.ms=30000

# 🔥Максимальное время блокировки вызова send(), если буферы заполнены.
max.block.ms=60000
```

## Метаданные и обновление кластера

```yaml
# Как часто обновлять метаданные о партициях и брокерах.
metadata.max.age.ms=300000

# Максимальное время ожидания получения метаданных от брокера.
metadata.fetch.timeout.ms=60000
```

## Управление соединениями
```yaml
# Время простоя соединения перед его закрытием.
connections.max.idle.ms=540000

# Размер TCP send буфера (в байтах). 0 — использовать настройки ОС.
send.buffer.bytes=131072

# Размер TCP receive буфера (в байтах). 0 — использовать настройки ОС.
receive.buffer.bytes=32768

# Задержка между попытками переподключения.
reconnect.backoff.ms=50

# Максимальная задержка между переподключениями.
reconnect.backoff.max.ms=1000
```

## Таймауты и обратная совместимость

```yaml
# Сколько времени клиент ждёт ответа при инициализации транзакции.
init.transactions.timeout.ms=60000

# Время ожидания для close() перед завершением отправки.
close.timeout.ms=5000
```

## Мониторинг и отладка
```yaml
# Включает логирование пропущенных метрик для продюсера.
metrics.recording.level=INFO

# Интервал обновления метрик (в мс).
metrics.sample.window.ms=30000

# Количество сэмплов метрик.
metrics.num.samples=2
```
## Контроль очередей и потоков
```yaml
# Максимальное количество сообщений в очереди перед отправкой.
max.block.ms=60000

# Сколько памяти (в байтах) может занять очередь перед блокировкой send().
buffer.memory=33554432
```
