### Пояснения к ключевым параметрам:
- **cleanup.policy** - определяет стратегию очистки: delete (удаление по времени/размеру) или compact (сохранение последнего значения по ключу)
    
- **[retention.ms/retention.bytes](https://retention.ms/retention.bytes)** - контроль времени и размера хранения данных
    
- **segment.bytes/[segment.ms](https://segment.ms/)** - влияют на размер файлов журнала, что важно для резервного копирования и удаления
    
- **max.message.bytes** - предотвращает попадание слишком больших сообщений, которые могут нарушить производительность
    
- **unclean.leader.election.enable=false** - обеспечивает безопасную работу, но может влиять на доступность при отказах
    
- **min.insync.replicas** - определяет минимальное количество реплик для подтверждения записи
    
- **min.cleanable.dirty.ratio** - контролирует частоту компактации логов
    
- **compression.type** - определяет алгоритм сжатия по умолчанию для топика
    
- **message.timestamp.type** - выбор между временем создания и временем добавления в лог

## Политики очистки и хранения
```yaml
# Политика очистки: delete (по времени/размеру) или compact (компактация ключей)
cleanup.policy=delete

# Комбинированная политика: компактация и удаление
cleanup.policy=compact,delete

# Максимальное время хранения сообщений (ms) до удаления
retention.ms=604800000   # 7 дней

# Максимальный общий размер хранения топика (в байтах)
retention.bytes=-1       # -1 означает без ограничения

# Минимальное время хранения сообщений (ms) - защита от преждевременного удаления
min.compaction.lag.ms=0

# Максимальное время задержки перед компактацией (ms)
max.compaction.lag.ms=9223372036854775807
```

## Сегментация и файлы
```yaml
# Максимальный размер сегмента логов (в байтах)
segment.bytes=1073741824   # 1 GB

# Максимальное время (ms) до смены сегмента
segment.ms=604800000        # 7 дней

# Задержка перед удалением файла после сегментации (ms)
file.delete.delay.ms=60000  # 1 минута

# Интервал проверки сегментов для удаления (ms)
segment.jitter.ms=0

# Время индексации сегментов (ms)
segment.index.bytes=10485760  # 10 MB
```

## Размеры сообщений и сжатие
```yaml
# Максимальный размер одного сообщения (в байтах)
max.message.bytes=1048588   # пример ≈1 MB

# Политика сжатия сообщений, если producer не задаёт явно
compression.type=producer

# Минимальный размер для сжатия (байты)
min.insync.replicas=1

# Размер индекса сегмента (байты)
index.interval.bytes=4096
```

## Репликация и лидерство
```yaml
# Фактор репликации топика
replication.factor=3

# Минимальное количество in-sync реплик
min.insync.replicas=2

# 🔥Разрешены ли выборы несинхронизированного лидера (не ISR)
unclean.leader.election.enable=false

# Таймаут для репликации (ms)
replica.lag.time.max.ms=30000

# Максимальное отставание реплики (сообщения)
replica.lag.max.messages=4000
```

## Временные метки
```yaml
# Тип временной метки сообщений: CreateTime или LogAppendTime
message.timestamp.type=CreateTime

# Максимальная разница во времени метки сообщения от брокера (ms)
message.timestamp.difference.max.ms=9223372036854775807

# Время задержки перед добавлением временной метки (ms)
message.timestamp.before.max.ms=9223372036854775807

# Время задержки после добавления временной метки (ms)
message.timestamp.after.max.ms=9223372036854775807
```

## Компактация (для cleanup.policy=compact)
```yaml
# Минимальное допустимое соотношение "грязной" части для компактации
min.cleanable.dirty.ratio=0.5

# Максимальное количество сообщений в одном пакете компактации
max.compaction.lag.ms=9223372036854775807

# Удалять ли записи с null ключами при компактации
delete.retention.ms=86400000  # 1 день

# Минимальное время хранения удаленных записей (ms)
min.compaction.lag.ms=0

# Размер сегмента для компактации (байты)
segment.bytes=268435456  # 256 MB для более частой компактации
```
