## 1. Broker — параметры надежности
```properties
# 💡Минимальное количество ISR (in-sync replicas), необходимых для подтверждения записи.
min.insync.replicas=2

# 💡Количество реплик (копий) по умолчанию при создании топика
default.replication.factor=3

# Таймаут ожидания подтверждения записи от ISR (мс)
replica.lag.time.max.ms=10000

# Частота синхронизации данных с диском для логов
log.flush.interval.messages=10000
log.flush.interval.ms=1000

# Включает fsync при каждой записи (медленно, но максимально надежно)
flush.messages=1

# Разрешает автоматическое создание топиков (лучше выключить для контроля)
auto.create.topics.enable=false

# Интервал проверки статуса реплик
replica.fetch.backoff.ms=1000

# Размер буфера для чтения репликами
replica.fetch.max.bytes=1048576
```

**Комментарии:**
- `min.insync.replicas` = 2 гарантирует, что запись считается успешной **только если данные записаны хотя бы в две реплики** (лидер + хотя бы одна копия).
    
- `default.replication.factor=3` — стандарт индустрии для высокой отказоустойчивости.
    
- `auto.create.topics.enable=false` предотвращает случайное создание топиков без нужного уровня репликации.
    
- `log.flush.*` управляет частотой записи на диск — чем чаще, тем надежнее, но медленнее.

## 2. Topic — параметры надежности
```properties
# Количество реплик для каждого партиции
replication.factor=3

# Минимум ISR для подтверждения записи
min.insync.replicas=2

# Политика удаления: удерживать, пока не удалится вручную или не достигнет лимита
cleanup.policy=delete
retention.ms=604800000  # 7 дней
retention.bytes=-1      # без ограничения по размеру

# Политика сжатия на уровне брокера
compression.type=producer

# Контроль максимального размера сообщения
max.message.bytes=1048576
```

**Комментарии:**
- `min.insync.replicas` + `acks=all` (см. ниже) дают гарантию **"exactly-once delivery"** при включенных транзакциях.
    
- `retention.ms` важно, чтобы не потерять данные из-за преждевременного удаления.
    
- `compression.type=producer` позволяет использовать ту же компрессию, что задает продюсер.

## 3. Producer — параметры надежной доставки
```properties
# Адреса брокеров
bootstrap.servers=broker1:9092,broker2:9092,broker3:9092

# 💡Гарантирует, что сообщение записано на все ISR
acks=all

# 💡Количество попыток при временных сбоях
retries=2147483647

# ⭐⭐Включить идемпотентность (предотвращает дубли)
enable.idempotence=true

# 💡Максимум неотправленных запросов в полете (важно при идемпотентности)
max.in.flight.requests.per.connection=5  # при идемпотентности ≤5

# Таймаут ожидания подтверждения от брокера
request.timeout.ms=30000

# Таймаут доставки сообщения
delivery.timeout.ms=120000

# Интервал между повторными попытками
retry.backoff.ms=100

# Контроль размера буфера
buffer.memory=33554432

# Таймаут записи в топик
linger.ms=10
batch.size=32768

# 🧠Включить транзакции (для atomic send)
transactional.id=tx-producer-1
```
**Комментарии:**
- `acks=all` + `min.insync.replicas >= 2` — база надежности.
    
- `enable.idempotence=true` предотвращает дубликаты при повторных попытках.
    
- `transactional.id` включает **Kafka Transactions**, позволяя атомарно писать в несколько топиков и добиться exactly-once.
    
- `retries` и `delivery.timeout.ms` контролируют время до отказа при сбое.
    
- `max.in.flight.requests.per.connection` должен быть ≤5 для корректной работы идемпотентности.

## 4. Consumer — параметры надежного чтения

Потребители в основном теряют сообщения, когда фиксируют смещения для прочитанных, но еще не полностью обработанных событий.

```properties
# Уникальный ID группы
group.id=my-consumer-group

# ⭐Смещение сохраняется автоматически (или вручную)
enable.auto.commit=false

# Максимальное время обработки сообщения
max.poll.interval.ms=300000

# Максимальное число сообщений за один poll
max.poll.records=500

# Таймаут сессии с брокером
session.timeout.ms=10000

# Интервал между heartbeat (для координации с группой)
heartbeat.interval.ms=3000

# Интервал повторных попыток при сбое
retry.backoff.ms=100

# Чтение только зафиксированных транзакций
isolation.level=read_committed

# 💡Минимизирует повторное чтение старых сообщений
auto.offset.reset=latest

# Частота авто-коммитов
auto.commit.inter val.ms.
```
