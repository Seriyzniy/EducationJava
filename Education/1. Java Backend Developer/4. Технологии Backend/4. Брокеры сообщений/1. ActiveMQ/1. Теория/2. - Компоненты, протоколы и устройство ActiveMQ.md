# 📕1. Брокер 
Центральная часть промежуточного программного обеспечения, распределяющая сообщения.

# 📕2. Клиент

> [!NOTE] Клиент
> **Часть программного обеспечения**, которая **перебрасывает сообщения с помощью брокера**. 


Она, в свою очередь, состоит из следующих артефактов:  
- **Код**, использующий **API JMS** (==Java Mesage Service==).
	
- JMS API — это набор интерфейсов для взаимодействия с брокером в соответствии с гарантиями, изложенными в спецификации JMS.
	
- Клиентская библиотека системы, которая обеспечивает реализацию API и взаимодействует с брокером.


Клиент и брокер **общаются друг с другом через протокол прикладного уровня**, также известный, как _протокол взаимодействия_

Спецификация JMS оставила детали этого протокола конкретным реализациям.
![[Pasted image 20251101102719.png]]

JMS использует термин **провайдер** для ==обозначения реализации системы обмена сообщениями==, включающей брокер и клиентские библиотеки.

Реализация ActiveMQ как **JMS-провайдера** определила многие архитектурные решения: 
- спецификация возлагает основную ответственность за доставку сообщений на **брокер**, 
- а клиент лишь взаимодействует с адресатом (очередью или топиком).

Такой подход упростил работу клиента, но усложнил брокер и повлиял на **производительность ActiveMQ**. 

Кроме того, для соответствия стандарту JMS пришлось учитывать множество особенностей пакета совместимости от Sun Microsystems, что также сказалось на эффективности.

# 📕3. Сохранение сообщений
**ActiveMQ** использует подключаемые **АДАПТЕРЫ персистентности** — ==механизмы хранения сообщений==. 

> Основные из них: 
> - **KahaDB**, 
> - **LevelDB** 
> - и на возможности использования базы данных через **JDBC**

## Сохранение в журнал в двоичной форме
Когда **брокер получает** персистентные **сообщения**, **они СНАЧАЛА ЗАПИСЫВАЮТСЯ на диск В ЖУРНАЛ**. 
==Журнал== — это структура данных на диске, в которую **можно только добавлять данные** и состоящая из нескольких файлов. 

Входящие сообщения сериализуются брокером в независимое от протокола представление объекта, а затем маршализируются в двоичную форму, которая затем записывается в конец журнала. 

Журнал содержит лог всех входящих сообщений, а также сведения о тех сообщениях, которые были подтверждены, как прочитанные клиентом.

## Индексные файлы
Для ускорения доступа используются **индексы**, указывающие позиции сообщений в журнале. 
Когда все сообщения из файла журнала будут прочитаны, они будут либо удалены, либо заархивированы фоновым рабочим потоком ActiveMQ.
Если журнал повреждён, **ActiveMQ восстанавливает его** по логам.

## Общий журнал для ВСЕХ очередей
**Сообщения из всех очередей записываются в одни и те же файлы журнала**, что означает, что, если одно сообщение не прочитано, то весь файл (обычно по умолчанию 32 МБ или 100 МБ, в зависимости от адаптера персистентности) не может быть очищен, что может привести к нехватке места. 

> ActiveMQ НЕ ПРЕДНАЗНАЧЕН для долгосрочного хранения — **сообщения нужно регулярно вычитывать**.

Журналы эффективны благодаря **последовательному доступу к диску**, что особенно выгодно как для HDD, так и для SSD.

# 📕4. Связь
Хотя спецификация **JMS** чётко определяет API и поведение, она **не регламентирует протокол обмена** между клиентом и брокером. 

Поэтому ==ActiveMQ разработал собственный протокол== — **OpenWire**, используемый клиентскими библиотеками для Java, .NET (NMS) и C++ (CMS).

Позже ActiveMQ добавил поддержку других протоколов (например, **AMQP, MQTT, STOMP, REST, WebSocket**), что обеспечило широкую совместимость с различными языками и платформами.

## AMQP 1.0 
**AMQP 1.0** (Advanced Message Queuing Protocol) — двоичный протокол обмена сообщениями общего назначения (ISO/IEC 19464:2014), не совместимый с ранними версиями (например, AMQP 0.9.1 в RabbitMQ). 

> Он НЕ РАЗЛИЧАЕТ клиентов и брокеров, поддерживает **управление потоками, транзакции** и различные **уровни QoS**: 
> - `_at most once_` (не более одного раза)
> - `_at least once_` (не менее одного раза) 
> - ❗❗`_exactly once_`❗❗ (и точно один раз).

## STOMP
**STOMP** (Simple/Streaming Text Oriented Messaging Protocol) — простой текстовый протокол обмена сообщениями, легко реализуемый и поддерживаемый множеством клиентов на разных языках.

## XMPP
**XMPP** (Extensible Messaging and Presence Protocol) — расширяемый протокол для обмена сообщениями и информации о присутствии, основанный на XML. 

Изначально создавался для чатов (под именем Jabber), но позже был расширен для поддержки сценариев pub/sub (==публикация-подписка==), VoIP и передачи данных.

## **MQTT**
**MQTT** — легковесный протокол «==публикация-подписка==» (стандарт ISO/IEC 20922), широко используемый для M2M и IoT-приложений.

**ActiveMQ** ==поддерживает различные протоколы== (ВЫШЕ ПЕРЕЧИСЛЕННЫЕ) ==поверх WebSockets==, обеспечивая полнодуплексную связь между веб-браузерами и брокером. 

❗❗❗**Важные ограничения**❗❗❗:
При работе ==со сторонними библиотеками== и внешними компонентами необходимо учитывать, что они ==имеют переменное качество и могут быть несовместимы== с функциями, предоставляемыми в ActiveMQ. 

> Как очень простой пример — невозможно отправить сообщения в очередь через MQTT (без настройки маршрутизации в брокере). 
> 
> Таким образом, вам нужно будет потратить некоторое время на работу с опциями, чтобы определить стек системы обмена сообщений, наиболее подходящий для требований вашего приложения.

# 📕 4. Компромисс между производительностью и надежностью

Перед изучением работы режима **«точка-точка»** в ActiveMQ важно понять ==основной компромисс== систем с интенсивной обработкой данных — **между производительностью и надежностью**.

==Любая система==, принимающая данные, будь то брокер сообщений или база данных, ==должна быть проинструктирована== о том, ==как следует обрабатывать эти данные **В СЛУЧАЕ СБОЕВ==**.

==Если брокер сообщений== теряет питание (==отрубается==): 
- **данные**, находящиеся **в памяти (ОЗУ) - `пропадают`**, 
- **а записанные на диск — `сохраняются`**. 

Поэтому для надежности сообщения нужно **сохранять в постоянное хранилище**, что значительно снижает скорость: запись на диск в **100–1000 раз медленнее**, чем в память.

Разработчик должен выбрать баланс — высокая надежность снижает производительность, а хранение сообщений только в памяти ускоряет работу, но повышает риск потери данных.

> По умолчанию **ActiveMQ и JMS ориентированы на надежность**(БЕЗ Персистентности (записи на диск)), НО позволяют гибко настраивать этот баланс. 

После оптимизации одного брокера систему можно масштабировать — распределяя потоки сообщений между несколькими брокерами или выделяя отдельные очереди.
