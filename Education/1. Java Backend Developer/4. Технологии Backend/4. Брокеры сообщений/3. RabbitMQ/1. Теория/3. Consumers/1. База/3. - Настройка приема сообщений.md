# 1. Режимы подтверждения (Acknowledgement Modes)

При регистрации потребителя (consumer) приложения могут выбрать один из двух режимов доставки сообщений:

- **Автоматический** (сообщения не требуют подтверждения получения, известен как принцип "выстрелил и забыл").
    
- **Ручной** (сообщения требуют подтверждения от клиента).


Подтверждения от потребителей — это тема отдельного руководства в документации, вместе с тесно связанной концепцией **подтверждений от издателя (publisher confirms)**.
[Consumer Acknowledgements and Publisher Confirms | RabbitMQ](https://www.rabbitmq.com/docs/confirms)

---
# 2. Ограничение одновременных доставок с помощью Prefetch (Предварительной выборки)
При использовании ручного режима подтверждения у потребителей есть способ ограничить количество сообщений, которые могут находиться "в обработке" (в пути по сети или доставлены, но не подтверждены). Это помогает избежать перегрузки потребителя.

Настраивается при помощи свойства `prefetch_count`

Эта функция вместе с подтверждениями потребителей также рассматривается в отдельном руководстве.
[Consumer Acknowledgements and Publisher Confirms | RabbitMQ](https://www.rabbitmq.com/docs/confirms)

# 3. Метрика Consumer Capacity (Пропускная способность потребителя)
Интерфейс управления RabbitMQ, а также конечные точки данных для мониторинга (например, для сбора метрик Prometheus) отображают для отдельных очередей метрику под названием **consumer capacity** (ранее — consumer utilisation, "использование потребителя").


`Consumer capacity` — метрика RabbitMQ, показывающая, какую долю времени **ОЧЕРЕДЬ способна немедленно доставлять сообщения потребителям**.

Она нужна, чтобы понять: хватает ли текущих потребителей или нужно добавить ещё экземпляров сервиса.

### Как интерпретировать
- **100%** — потребители успевают обрабатывать сообщения; очередь в любой момент готова отдавать сообщения дальше.  
    Даже если сообщений нет, значение будет 100% — потребители справляются с нагрузкой.
    
- **<100%** — очередь могла бы отдавать быстрее, но ограничена:
    - мало потребителей,
        
    - потребители медленно обрабатывают сообщения,
        
    - слишком маленький `prefetch` (слишком мало сообщений в работе у потребителей).
        
- **0%** — в очереди **нет активных потребителей**.

### Зачем это нужно
Метрика помогает решить, нужно ли:
- увеличить количество потребителей,
- ускорить обработку,
- поднять `prefetch`,
- изменить архитектуру очередей.  

Это **ориентировочный индикатор**. Для точной оценки приложения должны собирать свои метрики обработки сообщений.

# 4. Тайм-аут подтверждения доставки (Delivery Acknowledgement Timeout)
==RabbitMQ устанавливает тайм-аут на подтверждение получения доставки== потребителем. 

Это защитный механизм, который обнаруживает ситуации, когда потребители не подтверждают доставку сообщений. 

Настройка тайм-аута подтверждения доставки может помочь предотвратить компактизацию данных на диске и исчерпание дискового пространства на узлах.

## **Как это работает**  
**Если потребитель не подтверждает (ack) свою доставку** в течение значения тайм-аута, **его канал закрывается С ИСКЛЮЧЕНИЕМ КАНАЛА** `PRECONDITION_FAILED`. 

Сообщение об ошибке будет выглядеть примерно так:
```bash
Consumer 'consumer-tag-998754663370' on channel 1 and queue 'qq.1' in vhost '/' has timed out
waiting for a consumer acknowledgement of a delivery with delivery tag = 10. Timeout used: 180000 ms.
```

> Это значение тайм-аута можно настроить. Подробнее см. в руководстве по потребителям.

***Ошибка записывается в лог узла, к которому был подключен потребитель***. *Все последующие сообщения* по этому каналу (от всех потребителей) затем *возвращаются в очередь*.

Чтобы устранить исключение канала `PRECONDITION_FAILED`, пересмотрите работу вашего потребителя и рассмотрите возможность увеличения значения тайм-аута.

==Значение тайм-аута по умолчанию== в RabbitMQ составляет **30 минут**
> Проверка необходимости применения тайм-аута выполняется периодически, с интервалом в одну минуту. 
> Значения меньше одной минуты не поддерживаются, а значения меньше пяти минут не рекомендуются.

## **Конфигурация на уровне узла**  
Значение тайм-аута можно настроить в файле `rabbitmq.conf` (в миллисекундах):

```bash
# 30 minutes in milliseconds
consumer_timeout = 1800000
```

```bash
# one hour in milliseconds  
consumer_timeout = 3600000
```

Тайм-аут можно отключить с помощью файла `advanced.config`. Это **не рекомендуется**:

```erlang
%% advanced.config
[
  {rabbit, [
    {consumer_timeout, undefined}
  ]}
].
```

Вместо полного отключения тайм-аута рассмотрите возможность использования большого значения (например, несколько часов).

## **Конфигурация на уровне очереди**  
Начиная с RabbitMQ 3.12, значение тайм-аута также можно настраивать для каждой очереди отдельно.

## **Тайм-ауты доставки на уровне очереди с использованием политики (Policy)**  
Установите ключ политики `consumer-timeout`.

Значение должно быть указано в миллисекундах. Проверка необходимости применения тайм-аута выполняется периодически, с интервалом в одну минуту.

```bash
# override consumer timeout for a group of queues using a policy
rabbitmqctl set_policy queue_consumer_timeout "with_delivery_timeout\.*" '{"consumer-timeout":3600000}' --apply-to classic_queues
```

## **Тайм-ауты доставки на уровне очереди с использованием опционального аргумента очереди**  
Установите опциональный аргумент очереди `x-consumer-timeout` ==при объявлении очереди.== 
Тайм-аут указывается в миллисекундах. 

Проверка необходимости применения тайм-аута выполняется периодически, с интервалом в одну минуту.

# 5. **Ограничение количества потребителей на канал**
В некоторых сценариях, где возможны утечки потребителей, полезно о**граничить количество активных потребителей на каждом канале.** 

Это можно настроить в файле `rabbitmq.conf` с помощью параметра `consumer_max_per_channel`:
```bash
consumer_max_per_channel = 100
```

# 6. **Эксклюзивность (Exclusivity)**
**Только для классических очередей** (Classic Queues): **при регистрации потребителя** с помощью клиента AMQP 0-9-1 **ФЛАГ** `exclusive` метода `basic.consume` можно установить в `true`, чтобы запросить, чтобы данный потребитель был единственным в целевой очереди. 
> Вызов завершится успешно, только если на момент регистрации в очереди еще нет зарегистрированных потребителей. 
> 
> Это ==позволяет гарантировать, что одновременно из очереди будет потреблять только один потребитель.==

**Если эксклюзивный потребитель отменяется** или завершает работу, **ответственность за регистрацию нового** потребителя для продолжения потребления из очереди **лежит на приложении**.

Если требуется эксклюзивное потребление и непрерывность обработки сообщений, используйте **single active consumer** (единственный активный потребитель).

> **Важно:**  
> Очереди Quorum игнорируют флаг `exclusive` в кадре `basic.consume`. Для очередей Quorum вместо этого используйте механизм **single active consumer**.

# 7. **Единственный активный потребитель (Single Active Consumer - SAC)**
**Единственный активный потребитель (Single Active Consumer, SAC)** ==гарантирует, что из очереди будет потреблять только один активный потребитель.== 
Если он отменяется или завершает работу, очередь автоматически переключается на другого зарегистрированного потребителя. 
Это важно, когда сообщения должны обрабатываться строго по порядку.

Типичная последовательность событий выглядит следующим образом:
1. **Объявляется очередь, и несколько потребителей регистрируются в ней** примерно в одно и то же время.
    
2. **Самый первый** зарегистрированный потребитель **становится единственным активным потребителем**: сообщения направляются ему, а остальные потребители игнорируются.
    
3. **Если очередь является очередью Quorum и регистрируется новый потребитель с более высоким приоритетом**, ==то очередь прекращает доставлять сообщения текущему активному потребителю.== Когда все сообщения будут подтверждены, новый потребитель становится активным.
    
4. Если единственный активный потребитель по какой-либо причине отменяется или просто завершает работу, выбирается другой потребитель в качестве активного. Другими словами, **очередь автоматически переключается (осуществляет failover) на другого потребителя.** Более подробную информацию о том, как выбирается новый потребитель, смотрите в разделе **Поведение SAC**. ([Consumers | RabbitMQ](https://www.rabbitmq.com/docs/consumers#initial-sac-selection))


> **ВАЖНО**
> Обратите внимание, что без включенной функции "Единственный активный потребитель" сообщения распределялись бы между всеми потребителями по принципу циклического перебора (round-robin).

> **Предупреждение:**  
> Этот раздел посвящен механизму "Единственный активный потребитель", доступному для клиентов AMQP 0-9-1 и AMQP 1.0 на классических очередях и очередях Quorum. Он значительно отличается от функции **Single Active Consumer для потоков (streams)**.
> 
> ==Попытка включить SAC с помощью клиента AMQP 0-9-1 НА ПОТОКЕ НЕ СРАБОТАЕТ==. 
> Для использования SAC на потоке необходимо использовать нативный клиент протокола RabbitMQ Stream.

## **Включение "Единственного активного потребителя" на очередях Quorum и Classic**
"Единственный активный потребитель" ==можно включить при объявлении очереди==, установив аргумент `x-single-active-consumer` в значение `true`. 

Например, с клиентом Java:
```java
Channel ch = ...;

Map<String, Object> arguments = new HashMap<String, Object>();
arguments.put("x-single-active-consumer", true);

ch.queueDeclare("my-queue", false, false, false, arguments);
```

## **Отличие от эксклюзивных потребителей (Exclusive Consumers)**
По сравнению с эксклюзивным потребителем AMQP 0-9-1, механизм **"Единственный активный потребитель" СОЗДАЕТ МЕНЬШЕ НАГРУЗКИ НА СТОРОНУ ПРИЛОЖЕНИЯ для поддержания непрерывности потребления.** 

> **ПРОСТОТА**
> Потребителям просто нужно быть зарегистрированными, а ***переключение при сбое обрабатывается автоматически*** — НЕТ НЕОБХОДИМОСТИ ОБНАРУЖИВАТЬ СБОЙ АКТИВНОГО ПОТРЕБИТЕЛЯ и регистрировать нового.

## Determining Which Consumer is Currently Active
Интерфейс управления (management UI) и CLI могут сообщать, какой потребитель в данный момент является активным для очереди, где включена эта функция.

## **Первоначальный выбор SAC**
**При использовании с классической** очередью ==первоначальный активный потребитель== выбирается **случайным образом**, *даже если используются приоритеты потребителей.*

**Если очередь является очередью Quorum** ==и регистрируется новый потребитель с более высоким приоритетом==, **то очередь прекращает доставлять сообщения текущему активному** потребителю. 
Когда все сообщения будут подтверждены, новый потребитель становится активным.

Подробнее об этом поведении читайте в блоге, анонсирующем эту функцию.
([RabbitMQ 4.0: New Quorum Queue Features | RabbitMQ](https://www.rabbitmq.com/blog/2024/08/28/quorum-queues-in-4.0#consumer-priorities-combined-with-single-active-consumer))

## **SAC и эксклюзивные потребители являются взаимоисключающими**
Попытка зарегистрировать эксклюзивного потребителя при включенном SAC приведет к ошибке. По определению SAC предполагает наличие нескольких подключенных потребителей.

## **SAC и предварительная выборка (prefetch) на канале**
**Сообщения всегда доставляются активному потребителю**, ==даже если в какой-то момент он слишком занят==. Это может произойти при использовании ручного подтверждения и `basic.qos`: потребитель может быть занят обработкой максимального количества неподтвержденных сообщений, которое он запросил с помощью `basic.qos`. 

В этом случае другие потребители игнорируются, а доставка возвращается в очередь.

## SAC нельзя включить через политику.
Функция "Единственный активный потребитель" задаётся только при объявлении очереди через аргументы. 

**Политики динамические, и их использование могло бы нарушить порядок обработки**: внезапное отключение SAC заставит брокер отправлять сообщения нескольким потребителям одновременно, что противоречит цели SAC.
