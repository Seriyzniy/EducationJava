Метод `basic.qos` в AMQP 0-9-1 ограничивает число **неподтверждённых сообщений** на канале (или соединении) при потреблении (так называемый "prefetch count", лимит предварительной выборки).

По стандарту этот лимит **делится между всеми потребителями на канале** — что приводит к сложной и медленной координации, особенно в кластере.


В RabbitMQ реализовано более практичное поведение:

==В RabbitMQ prefetch применяется отдельно к каждому потребителю==, даже если они работают на одном канале.

| Значение `prefetch_count` в AMQP 0-9-1                           | Значение `prefetch_count` в RabbitMQ                                         |
| ---------------------------------------------------------------- | ---------------------------------------------------------------------------- |
| **Разделяется (shared)** **между всеми потребителями** на канале | Применяется **отдельно (separately)** к каждому новому потребителю на канале |

## **Один потребитель**
Следующий простой пример на Java будет получать максимум **10 неподтвержденных сообщений одновременно**:
```java
Channel channel = ...;
Consumer consumer = ...;

channel.basicQos(10); // Лимит на потребителя

channel.basicConsume("my-queue", false, consumer);
```

Значение **0** рассматривается как **бесконечность**, разрешая любое количество неподтвержденных сообщений.
```java
Channel channel = ...;

Consumer consumer = ...;

channel.basicQos(0); // Нет лимита для этого потребителя
channel.basicConsume("my-queue", false, consumer);
```

## **Независимые потребители**
Этот пример запускает двух потребителей на одном канале, каждый из которых будет независимо получать максимум **10 неподтвержденных сообщений одновременно**:
```java
Channel channel = ...;

Consumer consumer1 = ...;
Consumer consumer2 = ...;

channel.basicQos(10); // Лимит на потребителя

channel.basicConsume("my-queue1", false, consumer1);
channel.basicConsume("my-queue2", false, consumer2);
```

----
## **Несколько потребителей, разделяющих лимит**
Спецификация AMQP 0-9-1 не объясняет, что происходит, если вызывать `basic.qos` несколько раз с разными глобальными значениями. 

RabbitMQ интерпретирует это как то, что **два лимита предварительной выборки должны применяться независимо друг от друга**; ==потребители будут получать новые сообщения только тогда, когда== **ни один из лимитов** на неподтвержденные сообщения НЕ достигнут.

```java
Channel channel = ...;

Consumer consumer1 = ...;
Consumer consumer2 = ...;

channel.basicQos(10, false); // Лимит на потребителя
channel.basicQos(15, true);  // Лимит на канал

channel.basicConsume("my-queue1", false, consumer1);
channel.basicConsume("my-queue2", false, consumer2);
```

Эти два потребителя в сумме смогут иметь только **15 неподтвержденных сообщений**, при этом максимум **по 10 сообщений для каждого потребителя**. 

Это будет работать **медленнее**, чем в предыдущих примерах, из-за дополнительных накладных расходов на координацию между каналом и очередями для соблюдения глобального лимита.

## **Настраиваемый лимит предварительной выборки по умолчанию**
RabbitMQ может использовать **значение предварительной выборки по умолчанию**, которое будет применяться, если потребитель не указывает его явно. 

Это значение можно настроить через параметр `rabbit.default_consumer_prefetch` в файле расширенной конфигурации (`advanced.config`):
```erlang
%% Файл advanced.config
[
 {rabbit, [
       {default_consumer_prefetch, {false,250}}
     ]
 }
].
```
