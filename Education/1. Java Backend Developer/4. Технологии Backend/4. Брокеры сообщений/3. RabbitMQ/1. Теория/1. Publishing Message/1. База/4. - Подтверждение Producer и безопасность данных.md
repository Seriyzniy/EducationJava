Обеспечение безопасности данных — общая ответственность приложений, клиентских библиотек и узлов RabbitMQ.

Сети могут давать сбои, и клиент не может быть уверен, что отправленное сообщение достигло сервера и обработано. Сообщение может потеряться или задержаться.

Для этого разработан **механизм подтверждений на стороне издателя**, аналогичный механизму подтверждений потребителей (ack).

---
# 1. Стратегии использования подтверждений издателя (Publisher Confirms)
**Подтверждения издателя** позволяют отслеживать, какие сообщения успешно приняты RabbitMQ.

Основные ==стратегии==:
1. Публиковать по отдельности и использовать **асинхронные подтверждения** (обработчики событий, futures/promises).
    
2. Публиковать **пакет сообщений** и ждать подтверждения для всех сразу.
    
3. Публиковать по отдельности и ждать подтверждения каждого (**не рекомендуется** из-за сильного снижения пропускной способности).

Для ручного подтверждения, необходимо
```java
// ВАЖНО: Включить перед публикацией!
Channel channel = connection.createChannel();

channel.confirmSelect();  // <-- Включаем режим подтверждений
```

## 1. Асинхронное подтверждение
Большинство клиентских библиотек позволяют **асинхронно обрабатывать подтверждения издателя** по мере их поступления. Это безопасный и эффективный способ публикации.

**Алгоритм работы:**
1. Включить подтверждения издателя на канале.
    
2. Для каждого опубликованного сообщения добавить запись в **map** (словарь), связывая **порядковый номер сообщения** с самим сообщением.
    - Эта карта нужна, чтобы при получении ack/nack понять, какое сообщение подтверждено или нужно повторно отправить.
        
3. При получении **ack** — удалить запись из карты.
    
4. При получении **nack** — удалить запись и повторно отправить сообщение или обработать иначе.

В Java-клиенте RabbitMQ используются интерфейсы `ConfirmCallback` и `ConfirmListener`, слушатели которых добавляются к каналу для обработки подтверждений.

### Пример
```java
// ВАЖНО: Включить перед публикацией!
Channel channel = connection.createChannel();
channel.confirmSelect();  // <-- Включаем режим подтверждений


// 1. Добавляем слушателя подтверждений
channel.addConfirmListener(new ConfirmListener() {
    @Override
    public void handleAck(long deliveryTag, boolean multiple) {
        // УСПЕХ: Сообщение обработано брокером
        System.out.println("Confirmed: " + deliveryTag);
        if (multiple) {
            // Подтверждены все сообщения до этого deliveryTag
            System.out.println(
	            "Multiple messages confirmed up to: " + deliveryTag);
        }
    }
    
    @Override
    public void handleNack(long deliveryTag, boolean multiple) {
        // НЕУДАЧА: Брокер не смог обработать сообщение
        System.out.println("Not confirmed: " + deliveryTag);
        // Повторная отправка или обработка ошибки
        handlePublishFailure(deliveryTag);
    }
});

// 2. Публикуем сообщения
for (int i = 0; i < 100; i++) {
    // Каждое сообщение получает sequence number
    channel.getNextPublishSeqNo(); // Можно сохранить для отслеживания
    channel.basicPublish("exchange", "key", props, body.getBytes());
}
```

## **Пакетная публикация (Batch Publishing)** — отправка пакетов сообщений с ожиданием подтверждений для всего пакета.

**Алгоритм:**
1. Включить подтверждения издателя на канале.
    
2. Отправить пакет сообщений и ждать подтверждений.
    
3. Если все **ack** → перейти к следующему пакету.
    
4. Если есть **nack** или таймаут → повторно отправить пакет или только проблемные сообщения.

Java-клиент: `Channel#waitForConfirms(timeout)` для ожидания подтверждений.

Недостаток: снижает пропускную способность, но чем больше пакет, тем меньше накладные расходы на подтверждения.


### Пример: Синхронное ожидание
```java
// ВАЖНО: Включить перед публикацией!
Channel channel = connection.createChannel();
channel.confirmSelect();  // <-- Включаем режим подтверждений

// 1. Публикация сообщения
channel.basicPublish("exchange", "key", props, body);

// 2. Ожидание подтверждения (блокирующий вызов)
try {
    boolean confirmed = channel.waitForConfirms(5000); // 5 секунд таймаут
    if (confirmed) {
        System.out.println("Message confirmed by broker");
    } else {
        System.out.println("Message not confirmed within timeout");
        // Повтор или обработка ошибки
    }
} catch (InterruptedException e) {
    // Таймаут или разрыв соединения
    System.out.println("Confirmation wait interrupted");
}
```

---
## 3. **Публикация-и-ожидание (Publish-and-Wait)** 
**Публикация-и-ожидание (Publish-and-Wait)** — отправка одного сообщения с немедленным ожиданием подтверждения.
- По сути, это **пакетная публикация с размером пакета = 1**.
    
- Сильно снижает пропускную способность издателя.
    
- Рассматривается как **антипаттерн** и не рекомендуется к использованию.

# 2. Восстановление после ошибки соединения
Сетевое соединение между клиентом и RabbitMQ может оборваться, что влияет на сохранность данных.

Многие клиенты (Java, .NET, Bunny) поддерживают **автоматическое восстановление соединений и топологии**. Другие клиенты предоставляют примеры, как реализовать это вручную.

**Процесс восстановления:**
1. Переподключение к узлу.
2. Восстановление слушателей соединения.
3. Повторное открытие каналов и восстановление их слушателей.
4. Восстановление настроек каналов: `basic.qos`, подтверждения издателя, транзакции.
    
5. Восстановление топологии для каждого канала:
    - обменники
    - очереди
    - привязки (bindings)
    - потребители (consumers)

## Пример
### **Автоматическое восстановление в Java клиенте**
```java
ConnectionFactory factory = new ConnectionFactory();
factory.setHost("rabbitmq-host");

// ВКЛЮЧЕНИЕ автоматического восстановления
factory.setAutomaticRecoveryEnabled(true);

// Дополнительные настройки восстановления
factory.setNetworkRecoveryInterval(5000); // Попытки каждые 5 секунд
factory.setTopologyRecoveryEnabled(true); // Восстановление топологии


Connection connection = factory.newConnection();
```

### **Ручное восстановление (если автоматическое недоступно)**
```java
public class ResilientRabbitMQClient {
    private Connection connection;
    private Channel channel;
    private final ConnectionFactory factory;
    private volatile boolean recovering = false;
    
    public ResilientRabbitMQClient() {
        factory = new ConnectionFactory();
        factory.setHost("localhost");
        setupConnection();
    }
    
    private void setupConnection() {
        int maxRetries = 10;
        int retryDelay = 5000; // 5 секунд
        
        for (int attempt = 1; attempt <= maxRetries; attempt++) {
            try {
                // 1. Переподключение к доступному узлу
                connection = factory.newConnection();
                
                // 2. Восстановление слушателей соединения
                connection.addShutdownListener(cause -> {
                    if (!cause.isInitiatedByApplication()) {
                        scheduleRecovery();
                    }
                });
                
                // 3. Повторное открытие каналов
                channel = connection.createChannel();
                
                // 4. Восстановление слушателей каналов
                channel.addShutdownListener(cause -> {
                    if (!cause.isInitiatedByApplication()) {
                        scheduleRecovery();
                    }
                });
                
                // 5. Восстановление настроек канала
                restoreChannelSettings(channel);
                
                // Восстановление топологии
                recoverTopology(channel);
                
                System.out.println(
	                "Successfully connected to RabbitMQ");
                return;
                
            } catch (Exception e) {
                System.err.printf("Connection attempt %d failed: %s%n", 
                    attempt, e.getMessage());
                
                if (attempt < maxRetries) {
                    try {
                        Thread.sleep(retryDelay);
                    } catch (InterruptedException ie) {
                        Thread.currentThread().interrupt();
                        break;
                    }
                }
            }
        }
        throw new RuntimeException(
	        "Failed to connect after " + maxRetries + " attempts");
    }
    
    private void restoreChannelSettings(Channel channel) throws IOException {
        // Восстановление basic.qos (качество обслуживания)
        channel.basicQos(10); // prefetch count
        
        // Восстановление подтверждений издателя, если использовались
        if (publisherConfirmsEnabled) {
            channel.confirmSelect();
            channel.addConfirmListener(ackListener, nackListener);
        }
        
        // Восстановление транзакций, если использовались
        if (transactionsEnabled) {
            channel.txSelect();
        }
    }
    
    private void recoverTopology(Channel channel) throws IOException {
        // 1. Повторное объявление обменников
        for (String exchange : declaredExchanges) {
            if (!isPredefinedExchange(exchange)) {
                // AMQP.Direct, AMQP.Fanout и т.д.
                channel.exchangeDeclare(exchange, "direct", true);
            }
        }
        
        // 2. Повторное объявление очередей
        for (String queue : declaredQueues) {
            channel.queueDeclare(queue, true, false, false, null);
        }
        
        // 3. Восстановление всех привязок
        for (Binding binding : declaredBindings) {
            channel.queueBind(
                binding.getQueue(),
                binding.getExchange(),
                binding.getRoutingKey()
            );
        }
        
        // 4. Восстановление всех потребителей
        for (ConsumerRegistration consumer : registeredConsumers) {
            channel.basicConsume(
                consumer.getQueue(),
                consumer.isAutoAck(),
                consumer.getConsumerTag(),
                consumer.getCallback()
            );
        }
    }
    
    private void scheduleRecovery() {
        if (!recovering) {
            recovering = true;
            // Используем отдельный поток для восстановления
            new Thread(this::setupConnection).start();
        }
    }
}
```
