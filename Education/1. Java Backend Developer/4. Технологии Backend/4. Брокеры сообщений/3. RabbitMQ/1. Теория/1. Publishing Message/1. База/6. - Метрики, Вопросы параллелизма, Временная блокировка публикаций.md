# 1. Метрики
Для издателей RabbitMQ важно собирать метрики, чтобы понимать их производительность и корректность работы. Наиболее значимые показатели:

- **Outgoing message rate** — скорость отправки сообщений.
    
- **Publisher confirmation rate** — скорость получения подтверждений.
    
- **Connection churn rate** — как часто создаются/закрываются соединения (высокое значение = плохая архитектура).
    
- **Channel churn rate** — как часто создаются/закрываются каналы.
    
- **Unroutable dropped rate** — сколько сообщений было отброшено из-за отсутствия маршрута.
    
- **Unroutable returned rate** — сколько сообщений вернулось издателю (mandatory=true).
    

Churn-метрики важны, потому что помогают выявлять приложения, которые создают слишком много соединений/каналов и тратят ресурсы впустую.

Немаршрутизируемые сообщения указывают на ошибки конфигурации или неправильные routing key.

Помимо RabbitMQ, метрики может собирать и клиентская библиотека (например, Java client), что даёт понимание логики приложения, недоступной самому RabbitMQ.

# 2. Вопросы параллелизма (Concurrency Considerations)
Темы параллелизма в основном касаются **специфики реализации клиентских библиотек**, но можно дать некоторые общие рекомендации.

Параллельная публикация в один и тот же “контекст публикации” (**канале** в `AMQP 0-9-1`, **соединении** в `STOMP`, **сессии** в `AMQP 1.0` и т.д) небезопасна: конкурентные записи могут повредить поток AMQP-кадров, что приводит к закрытию соединения.

Для небольшого числа параллельных издателей ==лучше использовать отдельный поток на каждого==. Для большого количества — применять пул потоков, чтобы каждый поток работал со своим каналом.

## Пример
### Плохо
```java
// ⚠️ ОПАСНО: Не делайте так!
public class UnsafePublisher {
    private final Channel sharedChannel; // Общий для всех потоков
    
    public void publishFromMultipleThreads(Message message) {
        // Поток 1: начинает публикацию
        // Поток 2: ВТОРГАЕТСЯ и тоже публикует
        // Результат: смешанные кадры, разрыв соединения
        sharedChannel.basicPublish(...);
    }
}
```
- **Channel — не потокобезопасный объект.**
    
- Если два потока одновременно вызывают `channel.basicPublish(...)`, их байты могут “смешаться” в TCP-потоке.
    
- RabbitMQ увидит повреждённый протокол → **немедленно разорвёт соединение**.
    
- Поэтому:
    - 1 канал → 1 поток.
        
    - Если потоков много — создать пул потоков, каждый со своим каналом.


### Правильно
```java
// Стратегия 1: Dedicated publisher per thread
public class DedicatedPublisher implements Runnable {
    private final Channel dedicatedChannel; // Свой канал для этого потока
    private final BlockingQueue<Message> queue;
    
    public DedicatedPublisher(Connection connection, 
                              BlockingQueue<Message> queue) throws IOException {
        this.dedicatedChannel = connection.createChannel();
        this.queue = queue;
        
        // Настройка канала (делается один раз)
        dedicatedChannel.confirmSelect();
        dedicatedChannel.basicQos(100);
    }
    
    @Override
    public void run() {
        while (!Thread.currentThread().isInterrupted()) {
            try {
                Message message = queue.take();
                publishSafely(message);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
                break;
            } catch (IOException e) {
                handlePublishError(e);
            }
        }
    }
    
    private void publishSafely(Message message) throws IOException {
        // БЕЗОПАСНО: только этот поток использует dedicatedChannel
        dedicatedChannel.basicPublish(
            message.getExchange(),
            message.getRoutingKey(),
            message.getProperties(),
            message.getBody()
        );
    }
}

// Использование
public class Application {
    public static void main(String[] args) throws IOException {
        Connection connection = factory.newConnection();
        
        // Создаем отдельных издателей для каждой задачи
        List<Thread> publisherThreads = new ArrayList<>();
        
        for (int i = 0; i < 5; i++) { // 5 конкурентных издателей
            BlockingQueue<Message> queue = new LinkedBlockingQueue<>();
            DedicatedPublisher publisher = 
                new DedicatedPublisher(connection, queue);
            
            Thread thread = new Thread(publisher, "publisher-" + i);
            thread.start();
            publisherThreads.add(thread);
            
            // Каждая задача имеет свою очередь
            messageRouter.registerQueue(i, queue);
        }
    }
}
```

> Главная мысль: **нельзя публиковать сообщения из нескольких потоков в один и тот же канал (Channel)**.
> 
> В AMQP 0-9-1 _Channel не является потокобезопасным_.

# 3. Временная блокировка публикации (Temporarily Blocking Publishing)
Можно эффективно **заблокировать всю публикацию в кластере**, установив **верхний предел использования памяти (memory high watermark) в 0**, тем самым вызывая немедленное срабатывание сигнала тревоги ресурса:

```bash
rabbitmqctl set_vm_memory_high_watermark 0
```
