## 📙 1. Структура репозитория и ветвления

### ✅ Основные ветки:

- **`main` / `master`** — стабильная версия, всегда рабочая.  
    ➜ Только проверенные, протестированные коммиты.
    
- **`develop`** (если используется Git Flow) — интеграционная ветка.  
    ➜ Все фичи вливаются сюда перед попаданием в `main`.


- **Feature-ветки** — для разработки отдельных задач:
```bash
feature/add-login-form 
feature/refactor-auth
```
    
- **Bugfix-ветки** — для исправлений ошибок:
```bash
bugfix/fix-null-pointer
```
    
- **Release-ветки** — подготовка к релизу:
```bash
release/v1.2.0
```
    
- **Hotfix-ветки** — срочные фиксы из `main`:
```bash
hotfix/v1.2.1
```


> [!NOTE] ⭐⭐Рекомендация:⭐⭐
> Никогда не коммить прямо в `main` / `develop`.  
> Всегда делай изменения в отдельной ветке и потом merge/pull request.
>

## 📙 2. Именование и коммиты

### ✅ Правила именования веток:
- Используй **маленькие буквы**, слова через дефис `-`.
    
- Начало ветки = тип задачи (`feature`, `fix`, `hotfix`, `release`).
    
- Можно добавлять номер тикета:
```bash
feature/PROJ-123-add-login
```

### ✅ Оформление коммитов:

Пиши **осмысленные, атомарные** коммиты:

```bash
feat: добавлен фильтр пользователей по дате 
fix: исправлен баг с падением при пустом списке 
refactor: оптимизирован парсер XML 
test: добавлены unit-тесты на сервис аутентификации 
docs: обновлена инструкция по запуску
```

> Формат “Conventional Commits” помогает при автогенерации changelog'ов и семантических версий.

## 📙⭐⭐3. Pull / Fetch / Merge / Rebase ⭐⭐

### ✅ ❗❗❗❗Перед началом работы:❗❗❗❗
```bash
git fetch origin 
git pull origin main
```
> Это уменьшает вероятность конфликтов и "diverged branches".

### ✅ Для обновления своей ветки:
```bash
git fetch origin 
git rebase origin/main
```

> **`rebase`** сохраняет чистую историю без merge-коммитов.
> ⚠️ ❗❗Не делай rebase на ветках, которые уже запушены❗❗ и используются другими.

### ✅ ❗❗❗❗Для объединения веток:❗❗❗❗
- Если ветка **еще НЕ на сервере**, делай `rebase`.
- Если ветка **УЖЕ в общем репо**, делай `merge` через Pull Request.

## 📙 4. Force push и история

### ⚠️ ❗❗❗Никогда не делай:❗❗❗

```bash
git push --force origin main
```

### ✅ Вместо этого:
```bash
git push --force-with-lease
```

> Эта команда проверяет, не изменился ли origin с момента твоего последнего `fetch`, и предотвращает случайное затирание чужих коммитов.

### ✅ Ограничь force push:
Настрой **protected branches** (в GitHub / GitLab / Bitbucket):

- Запретить прямые коммиты в `main`
    
- Запретить `force push`
    
- Обязательный code review

## 📙 5. Работа с коммитами

### ✅ Объединяй мелкие коммиты перед пушем:

```bash
git rebase -i HEAD~N
```
(команды `squash` или `fixup`)

### ✅ Если нужно изменить последний коммит:

```bash
git commit --amend
```

### ✅ Проверяй историю перед пушем:
```bash
git log --oneline --graph --decorate
```

## 📙 6. Code review и Pull Requests

### ✅ Перед созданием PR:
1. Убедись, что ветка обновлена:
```bash
git fetch origin 
git rebase origin/main
```
    
2. Убедись, что коммиты чистые, без мусора.
    
3. Тесты проходят локально.
    
4. PR — решает **одну задачу**.

### ✅ В PR:
- Описание: зачем, что сделано, на что влияет.
    
- Лаконичная история коммитов (лучше squash перед merge).
    
- Не забудь ссылку на задачу (например, JIRA или issue ID).

## 📙 7. Работа с удалённым репозиторием

### ✅ Добавление нового origin:
```bash
git remote add origin <url>
```

### ✅ Просмотр и переименование:
```bash
git remote -v git remote rename origin upstream
```

### ✅ Обновление всех ссылок:
```bash
git fetch --all
```

### ✅ Удаление старых ссылок:
```bash
git remote prune origin
```

---

## 8. Очистка и управление локальными ветками

### ✅ Удалить слитые ветки:
```bash
git branch --merged git branch -d feature/old-branch
```

### ✅ Удалить неактуальные удалённые ссылки:
```bash
git fetch -p
```

---
## 9. Работа с тегами и релизами

### ✅ Создание аннотированного тега:
```bash
git tag -a v1.0.0 -m "Релиз 1.0.0"
```

### ✅ Отправка на сервер:
```bash
git push origin v1.0.0
```

### ✅ Удаление:
```bash
git tag -d v1.0.0 git push origin --delete v1.0.0
```
> 🏷️ Теги — это “чекпоинты” истории, их нельзя переписывать.

---

## 📙 10. Безопасность и контроль

✅ Проверяй что пушишь:

```bash
git diff --staged
```

✅ Настрой `.gitignore` корректно (чтобы не пушить секреты, IDE-файлы и кэши).

✅ Если секрет утёк в коммит:  
Используй `git filter-repo` или `BFG Repo-Cleaner` для очистки истории.

✅ Перед деплоем — теги и changelog обязателен.
