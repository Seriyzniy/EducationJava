# 📕 1. Ветки разошлись
## **Что происходит:**  
Ты изменил историю локальной ветки (например, сделал `git rebase` или `git commit --amend`), и теперь локальная ветка **имеет другую историю**, чем ветка на сервере. Git не может просто "перемотать" удалённую ветку, потому что тогда чужие коммиты потеряются.

## Пример
Допустим ты начал работу с веткой в понедельник и вносил изменения в течении недели.
А твой коллега точно также работал, только каждое изменение push-ил в удаленный репозиторий. 
👉В результате у вас РАЗОШЛИСЬ ВЕТКИ

И теперь ты хочешь запушить изменения в ту же ветку. НО что должен делать Git?
- У твоего последнего коммита РОДИТЕЛЬ - B
- В удаленном репозитории у последнего коммита РОДИТЕЛЬ тоже - B

Что делать Git-у? Изменить удаленный репозиторий добавив твой, или наоборот твой коммит проигнорировать и оставить как есть? НЕДОНОЗНАЧНОСТЬ

## 📙 Решения
### Решение - 1. `fetch` + `rebase` = `git pull --rebase origin main`

Сначала скачать коммиты из удаленной ветки. А затем ПЕРЕНЕСТИ СВОИ КОММИТЫ ПОВЕРХ удаленного

🔶 Результат `git fetch`

🔶 Результат `git rebase origin/main main` 
![[Pasted image 20251026113214.png]]

➡После этого можно ПУШИТЬ

> [!NOTE] ⭐ВСЕ ЭТО ДЕЛАЕТ `git pull --rebase origin main`⭐
> 

---
### Решение - 2. `fetch` + `merge` = `git pull`
Подтянуть изменения при помощи fetch, затем merge в свою main

🔶 Результат `git fetch`
![[Pasted image 20251026113051.png]]

🔶 Результат `git merge origin/main`
![[Pasted image 20251026113955.png]]

➡После этого можно ПУШИТЬ

Сервер добавит коммит D, к себе и добавит так же коммит E - который объединил C и D. Поэтому сервер просто переместит указатель ветки

> [!NOTE] ⭐ВСЕ ЭТО ДЕЛАЕТ `git pull`⭐

# 2. Случайный `force push` и потеря чужих коммитов

## **Симптом:**  
После `git push --force` чужие коммиты пропали с сервера.

## **Что происходит:**  
Ты переписал историю ветки, в которой уже кто-то добавил свои коммиты. Это одна из самых опасных ситуаций.

## 📙 Пример
Пусть у нас есть основная ветка `main`, и два разработчика — **ты (DevA)** и **твоя коллега (DevB)**.  

На сервере (GitHub/GitLab) история такая:
```css
A -- B -- C (origin/main)
```

Оба разработчика делают `git pull`, и у них одинаковое состояние локально:
```css
Локально у DevA и DevB:
A -- B -- C (main, origin/main)
```

### 👩‍💻 DevA делает коммит и пушит
Ты (DevA) делаешь новый коммит:
```css
A -- B -- C -- D (main)
```

И пушишь:
```bash
git push origin main
```

Теперь на сервере:
```css
A -- B -- C -- D (origin/main)
```

### 👨‍💻 DevB делает свой коммит (до обновления)

Коллега (DevB) **ещё НЕ СДЕЛАЛ `pull`**, у него осталась старая история (до твоего коммита):
```css
A -- B -- C (main)
```

Он делает свой коммит `E`:
```css
A -- B -- C -- E (main)
```

И теперь пытается пушить:
```bash
git push origin main
```

Git отказывает:
```bash
! [rejected] main -> main (non-fast-forward)
```
Потому что на сервере уже есть `D`, а у DevB — `E`.  

Истории разошлись:
```css
        D (origin/main)
       /
A -- B -- C
       \
        E (DevB main)
```

#### ⚠️ Ошибка — DevB делает `git push --force`
DevB решает “починить” пуш:

```bash
git push --force origin main
```

Что происходит “под капотом”:
- Git **берёт его локальную ветку** `A–B–C–E`
    
- И **переписывает указатель `origin/main`** на `E`
    
- Коммит `D` (твоя работа!) **теряется** в удалённой ветке, потому что на него **больше не ссылается ни одна ветка**
    

Теперь на сервере:
```css
A -- B -- C -- E (origin/main)
```

❌Твоего `D` больше нет на сервере! ❌

💡Git его физически не удалил (он остался в объектной базе), но **С ТОЧКИ ЗРЕНИЯ ИСТОРИИ — ЕГО “ОТРЕЗАЛИ”**.


> [!NOTE] НЕЛЬЗЯ ВОСТАНОВИТЬ ТАКЖЕ КАК И ПРИ РАСХОЖДЕНИИ ВЕТОК
> Так как при расхождении веток, ИСТОРИЯ не изменялась. Теперь общий родитель НЕ ЗНАЕТ О ТОМ, что ты СОЗДАВАЛ КОММИТ D. И поэтьому даже при использовании pull, коммит D будет висеть отдельно
> А при `git force` меняется история РОДИТЕЛЬСКОГО коммита, от которого пошли изменения

## Решение
Если ты _знаешь_, что кто-то переписал историю, но твои коммиты должны остаться,  

можно:
1. Восстановить “потерянные” коммиты (через `reflog`):
```bash
git reflog
git checkout <SHA потерянного коммита>
```

2. Перенести их поверх новой истории:
```bash
git rebase origin/main
```
    
3. Затем снова пушить (уже безопасно):
```bash
git push origin main
```
    

Так ты **воссоздаёшь историю заново**, с новыми SHA, но коммиты сохраняются.
