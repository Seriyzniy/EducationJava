
> [!NOTE] Тег в Git
> Тег — это просто **имя (метка)**, указывающее на **определённый коммит (объект)**.  
> ```bash
> git tag [имя_тега] [имя_коммита] 
> #Создать тег на коммит 
> -a <имя> -m "сообщение"  # аннотированный тег 
> -d <имя>                 # удалить тег 
> -l                       # показать список тегов
> ```
> Можно сказать:  
> - Тег — это как «ветка без движения»: он **ФИКСИРУЕТСЯ на конкретном коммите и больше НЕ ДВИГАЕТСЯ**.
> - Если конкретный коммит не указан, гит пометит тегом `HEAD`.

# 📕 1. Виды тегов
## 📙 1. Лёгкий тег (lightweight tag)

> [!NOTE] Это просто **указатель на коммит**, БЕЗ дополнительной информации.
> 

```bash
git tag v1.0
```
- Не содержит имени автора, даты, комментария.
    
- По сути — «ярлык» на коммит, аналогичный ветке, но без перемещения.
    
- Используется для временных отметок.
    

Посмотреть, на какой коммит указывает тег:
```bash
git show v1.0
```


## 📙 2. Аннотированный тег (annotated tag)

> [!NOTE] **Создаётся с описанием** и сохраняется как **полноценный объект** в базе Git
> 

```bash
git tag -a v1.0 -m "Release version 1.0"
```
- Сохраняет:
    - имя автора,
    - дату,
    - комментарий (сообщение),
    - подпись GPG (если используется `-s`),
    - и ссылку на коммит.
        
- Рекомендуется для **релизов**.

Просмотр информации о теге:
```bash
git show v1.0
```

# 📕 2. Работа с тегами
## 📙 1. Создание тегов на конкретный коммит
Ты можешь поставить тег **не на последний коммит**, а на любой:
```bash
git tag -a v0.9 1a2b3c4
```
(где `1a2b3c4` — это hash нужного коммита)

## 📙 2. Отправка тегов в удалённый репозиторий
❗❗По умолчанию `git push` **не отправляет теги**.

Отправить один тег:
```bash
git push origin v1.0
```

Отправить все теги:
```bash
git push origin --tags
```

## 📙3. Удаление тегов

Удалить локально:
```bash
git tag -d v1.0
```

Удалить на удалённом репозитории:
```bash
git push origin --delete tag v1.0
```

## 📙 4. Просмотр тегов

Посмотреть все теги:
```bash
git tag
```

По шаблону:
```bash
git tag -l "v1.*"
```

## 📙 5. Перейти на тег

Можно перейти в состояние **detached HEAD**:
```bash
git checkout v1.0
```
В этом режиме ты «просто смотришь» старую версию, не находясь ни в одной ветке.

## 📙 6. Подписанные теги (GPG)

Для подтверждения подлинности релиза:
```bash
git tag -s v1.0 -m "Signed release v1.0"
```

Проверка подписи:
```bash
git tag -v v1.0
```

# 📕 3. Команды `bisect` и `describe`
## 📙 1. Команда: `git bisect`

> [!NOTE] `git bisect`
> использует **бинарный поиск** (binary search) по истории коммитов,  чтобы быстро найти **первый плохой (ошибочный)** коммит между известными "хорошим" и "плохим".

### Пример сценария
Допустим, у тебя есть репозиторий, и ты знаешь:
- Коммит **`A`** — всё работает ✅
- Коммит **`F`** — программа ломается ❌

Ты не знаешь, **в каком из промежуточных коммитов (B–E)** появилась ошибка.

```css
A -- B -- C -- D -- E -- F
^                        ^
good                     bad
```

### Команды по шагам

#### 1️⃣ Начать процесс поиска
```bash
git bisect start
```

#### 2️⃣ Указать плохой коммит (где баг уже есть)
```bash
git bisect bad F
```

#### 3️⃣ Указать хороший коммит (где всё работало)
```bash
git bisect good A
```

Теперь Git сам:
- возьмёт **середину диапазона** (например, `C`);
    
- переключится на этот коммит;
    
- попросит тебя **проверить, работает ли код**.
    

---
#### 4️⃣ Проверить и сообщить Git результат

Если на `C` программа работает нормально:

```bash
git bisect good
```

Если баг уже есть:
```bash
git bisect bad
```

После этого Git снова **делит оставшийся диапазон пополам**  
и **переключает HEAD на следующий коммит**.

---

Так Git **повторяет процесс**, пока не найдёт **точный коммит, где впервые появилась ошибка**.  
Обычно для этого нужно всего `log₂(N)` проверок.  
Например, при 1000 коммитах — всего **около 10 шагов**!

#### 5️⃣ Завершение

Когда найден проблемный коммит:
```bash
git bisect reset
```
— возвращает тебя на исходную ветку и состояние HEAD до начала поиска.

---
### 🧠 Автоматизация

Если ты можешь **автоматически проверить**, работает ли код (например, через скрипт тестов),  
Git может **сам пройти все шаги** без твоего участия.

```
git bisect start 
git bisect bad 
git bisect good <commit> 

git bisect run ./test_script.sh
```

Где `./test_script.sh` — возвращает:
- `0`, если всё хорошо (good)
- `1`, если ошибка найдена (bad)

## 📙 2. Команда: `git describe`

> [!NOTE] `git describe`
> Она **описывает текущий коммит (или любой указанный коммит)**  через **ближайший аннотированный тег**, дополняя его информацией о расстоянии до этого коммита.
> 

### Формат вывода
```bash
git describe [коммит]
```

Результат обычно выглядит так:
```bash
v1.2-5-gd7e8f9a
```

где:
- `v1.2` — ближайший **аннотированный тег** (по истории назад);
- `5` — количество коммитов от тега до текущего состояния;
- `gd7e8f9a` — префикс хеша текущего коммита (7 символов),  
    где `g` — просто обозначает “git”.

### Пример

Пусть история репозитория такая:

```css
A -- B -- C -- D -- E -- F (HEAD)
           \
            v1.0
```

Теперь выполним:

```bash
git describe
```

Git найдёт ближайший тег **v1.0** позади HEAD и покажет:

```bash
v1.0-3-g<hash_F>
```
👉 значит: от тега `v1.0` `прошло 3 коммита`, и текущий — `F`.

⭐Обычно теги обозначают релизы⭐
