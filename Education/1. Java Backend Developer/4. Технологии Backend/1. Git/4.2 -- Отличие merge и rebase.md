# 📕 1. Отличия merge от rebase
| Команда      | Что делает                                                      | История                               |
| ------------ | --------------------------------------------------------------- | ------------------------------------- |
| `git merge`  | Создаёт **новый коммит слияния**                                | Ветвистая (развилки сохраняются)      |
| `git rebase` | **Переписывает** коммиты НА НОВОЕ ОСНОВАНИЕ<br>❗Меняет историю❗ | Линейная (чистая, без merge-коммитов) |

## Пример: исходная ситуация
Есть ветка `main` и от неё ответвилась `feature`:
![[Pasted image 20251025211505.png]]

- `A` — общий предок
- 
- `B`, `C` — коммиты в main
- `D`, `E` — коммиты в feature


Теперь нужно объединить изменения.

## 📙 Вариант 1: `git merge`
```bash
git checkout main
git merge feature
```

Git создаст **новый merge-коммит M**:
![[Pasted image 20251025211808.png]]
Что делает merge:
- сохраняет **всю историю** обеих веток;
- создаёт merge-коммит `M`, у которого **два родителя** (C и E);
- история остаётся **разветвлённой**.
    

📘 Преимущества:
- История полностью сохраняет, _как реально развивалась работа_;
- Удобно для командной работы — видно, где ветка была слита.
    

⚠️ Недостатки:
- История становится “шумной” — появляется множество merge-коммитов;
- Иногда сложнее читать `git log`.

## 📙 Вариант 2: `git rebase`
```bash
git checkout feature
git rebase main
```

Git «переписывает» коммиты D и E так, как будто они были созданы **поверх C**:
![[Pasted image 20251025212122.png]]

Что делает rebase:
- Находит общий предок (A),
- Вычисляет изменения, внесённые коммитами D, E,
- Применяет их поверх `main` (C),
- Создаёт **новые коммиты** D' и E' (с новыми hash’ами).
    

📘 Преимущества:
- История становится **линейной и чистой**;
- Удобно для чтения: можно просто пролистывать `git log`;
- Легко применить «feature» поверх свежих изменений `main`.
    

⚠️ ⭐⭐⭐Недостатки:⭐⭐⭐ ⚠️
- История **переписывается** (❗❗❗меняются hash коммитов❗❗❗);
	
- Нельзя использовать на ❗❗❗**общих (push-нутых)** ветках❗❗❗, иначе у других разработчиков ❗❗❗будут конфликты истории❗❗❗;

- Может быть риск потерять контекст merge.

### ⚠️Нельзя использовать rebase на уже запушенных ветках⚠️
Допустим два разработчика начали разрабатывать свой функционал над общим проектом
![[Pasted image 20251026140532.png]]

🟢Зеленый разраб:
- завершил работу над функцией
- Решил сделать REBASE для более чистой истории и ЗАПУШИТЬ НА СЕРВЕР "ЧИТСУЮ" историю
![[Pasted image 20251026141011.png]]

🔴Красный разраб:
- завершил работу над функционалом
- Пытается запушить свой функционал
У него локально
```bash
origin/main: A -- B -- D' -- E'

local main : A -- B
local red: A -- B -- E -- F
```

При попытке push - ОШИБКА 
Потому что:
- У `origin/main` теперь история `A--B--D'--E'`
    
- У RED `main` всё ещё на `B`

- Git **не может перемотать** `origin/main` назад на `B` без потери коммитов `D'` и `E'`
    
📌 То есть Git не знает “куда двигать указатель main”,  
потому что Red предлагает “старую” историю — это **non-fast-forward push**.

> Хотя запушить именно ветку RED он сможет, ОДНАКО при слиянии все в MAIN, будет ошибка

### ✅Решение
Локально у Red выполнить pull (fetch+merge), и только потом push
