## Использование кавычек
|Случай|Кавычки нужны?|Почему|
|---|---|---|
|Простая строка (`prod`)|❌|YAML корректно воспринимает|
|Строка с пробелами|✅|Пробелы нарушают синтаксис YAML|
|Начинается с `$`|✅|Иначе подставится переменная окружения|
|Значение `"false"`, `"on"`, `"yes"`|✅|YAML может интерпретировать как boolean|
|Число с нуля (`0123`)|✅|YAML считает восьмеричным|
|Символы `!`, `@`, `#`, `{`|✅|YAML зарезервировал их для спец. конструкций|

# 📕 2. Ключевые слова внутри `services:`
Ниже — все **основные ключевые слова**, которые можно писать внутри каждого сервиса (контейнера).

## 📙 Базовые 📙
### Основные параметры
```yml
services:
  app:
    image: nginx:latest      # Используем готовый образ
    container_name: my_app   # Имя контейнера (не обязательно)
    build: ./app             # Собрать образ из Dockerfile
    command: ["npm", "start"]# Переопределить CMD
    entrypoint: ["bash"]     # Переопределить ENTRYPOINT
    restart: always          # Политика перезапуска
```

> [!NOTE] Вместо `image` можно указать `dockerfile` при помощи `build .`
> 

📝 **`restart`** варианты:
- `no` — не перезапускать (по умолчанию)
- `always` — всегда перезапускать
- `on-failure` — при неуспешном завершении
- `unless-stopped` — перезапускать, пока не остановишь вручную

### Сеть и порты
```yaml
    ports:
      - "8080:80"            # Проброс порта (host:container)
    expose:
      - "8080"               # Доступен другим контейнерам, но не хосту
    networks:
      - backend              # Подключение к сети (описана ниже)

```

### Томá и файлы (папка сервера : папка контейнера)
```yaml
    volumes:
      - ./app:/usr/src/app   # Монтируем локальную папку
      - data_volume:/data    # Используем именованный том
```

### Переменные окружения
> [!NOTE] 💡 Можно подключить `env_file` - файл `.env` 💡
> 

```yml
    environment:
      SPRING_PROFILES_ACTIVE: prod
      DB_USER: user
      DB_PASSWORD: password
    env_file:
      - .env                 # ⭐⭐Подключить .env файл⭐⭐
```

### Зависимости и порядок запуска
```yaml
    depends_on:
      - db                   # db должен быть запущен раньше
```

📌 _Важно:_ `depends_on` не ждёт готовности контейнера, ==только его запуск==.  
Для “ожидания готовности” используют healthcheck или скрипт ожидания (например, `wait-for-it.sh`).

## 📙 Дополнительные 📙
### 💡 Проверка состояния (healthcheck) 💡
```yml
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8080"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 5s
```
→ Docker проверяет контейнер и помечает его как **healthy/unhealthy**.

### 💡 Секреты и конфиги (для продакшн) 💡
```yaml
    secrets:
      - db_password
    configs:
      - nginx_config

```

📁 Определяются на верхнем уровне:
```yaml
secrets:
  db_password:
    file: ./secrets/db_password.txt

configs:
  nginx_config:
    file: ./configs/nginx.conf
```

### Пользователь, рабочая директория, UID
```yaml
    working_dir: /usr/src/app  # Аналог WORKDIR
    user: "1000:1000"          # UID:GID, от чьего имени работает процесс
```

### Переменные для сборки
```yaml
    build:
      context: ./app           # Папка с Dockerfile
      dockerfile: Dockerfile
      args:
        JAR_FILE: app.jar      # Аргументы для Dockerfile (ARG)
```

### 💡 Лимиты ресурсов 💡
```yaml
    deploy:
      resources:
        limits:
          cpus: "0.50"
          memory: 512M
        reservations:
          cpus: "0.25"
          memory: 128M
```
В основном используется в Docker Swarm, но часть настроек работает и локально.

### Дополнительные параметры
```yaml
    hostname: myapp           # Устанавливает hostname внутри контейнера
    stdin_open: true          # Аналог -i (интерактивный режим)
    tty: true                 # Аналог -t (псевдотерминал)
    privileged: true          # Повышенные права (например, для Docker-in-Docker)
    extra_hosts:
      - "myhost:192.168.1.10" # Аналог --add-host
    depends_on:
      - db
    links:
      - db                    # Устарело, заменено на networks
```

# 📕 3. networks и volumes на верхнем уровне
## 📙 Сети
```yaml
networks:
  backend:
    driver: bridge
  frontend:
    driver: bridge

```

## 📙 Томá
```yaml
volumes:
  db_data:
    driver: local

```

Верхнеуровневый блок `volumes:` используется, чтобы:
> явно ==объявить именованные тома==, которыми могут ==совместно пользоваться несколько контейнеров== или которые **нужно сохранять между запусками**.

### 🔶 Именованный том
```yaml
services:
  db:
    image: postgres
    volumes:
      - 💡db_data:/var/lib/postgresql/data

volumes:
  💡db_data:
```
🔹 Здесь:
- `db_data` — **именованный том**;
- описан **на верхнем уровне** (ниже `volumes:`);
- используется в контейнере `db` для хранения данных БД.

> Если этот том не объявить на верхнем уровне — Docker Compose **всё равно его создаст**, но как **анонимный том** (без описания), что **менее предсказуемо** и **труднее управляется**.

### 🔶 Анонимный том
Создаются автоматически без имени, ЕСЛИ НЕ УКАЗАТЬ `volumes:` на верхнем уровне.  
==Удаляются вместе с контейнером.==
```yaml
services:
  app:
    image: myapp
    volumes:
      - /data
```
→ Docker создаст анонимный том (например, `8f02c81234d5e3e6d6c...:/data`),  
который не имеет имени и не сохраняется при пересоздании контейнера.

### 🔶 **Тома, привязанные к хосту (Bind mounts)** (папка сервера : папка контейнера)

Подключают **папку с хоста** к контейнеру.  
==Создаются без блока== `volumes:` вообще.

```yaml
services:
  app:
    image: myapp
    volumes:
      - ./app:/usr/src/app  # локальная папка
```
→ Здесь используется не именованный том, а **локальный путь** — просто bind mount.  
Docker Compose ничего не создаёт и не управляет этим томом.

### Сравнение
| Что                                   | Где создаётся         | Удаляется при down | Для чего                                |
| ------------------------------------- | --------------------- | ------------------ | --------------------------------------- |
| **Bind mount** (`./dir:/data`)        | На хосте              | Нет                | Для разработки (реальные файлы)         |
| **Именованный том** (`db_data:/data`) | В Docker Volume Store | Только с `-v`      | Для хранения данных между перезапусками |
| **Анонимный том** (`/data`)           | Автоматически Docker  | Да                 | Временные данные, кеши                  |

# 💡4.  Дополнительные ключи
|Ключ|Назначение|
|---|---|
|`profiles`|Позволяет активировать сервисы по профилю (`docker compose --profile dev up`)|
|`depends_on`|Указывает порядок запуска контейнеров|
|`logging`|Настройки логирования (драйвер, лимиты, опции)|
|`security_opt`|Опции безопасности (AppArmor, SELinux)|
|`sysctls`|Настройки ядра (например, `net.core.somaxconn`)|
|`cap_add`, `cap_drop`|Добавление/удаление Linux capabilities|
|`tmpfs`|Монтирование временной файловой системы|
|`device`|Проброс устройств с хоста (`/dev/ttyUSB0`)|
