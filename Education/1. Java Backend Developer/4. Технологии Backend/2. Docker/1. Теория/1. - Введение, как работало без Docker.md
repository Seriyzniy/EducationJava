Любое приложение, БД, kafka, redis- все лишь программа, которая запускается на каком-то компьютере.

К примеру, любая программа на Java или Python, которую ты напишешь, запускается на твоей компе.

Однако, если необходимо разработать более МАСШТАБНОЕ приложение, то одного компьютера не хватит для нормальной работы. Для этого необходимо развернуть (запустить) программу на севрере

# 📕 1. Как работало раньше (до Docker)

## 📙1. **Физические серверы**
Раньше каждое приложение ставили **на отдельный сервер** (мощный комп):
- Устанавливали ОДНУ ОС, библиотеки, зависимости, сервисы (приложения).
- Настраивали вручную окружение (например, `/etc`, переменные, users, конфиги).

### ПРОБЛЕМА, все приложения ПОТРЕБЛЯЛИ одни и те же ресурсы. 
Могла случиться ситуация, когда (допустим) Java приложение, сильно нагружалось из-за объемного количества запросов пользователей. 

И поэтому оно начинает потреблять БОЛЬШЕ РЕСУРСОВ, даже тех, на которые рассчитывали другие сервисы (БД, Redis и прочее). А значит они не смогут выполнять собственные задачи 
![[Pasted image 20251020094022.png]]

Минусы:
- Огромный расход ресурсов — на один сервер одно приложение.
- Масштабировать было дорого и долго.
- Любая поломка требовала пересборки вручную.
- ==КОНФЛИКТЫ зависимостей==

---
## 📙2. **Виртуальные машины (VM)**
Затем появились **виртуальные машины (VMware, VirtualBox, Hyper-V)** — внутри одного сервера можно было запустить несколько виртуальных “мини-компьютеров”, каждый со своей ОС.

Плюсы:
- ==Изоляция==: приложения не мешают друг другу.
- Можно разделить и ограничить ресурсы сервера для каждом виртуалки

![[Pasted image 20251020094908.png]]

## Проблема, каждую виртуальную машину нужно также НАСТРАИВАТЬ 
![[Pasted image 20251020095304.png]]

При этом, если мощностей сервера не хватает, необходимо ставить еще один такой, ПРИ ЭТОМ ПОВТОРИВ НАСТРОЙКУ

Минусы:
- **==Тяжеловесность==** — каждая VM содержит _полную ОС_ (гигабайты данных).
- **Медленный старт** (минуты).
- Плохое использование ресурсов: 10 VM = 10 Linux с одними и теми же файлами ядра.
- Обновления и настройка тоже сложны.

## 3. Так появился Docker
Современное устройство докер

На сервере установлен Гипервизор, для управления виртуальными машинами.
