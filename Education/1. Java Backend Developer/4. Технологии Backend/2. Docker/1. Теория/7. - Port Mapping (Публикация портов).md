
> [!NOTE] **Port mapping** (или **проброс портов**)
> Это механизм, при котором `порт внутри контейнера` ==связывается== с `портом на хост-машине` (твоём компьютере).
> 


Контейнер по своей природе изолирован — всё, что в нём, **не доступно снаружи** по умолчанию.  
Чтобы, например, открыть веб-сервер внутри контейнера (Nginx, Spring Boot, Node.js),  
Docker должен "пробросить" порт наружу.

## Пример
Допустим хотим запустить Tomcat внутри контейнера, и чтобы можно было к нему подключаться
```bash
docker run -p 7070:8080 tomcat
```

Docker делает следующее:

| Уровень        | Действие                                                                                      |
| -------------- | --------------------------------------------------------------------------------------------- |
| 🧱 Контейнер   | Внутри работает Tomcat, слушающий порт `8080`                                                 |
| 🖥️ Хост       | Docker создаёт прокси через сетевой мост (bridge) и открывает порт `7070`                     |
| 🌍 Внешний мир | Теперь можно обращаться к `http://localhost:7070`, и запрос попадёт на порт `8080` контейнера |
### 🟢 1. Простой проброс

```
docker run -d -p 8080:8080 tomcat
```
⮕ На хосте порт `8080` связан с портом `8080` внутри контейнера.

### 🟢 2. Несколько портов
```bash
docker run -d -p 8080:80 -p 4430:443 nginx
```
⮕ Пробрасываются и HTTP (80), и HTTPS (443).

### 🟢 3. Указать IP

```bash
docker run -d -p 127.0.0.1:8080:80 nginx
```
⮕ Проброс доступен **только с localhost**, а не из сети.

### 🟢 4. Автоматический проброс всех портов

```bash
docker run -d -P nginx
```
⮕ Docker автоматически выберет случайные свободные порты на хосте.

### 🟢 5. Проверка портов контейнера

```bash
docker port <container_name>
```

Пример:
```bash
docker port web 80/tcp -> 0.0.0.0:8080
```

# 📕 1. Как Docker реализует проброс ?

Когда ты используешь `-p`:
- Docker добавляет **правила NAT (iptables)** на хостовой системе;
    
- создаёт **виртуальную сеть bridge** (по умолчанию `docker0`);
    
- маршрутизирует пакеты от `localhost:порт_хоста` → `контейнер:порт`.
    

> [!NOTE] 🧠 По сути, Docker работает как **обратный прокси** между сетью хоста и контейнером. 🧠
> 

---
## 🧠 Важные детали

| Ситуация                                         | Поведение                                                                                                         |
| ------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------- |
| Не указал `-p`                                   | Контейнер недоступен снаружи (только внутри Docker-сети)                                                          |
| Указал только порт контейнера (например `-p 80`) | Docker выберет случайный порт на хосте                                                                            |
| 💡Несколько контейнеров с одним портом внутри 💡 | Можно пробросить разные порты на хосте (`8080:80`, `8081:80`)                                                     |
| ==Compose автоматически создаёт сеть==           | И контейнеры могут обращаться друг к другу **по имени сервиса**, без `-p`                                         |
| `EXPOSE` в Dockerfile                            | Это просто _декларация_, а не реальный проброс — она говорит Docker, какие порты контейнер "ожидает" использовать |

### Пример в Docker **Compose**
```yaml
version: '3'
services:
  web:
    image: nginx
    ports:
      - "8080:80"
      - "4430:443"
```

После запуска:
```bash
docker compose up -d
```

⮕ ты можешь открыть `http://localhost:8080`, и увидишь страницу Nginx из контейнера.

# 📕 2. Взаимодействие между контейнерами без Port Mapping

Если ==контейнеры в одной **Docker-сети**==, то они ==могут общаться== 
**по внутренним портам напрямую**:
```bash
docker network create mynet

docker run -d --network mynet --name backend myapp
docker run -d --network mynet --name frontend nginx
```

Тогда `frontend` может обращаться к `http://backend:8080`  
— без `-p`, т.к. это внутренняя связь в Docker network.
