Проблемы связанные с использованием виртуальных машин, решает Docker

> [!NOTE] ⭐⭐ Программа Docker ⭐⭐
> **Docker** — это **платформа и набор инструментов**, которые позволяют:
> 1. **Создавать (build)** образы приложений, 
> 2. **Хранить (share)** их (например, в Docker Hub или частном registry), 
> 3. **Запускать (run)** контейнеры из этих образов.

# 📕 1. Docker Image - Шаблон приложения
## Зачем?
Так как каждый раз настраивать виртуальную машину не удобно и очень долго(особенно при масштабировании) 
то было бы удобно, ==собрать все необходимое один раз== с нужными версиями и зависимостями, ==и просто запускать на сервере==, без дополнительных настроек

Для этого используется Docker Image - шаблон приложения, из которого потом создаются Docker Container. 

## Docker Image
Чтобы развернуть приложение на сервере необходимо собрать Docker Image

> [!NOTE] ⭐`Dokcer Image` - ==ШАБЛОН приложения==⭐ 
> `Docker Image` - Шаблон, из которого СОЗДАЮТСЯ КОНТЕЙНЕРЫ. 
> 
> 💡Его можно и нужно переиспользовать


## Слои в Docker Image

> [!NOTE] **Docker Image** состоит из **нескольких слоёв (layers)** 
> Каждый `слой` `представляет` собой `изменения в файловой системе`, внесённые на определённом этапе сборки (обычно — одной инструкцией из `Dockerfile`)
> 
> Образ `состоит из СЛОЕВ`
> - Все слои ==read-only==
> - Образы можно перемещать и удалять
> - Образы хранятся в репозиториях
> - Есть официальные образы и образы сообщества
> 

![[Pasted image 20251020134948.png]]

Каждый из этих слоев, Docker использует для создания контейнера.
При помощи слоев можно собирать другие образы, основываясь на слоях других образов

### Пример Docker File
```dockerfile
FROM ubuntu:22.04
RUN apt-get update && apt-get install -y python3
COPY app.py /app/
CMD ["python3", "/app/app.py"]
```

Что произойдёт:

| Этап | Инструкция                       | Что создаётся                                         | Слой            |
| ---- | -------------------------------- | ----------------------------------------------------- | --------------- |
| 1    | `FROM ubuntu:22.04`              | Базовый слой — файловая система Ubuntu                | 🧩 Layer 1      |
| 2    | `RUN apt-get ...`                | Добавляется установленный Python и изменённая система | 🧩 Layer 2      |
| 3    | `COPY app.py /app/`              | Добавляется твой файл `app.py`                        | 🧩 Layer 3      |
| 4    | `CMD ["python3", "/app/app.py"]` | Только метаданные (не слой файловой системы)          | ⚙️ Конфигурация |
### Польза от многослойности

✅ Быстрая пересборка:  
Docker использует кэш — если `RUN`, `COPY`, `ADD` не изменились, он не пересоздаёт слои.

✅ Экономия места:  
Образы, основанные на одной базе (например, `ubuntu`), **делят общие слои**.


## Пример состава Docker Image
Образ содержит:
1. **Base layer** — базовый слой, вроде `ubuntu`, `alpine`, `debian`, `busybox`

> [!NOTE] ⭐Базовый слой (или **базовый образ**) — это `снимок файловой системы` (user space) определённой Linux-среды.  ⭐
> 
> Он 
> - **НЕ содержит ядро**, 
> - НЕ управляет процессами, 
> - НЕ загружается как полноценная ОС
> 
> Это просто набор файлов и каталогов, **которые обычно есть в Linux-системе после установки**.
> 
> Пример
> ```bash
> /bin/         → bash, ls, cat …
> /lib/         → стандартные библиотеки (glibc)
> /usr/         → приложения и библиотеки
> /etc/         → конфигурационные файлы
> /tmp/, /var/  → временные и системные каталоги
> ```
> Эти файлы позволяют твоему приложению работать так, ==**как будто оно запущено в Ubuntu**==, хотя на самом деле ядро и ресурсы — от хоста.

2. **Приложение**  
	(например, бинарник `myapp` или `python`).
	
3. **Библиотеки и зависимости**  
    (`glibc`, `libssl`, `libz`, и т. д.).
    
4. **Минимальный набор утилит и каталогов**, чтобы всё работало  
    (например, `/bin/sh`, `/etc/passwd`, `/usr/lib/...`).

# 📕 2. Архитектура Docker и Docker Container

> [!NOTE] Docker `Container` - `ЗАПУЩЕННЫЙ Image`
> **Docker container** — это уже `собранное и запущенное приложение` ==в изолированном окружении.== 
> 
> Оно формируется послойно, в соответствии с образом. 
> Каждый новый слой расширяет функциональность предыдущего, формируя стек используемых инструментов, платформ и настроек системных служб.

2) Любая ОС, на которой можно установить Docker (ТОЛЬКО Linux, а если на другую ОС, то ПОВЕРХ Linux)
3) Программа Docker

## 📙 1. Принцип работы контейнеров

При создании контейнера, для него ==создадутся ФАЙЛЫ==, ==доступные ТОЛЬКО ЭТОМУ КОНТЕЙНЕРУ== - файлы из Image (зависимости, исходный код приложения и т.д)

Далее, у этого контейнера ==запустится процесс в котором будет работать приложение== (настройки запуска также указываются в в Image)

### Процесс контейнера ИСПОЛЬЗУЕТ ядро ОС, на которой запущен Docker
Когда ты запускаешь контейнер из образа, происходит следующее:
1. Docker Engine берёт твой image (user space).
2. Создаёт **==изолированное пространство процессов== (namespace)**.
3. Применяет ограничения **cgroups**.
4. Подключает файловую систему image’а как root (`/`).
    
5. Всё, что внутри контейнера — выполняется **на том же ядре Linux**, где установлен Docker.

> Контейнер просто «видит» это ядро как будто своё.

> [!NOTE] ⭐ Каждый контейнер работает независимо друг от друга ⭐
> 

![[Pasted image 20251020130825.png]]

> [!NOTE] Все запущенный `контейнеры` `используют одни и те же ресурсы` 
> Поэтому необходимо планировать допустимые ресурсы для каждого контейнера

### Основные технологии контейнеризации Linux
`Контейнеры` — НЕ ОТДЕЛЬНАЯ ВИРТУАЛИЗАЦИЯ, а просто ==изолированные процессы==, которым Linux ограничивает:
- видимость системы,
- доступ к ресурсам,
- и окружение.

Для этого Linux использует несколько ключевых механизмов ядра:

|Механизм ядра|Назначение|
|---|---|
|**Namespaces**|Изолируют системные ресурсы (PID, сеть, монтирование, пользователи, хостнеймы и т.д.) — контейнер “думает”, что он отдельная система.|
|**Control Groups (cgroups)**|Ограничивают ресурсы (CPU, RAM, I/O и т.п.) и управляют приоритетами процессов.|
|**UnionFS / OverlayFS**|Позволяют собирать образы из слоёв, эффективно переиспользуя данные.|
|**Capabilities / seccomp / AppArmor**|Гранулярная безопасность и изоляция процессов.|

### Копии файлов одних их тех же образов - НЕ СОЗДАЮТСЯ
При создании НЕСКОЛЬКИХ КОНТЕЙНЕРОВ на одном хосте, из одного и того же ОБРАЗА - то у таких контейнеров БУДУТ ОБЩИЕ ФАЙЛЫ, а не копии для каждого из контейнеров

### Остановка контейнеров
После остановки контейнеров, все файлы и процессы удаляются

При этом, Docker также сам останавливает контейнеры, которые не имеют активных процессов

# 📕 2. Docker на Linux и Windows

> [!NOTE] Docker и Linux
> Docker `построен` `поверх механизмов` `ядра Linux`, которые `ОБЕСПЕЧИВАЮТ КОНТЕЙНЕРИЗАЦИЮ контейнеризацию`.  
> 
> ❗❗❗Эти механизмы **существуют только в Linux**, поэтому Docker “нативно” работает именно там.
## Docker на Windows
![[Pasted image 20251020133444.png]]

Для запуска Docker на Windows, необходимо установить виртуальную ОС Linux

Docker использует **виртуальную машину Linux** — через **WSL 2 (Windows Subsystem for Linux)**

То есть:
- Docker Desktop ==запускает небольшое **ядро Linux внутри WSL2**==,
    
- все контейнеры живут внутри этой VM,
    
- а Windows CLI просто прокидывает команды туда.

# 📕3. Docker или Виртуальная машина
![[Pasted image 20251020134204.png]]

## 🐳 Docker 

Docker работает **на уровне операционной системы**.

- Внизу — **железо (hardware)**.
    
- Поверх него — **операционная система** (например, Linux).
    
- На ней установлен **Docker Engine**.
    
- А уже внутри Docker запускаются **контейнеры** — каждый содержит:
    - само приложение,
    - его библиотеки и зависимости,
    - лёгкую среду Linux ==(без отдельного ядра!)==.

Главное:  
👉 ==**все КОНТЕЙНЕРЫ РАЗДЕЛЯЮТ одно и то же ядро ОС.**==

Они изолированы с помощью функций ядра (namespaces, cgroups),  
но не дублируют саму систему.

### 🔹 Преимущества Docker:
- **Быстрый старт** (секунды, без загрузки ОС).
    
- **Лёгкие контейнеры** (занимают мегабайты, а не гигабайты).
    
- **Единое окружение**: можно перенести образ и ==запустить его где угодно==.
    
- **Гибкость** — десятки контейнеров могут работать на одном хосте.

### 🔹 Недостатки:
- Контейнеры **зависят от ядра хост-системы**.  
    Например, Linux-контейнеры нельзя напрямую запустить на Windows без слоя совместимости (WSL2).
    
- Изоляция чуть слабее, чем у виртуалок (всё же одно ядро).

## 🖥 Виртуальные машины 

Виртуальные машины работают **на уровне оборудования (железа)**.

- На физическом сервере стоит **гипервизор (Hypervisor)** — программа, которая эмулирует “виртуальное железо”.
    
- Каждая виртуальная машина содержит:
    - **полную копию операционной системы (гостевую ОС)**,
    - библиотеки,
    - и приложение.

Таким образом, ==каждая VM — это **полноценный “мини-компьютер”**==.

### 🔹 Преимущества VM:
- **Полная изоляция** — у каждой VM своё ядро и ОС.
- Можно запускать **разные ОС** (например, Windows и Linux на одном сервере).
    
- 🔒 Безопасность на уровне ядра (никакого общего окружения).

### 🔹 Недостатки:
- **Медленный запуск** (минуты, как обычный компьютер).
    
- **Большие размеры** (гигабайты на каждую ОС).
    
- **Дублирование системных ресурсов** — каждая VM держит своё ядро, службы, драйверы и т.п.

---
## Ключевая разница

| Критерий                | Docker (контейнеры)          | Виртуальные машины              |
| ----------------------- | ---------------------------- | ------------------------------- |
| Уровень изоляции        | На уровне ОС (ядра)          | На уровне железа                |
| 🔶 Ядро ОС              | Общее для всех контейнеров   | Свое ядро в каждой VM           |
| 🔶Размер                | Легковесные (МБ)             | Тяжелые (ГБ)                    |
| 🔶Время запуска         | Секунды                      | Минуты                          |
| 🔶Производительность    | Почти как у нативной системы | Есть накладные расходы          |
| Изоляция и безопасность | Ниже, чем у VM               | Полная изоляция                 |
| Гибкость окружений      | Только под одно ядро (Linux) | Любые ОС (Linux, Windows, etc.) |
