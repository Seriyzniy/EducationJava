# 1. Что такое Keycloak и зачем он нужен
`Keycloak` — это ***opensource сервис написанный на Java***, который **берет на себя всю работу с авторизацией и аутентификацией**. Вместо того чтобы выстраивать свою систему логинов, паролей, подтверждений почты, ролей и токенов, вы просто подключаете Keycloak — и получаете всё необходимое.

> Исходный код: ([keycloak/keycloak: Open Source Identity and Access Management For Modern Applications and Services](https://github.com/keycloak/keycloak/tree/main))

Keycloak функционирует как самостоятельный сервис: 
- у него есть панель управления, 
- REST API, 
- интеграция с внешними системами 
- и клиенты для популярных языков и фреймворков. 
По сути, Keycloak становится центральным хабом авторизации в проекте: `пользователи авторизуются через него, получают токены, а далее попадают в ваши приложения.`

## Сценарии где полезен Keycloak
Ключевые сценарии, где Keycloak раскрывает себя лучше всего:
- **Single Sign-On** (SSO) — один вход для всех ваших сервисов.
	
- **OAuth2 и OpenID Connect и SAML** — готовая реализация стандартов.
	
- *Роли и группы* — определяют, какие действия доступны пользователю.
	
- **Соцлогины** — вход через Google, GitHub, VK и т.д.
	
- *Управление пользователями* — создание, бан, сброс пароля, подтверждение email.
	
- Интеграция с любыми фронтами и бэками — Java, Python, Node.js, React, Angular.
	
- User Federation - синхронизация пользователей из LDAP и Active Directory серверов и других идентификационных провайдеров.
	
- **Kerberos bridge** - использование Kerberos сервера для автоматической аутентификации пользователей.
	
- **Admin Console** - для единого управления настройками и параметрами решения через Web.
	
- *CORS Support* – клиентские адаптеры имеют встроенную поддержку CORS.
    
- Service Provider Interfaces (SPI) – большое количество SPI, позволяющих настраивать различные аспекты работы сервера: потоки аутентификации, идентификационных провайдеров, сопоставление протоколов и многое другое.
    
- Клиентские адаптеры для JavaScript applications, WildFly, JBoss EAP, Fuse, Tomcat, Jetty, Spring.
    
- Поддержка работы с различными приложениями, поддерживающими OpenID Connect Relying Party library или SAML 2.0 Service Provider Library.

Keycloak помогает не только быстро запустить вход через логин и пароль, но и масштабировать доступ: от одного лендинга до десятков микросервисов с разными правами.

# 2. Установка Keycloak контейнера,  настройка и базовые понятия
Развертывать KeyCloak будем в Docker.

> [!IMPORTANT] Для KeyCloak необходима БД
> В этой базе данных будут храниться:
> - логины и пароли пользователей
> - `realms` (условно профиль настройки)
> - OAuth клиенты
> - роли и разрешение
> - и т.д
> 
> Эту **БД можно поднять самим** (Postgres или другую), **либо** KeyCloak будет **использовать встроенную** 

## 1. Поднятие контейнеров
Мы же поднимем собственную БД, для наглядности (можно посмотреть что там хранится)

> В ДАННОЙ ПРИМЕРЕ ИСПОЛЬЗУЕТСЯ **НЕ ОРИГИНАЛЬНЫЙ ОБРАЗ KeyCloak**

> [!WARNING] Liquebase
> Под капотом KeyCloak использует Liquebase для контроля миграций

Опишем docker-compose
```yaml
version: '3.8'  
  
services:  
  postgres:  
    image: postgres:15  
    container_name: keycloak_postgres  
    environment:  
      POSTGRES_DB: keycloak  
      POSTGRES_USER: keycloak  
      POSTGRES_PASSWORD: keycloak  
    ports:  
      - "5433:5432"  
    volumes:  
      - postgres_data:/var/lib/postgresql/data  
    networks:  
      - keycloak_net  
  
  keycloak:  
    image: quay.io/keycloak/keycloak:26.2  
    container_name: keycloak  
    command: start-dev  
    environment:  
      KC_DB: postgres  
      KC_DB_URL_HOST: postgres  
      KC_DB_URL_DATABASE: keycloak  
      KC_DB_USERNAME: keycloak  
      KC_DB_PASSWORD: keycloak  
      KC_HOSTNAME: localhost  
        
      KEYCLOAK_ADMIN: admin  
      KEYCLOAK_ADMIN_PASSWORD: admin  
    ports:  
      - "9090:8080"  
    depends_on:  
      - postgres  
    networks:  
      - keycloak_net  
  
volumes:  
  postgres_data:  
  
networks:  
  keycloak_net:
```

# 3. Принцип работы сервисов с KeyCloak

## 1. Общий принцип работы
![[Pasted image 20260120125248.png]]
### Работа системы(приложения) с KeyCloak
![[Pasted image 20260120124956.png]]

1. Пользователь открывает в браузере приложение
2. Для авторизации пользователя, мы должны получить код авторизации, поэтому приложение отправляет запрос на `AuthZ Endpoint`
3. В приложение возвращается код авторизации
4. Этот код авторизации приложение обмениваем у KeyCloak (через Token Endpoint) на токен
5. ОПЦИОНАЛЬНО: Приложение по токену может получить инфу о пользователе

### Работа системы(приложения) с микросервисами 
![[Pasted image 20260120125100.png]]

6. Получив токен доступа (Access Token), приложение может воспользоваться каким-то backend сервисом
7. Сервис, будет проверять токен через JWKS (JSON WebToken Key Signature) - ПРОВЕРЯЕТ ПОДПИСЬ.
	И только после проверки подписи дает доступ 

## 2. Работа с токенами
### При помощи Opaque
![[Pasted image 20260120113657.png]]

Сначала пользователь регистрируется в системе. Система отправляет запрос на регистрацию пользователя в KeyCloak, который сохраняет инфу о пользователе в своей БД

Далее
1. Приложение (сервис) запрашивает токен у KeyCloak (пользователь вошел в систему)
2. KeyCLoak возвращает токен пользователя
3. С этим токеном пользователь (через приложение), пытается получить доступ к Ресурсному серверу 1
4. Ресурсный сервер 1 **отправляет запрос в KeyCloak** на проверку токена. После чего KeyCloak дает ответ "Токен валиден" или "Токен не валиден"

### При помощи JWT
![[Pasted image 20260120114559.png]]

Приложение (сервис) запрашивает токен у KeyCloak (пользователь вошел в систему)
2. KeyCLoak возвращает токен пользователя
3. С этим токеном пользователь (через приложение), пытается получить доступ к Ресурсному серверу 1
4. Ресурсный сервер 1 **ЛОКАЛЬНО ПРОВЕРЯЕТ ВАЛИДНОСТЬ ТОКЕНА** (проверяет срок истечения и проверяет подпись при помощи публичного ключа)
