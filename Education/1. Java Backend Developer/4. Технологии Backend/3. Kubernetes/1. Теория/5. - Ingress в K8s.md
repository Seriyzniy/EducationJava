## Отличие от Service
| Характеристика            | **Service**                                                                             | **Ingress**                                                                                    |
| ------------------------- | --------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------- |
| **Назначение**            | Обеспечивает ==доступ к подам ВНУТРИ кластера или ИЗВНЕ== (через NodePort/LoadBalancer) | Управляет ==доступом ИЗВНЕ к сервисам по HTTP/HTTPS==, обеспечивает маршрутизацию по URL/хосту |
| **Тип<br>трафика**        | TCP/UDP (в основном `Layer 4`)                                                          | HTTP/HTTPS (`Layer 7`, может работать с TLS)                                                   |
| **Стабильный <br>IP**     | ClusterIP — да, <br>NodePort/LoadBalancer — внешние IP                                  | IP нет, Ingress использует Controller + внешний LB                                             |
| **Балансировка нагрузки** | kube-proxy распределяет <br>трафик на поды                                              | Контроллер Ingress (nginx, Traefik, HAProxy и др.) распределяет трафик                         |
| **Доступ<br>извне**       | NodePort / <br>LoadBalancer                                                             | Через Ingress Controller + DNS                                                                 |
| **Маршрутизация**         | На уровне IP/порт (Layer 4)                                                             | На уровне URL, host, path, header (Layer 7)                                                    |
| **TLS/HTTPS**             | Только через внешний LB или NodePort                                                    | Поддержка TLS напрямую, можно использовать один IP для нескольких сервисов                     |
| **Использование**         | Подключение <br>приложений внутри кластера и базовая публикация наружу                  | Публикация веб-приложений и маршрутизация запросов по доменам/пути                             |


# 📕 1. Ingress - объект K8s
**Ingress в Kubernetes** — это объект, который ==описывает **правила маршрутизации ВНЕШНЕГО HTTP/HTTPS-трафика== к сервисам внутри кластера**.

> [!NOTE] **Объект Ingress ТРЕБУЕТ** наличия **КОНТРОЛЛЕРА Ingress** (Controller Ingress)
> 

## Зачем нужен Ingress, если есть Service (NodePort/LoadBalancer)?
Для каждого сервиса типа LoadBalancer, необходим ОТДЕЛЬНЫЙ собственный LoadBlancer, который ПРЕДОСТАВЛЯЕТСЯ провайдером

К тому же, Ingress работают на 7-ом уровне OSI, что дает нам использование
- сессий
- куки

## Идея Ingress
- **Ingress** не сам обрабатывает трафик.
    
- Это **набор правил**, которые говорят **Ingress Controller**, как направлять запросы.
    
- Трафик идёт извне → Ingress Controller → нужный Service → нужный Pod.

> Без Ingress пришлось бы создавать **LoadBalancer** или **NodePort** для каждого сервиса, что неудобно и дорого.

## Что умеет Ingress
1. **Маршрутизация по URL**
```css
example.com/app1 → service1 example.com/app2 → service2
```
    
2. **Маршрутизация по хосту**
```css
app1.example.com → service1 app2.example.com → service2
```
    
3. **TLS/HTTPS**
    - Можно подключить сертификаты к одному IP и обслуживать несколько доменов.
        
4. **Базовая балансировка нагрузки** на уровне HTTP (Layer 7).

## Как работает Ingress
```rust
[Интернет] 
     |
     v
[Ingress Controller]   <- читает правила Ingress объекта
     |
     +----> [Service A] ---> Pod A
     |
     +----> [Service B] ---> Pod B
```

- **Ingress объект** хранит правила: какой URL → какой сервис.
    
- **Ingress Controller** реализует эти правила, обычно ==через NGINX, Traefik или другой LB==.
