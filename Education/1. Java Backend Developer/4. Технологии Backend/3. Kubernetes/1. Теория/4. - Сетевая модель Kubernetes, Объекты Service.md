# 📕 1.Базовые сетевые концепции Kubernetes
Kubernetes изначально следует модели **плоской сети**, где все поды и узлы свободно обмениваются трафиком по IP без NAT. 

## 📙 Плагин CNI
==За настройку сети отвечает== **CNI-плагин** (Container Network Interface), который 
- ==выделяет IP подам==, 
- создаёт интерфейсы и маршруты. 

Kubernetes не реализует сеть самостоятельно — это делают плагины вроде **Calico, Flannel, Cilium**, используя разные подходы (оверлей, BGP, eBPF и др.).

## 📙 Уникальный Ip для каждого Пода
По умолчанию **Pod сам по себе не имеет IP**, пока:
- kubelet не запустит контейнер,
- и CNI-плагин не выполнит процедуру `ADD`.
- ==ПРИ ПЕРЕСОЗДАНИИ **ПОДА** IP МЕНЯЕТСЯ==

После этого:
- внутри Pod появляется интерфейс (обычно `eth0`);
- ему назначается **уникальный IP** из пула CNI;
- этот IP становится **доступен из всего кластера** (CNI обеспечивает маршрутизацию);
- этот же IP виден в `kubectl get pods -o wide`.

> 💡Например, с Flannel IP-поды обычно вида `10.244.x.y`,  
> а с Calico — `192.168.x.y` или что-то похожее, в зависимости от настроек.

## 📙 Ip у Нода
Node (рабочая машина, где работает kubelet) тоже имеет ❗❗❗❗IP — но **это обычный IP хоста** в физической/облачной сети (eth0 узла).❗❗❗❗

Он ==не связан с Kubernetes напрямую==, просто используется для:
- взаимодействия kubelet с API-сервером,
- общения между нодами,
- маршрутизации межнодового трафика подов.

==Компонент **kube-proxy**==, работающий на каждом узле, ==обеспечивает доступ к сервисам (Service)== и балансировку трафика. 
Он создаёт правила NAT для IP сервисов и направляет запросы на целевые поды. 
Поддерживаются два режима работы: 
- **iptables** (прост, но плохо масштабируется) 
- **IPVS** (эффективен на больших кластерах, поддерживает больше алгоритмов балансировки).


> [!NOTE] IP есть и БЕЗ всяких SERVICE
> ⭐⭐⭐Каждый **Pod в Kubernetes получает свой уникальный IP-адрес**, 
> даже если **никакого Service вообще нет**.⭐⭐⭐

## 📙 Взаимодействие внутри узла (intra-node)
==Каждый узел== Kubernetes — это Linux-машина с корневым сетевым пространством имён (_root netns_) и интерфейсом `eth0`.  
![[Pasted image 20251030121252.png]]


У ==каждого пода== — своё _netns_ с виртуальным интерфейсом `eth0`, который связан с хостом через пару виртуальных интерфейсов `veth`:
- 🔹конец в поде — `eth0`, 
> 	(потому что под не знает о нижележащем хосте и думает, что у него своя корневая сетевая конфигурация.)
- 🔹конец в root netns — `vethXXX`.
![[Pasted image 20251030121323.png]]

Для связи подов на узле используется **сетевой мост Linux** (`cbr0`), аналогичный `docker0` в Docker.  
Мост соединяет все `veth`-интерфейсы подов и обеспечивает их взаимодействие.
![[Pasted image 20251030121501.png]]


📦 Пример обмена пакетами:
1. Пакет из pod1 выходит через `eth0` → `vethXXX` → `cbr0`.
    
2. `cbr0` по ARP-запросу определяет, какому `vethYYY` принадлежит нужный IP.
    
3. Пакет пересылается через `vethYYY` в `netns` pod2.

➡️ Таким образом, поды одного узла обмениваются данными через виртуальный мост `cbr0`.

## 📙 Взаимодействие между узлами (inter-node)
Чтобы ==поды разных узлов могли общаться==, каждый узел получает свой диапазон IP-адресов (CIDR) для подов.  

> Связь между узлами может быть реализована разными способами 
> - через L2 (ARP), 
> - L3 (IP-маршрутизацию), 
> - оверлейные сети (VXLAN, GRE) и т.п.  

Обычно это **обеспечивается маршрутизацией** _на уровне облачного провайдера_ **или CNI-плагином** (Calico, Flannel и др.).


![[Pasted image 20251030122137.png]]
📦 Пример: пакет идёт из pod1 (на node1) → pod4 (на node2):
1. Пакет выходит из `pod1` через `eth0 → vethXXX → cbr0`.
    
2. `cbr0` не находит получателя на этом узле и отправляет пакет через основной интерфейс `eth0`.
    
3. Таблица маршрутизации node1 направляет трафик на node2, так как IP pod4 входит в CIDR node2.
    
4. На node2 пакет принимается `eth0`, перенаправляется на `cbr0` (включён IP forwarding).
    
5. `cbr0` по ARP-запросу находит интерфейс `vethYYY`, связанный с pod4.
    
6. Пакет доставляется в `pod4` через виртуальный линк.

➡️ Таким образом, поды разных узлов общаются напрямую по своим IP без NAT, благодаря маршрутам и CNI-плагину.

## 💡 СЕТЬ ТОЛЬКО ВНУТРИ КЛАСТЕРА 💡
По умолчанию **поды в Kubernetes могут свободно общаться друг с другом внутри кластера**, но **не имеют прямого доступа во внешнюю сеть** (Интернет).

### Почему
- Kubernetes создаёт **внутреннюю плоскую сеть для подов**, где каждому поду выделяется IP из **кластера** (CIDR).
    
- Эти IP уникальны внутри кластера, но **не routable в интернете**.
    
- Любой трафик на внешние сети должен проходить через **узел (Node) или NAT/маршрутизатор)**.

### Как получают доступ к Интернету
Чтобы поды могли обращаться ВО внешнюю сеть, обычно делают одно из двух:
1. **NAT через узел (SNAT / Masquerade)**
    - CNI-плагин (например, Flannel, Calico) настраивает NAT на узлах;
    - пакеты из пода заменяют source IP на IP ноды при выходе наружу.
        
2. **Внешний шлюз / egress**
    - Используется специальный узел или сервис (например, egress gateway), который маршрутизирует трафик подов во внешнюю сеть.

А чтобы ПРИНИМАТЬ ЗАПРОСЫ из ВНЕШНЕЙ сети:
1. **Ingress** 
	- Трафик извне в сервисы внутри кластера (доступ к веб-приложениям, API и т.д.)

# 📕2.Сетевые сервисы Kubernetes.

> [!NOTE] Объект `Service`
> **Сервис** – тип ресурса Kubernetes, который заставляет прокси настраиваться `для пересылки запросов на набор контейнеров`. 
> 
> ==Набор контейнеров==, которые будут получать трафик, ==определяется селектором== соответствующим присвоенным ==при создании контейнеров меткам==.
> 
> Существует **на уровне всего кластера** и ==предоставляет СТАБИЛЬНУЮ ТОЧКУ ДОСТУПА (IP) к группе подов== через селектор, независимо от их расположения на нодах.
> 

Причины необходимости ПОСТОЯННОГО IP для ПОДА:
- Pod — сущность **временная** (может быть пересоздан, IP изменится);
- приложение должно иметь **постоянную точку доступа** (DNS-имя);

### Задачи которые решает Service
✅ ⭐⭐⭐ **Стабильный IP/DNS** ДЛЯ ПОДЫ⭐⭐⭐
```css
# Под может поменять IP
pod-frontend: 10.244.1.5 → 10.244.1.8 (после перезапуска)

# Service остается тем же
service-frontend: 10.96.123.45 (не меняется)
frontend-service.default.svc.cluster.local (DNS)
```

✅ **Балансировка нагрузки**
```bash
# 3 пода за одним Service
$ kubectl get pods -l app=backend
backend-pod-1: 10.244.1.5
backend-pod-2: 10.244.1.6  
backend-pod-3: 10.244.2.3

# Service распределяет трафик между ними
service-backend: 10.96.123.46 → [pod-1, pod-2, pod-3]
```

✅ **Service Discovery** (Обнаружение служб - приложений)
```javascript
// Вместо "куда подключиться?"
const unstableBackendIP = "10.244.1.5"; // Может измениться!

// ⭐Просто "подключиться к бэкенду"⭐
const stableBackend = "http://backend-service:3000";
```

### 🟡 ClusterIP (ВНУТРИ кластера)🟡
**ClusterIP** — сервис Kubernetes ==по умолчанию==. 
	Он обеспечивает сервис внутри кластера, **к которому могут обращаться другие приложения внутри кластера**. ❗❗❗==Внешнего доступа нет==❗❗❗

**Для чего используется:** Связь между микросервисами внутри кластера.

YAML для сервиса ClusterIP выглядит следующим образом:
```yaml
apiVersion: v1
kind: Service
metadata: 
  name: my-internal-service
  
spec:
  type: ClusterIP
  selector:          
    app: my-app
  ports: 
  - name: http
    port: 80
    targetPort: 80
    protocol: TCP
```

![[Pasted image 20251030133540.png]]

Теперь если залогиниться от определенного КОНТЕЙНЕРА, то можно обращаться к другим контейнерам из этого же Deployment.

Например, если контейнеры в ответ на обращение возвращают строку "Hello from <имя_контейнера>", то если выполнить на контейнере kuber-dh4zg:
```bash
curl http://172.20.8.191

//❗❗Или по имени сервиса❗❗ (можно так как есть встроенные DNS)
```

Получим ОТВЕТ от РАНДОМНОГО контейнера, который является частью того же Deployment = `labels: http-server`
```bash
Hello from kuber:tzkt4
```


🧠Однако внешний доступ можно обеспечить при помощи Proxy-сервера
![[Pasted image 20251030125229.png]]
Запускаем прокси-сервер Kubernetes:
```
$ kubectl proxy --port=8080
```

Теперь можно выполнять навигацию по API Kubernetes для доступа к этому сервису, используя схему:
```http
http://localhost:8080/api/v1/proxy/namespaces/services/<SERVICE-NAME>:<PORT-NAME>/  
```

Используем этот адрес для доступа к вышеуказанному сервису:
```http
http://localhost:8080/api/v1/proxy/namespaces/default/services/my-internal-service:http/
```


> **Headless Service** (`ClusterIP=None`) — особый случай, когда DNS-сервис Kubernetes возвращает IP всех подов без виртуального IP, позволяя клиентам реализовать свой алгоритм балансировки или обнаружения.


### 🟡 ExternalName (ВНУТРИ кластера)🟡
**ExternalName** - сервис, который служит ПСЕВДОНИМОМ для приложения
	Бывает полезно, когда нам ==нужно подключаться к ВНЕШНЕМУ приложению ПО ИМЕНИ== (DNS-name нашего сервиса, а не IP)

**Для чего используется:** Доступ к внешним сервисам через DNS имя внутри кластера.

```yaml
apiVersion: v1
kind: Service
metadata:
  name: external-database
  namespace: default
  
spec:
  type: ExternalName
  externalName: database.company.com  # ← ❗❗Внешний DNS❗❗
  
  # НЕТ selector - не указывает на поды!
  ports:
  - name: postgresql
    port: 5432
    protocol: TCP
```


### 🟡 NodePort (ДОСТУП ИЗВНЕ) 🟡
**NodePort** — как следует из названия, ==сервис получает фиксированный порт на КАЖДОМ УЗЛЕ==, через который доступен извне 
(подходит для простого проброса порта, но вручную управлять не очень удобно).

> [!NOTE] ⭐⭐При открытии NodePort, открывается указанный порт НА ВСЕХ НОДАХ⭐⭐
> 


```yaml
apiVersion: v1
kind: Service
metadata:
  name: web-service
  namespace: production
  annotations:
    description: "Web application service"
    
spec:
  type: NodePort
  
  selector:
    app: frontend
    environment: production
    
  ports:
  - name: http
    port: 80
    targetPort: 3000
    nodePort: 30080 # Фиксированный порт на нодах (30000-32767)
    protocol: TCP
    
  - name: https
    port: 443
    targetPort: 3443
    nodePort: 30443
    protocol: TCP
```

![[Pasted image 20251030141228.png]]

То есть, мы стучимся на ПУБЛИЧНЫЙ IP-адрес НОДЫ (ПУБЛИЧНЫЙ IP нода получается когда хостится на серверах), при этом УКАЗЫВАЕМ ПОРТ сервиса `NodePort`, а он в свою очередь "проксирует" запросы на ЛЮБОЙ из подов

#### sessionAffinity
Специальный атрибут, с помощью которого сервис NodePort перенаправляет все запросы НА ОДИН И ТОТ ЖЕ под.

```yaml
spec:
   sessionAffinity: ClientIp
```

И в следующий раз, когда клиент отправит запрос в приложение, Service отправит ПЕРВЫЙ запрос рандомно на одну из под. А в следующие разы Kubernetes запомнит с какого Ip стучался клиент, и перенаправит запрос на тот же под что и первый раз

### 🟡 LoadBalancer (ДОСТУП ИЗВНЕ) 🟡
**LoadBalancer** — интеграция с внешним балансировщиком (обычно облачным, ЛОКАЛЬНО НЕВОЗМОЖНО). 
Kubernetes сам запрашивает у провайдера внешний IP/балансировщик и привязывает его к сервису.
> 💡Аналог NodePort💡

![[Pasted image 20251030142520.png]]

Просто равномерно распределяет трафик по нодам, ==и по порту==, указанному в .yaml файле при конфигурации (указывается в nodePort, ==НО ОБЫЧНО ВЫБИРАЕТСЯ РАНДОМНО Облачным провайдером==)
