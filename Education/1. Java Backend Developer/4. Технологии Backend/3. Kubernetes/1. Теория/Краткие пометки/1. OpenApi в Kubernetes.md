В **Kubernetes** формат **OpenAPI** используется **точно по той же логике, что и Swagger/OpenAPI в Java-приложениях** — это спецификация, описывающая:

- какие есть **эндпоинты** (например `/api/v1/pods`, `/apis/apps/v1/deployments`);
    
- какие **типы ресурсов** доступны;
    
- какие **поля** и **типы данных** у каждого ресурса (например, что `spec.replicas` — `integer`);
    
- какие **операции** допустимы (`GET`, `POST`, `PUT`, `DELETE`);

### Пример
Если ты обратишься к Kubernetes API напрямую:
```http
https://<api-server>/openapi/v2
```
(или `/openapi/v3` — для новых версий),

ты получишь **JSON-документ** с описанием всех ресурсов кластера, их схем и эндпоинтов.

Этот документ и есть спецификация **OpenAPI Schema**, по которой:

- `kubectl` знает, какие поля допустимы в YAML;
    
- IDE и утилиты (например, `Lens`, `Octant`, `kubectl explain`) могут подсказывать структуру манифестов;
    
- клиенты (kubectl, SDK, контроллеры) могут динамически находить версии API и типы ресурсов — это и называется **API discovery**.
