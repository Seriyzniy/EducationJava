> ==Транзакция== — это **минимальная неделимая единица работы с БД**. 
>  
> Либо выполняется полностью, либо не выполняется вовсе.**

Транзакция — это гарантия **ACID**:
- **Atomicity** — атомарность (_всё или ничего_)
    
- **Consistency** — согласованность данных
    
- **Isolation** — изоляция от других транзакций
    
- **Durability** — данные надёжно сохранены

---
## Высокоуровневый взгляд на работу транзакции
Внутри транзакции могут происходить:
- чтения
- обновления
- вставки
- удаления
- проверка ограничений
- триггеры
- логирование изменений

СУБД должна:
1. **запомнить все изменения (но не применять их “по-настоящему”, пока нет COMMIT)**
    
2. **дать тебе читать “логику транзакции”, а не чужие изменения**
    
3. **обеспечить rollback — вернуть всё в исходное состояние**

---
## Механизмы, из которых состоит транзакция

Транзакция обеспечивается комбинацией:
### ✔ 1. **MVCC (многоверсионность данных)**
Строки НЕ ПЕРЕЗАПИСЫВАЮТСЯ — создаются новые версии.

### ✔ 2. **Журнал транзакций (WAL / Redo log)**
Правило WAL:
> **Сначала пишем в журнал, потом в таблицу.**

Это обеспечивает Durability — безопасность данных.

Если сервер упадёт:
- по журналу можно полностью восстановить состояние
- незакоммиченные транзакции просто не будут применены

#### Структура WAL
WAL — это:
- **не набор отдельных записей**,
- **не база данных**,
- **не список транзакций**,

а ==последовательный бинарный поток, разбитый на файлы== фиксированного размера (обычно 16 MB). В каждом файле хранится последовательность изменений

Пример
```css
WAL файл 00000001000000000000000A  (содержит LSN 1–10)
WAL файл 00000001000000000000000B  (содержит LSN 11–20)
WAL файл 00000001000000000000000C  (содержит LSN 21–30)
```



### ✔ 3. **Undo log / rollback segments**
**Хранят “предыдущее состояние” строк**— для отката.

Перед изменением строки СУБД сохраняет старую версию:
- **либо в буфере** (InnoDB)
- **либо через MVCC** *как отдельную версию* (PostgreSQL)


Чтобы сделать rollback:
- система просто откатывает версии, возвращаясь к старым


Rollback НЕ переписывает таблицы — он просто:
- помечает новые версии как невидимые
- восстанавливает старые метки xmin/xmax
- снимает блокировки

Это _очень дешёвая операция_.

### ✔ 4. **Блокировки**
Гарантируют isolation.

### ✔ 5. **Буферное изменение данных**
**Изменения лежат в памяти, пока не будет COMMIT** → только тогда идут на диск.

### ✔ 6. Атомарный COMMIT
COMMIT — это запись _одного крошечного события_ в WAL:
```nginx
XID 150 COMMITTED
```
Если этой записи нет — транзакция считается незавершённой → откатывается.

> Вся атомарность транзакции держится на атомарности COMMIT-лог-записи.

# Пример
Представим такую последовательность:
```sql
BEGIN;
SELECT ...
INSERT ...
UPDATE ...
DELETE ...
INSERT ...

COMMIT;
```

Разберём что происходит “под капотом”.
## Шаг 1. BEGIN — открытие транзакции
СУБД создаёт:
- Transaction ID (XID)
    
- Snapshot текущего состояния БД (для MVCC)
    
- Пустой список изменений (write set)


Ничего не блокируется глобально.

## Шаг 2. SELECT — чтение данных
PostgreSQL и MySQL InnoDB:
- Чтение идёт НЕ НАПРЯМУЮ из таблицы
    
- Читаются **версии строк**, которые разрешены твоим snapshot’ом
    
- Незакоммиченные изменения других транзакций скрываются


Isolation работает через **MVCC**, а не через блокировки (кроме Serializable).

## Шаг 3. 🔥INSERT — добавление строки🔥
Что реально происходит:
1. **Создаётся новая версия строки** с:
    - xmin = ID текущей транзакции
    - xmax = NULL
        
2. Строка кладётся в буфер (данные ещё не на диске)
    
3. **В** журнал WAL/REDO **log записывается операция вставки**
    
4. Возможно берутся **локальные блокировки** (row-level locks)


Но ==другие транзакции НЕ видят эту строку==, пока не будет COMMIT

## Шаг 4. UPDATE — обновление строки
Обновление _не перезаписывает строку_!  

В MVCC это выглядит так:
1. **Старая строка → получает xmax = ID транзакции**
2. **Создаётся новая строка → с другим xmin и xmax = NULL**


Т.е. происходит:
> Вставка новой версии + пометка старой версии как “устаревшей”.

Rollback может просто “отменить” новую, и старая снова станет актуальной.

## Шаг 5. DELETE — удаление строки
Тоже не удаление “по-настоящему”.

Под капотом:
- Ставится `xmax = XID транзакции`
    
- Строка считается “невидимой” для новых транзакций
    
- Но физически остаётся до вакуума (PostgreSQL)
    

Rollback просто снимает `xmax`.

## Шаг 6. 🔥COMMIT — ключевая операция🔥
COMMIT делает самые важные вещи.
1. В журнал WAL/Redo log записывается **COMMIT RECORD**
    
2. Данные гарантированно записаны на диск
    
3. Работа с блокировками завершается
    
4. Все версии строк становятся видимыми для других транзакций


> До COMMIT никакие изменения **не становятся частью базы данных**.

# 2. Savepoint
**Savepoint (точка сохранения)** — это механизм внутри транзакции, который позволяет **частично откатить её до определённого момента**, не откатывая всё целиком.

Проще говоря:
> Savepoint — это **“чекпоинт” внутри транзакции**.  
> Если случилась ошибка, можно откатить только часть операции.

## Для чего нужны?
Они позволяют:
### 1. 🔥**Откатывать только часть транзакции**🔥
Без savepoint:
- Ошибка → весь транзакционный блок откатывается.

С savepoint:
- Ошибка → откатывается только участок **между savepoint и текущим моментом**.

### 2. **Продолжать транзакцию после отката**
Это важно: откат до savepoint **не завершает транзакцию**, она продолжается.

### 3. **Строить сложные операции**
Например:
- Выполнить 10 вставок
- Если 8-я не проходит — откатить только её, остальные оставить

### 4. **Делать вложенные логические транзакции**
Хотя SQL не поддерживает настоящие nested transactions,  
**savepoint имитируют “локальную транзакцию”**.

---
## Под капотом
Механизм зависит от СУБД, но общая идея:
1. Транзакция состоит из множества внутренних шагов.
    
2. **Savepoint фиксирует состояние, которое можно “отмотать назад”**, не влияя на остальные шаги.
    
3. Система:
	- помнит логическую точку в логах транзакций (WAL/Redo log)
	- позволяет откатить операции до этой точки
	- не закрывает locks и не закрывает саму транзакцию (в отличие от ROLLBACK)

### Важные детали:
- Откат до savepoint не снимает блокировки, взятые транзакцией до этого момента.
    
- Откат затрагивает только _данные_, а не _структуру транзакции_.
