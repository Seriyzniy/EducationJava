# 1. ACID
ACID — это четыре свойства транзакции:
1. **Atomicity (Атомарность)** — либо вся транзакция выполняется, либо откатывается полностью.
    
2. **Consistency (Согласованность)** — после транзакции база остаётся в корректном состоянии (соблюдены все ограничения, правила).
    
3. **Isolation (Изоляция)** — параллельно выполняющиеся транзакции не должны «мешать» друг другу.
    
4. **Durability (Надёжность)** — результат транзакции сохраняется даже при сбоях.
    

👉 **Уровни изоляции** — это конкретные _режимы_, которые определяют, насколько сильно транзакции изолированы друг от друга.  

> По сути, уровни изоляции — это _детальная настройка буквы "I" из ACID_.

# 2. Проблемы конкурентного доступа (anomalies)
## 1. Dirty Read (грязное чтение)
**Сценарий:** одна транзакция читает данные, которые другая ещё не закоммитила — и которые могут быть откатаны.

### Пример
Есть таблица `accounts(balance)`.
Начальное состояние:
```java
balance = 100
```

### Ход событий
T1 (покупка)
```sql
UPDATE accounts SET balance = 0;
-- 🔥пока не коммитит!🔥
```

T2 (проверка баланса)
```sql
SELECT balance FROM accounts;
--> получит 0   <--- ❗❗грязное чтение❗❗
```

T1 (передумала)
```sql
ROLLBACK;
-- баланс снова 100
```

👉 **T2 прочитала данные, которых никогда "на самом деле" не было.**

## 2. Non-repeatable read (неповторяемое чтение)
Это когда **транзакция ДВАЖДЫ ЧИТАЕТ одну и ту же строку**, НО ВИДИТ _разные значения_, 
потому что **другая транзакция успела сделать UPDATE/DELETE и закоммитить.** (ОБНОВЛЯЮТСЯ старые данные)

### Пример
Начальное состояние:
```ini
price = 200
```

### Ход событий
T1
```sql
SELECT price FROM products WHERE id=1;  -- результат: 200
```

T2
```sql
UPDATE products SET price = 300 WHERE id=1;
COMMIT;
```

T1 (повторное чтение)
```sql
SELECT price FROM products WHERE id=1;
--> результат: 300  <--- non-repeatable read
```
👉 T1 увидела, что значение "поменялось у неё на глазах".

## 3.3. Phantom read (фантомное чтение)
Это когда **при повторном выполнении запроса ПОЯВЛЯЮТСЯ _НОВЫЕ СТРОКИ_**, подходящие под условие (или исчезают).

> ❗❗ В РАМКАХ ТОЙ ЖЕ ТРАНЗАКЦИИ, а не новой ❗❗
> Допустим блок кода начинает ТРАНЗАКЦИЮ Т1. Читает данные первый раз, обрабатывает и еще раз читает данные В ТОЙ же транзакции. 
> 
> И получили два разных ответа (НОВЫЕ ДАННЫЕ, которых раньше не было)
### Пример
Есть таблица заказов:

|id|amount|
|---|---|
|1|100|

### Ход событий
T1
```sql
SELECT COUNT(*) FROM orders WHERE amount > 50;
-- результат: 1
```

T2
```sql
INSERT INTO orders(amount) VALUES (70);
COMMIT;
```

T1 снова выполняет тот же SELECT:
```sql
SELECT COUNT(*) FROM orders WHERE amount > 50;
-- результат: 2   <--- фантомное чтение
```

👉 **Строки не менялись, но “появились новые”, подходящие под условие.**

---
## 4. Lost Update (потерянное обновление)
Это когда изменения одной транзакции затирают изменения другой — одна "переписывает" другую.

### Пример
Общий ресурс:
```ini
likes = 10
```

### Ход событий
T1 читает
```sql
SELECT likes FROM posts WHERE id=1;
--> 10
```

T2 читает
```sql
SELECT likes FROM posts WHERE id=1;
--> 10
```

#### Обе транзакции считают, что в строке `likes = 10`.
T1 увеличивает на 1
```sql
UPDATE posts SET likes = 11 WHERE id=1;
COMMIT;
```

T2 тоже увеличивает на 1
```sql
UPDATE posts SET likes = 11 WHERE id=1;
COMMIT;
```

👉 Итог:  
**likes = 11**, хотя должно быть 12.  
Обновление T1 фактически **потеряно**.
