Когда вы используете `@KafkaListener` на уровне класса, необходимо указать `@KafkaHandler` на уровне методов. 

При доставке сообщений, тип преобразованной полезной нагрузки используется для определения, какой метод вызывать.
```java
@KafkaListener(id = "multi", topics = "myTopic")
static class MultiListenerBean {

    @KafkaHandler
    public void listen(String foo) {
        // Обрабатывает строковые сообщения
    }

    @KafkaHandler
    public void listen(Integer bar) {
        // Обрабатывает целочисленные сообщения
    }

    @KafkaHandler(isDefault = true)
    public void listenDefault(Object object) {
        // Обрабатывает все остальные типы (по умолчанию)
    }
}
```

## Ключевые особенности:
**Версия 2.1.3+:** Можно назначить метод `@KafkaHandler` как обработчик по умолчанию, если нет совпадения с другими методами. Только один метод может быть таким.

**Требования к конвертации:** Полезная нагрузка должна быть уже преобразована в доменный объект для сопоставления. 

Используйте:
- Кастомный десериализатор
- `JsonDeserializer`
- `JsonMessageConverter` с `TypePrecedence.TYPE_ID`
 

**Ограничение для default-методов:** Не могут получать отдельные заголовки через `@Header`, должны использовать `ConsumerRecordMetadata`:

```java
@KafkaHandler(isDefault = true)
public void listenDefault(
	Object object, 
	@Header(KafkaHeaders.RECEIVED_TOPIC) String topic) 
{
    ...
}
```

### 🎯 **Идеальные use-cases:**
- 🔥Обработка **разных типов сообщений** из одного топика
    
- **Polymorphic DTOs** - когда в топик пишутся разные типы объектов
    
- **Versioned APIs** - обработка разных версий сообщений

### ⚠️ **Важные ограничения:**
- Default-методы **не могут использовать `@Header`** отдельно
    
- Требуется **правильная настройка десериализации**
    
- **Только один default-метод** на класс
