При использовании конкурентного контейнера слушателей сообщений (concurrent message listener container) один экземпляр слушателя вызывается на всех потоках потребителя. 

Когда вы используете:
```java
@KafkaListener(concurrency = "5")
```
Spring создаёт **5 потоков**, но слушатель (метод) — **один объект**.  
То есть 5 потоков вызывают **один и тот же экземпляр слушателя** → он должен быть **потокобезопасным**.

Лучше всего, если слушатель — **stateless** (не хранит mutable состояние).

Поэтому слушатели должны быть потокобезопасными, и предпочтительно использовать слушатели без состояния (stateless). 


## Если слушателя невозможно сделать потокобезопасным?
Если невозможно сделать ваш слушатель потокобезопасным или добавление синхронизации значительно снизит преимущество добавления параллелизма, **вы можете использовать одну из нескольких техник**:
1. Используйте n контейнеров с `concurrency=1` и бином `MessageListener` с областью видимости prototype, чтобы каждый контейнер получал свой собственный экземпляр (это невозможно при использовании `@KafkaListener`).
    
2. Храните состояние в экземплярах `ThreadLocal<?>`.
    
3. Поручите синглтон-слушателю делегирование бину, который объявлен в `SimpleThreadScope` (или аналогичной области видимости).

## Очистка состояния потока
Чтобы облегчить очистку состояния потока (для второго и третьего пунктов в предыдущем списке), 
начиная с версии 2.2, **контейнер слушателя публикует событие** `ConsumerStoppedEvent`, когда каждый поток завершает работу. 
```java
ConsumerStoppedEvent
```


Вы можете потреблять эти события с помощью `ApplicationListener` или метода `@EventListener`, чтобы удалить экземпляры `ThreadLocal<?>` или удалить (`remove()`) бины с областью видимости потока из области видимости. 

Вы можете его ловить:
```java
@EventListener
public void onStop(ConsumerStoppedEvent event) {
    threadLocal.remove();
}
```
Это нужно для:
- очистки ThreadLocal
- уничтожения ThreadScope bean вручную

> Обратите внимание, что `SimpleThreadScope` не уничтожает бины, которые имеют интерфейс уничтожения (такой как `DisposableBean`), поэтому вы должны уничтожить (`destroy()`) экземпляр самостоятельно.

По умолчанию, вещатель событий (event multicaster) контекста приложения вызывает слушателей событий в вызывающем потоке. Если вы измените вещатель на использование асинхронного исполнителя, очистка потока не будет эффективной.


## Особое примечание о виртуальных потоках и конкурентных контейнерах слушателей сообщений
ConcurrentMessageListenerContainer **пока плохо совместим с виртуальными потоками**, потому что:
- внутри Kafka client используются synchronized-блоки
    
- виртуальные потоки могут зависнуть (pinned) на платформенных потоках
    
- возможны race conditions
 

✔ Рекомендация:  
**Не ставить concurrency выше количества платформенных потоков**, пока Kafka и Spring не обновят реализацию.
