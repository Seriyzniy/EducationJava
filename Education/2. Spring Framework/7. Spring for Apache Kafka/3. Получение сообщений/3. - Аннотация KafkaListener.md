# 1. Аннотация @KafkaListener  
Аннотация `@KafkaListener` используется **для ОБОЗНАЧЕНИЯ МЕТОДА бина как слушателя для контейнера** слушателя. 

```java
@KafkaListener(topics = "orders")
public void listen(String msg) { ... }
```

Spring **автоматически превращает этот метод в Kafka-consumer**.

Но метод сам по себе не умеет слушать Kafka.  
Поэтому Spring создаёт специальный адаптер:

### MessagingMessageListenerAdapter
Это “обёртка”, которая:
- принимает `ConsumerRecord` от Kafka
- конвертирует полученные данные (ключ, значение, заголовки)
    
- **вызывает ваш метод с правильными аргументами**
    
- выполняет ack/nack при необходимости
- применяет message converter (например, JSON → объект)
- работает с `@SendTo`, параметрами, batch/listener modes и т. д.

---
# 1.1 Свойства аннотации

## 1. Свойство id и groupId
С версии **2.0** свойство `id` аннотации может использоваться как `group.id` Kafka-потребителя, **переопределяя** настройку фабрики.

Если нужно использовать старое поведение фабрики, можно установить `idIsGroup = false`.
 
Можно задавать `groupId` **явно**, независимо от `id`.

## 2. topics
Список топиков, на которые подписывается слушатель.
```java
@KafkaListener(topics = {"topic1", "topic2"})
public void listen(String msg) { ... }
```

## 3. topicPartitions
Явное указание топиков и разделов, можно с начальными смещениями.
```java
@KafkaListener(topicPartitions = {
    @TopicPartition(topic = "topic1", partitions = {"0","1"}),
    @TopicPartition(topic = "topic2", partitionOffsets = @PartitionOffset(partition = "0", initialOffset = "100"))
})
public void listen(ConsumerRecord<?, ?> record) { ... }
```

## 4. containerFactory
Указывает, какую фабрику контейнера использовать для слушателя (`ConcurrentKafkaListenerContainerFactory`).
```java
@KafkaListener(
	topics = "myTopic", 
	containerFactory = "kafkaListenerContainerFactory")
public void listen(String msg) { ... }
```
> Указываем имя бина

## 5. **concurrency**
Количество потоков для параллельной обработки (работает с ConcurrentKafkaListenerContainerFactory).
```java
@KafkaListener(topics = "myTopic", concurrency = "3")
public void listen(String msg) { ... }
```

## 6. batch
Указывает, что слушатель принимает **пачки сообщений**.
```java
@KafkaListener(
	topics = "myTopic", 
	containerFactory = "batchFactory", 
	batch = "true")
public void listen(List<String> messages) { ... }

```

## 7. Свойства потребителя (`properties`)
С версии **2.2.4** можно задавать Kafka-свойства прямо в аннотации:
```java
@KafkaListener(
	topics = "myTopic", 
	groupId = "group", 
	properties = {
	    "max.poll.interval.ms:60000",
	    ConsumerConfig.MAX_POLL_RECORDS_CONFIG + "=100"
})
```
❗❗❗Исключения: `group.id` и `client.id` **игнорируются** в `properties`. Используйте `groupId` и `clientIdPrefix` вместо них.


## Использование плейсхолдеров и SpEL
Аннотация поддерживает:
- `${…}` — значения из *application.properties*
```java
@KafkaListener(topics = "${kafka.topic.orders}")
public void listen(String msg) {}
```

- `#{…}` — Spring Expression Language (*SpEL*)
```java
@KafkaListener(topics = "#{myBean.topicName}")
public void listen(String msg) {}
```

# 1.2 Допустимые сигнатуры
Метод, помеченный `@KafkaListener`, может иметь следующие **параметры**
## 1.Основные варианты
### **payload** сообщения (обычно String, Integer, POJO, byte[])
```java
@KafkaListener(topics = "myTopic")
public void listen(String message) { ... }

@KafkaListener(topics = "myTopic") 
public void listen(Integer message) { ... }

@KafkaListener(topics = "myTopic")
public void listen(MyPojo message) { ... }

@KafkaListener(topics = "myTopic")
public void listen(byte[] message) { ... }
```

### `ConsumerRecord<?, ?>` для получения нативного объекта Kafka
```java
@KafkaListener(topics = "myTopic")
public void listen(ConsumerRecord<Integer, String> record) { ... }
```

### `ConsumerRecords<?, ?>` для batch-listeners (полный результат `poll()`)
```java
@KafkaListener(
	topics = "myTopic", 
	containerFactory = "batchFactory")
public void listen(ConsumerRecords<Integer, String> records) 
{ 
	... 
}
```

### `Message<?>` (Spring Messaging), чтобы получить payload и headers
```java
@KafkaListener(
	topics = "myTopic", 
	containerFactory = "batchFactory")
public void listen(List<Message<String>> messages) { ... }
```


## 2. Метаданные через @Header
Можно получить topic, partition, offset, timestamp и key:
```java
@KafkaListener(topics = "myTopic")
public void listen(String msg,
   @Header(KafkaHeaders.RECEIVED_TOPIC) String topic,
   @Header(KafkaHeaders.RECEIVED_PARTITION) int partition,
   @Header(KafkaHeaders.OFFSET) long offset) 
{ 
	...
}
```
- `@Payload` — явное указание payload (опционально)
- `@Header` — KafkaHeaders.* или любые другие headers

## 3. Acknowledgment для ручного подтверждения
Если используется `AckMode.MANUAL`:
```java
@KafkaListener(
	topics = "myTopic", 
	containerFactory = "manualAckFactory")
public void listen(String msg, Acknowledgment ack) {
    // обработка
    ack.acknowledge();
}
```

## 4. Batch Listeners
### Список payload:
```java
@KafkaListener(
	topics = "myTopic", 
	containerFactory = "batchFactory")
public void listen(List<String> messages) { ... }
```

### Список ConsumerRecord
```java
@KafkaListener(
	topics = "myTopic", 
	containerFactory = "batchFactory")
public void listen(
	List<ConsumerRecord<Integer, String>> records, 
   Acknowledgment ack) 
{ 
	... 
}
```

### Список Message
```java
@KafkaListener(
	topics = "myTopic", 
	containerFactory = "batchFactory")
public void listen(
	List<Message<String>> messages, 
    Acknowledgment ack)
{ 
	... 
}
```

❗❗В методе **может быть только один “основной параметр”** типа ConsumerRecords, `List<ConsumerRecord>` или `List<Message<?>>` (кроме опциональных Acknowledgment или `Consumer<?, ?>`).

# 1. Слушатели записей (Record Listeners) - автоматическое распределение партиций
## Record Listener
Это слушатель, который обрабатывает **одну запись Kafka за вызов**:
```java
public void listen(String data)
```

То есть метод принимает одну запись (значение), и Spring Kafka сам:
- создаёт consumer
- выполняет poll
- передаёт запись вашему методу
- делает commit (если включено)
- обрабатывает ошибки

И всё это — автоматически.

### Record Listener и аннотация KafkaListener
Аннотация **превращает метод в Kafka listener**.

Следующий пример показывает, как ее использовать:
```java
public class Listener {

    @KafkaListener(
	    id = "foo", 
	    topics = "myTopic", 
	    clientIdPrefix = "myClientId")
    public void listen(String data) {
        ...
    }
```

Что происходит под капотом:
1. Spring на старте обнаруживает методы с `@KafkaListener`
	
2. Для каждого метода создаётся **KafkaMessageListenerContainer** или **ConcurrentMessageListenerContainer** (если используется фабрика concurrent).
    
3. Ваш метод оборачивается в `MessagingMessageListenerAdapter` — адаптер, который преобразует входные Kafka записи и вызывает ваш метод.


Этот механизм требует 
- аннотацию `@EnableKafka` на одном из ваших классов `@Configuration` (для поиска всех методов с @KafkaListener)
- фабрику контейнеров слушателя

*По умолчанию* ожидается бин с именем 
```java
kafkaListenerContainerFactory
```

Следующий пример показывает, как использовать `ConcurrentMessageListenerContainer`:
```java
@Configuration
@EnableKafka
public class KafkaConfig {

	//Фабрика для слушателей
    @Bean
    KafkaListenerContainerFactory
	<ConcurrentMessageListenerContainer<Integer, String>>
	    kafkaListenerContainerFactory() 
	{
        ConcurrentKafkaListenerContainerFactory<Integer, String> 
		    factory = 
			    new ConcurrentKafkaListenerContainerFactory<>();
        factory.setConsumerFactory(consumerFactory());
        factory.setConcurrency(3);
        factory.getContainerProperties().setPollTimeout(3000);
        return factory;
    }

	// Фабрика Consumer
    @Bean
    public ConsumerFactory<Integer, String> consumerFactory() {
        return new DefaultKafkaConsumerFactory<>(consumerConfigs());
    }

	// Настройки для фабрики
    @Bean
    public Map<String, Object> consumerConfigs() {
        Map<String, Object> props = new HashMap<>();
        props.put(
	        ConsumerConfig.BOOTSTRAP_SERVERS_CONFIG,
	        "localhost:9092");
        ...
        return props;
    }
}
```

### Что такое clientIdPrefix?
Spring позволяет автоматически назначать `client.id` потребителям:
```java
@KafkaListener(..., clientIdPrefix = "myClientId")
```

Если concurrency = 3, будут созданы клиентские ID:
```bash
myClientId-0
myClientId-1
myClientId-2
```

### Настройки контейнера через фабрику
Контейнер listener’a создаётся фабрикой, копируя параметры из:
```java
factory.getContainerProperties()
```
То есть фабрика выступает **шаблоном**, из которого создаются реальные контейнеры.

Если вы меняете какое-то свойство:
```java
factory.getContainerProperties().setPollTimeout(3000);
```
— это применится ко всем listener’ам, использующим эту фабрику.

# 2. Явное назначение партиций для слушателя

Обычно `@KafkaListener` сам берет список разделов (partition) у Kafka и автоматически распределяет их между инстансами контейнеров.

**Explicit partition assignment** (ручное назначение) позволяет вам _жестко указать_, какие конкретные partition из каких topic должен читать данный слушатель.

Это отключает групповое распределение и передаёт полный контроль разработчику.

```java
@KafkaListener(id = "thing2", topicPartitions = {
    @TopicPartition(topic = "topic1", partitions = { "0", "1" }),
    @TopicPartition(
	    topic = "topic2", 
	    partitions = "0",
        partitionOffsets = @PartitionOffset(
	        partition = "1", 
	        initialOffset = "100"
	    )
	)
})
public void listen(ConsumerRecord<?, ?> record) { ... }
```

Для каждого `@TopicPartition` можно указывать:
#### 1. partitions = { ... }
Просто список разделов — начнём читать “как обычно” → с committed offset.

#### 2. partitionOffsets = { ... }
Задаем:
- partition
- initialOffset

⚠ **Нельзя смешивать один и тот же partition в обоих списках.**

### # SpEL (#{...}) и property placeholders (${...})
То есть можно писать:
```java
@KafkaListener(
   topics = "#{environment.getProperty('app.kafka.topic')}",
   groupId = "${kafka.group-id}"
)
```
Используется для конфигурации из:
- application.properties
- bean-методов
- выражений SpEL

### Поддержка wildcard `"*"`
С версии **2.5.5** можно задать:
```java
@TopicPartition(
    topic = "topic1",
    partitions = { "0", "1" },
    partitionOffsets = @PartitionOffset(partition = "*", initialOffset = "0")
)
```
Значение `"*"` означает:
➡ Применить offset **ко всем partition**, указанным в `partitions`.

### Новые seekPosition (BEGINNING, END, TIMESTAMP)
С версии **3.2** `@PartitionOffset` поддерживает новый параметр:
```java
seekPosition = "BEGINNING"
seekPosition = "END"
seekPosition = "TIMESTAMP"
```

```java
@KafkaListener(id = "seekPositionTime", topicPartitions = {
    @TopicPartition(topic = "topicX", partitionOffsets = {
        @PartitionOffset(
	        partition = "0", 
	        initialOffset = "723916800000", 
	        seekPosition = "TIMESTAMP"),
        @PartitionOffset(
	        partition = "1", 
	        initialOffset = "0", 
	        seekPosition = "BEGINNING"),
        @PartitionOffset(
	        partition = "2", 
	        initialOffset = "0", 
	        seekPosition = "END")
    })
})
public void listen(ConsumerRecord<?, ?> record) { }
```

# 3. Ручное подтверждение коммита (Manual Acknowledgment)
Когда Kafka listener обрабатывает сообщение, он должен подтвердить его обработку — это называется **commit offset**.

По умолчанию Spring Kafka делает это автоматически (auto commit / manual_immediate / batch и т.д.).

Но если вы хотите **самостоятельно решать, когда считать сообщение обработанным**, нужно включить ручной режим подтверждения — **Manual AckMode**.

## Настройка ручного AckMode
При использовании `@KafkaListener` нужно создать **кастомный контейнерный factory**, где:
1. Получаем `containerProperties`
2. Устанавливаем `AckMode` в `MANUAL` или `MANUAL_IMMEDIATE` при помощи `setAckMode()`

```java
@Bean
public ConcurrentKafkaListenerContainerFactory<String, String> kafkaManualAckListenerContainerFactory(
        ConsumerFactory<String, String> consumerFactory)
 {
    ConcurrentKafkaListenerContainerFactory<String, String> 
    factory =
            new ConcurrentKafkaListenerContainerFactory<>();

    factory.setConsumerFactory(consumerFactory);

    // включаем manual ack mode
	factory
		.getContainerProperties()
		.setAckMode(ContainerProperties.AckMode.MANUAL);

    return factory;
}
```

Указываем фабрику в Listenre
```java
@KafkaListener(
    id = "cat",
    topics = "myTopic",
    containerFactory = "kafkaManualAckListenerContainerFactory"
)
public void listen(String data, Acknowledgment ack) {
    System.out.println("Принял: " + data);

    // говорим Kafka: "Да, это сообщение успешно обработано"
    ack.acknowledge();
}
```

# 4. Метаданные сообщения (Consumer Record Metadata)
Каждое Kafka-сообщение (`ConsumerRecord`) содержит полезную нагрузку (key/value) и **метаданные**, такие как:
- номер смещения (offset)
- партиция
- топик
- timestamp
- тип timestamp
- ключ сообщения (если есть)

Spring Kafka автоматически прокладывает эти данные в **заголовки сообщения** (`MessageHeaders`).

Вы можете использовать следующие имена заголовков для извлечения заголовков сообщения:
```java
KafkaHeaders.OFFSET
    
KafkaHeaders.RECEIVED_KEY
    
KafkaHeaders.RECEIVED_TOPIC
    
KafkaHeaders.RECEIVED_PARTITION
    
KafkaHeaders.RECEIVED_TIMESTAMP
    
KafkaHeaders.TIMESTAMP_TYPE
```

### Пример
```java
@KafkaListener(id = "qux", topicPattern = "myTopic1")
public void listen(
	@Payload String data,
    @Header(name = KafkaHeaders.RECEIVED_KEY, required = false) 
	Integer key,
	
	@Header(KafkaHeaders.RECEIVED_PARTITION) int partition,
    @Header(KafkaHeaders.RECEIVED_TOPIC) String topic,
    @Header(KafkaHeaders.RECEIVED_TIMESTAMP) long ts) {

    System.out.println("Получил: " + data);
}
```

> Важно:  
> Аннотации `@Payload` и `@Header` должны быть **в методе**, а не в интерфейсе — иначе Spring их не найдёт.

> Начиная с версии 2.5, `RECEIVED_KEY` отсутствует, если входящая запись имеет `null` ключ; ранее заголовок заполнялся значением `null`. 
> Это изменение сделано, чтобы сделать фреймворк согласованным с соглашениями `spring-messaging`, где заголовки со значением `null` отсутствуют.

## ConsumerRecordMetadata (начиная с 2.5)
Вместо отдельных заголовков можно получить **все метаданные разом**:
```java
@KafkaListener(...)
public void listen(String payload, ConsumerRecordMetadata meta) {
    System.out.println("Topic = " + meta.topic());
    System.out.println("Partition = " + meta.partition());
    System.out.println("Offset = " + meta.offset());
    System.out.println("Timestamp = " + meta.timestamp());
}
```

Но **не содержит** key и value — они уже передаются отдельно.

# 5. Слушатели пачек (Batch Listeners)
**Batch Listener** — это слушатель Kafka, который **получает сразу всю пачку сообщений**, пришедших от `poll()`, вместо обработки каждого `ConsumerRecord` по отдельности.

Используется, когда:
- нужно уменьшить накладные расходы на обработку
- требуется групповая логика
- важна скорость при больших объемах
    

❗❗ _Non-blocking retries_ с batch listener **не поддерживаются**.

## Настройка bath-режима 
Начиная с версии 1.1, вы можете настраивать методы `@KafkaListener` для получения всей пачки записей потребителя, полученных из опроса потребителя.
### В контейнере
Чтобы настроить фабрику контейнеров слушателя для создания слушателей пачек, вы можете установить свойство `batchListener`. 

Следующий пример показывает, как это сделать:
```java
@Bean
public KafkaListenerContainerFactory<?> batchFactory() {
    ConcurrentKafkaListenerContainerFactory<Integer, String> 
	    factory =
            new ConcurrentKafkaListenerContainerFactory<>();
            
    factory.setConsumerFactory(consumerFactory());
    factory.setBatchListener(true); // включаем batch
    return factory;
}
```

### В аннотации
Начиная с версии 2.8, вы можете переопределять свойство `batchListener` фабрики, используя свойство `batch` в аннотации `@KafkaListener`.

Теперь можно указать, что конкретный метод — batch или record listener, даже если фабрика одна.
```java
@KafkaListener(topics = "myTopic", batch = "true")
```

## Message Converters для batch listener (с 2.9.6)
Начиная с версии 2.9.6, фабрика контейнеров имеет отдельные сеттеры для свойств конвертеров

Два отдельных конвертера:
- `recordMessageConverter` — для record listeners
- `batchMessageConverter` — для batch listeners


Позволяет настраивать разный тип преобразования payload.

## Сигнатуры методов
### Получение только полезных нагрузок (`List<String>`)
```java
@KafkaListener(
	id = "list", 
	topics = "myTopic", 
	containerFactory = "batchFactory")
public void listen(List<String> list) {
    ...
}
```

### Получение метаданных пачки?
Заголовки приходят как **List параллельных значений**:
```java
@KafkaListener(
	id = "list", 
	topics = "myTopic", 
	containerFactory = "batchFactory")
public void listen(List<String> list,
        @Header(KafkaHeaders.RECEIVED_KEY) List<Integer> keys,
        @Header(KafkaHeaders.RECEIVED_PARTITION) List<Integer> partitions,
        @Header(KafkaHeaders.RECEIVED_TOPIC) List<String> topics,
        @Header(KafkaHeaders.OFFSET) List<Long> offsets) {
    ...
}
```

### Получение `List<Message<?>>`
Вы можете получить `List` объектов `Message<?>` с каждым смещением и другими деталями в каждом сообщении, но это должен быть единственный параметр (кроме опциональных `Acknowledgment`, при использовании ручных подтверждений, и/или параметров `Consumer<?, ?>`), определенный в методе. 

Следующий пример показывает, как это сделать:
```java
@KafkaListener(
	id = "listMsg", 
	topics = "myTopic", 
	containerFactory = "batchFactory")
public void listen1(List<Message<?>> list) {
    ...
}

@KafkaListener(
	id = "listMsgAck", 
	topics = "myTopic", 
	containerFactory = "batchFactory")
public void listen2(List<Message<?>> list, Acknowledgment ack) {
    ...
}

@KafkaListener(
	id = "listMsgAckConsumer", 
	topics = "myTopic", 
	containerFactory = "batchFactory")
public void listen3(
	List<Message<?>> list, 
	Acknowledgment ack, 
	Consumer<?, ?> consumer) {
    ...
}
```
Можно добавить:
- `Acknowledgment`
- `Consumer<?, ?>`

### Получение `List<ConsumerRecord<?, ?>>` (Kafka API объекты)
```java
@KafkaListener(
	id = "listCRs", 
	topics = "myTopic", 
	containerFactory = "batchFactory")
public void listen(List<ConsumerRecord<Integer, String>> list) { }
```

Можно также добавить Acknowledgment и Consumer:
```java
public void listen(
	List<ConsumerRecord<Integer, String>> list, 
	Acknowledgment ack)
```

### Получение полного `ConsumerRecords<?, ?>` (poll result)
Начиная с версии 2.2, слушатель может получать полный объект `ConsumerRecords<?, ?>`, возвращаемый методом `poll()`, позволяя слушателю получать доступ к дополнительным методам 
- таким как `partitions()` 
	(который возвращает экземпляры `TopicPartition` в списке) 

- и `records(TopicPartition)` 
	(который получает выборочные записи). 

Опять же, это должен быть единственный параметр (кроме опциональных `Acknowledgment`, при использовании ручных подтверждений или параметров `Consumer<?, ?>`) в методе.
```java
@KafkaListener(
	id = "pollResults", 
	topics = "myTopic", 
	containerFactory = "batchFactory")
public void pollResults(ConsumerRecords<?, ?> records) {
    ...
}

@KafkaListener(
	id = "listCRsAck", 
	topics = "myTopic", 
	containerFactory = "batchFactory")
public void pollResults(
	ConsumerRecords<?, ?> records,
	Acknowledgment ack
) {
    ...
}
```
Позволяет:
- получать `records.partitions()`
- `records.records(TopicPartition tp)`
- работать с _нативным_ API Kafka

❗❗RecordFilterStrategy **игнорируется** для таких слушателей.
