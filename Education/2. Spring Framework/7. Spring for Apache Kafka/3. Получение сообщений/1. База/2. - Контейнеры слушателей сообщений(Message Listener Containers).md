## 💡 **Практические советы:**
### 🎯 Что важно запомнить:
1. **Используйте `ConcurrentMessageListenerContainer`** для многопоточности - это стандартный выбор в 99% случаев.
    
2. **AckMode BATCH** - это разумный выбор по умолчанию для большинства сценариев.
    
3. **Всегда отключайте `enable.auto.commit`** - Spring Kafka управляет коммитами лучше.

### Что использовать на практике
```java
@Bean
public ConcurrentKafkaListenerContainerFactory<String, String> kafkaListenerContainerFactory() {
    ConcurrentKafkaListenerContainerFactory<String, String> 
	    factory = 
		    new ConcurrentKafkaListenerContainerFactory<>();
    
    factory.setConsumerFactory(consumerFactory());
    factory.setConcurrency(3); // Оптимально для 3 партиций
    factory
	    .getContainerProperties()
	    .setAckMode(ContainerProperties.AckMode.BATCH);
    return factory;
}
```
`
ConcurrentKafkaListenerContainerFactory<K, V>` — это дженерик-класс из Spring Kafka, который параметризуется:
- **K** — тип _ключа сообщения_
- **V** — тип _значения сообщения_
### ⚠️ **Чего избегать:**
- Не вызывайте методы Consumer вручную в listener
	
- Не используйте RECORD AckMode для batch listeners
    
- Не забывайте про `missingTopicsFatal=true` в production

### Простая настройка в Spring Boot
```properties
spring.kafka.listener.ack-mode=BATCH
spring.kafka.listener.concurrency=3
spring.kafka.consumer.enable-auto-commit=false
```

# 0. Контейнеры слушателей
Предоставляются две реализации `MessageListenerContainer`:
- **KafkaMessageListenerContainer**
- **ConcurrentMessageListenerContainer**

**KafkaMessageListenerContainer** получает все сообщения со всех топиков или разделов в одном потоке.

**ConcurrentMessageListenerContainer** делегирует одному или нескольким экземплярам `KafkaMessageListenerContainer` для обеспечения многопоточного потребления.


> [!NOTE] ❗❗Для слушателей есть специальные фабрики❗❗
> По этой фабрике, Spring понимает, какой тип контейнера использовать
> 
> Фабрика - KafkaListenerContainerFactory


# 1. RecordInterceptor - перехватчик сообщения
Начиная с версии 2.2.7, вы можете добавить `RecordInterceptor` в контейнер слушателя; 

Он будет **вызываться перед вызовом слушателя**, позволяя проверить или изменить запись. 
> Если перехватчик возвращает `null`, слушатель не вызывается. 

### Нововведения
Начиная с версии 2.7, у него есть **дополнительные методы**, которые **вызываются ПОСЛЕ выхода слушателя** (нормально или путем выброса исключения). 

Также, начиная с версии 2.7, теперь есть `BatchInterceptor`, предоставляющий аналогичную функциональность для пакетных слушателей (Batch Listeners). 

Кроме того, `ConsumerAwareRecordInterceptor` (и `BatchInterceptor`) предоставляют доступ к `Consumer<?, ?>`. Это может быть использовано, например, для доступа к метрикам потребителя в перехватчике.

## НЕ ТРОГАЙ топик, партицию и смещение
**Вы не должны выполнять любые методы, которые влияют на позиции потребителя и/или подтвержденные смещения** (committed offsets) в этих перехватчиках; 
**контейнер должен управлять такой информацией.** 

Если перехватчик изменяет запись (создавая новую), топик, раздел и смещение должны оставаться неизменными, чтобы избежать непредвиденных побочных эффектов, таких как потеря записей.  

> `CompositeRecordInterceptor` и `CompositeBatchInterceptor` 
> могут быть использованы для вызова нескольких перехватчиков.

## Транзакции и RecordInterceptor
По умолчанию, начиная с версии 2.8, при использовании транзакций, **перехватчик вызывается до начала транзакции**. 

Вы можете установить свойство `interceptBeforeTx` контейнера слушателя в `false`, *чтобы вызывать перехватчик после начала транзакции*. 

Начиная с версии 2.9, это будет применяться к любому менеджеру транзакций, а не только к `KafkaAwareTransactionManagers`. 
> Это позволяет, например, перехватчику участвовать в транзакции JDBC, запущенной контейнером.

## Нововведения ConcurrentMessageListenerContainer
Начиная с версий 2.3.8, 2.4.6, `ConcurrentMessageListenerContainer` теперь поддерживает статическое членство (Static Membership), когда параллелизм (concurrency) больше единицы. 

`group.instance.id` получает суффикс `-n`, где n начинается с 1. Это, вместе с увеличенным `session.timeout.ms`, может быть использовано **для сокращения событий ребалансировки**, например, когда экземпляры приложения перезапускаются.

# 2. KafkaMessageListenerContainer
Доступен следующий конструктор:
```java
public KafkaMessageListenerContainer(
	ConsumerFactory<K, V> consumerFactory,
    ContainerProperties containerProperties)
```

Он получает `ConsumerFactory` и информацию о топиках и разделах, а также другую конфигурацию, в объекте `ContainerProperties`. 

## 1. ContainerProperties
`ContainerProperties` имеет следующие конструкторы:
```java
public ContainerProperties(
	TopicPartitionOffset... topicPartitions)
	
public ContainerProperties(String... topics)
public ContainerProperties(Pattern topicPattern)
```

### Конструктор 1
Первый конструктор принимает **массив аргументов** `TopicPartitionOffset`, чтобы явно указать контейнеру, **какие разделы использовать** (используя метод `assign()` потребителя) и с необязательным начальным смещением. 
- Положительное значение по умолчанию является абсолютным смещением. 
- Отрицательное значение по умолчанию относительно текущего последнего смещения в разделе. 

Предоставлен конструктор для `TopicPartitionOffset`, который принимает дополнительный логический аргумент. Если он `true`, начальные смещения (положительные или отрицательные) относительны к текущей позиции этого потребителя. Смещения применяются при запуске контейнера.

### Конструктор 2
Второй принимает **массив топиков**, и *Kafka распределяет разделы на основе свойства* `group.id` — распределяя разделы по группе. 

### Конструктор 3
Третий использует regex `Pattern` для выбора топиков.

## 2. Назначение слушателя контейнеру
Чтобы назначить `MessageListener` контейнеру, вы можете использовать метод `ContainerProps.setMessageListener` при создании контейнера. 

Следующий пример показывает, как это сделать:
```java
ContainerProperties containerProps = 
	new ContainerProperties("topic1", "topic2");
	
containerProps.setMessageListener(
	new MessageListener<Integer, String>() {
    ...
});


DefaultKafkaConsumerFactory<Integer, String> cf =
	new DefaultKafkaConsumerFactory<>(consumerProps());


KafkaMessageListenerContainer<Integer, String> container =
	new KafkaMessageListenerContainer<>(cf, containerProps);

return container;
```

### Конфигурация фабрики Consumer
Обратите внимание, что при создании `DefaultKafkaConsumerFactory` **использование конструктора, который просто принимает свойства**, означает, что классы десериализаторов **ключа и значения подбираются из конфигурации**. 
- Альтернативно, экземпляры `Deserializer` могут быть переданы в конструктор `DefaultKafkaConsumerFactory` для ключа и/или значения, в этом случае все `Consumer` используют одни и те же экземпляры. 
	
- Другой вариант — предоставить `Supplier<Deserializer>` (начиная с версии 2.3), которые будут использоваться для получения отдельных экземпляров `Deserializer` для каждого `Consumer`:
```java
DefaultKafkaConsumerFactory<Integer, CustomValue> cf =
	new DefaultKafkaConsumerFactory<>(
		consumerProps(), 
		null, 
		() -> new CustomValueDeserializer());
		
KafkaMessageListenerContainer<Integer, String> container =
	new KafkaMessageListenerContainer<>(cf, containerProps);

return container;
```

> Обратитесь к Javadoc для `ContainerProperties` для получения дополнительной информации о различных свойствах, которые вы можете установить.

## Нововведения
### Уровень логирования
Начиная с версии 2.1.1, доступно новое свойство `logContainerConfig`. Когда оно `true` и **включено логирование уровня INFO**, каждый контейнер слушателя записывает сообщение журнала, суммирующее его свойства конфигурации.

По умолчанию, *логирование подтверждений смещений топиков выполняется на уровне логирования DEBUG*. 

Начиная с версии 2.1.2, свойство в `ContainerProperties` под названием `commitLogLevel` позволяет вам указать уровень журнала для этих сообщений. 
Например, чтобы изменить уровень журнала на INFO, вы можете использовать 
```java
containerProperties.setCommitLogLevel(
	LogIfLevelEnabled.Level.INFO);
```


### Поведение при отсутствии топиков
Начиная с версии 2.2, было добавлено новое свойство контейнера `missingTopicsFatal` (по умолчанию: `false` начиная с 2.3.4). 
Это **предотвращает запуск контейнера**, если какие-либо из настроенных **топиков отсутствуют в брокере**. 
> Это не применяется, если контейнер настроен на прослушивание шаблона топиков (regex). 

Ранее потоки контейнера зацикливались внутри метода `consumer.poll()`, ожидая появления топика, при этом записывая много сообщений в лог. Помимо логов, не было никаких указаний на проблему.

### Повторная попытка получения сообщения при AuthException...
Начиная с версии 2.8, введено новое свойство контейнера `authExceptionRetryInterval`. Это заставляет контейнер повторять попытку получения сообщений после получения любого `AuthenticationException` или `AuthorizationException` от `KafkaConsumer`. 

Это может произойти, когда, например, настроенному пользователю отказано в доступе на чтение определенного топика или учетные данные неверны. Определение `authExceptionRetryInterval` позволяет контейнеру восстановиться, когда предоставлены соответствующие разрешения.

> По умолчанию, интервал не настроен - ошибки аутентификации и авторизации считаются фатальными, что приводит к остановке контейнера.

### Поведение при добавлении десериализаторов
Начиная с версии 2.8, при создании фабрики потребителя, если вы предоставляете десериализаторы как объекты (в конструкторе или через сеттеры), фабрика будет вызывать метод `configure()`, чтобы настроить их с помощью свойств конфигурации.

## Примеры
### KafkaMessageListenerContainer — БЕЗ Acknowledgment
Используем `AckMode.BATCH` (или любое авто-подтверждение).
```java
@Bean
public KafkaMessageListenerContainer<String, String> singleContainer(
        ConsumerFactory<String, String> cf) {

    ContainerProperties props = new 
	    ContainerProperties("topic1");
    props.setGroupId("group1");
    props.setMessageListener(
	    (MessageListener<String, String>) record -> 
	{
        System.out.println("Получено: " + record.value());
    });

    props.setAckMode(ContainerProperties.AckMode.BATCH); // авто-commit

    return new KafkaMessageListenerContainer<>(cf, props);
}
```

### KafkaMessageListenerContainer — С MANUAL Acknowledgment
```java
@Bean
public KafkaMessageListenerContainer<String, String> singleContainerManual(
        ConsumerFactory<String, String> cf) {

    ContainerProperties props = new 
	    ContainerProperties("topic1");
    props.setGroupId("group2");
    props.setMessageListener(
	    (AcknowledgingMessageListener<String, String>) (record, ack) -> {
        System.out.println("Обработка: " + record.value());
        ack.acknowledge(); // ручное подтверждение
    });

    props.setAckMode(ContainerProperties.AckMode.MANUAL);

    return new KafkaMessageListenerContainer<>(cf, props);
}
```

#### Каст к определенному слушателю
Мы делаем каст, потому что:
- `setMessageListener()` перегружен
    
- лямбда подходит под несколько функциональных интерфейсов
    
- Java не знает, какой выбрать
    
- приведение к `MessageListener` делает выбор однозначным

# 3.ConcurrentMessageListenerContainer

Единственный конструктор похож на конструктор `KafkaMessageListenerContainer`. 

Следующий листинг показывает сигнатуру конструктора:
```java
public ConcurrentMessageListenerContainer(
	ConsumerFactory<K, V> consumerFactory,
    ContainerProperties containerProperties)
```

Он также имеет свойство `concurrency`. 
> Например, `container.setConcurrency(3)` **создает три экземпляра** `KafkaMessageListenerContainer`.

Если свойства контейнера настроены для топиков (или шаблона топиков), Kafka распределяет разделы между потребителями, используя свои возможности управления группами.

## Распределение партиций
При прослушивании нескольких топиков, **распределение разделов по умолчанию может быть не таким**, как вы ожидаете. 

Например, если у вас есть три топика с пятью разделами каждый, и вы хотите использовать `concurrency=15`, вы увидите только пять активных потребителей, каждому назначен один раздел из каждого топика, а остальные 10 потребителей будут простаивать. 
(см [[5. - Репликация партиций, балансировка и надежность]])

Это потому, что `ConsumerPartitionAssignor` по умолчанию в Kafka — это `RangeAssignor` (см. его Javadoc). 

Для этого сценария вы можете рассмотреть возможность использования `RoundRobinAssignor` вместо этого, который распределяет разделы между всеми потребителями. 
Тогда каждому потребителю назначается один топик или раздел. 

### Изменение стратегии распределения в свойствах
Чтобы изменить `ConsumerPartitionAssignor`, вы можете установить свойство потребителя 
```
partition.assignment.strategy 
или
ConsumerConfig.PARTITION_ASSIGNMENT_STRATEGY_CONFIG
``` 
в свойствах, предоставленных `DefaultKafkaConsumerFactory`.

### Изменение стратегии распределения в Spring Boot
При использовании Spring Boot, вы можете установить стратегию следующим образом:
```properties
spring.kafka.consumer.properties.partition.assignment.strategy=
	\org.apache.kafka.clients.consumer.RoundRobinAssignor
```

Когда свойства контейнера настроены с `TopicPartitionOffsets`, **ConcurrentMessageListenerContainer распределяет экземпляры** `TopicPartitionOffset` по делегированным экземплярам `KafkaMessageListenerContainer`.

- Если, скажем, предоставлено шесть экземпляров `TopicPartitionOffset` и параллелизм равен 3; каждый контейнер получает два раздела. 
- Для пяти экземпляров `TopicPartitionOffset` два контейнера получают по два раздела, а третий получает один. 
- Если параллелизм больше количества `TopicPartitions`, параллелизм уменьшается так, что каждый контейнер получает один раздел.

Свойство `client.id` (если установлено) дополняется `-n`, где n — это экземпляр потребителя, соответствующий параллелизму. Это необходимо для предоставления уникальных имен для MBean, когда включен JMX.

## Метрики в MessageListenerContainer
Начиная с версии 1.3, `MessageListenerContainer` предоставляет доступ к метрикам базового `KafkaConsumer`. 

В случае `ConcurrentMessageListenerContainer`, метод `metrics()`
возвращает метрики для всех целевых экземпляров `KafkaMessageListenerContainer`. 

Метрики сгруппированы в `Map<MetricName, ? extends Metric>` по `client-id`, предоставленному для базового `KafkaConsumer`.

## Сон между вызовами poll()
Начиная с версии 2.3, `ContainerProperties` предоставляет опцию `idleBetweenPolls`, чтобы позволить основному циклу в контейнере слушателя спать между вызовами `KafkaConsumer.poll()`. 

Фактуальный интервал сна выбирается как минимум из предоставленной опции и разницы между `max.poll.interval.ms` конфигурации потребителя и временем обработки текущей пачки записей.

## Примеры
### ConcurrentMessageListenerContainer — БЕЗ Acknowledgment
```java
@Bean
public ConcurrentMessageListenerContainer<String, String> concurrentContainer(
        ConsumerFactory<String, String> cf) {

    ContainerProperties props = new 
	    ContainerProperties("topic2");
    props.setGroupId("group3");
    props.setMessageListener((MessageListener<String, String>) record -> {
        System.out.println(Thread.currentThread().getName()
                + " обработал: " + record.value());
    });

    props.setAckMode(ContainerProperties.AckMode.BATCH);

    ConcurrentMessageListenerContainer<String, String> 
    container = new ConcurrentMessageListenerContainer<>(
	    cf, props);
    container.setConcurrency(5);  // 5 consumer потоков

    return container;
}
```

# 4. Подтверждение смещений (Committing Offsets)

Предоставлено несколько вариантов для подтверждения смещений. 
- Если свойство потребителя `enable.auto.commit` равно `true`, Kafka автоматически подтверждает смещения в соответствии со своей конфигурацией. 
- Если оно `false`, **контейнеры поддерживают несколько настроек** `AckMode` (описанных в следующем списке). AckMode *по умолчанию* — `BATCH`. 

Начиная с версии 2.3, фреймворк устанавливает `enable.auto.commit` в `false`, **если оно явно не установлено в конфигурации.** 
Ранее использовалось значение по умолчанию Kafka (`true`), если свойство не было установлено.

## Действие контейнера для каждого варианта коммита
Метод `poll()` потребителя возвращает один или несколько `ConsumerRecords`. `MessageListener` вызывается для каждой записи. 

Следующий список описывает действие, выполняемое контейнером для каждого `AckMode` (когда транзакции не используются):
- **RECORD**: Подтвердить смещение, *когда слушатель возвращается после обработки записи*.
    
- **BATCH**: Подтвердить смещение, *когда все записи*, возвращенные `poll()`, *были обработаны*.
    
- **TIME**: Подтвердить смещение, когда все записи, возвращенные `poll()`, были обработаны, при условии, что `ackTime` с последнего подтверждения был превышен.
    
- **COUNT**: Подтвердить смещение, когда все записи, возвращенные `poll()`, были обработаны, при условии, что `ackCount` записей было получено с последнего подтверждения.
    
- **COUNT_TIME**: Подобно TIME и COUNT, но подтверждение выполняется, если выполняется любое из условий.
    
- **MANUAL**: *Ответственность за вызов* `acknowledge()` у `Acknowledgment` *лежит на слушателе сообщений*. После этого применяется та же семантика, что и для BATCH.
    
- **MANUAL_IMMEDIATE**: Немедленно подтвердить смещение, когда метод `Acknowledgment.acknowledge()` вызывается слушателем.


> При использовании транзакций, смещение(я) отправляются в транзакцию, и семантика эквивалентна RECORD или BATCH, в зависимости от типа слушателя (record или batch).

MANUAL и MANUAL_IMMEDIATE требуют, чтобы слушатель был `AcknowledgingMessageListener` или `BatchAcknowledgingMessageListener`. 
(см [[1. - Слушатели сообщений (Message Listeners)]])

## Свойство syncCommits и callback
В зависимости от свойства контейнера **syncCommits**, используется метод `commitSync()` или `commitAsync()` потребителя. 

`syncCommits` по умолчанию `true`; 
(также см. `setSyncCommitTimeout`. См. `setCommitCallback`, чтобы получать результаты асинхронных подтверждений)

**callback** по умолчанию — `LoggingCommitCallback`, который логирует ошибки (и успехи на уровне debug).

## Контейнер имеет собственный механизм подтверждения
Поскольку контейнер слушателя имеет свой собственный механизм для подтверждения смещений, **он предпочитает, чтобы свойство Kafka** `ConsumerConfig.ENABLE_AUTO_COMMIT_CONFIG` было `false`. 

Начиная **с версии 2.3, он безусловно устанавливает его в** `false`, *если оно специально не установлено в фабрике* потребителя или переопределениях свойств потребителя контейнера.

## Интерфейс Acknowledgment
Интерфейс **позволяет вручную подтверждать смещения**

Имеет следующий метод:
```java
public interface Acknowledgment {
    void acknowledge();
}
```

Этот метод дает слушателю контроль над тем, когда смещения подтверждаются.

### Новые методы
Начиная с версии 2.3, интерфейс `Acknowledgment` имеет два дополнительных метода 
- `nack(long sleep)` 
- `nack(int index, long sleep)`. 

#### Для слушателя _записей_:
- подтверждает уже обработанные смещения;
    
- сбрасывает оставшиеся записи текущего `poll`;
    
- делает `seek` на проблемную запись → её повторно доставят;
    
- приостанавливает consumer на `sleep` (кратно `pollTimeout`).

#### Для слушателя _пачек_:
- подтверждает записи **до указанного индекса**;
    
- оставшиеся — перерабатываются при следующем `poll`.

> `nack()` работает **только в потоке consumer**,  
> не поддерживается с Out-of-Order commits.

### Использование с транзакциями
Требуется:
```java
AckMode.MANUAL
```
`nack()` отправляет смещения успешных записей в транзакцию.

### Приостановка
`sleep` — пауза для всех назначенных партиций.  
Минимальная пауза = `pollTimeout` (по умолчанию 5 сек).

### Частичное подтверждение пачки (с версии 3.0.10)
Можно подтверждать часть batch:
```java
acknowledgment.acknowledge(index)
```
Требования:
- `AckMode.MANUAL_IMMEDIATE`
- метод вызывается в consumer thread
- слушатель получает `List`
- индекс по порядку, не меньше предыдущего
- неверные условия → исключение

# 5. Автозапуск контейнера слушателя (Listener Container Auto Startup)

Контейнеры слушателя реализуют `SmartLifecycle`, и `autoStartup` по умолчанию `true`. 

Контейнеры запускаются на поздней фазе (`Integer.MAX_VALUE - 100`). 

Другие компоненты, которые реализуют `SmartLifecycle`, для обработки данных от слушателей, должны запускаться на более ранней фазе. `- 100` оставляет место для более поздних фаз, чтобы позволить компонентам автоматически запускаться после контейнеров.
