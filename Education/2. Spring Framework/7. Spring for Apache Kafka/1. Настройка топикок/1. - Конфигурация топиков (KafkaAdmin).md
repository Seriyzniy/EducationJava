# 1. Бин KafkaAdmin

Если вы определите бин `KafkaAdmin` в контексте своего приложения, он может автоматически добавлять топики в брокер. 
Чтобы это сделать, вы можете добавить бин `@Bean` типа `NewTopic` для каждого топика в контекст приложения. 

## Использование TopicBuilder для создания топиков
В версии 2.3 был представлен новый класс `TopicBuilder`, чтобы сделать создание таких бинов более удобным. 

Следующий пример показывает, как это сделать:
```java
@Bean
public KafkaAdmin admin() {
    Map<String, Object> configs = new HashMap<>();
    configs.put(
	    AdminClientConfig.BOOTSTRAP_SERVERS_CONFIG, 
	    "localhost:9092"
	);
    return new KafkaAdmin(configs);
}

@Bean
public NewTopic topic1() {
    return TopicBuilder.name("thing1")
            .partitions(10)
            .replicas(3)
            .compact()
            .build();
}

@Bean
public NewTopic topic2() {
    return TopicBuilder.name("thing2")
            .partitions(10)
            .replicas(3)
            .config(TopicConfig.COMPRESSION_TYPE_CONFIG, "zstd")
            .build();
}

@Bean
public NewTopic topic3() {
    return TopicBuilder.name("thing3")
            .assignReplicas(0, List.of(0, 1))
            .assignReplicas(1, List.of(1, 2))
            .assignReplicas(2, List.of(2, 0))
            .config(TopicConfig.COMPRESSION_TYPE_CONFIG, "zstd")
            .build();
}
```

### Значение по умолчанию
Начиная с версии 2.6, вы можете опускать `partitions()` и/или `replicas()`, и тогда значения по умолчанию от брокера будут применены к этим свойствам. 
Версия брокера должна быть как минимум 2.4.0 для поддержки этой функции - смотрите KIP-464.

```java
@Bean
public NewTopic topic4() {
    return TopicBuilder.name("defaultBoth")
            .build();
}

@Bean
public NewTopic topic5() {
    return TopicBuilder.name("defaultPart")
            .replicas(1)
            .build();
}

@Bean
public NewTopic topic6() {
    return TopicBuilder.name("defaultRepl")
            .partitions(3)
            .build();
}
```

### Несколько топиков в одном бине NewTopic
Начиная с версии 2.7, вы можете объявлять несколько `NewTopics` в одном определении бина `KafkaAdmin.NewTopics`:
```java
@Bean
public KafkaAdmin.NewTopics topics456() {
    return new NewTopics(
            TopicBuilder.name("defaultBoth")
                .build(),
            TopicBuilder.name("defaultPart")
                .replicas(1)
                .build(),
            TopicBuilder.name("defaultRepl")
                .partitions(3)
                .build());
}
```

## Поведение при ошибках
> [!NOTE] ❗ Важно ❗
> **При использовании Spring Boot** бин `KafkaAdmin` регистрируется автоматически, поэтому вам нужны только бины `@Bean` типа `NewTopic` (и/или `NewTopics`).

По умолчанию, если брокер недоступен, сообщение записывается в лог, но контекст продолжает загружаться. 

> Вы можете программно вызвать метод `initialize()` у администратора позже, чтобы попробовать снова. Если вы хотите, чтобы это условие считалось фатальным, установите свойство `fatalIfBrokerNotAvailable` администратора в `true`. Тогда контекст не сможет инициализироваться (загрузиться с ошибкой).
> 
> Если брокер это поддерживает (версия 1.0.0 или выше), администратор увеличит количество разделов, если будет обнаружено, что существующий топик имеет меньше разделов, чем указано в `NewTopic.numPartitions`.


## Методы проверки топиков и AdminClient напрямую
Начиная с версии 2.7, `KafkaAdmin` предоставляет **методы для создания и проверки топиков** во время выполнения (runtime):
- `createOrModifyTopics`
- `describeTopics`


Для более продвинутых функций вы можете использовать `AdminClient` напрямую
(см [[1. - Интерфейс Admin]] и [[1.2 -- Класс KafkaAdminClient]]) 

Следующий пример показывает, как это сделать:
```java
@Autowired
private KafkaAdmin admin;

...

    AdminClient client = 
	    AdminClient.create(admin.getConfigurationProperties());
    ...
    client.close();
```

## Предикат при создании топиков
Начиная с версий 2.9.10, 3.0.9, вы можете предоставить `Predicate<NewTopic>` (предикат), который может использоваться для определения, следует ли рассматривать конкретный бин `NewTopic` для создания или модификации.

> Это полезно, например, если у вас есть несколько экземпляров `KafkaAdmin`, указывающих на разные кластеры, и вы хотите выбрать, какие топики должны создаваться или изменяться каждым администратором.

```java
admin.setAutoCreateOrModifyTopic(nt -> 
	!nt.name().equals("dontCreateThisOne"));
```
