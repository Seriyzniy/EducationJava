Как видно в разделе "Using KafkaTemplate", `ProducerFactory` используется для создания продюсера.

## Без транзакции - синглтон-продюсер
**Когда транзакции не используются**, по умолчанию `DefaultKafkaProducerFactory` **создает синглтон-продюсер**, используемый всеми клиентами, как рекомендуется в JavaDocs для `KafkaProducer`. 

💡Однако, если вы вызываете `flush()` в шаблоне, *это может вызвать задержки для других потоков*, использующих того же продюсера. 

### Отдельный продюсер для каждого потока
Начиная с версии 2.3, `DefaultKafkaProducerFactory` имеет **новое свойство** `producerPerThread`. 

Когда оно установлено в `true`, *фабрика будет создавать (и кэшировать) отдельный продюсер для каждого потока*, чтобы избежать этой проблемы.

> ==Закрытие продюсера через фабрику==
> Когда `producerPerThread` равно `true`, пользовательский код должен вызывать `closeThreadBoundProducer()` на фабрике, когда продюсер больше не нужен. 

Это физически закроет продюсер и удалит его из `ThreadLocal`. Вызов `reset()` или `destroy()` не очистит этих продюсеров.  
Также смотрите "KafkaTemplate Transactional and non-Transactional Publishing".
(см [[KafkaTemplate Transactional and non-Transactional Publishing]])

# 1. DefaultKafkaProducerFactory
## Конфигурация через конструктор 2-мя способами
При создании `DefaultKafkaProducerFactory`, классы сериализаторов ключа и/или значения могут быть
- 🔶подхвачены из конфигурации путем вызова конструктора, который принимает только **Map свойств** 
	(см [[2.2 -- RoutingKafkaTempalte]])
	
- 🔶либо экземпляры `Serializer` могут быть переданы в конструктор `DefaultKafkaProducerFactory` (в этом случае все `Producer` используют одни и те же экземпляры). 

Альтернативно, вы можете предоставить `Supplier<Serializer>` (начиная с версии 2.3), которые будут использоваться для получения отдельных экземпляров `Serializer` для каждого `Producer`:
```java
@Bean
public ProducerFactory<Integer, CustomValue> producerFactory() {
    return new DefaultKafkaProducerFactory<>(
	    producerConfigs(), 
	    null, 
	    () -> new CustomValueSerializer());
}

@Bean
public KafkaTemplate<Integer, CustomValue> kafkaTemplate() {
    return new KafkaTemplate<Integer, CustomValue>(producerFactory());
}
```

## Обновление конфигураций
Начиная с версии 2.5.10, вы теперь **можете обновлять свойства** продюсера **после создания фабрики**. 

Это может быть полезно, например, если вам нужно обновить местоположения хранилищ SSL ключей/доверенных сертификатов после смены учетных данных. 

Изменения не повлияют на существующие экземпляры продюсеров; 
> вызовите `reset()`, чтобы закрыть любые существующие продюсеры, чтобы новые продюсеры создавались с использованием новых свойств.


> [!NOTE] ❗ Настройки ТРАНЗАКЦИИ менять нельзя  ❗
> Вы не можете изменить транзакционную фабрику продюсеров на нетранзакционную и наоборот.  

Теперь предоставлены два новых метода:
```java
void updateConfigs(Map<String, Object> updates);
void removeConfig(String configKey);
```

### Обновление через set/get методы
Начиная с версии 2.8, если вы предоставляете сериализаторы как объекты (в конструкторе или через сеттеры), фабрика будет вызывать метод `configure()`, чтобы настроить их с помощью свойств конфигурации.
