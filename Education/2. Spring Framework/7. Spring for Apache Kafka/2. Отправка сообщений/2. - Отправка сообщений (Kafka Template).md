Этот раздел рассказывает о том, как использовать `KafkaTemplate` для отправки сообщений.

---
# 1. Обзор
`KafkaTemplate` оборачивает продюсера и предоставляет удобные методы для отправки данных в топики Kafka. 

Следующий листинг показывает соответствующие методы из `KafkaTemplate`:
```java
CompletableFuture<SendResult<K, V>> sendDefault(V data);

CompletableFuture<SendResult<K, V>> sendDefault(K key, V data);

CompletableFuture<SendResult<K, V>> sendDefault(
	Integer partition, 
	K key, 
	V data);

CompletableFuture<SendResult<K, V>> sendDefault(
	Integer partition, 
	Long timestamp, 
	K key, 
	V data);


CompletableFuture<SendResult<K, V>> send(String topic, V data);

CompletableFuture<SendResult<K, V>> send(String topic, K key, V data);

CompletableFuture<SendResult<K, V>> send(
	String topic, 
	Integer partition, 
	K key, 
	V data);

CompletableFuture<SendResult<K, V>> send(
	String topic, 
	Integer partition, 
	Long timestamp, 
	K key, 
	V data);

CompletableFuture<SendResult<K, V>> send(ProducerRecord<K, V> record);

CompletableFuture<SendResult<K, V>> send(Message<?> message);

interface ProducerCallback<K, V, T> {
    T doInKafka(Producer<K, V> producer);
}

interface OperationsCallback<K, V, T> {
    T doInOperations(KafkaOperations<K, V> operations);
}
```
Смотрите Javadoc для более подробной информации.

## Метод sendDefault()
API `sendDefault` требует, чтобы топик по умолчанию был предоставлен шаблону (KafkaTemplate судя по всему).

API принимает временную метку (timestamp) в качестве параметра и сохраняет эту метку в записи. 
То, как предоставленная пользователем временная метка сохраняется, зависит от типа временной метки, настроенного в топике Kafka. 
- Если топик настроен на использование `CREATE_TIME`, записывается указанная пользователем метка (или генерируется, если не указана). 
- Если топик настроен на использование `LOG_APPEND_TIME`, указанная пользователем метка игнорируется, и брокер добавляет локальное время брокера.

## Методы metrics и partitionsFor
Методы `metrics` и `partitionsFor` делегируют вызов одноименным методам базового `Producer`. 

## Метод execute
Метод `execute` предоставляет прямой доступ к базовому `Producer`.

# 2. Настройка KafkaTemplate через фабрику
Чтобы использовать шаблон, вы можете настроить фабрику продюсеров и предоставить ее в конструкторе шаблона. 

> [!NOTE] Порядок создания
> - Создание фабрики ProducerFactory с настройками
> - Описание настроек фабрики
> - Передача фабрики в KafkaTemplate

Следующий пример показывает, как это сделать:
```java
// 1️⃣ Создание фабрики с настройками
@Bean
public ProducerFactory<Integer, String> producerFactory() {
    return new DefaultKafkaProducerFactory<>(producerConfigs());
}

//2️⃣ Настройки для фабрики
@Bean
public Map<String, Object> producerConfigs() {
    Map<String, Object> props = new HashMap<>();
    props.put(
	    ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, 
	    "localhost:9092");
    props.put(
	    ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, 
	    IntegerSerializer.class);
    props.put(
	    ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, 
	    StringSerializer.class);
    // See https://kafka.apache.org/documentation/#producerconfigs for more properties
    return props;
}

//3️⃣ Создание KafkaTemplate согласно фабрики
@Bean
public KafkaTemplate<Integer, String> kafkaTemplate() {
    return new KafkaTemplate<Integer, String>(producerFactory());
}
```

## Настройка по фабрике + доп параметры
Начиная с версии 2.5, вы теперь можете переопределять свойства `ProducerConfig` фабрики, чтобы создавать шаблоны с разными конфигурациями продюсера из одной и той же фабрики.
```java
@Bean
public KafkaTemplate<String, String> stringTemplate(
	ProducerFactory<String, String> pf
){
    return new KafkaTemplate<>(pf);
}

@Bean
public KafkaTemplate<String, byte[]> bytesTemplate(
	ProducerFactory<String, byte[]> pf
) {
    return new KafkaTemplate<>(
	    pf,
        Collections.singletonMap(
	        ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, 
	        ByteArraySerializer.class
	    )
	);
}
```

Обратите внимание, что на бин типа `ProducerFactory<?, ?>` (такой как автоматически настраиваемый Spring Boot) можно ссылаться с различными суженными (narrowed) generic-типами.

Вы также можете настроить шаблон, используя стандартные определения `<bean/>`.

## Использование KafkaTemplate
Затем, чтобы использовать шаблон, вы можете вызвать один из его методов.

Когда вы используете методы с параметром `Message<?>`, **информация о топике, разделе, ключе** и временной метке **предоставляется В ЗАГОЛОВКЕ СООБЩЕНИЯ**, который включает следующие элементы:
- `KafkaHeaders.TOPIC`
- `KafkaHeaders.PARTITION`
- `KafkaHeaders.KEY`
- `KafkaHeaders.TIMESTAMP`

Полезная нагрузка сообщения (payload) — это данные.

## ProducerListener 
При желании вы можете настроить `KafkaTemplate` с `ProducerListener`, чтобы получать асинхронный callback с результатами отправки (успех или неудача) вместо ожидания завершения `Future`. 

Следующий листинг показывает определение интерфейса `ProducerListener`:
```java
public interface ProducerListener<K, V> {
    default void onSuccess(
	    ProducerRecord<K, V> producerRecord, 
	    RecordMetadata recordMetadata) 
	{ }

    default void onError(
	    ProducerRecord<K, V> producerRecord, 
	    RecordMetadata recordMetadata, 
	    Exception exception) 
	{ }
}
```

💡По умолчанию шаблон настроен с `LoggingProducerListener`, который логирует ошибки и ничего не делает при успешной отправке.

> Для удобства предоставлены реализации методов по умолчанию, на случай если вы захотите реализовать только один из методов.

## Метод send() в KafkaTemplate
Обратите внимание, что методы `send` возвращают `CompletableFuture<SendResult>`. 
> Вы можете зарегистрировать callback с помощью слушателя, чтобы асинхронно получать результат отправки. 

Следующий пример показывает, как это сделать:
```java
CompletableFuture<SendResult<Integer, String>> future = 
	template.send("myTopic", "something");

future.whenComplete((result, ex) -> {
    ...
});
```

`SendResult` имеет два свойства: `ProducerRecord` и `RecordMetadata`. 
Смотрите документацию по Kafka API для получения информации об этих объектах. 
(Дока по Spring Kafka - [Overview (Spring for Apache Kafka 3.3.10 API)](https://docs.spring.io/spring-kafka/docs/3.3.10/api/)) 

`Throwable` (исключение) может быть приведено к `KafkaProducerException`; его свойство `producerRecord` содержит неудавшуюся запись.

### Блокировка потока для ожидания ответа
Если вы хотите заблокировать отправляющий поток для ожидания результата, вы можете вызвать метод `get()` у future; Рекомендуется использовать метод с таймаутом. 

Если вы установили `linger.ms`, вы можете вызвать `flush()` перед ожиданием или, для удобства, шаблон имеет конструктор с параметром `autoFlush`, который заставляет шаблон вызывать `flush()` при каждой отправке. 
Сброс (Flushing) необходим только если вы установили свойство продюсера `linger.ms` и хотите немедленно отправить частичный батч.

# 3.Примеры отправки сообщений
Этот раздел показывает примеры отправки сообщений в Kafka:

## 1. Non Blocking (Async) 
```java
public void sendToKafka(final MyOutputData data) {
	//1️⃣ Создание данных (сообщения)
    final ProducerRecord<String, String> record = 
	    createRecord(data);
	
	//2️⃣ Отправка
    CompletableFuture<SendResult<String, String>> future = 
	    template.send(record);
	
	//3️⃣ Обработка результата отпарвки (ACK или Exception)   
    future.whenComplete((result, ex) -> {
        if (ex == null) {
            handleSuccess(data);
        }
        else {
            handleFailure(data, record, ex);
        }
    });
}
```

## 2. Blocking (Sync)

```java
public void sendToKafka(final MyOutputData data) {
    //1️⃣ Создание данных (сообщения)
    final ProducerRecord<String, String> record =
	    createRecord(data);
	
	//2️⃣ Отправка с блокировкой потока (при помощи get())
    try {
        template.send(record).get(10, TimeUnit.SECONDS);
        handleSuccess(data);
    }
    catch (ExecutionException e) {
        handleFailure(data, record, e.getCause());
    }
    catch (TimeoutException | InterruptedException e) {
        handleFailure(data, record, e);
    }
}
```
> Обратите внимание, что причина `ExecutionException` — это `KafkaProducerException` со свойством `producerRecord`.
