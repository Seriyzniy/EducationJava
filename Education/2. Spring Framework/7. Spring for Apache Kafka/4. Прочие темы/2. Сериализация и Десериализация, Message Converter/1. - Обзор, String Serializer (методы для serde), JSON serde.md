Apache Kafka предоставляет высокоуровневый API для сериализации и десериализации значений записей, а также их ключей. Он представлен абстракциями `org.apache.kafka.common.serialization.Serializer<T>` и `org.apache.kafka.common.serialization.Deserializer<T>` с некоторыми встроенными реализациями. 

В то же время мы можем указать классы сериализатора и десериализатора, используя свойства конфигурации Producer или Consumer. 

Следующий пример показывает, как это сделать:
```java
props.put(
	ConsumerConfig.KEY_DESERIALIZER_CLASS_CONFIG, 
	IntegerDeserializer.class);
props.put(
	ConsumerConfig.VALUE_DESERIALIZER_CLASS_CONFIG, 
	StringDeserializer.class);
...
props.put(
	ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, 
	IntegerSerializer.class);
props.put(
	ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, 
	StringSerializer.class);
```

**Для сложных случаев:**
- `KafkaConsumer` и `KafkaProducer` принимают готовые экземпляры сериализаторов
    
- Spring Kafka фабрики (`DefaultKafkaProducerFactory`, `DefaultKafkaConsumerFactory`) поддерживают:
    - Прямую передачу экземпляров `Serializer`/`Deserializer`
        
    - Использование `Supplier<Serializer>`/`Supplier<Deserializer>`
        
    - Suppliers вызываются при создании каждого Producer/Consumer
        

**Ключевое преимущество:** Гибкость в выборе стратегии сериализации для разных сценариев использования.

# 1. String serialization
Начиная с версии 2.5, Spring for Apache Kafka предоставляет классы
- `ToStringSerializer` - сериализует объекты в String через метод `toString()`
    
- `ParseStringDeserializer` - десериализует String обратно в объекты

Они полагаются на методы `toString` и некоторую `Function<String>` или `BiFunction<String, Headers>` для разбора строки и заполнения свойств экземпляра. 

Обычно это вызывает некоторый статический метод в классе, такой как `parse`:
```java
ToStringSerializer<Thing> thingSerializer = 
	new ToStringSerializer<>();
//...
ParseStringDeserializer<Thing> deserializer = 
	new ParseStringDeserializer<>(Thing::parse);
```

## Особенности:
- По умолчанию передает информацию о типе в заголовках Kafka
    
- Можно отключить: `ToStringSerializer.ADD_TYPE_INFO_HEADERS = false`
	Эта информация может быть использована `ParseStringDeserializer` на принимающей стороне.
    
- Поддерживает кастомные парсеры через `BiFunction<String, Headers>`

```java
ParseStringDeserializer<Object> deserializer = 
new ParseStringDeserializer<>((str, headers) -> {
    byte[] header = headers
	    .lastHeader(ToStringSerializer.VALUE_TYPE).value();
    String entityType = new String(header);

    if (entityType.contains("Thing")) {
        return Thing.parse(str);
    }
    else {
        // ...логика разбора
    }
});
```

Вы можете настроить `Charset`, используемый для преобразования `String` в/из `byte[]`, по умолчанию используется `UTF-8`.

## Метод для десериализации
Вы можете настроить десериализатор с именем метода парсера, используя свойства `ConsumerConfig`:
- `ParseStringDeserializer.KEY_PARSER`
    
- `ParseStringDeserializer.VALUE_PARSER`

Свойства должны содержать полностью квалифицированное имя класса, за которым следует имя метода, разделенные точкой `.`. Метод должен быть статическим и иметь сигнатуру либо `(String, Headers)`, либо `(String)`.

```java
@Bean
public ConsumerFactory<String, Object> consumerFactory() {
    Map<String, Object> props = new HashMap<>();
    props.put(
	    ConsumerConfig.BOOTSTRAP_SERVERS_CONFIG, 
	    "localhost:9092");
    
    // Указываем метод парсера для значения
    props.put(
	    ParseStringDeserializer.VALUE_PARSER, 
	    "com.example.Thing.parse");
    
    // Указываем метод парсера для ключа
    props.put(
	    ParseStringDeserializer.KEY_PARSER, 
	    "com.example.Key.parseWithHeaders");
    
    props.put(
	    ConsumerConfig.VALUE_DESERIALIZER_CLASS_CONFIG, 
	    ParseStringDeserializer.class);
    props.put(
	    ConsumerConfig.KEY_DESERIALIZER_CLASS_CONFIG, 
	    ParseStringDeserializer.class);
    
    return new DefaultKafkaConsumerFactory<>(props);
}
```


Также предоставляется `ToFromStringSerde` для использования с Kafka Streams.

# 2. JSON
## JacksonJsonSerializer и JacksonJsonDeserializer
Spring for Apache Kafka также предоставляет реализации `JacksonJsonSerializer` и `JacksonJsonDeserializer`, **основанные на Jackson JSON** object mapper.

`JacksonJsonSerializer` позволяет записывать любой Java-объект как JSON `byte[]`. 

`JacksonJsonDeserializer` **требует дополнительный аргумент** `Class<?> targetType`, чтобы позволить десериализацию потребленного `byte[]` в правильный целевой объект. 
(Из байты в нужный объект ПРИ ДЕСЕРИАЛИЗАЦИИ)

Следующий пример показывает, как создать `JacksonJsonDeserializer`:
```java
JacksonJsonDeserializer<Thing> thingDeserializer = 
	new JacksonJsonDeserializer<>(Thing.class);
```

Вы можете настроить как `JacksonJsonSerializer`, так и `JacksonJsonDeserializer` с помощью `ObjectMapper`. 
Вы также можете расширить их, чтобы реализовать некоторую особую логику конфигурации в методе `configure(Map<String, ?> configs, boolean isKey)`.

### Особенности версии 2.3+:
Начиная с версии **Spring Kafka 2.3**, все JSON-компоненты (например, `JsonSerializer`, `JsonDeserializer`, `JsonSerde`) используют **особый ObjectMapper из Spring Framework**, а не стандартный Jackson - ==JacksonUtils.enhancedObjectMapper()==

Этот объект настроен так:
#### 🔹 Отключён `MapperFeature.DEFAULT_VIEW_INCLUSION`
По умолчанию Jackson включает поля всех `JsonView`.  
Теперь — **нет**. Если используется `@JsonView`, сериализуются **только поля конкретного View**.

#### 🔹 Отключён `DeserializationFeature.FAIL_ON_UNKNOWN_PROPERTIES`
Если в JSON есть лишние поля, которых нет в вашем классе — **ошибки не будет**.  
Это удобно при интеграциях, когда структура сообщений может меняться.

#### 🔹 Модули для Java Time и Kotlin
Он автоматически включает модули:
- `JavaTimeModule` — поддержка `LocalDate`, `Instant`, `OffsetDateTime` и т.д.
    
- `KotlinModule` — корректная работа Kotlin-классов.
    

То есть: **не нужно вручную регистрировать эти модули**.

### JacksonMimeTypeModule
Этот метод также регистрирует `org.springframework.kafka.support.JacksonMimeTypeModule` для сериализации объектов `org.springframework.util.MimeType` в обычную строку для межплатформенной совместимости по сети. 

`JacksonMimeTypeModule` **может быть зарегистрирован как бин** в контексте приложения, и он будет автоматически настроен в экземпляре `ObjectMapper` Spring Boot.
```java
@Bean
public JacksonMimeTypeModule jacksonMimeTypeModule() {
    return new JacksonMimeTypeModule();
}
```

### JsonDeserializer теперь принимает TypeReference
Также начиная с версии 2.3, `JsonDeserializer` предоставляет конструкторы на основе `TypeReference` для лучшей обработки целевых generic-типов контейнеров.

```java
new JsonDeserializer<>(
	new TypeReference< List<MyDto> >() {});
```

Раньше так сделать было сложно, приходилось писать костыли для generic-типов.

Теперь deserializer понимает типы коллекций лучше.

### Передача информации о типе в заголовках Kafka (type headers)
Начиная с 2.1, Serializer и Deserializer могут использовать **заголовки Kafka**:
```java
__TypeId__
__KeyTypeId__
__ContentTypeId__
```

Например, если ты отправляешь разные типы в один топик:
- `OrderCreatedEvent`
- `OrderUpdatedEvent`
- `OrderCancelledEvent`

то KafkaMessageConverter (или JsonDeserializer) ==из заголовков понимает==, **какой конкретно класс нужно создать** при десериализации.

Без этого тебе пришлось бы использовать:
- разные топики, или
- switch-case по полям JSON.

Это делает JSON-обработку **полиморфной и гибкой**.


Кроме того, вы можете настроить сериализатор и десериализатор, используя следующие свойства Kafka. 
Они не имеют эффекта, если вы предоставили экземпляры `Serializer` и `Deserializer` для `KafkaConsumer` и `KafkaProducer` соответственно.

### Свойства конфигурации
#### Сериализатор:
- **`JacksonJsonSerializer.ADD_TYPE_INFO_HEADERS`** (по умолчанию `true`): Вы можете установить его в `false`, чтобы отключить эту функцию в `JacksonJsonSerializer` (устанавливает свойство `addTypeInfo`).
    
- **`JacksonJsonSerializer.TYPE_MAPPINGS`** (по умолчанию пусто): См. "Mapping Types".

#### Десериализатор:
- **`JacksonJsonDeserializer.USE_TYPE_INFO_HEADERS`** (по умолчанию `true`): Вы можете установить его в `false`, чтобы игнорировать заголовки, установленные сериализатором.
    
- **`JacksonJsonDeserializer.REMOVE_TYPE_INFO_HEADERS`** (по умолчанию `true`): Вы можете установить его в `false`, чтобы сохранить заголовки, установленные сериализатором.
    
- **`JacksonJsonDeserializer.KEY_DEFAULT_TYPE`**: Резервный тип для десериализации ключей, если информация о заголовке отсутствует.
    
- **`JacksonJsonDeserializer.VALUE_DEFAULT_TYPE`**: Резервный тип для десериализации значений, если информация о заголовке отсутствует.
    
- **`JacksonJsonDeserializer.TRUSTED_PACKAGES`** (по умолчанию `java.util, java.lang`): Разделенный запятыми список шаблонов пакетов, разрешенных для десериализации. `*` означает десериализацию всего.
    
- **`JacksonJsonDeserializer.TYPE_MAPPINGS`** (по умолчанию пусто): См. "Mapping Types".
    
- **`JacksonJsonDeserializer.KEY_TYPE_METHOD`** (по умолчанию пусто): См. "Using Methods to Determine Types".
    
- **`JacksonJsonDeserializer.VALUE_TYPE_METHOD`** (по умолчанию пусто): См. "Using Methods to Determine Types".


> Начиная с версии 2.2, **заголовки информации о типе** (если добавлены сериализатором) **удаляются десериализатором.** 
> 
> Вы можете вернуться к предыдущему поведению, установив свойство `removeTypeHeaders` в `false`, либо непосредственно на десериализаторе, либо с помощью свойства конфигурации, описанного ранее.

Смотрите также "Customizing the JacksonJsonSerializer and JacksonJsonDeserializer".


Начиная с версии 2.8, если вы создаете сериализатор или десериализатор программно, как показано в "Programmatic Construction", вышеуказанные свойства будут применены фабриками, при условии, что вы не установили никакие свойства явно (используя методы `set*()` или используя fluent API). 

Ранее, при создании программно, свойства конфигурации никогда не применялись; это все еще так, если вы явно устанавливаете свойства на объекте напрямую.

## Сопоставление типов (Mapping Types)
Начиная с версии 2.2, при использовании JSON вы теперь можете предоставлять сопоставления типов, используя свойства из предыдущего списка. 

Ранее вам приходилось настраивать type mapper внутри сериализатора и десериализатора. 

**Сопоставления состоят из разделенного запятыми списка пар** `token:className`. 
На исходящей стороне имя класса полезной нагрузки сопоставляется с соответствующим токеном. На входящей стороне токен в заголовке типа сопоставляется с соответствующим именем класса.

Следующий пример создает набор сопоставлений:
```java
senderProps.put(
	ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, 
	JsonSerializer.class);
	
senderProps.put(
	JsonSerializer.TYPE_MAPPINGS, 
	"cat:com.mycat.Cat, hat:com.myhat.Hat");
...
consumerProps.put(
	ConsumerConfig.VALUE_DESERIALIZER_CLASS_CONFIG, 
	JsonDeserializer.class);
consumerProps.put(
	JsonDeserializer.TYPE_MAPPINGS, 
	"cat:com.yourcat.Cat, hat:com.yourhat.Hat");
```

Соответствующие объекты должны быть совместимы.

Если вы используете Spring Boot, вы можете предоставить эти свойства в файле `application.properties` (или yaml). Следующий пример показывает, как это сделать:
```java
spring.kafka.producer.value-serializer=org.springframework.kafka.support.serializer.JsonSerializer

spring.kafka.producer.properties.spring.json.type.mapping=cat:com.mycat.Cat,hat:com.myhat.Hat
```

Вы можете выполнять только простую конфигурацию со свойствами. 

Для более продвинутой конфигурации (например, использование пользовательского `ObjectMapper` в сериализаторе и десериализаторе) вы должны использовать конструкторы фабрик producer и consumer, которые принимают предварительно собранные сериализатор и десериализатор. 

Следующий пример Spring Boot переопределяет фабрики по умолчанию:
```java
@Bean
public ConsumerFactory<?, ?> kafkaConsumerFactory(JsonDeserializer customValueDeserializer) {
    Map<String, Object> properties = new HashMap<>();
    // properties.put(..., ...)
    // ...
    return new DefaultKafkaConsumerFactory<>(
	    properties,
        new StringDeserializer(), 
        customValueDeserializer);
}

@Bean
public ProducerFactory<?, ?> kafkaProducerFactory(
	JsonSerializer customValueSerializer) 
{
    return new DefaultKafkaProducerFactory<>(
	    properties.buildProducerProperties(),
        new StringSerializer(), 
        customValueSerializer
    );
}
```

Сеттеры также предоставлены как альтернатива использованию этих конструкторов.

> При использовании Spring Boot и переопределении `ConsumerFactory` и `ProducerFactory`, как показано выше, необходимо использовать generic-типы с подстановочными знаками с типом возвращаемого значения метода бина. Если вместо этого предоставлены конкретные generic-типы, то Spring Boot проигнорирует эти бины и все равно будет использовать те, что по умолчанию.


Начиная с версии 2.2, вы можете **явно настроить десериализатор на использование предоставленного целевого типа** и **ИГНОРИРОВАТЬ информацию о типе в ЗАГОЛОВКАХ,** используя один из перегруженных конструкторов, которые имеют аргумент `boolean useHeadersIfPresent` (который по умолчанию `true`). 

Следующий пример показывает, как это сделать:
```java
DefaultKafkaConsumerFactory<Integer, Cat1> cf = 
	new DefaultKafkaConsumerFactory<>(
		props,
        new IntegerDeserializer(), 
        new JsonDeserializer<>(Cat1.class, false)
    );
```

## Использование методов для определения типов (Using Methods to Determine Types)
С версии **Spring Kafka 2.5**, `JsonDeserializer` может **определять ЦЕЛЕВОЙ тип** НЕ через:
- заголовки `__TypeId__`
- свойства `spring.json.default.type`
- явно заданный тип в конструкторе

А может вызвать **статический МЕТОД**, указанный через properties:

### Свойства в фабрике для установки метода

|Свойство|Полное имя|
|---|---|
|`JsonDeserializer.KEY_TYPE_METHOD`|`spring.json.key.type.method`|
|`JsonDeserializer.VALUE_TYPE_METHOD`|`spring.json.value.type.method`|

### Допустимых сигнатур метода:
Метод обязан быть:
- `public static`
- возвращать **`com.fasterxml.jackson.databind.JavaType`**

И иметь одну из сигнатур:
```java
JavaType method(String topic, byte[] data, Headers headers)
JavaType method(byte[] data, Headers headers)
JavaType method(byte[] data)
```

### Как работает под капотом
#После получения записи Kafka:
1. Spring Kafka вызывает `JsonDeserializer.deserialize()`
    
2. Вместо того чтобы определять тип через заголовки/конфигурацию, он вызывает метод:
```java
JavaType type = resolverMethod.invoke(null, data, headers)
```

3. Тип `JavaType` создаётся через:
```java
com.fasterxml.jackson.databind.type.TypeFactory
```
Пример:
```java
JavaType thing1Type = TypeFactory.defaultInstance().constructType(Thing1.class);
```

4. Затем JSON десериализуется в этот JavaType:
```java
objectMapper.readValue(data, type);
```
➡️ Здесь используется как раз **тот самый ObjectMapper из `JacksonUtils.enhancedObjectMapper()`**, описанный выше.


### Пример
Допустим, нам нужно динамически определить, в какой тип преобразовать JSON: `Thing1` или `Thing2`.\
#### Конкретные типы:
```java
JavaType thing1Type = TypeFactory.defaultInstance().constructType(Thing1.class);

JavaType thing2Type = TypeFactory.defaultInstance().constructType(Thing2.class);
```

#### Метод определения типа:
```java
public static JavaType thingOneOrThingTwo(byte[] data, Headers headers) {
    if (data[21] == '1') {
        return thing1Type;
    } else {
        return thing2Type;
    }
}
```

#### 1️⃣ Включение через Kafka properties:
```yaml
spring
  .kafka
	.consumer
      .properties
	    .spring
		  .json
		    .value.type.method=
					com.example.TypeResolver.thingOneOrThingTwo
```

#### 2️⃣ Программная настройка
Следующее является примером создания десериализатора программно (при предоставлении фабрике consumer десериализатора в конструкторе):
```java
JsonDeserializer<Object> deser = new JsonDeserializer<>()
        .trustedPackages("*")
        .typeResolver(SomeClass::thing1Thing2JavaTypeForTopic);
```

Метод:
```java
public static JavaType thing1Thing2JavaTypeForTopic(
	String topic,
	byte[] data, 
	Headers headers) {
    ...
}
```

## Программное создание (Programmatic Construction)

**При создании** сериализатора/десериализатора **ПРОГРАММНО** **для использования в фабрике producer/consumer**, начиная с версии 2.3, вы можете использовать fluent API, который упрощает конфигурацию.
```java
@Bean
public ProducerFactory<MyKeyType, MyValueType> pf() {
    Map<String, Object> props = new HashMap<>();
    // props.put(..., ...)
    // ...
    DefaultKafkaProducerFactory<MyKeyType, MyValueType> pf = new DefaultKafkaProducerFactory<>(props,
        new JsonSerializer<MyKeyType>()
            .forKeys()
            .noTypeInfo(),
        new JsonSerializer<MyValueType>()
            .noTypeInfo());
    return pf;
}

@Bean
public ConsumerFactory<MyKeyType, MyValueType> cf() {
    Map<String, Object> props = new HashMap<>();
    // props.put(..., ...)
    // ...
    DefaultKafkaConsumerFactory<MyKeyType, MyValueType> cf = new DefaultKafkaConsumerFactory<>(props,
        new JsonDeserializer<>(MyKeyType.class)
            .forKeys()
            .ignoreTypeHeaders(),
        new JsonDeserializer<>(MyValueType.class)
            .ignoreTypeHeaders());
    return cf;
}
```

Чтобы предоставить сопоставление типов программно, аналогично "Using Methods to Determine Types", используйте свойство `typeFunction`.

```java
JsonDeserializer<Object> deser = new JsonDeserializer<>()
        .trustedPackages("*")
        .typeFunction(MyUtils::thingOneOrThingTwo);
```


### Автоматическая настройка сериализаторов/десериализаторов
**Когда работает автоматическая настройка:**
- Вы создаете сериализатор/десериализатор через конструктор
    
- НЕ используете fluent API (`.noTypeInfo()`, `.ignoreTypeHeaders()` и т.д.)
    
- НЕ вызываете методы `set*()` напрямую


**Пример АВТОМАТИЧЕСКОЙ настройки**
```java
// Фабрика сама применит свойства конфигурации
@Bean
public ConsumerFactory<String, Object> consumerFactory() {
    Map<String, Object> props = new HashMap<>();
    props.put(
	    JsonDeserializer.TRUSTED_PACKAGES, 
	    "com.example.models");
    props.put(
	    JsonDeserializer.USE_TYPE_INFO_HEADERS, 
	    false);
    
    // Десериализатор получит настройки из props автоматически
    return new DefaultKafkaConsumerFactory<>(
        props,
        new StringDeserializer(),
        new JsonDeserializer<>()  // ← автоматическая настройка
    );
}
```

**Пример РУЧНОЙ настройки (автоматика отключена):**
```java
@Bean
public ConsumerFactory<String, Object> consumerFactory() {
    Map<String, Object> props = new HashMap<>();
    props.put(JsonDeserializer.TRUSTED_PACKAGES, "com.example.models");
    
    // Fluent API отключает автоматическую настройку
    JsonDeserializer<Object> deserializer = new JsonDeserializer<>()
        .trustedPackages("*")  // ← ручная настройка
        .ignoreTypeHeaders();  // ← игнорирует USE_TYPE_INFO_HEADERS из props
    
    return new DefaultKafkaConsumerFactory<>(
        props,
        new StringDeserializer(),
        deserializer  // ← настройки уже установлены вручную
    );
}
```
