# 1. Message в Spring Messaging
`Message<T>` - это основная абстракция Spring Messaging для представления сообщений, которая служит **оберткой** над Kafka `ProducerRecord` и `ConsumerRecord`.

## Структура Message
```java
public interface Message<T> {
    T getPayload();           // Полезная нагрузка (тело сообщения)
    MessageHeaders getHeaders(); // Заголовки сообщения
}
```

## Маппинг между Kafka Records и Message

### Для ConsumerRecord → Message:
```java
ConsumerRecord<String, String> kafkaRecord = ...;
Message<String> springMessage = ...;

// Маппинг полей:
kafkaRecord.key()      → header(KafkaHeaders.RECEIVED_KEY)
kafkaRecord.topic()    → header(KafkaHeaders.RECEIVED_TOPIC) 
kafkaRecord.partition()→ header(KafkaHeaders.RECEIVED_PARTITION)
kafkaRecord.offset()   → header(KafkaHeaders.OFFSET)
kafkaRecord.timestamp()→ header(KafkaHeaders.RECEIVED_TIMESTAMP)
kafkaRecord.value()    → getPayload()
```

### Для Message → ProducerRecord:
```java
Message<String> springMessage = ...;
ProducerRecord<String, String> kafkaRecord = ...;

// Маппинг полей:
header(KafkaHeaders.TOPIC)        → kafkaRecord.topic()
header(KafkaHeaders.KEY)          → kafkaRecord.key()
header(KafkaHeaders.PARTITION)    → kafkaRecord.partition()
header(KafkaHeaders.TIMESTAMP)    → kafkaRecord.timestamp()
getPayload()                      → kafkaRecord.value()
```

## KafkaHeaders - КЛЮЧЕВЫЕ константы
```java
public interface KafkaHeaders {
    String PREFIX = "kafka_";
    
    // Входящие заголовки (из ConsumerRecord)
    String RECEIVED_KEY = "kafka_receivedKey";
    String RECEIVED_TOPIC = "kafka_receivedTopic";
    String RECEIVED_PARTITION = "kafka_receivedPartition";
    String RECEIVED_TIMESTAMP = "kafka_receivedTimestamp";
    String OFFSET = "kafka_offset";
    
    // Исходящие заголовки (для ProducerRecord)
    String KEY = "kafka_key";
    String TOPIC = "kafka_topic";
    String PARTITION = "kafka_partitionId";
    String TIMESTAMP = "kafka_messageTimestamp";
    
    // Служебные заголовки
    String NATIVE_HEADERS = "kafka_nativeHeaders";
    String BATCH_CONVERTED_HEADERS = "kafka_batchConvertedHeaders";
}
```

## Практическое использование

### В @KafkaListener:
```java
@KafkaListener(topics = "my-topic")
public void listen(Message<String> message) {
    String payload = message.getPayload();
    
    String topic = message
	    .getHeaders()
	    .get(KafkaHeaders.RECEIVED_TOPIC, String.class);
	    
    Integer partition = message
	    .getHeaders()
	    .get(KafkaHeaders.RECEIVED_PARTITION, Integer.class);
	    
    Long offset = message
	    .getHeaders()
	    .get(KafkaHeaders.OFFSET, Long.class);
    
    System.out.println(
	    "Received from " + topic + 
	    "[" + partition + "]@" + 
	    offset + ": " + payload);
}
```

### Создание сообщения для отправки:
```java
Message<String> message = MessageBuilder
    .withPayload("Hello Kafka")
    .setHeader(KafkaHeaders.TOPIC, "my-topic")
    .setHeader(KafkaHeaders.KEY, "message-key")
    .setHeader(KafkaHeaders.PARTITION, 0)
    .setHeader("custom-header", "custom-value")
    .build();

kafkaTemplate.send(message);
```

### MessageBuilder - удобное создание сообщений
```java
// Простое сообщение
Message<String> simpleMessage = MessageBuilder.withPayload("content").build();

// Сообщение с заголовками
Message<String> complexMessage = MessageBuilder
    .withPayload("content")
    .setHeader("header1", "value1")
    .setHeader(KafkaHeaders.TOPIC, "topic1")
    .setHeader(KafkaHeaders.KEY, "key123")
    .build();

// Копирование сообщения с изменениями
Message<String> copiedMessage = MessageBuilder
    .fromMessage(originalMessage)
    .setHeader("new-header", "new-value")
    .build();
```

Поддержка заголовков в Apache Kafka появилась, начиная с версии **0.11.0.0**.

Поскольку Spring Messaging (из Spring Framework) тоже оперирует объектами `Message<?>`, которые имеют `MessageHeaders`, начиная с Spring Kafka 2.0 появилась возможность **отображать (маппить)** заголовки Kafka в заголовки Spring и обратно.

В старых версиях Spring Kafka заголовки Kafka не переносились полностью — переносились только те свойства `ConsumerRecord` и `ProducerRecord`, которые не относились к payload (тело сообщения).

Теперь же Spring Kafka может маппить **произвольные**, созданные пользователем заголовки.

# 2. Kafka Message Headers
**Заголовки Kafka** — это дополнительные метаданные сообщения **вне payload**.  
То же самое, что headers в HTTP или e-mail.

Payload — тело сообщения  
Header — вспомогательная информация, которая помогает сервисам правильно работать с payload

Kafka headers — это пара:
```java
key: String
value: byte[]
```

### Применение
#### 1️⃣ Передавать метаданные для обработки сообщений

Например:
- `correlationId` — для связывания запрос/ответ
    
- `retry-count` — сколько раз сообщение уже ретраилось
    
- `source-system` — откуда пришло сообщение
    
- `message-type` — тип события
    
- `schema-version` — версия модели
    
- `createdAt` — timestamp создания 

Это ключевой кейс.  
Так payload остаётся «чистым», а логика контролируется заголовками.

#### 2️⃣ Управлять маршрутизацией и бизнес-логикой
Например, в Listener:
```java
@KafkaListener(...)
public void listen(String payload,
   @Header("type") String type
) {
    if (type.equals("ALERT")) processAlert(payload);
    else processDefault(payload);
}
```

#### 3️⃣ Позволяет брокеру и продюсеру правильно сериализовать/десериализовать
Spring Kafka очень зависит от типов.

Когда у тебя:
```java
payload — JSON
headers — типы полей, служебная инфа
```
Spring может автоматически восстановить объект.

#### 4️⃣ Позволяет отличать DLQ-сообщения от обычных
Dead-Letter сообщения почти всегда обогащаются заголовками:
- причина ошибки
- stacktrace
- имя consumer-а
- время ошибки


# Интерфейс Apache Kafka Header
Kafka предоставляет очень простой интерфейс заголовка
```java
public interface Header {
    String key();   // имя (ключ) заголовка
    byte[] value(); // значение заголовка как массив байт
}
```

Все значения всех заголовков Kafka — это **byte[]**. Не строки, не числа — именно байты.

Spring Kafka обеспечивает преобразование различных типов данных в такие байты и обратно.

---
# KafkaHeaderMapper — стратегия маппинга заголовков
Spring Kafka использует интерфейс:
```java
public interface KafkaHeaderMapper {
    void fromHeaders(MessageHeaders headers, Headers target);

    void toHeaders(Headers source, Map<String, Object> target);
}
```
- **fromHeaders** — копирует заголовки из `MessageHeaders` Spring в `Headers` Kafka перед отправкой (Producer).
    
- **toHeaders** — переносит заголовки из Kafka `Headers` в `MessageHeaders` Spring при получении (Consumer).


То есть используется единый механизм в обе стороны.

---
# Реализации: SimpleKafkaHeaderMapper и JsonKafkaHeaderMapper
Spring предоставляет две реализованные стратегии маппинга:
## ✔ SimpleKafkaHeaderMapper
Этот маппер:
- работает с “сырыми” значениями (`byte[]`)
    
- может преобразовывать строки → байты
    
- НЕ использует JSON
    
- НЕ сохраняет типы заголовков
    
- подходит, когда интегрируемся с системами, которые **не используют Spring** и **не понимают JSON-структуры заголовков**

---
## ✔ JsonKafkaHeaderMapper
Это **основной и продвинутый** маппер Spring Kafka.
Он позволяет:
### 1) сериализовать заголовки в JSON
При сериализации:
- имя заголовка → `key`
    
- **значение → JSON в байтах**
    
- информация о типе записывается в специальный заголовок:  
    **`spring_json_header_types`**
Этот заголовок содержит **JSON-карту** вида:
```java
{
  "headerName1": "java.lang.Integer",
  "headerName2": "java.time.LocalDateTime"
}
```

Spring потом использует это для корректной десериализации в изначальный тип.

### 2) поддерживать произвольные пользовательские типы
Например:
- `UUID`
    
- `LocalDateTime`
    
- `MimeType`
    
- `Enum`
    
- любые Java-классы (при условии, что они в доверенных пакетах)

### 3) выполнять выборочный маппинг заголовков по паттернам
Можно указать шаблоны имен заголовков:
- `*` — подстановочный символ
    
- `foo*` — все заголовки, начинающиеся на foo
    
- `*bar` — оканчиваются на bar
    
- `*baz*` — содержат baz
    
- `!id` — исключить
    
- `!timestamp` — исключить

**Первый совпавший шаблон выигрывает** — это важно.

Примеры конструкторов:
```java
new JsonKafkaHeaderMapper()// маппит почти все заголовки

// используем свой ObjectMapper
new JsonKafkaHeaderMapper(objectMapper) 

// маппинг по шаблонам
new JsonKafkaHeaderMapper("foo*", "!id")  
new JsonKafkaHeaderMapper(objectMapper, patterns...)
```

### 4) использовать доверенные пакеты
По умолчанию разрешено десериализовать только из `java.lang` и `java.util`.

Если сообщения приходят из ненадёжного источника — это правильно.

Чтобы разрешить десериализацию своих классов:
```java
mapper.addTrustedPackages("com.example.*");
```

Чтобы разрешить всё:
```java
mapper.addTrustedPackages("*");
```

## Как работает JsonKafkaHeaderMapper?
Допустим, в заголовок кладу:
```java
headers.put("user", new User("Tom", 22));
```

JsonKafkaHeaderMapper сохранит:
```java
"user": <JSON>      ← байтовый массив JSON
```

И добавит **вспомогательный заголовок**:
```java
spring_json_header_types = { "user": "com.example.User" }
```

При приёме Spring сможет восстановить:
```java
User("Tom", 22)
```

# Пример
Допустим:
### Микросервис A → отправляет заказы в Kafka:
Payload = JSON заказа  
Headers = метаданные, которые нельзя класть в payload:
- `request-id`: `UUID` — нужен для трассировки
    
- `auth-user`: кастомный объект с инфой о пользователе
    
- `retry-count`: `Integer`
    
- `mime-type`: `MimeType` (Spring тип)
    
- `source-service`: `String`

### Микросервис B → должен их обработать и прочитать заголовки повторно **в исходных типах**, а не просто byte[].

---
## **ПРОБЛЕМА без JsonKafkaHeaderMapper**
Kafka умеет хранить только `byte[]`.  

Значит заголовок:
```java
"request-id": UUID("df21...")
```

превращается в:
```java
"request-id": byte[]
```

При чтении B **не знает**, что это UUID.

То же с:
- кастомными объектами
- картами/списками
- MimeType
- Integer vs String — невозможно различить
- **тип теряется полностью** (хотя можно установить договоренность при обмене)

❗ То есть без маппера A отправит, B прочитать **не сможет**, кроме вручную через `new String(bytes)` (что даст только строку).

---
## **КАК ЭТО РЕШАЕТ JsonKafkaHeaderMapper**
Spring Kafka делает следующее:
### 1) сериализует заголовки в JSON

### 2) добавляет спец-заголовок `"spring_json_header_types"`:
```java
{
  "request-id": "java.util.UUID",
  "auth-user": "com.example.auth.UserInfo",
  "retry-count": "java.lang.Integer"
}
```

### 3) Consumer использует эти типы при чтении.
Результат:  
Ты получаешь **ровно те объекты**, которые отправил.

public class UserInfo {

## 2. PRODUCER — отправляем сообщение
```java
@Bean
public KafkaTemplate<String, Order> kafkaTemplate(
        ProducerFactory<String, Order> pf
) {
    KafkaTemplate<String, Order> template = new KafkaTemplate<>(pf);

    // Включаем JSON маппинг заголовков
    JsonKafkaHeaderMapper mapper = new JsonKafkaHeaderMapper();
    mapper.addTrustedPackages("*"); // позволяем десериализацию всех пакетов

    MessagingMessageConverter converter = 
	    new MessagingMessageConverter();
    converter.setHeaderMapper(mapper);

    template.setMessageConverter(converter);
    return template;
}
```

И отправка:
```java
template.send(
    MessageBuilder
        .withPayload(order)
        .setHeader("request-id", UUID.randomUUID())
        .setHeader("auth-user", new UserInfo("mike", "admin"))
        .setHeader("retry-count", 5)
        .setHeader("mime-type", MimeType.valueOf("application/json"))
        .setHeader("source-service", "order-service")
        .build()
);
```

## CONSUMER — читаем
```java
@Bean
public ConcurrentKafkaListenerContainerFactory<String, Order> kafkaListenerFactory(
        ConsumerFactory<String, Order> cf
) {
    ConcurrentKafkaListenerContainerFactory<String, Order> 
	factory =
        new ConcurrentKafkaListenerContainerFactory<>();

    factory.setConsumerFactory(cf);

    // Тот же самый маппер!
    JsonKafkaHeaderMapper mapper = new JsonKafkaHeaderMapper();
    mapper.addTrustedPackages("*");

    MessagingMessageConverter converter = 
	    new MessagingMessageConverter();
    converter.setHeaderMapper(mapper);

    factory.setMessageConverter(converter);
    return factory;
}
```

И слушатель:
```java
@KafkaListener(topics = "orders-topic")
public void listen(Order order,
                  @Header("request-id") UUID requestId,
                  @Header("auth-user") UserInfo user,
                  @Header("retry-count") Integer retryCount,
                  @Header("mime-type") MimeType mimeType,
                  @Header("source-service") String service) {

    System.out.println(order);
    System.out.println("requestId = " + requestId);
    System.out.println("authUser = " + user.getUsername());
    System.out.println("retry-count = " + retryCount);
    System.out.println("mime-type = " + mimeType);
    System.out.println("source = " + service);
}
```
