
При использовании транзакций, если слушатель выбрасывает исключение (и обработчик ошибок, если присутствует, выбрасывает исключение), транзакция откатывается. 

*По умолчанию* **любые необработанные записи** (включая неудавшуюся запись) **повторно извлекаются при следующем опросе.**
Это достигается выполнением операций поиска в `DefaultAfterRollbackProcessor`. 

## AfterRollbackProcessor
`AfterRollbackProcessor` — компонент, который управляет тем, **что делать с сообщениями после отката транзакции**, если слушатель выбросил исключение.


> [!NOTE] Важно
> Используется **только при транзакциях**.


Основные задачи:
- повторное получение сообщений (seek)
- определение retry-поведения
- отправка проблемных сообщений в DLT
- контроль количества попыток
- применение BackOff между попытками
    

Фактически — это **"error handler" на уровне транзакции после rollback**, в отличие от `DefaultErrorHandler`, который работает без транзакций.

> Начиная с версии 2.2, `DefaultAfterRollbackProcessor` теперь может восстанавливать (пропускать) запись, которая продолжает терпеть неудачу. 

### Применение
Используется в сценариях:
#### ✔ транзакционный consumer + producer
Когда слушатель работает внутри Kafka-транзакции.

#### ✔ необходимо контролировать повторную обработку после rollback
Например:
- не повторять бесконечно
- отправлять в DLT после нескольких попыток
- применять backoff

#### ✔ batch listener — особое поведение
В batch режиме контейнер не знает, какая запись упала → вся пачка повторяется, если не включено дополнительное поведение.

### Поведение по умолчанию (DefaultAfterRollbackProcessor)
По умолчанию:
- после rollback делает **seek** на начальные offset текущей партии
    
- каждое сообщение повторяется заново
    
- после 10 неудач проблемное сообщение → ERROR лог
    
- восстановление (recovery) не делается автоматически
    

Для batch:
- **вся пачка повторяется**, потому что фреймворк не знает индекс упавшей записи  
    (исключение — см. batchRecoverAfterRollback ниже)

### Настройка пользовательского AfterRollbackProcessor
Следующий пример настраивает восстановление после 3 попыток:
```java
AfterRollbackProcessor<String, String> processor =
    new DefaultAfterRollbackProcessor((record, exception) -> {
        // восстановление после 3 неудач, без back off - например, отправка в топик мертвых писем
    }, new FixedBackOff(0L, 2L));
```
Объяснение:
- `recoverer` — что делать после maxFailures
- `BackOff` — количество и интервал retry

> *Когда вы НЕ используете транзакции*, вы можете достичь аналогичной функциональности, настроив `DefaultErrorHandler`. Смотрите "Container Error Handlers".


### Основные настройки и возможности
#### ✔ maxFailures, backoff, recoverer
Можно настроить:
- сколько раз пытаться
- интервал
- что делать при исчерпании retry (DLT, ignore, log)

#### ✔ commitRecovered — фиксация смещения после DLT
Если сообщение отправлено в DLT, и вы хотите зафиксировать offset:
```java
processor.setCommitRecovered(true);

processor.setKafkaTemplate(kafkaTemplate); // обязательный параметр
```

#### ✔ Настройка BackOff для разных сообщений
С версии **2.6**:
```java
processor.setBackOffFunction((record, ex) -> {
    if (ex instanceof ValidationException)
        return new FixedBackOff(0,0); // нет retry
    return new FixedBackOff(1000,5); // обычный retry
});
```

#### ✔ resetStateOnExceptionChange
С версии **2.6.3**:
Перезапускает retry, если тип исключения изменился.
```java
processor.setResetStateOnExceptionChange(true);
```

#### ✔ Неретрайбл исключения
Можно добавить:
```java
processor.addNotRetryableException(
	IllegalArgumentException.class);
```
По умолчанию неретрайбл:
- DeserializationException
- MessageConversionException
- ConversionException
- MethodArgumentResolutionException
- NoSuchMethodException
- ClassCastException

#### ✔ batchRecoverAfterRollback (новое!)
С версии **3.2** можно восстанавливать **всю batch**, если она постоянно падает:
```java
containerProps.setBatchRecoverAfterRollback(true);
```

Начиная с версии 3.2, восстановление теперь может восстанавливать (пропускать) всю пачку записей, которая продолжает терпеть неудачу.
Установите `ContainerProperties.setBatchRecoverAfterRollback(true)`, чтобы включить эту функцию.

По умолчанию в batch восстановление невозможно → слушатель должен сам обрабатывать ошибочную запись.  
Смотрите также "Publishing Dead-letter Records".

## Взаимодействие с транзакциями и DLT
Если recoverer использует `DeadLetterPublishingRecoverer`:
- после DLT можно зафиксировать offset (**commitRecovered=true**)
- публикация происходит в **новой транзакции**, если включено

Это важно для "exactly-once".

## Ошибки восстановителя (recoverer)
Если recoverer выбрасывает исключение:
- запись снова будет подлежать seek → повторная обработка
    
- BackOff сбрасывается (с версии 2.5.5)

Можно вернуть старое поведение:
```java
processor.setResetStateOnRecoveryFailure(false);
```

## RetryListener (с версии 2.7)
Для логгинга или метрик:
Начиная с версии 2.7, процессор может быть настроен с одним или несколькими `RetryListener`, получающими уведомления о ходе повторной попытки и восстановления.

```java
@FunctionalInterface
public interface RetryListener {
    void failedDelivery(
	    ConsumerRecord<?, ?> record, 
	    Exception ex, 
	    int deliveryAttempt);

    default void recovered(
	    ConsumerRecord<?, ?> record, 
	    Exception ex) {
    }

    default void recoveryFailed(
	    ConsumerRecord<?, ?> record, 
	    Exception original, 
	    Exception failure) {
    }
}
```

```java
processor.addRetryListener(new RetryListener() {
    @Override
    public void failedDelivery(ConsumerRecord<?, ?> rec, Exception ex, int attempt) {}

    @Override
    public void recovered(ConsumerRecord<?, ?> rec, Exception ex) {}

    @Override
    public void recoveryFailed(ConsumerRecord<?, ?> rec, Exception original, Exception failure) {}
});
```

Смотрите JavaDocs для получения дополнительной информации.

## # Сложный кейс: ProducerFencedException
Контейнер **не вызывает** AfterRollbackProcessor, если:
- transactional.id потерял актуальность
- fence произошёл при rebalance

Можно:
- увеличить timeout
- выполнять пустые транзакции периодически
- включить автоматическую остановку контейнера:
```java
containerProps.setStopContainerWhenFenced(true);
```
