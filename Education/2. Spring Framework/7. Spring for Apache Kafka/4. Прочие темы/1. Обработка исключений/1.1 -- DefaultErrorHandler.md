`DefaultErrorHandler` — основной обработчик ошибок Spring Kafka, заменивший:
- `SeekToCurrentErrorHandler`
- `RecoveringBatchErrorHandler`

### Реализуется CommonErrorHandler
```java
public class DefaultErrorHandler 
extends FailedBatchProcessor
implements CommonErrorHandler
```

Он управляет повторными попытками доставки, откатом смещений, восстановлением сообщений (recover), логированием и доступен для Record- и Batch-listeners.

> Одно отличие состоит в том, что резервное поведение для слушателей пачек (когда выбрасывается исключение, отличное от `BatchListenerFailedException`) эквивалентно "Retrying Complete Batches".

---
# 1. Поведение при ошибках
## 1) **Повторные попытки (Retries)**
Для RecordListener *по умолчанию*:  
`FixedBackOff(0, 9)` → 10 попыток обработки записи.

Можно настроить:
```java
new DefaultErrorHandler(recoverer, new FixedBackOff(0, 2)); 
```
Значит: 3 попытки (исходная + 2 повторные), иначе попытка "восстановления"

## 2) **Восстановление (Recovery)**
Если попытки исчерпаны — вызывается **ConsumerRecordRecoverer**:
- по умолчанию — просто логирует ошибку
    
- *можно использовать DLT*(DeadLetterPublishingRecoverer)
    
- можно сделать свой recoverer

---
# 2. Поведение при ошибках: два режима
## ✔ Режим поиска (Seek) — классический
**По умолчанию** DefaultErrorHandler **делает seek()** к смещениям необработанных записей при ошибке. 

То есть Kafka заново вернёт только «проблемные» сообщения.

**Что происходит:**
1. **Ошибка в сообщении 3** из пачки [1, 2, 3, 4, 5]
2. **Seek к offset сообщения 3** в Kafka
3. **Следующий poll()** вернет [3, 4, 5] заново из Kafka
4. **Kafka "не знает"** о нашей ошибке - мы просто перечитали сообщения
```java
Брокер Kafka: [1, 2, 3, 4, 5, 6, 7...]
После seek:    [3, 4, 5, 6, 7...] ← читаем заново
```
## ✔ Режим без seek — «переподача» сообщений
С версии 2.9:
```java
handler.setSeekAfterError(false);
```

Тогда:
- контейнер _сам_ хранит необработанные записи
    
- после ошибки → listener вызывается заново с новыми ConsumerRecords без seek
    
- один лишний poll() нужен, чтобы не упасть по timeout

**Что происходит:**
1. **Ошибка в сообщении 3** из пачки \[1, 2, 3, 4, 5]
    
2. **НЕ делаем seek** - offset в Kafka не меняется
    
3. **Контейнер сохраняет** \[3, 4, 5] в памяти
    
4. **Следующий poll()** делается только чтобы **"поддержать heartbeat" consumer** (не дать consumer отвалиться по таймауту)
    
5. **Listener вызывается** с сохраненными \[3, 4, 5] из памяти

```java
Брокер Kafka: [1, 2, 3, 4, 5, 6, 7...]
В памяти:     [3, 4, 5] ← обрабатываем повторно

Следующий poll вернет: [6, 7, 8...] ← продолжаем нормально
```


## Пример настройки DefaultErrorHandler
Следующий пример настраивает восстановление после трех попыток:
```java
DefaultErrorHandler errorHandler =
    new DefaultErrorHandler((record, exception) -> {
        // восстановление после 3 неудач, без back off - 
        // например, отправка в топик мертвых писем
    }, new FixedBackOff(0L, 2L));
```

## Добавление DefaultErrorHandler в фабрику контейнеров слушателей
Чтобы настроить контейнер слушателя с настроенным экземпляром этого обработчика, добавьте его в фабрику контейнеров.

Например, с фабрикой контейнеров `@KafkaListener` вы можете добавить `DefaultErrorHandler` следующим образом:
```java
@Bean
public ConcurrentKafkaListenerContainerFactory<String, String> kafkaListenerContainerFactory() {
    ConcurrentKafkaListenerContainerFactory<String, String> 
	    factory = 
		    new ConcurrentKafkaListenerContainerFactory<>();
    
    factory.setConsumerFactory(consumerFactory());
    factory.getContainerProperties().setAckMode(AckMode.RECORD);
    
    factory.setCommonErrorHandler(
	    new DefaultErrorHandler(new FixedBackOff(1000L, 2L)));
    return factory;
}
```

Для слушателя записей это **повторит доставку до 2 раз** (3 попытки доставки) **с back off в 1 секунду**, вместо конфигурации по умолчанию (`FixedBackOff(0L, 9)`). 

Неудачи просто логируются после исчерпания повторных попыток.

# 4. DefaultErrorHandler и транзакции
Если используется **транзакционный контейнер**, то `DefaultErrorHandler` **не применяется**. 

Для транзакций используется **DefaultAfterRollbackProcessor**, который выполняет аналогичную работу, но _после отката транзакции_.

(см [[1.3 -- Процессор после отката (After Rollback Processor)]])

# 5. Фатальные исключения
`DefaultErrorHandler` считает **некоторые исключения фатальными** — они _не будут повторены_, а сразу передадутся в **восстановитель (recoverer)**.

По умолчанию фатальные исключения:
- DeserializationException 
	
- MessageConversionException
    
- ConversionException
    
- MethodArgumentResolutionException
    
- NoSuchMethodException
    
- ClassCastException
    

Причина: повторная доставка **не исправит** проблему.

## Можно расширять список фатальных исключений

Пример: пометить `IllegalArgumentException` как не повторяемое:
```java
@Bean
public DefaultErrorHandler errorHandler(
ConsumerRecordRecoverer recoverer) {
    DefaultErrorHandler handler = new 
	    DefaultErrorHandler(recoverer);
    
    handler
	    .addNotRetryableExceptions(
		    IllegalArgumentException.class
		);
    return handler;
}
```

## Error vs Exception
⚠️ Важно: **DefaultErrorHandler работает только с RuntimeException**.

Если бросить **Error** (например, `OutOfMemoryError`):
- обработчик ошибок _пропускается_
    
- потребитель **умирает**
    
- консьюмер перестаёт обрабатывать сообщения, хотя приложение может выглядеть «здоровым»


➡️ *В обработчиках сообщений всегда бросай только RuntimeException*.

# 6. RetryListener — наблюдение за ретраями
Можно прикрепить один или несколько `RetryListener`, чтобы отслеживать:
- каждую попытку повторной доставки
    
- успешное восстановление
    
- падение восстановления


Интерфейс:
```java
public interface RetryListener {
    void failedDelivery(
	    ConsumerRecord<?, ?> record, 
	    Exception ex, 
	    int attempt);
    
    default void recovered(
	    ConsumerRecord<?, ?> record, 
	    Exception ex) {}
    
    default void recoveryFailed(
	    ConsumerRecord<?, ?> record, 
	    Exception original, 
	    Exception failure) {}
}
```

Это удобно для логирования, метрик, алертов.

# 7. Поведение при провале Recoverer
Если recoverer бросает исключение:
- запись снова попадает в механизм retry
    
- **BackOff сбрасывается**
    
- цикл повторных попыток начинается сначала


Можно отключить сброс:
```java
handler.setResetStateOnRecoveryFailure(false);
```
