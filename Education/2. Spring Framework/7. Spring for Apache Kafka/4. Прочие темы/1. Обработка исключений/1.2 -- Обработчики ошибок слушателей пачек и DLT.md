> [!NOTE] 
> Неблокирующие повторные попытки (аннотация `@RetryableTopic`) НЕ поддерживаются со слушателями пачек. 
> 
> Для функциональности топиков мертвых писем со слушателями пачек используйте `DefaultErrorHandler` с `DeadLetterPublishingRecoverer`.
# 1. Использование BatchListenerFailedException
При обработке сообщений **пачками** (`batchListener = true`) **важно знать**, _какая конкретная запись в пачке упала_.

```java
@KafkaListener(id = "batch", topics = "myTopic")
public void listen(List<Thing> things) {
    for (int i = 0; i < things.size(); i++) {
        try {
            process(things.get(i));
        } catch (Exception e) {
            throw new BatchListenerFailedException(
	            "Failed to process", i); // 🔥Указываем индекс🔥
        }
    }
}
```


Для этого в слушателе выбрасывается (передаем сам ConsumerRecord):
```java
new BatchListenerFailedException("msg", cause, record)
```

или, если нет `ConsumerRecord` (указываем индекс):
```java
new BatchListenerFailedException("msg", cause, index)
```

Это даёт контейнеру точный индекс сбоя.

## Что делает DefaultErrorHandler при работе с BatchListenerFailedException
Когда в пачке, например, 6 записей → упала запись №2:
1. **Смещения 0 и 1 фиксируются**
    
2. Записи **2,3,4,5** повторяются согласно BackOff
    
3. Если попытки исчерпаны → **запись 2 отправляется в DLT**
    
4. Смещение записи 2 фиксируется
    
5. Обработка продолжается с **3,4,5**
    

То есть _DLT получает только ту запись, что упала_, а не всю пачку.

## Алгоритм обработки ошибок:
Последовательность событий:
1. (Фиксация успешных) Зафиксировать смещения записей до индекса.
    
2. (Ретрай всех) Если повторные попытки не исчерпаны, выполнить поиски так, чтобы все оставшиеся записи (включая неудавшуюся запись) были доставлены заново.
    
3. (Восстановление) Если повторные попытки исчерпаны, попытаться восстановить неудавшуюся запись (по умолчанию только логирование) и выполнить поиски так, чтобы оставшиеся записи (исключая неудавшуюся запись) были доставлены заново. Смещение восстановленной записи фиксируется.
    
4. Если повторные попытки исчерпаны и восстановление не удалось, поиски выполняются так, как если бы повторные попытки не были исчерпаны.


Когда контейнер настроен с `AckMode.MANUAL_IMMEDIATE` (ручной режим управления offset), обработчик ошибок может быть настроен для фиксации смещения восстановленных записей; 
установите свойство `commitRecovered` в `true`.

# 2. Настройка DefaultErrorHandler + DLT для batch listener
Настройте `DefaultErrorHandler` с `DeadLetterPublishingRecoverer` на вашей фабрике контейнеров слушателей пачек:

```java
@Bean
public ConcurrentKafkaListenerContainerFactory<String, Order> batchFactory(
        ConsumerFactory<String, Order> consumerFactory,
        KafkaTemplate<String, Order> kafkaTemplate) 
{
    ConcurrentKafkaListenerContainerFactory<String, Order> 
	    factory =
	        new ConcurrentKafkaListenerContainerFactory<>();
    factory.setConsumerFactory(consumerFactory);
    factory.setBatchListener(true);

    // Настройка публикации мертвых писем
    DeadLetterPublishingRecoverer recoverer = 
	    new DeadLetterPublishingRecoverer(
		    kafkaTemplate,
            (record, ex) -> new TopicPartition(
	            record.topic() + "-dlt", record.partition()
	        )
	    );

    // Настройка повторных попыток: 3 попытки с интервалом 1 секунда между каждой
    DefaultErrorHandler errorHandler = 
	    new DefaultErrorHandler(
		    recoverer,
            new FixedBackOff(1000L, 2L)); // 2 повторные попытки = 3 всего попыток

    factory.setCommonErrorHandler(errorHandler);
    return factory;
}
```

Это включает retry + DLT для batch-режима.

# 3. Пропуск retry для определённых исключений
По умолчанию retry не делается только для **фатальных исключений**  
(например, DeserializationException, MessageConversionException).

Можно добавить свои "не ретраящиеся" исключения:
```java
errorHandler
	.addNotRetryableExceptions(ValidationException.class);
```

```java
@Bean
public ConcurrentKafkaListenerContainerFactory<String, Order> batchFactory(
        ConsumerFactory<String, Order> consumerFactory,
        KafkaTemplate<String, Order> kafkaTemplate) 
    {

    ConcurrentKafkaListenerContainerFactory<String, Order> 
	    factory =
	        new ConcurrentKafkaListenerContainerFactory<>();
    factory.setConsumerFactory(consumerFactory);
    factory.setBatchListener(true);
	
	//DLT
    DeadLetterPublishingRecoverer recoverer = 
	    new DeadLetterPublishingRecoverer(kafkaTemplate);
    // Обработчик ошибок
    DefaultErrorHandler errorHandler = 
	    new DefaultErrorHandler(
		    recoverer,
            new FixedBackOff(1000L, 2L)
        );

    // Добавление пользовательских типов исключений, которые должны пропускать повторные попытки и идти напрямую в DLT
    errorHandler
	    .addNotRetryableExceptions(
		    ValidationException.class, 
		    InvalidFormatException.class);

    factory.setCommonErrorHandler(errorHandler);
    return factory;
}
```

Если причина `BatchListenerFailedException` относится к таким типам → запись сразу идёт в DLT.

Теперь в вашем слушателе:
```java
@KafkaListener(
	id = "batch-listener", 
	topics = "orders", 
	containerFactory = "batchFactory")
public void processOrders(
	List<ConsumerRecord<String, Order>> records) 
{
    for (ConsumerRecord<String, Order> record : records) {
        try {
            process(record.value());
        }
        catch (DatabaseException e) {
            // Будет повторено 3 раза (в соответствии с конфигурацией BackOff)
            throw new BatchListenerFailedException(
	            "Database error", e, record);
        }
        catch (ValidationException e) {
            // Пропускает повторные попытки - идет напрямую в DLT
            // (потому что ValidationException настроен как не повторяемый)
            throw new BatchListenerFailedException(
	            "Validation failed", e, record);
        }
    }
}
```

Обработчик ошибок проверяет причину (второй параметр) `BatchListenerFailedException`. 

Если причина классифицирована как не повторяемая, запись немедленно отправляется в DLT без повторных попыток.

# 5. Поведение commit в batch mode
### AckMode.BATCH
(используется по умолчанию с batchListener=true)
- смещения _до_ упавшей записи фиксируются сразу
    
- смещение _упавшей_ записи фиксируется после восстановления (DLT)
    

### AckMode.MANUAL_IMMEDIATE
Нужно вручную подтверждать:
```java
ack.acknowledge();
```

Если хотите фиксировать после DLT:
```java
errorHandler.setCommitRecovered(true);
```

Пример с ручным подтверждением:
```java
@KafkaListener(
	id = "manual-batch", 
	topics = "myTopic", 
	containerFactory = "manualBatchFactory")
public void listen(
	List<ConsumerRecord<String, Order>> records, 
	Acknowledgment ack) 
{
    for (ConsumerRecord<String, Order> record : records) {
        try {
            process(record.value());
        }
        catch (Exception e) {
            throw new BatchListenerFailedException(
	            "Processing failed", 
	            e, 
	            record);
        }
    }
    ack.acknowledge();
}
```

# 6. Ошибки при работе с обработчиками ошибок

## 1. Ошибки преобразования с обработчиками ошибок пачек
Начиная с версии 2.8, **слушатели пачек теперь могут правильно обрабатывать ошибки преобразования**, при использовании `MessageConverter` с
- `ByteArrayDeserializer`, 
- `BytesDeserializer` 
- или `StringDeserializer`, а также `DefaultErrorHandler`.

Когда происходит ошибка преобразования,
- полезная нагрузка устанавливается в `null`
    
- исключение десериализации добавляется в заголовки записи, `KafkaHeaders.CONVERSION_FAILURES`

Список `ConversionExceptions` доступен в слушателе, чтобы слушатель мог выбросить `BatchListenerFailedException`, указывая первый индекс, на котором произошло исключение преобразования.

```java
@KafkaListener(
	id = "test", 
	topics = "topic")
void listen(
	List<Thing> in, 
	@Header(KafkaHeaders.CONVERSION_FAILURES) 
		List<ConversionException> exceptions) 
{
    for (int i = 0; i < in.size(); i++) {
        Foo foo = in.get(i);
        if (foo == null && exceptions.get(i) != null) {
            throw new BatchListenerFailedException(
	            "Conversion error", 
	            exceptions.get(i),
	            i);
        }
        process(foo);
    }
}
```

## 2.Ошибки десериализации в batch listener
**Слушатели пачек** *требуют ручной обработки ошибок десериализации.* 
(Так как не ясно, какое сообщение из пачки нельзя десериализовать)

В отличие от слушателей записей, **нет автоматического обработчика ошибок, который обнаруживает и направляет сбои десериализации в DLT**. 
Вы должны явно проверять неудавшиеся записи и выбрасывать `BatchListenerFailedException`.

Для этого нужно:
### 1. Включить ErrorHandlingDeserializer
Используйте `ErrorHandlingDeserializer`, чтобы предотвратить остановку всей пачки из-за сбоев десериализации:
```java
@Bean
public ConsumerFactory<String, Order> consumerFactory() {
    Map<String, Object> props = new HashMap<>();
    props.put(
	    ConsumerConfig.BOOTSTRAP_SERVERS_CONFIG, 
	    bootstrapServers);
	    
	    
    props.put(
	    ConsumerConfig.KEY_DESERIALIZER_CLASS_CONFIG, 
	    StringDeserializer.class);
	    
    // 🔥Оберните ваш десериализатор с ErrorHandlingDeserializer
    props.put(
	    // Конфигурация НАСТРОЙКИ для десериалзации 
	    ConsumerConfig.VALUE_DESERIALIZER_CLASS_CONFIG, 
	    ErrorHandlingDeserializer.class);
	    
	    
    props.put(
	    ErrorHandlingDeserializer.VALUE_DESERIALIZER_CLASS, 
	    JsonDeserializer.class.getName());

    return new DefaultKafkaConsumerFactory<>(props);
}
```

### 2. В слушателе проверять записи с value == null
В вашем слушателе вы должны вручную проверять значения `null`, которые указывают на сбои десериализации:
```java
@KafkaListener(
	id = "batch-deser", 
	topics = "orders", 
	containerFactory = "batchFactory")
public void listen(List<ConsumerRecord<String, Order>> records) {
    for (ConsumerRecord<String, Order> record : records) {
        if (record.value() == null) {
            // Десериализация не удалась - выбросить исключение для отправки в DLT
            throw new BatchListenerFailedException("Deserialization failed", record);
        }
        process(record.value());
    }
}
```

### 3. Recoverer публикует в DLT:
Когда `DeadLetterPublishingRecoverer` публикует сбой десериализации в DLT:
- Исходные данные `byte[]`, которые не удалось десериализовать, восстанавливаются как значение записи
    
- Информация об исключении (имя класса, сообщение, трассировка стека) добавляется в стандартные заголовки исключений DLT
    
- Исходный заголовок исключения `ErrorHandlingDeserializer` удаляется по умолчанию (установите `setRetainExceptionHeader(true)` на восстановителе, чтобы сохранить его)

Можно сохранить оригинальный заголовок ErrorHandlingDeserializer:
```java
recoverer.setRetainExceptionHeader(true);
```
