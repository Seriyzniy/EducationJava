Этот раздел описывает, как обрабатывать различные исключения, которые могут возникнуть при использовании Spring for Apache Kafka.

# 1. Обработчики ошибок слушателей (Listener Error Handlers)
Начиная с версии 2.0, аннотация `@KafkaListener` имеет новый атрибут: `errorHandler`.
```java
@KafkaListener(errorHandler = "myHandler")
```
## 1. Интерфейс KafkaListenerErrorHandler
Вы можете использовать `errorHandler` для предоставления имени бина реализации `KafkaListenerErrorHandler`. 

Этот функциональный интерфейс имеет один метод, как показано в следующем листинге:
```JAVA
@FunctionalInterface
public interface KafkaListenerErrorHandler {
    Object handleError(
	    Message<?> message, 
	    ListenerExecutionFailedException exception
	) throws Exception;
}
```

У вас есть доступ к объекту `Message<?>` spring-messaging, созданному конвертером сообщений, и исключению, которое было выброшено слушателем, которое обернуто в `ListenerExecutionFailedException`. 

Обработчик ошибок может выбросить оригинальное или новое исключение, которое выбрасывается в контейнер. Все, что возвращается обработчиком ошибок, игнорируется.

## 2.Нововвдения
Начиная с версии 2.7, вы можете установить свойство `rawRecordHeader` на `MessagingMessageConverter` и `BatchMessagingMessageConverter`, что приводит к добавлению сырого `ConsumerRecord` в преобразованный `Message<?>` в заголовке `KafkaHeaders.RAW_DATA`. 
```java
converter.setRawRecordHeader(true);
```


Это полезно, например, если вы хотите использовать `DeadLetterPublishingRecoverer` в обработчике ошибок слушателя. 
Это может быть использовано в сценарии request/reply, где вы хотите отправить результат сбоя отправителю после некоторого количества повторных попыток, после захвата неудавшейся записи в топик мертвых писем (dead letter topic).

Пример: после 10 попыток → DLT
```java
@Bean
public 
KafkaListenerErrorHandler eh(
	DeadLetterPublishingRecoverer recoverer
) {
	// Возвращаем экзепляр функционального интефрейса (анонимный класс, а точнее функционалоьный объект)
    return (msg, ex) -> {
        if (msg
	        .getHeaders()
	        .get(
		        KafkaHeaders.DELIVERY_ATTEMPT, 
		        Integer.class) > 9) 
		{
            recoverer
	            .accept(msg
			            .getHeaders()
			            .get(KafkaHeaders.RAW_DATA,
				             ConsumerRecord.class
				        ), 
				        ex);
            return "FAILED";
        }
        throw ex;
    };
}
```

## 3. Под-интерфейсы KafkaListenerErrorHandler

Spring предоставляет расширенные варианты:

### 1. **ConsumerAwareListenerErrorHandler**
дополнительно получает `Consumer<?, ?>`:
```java
handleError(message, exception, consumer)
```

### 2. **ManualAckListenerErrorHandler**
дополнительно получает `Acknowledgment` при ручном ack:
```java
handleError(message, exception, consumer, ack)
```

## 🚫 Важно: нельзя делать consumer.seek()
Т.к. контейнер не узнает о seek и нарушится управление offset'ами.

# 2. Обработчики ошибок контейнеров (Container Error Handlers)
Начиная с версии 2.8, устаревшие интерфейсы `ErrorHandler` и `BatchErrorHandler` были заменены **новым** `CommonErrorHandler`

Он работает **и для одиночных записей, и для батчей**, поэтому фабрика контейнеров может создавать любые слушатели с одним обработчиком.

## 1. Транзакции и ошибки
Если Kafka Listener работает **в транзакциях**, Spring **НЕ ПОДКЛЮЧАЕТ ОБРАБОТЧИК ошибок по умолчанию**, чтобы:
- исключение автоматически вызвало **rollback** транзакции.

==Ошибка при транзакциях== обрабатывается компонентом **AfterRollbackProcessor**.

Если вы используете собственный CommonErrorHandler —  
💡 **он должен выбросить исключение**, чтобы откатить транзакцию.

## 2. Как работает CommonErrorHandler
У *интерфейса* есть метод:
```java
default boolean isAckAfterHandle()
```
Он определяет:
- ❗ Если обработчик ошибок _не_ выбрасывает исключение — Spring **зафиксирует смещение** (commit offset).
    
- Возвращает `true` по умолчанию.
    

**Обработчики** из Spring обычно **выбрасывают исключение**, **если ошибка не обработана**, чтобы позволить контейнеру решить дальнейшие действия.
По умолчанию такие исключения **логируются контейнером на уровне** `ERROR`. 

Все обработчики ошибок фреймворка расширяют `KafkaExceptionLogLevelAware`, что позволяет вам контролировать уровень, на котором эти исключения логируются.
```java
/**
 * Set the level at which the exception thrown by this handler is logged.
 * @param logLevel the level (default ERROR).
 */
public void setLogLevel(KafkaException.Level logLevel) {
    ...
}
```

## 3. Глобальный обработчик для всех слушателей
```java
@Bean
public KafkaListenerContainerFactory
	<ConcurrentMessageListenerContainer<Integer, String>>
    kafkaListenerContainerFactory() 
{
    var factory = 
	    new ConcurrentKafkaListenerContainerFactory<Integer, String>();
    ...
    
    // Установка ГЛОБАЛЬНОГО обработчика
    factory.setCommonErrorHandler(myErrorHandler);
    return factory;
}
```


Если используется **Spring Boot**, достаточно просто объявить:
```java
@Bean
public CommonErrorHandler errorHandler() {
    return new DefaultErrorHandler();
}
```

> По умолчанию, 
> **если** аннотированный **метод слушателя выбрасывает исключение, оно выбрасывается в контейнер**, и сообщение обрабатывается в соответствии с конфигурацией контейнера.

Контейнер фиксирует любые ожидающие подтверждения смещений перед вызовом обработчика ошибок.

# 3. Обработчики Back Off (Back Off Handlers)
`DefaultErrorHandler` использует `BackOff` (например, `FixedBackOff`) для задержки между повторными попытками.

С версии **2.9** можно менять механизм:
### ✔ BackOffHandler по умолчанию
— просто `Thread.sleep()` на время backOff.

### ✔ ContainerPausingBackOffHandler
— **останавливает контейнер** на время backOff, затем возобновляет его.

Это полезно, когда задержка больше `max.poll.interval.ms`.
