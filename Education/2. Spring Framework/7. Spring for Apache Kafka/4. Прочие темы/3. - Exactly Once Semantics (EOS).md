EOS гарантирует, что цепочка:
```
read → process → write → commit offset
```

произойдёт **строго один раз**, даже если произойдут сбои, рестарты или повторные попытки.

**Важно:**  
Kafka _не может_ гарантировать, что сообщение будет _прочитано_ ровно один раз.  
Чтение всегда **at-least-once** — потому что offset пишется после обработки.

Но **вся цепочка**, включая запись в новые топики, выполняется **точно один раз**.

# 1."Exactly Once Semantics"
Вы можете предоставить контейнеру слушателя экземпляр `KafkaAwareTransactionManager`. 
При такой конфигурации контейнер начинает транзакцию перед вызовом слушателя. 
Любые операции `KafkaTemplate`, выполняемые слушателем, участвуют в транзакции. 
- Если слушатель успешно обрабатывает запись (или несколько записей, при использовании `BatchMessageListener`), контейнер отправляет смещение(я) в транзакцию с помощью `producer.sendOffsetsToTransaction()`, до того как менеджер транзакций зафиксирует транзакцию. 
- Если слушатель выбрасывает исключение, транзакция откатывается, и потребитель перепозиционируется (repositioned) так, чтобы откаченные записи могли быть получены при следующем опросе


## Экземпляр KafkaAwareTransactionManager
Spring предоставляет механизм:
```
KafkaAwareTransactionManager + транзакционный Kafka producer
```

Чтобы добиться EOS Spring Kafka делает так:
### 📌 1. Контейнер слушателя начинает Kafka-транзакцию
**Перед вызовом** @KafkaListener:
```java
beginTransaction()
```

Теперь и consumer (чтение), и producer (запись) «в одной транзакции».

### 📌 2. Вы обрабатываете сообщение
Код слушателя:
```java
@KafkaListener
public void listen(String msg) {
    template.send("out-topic", "processed"); // часть транзакции
}
```
Любой `KafkaTemplate.send()` попадает в транзакцию.

### 📌 3. Если обработка успешна → смещение коммитится _внутри той же транзакции_
Spring **делает это автоматически**:
```java
producer.sendOffsetsToTransaction(offsets, consumerGroupId)

commitTransaction()
```
То есть:
- запись в новый топик
- коммит offset

станут **одной атомарной операцией**.

Брокер гарантирует: либо _оба успех_, либо _оба не выполнены_

### 📌 4. Если слушатель выбросил исключение
Spring делает:
```java
rollbackTransaction()

consumer.seek(back)
```

То есть:
- все посланные сообщения откатываются
- offset НЕ зафиксирован
- чтение будет повторено

Гарантия: **не будет ситуации, когда offset зафиксирован, а запись в новый топик — нет**.
