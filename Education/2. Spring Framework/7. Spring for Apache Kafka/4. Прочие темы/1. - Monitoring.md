# 1) Мониторинг производительности слушателя                (Monitoring Listener Performance)
Начиная с версии 2.3, **контейнер слушателя** **будет автоматически создавать и обновлять Micrometer Timers для слушателя**, если 
- Micrometer обнаружен в classpath, 
- и единственный `MeterRegistry` присутствует в контексте приложения. 

…то **Spring сам начнёт собирать измерения времени работы слушателя**.

Таймеры могут быть отключены путем установки свойства `micrometerEnabled` в `ContainerProperties` в `false`.
```java
containerProperties.setMicrometerEnabled(false);
```

## 1) Два таймера и свойства
Поддерживаются два таймера - один для успешных вызовов слушателя и один для сбоев.

**Таймеры называются** `spring.kafka.listener` и имеют следующие **теги**:
- `name`: (имя бина контейнера)
    
- `result`: успех (success) или сбой (failure)
    
- `exception`: none или `ListenerExecutionFailedException`

### Таймер успешных вызовов
```yaml
spring.kafka.listener
```
Тег `result = success`.

### Таймер вызовов, завершившихся исключением
Имя то же самое, но тег:
```yaml
result = failure
exception = ListenerExecutionFailedException
```
> 💡 Метрики показывают: *сколько времени занимал каждый вызов слушателя* и сколько *их было — успешных и с ошибками.*

## 2) Свои теги
### Статические теги (micrometerTags)
```java
containerProperties.setMicrometerTags(
        Tags.of("service", "billing", "env", "prod")
);
```

### Динамические теги: micrometerTagsProvider (2.9.8+ и 3.0.6+)
Можно генерировать теги **на основе ConsumerRecord**.

Например, добавить тег с номером раздела:
```java
containerProperties.setMicrometerTagsProvider(record ->
        Tags.of("partition", String.valueOf(record.partition()))
);

```

функция получает `ConsumerRecord<?, ?>` и возвращает теги, которые могут быть основаны на этой записи, и объединены с любыми статическими тегами в `micrometerTags`.

## 3) Параллельный контейнер(concurrency > 1)
С параллельным контейнером **таймеры создаются для каждого потока**, и тег `name` получает суффикс `-n`, где n от 0 до concurrency-1.

## 4) Вид метрик в Prometheus
Пример успешной обработки:
```java
spring_kafka_listener_seconds_count{
    name="orderListener-0",
    result="success",
    exception="none"
}  1523
```

Пример ошибки:
```java
spring_kafka_listener_seconds_count{
    name="orderListener-0",
    result="failure",
    exception="ListenerExecutionFailedException"
}  12
```

# 2) Мониторинг производительности KafkaTemplate
Если в проекте есть:
- Micrometer в classpath,
- и в контексте Spring есть единственный `MeterRegistry`,

то **KafkaTemplate автоматически начинает публиковать Micrometer метрики**.

Это значит, что **каждый вызов `kafkaTemplate.send()` измеряется таймером** — сколько он длился, успешно или с ошибкой.

Отключить можно так:
```java
kafkaTemplate.setMicrometerEnabled(false);
```

## 1) Какие таймеры создаются?
### Имя таймера всегда:
```yaml
spring.kafka.template
```
и имеют следующие теги:
- `name`: (имя бина шаблона)
- `result`: успех (success) или сбой (failure)
- `exception`: none или имя класса исключения для сбоев


Создаются **два таймера**:
### Успешные отправки
Тег `result = success`, `exception = none`.

### Ошибочные отправки
Тег `result = failure`, `exception = <ExceptionClass>`  
Например:
```java
exception = TimeoutException
```

## 2) Собственные теги
### Статические теги (micrometerTags)
```java
kafkaTemplate.setMicrometerTags(
        Tags.of("service", "billing")
);
```
Они будут добавлены к каждому измерению.

### Динамические теги (2.9.8+ / 3.0.6+)
Можно добавить функцию, которая получает `ProducerRecord`, из которого извлекаешь нужные поля.
```java
kafkaTemplate.setMicrometerTagsProvider(record ->
        Map.of(
                "topic", record.topic(),
                "key", String.valueOf(record.key())
        )
);
```
функция получает `ProducerRecord<?, ?>` 
и возвращает теги, которые могут быть основаны на этой записи, и объединены с любыми статическими тегами в `micrometerTags`.

Эти теги добавятся **к статическим**, то есть формируют объединённый набор.

# 3) Нативные метрики Micrometer
Spring Kafka умеет собирать **метрики не только на уровне Spring (KafkaTemplate, Listener)**, НО и **нативные Kafka-клиентовые метрики**:
- метрики продюсера (`org.apache.kafka.clients.producer.Producer`)
    
- метрики консюмера (`org.apache.kafka.clients.consumer.Consumer`)
    
- метрики Kafka Streams (`KafkaStreams`)


Эти метрики предоставляет сам Kafka Client — например:
```yaml
kafka.producer.node.incoming.byte.total
kafka.consumer.fetch.latency.avg
kafka.producer.request.size.max
kafka.consumer.poll.idle.ratio
```

Spring Kafka **автоматически подключает эти метрики к Micrometer**, _если включены специальные listeners у фабрик продюсеров и консюмеров_.

## 1) Подключение фабрик слушателей метрик
Для этого нужно добавить **слушателей фабрики**:
- `MicrometerConsumerListener` → для ConsumerFactory
    
- `MicrometerProducerListener` → для ProducerFactory

Пример:
```java
@Bean
public ConsumerFactory<String, String> myConsumerFactory() {
    Map<String, Object> configs = consumerConfigs();

    DefaultKafkaConsumerFactory<String, String> cf =
            new DefaultKafkaConsumerFactory<>(configs);

    cf.addListener(
        new MicrometerConsumerListener<>(
            meterRegistry(),
            List.of(
	            new ImmutableTag("customTag", "customTagValue")
	        )
        )
    );

    return cf;
}

@Bean
public ProducerFactory<String, String> myProducerFactory() {
    Map<String, Object> configs = producerConfigs();
    configs.put(ProducerConfig.CLIENT_ID_CONFIG, "myClientId");
    ...
    DefaultKafkaProducerFactory<String, String> pf = 
	    new DefaultKafkaProducerFactory<>(configs);
    ...
    
    pf.addListener(
	    new MicrometerProducerListener<String, String>(
			meterRegistry(),
            Collections.singletonList(
	            new ImmutableTag(
		            "customTag", "customTagValue")
		        )
		    )
		);
    ...
    return pf;
}
```

### Принцип работы
Когда твой ConsumerFactory или ProducerFactory:
- **создаёт** новый Consumer/Producer → listener вызывает _bind_
- **закрывает** Consumer/Producer → listener вызывает _unbind_

То есть **метрики Kafka-клиента автоматически подключаются к Micrometer**


В результате в `/actuator/metrics` появляются десятки Kafka метрик, например:
```css
kafka.consumer.fetch.rate
kafka.producer.request.size.avg
kafka.consumer.records.lag.max
...
```

## 2) Что такое тег `spring.id`?
Идентификатор потребителя/производителя, переданный слушателю, добавляется к тегам измерителя с именем тега `spring.id`.

Каждому Kafka-клиенту назначается Kafka client id, например:
```java
myClientId-1
```

Listener формирует из него тег:
```yaml
spring.id = <factoryName>.<clientId>
```

Например:
```yaml
spring.id="myProducerFactory.myClientId-1"
```

### Пример получения одной из метрик Kafka:
```java
double count = this.meterRegistry.get("kafka.producer.node.incoming.byte.total")
		.tag("customTag", "customTagValue")
        .tag("spring.id", "myProducerFactory.myClientId-1")
        .functionCounter()
        .count();
```

## 3) Micrometer Listener для Kafka Streams
Для Kafka Streams тоже есть аналогичный слушатель:
- `KafkaStreamsMicrometerListener`

Он делает то же самое, что Producer/Consumer listeners — привязывает KafkaStreams метрики (узлы, задачи, состояние, буферы, лаги) к Micrometer.

## 4) Абстрактный класс KafkaMetricsSupport
В версии 3.3 добавили удобный суперкласс:
```java
KafkaMetricsSupport
```

Он умеет:
- привязывать Kafka метрики (`io.micrometer.core.instrument.binder.kafka.KafkaMetrics`)
    
- отвязывать их при закрытии
    
- работать с любым Kafka клиентом (не только Producer, Consumer или Streams)  

📌 Поэтому он стал базовым классом для:
- `MicrometerConsumerListener`
    
- `MicrometerProducerListener`
    
- `KafkaStreamsMicrometerListener`


Ты можешь **расширить KafkaMetricsSupport** для любых нестандартных клиентских сценариев.

### Пример кастомного класса
```java
public class MyCustomKafkaMetrics extends KafkaMetricsSupport {

    public MyCustomKafkaMetrics(MeterRegistry registry) {
        super(registry);
    }

    public void bind(Producer<byte[], byte[]> producer) {
        bindClient(producer);
    }

    public void unbind(Producer<byte[], byte[]> producer) {
        unbindClient(producer);
    }
}
```

# 4) Micrometer Observation (Наблюдение Micrometer)
**Micrometer Observation** — это новая система в Micrometer 2.x, которая объединяет:
- **метрики** (timers, counters)
    
- **трассировку** (tracing, OpenTelemetry)
    
- **логирование событий**

в единую концепцию **"наблюдения"** (Observation).


С точки зрения Kafka:
> Observation = единица измерения обработки одного сообщения  
> (для продюсера — отправка, для консюмера — получение)

### Что изменилось в Spring Kafka 3.0+?
#### KafkaTemplate и KafkaListener контейнеры могут работать через Observations

Это включается явно:
- KafkaTemplate:
```java
template.setObservationEnabled(true);
```

- Listener Container Properties:
```java
containerProps.setObservationEnabled(true);
```

### ✔Когда Observation включён

Micrometer Timers **отключаются**, потому что:

- теперь время, исключения, теги и т.п. собираются через Observations
    
- Observations создают и таймеры, и трассировки вместе

## 1) Batch Listener НЕ поддерживает Observation
Если у тебя batch-listener:
```java
@KafkaListener(batch = "true")
public void listen(List<ConsumerRecord> batch) { ... }
```
❌ Observation работать **не будет**  
Spring автоматически ловит это и переключает это на **старые Micrometer timers**

Причина:
> Observation привязан к одному событию (одному рекорду),  
> а batch-listener получает пачку, где нет 1:1 соответствия.

## 2) Собственные теги
Чтобы добавить теги к таймерам/трассировкам, настройте Observation Conventions:
- **KafkaTemplate**ObservationConvention
- **KafkaListener**ObservationConvention


Они позволяют:
- добавлять свои теги
- переопределять имена таймеров
- вставлять бизнес-информацию 

### Стандартные реализации
Spring предоставляет стандартные реализации:
- `DefaultKafkaTemplateObservationConvention`
- `DefaultKafkaListenerObservationConvention`

От них можно наследоваться
```java
class MyListenerConvention 
extends DefaultKafkaListenerObservationConvention { 
	... 
}
```

> Реализации по умолчанию добавляют 
> - тег `bean.name` для наблюдений шаблона 
> - тег `listener.id` для контейнеров.

### Динамические теги
Начиная с 3.0.6 — можно добавлять **динамические теги** основанные на информации в записях потребителя или производителя.

Для этого добавьте пользовательскую *KafkaListenerObservationConvention* и/или *KafkaTemplateObservationConvention* 
в свойства 
- контейнера слушателя 
- или KafkaTemplate соответственно. 

Свойство `record` в обоих контекстах наблюдения содержит `ConsumerRecord` или `ProducerRecord` соответственно.

Теперь можно использовать сам `ConsumerRecord` или `ProducerRecord`:
```java
@Override
public KeyValues getLowCardinalityKeyValues(KafkaRecordReceiverContext context) {
    return KeyValues.of(
        "topic", context.getRecord().topic(),
        "key", String.valueOf(context.getRecord().key())
    );
}
```

# 5) Наблюдения за слушателем пачек (Batch Listener Observations)
По умолчанию batch listener не создаёт Observation — слишком много проблем с 1:N записями.

Но можно включить:
```java
factory.getContainerProperties()
	.setRecordObservationsInBatch(true);
```

```java
@Bean
ConcurrentKafkaListenerContainerFactory<?, ?> kafkaListenerContainerFactory(
	ConcurrentKafkaListenerContainerFactoryConfigurer configurer,
    ConsumerFactory<Object, Object> kafkaConsumerFactory
) {
    ConcurrentKafkaListenerContainerFactory<Object, Object> 
	    factory = 
		    new ConcurrentKafkaListenerContainerFactory<>();
    
    configurer.configure(factory, kafkaConsumerFactory);
    
    factory
	    .getContainerProperties()
	    .setRecordObservationsInBatch(true);
    return factory;
}
```

Тогда:
- для **каждого ConsumerRecord в пачке** создаётся отдельное Observation
    
- Observation **НЕ передаётся** в метод слушателя
    
- но доступно в контексте (`ObservationRegistry.getCurrentObservation()`)

## 1) Как это работает внутри?

Spring создаёт Observation вокруг:

### 🔹 Producer
вокруг `KafkaTemplate.doSend()`:
- start observation → send → end observation
    
- собирает:
    - время отправки
    - исключения
    - теги producer'а
    - динамические теги

### 🔹 Consumer
вокруг получения записи:
- start observation → обработка записи → end observation

Если batch mode с `recordObservationsInBatch = true`, то:
- для каждой записи создаётся Observation
    
- но слушатель всё равно получает List
