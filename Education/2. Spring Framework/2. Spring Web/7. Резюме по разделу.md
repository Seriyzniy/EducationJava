Spring MVC - один из компонентов Spring Framework, который позволяет разрабатывать web приложения.

Spring MVC предполагает разработку web-приложений с использованием архитектуры ==Model-View-Controller==

# 1. Модель MVC
![[Pasted image 20241022183151.png]]
### Model
Получает данные от контроллера, и передает их в представление при помощи создания атрибутов, которые можно использовать в представлении Views
### View
Используется для представления данных пользователю
### Controller
Обрабатывает действия пользователя (запросы от пользователя), обрабатывает данные и передает их в модель

### Порядок работы системы MVC:
- Пользователь отправил запрос HTTP
	
- Сервер принимает запрос и отправляет на Spring MVC приложение 
	
- Запрос попадает в DispatcherServlet
	
- ==DispatcherServlet отправляет запрос КОНКРЕТНОМУ контроллеру, ответственному за данный запрос==
	
- Контроллер обращается к модели данных
	
- Модель данных выбирает информацию, в соответствии с запросом, из БД
	
- Модель данных передается контроллеру
	
- Контроллер отправляет ответ

### 1. Интерфейс Model
> [!NOTE] Interface Model
> Model является интерфейсом
> ```java
> public interface Model {
> ```
> 
> Поэтому НЕЛЬЗЯ сделать так:
> ```java
> @GetMapping("/greeting") 
> public String showGreeting() { 
> 	Model model = new Model();  //Так НЕЛЬЗЯ
> 	
> 	model.addAttribute("message", "Hello, World!"); 
> 	return "greeting"; // Имя представления }
> ```

#### Варианты передачи/использования интерфейса Model
- Передача в качестве аргумента (Spring сам внедрит необходимый объект)
```java
@GetMapping("/greeting") 
public String showGreeting(Model model) { 
	model.addAttribute("message", "Hello, World!"); 
	return "greeting"; // Имя представления 
}
```

- Использование класса `ModelMap`
```java
@GetMapping("/greeting") 
public String showGreeting() { 
	ModelMap model = new ModelMap();
	model.addAttribute("message", "Hello, World!"); 
	return "greeting"; // Имя представления 
}
```

- Использование класса `ModelAndView`, 
Класс ==объединяет== в себе ==модель и представление==. Его можно использовать, чтобы одновременно передать данные и указать, какое представление нужно отобразить.
```java
@GetMapping("/welcome") 
public ModelAndView welcome() { 
	ModelAndView modelAndView = new ModelAndView("welcomeView"); 
	
	modelAndView.addObject("name", "John"); 
	return modelAndView; 
}
```

### 2. Аннотация `@ModelAtribute`

> [!NOTE] `@ModelAttribute`
> Аннотация может аннотировать:
> - **Метод** (создает пару ==ключ-значение:== headerMessage="Welcome to our website!")
>  ```java
> @ModelAttribute("headerMessage")
> public String populateHeaderMessage() {
> 	return "Welcome to our website!";
> }
> ```
> - **Аргумент метода**
> ```java
> @PostMapping()
> public String create(@ModelAttribute("person") Person person) {
> 	personDAO.save(person);
> 	return "redirect:/people";
> }
> ```
#### Аннотирование МЕТОДА
![[Pasted image 20241117133630.png]]

> [!NOTE] Аннотация в МЕТОДЕ
> Создаст `пару ключ-значение` для `ВСЕХ моделей` данного `контроллера` со значением по умолчанию
> 
> Можно создавать ==любые пары== `ключ-значение`, не только строковые

#### Аннотация в ПАРАМЕТРЕ метода
![[Pasted image 20241117135217.png]]

> [!NOTE] Аннотация в ПАРАМЕТРЕ метода
> Ранее мы создавали контроллер для POST запроса, который берет параметры из HTML-формы и по этим параметрам:
> - Создает пустой объект класса `Person`
> - Устанавливает значения, полученные из формы, в поля объекта
> - Добавляет полученный объект в атрибут модели
> 
> ==Все это== теперь ==делает аннотация== `@ModelAttribute`, ==если передать ее в качестве параметра== метода
> 
> Если в HTML-форме не передавать параметры, то поскольку `Person` это класс, то установятся значения по умолчанию

# 3. Протокол HTTP

### 1. Доступные `Get` и `Post` методы
> [!NOTE] Протокол HTTP
> Протокол HTTP предлагает множество методов для работы с сервером, ==ОДНАКО при отправке данных через HTML-формы== (HTML - 5 version) ==ДОСТУПНЫ только 2 метода== GET и POST
> Все браузеры работают с HTML-5
#### Решение проблемы с 2-мя доступными методами

![[Pasted image 20241118144738.png]]

`PATCH`, `DELETE`, `PUT` запросы передаются с помощью `POST` запроса, НО ==в СКРЫТОМ поле ==`_method` указывается желаемый HTTP-метод 

Поскольку при нажатии на кнопку в форме (кнопка отправки), все введенные данные в полях отправятся на сервер. 
Поэтому ==скрытое поле== также отправится, и в нем будет пара `_method = "patch"`.

#### Принятие данных со скрытым полем

На стороне Spring приложения, чтение скрытого поля осуществляет ФИЛЬТР

> [!NOTE] Фильтр для чтения скрытого поля
> ==Фильтр== - объект, который ==перехватывает все входящие HTTP-запросы==.
>  
>  В данном случае фильтр используется для того ==чтобы прочитать ЗНАЧЕНИЕ скрытого поля== `_method` в поступающих HTTP-запросах (если такие есть)

```java
private void registerHiddenFieldFilter(ServletContext aContext) {
	aContext.addFilter("hiddenHttpMethodFilter",
		new HiddenHttpMethodFilter()).addMappingForUrlPatterns(null ,true, "/*");
}
```
Хотя мы и реализовали ==метод для добавления фильтра==, ==этот метод  необходимо вызвать== при старте сервера (диспетчер сервлета)
```java
@Override
public void onStartup(ServletContext aServletContext) throws ServletException {
	super.onStartup(aServletContext);
	registerHiddenFieldFilter(aServletContext);
}
```

# 4. CRUD и REST

> [!NOTE] CRUD
> ==CRUD== - классификация функций по манипуляциям с БД, т.е 4 базовые `функции`, используемые `при работе с БД`
> 
> ==REST== - `паттерн` проектирования `web-приложений`.
> - Описывает как при помощи HTTP взаимодействовать с сервером для выполнения основных операций CRUD (создание/добавление, чтение, изменение, удаление данных)
> - Иногда, помимо акронима REST, можно встретить слово ==**RESTful**==. Это не термин, но в сообществе его применяют к веб-сервисам, которые соответствуют REST-архитектуре.


> [!NOTE] Требования REST
> Всего в REST есть шесть требований к проектированию API. Пять из них обязательные, одно — опциональное:
>- Клиент-серверная модель (client-server model).
>- Отсутствие состояния (statelessness).
>	- На сервере не хранится никаких данных о прошлых взаимодействиях с клиентом — > каждый запрос должен содержать всю информацию для его обработки.
>- Кэширование (cacheability).
>	- Сохранение части данных у клиента или на промежуточных серверах.
>- Единообразие интерфейса (uniform interface).
>	- Ответ на запросы к новому ресурсу должен приходить в том же формате, что и к старым
>- Многоуровневая система (layered system).
>	- Несколько промежуточных узлов, выполняющих вспомогательные функции, — **прокси-серверы**.
>- Код по требованию (code on demand) — необязательно.
