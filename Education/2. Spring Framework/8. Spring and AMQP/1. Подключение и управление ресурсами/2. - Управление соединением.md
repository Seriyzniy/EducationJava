# 1. AddressResolver
**Начиная с версии 2.1.15** можно использовать **`AddressResolver`** для определения адреса (или адресов) подключения.

Использование `AddressResolver`:
- **переопределяет** любые настройки:
    - `addresses`
    - `host` / `port`


То есть если задан `AddressResolver`, все остальные способы задания адреса подключения **игнорируются**.

📌 Назначение:
- динамическое разрешение адресов
- DNS / сервис-дискавери 
- кастомная логика выбора узла RabbitMQ

# 2. Именование соединений (Naming Connections)
**Начиная с версии 1.7**, предоставляется интерфейс **`ConnectionNameStrategy`**, который можно внедрить в `AbstractConnectionFactory`.

Сгенерированное имя:
- используется для **идентификации соединения на стороне RabbitMQ**
- отображается в **RabbitMQ Management UI** (если сервер поддерживает эту возможность)


⚠️ Важно:
- имя **не обязано быть уникальным**
    
- оно **не является идентификатором соединения**
    
- **нельзя использовать** его, например, в HTTP API
    
- имя должно быть **человекочитаемым**
    

Это имя:
- передаётся в `client_properties`
- хранится под ключом `connection_name`

Например
```java
connectionFactory.setConnectionNameStrategy(
    cf -> "MY_CONNECTION"
);
```
Аргумент `ConnectionFactory` можно использовать:
- для различения соединений
- для применения логики именования

## Поведение по умолчанию
Если стратегия не задана:
- используется:
    - `beanName` фабрики
    - hex-представление объекта
    - внутренний счётчик
    
- из этого формируется значение `connection_name`

В XML-конфигурации `<rabbit:connection-factory>` также есть атрибут `connection-name-strategy`.

## SimplePropertyValueConnectionNameStrategy 
Существует **готовая реализация**:  
**`SimplePropertyValueConnectionNameStrategy`**

Она:
- берёт имя соединения из свойства приложения

Пример:
```java
@Bean
public SimplePropertyValueConnectionNameStrategy cns() {
    return new SimplePropertyValueConnectionNameStrategy(
        "spring.application.name"
    );
}
```

```java
@Bean
public ConnectionFactory rabbitConnectionFactory(
        ConnectionNameStrategy cns) {

    CachingConnectionFactory connectionFactory =
        new CachingConnectionFactory();
    ...
    connectionFactory.setConnectionNameStrategy(cns);
    
    return connectionFactory;
}
```

📌 Требование:
- свойство **должно существовать** в `Environment` приложения

### Spring Boot
При использовании Spring Boot:
- достаточно объявить `ConnectionNameStrategy` как `@Bean`
- Boot **автоматически обнаружит** его
- и внедрит в `ConnectionFactory`

# 3. Заблокированные соединения и ограничения ресурсов
Соединение может быть **заблокировано брокером**, если на стороне RabbitMQ сработал **Memory Alarm**

**Начиная с версии 2.0**:
- `org.springframework.amqp.rabbit.connection.Connection`  
    может принимать:
    - `com.rabbitmq.client.BlockedListener`

Это позволяет получать уведомления:
- о блокировке соединения
- о снятии блокировки


Дополнительно:
- `AbstractConnectionFactory` публикует события:
    - `ConnectionBlockedEvent`
        
    - `ConnectionUnblockedEvent`


Через них можно:
- реализовать бизнес-логику реакции
- предпринимать корректирующие действия

## 1. Проблема одного соединения (deadlock)
Если приложение использует:
- **один `CachingConnectionFactory`** (поведение по умолчанию в Spring Boot)

И брокер блокирует соединение
👉 приложение **полностью перестаёт работать**.

Это может произойти если в одном приложении:
- есть **producers**
- есть **consumers**

И происходит следующее:
- producers блокируют соединение (из-за нехватки ресурсов)
- consumers не могут освободить ресурсы
	
- потому что соединение заблокировано

### Решение
Использовать:
- **два отдельных `CachingConnectionFactory`**
    - один — для producers
    - второй — для consumers
- с одинаковыми настройками

❗ Исключение:
- **транзакционные producers**, выполняемые в consumer-потоке
- они **должны использовать тот же Channel**, что и consumer
- для них отдельная фабрика невозможна

## 2. Отдельное соединение для RabbitTemplate
**Начиная с версии 2.0.2**:
- `RabbitTemplate` может:
    - автоматически *использовать вторую фабрику соединений*
    - если **не используются транзакции**

> Подробнее: _Using a Separate Connection_


Имя publisher-соединения:
- формируется так же, как основное
- к имени добавляется суффикс `.publisher`

## 3. Новое исключение AmqpResourceNotAvailableException
**Начиная с версии 1.7.7** добавлено исключение:
```java
AmqpResourceNotAvailableException
```

Оно выбрасывается, если:
- `SimpleConnection.createChannel()` **не может создать канал**
    
- например:
    - *достигнут лимит* `channelMax`
    - *нет свободных каналов в кеше*

Это исключение:
- можно использовать в `RetryPolicy`
- чтобы выполнить повторную попытку
- с задержкой (backoff)

# 4. Настройка БАЗОВОГО RabbitMQ ConnectionFactory
`CachingConnectionFactory` использует:
- `com.rabbitmq.client.ConnectionFactory` (клиент RabbitMQ)


Некоторые свойства:
- автоматически прокидываются:
    - host
    - port
    - username
    - password
    - requestedHeartBeat
    - connectionTimeout

## 1. Указание дополнительных свойств
Например: `clientProperties`

Нужно:
1. создать свой `RabbitMQ ConnectionFactory`
    
2. передать его в конструктор `CachingConnectionFactory`


При использовании XML ссылка передаётся через `connection-factory` атрибут
```xml
<rabbit:connection-factory
    id="connectionFactory"
    connection-factory="rabbitConnectionFactory"/>
```

## 2. Добавление пользовательских свойств клиентского подключения
`CachingConnectionFactory` теперь *позволяет* вам *получить доступ к базовой фабрике подключений*, что *даёт возможность*, например, **устанавливать пользовательские свойства клиента**. 

В следующем примере показано, как это сделать:
```java
connectionFactory
	.getRabbitConnectionFactory()
	.getClientProperties()
		.put("thing1", "thing2");
```
Эти свойства отображаются в административном интерфейсе RabbitMQ при просмотре подключения.

### Автоматическое восстановление соединений (auto-recovery)
Клиент RabbitMQ **версии 4.0.x** включает auto-recovery **по умолчанию**


Spring AMQP:
- *совместим с этим*
- НО имеет **собственные механизмы восстановления**

#### Рекомендация

**Отключить auto-recovery RabbitMQ client**, потому что:
- могут возникать:
    - `AutoRecoverConnectionNotCurrentlyOpenException`
    
- особенно:
    - при использовании `RetryTemplate`
    - при failover в кластере


Причина:
- клиент восстанавливается по таймеру
- Spring может восстановить соединение **быстрее**

### Поведение Spring AMQP
- **Начиная с версии 1.7.1**
    
- Spring AMQP:
    - **отключает auto-recovery**
        
    - если ты **не создаёшь RabbitMQ ConnectionFactory вручную**


Также:
- `RabbitConnectionFactoryBean`
    - по умолчанию отключает auto-recovery

# 5. Publisher Confirms and Returns
**Подтверждённые** (с корреляцией) **и возвращённые сообщения** *поддерживаются при установке у* `CachingConnectionFactory` 
- свойства `publisherConfirmType` в значение `ConfirmType.CORRELATED`
- и свойства `publisherReturns` в `true`.

Когда эти параметры включены, экземпляры `Channel`, создаваемые фабрикой, **оборачиваются** в `PublisherCallbackChannel`. *Этот канал используется* **для обеспечения работы callback-механизмов**. 

После получения такого канала *клиент может зарегистрировать* `PublisherCallbackChannel.Listener`. 

Реализация `PublisherCallbackChannel` содержит логику, которая направляет подтверждение (confirm) или возврат (return) сообщения соответствующему слушателю. Более подробно эти возможности рассматриваются в следующих разделах документации.

Дополнительную информацию см. в разделах _Correlated Publisher Confirms and Returns_ и _simplePublisherConfirms in Scoped Operations_. Для общего понимания механизма подтверждений полезна статья команды RabbitMQ _Introducing Publisher Confirms_.

# 6. Слушатели соединений и каналов (Connection and Channel Listeners)
Фабрика соединений поддерживает регистрацию реализаций:
- `ConnectionListener` 
- и `ChannelListener`. 

Это **позволяет ПОЛУЧАТЬ УВЕДОМЛЕНИЯ О СОБЫТИЯХ, связанных с соединениями и каналами.**

Начиная с версии 2.0, объект `org.springframework.amqp.rabbit.connection.Connection` **может принимать реализации** `com.rabbitmq.client.BlockedListener`, позволяя приложению получать уведомления о блокировке и разблокировке соединения со стороны брокера (например, из-за memory alarm).

Интерфейс `ChannelListener` **позволяет получать уведомления о создании канала**, включая информацию о том, является ли он транзакционным, а также об аварийном завершении работы канала. 
> Один из сценариев, где это может быть полезно, описан в разделе _Publishing is Asynchronous — How to Detect Successes and Failures_.

# 7. Логирование закрытия каналов (Logging Channel Close Events)
Начиная с версии 1.5, был *добавлен механизм, позволяющий управлять* **уровнями логирования событий закрытия каналов**.

`AbstractConnectionFactory` использует стратегию логирования по умолчанию. 
- *При нормальном закрытии* канала (код 200 OK) сообщения в *лог не пишутся*. 
- Если *канал закрывается из-за неудачной пассивной декларации* очереди, событие *логируется на уровне DEBUG*. 
- *Закрытие канала из-за отказа* `basic.consume`, связанного с эксклюзивным consumer, также *логируется на уровне DEBUG* (начиная с версии 3.1; ранее использовался уровень INFO). 
- *Все остальные причины закрытия* каналов логируются *на уровне ERROR*.

> При необходимости это поведение **можно изменить, внедрив собственную реализацию** `ConditionalExceptionLogger` в `CachingConnectionFactory` через свойство `closeExceptionLogger`. 

Кроме того, класс `AbstractConnectionFactory.DefaultChannelCloseLogger` теперь является публичным и может быть расширен.

Дополнительную информацию см. в разделе _Consumer Events_.

# 8. Runtime Cache Properties
Начиная с версии 1.6, `CachingConnectionFactory` теперь **предоставляет статистику кэша** через метод `getCacheProperties()`. 

Эти статистические данные можно использовать для настройки кэша, чтобы оптимизировать его в рабочей среде. 

Например, 
максимальные значения (high water marks) *могут применяться для определения, следует ли увеличивать размер кэша*. Если значение равно размеру кэша, возможно, стоит рассмотреть дальнейшее его увеличение.

В следующей таблице описаны свойства CacheMode.CHANNEL:

## Таблица 1. Свойства кеша для `CacheMode.CHANNEL`

|Свойство|Значение|
|---|---|
|`connectionName`|Имя соединения, сгенерированное `ConnectionNameStrategy`.|
|`channelCacheSize`|Текущее сконфигурированное максимальное количество каналов, которые могут находиться в простое (в кеше).|
|`localPort`|Локальный порт соединения (если доступен). Может использоваться для сопоставления соединений и каналов в RabbitMQ Admin UI.|
|`idleChannelsTx`|Количество транзакционных каналов, которые в данный момент находятся в простое (закешированы).|
|`idleChannelsNotTx`|Количество нетранзакционных каналов, которые в данный момент находятся в простое (закешированы).|
|`idleChannelsTxHighWater`|Максимальное количество транзакционных каналов, которые одновременно находились в простое (в кеше).|
|`idleChannelsNotTxHighWater`|Максимальное количество нетранзакционных каналов, которые одновременно находились в простое (в кеше).|
