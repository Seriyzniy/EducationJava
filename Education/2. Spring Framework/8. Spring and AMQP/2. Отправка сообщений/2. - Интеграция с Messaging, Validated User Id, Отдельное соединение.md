# 1. Интеграция с Messaging
Начиная с версии 1.4, `RabbitMessagingTemplate` (построенный поверх `RabbitTemplate`) обеспечивает интеграцию с абстракцией messaging Spring Framework — то есть с `org.springframework.messaging.Message`. 

Это позволяет ==отправлять и получать сообщения, используя абстракцию== `spring-messaging Message<?>`.

Эта абстракция используется другими проектами Spring, такими как Spring Integration и поддержка STOMP в Spring. 

## 1. Два конвертора сообщений
В процессе задействовано два конвертера сообщений: 
- один для преобразования между `spring-messaging Message<?>` и абстракцией сообщения Spring AMQP (`Message`), 
- а другой — для преобразования между абстракцией сообщения Spring AMQP и форматом, требуемым базовой клиентской библиотекой RabbitMQ.

По умолчанию *полезная нагрузка сообщения* (payload) **преобразуется конвертером сообщений, предоставленным экземпляром** `RabbitTemplate`. 

### Пользовательский конвертер
В качестве альтернативы вы можете внедрить пользовательский `MessagingMessageConverter` с каким-либо другим конвертером полезной нагрузки, как показано в следующем примере:
```java
MessagingMessageConverter amqpMessageConverter = 
	new MessagingMessageConverter();
	
amqpMessageConverter.setPayloadConverter(myPayloadConverter);


rabbitMessagingTemplate.setAmqpMessageConverter(amqpMessageConverter);
```

# 2. Проверенный идентификатор пользователя (Validated User Id)
Начиная с версии 1.6, шаблон теперь поддерживает `user-id-expression` (`userIdExpression` при использовании Java-конфигурации). 

При отправке сообщения свойство идентификатора пользователя (user id) устанавливается (если оно еще не установлено) после вычисления этого выражения. 
Корневым объектом для вычисления является отправляемое сообщение.

В следующих примерах показано, как использовать атрибут `user-id-expression`:
```java
<rabbit:template ... user-id-expression="'guest'" />

<rabbit:template ... user-id-expression="@myConnectionFactory.username" />
```
- Первый пример использует литеральное выражение. 
- Второй получает свойство `username` из бина фабрики соединений в контексте приложения.

# 3. Использование отдельного соединения 
Начиная с версии 2.0.2, вы можете установить свойство `usePublisherConnection` в `true`, **чтобы использовать соединение, отличное от того, которое используется контейнерами слушателей** (listener containers), когда это возможно.

Это **позволяет избежать блокировки потребителей**, когда производитель заблокирован по какой-либо причине.

==Фабрики соединений поддерживают вторую внутреннюю фабрику соединений для этой цели==; по умолчанию она того же типа, что и основная фабрика, но можно явно задать другой тип фабрики для публикации, если нужно. 
Если шаблон Rabbit выполняется в транзакции, запущенной контейнером слушателя, используется канал контейнера, независимо от этой настройки.

> Как правило, не следует использовать `RabbitAdmin` с шаблоном, у которого это свойство установлено в `true`.
 
**Используйте конструктор** `RabbitAdmin`, **который принимает фабрику соединений.** 
Если вы используете другой конструктор, принимающий шаблон, убедитесь, что свойство шаблона установлено в `false`. 
Это связано с тем, что ==админ часто используется для объявления очередей для контейнеров слушателей.== 
- *Использование шаблона с установленным свойством* `true` означает, *что эксклюзивные очереди* (такие как `AnonymousQueue`) **будут объявлены на другом соединении**, *отличном от используемого контейнерами слушателей*. В этом случае очереди не могут быть использованы контейнерами.
