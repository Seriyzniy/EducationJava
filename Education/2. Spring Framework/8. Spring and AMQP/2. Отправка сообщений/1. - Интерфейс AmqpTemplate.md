## 1. Интерфейс AmqpTemplate
Как и многие другие высокоуровневые абстракции, предоставляемые Spring Framework и связанными проектами, Spring AMQP предоставляет "шаблон" (template), который играет центральную роль.

> [!NOTE] Интерфейс **AmqpTemplate** 
> это центральная высокоуровневая абстракция Spring AMQP для отправки и получения сообщений.
> 

Он **определяет общий набор операций**, *не зависящий от конкретной реализации брокера* (отсюда название _AMQP_). 

Поскольку AMQP — это wire-level протокол (протокол на уровне передачи данных), каждая реализация использует собственную клиентскую библиотеку. Поэтому конкретные реализации `AmqpTemplate` привязаны к клиентам AMQP.

## 2. Реализация - RabbitTemplate
На данный момент существует **ОДНА РЕАЛИЗАЦИЯ** — ==RabbitTemplate==, **основанная на RabbitMQ.** В коде часто используется интерфейс `AmqpTemplate`, но при конфигурации обычно фигурирует конкретная реализация (`RabbitTemplate`).

Интерфейс охватывает базовые операции отправки и получения сообщений, которые подробно рассматриваются отдельно.  
> Также см. **AsyncRabbitTemplate**.

# 1. Добавление возможности повторных попыток (Retry Capabilities)
Начиная с версии 1.3, вы **можете настроить** `RabbitTemplate` на использование `RetryTemplate` *для помощи в обработке проблем с подключением к брокеру.* 

> [!NOTE] **Retry**
> Для получения дополнительной информации см. "Основная поддержка повторных попыток (Retry) в Spring Framework".
> [org.springframework.core.retry (Spring Framework 7.0.0 API)](https://docs.spring.io/spring-framework/docs/7.0.0/javadoc-api/org/springframework/core/retry/package-summary.html)

Ниже приведен лишь один пример, использующий политику экспоненциальной задержки (exponential back off) и стандартную политику `SimpleRetryPolicy`, которая делает три попытки, прежде чем выбросить исключение вызывающей стороне.
```java
public RabbitTemplate rabbitTemplate() {
    RabbitTemplate template = new RabbitTemplate(connectionFactory());
    
    RetryPolicy retryPolicy = RetryPolicy.builder()
            .delay(Duration.ofMillis(500))
            .multiplier(2.0)
            .maxDelay(Duration.ofSeconds(10))
            .build();
            
    template.setRetryTemplate(new RetryTemplate(retryPolicy));
    
    return template;
}
```


Начиная с версии 1.4, помимо свойства `retryTemplate`, в `RabbitTemplate` поддерживается опция `recoveryCallback`. Она используется в качестве второго аргумента для
```java
RetryTemplate.execute(
	RetryCallback<T, E> retryCallback, 
	RecoveryCallback<T> recoveryCallback)
```

`RecoveryCallback` несколько ограничен, так как контекст повторных попыток содержит только поле `lastThrowable`. 

**Для более сложных случаев** использования **следует использовать ВНЕШНИЙ** `RetryTemplate`, чтобы можно было передать дополнительную информацию в `RecoveryCallback` через атрибуты контекста. 
Следующий пример показывает, как это сделать:
```java
retryTemplate.execute(
    new RetryCallback<Object, Exception>() {
        @Override
        public Object doWithRetry(RetryContext context) throws Exception {
            context.setAttribute("message", message);
            return rabbitTemplate.convertAndSend(exchange, routingKey, message);
        }

    }, new RecoveryCallback<Object>() {

        @Override
        public Object recover(RetryContext context) throws Exception {
            Object message = context.getAttribute("message");
            Throwable t = context.getLastThrowable();
            // Сделать что-то с сообщением
            return null;
        }
    });
}
```
**В этом случае вы не будете внедрять** `RetryTemplate` в `RabbitTemplate`.

# 2. Публикация является асинхронной — Как обнаружить успехи и неудачи

> [!NOTE] Заметка
> **Публикация сообщений** — это *асинхронный механизм*, и по умолчанию сообщения, которые не могут быть маршрутизированы, отбрасываются RabbitMQ.

**Для успешной публикации** вы *можете получить асинхронное* **ПОДТВЕРЖДЕНИЕ**, как описано в разделе "[Correlated Publisher Confirms and Returns](https://docs.spring.io/spring-amqp/reference/amqp/template.html#template-confirms)"

Рассмотрим **два сценария неудачи**:
1. Публикация в обменник (exchange), но нет соответствующей очереди назначения.
    
2. Публикация в несуществующий обменник.


*Первый случай* покрывается возвратами от издателя (publisher returns), как описано в разделе "Связанные подтверждения и возвраты от издателя".("[Correlated Publisher Confirms and Returns](https://docs.spring.io/spring-amqp/reference/amqp/template.html#template-confirms)")

Для *второго случая* **сообщение отбрасывается**, и возврат не генерируется. **Базовый канал закрывается с исключением**. 

> По умолчанию это *исключение логируется*, 
> НО 
> вы можете зарегистрировать `ChannelListener` в `CachingConnectionFactory` для получения уведомлений о таких событиях. 

Следующий пример показывает, как добавить `ConnectionListener`:
```java
this.connectionFactory.addConnectionListener(new ConnectionListener() {

    @Override
    public void onCreate(Connection connection) {
    }

    @Override
    public void onShutDown(ShutdownSignalException signal) {
        ...
    }

});
```
Вы можете изучить свойство `reason` сигнала, чтобы определить возникшую проблему.

Чтобы обнаружить исключение в потоке отправки, вы можете установить `setChannelTransacted(true)` в `RabbitTemplate`, и исключение будет обнаружено при вызове `txCommit()`. 

Однако **транзакции значительно снижают производительность**, поэтому тщательно обдумайте это, прежде чем включать транзакции только для этого одного случая использования.

# 3. Связанные подтверждения и возвраты от издателя (Correlated Publisher Confirms and Returns)

> [!NOTE] Поддержка подтверждений
> Реализация `AmqpTemplate` в виде `RabbitTemplate` поддерживает подтверждения (confirms) и возвраты (returns) от издателя.

## 1. Возвращаемые сообщения
Для возвращаемых сообщений **свойство** `mandatory` **шаблона должно быть установлено** в `true`, или `mandatory-expression` должно вычисляться в `true` для конкретного сообщения. 

Для работы этой функции требуется `CachingConnectionFactory`, у которого свойство `publisherReturns` установлено в `true` (см. "Подтверждения и возвраты от издателя"). 

**Возвраты отправляются клиенту путем регистрации** `RabbitTemplate.ReturnsCallback` через вызов `setReturnsCallback(ReturnsCallback callback)`. 

Колбэк должен реализовывать следующий метод:
```java
void returnedMessage(ReturnedMessage returned);
```

`ReturnedMessage` *имеет следующие свойства*:
- `message` - само возвращенное сообщение
    
- `replyCode` - код, указывающий причину возврата
    
- `replyText` - текстовое описание причины возврата, например, `NO_ROUTE`
    
- `exchange` - обменник, в который было отправлено сообщение
    
- `routingKey` - использованный ключ маршрутизации

> [!NOTE] 
> Каждым `RabbitTemplate` поддерживается только один `ReturnsCallback`. См. также **Reply Timeout**.

## 2. Подтверждения издателя
**Для подтверждений от издателя** (publisher confirms, также известных как publisher acknowledgements) **шаблону требуется** 
- `CachingConnectionFactory`, у которого свойство `publisherConfirm` установлено в `ConfirmType.CORRELATED`. 

**Подтверждения отправляются клиенту путем регистрации** `RabbitTemplate.ConfirmCallback` через вызов `setConfirmCallback(ConfirmCallback callback)`. 

Колбэк должен реализовывать этот метод:
```java
void confirm(CorrelationData correlationData, boolean ack, String cause);
```

==CorrelationData== передаётся клиентом при отправке сообщения и используется для сопоставления подтверждений.

В `ConfirmCallback`:
- `ack = true` — сообщение подтверждено,
	
- `ack = false` — отрицательное подтверждение (nack),
	
- `cause` (с версии 1.4) *содержит причину nack*, если она известна (например, отправка в несуществующий exchange → закрытие канала).

> [!NOTE] 
> `RabbitTemplate` поддерживает **только один** `ConfirmCallback`.

### Каналы и кэш
**После отправки** `RabbitTemplate` логически **закрывает канал**.
Это *препятствует приему подтверждений или возвратов*, когда кэш фабрики соединений полон (*когда в кэше есть место, канал физически не закрывается, и возвраты и подтверждения обрабатываются нормально*)

Если кэш каналов заполнен, закрытие откладывается до **5 секунд**, чтобы успели прийти confirm/return.
- С confirm — канал закрывается после последнего подтверждения.
	
- Только с return — канал держится открытым все 5 секунд.

Рекомендуется увеличить `channelCacheSize`, чтобы каналы возвращались в кэш, а не постоянно открывались/закрывались.

> [!NOTE] Плагин для отслеживания соединений
> Вы можете отслеживать использование каналов с помощью плагина управления RabbitMQ. Если вы видите, что каналы быстро открываются и закрываются, вам следует подумать об увеличении размера кэша, чтобы снизить нагрузку на сервер.

### Изменения по версиям:
- До **2.1** канал мог вернуться в кэш до получения confirm,  а другой процесс мог взять канал и выполнить операцию, вызывающую закрытие канала → возможна потеря подтверждений.
    
- С **2.1+** канал не возвращается в кэш, пока ожидаются confirm; на канале обрабатывается только одно подтверждение.
    
- С **2.2** колбэки вызываются в executor фабрики соединений, а не в IO-потоке клиента → устранён риск deadlock при выполнении Rabbit-операций внутри колбэков.


### Порядок return / confirm:
- Гарантируется, что **return вызывается до ack**, если callback return выполняется ≤ 60 секунд.
    
- Ack отправляется после завершения return-callback или по таймауту 60 секунд.

### Объект CorrelationData
Объект **CorrelationData** содержит `CompletableFuture`, который можно использовать для получения результата отправки вместо применения `ConfirmCallback` в шаблоне. 

Следующий пример показывает, как настроить экземпляр `CorrelationData`:
```java
CorrelationData cd1 = new CorrelationData();

this.templateWithConfirmsEnabled.convertAndSend(
	"exchange", 
	queue.getName(), 
	"foo", 
	cd1);

assertTrue(cd1.getFuture().get(10, TimeUnit.SECONDS).isAck());

ReturnedMessage = cd1.getReturn();
...
```

Поскольку это `CompletableFuture<Confirm>`, 
- вы можете либо синхронно получить результат с помощью `get()`, когда он будет готов, 
- либо использовать `whenComplete()` для асинхронного колбэка.

Объект **Confirm** — это простой bean с двумя свойствами:
- `ack` — признак подтверждения,
- `reason` — причина отрицательного подтверждения (nack).

*Для nack, сгенерированных брокером*, поле `reason` **не заполняется**. 
> Оно **заполняется для nack, сгенерированных фреймворком** (например, при закрытии соединения, пока подтверждения ожидают обработки).

Кроме того, ==если включены и подтверждения, и возвраты==, свойство **return** объекта `CorrelationData` заполняется возвращённым сообщением, если его не удалось маршрутизировать ни в одну очередь. 

Гарантируется, что возвращённое сообщение будет установлено **до того**, как `future` получит подтверждение (`ack`).


Метод `CorrelationData.getReturn()` возвращает объект **ReturnMessage** со следующими свойствами:
- `message` — возвращённое сообщение,
    
- `replyCode`,
    
- `replyText`,
    
- `exchange`,
    
- `routingKey`.

См. также раздел **Scoped Operations** — более простой механизм ожидания подтверждений от издателя.

# 4. Scoped Operations
Обычно `RabbitTemplate` **берёт канал из кэша на одну операцию и сразу возвращает его обратно**. 

В многопоточной среде *это означает*, **что ПОСЛЕДОВАТЕЛЬНЫЕ ВЫЗОВЫ** могут **ВЫПОЛНЯТЬСЯ НА РАЗНЫХ КАНАЛАХ**. 
> Иногда требуется гарантировать, что несколько операций будут выполнены на одном и том же канале.

Начиная с версии 2.0 для этого используется метод `invoke(OperationsCallback)`. 
```java
@FunctionalInterface
public interface OperationsCallback<T> {

    T doInRabbit(RabbitOperations operations);

}
```

**Все операции, выполненные внутри этого колбэка** через переданный `RabbitOperations`, ==используют один выделенный канал==. 

После завершения блока канал закрывается или возвращается в кэш после получения подтверждений издателя.

Такой режим нужен, **например, для ожидания подтверждений** публикации. Методы `waitForConfirms()` и `waitForConfirmsOrDie()` работают только внутри `invoke`, так как опираются на тот самый выделенный канал:
```java
Collection<?> messages = getMessagesToSend();

Boolean result = this.template.invoke(t -> {
    messages.forEach(m -> t.convertAndSend(ROUTE, m));
    t.waitForConfirmsOrDie(10_000);
    return true;
});
```

> Если операции выполняются уже внутри транзакции (например, в транзакционном listener container), `invoke` использовать не нужно — все действия и так выполняются на одном канале и фиксируются при завершении транзакции.

## Простой способ принятия подтверждений
С версии 2.2, **если требуется лишь дождаться подтверждений от брокера** *без связывания их с конкретными сообщениями*, **можно включить** 
`publisherConfirmType = ConfirmType.SIMPLE`. 

В этом случае сложная инфраструктура подтверждений не используется, и обработка становится эффективнее.

Также `RabbitTemplate` проставляет `publisherSequenceNumber` в `MessageProperties`. 

При необходимости можно перехватывать и логировать конкретные ack/nack, используя перегруженный `invoke` с клиентскими `ConfirmCallback`. Скоупированные операции привязаны к потоку, что важно учитывать в многопоточной среде.
```java
public <T> T invoke(
	OperationsCallback<T> action, 
	com.rabbitmq.client.ConfirmCallback acks,
    com.rabbitmq.client.ConfirmCallback nacks);
```


> Эти объекты `ConfirmCallback` (для ack и nack) являются колбэками Rabbit-клиента, а не колбэками шаблона.

Следующий пример логирует ack и nack:
```java
Collection<?> messages = getMessagesToSend();

Boolean result = this.template.invoke(t -> {
    messages.forEach(m -> t.convertAndSend(ROUTE, m));
    t.waitForConfirmsOrDie(10_000);
    return true;
}, (tag, multiple) -> {
        log.info("Ack: " + tag + ":" + multiple);
}, (tag, multiple) -> {
        log.info("Nack: " + tag + ":" + multiple);
}));
```

> Скоупированные операции привязаны к потоку. 
> См. "Строгий порядок сообщений в многопоточной среде" для обсуждения строгого упорядочивания в многопоточной среде.

# 5. Строгий порядок сообщений в многопоточной среде

Обсуждение в разделе "Скоупированные операции" применимо только тогда, когда операции выполняются в одном потоке.

Рассмотрим следующую ситуацию:
1. `thread-1` отправляет сообщение в очередь и передал управление другому потоку `thread-2`.
    
2. `thread-2` отправляет сообщение в ту же очередь.

Из-за асинхронной природы RabbitMQ и использования кешированных каналов **нет уверенности, что будет использован один и тот же канал, и, следовательно, не гарантирован порядок**, в котором сообщения поступят в очередь. (В большинстве случаев они будут приходить по порядку, но вероятность нарушения порядка не равна нулю).

## Как гарантировать порядок между потоками:
Чтобы решить эту проблему, *вы можете использовать ограниченный кеш каналов размером 1* (вместе с `channelCheckoutTimeout`).
Тогда второй поток будет ждать освобождения канала, и сообщения уйдут строго последовательно.

На практике это означает:
- либо выделить **отдельную ConnectionFactory** только для публикации,
    
- либо настроить `RabbitTemplate` на использование **publisher connection** основной фабрики (отдельное соединение для издателя).

Это лучше всего проиллюстрировать на простом Spring Boot приложении:
```java
@SpringBootApplication
public class Application {

	private static final Logger log = 
		LoggerFactory.getLogger(Application.class);

	public static void main(String[] args) {
		SpringApplication.run(Application.class, args);
	}

	@Bean
	TaskExecutor exec() {
		ThreadPoolTaskExecutor exec = new ThreadPoolTaskExecutor();
		exec.setCorePoolSize(10);
		return exec;
	}

	@Bean
	CachingConnectionFactory ccf() {
		CachingConnectionFactory ccf = 
			new CachingConnectionFactory("localhost");
			
			
		CachingConnectionFactory publisherCF = 
			(CachingConnectionFactory)ccf.getPublisherConnectionFactory();
			
		publisherCF.setChannelCacheSize(1); // Ограничение КЕША
		
		publisherCF.setChannelCheckoutTimeout(1000L);
		return ccf;
	}

	@RabbitListener(queues = "queue")
	void listen(String in) {
		log.info(in);
	}

	@Bean
	Queue queue() {
		return new Queue("queue");
	}


	@Bean
	public ApplicationRunner runner(Service service, TaskExecutor exec) {
		return args -> {
			exec.execute(() -> service.mainService("test"));
		};
	}

}


@Component
class Service {
	private static final Logger LOG = 
		LoggerFactory.getLogger(Service.class);

	private final RabbitTemplate template;

	private final TaskExecutor exec;

	Service(RabbitTemplate template, TaskExecutor exec) {
		template.setUsePublisherConnection(true);
		this.template = template;
		this.exec = exec;
	}

	void mainService(String toSend) {
		LOG.info("Publishing from main service");
		this.template.convertAndSend("queue", toSend);
		this.exec.execute(() -> secondaryService(toSend.toUpperCase()));
	}

	void secondaryService(String toSend) {
		LOG.info("Publishing from secondary service");
		this.template.convertAndSend("queue", toSend);
	}
}
```

Несмотря на то, что публикация выполняется в двух разных потоках, они оба будут использовать один и тот же канал, потому что кеш ограничен одним каналом.

Начиная с версии 2.3.7, `ThreadChannelConnectionFactory` **поддерживает передачу канала(ов) потока** *другому потоку* с помощью методов 
- `prepareContextSwitch` 
- и `switchContext`. 

**Первый метод возвращает контекст**, который передается второму потоку, вызывающему второй метод. 

Поток может иметь 
- либо нетранзакционный канал, 
- либо транзакционный канал (или один и другой), привязанный к нему; 
вы не можете передать их по отдельности, если не используете две фабрики соединений. 

Пример:
```java
@SpringBootApplication
public class Application {

	private static final Logger log = LoggerFactory.getLogger(Application.class);

	public static void main(String[] args) {
		SpringApplication.run(Application.class, args);
	}

	@Bean
	TaskExecutor exec() {
		ThreadPoolTaskExecutor exec = new ThreadPoolTaskExecutor();
		exec.setCorePoolSize(10);
		return exec;
	}

	@Bean
	ThreadChannelConnectionFactory tccf() {
		ConnectionFactory rabbitConnectionFactory = 
			new ConnectionFactory();
			
		rabbitConnectionFactory.setHost("localhost");
		return 
			new ThreadChannelConnectionFactory(rabbitConnectionFactory);
	}

	@RabbitListener(queues = "queue")
	void listen(String in) {
		log.info(in);
	}

	@Bean
	Queue queue() {
		return new Queue("queue");
	}


	@Bean
	public ApplicationRunner runner(Service service, TaskExecutor exec) {
		return args -> {
			exec.execute(() -> service.mainService("test"));
		};
	}

}

@Component
class Service {

	private static final Logger LOG = 
		LoggerFactory.getLogger(Service.class);

	private final RabbitTemplate template;

	private final TaskExecutor exec;

	private final ThreadChannelConnectionFactory connFactory;

	Service(RabbitTemplate template, TaskExecutor exec,
			ThreadChannelConnectionFactory tccf) {

		this.template = template;
		this.exec = exec;
		this.connFactory = tccf;
	}

	void mainService(String toSend) {
		LOG.info("Publishing from main service");
		this.template.convertAndSend("queue", toSend);
		
		Object context = this.connFactory.prepareSwitchContext();
		this.exec.execute(() -> secondaryService(
			toSend.toUpperCase(), 
			context)
		);
	}

	void secondaryService(String toSend, Object threadContext) {
		LOG.info("Publishing from secondary service");
		
		this.connFactory.switchContext(threadContext);
		
		this.template.convertAndSend("queue", toSend);
		this.connFactory.closeThreadChannel();
	}

}
```

После вызова `prepareSwitchContext`, если текущий поток выполняет любые дальнейшие операции, они будут выполняться на новом канале. 
> Важно закрыть привязанный к потоку канал, когда он больше не нужен.
