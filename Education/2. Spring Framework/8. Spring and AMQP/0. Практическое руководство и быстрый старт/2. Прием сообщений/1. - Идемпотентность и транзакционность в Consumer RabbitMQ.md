# 0. Введение в гарантии доставки в RabbitMQ
В отличие от Kafka, RabbitMQ использует другие механизмы для обеспечения гарантий доставки. 

Для достижения семантики **Exactly-once** в RabbitMQ требуется комбинация:
1. **Транзакций** (реже используется в RabbitMQ)
    
2. **Publisher Confirms** (аналог идемпотентного продюсера)
    
3. **Идемпотентного консюмера**
    
4. **Ручного подтверждения сообщений**


> [!IMPORTANT]  
> Полное Exactly-once в RabbitMQ сложно реализовать на практике. 
> 
> Чаще используют **At-least-once** с идемпотентной обработкой.

# 📕 1. Транзакционный Consumer в RabbitMQ

## 1.1. Транзакции в RabbitMQ
RabbitMQ поддерживает AMQP-транзакции, но они **не рекомендуются** для высоконагруженных систем ***из-за низкой производительности***.

```java
// Включение транзакций в Channel
channel.txSelect(); // Начало транзакции
try {
    // Получение сообщения
    GetResponse response = channel.basicGet("queue", false);
    
    // Обработка сообщения...
    
    channel.txCommit(); // Подтверждение транзакции
    channel.basicAck(
	    response.getEnvelope().getDeliveryTag(), 
	    false
	);
} catch (Exception e) {

    channel.txRollback(); // Откат транзакции
    
    channel.basicNack(
	    response.getEnvelope().getDeliveryTag(), 
	    false, 
	    true
	);
}
```

## 1.2. Spring AMQP с транзакциями
Все по аналогии с транзакционным Producer Rabbit
```java
@Bean
public SimpleRabbitListenerContainerFactory transactionalContainerFactory(
        ConnectionFactory connectionFactory) {
    
    SimpleRabbitListenerContainerFactory factory = 
        new SimpleRabbitListenerContainerFactory();
    factory.setConnectionFactory(connectionFactory);
    
    // Включение транзакций
    factory.setChannelTransacted(true);
    
    // Настройка менеджера транзакций
    factory.setTransactionManager(rabbitTransactionManager());
    
    return factory;
}

@Bean
public RabbitTransactionManager rabbitTransactionManager(
        ConnectionFactory connectionFactory) {
    return new RabbitTransactionManager(connectionFactory);
}
```

### Использование с @RabbitListener
```java
@RabbitListener(
    queues = "order-queue",
    containerFactory = "transactionalContainerFactory"
)
@Transactional // Spring Transaction Management
public void processOrder(Order order) {
    // 1. Сохраняем заказ в БД
    orderRepository.save(order);
    
    // 2. Отправляем событие в другую очередь
    rabbitTemplate.convertAndSend("order-processed", order.getId());
    
    // Если произойдет ошибка, обе операции откатятся
}
```

> [!IMPORTANT] 
> **Важно**: При использовании `channelTransacted = true`:
> - Подтверждение сообщения (`basicAck`) происходит автоматически при коммите транзакции
> - При откате транзакции сообщение возвращается в очередь 
> - Транзакция охватывает и получение, и отправку сообщений

# 📕 2. Идемпотентный Consumer в RabbitMQ

## 2.1. Проблема дублирования сообщений
В RabbitMQ сообщения могут доставляться повторно:
- При таймауте подтверждения
    
- При переподключении consumer-а
    
- При перезапуске брокера

## 2.2. Стратегия идемпотентной обработки
### Шаг 1: Добавление уникального идентификатора сообщения
```java
// Producer отправляет сообщение с ID
MessageProperties properties = new MessageProperties();
properties.setMessageId(UUID.randomUUID().toString()); // Уникальный ID
properties.setTimestamp(new Date());

Message message = new Message(
    "Данные сообщения".getBytes(),
    properties
);

rabbitTemplate.send("exchange", "routing.key", message);
```

### Шаг 2: Идемпотентный Consumer с проверкой в БД
```java
@Component
public class IdempotentMessageConsumer {
    
    @Autowired
    private MessageIdRepository messageIdRepository;
    
    @RabbitListener(queues = "processing-queue")
    public void processMessage(
		Order order,
		@Header(AmqpHeaders.MESSAGE_ID) String messageId,
		Channel channel,
		@Header(AmqpHeaders.DELIVERY_TAG) long deliveryTag)
    {
        // Шаг 1: Проверяем, обрабатывали ли уже это сообщение
        if (messageIdRepository.existsById(messageId)) {
            // Сообщение уже обработано - подтверждаем и пропускаем
            channel.basicAck(deliveryTag, false);
            return;
        }
        
        try {
            // Шаг 2: Бизнес-логика
            processOrder(order);
            
            // Шаг 3: Сохраняем ID сообщения ПОСЛЕ успешной обработки
            messageIdRepository.save(
				new ProcessedMessage(messageId, new Date()));
            
            // Шаг 4: Подтверждаем сообщение
            channel.basicAck(deliveryTag, false);
            
        } catch (Exception e) {
            // Шаг 5: При ошибке - возвращаем в очередь
            channel.basicNack(deliveryTag, false, true);
        }
    }
    
    private void processOrder(Order order) {
        // Идемпотентная бизнес-логика
        // Например, обновление статуса "если статус не X, то установить Y"
    }
}
```
