## 1. Использование AmqpTemplate
Сам `AmqpTemplate` **можно использовать для получения сообщений методом опроса** (polled `Message` reception). 

```java
amqpTemplate.receive()
```

По умолчанию, *если сообщение недоступно, немедленно возвращается* `null`. ==Блокировки нет==. 

Начиная с версии 1.5, вы можете установить `receiveTimeout` в миллисекундах, и методы `receive` будут блокироваться до этого времени, ожидая сообщение. 
> Значение меньше нуля означает бесконечную блокировку (или, по крайней мере, до потери соединения с брокером). 

В версии 1.6 появились варианты методов `receive`, позволяющие передавать таймаут при каждом вызове.


> [!WARNING] ОГРАНИЧЕНИЯ
> Поскольку операция `receive` ==создает новый== `QueueingConsumer` **для каждого сообщения**, эта техника не очень подходит для высоконагруженных сред. 
> 
> Для таких случаев использования рассмотрите возможность 
> - использования асинхронного потребителя 
> - или установите `receiveTimeout` равным нулю.


> [!NOTE] Передача аргументов `basicConsumer`
> Начиная с версии 2.4.8, при использовании *ненулевого таймаута* **можно указать аргументы, передаваемые в метод** `basicConsume`, который используется для связывания потребителя с каналом. 
> 
> Например: `template.addConsumerArg("x-priority", 10)`.

## 2. Перегрузки метода receive()
Доступны ==четыре простых метода== `receive`. 

Как и в случае с обменником (`Exchange`) на стороне отправки, существует метод, 
- требующий, чтобы свойство очереди по умолчанию было установлено непосредственно в самом шаблоне,
- и метод, который принимает параметр очереди во время выполнения. 

> В версии 1.6 появились варианты, принимающие `timeoutMillis` для переопределения `receiveTimeout` на каждый запрос.

Следующий список показывает определения четырех методов:
```java
Message receive() throws AmqpException;

Message receive(String queueName) throws AmqpException;

Message receive(long timeoutMillis) throws AmqpException;

Message receive(String queueName, long timeoutMillis) throws AmqpException;
```

## 3. Конвертация в Message 
Как и в случае отправки сообщений, `AmqpTemplate` **имеет несколько удобных методов для получения POJO** вместо экземпляров `Message`. 

> Реализации предоставляют способ настройки `MessageConverter`, используемого для создания возвращаемого `Object`. 

Следующий список показывает эти методы:
```java
Object receiveAndConvert() throws AmqpException;

Object receiveAndConvert(String queueName) throws AmqpException;

Object receiveAndConvert(long timeoutMillis) throws AmqpException;

Object receiveAndConvert(String queueName, long timeoutMillis) throws AmqpException;
```

Начиная с версии 2.0, существуют **варианты этих методов, принимающие дополнительный аргумент** `ParameterizedTypeReference` для конвертации сложных типов. 
- Шаблон должен быть настроен с `SmartMessageConverter`. 

> Для получения дополнительной информации см. "Преобразование из сообщения с помощью RabbitTemplate".

## 4. Прием и ответ
Подобно методам `sendAndReceive`, начиная с версии 1.3, `AmqpTemplate` имеет **несколько удобных методов** `receiveAndReply` **для СИНХРОННОГО получения**, ==обработки и ответа== на сообщения. 

Следующий список показывает определения этих методов:
> ❗❗❗ Методы имеют варианты с Callback ❗❗❗
```java
<R, S> boolean receiveAndReply(ReceiveAndReplyCallback<R, S> callback)
       throws AmqpException;

<R, S> boolean receiveAndReply(
	String queueName, 
	ReceiveAndReplyCallback<R, S> callback) throws AmqpException;

<R, S> boolean receiveAndReply(
	ReceiveAndReplyCallback<R, S> callback,
    String replyExchange, 
    String replyRoutingKey) throws AmqpException;

<R, S> boolean receiveAndReply(
	String queueName, 
	ReceiveAndReplyCallback<R, S> callback,
    String replyExchange, String replyRoutingKey) throws AmqpException;

<R, S> boolean receiveAndReply(
	ReceiveAndReplyCallback<R, S> callback,
    ReplyToAddressCallback<S> replyToAddressCallback) throws AmqpException;

<R, S> boolean receiveAndReply(
	String queueName, 
	ReceiveAndReplyCallback<R, S> callback,
    ReplyToAddressCallback<S> replyToAddressCallback) throws AmqpException;
```

Реализация `AmqpTemplate` берет на себя **фазы получения и ответа.** 

В большинстве случаев вам **нужно предоставить только реализацию** `ReceiveAndReplyCallback` *для выполнения некоторой бизнес-логики над полученным сообщением* и, если необходимо, построения объекта или сообщения для ответа. 

> Обратите внимание, `ReceiveAndReplyCallback` может вернуть `null`.
> 
> В этом случае ответ не отправляется, и `receiveAndReply` работает как метод `receive`. 
> 
> Это позволяет использовать одну и ту же очередь для смеси сообщений, некоторым из которых может не требоваться ответ.

> [!NOTE] ПРЕОБРАЗОВАНИЕ запроса и ответа
> **Автоматическое преобразование** сообщений (запроса и ответа) *применяется только в том случае*, **если предоставленный колбэк НЕ ЯВЛЯЕТСЯ экземпляром** `ReceiveAndReplyMessageCallback`, который предоставляет контракт обмена сырыми сообщениями.


## 5. ReplyTo
`ReplyToAddressCallback` **полезен** в случаях, **когда требуется пользовательская логика для определения адреса** `replyTo` во время выполнения на основе полученного сообщения и ответа от `ReceiveAndReplyCallback`. 

> По умолчанию для маршрутизации ответа используется информация `replyTo` в сообщении-запросе.

Следующий пример показывает пример получения и ответа на основе POJO:
```java
boolean received =
	this.template.receiveAndReply(
		ROUTE, 
		new ReceiveAndReplyCallback<Order, Invoice>() {

			public Invoice handle(Order order) {
					return processOrder(order);
			}
	});
        
if (received) {
	log.info("We received an order!");
}
```
