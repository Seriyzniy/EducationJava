==Самый ПРОСТОЙ СПОСОБ асинхронно получать сообщения== — использовать инфраструктуру аннотированных конечных точек слушателя. 

В двух словах, она **позволяет вам предоставить метод** управляемого бина **КАК КОНЕЧНУЮ ТОЧКУ СЛУШАТЕЛЯ Rabbit**. 

Следующий пример показывает, как использовать аннотацию `@RabbitListener`:
```java
@Component
public class MyService {

    @RabbitListener(queues = "myQueue")
    public void processOrder(String data) {
        ...
    }

}
```

Когда **в указанной очереди появляется сообщение,** 
- соответствующий **метод вызывается**, 
- а полезная нагрузка сообщения передаётся в параметры метода.

Вся «техническая» работа скрыта: инфраструктура `@RabbitListener` **под капотом создаёт и настраивает контейнер слушателя сообщений** для каждого такого метода, используя `RabbitListenerContainerFactory`. Разработчику не нужно напрямую работать с контейнерами и listener’ами.

**Очередь**, указанная в `@RabbitListener`, **должна существовать и быть привязана к обменнику.** 
> При этом она может быть объявлена автоматически, если в контексте приложения присутствует `RabbitAdmin`.

## 1. Заполнители свойств
**Аннотация `@RabbitListener` поддерживает заполнители свойств (`${...}`) и выражения SpEL (`#{...}`)** — их можно использовать в параметрах аннотации (например, `queues`, `bindings`). Это позволяет гибко настраивать очереди и биндинги без жёсткой привязки к именам.

Примеры объявления слушателей:
```java
@Component
public class MyService {

  @RabbitListener(bindings = @QueueBinding(
        value = @Queue(
	        value = "myQueue", 
	        durable = "true"),
	        
        exchange = @Exchange(
	        value = "auto.exch", 
	        ignoreDeclarationExceptions = "true"),
	        
        key = "orderRoutingKey")
  )
  public void processOrder(Order order) {
    ...
  }

  @RabbitListener(bindings = @QueueBinding(
        value = @Queue,
        exchange = @Exchange(value = "auto.exch"),
        key = "invoiceRoutingKey")
  )
  public void processInvoice(Invoice invoice) {
    ...
  }

  @RabbitListener(
	  queuesToDeclare = @Queue(
		  name = "${my.queue}", 
		  durable = "true"))
  public String handleWithSimpleDeclare(String data) {
      ...
  }
}
```

В первом примере **очередь, обменник и биндинг ОБЪЯВЛЯЮТСЯ АВТОМАТИЧЕСКИ**, *если их ещё нет, и привязывается к обменнику с ключом маршрутизации.*
Параметр `ignoreDeclarationExceptions` позволяет подключаться к уже существующему обменнику, даже если его параметры отличаются.

Во втором примере создаётся **анонимная (эксклюзивная, auto-delete) очередь**, имя которой генерируется фреймворком с использованием `Base64UrlNamingStrategy`. 
Такие ==очереди нельзя объявлять, если имя задаётся брокером== — в этом случае нужны обычные `@Bean`.
Можно предоставить несколько записей `QueueBinding`, что позволяет слушателю прослушивать несколько очередей.

В третьем примере очередь объявляется **по имени из конфигурационного свойства** `my.queue`, с привязкой к обменнику по умолчанию.

С версии **2.0** аннотация `@Exchange` поддерживает **ЛЮБЫЕ ТИПЫ ОБМЕННИКОВ**, включая пользовательские. 
Для сложных сценариев по-прежнему можно использовать стандартные определения `@Bean`.

## 2. Конфигурация при помощи @Bean
Когда требуется **более сложная конфигурация**, вместо аннотаций можно использовать **обычные определения `@Bean`**.

Параметр **`ignoreDeclarationExceptions`** у `@Exchange` позволяет привязаться к **уже существующему обменнику**, даже если его параметры отличаются (например, `internal`). По умолчанию свойства существующего обменника должны полностью совпадать.

Начиная с версии **2.0**, очередь можно привязывать к обменнику **с НЕСКОЛЬКИМИ КЛЮЧАМИ МАРШРУТИЗАЦИИ**:
```java
...
key = { "red", "yellow" }
...
```

Также аннотация **`@QueueBinding`** поддерживает задание **аргументов** для очередей, обменников и привязок:
```java
@RabbitListener(bindings = @QueueBinding(
        value = @Queue(
                value = "auto.headers",
                autoDelete = "true",
                arguments = @Argument(
                        name = "x-message-ttl",
                        value = "10000",
                        type = "java.lang.Integer"
                )
        ),
        exchange = @Exchange(
                value = "auto.headers",
                type = ExchangeTypes.HEADERS,
                autoDelete = "true"
        ),
        arguments = {
                @Argument(name = "x-match", value = "all"),
                @Argument(name = "thing1", value = "somevalue"),
                @Argument(name = "thing2")
        })
)
public String handleWithHeadersExchange(String foo) {
    ...
}
```

В этом примере для очереди задан аргумент **`x-message-ttl`** со значением 10 секунд; поскольку тип не `String`, он явно указан как `Integer`. Если очередь уже существует, её аргументы должны совпадать с объявленными.

Для **`ExchangeTypes.HEADERS`** аргументы привязки настраиваются так, чтобы сообщение содержало заголовок `thing1 = somevalue` и заголовок `thing2` с любым значением. Аргумент **`x-match = all`** означает, что оба условия обязательны.

Поля **`name`**, **`value`** и **`type`** в `@Argument` могут быть заданы через **property placeholders (`${...}`)** или **SpEL (`#{...}`)**. 

Имя должно разрешаться в `String`, тип — в `Class` или полное имя класса, значение — в объект, который может быть преобразован `DefaultConversionService`. 

Если имя аргумента разрешается в `null` или пустую строку, такой `@Argument` игнорируется.
