Хотя ==модель AMQP==, которую мы описали в предыдущем разделе, является **универсальной и применимой ко всем реализациям**, когда мы переходим к управлению ресурсами, детали **специфичны для реализации брокера**.

Поэтому ==в этом разделе мы сосредоточимся на коде==, который существует только в нашем модуле **"spring-rabbit"**, поскольку на данный момент **RabbitMQ является единственной поддерживаемой реализацией**.

# 1. Центральный компонент: ConnectionFactory - Фабрика Соединений
Центральным компонентом для управления соединением с брокером RabbitMQ является ==интерфейс== **`ConnectionFactory`**.

**Обязанность реализации `ConnectionFactory`** — предоставить экземпляр `org.springframework.amqp.rabbit.connection.Connection`, который является оберткой для `com.rabbitmq.client.Connection`.

## 1. Выбор фабрики соединений (Connection Factory)
Есть **3 фабрики соединений** на выбор:
1. **`PooledChannelConnectionFactory`**
    
2. **`ThreadChannelConnectionFactory`**
    
3. **`CachingConnectionFactory`**

Первые две были добавлены в версии **2.3**.

### Рекомендации по выбору
- **Для большинства случаев использования** следует использовать **`CachingConnectionFactory`**.
    
- **`ThreadChannelConnectionFactory`** можно использовать, если вы хотите обеспечить **строгий порядок сообщений** без необходимости использования *Scoped Operations*.
    
- **`PooledChannelConnectionFactory`** похож на `CachingConnectionFactory` в том, что он использует **одно соединение и пул каналов**. Его реализация проще, но он **не поддерживает correlated publisher confirmations**.

### Поддержка подтверждений публикации (Publisher Confirms)
- **Простые подтверждения публикации** поддерживаются всеми тремя фабриками.
    
- **Correlated publisher confirmations** НЕ поддерживаются `PooledChannelConnectionFactory`.

## 2. RabbitTemplate
В Spring AMQP есть основной объект **для отправки сообщений** — **`RabbitTemplate`**.  

Он использует **`ConnectionFactory`**, из которой берутся:
- TCP-соединения (`Connection`)
- каналы (`Channel`)


По умолчанию:
- **публикация** сообщений (producer)
- **потребление** сообщений (consumer)
могут использовать **одно и то же соединение** (но разные каналы).

### С версии **2.3.2**
Появилась возможность:
👉 **использовать отдельную фабрику соединений именно для публикации сообщений**

То есть логически:
```java
Consumer connection(s)  <---> RabbitMQ
Producer connection(s)  <---> RabbitMQ
```
а не всё через одно TCP-соединение.

## 3. Стратегия Backoff (начиная с версии 3.1)
Начиная с версии 3.1, `AbstractConnectionFactory` включает свойство **`connectionCreatingBackOff`**, которое поддерживает **политику backoff (отката)** в модуле соединений.

> [!NOTE] Backoff = стратегия вида:
> ```bash
> Попытка → ошибка → ПОДОЖДАТЬ → новая попытка
> ```

В настоящее время существует поддержка в поведении `createChannel()` для обработки исключений, которые возникают при достижении лимита `channelMax`, реализующая **стратегию backoff** на основе попыток и интервалов.

## 4. ФАБРИКИ
### Фабрика соединений: PooledChannelConnectionFactory
Эта фабрика управляет ==одним соединением== и ==2 пулами каналов==, основанными на *Apache Pool2*. 

- Один пул предназначен для **транзакционных каналов**, 
- другой — для **нетранзакционных каналов**.

Пулы представляют собой `GenericObjectPool` с конфигурацией по умолчанию; предоставляется callback для настройки пулов. 
> Обратитесь к документации Apache для получения дополнительной информации.

**JAR Apache commons-pool2 должен быть в classpath** для использования этой фабрики.
```java
@Bean
PooledChannelConnectionFactory pcf() throws Exception {

    ConnectionFactory rabbitConnectionFactory = 
	    new ConnectionFactory();
    rabbitConnectionFactory.setHost("localhost");
    
    
    PooledChannelConnectionFactory pcf = 
	    new PooledChannelConnectionFactory(rabbitConnectionFactory);
    
    pcf.setPoolConfigurer((pool, tx) -> {
        if (tx) {
            // configure the transactional pool
        }
        else {
            // configure the non-transactional pool
        }
    });
    return pcf;
}
```

> В качестве `ConnectionFactory` используется из RabbitMQ API - `com.rabbitmq.client.ConnectionFactory`

### Фабрика соединений: ThreadChannelConnectionFactory
Эта фабрика управляет ==одним соединением== и ==двумя `ThreadLocal`== 
- один для транзакционных каналов, 
- другой для нетранзакционных каналов. 

Эта фабрика гарантирует, что **все операции в одном и том же потоке используют один и тот же канал** (пока он остается открытым). Это обеспечивает **строгий порядок сообщений** без необходимости использования Scoped Operations.

Чтобы избежать утечек памяти, (*если ваше приложение использует множество кратковременных потоков*) вы должны вызывать метод фабрики **`closeThreadChannel()`** для освобождения ресурсов канала.

> Начиная с версии 2.3.7, поток может передавать свои каналы другому потоку. 
> См. раздел "Strict Message Ordering in a Multi-Threaded Environment" для получения дополнительной информации.

### Фабрика соединений: CachingConnectionFactory
Третья предоставляемая реализация — **`CachingConnectionFactory`**, которая *по умолчанию устанавливает* **один прокси-объект соединения**, который может использоваться приложением совместно. 

Совместное использование соединения возможно, поскольку **"единицей работы"** для обмена сообщениями с AMQP фактически является **"канал"** (в некотором смысле это похоже на связь между соединением и сессией в JMS).

> Экземпляр соединения предоставляет метод `createChannel`.

Реализация `CachingConnectionFactory` поддерживает **кеширование этих каналов** и поддерживает **отдельные кеши** для каналов в зависимости от того, являются ли они транзакционными.

#### Конфигурация CachingConnectionFactory
При создании экземпляра `CachingConnectionFactory` вы можете указать **'hostname'** через конструктор. 
Также следует указать свойства **'username'** и **'password'**. 

> Для настройки размера кеша каналов (по умолчанию 25) можно вызвать метод **`setChannelCacheSize()`**.

---
#### Кеширование соединений (с версии 1.3)

Начиная с версии 1.3, вы можете настроить `CachingConnectionFactory` на **КЕШИРОВАНИЕ СОЕДИНЕНИЙ**, а не только каналов. 
В этом случае каждый вызов `createConnection()` создает новое соединение (или получает простаивающее из кеша). 
> Закрытие соединения возвращает его в кеш (если размер кеша не достигнут). Каналы, созданные на таких соединениях, также кешируются.

Использование отдельных соединений может быть полезно в некоторых средах, например:
- Потребление из HA-кластера (***High Availability-кластер (кластер высокой доступности***))
    
- В сочетании с балансировщиком нагрузки для подключения к разным членам кластера
    
- И другие сценарии

Для кеширования соединений установите `cacheMode` в значение **`CacheMode.CONNECTION`**.

**Это не ограничивает количество соединений.** Скорее, это указывает, сколько простаивающих открытых соединений разрешено.

> [!WARNING] Важное понимание размера кеша
> Важно понимать, что размер кеша (по умолчанию) **НЕ ЯВЛЯЕТСЯ ЛИМИТОМ**, а лишь **количеством каналов, КОТОРЫЕ МОГУТ БЫТЬ ЗАКЕШИРОВАНЫ**. 
> 
> При размере кеша, скажем, 10, фактически может использоваться **любое количество каналов**. 
> Если используется более 10 каналов и все они возвращаются в кеш, 10 помещаются в кеш. Остальные **физически закрываются**.
> 


##### Изменение размера кеша по умолчанию (с версии 1.6)

Начиная с версии 1.6, **размер кеша каналов по умолчанию увеличен с 1 до 25**. 

> Используется свойство `channelCacheSize`

В средах с высокой нагрузкой и многопоточностью маленький кеш означает, что каналы создаются и закрываются с высокой скоростью. 
> Увеличение размера кеша по умолчанию может избежать этих накладных расходов.

Следует отслеживать используемые каналы через **RabbitMQ Admin UI** и рассмотреть возможность дальнейшего увеличения размера кеша, если вы видите, что многие каналы создаются и закрываются. 

Кеш растет **только по требованию** (в соответствии с требованиями параллелизма приложения), поэтому это изменение не влияет на существующие приложения с низкой нагрузкой.

---
#### Таймаут получения канала (с версии 1.4.2)
Начиная с версии 1.4.2, `CachingConnectionFactory` имеет **свойство** **`channelCheckoutTimeout`**. 

Когда это свойство **БОЛЬШЕ НУЛЯ**, `channelCacheSize` становится **ограничением на количество каналов**, которые могут быть созданы на соединении. 

Если лимит достигнут, вызывающие потоки блокируются до тех пор, пока канал не станет доступен или не будет достигнут этот таймаут, в котором случае выбрасывается исключение **`AmqpTimeoutException`**.

---
#### Возврат каналов
Каналы, используемые в рамках фреймворка (например, `RabbitTemplate`), надежно возвращаются в кеш. Если вы **создаете каналы вне фреймворка** (например, получая доступ к соединениям напрямую и вызывая `createChannel()`), вы должны надежно возвращать их (путем закрытия), возможно, в блоке `finally`, чтобы избежать исчерпания каналов.

----
#### Ограничение соединений (с версии 1.5.5)

Начиная с версии 1.5.5, предоставлено новое свойство **`connectionLimit`**. 
Когда это свойство установлено, оно *ограничивает* **общее количество разрешенных соединений**. 
- Если лимит достигнут, используется `channelCheckoutTimeLimit` для ожидания освобождения соединения. 
- Если время превышено, выбрасывается исключение **`AmqpTimeoutException`**.

> [!WARNING] **Важно:** 
> Когда режим кеширования `CONNECTION`, **автоматическое объявление очередей и других сущностей** (*см. Automatic Declaration of Exchanges, Queues, and Bindings*) **НЕ поддерживается**.
> 

---
#### Пул потоков и Executor

Также на момент написания библиотека `amqp-client` *по умолчанию создает* **фиксированный пул потоков для каждого соединения** (размер по умолчанию: `Runtime.getRuntime().availableProcessors() * 2` потоков). 

При использовании большого количества соединений следует рассмотреть возможность установки **пользовательского исполнителя (executor)** на `CachingConnectionFactory`. 
> Затем один и тот же исполнитель может использоваться всеми соединениями, и его потоки могут быть общими.

Пул потоков исполнителя должен быть:
- **Неограниченным**
- Или установлен соответствующим образом для ожидаемого использования (обычно **хотя бы один поток на соединение**)


> [!NOTE] Заметка
> Если на каждом соединении создается несколько каналов, размер пула влияет на параллелизм, поэтому **исполнитель с переменным (или простым кешированным) пулом потоков** был бы наиболее подходящим.

---
#### Примеры создания соединения
```java
// Java конфигурация
// ИСПОЛЬЗУЕТСЯ фабрика соединений поставляемая Spring Rabbit
CachingConnectionFactory connectionFactory = 
	new CachingConnectionFactory("somehost");
connectionFactory.setUsername("guest");
connectionFactory.setPassword("guest");

// ИСПОЛЬЗУЕТСЯ эта фабрика, для получения соединения из RabbitMQ API
Connection connection = connectionFactory.createConnection();
```

```xml
<!-- XML конфигурация -->
<bean id="connectionFactory"
      class="org.springframework.amqp.rabbit.connection.CachingConnectionFactory">
    <constructor-arg value="somehost"/>
    <property name="username" value="guest"/>
    <property name="password" value="guest"/>
</bean>
```

> [!NOTE] Заметка - `SingleConnectionFactory`
> Существует также реализация **`SingleConnectionFactory`**, которая доступна только в юнит-тестах фреймворка. 
> 
> Она проще, чем `CachingConnectionFactory`, поскольку **не кеширует каналы**, но она ==не предназначена для практического использования== вне простых тестов из-за отсутствия производительности и отказоустойчивости. 
> 
> Если по какой-то причине вам нужно реализовать свою собственную `ConnectionFactory`, базовый класс **`AbstractConnectionFactory`** может предоставить хорошую отправную точку.

##### Пространство имен rabbit

`ConnectionFactory` может быть быстро и *удобно создана с использованием пространства имен* `rabbit`, следующим образом:
```xml
<rabbit:connection-factory id="connectionFactory"/>
```
В большинстве случаев этот подход предпочтительнее, поскольку фреймворк может выбрать для вас наилучшие значения по умолчанию. Созданный экземпляр — это `CachingConnectionFactory`.

**Имейте в виду**, что размер кеша каналов по умолчанию равен **25**. Если вы хотите, чтобы кешировалось больше каналов, установите большее значение, задав свойство `channelCacheSize`. 

В XML это будет выглядеть так:
```xml
<bean id="connectionFactory"
      class="org.springframework.amqp.rabbit.connection.CachingConnectionFactory">
    <constructor-arg value="somehost"/>
    <property name="username" value="guest"/>
    <property name="password" value="guest"/>
    <property name="channelCacheSize" value="50"/>
</bean>
```

Также с пространством имен можно добавить атрибут **'channel-cache-size'**, следующим образом:
```xml
<rabbit:connection-factory
    id="connectionFactory" channel-cache-size="50"/>
```

*Режим кеширования по умолчанию* — `CHANNEL`, но вы можете настроить его на кеширование соединений. 

В следующем примере мы используем **connection-cache-size**:
```xml
<rabbit:connection-factory
    id="connectionFactory" cache-mode="CONNECTION" connection-cache-size="25"/>
```

---
##### Указание хоста и порта, и адресов нод

Вы можете указать атрибуты **host** и **port** с помощью пространства имен, следующим образом:
```xml
<rabbit:connection-factory
    id="connectionFactory" host="somehost" port="5672"/>
```

Альтернативно, если вы работаете в кластерной среде, вы можете использовать атрибут **addresses**, следующим образом:
```xml
<rabbit:connection-factory
    id="connectionFactory" addresses="host1:5672,host2:5672" address-shuffle-mode="RANDOM"/>
```
> См. раздел "Connecting to a Cluster" для получения информации о `address-shuffle-mode`.

---
#### Пример с ПОЛЬЗОВАТЕЛЬСКОЙ фабрикой потоков

Следующий пример с пользовательской фабрикой потоков, которая добавляет префикс `rabbitmq-` к именам потоков:
```xml
<rabbit:connection-factory 
	id="multiHost" 
	virtual-host="/bar" 
	addresses="host1:1234,host2,host3:4567"
    thread-factory="tf"
    channel-cache-size="10" username="user" password="password" />

<bean 
	id="tf" class="org.springframework.scheduling.concurrent.CustomizableThreadFactory">
    <constructor-arg value="rabbitmq-" />
</bean>
```
