# 1. API построителя для очередей и обменников
Версия 1.6 представляет удобный **fluent API для настройки объектов** `Queue` и `Exchange` **при использовании Java-конфигурации**. 

Следующий пример показывает, как его использовать:
```java
@Bean
public Queue queue() {
    return QueueBuilder.nonDurable("foo")
        .autoDelete()
        .exclusive()
        .withArgument("foo", "bar")
        .build();
}

@Bean
public Exchange exchange() {
  return ExchangeBuilder.directExchange("foo")
      .autoDelete()
      .internal()
      .withArgument("foo", "bar")
      .build();
}
```

Для получения дополнительной информации см. Javadoc для [QueueBuilder (Spring AMQP 4.0.1 API)](https://docs.spring.io/spring-amqp/docs/4.0.1/api/org/springframework/amqp/core/QueueBuilder.html) 
[BindingBuilder (Spring AMQP 4.0.1 API)](https://docs.spring.io/spring-amqp/docs/4.0.1/api/org/springframework/amqp/core/BindingBuilder.html)
и [ExchangeBuilder (Spring AMQP 4.0.1 API)](https://docs.spring.io/spring-amqp/docs/4.0.1/api/org/springframework/amqp/core/ExchangeBuilder.html).

> Начиная с версии 2.0, `ExchangeBuilder` **теперь по умолчанию создает устойчивые (durable) обменники**, чтобы быть последовательным с простыми конструкторами в отдельных классах `AbstractExchange`. 
> 
> Чтобы создать неустойчивый (non-durable) обменник с помощью построителя, используйте `.durable(false)` перед вызовом `.build()`. 
> 
> Метод `durable()` без параметра больше не предоставляется.

Версия 2.2 представила fluent API для добавления "известных" аргументов обменника и очереди...
```java
@Bean
public Queue allArgs1() {
    return QueueBuilder.nonDurable("all.args.1")
            .ttl(1000)
            .expires(200_000)
            .maxLength(42)
            .maxLengthBytes(10_000)
            .overflow(Overflow.rejectPublish)
            .deadLetterExchange("dlx")
            .deadLetterRoutingKey("dlrk")
            .maxPriority(4)
            .lazy()
            .leaderLocator(LeaderLocator.minLeaders)
            .singleActiveConsumer()
            .build();
}

@Bean
public DirectExchange ex() {
    return ExchangeBuilder.directExchange("ex.with.alternate")
            .durable(true)
            .alternate("alternate")
            .build();
}
```

# 2. Объявление коллекций обменников, очередей и привязок (Declarables)
Вы **можете обернуть коллекции объектов** `Declarable` 
- `Queue`, 
- `Exchange` 
- и `Binding` 
в объекты `Declarables`. 


### Declarables
Это **контейнер (обёртка)** для **группы** объектов `Declarable`.
То есть вместо:
```java
@Bean
public Queue q1() { ... }

@Bean
public Queue q2() { ... }

@Bean
public Exchange e1() { ... }
```

можно сделать:
```java
@Bean
public Declarables infra() {
    return new Declarables(
        new Queue("q1"),
        new Queue("q2"),
        new DirectExchange("e1")
    );
}
```

### RabbitAdmin
`RabbitAdmin` — это **инфраструктурный бин**, который:
- сканирует Spring-контекст
    
- **находит все бины типа `Declarable`**
    - одиночные (`Queue`, `Exchange`, `Binding`)
    - и обёрнутые в `Declarables`
        
- **объявляет их в RabbitMQ**

#### Когда происходит объявление?
- **При каждом установлении соединения с брокером**
	
- при старте приложения
    
- после обрыва и восстановления соединения


Это очень важно:
> инфраструктура **самовосстанавливается** автоматически

Следующий пример показывает, как это сделать:
```java
@Configuration
public static class Config {

    @Bean
    public CachingConnectionFactory cf() {
        return new CachingConnectionFactory("localhost");
    }

    @Bean
    public RabbitAdmin admin(ConnectionFactory cf) {
        return new RabbitAdmin(cf);
    }

    @Bean
    public DirectExchange e1() {
        return new DirectExchange("e1", false, true);
    }

    @Bean
    public Queue q1() {
        return new Queue("q1", false, false, true);
    }

    @Bean
    public Binding b1() {
        return BindingBuilder.bind(q1()).to(e1()).with("k1");
    }



    @Bean
    public Declarables es() {
        return new Declarables(
                new DirectExchange("e2", false, true),
                new DirectExchange("e3", false, true));
    }

    @Bean
    public Declarables qs() {
        return new Declarables(
                new Queue("q2", false, false, true),
                new Queue("q3", false, false, true));
    }

    @Bean
    @Scope(ConfigurableBeanFactory.SCOPE_PROTOTYPE)
    public Declarables prototypes() {
        return new Declarables(
	        new Queue(
		        this.prototypeQueueName, false, false, true));
    }

    @Bean
    public Declarables bs() {
        return new Declarables(
			new Binding("q2", DestinationType.QUEUE, "e2", "k2", null),
			new Binding("q3", DestinationType.QUEUE, "e3", "k3", null));
    }

    @Bean
    public Declarables ds() {
        return new Declarables(
			new DirectExchange("e4", false, true),
			new Queue("q4", false, false, true),
			new Binding("q4", DestinationType.QUEUE, "e4", "k4", null));
    }
}
```

> В **версиях ДО 2.1** ***можно было объявлять несколько экземпляров*** `Declarable`, определяя бины типа `Collection<Declarable>`. 
> 
> Это может вызывать нежелательные побочные эффекты в некоторых случаях, потому что администратор должен перебирать все бины `Collection<?>`.

Версия 2.2 добавила метод `getDeclarablesByType` в `Declarables`; он может быть использован как удобство, например, при объявлении бина(ов) контейнера слушателя.
```java
public SimpleMessageListenerContainer container(
	ConnectionFactory connectionFactory,
    Declarables mixedDeclarables, 
    MessageListener listener) 
{
    SimpleMessageListenerContainer container = 
	    new SimpleMessageListenerContainer(connectionFactory);
	    
    container.setQueues(
		mixedDeclarables
			.getDeclarablesByType(Queue.class).toArray(new Queue[0]));
			
    container.setMessageListener(listener);
    
    return container;
}
```

# 3. Условное объявление
Речь идёт о том, **кто и на каком брокере объявляет очереди, обменники и привязки**, когда в приложении используется Spring AMQP и **несколько `RabbitAdmin`**.

По умолчанию всё очень просто и довольно грубо: **каждый `RabbitAdmin`**, у которого `auto-startup=true`, при подключении к брокеру пытается **объявить все `Queue`, `Exchange` и `Binding`, найденные в контексте Spring**. 
- Если брокер один — проблем нет. 
- Если брокеров несколько, начинается хаос: одна и та же инфраструктура объявляется везде.

Чтобы это контролировать, все элементы инфраструктуры (`Queue`, `Exchange`, `Binding`) реализуют интерфейс `Declarable`. 
В нём есть логика, позволяющая ответить на два вопроса: 
1) _нужно ли вообще объявлять этот объект_ 
2) и _какие именно `RabbitAdmin` имеют право это делать_. 
Каждый `RabbitAdmin` ==перед объявлением проверяет эти условия.==


Начиная с версии 2.1.9, `RabbitAdmin` имеет новое свойство `explicitDeclarationsOnly` (по умолчанию `false`). 
Когда оно установлено в `true`, администратор **перестаёт объявлять всё подряд** и объявляет **только те элементы**, которые явно привязаны именно к нему. 
> Это режим строгого контроля, полезный в сложных конфигурациях и при работе с несколькими брокерами.администратором.


Если у элемента инфраструктуры(который наследует Declarable) **не указано никаких ограничений**, он будет объявлен **всеми администраторами**. 
Если же у него ==явно задан список администраторов== 
- через `declared-by` в XML 
- или `setAdminsThatShouldDeclare()` в Java, 
то объявлять его будет **только указанный `RabbitAdmin`**, остальные его проигнорируют.

Свойства доступны как атрибуты в пространстве имен, как показано в следующих примерах:

```xml
<rabbit:admin id="admin1" connection-factory="CF1" />
<rabbit:admin id="admin2" connection-factory="CF2" />
<rabbit:admin id="admin3" connection-factory="CF3" explicit-declarations-only="true" />

<rabbit:queue id="declaredByAdmin1AndAdmin2Implicitly" />
<rabbit:queue id="declaredByAdmin1AndAdmin2" declared-by="admin1, admin2" />
<rabbit:queue id="declaredByAdmin1Only" declared-by="admin1" />
<rabbit:queue id="notDeclaredByAllExceptAdmin3" auto-declare="false" />

<rabbit:direct-exchange name="direct" declared-by="admin1, admin2">
    <rabbit:bindings>
        <rabbit:binding key="foo" queue="bar"/>
    </rabbit:bindings>
</rabbit:direct-exchange>
```

По умолчанию атрибут `auto-declare` имеет значение `true`, и,
- если `declared-by` не предоставлен (или пуст), 
- то все экземпляры `RabbitAdmin` объявляют объект 
	- (при условии, что атрибут `auto-startup` администратора — `true` (по умолчанию), 
	- и атрибут `explicit-declarations-only` администратора — `false`).

Аналогично, вы можете использовать Java-конфигурацию `@Configuration` для достижения того же эффекта. 

**В следующем примере** ==компоненты объявляются== `admin1`, но не `admin2`:
```java
@Bean
public RabbitAdmin admin1() {
    return new RabbitAdmin(cf1());
}

@Bean
public RabbitAdmin admin2() {
    return new RabbitAdmin(cf2());
}


@Bean
public Queue queue() {
    Queue queue = new Queue("foo");
    queue.setAdminsThatShouldDeclare(admin1());
    return queue;
}

@Bean
public Exchange exchange() {
    DirectExchange exchange = new DirectExchange("bar");
    exchange.setAdminsThatShouldDeclare(admin1());
    return exchange;
}

@Bean
public Binding binding() {
    Binding binding = new Binding(
	    "foo", 
	    DestinationType.QUEUE, 
	    exchange().getName(), 
	    "foo", 
	    null);
	    
    binding.setAdminsThatShouldDeclare(admin1());
    return binding;
}
```

# 4. Замечание об атрибутах id и name
Атрибут `name` в `<rabbit:queue/>` и `<rabbit:exchange/>` задаёт **имя сущности на брокере**. 
> Если для очереди `name` не указан, ==создаётся анонимная очередь==.


В версиях до 2.0 `name` также регистрировался как псевдоним имени бина (аналогично `name` в элементах `<bean/>`).

Это вызывало две проблемы:
1. Это препятствовало объявлению очереди и обменника с одинаковым именем.
    
2. Псевдоним не разрешался, если он содержал выражение SpEL (`#{…​}`).


Начиная **с версии 2.0**, при указании **и `id`, и `name`**, `name` больше не считается именем бина — ==для ссылок используется== `id`. 
> Это позволяет иметь очередь и обменник с одинаковым именем. 

Если задан только `name`, бин по-прежнему доступен по нему, но при использовании SpEL в `name` для ссылок обязательно нужно указывать `id`.

# 5. AnonymousQueue
**AnonymousQueue** рекомендуется использовать, когда нужна
- уникальная, 
- эксклюзивная 
- и автоудаляемая очередь. 

Она предпочтительнее очередей с именами, генерируемыми брокером (через пустое имя), потому что ==имя очереди должно быть известно бинам ещё до подключения к брокеру==, а при переподключении `RabbitAdmin` повторно объявляет `AnonymousQueue` **с тем же именем**, тогда как имя, сгенерированное брокером, изменилось бы.

`AnonymousQueue` позволяет управлять форматом имени. 
*По умолчанию* используется префикс `spring.gen-` и UUID в base64 (URL-safe). 
Формат можно изменить, передав `NamingStrategy` — стандартную или свою, например с префиксом приложения или хоста. 
> Это настраивается как в Java-конфигурации, так и через XML с атрибутом `naming-strategy`.

Начиная с версии 2.1, анонимные очереди объявляются с параметром `Queue.X_QUEUE_LEADER_LOCATOR=client-local`, **чтобы очередь создавалась на узле, к которому подключено приложение.** При необходимости это поведение можно отключить.

# 6. Восстановление автоматически удаляемых объявлений (Recovering Auto-Delete Declarations)
Обычно `RabbitAdmin` (или несколько администраторов) **ВОССТАНАВЛИВАЕТ ТОЛЬКО очереди/обменники/привязки, **которые объявлены как БИНЫ** в контексте приложения; 
> если какие-либо такие объявления являются автоматически удаляемыми (`auto-delete`), они будут удалены брокером при потере соединения. 

Когда соединение восстанавливается, администратор повторно объявит эти сущности. 

Обычно **сущности, созданные путем вызова** 
- `admin.declareQueue(…)`, 
- `admin.declareExchange(…)` 
- и `admin.declareBinding(…)`, ==НЕ восстанавливаются.==

> (Логично, так как метод не вызывается)

Начиная с версии 2.4, у администратора появилось новое свойство `redeclareManualDeclarations`; когда оно установлено в `true`, администратор будет восстанавливать эти сущности в дополнение к бинам в контексте приложения.

Восстановление отдельных объявлений не будет выполнено, если вызывается 
- `deleteQueue(…)`, 
- `deleteExchange(…)` 
- или `removeBinding(…)`. 
Связанные привязки удаляются из восстанавливаемых сущностей при удалении очередей и обменников.

Наконец, вызов `resetAllManualDeclarations()` предотвратит восстановление любых ранее объявленных сущностей.
