*Спецификация AMQP описывает*, ***как протокол может использоваться для настройки*** очередей, обменников и привязок на брокере. 

Эти операции (которые переносимы начиная со спецификации 0.8 и выше) представлены
- **в интерфейсе** `AmqpAdmin` из пакета `org.springframework.amqp.core`
	
- Реализация этого класса для RabbitMQ — `RabbitAdmin`, **расположенный** в пакете `org.springframework.amqp.rabbit.core`.

# 1. Интерфейс AmqpAdmin
Интерфейс `AmqpAdmin` основан на использовании доменных абстракций Spring AMQP и показан в следующем листинге:
```java
public interface AmqpAdmin {

    // Операции с обменниками
    void declareExchange(Exchange exchange);
    void deleteExchange(String exchangeName);

    // Операции с очередями
    Queue declareQueue();
    String declareQueue(Queue queue);
    boolean deleteQueue(String queueName);
    void deleteQueue(String queueName, boolean unused, boolean empty);
    void purgeQueue(String queueName, boolean noWait);

    // Операции с привязками
    void declareBinding(Binding binding);
    void removeBinding(Binding binding);

    Properties getQueueProperties(String queueName);
    QueueInformation getQueueInfo(String queueName);
}
```

## Методы получения информации
Метод `getQueueProperties()` **возвращает некоторую ограниченную информацию об очереди** (количество сообщений и количество потребителей). 
Ключи для возвращаемых свойств доступны в виде констант в `RabbitAdmin` 
- `QUEUE_NAME`, 
- `QUEUE_MESSAGE_COUNT` 
- и `QUEUE_CONSUMER_COUNT`. 

Метод `getQueueInfo()` возвращает удобный объект данных `QueueInformation`.

## Методы объявления очереди
### 1. Без аргументов 
Метод `declareQueue()` без аргументов **определяет очередь на брокере с автоматически сгенерированным именем.** 
> *Дополнительные свойства* этой автоматически сгенерированной очереди: 
> - `exclusive=true`, 
> - `autoDelete=true` 
> - и `durable=false`.

### 2. С аргументом Queue
Метод `declareQueue(Queue queue)` принимает объект `Queue` и возвращает имя объявленной очереди. 

*Если свойство* `name` ***предоставленной очереди*** — **пустая строка**, **брокер объявляет очередь со сгенерированным именем.** 
Это имя возвращается вызывающей стороне. Оно также добавляется в свойство `actualName` объекта `Queue`. 

Эту функциональность можно использовать программно, только вызывая `RabbitAdmin` напрямую. 
> При использовании автоматического объявления администратором при определении очереди декларативно в контексте приложения можно установить свойство `name` в `""` (пустую строку). 
> Затем брокер создает имя. 
> 
> Начиная с версии 2.1, контейнеры слушателей могут использовать очереди такого типа. Для получения дополнительной информации см. **Контейнеры и очереди, именованные брокером (Containers and Broker-Named queues**

### 3. Анонимная очередь
Это отличается от `AnonymousQueue`: в этом случае **имя генерирует сам фреймворк** (UUID), 
а *очередь создаётся с параметрами* 
- `durable=false`, 
- `exclusive=true`, 
- `autoDelete=true`. 

В XML-конфигурации `<rabbit:queue/>` с пустым или отсутствующим атрибутом `name` всегда создаёт именно `AnonymousQueue`.

`AnonymousQueue` обычно предпочтительнее брокер-именованных очередей, так как даёт полный контроль над форматом имени и поведением. 
Начиная с версии **2.1**, ***такие очереди объявляются с аргументом*** 
`x-queue-leader-locator=client-local` (`Queue.X_QUEUE_LEADER_LOCATOR`), что **гарантирует создание очереди на том узле брокера**, к **которому ПОДКЛЮЧЕНО ПРИЛОЖЕНИЕ**. 

Декларативные очереди, напротив, должны иметь **фиксированные имена**, поскольку на них могут ссылаться другие бины и компоненты контекста (например, контейнеры слушателей).
```xml
<rabbit:listener-container>
    <rabbit:listener ref="listener" queue-names="#{someQueue.name}" />
</rabbit:listener-container>
```
> См. **Автоматическое объявление обменников, очередей и привязок (Automatic Declaration of Exchanges, Queues, and Bindings)**.

## РЕАЛИЗАЦИЯ - RabbitAdmin
Реализация этого интерфейса для RabbitMQ — `RabbitAdmin`, который при настройке с использованием Spring XML выглядит следующим образом:
```xml
<rabbit:connection-factory id="connectionFactory"/>

<rabbit:admin id="amqpAdmin" connection-factory="connectionFactory"/>
```

**Когда режим кеширования** `CachingConnectionFactory` — `CHANNEL` (по умолчанию), реализация `RabbitAdmin` ***автоматически*** **ЛЕНИВО ОБЪЯВЛЯЕТ очереди, обменники и привязки**, объявленные в том же `ApplicationContext`. 
> Эти компоненты ==объявляются, как только устанавливается соединение== `Connection` с брокером. 

Существуют некоторые возможности пространства имен, которые делают это очень удобным — например, в примере приложения Stocks у нас есть следующее:
```xml
<rabbit:queue id="tradeQueue"/>
<rabbit:queue id="marketDataQueue"/>

<fanout-exchange name="broadcast.responses"
                 xmlns="http://www.springframework.org/schema/rabbit">
    <bindings>
        <binding queue="tradeQueue"/>
    </bindings>
</fanout-exchange>


<topic-exchange name="app.stock.marketdata"
                xmlns="http://www.springframework.org/schema/rabbit">
    <bindings>
        <binding queue="marketDataQueue" 
		         pattern="${stocks.quote.pattern}"/>
    </bindings>
</topic-exchange>
```
В предыдущем примере мы 
- *используем анонимные очереди* (***фактически, внутренне просто очереди с именами, сгенерированными фреймворком, а не брокером***) 
- и ссылаемся на них по ID. 

Мы также можем объявлять очереди с явными именами, которые также служат идентификаторами их определений бинов в контексте.

Следующий пример настраивает очередь с явным именем:
```xml
<rabbit:queue name="stocks.trade.queue"/>
```

Вы можете предоставить как атрибуты `id`, так и `name`. 
Это позволяет ссылаться на очередь (например, в привязке) по ID, который не зависит от имени очереди. 

> Это также позволяет использовать стандартные функции Spring (например, плейсхолдеры свойств и выражения SpEL для имени очереди). 
> 
> Эти функции НЕДОСТУПНЫ, когда вы используете имя в качестве идентификатора бина.


### 1. Аргументы очередей
Очереди могут быть настроены с ==дополнительными аргументами== — например, `x-message-ttl`. 
> *При использовании поддержки пространства имен* они **предоставляются в виде** `Map` пар 
> - имя-аргумента/значение-аргумента, 
> 
> которые определяются с помощью элемента `<rabbit:queue-arguments>`. 

Следующий пример показывает, как это сделать:
```xml
<rabbit:queue name="withArguments">

    <rabbit:queue-arguments>
        <entry key="x-dead-letter-exchange" value="myDLX"/>
        <entry key="x-dead-letter-routing-key" value="dlqRK"/>
    </rabbit:queue-arguments>
    
</rabbit:queue>
```

По умолчанию **аргументы считаются строками**.

***Для аргументов других типов*** необходимо указать тип. 
Следующий пример показывает, как указать тип:
```xml
<rabbit:queue name="withArguments">

    <rabbit:queue-arguments value-type="java.lang.Long">
        <entry key="x-message-ttl" value="100"/>
    </rabbit:queue-arguments>
    
</rabbit:queue>
```

При предоставлении ***аргументов смешанных типов*** необходимо указать тип для каждого элемента `entry`. 
Следующий пример показывает, как это сделать:
```xml
<rabbit:queue name="withArguments">

    <rabbit:queue-arguments>
        <entry key="x-message-ttl">
            <value type="java.lang.Long">100</value>
        </entry>
        <entry key="x-dead-letter-exchange" value="myDLX"/>
        <entry key="x-dead-letter-routing-key" value="dlqRK"/>
    </rabbit:queue-arguments>
    
</rabbit:queue>
```

Начиная с Spring Framework 3.2 и выше, это можно объявить немного более кратко, следующим образом:
```xml
<rabbit:queue name="withArguments">


    <rabbit:queue-arguments>
    
        <entry key="x-message-ttl" value="100" value-type="java.lang.Long"/>
        
        <entry key="x-ha-policy" value="all"/>
        
    </rabbit:queue-arguments>
    
    
</rabbit:queue>
```


При использовании ==Java-конфигурации== аргумент `Queue.X_QUEUE_LEADER_LOCATOR` поддерживается как свойство первого класса через метод `setLeaderLocator()` в классе `Queue`. 

> [!NOTE] 
> Начиная с версии 2.1, анонимные очереди объявляются с этим свойством, **установленным по умолчанию в** `client-local`. 
> 
> Это гарантирует, что очередь объявляется на узле, к которому подключено приложение.

> Брокер RabbitMQ ***НЕ ПОЗВОЛЯЕТ объявлять очередь*** **с несовпадающими аргументами.** 
> 
> Например, если очередь уже существует без аргумента времени жизни (time to live), 
> 
> и вы пытаетесь объявить ее с (например) `key="x-message-ttl" value="100"`, будет выброшено исключение.


> [!WARNING] Исключение при повторном объелении
> Важно: **RabbitMQ не разрешает повторно объявлять очередь с другими аргументами**. 
> Если очередь уже существует с иными параметрами — будет выброшено исключение

### 2. Ошибки при объявлении
По умолчанию `RabbitAdmin` **прерывает все объявления при первой ошибке**.  

Это поведение можно изменить:
- `ignoreDeclarationExceptions=true` — логировать ошибку и продолжать объявление остальных элементов.
	(`ignore-declaration-exceptions` - аналогичное свойство в xml)
    
- ==Это глобальная настройка==, применяемая ко всем элементам. 
	- *Очереди, обменники и привязки* **имеют АНАЛОГИЧНОЕ свойство**, которое **применяется только к этим** элементам.


Начиная с версии **1.6**, эта настройка действует **ДЛЯ ЛЮБЫХ ИСКЛЮЧЕНИЙ**, а не только `IOException`.

Кроме того, любые исключения при объявлении приводят к публикации события `DeclarationExceptionEvent`, которое является `ApplicationEvent` и может быть использовано любым `ApplicationListener` в контексте. 
Событие содержит ссылку на администратора, элемент, который объявлялся, и `Throwable`.

# 2. Обменник типа Headers
**HeadersExchange** позволяет маршрутизировать сообщения по заголовкам.

Начиная с версии **1.3**, 
- можно задавать несколько заголовков 
- и указывать, должны совпасть **все** или **любые** из них (через `x-match=all|any`). 
> Это настраивается в binding’ах.

Следующий пример показывает, как это сделать:
```xml
<rabbit:headers-exchange name="headers-test">

    <rabbit:bindings>
    
        <rabbit:binding queue="bucket">
            <rabbit:binding-arguments>
                <entry key="foo" value="bar"/>
                <entry key="baz" value="qux"/>
                <entry key="x-match" value="all"/>
            </rabbit:binding-arguments>
        </rabbit:binding>
        
    </rabbit:bindings>
    
</rabbit:headers-exchange>
```

##### Внутренний обменник (Internal Exchange)
С версии **1.6** обменники можно помечать как **internal**. 

Внутренний обменник корректно объявляется через `RabbitAdmin`

Если **internal** флаг установлен в `true` для обменника, RabbitMQ ==НЕ ПОЗВОЛЯЕТ КЛИЕНТАМ использовать этот обменник==. 
> Это полезно для обменника недоставленных сообщений (dead letter exchange) или для обменника-к-обменнику привязки, где вы не хотите, чтобы обменник использовался издателями напрямую.

## Java-конфигурация (пример Stock)
Чтобы увидеть, как использовать Java для настройки инфраструктуры AMQP, посмотрите пример приложения Stock.

У нас есть класс `@Configuration` `AbstractStockRabbitConfiguration`, который, в свою очередь, имеет подклассы 
- `RabbitClientConfiguration` 
- и `RabbitServerConfiguration`. 

Следующий листинг показывает код для `AbstractStockRabbitConfiguration`
```java
@Configuration
public abstract class AbstractStockAppRabbitConfiguration {

    @Bean
    public CachingConnectionFactory connectionFactory() {
    
        CachingConnectionFactory connectionFactory =
            new CachingConnectionFactory("localhost");
            
        connectionFactory.setUsername("guest");
        connectionFactory.setPassword("guest");
        
        return connectionFactory;
    }

    @Bean
    public RabbitTemplate rabbitTemplate() {
    
        RabbitTemplate template = 
	        new RabbitTemplate(connectionFactory());
	        
        template.setMessageConverter(jsonMessageConverter());
        configureRabbitTemplate(template);
        
        return template;
    }

    @Bean
    public Jackson2JsonMessageConverter jsonMessageConverter() {
        return new Jackson2JsonMessageConverter();
    }

    @Bean
    public TopicExchange marketDataExchange() {
        return new TopicExchange("app.stock.marketdata");
    }

    // дополнительный код опущен для краткости
}
```

### 1. Серверная конфигурация
В приложении Stock **сервер настроен с использованием следующего класса** `@Configuration`:
```java
@Configuration
public class RabbitServerConfiguration 
extends AbstractStockAppRabbitConfiguration  {

    @Bean
    public Queue stockRequestQueue() {
        return new Queue("app.stock.request");
    }
}
```
**Серверная конфигурация** добавляет очередь и при старте приложения объявляет её вместе с exchange’ом на брокере. 

Явной привязки нет — очередь автоматически привязывается к **дефолтному exchange** (поведение по спецификации AMQP).

### 2. Клиентская конфигурация:
Клиентский класс `@Configuration` немного интереснее. Его объявление следующее:
```java
@Configuration
public class RabbitClientConfiguration 
extends AbstractStockAppRabbitConfiguration {

    @Value("${stocks.quote.pattern}")
    private String marketDataRoutingKey;

    @Bean
    public Queue marketDataQueue() {
        return amqpAdmin().declareQueue();
    }

    /**
     * Привязывается к обменнику рыночных данных.
     * Интересуется любыми котировками акций,
     * которые соответствуют его ключу маршрутизации.
     */
    @Bean
    public Binding marketDataBinding() {
        return BindingBuilder
	        .bind(marketDataQueue())
	        .to(marketDataExchange())
	        .with(marketDataRoutingKey);
    }

    // дополнительный код опущен для краткости
}
```
- программно объявляет очередь через `AmqpAdmin.declareQueue()` (часто с автосгенерированным именем),
    
- привязывает её к `TopicExchange` с routing key, вынесенным в свойства.


Итог: инфраструктура AMQP (exchange, очереди, привязки) объявляется **декларативно при старте**, сервер и клиент разделяют ответственность, а маршрутизация управляется конфигурацией.
