Существует довольно много параметров для настройки 
- `SimpleMessageListenerContainer` (SMLC) 
- и `DirectMessageListenerContainer` (DMLC), 

связанных с транзакциями и качеством обслуживания, причем некоторые из них взаимодействуют друг с другом. 

С**войства, применимые к SMLC, DMLC или** `StreamListenerContainer` (StLC) (см. "Использование плагина RabbitMQ Stream"), **обозначены галочкой в соответствующем столбце**. 
> См. "Выбор контейнера" для получения информации, которая поможет вам решить, какой контейнер подходит для вашего приложения.

В следующей таблице показаны имена свойств контейнера и их эквивалентные имена атрибутов (в скобках) при использовании пространства имен для настройки `<rabbit:listener-container/>`. 

Атрибут `type` этого элемента может быть `simple` (по умолчанию) или `direct` для указания SMLC или DMLC соответственно. 

Некоторые свойства не предоставляются пространством имен. Они обозначены как N/A для атрибута.

# Свойства
## Свойства, связанные с подтверждениями (Acknowledgment) и таймаутами

### Время ожидания подтверждения (ackTimeout)
**N/A** - Когда установлено свойство `messagesPerAck`, этот таймаут используется в качестве альтернативы для отправки подтверждения (ack). При поступлении нового сообщения количество неподтвержденных сообщений сравнивается с `messagesPerAck`, а время, прошедшее с последнего подтверждения, сравнивается с этим значением. Если выполняется любое из условий, сообщение подтверждается. Когда новые сообщения не поступают и есть неподтвержденные сообщения, этот таймаут является приблизительным, так как условие проверяется только с интервалом `monitorInterval`. См. также `messagesPerAck` и `monitorInterval` в этой таблице.

**SMLC (SimpleMessageListenerContainer)** ✅  
**DMLC (DirectMessageListenerContainer)** ✅  
**StLC (StreamListenerContainer)** ❌

---
### Режим подтверждения (acknowledgeMode)
**acknowledge** -
- `NONE`: Подтверждения не отправляются (несовместимо с `channelTransacted=true`). RabbitMQ называет это "autoack", потому что брокер считает все сообщения подтвержденными без каких-либо действий со стороны потребителя.
    
- `MANUAL`: Слушатель должен подтвердить все сообщения, вызвав `Channel.basicAck()`.
    
- `AUTO`: Контейнер подтверждает сообщение автоматически, если только `MessageListener` не выбрасывает исключение. Обратите внимание, что `acknowledgeMode` дополняет `channelTransacted` — если канал является транзакционным, брокеру требуется уведомление о фиксации (commit) в дополнение к подтверждению (ack). Это режим по умолчанию. См. также `batchSize`.
    

**SMLC** ✅  
**DMLC** ✅  
**StLC** ✅

## Свойства, связанные с автоматическим объявлением и запуском

### Автоматическое объявление (autoDeclare)
**auto-declare** - Когда установлено в `true` (по умолчанию), контейнер использует `RabbitAdmin` для повторного объявления всех объектов AMQP (очередей, обменников, привязок), если обнаруживает, что хотя бы одна из его очередей отсутствует во время запуска, возможно, потому что это автоудаляемая (auto-delete) или истекшая очередь, но повторное объявление происходит, если очередь отсутствует по любой причине. Чтобы отключить это поведение, установите это свойство в `false`. Обратите внимание, что контейнер не запустится, если все его очереди отсутствуют.

> До версии 1.6, если в контексте было больше одного администратора, контейнер выбирал случайный. Если администраторов не было, он создавал его внутри себя. В любом случае это могло привести к неожиданным результатам. Начиная с версии 1.6, для работы `autoDeclare` в контексте должен быть ровно один `RabbitAdmin`, или ссылка на конкретный экземпляр должна быть настроена в контейнере с помощью свойства `rabbitAdmin`.

**SMLC** ✅  
**DMLC** ✅  
**StLC** ❌

По умолчанию `true`, но можно установить `false`, если брокер может быть недоступен при запуске, и позже вручную вызвать `start()`, когда брокер будет готов.

**SMLC** ✅

## Свойства, связанные с пакетной обработкой (Batching)

### Размер пачки (batchSize)
**transaction-size, batch-size** - При использовании с `acknowledgeMode`, установленным в `AUTO`, контейнер пытается обработать до этого количества сообщений перед отправкой подтверждения (ожидая каждое до таймаута приема `receiveTimeout`). Также в этот момент фиксируется транзакционный канал. Если `prefetchCount` меньше, чем `batchSize`, он увеличивается до значения `batchSize`.

**SMLC** ✅  
**DMLC** ❌  
**StLC** ❌

---
### Включение пакетной обработки в потребителе (consumerBatchEnabled)
**batch-enabled** - Если `MessageListener` поддерживает это, установка значения `true` включает пакетирование отдельных сообщений, до `batchSize`; неполная пачка будет доставлена, если новые сообщения не поступают в течение `receiveTimeout` или время сбора пачки превысило `batchReceiveTimeout`. Когда `false`, пакетирование поддерживается только для пачек, созданных производителем; см. "Пакетирование (Batching)".

**SMLC** ✅  
**DMLC** ❌  
**StLC** ❌

---
### Таймаут приема пачки (batchReceiveTimeout)
**batch-receive-timeout** - Таймаут в миллисекундах для сбора сообщений в пачку. Ограничивает время ожидания заполнения `batchSize`. Когда `batchSize` > 1 и время сбора сообщений в пачку больше `batchReceiveTime`, пачка будет доставлена. По умолчанию 0 (без таймаута).

**SMLC** ✅  
**DMLC** ❌  
**StLC** ❌

---
### Стратегия пакетирования (batchingStrategy)
**N/A** - Стратегия, используемая при разбиении пачек сообщений (debatching). По умолчанию `SimpleDebatchingStrategy`. См. "Пакетирование и @RabbitListener с пакетированием".

**SMLC** ✅  
**DMLC** ✅  
**StLC** ❌

---
### Включение разбиения пачек (debatchingEnabled)

**N/A** - Если `true`, контейнер слушателя будет разбивать пачки сообщений и вызывать слушатель для каждого сообщения из пачки. Начиная с версии 2.2.7, пачки, созданные производителем, будут разбиты как `List<Message>`, если слушатель является `BatchMessageListener` или `ChannelAwareBatchMessageListener`. В противном случае сообщения из пачки представляются по одному. По умолчанию `true`. См. "Пакетирование и @RabbitListener с пакетированием".

**SMLC** ✅  
**DMLC** ✅  
**StLC** ❌

### Группа (group)

**(group)** - Доступно только при использовании пространства имен. При указании регистрируется бин типа `Collection<MessageListenerContainer>` с этим именем, и контейнер для каждого элемента `<listener/>` добавляется в коллекцию. ### Количество предварительной выборки (prefetchCount)

**prefetch** - Количество неподтвержденных сообщений, которые могут быть на обработке у каждого потребителя. Чем выше это значение, тем быстрее могут доставляться сообщения, но выше риск неупорядоченной обработки. Игнорируется, если `acknowledgeMode` — `NONE`. Увеличивается при необходимости, чтобы соответствовать `batchSize` или `messagePerAck`. Начиная с версии 2.0 по умолчанию 250. Можно установить в 1, чтобы вернуться к предыдущему поведению.

> Есть сценарии, когда значение предварительной выборки должно быть низким — например, при работе с большими сообщениями, особенно если обработка медленная (сообщения могут накапливаться в большом объеме памяти в процессе клиента), и если необходим строгий порядок сообщений (в этом случае значение предварительной выборки должно быть установлено обратно в 1). Также при низком объеме сообщений и нескольких потребителях (включая конкурентность в одном экземпляре контейнера слушателя) вы можете уменьшить предварительную выборку, чтобы получить более равномерное распределение сообщений между потребителями.  
> Также см. `globalQos`.

## Свойства, связанные с несовпадением и отсутствием очередей

### Фатальное несовпадение очередей (mismatchedQueuesFatal)
**mismatched-queues-fatal** - Когда контейнер запускается, если это свойство `true` (по умолчанию: `false`), контейнер проверяет, что все очереди, объявленные в контексте, совместимы с очередями, уже существующими на брокере. Если существуют несовпадающие свойства (такие как `auto-delete`) или аргументы (такие как `x-message-ttl`), контейнер (и контекст приложения) не запускается с фатальным исключением.

> Если проблема обнаружена во время восстановления (например, после потери соединения), контейнер останавливается.
> 
> В контексте приложения должен быть ровно один `RabbitAdmin` (или один, специально настроенный на контейнере с помощью свойства `rabbitAdmin`). В противном случае это свойство должно быть `false`.
> 
> Если брокер недоступен во время первоначального запуска, контейнер запускается, и условия проверяются при установлении соединения.
> 
> Проверка выполняется для всех очередей в контексте, а не только для очередей, которые настроены для использования конкретным слушателем. Если вы хотите ограничить проверку только теми очередями, которые использует контейнер, вы должны настроить отдельный `RabbitAdmin` для контейнера и предоставить ссылку на него, используя свойство `rabbitAdmin`. См. "Условное объявление" для получения дополнительной информации.
> 
> Обнаружение несовпадения аргументов очереди отключено при запуске контейнера для `@RabbitListener` в бине, помеченном как `@Lazy`. Это сделано для того, чтобы избежать потенциальной взаимной блокировки (deadlock), которая может задержать запуск таких контейнеров до 60 секунд. Приложения, использующие ленивые бины слушателей, должны проверять аргументы очереди перед получением ссылки на ленивый бин.

**SMLC** ✅  
**DMLC** ✅  
**StLC** ❌

### Фатальное отсутствие очередей (missingQueuesFatal)

**missing-queues-fatal** - Когда установлено в `true` (по умолчанию), если ни одна из настроенных очередей недоступна на брокере, это считается фатальным. Это приводит к неудачной инициализации контекста приложения во время запуска. Кроме того, когда очереди удаляются во время работы контейнера, по умолчанию потребители делают три попытки подключиться к очередям (с интервалом в пять секунд) и останавливают контейнер, если эти попытки не удаются.

> Это свойство не было настраиваемым в предыдущих версиях.
> 
> Когда установлено в `false`, после трех попыток контейнер переходит в режим восстановления, как и при других проблемах, таких как отключение брокера. Контейнер пытается восстановиться в соответствии со свойством `recoveryInterval`. Во время каждой попытки восстановления каждый потребитель снова пытается четыре раза пассивно объявить очереди с интервалом в пять секунд. Этот процесс продолжается бесконечно.
> 
> Вы также можете использовать бин свойств, чтобы установить это свойство глобально для всех контейнеров.

**SMLC** ✅  
**DMLC** ✅  
**StLC** ❌
