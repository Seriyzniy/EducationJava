Понимание формата **WAR-файла** (Web Application Archive) — это базовый навык для любого Java-веб-разработчика, особенно если ты работаешь с **сервлетами, JSP или Jakarta EE**

### WAR vs JAR

| Критерий     | WAR                               | JAR                                 |
| ------------ | --------------------------------- | ----------------------------------- |
| Назначение   | Веб-приложения                    | Обычные Java-программы / библиотеки |
| 🟢Запуск     | Через `контейнер` (Tomcat, Jetty) | Через `JVM` (`java -jar`)           |
| Содержимое   | Сервлеты, JSP, web.xml, ресурсы   | Классы, ресурсы, `MANIFEST.MF`      |
| Главный файл | `WEB-INF/web.xml`                 | `META-INF/MANIFEST.MF`              |
| Расширение   | `.war`                            | `.jar`                              |
# 📕 1. Архив WAR
> [!NOTE] **WAR (Web Application ARchive)** — это `архив` (по сути, обычный `.zip`), 
> в котором `упаковано` `веб-приложение на Java` ==в формате==, который ==понимают **веб-контейнеры**== (Tomcat, Jetty, GlassFish и др).

WAR-файл содержит:
- 💡твои `.class`-файлы (скомпилированный код сервлетов),
- ресурсы (HTML, JSP, CSS, JS, картинки),
- 💡файл конфигурации `web.xml`,
    
- 🧠и все зависимости (в `WEB-INF/lib`).

Контейнер просто «распаковывает» `.war` и разворачивает приложение как веб-сайт.

## Структура WAR-файла
Пример типовой структуры:

```pgsql
myapp.war
│
├── META-INF/
│   └── MANIFEST.MF
│
├── WEB-INF/
│   ├── web.xml              ← обязательно здесь!
│   ├── classes/
│   │   └── com/example/
│   │       └── MyServlet.class
│   └── lib/
│       └── (тут твои .jar-зависимости, если есть)
│
└── (опционально статические файлы)
    ├── index.html
    ├── css/
    └── js/
```

| Папка / файл                  | Назначение                                               |
| ----------------------------- | -------------------------------------------------------- |
| `/`                           | Статические файлы (HTML, CSS, JS, изображения)           |
|                               |                                                          |
| `WEB-INF/`                    | ==Закрытая часть== (невидима пользователю через браузер) |
| `WEB-INF/web.xml`             | 💡Конфигурация сервлетов, фильтров, слушателей           |
| `WEB-INF/classes/`            | 💡Скомпилированные `.class`-файлы твоего приложения      |
| `WEB-INF/lib/`                | 💡JAR-библиотеки, используемые внутри приложения         |
| `META-INF/` _(необязательно)_ | Метаинформация об архиве (например, `MANIFEST.MF`)       |

## Чтение WAR файла контейнером
Когда ты помещаешь `MyWebApp.war` в каталог `webapps/` Tomcat, он делает следующее:
1. Распаковывает `.war` во временную папку:
```bash
$CATALINA_HOME/webapps/MyWebApp/
```

2. Читает `WEB-INF/web.xml` (или аннотации `@WebServlet`).
    
3. Регистрирует сервлеты, фильтры, слушателей.
    
4. Делает ресурсы из `/` доступными по HTTP (например `/index.html`).

# 📕 2. Сборка файла
## Создание WAR вручную (без Maven)
Для WAR архива необходимо
- Скомпилированные файлы .class
- Файл web.xml с необходимыми настройками
- Другие библиотеки по необходимости в формате JAR

Затем
```bash
jar --create --file MyWebApp.war -C MyWebApp .
```

- `jar --create` — создать архив;
    
- `--file MyWebApp.war` — имя выходного файла;
    
- `-C MyWebApp .` — взять содержимое каталога `MyWebApp`.

## Сборка через Maven
Maven сам умеет собирать WAR, если указать тип упаковки:
```xml
<packaging>war</packaging>
```

и структуру проекта (Maven сам создаст нужные каталоги):
```css
src/
 ├── main/
 │   ├── java/
 │   │   └── com/example/HelloServlet.java
 │   ├── resources/
 │   └── webapp/
 │       ├── WEB-INF/
 │       │   └── web.xml
 │       └── index.html
```

Сборка:
```bash
mvn clean package
```

Результат:
```css
target/MyWebApp.war
```

---
# ⚠️ Важно: `servlet-api.jar` не кладётся в WAR

Tomcat уже содержит Jakarta Servlet API.  
Если ты положишь его в `WEB-INF/lib` — получишь **ClassCastException**  
(два разных `jakarta.servlet.Servlet` в разных ClassLoader-ах).

# 🐋WAR и Docker

В Docker-контейнерах Tomcat ожидает именно `.war`.

Пример Dockerfile:
```dockerfile
FROM tomcat:10.1-jdk17-temurin
COPY MyWebApp.war /usr/local/tomcat/webapps/ROOT.war
EXPOSE 8080
```

После сборки:
```bash
docker build -t mywebapp .
docker run -d -p 8080:8080 mywebapp
```
