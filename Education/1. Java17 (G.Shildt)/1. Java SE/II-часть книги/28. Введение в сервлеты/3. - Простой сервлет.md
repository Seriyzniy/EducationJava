После чего можно приступать к написанию кода сервелета, необходимые импорты ИЗ СТОРОННЕЙ БИБЛИОТЕКИ (.jar) подтянутся

Первым делом нужно создать файл по имени HelloServlet.java
```java
package servlets; // ✅ПАКЕТ servlets, ЗНАЧИТ И В classes должна быть эта папка

import java.io.IOException;
import java.io.PrintWriter;

import jakarta.servlet.GenericServlet;
import jakarta.servlet.ServletException;
import jakarta.servlet.ServletRequest;
import jakarta.servlet.ServletResponse;

public class HelloServlet extends GenericServlet {

	@Override
	public void service(
		ServletRequest arg0, 
		ServletResponse arg1) throws ServletException, IOException 
	{
		arg1.setContentType("text/html");
		
        PrintWriter pw = arg1.getWriter();
        pw.println("<B>Hello!</B>");
        pw.close();
		
	}
	
}
```

## Проанализируем код. 

Прежде всего, обратите внимание, что в нем импортируется пакет `jakarta.servlet`, который содержит классы и интерфейсы, необходимые для создания сервлетов.

Затем определяется класс `HelloServlet` ==как подкласс== `GenericServlet`. 

Класс `GenericServlet` обеспечивает функциональность, упрощающую создание сервлета. Например, он предоставляет версии методов `init()` и `destroy()` , которые можно применять в том виде как есть. 
💡Понадобится реализовать только метод `service()` .

Внутри класса `HelloServlet` переопределяется метод `service()` , унаследованный из `GenericServlet`, который обрабатывает запросы от клиента. 

Обратите внимание, что:
- 🔶Первый аргумент представляет собой объект `ServletRequest`, который дает сервлету ==возможность читать данные==, предоставленные через ==клиентский запрос==. 
- 🔶Второй аргумент - объект `ServletResponse`, позволяющий сервлету формировать ==ответ для клиента==.

Вызов `setContentType()` устанавливает МIМЕ-тип НТТР-ответа. 
Здесь МIМЕ-типом является `text/html`, указывающий на то, что браузер должен интерпретировать содержимое как исходный код HTML.

Далее метод `getWriter()` получает поток `PrintWriter` ([[4.6 - PrintWriter]]). 
Все, что записывается в данный поток, отправляется клиенту в виде части НТТР-ответа. 
Затем в качестве НТТР-ответа с помощью `println()` записывается простой исходный код HTML.

После того, как добавили `servlet-api.jar` в проект, и написали класс, скомпилируем его.

Т.к в класе используется импорт из сторонней библиотеки, для компиляции НЕОБХОДИМО УКАЗАТЬ CLASSPATH до библиотеки

💡Раз мы уже добавили в CLASSPATH путь до библиотеки, то `в папке с проектом будет файл` `classpath`
```xml

<classpath>
	<classpathentry exported="true" kind="con" path="org.eclipse.jdt.launching.JRE_CONTAINER/org.eclipse.jdt.internal.debug.ui.launcher.StandardVMType/JavaSE-21">
		<attributes>
			<attribute name="module" value="true"/>
		</attributes>
	</classpathentry>
	<classpathentry kind="src" path="src"/>
	<classpathentry kind="lib" path="D:/Education_Java/JavaProgramming/ToolsAndConfig/ApacheServers/apache-tomcat-10.1/apache-tomcat-10.1.28/lib/servlet-api.jar"/>
	<classpathentry kind="output" path="bin"/>

```

Здесь указан путь до `servlet-api.jar`

Теперь скопируем путь и напишем команду для компиляции в консоле

> [!NOTE] Ключ `-cp` == `--classpath`

# 📕 3. Тест сервлета
Для тестирования используется URL:
```url
http://localhost:8080/examples/servlets/servlet/HelloServlet
```

## Почему в URL появляется `/examples`
Tomcat по умолчанию разворачивает каждое веб-приложение **в свой контекст (context path)**.

Этот контекст — **имя папки внутри `webapps`**, где лежит твой проект.

Папка `examples` - это просто одно из стандартных демо-приложений Tomcat, куда по умолчанию помещают сервлеты для тестирования.****
