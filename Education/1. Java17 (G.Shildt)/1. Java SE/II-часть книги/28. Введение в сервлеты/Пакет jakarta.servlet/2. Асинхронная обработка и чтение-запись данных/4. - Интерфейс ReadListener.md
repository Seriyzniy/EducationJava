
> [!NOTE] `ReadListener` — интерфейс из пакета **`javax.servlet`** (Servlet API 3.1+),  
> Предназначенный для `асинхронного и неблокирующего` `ЧТЕНИЯ ДАННЫХ` из `ServletInputStream`.

Он работает в паре с:
- `AsyncContext` — для асинхронного режима,
    
- `ServletInputStream` — в неблокирующем режиме (`isReady()` / `setReadListener()`).


**Используется с:** `ServletInputStream.setReadListener(ReadListener listener)`

---
## Интерфейс
```java
public interface ReadListener extends java.util.EventListener {
    void onDataAvailable() throws IOException;
    void onAllDataRead() throws IOException;
    void onError(Throwable t);
}
```

## Методы

### Методы обработки состояния потока
```java
// Вызывается, когда ❗часть данных доступна для чтения без блокировки❗
void onDataAvailable() throws IOException;

// Вызывается, ❗когда все данные из потока были успешно прочитаны❗
void onAllDataRead() throws IOException;
```

### Методы обработки ошибок
```java
// Вызывается, если при чтении произошла ошибка (например, разрыв соединения)
void onError(Throwable t);
```

---
## 🧠Под капотом
Все ==методы== `ReadListener` ==вызываются **контейнером сервлетов** ==(например, Tomcat, Jetty, Undertow, GlassFish и т.д.),  
а именно — **внутренним механизмом асинхронного ввода-вывода (non-blocking I/O)**, который основан на **NIO Selector / Poller** внутри контейнера.

### Как это работает «под капотом»

#### 🔹 1. Клиент делает POST-запрос
Контейнер принимает соединение, создаёт `HttpServletRequest`, `ServletInputStream` и передаёт управление в `doPost()`.

#### 🔹 2. Ты вызываешь `req.startAsync()`
→ Контейнер переводит запрос в **асинхронное состояние**,  

создаёт объект `AsyncContext` и **отвязывает** текущий поток от обработки запроса.  
(т.е. поток-воркер, который принял запрос, теперь свободен и возвращается в пул)

#### 🔹 3. Ты вызываешь `input.setReadListener(listener)`
→ Контейнер регистрирует слушателя в **внутреннем I/O механизме**.  
Теперь этот `ReadListener` привязан к сокету клиента.

#### ⚡ Что происходит дальше
| Событие                                              | Кто вызывает                                    | Что делает контейнер                                                                                   |
| ---------------------------------------------------- | ----------------------------------------------- | ------------------------------------------------------------------------------------------------------ |
| ♻Входящие данные появляются в сокете                 | Поток-поллер (`PollerThread`, `SelectorThread`) | Проверяет, что `ServletInputStream.isReady() == true`, <br><br>и вызывает `listener.onDataAvailable()` |
| ✅Все данные получены (конец потока)                  | Контейнер                                       | Вызывает `listener.onAllDataRead()`                                                                    |
| ❌ Ошибка при чтении (разрыв соединения, IOException) | Контейнер                                       | Вызывает `listener.onError(Throwable)`                                                                 |
| ⏱ Таймаут асинхронного контекста                     | Контейнер                                       | Вызывает `AsyncListener.onTimeout()` (а не `ReadListener`)                                             |
### ❗🧠Важно понимать

- Методы `onDataAvailable()` и `onAllDataRead()` **не вызываются пользователем вручную**.  
    Они **триггерятся событиями на уровне сетевого сокета**, аналогично `SelectionKey.OP_READ` в Java NIO.
    
- Контейнер **следит за готовностью канала к чтению**.  
    Когда есть новые данные в сокете, он вызывает твой `ReadListener` в **рабочем потоке контейнера**,  
    а не в основном потоке запроса (тот уже освобождён).
    
- Каждый вызов происходит **асинхронно** и **не блокирует** поток.

```scss
┌───────────────┐
│ Клиент пишет  │
│   в сокет     │
└──────┬────────┘
       │ (данные поступают)
       ▼
┌──────────────────────────┐
│  Container Poller Thread │ — следит за готовностью каналов
└────────┬─────────────────┘
         │
         ▼
┌─────────────────────────────────────────────┐
│ вызывает listener.onDataAvailable()         │
│ контейнерный поток читает данные            │
│ через ServletInputStream.read()             │
└─────────────────────────────────────────────┘
         │
         ▼
┌────────────────────────┐
│ listener.onAllDataRead │ — все данные прочитаны
└────────────────────────┘
         │
         ▼
┌──────────────────────────┐
│ asyncContext.complete()  │ — закрывает ответ
└──────────────────────────┘

```

## 💡 Особенности и подводные камни 💡

1️⃣ 💡 **Работает ==только в асинхронном режиме==.**  
Перед установкой `ReadListener` нужно вызвать `req.startAsync()`.  
Иначе — `IllegalStateException`.

2️⃣ ❗❗❗**НЕ ИСПОЛЬЗУЙ ЦИКЛ ЧТЕНИЯ!**  
Метод `onDataAvailable()` может ==вызываться **многократно**==, ==по мере поступления данных==,  
❗❗❗поэтому внутри него **нельзя просто делать `read()` в цикле до конца**,  
нужно проверять `inputStream.isReady()`.

3️⃣ **`onAllDataRead()` вызывается ровно один раз**,  
когда все данные считаны. После этого можно вызывать `asyncContext.complete()`.

4️⃣ **Ошибки и таймауты не вызывают `onAllDataRead()`.**  
Если чтение прерывается — будет вызван `onError(Throwable)`.

5️⃣ ⭐⭐**==Методы вызываются контейнером==**, а не пользователем,  
и обычно — в **потоках контейнера**, а не в пользовательском `Executor`.

6️⃣ **`onDataAvailable()` может быть вызван снова, даже если данные закончились**,  
❗❗❗поэтому всегда проверяй `inputStream.isFinished()` перед чтением.

7️⃣ ❗❗❗**Не забывай завершать асинхронный контекст (`asyncContext.complete()`)**  
— иначе соединение «повиснет» навсегда.

---
## 💎 Best Practices 💎

✅ 💡Перед установкой слушателя обязательно вызвать `request.startAsync()`. 

✅ 💡Используй цикл чтения **только при `inputStream.isReady()`**.  

✅ ❗❗❗После `onAllDataRead()` всегда вызывай `asyncContext.complete()`.  

✅ Обрабатывай исключения внутри `onError()` корректно (например, логируй и заверши контекст).  

✅ Избегай тяжёлых вычислений в методах — они выполняются в потоках контейнера.  

✅ ⭐⭐Храни `AsyncContext` как поле, чтобы иметь доступ к нему из всех колбэков.

## Пример
```java
@WebServlet(value = "/async-read", asyncSupported = true)
public class AsyncReadServlet extends HttpServlet {

    @Override
    protected void doPost(
	    HttpServletRequest req, 
	    HttpServletResponse resp) throws ServletException, IOException 
	{
        // Инициализируем асинхронный контекст
        AsyncContext asyncContext = req.startAsync();

        // Получаем неблокирующий поток
        ServletInputStream input = req.getInputStream();


        // Устанавливаем слушателя чтения
        input.setReadListener(new ReadListener() {
            private StringBuilder buffer = new StringBuilder();

            @Override
            public void onDataAvailable() throws IOException {
                byte[] data = new byte[1024];
                int len;
                // 🟢Читаем только когда поток готов🟢
                while (//всспомни про коротко замыкающие булевы выражения
	                input.isReady() && (len = input.read(data)) != -1
	            ) {
                    buffer.append(new String(data, 0, len));
                }
            }

            @Override
            public void onAllDataRead() throws IOException {
                // Все данные прочитаны — формируем ответ
                HttpServletResponse response = 
	                (HttpServletResponse) asyncContext.getResponse();
                
                response.setContentType("text/plain;charset=UTF-8");
                response.getWriter().write("Received: " + buffer);
                asyncContext.complete();
            }

            @Override
            public void onError(Throwable t) {
                t.printStackTrace();
                asyncContext.complete();
            }
        });
    }
}
```
