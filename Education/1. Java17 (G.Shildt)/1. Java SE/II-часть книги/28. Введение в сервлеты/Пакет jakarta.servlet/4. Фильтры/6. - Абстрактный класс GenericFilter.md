
> [!NOTE] `GenericFilter` — это абстрактный класс, предоставляющий базовую `(скелетную) реализацию интерфейса` `Filter`
> Он упрощает создание собственных фильтров, аналогично тому, как `GenericServlet` упрощает создание сервлетов.
> 
> Благодаря этому, разработчик может просто наследоваться от `GenericFilter` и переопределять только нужные методы (чаще всего — `doFilter()`).

## Конструктор - ТОЛЬКО ПРИ НАСЛЕДОВАНИ
```java
// Конструктор без параметров.
// Вызывается контейнером при инициализации фильтра.
public GenericFilter() { }
```

## Методы
> Класс `GenericFilter` реализует интерфейс `Filter`, а также добавляет методы для работы с `FilterConfig`.

### Методы жизненного цикла фильтра
```java
// Инициализация фильтра контейнером.
// Сохраняет ссылку на объект FilterConfig.
public void init(FilterConfig filterConfig) throws ServletException

// Освобождение ресурсов фильтра при его уничтожении.
public void destroy()

// Основной метод фильтрации.
// Выполняет фильтрацию запроса и передает управление дальше по цепочке.
public abstract void doFilter(
	ServletRequest request, 
	ServletResponse response, 
	FilterChain chain)
```

### Методы для доступа к конфигурации фильтра
```java
// Возвращает объект FilterConfig, сохранённый при инициализации.
public FilterConfig getFilterConfig()

// Устанавливает объект FilterConfig (обычно вызывается контейнером).
public void setFilterConfig(FilterConfig filterConfig)

// Возвращает имя фильтра, заданное в web.xml.
public String getFilterName()

// Возвращает ServletContext, связанный с данным фильтром.
public ServletContext getServletContext()


// Возвращает параметр инициализации по имени (из web.xml).
public String getInitParameter(String name)

// Возвращает все имена параметров инициализации фильтра.
public Enumeration<String> getInitParameterNames()
```

---
## 💡 Особенности и подводные камни 💡

1️⃣ `GenericFilter` — **абстрактный класс**, ==нельзя создавать напрямую==. Нужно наследоваться и реализовать `doFilter()`.  

2️⃣ Контейнер **инициализирует** фильтр только один раз, вызывая `init()` с объектом `FilterConfig`.  

3️⃣ Если забыть вызвать `chain.doFilter()` — цепочка фильтров и сам сервлет **не будут выполнены**.  

4️⃣ ❗❗❗Не стоит хранить в полях фильтра **состояние запроса/ответа**, так как фильтры многопоточны.  

5️⃣ Если фильтр использует ресурсы (например, потоки, коннекты), их нужно **закрывать в `destroy()`**.

---
## 💎 Best Practices 💎

✅ Всегда вызывать `chain.doFilter(request, response)` — если нет особой причины завершить цепочку.  

✅ 💡Не выполнять длительные операции внутри фильтра (они блокируют запрос).  

✅ Использовать `getInitParameter()` для параметров, а не хардкодить значения.  

✅ Для логирования, метрик и авторизации лучше использовать отдельные фильтры — один фильтр = одна ответственность.  

✅ В `destroy()` освобождать любые внешние ресурсы (сокеты, потоки, пулы).
