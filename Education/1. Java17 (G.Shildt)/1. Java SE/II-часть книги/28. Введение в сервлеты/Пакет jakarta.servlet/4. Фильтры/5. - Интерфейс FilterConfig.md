
> [!NOTE] Интерфейс **`FilterConfig`** используется контейнером сервлетов 
> `для передачи` `конфигурационной информации фильтру` при его инициализации.
> 
> ==Контейнер создаёт== объект `FilterConfig` и передаёт его в метод `init(FilterConfig config)` фильтра.
> 
> С помощью этого объекта фильтр может: 
> - получить своё **имя** (`getFilterName()`),
> - получить **инициализационные параметры** (`getInitParameter()`),
> - 💡получить **`ServletContext`**, чтобы взаимодействовать с окружением приложения.

## Интерфейс
```java
package jakarta.servlet;

import java.util.Enumeration;

public interface FilterConfig {
	// Возвращает имя фильтра, указанное в web.xml
    String getFilterName();
	
	// Возвращает объект ServletContext, через который можно
	// взаимодействовать с окружением (ресурсы, логи, атрибуты)
    ServletContext getServletContext();

	// Возвращает значение параметра инициализации по имени
    String getInitParameter(String name);
	
	// Возвращает перечисление всех имён параметров инициализации
    Enumeration<String> getInitParameterNames();
}
```

---
## 💡 Особенности и подводные камни 💡

1️⃣ **Объект `FilterConfig` ==создаётся контейнером==.**  
Программист сам его не создаёт — он передаётся контейнером в `init()`.

2️⃣ 💡**Используется ==только при инициализации==.**  
После вызова `init()` фильтр обычно сохраняет нужные значения в свои поля, а сам объект `FilterConfig` не трогает.

3️⃣ 🧠**Не путать с `ServletConfig`.**  
У них одинаковая идея (конфигурация компонента), но первый — для фильтров, второй — для сервлетов.

4️⃣ 💡**Данные из `web.xml` можно получать здесь.**  
Все `<init-param>` внутри тега `<filter>` доступны через `FilterConfig`.

5️⃣ **`ServletContext` внутри `FilterConfig` — это ключ к среде приложения.**  
Через него можно читать ресурсы (`getResourceAsStream`), писать в лог (`log()`), обращаться к атрибутам приложения и т.д.

---
## 💎 Best Practices 💎

✅ 💡В методе `init(FilterConfig config)` **сохраняй нужные параметры** в поля фильтра, чтобы не обращаться к `FilterConfig` на каждом запросе.  

✅ 💡Проверяй наличие и корректность init-параметров при старте фильтра.  

✅ ❗❗❗Не изменяй объект `FilterConfig` — он предназначен ==только для чтения==.  
✅ Используй `getServletContext()` для логирования и доступа к ресурсам (например, конфигурационным файлам).  

✅ 💡Для сложных параметров можно использовать JSON или properties-файлы, загружаемые через `ServletContext`.

## 🧠 Пример использования

**web.xml**
```xml
<filter>
    <filter-name>LoggingFilter</filter-name>
    <filter-class>com.example.filters.LoggingFilter</filter-class>
    
    <init-param>
        <param-name>prefix</param-name>
        <param-value>[MyApp]</param-value>
    </init-param>
</filter>

<filter-mapping>
    <filter-name>LoggingFilter</filter-name>
    <url-pattern>/*</url-pattern>
</filter-mapping>
```

LoggingFilter.java
```java
package com.example.filters;

import jakarta.servlet.*;
import java.io.IOException;

public class LoggingFilter implements Filter {
    private String prefix;
    private ServletContext context;

    @Override
    public void init(FilterConfig config) throws ServletException {
        // Получаем init-параметр
        prefix = config.getInitParameter("prefix");
 
        // Сохраняем контекст для логирования
        context = config.getServletContext();
        context.log(prefix + " LoggingFilter инициализирован");
    }

    @Override
    public void doFilter(ServletRequest req, ServletResponse res, FilterChain chain)
            throws IOException, ServletException {
        context.log(prefix + " Запрос получен");
        chain.doFilter(req, res);
        context.log(prefix + " Ответ отправлен");
    }

    @Override
    public void destroy() {
        context.log(prefix + " LoggingFilter уничтожен");
    }
}
```
