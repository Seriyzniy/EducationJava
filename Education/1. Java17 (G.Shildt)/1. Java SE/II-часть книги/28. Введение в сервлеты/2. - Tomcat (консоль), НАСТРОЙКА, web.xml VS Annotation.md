## Web.xml VS Annotation
> [!NOTE] ⭐⭐`web.xml` VS `Annotation` ⭐⭐
> **`web.xml` всегда ==имеет приоритет над аннотациями==.**
> 
> То есть:
> - Если **сервлет описан и в аннотации, и в `web.xml`**, контейнер **игнорирует аннотацию** для этого сервлета.
> - Если **в `web.xml` нет упоминания этого сервлета**, тогда контейнер использует данные из аннотации.

```xml
<web-app xmlns="https://jakarta.ee/xml/ns/jakartaee"
  xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
  xsi:schemaLocation="https://jakarta.ee/xml/ns/jakartaee
                      https://jakarta.ee/xml/ns/jakartaee/web-app_6_0.xsd"
  version="6.0"
  metadata-complete="true">
```

⭐⭐⭐`metadata-complete="true"`
⚠️ Очень важный флаг! Указывает контейнеру ==не СКАНИРОВАТЬ АННОТАЦИИ== (@WebServlet, @WebFilter, @WebListener) и использовать только web.xml для регистрации компонентов.

---

Скажем, для первого примера под названием `HelloServlet` в раздел определения сервлетов будут добавлены следующие строки:
```xml
 <servlet>
	<servlet-name>HelloServlet</servlet-name>
	<servlet-class>servlets.HelloServlet</servlet-class>
</servlet>
```

А в раздел сопоставления (МАППИНГА) сервлетов - такие строки:
```xml
<servlet-mapping>
	<servlet-name>HelloServlet</servlet-name>
	<url-pattern>/servlets/servlet/HelloServlet</url-pattern>
</servlet-mapping>
```
Та же самая общая процедура используется во всех примерах.

# 📕 2. Пулы потоков контейнера сервлетов

🔶**Синхронность по умолчанию.** Все фильтры и сервлеты выполняются в **одном потоке** запроса

Контейнер сервлетов (Tomcat, Jetty, и т.д.) управляет ==**несколькими пулами потоков**==, основные из которых:

| Назначение                                    | Описание                                                                                                             | Источник                                                                                    |
| --------------------------------------------- | -------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------- |
| **Request-handling threads** (worker threads) | Потоки, которые **==принимают и обрабатывают HTTP-запросы==** (выполняют `service()`, `doGet()`, `doPost()`, и т.д.) | Из **thread pool** сервера (например, `maxThreads` в `server.xml` Tomcat’а, обычно 200–400) |
| **Async threads**                             | Потоки, в которых ==выполняются асинхронные задачи==, запущенные через `AsyncContext.start(Runnable)`                | Зависит от контейнера: в Tomcat — ==**тот же пул** (по умолчанию)==, но можно подменить     |
| **Container management threads**              | Потоки ==для фоновых задач== контейнера (мониторинг, очистка, async callbacks)                                       | Внутренние служебные потоки контейнера                                                      |
| **Listener callback threads**                 | Потоки, ==в которых вызываются методы== `AsyncListener` (`onComplete`, `onTimeout` и т.д.)                           | Обычно те же **container threads**, не пользовательские                                     |

### Нет пула для асинхронных задач
- Tomcat **не имеет отдельного пула для async задач**.  
    По умолчанию `AsyncContext.start()` использует **тот же Executor**, что и HTTP worker threads.
    
- Это значит, что async-задачи **конкурируют** за потоки с обычными запросами. 
    То есть — это _тот же пул_, но используется **другой поток** (не тот, что обрабатывал запрос).

## ⚙️ Что происходит при обычном (синхронном) запросе
1. Приходит HTTP-запрос.
    
2. Контейнер выделяет **один поток из пула worker threads**.
    
3. Этот поток вызывает `service()` → `doGet()` ==и ждёт, пока метод завершится==.
    
4. Поток ==блокируется==, ==пока выполняется обработка== (например, чтение из БД).
    
5. После завершения — поток освобождается и возвращается в пул.

❌ Минус: если запрос «висит» (например, 5 секунд), поток всё это время занят, и другие клиенты ждут.

## ⚙️ Что происходит при **асинхронной** обработке (`startAsync()`)

> [!NOTE] ❗ВСЕ УЧАСТНИКИ(сервлеты, фильтры) ДОЛЖНЫ ИМЕТЬ ПОДДЕРЖКУ АСНХРОННОСТИ❗
> ⭐⭐⭐Tomcat не может начать `request.startAsync()`, если **все участники цепочки** (фильтры + сервлет) не имеют `async-supported = true`.


1. Приходит HTTP-запрос.
    
2. Контейнер выделяет поток из пула и вызывает `service()` или `doGet()`.
    
3. Ты вызываешь:
```java
AsyncContext async = req.startAsync();
```
👉 Контейнер **развязывает** текущий поток с этим запросом.  
Теперь поток возвращается в пул и может обрабатывать другой запрос.

4. Асинхронная задача запускается:
```java
async.start(() -> { 
	... 
});
```
- ❗❗❗Тут происходит важноe:
    - По умолчанию, контейнер использует **тот же пул потоков (worker thread pool)**,  
        но ты можешь **назначить свой Executor**, если хочешь полностью отделить логику от контейнера.


5. После завершения фоновой работы (например, запрос в БД, чтение файла):
```java
async.complete();
```

Контейнер снова берёт **один из своих потоков** (не обязательно тот же)  
и выполняет операции завершения:
- формирование ответа,
    
- вызов `AsyncListener.onComplete()`.
