
> [!NOTE] `CompletionHandler<V, A>` — это **обобщённый интерфейс** из пакета `java.nio.channels`.
> Он представляет ==**callback-механизм**== для уведомления о завершении асинхронной операции (успешной или неуспешной).

Каждый раз, когда вы вызываете асинхронный метод вроде  
`read()`, `write()`, `connect()` или `accept()` с параметром `CompletionHandler`,  
==JVM регистрирует этот обработчик и вызывает его при завершении операции — **в отдельном потоке**== из группы (`AsynchronousChannelGroup`).

## Future vs CompletionHandler
Java NIO.2 определяет два пути обработки **асинхронных результатов**:
1. 🔸**Pull-модель** — через `Future<V>` (вытягиваем результат вручную);
    
2. 🔸**Push-модель** — через `CompletionHandler` (результат “приходит” сам).

## Назначение
- Используется для **асинхронного взаимодействия без блокировок**.

- Позволяет **реагировать на завершение операций** (вместо Future и `get()`).
    
- Применяется во всех каналах:
    - `AsynchronousSocketChannel`,
        
    - `AsynchronousServerSocketChannel`,
        
    - `AsynchronousFileChannel`.

## Интерфейс
```java
public interface CompletionHandler<V, A> {
    /**
     * Вызывается при успешном завершении асинхронной операции.
     *
     * 💡result — результат операции (тип зависит от контекста💡):
     * - Integer — количество прочитанных/записанных байт,
     * - AsynchronousSocketChannel — для accept(),
     * - Void — для connect().
     * 
     * ❗❗attachment❗❗ — объект, 
     * переданный при вызове асинхронного метода.
     * Обычно содержит контекст: состояние, буфер, канал и т. п.
     */
    void completed(V result, A attachment);


    /**
     * Вызывается, если асинхронная операция завершилась с ошибкой.
     *
     * exc — исключение, описывающее причину сбоя 
     * (IOException, TimeoutException и т. д.)
     * 
     * ❗❗attachment❗❗ — тот же объект, который был передан при запуске операции.
     */
    void failed(Throwable exc, A attachment);
}
```

---
## 💡 Особенности и подводные камни 💡

### 1. Обобщённость
- `V` — тип результата (`Integer`, `Void`, `AsynchronousSocketChannel` и т. д.);
    
- `A` — тип "контекста" (обычно объект, который хранит состояние операции, например `ByteBuffer`).
### 2. Асинхронность выполнения
- Методы `completed()` и `failed()` вызываются **в потоке группы** (`AsynchronousChannelGroup`),  
    а не в том потоке, где операция была запущена.
### 💡 3. Цепочки операций
- Типичный паттерн — вызывать в `completed()` **следующую асинхронную операцию**,  
    например, `read()` → после завершения → `write()` → после завершения → `close()`.
### ❗❗❗ 4. Ошибки не пробрасываются
- Любое исключение в `completed()` или `failed()` **не выбрасывается наружу** — оно теряется.  
    Поэтому всегда **логируй** и **обрабатывай ошибки** внутри callback-а.
### 5. Многопоточность
- Один `CompletionHandler` может быть вызван из разных потоков (если используется для нескольких операций).  
    Поэтому при необходимости нужно обеспечивать синхронизацию вручную.

## ✅ Best Practices ✅
- Храни в `attachment` контекст операции — например:
```java
class Session {
    AsynchronousSocketChannel channel;
    ByteBuffer buffer;
}
```

- ❗❗❗Всегда вызывай **следующую операцию** (`read` или `write`) внутри `completed()`, если нужно “держать” соединение.
    
- ❗❗❗Обрабатывай ошибки **только в `failed()`**, не смешивай с основной логикой.
    
- 🔒 Не делай блокирующих вызовов (`get()`, `Thread.sleep()`) внутри `completed()`.
    
- 💡Логируй исключения, чтобы не потерять контекст.
    
- 🧠Используй **анонимные классы или лямбды** для компактности, но не злоупотребляй —  
    лучше вынеси сложную логику в отдельный `Handler`-класс.

## Пример 1 — Асинхронное чтение из сокета
```java
import java.net.*;
import java.nio.*;
import java.nio.channels.*;

public class AsyncReadExample {
    public static void main(String[] args) throws Exception {
        AsynchronousSocketChannel client = AsynchronousSocketChannel.open();
        client.connect(new InetSocketAddress("example.com", 80)).get();

        String req = "GET / HTTP/1.1\r\nHost: example.com\r\n\r\n";
        ByteBuffer out = ByteBuffer.wrap(req.getBytes());
        
        client.write(out).get();

        ByteBuffer in = ByteBuffer.allocate(1024);

        client.read(in, in, new CompletionHandler<Integer, ByteBuffer>() {
            @Override
            public void completed(Integer bytesRead, ByteBuffer buf) {
                if (bytesRead == -1) {
                    System.out.println("Server closed connection");
                    try { 
	                    client.close(); 
	                } catch (Exception ignored) {}
                    return;
                }

                buf.flip();
                System.out.println(
	                "Received: " + 
	                new String(buf.array(), 0, buf.limit()));
            }

            @Override
            public void failed(Throwable exc, ByteBuffer buf) {
                System.err.println("Read failed: " + exc);
                try { 
	                client.close(); 
	            } catch (Exception ignored) {}
            }
        });

        Thread.sleep(2000);
    }
}

```

# `Future` vs `CompletionHandler`
|Критерий|`Future`|`CompletionHandler`|
|---|---|---|
|Модель|Pull (ждём результат)|Push (обратный вызов)|
|Блокирует поток|✅ Да (`get()`)|❌ Нет|
|Подходит для|простых операций, тестов|реактивных серверов, цепочек I/O|
|Сложность|низкая|выше, но гибче|
|Поток выполнения|вызывающий поток|поток из `AsynchronousChannelGroup`|
|Контекст операции|нет (`Future` хранит только результат)|есть (`attachment`)|
|Ошибки|через `ExecutionException`|через `failed(Throwable, A)`|
|Таймаут|встроенный (`get(timeout)`)|обрабатывается вручную|
