# 📕 1. java.io - Классическая модель
![[Pasted image 20251005112435.png]]

## Буферы
В IO буфер — **внутренняя деталь** потока, ==ОДНОНАПРАВЛЕННЫЙ== (или на ввод, или на вывод).

💬 То есть:
- в IO ты используешь **два разных класса** — `InputStream` и `OutputStream`;

При этом, буферы просто уменьшают число системный вызовов, как бы собирая в кучу необходимые данные, и все разом записывает
## Принцип действия

> [!NOTE] В IO мы получаем доступ к файлу "напрямую"…
> “Напрямую” в IO — не совсем физически напрямую, а логически (`через блокирующие системные вызовы`).

**В IO** ==“режим”== чтения или записи ==жёстко зашит в ТИП ПОТОКА (Input/Output),==

Технически:
- `read()` вызывает системный вызов `read(fd, byte[], len)`
    
- ядро ОС блокирует поток, пока данные не будут готовы
    
- данные копируются **в Java heap**
    
- JVM отдаёт тебе байты

# 📕 2. java.nio - Новый IO
![[Pasted image 20251005112949.png]]

## Буферы
В NIO буфер — **основная модель данных**, и он **==ДВУНАПРАВЛЕННЫЙ==**, но **режимный** (чтение/запись управляются вручную).

**Один `ByteBuffer`**, который ты сам “переключаешь” с помощью `flip()`, `clear()`, `rewind()`.

## Принцип действия

> [!NOTE] В NIO есть ==КАНАЛЫ==, которые ==связывают ОС и программу==.
> 

Технически:
- `read()` или `write()` у `Channel` используют **non-blocking дескрипторы** (через `fcntl(O_NONBLOCK)`)
    
- ОС немедленно возвращает управление
    
- ==канал регистрируется== в `Selector`, который отслеживает готовность
    
- когда ядро говорит “данные пришли”, ты читаешь их в `ByteBuffer`

# ⭐ ИТОГО ⭐

> [!NOTE] Разница
> 🟢В IO модель `основана на ПОТОКХ`, которые блокируются при чтении/записи,  
> а буфер — это лишь внутренняя оптимизация. 
> 
> 🟠В NIO модель `основана на КАНАЛАХ`, которые работают `С БУФЕРАМИ данных` напрямую и **не блокируют поток**, позволяя ОС уведомлять программу о готовности данных (через `Selector` или `CompletionHandler`).

## 💬 Если совсем по-человечески:
- В IO ты говоришь:  
    👉 _“Прочитай мне данные прямо сейчас.”_  
    (и ждёшь, пока они придут)
    
- В NIO ты говоришь:  
    👉 _“Скажи, когда данные будут готовы, и я сам их возьму.”_  
    (и занимаешься другими задачами, пока ждёшь)

|Критерий|IO|NIO|
|---|---|---|
|Модель|Потоковая|Канальная|
|Буфер|Внутренний, однонаправленный|Управляемый вручную, двунаправленный|
|Режим|Блокирующий|Неблокирующий|
|Компоненты|Stream / Reader / Writer|Channel / Buffer / Selector|
|Контроль|JVM|Разработчик|
|Асинхронность|Через многопоточность|Встроенная на уровне ОС|
|Использование|Простые I/O операции|Масштабируемые, высокопроизводительные системы|
# ❓❓Non-blocking vs Asynchronous❓❓

| Модель                            | Non-blocking (NIO)                                        | Asynchronous (NIO.2)                                                                            |
| --------------------------------- | --------------------------------------------------------- | ----------------------------------------------------------------------------------------------- |
| Кто выполняет операцию            | **Твой поток**                                            | **Операционная система**                                                                        |
| Кто ждёт завершения               | **Ты сам** (через `Selector`)                             | **ОС** (уведомит, когда готово)                                                                 |
| read()/write()                    | **возвращают сразу**, но могут вернуть 0 (ещё нет данных) | **не возвращают данные сразу**, результат приходит позже через `Future` или `CompletionHandler` |
| Поток блокируется                 | Нет                                                       | Нет                                                                                             |
| ❗❗❗Кто уведомляет о готовности❗❗❗ | `Selector` (polling loop)                                 | `ОС` вызывает callback                                                                          |
| Пример API                        | `SocketChannel`, `Selector`                               | `AsynchronousSocketChannel`, `CompletionHandler`                                                |

##  💥 Принципиальная разница — кто “ждёт” данные 💥
- В **NIO (non-blocking)**:
    - поток **ждёт событий готовности** (через `Selector.select()`);
        
    - потом сам **выполняет чтение/запись**.
        
- В **NIO.2 (asynchronous)**:
    - поток **вообще НЕ ЖДЕТ —  
        ОС выполняет I/O асинхронно и **сама вызывает callback**, когда всё готово.

## ⚙️ 1. Что такое “non-blocking” на самом деле

Когда ты делаешь:
```java
channel.configureBlocking(false);
channel.read(buffer);
```

Операция:
- **не блокирует** поток,
- но может вернуть **0 байт**, если данных пока нет.

Ты сам должен потом проверить, _когда канал готов для чтения_, используя `Selector`.

То есть в non-blocking IO ты 
- ==**сам следишь** за готовностью канала== и 
- ==**сам вызываешь read() в нужный момент**.==

> [!NOTE] 💡 Аналогия:
> Ты подходишь к окошку почты каждые 10 минут и спрашиваешь —  
> “Пришла ли моя посылка?”  
> Если нет — идёшь заниматься дальше.  
> Если да — получаешь

## ⚙️ 2. Что делает “asynchronous” модель (NIO.2)

Когда ты делаешь:
```java
channel.read(
	buffer, 
	attachment, 
	new CompletionHandler<Integer, ByteBuffer>() {
		...
	});

```

Тут происходит другое:
- JVM просит **ОС выполнить операцию чтения** асинхронно;
    
- ОС начинает I/O операцию в фоновом режиме;
- Твой поток **немедленно освобождается**;
    
- ❗❗❗Когда данные реально прочитаны — ОС уведомляет JVM,  
    а та вызывает твой `CompletionHandler.completed()` (в пуле потоков).


> [!NOTE] 💡 Аналогия:
> Ты оставляешь заявку на доставку и идёшь по делам.  
> Почта сама позвонит тебе, когда посылка придёт.
