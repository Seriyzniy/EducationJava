## 🧠 Под капотом 🧠
Если `Channel` — это “труба” между программой и устройством,  
то `Selector` — это “диспетчер каналов”, который следит, какие трубы готовы к работе.

### Selector “держит” пул каналов
Но не в смысле “открытых файлов”, а в смысле **зарегистрированных неблокирующих каналов**,   за которыми он следит.

Когда ты делаешь:
```java
channel.register(selector, SelectionKey.OP_READ);
```
— ты добавляешь канал в **внутренний набор** `Selector` (реально — это структура на уровне ОС: `epoll`, `kqueue`, `select`, в зависимости от платформы).

Таким образом:
- 💥один `Selector` может отслеживать **десятки тысяч** каналов (да, тысячи!);
    
- 💥каждый канал связан с **одним `SelectionKey`** внутри этого селектора;
- каждый `SelectionKey` знает, какие операции интересуют (`interestOps`) и какие готовы (`readyOps`).

### Каналов на один селектор — сколько угодно (в разумных пределах)

```java
Selector selector = Selector.open();

SocketChannel ch1 = SocketChannel.open();
SocketChannel ch2 = SocketChannel.open();
ch1.configureBlocking(false);
ch2.configureBlocking(false);

//ОДИН и тот же селектор для ДВУХ каналов
ch1.register(selector, SelectionKey.OP_READ);
ch2.register(selector, SelectionKey.OP_READ);
```
Теперь `selector` следит **одновременно** за `ch1` и `ch2`.  
Когда у одного из них появятся данные, система пометит этот канал как “готовый”.

### Как узнать, кто готов: `selector.select()` + `selectedKeys()`
Когда ты вызываешь:
```JAVA
selector.select();
```
— JVM делает системный вызов (`epoll_wait` / `select`) и **ждёт**, пока хотя бы один канал не будет готов.

Когда событие происходит (например, клиент прислал данные),  `Selector` наполняет внутреннее множество **“готовых ключей”** (`selectedKeys()`).

И да — ты **получаешь к ним доступ через итератор**:
```java
Set<SelectionKey> keys = selector.selectedKeys();
Iterator<SelectionKey> iter = keys.iterator();

while (iter.hasNext()) {
    SelectionKey key = iter.next();
    iter.remove(); // обязательно! иначе событие повторится
    if (key.isReadable()) {
        SocketChannel ch = (SocketChannel) key.channel();
        // читаем данные
    }
}
```
💬 То есть ты не “перебираешь все каналы”,  
а **перебираешь только готовые** (те, у кого реально есть событие).  
Это делает работу очень эффективной — никаких пустых проверок.

### Low-Level
Когда ты регистрируешь канал:
```java
channel.register(selector, SelectionKey.OP_READ);
```
- JVM вызывает системный вызов `epoll_ctl(EPOLL_CTL_ADD, fd, ...)`;
    
- `fd` (file descriptor) добавляется в системную таблицу наблюдения;
    
- 💡когда в ядре ОС в дескрипторе появляются данные — оно ставит “флаг готовности”;
    
- 💡при вызове `selector.select()` этот флаг считывается, и `SelectionKey` попадает в `selectedKeys()`.

# 📕 Selector 📕

![[Pasted image 20251005133526.png]]

> [!NOTE] `Selector` — это **центральный класс неблокирующего I/O** (из пакета `java.nio.channels`),
> Позволяет ==ОДНОМУ ПОТОКУ== одновременно ==управлять **МНОЖЕСТВОМ КАНАЛОВ**== (`SocketChannel`, `ServerSocketChannel`, `DatagramChannel` и т.д.).
> 
> а

Именно `Selector` делает возможной архитектуру типа **Reactor pattern**, лежащую в основе Netty, Kafka, Tomcat, и множества других фреймворков.

## Назначение
- Позволяет отслеживать **готовность** нескольких каналов к операциям чтения, записи, подключения и т.д.
    
- Работает в связке с `SelectableChannel`, который может быть зарегистрирован у селектора.
    
- Позволяет **избежать создания отдельного потока на каждое соединение**.

💡 Проще говоря:
> Вместо того чтобы “блокироваться” на каждом `read()`,  
> ты спрашиваешь у `Selector`:  
> — “Какие каналы сейчас готовы для чтения/записи?”
> 
> И `Selector` возвращает тебе список таких каналов.

___
## Конструктор
`Selector` не создаётся напрямую — используются статические методы `open()`:
```JAVA
// Создаёт новый селектор, привязанный к системному multiplexor (epoll/kqueue/Select)
public static Selector open() throws IOException;
```

## Методы
```java
// ✅ РЕГИСТРАЦИЯ каналов
/**
 * Возвращает объект-помощник для регистрации канала.
 * Каналы регистрируются методом channel.register().
 */
public abstract SelectorProvider provider();

/**
 * Возвращает множество ключей всех зарегистрированных каналов.
 * 💡Каждый ключ представляет регистрацию (channel + ops).
 */
public abstract Set<SelectionKey> keys();

/**
 * 💡💡Возвращает множество "готовых" ключей💡💡 (т.е. каналов, готовых к I/O).
 * После select() оно заполняется, а потом очищается вручную.
 */
public abstract Set<SelectionKey> selectedKeys();



// ✅ ОЖИДАНИЕ готовности
/**
 * Блокирует поток, пока хотя бы один канал не станет готовым.
 * Возвращает количество готовых каналов.
 */
public abstract int select() throws IOException;

/**
 * Блокирует максимум `timeout` миллисекунд.
 */
public abstract int select(long timeout) throws IOException;

/**
 * Не блокирует поток вообще. Просто проверяет готовые каналы и возвращает немедленно.
 */
public abstract int selectNow() throws IOException;

/**
 * Принудительно пробуждает селектор (если кто-то вызвал select() в другом потоке).
 */
public abstract Selector wakeup();



// ✅ УПРАВЛЕНИЕ состоянием
/**
 * Проверяет, открыт ли селектор.
 */
public abstract boolean isOpen();

/**
 * Закрывает селектор и все связанные системные ресурсы.
 */
@Override
public abstract void close() throws IOException;
```

## 💡 Особенности и подводные камни 💡

###  1. ❗❗❗Работает только с `SelectableChannel`❗❗❗
- Только те каналы, которые наследуют `SelectableChannel` (`SocketChannel`, `ServerSocketChannel`, `DatagramChannel`)  
    могут регистрироваться в `Selector`.
### 2. Регистрация каналов
⭐⭐Канал регистрируется в `Selector` методом :
```java
SelectionKey key = channel.register(
	selector, 
	SelectionKey.OP_READ | SelectionKey.OP_WRITE
);
```
❗❗`register()` есть у классов каналов, наследников `SelectableChannel`

📌 `SelectionKey` хранит информацию:
- о **канале**;
- о **интересующих событиях** (read, write, accept, connect);
- о **прикреплённом объекте** (attachment, например `ByteBuffer` или объект сессии).

### 3. События готовности SelectionKey

|Константа|Значение|Что значит|
|---|---|---|
|`OP_READ`|1|Канал готов к чтению|
|`OP_WRITE`|4|Канал готов к записи|
|`OP_CONNECT`|8|Клиент готов к установлению соединения|
|`OP_ACCEPT`|16|Сервер готов принять новое соединение|

### 4. 🧠select() — как работает под капотом

1. `select()` вызывает системный вызов (`select()`, `poll()`, `epoll_wait()` или `kqueue` в зависимости от ОС).
    
2. ОС проверяет состояние всех зарегистрированных сокетов.
    
3. Когда хотя бы один из них готов, `select()` возвращает количество готовых.
    
4. `selectedKeys()` содержит набор каналов, которые можно обрабатывать без блокировки.

### 5. Очистка selectedKeys()

После обработки каждого события **обязательно** нужно очищать обработанные ключи:
```java
Iterator<SelectionKey> it = selector.selectedKeys().iterator();

while (it.hasNext()) {
    SelectionKey key = it.next();
    it.remove(); // обязательно удаляем!
    handle(key);
}
```

💡❗Если этого не делать — одни и те же события будут срабатывать снова и снова.

### 6. Потокобезопасность
- `Selector` можно пробудить из другого потока методом `wakeup()`.  
    Это полезно, если нужно завершить цикл `select()` (например, при остановке сервера).
    
- Но **регистрировать каналы и вызывать select() из разных потоков одновременно нельзя** — будет `ConcurrentModificationException`.

### 7.  ❗❗Подводные камни
- `Selector` не гарантирует мгновенного уведомления (может быть “spurious wakeup”).
    
- При большом числе каналов (`>10k`) — лучше использовать **epoll-based селекторы** (они в Java используются автоматически).

---
## ✅ Best Practices ✅
✅ Один `Selector` на один поток 
✅ Используй `key.attach(sessionObject)` для хранения состояния клиента.  
✅ После `select()` **всегда** очищай `selectedKeys()`.  
✅ Используй `wakeup()` для graceful shutdown.  
✅ Не обрабатывай долго операции прямо в select-цикле — выноси в отдельный executor.  
✅ При высокой нагрузке — можешь создавать **несколько селекторов** и распределять каналы между ними.

## Пример 1 — Echo-сервер на Selector (полностью неблокирующий)
```java
import java.io.IOException;
import java.net.InetSocketAddress;
import java.nio.*;
import java.nio.channels.*;
import java.util.*;

public class NioEchoServer {
    public static void main(String[] args) throws IOException {
        Selector selector = Selector.open();
        
        ServerSocketChannel serverChannel = ServerSocketChannel.open();
        serverChannel.bind(new InetSocketAddress(9000));
        serverChannel.configureBlocking(false);
		
		// Регистрация селектора
        serverChannel.register(selector, SelectionKey.OP_ACCEPT);
        
        System.out.println("NIO Echo Server started on port 9000...");

        while (true) {
            selector.select(); // блокируемся до готовности хотя бы одного канала

            Iterator<SelectionKey> keys = selector
	            .selectedKeys()
	            .iterator();
	            
            while (keys.hasNext()) {
                SelectionKey key = keys.next();
                keys.remove();

                if (key.isAcceptable()) {
                    // новое подключение
                    SocketChannel client = serverChannel.accept();
                    client.configureBlocking(false);
                    
                    client.register(
	                    selector, 
	                    SelectionKey.OP_READ, 
	                    ByteBuffer.allocate(256)
	                );
	                
                    System.out.println(
	                    "Client connected: " +
	                     client.getRemoteAddress()
	                );
                }

                if (key.isReadable()) {
                    // читаем данные
                    SocketChannel client = 
	                    (SocketChannel) key.channel();
	                    
                    ByteBuffer buffer = (ByteBuffer) key.attachment();
                    int read = client.read(buffer);
                    if (read == -1) {
                        client.close();
                        System.out.println("Client disconnected");
                        continue;
                    }
                    buffer.flip();
                    client.write(buffer);
                    buffer.clear();
                }
            }
        }
    }
}
```
🧠 Здесь:

- Один поток (`main`) обслуживает все соединения;
    
- Selector сообщает, когда канал готов к чтению или приёму;
    
- Никаких блокировок и сотен потоков.
