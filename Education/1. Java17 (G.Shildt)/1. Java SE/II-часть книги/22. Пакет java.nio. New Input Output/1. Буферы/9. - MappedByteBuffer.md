
> [!NOTE] Подкласс `ByteBuffer`, получаемый через `FileChannel.map()`.  
> Он ==позволяет отобразить файл== или его часть ==в память (mmap).== 
> 
> После этого доступ к файлу осуществляется как к массиву байтов.
> 

Когда мы говорим _отобразить файл в память_ (memory map), мы имеем в виду, что **операционная система сопоставляет (map) участок файла с участком виртуальной памяти процесса**.


> [!NOTE] 🧠Грубо говоря:🧠
> Отобразить файл в память = дать программе иллюзию, что файл — это массив в памяти, а ОС сама подгружает и сбрасывает данные на диск.

## Отличия от обычного `FileInputStream.read()` или `Files.readAllBytes()`

|Обычное чтение (`read`)|Memory-mapped file (`MappedByteBuffer`)|
|---|---|
|Нужно явно вызывать `read()` / `write()`|Просто работаешь с массивом в памяти|
|Копирование: диск → kernel buffer → твой массив|Данные мапятся напрямую в память процесса|
|Чтение больших файлов — куча системных вызовов|Чтение ленивое (OS сама подгружает страницы по мере доступа)|
|Обычно быстрее для мелких файлов|Обычно быстрее для **очень больших файлов**|

Допустим, у тебя есть **файл размером 10 ГБ**.
- ❗❗Если ты читаешь его через `read()`, то обычно надо грузить кусками в массивы байтов, вручную крутить цикл.
    
- ✅Если через `MappedByteBuffer` — то можно сразу сказать: "Отобрази этот файл (или кусок) в память".  
    Теперь доступ к файлу похож на доступ к `byte[]`:
```java
MappedByteBuffer mbb = channel
	.map(FileChannel.MapMode.READ_WRITE, 0, 1024*1024);

byte b = mbb.get(123456); // ← автоматически подтянется страница с диска
mbb.put(123457, (byte) 42); // ← ОС сама сбросит на диск

```

## Конструктор - нет конструктора
```java
// Через FileChannel
MappedByteBuffer map(FileChannel.MapMode mode, long position, long size)

// Режимы:
// MapMode.READ_ONLY
// MapMode.READ_WRITE
// MapMode.PRIVATE (copy-on-write)
```

## Методы
Все остальные методы унаследованы от `ByteBuffer`: get/put, slice, duplicate, compact, position/limit/capacity и т.д.)
```java
// ✅ Специфичные методы
// Гарантированная запись изменений на диск
public final void force()

// Проверка, загружен ли буфер в память
public final boolean isLoaded()

// Подгрузить содержимое в память
public final MappedByteBuffer load()
```

## 💡 Особенности и подводные камни 💡
- `MappedByteBuffer` ==напрямую работает с **файловой системой**==.
    
- ❗❗На Windows файл нельзя удалить, пока существует сопоставленный буфер.
    
- Нет явного API для `unmap()` до JDK 9; ранее приходилось использовать хаки через `Cleaner`/`Unsafe`.
    
- `PRIVATE` режим — copy-on-write: изменения не попадают обратно в файл.
    

## ✅ Best Practices ✅
- ❗❗Используй mmap для **больших файлов** (логов, БД, индексных структур).
    
- Для корректного завершения работы — вызывай `force()` для сохранения изменений.
    
- Если нужна явная выгрузка: начиная с JDK 9, можно использовать `sun.misc.Unsafe::invokeCleaner` (или `FileChannel::unmap` в новых версиях).
    
- Не злоупотребляй — `MappedByteBuffer` может удерживать файл «залоченным».

## Пример
```java
import java.io.*;
import java.nio.*;
import java.nio.channels.*;

public class MappedExample {
    public static void main(String[] args) throws Exception {
        try (RandomAccessFile raf = new RandomAccessFile("test.dat", "rw");
             FileChannel fc = raf.getChannel()) {

            // Маппим первые 1KB файла
            MappedByteBuffer mbb = fc.map(FileChannel.MapMode.READ_WRITE, 0, 1024);

            // Пишем данные
            mbb.put(0, (byte)65); // 'A'
            mbb.put(1, (byte)66); // 'B'

            // Сохраняем изменения
            mbb.force();
        }
    }
}

```
