Прокси — это ==шаблон проектирования==. Мы создаем и используем его для добавления и изменения функционала уже существующих классов.

**Прокси-объект в Java** — это **объект, который ==выступает в качестве заменителя другого объекта== и КОНТРОЛИРУЕТ ДОСТУП к нему**. 

Внешне он выглядит и действует как другой объект, реализуя его интерфейсы, но внутренне перенаправляет вызовы своих методов реальному экземпляру объекта, который он имитирует.

---

# 📕1. Класс Proxy

> [!NOTE] Класс `Proxy` — это **`механизм динамического создания` `прокси-объектов` во время выполнения**.
> Он позволяет `создавать объекты`, `реализующие один` или `несколько` `ИНТЕРФЕЙСОВ`, 
> и 
> `перехватывать` все `вызовы` их `методов` через объект типа `InvocationHandler`.

Основное применение:
- логирование вызовов;
- проверка доступа;
- ленивые вычисления;
    
- реализация **AOP** (аспектно-ориентированного программирования) в **Spring** и других фреймворках;
    
- создание **моков** в тестах (например, Mockito)

---
## Принцип работы
Самое простое — использовать `java.lang.reflect.Proxy`, который является частью JDK. Этот класс может создать прокси-класс или напрямую его инстанс. 

Пользоваться прокси, встроенным в Java, очень просто.

Все что нужно — реализовать `java.lang.InvocationHandler`, чтобы прокси-объект мог его вызывать. 

Интерфейс `InvocationHandler` крайне прост и содержит только один метод: `invoke()`. 
При его вызове, аргументы содержат проксируемый оригинальный объект, вызванный метод (как отражение объекта `Method`) и массив объектов исходных аргументов.

## Конструктор
Класс `Proxy` имеет **приватный и защищенный конструктор**, так как объекты создаются только через фабричные методы.
```java
/**
 * Создает новый экземпляр Proxy из подкласса
 * (обычно динамического прокси-класса) с указанным обработчиком вызовов.
 *
 * @param h обработчик вызовов для данного прокси-объекта
 *
 * @throws NullPointerException если указанный обработчик вызовов {@code h}
 *         имеет значение {@code null}
 */
protected Proxy(InvocationHandler h) {
    Objects.requireNonNull(h);
    this.h = h;
}
```

### Фабричные методы для создания
```java
// Создаёт новый прокси-объект, реализующий указанные интерфейсы
// loader — загрузчик классов, который будет использоваться для загрузки прокси-класса
// interfaces — список интерфейсов, которые прокси должен реализовать
// h — обработчик вызовов, определяющий поведение методов
public static Object newProxyInstance(ClassLoader loader,
                                     Class<?>[] interfaces,
                                     InvocationHandler h)

// Возвращает класс прокси для заданного загрузчика и интерфейсов
// Если класс уже создан, возвращается кэшированный вариант
public static Class<?> getProxyClass(ClassLoader loader,
                                    Class<?>... interfaces)
```

---
## Методы
### Методы работы с прокси-классами
Позволяют получить информацию или проверить, является ли объект прокси
```java
// Проверяет, является ли класс прокси-классом
public static boolean isProxyClass(Class<?> cl)

// Возвращает обработчик (InvocationHandler), связанный с прокси-объектом
// Если объект не является прокси, выбрасывается IllegalArgumentException
public static InvocationHandler getInvocationHandler(Object proxy)
```

---
## 💡 **Особенности и подводные камни** 💡

1️⃣ `Proxy` ==работает **только с интерфейсами**== — вы не можете создать прокси для обычного класса (в отличие от библиотек вроде CGLIB).  

2️⃣💡Все ==вызовы методов интерфейса== ПЕРЕНАПРАВЛЯЮТСЯ в `InvocationHandler.invoke()`.  

3️⃣ 💡Прокси создаются **во время выполнения**, поэтому производительность ниже, чем у обычных объектов.  

4️⃣ Если интерфейс наследует методы `Object` (например, `toString()`), их тоже можно перехватывать.  

5️⃣ 💡Нельзя напрямую кастовать прокси к типу, не указанному в списке `interfaces` при создании — будет `ClassCastException`.

---

## 💎 **Best Practices** 💎

✅ Использовать `Proxy` для кросс-срезных задач (AOP, логирование, кэширование).  

✅ Передавать минимально необходимый список интерфейсов.  

✅ Проверять типы во время создания прокси, чтобы избежать `ClassCastException`.  

✅ Для прокси классов (а не интерфейсов) — использовать **CGLIB** или **ByteBuddy**. 

✅ Хранить `InvocationHandler` неизменяемым (stateless), чтобы избежать непредсказуемого поведения.

# 📕2. Интерфейс InvocationHandler

**Назначение:**  
Интерфейс `InvocationHandler` используется вместе с `Proxy` и ==ОПРЕДЕЛЯЕТ ПОВЕДЕНИЕ== динамического прокси-объекта_ при вызове любого метода.

Когда ты создаёшь прокси через `Proxy.newProxyInstance()`, **==все вызовы методов== этого прокси ==перенаправляются в один метод==** —  
👉 `invoke()` твоего обработчика `InvocationHandler`.

Это **функциональный интерфейс**, т.е. содержит **один метод** и может быть реализован через **лямбду**.

## Методы
```java
// Вызывается при каждом вызове метода на прокси
Object invoke(
	Object proxy, 
	Method method, 
	Object[] args) throws Throwable;
```

- `proxy` — сам прокси-объект (на котором вызван метод);
    
- `method` — объект `Method`, описывающий, какой именно метод был вызван;
    
- `args` — массив аргументов, переданных в этот метод (может быть `null`).
    

Возвращаемое значение — результат, который увидит вызывающий код (аналогично реальному методу).

---
## 💡 Особенности и подводные камни 💡

1️⃣ 💡❗**Каждый вызов метода** интерфейса → **один вызов `invoke()`**.  

2️⃣ Если метод возвращает `void`, `invoke()` должен вернуть `null`.  

3️⃣ Можно использовать `method.invoke(target, args)` для делегирования вызова реальному объекту.  

4️⃣ 🧠В `invoke()` ==можно добавлять== ==кросс-функциональную логику==: логирование, безопасность, транзакции, кэш и т.д.  

5️⃣ Прокси можно создавать **==только для интерфейсов==**, не для классов.

---
## 💎 Best Practices 💎

✅ Проверяй `method.getDeclaringClass()`, чтобы различать системные методы (`toString`, `equals`, `hashCode`).  
✅ Лаконично логируй или перехватывай исключения внутри `invoke()`.  
✅ Если цель — просто делегировать вызовы, не забудь вернуть результат `method.invoke(target, args)`.

---
# Пример
```java
import java.lang.reflect.*;

interface Service {
    void process();
}

class ServiceImpl implements Service {
    public void process() {
        System.out.println("Выполняется логика сервиса...");
    }
}

class LoggingInvocationHandler implements InvocationHandler {
    private final Object target;

    public LoggingInvocationHandler(Object target) {
        this.target = target;
    }

    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        System.out.println("Метод " + method.getName() + " вызван");
        
        Object result = method.invoke(target, args);
         
        System.out.println("Метод " + method.getName() + " завершён");
        return result;
    }
}


public class ProxyExample {
    public static void main(String[] args) {
        Service target = new ServiceImpl();

        // Создаём прокси, реализующий интерфейс Service
        Service proxy = (Service) Proxy.newProxyInstance(
                Service.class.getClassLoader(),
                new Class<?>[]{Service.class},
                new LoggingInvocationHandler(target)
        );

        proxy.process();
    }
}
```

## 1. Что происходит при вызове метода у прокси

Когда ты создаёшь прокси:
```java
Service proxy = (Service) Proxy.newProxyInstance(
    Service.class.getClassLoader(),
    new Class<?>[]{Service.class},
    new LoggingInvocationHandler(target)
);
```

JVM **динамически ==создаёт новый класс==** (в runtime), который:
- реализует интерфейс `Service`;
    
- наследуется от `java.lang.reflect.Proxy`;
    
- ❗❗❗==внутри каждого метода== интерфейса (`process()`, `toString()`, `equals()` и т.д.)  
    автоматически ==вставляет вызов== `handler.invoke(proxy, method, args)`.

## 2. Что внутри динамически сгенерированного класса

Для примера, JVM создаёт что-то вроде (упрощённо):
```java
public final class $Proxy0 extends Proxy implements Service {
    private static Method m1;

    static {
        try {
            // Сохраняет ссылку на реальный Method объекта ServiceImpl
            m1 = Service.class.getMethod("process");
        } catch (NoSuchMethodException e) {
            throw new RuntimeException(e);
        }
    }

    public $Proxy0(InvocationHandler h) {
        super(h);
    }

    @Override
    public void process() {
        try {
            // ❗❗✅При вызове proxy.process() выполняется этот код:✅❗❗
            super.h.invoke(this, m1, null);
        } catch (Throwable t) {
            throw new UndeclaredThrowableException(t);
        }
    }
}
```

## 3. Объяснение ключевого момента

➡️ То есть **JVM сама вызывает `invoke()`** и **сама передаёт объект `Method`** (`m1` в примере выше).  
Этот объект `Method` создаётся рефлексией **при генерации класса прокси**.

Поэтому, когда ты вызываешь:
```java
proxy.process();
```

выполняется такой стек вызовов:

1. Вызов `proxy.process()` → JVM попадает в метод `process()` динамического класса `$Proxy0`.
    
2. Внутри `$Proxy0.process()` вызывается  
    `handler.invoke(proxy, m1, null)`
    
3. Твой `LoggingInvocationHandler` получает:
    - `proxy` — сам прокси-объект;
        
    - `m1` — объект `Method`, соответствующий `Service.process()`;
        
    - `args` — массив аргументов метода (в данном случае `null`, т.к. `process()` без параметров).
