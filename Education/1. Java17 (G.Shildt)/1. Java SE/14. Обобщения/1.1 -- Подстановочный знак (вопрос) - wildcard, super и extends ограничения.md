
В **обобщениях (generics)** ==подстановочный знак== `?` называется ==**wildcard== (джокер, метасимвол)**

Он используется тогда, когда мы **не знаем или не хотим указывать точный параметр типа**, а лишь задаём ограничения на него.

> [!NOTE] ==Подстановочный== знак `?`представляет `НЕИЗВЕСТНЫЙ тип`:
> ```java
> boolean isSameAvg(Stats<?> ob)
> ```

***
## 📙 1.1 ⭐⭐**Где используется**⭐⭐
### 🔶 1. В параметрах МЕТОДА
Это самое частое место:
```java
void printList(List<?> list) { ... }
void printNumbers(List<? extends Number> list) { ... }
void addIntegers(List<? super Integer> list) { ... }
```
Здесь wildcard помогает описать **диапазон допустимых типов**, которые можно передать в метод.

### 🔶2. В аргументах обобщённых классов / интерфейсов
Можно ==создавать ссылки на коллекции== или контейнеры:
```java
List<?> list;               // список неизвестного типа
List<? extends Number> nums; // список чисел или их подклассов
List<? super Integer> ints;  // список Integer или его супертипов
```

### 🔶 3. В полях класса
```java
class Container {
    List<?> data;
    List<? extends Number> numbers;
    List<? super Integer> integers;
}
```
Это легально, но обычно затрудняет добавление элементов в поле → ==такие конструкции используют для **хранения ссылок на коллекции**, которые ТОЛЬКО ЧИТАЮТСЯ==

### 🔶❓4. В возвращаемом типе метода
```java
List<? extends Number> getNumbers() { ... }
```
⚠️ Но на практике это встречается редко — чаще удобнее использовать параметр типа `<T>` в сигнатуре метода, чтобы код был более выразительным.

***
## 📙 **Что можно подставить?**

> [!NOTE] ⭐⭐**Ключевое правило (PECS):**⭐⭐
> **Producer Extends** — если коллекция только отдаёт значения (==читаем==), используем `? extends T`. 
> 
> **Consumer Super** — если коллекция только принимает значения (==пишем==), используем `? super T`.

### 1. 🟣 Базовое `?`
```java
List<?> list = new ArrayList<String>();
```
Здесь `list` может хранить ссылку на `List<String>`, `List<Integer>`, `List<Object>` — любой `List<SomeType>`.  

🛑Но **ограничение**: мы ==НЕ МОЖЕМ добавлять элементы== (💡кроме `null`💡, с `Object` - ошибка), так как компилятор не знает, какой именно тип у списка.  
Читать можно, но возвращаемое значение будет иметь тип `Object`.

👉 Применение: когда важен **контейнер**, но не важно, что именно внутри.
> [!NOTE] `Добавить нельзя` даже `Object`

***
### 2. 🟣 Ограничение сверху `? extends T`
```java
List<? extends Number> numbers = new ArrayList<Integer>();
```
- Можно подставить: `Number`, `Integer`, `Double` и т.д. (все подклассы `Number`).
    
- Разрешено **читать** элементы как `Number`.
    
- ==**Добавлять нельзя**==, потому что неизвестно, какой именно подтип `Number` используется внутри (например, список может быть `List<Double>`, а вы попытаетесь вставить `Integer` → ошибка).

> [!NOTE] Пример

👉 Используется, когда контейнер **только источник данных** (PECS — Producer Extends).

***
### 3. 🟣 ⭐⭐Ограничение снизу `? super T`⭐⭐
```java
List<? super Integer> integers = new ArrayList<Number>();
```

- Можно подставить: `Integer`, `Number`, `Object`.
- ==Можно **ДОБАВЛЯТЬ** объекты типа== `Integer` (и его СУПЕРклассы).
    
- Чтение возможно только как `Object`, потому что компилятор не знает, какой именно супертип внутри (`List<Object>` или `List<Number>`).
    
👉 Используется, когда контейнер **приёмник данных** (PECS — Consumer Super).

# 📕 1. ⭐⭐ Ограниченные аргументы с подстановочными знаками⭐⭐
🟣 **Верхняя граница**:  - только указанный класс и его подклассы
```java
<? extends Superclass>
```

🟣 **Нижняя граница**:  - только указанный класс и его суперклассы (`super`)
```java
<? super Subclass>
```

***

```java
// Ограниченный аргумент с подстановочным знаком.

// Двумерные координаты.
class TwoD {
    int x, y;
    
    TwoD(int a, int b) {
        x = a;
        y = b;
    }
}

// Трехмерные координаты.
class ThreeD extends TwoD {
    int z;
    
    ThreeD(int a, int b, int c) {
        super(a, b);
        z = c;
    }
}

// Четырехмерные координаты.
class FourD extends ThreeD {
    int t;
    
    FourD(int a, int b, int c, int d) {
        super(a, b, c);
        t = d;
    }
}

// Этот класс хранит массив объектов координат.
class Coords<T extends TwoD> {
    T[] coords;
    
    Coords(T[] o) {
        coords = o;
    }
}

// Демонстрация использования ограниченного 
// аргумента с подстановочным знаком.
class BoundedWildcard {
    static void showXY(Coords<?> c) {
        System.out.println("Координаты X Y:");
        for(int i = 0; i < c.coords.length; i++)
            System.out.println(c.coords[i].x + " " + c.coords[i].y);
        System.out.println();
    }
    
    static void showXYZ(Coords<? extends ThreeD> c) {
        System.out.println("Координаты X Y Z:");
        for(int i = 0; i < c.coords.length; i++)
            System.out.println(c.coords[i].x + " " + 
                             c.coords[i].y + " " + 
                             c.coords[i].z);
        System.out.println();
    }
    
    static void showAll(Coords<? extends FourD> c) {
        System.out.println("Координаты X Y Z T:");
        for(int i = 0; i < c.coords.length; i++)
            System.out.println(c.coords[i].x + " " + 
                             c.coords[i].y + " " + 
                             c.coords[i].z + " " + 
                             c.coords[i].t);
        System.out.println();
    }
    
    
    // ✅ MAIN ✅
    public static void main(String[] args) {
        TwoD[] td = {
            new TwoD(0, 0),
            new TwoD(7, 9),
            new TwoD(18, 4),
            new TwoD(-1, -23)
        };
        
        Coords<TwoD> tdlocs = new Coords<TwoD>(td);
        
        System.out.println("Содержимое tdlocs.");
        showXY(tdlocs); // Нормально, это объект TwoD
        // showXYZ(tdlocs); // Ошибка, не объект ThreeD
        // showAll(tdlocs); // Ошибка, не объект FourD
        
        // Создать несколько объектов FourD.
        FourD[] fd = {
            new FourD(1, 2, 3, 4),
            new FourD(6, 8, 14, 8),
            new FourD(22, 9, 4, 9),
            new FourD(3, -2, -23, 17)
        };
        
        Coords<FourD> fdlocs = new Coords<FourD>(fd);
        
        System.out.println("Содержимое fdlocs.");
        // Все вызовы выполнятся успешно.
        showXY(fdlocs);
        showXYZ(fdlocs);
        showAll(fdlocs);
    }
}
```

### Разбор примера выше 
Есть три уровня иерархии:
```nginx
TwoD  ←  ThreeD  ←  FourD
```

и обобщённый класс:
```java
class Coords<T extends TwoD>
```
> [!NOTE] Значит, в `Coords` могут быть ==ТОЛЬКО ТИПЫ, НАСЛЕДУЮЩИЕ== `TwoD` (сам `TwoD`, `ThreeD`, `FourD`).
> 

#### 1. Метод `showXY(Coords<?> c)`
```java
static void showXY(Coords<?> c)
```
- Здесь `?` означает «==любой тип, который удовлетворяет ограничению== `T extends TwoD`».
    
- Следовательно, можно передать:
    - `Coords<TwoD>`
    - `Coords<ThreeD>`
    - `Coords<FourD>`

👉 В этом методе доступны **только поля `TwoD`** (x и y), потому что компилятор не знает, конкретный это `TwoD`, `ThreeD` или `FourD`.

#### 2. Метод `showXYZ(Coords<? extends ThreeD> c)`
```java
static void showXYZ(Coords<? extends ThreeD> c)
```
- Здесь `? extends ThreeD` означает «==любой тип, который является== `ThreeD` или его ==подклассом==».
    
- Следовательно, можно передать:
    - `Coords<ThreeD>`
    - `Coords<FourD>`

👉 В этом методе доступны **поля `ThreeD` и его родителей**: `x`, `y`, `z`. Но поля `t` (из `FourD`) использовать нельзя, потому что не факт, что это `FourD`.

#### 3. Почему нельзя `Coords<TwoD>` в `showXYZ`?
`TwoD` не наследует `ThreeD`, поэтому `Coords<TwoD>` не подходит. Иначе можно было бы обратиться к `z`, которого в `TwoD` нет → ошибка компиляции.
