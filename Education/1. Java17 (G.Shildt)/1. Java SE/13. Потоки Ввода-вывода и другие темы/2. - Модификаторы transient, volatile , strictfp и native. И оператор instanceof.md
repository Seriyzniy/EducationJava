# 📕1. Модификатор `transient`
- `transient` — метка для сериализации: поле не участвует в _де/сериализации_ при `Serializable` (и обычно должно восстанавливаться вручную при десериализации).
```java
class T {
    transient int a; // не сохраняется при сериализации
    int b; // сохраняется
}
```

> [!NOTE] ⭐`transient` JSON/XML сериализаторы (Gson/Jackson)
> `transient` не влияет на JSON/XML сериализаторы (Gson/Jackson) — они используют свои правила. Для них отдельные аннотации (`@JsonIgnore`) нужны.

# 📕2. Модификатор `volatile`
- `volatile` — метка для многопоточности: гарантирует **видимость** и **специальные правила упорядочивания** (acquire/release), но **не делает составные операции атомарными**.
```java
class SharedData {
    volatile boolean flag; // гарантирует актуальность значения для всех потоков
}
```

# 📕 3. Модификатор `native`

`native` — это **модификатор метода**, который говорит JVM:  
👉 «==Тело метода реализовано не на Java, а на другом языке== (обычно C или C++), и загружается через JNI (Java Native Interface)».

То есть в Java ты только объявляешь сигнатуру метода, а сама реализация лежит в нативной библиотеке (`.dll` на Windows, `.so` на Linux, `.dylib` на macOS).

```java
class NativeExample {
    // объявление нативного метода
    public native void printHello();

    static {
        // загружаем библиотеку mylib.dll (или libmylib.so)
        System.loadLibrary("mylib");
    }
}
```

## Как это работает (пошагово)
1. **Java-класс** объявляет метод с `native`.
    
2. В `static {}` блоке вызывается `System.loadLibrary("имя")`.
    - JVM ищет библиотеку в `java.library.path`.
    - Подгружает её (`.dll`/`.so`).
        
3. Когда вызывается метод:
    - JVM ищет функцию с определённой сигнатурой в библиотеке.
    - Передаёт туда параметры.
    - Возвращает результат обратно в Java-мир.

## Ключевые особенности `native`
1. ✅ **Только для методов**.  
    Нельзя применить к классам, переменным, конструкторам.
    
2. ✅ **Нет реализации в Java**.  
    Если ты объявил метод `native`, то тело у него **пустое** (заканчивается `;`).

 3. **Вызов работает через JNI**.  
    JVM идёт в динамическую библиотеку и вызывает функцию по имени.
    
4. ✅ **Может бросать исключения**, но они должны быть "проброшены" из нативного кода через JNI API.

# 📕4. 🧠Модификатор `strictfp`🧠
`strictfp` — это **модификатор**, который ==гарантирует строгую== (детерминированную) ==семантику вычислений с плавающей точкой== (floating-point) в Java.

> [!NOTE] ⭐⭐⭐==УСТАРЕЛ С JDK 17==⭐⭐⭐
> Помните! Начиная с JDK 17, модификатор stricfp был объявлен устаревшим и теперь его использование приводит к выдаче предупреждающего сообщения.


> [!NOTE] Он говорит компилятору и JVM:  
👉 "Используй _только IEEE 754 strict mode_ для всех операций с `float` и `double`, не применяй аппаратно-зависимые расширения".
> 

## Где можно применять `strictfp`?
Он применяется к:
- **Классам**
- **Интерфейсам**
- **Методам**
    
❌ Нельзя применить к **переменным** или **конструкторам**.

> [!NOTE] ⭐⭐Если модификатор стоит у класса/интерфейса → **все его методы** автоматически `strictfp`.
>

```java
strictfp class Calculator {
    double calc(double a, double b) {
        return a / b * Math.PI;
    }
}

class Demo {
    strictfp double divide(double a, double b) {
        return a / b;
    }
}
```

## Что именно делает `strictfp`?
1. **Все операции с float/double соответствуют IEEE 754**:
    - строго 32 бита для `float`
    - строго 64 бита для `double`
        
    - никаких промежуточных 80-битных расширений
        
2. **Округление и переполнение тоже строго фиксированы**:
    - `Infinity`, `NaN`, `+0.0`, `-0.0` обрабатываются одинаково везде.
        
3. **Одинаковый результат на всех платформах**.

# 📕5. Оператор `instanceof`
Иногда ==полезно знать ТИП ОБЪЕКТА во время выполнения.== 

Например, у вас может быть один поток выполнения, генерирующий объекты различных типов, и другой поток, который эти обьекты обрабатывает. В такой ситуации может быть удобно, чтобы обрабатывающий поток знал типы всех объектов при их получении. 
Еще одна ситуация, коrда важно знать тип объекта во время вы полнения, связана с приведением. В Java недопустимое приведение становится причиной ошибки времени выполнения

> [!NOTE] Оператор `instanceof`
> `instanceof` — это **бинарный оператор** в Java, который проверяет, является ли объект экземпляром указанного класса/подкласса/реализации интерфейса.

```java
obj instanceof Type
```
- `obj` — выражение, дающее ссылку на объект (может быть `null`).
    
- `Type` — имя класса, интерфейса или обобщённого типа (generic erased type).

Результат — `boolean`.


> [!NOTE] Generic и instanceof ЗАПРЕЩЕН НО...
> Generics в Java работают через **type erasure**, поэтому `instanceof` с параметризованным типом запрещён.
> ```JAVA
> List<String> list = new ArrayList<>();
> ❌if (list instanceof List<String>) { } // ошибка: illegal generic type
> 
> ✅if (list instanceof List<?>)
> ```

## Сравнение с `getClass()`

Иногда путают `instanceof` и `getClass()`. Разница:

- `instanceof` проверяет **иерархию** (суперклассы, интерфейсы).
    
- `getClass()` проверяет **точный класс**.
