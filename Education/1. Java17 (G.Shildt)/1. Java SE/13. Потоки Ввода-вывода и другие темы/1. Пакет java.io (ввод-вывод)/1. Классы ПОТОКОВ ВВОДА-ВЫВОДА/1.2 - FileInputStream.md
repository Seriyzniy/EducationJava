📌 `FileInputStream` — это **байтовый поток ввода**, который ==читает данные из **файла** (или дескриптора файла на уровне ОС)==.
- Наследуется от `InputStream`.
- Используется для **чтения байтов** (==файлы любого типа==: текстовые, бинарные, изображения).
    
- ❗❗❗В отличие от `ByteArrayInputStream`, работает НЕ с памятью, а с **диском** → может выбрасывать `IOException`.
## Конструкторы
```java
// ⭐`name` — путь к файлу.
// Если файл не существует или нет прав доступа → `FileNotFoundException`.
public FileInputStream(String name) throws FileNotFoundException

// ⭐ То же самое, но принимает объект `File`.
// Удобнее, если работаешь с API `java.io.File`.
public FileInputStream(File file) throws FileNotFoundException

//Открывает поток по дескриптору файла.
//Обычно не используешь напрямую, а получаешь через `FileOutputStream.getFD()` или системные вызовы.  
//⚠️ Опасно в плане безопасности: позволяет работать с низкоуровневым дескриптором ОС.
public FileInputStream(FileDescriptor fdObj)
```

## Методы

📌 Наследуются от `InputStream`, но реализованы «нативно» (через вызовы ОС).

```java
// Читает **1 байт** (возвращает `0–255`).
// Возвращает `-1`, если достигнут конец файла (EOF).
public int read() throws IOException

// Читает данные в массив `b`.
// Возвращает количество реально прочитанных байтов, либо `-1` при EOF.
public int read(byte[] b) throws IOException

// Читает в массив `b`, начиная с позиции `off`, до `len` байтов.   
// Возвращает фактическое количество прочитанного или `-1`.
public int read(byte[] b, int off, int len) throws IOException

// Пропускает до `n` байтов.
// Возвращает фактическое число пропущенных байтов. 
// ⚠️ В больших файлах может не сработать "идеально" (зависит от ОС).
public long skip(long n) throws IOException

// Возвращает **оценку** количества доступных байтов без блокировки.    
//⚠️ Не использовать для определения размера файла! Это не `file.length()`.
public int available() throws IOException

// Закрывает поток (и дескриптор файла в ОС). 
// После закрытия вызовы `read()` дадут `IOException`.  
//⚠️ Если не закрыть — утечка файловых дескрипторов.
public void close() throws IOException
```

```java
// Возвращает объект `FileDescriptor`, связанный с этим потоком.
// Используется редко (например, для низкоуровневого доступа).
public final FileDescriptor getFD() throws IOException
```

## Пример чтение по одному байту
```java
import java.io.FileInputStream;
import java.io.IOException;

public class Demo {
    public static void main(String[] args) {
        try (FileInputStream fis = new FileInputStream("test.txt")) {
            int b;
            while ((b = fis.read()) != -1) {
                System.out.print((char) b);
            }
        } catch (IOException e) {
            e.printStackTrace();
        }
    }
}
```

> [!NOTE] ❗❗❗ `Нельзя читать ВСЕ текстовые файлы`, т.к кодировка может отличаться ❗❗❗

Что здесь происходит:
- `fis.read()` читает **один байт** (`0..255`) из файла.
    
- `(char) ch` просто берёт этот байт и превращает его в символ, **без какой-либо кодировки**.
    
- Ты фактически читаешь **сырые байты**, а не текстовые символы.

- ⭐⭐Это **работает корректно только если файл в ASCII** или в кодировке, где символ = один байт (например ISO‑8859‑1).
    
- ⭐⭐Если файл в UTF‑8, UTF‑16 или другой многобайтовой кодировке → ты получишь **мусор или неверные символы** (используй классы `Reader`)

## ✅Пример чтение блоками (правильнее)
```java
import java.io.FileInputStream;
import java.io.IOException;

public class BufferDemo {
    public static void main(String[] args) {
        byte[] buffer = new byte[1024]; // читаем по 1 KB
        try (FileInputStream fis = new FileInputStream("test.txt")) {
            int bytesRead;
            while ((bytesRead = fis.read(buffer)) != -1) {
                System.out.print(new String(buffer, 0, bytesRead));
            }
        } catch (IOException e) {
            e.printStackTrace();
        }
    }
}
```

## 🛑Подводные камни🛑
1. **`available()` не равен размеру файла.**  
    Если нужен размер → `new File("file.txt").length()`.
    
2. **Утечки ресурсов.**
    - Если не закрывать поток (`close()`), ОС может выдать `Too many open files`.
    - Решение → всегда использовать `try-with-resources`.
        
3. **Производительность.**
    - Чтение по одному байту → крайне неэффективно.
    - Используй `BufferedInputStream`, если файл большой.
        
4. **Проблемы с кодировкой.**
    - `FileInputStream` работает с **байтами**, а не с символами.
        
    - Для текста используй `InputStreamReader` (с указанием кодировки).

## ✅Best Practices✅

1. ✅ **Используй `try-with-resources`** для автоматического закрытия.
    
2. ✅ Для текстовых файлов → `InputStreamReader` с `Charset`.
    
3. ✅ Для больших файлов → `BufferedInputStream` или `Files.newBufferedReader()`.
    
4. ✅ Если надо копировать файл → используй `Files.copy()`, а не ручные циклы.
    
5. ❌ Не используй `available()` для размера.
    
6. ✅ Для тестов, когда нужен поток из строки → используй `ByteArrayInputStream`, а не `FileInputStream`.
