`FilterInputStream` — это ==**абстракция-декоратор**== над другим `InputStream`.
- Он **наследуется от `InputStream`**.
    
- Его главная задача — **обернуть существующий поток** и "фильтровать" данные при чтении.
	
- Все его методы по умолчанию просто **делегируют вызовы** другому `InputStream`.
***
## 💡💡Особенность💡💡

У `FilterInputStream` ==нет "своей" логики — он **чистый декоратор**.==
- ⭐⭐Его прямое использование не имеет смысла.⭐⭐
    
- Все реальные фильтры наследуются от него и **переопределяют методы**.

***
## Конструктор
```java
protected FilterInputStream(InputStream in)
```
- `in` — ==поток, который мы оборачиваем==.
- Обычно вызывается **только в наследниках** (например, `BufferedInputStream`, `DataInputStream`).

## Методы
Все методы **делегируют** в `in` (тот поток, который был передан в конструктор).

```java
// Читает **1 байт** (возвращает `0–255`).
// Возвращает `-1`, если достигнут конец файла (EOF).
// ✅Делегирует в `in.read()`.✅
public int read() throws IOException

// Читает данные в массив `b`.
// Возвращает количество реально прочитанных байтов, либо `-1` при EOF.
// ✅Делегирует в `in.read(b)`.✅
public int read(byte[] b) throws IOException

// Читает в массив `b`, начиная с позиции `off`, до `len` байтов.   
// Возвращает фактическое количество прочитанного или `-1`.
// ✅Делегирует в `in.read(b, off, len)`.✅
public int read(byte[] b, int off, int len) throws IOException

// Пропускает до `n` байтов.
// Возвращает фактическое число пропущенных байтов. 
// ⚠️ В больших файлах может не сработать "идеально" (зависит от ОС).
// ✅ Делегирует в `in.skip(n)`.✅
public long skip(long n) throws IOException

// Возвращает **оценку** количества доступных байтов без блокировки.    
//⚠️ Не использовать для определения размера файла! Это не `file.length()`.
// ✅ Делегирует в `in.available()`.✅
public int available() throws IOException

// Закрывает поток (и дескриптор файла в ОС). 
// После закрытия вызовы `read()` дадут `IOException`.  
//⚠️ Если не закрыть — утечка файловых дескрипторов.
// ✅ Делегирует в `in.close()`.✅
public void close() throws IOException

// Устанавливает «закладку» в текущей позиции (`pos`).
// `readAheadLimit` для этого класса игнорируется.
// ✅ Делегирует в `in.mark(readAheadLimit)`.✅
public synchronized void mark(int readAheadLimit)

// Возвращает курсор (`pos`) обратно на позицию, сохранённую вызовом `mark()`.
// Если `mark()` не вызывался → вернёт в начало.
// ✅ Делегирует в `in.reset()`.✅
public synchronized void reset()

//Возвращает `true` если `mark()` и `reset()` поддерживаются.
// ✅ Делегирует в `in.markSupported()`.✅
public boolean markSupported()
```

## Пример
```java
// ⭐⭐FilterInputStream  НЕ ИСПОЛЬЗУЕТСЯ САМОСТОЯТЕЛЬНО!
class UpperCaseInputStream extends FilterInputStream {
    protected UpperCaseInputStream(InputStream in) {
        super(in);
    }

    @Override
    public int read() throws IOException {
        int c = super.read();
        return (c == -1 ? c : Character.toUpperCase((char) c));
    }
}

public class Demo {
    public static void main(String[] args) {
        try (InputStream fis = new FileInputStream("test.txt");
             InputStream upper = new UpperCaseInputStream(fis)) {

            int c;
            while ((c = upper.read()) != -1) {
                System.out.print((char) c);
            }

        } catch (IOException e) {
            e.printStackTrace();
        }
    }
}
```

## Best Practices

1. ✅ **Никогда не используй `FilterInputStream` напрямую.**  
    Он задуман как **базовый класс для наследников**.
    
2. ✅ Используй **готовые фильтры** (`BufferedInputStream`, `DataInputStream`, `DigestInputStream`) вместо велосипедов.
    
3. ✅ Если делаешь свой — **переопредели только нужные методы**. Обычно это `read()`, иногда `close()`.
    
4. ✅ Комбинируй декораторы:
```JAVA
InputStream in = new BufferedInputStream(new FileInputStream("file.bin"));
DataInputStream din = new DataInputStream(in);
int x = din.readInt();
```

5. ❌ Не забывай про `close()` — лучше `try-with-resources`.
