Чтобы создать собственное исключение необходимо просто наследоваться от одного из классов Exception или RuntimeException
```java
// Checked — т.к. наследуем от Exception
public class MyCheckedException extends Exception {
    public MyCheckedException() {
        super();
    }

    public MyCheckedException(String message) {
        super(message);
    }

	// Связка исключений
    public MyCheckedException(String message, Throwable cause) { /
        super(message, cause);
    }

    public MyCheckedException(Throwable cause) {
        super(cause);
    }
}
```

```java
// Unchecked — т.к. наследуем от RuntimeException
public class MyUncheckedException extends RuntimeException {
    public MyUncheckedException() {
        super();
    }

    public MyUncheckedException(String message) {
        super(message);
    }

    public MyUncheckedException(String message, Throwable cause) {
        super(message, cause);
    }

    public MyUncheckedException(Throwable cause) {
        super(cause);
    }
}
```

# 💡 Особенности 💡
## 🔶1.  Почему лучше наследоваться от `Exception` или `RuntimeException`?
Потому что **они — часть контрактов Java**: JVM, компилятор и сторонние библиотеки **понимают, как работать** с ними.

Не стоит напрямую наследоваться от `Throwable` — это **технически возможно**, но **плохо читаемо и не рекомендуется**.

## 🔶 2.  Не забудь реализовать **все конструкторы**
Чтобы твое исключение можно было использовать гибко:
- с сообщением;
    
- с причиной (`Throwable cause`);
    
- без аргументов.

## Что выбрать Checked или unchecked?

| Если...                                                        | Тогда наследуй от...           |
| -------------------------------------------------------------- | ------------------------------ |
| Ошибка связана с ==**внешними условиями**== (файлы, сеть, БД)  | `Exception` (Checked)          |
| Ошибка связана с **логикой программы** (некорректный аргумент) | `RuntimeException` (Unchecked) |

# 💡 Хорошая практика 💡

- Именуй свои исключения с постфиксом `Exception`.
    
- Используй `@Override` для `getMessage()` или `toString()`, если хочешь персонализировать.
    
- Документируй свои исключения: когда и почему они выбрасываются.
    
- Используй `cause` при ==оборачивании других исключений== (`new MyException("fail", e)`).

# 📕1. Что такое `cause` (СЦЕПЛЕНИЕ ИСКЛЮЧЕНИЙ)
Каждое исключение в Java (наследник `Throwable`) может **содержать ссылку на другое исключение**, называемое **"причиной"** (`cause`).

Это удобно, когда ты хочешь **создать своё собственное исключение**, но **не терять информацию об исходной ошибке**, которая его вызвала.

Чтобы сделать возможными сцепленные исключения, в класс Throwable были добавлены два конструктора и два метода. 

✅Конструкторы приведены ниже: 
```java
Throwable(Throwable causeExc) 
Throwable(String msg, Throwable causeExc)
```

✅Методы для сцепленных исключений `getCause()` и `initCause()`
```java
Throwable getCause() 
Throwable initCause(Throwable causeExc)
```

> [!NOTE] Одно исключение-причина на один объект исключения
> Таким образом, вы можете ассоциировать причину с исключением после его создания. Однако исключение-причина может быть установлено только один раз, т.е. вызывать initCause ( ) для каждого объекта исключения допускается только один раз.


## Допустим, у тебя есть метод, который выбрасывает `SQLException`:

```java
public void saveUser(User user) throws SQLException {
    // что-то пошло не так при работе с БД
    throw new SQLException("Ошибка соединения с БД");
}
```

А ты хочешь обернуть это в своё исключение:
```java
public class UserSaveException extends Exception {
    public UserSaveException(String message, Throwable cause) {
        super(message, cause); // 🧠🟢<-- связываем причину 
		                       //(просто ссылка на объект другого исключения)
    }
}
```

И используешь так:
```java
public void registerUser(User user) throws UserSaveException {
    try {
        saveUser(user);
    } catch (SQLException e) {
        throw new UserSaveException("Не удалось сохранить пользователя", e);
    }
}
```

После чего можно получить информацию о верхнем уровне исключения (UserSaveException) а затем уже о причине исключения (SQLException)
