В программировании часто бывает полезно отделять то, что должно быть сделано, от того, как оно делается. В главе 9 вы узнали, что один из способов достижения такой цели в языке Java предусматривает применение интерфейсов. 

> [!NOTE] Интерфейс определяет что, а реализующий его класс - как. 

Концепцию можно расширить так, чтобы реализующий класс предоставлялся кодом, который находится за пределами программы, за счет использования подключаемого модуля. 

> [!NOTE] ⭐Можно `просто подключить другую реализацию`, `без необходимости компилировать` проект⭐
> При таком подходе возможности приложения можно совершенствовать, модернизировать либо изменять, просто меняя подключаемый модуль.

# 📕 1. Основы служб и поставщиков служб

## 📙1. Ключевые понятия:
- **Служба (Service)**: ==Определяется интерфейсом== или абстрактным классом (например, `BinaryFunc`).
- **Поставщик службы (Service Provider)**: Класс, который предоставляет конкретную ==реализацию службы== (например, `AbsPlusProvider`).
    
- 💡✅**Загрузчик служб (`ServiceLoader<S>`)**💡: ==Класс== из `java.util`, который ==находит и загружает всех поставщиков== для данной службы.
```java
class ServiceLoader<S>
```

## 📙2. ServiceLoader
Этот класс появился ещё в **Java** 6, но с приходом модулей (Java 9+) стал ещё важнее. Его задача — находить и загружать реализации сервисов (интерфейсов/абстрактных классов), которые объявлены как провайдеры (в `META-INF/services/` или через `provides ... with ...` в `module-info.java`).

### 💡Ключевые моменты:
- У `ServiceLoader` ==**нет публичных конструкторов**== — ты создаёшь его через фабричные методы (`load`, `loadInstalled`, `loadWith...`).
    
- Это по сути ленивый итератор, который подгружает реализации по мере обхода.
    
- Позволяет перебирать все реализации, а не только одну.

### Фабричные методы для создания
```java
/*
 * ⭐⭐static <S> ServiceLoader<S> load(Class<S> service)⭐⭐
 * Загружает сервисы с использованием контекстного ClassLoader.
 */
ServiceLoader<Runnable> loader1 = ServiceLoader.load(Runnable.class);

/*
 * static <S> ServiceLoader<S> load(Class<S> service, ClassLoader loader)
 * То же самое, но с явным загрузчиком.
 */
ServiceLoader<Runnable> loader2 = ServiceLoader.load(Runnable.class, ClassLoader.getSystemClassLoader());

/*
 * static <S> ServiceLoader<S> loadInstalled(Class<S> service)
 * Загружает сервисы через установленные (system/bootstrap) загрузчики.
 */
ServiceLoader<Runnable> loader3 = ServiceLoader.loadInstalled(Runnable.class);

/*
 * static <S> ServiceLoader<S> loadWith(Class<S> service, ModuleLayer layer)
 * ⭐Загружает сервисы из указанного ModuleLayer.
 */
ServiceLoader<Runnable> loader4 = ServiceLoader.loadWith(Runnable.class, ModuleLayer.boot());
```

### Методы
```java
ServiceLoader<Runnable> loader = ServiceLoader.load(Runnable.class);

/*
 * Iterator<S> iterator()
 * Перебор сервисов (лениво).
 */
Iterator<Runnable> it = loader.iterator();

/*
 * Stream<Provider<S>> stream()
 * Поток провайдеров (Java 9+).
 */
Stream<ServiceLoader.Provider<Runnable>> stream = loader.stream();

/*
 * void reload()
 * Сброс кеша, пересканирование.
 */
loader.reload();

/*
 * Spliterator<S> spliterator()
 * Для параллельного обхода сервисов.
 */
Spliterator<Runnable> spl = loader.spliterator();

/*
 * Spliterator<Provider<S>> spliterator()
 * Для параллельного обхода провайдеров (Java 9+).
 */
Spliterator<ServiceLoader.Provider<Runnable>> spl2 = loader.stream().spliterator();

/*
 * toString()
 * Строковое представление.
 */
System.out.println(loader);

/*
 * Iterable<S>
 * Можно использовать foreach.
 */
for (Runnable r : loader) {
    r.run();
}
```

# 📕 2. Ключевые слова, связанные со службами
1. 🔶**`provides ... with ...`**: Объявляет, что ==модуль ПРЕДОСТАВЛЯЕТ РЕАЛИЗАЦИЮ службы==.

> [!NOTE] ⭐УКАЗЫВАЕМ КЛАСС `Provider`⭐, с помощью которого получаем определенные реализации
> 
```java
provides service.Interface with impl.ProviderClass; // ✅Класс provider
```

2. **`uses`**: Объявляет, что ==модуль ИСПОЛЬЗУЕТ службу==.
```java
uses service.Interface;
```

Когда они используются все вместе, то позволяют задать модуль, предоставляющий службу, модуль, который нуждается в этой службе, и конкретную реализацию самой службы.

## 📙 1. Интерфейс службы
Понадобятся ==два интерфейса, связанных со службами.== 
1) Один задает форму действия,
2) Второй указывает форму поставщика этого действия. 

Определения обоих интерфейсов находятся в каталоге `binaryfuncs` и оба они входят в состав пакета `userfuncs.binaryfuncs`.
##### Первый интерфейс по имени BinaryFunc объявляет форму бинарной функции:
Интерфейс `BinaryFunc` ==объявляет форму объекта, который может реализовывать бинарную целочисленную функцию==, что указывается методом `func()`. 
С помощью метода `getName()` можно получить имя функции, которое будет использоваться для выяснения типа реализуемой функции.
```java
package userfuncs.binaryfuncs;

public interface BinaryFunc {
	// Получить имя функции
    public String getName();
    
    // Это функция, подлежащая выполнению. Она будет
    // предоставляться конкретными реализациями
    public int func(int a, int b);
}
```

##### Второй интерфейс объявляет форму поставщика служб и называется BinFuncProvider:
В интерфейсе `BinFuncProvider` объявлен только один метод `get()`, который применяется для получения экземпляра `BinaryFunc`. 
==Этот интерфейс должен быть реализован классом, желающим предоставлять экземпляры== `BinaryFunc`.
```java
package userfuncs.binaryfuncs;

public interface BinFuncProvider {
	// Получить экземпляр BinaryFunc
    public BinaryFunc get();
}
```

## 📙 2. Классы реализации
В рассматриваемом примере поддерживаются две конкретные реализации BinaryFunc. 
1) Первая реализация, `AЬsPlus`, возвращает ==сумму== абсолютных значений своих аргументов. 
2) Вторая реализация, `AЬsMinus`, возвращает результат ==вычитания== абсолютного значения второго аргумента из абсолютного значения первого аргумента.

Исходный код этих классов должен храниться в каталоге `binaryfuncsimpl` и принадлежать пакету `userfuncsimpl.binaryfuncsimpl`
```java
package userfuncsimpl.binaryfuncsimpl;

import userfuncs.binaryfuncs.BinaryFunc;

public class AbsPlus implements BinaryFunc {
	// Возвратить имя этой функции
    public String getName() {
        return "absPlus";
    }
    
    // Реализовать функцию AbsPlus
    public int func(int a, int b) {
        return Math.abs(a) + Math.abs(b);
    }
}
```

Ниже приведен код AbsMinus:
```java
package userfuncsimpl.binaryfuncsimpl;

import userfuncs.binaryfuncs.BinaryFunc;

public class AbsMinus implements BinaryFunc {
	// Возвратить имя этой функции
    public String getName() {
        return "absMinus";
    }
    
    // Реализовать функцию AbsMinus
    public int func(int a, int b) {
        return Math.abs(a) - Math.abs(b);
    }
}
```

Для получения экземпляра `AbsPlus` используется класс `AbsPlusProvider`. Он реализует интерфейс `BinFuncProvider` и выглядит следующим образом:
```java
package userfuncsimpl.binaryfuncsimpl;

import userfuncs.binaryfuncs.BinFuncProvider;
import userfuncs.binaryfuncs.BinaryFunc;

public class AbsPlusProvider implements BinFuncProvider {
	// Предоставить объект AbsPlus
    public BinaryFunc get() {
        return new AbsPlus();
    }
}
```

Поставщик для AЬsMinus называется AЬsMinusProvider и показан ниже:
```java
package userfuncsimpl.binaryfuncsimpl;

import userfuncs.binaryfuncs.BinFuncProvider;
import userfuncs.binaryfuncs.BinaryFunc;

public class AbsMinusProvider implements BinFuncProvider {
    // Предоставить объект AbsMinus
    public BinaryFunc get() {
        return new AbsMinus();
    }
}
```



## 📙 3. Файлы определения модулей
Далее необходимы два файла определения модулей. 

Модуль для проекта `MyModuleUserFuncsInterface`
```java
module userfuncs {
	exports userfuncs.binaryfuncs;
}
```
В нем экспортируется пакет `userfuncs.binaryfuncs`, где определены интерфейсы `BinaryFunc` и `BinFuncProvider`.

Модуль для проекта `MyModuleUserFuncsImplementation`
```java
module userfuncsimpl {
    requires userfuncs;
    
    provides userfuncs.binaryfuncs.BinFuncProvider with
        userfuncsimpl.binaryfuncsimpl.AbsPlusProvider,
        userfuncsimpl.binaryfuncsimpl.AbsMinusProvider;
}
```
> [!NOTE] ❗Что реализует❗
> В модуле `userfuncsimpl` требуется модуль `userfuncs`, поскольку последний содержит интерфейсы `BinaryFunc` и `BinFuncProvider`, которые нужны реализациям. 
> 
> Модуль `userfuncsimpl` предоставляет реализации `BinFuncProvider` в виде классов `AЬsPlusProvider` и `AЬsMinusProvider`.

## 📙 5. Демонстрация поставщиков служб в классе ModAppDemo
Чтобы продемонстрировать работу со службами, метод `main()` класса `ModAppDemo` расширен с целью применения `AЬsPlus` и `AbsMinus`, для чего загружает их во время выполнения с помощью `ServiceLoader.load()`.

### 1. Изменение файла `module-info.java`
Для начала изменим файл модуля, 
1) Объявив, что нам необходим модуль `userfuncs`, ==где объявлены интерфейсы==, 
2) Используя ключевое слово `uses` ==скажем что текущий модуль использует РЕАЛИЗАЦИЮ СЛУЖБЫ== `BinFuncProvider` (⭐именно интерфейс, т.к НЕ НУЖНО привязываться к конкретной реализации⭐)

```java
module modAppDemo {
	requires modMathFuncs;
	
	// ✅ Необходим МОДУЛЬ (имя_модуля) в котором находятся интерфейсы служб
	requires userfuncs;
	
	//✅ Модуль использует реализацию служб, объявленных в этом пакете
	uses userfuncs.binaryfuncs.BinFuncProvider; 
}
```

### 2. Добавление пути в `Module path`

### Изменение класса ModAppDemo

```java
package org.appstart;

import java.util.ServiceLoader;

import com.mathfuncs.SimpleMathFuncs;

import userfuncs.binaryfuncs.BinFuncProvider;
import userfuncs.binaryfuncs.BinaryFunc;

public class ModAppDemo {
	public static void main(String[] args) {
		// Первым делом использовать встроенные функции, как и ранее
        if (SimpleMathFuncs.isFactor(2, 10)) {
            System.out.println("2 является делителем 10");
        }
        
        System.out.println("Наименьший общий делитель для 35 и 105 равен " +
            SimpleMathFuncs.lcf(35, 105));
            
        System.out.println("Наибольший общий делитель для 35 и 105 равен " +
            SimpleMathFuncs.gcf(35, 105));
            
            
        //✅Теперь использовать пользовательские операции, основанные на службах
        // Получить загрузчик служб для бинарных функций
        ServiceLoader<BinFuncProvider> ldr = 
            ServiceLoader.load(BinFuncProvider.class);
            
        BinaryFunc binOp = null;
        
        // ⭐ Найти поставщика для absPlus ⭐ и получить реализацию функции
        for (BinFuncProvider bfp : ldr) {
            if (bfp.get().getName().equals("absPlus")) {
                binOp = bfp.get();
                break;
            }
        }
        
        if (binOp != null) {
            System.out.println("Результат выполнения функции absPlus: " +
                binOp.func(12, -4));
        } else {
            System.out.println("Функция absPlus не найдена.");
        }
        
        binOp = null;
        
        // ⭐ Найти поставщика для absMinus⭐ и получить функцию
        for (BinFuncProvider bfp : ldr) {
            if (bfp.get().getName().equals("absMinus")) {
                binOp = bfp.get();
                break;
            }
        }
        
        if (binOp != null) {
            System.out.println("Результат выполнения функции absMinus: " +
                binOp.func(12, -4));
        } else {
            System.out.println("Функция absMinus не найдена.");
        }
    }
}
```
