# 📕 1. Exports to ...
Оператор `exports` можно ограничить с помощью конструкции `to`, чтобы экспортировать пакет только определенным модулям (уточненный экспорт).

- **Синтаксис**: `exports package.name to module1, module2;`
    
```bash
// ❗Пример команды из книги (не проверял)❗
javac -d appmodules --module-source-path appsrc appsrc\appstart\appstart\mymodappdemo\MyModAppDemo.java
```


**Компиляция:** Для компиляции таких модулей удобно использовать ==**многомодульный режим** компилятора== `javac` с параметром `--module-source-path`, который компилирует все модули сразу, учитывая их взаимные зависимости.

> [!NOTE] `--module-source-path`
> Параметр `--module-source-path` обеспечивает ==автоматическую компиляцию файлов в дереве ниже указанного каталога==, которым является appsrc. 
> 
> Параметр `--module-source-path` должен применяться вместе с параметром -d, чтобы гарантировать сохранение скомпилирован ных модулей в надлежащих каталогах внутри appmodules.

# 📕 2. Использование requires transitive
Рассматривается ситуация с транзитивными зависимостями (если модуль A зависит от B, а B зависит от C, то A косвенно зависит от C).

- ❌❓**Проблема**: Если модулю A нужно использовать типы из модуля C, пришлось бы добавлять `requires C` в модуль A. Это неудобно, если модуль B используется многими модулями.
    
- ✅**Решение**: ==Ключевое слово== `transitive` в операторе `requires`.
    - Если в модуле B объявить `requires transitive C`, то любой модуль, который зависит от B (как модуль A), будет автоматически получать доступ к C.
```java
// Определение модуля для appfuncs
module modMathFuncs {
    // Экспортировать пакет appfuncs.simplefuncs
    exports com.mathfuncs;
    
    // Затребовать модуль appsupport и сделать его транзитивным
    requires transitive appsupport;
}
```
