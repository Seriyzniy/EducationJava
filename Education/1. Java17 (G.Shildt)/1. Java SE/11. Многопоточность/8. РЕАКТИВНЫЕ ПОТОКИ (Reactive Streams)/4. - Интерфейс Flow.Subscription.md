
> [!NOTE] `Flow.Subscription` — это интерфейс, представляющий `канал управления` между `Publisher` и `Subscriber`.
> Он позволяет подписчику:
> - запрашивать нужное количество элементов (`request(long n)`), 
> - или отменить подписку (`cancel()`),  
> 
> таким образом регулируя поток данных и предотвращая перегрузку

## Пример создания
```java
Flow.Subscription subscription = new Flow.Subscription() {
    @Override
    public void request(long n) {
        // Запросить n элементов
    }

    @Override
    public void cancel() {
        // Отменить подписку
    }
};
```

## Методы

Интерфейс определяет **два метода**, которые полностью описывают его поведение.
```java
// Запрашивает у издателя (Publisher) n элементов.
// Publisher после этого может вызывать onNext() до n раз.
void request(long n);
```
🟩 **Пояснение:**
- Вызывается **подписчиком** для указания, сколько элементов он готов принять.
    
- `n` — количество запрашиваемых элементов.
    
- Если `n <= 0`, то Publisher **должен вызвать `onError(IllegalArgumentException)`** и завершить подписку.
    
- Может вызываться **многократно**, и общее количество запрошенных элементов суммируется.
    
- 💡Publisher не обязан немедленно отправлять данные — он может их выдавать асинхронно.
    
- Поддерживает _backpressure_: подписчик управляет скоростью поступления данных.

---
```java
// Отменяет подписку.
// После вызова этого метода Publisher должен прекратить отправку данных.
void cancel();
```
🟩 **Пояснение:**
- 💡Отменяет поток данных, даже если не все элементы были получены.
    
- После `cancel()` ни один метод (`onNext`, `onError`, `onComplete`) не должен вызываться.
    
- Можно использовать, если подписчику больше не нужны данные, либо при ошибках обработки.

---
## 💡 Особенности и подводные камни 💡

1️⃣ ❗❗❗Publisher обязан быть **потокобезопасным** при обработке `request()` и `cancel()`.

2️⃣ 💡Некоторые реализации (например, `SubmissionPublisher`) выполняют выдачу элементов асинхронно, используя очереди и ExecutorService.

3️⃣ Контракт Reactive Streams **запрещает бесконечную отправку данных без запроса** — это нарушает принцип backpressure.

---
## Best Practices 💎
✅ Всегда вызывай `request(n)` внутри `onSubscribe()`, чтобы запустить получение данных.

✅ Запрашивай данные **постепенно** (`request(1)` или `request(batchSize)`), если обработка тяжёлая.

✅ Если подписчику больше не нужны данные, обязательно вызывай `cancel()` — иначе могут остаться ресурсы (например, потоки).

✅ 💡Если используешь `request(Long.MAX_VALUE)` — это сигнал Publisher'у, что можно отправлять **все элементы без ограничений** (режим “unbounded”).

✅ 💡 Гарантируй потокобезопасность в реализации `request()` и `cancel()` (особенно в кастомных Publisher’ах).
