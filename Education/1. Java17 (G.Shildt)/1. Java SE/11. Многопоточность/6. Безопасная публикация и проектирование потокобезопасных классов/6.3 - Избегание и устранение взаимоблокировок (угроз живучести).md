# 10.1 📕 Взаимоблокировки (Deadlocks)
💡 **Ключевая идея:** Deadlock возникает, когда два или более потоков ждут друг друга, чтобы освободить заблокированные ресурсы, создавая цикличную зависимость. Программа замирает навсегда.

==JVM не оказывает помощи в разрешении ситуаций возникновения взаимоблокировок==. Когда набор Java потоков попадает в состояние взаимоблокировки, это конец игры - эти потоки мгновенно выходят из строя. В зависимости от того, что делают эти потоки, приложение может полностью зависнуть или может зависнуть определенная подсистема, или может пострадать производительность.

## **10.1.1 📙 Взаимоблокировки, вызванные порядком наложения блокировок (Lock-Ordering Deadlocks)**
💡 **Суть:** Самый **простой и распространенный тип deadlock.** 
Возникает, когда два потока пытаются ==захватить одни и те же блокировки, но в **разном порядке**.==

### ❌ Deadlock
```java
public class LeftRightDeadlock {
    private final Object left = new Object();
    private final Object right = new Object();

    public void leftRight() {
        synchronized (left) {       // Поток А захватывает left
            synchronized (right) {  // ...и ждет right
                doSomething();
            }
        }
    }

    public void rightLeft() {
        synchronized (right) {      // Поток Б захватывает right
            synchronized (left) {   // ...и ждет left
                doSomethingElse();
            }
        }
    }
}
```
> [!NOTE] **Решение:** ==Всегда захватывать блокировки в **строго определенном глобальном порядке**== (например, сначала всегда блокировать объект с меньшим `hashCode`).

## **10.1.2 ⭐⭐ Взаимоблокировки, вызванные динамическим порядком блокировок**
💡**Суть:** ==Порядок блокировок зависит от внешних входных данных== (например, аргументов метода), поэтому его сложнее контролировать.
### ❌ Deadlock
```java
public void transferMoney(Account fromAcct, Account toAcct, DollarAmount amount)
        throws InsufficientFundsException {
    synchronized (fromAcct) {
        synchronized (toAcct) {
            if (fromAcct.getBalance().compareTo(amount) < 0)
                throw new InsufficientFundsException();
            else {
                fromAcct.debit(amount);
                toAcct.credit(amount);
            }
        }
    }
}
```

### ✔Решение
**Индуцировать порядок захвата**: сравнить `System.identityHashCode(obj)` и захватывать локи в порядке hash — если хэши равны, использовать глобальный `tieLock` как «разрыватель» коллизий.

> [!NOTE] System.identityHashCode
> `System.identityHashCode`, который возвращает то же значение, которое будет возвращено методом `Object.hashCode`

```java
private static final Object tieLock = new Object();

int fromHash = System.identityHashCode(fromAcct);
int toHash = System.identityHashCode(toAcct);

if (fromHash < toHash) {
    synchronized (fromAcct) {
        synchronized (toAcct) { helper.transfer(); }
    }
} else if (fromHash > toHash) {
    synchronized (toAcct) {
        synchronized (fromAcct) { helper.transfer(); }
    }
} else {
    synchronized (tieLock) {
	    //Блокировка на двух акк нужна, т.к этот акк может задействован для перевода на другой счет
        synchronized (fromAcct) { 
            synchronized (toAcct) { helper.transfer(); }
        }
    }
}
```
> [!NOTE] Best practices
> Тогда, ВНЕ ЗАВИСИМОСТИ ОТ ПОРЯДКА АРГУМЕНТОВ, разные потоки будут синхронизироваться на том объекте, у которого меньший хеш

## 10.1.3 📙 Взаимоблокировки между взаимодействующими объектами (вызов чужого метода С УДЕРАЖНИЕМ блокировки)
💡 **Суть:** Deadlock может возникать, даже если вы явно не захватываете две блокировки в одном методе. 
Проблема появляется, когда вы вызываете **"чужой" синхронизированный метод** (method of an alien class), удерживая свою блокировку.


> [!NOTE] ❗❗❗Проблема `Synchronized` методов❗❗❗
> 
### ❌ Deadlock
```java
class Taxi {
    @GuardedBy("this")
    private Point location, destination;
    private final Dispatcher dispatcher;

    public Taxi(Dispatcher dispatcher) {
        this.dispatcher = dispatcher;
    }

    public synchronized Point getLocation() {
        return location;
    }

    public synchronized void setLocation(Point location) {
        this.location = location;
        if (location.equals(destination))
            dispatcher.notifyAvailable(this);
    }
}

class Dispatcher {
    @GuardedBy("this")
    private final Set<Taxi> taxis;
    @GuardedBy("this")
    private final Set<Taxi> availableTaxis;

    public Dispatcher() {
        taxis = new HashSet<Taxi>();
        availableTaxis = new HashSet<Taxi>();
    }

    public synchronized void notifyAvailable(Taxi taxi) {
        availableTaxis.add(taxi);
    }

    public synchronized Image getImage() {
        Image image = new Image();
        for (Taxi t : taxis)
            image.drawMarker(t.getLocation());
        return image;
    }
}
```

Почему тут дедлок?
1. Поток 1: Вызывает `taxi.setLocation(newLocation)`. Захватывает блокировку `Taxi`.
    
2. Внутри `setLocation` вызывается `dispatcher.notifyAvailable(this)`. Поток 1 теперь пытается захватить блокировку `Dispatcher`.
    
3. Поток 2: Вызывает `dispatcher.getImage()`. Захватывает блокировку `Dispatcher`.
    
4. Внутри `getImage()` для каждого такси вызывается `taxi.getLocation()`. Поток 2 теперь пытается захватить блокировку `Taxi`.
    
- **Результат:** Классический deadlock. Поток 1 ждет блокировку `Dispatcher` от Потока 2, а Поток 2 ждет блокировку `Taxi` от Потока 1.

✅==Решение в Open-calls==(открытые вызовы - вызов другого метода без удержания блокировки)
👇👇👇
## 10.1.4 📙 Открытые вызовы (Open Calls)
**Суть:** Это главное лекарство от проблем из предыдущего раздела. ==**Открытый вызов** — это вызов метода без удерживания какой-либо блокировки==.

**Решение для Taxi и Dispatcher:** Сузить синхронизированные блоки до минимума, чтобы ==вызывать "чужие" методы== (`dispatcher.notifyAvailable`) уже ==ПОСЛЕ ТОГО, как собственная блокировка отпущена.==
### ✅Решение
```java
//Class Taxis
public void setLocation(Point location) {
    boolean reachedDestination;
    
    synchronized (this) { // Синхронизированный блок ТОЛЬКО для локальных данных
        this.location = location;
        reachedDestination = location.equals(destination);
    } // БЛОКИРОВКА ОТПУЩЕНА!
    
    if (reachedDestination) {
        dispatcher.notifyAvailable(this); // ОТКРЫТЫЙ ВЫЗОВ!
    }
}

//Class Dispatcher
public Image getImage() { 
	Set copy; 
	
	synchronized (this) { 
		copy = new HashSet(taxis); 
	} 
	
	Image image = new Image(); 
	for (Taxi t : copy) 
		image.drawMarker(t.getLocation());
		 
	return image; 
}
```

> [!NOTE] ✅ Best practices
> Старайтесь в своей программе использовать открытые вызовы. 
> 
> Программы, которые полагаются на открытые вызовы, гораздо легче анализировать на отсутствие взаимоблокировок, чем те, которые допускают вызов чужих методов в процессе удержания блокировки.
> 
> Во многих случаях потеря атомарности вполне приемлема; нет никаких причин, по которым обновление местоположения такси и уведомление диспетчера о том, что оно готово получить новое место назначения, должны быть атомарными операциями.
> 
> В других случаях потеря атомарности заметна, но семантические изменения все еще приемлемы.
> 💡**Если атомарность критична:** применяются другие протоколы (флаги состояния + single-thread owner для «shutdown» и т.д.).

## 10.1.5 📙 Взаимоблокировки ресурсов (Resource deadlocks)
Точно так же, как потоки могут попасть в ситуацию взаимоблокировки, когда каждый из них ожидает блокировку, удерживаемую другим потоком, и не освобождает свою, ==они также могут попасть в ситуацию взаимоблокировки при ожидании ресурсов.==

> [!NOTE] ⭐ Используйте семафор!
> Пулы ресурсов, как правило, реализуются с использованием семафоров (см. раздел 5.5.3) для облегчения блокирования, когда пул пуст.
> 
> **Решение:** упорядочивать доступ к ресурсам, использовать таймауты, разделять пулы, ==проектировать так, чтобы не требовать множественных ресурсов одновременно, 
> ИЛИ 
> использовать семафоры/контроль «взятия набора ресурсов» (например, acquire оба resource в СОГЛАСОВАННОМ порядке==).

# 10.2 📕 Предотвращение и диагностика взаимоблокировок

> [!NOTE] ⭐**Общие рекомендации:**  ⭐
> 1. Если можно — **избегать получения более одной блокировки** одновременно.
> 2. Если нужно — **ввести глобальный порядок** захвата и придерживаться его.
> 3. Использовать открытые вызовы. 
> 4. Выполнить **аудит** кода: найти места с несколькими блокировками и проверить согласованность порядка.

## 10.2.1 Блокировки, ограниченные по времени (Timed locks)
💡 **Идея:** вместо бесконечного `synchronized`, применять `Lock.tryLock(timeout, unit)`. Если не удалось — отступить, освободить уже захваченные ресурсы, возможно сделать задержку + повтор (backoff) или лог/перезапуск.

```java
boolean success = false;
long timeout = 1; TimeUnit unit = TimeUnit.SECONDS;
while (!success) {
    if (fromLock.tryLock(timeout, unit)) {
        try {
            if (toLock.tryLock(timeout, unit)) {
                try {
                    // transfer
                    success = true;
                } finally {
                    toLock.unlock();
                }
            }
        } finally {
            if (!success) fromLock.unlock();
        }
    }
    if (!success) {
        // Засыпаем на рандомное время чтобы избежать блокировки
        Thread.sleep(ThreadLocalRandom.current().nextInt(1, 10));
    }
}
```
💡 **Ключевая идея:** таймаут даёт шанс восстановиться — можно записать причину, логировать, повторить, вместо того, чтобы ждать вечно.

## 10.2.2 Анализ взаимоблокировок с использованием дампов потоков
Несмотря на то, что предотвращение взаимоблокировок является, в основном, вашей проблемой, JVM может помочь идентифицировать их, когда они происходят, с помощью использования дампа потоков. 

==Дамп потока содержит трассировку стека для каждого запущенного потока==, аналогичную трассировке стека, сопровождающей исключение. 
Дампы потоков ==также включают в себя информацию о блокировании==, например о том, ==какие блокировки удерживаются== каждым потоком, в ==каких кадрах стека они были захвачены, и какую блокировку ожидает захватить== заблокированный поток

> [!NOTE] 🧠 Как происходит поиск блокировок под капотом?
> Перед созданием дампа потока, JVM просматривает граф ожидания (is-waiting-for graph) в поисках циклов, для определения наличия взаимоблокировок. Если среда JVM находит цикл, она включает информацию о взаимоблокировке, идентифицирующую, какие блокировки и потоки участвуют, и где в программе нарушен захват блокировки

> [!NOTE] 🧠 Ограничения с использованием дампов
> Если вы используете явные классы блокировок Lock, вместо встроенной блокировки, 
> - Среда Java 5.0 не поддерживает связывание информации класса Lock с дампом потока; явные блокировки вообще не отображаются в дампах потоков. 
> - Среда Java 6 включает поддержку дампа потока и обнаружение взаимоблокировки с явными блокировками класса Lock, но информация о том, где блокировки были захвачены, будет неизбежно менее точна, чем для встроенных блокировок. 
> 
> Встроенные блокировки связаны с кадром стека, в котором они были захвачены; явные блокировки Lock связаны только с захватившим их потоком.


> [!NOTE] 💡 **Как диагностировать:**
> - Unix: `kill -3 <pid>` или `jstack <pid>` — получить thread dump. IDE (IntelliJ, Eclipse) тоже умеют это делать. 
> - В дампе JVM может автоматически показать «Found one Java-level deadlock» и перечислить потоки/мониторы (листинг 10.7 — пример). 
> - В Java коде: `ThreadMXBean`:

```java
import java.lang.management.ManagementFactory;
import java.lang.management.ThreadInfo;
import java.lang.management.ThreadMXBean;

public class MyMain {
    public static void main(String[] args) {
    	ThreadMXBean tmx = ManagementFactory.getThreadMXBean();
    	long[] deadlocked = tmx.findDeadlockedThreads();
    	if (deadlocked != null) {
    	    ThreadInfo[] infos = tmx.getThreadInfo(deadlocked);
    	    // логировать infos
    	}
    }
}
```

# 10.3 📕 Прочие угрозы живучести
## 10.3.1 Голодание (Starvation)
Голодание возникает тогда, когда ==потоку ПОСТОЯННО ОТКАЗЫАЮТ в доступе к ресурсам, в которых он нуждается для достижения прогресса==; наиболее подверженный голоданию ресурс - циклы ЦП

Может быть вызвано:
- неправильным использованием приоритетов потоков;
    
- «жёсткими» циклами/вызовами без прерывания под удерживаемой блокировкой;
    
- unfair locks (нечестные реализации).

⭐⭐Советы:⭐⭐
- избегать ручного изменения приоритетов (это платформозависимо и вводит риск);
    
- для необходимости «справедливости» использовать fair варианты `ReentrantLock(true)` или семафоры с fair=true;
    
- проектировать так, чтобы долгие работы не удерживали важные блоки.

## 10.3.2 Плохая отзывчивость (Poor responsiveness)

**Что:** UI/Front-end «залипает» либо сервис медленно отвечает. 
Причины: долгие фоновые вычисления, долгие synchronized-блоки, блокировки на ресурсах.  

**Решение:**
- отделять UI/сервисный поток от тяжёлых задач (`Executor`, background threads);
    
- уменьшать критические секции;
    
- ☢✔при необходимости снизить приоритет фоновых потоков (как исключение);
    
- применять таймауты, chunk-processing (делить процесс на куски).

## 10.3.3 Динамическая взаимоблокировка
**Что:** потоки не блокируются, но всё время повторяют действия, не добиваясь прогресса (пример — poisoned message: транзакция откатывается, сообщение снова ставится в очередь, повтор и т.д.; пример «двое в коридоре отступают в одну и ту же сторону»).

**Решение:**
- 💡использовать **рандомизированные задержки / экспоненциальный backoff** перед повтором;
    
- 💡детектировать «отравленные» сообщения и отправлять их в dead-letter queue;
    
- сократить агрессивную политику восстановления, если ошибка неустранима.
