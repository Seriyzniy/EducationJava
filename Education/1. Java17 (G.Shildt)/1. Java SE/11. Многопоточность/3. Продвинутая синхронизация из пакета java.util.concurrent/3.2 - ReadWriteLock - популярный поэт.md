Во многих случаях, структуры данных предназначены “в основном для чтения” - они изменяемы и иногда модифицируются, но большинство обращений к ним происходит только для чтения данных. 

# 1. 📕 Интерфейс ReadWriteLock
`ReadWriteLock` — это ==интерфейс==, который позволяет повысить параллелизм для структур данных, которые **«чаще читаются, чем изменяются»**.
==Он разрешает множественный доступ для **читателей** НО ЭКСКЛЮЗИВЫНЙ доступ для **писателя**.==
```java
/**
 * Интерфейс ReadWriteLock предоставляет пару связанных блокировок:
 * одна для операций чтения и одна для операций записи.
 * 
 * ReadWriteLock позволяет иметь множественные читатели одновременно,
 * но только одного писателя в исключительном режиме.
 */
public interface ReadWriteLock { //🛑ИНТЕРФЕЙС🛑
    
    /**
     * Возвращает блокировку, используемую для операций чтения.
     * 
     * @return блокировка для чтения
     */
    Lock readLock();
    
    /**
     * Возвращает блокировку, используемую для операций записи.
     * 
     * @return блокировка для записи
     */
    Lock writeLock();
}
```

Как и класс ReentrantLock, класс ReentrantReadWriteLock может быть построен несправедливым (по умолчанию) или справедливым. 
- 🟢В случае справедливой блокировки предпочтение отдается потоку, который ожидал дольше; если блокировка удерживается читателями и поток запрашивает блокировку на запись, читателям не будет позволяться захватывать блокировку на чтение, пока писатель не будет обслужен и не освободит блокировку на запись. 
- 🔴В случае несправедливой блокировки, порядок в котором потокам предоставляется доступ, не определен. ==Понижение от писателя до читателя ДОПУСКАЕТЯ==; а повышение с читателя до писателя НЕТ (попытка выполнить это приведёт к взаимоблокировке).

> [!NOTE] Настройка справедливости
>**Fair vs non-fair:** конструктор `ReentrantReadWriteLock(boolean fair)` — если `fair=true`, порядок ожидания будет FIFO (но меньшая производительность). 
>
>Non-fair даёт более высокую пропускную способность, но риск starvation/барража. `tryLock()` может «bargе» даже при fair=true (см. ниже).

### Пример
```java
class DataHolder {
    private final ReentrantReadWriteLock rw = new ReentrantReadWriteLock();
    private final Lock r = rw.readLock();
    private final Lock w = rw.writeLock();
    private int value;

    public int read() {
        r.lock();
        try {
            return value;            // быстрый чтение — параллельно
        } finally {
            r.unlock();
        }
    }

    public void write(int v) {
        w.lock();
        try {
            value = v;              // эксклюзивное изменение
        } finally {
            w.unlock();
        }
    }
}
```

> [!NOTE] 🛑
> Всегда `try { } finally { unlock(); }` — иначе при исключении заблокируете ресурс

## Политики доступа:
- Если писатель активен, читатели блокируются
- Если читатели активны, писатель блокируется
    
- Писатель имеет приоритет над читателями

# 2. 📕 Реализация ReerantReadWriteLock
```java
import java.util.concurrent.TimeUnit;
import java.util.concurrent.locks.Condition;
import java.util.concurrent.locks.Lock;
import java.util.concurrent.locks.ReadWriteLock;

public class ReentrantReadWriteLock implements ReadWriteLock {
    
    private final ReadLock readLock;
    private final WriteLock writeLock;
    private final boolean fair;
    
    // Внутренние счетчики для отслеживания состояния блокировок
    private int readCount = 0;
    private int writeCount = 0;
    private Thread writingThread = null;
    
    //По умолчанию НЕсправедливая блокировка
    public ReentrantReadWriteLock() {
        this(false);
    }
    
    //С параметром true - справедливая блокировка
    public ReentrantReadWriteLock(boolean fair) {
        this.fair = fair;
        this.readLock = new ReadLock();
        this.writeLock = new WriteLock();
    }
```


> [!NOTE] ⭐ Особенности использования СПРАВЕДЛИВОЙ и НЕСПРАВЕДЛИВОЙ блокировки
> Важно: у методов `tryLock()` поведение «barge» — даже при fair=true `tryLock()` может успешно захватить замок, несмотря на очередь ждующих потоков. 
> 
> Таймед `tryLock(timeout, ...)` обычно учитывает очередь.

## Полезные методы самого `ReentrantReadWriteLock`:
- `getReadLockCount()` — общее число прочитанных захватов;
- `getReadHoldCount()` — число read-холдов текущего потока;
- `getWriteHoldCount()` — число write-холдов текущего потока;
    
- `isWriteLocked()`, `isWriteLockedByCurrentThread()`, `isFair()` и др.

# 3. 📕 Downgrade (понижение писателя в читателя) — паттерн и пример

**Когда нужен:** вы сделали изменение (write) и хотите дальше безопасно читать без потери  других читателей — то есть перейти с эксклюзивного режима в совместный, не выпуская «пробел» без блокировки.

**Правильная последовательность:**
1. Захватить `writeLock`.
2. Работаем с данными
3. Захватить `readLock` **пока всё ещё держите** `writeLock`.
4. Освободить `writeLock`.
    
5. Теперь вы держите только `readLock` — downgrade выполнен.

```java
w.lock();
try {
    // Изменяем данные
    // ...
    r.lock();                // Захватываем блокировку читалея УДЕРЖИВАЯ писателя
} finally {
    w.unlock();             // Освобождаем блокировку писателя УДЕРЖИВАЯ ЧИТАТЕЛЯ
}

try {
    // do read-only work while holding read lock
} finally {
    r.unlock();
}
```

# 4. 📕 Upgrade (read → write) — почему опасно и как правильно делать

**Опасность:** если поток держит `readLock` и пытается получить `writeLock()`, он будет ждать, пока **все** другие читатели не освободят read; но другие читатели могут ждать на то, чтобы получить read (или быть запущены), и получится deadlock/зависание — типичный случай.

1. ❌ Нельзя получить write lock, уже имея read lock
2. ✅ Правильный подход: освободить read lock → получить write lock
## ❌ Плохой (deadlock) пример — НЕ делайте так:
```java
r.lock();
try {
    // проверяем условие
    if (needToWrite()) {
        w.lock();  // может блокировать навсегда — НЕБЕЗОПАСНО
        try { ... } finally { w.unlock(); }
    }
} finally {
    r.unlock();
}
```

## Два безопасных подхода:
1. **Отпустить read, затем попытаться захватить write (с повторной проверкой):**
```java
// 🛑Сначала читаем под read lock
lock.readLock().lock();
String currentData;
try {
	currentData = data;
	System.out.println(Thread.currentThread().getName() + ": Прочитал: " + currentData);
} finally {
	lock.readLock().unlock();
}

// 🟢Затем получаем write lock для модификации (пытаемся "вклиниься" в очередь)
lock.writeLock().lock();
try {
	System.out.println(Thread.currentThread().getName() + ": Получил write lock");
	data = "modified: " + currentData + " by " + Thread.currentThread().getName();
	System.out.println(Thread.currentThread().getName() + ": Записал: " + data);
} finally {
	lock.writeLock().unlock();
}
```

2. Использовать tryLock с таймаутом (попытка апгрейда с откатом и backoff):
```java
r.lock();
try {
    if (!needToWrite()) return;
    // release read and try to acquire write with timeout
} finally {
    r.unlock();
}

if (w.tryLock(1, TimeUnit.SECONDS)) {
    try {
        if (needToWrite()) { ... }
    } finally {
        w.unlock();
    }
} else {
    // не удалось — повторить/логировать/отказать
}
```

# 5. 📕⭐ Conditions и ожидание/сигнал (Condition)

`writeLock.newCondition()` — условие доступно **только** от write lock; 
`readLock` не предоставляет `newCondition()` (попытка — `UnsupportedOperationException`). 

Это значит: ==если ЧИТАТЕЛЬ должен ждать события через `Condition`, ему нужно РАБОТАТЬ С `writelock` (что тяжело)==

```java
import java.util.concurrent.locks.Condition;
import java.util.concurrent.locks.ReentrantReadWriteLock;

public class ProducerConsumerWithCondition {
    private final ReentrantReadWriteLock lock = new ReentrantReadWriteLock();
    private final Condition notEmpty = lock.writeLock().newCondition();
    private final Condition notFull = lock.writeLock().newCondition();
    
    private final String[] buffer;
    private int count = 0;
    private int putIndex = 0;
    private int takeIndex = 0;
    
    public ProducerConsumerWithCondition(int capacity) {
        this.buffer = new String[capacity];
    }
    
    /**
     * Производитель - добавляет данные в буфер
     */
    public void produce(String item) throws InterruptedException {
        lock.writeLock().lock(); // Получаем write lock
        try {
            // Ждем, пока буфер не освободится
            while (count == buffer.length) {
                System.out.println("Буфер полон, производитель ждет...");
                // Ожидаем сигнала о наличии свободного места 
                // временно освобождая блокировку
                notFull.await(); 
            }
            
            // Добавляем элемент в буфер
            buffer[putIndex] = item;
            putIndex = (putIndex + 1) % buffer.length;
            count++;
            
            System.out.println("Произведено: " + item + " (размер: " + count + ")");
            
            // Сигнализируем потребителям, что буфер не пуст
            notEmpty.signalAll();
            
        } finally {
            lock.writeLock().unlock();
        }
    }
    
    /**
     * Потребитель - забирает данные из буфера
     */
    public String consume() throws InterruptedException {
        lock.writeLock().lock(); // Получаем write lock
        try {
            // Ждем, пока в буфере появятся данные
            while (count == 0) {
                System.out.println("Буфер пуст, потребитель ждет...");
                // // Ожидаем сигнала о наличии данных
                // временно освобождая блокировку
                notEmpty.await(); 
            }
            
            // Забираем элемент из буфера
            String item = buffer[takeIndex];
            
            //🛑Изменяем массив🛑 
            //🛑Поэтому потребитель захватывает writeLock🛑
            buffer[takeIndex] = null; 
            takeIndex = (takeIndex + 1) % buffer.length;
            count--;
            
            System.out.println("Потреблено: " + item + " (размер: " + count + ")");
            
            // Сигнализируем производителям, что появилось свободное место
            notFull.signalAll();
            
            return item;
            
        } finally {
            lock.writeLock().unlock();
        }
    }
    
    /**
     * Получение текущего размера буфера (с использованием read lock)
     */
    public int getSize() {
        lock.readLock().lock();
        try {
            return count;
        } finally {
            lock.readLock().unlock();
        }
    }
    
    /**
     * Очистка буфера (с использованием write lock)
     */
    public void clear() {
        lock.writeLock().lock();
        try {
            for (int i = 0; i < buffer.length; i++) {
                buffer[i] = null;
            }
            count = 0;
            putIndex = 0;
            takeIndex = 0;
            notFull.signalAll(); // Уведомляем ожидающих производителей
        } finally {
            lock.writeLock().unlock();
        }
    }
}
```
