`StampedLock` это относительно ==новая альтернатива== `ReentrantReadWriteLock` (появился в **Java 8**)

> [!NOTE] ВАЖНО
> Используется `stamp` - счетчик, который меняется при его захвате.
> 1. При `readLock` он не меняется
> 2. При `writeLock` - меняется
> 3. При `tryOptimisticRead` не меняется, но после этого необходимо проверить что `stamp` не изменился при помощи метода `validate`


## 📙 Что такое `StampedLock`
`StampedLock` — ==это **замок с метками (stamp)**==, позволяющий работать в трёх режимах:

### 🔶 1. Write Lock (эксклюзивная блокировка)
- Аналогичен `ReentrantReadWriteLock.writeLock()`
- 🛑🛑🛑Только ==ОДИН ПОТОК может получить блокировку на запись==
- Возвращает штамп (stamp) для разблокировки

Если один поток вызвал `writeLock()`, то:
- он получает **новый уникальный stamp** (долгое `long` число);
- остальные потоки, которые вызовут `writeLock()`, будут **ждать** (блокироваться);
- потоки, которые вызовут `readLock()`, тоже будут ждать, пока writer не освободит замок.
### 🔶 2. Read Lock (разделяемая блокировка)
- Аналогичен `ReentrantReadWriteLock.readLock()`
- 🛑🛑🛑==Несколько потоков могут читать одновременно==
- Блокирует запись
    
### 🔶 3. Optimistic Read (оптимистичное чтение)
- ⭐⭐⭐==Уникальная особенность StampedLock==⭐⭐⭐
- Не блокирует другие потоки
- Требует проверки валидности после чтения

Этот метод **не блокирует вообще** — он ==просто возвращает stamp== (например, 300);
Поток читает данные как есть;

После чтения поток проверяет `validate(stamp)`:
- если в это время никто не делал `writeLock()`, то stamp всё ещё валиден → данные корректны;
- если был writer → stamp недействителен → нужно повторить чтение с `readLock()`.
***
## 📙 Как работает
- Каждый вызов `lockXXX()` или `tryOptimisticRead()` возвращает **долгоживущую метку (`long stamp`)**, которая нужна для `unlock(stamp)` или `validate(stamp)`.
    
- Метка гарантирует, что **именно тот lock/unlock** будут связаны.
    
- Это отличается от `ReentrantLock`/`ReadWriteLock`, где разблокировка идёт по текущему потоку

### Основные методы
```java
// Блокировки записи
long writeLock() // Блокирующая запись
long tryWriteLock() // Неблокирующая попытка
long tryWriteLock(long time, TimeUnit unit) // С таймаутом

// Блокировки чтения  
long readLock() // Блокирующее чтение
long tryReadLock() // Неблокирующая попытка
long tryReadLock(long time, TimeUnit unit) // С таймаутом

// Оптимистичное чтение
long tryOptimisticRead() // Получить оптимистичный штамп

// Разблокировка
void unlock(long stamp) // Разблокировать
boolean validate(long stamp) // Проверить валидность штампа

// Конвертация
long tryConvertToWriteLock(long stamp) // Попробовать конвертировать
long tryConvertToReadLock(long stamp) // Попробовать конвертировать
```

## Пример
```css
T1: readLock() → stamp = 200
T2: readLock() → stamp = 200   ← получает ТОТ ЖЕ stamp

T3: writeLock() → блокируется, пока все читатели (T1, T2) не вызовут unlockRead()
После освобождения → stamp меняется (200 → 201)
```

```java
import java.util.concurrent.locks.StampedLock;

public class Point {
    private double x, y;
    private final StampedLock lock = new StampedLock();

    // Изменение координат (write lock)
    void move(double deltaX, double deltaY) {
        long stamp = lock.writeLock(); // блокировка записи
        try {
            x += deltaX;
            y += deltaY;
        } finally {
            lock.unlockWrite(stamp); // разблокировка по stamp
        }
    }

    // Оптимистичное чтение
    double distanceFromOrigin() {
        long stamp = lock.tryOptimisticRead(); // не блокируем
        double currentX = x, currentY = y;
        if (!lock.validate(stamp)) { // проверка: не было ли write
            stamp = lock.readLock(); // fallback на обычный read lock
            try {
                currentX = x;
                currentY = y;
            } finally {
                lock.unlockRead(stamp);
            }
        }
        return Math.sqrt(currentX * currentX + currentY * currentY);
    }

    // Обычное чтение
    double[] get() {
        long stamp = lock.readLock();
        try {
            return new double[]{x, y};
        } finally {
            lock.unlockRead(stamp);
        }
    }
}
```

## Особенности StampedLock
1. 🔶**Не является reentrant**  
В отличие от `ReentrantLock`, один и тот же поток не может захватить замок повторно — будет дедлок.

2. 🔶**Нужно хранить и передавать `stamp`**  
Забыл вызвать `unlockXXX(stamp)` с правильной меткой → замок навсегда зависнет.
    
3. 🔶**Оптимистичное чтение = риск ложного чтения**  
Если параллельно идёт `write`, `tryOptimisticRead()` может вернуть «грязные» данные.  
Поэтому **обязательно** проверять `validate(stamp)`.

4. 🔶**Сложнее отлаживать**  
Так как нет привязки к потоку (unlock может вызвать другой поток), ошибки легко допустить.

5. 🔶Почему `readLock()` блокирует `writeLock()`?
Если поток держит `readLock`, то `writeLock` **не может быть получен**, потому ==что writer должен быть **единственным владельцем данных**==, а значит, никто не может читать в этот момент.

📌 То есть `readLock` не мешает другим читателям, но **мешает writer'ам**.
