
> [!NOTE] Интерфейс **`BlockingQueue<E>`** — это **расширение `Queue<E>`**, предназначенное `для потокобезопасного обмена` `элементами` между потоками.
> ❗❗❗Он используется в сценариях **Producer–Consumer**, где один поток производит данные, а другой их потребляет.
> 
> ⭐Главная особенность — `методы` `могут БЛОКИРОВАТЬСЯ`, ожидая, пока очередь не освободится (для вставки) или не появится элемент (для извлечения).
> 

## Интерфейс
```java
public interface BlockingQueue<E> extends Queue<E> {
    /**
     * Добавляет элемент в очередь. Если очередь заполнена, бросает исключение.
     */
    boolean add(E e);


    /**
     * Добавляет элемент в очередь. При заполнении возвращает false вместо исключения.
     */
    boolean offer(E e);
    /**
     * Добавляет элемент в очередь, ожидая не более указанного времени.
     */
    boolean offer(E e, long timeout, TimeUnit unit) throws InterruptedException;
	/**
     * Извлекает и удаляет элемент из головы очереди, ожидая не более указанного времени.
     */
    E poll(long timeout, TimeUnit unit) throws InterruptedException;


    /**
     * Добавляет элемент в очередь, при необходимости ожидая освобождения места.
     */
    void put(E e) throws InterruptedException;
    /**
     * Извлекает и удаляет элемент из головы очереди, ожидая при необходимости.
     */
    E take() throws InterruptedException;


    /**
     * Возвращает оставшуюся вместимость очереди.
     */
    int remainingCapacity();



    /**
     * Удаляет указанный элемент из очереди, если он присутствует.
     */
    boolean remove(Object o);



    /**
     * Проверяет, содержит ли очередь указанный элемент.
     */
    boolean contains(Object o);

    /**
     * Перемещает все элементы из этой очереди в указанную коллекцию.
     */
    int drainTo(Collection<? super E> c);

    /**
     * Перемещает не более указанного количества элементов в коллекцию.
     */
    int drainTo(Collection<? super E> c, int maxElements);
}
```

## 💡 Особенности и подводные камни 💡

1️⃣ ❗❗❗Методы `put()` и `take()` **==блокируют поток==**, пока операция не может быть выполнена.  

2️⃣ ❗❗❗Методы `offer()` и `poll()` **==НЕ блокируют==**, если не указано время ожидания. 

3️⃣ `BlockingQueue` **не поддерживает `null`** — вставка `null` вызывает `NullPointerException`.  

4️⃣ `BlockingQueue` **не допускает** операции, нарушающие порядок, такие как `addFirst()` или `removeLast()` — это **не `Deque`**.  

5️⃣ Не допускается **прямая синхронизация** (`synchronized`) при работе с `BlockingQueue` — она уже потокобезопасна.  

6️⃣ Очереди могут быть **ограниченными** (с заданной ёмкостью) и **неограниченными** (например, `LinkedBlockingQueue()` без параметра). 

7️⃣ ⭐Метод `drainTo()` эффективнее, чем многократное извлечение через `take()` — блокировка захватывается один раз.

---

## 💎 Best Practices 💎

✅ Используйте `put()` и `take()` в сценариях **Producer–Consumer**, где важно дождаться места или элемента.  

✅ Применяйте `offer(e, timeout, unit)` и `poll(timeout, unit)` при необходимости **контролировать время ожидания**.  

✅ Используйте **ограниченные очереди** (`new ArrayBlockingQueue<>(capacity)`) для предотвращения переполнения памяти.  

✅ Не используйте `null` — для обозначения "отсутствия элемента" используйте `poll()`, возвращающий `null`.  

✅ Для массовой выгрузки данных из очереди применяйте `drainTo()` вместо циклов `while(!queue.isEmpty()) take()`.
