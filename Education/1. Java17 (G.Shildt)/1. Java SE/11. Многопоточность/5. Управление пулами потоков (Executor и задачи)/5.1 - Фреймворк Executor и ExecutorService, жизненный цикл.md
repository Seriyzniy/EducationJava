```java
public static void main(String[] args) throws IOException { 
	ServerSocket socket = new ServerSocket(80);
	//Бесконечное создание потоков в цикле🛑
	while (true) { 
		final Socket connection = socket.accept(); 
		Runnable task = new Runnable() { 
			public void run() { 
				handleRequest(connection); 
			} 
		}; 
		new Thread(task).start(); 
	} 
} 
```
Использование в продукте подхода “один поток на одну задачу” имеет некоторые практические недостатки, особенно при создании огромного числа потоков:

- 🔶Накладные расходы на поддержание жизненного цикла. 
==Создание потока и его завершение не даются даром==. Фактические накладные расходы варьируются в зависимости от платформы, но создание потока занимает время, добавляю задержку ко времени обработки запроса, и требует некоторой обработки активностями JVM и ОС.
	
- 🔶Потребление ресурсов. 
Активные потоки потребляют системные ресурсы, особенно память. Если количество запускаемых потоков превышает количество доступных процессоров, потоки бездействуют.
	
- 🔶Масштабируемость. 
Существует ограничение на количество создаваемых потоков. Ограничение зависит от платформы и зависит от таких факторов, как параметры вызова JVM, запрошенный размер стека в конструкторе потоков и ограничения на потоки, накладываемые базовой операционной системой. При достижении этого предела, наиболее вероятным результатом является возбуждение исключения `OutOfMemoryError.`

# 📕 1. Executor 
Пулы потоков предлагают методы, для управления потоками, пакет `java.util.concurrent` обеспечивает гибкую реализацию пула потоков в рамках фреймворка Executor. 

==Основной абстракцией== для выполнения задач в библиотеках классов Java является не поток, а ==интерфейс Executor==

Фреймворк **Executor** в Java — это гибкий способ **разделить задачу и поток**, в котором она будет выполнена. Вместо того чтобы создавать и запускать `Thread` вручную, ты делегируешь выполнение **объекту-исполнителю (Executor)**.

## Методы
```java
public interface Executor {
    void execute(Runnable command);
}
```
`execute(Runnable command)` — принимает задачу (`Runnable`) и выполняет её в каком-то потоке (например, в пуле потоков).

Пример:
```java
Executor MyExecutor = Executors.newSingleThreadExecutor();

MyExecutor.execute(() -> {
    System.out.println("Hello from " + Thread.currentThread().getName());
});
```
📌 Здесь `Executors.newSingleThreadExecutor()` вернёт `ExecutorService`, который реализует `Executor`.

🛑🛑🛑Это абстракция над запуском задачи (`Runnable`). 🛑🛑🛑
==Основная идея==: **тебе не нужно знать, как именно запускается поток — ты просто передаёшь задачу**

==РЕАЛИЗАЦИЯ интерфейса Executor основана на шаблоне производитель-потребитель==, где активности, которые предоставляют задачи, являются производителями (производят единицы работы, которые должны быть выполнены), а выполняющие задачи потоки, являются потребителями (потребители этих единиц работы). Использование интерфейса Executor обычно является самым простым способом реализации архитектуры производитель-потребитель в вашем приложении.

==Пул потоков==, как следует из названия,==управляет однородным пулом рабочих потоков==. Пул потоков тесно связан с рабочей очередью, содержащей задачи, ожидающие выполнения. Рабочие потоки живут достаточно просто: запрашивают следующую задачу из рабочей очереди, выполняют ее и возвращаются к ожиданию других задач.

# 📕 2. ExecutorS
Библиотека классов предоставляет гибкую реализацию пула потоков, наряду с некоторыми полезными предопределенными конфигурациями. 

==Пул потоков можно создать==, вызвав один из статических ==фабричных методов класса== `Executors`:

- 🔶 `newFixedThreadPool`. **Пул потоков фиксированного размера, создает потоки по мере отправки задач, вплоть до максимального размера пула**, а затем старается сохранять размер пула неизменным (добавляя новые потоки, если поток умирает из-за возникновения неожиданного исключения Exception). 
```java
public class FixedPoolDemo {
    public static void main(String[] args) throws InterruptedException {
        ExecutorService pool = Executors.newFixedThreadPool(2);

        for (int i = 1; i <= 5; i++) {
            int taskId = i;
            pool.submit(() -> {
                System.out.println("Task " + taskId + " executed by " + Thread.currentThread().getName());
                try { Thread.sleep(1000); } catch (InterruptedException ignored) {}
            });
        }

        pool.shutdown();
    }
}
```

- 🔶 `newCachedThreadPool`. создаёт пул потоков с **динамическим регулированием количества потоков для выполнения задач**. Метод создаёт новые потоки по мере необходимости и использует ранее созданные потоки повторно, когда они доступны.

> [!NOTE] 📖 Создаёт пул с **НЕОГРАНИЧЕННЫМ количеством потоков**. Если поток простаивает > 60 сек — он уничтожается.
> 
```java
public class CachedPoolDemo {
    public static void main(String[] args) {
        ExecutorService pool = Executors.newCachedThreadPool();

        for (int i = 1; i <= 5; i++) {
            int taskId = i;
            pool.submit(() -> {
                System.out.println("Task " + taskId +
                                   " executed by " + Thread.currentThread().getName());
            });
        }

        pool.shutdown();
    }
}
```

- 🔶 `newSingleThreadExecutor`. **Однопоточная реализация** интерфейса Executor **создает единственный рабочий поток для обработки задач, заменяя его, если он неожиданно умирает.** Гарантируется, что задачи будут обрабатываться последовательно в соответствии с порядком, определяемым очередью задач (FIFO, LIFO, в порядке приоритета).
```java
public class SingleThreadDemo {
    public static void main(String[] args) {
        ExecutorService executor = Executors.newSingleThreadExecutor();

        for (int i = 1; i <= 3; i++) {
            int taskId = i;
            executor.submit(() ->
                System.out.println("Task " + taskId +
                                   " executed by " + Thread.currentThread().getName()));
        }

        executor.shutdown();
    }
}
```


- 🔶 `newScheduledThreadPool`. **Пул потоков фиксированного размера, поддерживающий отложенное и периодичное выполнение задач**, подобно классу Timer

> [!NOTE] 📖 Пул, где задачи могут выполняться **с задержкой или периодически**.
> `newScheduledThreadPool` → аккуратно с `scheduleAtFixedRate`: если ЗАДАЧА выполняется ДОЛЬШЕ ПЕРИОДА, она МОЖЕТ НАКЛЫДЫВАТЬСЯ.
```java
public class ScheduledDemo {
    public static void main(String[] args) {
        ScheduledExecutorService scheduler = Executors.newScheduledThreadPool(2);

        Runnable task = () -> System.out.println("Running at " + System.currentTimeMillis());

        // Запуск через 2 секунды (ОТЛОЖЕННЫЙ ЗАПУСК)
        scheduler.schedule(task, 2, TimeUnit.SECONDS);

        // Запуск каждые 3 секунды (начиная через 1 сек) (ПЕРЕОДИЧЕСКИЙ ЗАПУСК)
        scheduler.scheduleAtFixedRate(task, 1, 3, TimeUnit.SECONDS);

        // scheduler.shutdown(); // останавливать вручную
    }
}
```

> [!NOTE] 🛑Возвращаемые значения методов `ExecutorS` 🛑
> Фабричные ==**методы newFixedThreadPool, newSingleThreadExecutor и newCachedThreadPool**== возвращают экземпляры класса общего назначения - ==ThreadPoolExecutor==, которые могут непосредственно использоваться для создания более специализированных экземпляров интерфейса Executor

- ⭐ `unconfigurableExecutorService(ExecutorService executor)`. Оборачивает пул так, чтобы нельзя было менять его настройки.
```java
public class UnconfigurableDemo {
    public static void main(String[] args) {
        ExecutorService realPool = Executors.newFixedThreadPool(2);
        ExecutorService safePool = Executors.unconfigurableExecutorService(realPool);

        safePool.submit(() -> System.out.println("Hello from safe pool"));
        safePool.shutdown();
    }
}
```
👉 Используется, чтобы отдать пул наружу, но **скрыть возможности кастомизации**.

# 💡 Жизненный цикл экземпляра Executor 💡
Реализация Executor, вероятнее всего, создаст потоки для обработки задач. 
🛑Но JVM НЕ СМОЖЕТ завершить свою работу до тех пор, пока все потоки (не демоны) не завершаться🛑, поэтому неудача с закрытием экземпляра Executor может помешать завершению работы JVM.

Поскольку экземпляр Executor обрабатывает задачи асинхронно, то есть в любой момент времени, состояние ранее переданных на обработку задач не сразу становится очевидным. 
Некоторые из них могут быть завершены, некоторые могут быть запущены в текущий момент, а другие могут быть поставлены в очередь ожидания выполнения. 

Жизненный цикл, подразумеваемый интерфейсом ExecutorService, включает в себя три состояния: 
1. запущено (running), 
2. завершение работы (shutting down)
3. завершено (terminated). 
==Экземпляры интерфейса ExecutorService изначально создаются в состоянии запущено==


> [!NOTE] 🛑Жизненный цикл ЗАДАЧИ 🛑
> Жизненный цикл ЗАДАЧИ, выполняемой экземпляром Executor, состоит из ==четырех фаз==:  
> 1. создано (created), 
> 2. отправлено (submitted), 
> 3. запущено (started)
> 4. завершено (completed).


Существует целый спектр вариантов завершения работы приложения - от ==мягкого выключения== (завершается то, что было ранее начато, но новая работа не принимается) до ==внезапного выключения== (выключается питание в комнате с компьютером), а также различные варианты между ними.

# 📕 3. ExecutorService

> [!NOTE] 🛑ЭКЗЕМПЛЯРЫ интерфейса ExecutorService изначально СОЗДАЮТСЯ в состоянии ЗАПУЩЕНО (running).
> 

Рассматривая вопросы жизненного цикла службы выполнения отметим, что ==интерфейс ExecutorService== расширяет интерфейс Executor,**добавляя ряд методов для управления жизненным циклом** (а также некоторые удобные методы для отправки задач).

```java
public interface ExecutorService extends Executor { 
	<T> Future<T> submit(Callable<T> task); 
	<T> Future<T> submit(Runnable task, T result);
		Future<?> submit(Runnable task);

	void shutdown(); 
	List<Runnable> shutdownNow(); 
	
	boolean isShutdown(); 
	boolean isTerminated(); 
	boolean awaitTermination(long timeout, TimeUnit unit)  throws InterruptedException; 
} 
```
***
## ⭐⭐⭐РАЗНИЦА между `execute` и `submit`⭐⭐⭐
### 1. Возвращаемое значение
execute():
```java
void execute(Runnable command);
```
- **Не возвращает результат** (void)
- Не может быть использован для получения статуса выполнения задачи

submit():
```java
Future<?> submit(Runnable task);
<T> Future<T> submit(Callable<T> task);
<T> Future<T> submit(Runnable task, T result);
```
- **Возвращает `Future` объект**
- Через `Future` можно отслеживать статус выполнения, получать результаты, отменять задачи

### 2. Тип принимаемых задач
execute():
- Только `Runnable`

submit():
- `Runnable`
- `Callable` (может возвращать результат и бросать проверяемые исключения)

### 3. Обработка исключений
execute():
```java
executor.execute(() -> {
    throw new RuntimeException("Ошибка!");
});
```
- Исключения **теряются** (не видны вызывающему коду)
- Можно обрабатывать через `UncaughtExceptionHandler`

submit():
```java
Future<?> future = executor.submit(() -> {
    throw new RuntimeException("Ошибка!");
});

future.get(); // Бросит ExecutionException с причиной исходного исключения
```
- Исключения **оборачиваются** в `ExecutionException`
- Доступны при вызове `future.get()

***
## Основные методы:
1. 🔶**Выполнение задач**
-  `Future<?> submit(Runnable task, T result)` → возвращает `Future<?>` (можно проверить статус).
    
- `Future<T> submit(Callable<T> task)` → возвращает `Future<T>` (результат выполнения).
	♻Возвращает: `Future<T>` - содержит результат выполнения
    
- `List<Future<T>> invokeAll(Collection<? extends Callable<T>> tasks)` → запускает список задач, ждёт их выполнения, возвращает список `Future`.💡Также есть вариант с установкой Timeout
	♻Возвращает: `List<Future<T>>` - список Future для всех задач
    
- `T invokeAny(Collection<? extends Callable<T>> tasks)` → запускает задачи и 
	♻Возвращает результат первой успешно завершённой.
    

1. 🔶**Управление жизненным циклом**
- `void shutdown()` → мягкая остановка (ждёт завершения всех задач, новые не принимаются).
    
- `List<Runnable> shutdownNow()` → жёсткая остановка (пытается прервать все задачи, возвращает список невыполненных). 
	♻Возвращает: `List<Runnable>` - список задач, которые не начали выполняться
	🛑 ==НО не возвращает список задач, которые начали выполняться и не завершились==
    
- `boolean isShutdown()` → true, если shutdown уже вызван.
	♻Возвращает: boolean - true если shutdown вызван
    
- `boolean isTerminated()` → true, если все задачи завершены.
    

1. 🔶**Дополнительно**
- `boolean awaitTermination(long timeout, TimeUnit unit)` → блокируется, пока все задачи не завершатся или не истечёт время.
	♻Возвращает: boolean - true если executor завершил работу


> [!NOTE] Методы `shutdown()` и `shutdownNow()`
> - Метод shutdown инициирует МЯГКОЕ ЗАВЕРШЕНИЕ работы: **новые задачи не принимаются к обработке**, НО ==ранее переданные завершаются==, включая те, которые, что еще не принимались на выполнение. 
> 
> - Метод shutdownNow производит ВНЕЗАПНОЕ ЗАВЕРЩЕНИЕ работы: он пытается ==отменить выполняющиеся задачи и не запускает какие-либо задачи==, которые находятся в очереди, но не запущены.

🛑Жизненный цикл, подразумеваемый интерфейсом ExecutorService, ==включает в себя три состояния==: запущено (running), завершение работы (shutting down) и завершено (terminated). 

Пример:
```java
public class ExecutorServiceExample {
    public static Runnable runnableTask;
	public static Callable<Integer> callableTask;
	
	static {
		runnableTask = new Runnable() {
			public void run() {
				System.out.println("It`s a Runnable task");
			}
		};
		
		callableTask = new Callable<Integer>() {
			public Integer call() throws Exception {
				System.out.println("It`s a Callable task");
				Thread.sleep(3000);;
				return 100;
			}
		};
	}
	
	public static void main(String[] args) throws Exception {
        ExecutorService pool = Executors.newFixedThreadPool(2);
        //
        // Runnable (без возврата результата)
        //
        Future<?> runnableFuture = pool.submit(runnableTask);
        
        // Вызов get() у Runnable вернёт null
        System.out.println("Runnable result = " + runnableFuture.get()); // null


        //
        // Callable (с возвратом результата)
        //
        Future<Integer> callableFuture  = pool.submit(callableTask);

        // get() блокируется, пока результат не будет готов
        Integer result = callableFuture.get();
        System.out.println("Future2 result = " + result); // 100

        pool.shutdown(); // мягкая остановка
    }
}

Вывод:
It`s a Runnable task
Runnable result = null

It`s a Callable task
Future2 result = 100 //Через 3 секунды
```
## Обработчик задач, после завершения ExecutorService
Задачи, передаваемые экземпляру ExecutorService после того, как его работа была завершена, обрабатываются обработчиком отклонённых задач (rejected execution handler), который может тихо избавляться от задач или может бросать непроверяемое исключение `RejectedExecutionException`.


> [!NOTE] ✅Завершение работы ExecutorService
> Можно дождаться, пока служба ExecutorService достигнет завершенного состояния с помощью метода `awaitTermination()`, или опрашивать, завершилась ли она с помощью метода `isTerminated()`. 
> 
> Как правило, следом за вызовом метода `shutdown()` следует вызов метода `awaitTermination()`, тем самым создавая эффект синхронного завершения работы экземпляра ExecutorService

🛑🛑🛑==Все методы submit интерфейса ExecutorService возвращают экземпляры Future==, так что вы можете отправить экземпляры Runnable или Callable исполнителю и получить назад экземпляр Future, который может быть использован для получения результата или отмены задачи. 

Можно также явно создать экземпляр FutureTask для переданного экземпляра Runnable или Callable (поскольку класс FutureTask реализует интерфейс Runnable, он может быть передан экземпляру Executor для выполнения или непосредственно выполнен, путем вызова метода run).

Пример:
```java
public class FixedThreadPoolExample {
    public static void main(String[] args) {
        ExecutorService pool = Executors.newFixedThreadPool(3);

        for (int i = 0; i < 10; i++) {
            int taskId = i;
            pool.submit(() -> {
                System.out.println("Task " + taskId + " running in " + Thread.currentThread().getName());
            });
        }

        pool.shutdown(); // корректное завершение
    }
}
```

# ☢Подводные камни Executor Framework ☢

### ❗1. `OutOfMemoryError` при `unbounded` очереди (`LinkedBlockingQueue`)
- 🛑Если ты используешь `newCachedThreadPool()` или `newFixedThreadPool()` с **безразмерной очередью**, задачи накапливаются бесконтрольно.
    
- **Решение**: используй `ThreadPoolExecutor` с ограниченной очередью.

### ❗2. Забвение `shutdown()`
- 🛑Без `shutdown()` или `shutdownNow()` приложение может **никогда не завершиться**, потому что потоки остаются активны.
    
- **Решение**: всегда завершай `ExecutorService`.

### ❗3. `RejectedExecutionException`
- 🛑Если очередь полная и потоки все заняты, новые задачи могут быть отклонены.
    
- **Решение**: настрой `RejectedExecutionHandler`.
```java
executor.setRejectedExecutionHandler(
	new ThreadPoolExecutor.CallerRunsPolicy()
);
```

### ❗4. Ошибки в `Runnable` или `Callable` могут "пропасть"
- 🛑🛑Если `Runnable` выбросит исключение — оно **не попадёт в лог**, если ты не обернёшь задачу вручную.🛑🛑
    
- **Решение**: использовать `Callable` + `Future` + обработку `ExecutionException`.

### ❗5. Потеря управления при использовании `Executors.*`
- Методы `Executors.newXXX()` удобны, но **прячут реализацию** `ThreadPoolExecutor`.
- Ты не контролируешь ни очередь, ни обработку ошибок, ни политику отклонения задач.
    
- **Решение**: используй `ThreadPoolExecutor` напрямую.
