# 1. 📕 Неявные связи между задачами и политиками выполнения

Хотя Executor отделяет отправку задач от их выполнения, это не всегда идеально.

Выделяются четыре типа задач, требующих особого подхода:
1. **Зависимые задачи** — могут привести к взаимоблокировкам, если пул слишком мал.

2. **Задачи, использующие ограничение потока** — требуют однопоточного выполнения для обеспечения потокобезопасности без синхронизации.
  
3. **Задачи, чувствительные ко времени ответа** (например, GUI) — не должны блокировать потоки надолго.

4. **Задачи, использующие ThreadLocal** — не подходят для пулов, так как потоки переиспользуются.

> [!NOTE] 👉 **Идея:** задачи должны быть независимыми, короткими и однородными.
> 

## 1.1 📙Взаимоблокировка потоков, вызванная голоданием

 Ситуация: задача в пуле вызывает подзадачу и ждёт её результата.
 
- В однопоточном пуле это всегда deadlock (пример — `RenderPageTask`, который ждёт хедер/футер, тоже выполняемые в том же пуле).
    
- Может возникать и в многопоточных пулах, если все потоки заняты зависимыми задачами.

> [!NOTE] 👉 **Вывод:** не пихайте взаимозависимые задачи в один ограниченный пул.
>

### ✔ Хороший пример
```java
ExecutorService exec = Executors.newFixedThreadPool(3);

Future<String> header = exec.submit(() -> "HEADER");
Future<String> footer = exec.submit(() -> "FOOTER");

exec.submit(() -> {
    try {
        String page = header.get() + " BODY " + footer.get();
        System.out.println(page);
    } catch (Exception e) {
        e.printStackTrace();
    }
});
```
## 1.2 📙 Долговременные задачи

- Долгие задачи могут «засорять» пул, ухудшая отзывчивость.
    
- Рекомендация: использовать ограниченное по времени ожидание (timeouts) вместо бесконечного.

### ❌ Плохо: блокировка без таймаута
```java
BlockingQueue<String> queue = new LinkedBlockingQueue<>();
queue.take(); // если пустая — ждёт вечно
```

```java
BlockingQueue<String> queue = new LinkedBlockingQueue<>();
String item = queue.poll(1, TimeUnit.SECONDS); // ждёт максимум 1 сек
if (item == null) {
    System.out.println("Нет данных, продолжаем работу...");
}
```

# 2. 📕 Размеры пулов потоков

Размеры пула потоков редко когда должны быть жестко заданы; вместо этого размеры пула должны предоставляться механизмом конфигурации или вычисляться динамически, с помощью метода:
```java
int N_CPUS = Runtime.getRuntime().availableProcessors();
```

1. Если пул потоков `слишком ВЕЛИК`, ==потоки конкурируют за ограниченные ресурсы ЦП и памяти==, что приводит к ==увеличению использования памяти и возможному исчерпанию ресурсов==. 
2. Если размер пула `слишком МАЛ`, ==пропускная способность падает==, поскольку ==процессорные мощности не используются в полном объёме==, несмотря на доступную работу.

> [!NOTE] Формула вывода ОПТИМАЛЬНОГО количества потоков в пуле
> 						`Nthreads = Ncpu * Ucpu * (1 + W/C)`
> 
> где:
> - Ncpu = количество CPU
> - Ucpu =  = целевая загрузка CPU, 0 ≤ Ucpu ≤ 1
> - W/С = отношение времени ожидания к времени вычисления

## Best-Practic
### ✅ **Best practice: CPU-bound задачи** (вычисления)
```java
int nThreads = Runtime.getRuntime().availableProcessors();
ExecutorService exec = Executors.newFixedThreadPool(nThreads);
```

### ✅ **Best practice: IO-bound задачи** (зависящие от операций ввода/вывода)
```java
int nThreads = Runtime.getRuntime().availableProcessors() * 2;
ExecutorService exec = Executors.newFixedThreadPool(nThreads);
```

# 3. 📕⭐⭐⭐ Конфигурирование класса ThreadPoolExecutor

> [!NOTE] Класс `ThreadPoolExecutor`
> Класс `ThreadPoolExecutor` предоставляет ==базовую реализацию для ExecurorService, возвращаемых фабричными методами== `newCachedThreadPool`, `newFixedThreadPool`, и `newScheduledThreadExecutor` класса `Executors`. 
> 
> Класс `ThreadPoolExecutor` представляет собой гибкую и надежную реализацию пула, которая позволяет выполнять различные настройки.

```java
ThreadPoolExecutor(
  int corePoolSize,
  int maximumPoolSize,
  long keepAliveTime,
  TimeUnit unit,
  BlockingQueue<Runnable> workQueue,
  ThreadFactory threadFactory,
  RejectedExecutionHandler handler)
```

## Пример использования `ThreadPoolExecutor`
```java
public static void main(String[] args) {
	// Создаём пул потоков вручную через ThreadPoolExecutor
	ThreadPoolExecutor executor = new ThreadPoolExecutor(
			2,                      // corePoolSize: держим минимум 2 потока
			4,                      // maximumPoolSize: максимум 4 потока
			10, TimeUnit.SECONDS,   // keepAliveTime: "лишние" потоки живут 10 секунд
			new ArrayBlockingQueue<>(2), // очередь на 2 задачи
			Executors.defaultThreadFactory(), // ⭐стандартная фабрика потоков
			new ThreadPoolExecutor.CallerRunsPolicy() // политика насыщения
	);
	
	// Отправляем 6 задач
	for (int i = 1; i <= 6; i++) {
		final int taskId = i;
		executor.submit(() -> { //⭐
			System.out.println(Thread.currentThread().getName() +
					" выполняет задачу " + taskId);
			try {
				Thread.sleep(3000); // имитируем работу
			} catch (InterruptedException e) {
				Thread.currentThread().interrupt();
			}
			System.out.println(Thread.currentThread().getName() +
					" завершил задачу " + taskId);
		});
	}
	
	executor.shutdown(); // инициируем завершение после выполнения всех задач
	}
}
```

## 3.1 📙Создание и удаление потока

#### 1. corePoolSize 
- ==Потоки создаются по мере нагрузки, до== `corePoolSize`.
	
- Минимальное число потоков, которые пул **старается держать постоянно**, даже если они простаивают.
    
- Если задач меньше, чем `corePoolSize`, пул всё равно создаст эти потоки.
    
- По умолчанию потоки из core-пула **не завершаются**, пока не включено `allowCoreThreadTimeOut(true)`.

👉 Практика:
- `corePoolSize` ≈ количество ядер для CPU-bound задач.
    
- Для I/O-bound можно выше, т.к. многие будут простаивать.

---
#### 2. maximumPoolSize
- Максимальное число потоков, которые могут быть созданы.
    
- ==Если очередь задач заполнена==, и потоков меньше `maximumPoolSize`, ==создаётся новый поток.==
    
- Если достигнут предел → включается политика насыщения (`RejectedExecutionHandler`).


👉 Практика:
- Должен быть больше или равен `corePoolSize`.
    
- Если очередь безлимитная → `maximumPoolSize` **не играет роли**, всегда используется `corePoolSize`.

---
#### 3. keepAliveTime
- Время, сколько поток **сверх corePoolSize** может простаивать перед завершением.
    
- ==Работает только для «лишних» потоков.==
    
- Если включено `allowCoreThreadTimeOut(true)` → распространяется и на core-потоки.
    
👉 Практика:
- Для короткоживущих пиковых нагрузок — держать маленьким (10–60 сек).
    
- Для сервисов с постоянным фоном — можно ставить большим.

---
#### 4. unit
- Единица измерения `keepAliveTime`.
    
- Тип: `TimeUnit` (например, `TimeUnit.SECONDS`).

---

#### 5. workQueue
==Очередь задач, ожидающих выполнения==. Варианты:

- **SynchronousQueue** → задачи передаются сразу потоку (используется в `newCachedThreadPool`). Если потока нет → создаётся новый (до max).
    
- **LinkedBlockingQueue** (по умолчанию без лимита) → задачи копятся в очереди, новые потоки не создаются. ⚠️ риск OOM.
    
- **ArrayBlockingQueue** (фиксированный размер) → сбалансированный вариант.

👉 Практика:
- Для стабильной нагрузки → ограниченная очередь (`ArrayBlockingQueue`).
    
- Для очень динамической нагрузки → `SynchronousQueue`.

---
#### 6. threadFactory
- ==Фабрика для создания новых потоков==.
    
- Можно задать имя, приоритет, daemon-флаг, `UncaughtExceptionHandler`.
    
- По умолчанию используется `Executors.defaultThreadFactory()`.

👉 Практика:
- Лучше всегда использовать кастомную фабрику, чтобы давать потокам осмысленные имена.

---
#### 7. handler (RejectedExecutionHandler)

Политика, ==если задача **не может быть принята**== (очередь переполнена + достигнут максимум потоков):
- `AbortPolicy` → бросает `RejectedExecutionException`.
	
- `CallerRunsPolicy` → задача выполняется в том же потоке, который вызвал `submit()`.
	
- `DiscardPolicy` → задача тихо игнорируется.
	
- `DiscardOldestPolicy` → удаляется самая старая задача из очереди.
        

👉 Практика:
- Для «важных» задач → `CallerRunsPolicy`.
    
- Для «слабозначимых» → `DiscardPolicy`.

## 3.2 📙 Управление задачами в очереди

- Очереди:
    - **Неограниченные** (`LinkedBlockingQueue`) → риск OOM.
        
    - **Ограниченные** (`ArrayBlockingQueue`) → нужен баланс между размером очереди и пула.
        
    - **SynchronousQueue** → задачи передаются напрямую рабочему потоку (используется в `newCachedThreadPool`).

> [!NOTE] ⭐ Приоритетная очередь
> Для большего контроля над порядком выполнения задач можно использовать класс `PriorityBlockingQueue`, который упорядочивает задачи в соответствии с заданным приоритетом. 
> 
> Приоритет может быть определен естественным порядком (если задачи реализуют интерфейс Comparable) или с помощью интерфейса Comparator.

## 3.3 📙 Политики насыщения
Когда ограниченная рабочая очередь заполняется, вступает в игру политика насыщения (saturation ThreadPoolExecutor policy). может быть 

Политика изменена насыщения путем для вызова класса метода `setRejectedExecutionHandler`.

Предоставляется ==несколько реализаций интерфейса== `RejectedExecutionHandler`, ==реализующих== ту или другую ==политику насыщения==: 
- 🔶 `AbortPolicy` - бросить непроверяемое исключение RejectedExecutionException
	
- 🔶`CallerRunsPolicy` - Только что отправленная задача выполняется не в потоке пула, а в потоке, вызвавшем метод `execute`.
	
- 🔶`DiscardPolicy` - автоматически отбрасывает только что отправленную задачу, если она не может быть поставлена в очередь на выполнение
	
- 🔶`DiscardOldestPolicy` - сбрасывает задачу, которая в противном случае была бы выполнена следующей, и пытается повторно отправить новую задачу. (==Если рабочая очередь представляет собой очередь с приоритетом, это приводит к сбросу элемента с наивысшим приоритетом. Таким образом, комбинация политики насыщения путём сброса старейшего элемента и очереди с приоритетом не является хорошей идеей.==

```java
ExecutorService exec = new ThreadPoolExecutor(
    1, 1,
    0L, TimeUnit.MILLISECONDS,
    new ArrayBlockingQueue<>(1),
    new ThreadPoolExecutor.CallerRunsPolicy() // Best practice для graceful degradation
);
```
#### ⭐ Интересный пример - Ограничение семафором
Не существует предопределённой политики насыщения, выполняющей блокировку метода execute, когда очередь заполняется

Однако, ==похожий эффект может быть достигнут с помощью семафор==а, для ограничения частоты внедрения задач, как показано в классе BoundedExecutor
```java
public class BoundedExecutor { 
	private final Executor exec; 
	private final Semaphore semaphore; 
	
	public BoundedExecutor(Executor exec, int bound) {
		this.exec = exec; 
		this.semaphore = new Semaphore(bound); 
	} 
	public void submitTask(final Runnable command) throws InterruptedException { 
		semaphore.acquire(); 
		try { 
			exec.execute(new Runnable() { 
				public void run() {
					try { 
						command.run(); 
					} finally { 
						semaphore.release(); 
					} 
				} 
			}); 
		} catch (RejectedExecutionException e) { 
			semaphore.release(); 
		} 
	} 
}
```

> [!NOTE] Особенность такого подхода
> При таком подходе используйте неограниченную очередь (нет причин ограничивать и размер очереди, и частоту внедрения) и ==установите границу на семафоре равную размеру пула плюс количеству поставленных в очередь задач==, которые вы хотите обработать, так как семафор ограничивает количество задач, как выполняемых в данный момент, так и ожидающих выполнения.

## 3.4 📙 Фабрики задач
Всякий раз, когда пул потоков должен создать поток, он делает это с помощью фабрики потоков
Фабрика потоков по умолчанию создает новый поток, не демон, без специальной конфигурации. Определяемые фабрики потоков позволяет вам настроить конфигурацию пула потоков. 

⭐⭐==Интерфейс ThreadFactory имеет один метод, newThread==, который ==вызывается== всякий раз, когда пулу потоков необходимо ==создать новый поток.==
```java
public interface ThreadFactory { 
	Thread newThread(Runnable r); 
}
```
- Можно задать имена, приоритеты, `UncaughtExceptionHandler`.
- Удобно для логирования и отладки.


```java
public class MyAppThread extends Thread {
    public static final String DEFAULT_NAME = "MyAppThread";
    private static volatile boolean debugLifecycle = false;
    private static final AtomicInteger created = new AtomicInteger();
    private static final AtomicInteger alive = new AtomicInteger();
    private static final Logger log = Logger.getAnonymousLogger();

    public MyAppThread(Runnable r) { this(r, DEFAULT_NAME); }

    public MyAppThread(Runnable runnable, String name) {
        super(runnable, name + "-" + created.incrementAndGet());
        setUncaughtExceptionHandler(
            new Thread.UncaughtExceptionHandler() {
                public void uncaughtException(Thread t, Throwable e) {
                    log.log(Level.SEVERE, "UNCAUGHT in thread " + t.getName(), e);
                }
            });
    }

    public void run() {
        boolean debug = debugLifecycle;
        if (debug) log.log(Level.FINE, "Created " + getName());
        try {
            alive.incrementAndGet();
            super.run();
        } finally {
            alive.decrementAndGet();
            if (debug) log.log(Level.FINE, "Exiting " + getName());
        }
    }

    public static int getThreadsCreated() { return created.get(); }
    public static int getThreadsAlive() { return alive.get(); }
    public static boolean getDebug() { return debugLifecycle; }
    public static void setDebug(boolean b) { debugLifecycle = b; }
}

//Использование кастомного класса
public class MyThreadFactory implements ThreadFactory {
    private final String poolName;

    public MyThreadFactory(String poolName) {
        this.poolName = poolName;
    }

    public Thread newThread(Runnable runnable) {
        return new MyAppThread(runnable, poolName);
    }
}

//🛑🛑🛑 Где-то в main 🛑🛑🛑
ExecutorService exec = Executors.newFixedThreadPool(2, MyThreadFactory);
```

## 3.5 📙Настройка экземпляра `ThreadPoolExecutor` после построения
Большинство параметров, передаваемых конструкторам ThreadPoolExecutor, также можно изменить после построения с помощью сеттеров (например, корневой размер пула потоков, максимальный размер пула потоков, время ожидания, фабрику потоков и обработчик выполнения отклонённых запросов). 

Также можно менять параметры готовый реализаций интерфейса `Executor`, для этого необходимо привести реализацию к `ThreadPoolExecutor`
```java
ExecutorService exec = Executors.newCachedThreadPool(); 
if (exec instanceof ThreadPoolExecutor) 
	//Приводив к ThreadPoolExecutor и изменяем параметр
	((ThreadPoolExecutor) exec).setCorePoolSize(10); 
else 
	throw new AssertionError("Oops, bad assumption");
```
> [!NOTE] Если нужен «неизменяемый» Executor → `Executors.unconfigurableExecutorService()`.
> Этот метод принимает существующий экземпляр `ExecutorService` и ==обертывает его, раскрывая только методы интерфейса ExecutorService==, без возможности настройки в дальнейшем

# 4. 📕Расширение класса ThreadPoolExecutor
Класс `ThreadPoolExecutor` был разработан для расширения, предоставляя несколько “хуков” для переопределения в подклассах - `beforeExecute`, `afterExecute`, и `terminated` – что может быть использовано для расширения поведения класса ThreadPoolExecutor.

> [!NOTE] Хуки класса `ThreadPoolExecutor` - это...
> **Хуки (hooks)** в классе `ThreadPoolExecutor` — это ==защищённые методы==, предназначенные ==для переопределения в подклассах==. 
> 
> Они 🛑🛑==позволяют встраивать собственный код в ключевые моменты жизненного цикла пула==🛑🛑 потоков и выполняемых им задач, не меняя основную логику работы пула.

## 🔶beforeExecute(Thread t, Runnable r)
- **Когда вызывается:** Непосредственно ==**перед** выполнением задачи== `r` в рабочем потоке `t`.
    
- **Примеры применения:**
    - **Логирование:** Запись в лог о том, какая задача и в каком потоке начинает выполняться.
        
    - **Сбор статистики:** Фиксация времени начала выполнения задачи (как в примере `TimingThreadPool`).
        
    - **Установка контекста:** Загрузка специфичных для потока данных (например, из `ThreadLocal`) перед выполнением.
        
    - **Мониторинг:** Увеличить счетчик активно выполняющихся задач.
        
- ☢**Важно:** Если этот метод бросит исключение `RuntimeException`, выполнение задачи **будет отменено**, и метод `afterExecute` вызван не будет.

## 🔶afterExecute(Runnable r, Throwable t)
- **Когда вызывается:** Непосредственно *==*после** завершения выполнения задачи== `r` в рабочем потоке. Вызывается **всегда**, независимо от того, как задача завершилась — успешно или с исключением.
    
- ❗❗**Параметр `Throwable t`:**
    - Если задача завершилась **нормально**, значение `t` будет **`null`**.
        
    - Если задача завершилась с **проверяемым исключением (`Exception`)**, оно будет передано в параметре `t`.
        
    - Если задача завершилась с **непроверяемым исключением (`Error`)**, этот метод **НЕ** будет вызван. Пул просто завершит проблемный поток и создаст новый взамен.
    
- **Примеры применения:**
    - **Логирование:** Запись в ==лог об окончании задачи== и её статусе (успех/ошибка).
        
    - **Сбор статистики:** Расчет времени выполнения задачи (путем вычитания времени старта из `beforeExecute` из текущего времени).
        
    - ==**Закрытие ресурсов:**== Закрытие соединений с БД или файлов, которые открывались в задаче.
        
    - **Обработка ошибок:** Централизованная запись в лог всех исключений, возникших в задачах.

## 🔶terminated()
- **Когда вызывается:** ==Один раз, в самом конце жизненного цикла ВСЕГО ПУЛА==, **после того как**:
    1. Был вызван метод `shutdown()` или `shutdownNow()`.
        
    2. **Все рабочие потоки завершили свою работу.**
        
    3. **Все задачи в очереди обработаны** (или отброшены при `shutdownNow()`).
        
- **Примеры применения:**
    - **Логирование:** Запись финального сообщения о том, что ==пул потоков завершил работу.==
        
    - **Сбор и вывод статистики:** Агрегация собранных в `beforeExecute`/`afterExecute` данных (например, вывод среднего времени выполнения всех задач).
        
    - **Освобождение ресурсов:** Закрытие внешних соединений или ресурсов, которые использовались пулом в целом, а не отдельными задачами.

# 5. 📕 Распараллеливание рекурсивных алгоритмов
Рекурсивные и циклические задачи можно разбить на подзадачи и отдавать в Executor, если эти задачи не зависят от результата предыдущих
## Условный пример распараллеливания итеративного цикла
```java
void processInParallel(Executor exec, List elements) { 
	for (final Element e : elements) {
		exec.execute(new Runnable() { 
			public void run() { 
				process(e); 
			} 
		}); 
	}
}
```

## Условный пример распараллеливания РЕКУРСИВНОГО цикла
```java
public void parallelRecursive(
	final Executor exec, List> nodes, final Collection results) 
{ 
	for (final Node n : nodes) { 
		exec.execute(new Runnable() { 
			public void run() { 
				results.add(n.compute()); 
			} 
		}); 
		parallelRecursive(exec, n.getChildren(), results); 
	} 
}
```
Метод выполняет обход в глубину, но вместо вычисления результата при посещении каждого узла, она отправляет задачу для вычисления результата в отдельный поток.

К моменту возвращения управления методом parallelRecursive, каждый узел дерева уже был посещен (обход по-прежнему осуществляется последовательно: параллельно выполняются только вызовы метода compute) и вычисления, выполняемые для каждого узла, уже были поставлены в очередь экземпляра Executor.

Объекты, вызывающие метод parallelRecursive, могут ожидать получения всех результатов, создав специфичный, предназначенный для обхода, экземпляр Executor и используя методы shutdown и awaitTermination
```java
public<T> Collection<T> getParallelResults(List<Node<T>> nodes) 
    throws InterruptedException 
{ 
    ExecutorService exec = Executors.newCachedThreadPool(); 
    Queue<T> resultQueue = new ConcurrentLinkedQueue<T>();
     
    parallelRecursive(exec, nodes, resultQueue);
     
    exec.shutdown(); 
    exec.awaitTermination(Long.MAX_VALUE, TimeUnit.SECONDS); 
    return resultQueue; 
}
```
